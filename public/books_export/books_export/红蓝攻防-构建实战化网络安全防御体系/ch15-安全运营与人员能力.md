## 15.1 先从一个故事讲起

### 15.1.1 "买了一堆设备，然后呢？

先给你讲个真实的故事。

有一家公司，做互联网的，
我们叫它I公司吧。
I公司前几年发展很快，
融了不少钱，
老板也挺重视安全的，
安全预算给得很足。

安全负责人一上任，
大刀阔斧，
买了一堆安全设备：
防火墙、WAF、IDS/IPS、SIEM、EDR、DLP、漏扫...
市面上叫得上名的，基本都买了。
花了好几百万。

买完之后呢？
安全团队就两个人，
一个负责人加上一个工程师。
两个人，要管这么多设备，
管得过来吗？
管不过来。
怎么办？
厂家帮忙调一下设备装上了，
策略用的都是默认策略。
SIEM装上了，
规则用的都是默认规则。
告警嘛，反正都开着，响不响就不知道了。

一开始老板问：
"我们安全做得怎么样了？"
负责人说：
"挺好的，设备都买齐了，
该有的都有了。"
老板一听，挺满意。

结果呢？
没过多久，就出事了。
被人打进来了，
偷了不少数据。
怎么发现的？
还是人家在暗网上看到的。
自己的SIEM呢？
SIEM有告警，
但一天几万条告警，
没人看，
也分不清哪个是真的。
EDR呢？
EDR也有告警，
但没人看。

出事之后复盘的时候，
大家才发现，
原来这些设备，
跟没装差不多。
- 策略没调过，
规则没优化过，
告警没人处理，
漏洞没人修，

基本等于零。

花了几百万，

该出事还是出事。

因为没有人运营啊。

## 15.2 什么是安全运营？

首先，
什么是安全运营（Security Operations）？
这个词也挺火的，
但很多人说不清楚到底是什么。

我给你用人话解释一下。

简单说，
**安全运营就是：
让你的安全体系真正转起来，
持续地发挥作用。**

什么意思呢？
- 设备买了，能不能用好？
- 规则写了，能不能优化？
- 告警出来了，能不能处理？
- 漏洞发现了，能不能修？
- 事件发生了，能不能响应？
- 体系建好了，能不能持续改进？

这些，
都是安全运营的事。

如果说安全建设是"从零到一，
搭架子、买设备、建体系，
那安全运营就是"从一到一百"，
让这个架子真正转起来，
持续地发挥价值，
持续地变得更好。

### 15.2.1 安全运营都做什么？

安全运营具体都做什么事？
我给你列一下。

```
安全运营的日常工作：

  1. 监测与告警运营
     • 盯着各种安全设备的告警
     • 分析告警是真是假
     • 真的就处置，假的就优化规则
     • 降低误报，提升准确率

  2. 漏洞运营
     • 漏洞扫出来了
     • 跟进修复
     • 跟踪进度
     • 确保漏洞都能及时修了

  3. 资产运营
     • 资产清册维护
     • 新发现的资产加进去
     • 下线的去掉
     • 保持资产准确

  4. 事件响应运营
     • 安全事件的处理
     • 应急响应
     • 溯源分析
     • 复盘改进

  5. 威胁情报运营
     • 威胁情报的收集、分析、应用
     • 新的情报来了，看看跟我们有没有关系
     • 相关的就加检测、做狩猎

  6. 红蓝对抗/渗透测试运营
     • 定期搞红蓝对抗、渗透测试
     • 发现问题，跟进整改

  7. 安全配置运营
     • 安全设备策略的维护和优化
     • 规则的更新
     • 策略的调整

  8. 合规运营
     • 等保、合规
     • 各种审计
     • 制度流程的落地

  9. 安全培训
     • 给员工做安全培训
     • 提高全员安全意识

  10. 持续改进
      • 发现问题，解决问题
      • 优化流程，优化体系
      • 让安全能力越来越强
```

看到了吧？
安全运营的事多着呢。
不是买完设备就完事了？
差远了。
设备买了只是开始，
后面运营的路长着呢。

### 15.2.2 为什么安全运营这么重要？

为什么安全运营这么重要？
几个原因。

```
原因一：设备只是工具，用得好用得用得差远了
  • 同样一个SIEM
  • 会用的人能用出花来
  • 不会用的人，就是个日志收集器
  • 设备是死的，人是活的
  • 运营得好不好，效果天差地别

原因二：安全是动态的，不是一劳永逸的
  • 新的漏洞天天出
  • 新的攻击手法天天有
  • 你今天安全，不代表明天安全
  • 不运营、不更新、不优化
  • 很快就不行了

原因三：很多问题不是技术问题，是运营问题
  • 漏洞扫出来了没人修 → 运营问题
  • 告警出来了没人看 → 运营问题
  • 制度有了不落地 → 运营问题
  • 设备有了不会用 → 还是运营问题
  • 很多安全事件，根子上是运营没跟上

原因四：安全能力是运营出来的
  • 不是买出来的
  • 不是建出来的
  • 是一天天运营出来的
  • 每天处理告警、修漏洞、处理事件
  • 一点一滴积累出来的
```

简单说，
**安全建设决定你的下限，
安全运营决定你的上限。
建设得再好，
运营跟不上，
也是白搭。

现在越来越多的公司意识到这一点，
所以安全运营越来越受重视。

## 15.3 安全运营中心（SOC）

说到安全运营，
就不得不提SOC。

SOC是什么？
Security Operations Center，
安全运营中心。

很多人以为SOC就是一个房间，
有个大屏，
几个人盯着屏幕。
不对。
SOC不是一个地方，
是一套体系，是一支团队，是一种能力。

简单说，
SOC就是专门做安全运营的团队+平台+流程的组合。
- 人：安全运营团队
- 平台：各种安全设备、SIEM、分析平台
- 流程：运营流程、响应流程

三者加起来，
就是SOC。

### 15.3.1 SOC的核心能力

一个成熟的SOC，
一般有哪些核心能力？

```
核心能力：

  1. 7x24小时监测
     • 全天候盯着告警
     • 有攻击能及时发现

  2. 告警分析与研判
     • 告警是真是假？
     • 严重不严重？
     • 能不能处置？

  3. 应急响应
     • 出事了能快速响应
     • 止损、溯源、恢复

  4. 威胁狩猎
     • 主动找威胁
     • 不是光等告警

  5. 漏洞管理
     • 漏洞的发现、修复、验证
     • 闭环管理

  6. 态势感知
     • 对整体安全态势的把握
     • 心里有数

  7. 持续改进
     • 发现问题、解决问题
     • 越运营越强
```

当然，
不是所有SOC都能做到这些。
很多公司的SOC，
就只是接接告警都做不好。
SOC的成熟度，
差别很大。

### 15.3.2 SOC的成熟度分级

业界一般把SOC的成熟度分成几级，
我给你简单讲讲，
你可以看看你们大概在哪个级别。

```
第一级：初始级
  • 基本没有专门的安全运营团队
  • 设备买了一些，但没人管
  • 告警基本没人看
  • 出事了才临时抱佛脚
  • 基本靠运气
  • 很多中小公司很多都在这一级

第二级：基本级
  • 有专门的人做安全运营
  • 主要工作就是看告警、处理漏洞
  • 有基本的应急流程
  • 能处理常见的安全事件
  • 基本能用，但很初级

第三级：规范级
  • 有完整的安全运营团队
  • 分工明确，流程完善
  • 7x24小时监测
  • 应急响应比较成熟
  • 有威胁情报、漏洞管理都有
  • 体系化运营得比较规范了

第四级：量化级
  • 不仅有流程，还能量化
  • 有各种指标、有数据
  • 用数据驱动运营
  • 能衡量安全态势感知能力
  • 持续优化，持续改进

第五级：优化级/智能级
  • 高度自动化
  • 人工智能、机器学习辅助
  • 主动防御、自动响应
  • 持续改进能力很强
  • 这是最高级，很少有公司能达到
```

大部分公司，
都在第二级到第三级之间。
能做到第四级的，
已经很厉害了。
第五级，那是传说中的存在。

不用追求最高级，
适合自己的就好。
但至少，
得往好的方向努力，
总没错。

## 15.4 安全团队怎么建？

安全运营，
最终还是靠人。
那安全团队怎么建？
需要哪些角色？

这个问题没有标准答案，
不同规模、不同阶段的公司，
团队不一样。
我给你讲一个大概的框架，
你可以参考。

### 15.4.1 不同阶段的团队配置

```
阶段一：初创期/小公司
  （几十人到一两百人

  安全团队：1-2人
  • 什么都干，全能型
  • 漏洞、运维安全、合规、渗透...啥都管
  • 身兼数职
  • 这个阶段，
    不用太细，能覆盖就行

阶段二：成长期/中等公司
  （几百人到上千人）

  安全团队：5-10人
  • 开始有分工了
  • 大概分几个方向：
    - 安全运营/分析
    - 渗透测试/红队
    - 安全开发/平台
    - 合规/风险管理
  • 每个方向一两个人
  • 开始体系化建设

阶段三：成熟期/大公司
  （几千人以上）

  安全团队：几十人上百人
  • 分工很细
  • 各种角色都有
  • 专门的团队
  • 安全运营中心
  • 渗透测试团队
  • 安全研究团队
  • 数据安全团队
  • 云安全团队
  • ......
  • 体系完整，能力全面
```

团队建设，
是一步步来的。
不用一上来就建个大团队，
根据实际情况，
慢慢发展。

### 15.4.2 常见的安全角色

安全团队里，
常见的角色有哪些？

```
1. 安全运营工程师/分析师
 （Security Analyst
  • 安全运营的主力
  • 看告警、分析事件、应急响应
  • 是SOC的核心角色

2. 渗透测试工程师/红队
  • 打攻击、做渗透、搞红队
  • 找漏洞、检验防御
  • 攻击视角的角色

3. 安全开发工程师
  • 做安全工具、安全平台开发
  • 写脚本、搞自动化
  • 提高效率、提效的

4. 安全架构师
  • 安全体系、安全架构设计
  • 系统、应用、网络安全架构
  • 偏架构和设计

5. 数据安全工程师
  • 专门做数据安全
  • 数据分类分级、DLP、数据库安全
  • 数据防泄露防护

6. 云安全工程师
  • 专门做云安全
  • 云平台、云原生安全
  • 现在越来越重要

7. 安全研究员
  • 安全管理、安全制度、流程
  • 风险管理、合规

8. 安全意识培训
  • 给员工做安全培训
  • 钓鱼演练、安全意识提升
```

角色很多，
不是每个公司都要有这些角色。
小公司可能一个人干好几个角色，
大公司分得细。
根据自己的情况来。

但有一点很重要：
**安全团队不能全是技术也不行，
得有懂业务的，
得有懂管理的，
得有懂技术的，
搭配着来。

## 15.5 人员能力怎么提升？

团队有了，
人有了，
接下来就是能力问题。

安全这个行业，
技术更新快，
攻击手法天天变，
人不学习，
很快就落伍了。

那人员能力怎么提升？

几个方法。

### 方法一：实战练兵，以战养战

最好的学习，
是实战。
- 多做应急响应，
- 多搞红蓝对抗，
- 多做渗透测试，
- 多处理安全事件。

经历得多了，
能力自然就上去了。
纸上得来终觉浅，
绝知此事要躬行。

怎么创造实战机会，
让大家轮流上，
练得多了，
就练出来了。

### 方法二：培训学习，持续学习

第二，培训和学习。

```
学习的途径很多：
  • 内部培训：互相分享，互相学习
  • 外部培训：花钱去上课
  • 线上课程：各种在线学习平台
  • 技术大会：听讲座、论坛
  • 看书：经典书
  • 做实验：自己搭环境练手
  • 打CTF：练技术
  • ......

学习的途径很多，
关键是要有学习的氛围和习惯。
```

安全这个行业，
就得活到老，学到老。
不学习，
很快就跟不上了。

### 方法三：知识沉淀，知识管理

第三，知识沉淀和知识管理。

什么意思？
就是把做过的事、
学到的东西，
都记录下来，
沉淀下来，
变成团队的知识。

```
怎么做：
  • 应急响应的案例
  • 漏洞分析的报告
  • 研究的成果
  • 常用的工具和脚本
  • 经验总结

好处：
  • 新人来了，
  • 不用再从头踩坑
  • 经验不会随着人走
  • 团队的能力越来越强
```

很多团队，
都在人脑子里，
人一走，
就没了。
这不行。
知识要沉淀下来，
变成团队的资产。

### 方法四：交流分享

第四，交流和分享。

- 跟同行交流
- 跟业界交流
- 参加社区、参加会议
- 跟其他公司交流
- 跟厂商交流

别闭门造车，
多出去看看，
知道外面是什么样，
别人是怎么做的，
才能开阔眼界，
才能进步。

## 15.6 安全运营的关键指标（KPI）

安全运营做得好不好，
怎么衡量？
得有指标，
也就是KPI。

很多人说安全不好衡量，
看不见摸不着，
不知道做得好不好。
其实不是的，
安全运营是可以量化的。

我给你一些常见的指标，
你可以参考。

```
常见的安全运营KPI：

监测检测类：
  • 告警准确率（误报率多少？）
  • 检出率（攻击能不能发现？）
  • 平均检测时间（攻击发生了多久才发现？）
  • 覆盖率（检测覆盖了多少攻击场景？）

响应处置类：
  • 平均响应时间（告警多久响应？）
  • 平均处置时间（多久能处理完？）
  • 事件关闭率
  • 应急响应成功率

漏洞管理类：
  • 漏洞修复率
  • 漏洞平均修复时间
  • 高危漏洞修复SLA达标率
  • 漏洞复发率

资产类：
  • 资产覆盖率（准确率多少？）
  • 资产更新及时率

威胁情报类：
  • 情报利用率
  • 情报命中率

运营效率类：
  • 自动化程度（多少事自动处理了？）
  • 人均处理事件数
  • MTTD（平均检测时间）
  • MTTR（平均响应时间/平均恢复时间）

风险类：
  • 安全事件数量趋势
  • 高危事件数量
  • 风险敞口变化趋势
```

指标不用太多，
找几个核心的，
能反映真实情况的就行。
关键是要有，
有了指标，
你才知道做得好不好，
才知道有没有进步。

## 15.7 给安全团队的几个建议

最后，
给安全团队和安全负责人几个建议。

### 建议一：别光搞技术，也要懂业务

很多做安全的人，
容易陷入技术思维，
整天就想着技术技术。
但其实，
安全是为业务服务的。
脱离了业务，
安全就没有价值。

要懂业务，
- 公司的业务是什么？
- 核心业务是什么？
- 业务痛点是什么？
- 业务的风险在哪里？

懂了业务，
你才知道安全该怎么做，
该把劲往哪使。
不然的话，
你觉得重要的事，
业务觉得不重要，
你推不动，
也没价值。

### 建议二：别光说不，要说怎么办

安全人，
容易犯的毛病，
就是这个不行、那个有风险，
然后呢？
怎么办？
不知道。

别光说不，
要说怎么办。
- 业务要上一个功能，你说有风险，
  那你告诉我，
  怎么既能上，又能把风险控制住？
- 要给解决方案，
  而不是只说不行。

安全不是业务的绊脚石，
是业务的助推器。
帮业务安全地把事干成，
才是你的价值。

### 建议三：自动化，把人从重复劳动中解放出来

安全运营的事很多很杂，
光靠人堆，
堆不过来的。

要搞自动化，
- 告警分析能自动处理的，
就让机器去做。
人只做机器做不了的事。
- 告警分诊自动归类，
人只处理、研究、
决策的事。

这样效率才能上来，
人才能做更有价值的事。

### 建议四：安全是全员的事，不是安全团队一个部门的事

最后，
记住一句话：
**安全是全员的事，
不是安全团队一个部门的事。**

安全团队，
别什么都自己扛，
扛不住的。

要推动全员参与，
- 开发要写安全的代码
- 运维要做安全加固
- 业务要考虑安全
- 员工要有安全意识

安全团队的，
是搭体系、定标准、
做支撑、推动、赋能。
不是自己一个人把所有事都干了。

发动群众，
全员安全，
才能真的安全。

## 15.8 本章总结

这一章我们讲了安全运营与人员能力，总结一下：

**什么是安全运营：**
- 让安全体系真正转起来，持续发挥作用
- 建设是从零到一，运营是从一到一百
- 三分技术，七分运营

**安全运营做什么：**
- 监测告警运营
- 漏洞运营
- 资产运营
- 事件响应运营
- 威胁情报运营
- 红蓝对抗运营
- 安全配置运营
- 合规运营
- 安全培训
- 持续改进

**SOC（安全运营中心）：**
- 不是一个地方，是团队+平台+流程
- 成熟度分级：初始级→基本级→规范级→量化级→优化级

**安全团队建设：**
- 不同阶段不同配置，从小到大一队慢慢发展
- 常见角色：安全分析师、渗透测试、安全开发、安全架构、数据安全、云安全、安全管理...

**人员能力提升：**
- 实战练兵，以战养战
- 培训学习，持续学习
- 知识沉淀，知识管理
- 交流分享，开阔眼界

**安全运营KPI：**
- 检测类、响应类、漏洞类、资产类、效率类、风险类
- 有指标才能衡量，才能进步

最后送你三句话：

> **技术是银，人是金。
> 安全是运营出来的，
> 不是买出来的。
> 得人对了，
> 事就成了。
