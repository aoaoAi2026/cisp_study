## 12.2 什么是应急响应？

首先，
什么是应急响应（Incident Response，简称IR）？

官方点说：
应急响应是指安全事件发生后，
为了控制事态、减少损失、查明原因、
恢复正常、防止再次发生，
而采取的一系列措施。

用人话说就是：
**出事了，
赶紧处理，
别让事情变大，
搞清楚怎么回事，
恢复正常，
以后别再出同样的事。**

### 12.2.1 应急响应的目标

应急响应的目标是什么？
按优先级来排：

```
第一优先级：止损
  • 先把攻击拦住
  • 别让损失继续扩大
  • 比如：
    - 攻击者还在偷数据，先断了他
    - 勒索病毒还在加密，先拦住
  • 止损是第一位的
  • 其他都往后放

第二优先级：保业务
  • 业务不能停太久
  • 能恢复的赶紧恢复
  • 业务停一天，损失可能几百万、几千万
  • 先让业务跑起来再说
  • 其他的可以慢慢查

第三优先级：溯源
  • 搞清楚：
    - 攻击是怎么进来的？
    - 攻击者干了什么？
    - 偷了什么东西？
    - 怎么进来的、走到哪了？
  • 溯源是为了防止下次再出事
  • 也是为了留证据

第四优先级：取证和追责
  • 保留证据
  • 如果要报警、要打官司
  • 证据链要完整
  • 这个优先级相对靠后
  • 但该做的还是要做

第五优先级：复盘改进
  • 最后，总结经验教训
  • 哪里有问题？
  • 哪里没防住？
  • 怎么改进？
  • 防止下次再出事
  • 这是最重要的长期价值
```

很多人搞反了优先级，
一上来就想溯源、想抓黑客，
结果耽误了止损，
损失越来越大。
这是不对的。

记住，
**止损永远是第一位的。**
先把火灭了，
再查是谁放的火。

### 12.2.2 应急响应的重要性

应急响应有多重要？
这么说吧：
**防护做得再好，
也有被突破的时候。
应急响应，
就是你最后一道防线。**

前面的防线都被突破了，
就剩下应急响应了。
应急响应做得好，
能把损失降到最低。
做得不好，
小事能变成大事，
大事能变成灭顶之灾。

比如同样是被勒索病毒攻击：
- 应急响应做得好的，
  几小时就恢复了，
  没多大损失。
- 应急响应做得不好的，
  业务停了一个星期，
  数据没备份，
  只能交钱，
  损失几千万。

差距就是这么大。

所以，
应急响应能力，
是安全团队的核心能力之一。
平时可以不用，
但不能没有。

## 12.3 应急响应的标准流程

应急响应，
最怕的就是乱。
一乱就出错，
一错损失就大了。

所以，
一定要有流程。
出事了按流程来，
一步一步走，
就不会出大错。

那流程是什么样的？
行业里有很多标准，
比如NIST的、ISO的、SANS的，
虽然细节不太一样，
但核心思想差不多。

我给你讲一个最常用的，
六步流程。

### 12.3.1 六步流程

```
应急响应六步流程：

  ┌──────────────────────────────┐
  │  1. 准备（Preparation）     │
  └───────────────┬──────────────┘
                  │
  ┌───────────────▼──────────────┐
  │  2. 检测与分析               │
  │  （Detection & Analysis）   │
  └───────────────┬──────────────┘
                  │
  ┌───────────────▼──────────────┐
  │  3. 遏制（Containment）     │
  └───────────────┬──────────────┘
                  │
  ┌───────────────▼──────────────┐
  │  4. 根除（Eradication）     │
  └───────────────┬──────────────┘
                  │
  ┌───────────────▼──────────────┐
  │  5. 恢复（Recovery）        │
  └───────────────┬──────────────┘
                  │
  ┌───────────────▼──────────────┐
  │  6. 复盘改进                 │
  │  （Lessons Learned）        │
  └──────────────────────────────┘
```

这六步，
就是应急响应的标准流程。
我一步一步给你讲。

---

### 第一步：准备（Preparation）

第一步，准备。
这个是平时做的，
不是出事了才做。

准备什么？

```
准备工作包括：

  1. 组织和人员
     • 谁负责应急？
     • 每个人的分工是什么？
     • 联系人是谁？
     • 有没有应急团队？
     • 外部支援（厂商、公安、律师）的联系方式有没有？

  2. 工具和资源
     • 应急需要的工具有没有？
     • 取证工具、分析工具、排查工具...
     • 需要的权限有没有？
     • 备份有没有？

  3. 流程和预案
     • 有没有应急流程？
     • 不同类型的事件，有没有对应的预案？
     • 升级路径是什么？
     • 谁拍板？怎么汇报？

  4. 数据和日志
     • 日志全不全？
     • 流量有没有存？
     • 资产清不清楚？
     • 没有这些，出事了没法查

  5. 演练
     • 平时有没有演练？
     • 真出事了能不能按流程走？
     • 演练过才知道有没有问题
```

准备工作是应急响应的基础。
平时准备得越充分，
出事的时候就越从容。
平时啥也不准备，
出事了肯定手忙脚乱。

### 第二步：检测与分析（Detection & Analysis）

第二步，检测与分析。
也就是：
发现事件，
确认是不是真的出事了，
有多大事。

```
这一步做什么：

  1. 发现事件
     • 怎么发现的？
       - 告警？
       - 用户报告？
       - 威胁狩猎？
       - 外部通报？
     • 先搞清楚发生了什么

  2. 验证确认
     • 是不是真的安全事件？
     • 还是误报？
     • 严重程度怎么样？
     • 影响范围有多大？

  3. 初步分析
     • 什么类型的事件？
       - Web入侵？
       - 勒索病毒？
       - 数据泄露？
       - 内网横向？
     • 影响了哪些资产？
     • 业务有没有受影响？
     • 严重等级是几级？

  4. 启动应急
     • 确认是安全事件了
     • 严重程度到了
     • 就启动应急响应流程
     • 通知相关的人
     • 该拉群拉群，该开会开会
```

这一步的关键是：
**快、准。**
- 发现要快
- 判断要准
- 别把误报当真，也别把真攻击当误报

判断错了，
要么小题大做，
要么大事化小，
都不好。

### 第三步：遏制（Containment）

第三步，遏制。
也就是**止损**。
先把攻击拦住，
别让损失继续扩大。

这是最紧急的一步。

```
遏制的常见措施：

  • 封IP / 封域名
    把攻击者的IP、C2域名封了
  
  • 断网 / 隔离
    被攻陷的机器，先从网络上隔离
    防止攻击者继续横向移动
  
  • 关停服务
    如果攻击还在继续，必要时先关停服务
    业务停一下总比全没了强
  
  • 改密码
    被偷的账号密码，赶紧改
    防止攻击者继续用
  
  • 临时加固
    加一些临时的防护措施
    先挡住攻击再说
```

遏制的原则是：
**先止损，再考虑其他。**
别犹豫，
犹豫一分钟，
损失就多一分钟。

当然，
遏制的时候也要注意：
- 尽量别破坏证据
- 能留的证据先留一下
- 但如果情况紧急，
  先止损再说，
  证据的事后面再说

### 第四步：根除（Eradication）

第四步，根除。
也就是**把攻击者彻底赶出去，
把后门都清干净。**

遏制只是暂时拦住了攻击，
但攻击者可能还留了后门，
可能在其他机器上还有据点。
如果不根除干净，
过两天人家又回来了。

```
根除做什么：

  1. 全面排查
     • 攻击者进来了多深？
     • 哪些机器被攻陷了？
     • 拿了哪些权限？
     • 偷了什么数据？
     • 都查清楚

  2. 清除后门
     • WebShell删了
     • 恶意进程杀了
     • 后门账号删了
     • 恶意服务/计划任务清了
     • 病毒木马杀干净
     • 所有被攻陷的机器都要清

  3. 修补漏洞
     • 攻击是怎么进来的？
     • 那个漏洞补上
     • 相关的漏洞也一起补
     • 别再被同样的方法打进来

  4. 重置凭据
     • 所有可能被泄露的密码、密钥
     • 通通换掉
     • 别心疼，总比被人继续用好
```

根除这一步，
一定要彻底。
宁可多花点时间，
也别留尾巴。
不然后患无穷。

怎么判断根除干净了没有？
一个简单的标准：
**假设攻击者还在里面，
你再找一遍，
看能不能找到他。**
如果找不到了，
可能就差不多了。

### 第五步：恢复（Recovery）

第五步，恢复。
也就是**把业务恢复正常。**

遏制了，根除了，
接下来就是恢复业务。
业务不能停太久。

```
恢复做什么：

  1. 验证干净
     • 确认被攻陷的机器都清干净了
     • 确认没有后门了
     • 确认漏洞补上了
     • 再恢复，别刚恢复又被打了

  2. 恢复系统和数据
     • 从备份恢复
     • 或者重装系统
     • 数据恢复回来
     • 确保数据完整、可用

  3. 逐步恢复业务
     • 别一下全放开
     • 先恢复一部分，观察观察
     • 没问题了再全部恢复
     • 有问题还能及时收回来

  4. 加强监测
     • 恢复之后，加强监测
     • 盯着点，看看有没有异常
     • 防止攻击者卷土重来
```

恢复的时候别着急，
确认安全了再恢复。
不然刚恢复又被打了，
那就更惨了。

### 第六步：复盘改进（Lessons Learned）

第六步，也是最后一步，
复盘改进。

事件处理完了，
业务也恢复了，
但事情还没完。
得复盘，
得改进，
防止下次再出事。

```
复盘做什么：

  1. 事件回顾
     • 把整个事件从头到尾理一遍
     • 怎么发现的？怎么处理的？
     • 时间线是怎么样的？
     • 哪里做得好，哪里做得不好？

  2. 根因分析
     • 为什么会发生这次事件？
     • 根本原因是什么？
       - 是有漏洞没补？
       - 是配置不对？
       - 是员工点了钓鱼邮件？
       - 是监测没发现？
     • 挖到根上，别停留在表面

  3. 改进措施
     • 针对发现的问题
     • 制定改进措施
     • 谁负责？什么时候完成？
     • 一条一条落地
     • 别复盘完了就完了，啥也不改

  4. 更新预案
     • 这次事件中暴露的流程问题
     • 预案里没有的
     • 赶紧补上、更新
     • 下次遇到类似的就有经验了
```

复盘是应急响应最有价值的一步。
吃一堑，长一智。
出事不可怕，
可怕的是同样的事出第二次。

每一次事件，
都是一次学习和改进的机会。
抓住这个机会，
把安全体系补得更牢。

---

六步流程讲完了。
记住这个流程，
出事了按这个来，
就不会出大错。

## 12.4 常见事件类型的处理要点

不同类型的安全事件，
处理的重点不一样。
我给你讲几种常见的，
你心里有个数。

### 12.4.1 Web入侵事件

最常见的，Web入侵。
比如网站被黑了、被挂马了、有WebShell了。

```
处理要点：
  1. 先遏制
     • 先把WebShell的访问拦了
     • 把攻击者IP封了
     • 必要时先把网站停了
  
  2. 查找入口
     • 攻击者是怎么进来的？
     • 什么漏洞？SQL注入？文件上传？
     • 找到入口，补上
  
  3. 清除后门
     • 所有WebShell都找出来删掉
     • 被篡改的文件恢复
     • 有没有其他后门？
  
  4. 评估影响
     • 数据有没有泄露？
     • 服务器有没有被拿权限？
     • 有没有横向移动到内网？
  
  5. 加固
     • 漏洞补上
     • 相关的漏洞也查一查
     • WAF规则加上
     • 代码安全审计
```

Web入侵是最常见的，
也是最容易被忽视的。
很多人觉得删个WebShell就完事了，
其实不是。
一定要找到入口，
一定要查清楚影响范围。

### 12.4.2 勒索病毒事件

现在越来越多的，勒索病毒。

```
处理要点：
  1. 赶紧遏制
     • 第一时间隔离被感染的机器
     • 断网！断网！断网！
     • 防止继续扩散
     • 勒索病毒扩散很快的，犹豫一分钟就多感染一片
  
  2. 不要急着付钱
     • 先看看有没有备份
     • 有没有解密工具
     • 付了钱也不一定能解密
     • 还会助长攻击者的气焰
  
  3. 找入口
     • 勒索病毒是怎么进来的？
     • 钓鱼邮件？漏洞？弱口令？
     • 找到入口，堵住
  
  4. 清除病毒
     • 所有被感染的机器都要清
     • 别漏了，漏一个又全完
  
  5. 恢复数据
     • 有备份的话，从备份恢复
     • 没备份的话，看能不能解密
     • 实在不行...那就只能认了
  
  6. 加固
     • 打补丁
     • 关不必要的端口
     • 加强弱口令管理
     • 备份！备份！备份！
     • 重要数据离线备份
```

勒索病毒，
最关键的是**早发现、快隔离**。
发现得早，
可能只感染了几台。
发现晚了，
整个内网都感染了，
那损失就大了。

还有，
备份真的很重要。
有备份，
勒索病毒就是个麻烦事。
没备份，
那就是灾难。

### 12.4.3 数据泄露事件

最头疼的，数据泄露。

```
处理要点：
  1. 先止损
     • 先把数据外泄的通道掐断
     • 该封IP封IP，该断网断网
     • 能少泄露一点是一点
  
  2. 评估影响
     • 哪些数据泄露了？
     • 多少条？
     • 有多敏感？
     • 影响多少用户？
     • 搞清楚严重程度
  
  3. 溯源
     • 怎么泄露的？
     • 从哪出去的？
     • 谁干的？
     • 进来多久了？
  
  4. 通知
     • 按法规要求，该通知的通知
     • 通知用户、通知监管
     • 别瞒，瞒不住的，
       被爆出来更麻烦
  
  5. 补救
     • 给用户什么补偿/保护？
     • 密码要不要强制改？
     • 要不要加监控？
  
  6. 加固
     • 数据安全防护加强
     • 访问控制加强
     • 数据防泄漏（DLP）补上
     • 数据加密、脱敏
```

数据泄露是最麻烦的，
影响大、后果严重、还容易犯法。
处理的时候一定要谨慎，
该走的流程要走，
该通知的要通知，
别自作聪明想瞒，
最后只会更糟。

### 12.4.4 内网渗透/APT事件

最严重的，内网被打穿了，
或者遇到APT了。

```
处理要点：
  1. 别急着打草惊蛇
     • 别一上来就封IP、断网
     • 不然攻击者察觉到了，
       马上销毁证据、换C2、躲起来
     • 先观察观察，
       摸清楚他在哪、干了什么、有多少据点
  
  2. 全面排查
     • 不要只查一台机器
     • 整个内网都要查
     • 横向移动到哪了？
     • 拿了哪些权限？
     • 偷了什么数据？
     • 有多少后门？
     • 一定要查全
  
  3. 统一行动
     • 查清楚了，
       找个统一的时间
     • 一起动手
       - 一起封
       - 一起清
       - 一起改密码
     • 别今天清一台，明天清一台
       打草惊蛇，让他跑了
  
  4. 彻彻底底根除
     • 这种攻击，后门肯定很多
     • 一定要查仔细、清干净
     • 别留尾巴
     • 不然过几天又回来了
  
  5. 长期监测
     • 处理完之后，长期盯着
     • 看看有没有回来的迹象
     • APT组织很执着的，
       说不定过段时间又换个方式进来
```

内网渗透/APT事件，
是最复杂、最难处理的。
如果自己搞不定，
别硬扛，
赶紧找专业的团队来帮忙。

## 12.5 溯源与取证

应急响应中，
很重要的一块，
是溯源和取证。

什么是溯源？
就是搞清楚：
- 攻击是怎么进来的？
- 攻击者干了什么？
- 走到哪了？
- 偷了什么？
- 攻击者是谁？

什么是取证？
就是把证据固定下来，
将来可能要报警、要打官司。

溯源和取证，
是个技术活，
也是个细心活。

### 12.5.1 溯源怎么搞？

溯源怎么搞？
简单说就是：
**从发现点开始，
往前找入口，
往后找影响，
把整个攻击过程串起来。**

```
溯源的一般思路：

  1. 确定出发点
     • 你是从哪发现攻击的？
     • 是一台Web服务器？
     • 还是一台办公电脑？
     • 从这台机器开始查

  2. 往前找：攻击是怎么进来的？
     • 看日志、看流量
     • 攻击者最早是什么时候来的？
     • 通过什么进来的？
     • 是Web漏洞？钓鱼邮件？弱口令？
     • 一直找到最开始的入口

  3. 往后找：攻击者干了什么？
     • 进来之后干了什么？
     • 提权了吗？
     • 横向移动了吗？
     • 到了哪些机器？
     • 拿了什么权限？
     • 偷了什么数据？
     • 留了什么后门？
     • 把所有影响都找出来

  4. 串起来：还原攻击路径
     • 把整个过程串起来
     • 做一个时间线
     • 什么时候干了什么
     • 画一张攻击路径图
     • 一目了然
```

溯源就像破案一样，
从一点线索开始，
慢慢挖，
最后还原整个真相。

### 12.5.2 取证要注意什么？

取证的话，
最重要的是**证据的完整性和合法性**。
不然到了法庭上不算数。

```
取证的基本原则：

  1. 不破坏原始证据
     • 原始的磁盘、日志、流量
     • 别直接在上面操作
     • 先做镜像、做备份
     • 在副本上分析
     • 原始证据要保留好

  2. 记录操作过程
     • 你动了什么？
     • 什么时候动的？
     • 怎么操作的？
     • 都记下来
     • 操作日志要完整

  3. 证据链完整
     • 证据从哪来的？
     • 谁取的？
     • 谁保管的？
     • 中间有没有被篡改？
     • 都要有记录
     • 链不能断

  4. 哈希校验
     • 取完证，算个哈希
     • 证明证据没被改过
     • 原始的和副本的哈希要对得上

  5. 找专业的来
     • 如果要报警、要打官司
     • 最好找专业的取证人员/机构来做
     • 自己瞎搞，证据可能就废了
     • 到时候想用都用不了
```

取证是个专业活，
如果事情比较严重，
可能要走法律程序，
那一定要找专业的来，
别自己乱搞，
把证据搞废了。

## 12.6 怎么做好应急响应？

最后，聊聊怎么才能做好应急响应。

应急响应不是出事了才去想的，
功夫在平时。

### 12.6.1 平时的准备

平时准备什么？

```
1. 建组织、明分工
  • 要有应急响应团队
  • 谁负责、谁干什么，分清楚
  • 联系人、联系方式，都要有
  • 外部支援也要有（厂商、公安、律师）

2. 定流程、写预案
  • 应急流程要有
  • 不同类型的事件，要有预案
  • 越详细越好
  • 出事了照着做就行

3. 备好工具
  • 应急需要的工具，提前准备好
  • 取证工具、分析工具、排查脚本...
  • 要用的时候能马上拿到

4. 日志和数据
  • 日志要全、要存好
  • 流量最好也存
  • 资产要清楚
  • 这些是溯源的基础

5. 备份！备份！备份！
  • 重要数据一定要备份
  • 还要有离线备份
  • 备份要定期测，确保能用
  • 很多时候，备份是最后一根稻草

6. 演练
  • 定期做应急演练
  • 模拟各种事件
  • 看看流程顺不顺
  • 看看大家会不会处理
  • 练过了，真出事才不慌
```

平时准备得越充分，
出事了就越从容。
应急响应，
拼的就是平时的积累。

### 12.6.2 几个重要原则

应急响应的时候，
有几个原则要记住。

```
原则一：止损优先
  • 先止损，再溯源
  • 先把火灭了，再查谁放的火
  • 别搞反了

原则二：保持冷静
  • 越急越容易出错
  • 按流程来，一步一步走
  • 别乱操作，把证据搞没了

原则三：及时升级
  • 搞不定的，赶紧升级
  • 该找领导找领导，该找外援找外援
  • 别硬扛，越扛损失越大

原则四：沟通要及时
  • 该汇报的汇报
  • 该通知的通知
  • 别捂着、别瞒着
  • 瞒不住的，最后更麻烦

原则五：记录全过程
  • 从发现到处理到结束
  • 每一步都记录下来
  • 时间、人物、操作、结果
  • 事后复盘要用
  • 真要追责也有依据
```

这几个原则，
看起来简单，
但真出事的时候，
很多人就忘了。
时刻记着点，
能少犯很多错。

## 12.7 本章总结

这一章我们讲了应急响应，总结一下：

**什么是应急响应：**
- 出事了怎么处理
- 目标：止损 → 保业务 → 溯源 → 取证 → 复盘改进
- 止损永远是第一位的

**应急响应六步流程：**
1. 准备（平时做）
2. 检测与分析（发现、确认、定级）
3. 遏制（止损，先拦住攻击）
4. 根除（清后门、补漏洞、换密码，彻底清干净）
5. 恢复（业务恢复正常）
6. 复盘改进（总结经验、改进提高）

**常见事件处理要点：**
- Web入侵：找入口、清后门、补漏洞
- 勒索病毒：快隔离、看备份、别着急付钱
- 数据泄露：先止损、评估影响、按规定通知
- 内网渗透/APT：别打草惊蛇、全面排查、统一行动、彻底根除

**溯源与取证：**
- 溯源：从发现点往前找入口、往后找影响，还原整个攻击过程
- 取证：不破坏原始证据、记录过程、证据链完整、哈希校验

**怎么做好应急响应：**
- 平时准备：组织、流程、工具、日志、备份、演练
- 重要原则：止损优先、保持冷静、及时升级、沟通及时、记录全过程

最后送你一句话：

> **平时多流汗，
> 战时少流血。
> 应急响应，
> 功夫在平时。**
