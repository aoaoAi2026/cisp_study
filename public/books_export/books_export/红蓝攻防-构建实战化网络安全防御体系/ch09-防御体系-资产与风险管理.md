## 9.1 先从一个故事讲起

### 9.1.1 消失的"幽灵服务器"

先给你讲个真实的故事。

有一家公司，做电商的，我们叫它E公司吧。
E公司发展很快，
几年时间，从几十人发展到了上千人。
IT系统也从几台服务器，
变成了几百台服务器、几十个业务系统。

安全负责人是新来的，姓刘，我们叫他老刘。
老刘刚到任，
想先摸摸底，看看公司有多少资产。
他让运维提供一份资产清单。
运维给了他一份Excel，
上面列着200多台服务器、30多个业务系统。

老刘觉得不对啊，
他平时观察，
感觉远不止这么多。
于是他自己动手，
用了几个工具，
扫了扫公司的IP段，
又查了查域名、查了查云资产。

这不扫不知道，一扫吓一跳。

扫出来多少？
**400多台服务器，60多个业务系统。**
比运维给的清单，多了将近一倍！

多出来的这些是什么？
- 项目做完了没人管的"僵尸服务器"
- 各个部门自己偷偷买的云服务器
- 测试环境、演示环境
- 外包开发的系统，交完活就没人管了
- 还有几台，连谁的都不知道

老刘头都大了：
"我的天，这一半资产都不在清单里，
那安全防护怎么做？
漏洞扫描扫不到，
安全策略覆盖不到，
监控也没有，
这不就是裸奔吗？"

果然，
没过多久，
其中一台没人管的老系统就被黑了。
攻击者从那台老系统打进来，
跳到了内网，
幸好发现得还算及时，
没造成太大损失。

出事之后，
老板终于重视了。
给老刘授权，
让他牵头做资产梳理，
把所有资产都摸清楚。
花了好几个月，
才把家底差不多摸清楚。

这个故事是不是听着很耳熟？

资产不清，
是很多公司安全建设的通病。
- 不知道自己有多少资产
- 不知道哪些是重要的
- 不知道哪些有漏洞
- 不知道哪些没人管

这样的状态下，
安全建设就是空中楼阁。
你以为你防住了，
其实到处都是窟窿。

所以说，
**资产是安全的基础。
家底摸不清，
一切都是白搭。**

这一章，我们就来聊聊，
怎么做好资产和风险管理。

## 9.2 为什么资产是防御的基础？

很多人不理解：
"安全不就是买设备、建防护吗？
跟资产有啥关系？"

关系大了去了。
我给你讲讲为什么。

### 9.2.1 你不知道的资产，就是你最大的风险

首先第一个，
**未知的资产，是最大的风险。**

什么意思？
就是你不知道它存在的资产，
就是最危险的。

为什么？
因为：
- 漏洞扫描不会扫它
- 安全策略不会覆盖它
- 监控不会盯着它
- 补丁不会给它打
- 出了事你也不知道

它就像一个隐形的窟窿，
攻击者随时能从这钻进来，
而你完全不知道。

就像前面故事里的那些"幽灵服务器"，
没人管、没人问，
裸奔在公网上，
不打你打谁？

安全防护的前提，
是你得知道你要保护什么。
你都不知道它的存在，
你怎么保护它？

### 9.2.2 没有资产，就没有风险评估

第二个，
**没有资产，就没法做风险评估。**

风险评估是啥？
就是看看你哪里有风险、风险有多大。

但你连自己有多少资产都不知道，
你评估什么呢？
你都不知道你有什么，
怎么知道哪里有风险？

就像医生给人看病，
你连病人有哪些器官、
哪些器官重要都不知道，
你怎么看病？

资产是风险评估的基础。
资产搞清楚了，
才能一个一个评估风险，
才能知道哪里最危险，
才能知道先修哪里。

### 9.2.3 没有资产，安全就是一笔糊涂账

第三个，
**没有资产，安全投入就是一笔糊涂账。**

老板问你：
"我们今年安全预算100万，
都花在哪了？
值不值？"

你怎么回答？
"买了防火墙、买了WAF、买了SIEM..."
然后呢？
保护了哪些资产？
降低了多少风险？
不知道。

因为你连自己有多少资产都不知道，
你当然不知道钱花得值不值。

有了资产清单，
你就能算清楚：
- 我们有多少重要资产
- 这些资产面临什么风险
- 我们的防护覆盖了多少
- 每一分钱花在了哪里
- 降低了多少风险

清清楚楚，明明白白。
老板也知道钱花得值不值。

### 9.2.4 资产是所有安全工作的起点

最后一句话总结：
**资产是所有安全工作的起点。**

- 漏洞管理，得先有资产
- 安全加固，得先知道加固什么
- 安全监控，得知道监控哪些
- 应急响应，得知道影响了哪些资产
- 风险评估，得基于资产来评

一切的一切，
都从资产开始。

资产不清，
后面的所有工作都是空中楼阁。

所以，
做安全的第一步，
不是买设备，
不是建SOC，
是先把家底摸清楚。

## 9.3 资产梳理：怎么摸清家底？

资产这么重要，
那怎么梳理资产呢？
怎么把家底摸清楚？

我给你讲讲方法。

### 9.3.1 资产都有啥？

首先，
得先搞清楚，
什么是资产？
我们要梳理哪些东西？

很多人以为资产就是服务器，
不对。
资产的范围广着呢。

```
常见的资产类型：

一、网络资产
  • IP地址（公网IP、内网IP）
  • 域名（主域名、子域名）
  • 网络设备（路由器、交换机、防火墙...）
  • 安全设备（WAF、IDS/IPS、VPN...）

二、主机资产
  • 服务器（物理机、虚拟机、云服务器）
  • 终端（PC、笔记本、办公电脑）
  • 移动设备（手机、平板）
  • 操作系统（Windows、Linux、Unix...）

三、应用资产
  • Web应用（网站、后台、管理系统...）
  • 移动应用（APP、小程序...）
  • API接口
  • 中间件（Nginx、Tomcat、WebLogic...）
  • 数据库（MySQL、Oracle、SQL Server...）

四、数据资产
  • 用户数据
  • 业务数据
  • 财务数据
  • 代码
  • 文档
  • 等等

五、人员资产
  • 员工账号
  • 管理员账号
  • 第三方账号
  • 等等

六、其他
  • 云资产（云服务器、云存储、CDN...）
  • 第三方服务（外包、供应商、合作伙伴...）
  • 物理资产（机房、设备...）
  • ......
```

资产的范围非常广，
不只是服务器。
只要是有价值的、
需要保护的，
都是资产。

当然，
一开始不用追求大而全，
先把最核心的、
跟网络安全相关的资产梳理清楚，
再慢慢扩展。

### 9.3.2 资产梳理的方法

怎么梳理资产？
我给你几个方法，
结合着用。

```
方法一：人工盘点
  • 找运维、找开发、找各个部门
  • 一个一个问，一个一个统计
  • 优点：准确、详细
  • 缺点：慢、费人力、容易漏

方法二：工具扫描
  • 用扫描器扫网段
  • 用网络空间搜索引擎搜（FOFA、Shodan...）
  • 子域名枚举
  • 端口扫描、服务识别
  • 优点：快、全面、不容易漏
  • 缺点：不够详细，需要人工确认

方法三：系统对接
  • 跟CMDB（配置管理数据库）对接
  • 跟云平台对接，自动拉取云资产
  • 跟OA、HR系统对接，同步人员信息
  • 优点：自动化、实时更新
  • 缺点：需要有这些系统，前期建设成本高

方法四：流量分析
  • 分析网络流量
  • 看有哪些IP在通信
  • 看有哪些应用在跑
  • 能发现很多你不知道的资产
  • 优点：能发现"幽灵资产"
  • 缺点：需要流量分析能力
```

一般来说，
这几种方法结合着用：
1. 先人工盘点，搞个大概
2. 再用工具扫，查漏补缺
3. 能对接系统的就对接，自动化
4. 用流量分析，找出没发现的资产

多管齐下，
才能摸得比较全。

### 9.3.3 资产梳理的难点

资产梳理说起来简单，
做起来难。
难点在哪？

```
难点一：资产太多，变化太快
  • 公司大了，资产成百上千
  • 今天加几台，明天下几台
  • 天天在变，很难保持准确

难点二：部门墙
  • 各个部门都有自己的资产
  • 有的不想让你知道
  • 有的嫌麻烦，不配合
  • 跨部门协调难

难点三："影子IT"
  • 部门自己偷偷买的云服务器
  • 自己搞的小系统
  • 不上报、不备案
  • 你根本不知道它的存在

难点四：历史遗留
  • 老系统、老设备
  • 谁的？干啥的？没人知道
  • 也不敢随便停
  • 就那么一直跑着
```

这些都是实际工作中会遇到的问题。
没有什么完美的解决办法，
只能一点点推，
领导支持很重要，
制度流程也很重要。

### 9.3.4 怎么保持资产的准确性？

资产梳理不是一次性的工作，
不是说今天梳理完了就完事了。
资产天天在变，
今天加几台，明天下几台，
过一段时间，
你的资产清单就不准了。

那怎么保持准确性？

```
几个关键点：

1. 建立资产管理制度
   • 新上资产必须登记备案
   • 下线下线也要走流程
   • 不登记就不能上线
   • 有制度，才有依据

2. 自动化发现
   • 定期用扫描器扫
   • 跟云平台、CMDB对接
   • 自动发现新增资产
   • 自动更新资产信息
   • 减少人工，减少出错

3. 定期盘点
   • 每个季度或者每半年
   • 全面盘点一次
   • 人工核对一遍
   • 看看有没有遗漏
   • 看看信息准不准

4. 责任到人
   • 每个资产都要有负责人
   • 谁的资产谁负责
   • 信息不准了找负责人
   • 跟考核挂钩
   • 大家才会上心
```

资产梳理是个长期工作，
不是一劳永逸的。
得有制度、有工具、有流程，
才能一直保持比较准确。

## 9.4 资产分级：哪些重要？哪些不重要？

资产梳理完了，
有了资产清单，
接下来要干啥？
给资产分级。

为什么要分级？
因为资产不是平等的，
有的重要，有的不重要。
- 核心业务系统，那是命根子
- 一个测试系统，就算被打了也没啥事

你总不能把所有资产都按一个标准来防护吧？
那成本也太高了，
也没必要。

好钢要用在刀刃上，
把钱和精力，
花在最重要的资产上。

### 9.4.1 怎么给资产分级？

资产怎么分级？
一般是按重要性来分。
重要性看什么？
看**价值**和**影响**。

```
评估资产重要性，看几个方面：

一、业务价值
  • 这个资产支撑什么业务？
  • 业务有多重要？
  • 是核心业务还是边缘业务？
  • 没了它，业务能不能转？

二、数据价值
  • 上面存了什么数据？
  • 数据敏不敏感？
  • 数据量大不大？
  • 泄露了会怎么样？

三、影响范围
  • 出问题了影响多大？
  • 影响多少用户？
  • 影响多少业务？
  • 会不会影响其他系统？

四、经济损失
  • 出问题了会损失多少钱？
  • 直接损失多少？
  • 间接损失多少？
  • 声誉损失多少？
```

根据这些维度，
给资产打个分，
分成不同的等级。

一般分几级呢？
三级比较常见：
- **核心资产（重要）**：出了问题天塌下来那种
- **重要资产（中等）**：有影响，但还能接受
- **一般资产（普通）**：出点事也没啥大不了的

当然，
也可以分四级、五级，
看你们公司的情况。
不用太复杂，
够用就行。

### 9.4.2 分级有啥用？

资产分级有啥用？
用处大了。

```
用处一：指导安全投入
  • 核心资产，重点保护，多投钱
  • 一般资产，普通防护就行
  • 好钢用在刀刃上
  • 不浪费钱

用处二：指导优先级
  • 漏洞修复，核心资产的漏洞先修
  • 安全加固，核心资产先做
  • 监控告警，核心资产的告警优先级高
  • 应急响应，核心资产的事件先处理

用处三：指导风险评估
  • 同样一个漏洞
  • 在核心资产上，就是高危
  • 在一般资产上，可能就是中危
  • 风险是相对的，看在谁身上

用处四：业务连续性
  • 核心资产，要有备份、有容灾
  • 出问题了要能快速恢复
  • 一般资产，就没那么高要求
```

资产分级，
是很多安全工作的基础。
有了分级，
你才知道先干什么、后干什么，
才知道钱该往哪花。

不然的话，
胡子眉毛一把抓，
什么都想做，
什么都做不好。

## 9.5 风险管理：把钱花在刀刃上

资产搞清楚了，
也分级了，
接下来就是风险管理。

什么是风险管理？
简单说就是：
**搞清楚哪里有风险、风险有多大、
哪些先处理、哪些后处理、
怎么处理、花多少钱。**

风险管理的目标，
不是消灭所有风险（也不可能），
而是把风险控制在可接受的范围内，
用最少的钱，
获得最大的安全效果。

### 9.5.1 风险是怎么算的？

风险怎么评估？
怎么判断风险大还是小？

简单的公式：

> **风险 = 威胁 × 漏洞 × 资产价值**

什么意思？
- **威胁**：外部的攻击者有多厉害？发生攻击的可能性有多大？
- **漏洞**：你自己有多少漏洞？有多严重？
- **资产价值**：这个资产有多重要？出问题了损失有多大？

三个乘起来，
就是风险的大小。

举个例子：
- 核心数据库（资产价值高）
- 有个SQL注入漏洞（漏洞严重）
- 经常有攻击者打我们（威胁高）
→ 风险 = 高×高×高 = 极高，必须马上修

再举个例子：
- 一个测试用的小网站（资产价值低）
- 有个XSS漏洞（漏洞中等）
- 没什么人知道这个站（威胁低）
→ 风险 = 低×中×低 = 低，有空再修

就这么个意思。
风险不是绝对的，
是相对的。
同样一个漏洞，
在核心系统上就是天大的事，
在测试系统上可能就不是事。

### 9.5.2 风险都有哪些处置方式？

风险评估完了，
知道哪些风险高、哪些低，
接下来怎么处理呢？

风险处置有四种常见方式：

```
第一种：规避（避免）
  • 就是：不干了，躲开这个风险
  • 比如：这个系统风险太高，我们直接下线
  • 再比如：这个功能太危险，我们关了不用
  • 优点：彻底消除风险
  • 缺点：业务可能受影响

第二种：缓解（降低）
  • 就是：想办法降低风险
  • 比如：有漏洞，打补丁修了
  • 再比如：加个WAF，降低攻击成功的概率
  • 这是最常见的处理方式
  • 把风险降到可接受的范围内

第三种：转移
  • 就是：把风险转给别人
  • 比如：买安全保险，出事了保险公司赔
  • 再比如：把这块业务外包出去，风险也转出去了
  • 风险还在，只是不由你承担了

第四种：接受
  • 就是：这个风险我认了，不处理了
  • 为什么？因为风险太小，处理成本太高
  • 或者处理不了，只能接受
  • 比如：低危漏洞，修起来费半天劲，不修也没啥事
  • 但接受风险要领导拍板，不能自己偷偷接受
```

四种方式，
根据风险的大小、
处理的成本、
业务的影响，
来选最合适的方式。

不是所有风险都要修，
也不是所有风险都能修。
把钱花在刀刃上，
把高风险的处理了，
低风险的该接受就接受。

### 9.5.3 风险管理的流程

风险管理不是一次性的，
是个循环的过程。

```
风险管理的PDCA循环：

P（计划）：
  • 资产梳理
  • 风险评估
  • 定优先级
  • 制定处理计划

D（执行）：
  • 按照计划去做
  • 该修漏洞修漏洞
  • 该加防护加防护
  • 该接受的接受

C（检查）：
  • 检查处理效果
  • 风险降下来了吗？
  • 有没有新的风险？
  • 计划执行得怎么样？

A（改进）：
  • 总结经验教训
  • 优化流程
  • 持续改进
  • 然后进入下一轮循环
```

一直循环，
持续改进。
风险是动态的，
今天没风险，
不代表明天没风险。
新的漏洞天天在出，
新的攻击天天在有。
风险管理，
永远在路上。

## 9.6 漏洞管理：风险管理的重头戏

风险管理里，
最常见、也最重要的一块，
就是漏洞管理。

漏洞是最直接的风险，
也是最好量化的。
很多公司的风险管理，
其实就是从漏洞管理开始的。

那漏洞管理体系怎么建？

### 9.6.1 漏洞从哪来？

首先，
漏洞从哪来？
你怎么知道有漏洞？

```
漏洞来源：
  1. 漏洞扫描器
     • Nessus、AWVS、OpenVAS...
     • 定期扫，自动扫
     • 最常见的漏洞来源

  2. 渗透测试
     • 人工渗透测试
     • 能发现扫描器发现不了的漏洞
     • 比如逻辑漏洞、业务漏洞

  3. 红蓝对抗
     • 红队打进来的路径
     • 利用的漏洞
     • 都是真实的、高风险的

  4. 安全厂商/情报
     • 厂商通告的新漏洞
     • 威胁情报
     • 0day预警

  5. 内部反馈
     • 员工、开发、运维发现的
     •  SRC（安全应急响应中心）收来的
     • 白帽子提交的
```

漏洞的来源很多，
要多管齐下，
才能发现得比较全。

### 9.6.2 漏洞怎么分级？

漏洞发现了，
接下来要分级。
哪些高危、哪些中危、哪些低危？

怎么分？
一般看几个方面：
- **严重程度**：漏洞本身有多严重？
- **利用难度**：好不好利用？
- **影响范围**：影响多少资产？
- **资产价值**：在什么资产上？

行业里一般用CVSS评分，
但那个太复杂了，
一般公司不用那么精确，
简单分个高、中、低，
就够用了。

```
简单的分级标准（参考）：

高危漏洞：
  • 能直接getshell、远程代码执行
  • SQL注入（能拖库的）
  • 严重的逻辑漏洞（能直接拿钱、拿权限）
  • 核心系统的严重漏洞

中危漏洞：
  • 普通的SQL注入、XSS
  • 越权访问
  • 敏感信息泄露
  • 普通的系统漏洞

低危漏洞：
  • 反射型XSS（没什么危害的）
  • 信息泄露（不敏感的）
  • 版本过低（暂时没漏洞的）
  • 最佳实践类的建议
```

当然，
这只是个参考，
具体怎么分，
看你们公司的情况。
但一定要有标准，
不能凭感觉。

### 9.6.3 漏洞修复的SLA

漏洞分级了，
接下来就是修。
但不是所有漏洞都要马上修，
也修不过来。
得有个时间要求，
也就是SLA（服务水平协议）。

```
参考的SLA（可以根据实际情况调）：

紧急漏洞（可以直接被利用的严重0day）：
  • 24小时内修复
  • 或者马上上防护措施

高危漏洞：
  • 7天内修复
  • 核心系统的高危漏洞，优先修

中危漏洞：
  • 30天内修复
  • 按计划排期修

低危漏洞：
  • 90天内修复
  • 或者攒一批一起修
  • 甚至可以接受风险
```

有了SLA，
就有了依据。
什么时候该修完，
清清楚楚。
到时间没修完，
就升级、就通报。
不然的话，
漏洞放那半年没人管，
跟没发现一样。

### 9.6.4 漏洞管理的闭环

漏洞管理，
最重要的是闭环。
什么叫闭环？
就是从发现，
到修复，
到验证，
到关闭，
整个流程要完整。

```
漏洞管理的完整流程：

1. 发现
   • 扫描、渗透、情报...
   • 把漏洞找出来

2. 确认
   • 漏洞是不是真的？
   • 误报的排除
   • 分级、定优先级

3. 派发
   • 派给对应的负责人
   • 哪个系统的找谁
   • 定好修复时间

4. 修复
   • 负责人去修漏洞
   • 打补丁、改代码、加防护...

5. 验证
   • 修完了，再扫一遍
   • 看看是不是真的修好了
   • 有没有修出新问题

6. 关闭
   • 验证通过，关闭漏洞
   • 记录归档

7. 复盘（定期）
   • 这段时间有多少漏洞？
   • 修复率怎么样？
   • 有没有逾期的？
   • 为什么会有这些漏洞？
   • 怎么从根源上减少？
```

整个流程跑通了，
漏洞管理才算真正落地了。
不然的话，
扫出来一堆漏洞，
没人修，
扫了也白扫。

## 9.7 本章总结

这一章我们讲了资产与风险管理，
给你总结一下：

**为什么资产是基础：**
1. 未知的资产，是最大的风险
2. 没有资产，就没法做风险评估
3. 没有资产，安全投入就是糊涂账
4. 资产是所有安全工作的起点

**资产梳理：**
- 资产类型：网络、主机、应用、数据、人员、其他...
- 梳理方法：人工盘点 + 工具扫描 + 系统对接 + 流量分析
- 难点：变化快、部门墙、影子IT、历史遗留
- 保持准确：制度 + 自动化 + 定期盘点 + 责任到人

**资产分级：**
- 为什么分级：资产价值不一样，不能一概而论
- 怎么分：看业务价值、数据价值、影响范围、经济损失
- 分几级：一般分三级（核心、重要、一般）就够用
- 有啥用：指导投入、指导优先级、指导风险评估、指导业务连续性

**风险管理：**
- 风险公式：风险 = 威胁 × 漏洞 × 资产价值
- 处置方式：规避、缓解、转移、接受
- 流程：PDCA循环，持续改进

**漏洞管理（风险管理的重头戏）：**
- 漏洞来源：扫描器、渗透测试、红蓝对抗、情报、内部反馈
- 漏洞分级：高危、中危、低危
- 修复SLA：紧急24h、高危7天、中危30天、低危90天
- 闭环管理：发现→确认→派发→修复→验证→关闭→复盘

最后送你一句话：

> **先搞清楚你有什么，
> 再搞清楚哪里有问题，
> 然后针对性地去改。
> 这就是安全建设最朴素的道理。**
