## 11.2 什么是威胁狩猎？

说了半天，
到底什么是威胁狩猎（Threat Hunting）？

官方一点的定义：
威胁狩猎是一种主动的、
有假设驱动的安全活动，
通过分析各种数据，
来发现那些逃避了自动化检测的威胁。

用人话说就是：
**你已经有了防火墙、杀毒、SIEM，
但你知道这些东西不是万能的，
肯定有漏掉的攻击。
所以你主动出击，
去翻日志、分析流量、看数据，
把那些藏起来的攻击者找出来。**

就像猎人打猎一样，
不是等猎物自己送上门，
而是主动进山，
去找猎物的踪迹。
所以叫"狩猎"。

### 11.2.1 跟其他几个概念的区别

很多人容易把威胁狩猎跟其他几个概念搞混。
我给你区分一下。

```
跟告警响应的区别：
  • 告警响应：被动的，告警来了才去处理
  • 威胁狩猎：主动的，没告警也去找

  简单说：
  告警响应是"生病了去医院"
  威胁狩猎是"主动去体检，查有没有毛病"

跟渗透测试的区别：
  • 渗透测试：站在攻击者角度，主动去打
  • 威胁狩猎：站在防守方角度，在自己的地盘里找攻击者

  一个是进攻，一个是防守方主动搜索

跟应急响应的区别：
  • 应急响应：已经确定出事了，去处理
  • 威胁狩猎：不知道有没有事，去找事

  一个是救火，一个是找隐患
```

这样说，清楚了吧？

威胁狩猎是介于告警响应和渗透测试之间的一个东西。
它是防守方主动的、
假设已经被攻破了的前提下，
去寻找攻击者踪迹的活动。

### 11.2.2 为什么需要威胁狩猎？

为什么需要威胁狩猎？
几个原因。

```
原因一：没有100%的检测
  • 任何检测系统都不是万能的
  • 0day漏洞，检测不出来
  • 免杀的恶意代码，检测不出来
  • 高级攻击者，会躲着你的检测走
  • 总有漏网之鱼

原因二：高级攻击越来越多
  • 现在的攻击者越来越专业
  • APT组织、商业黑客团队
  • 人家有耐心、有技术、有资源
  • 进来了就悄悄待着，慢慢搞
  • 不触发任何告警
  • 你等告警，等到死也等不到

原因三：发现得越早，损失越小
  • 攻击者在内网待的时间越长
  • 偷的东西越多，造成的损失越大
  • 主动去找，早点发现
  • 就能早点止损

原因四：提升检测能力
  • 威胁狩猎的过程中
  • 你会发现很多检测盲区
  • 发现了，就可以补上
  • 让你的检测能力越来越强
  • 形成良性循环
```

简单说，
威胁狩猎，
就是给你的安全体系再加一层保险。
检测没发现的，
狩猎给它找出来。

现在越来越多大公司，
都在做威胁狩猎。
因为他们知道，
光靠被动检测，
是不够的。

## 11.3 威胁狩猎的三种类型

威胁狩猎，
一般分三种类型，
或者说三个层次。

### 11.3.1 情报驱动的狩猎

第一种，
情报驱动的狩猎。

什么意思？
就是先有情报，
根据情报去猎。

比如：
- 最近出了一个新的0day漏洞，
  你去看看你们系统有没有被利用的痕迹
- 威胁情报说某个APT组织最近在搞你们行业，
  你去看看有没有他们的踪迹
- 听说有个新的勒索软件在传播，
  你去查查有没有感染的迹象

特点：
- 有明确的目标
- 有明确的TTPs（战术、技术、流程）
- 相对容易，有方向
- 新人也能上手

这种是最常见的，
也是入门级的狩猎。

### 11.3.2 假设驱动的狩猎

第二种，
假设驱动的狩猎。

什么意思？
就是先提一个假设，
然后去验证这个假设对不对。

比如：
- 假设：攻击者可能通过钓鱼邮件进来了
  然后去查邮件日志、找可疑的邮件
- 假设：攻击者可能拿到了某个管理员的账号
  然后去查这个账号的登录记录、操作记录
- 假设：内网里可能有横向移动
  然后去查RDP、SMB的异常连接

特点：
- 有一个假设的方向
- 但不确定对不对
- 需要去验证
- 比情报驱动的难一点
- 需要对攻击手法比较了解

这种是进阶的狩猎。

### 11.3.3 探索性狩猎/异常狩猎

第三种，
探索性狩猎，
或者叫异常狩猎。

什么意思？
就是没有明确的目标，
也没有明确的假设，
就是去数据里找异常，
看看有没有什么不对劲的地方。

比如：
- 看看有没有奇怪的进程
- 看看有没有奇怪的网络连接
- 看看有没有异常的登录
- 看看有没有异常的用户行为
- 全凭经验和直觉去找

特点：
- 最难，最考验能力
- 没有明确方向，全靠经验
- 但也最容易发现未知的威胁
- 高手才玩得转

这种是高级的狩猎。

---

三种类型，
从易到难。
一般是从情报驱动的开始，
慢慢过渡到假设驱动，
最后才是探索性的。
别一上来就想搞探索性狩猎，
那跟大海捞针差不多，
容易打击信心。

## 11.4 威胁狩猎的一般流程

威胁狩猎怎么搞？
有没有一个标准的流程？

一般来说，
威胁狩猎大概是这么一个流程：

```
威胁狩猎流程：

  1. 明确目标/提出假设
     └─ 这次狩猎要找什么？
     └─ 是情报驱动？还是假设驱动？
     └─ 先定好方向

  2. 收集数据
     └─ 需要哪些数据？
     └─ 日志？流量？终端数据？
     └─ 把数据搞到手

  3. 分析数据
     └─ 分析、过滤、聚合
     └─ 找异常、找可疑的
     └─ 这是最核心的一步

  4. 验证/调查
     └─ 找到可疑的了，是不是真的攻击？
     └─ 深入调查，确认一下
     └─ 是误报？还是真的攻击？

  5. 响应/处置
     └─ 真的是攻击，那就按应急响应来
     └─ 该隔离隔离，该查杀查杀
     └─ 该溯源溯源

  6. 改进/复盘
     └─ 这次狩猎发现了什么问题？
     └─ 检测能力有没有盲区？
     └─ 补上检测规则
     └─ 下次不用再靠狩猎发现了，告警就能发现
     └─ 总结经验，持续改进
```

大概就是这么个流程。
从定目标，
到找数据，
到分析，
到验证，
到响应，
到改进。

其中最核心的，
是分析数据这一步，
也是最考验能力的一步。

## 11.5 威胁狩猎需要什么能力？

想做威胁狩猎，
需要具备哪些能力？
需要什么条件？

我给你列一下。

### 11.5.1 数据基础

首先，
你得有数据。
没有数据，
猎啥？

需要什么数据？
越全越好。

```
需要的数据：
  • 日志数据
    - 系统日志、安全日志
    - Web日志、应用日志
    - 安全设备日志（防火墙、WAF、IDS...）
    - 身份认证日志
    - ......
    能接的都接上

  • 网络流量数据
    - 全流量最好
    - 至少也要有NetFlow/元数据
    - 流量分析是狩猎的一大利器

  • 终端数据
    - 进程、文件、注册表、网络连接
    - EDR的数据最好
    - 终端是狩猎的重要数据来源

  • 资产数据
    - 有哪些资产？
    - 哪些重要？
    - 正常应该有什么进程？什么连接？
    - 有资产基线，才知道什么是异常

  • 威胁情报
    - 恶意IP、域名、哈希
    - 攻击手法、IOC
    - 情报驱动的狩猎，全靠这个
```

数据是威胁狩猎的基础。
没有数据，
巧妇难为无米之炊。

而且数据不光要有，
还得全、得准、得好查。
不然想找个东西找半天，
效率太低了。

### 11.5.2 工具基础

光有数据还不行，
还得有工具来分析。

需要什么工具？

```
常用工具：
  • SIEM —— 日志集中分析
  • 全流量分析平台 —— 流量分析
  • EDR平台 —— 终端数据分析
  • 威胁情报平台（TIP） —— 情报匹配
  • 大数据分析平台 —— 海量数据查询
  • 各种脚本工具 —— Python、Shell...
```

工具不用太高大上，
够用就行。
关键是要能用起来，
要能查到数据。

很多时候，
你就是用SIEM或者日志平台，
写点查询语句，
就能做很多狩猎了。

### 11.5.3 人的能力

最重要的，
还是人的能力。
威胁狩猎，
很考验分析师的水平。

需要什么能力？

```
需要的能力：
  1. 攻击知识
     • 得知道攻击者是怎么干活的
     • 知道攻击的步骤、手法
     • 知道攻击者会留下什么痕迹
     • 不然你根本不知道找什么

  2. 数据分析能力
     • 会查日志、会分析流量
     • 会写查询语句
     • 会从海量数据里找异常
     • 数据敏感度很重要

  3. 逻辑推理能力
     • 像侦探一样
     • 从蛛丝马迹中推理出真相
     • 把零散的线索串起来
     • 还原攻击过程

  4. 经验和直觉
     • 这个是练出来的
     • 狩猎做多了，看到什么不对劲
     • 一眼就能感觉出来
     • 这就是直觉

  5. 耐心和细心
     • 威胁狩猎是个细活
     • 海量数据里找线索
     • 没耐心不行
     • 不细心也不行
     • 漏掉一点线索，可能就错过了
```

威胁狩猎，
七分靠数据，
三分靠工具，
九十分靠人。
人的能力，
是最关键的。

## 11.6 常见的狩猎场景和方法

说了这么多，
那具体狩猎的时候，
都猎点啥？怎么猎？

我给你列一些常见的狩猎场景和方法，
你可以参考参考。

### 场景一：找WebShell

最常见的，找WebShell。

```
怎么找？
  1. 看Web访问日志
     • 找那些访问奇怪的文件的
     • 比如文件名很奇怪的、时间很奇怪的
     • 比如一个.php文件，之前从来没人访问过，
       突然有人访问了，还是POST请求
     • 那就可疑了

  2. 看文件系统
     • 找最近新增的Web文件
     • 找时间不对的文件
     • 找文件内容可疑的
     • 比如里面有eval、assert、base64_decode的

  3. 看Web进程
     • 看看Web进程有没有执行奇怪的命令
     • 有没有起奇怪的子进程
     • 比如nginx起了个cmd.exe，那肯定有问题

  4. 看异常流量
     • 看看有没有跟Web服务器交互的异常流量
     • 比如菜刀、蚁剑的流量特征
```

WebShell是最常见的后门，
也是狩猎的入门级项目。

### 场景二：找横向移动

第二个，找横向移动的痕迹。

```
怎么找？
  1. 看RDP登录
     • 异常的RDP连接
     • 比如一个IP登了好多台机器
     • 比如凌晨的RDP登录
     • 比如登录成功之后马上又登另一台

  2. 看SMB/PsExec
     • 异常的SMB连接
     • 比如一台机器连了好多台机器的445端口
     • 比如PsExec执行的痕迹

  3. 看WMI
     • WMI远程执行的痕迹
     • 一台机器往好多台机器发WMI请求，可疑

  4. 看Pass-the-Hash
     • 异常的登录类型
     • 大量的登录失败再成功
     • 等等

  5. 看异常的管理员登录
     • 管理员账号从异常的IP登录
     • 管理员账号登录了不常登的机器
```

横向移动是内网渗透的必经之路，
如果能在这个阶段发现，
就能止损。

### 场景三：找C2通信

第三个，找C2（命令与控制）通信。

就是找被控机器跟攻击者的服务器通信的痕迹。

```
怎么找？
  1. 看异常外联
     • 哪些机器在连外面的可疑IP/域名？
     • 尤其是那些不常见的端口
     • 尤其是那些平时不怎么上网的服务器

  2. 看心跳流量
     • C2通信一般有心跳
     • 很有规律，间隔差不多
     • 流量大小也差不多
     • 比如每5分钟发一个包，大小都是100多字节
     • 这种就很可疑

  3. 看DNS异常
     • 奇怪的DNS查询
     • 很长的随机子域名
     • 可能是DNS隧道

  4. 威胁情报匹配
     • 把所有外联的IP/域名
     • 跟威胁情报比一比
     • 看看有没有已知的恶意IP/域名
```

找到C2通信，
就找到了被控的机器，
也找到了攻击者的服务器。

### 场景四：找权限维持

第四个，找权限维持的痕迹。

就是找攻击者留的后门。

```
怎么找？
  1. 找账号
     • 有没有可疑的新增账号？
     • 有没有管理员组里多出来的账号？
     • 有没有很久不用突然激活的账号？

  2. 找启动项/计划任务
     • 有没有可疑的启动项？
     • 有没有奇怪的计划任务？
     • 尤其是指向奇怪路径的

  3. 找服务
     • 有没有可疑的服务？
     • 服务名奇怪的、路径奇怪的

  4. 找WebShell、后门文件
     • 这个前面说过了

  5. 找注册表异常
     • 注册表启动项有没有被改？
     • 有没有奇怪的注册表项？
```

攻击者进来了，
肯定会留后门。
把后门找出来，
才能防止他下次再来。

---

当然，
狩猎的场景还有很多很多，
说不完。
这里只是举几个常见的例子，
让你有个感觉。

具体猎什么，
要看你的环境、
你的威胁模型、
你的数据情况。

## 11.7 从零开始建威胁狩猎能力

说了这么多，
很多人可能会问：
"我们公司现在啥也没有，
想搞威胁狩猎，
怎么开始？"

别着急，
威胁狩猎不是一蹴而就的，
可以一步一步来。

我给你一个从零开始的路线图。

### 第一步：打好数据基础

首先，
先把数据基础打好。
- 日志先接全了
- 重要系统的日志都要有
- 最好能有终端数据、流量数据
- 数据是狩猎的基础，没有数据啥也别谈

不用追求一步到位，
先把核心的、重要的接上，
慢慢补。

### 第二步：先搞情报驱动的狩猎

一开始别搞复杂的，
先从情报驱动的狩猎开始。
- 有新漏洞出来了，去查查有没有被利用
- 有新的威胁情报，去匹配匹配
- 有新的攻击手法，去看看有没有痕迹

这种有明确目标的，
容易出成果，
也容易建立信心。

### 第三步：积累经验，慢慢扩展

做着做着，
经验多了，
数据全了，
再慢慢扩展。
- 开始做一些假设驱动的狩猎
- 开始研究一些常见的攻击场景
- 开始找一些常见的后门、常见的攻击痕迹

一步一步来，
别急。

### 第四步：形成机制，持续运营

狩猎不是搞一次就完事了，
要形成机制，
持续运营。
- 定期搞狩猎活动
- 每周、每月都有狩猎任务
- 每次狩猎完了都复盘
- 发现问题就补上检测
- 形成良性循环

慢慢的，
狩猎能力就越来越强了。

### 第五步：从"人找威胁"到"威胁找人"

最高境界是什么？
不是人天天去找威胁，
而是通过自动化的手段，
把可疑的东西主动找出来，
推送给分析师。

比如：
- 用UEBA（用户行为分析）自动找异常
- 用机器学习自动发现可疑行为
- 用自动化的狩猎剧本，定期自动跑
- 有可疑的了，再让分析师去研判

这样效率就高多了。
当然，
这个是高阶玩法，
需要一定的积累。

## 11.8 给防守方的建议

最后，给防守方几点建议。

### 建议一：别神话威胁狩猎，也别小看它

现在威胁狩猎被吹得很神，
好像什么都能发现。
其实不是。
威胁狩猎不是万能的，
它只是整个防御体系的一部分。

但也别小看它。
它确实能发现很多检测发现不了的东西。
尤其是高级威胁。

摆正心态，
它就是你安全体系中的一环，
跟监测、响应、渗透测试这些配合着来。

### 建议二：先有好的检测，再谈狩猎

很多人一上来就想搞威胁狩猎，
觉得这个高大上。
但其实，
基础的检测都没做好，
搞狩猎就是空中楼阁。

为什么？
因为：
- 连已知的攻击都检测不出来
- 你还指望能发现未知的？
- 连SIEM都没用好
- 你还能搞狩猎？

先把基础打好：
- 日志接全
- 检测规则写好
- 误报降下来
- 告警能正常处置

基础打好了，
再往上搞狩猎，
水到渠成。

### 建议三：狩猎的目的是改进检测

威胁狩猎最重要的目的是什么？
是找到攻击者吗？
是，但不全是。

更重要的是：
**通过狩猎发现检测盲区，
然后补上，
让下次同样的攻击，
不用狩猎就能告警发现。**

每狩猎一次，
都要问自己：
- 这个攻击为什么没被检测到？
- 我们的检测哪里有问题？
- 怎么补上？
- 下次能不能自动发现？

这样一次一次补，
你的检测能力就越来越强，
需要手动狩猎的就越来越少。

这才是良性循环。
不然你天天手动猎，
累死也猎不完。

### 建议四：从小处着手，循序渐进

威胁狩猎不是只有大公司才能搞，
小公司也能搞。
不用一上来就搞个庞大的狩猎团队，
不用一上来就全流量全终端。

从小处着手，
- 先从最常见的攻击开始猎
- 先从最重要的资产开始猎
- 先搞情报驱动的，再搞其他的
- 一步一步来，循序渐进

只要开始做，
就比不做强。

## 11.9 本章总结

这一章我们讲了威胁狩猎，总结一下：

**什么是威胁狩猎：**
- 主动出击，主动去找攻击者
- 不是等告警，是主动搜索
- 发现那些逃避了自动化检测的威胁

**跟其他概念的区别：**
- 跟告警响应：一个主动一个被动
- 跟渗透测试：一个防守方找，一个攻击方打
- 跟应急响应：一个找隐患，一个处理已经发生的事

**为什么需要威胁狩猎：**
1. 没有100%的检测
2. 高级攻击越来越多，会躲检测
3. 发现得越早，损失越小
4. 能提升检测能力，形成良性循环

**三种类型：**
1. 情报驱动的狩猎（入门）
2. 假设驱动的狩猎（进阶）
3. 探索性/异常狩猎（高级）

**一般流程：**
明确目标 → 收集数据 → 分析数据 → 验证调查 → 响应处置 → 改进复盘

**需要什么：**
1. 数据基础（日志、流量、终端、资产、情报）
2. 工具基础（SIEM、全流量、EDR...）
3. 人的能力（攻击知识、分析能力、推理能力、经验、耐心）

**常见狩猎场景：**
- 找WebShell
- 找横向移动
- 找C2通信
- 找权限维持/后门
- ......

**从零开始路线：**
1. 打好数据基础
2. 先搞情报驱动的狩猎
3. 积累经验，慢慢扩展
4. 形成机制，持续运营
5. 从"人找威胁"到"威胁找人"

最后送你一句话：

> **最好的防御，
> 是假设自己已经被攻破，
> 然后主动去找敌人。**
