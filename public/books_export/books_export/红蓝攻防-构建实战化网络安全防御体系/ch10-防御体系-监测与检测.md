## 📖 写在前面

上一章我们讲了资产与风险管理，
这一章我们讲监测与检测。

什么是监测与检测？
简单说就是：
**盯着你的系统和网络，
有攻击了能发现，
出问题了能知道。**

很多人觉得安全就是防——
买防火墙、买WAF、买杀毒软件，
把攻击挡在外面就行了。

但实际上呢？
没有100%的防御。
再强的防护，
也总有被突破的时候。
0day漏洞、钓鱼邮件、内鬼...
防不胜防。

所以，
除了防，
你还得能发现。
攻击发生了，
你得尽快知道，
尽快响应，
才能把损失降到最低。

发现得越早，
损失越小。
发现得越晚，
损失越大。

这一章我们就来聊聊：
- 监测与检测体系怎么建？
- 要监测哪些东西？
- SIEM怎么用？
- 检测规则怎么写？
- 误报太多怎么办？
- 怎么提高检测的准确率？

坐稳了，开唠。

## 10.1 先从一个故事讲起

### 10.1.1 "形同虚设"的SIEM

先给你讲个真实的故事。

有一家公司，做金融的，
安全投入挺大的，
买了全套的安全设备，
其中就包括一套几十万的SIEM。

SIEM是什么？
安全信息与事件管理，
说白了就是把所有日志都接过来，
集中分析，发现攻击。

这家公司的SIEM，
接了几十种设备的日志，
写了几百条检测规则，
大屏24小时亮着，
看起来挺像那么回事的。

但是呢，
实际用起来，
完全不是那么回事。

什么问题呢？
**告警太多了。**

一天多少条告警？
**几万条。**

几万条告警，
安全团队就那么几个人，
根本看不过来。
而且大部分都是误报——
不是攻击，是正常的业务。

一开始他们还认真看，
看来看去全是误报，
后来就麻木了。
告警来了，
扫一眼，哦，又是误报，忽略。
再后来，
干脆就不看了，
就让它在那响着。

SIEM买了，
规则也写了，
但实际上，
跟没有差不多。
因为告警太多，
等于没有告警。

直到有一天，
真的被攻击了。
攻击者从一个Web漏洞打进来，
在内网里横向移动，
偷了不少数据。

而SIEM呢？
其实SIEM是有告警的，
而且还不少。
但是呢，
混在几万条误报里，
根本没人注意到。
等发现的时候，
攻击者早跑了。

---

这个故事是不是听着很熟悉？
很多公司的SIEM都是这样：
- 买的时候挺期待
- 上线的时候挺热闹
- 用起来全是误报
- 最后形同虚设

问题出在哪？
是SIEM不好用吗？
不完全是。
很多时候是人的问题、方法的问题。

- 日志接得不对
- 规则写得不好
- 误报没人管
- 没人优化

最后好好一个SIEM，
就这么废掉了。

监测与检测，
不是买个SIEM、接几个日志、写几条规则就完事了。
这里面学问大着呢。

这一章，我们就来聊聊，
怎么建一个真正有用的监测与检测体系。

## 10.2 监测什么？

首先第一个问题：
我们要监测什么？
哪些东西需要盯着？

很多人以为监测就是看安全设备的告警，
不对。
要监测的东西多着呢。

我给你分分类。

### 10.2.1 网络层监测

最基础的，网络层。

```
监测什么：
  • 网络流量
  • 异常流量（大流量、异常连接）
  • 攻击流量（扫描、注入、XSS...）
  • 隧道通信（DNS隧道、ICMP隧道...）
  • 数据外泄
  • 异常连接（连到恶意IP、异常端口）

用什么监测：
  • IDS/IPS
  • 流量分析平台
  • 全流量分析
  • DDoS检测
  • 网络行为分析（NBA）
```

网络层是第一道监测线。
攻击从网络进来，
首先会在网络流量上留下痕迹。
网络层监测做好了，
很多攻击在网络层就能发现。

### 10.2.2 主机层监测

然后是主机层，
也就是服务器和终端。

```
监测什么：
  • 登录日志（成功/失败、异常时间、异常地点）
  • 进程创建（异常进程、恶意进程）
  • 文件操作（创建、修改、删除敏感文件）
  • 注册表修改（启动项、服务...）
  • 计划任务/服务变更
  • 权限变化
  • 漏洞利用痕迹
  • 恶意软件行为

用什么监测：
  • 操作系统日志
  • HIDS/EDR
  • 主机审计工具
  • 杀毒软件
  • 文件完整性检查
```

主机层的监测非常重要。
因为很多攻击，
最后都会落到主机上。
而且主机层的信息最丰富，
能看到最详细的攻击行为。

尤其是现在无文件攻击、白利用越来越多，
传统的杀毒软件查不出来，
只能靠行为监测。
EDR为什么火？
就是这个原因。

### 10.2.3 应用层监测

然后是应用层，
就是各种业务系统、Web应用。

```
监测什么：
  • Web攻击（SQL注入、XSS、文件上传...）
  • WebShell
  • 异常访问（爬虫、暴力破解、扫描...）
  • 业务异常（异常下单、异常转账...）
  • API调用异常
  • 账号异常（暴力破解、异常登录...）

用什么监测：
  • WAF日志
  • Web服务器日志
  • 应用日志
  • 业务安全监测
  • API安全监测
```

应用层的攻击最多，
也最常见。
Web攻击、业务逻辑攻击，
都在这一层。
应用层的监测，
能发现很多针对业务系统的攻击。

### 10.2.4 数据层监测

然后是数据层，
就是数据安全的监测。

```
监测什么：
  • 敏感数据访问（谁、什么时候、访问了什么）
  • 异常数据查询（批量查询、导出）
  • 数据外发（下载、外传）
  • 数据库异常操作（删表、改权限...）
  • 数据泄露

用什么监测：
  • 数据库审计
  • 数据防泄漏（DLP）
  • 数据库防火墙
  • 数据访问日志
```

数据是最核心的资产，
也是攻击者的最终目标之一。
数据层的监测，
就是盯着敏感数据，
看谁在访问、在干什么、
有没有被偷。

### 10.2.5 身份与访问监测

还有身份与访问监测，
就是盯着账号和权限。

```
监测什么：
  • 登录异常（异常时间、异常地点、异常IP）
  • 暴力破解
  • 权限提升（普通用户变管理员）
  • 异常权限变更
  • 高危操作
  • 账号共享、异常登录
  • 第三方接入异常

用什么监测：
  • 身份认证日志
  • 特权账号管理（PAM）
  • 堡垒机日志
  • IAM日志
  • UEBA（用户行为分析）
```

身份是安全的核心之一。
很多攻击到最后，
都是偷账号、用账号。
把账号管住了、盯着了，
很多攻击就能发现。

### 10.2.6 其他监测

除了上面说的，
还有很多其他的。

```
还有什么要监测：
  • 安全设备本身的状态（是不是正常运行、策略有没有被改）
  • 云安全监测（云配置、云资产、云操作...）
  • 威胁情报匹配（有没有跟恶意IP/域名通信）
  • 物理安全监测（门禁、监控...）
  • ......
```

总之，
能产生日志的、能产生数据的，
理论上都可以监测。
但实际中，
你不可能什么都监测，
人力、物力、财力都有限。

那怎么办？
**抓重点。**
- 重要资产重点监测
- 高风险的重点监测
- 攻击高发的重点监测

把钱和精力花在刀刃上。

## 10.3 SIEM：监测的大脑

说到监测与检测，
就不得不提SIEM。
SIEM是什么？
前面简单说了，
安全信息与事件管理。

简单说，
SIEM就是把所有设备、所有系统的日志，
都接到一个地方，
集中存起来，集中分析。
然后用规则、用关联分析，
发现攻击。

SIEM是监测体系的大脑。
没有SIEM，
你就是东一榔头西一棒子，
各个设备各报各的警，
你根本看不过来，
也发现不了复杂的攻击。

有了SIEM，
所有日志集中在一起，
你才能做关联分析，
才能发现跨设备、跨系统的复杂攻击。

### 10.3.1 SIEM能干嘛？

SIEM的核心功能是什么？

```
核心功能：

1. 日志收集
   • 把各种设备、系统的日志都接过来
   • 防火墙、WAF、IDS、操作系统、应用...
   • 统一收集、统一存储

2. 日志规范化
   • 不同设备的日志格式不一样
   • SIEM把它们都转成统一的格式
   • 方便分析和查询

3. 关联分析
   • 把不同来源的日志关联起来
   • 比如：
     防火墙告警 + Web服务器日志 + 数据库异常
     三个放一块一看，哦，是SQL注入攻击
   • 单条日志可能看不出问题，
     关联起来就清楚了

4. 告警
   • 有规则，命中了就告警
   • 告诉你哪里可能出问题了

5. 可视化
   • 大屏、仪表盘
   • 安全态势一目了然
   • 有多少告警、什么类型、严重程度...

6. 调查与溯源
   • 出了事，能快速查日志
   • 攻击怎么进来的？干了啥？
   • 时间线、攻击路径，都能查
```

简单说，
SIEM就是日志的大仓库，
加上分析和告警的能力。
是监测体系的核心。

### 10.3.2 SIEM为什么经常不好用？

很多公司买了SIEM，
但用得不好，
甚至形同虚设。
为什么？

常见的几个原因：

```
原因一：日志接得不全、不准
  • 日志没接全，重要的系统没接
  • 日志格式不对，解析错了
  • 垃圾日志太多，有用的被淹没了
  • 基础没打好，后面都白搭

原因二：规则写得不好
  • 规则太少，检测不出来
  • 规则太松，误报一大堆
  • 规则太死，只能检测已知攻击
  • 规则写好了就再也不改了，从不优化

原因三：误报太多，没人看
  • 一天几万条告警，全是误报
  • 看不过来，干脆不看了
  • 最后真的攻击来了也发现不了
  • 这是最常见的问题

原因四：没人运营、没人优化
  • SIEM上线了就完事了
  • 规则从来不更新
  • 误报从来不处理
  • 跟买了辆车不开，放那落灰一样

原因五：人员能力跟不上
  • SIEM很强大，但人不会用
  • 不会写规则，不会分析
  • 出了告警不知道怎么研判
  • 好工具也白搭
```

SIEM不是买了就完事了，
它是一个需要持续运营的东西。
三分建设，七分运营。
运营得好，
它就是你的火眼金睛。
运营得不好，
它就是个摆设。

### 10.3.3 怎么让SIEM真正有用？

那怎么才能让SIEM真正有用？
几个关键点：

```
第一，打好基础
  • 把日志接全、接准
  • 重要系统一个不能少
  • 日志解析要对
  • 没用的垃圾日志少接
  • 基础打牢了，后面才好办

第二，写好规则
  • 规则不是越多越好，是越准越好
  • 从高风险的开始写
  • 先写确定性高的
  • 慢慢加，慢慢优化
  • 别一上来就写几百条，全是误报

第三，治理误报
  • 误报一定要处理
  • 每条误报都要分析：为什么误报？
  • 是规则写得不好？那就优化规则
  • 是正常业务？那就加白名单
  • 误报降下来了，大家才愿意看
  • 不然全是狼来了，真的狼来了也不信了

第四，持续运营
  • 定期回顾告警：哪些准、哪些不准
  • 定期优化规则
  • 定期更新检测内容
  • 新的攻击手法出来了，赶紧加检测
  • 一直运营，一直优化

第五，提升人的能力
  • 工具再好也要人用
  • 分析师要会研判、会调查
  • 会写规则、会优化
  • 会用SIEM做溯源
  • 人的能力上去了，工具才能发挥价值
```

就这么简单？
就这么简单。
但真要做到，
不容易。
需要时间、需要耐心、需要积累。

## 10.4 检测规则怎么写？

监测与检测的核心是什么？
是检测规则。
规则写得好不好，
直接决定了检测准不准、
误报多不多。

那检测规则怎么写？

### 10.4.1 检测的三种方法

首先，
检测有三种基本方法，
或者说三种思路。

```
第一种：特征检测（Signature-based）
  • 什么意思？
    就是找"坏人的特征"
    比如：
    - 攻击载荷的特征（SQL注入的关键字、病毒的哈希）
    - 恶意IP、恶意域名
    - 工具的特征
  • 优点：准确率高，误报少
  • 缺点：只能检测已知的，未知的检测不出来
  • 就像查身份证，照片对上了就是你，
    但换个样子就认不出来了

第二种：异常检测（Anomaly-based）
  • 什么意思？
    先建立"正常"的基线
    然后偏离基线的，就是异常
    比如：
    - 正常情况这个IP一天登录10次，今天登了1000次 → 异常
    - 正常这个用户只在北京登录，突然从美国登了 → 异常
  • 优点：能检测未知攻击、0day
  • 缺点：误报多，因为"异常"不一定是攻击
  • 就像抓小偷，不一定非得看他偷东西，
    鬼鬼祟祟的就可以怀疑

第三种：行为检测/状态机
  • 什么意思？
    不看单条日志，看一连串的行为
    把攻击的过程拆成几步，
    几步都命中了，就是攻击
    比如：
    扫描端口 → 尝试登录 → 登录成功 → 创建管理员账号
    这几步按顺序都发生了，那大概率是攻击
  • 优点：准确率高，能检测复杂攻击
  • 缺点：规则复杂，写起来难
  • 就像看一个人鬼鬼祟祟、到处摸、还拿东西，
    那肯定是小偷
```

三种方法，
各有优缺点。
实际中，
一般是三种结合着用：
- 特征检测先拦掉大部分已知攻击
- 异常检测发现未知的
- 行为检测抓复杂的攻击

多管齐下，
才能检测得又全又准。

### 10.4.2 写规则的基本原则

写检测规则，
有几个基本原则。

```
原则一：从高风险的开始
  • 别一上来就写一堆低风险的规则
  • 先写最危险、最常见的攻击
  • 比如：WebShell、SQL注入、暴力破解、横向移动...
  • 先把高风险的覆盖了，再慢慢加

原则二：准确率优先
  • 宁肯少检，也别全是误报
  • 一条100%准确的规则，
    比十条50%准确的强多了
  • 误报太多，大家就不信了
  • 先准，再慢慢求全

原则三：持续优化
  • 规则不是写好了就完事了
  • 要持续优化
  • 有误报就改，有漏报就加
  • 新的攻击手法出来了就更新
  • 一直在路上

原则四：上下文很重要
  • 别光看单条日志
  • 要结合上下文
  • 同样一个行为，在普通服务器上正常，
    在域控上可能就异常
  • 同样是登录，工作时间正常，
    凌晨三点就可疑
  • 上下文越丰富，判断就越准

原则五：分级告警
  • 不是所有告警都一个级别
  • 高危的（比如WebShell、提权）—— 高优先级，马上处理
  • 中危的 —— 当天处理
  • 低危的 —— 定期看看
  • 别什么都当紧急事件，
    不然大家都疲了
```

写规则是个技术活，
也是个耐心活。
别急，
慢慢来，
慢慢优化，
会越来越准的。

### 10.4.3 常见的检测点

那一般要写哪些检测规则？
我给你列一些常见的、高价值的检测点，
你可以参考参考。

```
网络层：
  • 端口扫描
  • 暴力破解（SSH、RDP、Web...）
  • SQL注入/XSS/Web攻击
  • 异常外联（连到恶意IP/域名）
  • DNS隧道/异常DNS查询
  • 异常大流量/数据外泄
  • ......

主机层：
  • 异常登录（异常时间、异常地点、异常IP）
  • 可疑进程创建（比如mimikatz、cmd.exe的异常调用）
  • 可疑的PowerShell命令
  • 新增加管理员账号
  • 计划任务/服务变更（可疑的）
  • 注册表启动项变更
  • 敏感文件被修改
  • ......

应用层：
  • WebShell
  • 恶意文件上传
  • 异常访问后台
  • 业务逻辑异常（比如异常下单、异常转账）
  • 接口异常调用
  • ......

数据层：
  • 批量数据导出/查询
  • 敏感数据异常访问
  • 数据库异常操作（删表、改权限...）
  • 数据外发
  • ......

身份层：
  • 暴力破解
  • 异常登录
  • 权限提升
  • 高危操作
  • ......
```

这些都是比较基础、也比较高价值的检测点。
先把这些做好了，
就能发现大部分常见攻击。
然后再慢慢加，慢慢优化。

## 10.5 误报太多怎么办？

做检测的，
最头疼的就是误报。
误报太多，
人就麻木了，
真的攻击来了也发现不了。

那误报太多怎么办？
怎么降误报？

几个方法。

### 方法一：优化规则

最直接的，优化规则。

很多误报，
是因为规则写得太粗了。
比如：
- 只匹配了一个关键字，
  正常业务里也有这个关键字，
  那就误报了。

怎么优化？
- 加更多条件，缩小范围
- 结合上下文
- 排除正常的业务场景
- 规则越精确，误报越少

当然，
也不能为了降误报，
把规则写得太死，
那漏报又多了。
要在准确率和检出率之间找平衡。

### 方法二：加白名单

有些误报，
是正常的业务，
确实没法通过优化规则来消除。
那怎么办？
加白名单。

比如：
- 某个IP是漏洞扫描器，
  它扫出来的告警都是正常的，
  那就把它加白名单。
- 某个管理员经常凌晨加班，
  他凌晨登录是正常的，
  那就给他单独设个规则。

加白名单是很正常的，
不用觉得加白名单就是作弊。
关键是白名单要管起来，
不能乱加，
加了要定期回顾，
不用了就删掉。

### 方法三：分层告警/告警聚合

还有一个方法，
就是把告警分层、聚合。

什么意思？
- 单条低危的告警，先不通知人，
  攒着，或者放在低优先级的地方
- 多条相关的告警凑一块了，
  或者严重程度到了一定级别，
  再升级成人造告警

比如：
- 一次失败登录 → 忽略
- 十次失败登录 → 低危告警
- 一百次失败登录 → 中危告警
- 失败一百次之后成功了 → 高危告警，马上通知人

这样，
低危的、不确定的告警，
就不会打扰人了。
真正严重的告警，
才会拉响警报。

误报的问题就缓解了很多。

### 方法四：持续运营，持续优化

最后，
也是最根本的——
持续运营，持续优化。

误报不是一天就能降下来的，
是一天天、一条条，
慢慢优化出来的。

- 每天看告警，分析误报
- 为什么误报？怎么改？
- 改规则、加白名单
- 今天改几条，明天改几条
- 慢慢的，误报就越来越少了

这个过程急不得，
需要时间，需要积累。
但只要坚持做，
一定会越来越好。

## 10.6 给防守方的建议

最后，给防守方几点建议。

### 建议一：先建基础，再搞高大上

很多人一上来就想搞威胁狩猎、搞AI检测、搞态势感知，
但其实基础的东西都没做好。
- 日志接得不全
- 规则没几条
- 误报一大堆

别着急，
先把基础打好。
- 日志接全接准
- 把常见的攻击检测规则写好
- 把误报降下来
- 让监测体系真正能用

基础打好了，
再往上加东西。
不然就是空中楼阁。

### 建议二：检测能力要覆盖ATT&CK各阶段

之前讲过ATT&CK框架，
14个战术阶段。
你的检测能力，
最好每个阶段都有一些。
不用每个技术都覆盖，
但每个战术阶段至少要有一些检测能力。

比如：
- 初始访问阶段：有没有检测钓鱼、Web攻击的？
- 执行阶段：有没有检测恶意进程、PowerShell的？
- 横向移动阶段：有没有检测RDP、SMB、哈希传递的？
- 数据外泄阶段：有没有检测数据外发的？

每个阶段都有检测，
就能做到"层层有检测"。
就算攻击者突破了前几层，
后面还有层能发现他。
总比某一层完全没检测、
进去了就石沉大海强。

### 建议三：别光靠特征，行为检测也很重要

前面说了，
特征检测只能检测已知攻击。
现在攻击者越来越会躲了，
免杀、白利用、无文件攻击...
很多都没有明显的特征。

所以，
不能只靠特征检测。
行为检测、异常检测也要搞起来。
- 看行为模式正不正常
- 看用户习惯有没有变
- 看系统状态有没有异常

虽然误报会多一点，
但能发现很多特征检测发现不了的攻击。

### 建议四：监测体系要持续运营

最后再强调一遍：
**监测体系不是建完就完事了，
要持续运营。**

- 规则要持续优化
- 误报要持续处理
- 新的攻击手法要及时加检测
- 人员能力要持续提升
- 定期回顾、定期改进

监测体系是活的，
不是死的。
一直运营，一直优化，
它才会越来越好用。

## 10.7 本章总结

这一章我们讲了监测与检测体系，总结一下：

**监测什么：**
1. 网络层监测（流量、攻击、隧道、外泄...）
2. 主机层监测（登录、进程、文件、注册表...）
3. 应用层监测（Web攻击、业务异常、API...）
4. 数据层监测（数据访问、数据外泄、数据库操作...）
5. 身份与访问监测（登录异常、权限变化、高危操作...）

**SIEM——监测的大脑：**
- 核心功能：日志收集、规范化、关联分析、告警、可视化、调查溯源
- 为什么不好用：日志不全、规则不好、误报太多、没人运营、人不行
- 怎么用好：打好基础、写好规则、治理误报、持续运营、提升能力

**检测规则：**
- 三种方法：特征检测、异常检测、行为检测/状态机
- 基本原则：从高风险开始、准确率优先、持续优化、上下文、分级告警
- 常见检测点：网络层、主机层、应用层、数据层、身份层

**误报太多怎么办：**
1. 优化规则，写得更精确
2. 加白名单，排除正常业务
3. 分层告警/告警聚合，严重的才通知人
4. 持续运营，持续优化

**给防守方的建议：**
1. 先建基础，再搞高大上
2. 检测能力要覆盖ATT&CK各阶段，层层有检测
3. 别光靠特征，行为检测也很重要
4. 监测体系要持续运营

最后送你一句话：

> **没有100%的防御，
> 但可以有100%的监测。
> 发现得越早，损失越小。**
