# 第二十章 安全运营与持续改进

## 20.1 安全运营概述

安全加固不是一次性的工作，而是持续运营的过程。本章介绍如何建立安全运营体系，实现安全防护的持续改进。

### 20.1.1 什么是安全运营

```
安全运营（Security Operations）：

定义：
以持续改进的方式，通过人员、流程、技术的有机结合，
实现对信息系统全生命周期的安全管理，保障业务安全。

核心目标：
├── 风险可管：识别、评估、控制安全风险
├── 事件可控：及时发现、响应、处置安全事件
├── 合规可证：满足法律法规和标准要求
├── 业务可持续：保障业务连续性
└── 持续改进：不断提升安全防护水平

安全运营 vs 安全加固：

安全加固：一次性或阶段性的技术措施，解决具体问题
安全运营：持续性的管理过程，保障长期安全

关系：
安全加固是安全运营的一部分，安全运营让加固持续有效。
```

### 20.3.2 漏洞管理流程

```
漏洞管理全流程：

1. 漏洞发现
   ├── 定期扫描（每周/每月）
   ├── 渗透测试
   ├── 漏洞情报订阅
   ├── 白帽子报告
   ├── 厂商通告
   └── 自查发现

2. 漏洞确认
   ├── 漏洞验证
   ├── 影响范围评估
   ├── 严重等级判定
   │   ├── 严重（Critical）
   │   ├── 高危（High）
   │   ├── 中危（Medium）
   │   └── 低危（Low）
   └── 录入漏洞管理系统

3. 漏洞修复
   ├── 修复方案制定
   ├── 修复责任人分配
   ├── 修复时限要求
   │   ├── 严重：24小时内
   │   ├── 高危：7天内
   │   ├── 中危：30天内
   │   └── 低危：90天内
   ├── 修复实施
   └── 回归验证

4. 漏洞关闭
   ├── 修复验证
   ├── 漏洞关闭
   ├── 记录归档
   └── 统计分析

漏洞管理指标：
├── 漏洞数量趋势
├── 漏洞修复率
├── 平均修复时间（MTTR）
├── 逾期漏洞数
├── 重复漏洞率
└── 漏洞分布（类型/系统/严重级）
```

```

   ├── 遏制阶段
   │   ├── 隔离受影响系统

   │   └── 清除后门
   ├── 恢复阶段
   │   ├── 系统恢复
   │   ├── 业务验证
   │   └── 监控观察
   └── 总结阶段

   ├── 知识库更新
   └── 持续跟踪

事件管理指标：
├── MTTD（平均检测时间）

├── MTTF（平均修复时间）

├── 事件类型分布
├── 事件严重程度分布
└── 重复事件率
```

### 核心思想

```
安全加固的核心理念：

1. 纵深防御
   没有绝对安全的系统，多层防护才能抵御攻击。

2. 最小权限
   只授予必要的最小权限，减少攻击面。

3. 持续改进
   安全是过程不是终点，需要持续运营和改进。

4. 人防+技防
   技术是基础，人和管理更重要。

5. 安全左移
   安全越早介入，成本越低，效果越好。

6. 数据驱动
   用数据度量安全，用数据指导改进。

7. 业务对齐
   安全服务于业务，不能脱离业务空谈安全。
```
