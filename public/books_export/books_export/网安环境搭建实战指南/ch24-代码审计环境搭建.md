# 第24章 代码审计环境搭建

## 24.1 代码审计环境概述

### 24.1.1 什么是代码审计

代码审计（Code Audit）是指通过检查源代码或二进制代码，发现其中的安全漏洞、逻辑缺陷、编码规范问题等的过程。代码审计是软件安全保障的重要手段，可以在软件发布前发现并修复潜在的安全问题。

代码审计主要分为以下几类：
- **静态代码分析**：不运行程序，通过分析代码结构、语法、数据流等发现问题
- **动态代码分析**：运行程序，通过观察运行时行为发现问题
- **人工代码审计**：由安全专家逐行检查代码，发现自动化工具难以发现的逻辑漏洞
- **自动化代码审计**：使用工具自动扫描代码，发现已知的安全问题模式

### 24.1.2 代码审计工具分类

| 工具类型 | 代表工具 | 适用语言 | 主要用途 |
|---------|---------|---------|---------|
| IDE/编辑器 | PhpStorm、VS Code、PyCharm | 多语言 | 代码编写、调试、审计辅助 |
| PHP审计工具 | RIPS、Seay源代码审计系统 | PHP | PHP代码安全审计 |
| Java审计工具 | FindBugs、SpotBugs、PMD | Java | Java代码安全审计 |
| Python审计工具 | Bandit、Pylint、Flake8 | Python | Python代码安全审计 |
| 前端审计工具 | ESLint、Retire.js、npm audit | JavaScript | 前端代码安全审计 |
| 综合审计平台 | SonarQube、Fortify、CodeQL | 多语言 | 企业级代码审计平台 |
| 反编译工具 | JD-GUI、JADX、dnSpy | Java/.NET等 | 二进制代码反编译 |

### 24.1.3 环境要求

#### 硬件要求
- CPU：4核8线程以上
- 内存：8GB以上（推荐16GB）
- 硬盘：100GB以上可用空间
- 网络：稳定的互联网连接（用于下载工具和更新规则库）

#### 软件环境
- 操作系统：Windows 10/11 或 Linux（Ubuntu 20.04+）
- Java环境：JDK 8+（部分工具需要）
- Python环境：Python 3.7+
- Docker环境：用于部署部分容器化工具

### 24.2.5 常见问题

**Q1: PhpStorm扫描不出PHP代码问题怎么办？**

A: 请检查以下几点：
1. 确认PHP解释器已正确配置（File → Settings → PHP）
2. 确认代码文件夹已标记为"Sources"（右键文件夹 → Mark Directory as → Sources Root）
3. 确认已安装PHP Inspections插件并启用
4. 尝试手动触发检查：Code → Inspect Code

**Q2: Xdebug无法连接到PhpStorm怎么办？**

A: 排查步骤：
1. 使用`php -v`确认Xdebug已加载
2. 使用`php -m | grep xdebug`确认Xdebug扩展已启用
3. 检查php.ini中xdebug.client_host和xdebug.client_port配置
4. 确认PhpStorm已开启监听（电话图标已按下）
5. 检查防火墙是否阻止了9003端口
6. 在PhpStorm中设置断点，添加`xdebug_info()`到PHP页面查看详情

**Q3: RIPS扫描结果不准确怎么办？**

A: RIPS作为自动化工具，存在误报和漏报的可能：
1. 提高扫描级别（-l 参数，最高5级）
2. 结合人工审计验证结果
3. 使用多种工具交叉验证
4. 更新RIPS到最新版本
5. 理解工具原理，避免对扫描结果的过度依赖

**Q4: Seay源代码审计系统在Windows 11上无法运行？**

A: 可能需要安装.NET Framework：
1. 下载并安装 .NET Framework 4.7.2 或更高版本
2. 右键程序 → 属性 → 兼容性 → 勾选"以兼容模式运行" → 选择Windows 7
3. 以管理员身份运行

### 24.3.5 常见问题

**Q1: Java版本不兼容导致工具无法运行？**

A: 不同工具对Java版本有不同要求：
- SpotBugs 4.x：需要Java 11+
- SonarQube 10.x：需要Java 17+
- PMD 7.x：需要Java 11+
- Checkstyle 10.x：需要Java 11+

建议使用SDKMAN或update-alternatives管理多个Java版本。

**Q2: SonarQube启动失败，提示内存不足？**

A: 修改SonarQube配置：
```bash
docker run -d --name sonarqube \
  -p 9000:9000 \
  -e SONAR_ES_BOOTSTRAP_CHECKS_DISABLE=true \
  -e SONAR_WEB_JVM_OPTS="-Xmx2G -Xms2G" \
  -e SONAR_CE_JAVA_OPTS="-Xmx2G -Xms2G" \
  sonarqube:latest
```

**Q3: SpotBugs扫描结果有很多误报怎么办？**

A: 可以通过配置过滤误报：
1. 使用`-exclude`参数排除指定规则
2. 创建excludeFilter.xml文件自定义过滤
3. 在代码中使用`@SuppressFBWarnings`注解
4. 调整扫描级别（-low/-medium/-high）

A: 建议分步骤改进：

3. 使用.pylintrc配置文件禁用不适用的规则
4. 不要追求10分，8-9分已经很不错
5. 关注安全相关的规则，而不是纯风格问题

### 24.5.6 常见问题

**Q1: ESLint规则太多太严格怎么办？**

A: 可以根据项目需求调整规则：
1. 从推荐配置开始（eslint:recommended）
2. 逐步添加团队认可的规则
3. 使用注释临时禁用特定行的规则：`// eslint-disable-next-line rule-name`
4. 在配置文件的rules中调整规则级别（off/warn/error）
5. 不要为了通过检查而禁用安全相关规则

**Q2: npm audit发现很多漏洞但无法升级怎么办？**

A: 可以尝试以下方法：
1. 使用`npm audit fix`自动修复
2. 手动更新有问题的包
3. 使用`npm-force-resolutions`强制解析版本
4. 如果是间接依赖的问题，等待上游修复
5. 评估漏洞影响，风险可控的可以接受
6. 使用`overrides`字段（npm 8.3+）覆盖依赖版本

4. 确保所有依赖都已下载

**Q2: CodeQL数据库创建失败？**

A: 常见原因和解决方法：
1. 确保项目可以正常编译
2. Java项目需要Maven或Gradle
3. C/C++项目需要编译工具链
4. 使用`--command`参数指定正确的编译命令
5. 检查日志中的具体错误信息

**Q3: Semgrep扫描速度慢？**

A: 优化方法：
1. 排除不需要扫描的文件（.semgrepignore）
2. 减少使用的规则数量
3. 使用更快的规则集
4. 增量扫描（只扫描修改的文件）
5. 使用--jobs参数增加并发

### 24.7.5 常见问题

**Q1: JD-GUI反编译出来的代码有错误怎么办？**

A: 反编译工具不能保证100%还原源代码：
1. 尝试使用不同的反编译器对比（JADX、CFR、Procyon）
2. 结合字节码分析（Bytecode Viewer）
3. 混淆后的代码反编译效果会很差
4. 重要逻辑可以动态调试验证

**Q2: APK反编译失败怎么办？**

A: 常见原因：
1. APK加壳了，需要先脱壳
2. APK使用了加固服务（360加固、腾讯乐固等）
3. APKTOOL版本过低，升级到最新版
4. 使用`--no-src`参数跳过dex反编译，只提取资源
5. 尝试使用其他工具（JADX）

**Q3: .pyc文件无法反编译？**

A: 可能原因：
1. Python版本不匹配，uncompyle6支持2.7和3.0-3.10
2. 文件损坏或不是标准的pyc文件
3. 使用了混淆或加密
4. 尝试使用pycdc或decompyle3
5. 查看pyc文件头确认Python版本
