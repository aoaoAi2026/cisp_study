# 方法一（Burp 原生宏 + Session 会话规则）VS 方法三（Burp CSRF Token 插件）超详细对比

## 前置基础统一认知

两者**最终效果完全一致**：自动为 Intruder 每条爆破请求刷新页面、抓取全新`user_token`并替换，都能通关 DVWA Brute High；

核心底层逻辑相同：每次发包前置 GET 页面拿新 token；

本质区别：**实现载体、配置流程、依赖条件、考场合规性、可控性、报错排查方式完全不同**。

## 一、核心定义区分

### 方法一：Burp 原生宏 + 会话规则

依靠 Burp Suite 软件**出厂自带、无需额外下载**的内置功能：Macros（宏录制）、Session Handling Rules（会话处理规则）。

无任何外部扩展，是 Burp 官方原生设计用来处理 CSRF 一次性令牌的标准解决方案。

### 方法三：Burp CSRF Token Tracker 第三方插件

属于 BApp Store 商店里的**外部扩展程序**，不是 Burp 自带功能，必须联网下载安装；插件是别人开发的外挂程序，接管 Burp 的请求拦截逻辑，自动识别、刷新、替换 token，跳过手动配置宏的整套流程。

## 二、10 个维度极致详细对比（含操作差异、原理差异、优缺点、适用场景）

### 维度 1：环境依赖 & 网络要求

#### 方法一（原生宏）



1. **完全零网络依赖**：断网、内网、考场、离线虚拟机都能完整配置使用；

2. 仅依赖 Burp 本体，不需要下载任何文件、扩展；

3. Kali 本地离线 Burp、无互联网环境 100% 可用。

#### 方法三（CSRF 插件）



1. **必须有外网网络**：首次使用必须联网打开 BApp Store 下载插件安装；

2. 离线环境、机房封闭内网、考试断网环境无法安装，直接作废；

3. 就算之前装过插件，Burp 大版本更新后插件可能失效，需要重新联网修复兼容包。

#### 核心区别总结

原生宏无网络门槛，插件强依赖互联网，线下考试直接禁用。

### 维度 2：完整配置操作步骤差异（最大直观区别）

#### 方法一 完整配置流程（一共 6 大阶段，手动自定义全部规则）



1. 清空 Cookie Jar 消除会话冲突；

2. 手动录制**纯页面无参数 GET 宏**（必须手动区分包，不能选错登录包）；

3. 手动配置宏 Cookie 隔离规则：取消旧 Cookie 复用、开启新 Cookie 存入；

4. 手动手写 token 提取起止符：`name="user_token" value="` 和 `"`；

5. 手动新建会话规则，绑定宏，勾选同步 token、同步 Cookie；

6. 手动切换 Scope 标签，单独勾选 Intruder 工具，限定规则生效范围；

7. Intruder 手动设置单线程、永久跟随重定向。

> 全部配置
>
> **人工可控**
>
> ，每一步规则都是你手动定义，你能清楚每一行勾选的作用。

#### 方法三 插件配置流程（极简，几乎无手动规则编写）



1. 联网打开 BApp 商店安装插件，重启 Burp；

2. 打开插件面板点击`Auto Detect Tokens`，插件自动扫描页面识别`user_token`；

3. 插件默认自动开启「Intruder 自动更新 token」，无需录制宏、不用写提取规则、不用配置 Session 会话规则、不用手动设置 Scope；

4. 仅需要在 Intruder 手动设置单线程、跟随重定向。

> 插件全自动封装底层逻辑，跳过宏录制、token 提取、会话规则全套十几步手动操作。

#### 操作差异总结

原生宏：十几步手动精细化配置，每一步都需要理解原理；

插件：2 步简单操作，自动化屏蔽底层复杂配置，新手不用懂 token 提取逻辑。

### 维度 3：底层实现原理差异（根本区别）

#### 方法一 原生宏原理

Burp 内置两套独立功能联动工作：



1. Macros 宏：固定存储一条 GET 页面请求，作为 “刷新 token 模板”；

2. Session Handling Rules（会话规则引擎）：Burp 原生内置拦截器，在 Intruder 发出每一条 HTTP 请求前，强制优先执行绑定的宏，执行完成后读取宏响应里提取到的 token，替换当前爆破请求的`user_token`参数；

3. Cookie 控制、token 提取、工具生效范围全部由用户手动编写规则交给 Burp 引擎执行。

   **控制权在用户手上，Burp 只是执行你写好的规则**。

#### 方法三 插件原理

插件是独立运行在 Burp 内部的第三方程序，劫持 Burp 的 HTTP 请求流：



1. 插件自动爬虫靶场页面，自动识别表单里`user_token`参数；

2. 当 Intruder 准备发送爆破请求时，插件强行拦截请求，单独发起 GET 页面请求获取新 token；

3. 插件内部代码自动替换请求里的旧 token，再放行爆破请求；

4. Cookie 同步、token 提取、工具过滤全部写死在插件代码里，用户看不到底层规则。

   **控制权在插件开发者手上，用户无法自定义底层逻辑**。

### 维度 4：可控性 & 自定义修改空间

#### 方法一 原生宏：高度自定义，全可控



1. 可以自由修改宏请求内容：添加请求头、Cookie、代理、延时；

2. 可以自定义 token 提取规则，任意起止符、正则匹配任意页面令牌；

3. 会话规则可自由控制：只对 Proxy 生效 / 只对 Intruder 生效 / 全部工具生效；

4. 可手动测试宏（Test macro 按钮），单独查看宏是否能正常提取 token，方便排错；

5. Cookie 处理规则自由切换：是否复用旧 Cookie、是否自动存储新 Cookie 可独立勾选。

#### 方法三 插件：几乎无自定义空间，逻辑固定



1. 无法修改底层刷新页面的请求头、请求方式；

2. token 提取规则是插件内置固定代码，不能手动修改起止符、正则；

3. 工具生效范围固定同时作用于 Proxy+Intruder，无法单独关闭某一个工具；

4. 无单独测试 token 提取的功能，无法单独验证页面请求是否正常；

5. Cookie 同步逻辑写死在插件内，不能手动开关 Cookie 复用规则。

#### 差异总结

原生宏适合需要灵活调整请求逻辑、学习底层原理的人；插件黑盒运行，只能用默认逻辑，无法精细化调整。

### 维度 5：线下渗透考试 / 竞赛合规性

#### 方法一（原生宏）



1. Burp 官方自带功能，所有 Web 安全线下考试、DVWA 实训、职业渗透考核**完全允许使用**；

2. 考官只会要求掌握宏与会话规则原理，不会禁止原生功能；

3. 不存在 “第三方外挂扩展” 违规风险。

#### 方法三（插件）



1. 绝大多数线下考试、职业技能竞赛**明确禁止安装 BApp 第三方插件**；

2. 考试规则仅允许使用 Burp 出厂自带功能，外部扩展一律视为违规工具；

3. 部分机房会直接屏蔽 BApp 商店网络，无法安装插件。

### 维度 6：报错排查难度（出问题时区分巨大）

#### 方法一（原生宏）：报错清晰，可逐行定位问题

如果爆破全 302、token incorrect，可分层排查：



1. 点`Test macro`单独测试宏，看宏能不能提取到 token → 定位提取规则是否写错；

2. 打开 Cookie Jar 查看 Cookie 是否刷新 → 定位 Cookie 勾选是否错误；

3. 打开 Session 规则 Scope，检查是否勾选 Intruder → 定位工具生效范围；

4. 查看宏 Configure item，检查旧 Cookie 复用是否取消勾选；

   每一层配置独立可视，哪一步出错一目了然，适合学习排错思路。

#### 方法三（插件）：黑盒报错，排错极其困难



1. 插件无独立测试功能，无法单独查看页面请求是否拿到 token；

2. 插件底层逻辑不展示，出现全 302 跳转时，你无法区分是 Cookie 冲突、token 提取失败、还是工具未生效；

3. 插件无分层配置面板，没有单独的 Cookie、token、Scope 可视化选项，只能卸载重装尝试修复；

4. 插件崩溃、兼容报错只会弹出英文乱码弹窗，无分层报错提示。

### 维度 7：软件版本兼容稳定性

#### 方法一（原生宏）



1. Burp 全系列版本（社区版、专业版、新旧 2023/2024/2026 版本）宏与会话规则功能逻辑完全不变；

2. 升级 Burp 软件不会导致宏配置失效，配置永久兼容；

3. 无闪退、崩溃风险，属于核心底层功能，Burp 官方长期维护。

#### 方法三（插件）



1. 插件仅适配特定区间 Burp 版本，新版 Burp 经常出现插件不兼容、闪退；

2. Burp 大更新后插件大概率失效，必须等作者更新插件代码，再联网下载新版插件；

3. 部分插件内存泄漏，长时间跑爆破会导致 Burp 卡顿、直接闪退丢失爆破结果。

### 维度 8：学习价值（新手入门核心差距）

#### 方法一（原生宏）：学习价值极高

整套配置完整展示 CSRF 一次性令牌防护的底层绕过逻辑：



1. 理解为什么要隔离旧 Cookie；

2. 理解 HTML 内 token 如何通过起止符截取；

3. 理解会话规则区分不同工具生效范围；

4. 学会这套逻辑后，所有带一次性 CSRF 令牌的靶场（XSS、文件上传、SQLi 关卡）全部通用；

   适合系统学习 Web 安全、准备考证、积累渗透思路。

#### 方法三（插件）：几乎无学习价值

插件一键自动化屏蔽所有底层原理，你不需要知道 token 怎么提取、会话怎么同步、Cookie 如何隔离；

长期依赖插件会完全不懂 CSRF 绕过底层逻辑，更换靶场、更换无插件环境直接无从下手；

仅适合只想快速跑完靶场、不想学习原理的用户。

### 维度 9：操作容错率（新手容易踩坑点）

#### 方法一（原生宏）：容错率低，一步错全盘失效

任何一步漏操作都会导致全部 302 跳转：

漏取消旧 Cookie 复用、Scope 没勾选 Intruder、宏选错登录包、token 起止符写错、线程大于 1，任意一项出错直接爆破失败。

优点：出错能明确学到哪里配置错误，加深记忆。

#### 方法三（插件）：容错率极高，极少踩坑

插件自动处理 token 提取、宏录制、Scope、Cookie 同步，用户仅需要设置单线程、跟随重定向；

几乎不会出现配置类报错，门槛极低，零基础也能一键跑通爆破。
