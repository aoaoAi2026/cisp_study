方案一 Burp Suite 带 Token 爆破 完整分步通关方案

本方案针对 DVWA Brute Force High 难度，完整实现带 CSRF Token 的自动化暴力破解，每一步均附带原理说明与选型依据，彻底解决 Token 不更新、全量 302、攻击提前终止等常见问题。

# 方案一 Burp Suite 带 Token 爆破 完整分步通关方案

## 一、前置原理说明

### 1. High 级别核心防御机制

对比 Low/Medium 难度，High 新增了一层 **CSRF Token 防护**：

*   每次访问登录页面，服务端都会生成一个唯一的、一次性的 `user_token` 嵌入表单中；
*   提交登录请求时必须携带该 Token，服务端校验通过才会处理登录逻辑；
*   Token 仅单次有效，提交一次后立即失效，下次登录必须重新获取新 Token；
*   该难度**无登录失败次数锁定**，只要能持续获取新 Token，就可以无限次尝试密码。

### 2. 带 Token 爆破的核心原理

常规多线程爆破在 Token 机制下完全失效，因此必须采用「单线程串行 + 循环提取 Token」的逻辑：

1.  先获取初始页面，拿到第一个有效 Token；
2.  提交第一个密码 + 当前 Token；
3.  从登录响应页面中提取下一个全新的 Token；
4.  用新 Token 提交下一个密码，循环往复，遍历完整个密码字典；
5.  命中正确密码时，页面返回登录成功的欢迎页，与失败响应产生明显差异。

## 二、前置准备

1.  已搭建好 DVWA 环境，浏览器成功登录后台，将安全等级切换为 `High`，进入左侧「Brute Force」关卡；
2.  Burp Suite 可正常抓包，浏览器代理配置完成；
3.  准备一份密码字典（txt 格式，每行一个密码），必须包含 DVWA 默认正确密码 `password`，用于验证爆破效果；
4.  确保浏览器会话未过期，Cookie 中同时包含 `PHPSESSID` 与 `security=high`。

## 三、详细操作步骤（每步含原理说明）

### 步骤 1：抓取登录请求数据包

**操作**

1.  在 Brute Force 登录页输入测试内容：用户名 `admin`，密码随意填写（如 `123456`）；
2.  开启 Burp 代理拦截，点击页面「Login」按钮；
3.  抓到请求包后，确认参数包含 `username`、`password`、`Login`、`user_token` 四项，Cookie 中存在 `PHPSESSID` 和 `security=high`；
4.  右键数据包 → 选择「发送到 Intruder」，切换到 Intruder 标签页。

**原理** 必须抓取完整的登录请求，才能获取到初始的有效 Token、会话 Cookie 以及请求格式，这是后续自动化爆破的基础。

### 步骤 2：配置载荷位置与攻击类型

**操作**

1.  切换到「位置」子标签，点击右侧「清除 §」，清空所有自动标记；
2.  手动给两个参数值添加 § 标记：

    1.  `password=§123456§`（密码遍历位）
    2.  `user_token=§初始Token值§`（Token 更新位） 其余参数（`username`、`Login`）保持固定，不添加标记；
3.  攻击类型下拉选择 **Pitchfork（草叉）**。

**原理与选型说明**

*   为什么只标记两个参数：用户名固定为 `admin`，`Login` 是提交按钮固定值，无需修改；只有密码需要遍历、Token 需要动态更新，因此仅标记两处。
*   为什么选 Pitchfork 模式： Pitchfork 模式会让两组载荷一一对应，第 1 个密码搭配第 1 个 Token，第 2 个密码搭配第 2 个 Token，完全匹配我们「一个密码配一个新 Token」的串行逻辑。 其他模式均不适用：

    *   Sniper：只遍历一个载荷位，无法同时更新密码和 Token；
    *   Cluster bomb：两组载荷笛卡尔积排列，会出现旧 Token 配新密码的情况，全部 Token 校验失败。

### 步骤 3：配置 Grep-Extract 自动提取 Token

**操作**

1.  切换到「设置」子标签，下滑找到「检索 - 提取」模块；
2.  勾选「从响应包中提取以下项目」，点击「添加」弹出配置窗口；
3.  右侧勾选「从正则表达式组中提取」，输入正则表达式：

    1.  `user_token' value='(.*?)'`
4.  勾选「区分大小写」，不勾选「排除 HTTP 头」；
5.  点击「重新获取响应」，预览区出现页面源码且 32 位 Token 被高亮，即匹配成功，点击「确认」保存规则。

**原理**

*   正则表达式 `user_token' value='(.*?)'` 的作用：精准匹配页面中隐藏表单的 Token 值，括号 `()` 为提取分组，只提取 value 中的 32 位字符串，不提取多余代码。
*   `(.*?)` 是非贪婪匹配，保证只匹配到最近的一个单引号，不会把后面的页面内容都匹配进来。
*   该提取规则是「递归更新 Token」的前提，Burp 每次收到响应后，都会用这条规则提取下一次请求要用的新 Token。

### 步骤 4：配置 Grep-Match 成功结果自动标记

**操作**

1.  在「设置」标签页，找到「检索 - 匹配」模块（在「检索 - 提取」上方）；
2.  点击「清空」删除默认的 error、fail 等匹配项；
3.  点击「添加」，输入登录成功页面的特征文字：

    1.  `Welcome to the password protected area admin`
4.  匹配类型保持「简单字符串」，保持「排除 HTTP 头」勾选；
5.  无需额外保存，配置自动生效。

**原理**

*   登录成功与失败的页面内容不同，成功页面会包含固定的欢迎文字。
*   配置 Grep-Match 后，Burp 会自动给匹配到该文字的请求打上对勾标记，无需逐个点开响应查看，一眼就能定位到正确密码，大幅提升效率。

### 步骤 5：配置两组载荷参数

切换到「payload」子标签，分别对两个载荷集进行配置。

#### 载荷集 1（密码位）

**操作**

1.  「Payload 集」选择 `1`；
2.  「Payload 类型」选择「简单列表」；
3.  点击「加载」，导入准备好的密码字典文件。

**原理** 密码字典是固定的字符串列表，简单列表直接按顺序遍历即可，完全匹配密码爆破的需求。

#### 载荷集 2（Token 位，核心配置）

**操作**

1.  「Payload 集」切换为 `2`；
2.  「Payload 类型」选择「递归提取」；
3.  在下方列表中，选中步骤 3 配置的 Token 提取规则；
4.  在「首次请求初始 payload」输入框中，填入原始数据包里的 32 位初始 Token；
5.  **取消勾选「检测到重复 payload 时停止」**。

**原理与选型说明**

*   为什么用「递归提取」： 递归提取类型会从上一个请求的响应中，用 Grep-Extract 规则提取内容，作为下一个请求的载荷值，完美实现「提交一次 → 拿新 Token → 再提交」的循环逻辑，这是带 Token 爆破的核心。
*   为什么要填初始 Token：第一次请求没有上一个响应可以提取 Token，必须手动填入初始值来启动整个循环，否则第一次请求就会因 Token 为空直接失败。
*   为什么取消「检测到重复 payload 时停止」： Token 虽然是随机生成，但理论上存在极小概率重复；同时如果勾选该选项，Burp 会在检测到 Token 重复时直接终止攻击，通常跑 2-3 条就会误判停止，无法跑完整个字典。

### 步骤 6：配置单线程资源池

**操作**

1.  切换到「资源池」子标签；
2.  选择「自定义资源池」，将「并发请求」数值改为 `1`；
3.  请求延迟保持默认即可。

**原理** Token 是一次性串行生效的，必须等上一个请求完成、提取到新 Token 后，才能发起下一个请求。 如果开多线程，会出现多个请求同时争抢同一个 Token，或者新 Token 还没提取出来就用旧 Token 发起请求，最终全部 Token 校验失败，返回 302。单线程串行是带 Token 爆破的必要条件。

### 步骤 7：配置重定向规则（解决全 302 的关键）

**操作**

1.  切回「设置」子标签，下滑找到「重定向」选项；
2.  下拉选择「始终跟随重定向」（或「仅跟随同站点重定向」）。

**原理**

*   DVWA 登录采用 PRG（Post-Redirect-Get）模式：提交登录表单后，无论成功失败，服务端都会先返回 302 重定向，再跳转到登录页 / 欢迎页。
*   如果选择「永不跟随重定向」，Burp 只会拿到 302 的空响应头，响应体中没有登录表单，也就提取不到新的 user\_token。后续请求只能一直复用初始的过期 Token，导致全部校验失败，全是 302。
*   开启跟随重定向后，Burp 会自动完成跳转，拿到完整的页面响应，才能正常提取到新 Token，让循环持续下去。

### 步骤 8：启动攻击与结果筛选

**操作**

1.  所有配置完成后，点击右上角「开始攻击」，弹出攻击结果窗口；
2.  先验证配置是否正常：

    1.  查看「Payload 2」列：每一行的 Token 值都不相同 → Token 自动刷新正常；
    2.  状态码列：大部分请求为 200 → 重定向跟随正常。
3.  筛选正确密码，两种方法任选：

    1.  **方法一：Grep-Match 一键定位**：结果表格中会多出一列（列名为你设置的欢迎文字），打勾的那一行就是登录成功的请求，对应「Payload 1」列的值就是正确密码；
    2.  **方法二：按响应长度排序**：点击「长度」列标题排序，失败请求的页面长度几乎完全一致，成功请求的响应长度会明显与众不同，该行对应的密码即为正确密码。

**原理** 登录失败返回的是统一的登录页，内容长度固定；登录成功返回的是欢迎页，内容不同，长度差异明显。因此通过长度排序也能快速定位成功项，无需额外配置。

### 步骤 9：验证通关

回到浏览器 DVWA Brute Force 页面，输入用户名 `admin` 和爆破得到的密码，点击 Login，页面显示欢迎信息即通关成功。

## 四、核心配置选型总结

| 配置项           | 最终选型          | 核心原因                           |
| :------------ | :------------ | :----------------------------- |
| 攻击类型          | Pitchfork（草叉） | 两组载荷一一对应，匹配 “一密码一 Token” 的串行逻辑 |
| Token 载荷类型    | 递归提取          | 自动从响应中提取新 Token，实现循环更新         |
| 并发线程          | 1             | 多线程会导致 Token 不同步，全部校验失败        |
| 重定向           | 始终跟随          | 302 跳转后的页面才包含新 Token，不跟随则提取失败  |
| 重复 payload 停止 | 取消勾选          | 避免误判 Token 重复，提前终止攻击           |

## 五、常见问题排查

1.  **所有请求全是 302**

    1.  排查重定向是否设置为「始终跟随」；
    2.  排查 Token 载荷类型是否为「递归提取」；
    3.  排查初始 Token 是否填写正确。
2.  **攻击跑 2-3 条就停止**

    1.  确认已取消勾选「检测到重复 payload 时停止」。
3.  **Token 一直不更新，每行都相同**

    1.  确认 Grep-Extract 正则匹配正确，预览能抓到 Token；
    2.  确认重定向已开启跟随，响应包含完整页面。
4.  **跑完字典全失败**

    1.  确认字典中包含正确密码 `password`；
    2.  确认用户名固定为 `admin`，未被错误标记。

# 方案二 Burp CSRF Token Tracker 插件自动携带Token 完整通关方案

## 一、前置原理说明

### 1. 插件解决的核心痛点（对比手动 Recursive grep 爆破）

DVWA Brute Force High 关卡每次提交登录必须携带一次性`user_token`，手动递归 grep 配置繁琐，存在 302、Token 不刷新、攻击提前终止等问题。
`CSRF Token Tracker`是 Burp 官方插件，底层逻辑：

1.  自动监听所有请求响应，从页面 HTML 中持续抓取最新`user_token`；
2.  自动替换数据包中`user_token`参数的值，全程无需手动配置正则、递归载荷、初始 Token；
3.  自动维持会话 Cookie，保持 DVWA 登录状态，规避会话过期问题；
4.  兼容 Intruder 多模式爆破，不用限制单线程（虽然 DVWA 场景建议单线程，但插件支持并发）。

### 2. 插件工作流程原理

1.  浏览器登录 DVWA，插件读取浏览器代理流量，缓存当前页面有效 Token；
2.  Intruder 发起密码请求前，插件自动 GET 目标页面刷新最新 Token；
3.  把数据包里旧`user_token`替换为刚抓取的全新 Token，再发送登录请求；
4.  响应返回后自动重新抓取下一轮 Token，循环自动化完成，完全替代手动 Grep 提取逻辑。

### 3. 两种方案核心差异

| 对比项   | 方案一（手动递归 grep）        | 方案二（CSRF Token Tracker 插件） |
| ----- | --------------------- | -------------------------- |
| 正则配置  | 必须手写匹配 user\_token 正则 | 无需写正则，插件自动识别表单 token       |
| 载荷配置  | 需要两组载荷、递归提取、初始 Token  | 仅配置一组密码载荷即可，Token 插件自动维护   |
| 重定向配置 | 必须手动开启跟随重定向           | 插件内部自动处理跳转，无需手动设置          |
| 操作复杂度 | 步骤多，容易配置出错全 302       | 简化 70% 配置，减少人为失误           |

## 二、前置准备

1.  DVWA 环境正常，登录后台，安全等级切换为`High`，进入`Brute Force`暴力破解页面；
2.  Burp Suite 已安装`CSRF Token Tracker`插件（Extender - 应用商店搜索安装）；
3.  浏览器开启 Burp 代理，能正常抓包；
4.  准备 txt 密码字典，必须包含默认正确密码`password`；
5.  保持浏览器 DVWA 页面登录状态，会话 Cookie 有效。

## 三、分步详细操作（每步附带操作原理）

### 步骤 1：安装并启用 CSRF Token Tracker 插件

**操作**

1.  打开 Burp 顶部标签【扩展程序】→【应用程序商店】；
2.  搜索框输入`CSRF Token Tracker`，选中插件点击【安装】；
3.  切换到【已安装】列表，确认插件右侧复选框勾选启用；
4.  插件启用后，Burp 底部会出现`CSRF Tracker`独立标签页。

**原理**
该插件不在 Burp 原生功能内，属于第三方自动化 CSRF Token 管理工具，必须手动安装；启用后插件才会拦截流量、自动抓取页面 token，不启用则无自动替换功能。

### 步骤 2：配置插件规则（绑定 DVWA 的 user\_token）

**操作**

1.  切换到底部【CSRF Tracker】标签；
2.  点击左上角【Add】新建一条 Token 匹配规则；
3.  规则参数填写：
    *   Match URL（匹配 URL）：`http://192.168.226.129:9111/dvwa/vulnerabilities/brute/*`
        > 原理：通配符`*`匹配该关卡所有页面，插件只在这个域名路径下抓取 token，避免其他网站干扰
    *   Token Parameter Name（Token 参数名）：`user_token`
        > 原理：DVWA 表单隐藏域 name 固定为 user\_token，插件根据这个名称定位表单值
    *   Search Location（Token 抓取位置）：HTML Form（表单内隐藏输入框）
    *   Auto-update token before request（请求前自动更新 Token）：勾选
        > 核心开关：发起请求前先刷新页面拿新 token，解决一次性 token 失效问题
4.  点击【OK】保存规则，列表中出现刚创建的 DVWA 规则。

**原理选型说明**

1.  Match URL 不能填全局`*`：如果匹配所有网站，插件会抓取大量无关 token，造成冲突、请求报错；精准限定 Brute Force 关卡路径，只处理当前靶场页面。
2.  Search Location 选 HTML Form：DVWA 的 user\_token 存放在`<input type="hidden">`表单中，插件会解析 HTML 表单提取 value；Cookie/Header 选项适用于 token 存放在请求头的网站，本场景不适用。
3.  必须勾选 Auto-update token before request：High 关卡 token 单次有效，不提前刷新会一直使用旧 token，全部返回 302 跳转。

### 步骤 3：抓取登录基础数据包发送到 Intruder

**操作**

1.  DVWA Brute Force 页面输入用户名`admin`，密码随意填`123456`；
2.  Burp 代理拦截开启，点击 Login 抓包；
3.  确认请求参数：`username=admin&password=123456&Login=Login&user_token=xxx`，Cookie 携带`PHPSESSID; security=high`；
4.  右键数据包 → 【发送到 Intruder】，切换 Intruder 标签。

**原理**
插件仅自动替换`user_token`参数，用户名、会话 Cookie、提交参数格式仍需要原始数据包提供基准模板，必须携带完整会话 Cookie 保证登录状态。

### 步骤 4：配置 Intruder 位置（仅标记密码一处载荷）

**操作**

1.  切换 Intruder【位置】子标签；
2.  点击右侧【清除 §】清空全部自动标记；
3.  仅手动标记密码参数：`password=§123456§`
    *   `username=admin`、`Login=Login`、`user_token=xxx`**全部不添加标记**
4.  攻击类型下拉选择 **Sniper（狙击手）**

**原理与选型说明**

1.  仅标记 password：user\_token 由 CSRF 插件自动替换更新，不需要 Intruder 遍历，因此不用打标记；用户名固定 admin 无需修改。
2.  攻击类型选 Sniper：Sniper 只遍历单一载荷位（密码），完美适配当前场景；
    *   Pitchfork/Cluster bomb 需要两组及以上载荷，本方案无 Token 载荷，无需使用；
    *   Battering ram 适用于多位置填相同密码，本场景不适用。

### 步骤 5：配置 Payload 密码字典

**操作**

1.  切换 Intruder【payload】子标签；
2.  Payload 集默认选择`1`；
3.  Payload 类型：简单列表；
4.  点击【加载】导入本地密码 txt 字典，字典包含`password`；
5.  下方 Payload 处理保持默认，无需修改。

**原理**
仅需遍历密码字符串列表，简单列表按顺序逐条发送，是暴力破解标准载荷类型，无需递归、规则提取等复杂配置。

### 步骤 6：资源池线程配置（推荐单线程）

**操作**

1.  切换【资源池】子标签；
2.  自定义资源池，并发请求设置为`1`。

**原理**
虽然插件支持多线程自动刷新 Token，但 DVWA 提交后 302 跳转加载页面存在延迟，多线程会出现请求抢 Token、页面未加载完成就提交的情况，大概率出现 Token 校验失败；单线程串行执行最稳定，不会出现大量 302 报错。

### 步骤 7：配置 Grep-Match 自动标记成功请求（筛选正确密码）

**操作**

1.  切换 Intruder【设置】标签，找到上方【检索 - 匹配】模块；
2.  点击【清空】删除默认 error、fail 匹配项；
3.  点击【添加】，输入成功页面特征文本：
4.  匹配类型：简单字符串，勾选排除 HTTP 头；
5.  配置自动保存，无需额外确认。

**原理**
和方案一原理一致：登录成功页面包含固定欢迎文字，插件自动匹配并在结果列表打勾，不用逐个打开响应查看，快速定位正确密码。

### 步骤 8：无需手动配置 Grep-Extract、重定向

**关键说明（对比方案一简化点）**

1.  不用配置「检索 - 提取」：CSRF 插件自动完成 Token 抓取替换，不需要手动写正则提取；
2.  不用手动修改重定向选项：插件内部自动跟随 302 跳转刷新页面获取新 Token，Intruder 重定向保持默认即可，不会出现全 302 问题。

### 步骤 9：启动攻击，筛选正确密码

**操作**

1.  全部配置完成，点击右上角【开始攻击】；
2.  查看结果列表识别成功请求，两种方式：
    *   方法 1（Grep 匹配标记）：列表新增匹配列，带绿色对勾的行，Payload1 值就是正确密码；
    *   方法 2（长度排序）：点击列表【长度】列排序，长度明显更大的行即为登录成功。

**原理**
失败响应统一返回登录页面，页面长度固定；成功返回欢迎页面，HTML 内容更多，长度差异直观。

### 步骤 10：浏览器验证通关

复制爆破得到的正确密码`password`，回到 DVWA Brute Force 页面，用户名填`admin`、密码填入，点击 Login，页面显示欢迎提示，关卡通关。

## 四、CSRF Token Tracker 插件方案 核心配置选型总结

| 配置项           | 选型                                           | 操作原因                                |
| ------------- | -------------------------------------------- | ----------------------------------- |
| Token 管理工具    | CSRF Token Tracker 插件                        | 自动抓取、替换 user\_token，省去正则 + 递归载荷复杂配置 |
| Intruder 攻击类型 | Sniper                                       | 仅需要遍历密码单一参数，无 Token 载荷              |
| 并发线程          | 1                                            | 避免多线程 Token 同步冲突，减少 302 报错          |
| Token 抓取位置    | HTML Form                                    | DVWA user\_token 存放于表单隐藏 input 标签   |
| 自动更新 Token    | 勾选                                           | High 关卡 Token 单次生效，每次请求前刷新获取新 Token |
| Grep 匹配关键词    | Welcome to the password protected area admin | 自动标记登录成功请求，快速筛选密码                   |

## 五、方案二常见问题排查

1.  **全部请求返回 302 跳转**
    *   排查 CSRF Tracker 规则是否正确填写参数名`user_token`；
    *   确认规则勾选了【Auto-update token before request】请求前刷新 Token；
    *   检查 Match URL 是否匹配当前 Brute Force 页面地址，规则未生效插件不会更新 Token。
2.  **插件不抓取 Token**
    *   确认插件已启用，规则 URL 路径无误；
    *   浏览器 DVWA 页面保持登录，Cookie 有效；
    *   Search Location 选择 HTML Form，不要选 Cookie/Header。
3.  **攻击无匹配成功项，全部密码失败**
    *   密码字典内未包含`password`正确密码；
    *   Intruder 仅标记 password 参数，不要误标记 user\_token；
4.  **插件报错：Token 无法替换**
    *   原始数据包必须存在`user_token`参数，不能删除表单参数；
    *   关闭其他冲突的 Token 管理插件，避免互相覆盖参数值。

## 补充：方案二对比方案一优势

1.  配置步骤减少近一半，无需手动编写正则、配置递归提取载荷、填写初始 Token；
2.  自动处理 302 跳转刷新页面，不会出现手动 grep 常见的全 302 卡死问题；
3.  逻辑分离：插件单独负责 Token 生命周期，Intruder 只专注密码爆破，降低配置失误概率；
4.  通用性强，其他带 CSRF Token 的靶场关卡可复用这套插件规则，只需修改匹配 URL。

# 方案三 Burp Suite 宏（Macro）自动化获取Token爆破 DVWA Brute Force High 完整方案

## 一、前置原理说明

### 1. 宏的核心作用

Burp 宏是内置自动化脚本功能，无需安装插件，原生实现「前置请求获取 Token→替换到登录包→循环爆破」，完全替代手动 Recursive grep 与第三方 Token 插件。
DVWA High 一次性`user_token`逻辑适配宏工作流程：

1.  每次发送登录爆破请求前，**自动发起 GET 请求访问暴力破解页面**；

2.  从 GET 响应的 HTML 表单中提取最新`user_token`；

3.  自动把登录 POST 数据包里旧的`user_token`替换为刚提取的新 Token；

4.  再发送带新 Token + 当前遍历密码的登录请求；

5.  循环执行，实现自动刷新 Token，解决全 302、Token 过期问题。

### 2. 宏实现 Token 刷新完整工作链路

1.  Intruder 准备发送密码请求 → 触发前置宏；

2.  宏发送 GET `/dvwa/vulnerabilities/brute/` 获取页面源码；

3.  宏内置正则提取响应内 32 位`user_token`；

4.  将提取到的 Token 覆盖替换 POST 包`user_token`参数；

5.  Intruder 发送替换完成后的登录请求；

6.  收到登录响应后，进入下一条密码，重复以上流程。

### 3. 三种方案横向对比

| 对比维度  | 方案一 手动递归 Grep        | 方案二 CSRF Token Tracker 插件 | 方案三 Burp 宏（原生无插件）          |
| ----- | -------------------- | ------------------------- | -------------------------- |
| 依赖    | 原生功能，配置复杂            | 第三方插件，简化操作                | Burp 原生功能，无需安装插件           |
| 正则    | 必须手动配置 Grep 提取       | 插件自动识别，无需正则               | 宏内部需要配置提取正则                |
| 载荷数量  | 2 组载荷（密码 + 递归 Token） | 仅 1 组密码载荷                 | 仅 1 组密码载荷，宏自动维护 Token      |
| 重定向处理 | 必须手动开启跟随重定向          | 插件自动处理跳转                  | 宏单独处理 GET 页面，不受 POST 重定向影响 |
| 适用场景  | 无法安装插件、熟悉递归载荷        | 允许安装插件，追求最简配置             | 靶场限制安装插件、仅能使用 Burp 原生功能    |

## 二、前置准备

1.  DVWA 环境正常登录，安全等级切换为`High`，进入`Brute Force`页面；

2.  Burp 代理开启，浏览器可正常抓包，DVWA 会话 Cookie（PHPSESSID、security=high）有效；

3.  准备密码 TXT 字典，内置正确密码`password`；

4.  无第三方插件依赖，仅使用 Burp 原生「选项 - 宏」功能。

## 三、分步详细操作（每一步附带操作原理 + 选型理由）

### 步骤 1：抓取 2 个基础数据包（GET 页面包 + POST 登录包）

#### 1.1 抓取获取 Token 的 GET 请求包

**操作**

1.  浏览器停留在 Brute Force 页面，刷新页面；

2.  Burp 拦截到 GET 请求：`GET /dvwa/vulnerabilities/brute/ HTTP/1.1`；

3.  右键该数据包 → 【发送到 宏录制器】。

**原理**
只有访问 Brute Force 页面（GET），服务端才会返回携带全新`user_token`的 HTML 表单，宏需要这条请求作为前置动作，每次爆破前刷新页面拿新 Token。

#### 1.2 抓取登录 POST 请求包，送入 Intruder

**操作**

1.  页面用户名填`admin`，密码随意填写`123456`，点击 Login；

2.  拦截到 POST 登录包，确认参数：`username=admin&password=123456&Login=Login&user_token=xxx`，Cookie 完整；

3.  右键数据包 → 【发送到 Intruder】，切换到 Intruder 标签页。

**原理**
POST 登录包是爆破基准模板，宏后续会自动替换其中`user_token`参数，仅密码位留给 Intruder 遍历。

### 步骤 2：创建宏，配置自动获取 + 提取 Token

**操作**

1.  Burp 顶部菜单栏：【选项】→ 切换到【会话】子标签；

2.  页面下方找到【宏】区域，点击【添加】新建宏；

3.  在弹出的宏编辑器中，选中刚才发送到宏录制器的 GET 请求，点击【OK】；

4.  宏列表选中新建的宏，点击【编辑】，进入宏详细配置窗口；

5.  切换到【响应提取】标签，点击【添加】配置 Token 提取规则：

    *   勾选「从正则表达式组提取」；

    *   正则输入：`user_token' value='(.*?)'`；

    *   提取名称自定义：`dvwa_token`（后续替换参数要用到这个名称）；

6.  点击【确认】保存提取规则，再点击【确定】完成宏配置。

**原理与选型说明**

1.  宏绑定 GET 页面请求：每次执行宏都会刷新页面，获取一次性有效 Token，解决 High 关卡 Token 单次失效问题；

2.  正则`user_token' value='(.*?)'`：非贪婪匹配精准抓取表单隐藏域内 32 位 Token 字符串，括号分组代表提取内容；

3.  自定义提取名称`dvwa_token`：作为全局变量，用于自动替换 POST 数据包中的 user\_token 参数。

### 步骤 3：配置会话处理规则，自动替换 POST 中的 user\_token

**操作**

1.  仍在【选项 - 会话】页面，找到【会话处理规则】，点击【添加】新建规则；

2.  规则名称填写：`自动替换DVWA登录Token`；

3.  规则操作面板点击【添加】，选择【运行宏】；

4.  选中步骤 2 创建的 GET 刷新页面宏，勾选【在执行宏后使用宏的响应提取项替换参数 / 头】；

5.  继续点击【添加】，选择【替换参数】：

    *   参数名称：`user_token`；

    *   替换值：`%dvwa_token%`（百分号包裹宏定义的提取名称，代表引用宏抓取到的新 Token）；

6.  切换到【范围】标签，限定规则生效 URL：

    *   匹配 URL 输入：`http://192.168.226.129:9111/dvwa/vulnerabilities/brute/*`；

    *   勾选仅对 Intruder 工具生效，取消其他工具（重放器、代理等）。

7.  点击【确定】保存整条会话规则，启用该规则。

**原理**

1.  会话规则执行顺序：先运行宏刷新页面拿新 Token，再用提取到的 Token 替换 POST 包`user_token`参数；

2.  `%变量名%`是 Burp 内置变量引用语法，自动读取宏响应提取到的最新 Token；

3.  限定 URL 范围：避免这条规则作用于其他网站，造成参数错误；仅限定 Intruder，不干扰正常代理抓包。

### 步骤 4：配置 Intruder 位置载荷（仅标记密码）

**操作**

1.  切换到 Intruder【位置】子标签；

2.  点击右侧【清除 §】清空全部自动标记；

3.  仅给密码参数添加标记：`password=§123456§`；

    *   `username=admin`、`Login=Login`、`user_token=xxx`**不添加任何 § 标记**；

4.  攻击类型选择 **Sniper（狙击手）**。

**原理选型说明**

1.  user\_token 由会话宏规则自动替换更新，不需要 Intruder 遍历，因此无需标记；用户名固定为 admin 无需修改；

2.  Sniper 仅遍历单一载荷（密码），完美适配本场景；Pitchfork/Cluster bomb 需要两组载荷，本方案无 Token 载荷，不适用。

### 步骤 5：配置 Payload 密码字典

**操作**

1.  切换 Intruder【payload】标签；

2.  Payload 集固定为 1，Payload 类型选择「简单列表」；

3.  点击【加载】导入本地密码 TXT 字典，字典必须包含`password`；

4.  Payload 处理保持默认，无需修改。

**原理**
仅需顺序遍历密码字符串，简单列表是暴力破解基础载荷类型，无需递归、自定义规则。

### 步骤 6：资源池线程配置（强制单线程）

**操作**

1.  切换【资源池】子标签；

2.  自定义资源池，并发请求数值设置为`1`。

**原理**
宏执行、页面 GET 请求、参数替换存在时序延迟，多线程会出现：A 线程宏还未拿到新 Token，B 线程已经发送旧 Token 请求，导致大量 302 校验失败；单线程串行执行保证「先拿 Token、再发登录包」时序稳定。

### 步骤 7：配置 Grep-Match 自动标记登录成功请求（筛选密码）

**操作**

1.  切换 Intruder【设置】标签，找到上方【检索 - 匹配】模块；

2.  点击【清空】删除默认 error、fail 匹配关键词；

3.  点击【添加】，输入成功页面特征文本：

```Plain
Welcome to the password protected area admin
```

1.  匹配类型保持「简单字符串」，勾选【排除 HTTP 头】；

2.  配置自动保存，无需额外确认。

**原理**
登录成功页面存在固定欢迎文字，Burp 会自动给匹配该文本的请求打勾标记，无需逐个点开响应查看，快速定位正确密码。

### 步骤 8：无需配置 Grep-Extract、无需手动修改重定向

**关键简化说明**

1.  不用配置检索 - 提取：Token 提取逻辑完全封装在宏内部，Intruder 无需单独配置提取规则；

2.  不用手动修改重定向选项：宏单独发送 GET 页面请求不受 POST 登录 302 跳转影响，每次都能独立获取完整页面与新 Token，不会出现 Token 提取失败全 302 问题。

### 步骤 9：启动攻击，筛选正确密码

**操作**

1.  全部配置完成，点击右上角【开始攻击】；

2.  两种方式筛选成功请求：

    *   方法 1（Grep 匹配标记）：结果列表新增匹配列，带绿色对勾的行，Payload1 的值就是正确密码；

    *   方法 2（长度排序）：点击列表「长度」列标题排序，响应长度明显更大的一行即为登录成功。

**原理**
密码错误统一返回登录页面，页面 HTML 长度固定；密码正确跳转欢迎页面，页面内容更多，长度差异直观可辨。

### 步骤 10：浏览器验证通关

复制爆破得到的正确密码`password`，回到 DVWA Brute Force 页面，用户名输入`admin`、密码填入，点击 Login，页面展示欢迎提示，关卡通关。

## 五、方案三常见问题排查

1.  **所有请求全部返回 302 跳转**

    *   排查会话处理规则是否启用，范围 URL 是否匹配靶场地址；

    *   检查宏内响应提取规则正则是否能正常匹配 Token；

    *   确认替换参数名称为`user_token`，引用变量名和宏提取名称完全一致；

    *   确认资源池并发线程 = 1。

2.  **宏无法提取到 Token**

    *   检查宏内 GET 请求的 Cookie 是否携带`PHPSESSID; security=high`；

    *   正则表达式无书写错误，预览宏响应能看到 user\_token 字段；

    *   浏览器 DVWA 会话未过期，未退出登录。

3.  **Intruder 无匹配成功项，全部密码失败**

    *   密码字典内未包含正确密码`password`；

    *   仅标记 password 参数，不要误标记 user\_token；

    *   会话规则仅对 Intruder 生效，没有被禁用。

4.  **宏不自动执行，一直使用初始旧 Token**

    *   确认【选项 - 会话】中会话处理规则已勾选启用；

    *   规则操作顺序为先运行宏，再执行参数替换；

    *   匹配范围勾选仅作用于 Intruder 工具。

## 补充：方案三优缺点

### 优点

1.  纯 Burp 原生功能，靶场 / 限制环境无法安装插件时唯一可用自动化 Token 方案；

2.  逻辑解耦：宏单独负责获取 Token，Intruder 仅负责密码遍历，不易混淆配置；

3.  通用性极强，所有带一次性 CSRF Token 的表单都可复用这套宏 + 会话规则模板，仅修改 URL 与参数名。

### 缺点

1.  配置步骤比 CSRF 插件方案更多，需要同时配置宏、响应提取、会话规则三层功能；

2.  新手容易混淆宏变量名、参数替换名称，出现 Token 无法替换的问题；

3.  每一条密码请求都会额外发起一次 GET 页面请求，请求总量翻倍，爆破速度略慢。

# Burp Suite 宏配置通关 DVWA Brute Force High 级别详细教程

DVWA 第一关为 **Brute Force（暴力破解）**，High 难度加入了动态 `user_token`（CSRF 令牌）防护 —— 每次请求都会生成全新的一次性令牌，传统直接爆破会因 token 失效而失败。我们可以通过 Burp Suite 的「宏（Macro）+ 会话处理规则」实现自动获取并更新 token，完成爆破通关。

## 一、前置准备

1.  **环境要求**：已搭建好的 DVWA 靶场、Burp Suite 专业版 / 社区版、配置好 Burp 代理的浏览器（如 Firefox + FoxyProxy）。

2.  **靶场设置**：登录 DVWA，将左侧 `DVWA Security` 级别设置为 **High**，进入 `Brute Force` 关卡。

3.  **核心原理**：High 级别每次访问暴力破解页面，服务器都会生成唯一的 `user_token`，提交登录时必须携带该 token 且仅能使用一次。Burp 宏会先自动访问页面获取新 token，再将 token 代入爆破请求，实现令牌自动刷新。

## 二、步骤 1：抓取目标请求包

先获取一次登录请求，作为后续宏配置和爆破的基础。

1.  开启 Burp 代理拦截，浏览器访问 DVWA 的 Brute Force 页面。

2.  页面中输入用户名 `admin`，密码随意输入（如 `123456`），点击 `Login`。

3.  Burp 拦截到 GET 请求，格式如下：

```http
GET /dvwa/vulnerabilities/brute/?username=admin&password=123456&Login=Login&user_token=xxxxxx HTTP/1.1
Host: 你的靶场地址
Cookie: PHPSESSID=xxxx; security=high
```

1.  右键该请求 → 选择 `Send to Repeater`（用于测试宏是否生效），同时再右键 → `Send to Intruder`（用于后续爆破）。

2.  点击 `Forward` 放行该请求。

## 三、步骤 2：配置 Burp 宏（Macro）

宏的作用是模拟一次页面访问，从响应中自动提取最新的 `user_token`。

1.  打开 Burp 顶部的 **Settings**（旧版本为 `Project options` → `Sessions`），找到 **Sessions** 标签页。

2.  向下滚动找到 **Macros** 区域，点击 `Add` 按钮，弹出「Macro Editor」窗口。

3.  在左侧的请求历史列表中，选中**访问 Brute Force 页面的 GET 请求**（即返回登录表单、包含 `user_token` 的页面请求），点击中间的 `>>` 按钮将其添加到右侧宏序列中。

4.  点击右下角 `Configure item` 按钮，配置参数提取规则：

    *   在弹出的窗口中找到「Custom parameter locations in response」，点击 `Add`。

    *   「Parameter name」填写 `user_token`。

    *   提取方式选择「Define start and end」，填写边界：

        *   开始边界：`name='user_token' value='`

        *   结束边界：`'`

    *   也可点击下方 `Define from response`，在响应预览里直接选中 token 的值，Burp 会自动填充边界。

5.  连续点击 `OK` 保存配置，给宏命名（如 `get_token`）。

## 四、步骤 3：配置会话处理规则

会话处理规则用于告诉 Burp：在发起爆破请求前，先执行上面的宏，并用新 token 更新当前请求。

1.  仍在 Sessions 设置页，找到 **Session Handling Rules** 区域，点击 `Add`。

2.  给规则命名（如 `auto_token`），在「Rule Actions」区域点击 `Add` → 选择 `Run a macro`。

3.  在弹出的动作配置窗口中：

    *   `Select macro` 下拉选择刚才创建的 `get_token` 宏。

    *   勾选 `Update current request with parameters matched from final macro response`。

    *   选择 `Update only the following parameters`，输入框填写 `user_token`。

    *   其余保持默认，点击 `OK`。

4.  切换到「Scope」标签页，设置规则生效范围：

    *   **Tools Scope**：勾选 `Intruder`（仅爆破时生效），可同时勾选 `Repeater` 方便测试。

    *   **URL Scope**：选择 `Use suite scope`（需先将靶场地址加入 Target 范围），或直接选 `Include all URLs`。

5.  连续点击 `OK` 保存所有规则。

## 五、步骤 4：测试宏是否生效

1.  切换到 `Repeater` 标签，找到之前发送的请求包。

2.  点击 `Send` 发送第一次请求，复制响应中返回的 `user_token` 值。

3.  再次点击 `Send` 发送第二次请求，观察请求行里的 `user_token` 是否自动变化。

    *   若两次 token 不同，说明宏配置成功，每次请求前都会自动获取最新令牌。

    *   若 token 无变化，检查提取边界是否正确、会话规则作用范围是否包含 Repeater。

## 六、步骤 5：配置 Intruder 爆破

1.  切换到 `Intruder` 标签，进入 **Positions** 子标签页：

    *   点击 `Clear §` 清除所有默认标记。

    *   仅选中 `password` 参数的值，点击 `Add §`，将密码设为爆破位置。

    *   攻击模式（Attack type）选择 **Sniper**（用户名固定为 admin，仅爆破密码）。

    *   ⚠️ 注意：`user_token` 不要加标记，它会由宏自动更新。

2.  进入 **Payloads** 子标签页：

    *   Payload set 选择 1，Payload type 选择 `Simple list`。

    *   在 Payload Options 中加载密码字典，DVWA 默认正确密码为 `password`，可手动添加测试。

3.  进入 **Settings** 子标签页，找到 `Request Engine`：

    *   线程数（Number of threads）**必须设为 1**——token 是串行一次性的，多线程会导致令牌错乱、全部失败。

    *   可适当添加 100-300ms 的请求延迟，避免请求过快触发防护。

4.  （可选）在 `Grep - Match` 中添加 `Username and/or password incorrect`，方便快速筛选登录成功的结果。

## 七、步骤 6：启动攻击并通关

1.  点击右上角 `Start attack` 开始爆破。

2.  在攻击结果窗口中，通过响应长度或 Grep Match 结果判断：

    *   大部分请求响应长度一致，且包含「密码错误」提示；

    *   响应长度不同、没有错误提示的那条，对应的密码就是正确密码。

3.  DVWA Brute Force High 级别的默认正确凭证为：

    *   用户名：`admin`

    *   密码：`password`

4.  在浏览器中输入正确账号密码，页面显示 `Welcome to the password protected area admin` 即通关成功。

## 八、常见踩坑点

1.  **必须单线程**：High 级别 token 一次性有效，多线程会导致 token 复用失败，所有请求都返回错误。

2.  **Cookie 安全级别**：确保请求 Cookie 中 `security=high`，否则会命中低级别逻辑。

3.  **Token 提取边界**：边界多一个空格或少一个引号都会导致提取失败，建议用「从响应中选取」的方式自动生成。

4.  **规则作用范围**：会话规则必须勾选 Intruder，否则爆破时不会执行宏。
