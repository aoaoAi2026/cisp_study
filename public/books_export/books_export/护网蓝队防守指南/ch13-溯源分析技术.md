## 13.1 先从一个故事讲起

### 13.1.1 一次漂亮的溯源

先给你讲个真实的故事。

有一年护网，我认识的一个蓝队，姓李，大家叫他老李。

有一天，他们单位被打进来了。
红队从一个Web应用的文件上传漏洞进来的，上传了一个WebShell。

一般的蓝队，可能删了WebShell、补了漏洞就完事了。

但老李不是。
老李说："不行，得查清楚，他们是从哪来的？还有没有其他入口？还会不会再来？"

于是老李开始溯源。

第一步，先查Web日志。
找到了上传WebShell的IP：192.168.x.x（不对，是公网IP，打个比方）。
用这个IP去查威胁情报，发现是个云服务器。

第二步，查流量。
把这个IP的所有流量都导出来，仔细分析。
发现攻击者不光上传了WebShell，还扫了内网，还尝试了很多其他路径。

第三步，分析WebShell。
把WebShell样本拿过来，逆向分析。
发现这个WebShell是定制的，网上找不到现成的。
里面有一些特征，比如加密方式、变量命名风格。

第四步，查攻击者的操作。
把攻击者操作WebShell的所有流量都还原出来。
发现攻击者用的是冰蝎，连接密码有特点。
还发现攻击者的操作习惯：比如喜欢先执行 whoami、再 ipconfig、再 net user...

第五步，关联其他事件。
老李把这个攻击的特征，跟历史上其他几次攻击对比了一下。
发现！
三个月前有一次攻击，WebShell的写法、加密方式、操作习惯，几乎一模一样。
虽然IP不一样，但手法太像了。

"这肯定是同一个人干的！"老李说。

第六步，画像。
根据收集到的所有信息，给攻击者画了个像：
- 技术水平：中等偏上
- 常用工具：冰蝎、nmap、mimikatz
- 攻击习惯：喜欢从Web打点，然后内网横向
- 作息时间：喜欢半夜干活，尤其是凌晨1-3点
- 可能是个老手，经验丰富

最后，老李把这些信息整理成了一份溯源报告，上报给了护网指挥部。
指挥部根据这些信息，对比了其他单位的攻击事件，最后锁定了一支红队。

那次护网，老李他们单位因为溯源做得好，还加了分。

---

这个故事告诉我们什么？

**溯源，不只是"查个IP"那么简单。**

溯源是一个系统工程，要从多个维度、多个角度，把攻击者的方方面面都挖出来。

溯源能力强不强，直接体现了一个蓝队的水平。

这一章，我们就来聊聊溯源分析那些事。

## 13.2 什么是溯源分析？

### 13.2.1 溯源的定义

啥是溯源？

用一句话说：

> **溯源，就是追溯攻击的源头，搞清楚攻击从哪来、是谁干的、怎么干的、干了啥、还有啥目的。**

我再翻译翻译，说得更直白点：

被打了，不能白挨打。
得搞清楚：
- 谁打的？
- 从哪来的？
- 怎么打的？
- 打到哪了？
- 偷了啥？
- 还有没有后门？
- 还会不会再来？

这些问题都搞清楚了，溯源就到位了。

### 13.2.2 溯源的层次

溯源不是一蹴而就的，是分层次的。

一般来说，溯源有这么几个层次：

```
第一层：攻击来源溯源
  • 攻击从哪个IP来的？
  • 在哪个地区？
  • 是机房还是家庭宽带？
  • 是云服务器还是肉鸡？

第二层：攻击手法溯源
  • 用的什么工具？
  • 用的什么手法？
  • 攻击路径是怎样的？
  • 是哪个团伙的惯用手法？

第三层：攻击者溯源
  • 攻击者是谁？
  • 是个人还是组织？
  • 什么来头？
  • 动机是什么？

第四层：攻击归因
  • 是哪个团伙/组织干的？
  • 之前还干过啥？
  • 有什么背景？
  • 下一步可能干啥？
```

一层比一层深，一层比一层难。

一般的蓝队，能做到第一层、第二层就不错了。
厉害的蓝队，能做到第三层、第四层。

当然，越深越难，也越有价值。

### 13.2.3 溯源的目的

为啥要溯源？

```
1. 搞清楚攻击全貌
   • 攻击打到哪一步了？
   • 影响有多大？
   • 还有没有没发现的？

2. 彻底清除威胁
   • 找到所有后门
   • 找到所有被入侵的机器
   • 防止攻击者再回来

3. 改进防护
   • 知道攻击怎么进来的
   • 针对性地加固
   • 防止下次再被同样的方式打进来

4. 法律/合规
   • 收集证据
   • 万一要报警、起诉
   • 满足合规要求

5. 威胁情报
   • 积累攻击样本和IOC
   • 研究攻击者的手法
   • 提高检测能力
```

溯源不是为了"抓犯人"（当然能抓到更好），
更重要的是为了改进自己的防护，防止下次再被打。

## 13.3 网络溯源：从IP开始

最基础的溯源，就是网络溯源——
从攻击IP开始，一层一层往上查。

### 13.1 IP基础信息查询

拿到一个IP，第一步先查它的基本信息。

查什么？

```
归属地查询：
  • 哪个国家？哪个城市？
  • 哪个运营商？
  • 是机房还是家庭宽带？
  • 是公司还是个人？

WHOIS查询：
  • IP段属于哪个公司/机构？
  • 联系人是谁？
  • 联系方式是什么？

威胁情报查询：
  • 这个IP有没有恶意记录？
  • 之前有没有干过坏事？
  • 是什么类型的攻击？
  • 关联了哪些样本/域名？

端口/服务查询：
  • 这个IP开了哪些端口？
  • 跑了哪些服务？
  • 是什么操作系统？
  • 有没有漏洞？
```

用啥查？
- 威胁情报平台（微步在线、Virustotal、威胁情报中心...）
- IP查询网站（ipip.net、ip.cn、纯真IP库...）
- WHOIS查询
- 在线端口扫描（yougetsignal、shodan...）
- Shodan、FOFA这些网络空间搜索引擎

这些是最基础的，拿到一个IP先查一遍。

### 13.3.2 是真实IP还是跳板？

查到IP了，先别急着下结论。
这个IP是攻击者的真实IP吗？
还是说只是个跳板/肉鸡？

大概率是跳板。

现在的攻击者，谁会用自己的真实IP干活啊。
- VPN
- 代理
- 肉鸡
- 洋葱路由（Tor）
- 多重跳板

一层又一层，根本找不到真实IP。

那怎么办？

**几种思路：**

1. **看攻击时间和习惯**
   虽然IP不一样，但攻击时间、手法、工具、习惯是一样的。
   从这些特征，能判断是不是同一个人/同一个组织。

2. **找攻击者的失误**
   再厉害的攻击者，也有失误的时候。
   比如：
   - 某一次忘了挂代理，漏了真实IP
   - 木马的C2域名注册信息里有真实信息
   - 某个文件的元数据里有个人信息
   - 社交工程学信息泄露

   只要找到一个失误，就能顺藤摸瓜。

3. **流量分析**
   如果有全流量的话，可以从流量特征里找线索。
   比如：
   - 攻击流量的特征
   - 工具的特征
   - 心跳包的间隔
   - 加密方式

   这些特征，跟已知的攻击组织比对，就能大概知道是谁。

4. **蜜罐/欺骗**
   布置蜜罐，攻击者打进来之后，故意留一些"假"的信息。
   攻击者把这些信息带回去，你就能追踪到他们的C2。

   当然，这个比较高阶了。

总之，溯源真实IP很难，但不是不可能。
就看你细不细心，有没有耐心。

### 13.3.3 域名溯源

除了IP，域名也是重要的线索。

很多木马、WebShell、远控，都会连接C2域名。

查到一个恶意域名，能挖出来很多东西。

```
查WHOIS：
  • 域名是谁注册的？
  • 注册人是谁？
  • 注册邮箱是什么？
  • 注册商是哪家？
  • 什么时候注册的？

查解析历史：
  • 这个域名都解析过哪些IP？
  • 每个IP用了多久？
  • IP之间有没有关联？

查关联域名：
  • 同一个注册人还有哪些域名？
  • 同一个IP上还有哪些域名？
  • 同一个C段还有哪些恶意域名？

查威胁情报：
  • 这个域名有没有恶意记录？
  • 关联了哪些样本？
  • 是什么家族的木马？
```

域名里能挖的信息很多。
有时候一个注册邮箱，就能牵出一大堆东西。

## 13.4 样本溯源：分析恶意代码

光查IP和域名还不够。
还得分析恶意代码样本。

样本里藏着很多信息。

### 13.4.1 静态分析

拿到一个病毒/木马/WebShell样本，先做静态分析。

啥是静态分析？
就是不运行它，直接看文件本身。

看啥？

```
基本信息：
  • 文件类型？什么语言写的？
  • 文件大小？编译时间？
  • 有没有加壳？有没有混淆？
  • 版本信息、公司名、版权信息（可能泄露线索）

字符串分析：
  • 里面有哪些字符串？
  • 有没有URL、IP、域名？
  • 有没有用户名、密码？
  • 有没有独特的字符串（特征）？
  • 用的什么加密算法？

代码结构：
  • 程序的大致结构是啥样的？
  • 有哪些功能模块？
  • 用了什么技术？
  • 编码风格是啥样的？

对比分析：
  • 跟已知的恶意代码家族像不像？
  • 是不是某个已知家族的变种？
  • 跟之前见过的样本有没有相似之处？
```

静态分析是基础。
很多信息，从文件本身就能看出来。

### 13.4.2 动态分析

静态分析不够，就做动态分析。

啥是动态分析？
就是在沙箱里运行一下，看它干啥。

看啥？

```
文件操作：
  • 创建了哪些文件？
  • 修改了哪些文件？
  • 删除了哪些文件？
  • 释放了哪些文件？

注册表操作：
  • 修改了哪些注册表项？
  • 加了哪些启动项？
  • 改了哪些系统设置？

进程操作：
  • 创建了哪些进程？
  • 注入了哪些进程？
  • 起了哪些服务？
  • 加了哪些计划任务？

网络行为：
  • 连接了哪些IP/域名？
  • 用了什么协议？
  • 传了什么数据？
  • C2地址是啥？
  • 心跳间隔是多少？

系统行为：
  • 有没有提权？
  • 有没有开后门？
  • 有没有偷密码？
  • 有没有加密文件？
```

动态分析能看到恶意代码的实际行为。
C2地址、传播方式、攻击手法，很多都能看出来。

常用的沙箱：
- 微步在线沙箱
- 奇安信沙箱
- Virustotal
- Joe Sandbox
- Cuckoo Sandbox（开源的）

把样本扔进去，跑一遍，报告就出来了。

### 13.4.3 样本特征与家族归属

分析完样本，要搞清楚：
这是哪个家族的恶意代码？
是不是已知的攻击组织？

怎么判断？

看特征：
- 代码结构像不像？
- 加密方式一样吗？
- 字符串风格像不像？
- C2通信协议一样吗？
- 功能模块差不多吗？
- 有什么独特的"标志"？

这些特征，就像人的指纹一样。
每个攻击者、每个攻击组织，都有自己的风格和习惯。

见得多了，一眼就能认出来：
"哦，这个是XX组织的惯用手法。"

当然，这需要积累。
样本见得多了，自然就有感觉了。

## 13.5 攻击溯源：还原攻击全过程

除了溯源"是谁干的"，更重要的是溯源"怎么干的"。

也就是：还原整个攻击过程。

### 13.5.1 攻击时间线

还原攻击过程，第一步是梳理时间线。

什么时间，发生了什么事。

```
时间线应该包括：
  • 攻击者什么时候开始侦察的？
  • 什么时候第一次尝试攻击？
  • 什么时候成功突破的？
  • 什么时候上传的WebShell/木马？
  • 什么时候提的权？
  • 什么时候开始横向移动？
  • 什么时候访问的核心系统？
  • 什么时候偷的数据？
  • 什么时候清理的痕迹？
  • ......
```

把所有事件按时间顺序排好，整个攻击过程就清楚了。

怎么梳理时间线？
- 看日志（各种设备、各种系统的日志）
- 看流量（全流量记录）
- 看文件（文件创建/修改时间）
- 看进程（进程创建时间）
- 看注册表（修改时间）

把所有能找到的时间点都串起来。

时间线越完整，攻击过程就越清晰。

### 13.5.2 攻击路径

然后是攻击路径。

攻击者是怎么一步步打进来的？

```
攻击路径要搞清楚：
  • 入口点是哪里？（Web漏洞？钓鱼邮件？弱口令？）
  • 拿下的第一台机器是哪台？
  • 然后横向移动到了哪些机器？
  • 是怎么横向移动的？（RDP？WMI？PsExec？哈希传递？）
  • 最终目标是什么？（域控？数据库？核心业务？）
  • 到最后打到了哪一步？
```

就像破案一样，把攻击走过的路，一步一步还原出来。

### 13.5.3 影响范围

还要搞清楚影响范围。

```
影响范围包括：
  • 多少台机器被入侵了？
  • 哪些系统受影响了？
  • 哪些数据被偷了？
  • 多少用户受影响？
  • 业务中断了多久？
  • 经济损失有多大？
```

影响范围搞清楚了，才能评估严重程度，才能知道该怎么处置。

### 13.5.4 攻击手法

还有攻击手法。

```
攻击者用了哪些手法？
  • 初始访问：SQL注入？文件上传？钓鱼邮件？暴力破解？
  • 执行：WebShell？木马？宏？PowerShell？
  • 持久化：账号后门？计划任务？服务？注册表？
  • 提权：漏洞提权？配置错误提权？
  • 凭据窃取：Mimikatz？哈希传递？
  • 横向移动：RDP？WMI？PsExec？
  • 数据窃取：打包下载？外发邮件？
  • 痕迹清理：删日志？擦痕迹？
  • ......
```

手法搞清楚了，才能针对性地加固。
这次防住了，下次就不容易再被同样的方式打进来。

## 13.6 攻击者画像

溯源到最后，最好能给攻击者画个像。

啥是攻击者画像？

就是把攻击者的特征都总结出来，就像给嫌疑人画肖像一样。

### 13.6.1 画像都画啥？

```
技术特征：
  • 技术水平怎么样？（高？中？低？）
  • 擅长什么方向？（Web？内网？免杀？）
  • 常用工具有哪些？
  • 常用手法有哪些？
  • 代码风格是啥样的？

行为特征：
  • 喜欢什么时候动手？（白天？半夜？周末？）
  • 攻击节奏是快还是慢？
  • 操作习惯是啥样的？
  • 喜欢用什么工具？
  • 有没有什么特殊的癖好？

组织特征：
  • 是个人还是组织？
  • 大概有多少人？
  • 分工是怎样的？
  • 是哪个国家/地区的？

动机特征：
  • 攻击目的是啥？（偷数据？搞破坏？要钱？）
  • 目标偏好是什么？（喜欢打什么行业？什么系统？）
  • 是黑产？还是APT？还是 script kiddie？
```

画像越详细，对攻击者就越了解。
越了解，就越容易防住他。

### 13.6.2 画像有啥用？

画像是干嘛用的？

```
1. 预测下一步行动
   了解了攻击者的习惯，就能大概猜到他下一步会干啥。
   比如：这个攻击者喜欢半夜干活，那后半夜就重点盯防。

2. 关联历史攻击
   看看之前有没有类似的攻击，是不是同一个人/组织干的。
   历史上他们还用过什么手法？提前做好准备。

3. 针对性防护
   知道攻击者擅长什么、喜欢用什么，就针对性地加强防护。
   比如：他们喜欢用WebShell，那就加强WebShell检测。

4. 威胁情报积累
   把攻击者的特征、IOC、手法，都积累到威胁情报库里。
   下次再出现，立刻就能认出来。
```

画像不是为了"抓他"，是为了"防他"。

当然，如果能抓到那更好，但一般企业级的溯源，到画像这一步就差不多了。

## 13.8 溯源的注意事项

最后，聊聊溯源的几个注意事项。

### 13.8.1 合法合规

这个是第一位的。

溯源的时候，一定要在法律允许的范围内。

- 别去黑别人的系统
- 别去未经授权地扫描
- 别搞入侵、别搞破坏
- 数据来源要合法
- 隐私要保护

别为了溯源，自己先违法了。
那就得不偿失了。

### 13.8.2 证据链完整

溯源的过程中，要注意保留证据。

- 日志要留好
- 样本要保存好
- 流量要存好
- 操作要记录
- 时间线要清晰

证据链要完整。
万一要走法律程序，这些都是证据。

### 13.8.3 大胆假设，小心求证

溯源的时候，经常要猜。
- "会不会是这样？"
- "是不是同一个人？"
- "他是不是用了XX手法？"

猜是可以的，大胆假设。
但一定要小心求证。
有证据了才能下结论，别瞎猜。

没有证据的猜测，只能是推测，不能当结论。

### 13.8.4 别钻牛角尖

溯源不是要查个水落石出，把祖宗十八代都挖出来。

对企业来说，溯源到什么程度就够了？
- 知道攻击怎么进来的
- 知道打到哪一步了
- 知道影响有多大
- 知道该怎么加固
- 知道攻击者的大概手法和习惯

差不多就够了。

别为了"找到真实身份"，投入太多精力，反而耽误了加固和恢复。
主次要分清。

当然，如果是重大事件，或者涉及法律，那另说。

## 13.9 本章总结

这一章我们讲了溯源分析技术，内容很多，给你总结一下：

**什么是溯源：**
- 追溯攻击的源头，搞清楚从哪来、是谁干的、怎么干的
- 溯源分层次：IP→手法→攻击者→归因
- 溯源的目的：搞清楚全貌、彻底清除、改进防护、合规、积累情报

**网络溯源：**
- IP基础信息查询（归属地、WHOIS、威胁情报、端口）
- 判断是真实IP还是跳板（大概率是跳板）
- 域名溯源（WHOIS、解析历史、关联域名）

**样本溯源：**
- 静态分析（文件信息、字符串、代码结构）
- 动态分析（文件、注册表、进程、网络、系统行为）
- 样本特征与家族归属

**攻击溯源：**
- 攻击时间线（什么时候发生了什么）
- 攻击路径（怎么一步步打进来的）
- 影响范围（多少机器、多少数据、多大损失）
- 攻击手法（用了哪些技术）

**攻击者画像：**
- 技术特征、行为特征、组织特征、动机特征
- 作用：预测行动、关联历史、针对性防护、积累情报

**溯源工具：**
- 信息查询类（IP/域名、威胁情报、网络空间搜索）
- 样本分析类（沙箱、逆向工具、辅助工具）
- 日志/流量分析类

**注意事项：**
- 合法合规（最重要！）
- 证据链完整
- 大胆假设，小心求证
- 别钻牛角尖，适可而止

记住一句话：

> **溯源不是目的，改进才是目的。**

溯源是为了更好地防守，不是为了"抓坏人"。
把自己的防御建好了，比什么都强。
