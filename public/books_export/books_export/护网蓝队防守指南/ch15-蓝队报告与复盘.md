# 第十五章 蓝队报告与复盘 —— 吃一堑长一智

> 第15章 | 预计学习时间：3小时 | 难度：⭐⭐⭐

## 📖 写在前面

事情处理完了，是不是就结束了？

当然不是。

老祖宗说得好：
> "吃一堑，长一智。"

被打了不可怕，可怕的是被打了还不长记性，下次还在同一个地方挨打。

所以，事情处理完了，还得做两件事：
- **写报告**：把事情说清楚
- **复盘**：总结经验教训，改进

报告和复盘，是蓝队工作的重要组成部分。
做得好，能让领导满意、让团队成长、让防护更上一层楼。
做得不好，那就是白忙活。

这一章，我们就来聊聊：
- 安全事件报告怎么写？
- 复盘会怎么开？
- 经验教训怎么落地？
- 怎么把"吃一堑"变成"长一智"？

坐稳了，开唠。

## 15.1 先从一个故事讲起

### 15.1.1 两个团队的不同结局

先给你讲个故事。

有两家公司，A公司和B公司。
同一年，两家都被攻击了，情况差不多，都是Web漏洞被利用，拖了库。

我们来看看两家是怎么做的。

---

**A公司的做法：**

事件处理完了，漏洞补了，数据改了，系统恢复了。

领导问："怎么样了？"
安全负责人说："没事了，都处理好了。"
领导说："行，那就好。"

然后......就没有然后了。
报告？没写。
复盘？没开。
改进？再说吧。

过了半年，A公司又被打了。
这次还是类似的手法，还是从Web进来的，还是拖了库。
损失比上次还大。

领导怒了："你们安全部是干什么吃的？上次不是说处理好了吗？怎么又出事了？"

安全负责人也很委屈："这次是另一个系统的漏洞......"

---

**B公司的做法：**

事件处理完了，安全团队做了三件事：

第一件，写了一份详细的事件报告。
报告里写清楚了：
- 事件是怎么发生的？
- 影响有多大？
- 我们是怎么处理的？
- 花了多长时间？
- 还有哪些问题？
- 以后该怎么改进？

第二件，开了个复盘会。
安全、开发、运维、测试、业务部门都来了。
大家坐在一起，从头到尾过了一遍。
- 哪里做得好？
- 哪里做得不好？
- 漏洞为什么会出现？
- 检测为什么不够及时？
- 响应为什么慢了？
- 以后该怎么避免？
- 每个部门该做什么？

第三件，制定了改进计划。
- 三个月内，所有Web系统过一遍安全测试
- 新增WAF规则，增加代码审计
- 安全培训每个季度搞一次
- 建立漏洞管理流程，高危漏洞24小时内修复
- 日志接入SIEM，提高检测能力

每项都有负责人、有时间节点、有验收标准。

然后，每过一段时间，就跟进一下改进事项的进展。
没做完的，催；做完了的，验收。

半年后，B公司也遇到了类似的攻击尝试。
但这次，WAF直接拦下来了，根本没打进来。
安全团队在监控大屏上看到告警，点了个"已拦截"，就完事了。

---

同样是被攻击，两家公司的结局完全不同。
A公司在同一个坑里摔了两次，B公司爬起来后把坑填上了。

差别在哪？
就在**报告和复盘**。

这就是为什么我们要写报告、要复盘。
报告是总结，复盘是反思，最终目的都是改进。

这一章，我们就好好聊聊这件事。

## 15.2 安全事件报告怎么写？

先说说安全事件报告。

### 15.2.1 为什么要写报告？

很多人觉得写报告麻烦、没用。
"事情都处理完了，写那玩意儿干啥？"

错。报告很重要。

报告有啥用？

```
1. 给领导看的
   领导需要知道发生了什么事、影响多大、你们干了啥、花了多少钱/时间、以后怎么防。
   你不写报告，领导怎么知道你们干了啥？怎么向上面交差？怎么给你们申请资源？

2. 留档备查
   万一以后再出类似的事，可以翻出来看看，上次是怎么处理的？
   万一要走法律程序，报告就是证据。
   合规审计也需要。

3. 总结经验教训
   写报告的过程，就是梳理的过程。
   写着写着，你就会发现：哦，原来这里还有问题，当时应该这样做就好了。
   这些总结，是最宝贵的财富。

4. 给其他团队/其他单位参考
   你们踩过的坑，别人别再踩了。
   安全圈的分享精神，就是这么来的。
```

报告不是写给别人看的，是写给自己的。
是为了让自己成长，让团队成长。

### 15.2.2 报告的基本结构

一份好的安全事件报告，应该包含哪些内容？

我给你一个通用的模板：

```
一、事件概述
   • 事件名称/编号
   • 事件类型（Web入侵、数据泄露、勒索病毒、DDoS...）
   • 严重等级（特别重大、重大、较大、一般）
   • 发生时间、发现时间、处置时间
   • 影响范围
   • 事件摘要（一两句话说清楚）

二、事件详情
   • 事件背景
   • 攻击过程/时间线
   • 攻击手法
   • 影响评估
     - 业务影响
     - 数据影响
     - 经济损失
     - 声誉影响

三、处置过程
   • 发现过程
   • 响应团队和分工
   • 处置步骤（按时间顺序）
   • 做了哪些措施
   • 恢复过程
   • 花了多少时间、多少人力

四、原因分析
   • 根本原因是什么？
   • 为什么会发生？
   • 技术原因
   • 管理原因
   • 流程原因
   • 人员原因

五、经验教训
   • 哪里做得好？（值得肯定的）
   • 哪里做得不好？（需要改进的）
   • 有什么经验和教训？

六、改进建议
   • 技术改进
   • 管理改进
   • 流程改进
   • 人员改进
   • 短期措施
   • 长期措施

七、附录
   • 证据清单
   • 日志截图
   • 数据统计表
   • 相关链接
```

七大部分，基本上把该说的都说清楚了。

当然，根据事件的大小、公司的要求，可以适当调整。
小事件可以简化，大事件要详细。

### 15.2.3 写报告的注意事项

写报告有几个注意事项，我给你提个醒。

**第一，客观真实，别吹牛，也别甩锅。**

报告要实事求是。
- 是怎么样就怎么样，别夸大，别缩小
- 做得不好的地方，如实写，别藏着掖着
- 是谁的责任就是谁的责任，但别上来就甩锅
- 重点在解决问题，不在追究责任

一写报告就甩锅，下次没人愿意跟你配合了。

**第二，条理清晰，重点突出。**

别一大段一大段的，没人看得下去。
- 多用标题、分点
- 重要的内容放前面
- 数据说话，别光说"很严重"，要说"影响了X台机器、X条数据"
- 图文并茂，该放截图放截图

**第三，区分事实和推测。**

报告里，哪些是事实、哪些是推测，要分清楚。
- 有证据的就是事实
- 猜的、没证据的就是推测
- 别把推测当事实写
- 推测的内容，要注明"初步判断"、"疑似"、"可能"

**第四，不同对象，不同版本。**

给领导看的，就简洁点，重点说结论、说影响、说建议。
领导没时间看细节。

给技术团队看的，就详细点，把技术细节、攻击手法、处置过程都写清楚。
技术人员需要了解细节。

给外部的（客户、监管），就更要谨慎了，什么能说什么不能说，要把好关。

一份报告，多个版本，很正常。

**第五，及时写，别拖。**

事件刚处理完，印象最深刻，赶紧写。
拖久了，很多细节就忘了，写出来就不准了。

一般来说，事件处理完后3天内，报告初稿要出来。
一周内，正式报告要出来。

### 15.2.4 反面例子：一份"烂"报告长啥样？

给你看个反面例子，感受一下。

```
事件报告

今天我们系统被攻击了，好像是黑客从网站进来的，上传了个WebShell。
我们发现后，就把WebShell删了，漏洞也补了。
现在系统已经正常了，没什么事了。
以后我们会加强防护的。

——安全部
```

就这么几行，你觉得领导看了会满意吗？

- 攻击是怎么发生的？不知道
- 影响有多大？不知道
- 数据有没有泄露？不知道
- 为什么会有这个漏洞？不知道
- 以后怎么加强？不知道

等于没说。

这种报告，写了还不如不写。
领导一看，就知道你们不专业。

别写这种报告。

## 15.3 复盘会怎么开？

报告写完了，还要开复盘会。

复盘不是批斗会，是总结会。
目的只有一个：**下次做得更好**。

### 15.3.1 复盘的目的

开复盘会，是为了啥？

```
1. 把事情说清楚
   大家坐在一起，把整个事件从头到尾过一遍。
   每个人说说自己做了什么、看到了什么。
   把大家的信息拼在一起，还原全貌。

2. 找问题
   哪里做得不好？
   哪里出了问题？
   为什么会出问题？
   把问题都找出来。

3. 总结经验
   哪里做得好？
   哪些方法有效？
   好的经验要保留、要推广。

4. 定改进措施
   问题找到了，怎么改？
   谁来改？什么时候改完？
   定下来，下次就不会再犯。

5. 统一认识
   大家对这个事件达成共识。
   不是某个人的锅，是我们整体的问题。
   一起改进，一起进步。
```

复盘的核心是"对事不对人"。
大家是来解决问题的，不是来吵架的。

### 15.3.2 复盘的参与人员

哪些人要参加复盘会？

```
必须来的：
  • 安全团队（主角）
  • 运维团队（系统、网络）
  • 开发团队（业务系统）
  • 测试团队（质量保障）

根据需要来的：
  • 业务部门（了解业务影响）
  • 法务/合规（法律、合规问题）
  • 公关（舆情影响）
  • 客服（客户咨询）
  • 领导（拍板、要资源）

主持人：
  • 一般由安全负责人或部门负责人主持
  • 要能控场，别开成批斗会
```

相关的人都要叫来。
别安全部门自己关起门来复盘，那很多问题你找不到根本原因。

比如漏洞是开发写的，你不叫开发来，怎么讨论怎么避免？

### 15.3.3 复盘的步骤

复盘会怎么开？
给你一个通用的流程：

```
第一步：回顾目标（5分钟）
  • 当初我们的目标是什么？
  • 预期是怎样的？

第二步：评估结果（10分钟）
  • 实际情况是怎样的？
  • 哪些达到了预期？
  • 哪些没达到？
  • 有哪些意外？

第三步：分析原因（30-60分钟）
  • 为什么会出问题？
  • 根本原因是什么？
  • 做得好的地方，为什么做得好？
  • 一层一层往下挖，别停留在表面

第四步：总结经验（20分钟）
  • 从这件事里，我们学到了什么？
  • 哪些经验可以复用？
  • 哪些教训要记住？

第五步：制定行动计划（20分钟）
  • 接下来要做什么？
  • 谁来做？
  • 什么时候做完？
  • 怎么验证？

第六步：总结收尾（5分钟）
  • 总结一下会议结论
  • 强调一下"对事不对人"
  • 感谢大家参与
```

整个会议，控制在2小时以内比较好。
太长了，大家都累，效率也低。

### 15.3.4 复盘的关键技巧

复盘会能不能开好，有几个关键点。

**第一，对事不对人。**

这是最重要的。
复盘是来解决问题的，不是来追究责任的。

一上来就"这是谁的错？"，那没人敢说话了。
最后变成了批斗会，问题没解决，还伤了和气。

正确的做法是：
- "我们一起看看，问题出在哪？"
- "为什么会发生这种情况？"
- "我们怎么改进，才能避免下次再发生？"

把焦点放在"事"上，别放在"人"上。

**第二，多问几个"为什么"。**

别停留在表面，要挖到根。

举个例子：
- 问："为什么被攻击了？"
  答："因为有个漏洞。"
- 问："为什么有漏洞？"
  答："开发写代码的时候没注意。"
- 问："为什么没注意？"
  答："不知道要这么写，没接受过安全培训。"
- 问："为什么没培训？"
  答："公司没有相关的培训计划。"
- 问："那测试呢？测试没测出来？"
  答："测试只测功能，不测安全。"

挖到这，根本原因就找到了：
不是某个开发的错，是公司没有安全培训、没有安全测试流程。

那改进措施就不是"罚那个开发一顿"，而是：
- 建立安全培训机制
- 建立安全测试流程
- 代码安全审计

这样才能从根本上解决问题。

**第三，别光找问题，也要肯定做得好的地方。**

别复盘会一开，全是批评。
做得好的地方，也要说出来，也要肯定。

比如：
- "这次检测很及时，10分钟就发现了，值得表扬。"
- "运维响应很快，半个小时就把系统恢复了，辛苦了。"

有表扬有批评，大家才愿意来。
一上来就全是骂，下次没人愿意说话了。

**第四，行动项要具体，要有责任人、有时间点。**

复盘会最怕什么？
最怕说了一大堆，最后啥也没落地。

"以后要加强防护。"——怎么加强？谁来做？什么时候做完？
"要提高安全意识。"——怎么提高？谁负责？多久搞一次？

这种话，说了等于没说。

行动项一定要具体：
- 做什么？（具体的事）
- 谁来做？（责任人）
- 什么时候做完？（截止时间）
- 怎么算做完了？（验收标准）

比如：
> "在2024年6月30日前，由安全部小李牵头，完成所有在线Web系统的安全扫描，输出扫描报告和整改清单。"

这才叫行动项。

### 15.3.5 复盘之后呢？

复盘会开完了，就完了？

当然不是。

**第一，要有行动项跟踪。**

会议定的行动项，要有人跟进。
- 每个月看一下进展
- 到了时间点，检查有没有做完
- 没做完的，问问原因，催一催
- 做完了的，验一验效果

别开完会就扔一边了，那等于白开。

**第二，要定期回顾。**

比如每个季度，把所有的事件复盘拿出来看一下。
- 改进措施都落地了吗？
- 还有没有类似的问题？
- 有没有新的问题？

持续改进，越做越好。

**第三，要沉淀成知识。**

好的经验、好的方法，要沉淀下来。
- 写成文档
- 做成SOP
- 更新到应急预案里
- 培训给大家

让一个人的经验，变成整个团队的经验。

## 15.4 经验教训怎么落地？

最后，聊聊经验教训怎么落地。

很多单位，复盘也复了，改进措施也定了，
但过了半年一看，啥也没变。
该有的问题还是有，该出的事还是出。

为啥？
因为没落地。

经验教训怎么才能落地？

### 15.4.1 从技术上落地

技术上的改进，相对容易落地。

```
漏洞要补 → 补丁管理流程
检测要准 → 优化SIEM规则、加检测工具
响应要快 → 自动化响应、SOAR
防护要强 → 加设备、加策略、做加固

这些都是可以量化的：
  • 多少个补丁打完了？
  • 多少条规则加上了？
  • 多少个系统加固了？
  • 平均响应时间降到多少了？
```

技术上的事，只要有人做、有时间做，就能落地。

### 15.4.2 从流程上落地

流程上的改进，稍微难一点。

```
漏洞发现了不及时修 → 建立漏洞管理SLA
  • 高危漏洞24小时内修复
  • 中危漏洞7天内修复
  • 低危漏洞30天内修复

出了事不知道找谁 → 建立应急响应流程
  • 什么等级的事件，找谁，怎么上报
  • 每个角色的职责是什么
  • 处置流程是什么

开发写代码老出漏洞 → 把安全嵌入开发流程
  • 需求阶段考虑安全
  • 开发阶段遵循安全规范
  • 测试阶段有安全测试
  • 上线前有安全检查
```

流程这东西，定了就要执行。
不执行，就是一张废纸。

怎么保证执行？
- 有制度支撑
- 有考核挂钩
- 有工具保障（比如不通过安全检查就不让上线）

### 15.4.3 从人上落地

人的改进，是最难的。

安全意识怎么提高？
- 培训
- 演练
- 考核
- 文化

安全意识这东西，不是一次培训就能搞定的。
要常讲、常练、常提醒。

比如：
- 每月发一封安全提醒邮件
- 每季度搞一次安全培训
- 每半年搞一次钓鱼演练
- 新员工入职必须过安全培训

慢慢的，安全意识就上来了。

### 15.4.4 从管理上落地

最后，也是最重要的——管理上的支持。

很多改进，光靠安全部门推不动。
得有领导支持。

比如：
- 要加设备、加人，得领导批预算
- 要改流程、定制度，得领导拍板
- 要推动业务部门配合，得领导发话

所以，跟领导的沟通很重要。
- 让领导知道安全的重要性
- 让领导看到安全的价值
- 让领导愿意投入

领导重视了，很多事就好办了。

## 15.5 本章总结

这一章我们讲了报告和复盘，给你总结一下：

**安全事件报告：**
- 为什么写：给领导看、留档备查、总结经验、分享参考
- 基本结构：概述、详情、处置、原因、教训、建议、附录
- 注意事项：客观真实、条理清晰、区分事实和推测、不同对象不同版本、及时写
- 别写那种几句话的"烂报告"

**复盘会：**
- 目的：说清楚、找问题、总结经验、定措施、统一认识
- 参与人员：安全、运维、开发、测试、业务、领导...相关的都来
- 步骤：回顾目标→评估结果→分析原因→总结经验→制定计划→总结收尾
- 关键技巧：
  - 对事不对人
  - 多问几个"为什么"，挖到根
  - 既要批评，也要肯定
  - 行动项要具体，有责任人、有时间点、有验收标准
- 复盘之后：要跟踪、要回顾、要沉淀

**经验落地：**
- 技术上落地：设备、工具、策略，相对容易
- 流程上落地：建立SOP、制度、规范，要执行
- 人上落地：培训、演练、文化，最难但也最重要
- 管理上落地：领导支持是关键

最后送你一句话：

> **被攻击不可怕，不总结才可怕。**

吃一堑，就要长一智。
每一次被攻击，都是一次成长的机会。
就看你能不能抓住。
