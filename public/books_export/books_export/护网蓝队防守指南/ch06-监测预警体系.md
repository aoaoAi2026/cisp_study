## 6.1 先从一个故事讲起

### 6.1.1 小明的第一次值班

先给你讲个真实的故事。

小明是个刚入行的蓝队工程师，第一次值夜班。

晚上12点，小明坐在监控室里，盯着大屏。

大屏上花花绿绿的，各种曲线、各种图表、各种数字，看起来挺酷的。

小明看了一会儿，觉得挺无聊的。

"这不就是看看大屏吗？有啥难的？"

凌晨2点，小明有点困了，打了个哈欠。

就在这时，SIEM上跳出来一个告警："检测到异常的远程登录行为"。

小明看了一眼，IP是内部IP，用户名是一个普通员工。

"哦，可能是有人加班吧。"小明没当回事，随手把告警关了。

然后继续打哈欠。

又过了一会儿，又跳出来一个告警："检测到异常的端口扫描行为"。

来源还是那个IP。

"哦，可能是开发在测试吧。"小明又没当回事。

就这样，一晚上过去了，啥事儿也没有。

第二天早上，交班的时候，队长问："昨晚有啥情况没？"

小明说："没啥事，就几个告警，都是误报。"

队长点了点头。

结果呢？

上午10点，出事了。

业务部门说，核心数据库好像被人访问过了，有异常查询。

队长赶紧查。

一查，坏了。

昨晚凌晨2点多，有个员工的账号被盗了，攻击者用这个账号登录了服务器，然后横向移动，最后访问了核心数据库。

而这些，小明昨晚都看到告警了，但都当成误报给忽略了。

结果呢？数据被"偷"了。

小明吓得一身冷汗。

队长说："你呀，还是太年轻了。监测不是光盯着大屏看，是要动脑子的。告警来了，你得分析，得判断，得搞清楚到底是咋回事。"

"你以为监测就是看看大屏？那是保安看大门。"

"真正的监测，是从海量的信息里，找出那一根隐藏的针。"

小明记住了这句话。

---

这个故事告诉我们什么？

监测预警，不是光有设备就行的。

- 有设备，没人会用，等于白搭。
- 有告警，没人会分析，等于白搭。
- 有数据，没人会看，等于白搭。

监测预警，是技术+经验+细心的结合。

## 6.2 监测预警体系长啥样？

### 6.2.1 监测的三个层次

监测预警，不是一个单一的东西，它是有层次的。

我把它分成三个层次：

```
第一层：基础监测——看得见
  有日志、有流量、有告警
  知道发生了什么

第二层：智能监测——看得准
  告警降噪、关联分析、威胁情报
  能区分真假攻击

第三层：主动监测——看得早
  威胁狩猎、主动发现、预测预警
  攻击还没造成损失就发现
```

一层比一层高级。

很多单位，还停留在第一层——
有日志、有告警，但全是误报，根本看不过来。
等于看不见。

好一点的单位，到了第二层——
告警没那么多了，能区分真假了，
能把相关的告警串起来了。
能看得准了。

最厉害的单位，到了第三层——
不光等着告警，还主动去找攻击。
威胁狩猎，主动发现。
攻击刚露头，就被发现了。

这才是真正的监测预警。

### 6.2.2 监测的数据源

监测，得有数据吧？

数据从哪来？

主要有这么几类：

```
第一类：日志数据
  • 防火墙日志
  • IDS/IPS日志
  • WAF日志
  • 系统日志（Windows/Linux）
  • Web服务器日志
  • 数据库日志
  • 应用日志
  • 安全产品日志
  • ......

第二类：流量数据
  • 全流量PCAP
  • 流量元数据（NetFlow/IPFIX）
  • 流量统计数据
  • ......

第三类：端点数据
  • 进程信息
  • 文件信息
  • 注册表信息
  • 网络连接
  • 行为数据
  • ......

第四类：威胁情报
  • IOC（IP、域名、文件哈希）
  • 攻击手法
  • 威胁 actor
  • 漏洞情报
  • ......

第五类：其他数据
  • 身份认证日志
  • 运维操作日志
  • 物理门禁日志
  • ......
```

数据越多，越全，越容易发现攻击。

但也不是越多越好——
太多了，看不过来，反而成了负担。

关键是要有用。

## 6.3 常见的监测手段

### 6.3.1 日志监测

最基础、最常用的监测手段——日志监测。

啥是日志？

就是各种设备、系统、应用产生的记录。
谁什么时候干了啥，都记在日志里。

日志监测就是看这些日志，找异常。

**常见的日志类型：**

```
网络设备日志：
  • 防火墙日志：谁访问了啥、被拦了还是放行了
  • 路由器/交换机日志：接口状态、路由变化
  • VPN日志：谁登录了VPN、从哪登的

安全设备日志：
  • IDS/IPS日志：检测到什么攻击
  • WAF日志：检测到什么Web攻击
  • EDR/杀毒日志：查到了什么病毒木马
  • 邮件网关日志：拦截了什么垃圾邮件、钓鱼邮件

系统日志：
  • Windows事件日志：登录、登出、进程创建、权限变化...
  • Linux syslog：登录、命令执行、服务状态...
  • 认证日志：谁登录了、成功还是失败

应用日志：
  • Web访问日志：谁访问了哪个页面、状态码是啥
  • 应用运行日志：应用运行情况、错误信息
  • 数据库日志：谁查询了什么数据、修改了什么数据
```

**日志监测看啥？**

主要看异常：

- 异常的登录：半夜登录、陌生IP登录、多次失败后成功...
- 异常的操作：执行了奇怪的命令、访问了不该访问的文件...
- 异常的访问：访问了敏感页面、大量的404、SQL注入特征...
- 异常的时间：不该有人操作的时候有操作...

日志是蓝队的眼睛。

没有日志，就等于瞎子。

### 6.3.2 流量监测

比日志更"真"的，是流量监测。

为啥？

因为日志可以被篡改、可以被删除、可以被绕过。
但流量不会骗人。

只要有通信，就有流量。
攻击者可以删日志，但他删不掉流量记录（除非你没存）。

**流量监测有啥用？**

```
东西向流量监测（南北向=出入内网的流量）：
  • 发现端口扫描
  • 发现漏洞攻击
  • 发现WebShell通信
  • 发现C2通信
  • 发现数据外泄

东西向流量监测（内网之间的流量）：
  • 发现横向移动
  • 发现内网扫描
  • 发现远程登录
  • 发现内网渗透
  • 发现异常的内网访问
```

**常见的流量监测方式：**

1. **全流量存储（PCAP）**
   把所有流量包都存下来，想查的时候随时查。
   好处是全，啥都能查到。
   坏处是太占存储空间了，成本高。

2. **流量元数据（NetFlow/IPFIX）**
   不存完整的包，只存摘要信息：谁和谁通信、什么协议、多少字节、多长时间...
   好处是占空间小，能存很久。
   坏处是细节少，有些东西查不到。

3. **流量分析引擎**
   实时分析流量，发现异常就告警。
   比如入侵检测（IDS）就是干这个的。
   好处是实时。
   坏处是只能看有没有攻击包的内容，看不到。

流量监测很重要，特别是在溯源的时候。
日志被删了，流量还在，就能从流量里找线索。

### 6.3.3 端点监测

日志和流量，都是"外围"监测。
端点监测，是直接监控终端和服务器本身。

啥是端点？
就是服务器、个人电脑这些。

端点监测看啥？

```
进程监测：
  • 有没有可疑的进程？
  • 进程的父进程是什么？
  • 进程是从哪来的？
  • 进程干了啥？

文件监测：
  • 有没有新建可疑文件？
  • 系统文件被改了吗？
  • 有没有WebShell文件？
  • 有没有恶意文件？

注册表监测（Windows）：
  • 启动项被改了吗？
  • 注册表有异常修改吗？
  • 有没有注册表后门？

网络连接监测：
  • 进程连了什么IP？
  • 有没有可疑的外联？
  • 有没有异常的端口？

行为监测：
  • 有没有异常的行为？
  • 是不是像攻击者的操作习惯？
  • 有没有提权行为？
  • 有没有横向移动的迹象？
```

端点监测有啥好处？
- 更细，能看到系统内部发生了什么
- 更准，能发现很多网络层发现不了的攻击
- 更快，攻击刚执行就能发现

现在很火的EDR（终端检测与响应），就是干这个的。

### 6.3.4 身份监测

还有一种很重要的监测——身份监测。

很多攻击，最后都是偷账号、偷密码。
然后用偷来的账号登录，装作是"正常用户"。

这时候，传统的监测手段就不好使了——
因为他用的是真实的账号、真实的密码、正常的操作。
看起来跟正常用户一模一样。

怎么办？

身份监测。

身份监测看啥？

```
登录行为：
  • 登录时间正常吗？（半夜登录？）
  • 登录地点正常吗？（从没去过的地方？）
  • 登录设备正常吗？（新设备？）
  • 登录频率正常吗？（短时间内多次登录？）
  • 失败次数正常吗？（很多次失败后成功？）

操作行为：
  • 访问的资源正常吗？（平时不访问的系统？）
  • 操作量正常吗？（突然大量下载？）
  • 操作行为符合这个用户的习惯吗？

权限变化：
  • 权限突然提升了？
  • 加入了管理员组？
  • 有异常的权限申请？
```

简单说就是：通过行为分析，判断"这个人是不是真的是他本人"。

就像你平时都是早上9点上班，从公司登录。
突然有一天，凌晨3点，从国外登录了。
那大概率不是你本人，是账号被盗了。

这就是身份监测的思路。

现在很火的"UEBA"（用户和实体行为分析），就是干这个的。

### 6.3.5 威胁情报

最后一个，威胁情报。

啥是威胁情报？

简单说就是：
- 最近有什么新的攻击手法？
- 最近有什么新的漏洞？
- 最近有哪些黑客组织在活动？
- 哪些IP、域名、文件哈希是恶意的？

这些信息，就是威胁情报。

威胁情报有啥用？

```
检测：
  • 把威胁情报里的IOC（恶意IP、域名、哈希）导入到检测设备里
  • 一匹配上，就告警
  • 比如：这个IP在威胁情报里是恶意的，它访问我们了，赶紧拦

预警：
  • 最近有新的漏洞出来了，赶紧检查我们有没有受影响
  • 最近有针对我们这个行业的攻击，提前做好准备
  • 最近有新的攻击手法，提前更新检测规则

溯源：
  • 攻击手法符合哪个组织的风格？
  • 这个IP之前有没有干过坏事？
  • 这个木马是哪个团伙的？
```

威胁情报就像是"天气预报"——
告诉你最近可能会有什么"天气"，让你提前做好准备。

威胁情报从哪来？
- 商业威胁情报厂商（比如微步、天际友盟...）
- 开源威胁情报（比如VT、MalwareBazaar...）
- 行业共享（行业内互相交换情报）
- 自己积累（自己分析攻击，积累情报）

有了威胁情报，监测就不是"瞎摸"了，而是"精准打击"。

## 6.4 告警管理：从海量告警中找真攻击

### 6.4.1 告警的痛点

很多人以为，监测就是告警越多越好。

大错特错。

告警太多了，反而是坏事。

为啥？

因为：
- 一天几千几万条告警，看都看不过来
- 大部分都是误报，看着看着就麻木了
- 真的攻击告警，反而被淹没在误报里，没人发现

这就是"告警疲劳"。

很多单位，SIEM一天产生几万条告警。
谁看得过来？
看不过来，就只能不看。
那跟没有告警有啥区别？

所以，告警不是越多越好，是越准越好。

### 6.4.2 告警降噪

怎么解决告警太多的问题？

两个字：降噪。

怎么降噪？

**方法一：误报过滤**

先把已知的误报过滤掉。

比如：
- 内部的安全扫描器，经常触发IDS告警，把扫描器的IP加白名单
- 正常的业务访问，经常触发WAF告警，把这个URL加白
- 已知的正常行为，就别告警了

把那些确定是误报的，都过滤掉。
这一步就能干掉一大部分告警。

**方法二：告警分级**

告警不是都一样重要的。

分个级：
- 高危告警：必须立刻处理
- 中危告警：尽快处理
- 低危告警：有空再看
- 信息类：基本不用管

重点盯着高危的看，低危的没事了再扫一眼。
别在低危告警上浪费太多时间。

**方法三：告警聚合**

很多告警，其实是同一件事。

比如：
- 同一个IP，一分钟内触发了100条SQL注入告警
- 这其实就是一次攻击，不是100次

把它们聚合成一条告警。
这样数量不就少了？

**方法四：关联分析**

单个告警说明不了什么。
但好几个告警凑到一起，就说明问题了。

比如：
- 一个IP在扫描 → 低优先级
- 这个IP在尝试Web攻击 → 中优先级
- 这个IP成功上传了WebShell → 高优先级

单个看，可能都没啥。
串起来看，就是一次完整的攻击。

关联分析，就是把相关的告警串起来，形成一个完整的"攻击故事"。

这样告警就少了，而且更准了。

关于关联分析，我们下一章专门讲。

### 6.4.3 告警研判流程

告警来了，怎么判断是真攻击还是误报？

给你一个简单的研判流程：

```
第一步：看告警本身
  • 告警的规则是什么？
  • 触发的条件是什么？
  • 原始日志/原始数据包是什么？
  → 先搞清楚，这个告警在说啥

第二步：看上下文
  • 这个IP是谁？已知的恶意IP？还是正常业务IP？
  • 这个用户是谁？正常员工？还是管理员？
  • 这个时间正常吗？
  • 这个行为正常吗？
  → 放在上下文中看，是不是合理

第三步：查关联
  • 这个IP/用户，最近还有没有其他告警？
  • 有没有其他相关的行为？
  • 有没有后续的动作？
  → 关联起来看，是不是完整的攻击链

第四步：验证
  • 被攻击的机器，有没有被入侵的痕迹？
  • 有没有木马？有没有WebShell？有没有异常账号？
  • 数据有没有被偷？
  → 深入验证，确认是不是真的

第五步：定性
  • 是误报？那标记一下，以后别再告警了
  • 是真实攻击？那赶紧启动应急响应
  • 不确定？那继续查，直到搞清楚为止
```

按这个流程来，一步一步分析，基本不会错。

## 6.5 威胁狩猎：主动出击

### 6.5.1 什么是威胁狩猎？

前面讲的监测，都是"被动"的——
等着告警，告警来了再分析。

但真正厉害的蓝队，不满足于被动等着。
他们主动出击，去找攻击。

这就是**威胁狩猎（Threat Hunting）**。

啥叫威胁狩猎？

用一句话说：

> **主动地、有假设地去寻找网络中潜在的攻击者，而不是等着告警告诉你。**

翻译成人话就是：
别等着告警响了才动手。
自己主动去查，去翻，去找。
看看有没有藏得很深的、告警没发现的攻击。

就像警察巡逻——
不是等着有人报警才出警，而是主动在街上巡逻，看看有没有可疑的人。

### 6.5.2 为什么需要威胁狩猎？

为啥要主动狩猎？告警不好吗？

因为告警有局限性：

1. **告警只能发现已知的攻击。**
   告警是靠规则匹配的，规则是根据已知的攻击手法写的。
   新的攻击手法、0day漏洞、高级持续性威胁（APT），
   很可能没有告警，悄无声息就进来了。

2. **高级攻击者会隐蔽自己。**
   厉害的攻击者，会尽量不触发告警。
   免杀木马、慢速攻击、合法工具滥用...
   等告警响的时候，人家早就打完收工了。

3. **有些攻击根本没有明显的特征。**
   比如用偷来的账号正常登录，看起来跟正常用户一模一样。
   你怎么告警？没法告警。
   但行为分析、异常检测，可能会发现不对劲。

所以，光靠被动告警不够。
还得主动狩猎。

### 6.5.3 威胁狩猎怎么玩？

威胁狩猎不是瞎找，是有方法的。

常见的狩猎方法：

**方法一：基于假设的狩猎**

先提出一个假设，然后去验证这个假设对不对。

比如：
- 假设："我们可能被APT组织盯上了" → 去查有没有APT的IOC
- 假设："可能有WebShell没被发现" → 去查WebShell的特征
- 假设："可能有横向移动行为" → 去查异常的远程登录

先有假设，然后去找证据，验证假设。

**方法二：基于情报的狩猎**

根据威胁情报来狩猎。

比如：
- 最近出了个新漏洞 → 去查我们有没有被这个漏洞攻击
- 最近有个新的木马 → 去查我们网络里有没有这个木马的IOC
- 最近有针对我们行业的攻击 → 去查有没有相关的攻击痕迹

威胁情报给你指引方向，你顺着方向去查。

**方法三：基于异常的狩猎**

找异常行为。

比如：
- 找半夜登录的账号
- 找从没见过的进程
- 找异常的外联流量
- 找大量下载数据的行为
- 找不符合正常用户习惯的操作

不正常的，就重点查。

### 6.5.4 威胁狩猎的流程

威胁狩猎一般分几步：

```
第一步：确定狩猎目标
  这次猎啥？
  是找WebShell？还是找横向移动？还是找持久化？
  先定个目标，别瞎找。

第二步：收集数据
  要猎这个目标，需要哪些数据？
  日志？流量？端点数据？
  把数据准备好。

第三步：分析筛选
  从数据里，找出可疑的、异常的。
  缩小范围，把可疑的挑出来。

第四步：深入调查
  对可疑的目标，深入查。
  查它的来龙去脉，搞清楚到底是啥。
  是正常的？还是真的攻击？

第五步：总结反馈
  • 如果发现了攻击 → 赶紧处置，然后更新检测规则
  • 如果没发现 → 总结经验，优化狩猎方法
  • 把狩猎的结果，反馈到监测体系里
  → 这样下次就能自动检测了，不用再手动猎了
```

威胁狩猎不是一次性的，是持续性的。
一次又一次，越猎越准，越猎越快。

同时，把狩猎发现的新攻击手法，写成检测规则，加到监测系统里。
这样，监测系统就越来越强了。

## 6.6 监测体系的建设要点

最后，再聊聊建设监测体系的几个要点。

### 要点一：覆盖面要全

监测不能有死角。

- 边界要监测
- 内网也要监测
- 服务器要监测
- 终端也要监测
- 网络流量要监测
- 系统日志要监测
- 身份行为也要监测

有一个地方没监测到，攻击者就可能从那个地方钻进来。

木桶效应——你的监测能力，取决于最薄弱的那个环节。

### 要点二：多层验证

一个告警，要从多个角度验证。

- 流量上看到了 → 去系统日志里看看有没有记录
- 系统日志里有 → 去端点上看看进程和文件
- 端点上也有 → 再查查威胁情报

多个角度都对上了，那大概率是真的。

单点证据，容易误报。
多点印证，才靠谱。

### 要点三：持续优化

监测体系不是建完就完事了。

要持续优化：
- 误报太多 → 优化规则
- 有攻击没发现 → 加规则
- 新的攻击手法出来了 → 赶紧更更新检测能力
- 新的系统上线了 → 赶紧接入监测

监测体系是活的，不是死的。
要不断迭代，不断优化。

### 要点四：人是关键

最后，也是最重要的——人。

设备再先进，工具再好用，最后还是得靠人来分析、来判断、来决策。

再智能的AI，也替代不了人的经验和判断。

所以，团队的能力建设，比买设备更重要。

多培训、多练习、多实战。
人的水平上去了，监测水平才能上去。

## 6.7 本章总结

这一章我们讲了监测预警体系，内容很多，给你总结一下：

**监测的三个层次：**
1. 基础监测——看得见（有日志、有告警）
2. 智能监测——看得准（告警降噪、关联分析）
3. 主动监测——看得早（威胁狩猎、主动发现）

**常见的监测手段：**
- 日志监测：最基础，看各种日志找异常
- 流量监测：更真实，全流量/元数据/实时分析
- 端点监测：更细致，看进程、文件、注册表、行为
- 身份监测：看账号和行为，判断是不是本人
- 威胁情报：外部情报，精准检测、提前预警

**告警管理：**
- 告警不是越多越好，是越准越好
- 告警降噪：误报过滤、分级、聚合、关联分析
- 研判流程：看告警本身 → 看上下文 → 查关联 → 验证 → 定性

**威胁狩猎：**
- 主动出击，不是等着告警
- 方法：基于假设、基于情报、基于异常
- 狩猎完了要反馈，让监测系统越来越强

**建设要点：**
- 覆盖面要全，不能有死角
- 多层验证，多点印证才靠谱
- 持续优化，监测体系是活的
- 人是关键，比设备更重要

记住一句话：

> **监测是蓝队的眼睛。眼睛亮不亮，直接决定了防守的水平。**

看不见，就一切都白搭。
