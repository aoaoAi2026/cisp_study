## 7.1 先从一个故事讲起

### 7.1.1 老刑警的故事

先给你讲个故事。

我认识一个老蓝队，姓李，大家都叫他老李。

老李干蓝队干了快十年了，经验特别丰富。
别人查半天查不出来的事儿，老李过来，看几分钟日志，就说个八九不离十。

有一次，一个客户的服务器被入侵了。
客户自己的安全团队查了两天，啥也没查出来。
只知道服务器被黑了，但怎么进来的、什么时候进来的、进来干了啥，一概不知。

老李过去，坐下来，先问了几个问题：
"哪台服务器？"
"什么时候发现的？"
"有啥异常？"
"日志都有吗？"

然后打开日志，开始看。

看了大概半个小时，老李开口了：

"攻击者是三天前的晚上11点23分进来的。
从Web应用的文件上传漏洞进来的，上传了一个WebShell。
然后在第2天凌晨2点15分提权成功，拿到了系统权限。
接着抓了密码，用远程桌面登录过另外3台服务器。
在服务器上种了一个后门，还加了一个隐藏账号。
最后，在第3天下午4点05分，下载了大概200M的数据库备份文件。"

客户的安全团队都听傻了：
"你怎么知道这么清楚？"
"我们看了两天都没看明白。"

老李笑了笑，说：
"没啥，就是日志看的多了，就有感觉了。
就像老刑警看案发现场一样，看一眼就知道大概是怎么回事。"

---

这个故事告诉我们什么？

日志分析，是有方法、有套路的。
看多了、练多了，你也能有这种"感觉"。

这一章，我就把这些方法和套路教给你。

## 7.2 日志分析基础

### 7.2.1 常见的日志类型

首先，你得知道有哪些日志，每种日志长啥样、里面有啥信息。

我给你列几种最常见的：

**1. Windows事件日志**

Windows系统自带的日志，存在事件查看器里。

主要有这么几类：
- **安全日志**：登录、登出、权限变化、对象访问...（最常用）
- **系统日志**：系统服务、驱动、启动关机...
- **应用程序日志**：应用程序运行的日志

安全日志里，几个重要的事件ID要记住：
```
4624：登录成功
4625：登录失败
4672：管理员登录（特殊权限登录）
4634：注销
4688：进程创建
4697/7045：服务安装
5140：网络共享访问
4698：计划任务创建
4720：用户创建
4732：用户加入组
```

这些都是最常用的，得背下来。
看到这些ID，你得知道发生了啥。

**2. Linux系统日志**

Linux的日志一般存在 `/var/log/` 目录下。

常见的日志文件：
```
/var/log/messages   系统主日志（大部分系统事件都在这）
/var/log/secure     安全日志（登录、认证、sudo...）
/var/log/auth.log   认证日志（Debian/Ubuntu系）
/var/log/lastlog    最后登录日志
/var/log/wtmp       登录登出记录（用last命令看）
/var/log/btmp       失败登录记录（用lastb命令看）
/var/log/cron       计划任务日志
/var/log/maillog    邮件日志
/var/log/httpd/     Apache日志
/var/log/nginx/     Nginx日志
```

最常用的是 secure（或者 auth.log），登录、认证相关的都在这。

**3. Web服务器日志**

Web服务器的访问日志，比如Apache、Nginx、IIS。

一般长这样（Nginx为例）：
```
192.168.1.100 - - [24/Jun/2026:10:30:00 +0800] "GET /index.html HTTP/1.1" 200 1234 "-" "Mozilla/5.0..."
```

每个字段是啥：
- 客户端IP
- 时间
- 请求方法（GET/POST...）
- 请求的URL
- HTTP状态码（200/404/500...）
- 响应大小
- 来源页面（Referer）
- User-Agent（浏览器信息）

从Web日志里，能看到谁访问了什么页面、访问成功了还是失败了、用的什么浏览器。

Web攻击，大部分都能从Web日志里找到痕迹。

**4. 防火墙日志**

防火墙的日志，记录了什么流量被放行了、什么流量被拦截了。

一般包含：
- 时间
- 源IP/源端口
- 目的IP/目的端口
- 协议（TCP/UDP/ICMP...）
- 动作（允许/拒绝）
- 规则编号

从防火墙日志里，能看到谁在访问什么、有没有被拦住。

**5. 安全设备日志**

IDS、IPS、WAF、EDR这些安全设备产生的日志。

这些日志一般会直接告诉你：检测到了什么攻击、攻击类型是什么、严重程度怎么样。

这些是告警的主要来源。

### 7.2.2 日志分析的基本思路

面对一大堆日志，从哪下手？

别慌，有套路。

给你一个万能的分析思路：**"5W1H"分析法**

```
When  —— 什么时候发生的？
Where —— 在哪发生的？哪台机器？哪个网段？
Who   —— 谁干的？哪个IP？哪个账号？
What  —— 干了啥？执行了什么命令？访问了什么文件？
Why   —— 为什么？是误报还是真攻击？目的是什么？
How   —— 怎么干的？用了什么手法？攻击路径是什么？
```

把这几个问题搞清楚了，整个事件也就搞清楚了。

是不是很简单？

### 7.2.3 日志分析的一般步骤

具体操作的时候，一般分几步：

**第一步：确定时间范围**

先搞清楚，大概是什么时候的事。
- 什么时候发现的？
- 最早可能是什么时候发生的？
- 最晚可能是什么时候？

先把时间范围定下来，缩小范围。
不然日志那么多，看到天荒地老也看不完。

**第二步：找到入口点**

然后找攻击是从哪进来的。
- 是Web应用打进来的？
- 是远程登录弱口令进来的？
- 是钓鱼邮件进来的？
- 是从内网横向移动过来的？

找到第一个被攻破的点，就找到了整个事件的起点。

**第三步：顺着时间线往后查**

找到起点了，然后顺着时间线往后看。
- 进来之后干了啥？
- 执行了什么命令？
- 上传了什么文件？
- 创建了什么账号？
- 有没有横向移动？
- 有没有偷数据？

一步一步往后推，把整个攻击过程还原出来。

**第四步：反过来验证**

整个过程还原出来了，再反过来验证一下。
- 时间对不对得上？
- 逻辑上通不通？
- 有没有矛盾的地方？
- 有没有漏掉的环节？

都对得上，那基本就是真相了。

就像警察破案一样——
先找案发现场，然后找线索，然后还原过程，最后验证。

## 7.3 常见攻击的日志特征

光有方法还不够，你得知道常见的攻击手法，在日志里长啥样。

我给你列几个最常见的。

### 7.3.1 暴力破解攻击

暴力破解，就是一个一个试密码，直到试对为止。

日志特征很明显：

**Windows日志：**
- 一大堆 4625（登录失败）事件
- 都是同一个IP来的
- 试了很多用户名
- 最后可能跟着一个 4624（登录成功）—— 说明破解成功了！

**Linux日志：**
- secure日志里一大堆 `Failed password`
- 来自同一个IP
- 试了各种用户名
- 最后可能有一个 `Accepted password` —— 破解成功了

**怎么判断是暴力破解？**
- 短时间内大量登录失败
- 来源IP相同或相近
- 尝试的用户名很多（字典攻击）
- 可能最后跟着一个成功登录

遇到这种情况，如果最后有成功登录，那基本就是被攻破了。
赶紧处理！

### 7.3.2 WebShell攻击

WebShell，就是通过Web应用漏洞上传一个网页木马，然后用这个木马控制服务器。

**Web日志特征：**
- 有可疑的文件上传请求（POST到上传接口）
- 上传的文件名很可疑（比如 .php、.jsp、.asp 后缀）
- 上传之后，紧接着就有对这个文件的访问请求
- 访问这个文件的POST请求很多（因为WebShell是POST方式传命令的）
- User-Agent可能是工具（比如"antsword"、"behinder"、"godzilla"，或者干脆没有UA）

**系统日志特征：**
- Web服务器的进程（比如 w3wp.exe、nginx、apache）创建了可疑的子进程
- 比如 cmd.exe、powershell.exe、bash
- 执行了可疑的命令（whoami、ipconfig、net user...）

**怎么找WebShell？**
1. 先看Web日志，找可疑的上传和访问
2. 去服务器上找可疑的文件
3. 看进程，Web进程有没有起奇怪的子进程
4. 看网络连接，有没有可疑的外联

找到WebShell了，就找到入口了。

### 7.3.3 横向移动攻击

横向移动，就是攻击者拿到一台机器的权限后，从这台机器跳到其他机器上。

常见的手法：
- 哈希传递（Pass-the-Hash）
- 票据传递（Pass-the-Ticket）
- WMI远程执行
- PsExec
- 远程桌面（RDP）
- IPC$连接

**日志特征：**

```
IPC$连接 + 远程执行：
  • 4648 事件（尝试使用显式凭据登录）
  • 5140 事件（网络共享访问）
  • 4672 事件（特殊权限登录）
  • 4688 事件（进程创建，源是 services.exe 或者 wmiprvse.exe）

远程桌面登录：
  • 4624 事件，登录类型是10（远程交互）
  • 来源IP是内网的其他机器
  • 登录时间跟前面的攻击时间对得上

WMI执行：
  • WMI-Activity/Operational 日志里有异常的WMI活动
  • 进程创建事件，父进程是 wmiprvse.exe
```

横向移动的核心特征就是：
**从已经被攻破的机器，跳到了其他机器上。**

所以你得先找到"第一台被攻破的机器"，然后顺着它找，看它登录了哪些机器、连接了哪些机器。

### 7.3.4 权限提升攻击

提权，就是从普通用户权限，提升到管理员权限。

**日志特征：**

Windows下：
- 4672 事件（特殊权限登录）—— 突然有了管理员权限
- 进程创建事件，普通用户进程启动了一个高权限进程
- 系统日志里有漏洞利用的痕迹（比如某个服务崩溃了）
- 可能有 MS14-068、CVE-2021-40449 这些经典提权漏洞的特征

Linux下：
- secure日志里，突然有成功的sudo
- 普通用户执行了只有root才能执行的命令
- 有SUID程序被利用的痕迹
- cron、计划任务被篡改

提权成功的标志就是：**权限突然变大了。**

之前干啥都没权限，突然干啥都有权限了。
那肯定是提权了。

### 7.3.5 权限维持（留后门）

攻击者进来之后，肯定不想走了。
他会留后门，方便下次再来。

常见的后门手法：
- 新建管理员账号
- 注册表启动项
- 计划任务
- 服务后门
- WebShell
- 远控木马

**日志特征：**

新建账号：
- 4720 事件（用户账户已创建）
- 4732 事件（成员已添加到启用了安全性的本地组）—— 比如加入了管理员组

计划任务：
- 4698 事件（已创建计划任务）
- 4702 事件（已更新计划任务）
- 任务的执行程序很可疑

服务安装：
- 4697 事件（服务已安装）
- 7045 事件（服务已安装）
- 服务名很奇怪，或者服务指向的文件很可疑

注册表修改：
- 注册表启动项被修改
- 比如 Run、RunOnce 这些键值

发现这些，就说明攻击者已经留后门了。
得赶紧清理干净，不然你以为他走了，其实他还能随时回来。

## 7.4 流量分析基础

讲完了日志，再说说流量分析。

流量分析，也是蓝队的基本功。

### 7.4.1 为什么要做流量分析？

有人说，有日志不就行了？为啥还要看流量？

因为日志有局限性：

1. **日志可能被删。** 攻击者进来之后，第一件事可能就是删日志。
   日志删了，你啥都看不到了。
   但流量删不了——流量是存在旁路设备上的，攻击者碰不到。

2. **有些攻击没有日志。** 比如攻击是加密的、或者走的是正常的协议通道。
   日志里看不到，但流量里有痕迹。

3. **流量更真实。** 日志是系统记的，可能记错、漏记、被篡改。
   流量是真实发生的通信，不会骗人。

4. **溯源的时候需要流量。** 想完整还原攻击过程，光靠日志不够，得有流量。

所以，流量分析很重要。

### 7.4.2 常见的流量分析工具

工欲善其事，必先利其器。

常用的流量分析工具：

**1. Wireshark**

最经典、最常用的抓包分析工具。
功能强大，啥都能分析。
缺点是学习曲线有点陡，新手可能看着头大。

**2. Tcpdump**

Linux下的命令行抓包工具。
轻量、快速、强大。
适合在服务器上直接抓包。

**3. 全流量分析平台**

比如：科来、奇安信的天眼、深信服的全流量分析...
商业化的产品，功能强大，界面友好，适合企业用。
就是贵。

**4. 其他工具**
- NetworkMiner：自动提取文件、证书、密码什么的
- BruteShark：提取密码、哈希
- CapLoader：大PCAP文件快速浏览

新手的话，先把Wireshark学会，基本就够用了。

### 7.4.3 常见攻击的流量特征

跟日志一样，常见的攻击手法，在流量里也有明显的特征。

**1. 端口扫描**

端口扫描就是，攻击者扫你的端口，看哪些开着。

流量特征：
- 短时间内，一个源IP，访问同一个目的IP的很多不同端口
- 大部分是SYN包，没有后续的握手
- 端口号可能是连续的，也可能是常见的端口（21、22、80、443、3389...）

用Wireshark看，一眼就能看出来——一大堆SYN包，一个接一个。

**2. SQL注入**

SQL注入，就是往Web参数里插SQL语句。

流量特征：
- HTTP GET/POST请求里，有SQL关键字
- 比如：`' or 1=1--`、`union select`、`and 1=2`、`sleep(`...
- URL或者POST数据里，有单引号、双引号这些奇怪的字符
- 可能有报错信息（比如 SQL syntax error...）

在HTTP流量里搜这些关键字，一搜一个准。

**3. WebShell通信**

WebShell的流量，也有特征。

常见的特征：
- 同一个URL，有大量的POST请求（正常页面一般GET多）
- POST数据是加密的或者编码的（一句话木马）
- 响应包的大小变化很大（每次执行不同的命令，返回结果不一样）
- User-Agent很奇怪，或者是工具的UA（蚁剑、冰蝎、哥斯拉...）
- 连接时间不规律，有时候几分钟一次，有时候几秒钟一次

冰蝎、哥斯拉这些加密的WebShell，虽然内容看不懂，但流量模式还是能看出来的。

**4. C2通信（远控木马）**

C2就是Command & Control，远控木马的控制端通信。

C2流量的常见特征：
- 有规律的心跳包（每隔固定时间通信一次，比如1分钟、5分钟）
- 流量很小，每次就几个包
- 加密流量（HTTPS或者自定义加密）
- 域名很奇怪，或者是动态域名
- IP是冷门地区的
- 正常上班时间流量少，半夜流量多（黑客喜欢半夜干活）
- DNS隧道、ICMP隧道这些异常的协议

C2流量的核心就是：**规律、持续、隐蔽。**

**5. 数据外泄**

数据被偷了，流量里也能看出来。

特征：
- 某个机器，突然有大量的外发流量
- 平时没什么外联，突然往外传很多数据
- 传输的时间很长、数据量很大
- 目的IP/端口很可疑
- 可能是FTP、或者是HTTP POST上传
- 传输时间可能在非工作时间（半夜偷数据，不容易被发现）

数据外泄是最严重的后果之一。
发现大量异常外发流量，一定要高度重视。

## 7.5 分析实战技巧

讲完了理论，再讲点实战技巧。

这些都是老蓝队总结出来的经验，非常实用。

### 7.5.1 日志分析小技巧

**技巧一：先缩小范围**

日志那么多，不能一条一条看。
先想办法把范围缩小。

怎么缩小？
- 先定时间范围
- 再定IP范围
- 再定事件类型
- 最后再看具体内容

比如：你怀疑有人暴力破解。
那你就先搜 4625 事件，看有哪些失败登录。
找到可疑IP了，再去搜这个IP的所有登录记录。
这样就快多了。

**技巧二：善用工具**

别傻呵呵地用记事本打开日志慢慢翻。

用工具！

Windows下：
- 事件查看器自带的筛选和查找
- PowerShell 命令（Get-WinEvent）
- 第三方工具（比如 Log Parser）

Linux下：
- grep、awk、sed、sort、uniq 这些神器
- 组合起来用，效率超高

比如：
```bash
# 统计secure日志里失败登录最多的IP
grep "Failed password" /var/log/secure | awk '{print $11}' | sort | uniq -c | sort -nr

# 看某个IP的所有登录记录
grep "192.168.1.100" /var/log/secure

# 查看某个时间段的日志
sed -n '/Jun 24 10:00/,/Jun 24 12:00/p' /var/log/secure
```

这些命令熟练了，分析速度能快好几倍。

**技巧三：对比正常和异常**

什么是异常？
跟正常的不一样，就是异常。

所以你得知道正常情况下是什么样。

比如：
- 正常大家都是白天上班登录，突然有人半夜登录 → 异常
- 正常大家都从公司IP登录，突然有人从国外登录 → 异常
- 正常某个服务器没人登录，突然有人登录了 → 异常

对比正常基线，找不同。
这是最基本、也是最有效的方法。

**技巧四：找第一台被攻破的机器**

遇到内网渗透事件，最重要的是找到"第一台被攻破的机器"——也就是入口点。

找到了入口点，整个事件就清楚了一半。

怎么找？
- 看时间：哪个机器最早出现异常？
- 看路径：其他机器的攻击，都是从哪台机器来的？
- 看手法：最开始的攻击手法是什么？（比如Web漏洞、钓鱼邮件）

顺藤摸瓜，找到源头。

### 7.5.2 流量分析小技巧

**技巧一：先看统计，再看细节**

拿到一个PCAP文件，别上来就一个包一个包地看。

先看统计信息：
- 有哪些IP在通信？
- 哪些IP流量最大？
- 用了哪些协议？
- 端口分布是怎样的？

Wireshark里，Statistics菜单下有各种统计信息。
先看统计，找到可疑的点，再深入分析。
这样效率高多了。

**技巧二：用过滤表达式**

Wireshark的过滤表达式，一定要学会。

常用的过滤：
```
ip.addr == 192.168.1.100    只看这个IP的流量
ip.src == 192.168.1.100     只看源IP是这个的
ip.dst == 192.168.1.100     只看目的IP是这个的
tcp.port == 80              只看80端口的
http                        只看HTTP协议的
http.request.method == POST 只看POST请求
```

用好过滤，能从海量流量里，快速过滤出你想看的。

**技巧三：搜关键字**

如果怀疑是Web攻击，直接在HTTP流量里搜关键字。

比如：
- SQL注入：搜 `select`、`union`、`sleep`、`or 1=1`
- XSS：搜 `<script`、`alert(`、`onerror`
- 命令注入：搜 `cmd.exe`、`/bin/bash`、`;id`、`|whoami`

一搜一个准。

**技巧四：看会话而不是看包**

别一个包一个包地看，要看会话（流）。

Wireshark里，右键 → Follow → TCP Stream，就能看整个TCP会话的内容。
这样更直观，能看到完整的交互过程。

比如WebShell通信，Follow一下TCP流，就能看到完整的请求和响应。
比一个包一个包看，爽多了。

## 7.6 分析的常见误区

最后，再说说日志和流量分析中常见的误区，别掉进去。

### 误区一：只看告警，不看原始数据

很多人分析问题，就只看安全设备的告警。
"告警说啥就是啥。"

不对。

告警可能是误报，也可能只说了一部分。
一定要去看原始日志、原始流量。
自己去验证、去分析。

告警只是给你一个线索，真相还得你自己去找。

### 误区二：看到一个点就下结论

有些人，看到一个可疑的点，就立刻下结论：
"肯定是被黑了！"
"肯定是这个干的！"

别急着下结论。

一个点说明不了什么。
要多个点相互印证，形成完整的证据链，才能下结论。

孤证不立。

### 误区三：只看被攻击的机器，不看其他的

很多人分析入侵事件，就盯着那一台被攻破的机器看。
其他机器不管。

不对。

攻击者进来之后，不会只待在一台机器上。
他肯定会横向移动，去其他机器。

所以，你得查：
- 他还登录了哪些机器？
- 他还访问了哪些机器？
- 他还偷了哪些数据？

把所有受影响的机器都找出来。
不然你清了这台，那台还有，等于白清。

### 误区四：分析完就完了，不总结改进

很多人，事件分析完了、处置完了，就完事了。
然后就忘了。

然后下次，同样的攻击又来了，还是中招。

为啥？
因为没总结，没改进。

分析完了，一定要：
- 总结这次攻击是怎么进来的
- 为什么我们没防住
- 为什么我们没及时发现
- 以后怎么改进
- 把检测规则加上，下次再出现就自动告警了

吃一堑，长一智。
这样才能越做越好。

## 7.7 本章总结

这一章我们讲了日志分析和流量分析，内容很多，给你总结一下：

**常见的日志类型：**
- Windows事件日志（安全日志最重要，记住常用的事件ID）
- Linux系统日志（secure/auth.log 最常用）
- Web服务器日志（访问日志、错误日志）
- 防火墙日志
- 安全设备日志

**日志分析思路：**
- 5W1H：When/Where/Who/What/Why/How
- 步骤：定时间范围 → 找入口点 → 顺时间线往后查 → 验证

**常见攻击的日志特征：**
- 暴力破解：大量登录失败，最后可能有成功
- WebShell：可疑文件上传 + 可疑访问 + 异常子进程
- 横向移动：内网机器之间的异常登录和执行
- 提权：权限突然变大
- 权限维持：新建账号、计划任务、服务、注册表

**流量分析：**
- 为什么要做流量分析：日志可能被删、更真实、溯源需要
- 常用工具：Wireshark、Tcpdump、全流量分析平台
- 常见攻击流量特征：端口扫描、SQL注入、WebShell、C2通信、数据外泄

**实战技巧：**
- 先缩小范围，再深入分析
- 善用工具和命令（grep/awk/sed/过滤表达式）
- 对比正常基线找异常
- 找入口点、找第一台被攻破的机器
- 看会话不看单个包
- 多个证据相互印证

**常见误区：**
- 只看告警不看原始数据 → 要自己验证
- 看到一个点就下结论 → 孤证不立，要有完整证据链
- 只看被攻击的机器 → 要全面排查所有受影响的
- 分析完就完了 → 要总结改进，吃一堑长一智

最后送你一句话：

> **熟能生巧。**

日志和流量分析，没什么捷径。
就是多看、多练、多分析。
看的多了，自然就有感觉了。

就像老刑警一样，看一眼就知道大概是怎么回事。
