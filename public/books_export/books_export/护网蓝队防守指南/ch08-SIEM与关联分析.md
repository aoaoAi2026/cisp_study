## 8.1 先从一个故事讲起

### 8.1.1 被淹没的告警

先给你讲个真实的故事。

有一家公司，安全设备买了不少：
- 防火墙
- IDS/IPS
- WAF
- EDR
- 数据库审计
- 邮件网关

每个设备都会产生告警。
一开始还好，告警不多，大家还能看得过来。

后来，设备越来越多，日志越来越多，告警也越来越多。

多到什么程度？
一天几万条告警。

安全团队一共才5个人。
5个人，一天几万条告警，怎么看？

看不过来啊。

于是大家就挑着看：
- 高危的看看
- 中危的扫一眼
- 低危的直接忽略

结果呢？

有一次，红队打进来了。
整个攻击过程中，触发了大概20多条告警。
但是这些告警分散在不同的设备里，有低危的、有中危的，还有的被当成了误报。

没有一条告警被重视。
没有一个人发现，这些零散的告警，其实是同一次完整的攻击。

等红队都拿到域控了，大家才反应过来。
晚了。

---

为什么会这样？

因为告警是零散的、孤立的。
每条告警只告诉你"发生了一件事"，但不告诉你"这件事和其他事有什么关系"。

你得自己把它们串起来。
但告警太多了，你根本串不过来。

怎么办？

这时候，SIEM就派上用场了。

## 8.2 什么是SIEM？

### 8.2.1 SIEM的定义

SIEM，全称是 Security Information and Event Management。
中文叫：**安全信息与事件管理**。

名字听起来挺高大上的，其实说人话就是：

> **把所有设备的日志都收集到一起，集中存储、集中分析、集中告警，然后把相关的告警串起来，告诉你发生了什么攻击。**

我再翻译翻译，更直白一点：

SIEM就是安全团队的"中控台"。

- 所有日志都往这送
- 所有告警都在这看
- 所有事件都在这管
- 还能自动把相关的告警串成一个完整的事件

是不是一下子就明白了？

### 8.2.2 SIEM的核心功能

SIEM一般有这么几个核心功能：

```
1. 日志采集
   • 从各种设备、系统、应用采集日志
   • 防火墙、IDS、WAF、操作系统、数据库、应用...
   • 啥日志都能接

2. 日志存储
   • 把所有日志集中存在一起
   • 方便查询、方便分析
   • 一般存几个月到几年

3. 实时监控
   • 实时分析日志
   • 有异常立刻告警
   • 大屏展示，一目了然

4. 关联分析
   • 把相关的告警串起来
   • 从零散的告警里，找出完整的攻击
   • 这是SIEM最核心的功能

5. 告警管理
   • 告警分级、分派、处理
   • 告警工单流程
   • 跟踪处理进度

6. 报表与合规
   • 自动生成各种报表
   • 满足等保、ISO27001等合规要求
   • 定期出安全报告

7. 威胁情报
   • 集成威胁情报
   • IOC匹配
   • 提高检测准确率
```

简单说就是：收集 → 存储 → 分析 → 告警 → 处置 → 报告。
一条龙服务。

### 8.2.3 为什么需要SIEM？

没有SIEM行不行？

也行。
但你会很累。

有了SIEM之后：

**之前：**
- 你要登录10个不同的设备看日志
- 每个设备的界面都不一样
- 告警分散在各个地方
- 你得自己手动把它们串起来
- 出了问题，找日志找半天

**之后：**
- 所有日志都在一个地方
- 统一的界面、统一的查询
- 告警自动关联
- 出了问题，一查就有
- 报表自动生成

哪个爽？

当然是有SIEM爽啊。

## 8.3 什么是关联分析？

### 8.3.1 关联分析是啥

SIEM最核心的功能，就是关联分析。

啥叫关联分析？

用一句话说：

> **把零散的、孤立的告警，按照一定的规则串起来，形成一个完整的安全事件。**

我给你举个例子，你就懂了。

假设发生了一次攻击，过程是这样的：

```
1. 攻击者扫描端口 → IDS告警：端口扫描（低危）
2. 攻击者尝试SQL注入 → WAF告警：SQL注入尝试（中危）
3. 攻击者上传WebShell → WAF告警：文件上传（中危）
4. 攻击者提权 → EDR告警：异常提权行为（高危）
5. 攻击者横向移动 → IDS告警：内网扫描（中危）
6. 攻击者登录域控 → Windows告警：异常登录（高危）
```

没有关联分析的时候，你看到的是：
- 6条独立的告警
- 分散在不同的设备里
- 有低危、有中危、有高危
- 你可能只看了高危的，没看中危和低危的
- 你不知道它们是同一次攻击

有关联分析之后，你看到的是：
- **一个安全事件：一次完整的入侵攻击**
- 包含6个阶段，从扫描到域控沦陷
- 攻击过程一目了然
- 你直接就知道发生了啥、有多严重

这就是关联分析的威力。

它把"点"连成了"线"，把"零散的告警"变成了"完整的故事"。

### 8.3.2 关联分析的价值

关联分析有啥用？

**第一，减少告警数量。**

原来100条告警，关联之后可能就变成5个事件。
数量少了，你就看得过来了。
不会再有告警疲劳。

**第二，提高告警准确率。**

单个告警可能是误报。
但好几个告警串在一起，还都是误报的概率就小多了。
关联之后，误报率大大降低。

**第三，看清攻击全貌。**

原来你只能看到"发生了什么事"。
现在你能看到"整个攻击是怎么发生的、到了哪一步、有多严重"。
一目了然。

**第四，更快响应。**

原来你得自己慢慢拼、慢慢查。
现在SIEM直接给你拼好了。
你直接就知道怎么回事，直接就能响应。
响应速度快多了。

一句话：关联分析，让你从"只见树木，不见森林"，变成了"既见树木，又见森林"。

## 8.4 常见的关联分析场景

说了这么多理论，来点实战的。

常见的关联分析场景有哪些？

我给你举几个最常用的例子。

### 8.4.1 场景一：暴力破解成功

这是最经典的关联场景。

**场景描述：**
一个IP，在短时间内，多次登录失败，然后有一次登录成功。

**为什么要关联？**
- 单独的登录失败 → 可能是用户记错密码了（误报率高）
- 单独的登录成功 → 正常登录啊（根本不会告警）
- 但 **多次失败 + 最后成功** → 大概率是暴力破解成功了！（很危险）

**关联规则：**
```
同一个源IP
+ 同一个目标账号/目标机器
+ 短时间内（比如5分钟内）
+ N次以上登录失败（比如10次）
+ 之后紧跟着一次登录成功
→ 触发告警：暴力破解成功
```

这个场景非常经典，也非常实用。
强烈建议每个SIEM都配上这条规则。

### 8.4.2 场景二：Web攻击 + 服务器异常

第二个经典场景：Web攻击之后，服务器出现异常。

**场景描述：**
WAF检测到Web攻击，然后紧接着，那台Web服务器上出现了异常行为（比如异常进程、异常提权）。

**为什么要关联？**
- 单独的WAF攻击告警 → 可能被拦住了，没事
- 单独的服务器异常 → 可能是正常操作
- 但 **WAF告警 + 服务器异常 + 时间接近 + 同一台机器** → 说明攻击成功了，已经打进服务器了！

**关联规则：**
```
同一个目标服务器
+ 短时间内（比如10分钟内）
+ 先有WAF的高危攻击告警（比如文件上传、命令执行）
+ 之后服务器上出现异常（比如新建可疑进程、提权行为）
→ 触发告警：Web攻击成功，服务器被入侵
```

这个场景也很实用。
能帮你快速判断，Web攻击是被拦住了，还是真的打进来了。

### 8.4.3 场景三：内网横向移动

第三个场景：内网横向移动。

**场景描述：**
一台机器被攻陷后，用这台机器当跳板，去攻击其他内网机器。

**为什么要关联？**
- 单独一台机器的告警 → 只是一台机器的问题
- 多台机器都有告警，而且是从同一台机器来的 → 说明在横向移动，问题严重多了！

**关联规则：**
```
有一台机器（A），已经被确认被入侵
+ 短时间内，从A机器出发，有很多对内网其他机器的攻击行为
  （比如：登录尝试、端口扫描、WMI执行、PsExec...）
+ 其中有一些成功了
→ 触发告警：内网横向移动
```

发现横向移动，就说明事情闹大了。
这时候必须立刻响应。

### 8.4.4 场景四：异常时间 + 异常行为

第四个场景：异常时间的异常行为。

**场景描述：**
正常上班时间啥事没有，半夜三更的，突然有大量异常操作。

**为什么要关联？**
- 上班时间有操作 → 正常，大家都在干活
- 半夜有操作 → 不正常，谁半夜不睡觉干活？
  （除非是运维在加班，但一般不会天天加班到半夜）

**关联规则：**
```
时间在非工作时间（比如凌晨0点到6点）
+ 有敏感操作（比如：登录、数据查询、文件下载、系统修改...）
+ 这个用户平时这个时间不操作
→ 触发告警：非工作时间异常操作
```

这个场景也很有用。
很多攻击者喜欢半夜动手，因为这时候人少，不容易被发现。

### 8.4.5 场景五：数据外泄

第五个场景：数据外泄。

**场景描述：**
一个用户，平时流量不大，突然有大量的外发流量。

**为什么要关联？**
- 单独的大流量 → 可能是在看视频、下东西（正常）
- 但如果是 **敏感系统 + 大量外发 + 时间不对 + 用户平时不这样** → 可能是在偷数据！

**关联规则：
```
某个内部IP/某个用户
+ 短时间内（比如1小时内）
+ 向外网传输了大量数据（比如超过1G）
+ 目的IP可疑（或者不在白名单里）
+ 这个用户平时没有这么大的外发流量
→ 触发告警：疑似数据外泄
```

数据外泄是最严重的安全事件之一。
发现得早，损失就小。

### 8.4.6 场景六：多阶段攻击链

最后一个，最厉害的——多阶段攻击链关联。

就是把整个攻击过程，从开始到结束，完整地串起来。

比如：

```
阶段1：侦察 → 端口扫描、漏洞扫描
阶段2：初始访问 → Web攻击、钓鱼邮件、暴力破解
阶段3：执行 → 执行恶意代码、WebShell
阶段4：持久化 → 新建账号、计划任务、服务后门
阶段5：权限提升 → 提权漏洞、错误配置
阶段6：凭据访问 → 抓密码、哈希传递
阶段7：横向移动 → 远程登录、WMI、PsExec
阶段8：数据窃取 → 访问敏感数据、下载数据
阶段9：数据渗出 → 往外传数据
```

SIEM可以按照ATT&CK的战术阶段，把各个阶段的告警串起来，形成一条完整的攻击链。

这样，攻击进行到哪一步了、有多严重，一目了然。

这个是高级玩法，难度也比较大。
但一旦做起来，效果是最好的。

## 8.5 怎么写关联规则？

### 8.5.1 关联规则的基本结构

关联规则一般长啥样？

基本结构是这样的：

```
IF （条件1 AND 条件2 AND 条件3...）
   在 时间窗口 内
   针对 同一个实体（IP/用户/主机...）
THEN 触发告警
     告警级别：XX
     告警描述：XX
```

说白了就是：
**当满足某些条件，在一定时间内，针对同一个目标，就告警。**

### 8.5.2 写规则的步骤

怎么写一条好的关联规则？

一般分几步：

**第一步：明确场景**

先想清楚，你要检测什么攻击？
这个攻击的过程是怎样的？
有哪些特征？

比如：我要检测暴力破解成功。
特征是：多次失败 → 一次成功。

**第二步：确定数据源**

需要哪些日志？从哪来？
- 登录日志？（Windows安全日志、Linux secure日志）
- 认证日志？（VPN日志、应用登录日志）

把数据源确定好，确保日志已经接进SIEM了。

**第三步：定义关联条件**

规则怎么写？
- 同一个IP？同一个用户？同一个目标机器？
- 多长时间窗口？（5分钟？10分钟？1小时？）
- 多少次失败才算？（5次？10次？20次？）
- 必须要有成功吗？（还是只要大量失败也算？）

把这些条件都定义清楚。

**第四步：调优测试**

规则写好了，先别急着上线。
先测试测试。

- 拿历史日志跑一跑，看看误报多不多
- 误报太多 → 调严格一点
- 漏报太多 → 调宽松一点
- 找到一个平衡点

**第五步：上线迭代**

测试没问题了，再上线。

上线之后也不是就完事了。
- 持续观察
- 有问题就调
- 新的攻击手法出来了，更新规则

规则是活的，不是死的。
要持续迭代、持续优化。

### 8.5.3 写规则的注意事项

写关联规则，有几个坑要注意：

**坑一：规则太松，误报太多**

规则写得太松了，什么都告警。
结果一天几百条告警，全是误报。
等于没有。

**坑二：规则太严，漏报太多**

规则写得太严了，什么都查不出来。
真的攻击来了，它不告警。
那要你何用？

平衡很重要。
误报和漏报之间，要找到平衡点。

**坑三：时间窗口太大或太小**

时间窗口太大 → 把不相关的告警也串起来了，误报多。
时间窗口太小 → 该串的串不起来，漏报多。

时间窗口要根据实际情况调。

**坑四：规则写死了，不会变**

攻击手法在变，规则也要跟着变。
不能写好就不管了。

定期回顾、定期优化。
规则才能一直好用。

## 8.6 SIEM落地的常见坑

最后，聊聊SIEM落地的常见坑。

很多单位买了SIEM，最后没发挥作用，就是踩了这些坑。

### 坑一：只买设备，没人运营

买了SIEM，往那一放，就不管了。
没人管规则、没人管告警、没人优化。

结果呢？
告警一大堆，全是误报，没人看。
最后成了摆设，用来过合规的。

SIEM只是个工具，关键还得靠人去运营。
没人运营，再贵的SIEM也没用。

### 坑二：日志接不全

SIEM就接了几个设备的日志。
防火墙接了，IDS接了，操作系统没接，应用没接，终端也没接。

那你能分析出啥来？
巧妇难为无米之炊啊。

日志越全，SIEM越好用。
能接的都接上，别嫌麻烦。

### 坑三：规则不优化

规则写了一堆，全是默认规则。
也不根据自己的实际情况调。
结果误报一大堆，根本看不过来。

默认规则只能当个参考。
一定要根据自己的环境、自己的业务，去调优。

不然没用。

### 坑四：期望太高

很多人觉得，买了SIEM，就万事大吉了。
什么攻击都能自动发现、自动处置。

想多了。

SIEM只是个工具，它能帮你提高效率，但不能替代人。
该分析还得分析，该处置还得处置。

别把SIEM想成万能的。

### 坑五：光有SIEM，没有流程

告警来了，然后呢？
谁来处理？
怎么处理？
处理完了怎么办？
要不要升级？
要不要上报？

啥流程都没有。
告警来了，大家大眼瞪小眼。

SIEM只是技术层面的。
配套的流程、制度、人员，都得跟上。
不然发挥不了作用。

- 实时监控、关联分析

- 减少告警数量、提高准确率、看清攻击全貌、加快响应速度

**常见的关联场景：**
1. 暴力破解成功（多次失败 + 一次成功）
2. Web攻击 + 服务器异常（攻击打到服务器里了）
3. 内网横向移动（一台打多台）
4. 异常时间 + 异常行为（半夜搞事情）
5. 数据外泄（大量外发流量）
6. 多阶段攻击链（完整的攻击过程）

**怎么写关联规则：**
- 明确场景 → 确定数据源 → 定义条件 → 调优测试 → 上线迭代
- 注意平衡误报和漏报
- 规则要持续优化

**常见的坑：**
- 只买设备没人运营 → 要有人运营
- 日志接不全 → 能接的都接上
- 规则不优化 → 持续调优
- 期望太高 → SIEM是工具，不是万能的
- 没有流程 → 配套流程要跟上

记住一句话：

> **SIEM是工具，人才是关键。**

工具再好，不会用、没人用，也等于零。
把人、流程、工具结合起来，才能发挥最大的作用。
