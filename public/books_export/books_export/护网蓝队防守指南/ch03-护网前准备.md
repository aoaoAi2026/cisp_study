# 第三章 护网前准备 —— 不打无准备之仗

> 第3章 | 预计学习时间：4小时 | 难度：⭐⭐

---

## 📖 写在前面

孙子兵法有云：

> "兵马未动，粮草先行。"

打仗是这样，护网也是这样。

很多人觉得护网就是护网那14天的事，其实不然。

护网打得好不好，80%取决于护网前的准备工作做得怎么样。

准备工作做扎实了，护网期间就能从容应对。
准备工作没做好，护网期间就手忙脚乱，到处救火。

这一章，我们就来好好聊聊：护网前到底要准备些啥？怎么准备？准备到什么程度才算够？

坐稳了，开唠。

## 3.1 先从一个真实的故事讲起

### 3.1.1 两个单位的对比

先给你讲两个真实的故事。

有一年护网，我认识两个蓝队队长，一个姓李，一个姓张，分别在两家不同的单位。

**李队长的单位：**

护网前一个月，李队长就忙开了。

- 资产梳理：把单位里里外外的资产全梳理了一遍，列了个详细的清单，哪些是重要的，哪些是一般的，哪些暴露在公网，清清楚楚。
- 漏洞扫描：扫了三遍，高危漏洞全补了，中危的也补了80%。
- 安全加固：服务器加固、网络加固、应用加固，能加固的全加固了一遍。
- 安全产品检查：防火墙、IDS、WAF、EDR、SIEM......一个一个检查，确保都正常运行，规则都调好了。
- 应急预案：写了十几份应急预案，什么情况怎么办，谁负责什么，写得清清楚楚。还组织了两次演练。
- 安全培训：给全体员工做了三次安全意识培训，钓鱼邮件识别、密码安全、社会工程学......都讲了。

忙了一个月，李队长瘦了5斤。

**张队长的单位：**

张队长呢？护网前一周才想起来要准备。

"哎呀，快护网了，我们也准备准备吧。"

扫了个漏洞——扫出来一大堆，没时间补了。
加固了一下——加了几条防火墙规则，意思意思。
应急预案——找了个去年的模板，改了改名字。
安全培训——发了一封邮件，大家看一下啊。

然后就到护网了。

**护网的结果呢？**

李队长的单位：护网14天，被红队试探了很多次，但真正打进来的只有2次，而且都在24小时内就发现并处置了。最后排名靠前，还拿了奖。

张队长的单位：护网第3天就被打穿了，红队拿到了域控权限，还"偷"了一堆核心数据。最后排名倒数，被通报批评。

张队长委屈啊："我们护网期间也很努力啊，天天熬夜，为啥结果这么差？"

为啥？

因为功夫都在诗外。

护网的胜负，在护网开始之前，就已经决定了80%。

准备工作没做好，护网期间再努力，也事倍功半。

## 3.2 准备工作的时间线

护网准备工作，从什么时候开始？

一般来说，提前**1-2个月**开始比较合适。

太短了，时间不够，很多事情做不完。
太长了，容易松懈，前面做的后面又忘了。

我给你画一个时间线，你对照着看：

```
护网前2个月：
  • 启动准备工作
  • 成立护网工作组
  • 制定准备计划
  • 资产梳理启动

护网前1.5个月：
  • 第一轮漏洞扫描
  • 安全产品状态检查
  • 历史问题回顾

护网前1个月：
  • 第二轮漏洞扫描
  • 高危漏洞修复
  • 安全加固实施
  • 应急预案编写
  • 安全意识培训

护网前2周：
  • 第三轮漏洞扫描（验证）
  • 安全产品调优
  • 应急演练
  • 监控体系检查

护网前1周：
  • 最后一轮检查
  • 全员动员
  • 值班表安排
  • 通讯保障测试
  • 战前动员会

护网前1天：
  • 最后检查
  • 工具准备
  • 状态确认
  • 早点休息，养精蓄锐
```

大概就是这么个节奏。

当然，具体时间你可以根据自己单位的情况调整。但大的节奏是这样的：越早开始，越从容。

## 3.3 准备工作的八大模块

准备工作具体干啥？我给你总结了八大模块，一个一个来。

### 3.3.1 第一模块：组织准备——人是第一位的

打仗先点兵。护网也是一样，先把人组织好。

**第一件事：成立护网工作组**

别到时候谁干啥都不知道。

一般要有这么几个角色：
- **总指挥**：一般是领导，负责拍板、协调资源
- **蓝队队长**：负责技术指挥、整体调度
- **监测组**：负责7×24小时监控、告警研判
- **分析组**：负责深入分析、溯源取证
- **响应组**：负责应急处置、漏洞修复
- **保障组**：负责后勤、通讯、物资、协调
- **对接组**：负责跟上级、监管、裁判对接

人不用太多，但职责要分清楚，谁干啥要明确。

别到时候出事了，大家大眼瞪小眼："这该谁管？"

**第二件事：制定值班表**

护网是7×24小时的，人不能连轴转，得排班。

怎么排？
- 一般分三班倒：早班、中班、夜班
- 每班至少2个人，互相有个照应
- 关键岗位要有备份，万一有人临时有事，有人能顶上
- 周末也要安排好，不能没人

值班表提前排好，发到每个人手里，大家心里有数。

**第三件事：通讯保障**

真出事了，得能找到人。

- 建个护网工作群，重要信息在群里通报
- 关键人员的电话存好，24小时开机
- 万一内网不通了，有没有备用的联系方式？
- 跟上级单位、监管部门的对接方式要明确

别到时候出事了，找不到人，急死人。

**第四件事：战前动员**

护网开始前，开个动员会。

干啥？
- 讲清楚护网的重要性
- 讲清楚各自的职责
- 讲清楚纪律要求
- 鼓鼓劲，打打气

士气很重要。

### 3.3.2 第二模块：资产准备——知己知彼

资产梳理是基础中的基础。

你连自己家里有啥都不知道，怎么防？

**第一件事：全面资产盘点**

把所有资产都列出来：

```
网络资产：
  • IP地址段（公网+内网）
  • 域名（主域名、子域名）
  • 网络设备（路由器、交换机、防火墙、VPN...）
  • 安全设备（IDS、IPS、WAF、EDR、SIEM...）

服务器资产：
  • 物理服务器
  • 虚拟服务器
  • 云服务器
  • 操作系统（Windows/Linux/其他）
  • 用途（Web服务器、数据库服务器、文件服务器...）

应用系统：
  • 业务系统
  • Web应用
  • 中间件
  • 数据库
  • 版本号

数据资产：
  • 核心数据
  • 重要数据
  • 一般数据
  • 数据存在哪？谁管？
```

越全越好，越细越好。

很多单位被打进来，就是因为有"影子资产"——没人管的测试服务器、没人维护的老系统、忘了关的云主机......

这些都是红队最喜欢的突破口。

**第二件事：资产分级分类**

资产列出来了，然后分个级。

哪些是核心资产？哪些是重要资产？哪些是一般资产？

一般分三级：
- **核心资产**：域控、核心业务系统、核心数据库、重要数据......没了就要命的
- **重要资产**：普通业务系统、重要服务器......很重要，但不是致命的
- **一般资产**：办公电脑、测试系统......一般般，被黑了影响不大

为什么要分级？

因为防护资源是有限的，不能平均用力。
核心资产重点保护，重要资产次之，一般资产再次之。

好钢要用在刀刃上。

**第三件事：攻击面梳理**

哪些资产暴露在公网上？
- 公网IP有多少？
- 公网开放了哪些端口？
- 公网有哪些Web应用？
- 公网有哪些远程访问入口（VPN、远程桌面...）？

这些就是你的"攻击面"。

攻击面越小越好。能关的关，能藏的藏，能撤的撤。

比如：
- 测试系统别放公网上了，挪到内网去
- 不用的端口关了
- 不用的服务停了
- 远程访问加上白名单

攻击面小一点，红队就难打一点。

### 3.3.3 第三模块：漏洞准备——补好篱笆

漏洞是红队最好的入口。

把漏洞补好，能防住80%的攻击。

**第一件事：全面漏洞扫描**

扫什么？

```
系统漏洞扫描：
  • Windows漏洞
  • Linux漏洞
  • 网络设备漏洞
  • 安全设备漏洞

Web漏洞扫描：
  • SQL注入
  • XSS
  • 文件上传
  • 命令注入
  • 逻辑漏洞
  • 中间件漏洞

弱口令扫描：
  • SSH弱口令
  • RDP弱口令
  • 数据库弱口令
  • 后台管理弱口令
  • 一切有密码的地方

配置核查：
  • 不安全的配置
  • 默认账号默认密码
  • 多余的服务
  • 多余的端口
```

怎么扫？
- 自动扫描器扫一遍（Nessus、AWVS、AppScan...）
- 人工再检查一遍重点系统
- 最好找专业的人来扫，自己扫可能有遗漏

扫完了，出个漏洞清单。

**第二件事：漏洞修复**

扫出来了漏洞，然后补。

怎么补？按优先级来。

```
紧急（立即修复）：
  • 远程代码执行漏洞
  • 严重的权限绕过
  • 可直接getshell的漏洞
  • 公网上的高危漏洞

高危（1-3天内修复）：
  • 普通的高危漏洞
  • 重要系统的中危漏洞
  • 弱口令问题

中危（1周内修复）：
  • 一般的中危漏洞
  • 配置问题
  • 信息泄露

低危（有时间再修）：
  • 低危漏洞
  • 最佳实践建议
```

有些漏洞不好补怎么办？
- 不能直接打补丁？用WAF规则先挡一下
- 不能改代码？用网络访问控制限制一下
- 实在补不了？至少先发现了，重点盯着

总之，不能放着不管。

**第三件事：验证修复效果**

补完了漏洞，不是就完了。

得再扫一遍，看看是不是真的补上了。

- 漏洞真的修好了吗？
- 修了漏洞有没有引入新问题？
- 业务有没有受影响？

验证完了，这才算完。

### 3.3.4 第四模块：加固准备——筑牢防线

光补漏洞还不够，还得加固。

加固做得好，能防住很多攻击。

**第一件事：网络加固**

```
边界加固：
  • 防火墙策略优化：最小化原则，只开必要的端口和服务
  • 访问控制：重要系统加白名单
  • NAT策略检查：有没有不该映射的映射出去了？
  • VPN策略：VPN权限收紧，该关的关

内网加固：
  • VLAN划分：不同业务、不同级别分在不同VLAN
  • 内网访问控制：不是谁都能访问谁，该限的限
  • 东西向流量监测：内网之间的流量也要能看到
```

网络是第一道防线，得守好。

**第二件事：系统加固**

```
Windows加固：
  • 关闭不必要的服务
  • 关闭不必要的端口
  • 启用防火墙
  • 禁用Guest账号
  • 管理员账号改名
  • 密码策略（强密码、定期修改）
  • 开启审核策略（登录审核、对象访问审核...）
  • 安装EDR/杀毒软件
  • 禁用默认共享
  • 限制远程桌面

Linux加固：
  • 关闭不必要的服务
  • 关闭不必要的端口
  • 启用防火墙（iptables/firewalld）
  • 禁用root远程登录
  • 密码策略
  • 开启日志审计
  • 安装EDR/杀毒软件
  • 文件权限加固
  • SSH加固（改端口、用密钥、禁用空密码）
```

系统加固的东西很多，可以去找一些加固基线（等保2.0基线、CIS基线...），照着一条一条来。

**第三件事：应用加固**

```
Web应用加固：
  • 部署WAF
  • 配置WAF规则
  • 后台管理地址隐藏
  • 错误信息不泄露
  • 禁止目录遍历
  • 上传文件校验

中间件加固：
  • 升级到安全版本
  • 删除默认页面
  • 禁用不必要的组件
  • 运行权限最小化
  • 配置安全的HTTP头

数据库加固：
  • 不开公网
  • 改默认端口
  • 禁用默认账号
  • 强密码策略
  • 最小权限原则
  • 开启审计日志
```

应用层也很重要，很多攻击都是从Web应用打进来的。

**第四件事：账号加固**

```
• 清理无用账号：离职的、测试的、不用的，全删掉
• 禁用默认账号：admin、test、guest......能禁的禁
• 强密码策略：长度、复杂度、有效期......都安排上
• 多因素认证：重要系统能开MFA的就开
• 最小权限原则：每个人只给必要的权限
• 管理员账号：越少越好，越严越好
```

账号是最容易被忽视的，也是最容易出问题的。

一个弱口令，就能让所有的防护都白搭。

### 3.3.5 第五模块：产品准备——工欲善其事，必先利其器

安全设备买了，得确保它们都能正常工作，都好用。

**第一件事：全面检查**

把所有安全设备都检查一遍：

```
防火墙：
  • 运行正常吗？
  • 策略优化了吗？
  • 特征库更新了吗？
  • 日志能正常输出吗？

IDS/IPS：
  • 运行正常吗？
  • 规则库是最新的吗？
  • 有没有误报/漏报？
  • 告警能正常发到SIEM吗？

WAF：
  • 运行正常吗？
  • 规则库更新了吗？
  • 误报多不多？
  • 策略调优了吗？

EDR：
  • Agent都装了吗？覆盖率多少？
  • 病毒库是最新的吗？
  • 策略配置好了吗？
  • 告警正常吗？

SIEM/态势感知：
  • 日志都接进来了吗？
  • 解析规则对吗？
  • 关联规则写了吗？
  • 告警准确吗？
  • 大屏正常吗？

其他设备：
  • 数据库审计
  • 堡垒机
  • VPN
  • 邮件网关
  • ......
```

一个一个过，确保都正常。

别等护网开始了才发现，某个设备早就挂了，或者日志没接进来。

那就尴尬了。

**第二件事：策略调优**

设备光运行还不够，还得好用。

- IDS规则：是不是太松了？一大堆误报，看都看不过来
- WAF规则：是不是太严了？正常业务都给拦了
- EDR策略：是不是太松了？木马都查不出来

该调的调，该优的优。

平衡好安全性和可用性。

**第三件事：日志检查**

安全分析，靠的就是日志。

日志都有了吗？
- 防火墙日志 ✓
- IDS日志 ✓
- WAF日志 ✓
- 系统日志 ✓
- Web日志 ✓
- 数据库日志 ✓
- 应用日志 ✓

日志都接进SIEM了吗？
- 解析对不对？
- 字段全不全？
- 时间对不对？（时间同步很重要！）
- 有没有丢日志？

日志是蓝队的眼睛。眼睛瞎了，还怎么打？

### 3.3.6 第六模块：预案准备——有备无患

真出事了怎么办？

靠临场发挥？那可不行。

得有预案。

**第一件事：编写应急预案**

各种常见的情况，都写个预案。

```
常见预案：
  • Web应用被入侵应急预案
  • 服务器被入侵应急预案
  • 勒索病毒应急预案
  • 数据泄露应急预案
  • DDoS攻击应急预案
  • 钓鱼邮件应急预案
  • 内网横向移动应急预案
  • 账号被盗应急预案
  • ......
```

预案里写啥？

- 事件分级（什么级别的事件，对应什么级别的响应）
- 响应流程（第一步干啥，第二步干啥）
- 职责分工（谁负责啥）
- 联系方式（找谁、怎么找）
- 处置步骤（具体怎么操作）
- 上报要求（什么时候报、报给谁、报什么）

越详细越好，越具体越好。

别写得跟散文似的，到时候照着做就行。

**第二件事：应急演练**

预案写好了，得练练。

光写不练，等于白写。

怎么练？
- 桌面推演：大家坐下来，模拟一个场景，你说你干啥，我说我干啥，走一遍流程
- 实战演练：真的搞个"攻击"，看看大家的反应，看看预案好不好用

练完了，发现问题，改预案。

多练几次，大家就都熟练了。

真出事了，就不慌了。

**第三件事：工具准备**

应急响应要用的工具，提前准备好。

```
常用工具：
  • 病毒查杀工具
  • 进程查看工具
  • 网络连接查看工具
  • 日志分析工具
  • 流量分析工具
  • 取证工具
  • 漏洞扫描工具
  • ......
```

工具提前装好、提前测试好，确保能用。

别到时候出事了，找个工具找半天，或者下载不下来，急死人。

### 3.3.7 第七模块：人员准备——人是最关键的

设备再好，工具再好，最后还是得靠人。

**第一件事：安全意识培训**

给全体员工做安全意识培训。

为啥？因为人是最薄弱的环节。

红队打不进来技术层面，就会从人下手。

一封钓鱼邮件，员工一点，就中招了。

培训讲啥？
- 什么是钓鱼邮件？怎么识别钓鱼邮件？
- 陌生邮件别乱点开，陌生链接别乱点，陌生文件别乱下载
- 密码安全：强密码、不共用、定期改
- 社会工程学识别：有人打电话套信息别信
- 物理安全：陌生人跟着进门要盘问，U盘别乱插
- 发现可疑情况找谁报告

多讲点真实案例，大家印象深刻。

光讲还不够，最好搞个模拟钓鱼测试，看看大家的警惕性怎么样。

**第二件事：蓝队技能培训**

蓝队自己人也得培训。

- 告警分析培训
- 日志分析培训
- 流量分析培训
- 应急响应培训
- 溯源分析培训
- 工具使用培训

别到时候告警来了，大家都不知道怎么分析。

多培训，多练习，水平才能上去。

**第三件事：战前动员**

护网开始前，开个动员会。

- 讲清楚护网的重要性
- 讲清楚纪律要求
- 讲清楚奖惩机制
- 鼓鼓劲，打打气

士气这东西，很重要。

### 3.3.8 第八模块：后勤准备——兵马未动，粮草先行

别小看后勤，后勤跟不上，仗也打不好。

**吃：**
- 值班的人吃饭怎么解决？
- 夜班有没有夜宵？
- 零食、饮料、咖啡备上

**住：**
- 夜班的人有没有地方休息？
- 万一需要通宵，有没有地方躺一会儿？

**行：**
- 家远的同事，夜班打车能不能报销？
- 万一有紧急情况，能不能及时赶过来？

**钱：**
- 护网补贴有没有？
- 奖金怎么算？
- 报销流程怎么走？

别小看这些东西，这些都跟大家的积极性挂钩。

后勤保障做好了，大家才能安心干活。

## 3.4 准备工作的常见坑

最后，我再给你讲讲准备工作中常见的一些坑，别掉进去。

### 坑一：准备太晚

很多单位，护网前一周才开始准备。

一周时间能干啥？扫个漏洞都扫不完。

准备工作，越早开始越好。

提前两个月开始，轻轻松松。
提前一周开始，手忙脚乱。

### 坑二：资产不清

很多单位，连自己有多少资产都搞不清楚。

- "我们好像有台服务器在云上？"
- "这个域名是谁的？没人知道？"
- "这个系统还在用吗？好像没人管了。"

这些"影子资产"，最容易出问题。

红队最喜欢打没人管的系统——没人维护，漏洞一大堆，打进去了也没人发现。

资产梳理，一定要做细、做全。

### 坑三：漏洞补不完就不补了

扫出来一大堆漏洞，一看，这么多？补不完啊。算了，不补了。

大错特错。

补不完就不补了？那不是更危险？

补一个是一个，补两个是两个。
先补高危的，再补中危的，最后补低危的。
能补多少补多少。

总比一个都不补强吧。

### 坑四：重技术轻管理

很多单位，安全设备买了一堆，漏洞也补了不少，但就是不重视管理。

- 密码还是弱密码
- 账号还是乱开
- 权限还是乱给
- 员工还是乱点钓鱼邮件

结果呢？红队一个钓鱼邮件就打进来了。

技术防护重要，管理也重要。

两手都要抓，两手都要硬。

### 坑五：预案写了不练

预案写得漂漂亮亮的，装订成册，放在柜子里，从来没练过。

真出事了，拿出来一看，全是废话，根本没法照着做。

或者大家根本不知道预案里写了啥，出事了还是乱作一团。

预案不是写出来看的，是用来练的。

多演练，多修改，预案才有用。

### 坑六：只重边界，不重内网

很多单位，边界防护做得很好，防火墙、IDS、WAF......一应俱全。

内网呢？啥都没有。

内网一台机器被黑了，横着走，想去哪去哪。

现在的攻击，都是"边界突破+内网横向移动"的模式。

边界被突破是迟早的事，内网防护同样重要。

纵深防御，内外都要防。

## 3.5 本章总结

这一章我们讲了护网前的准备工作，内容很多，给你总结一下：

**准备时间线：**
- 提前1-2个月开始，越从容越好
- 分阶段推进，从粗到细

**八大模块：**
1. 组织准备：成立工作组、排值班表、通讯保障、战前动员
2. 资产准备：全面盘点、分级分类、攻击面梳理
3. 漏洞准备：全面扫描、按优先级修复、验证效果
4. 加固准备：网络加固、系统加固、应用加固、账号加固
5. 产品准备：全面检查、策略调优、日志检查
6. 预案准备：编写预案、应急演练、工具准备
7. 人员准备：安全意识培训、蓝队技能培训、战前动员
8. 后勤准备：吃住行钱，保障到位

**常见的坑：**
- 准备太晚 → 早动手
- 资产不清 → 摸清楚
- 漏洞补不完就不补 → 能补多少补多少
- 重技术轻管理 → 两手都要硬
- 预案写了不练 → 多演练
- 重边界轻内网 → 纵深防御

记住一句话：

> **护网的胜负，80%在护网开始之前就已经决定了。**

准备工作做得越扎实，护网就越轻松。

不打无准备之仗。
