## 📖 写在前面

再坚固的防线，也有被突破的时候。

攻击来了怎么办？
服务器被入侵了怎么办？
数据被偷了怎么办？
勒索软件来了怎么办？

这时候，就轮到应急响应出场了。

应急响应，就是安全事件发生之后，怎么处理、怎么应对、怎么把损失降到最小。

应急响应是蓝队的核心能力之一。
平时监测、检测做得再好，真出事了顶不上，也白搭。

这一章，我们就来聊聊应急响应：
- 什么是应急响应？
- 应急响应的流程是怎样的？
- 常见的安全事件怎么处置？
- 应急响应有哪些注意事项？
- 怎么写应急响应报告？

坐稳了，开唠。

## 12.1 先从一个故事讲起

### 12.1.1 一次"手忙脚乱"的应急响应

先给你讲个真实的故事。

有一家公司，安全建设一般般。
有一天，出事了。

早上9点，运维小王发现，有一台Web服务器好像不太对劲——
CPU占用特别高，还有可疑的进程。

小王赶紧给安全部门打电话。
安全部门的人过来一看，说："坏了，这是被入侵了，还种了挖矿木马。"

然后呢？
然后就乱套了。

- 有人说："赶紧拔网线！"
- 有人说："别拔！先查清楚怎么进来的！"
- 有人说："赶紧删文件！"
- 有人说："别删！留证据！"
- 有人去查日志，发现日志早就被删了
- 有人想去备份，发现备份服务器也中招了

乱哄哄一上午，啥也没整明白。
木马还在内网里传，又多了几台机器中招。

最后没办法，找了专业的应急响应团队来。
人家来了，三下五除二，两天就搞定了。

后来复盘的时候，人家说：
"你们最大的问题不是技术不行，是没有流程。
出事了，谁干啥、先干啥、后干啥，全乱套。
你一句我一句，耽误了最佳处置时间，还把证据都破坏了。"

---

这个故事告诉我们什么？

**应急响应，不是靠临场发挥，是靠流程。**

平时就要有预案、有流程、有分工。
真出事了，按流程来，一步一步走。
这样才能不乱，才能把损失降到最小。

这一章，我们就来讲讲应急响应的流程和方法。

## 12.2 什么是应急响应？

### 12.2.1 应急响应的定义

啥是应急响应？

用一句话说：

> **安全事件发生后，为了尽快恢复正常、减少损失、查明原因、防止再发生，而进行的一系列工作。**

我再翻译翻译：

出事了，别慌。
- 先把事按住，别让它扩大
- 然后搞清楚怎么回事
- 然后收拾干净
- 然后恢复正常
- 然后总结教训，防止下次再出事

这就是应急响应。

### 12.2.2 应急响应的目标

应急响应要达到什么目标？

```
1. 遏制事态：
   把事件控制住，别让它继续扩大
   把损失降到最小

2. 尽快恢复：
   尽快恢复业务正常运行
   减少业务中断时间

3. 查明原因：
   搞清楚是怎么回事
   攻击怎么进来的
   攻击者干了啥
   影响有多大

4. 根除隐患：
   把攻击者清理干净
   把漏洞补上
   把后门都清掉

5. 总结改进：
   总结经验教训
   改进防护措施
   防止下次再发生
```

这五个目标，一个都不能少。

很多人应急响应，只做到了"恢复"，就完事了。
原因也没查清楚，漏洞也没补上，后门也没清干净。
结果过了几天，攻击者又来了。

那应急响应等于白做了。

### 12.2.3 应急响应的原则

应急响应有几个基本原则：

**原则一：尽快遏制**

先把事情按住，别让它继续扩大。
遏制是第一位的。
损失已经造成了，别让损失变得更大。

**原则二：保护证据**

处置的时候，要注意保护证据。
别上来就删文件、就重装系统。
证据都没了，怎么查原因？
怎么知道攻击者干了啥？
怎么防止下次再发生？

当然，也不能为了保护证据就不遏制。
先遏制，再保护证据。
两个都要。

**原则三：最小影响**

处置的时候，尽量减少对业务的影响。
能不中断业务就不中断。
但如果不中断不行，那该断还是得断。
两害相权取其轻。

**原则四：按流程来**

别乱，别慌，按流程一步一步来。
谁干啥、先干啥、后干啥，都有章法。
乱了阵脚，只会越搞越糟。

## 12.3 应急响应的标准流程

应急响应有很多标准流程，最经典的是PDCERF。

啥是PDCERF？

```
P — Preparation        准备
D — Detection          检测
C — Containment        遏制
E — Eradication        根除
R — Recovery           恢复
F — Follow-up          跟踪/复盘
```

六个阶段，一个都不能少。

下面我们一个一个讲。

### 12.3.1 第一阶段：准备（Preparation）

出事之前的准备工作。

别等出事了才准备，那就晚了。

准备啥？

```
组织准备：
  • 应急响应团队（谁负责啥）
  • 联系人列表（找谁、怎么找）
  • 分工和职责（谁干啥）

技术准备：
  • 应急响应工具包（各种工具）
  • 取证工具
  • 分析工具
  • 干净的系统镜像
  • 备份

流程准备：
  • 应急预案
  • 事件分级标准
  • 上报流程
  • 处置流程
  • 沟通机制

人员准备：
  • 应急响应培训
  • 应急演练
  • 7×24小时值班
```

准备工作做得越充分，应急响应就越从容。

平时不准备，出事了手忙脚乱。

### 12.3.2 第二阶段：检测（Detection）

发现安全事件。

怎么发现？

```
告警发现：
  • SIEM告警
  • 安全设备告警（IDS、WAF、EDR...）
  • 系统告警
  • 日志异常

人工发现：
  • 员工发现（电脑变慢、文件被加密...）
  • 运维发现（系统异常、性能异常...）
  • 安全团队主动发现（威胁狩猎、巡检...）

外部发现：
  • 客户/合作方通知
  • 监管部门通报
  • 厂商通报漏洞
  • 威胁情报
```

发现事件之后，第一步是什么？
不是立刻处置，是——研判。

研判啥？
- 是不是真的安全事件？还是误报？
- 是什么类型的事件？（病毒、入侵、数据泄露...）
- 严重程度怎么样？（一般、重要、紧急）
- 影响范围有多大？（几台机器？还是整个内网？）

研判完了，确定是真的安全事件，那就启动应急响应。

### 12.3.3 第三阶段：遏制（Containment）

确认是安全事件了，第一件事——遏制。

先把事情按住，别让它继续扩大。

怎么遏制？

```
网络层面：
  • 断开受感染机器的网络
  • 隔离受感染的网段
  • 封禁恶意IP/域名
  • 关闭被利用的端口和服务

系统层面：
  • 禁用被盗的账号
  • 停止可疑的服务和进程
  • 断开可疑的连接

应用层面：
  • 暂时下线有漏洞的应用
  • 封堵漏洞利用的入口
  • 禁用危险功能
```

遏制的原则：
- 快！越快越好，时间就是损失
- 准！别把正常业务也搞停了
- 稳！别慌，别乱，按预案来

遏制的时候，要注意保护证据：
- 别上来就格式化、就重装
- 内存、日志、进程，这些都要保留
- 先做个镜像或者快照，再动手

当然，如果事态特别紧急，那先遏制再说。
两害相权取其轻。

### 12.3.4 第四阶段：根除（Eradication）

事态控制住了，然后就是根除——把攻击者彻底赶出去。

根除啥？

```
清除恶意代码：
  • 删病毒、删木马、删WebShell
  • 清Rootkit
  • 清挖矿程序
  • 所有被感染的机器都要清干净

清除后门：
  • 删可疑账号
  • 清启动项（注册表、计划任务、服务...）
  • 清后门程序
  • 清WebShell
  • 检查所有可能留后门的地方

修复漏洞：
  • 补上被利用的漏洞
  • 加固系统
  • 加固应用
  • 加固网络

排查影响：
  • 还有没有其他机器被入侵？
  • 还有没有其他账号被盗？
  • 数据有没有被偷？
  • 业务有没有被破坏？
```

根除要彻底。
别留尾巴。
留一个后门，等于没清。

怎么确认根除干净了？
- 再扫一遍病毒
- 再查一遍进程、启动项、账号
- 观察一段时间，看看有没有异常
- 确认没问题了，再进入下一阶段

### 12.3.5 第五阶段：恢复（Recovery）

清理干净了，然后就是恢复——恢复业务正常运行。

怎么恢复？

```
系统恢复：
  • 从干净的备份恢复
  • 或者重装系统（确保干净）
  • 恢复系统配置

数据恢复：
  • 从备份恢复数据
  • 验证数据的完整性
  • 确保数据没被篡改

应用恢复：
  • 恢复应用服务
  • 恢复配置
  • 测试功能是否正常

网络恢复：
  • 逐步恢复网络连接
  • 先恢复重要业务
  • 再恢复次要业务
  • 边恢复边观察
```

恢复的原则：
- 稳！一步一步来，别一下全放开
- 验证！恢复一个验证一个，确保没问题
- 监控！恢复之后密切监控，防止再出事

恢复的时候，一定要用干净的备份。
别把病毒也恢复了，那就尴尬了。

### 12.3.6 第六阶段：跟踪/复盘（Follow-up）

业务恢复了，应急响应就结束了？

没有。
还有最重要的一步——复盘。

复盘啥？

```
事件回顾：
  • 事件是怎么发生的？
  • 攻击路径是怎样的？
  • 时间线是怎样的？
  • 影响有多大？

原因分析：
  • 为什么会发生？
  • 是技术问题？还是管理问题？
  • 为什么我们没防住？
  • 为什么我们没及时发现？

处置总结：
  • 处置过程中做得好的地方
  • 处置过程中做得不好的地方
  • 有哪些经验教训？

改进计划：
  • 技术层面怎么改进？（补漏洞、加设备、加固...）
  • 流程层面怎么改进？（优化流程、完善预案...）
  • 人员层面怎么改进？（培训、演练...）
  • 什么时候完成？谁负责？
```

复盘是应急响应最有价值的一步。
吃一堑，长一智。
同样的错误，不能犯第二次。

很多单位，应急响应只做到"恢复"就完事了。
复盘也不做，改进也不做。
结果下次同样的攻击又来了，还是中招。

那应急响应的价值就大打折扣了。

## 12.4 常见安全事件的处置要点

讲完了通用流程，再讲讲几种常见安全事件的处置要点。

### 12.4.1 服务器被入侵

服务器被入侵了，怎么办？

```
第一步：遏制
  • 断开服务器网络（别让攻击者继续操作）
  • 禁用可疑账号
  • 停止可疑服务和进程

第二步：取证（注意保护证据）
  • 保存内存镜像
  • 保存硬盘镜像/快照
  • 导出系统日志、安全日志
  • 导出Web日志、应用日志
  • 记录可疑进程、文件、网络连接

第三步：分析
  • 攻击者是怎么进来的？（Web漏洞？弱口令？其他？）
  • 什么时候进来的？
  • 进来之后干了啥？（提权？横向移动？偷数据？）
  • 拿到了什么权限？
  • 有没有留后门？
  • 有没有扩散到其他机器？

第四步：根除
  • 清理WebShell、木马、后门
  • 清理可疑账号、启动项、服务
  • 补上漏洞
  • 加固系统

第五步：恢复
  • 从干净备份恢复，或者重装
  • 恢复业务
  • 密切监控

第六步：复盘
  • 为什么被入侵？
  • 怎么改进？
```

关键点：
- 先断网，保护证据
- 一定要溯源，找到入口点
- 一定要查清楚有没有横向移动
- 一定要清干净后门

### 12.4.2 勒索软件事件

中了勒索软件，怎么办？

```
第一步：立刻遏制
  • 立刻断开受感染机器的网络！
  • 隔离受感染的网段
  • 封禁恶意IP和域名
  • 禁用被盗的账号
  • （防止继续扩散是第一位的！）

第二步：评估情况
  • 是什么勒索软件？
  • 感染了多少机器？
  • 哪些业务受影响？
  • 备份有没有被加密？
  • 有没有解密工具？

第三步：决策
  • 交钱还是不交钱？（一般不建议交）
  • 从备份恢复？还是尝试解密？
  • 要不要报警？（严重的话建议报）

第四步：根除
  • 清除勒索软件
  • 找到并封堵攻击入口
  • 补漏洞、加固

第五步：恢复
  • 从干净备份恢复系统和数据
  • 逐步恢复业务
  • 密切监控

第六步：复盘
  • 怎么进来的？
  • 为什么传播这么快？
  • 怎么改进？
```

关键点：
- 第一时间断网隔离！防止扩散
- 别着急交钱，先想别的办法
- 备份是关键，有备份就不怕
- 一定要找到入口，不然还会再来

### 12.4.3 数据泄露事件

数据被偷了，怎么办？

```
第一步：遏制
  • 堵住数据泄露的通道
  • 禁用被盗的账号
  • 封攻击者的IP
  • 防止数据继续泄露

第二步：评估影响
  • 什么数据被泄露了？
  • 多少数据被泄露了？
  • 数据敏感级别有多高？
  • 影响哪些人？
  • 会不会触犯法律法规？

第三步：溯源
  • 数据是怎么被偷的？
  • 谁偷的？
  • 偷了多少？
  • 偷去干啥了？

第四步：处置
  • 封堵漏洞
  • 清理攻击者
  • 加固系统
  • 必要时报案
  • 必要时通知受影响的用户

第五步：合规
  • 按法律法规要求上报
  • 配合监管部门调查
  • 做好公关沟通

第六步：复盘改进
  • 为什么数据会被泄露？
  • 数据安全防护有哪些不足？
  • 怎么改进？
```

关键点：
- 先止损，别让数据继续泄露
- 评估影响范围和严重程度
- 该上报的上报，该通知的通知
- 数据安全要从根本上改进

### 12.4.4 DDoS攻击

被DDoS了，怎么办？

```
第一步：确认
  • 是不是真的DDoS？
  • 还是正常的业务高峰？
  • 什么类型的DDoS？（流量型？应用层？）

第二步：缓解
  • 开启抗DDoS服务/设备
  • 流量清洗
  • 封可疑IP
  • 扩容带宽
  • 必要时切换IP

第三步：排查
  • 攻击来源是哪？
  • 攻击目标是啥？
  • 有没有其他攻击同时发生？
  • 是不是调虎离山？

第四步：恢复
  • 攻击减弱后，逐步恢复业务
  • 监控流量，防止再次来袭

第五步：复盘
  • 为什么被DDoS？
  • 防护能力够不够？
  • 怎么改进？
```

关键点：
- 先确认是不是真的DDoS
- 抗D设备/服务是关键
- 注意会不会是声东击西，DDoS只是掩护

## 12.5 应急响应的注意事项

最后，聊聊应急响应的几个注意事项，都是血泪教训总结出来的。

### 12.5.1 保护好证据

这个怎么强调都不为过。

很多人一上来就：
- 删文件
- 杀进程
- 重装系统
- 格式化硬盘

证据全给毁了。

然后问："攻击者怎么进来的？"
不知道，证据都没了，怎么查？

所以，处置之前，先留证据：
- 内存镜像先做一个
- 硬盘快照先拍一个
- 日志先导出来
- 进程、网络连接、启动项，先截屏、先记录

当然，如果事态特别紧急，那先遏制再说。
但只要有条件，一定要先保留证据。

证据不只是用来查原因的，万一要报案、要走法律程序，也用得上。

### 12.5.2 按流程来，别乱

出事了，最忌讳的就是乱。

你一言我一语，你说这么干，他说那么干。
最后啥也没干成，还把事情搞砸了。

所以：
- 要有统一的指挥
- 要有明确的分工
- 要按流程来
- 一步一步走

别乱，越乱越坏事。

### 12.5.3 及时上报，别瞒

出事了，很多人的第一反应是：
"先别声张，我们自己处理，处理完了就没事了。"

大错特错。

该上报的一定要上报：
- 给领导上报
- 给监管部门上报（如果需要的话）
- 给客户通报（如果影响到客户的话）

为什么？
- 纸包不住火，早晚会被发现
- 瞒报的后果，比事件本身更严重
- 早点上报，早点获得支持，事情更好处理

当然，上报不是瞎嚷嚷。
先把情况搞清楚，搞清楚了再上报。
上报的时候说清楚：什么事、什么影响、正在怎么处理、需要什么支持。

### 12.5.4 别贸然反击

有些人被攻击了，气不过，想反击回去。
"你打我，我也打你！"

千万别。

为什么？
- 你反击，可能就违法了
- 你知道对方是谁吗？万一是被拐的肉鸡呢？
- 你反击，可能反而给对方口实
- 把事情搞大了，对谁都不好

遇到攻击，做好防护、做好溯源、做好记录就行了。
该报警报警，该走法律程序走法律程序。
别自己动手反击。

打赢了，你违法。
打输了，你更丢人。

没必要。

### 12.5.5 做好沟通

应急响应不是安全团队一个部门的事。
- 要跟运维沟通
- 要跟开发沟通
- 要跟业务部门沟通
- 要跟领导沟通
- 可能还要跟客户、跟监管沟通

沟通很重要。

怎么沟通？
- 及时：有事及时说，别等人家来问
- 准确：说清楚是什么事、什么影响
- 客观：实事求是，别隐瞒，别夸大
- 专业：用人家听得懂的话讲

沟通做好了，能少很多麻烦。

## 12.6 应急响应报告怎么写

应急响应做完了，要写报告。
报告怎么写？

给你一个模板：

```
一、事件概述
  • 事件类型
  • 发生时间
  • 发现时间
  • 影响范围
  • 严重程度
  • 事件描述

二、处置过程
  • 时间线（什么时候发现、什么时候启动应急、什么时候遏制、什么时候根除、什么时候恢复）
  • 处置措施（做了哪些事）
  • 参与人员

三、原因分析
  • 根本原因（为什么会发生？）
  • 攻击路径（攻击者是怎么进来的、干了啥）
  • 为什么没防住？
  • 为什么没及时发现？

四、影响评估
  • 业务影响（中断了多久、影响了多少用户...）
  • 数据影响（数据有没有泄露、有没有被篡改...）
  • 经济损失（大概损失多少钱）
  • 其他影响

五、改进措施
  • 技术改进（补漏洞、加设备、加固...）
  • 流程改进（完善预案、优化流程...）
  • 人员改进（培训、演练...）
  • 改进计划（什么时间、谁负责、完成标准）

六、经验教训
  • 做得好的地方
  • 做得不好的地方
  • 经验总结
```

报告要写得清楚、准确、专业。
让看的人一眼就能看明白：发生了啥、怎么处理的、原因是啥、以后怎么改。

## 12.7 本章总结

这一章我们讲了应急响应实战，内容很多，给你总结一下：

**什么是应急响应：**
- 安全事件发生后的一系列处置工作
- 目标：遏制、恢复、查明、根除、改进

**应急响应的原则：**
- 尽快遏制
- 保护证据
- 最小影响
- 按流程来

**PDCERF六阶段流程：**
1. 准备（Preparation）：组织、技术、流程、人员，提前准备
2. 检测（Detection）：发现事件、研判真假、评估严重程度
3. 遏制（Containment）：先按住，防止事态扩大
4. 根除（Eradication）：清理干净、补上漏洞、排查影响
5. 恢复（Recovery）：恢复业务、逐步上线、密切监控
6. 跟踪/复盘（Follow-up）：总结教训、改进提升、防止再犯

**常见事件处置要点：**
- 服务器被入侵：断网→取证→分析→根除→恢复→复盘
- 勒索软件：立刻隔离→评估→决策→根除→恢复→复盘
- 数据泄露：止损→评估→溯源→处置→合规→改进
- DDoS攻击：确认→缓解→排查→恢复→复盘

**注意事项：**
- 保护好证据（非常重要！）
- 按流程来，别乱
- 及时上报，别瞒
- 别贸然反击
- 做好沟通

**应急响应报告：**
- 事件概述
- 处置过程
- 原因分析
- 影响评估
- 改进措施
- 经验教训

记住一句话：

> **平时多流汗，战时少流血。**

预案写得好、演练搞得多、流程顺，
真出事了，就不慌，就能处理得好。
