## 9.1 先从一个故事讲起

### 9.1.1 一个漏洞毁了所有

先给你讲个真实的故事。

有一家公司，安全建设做得还不错：
- 防火墙买的最好的
- IDS/IPS也有
- 终端装了EDR
- 服务器也做了加固
- 内网也分段了

老板觉得，我们这么多安全设备，应该挺安全的吧。

结果呢？

护网第一天，就被打进来了。

怎么进来的？

很简单——一个Web应用的文件上传漏洞。

攻击者扫到了一个政务系统，这个系统有个文件上传的功能，没做严格校验。
攻击者上传了一个WebShell，就这么进去了。

然后呢？
然后提权、横向移动、打域控、偷数据...
一路畅通。

花了几百万买的安全设备，啥用都没有。

为啥？
因为攻击是从Web应用进来的，走的是80和443端口。
防火墙得放行啊，不然用户怎么访问网站？
WAF也有，但这个漏洞是个0day，WAF规则还没更更新。
EDR也装了，但WebShell是文件，一开始没检测出来。

就一个Web漏洞，整个防线全废了。

---

这个故事告诉我们什么？

**Web安全，是边界防护的重中之重。**

Web应用是最常见的攻击入口。
Web防护做不好，其他安全设备再多也没用。

人家从大门光明正大地走进来的，你能怪城墙不高吗？

所以，Web攻击的检测和防护，是蓝队必须掌握的基本功。

## 9.2 常见的Web攻击类型

首先，你得知道常见的Web攻击有哪些。

我给你列一下最常见的，也是最危险的几种。

### 9.2.1 SQL注入

最经典、最古老、也是最危险的Web攻击之一。

啥是SQL注入？

简单说就是：
攻击者把SQL命令插到Web参数里，后端的数据库执行了这些恶意SQL，结果就被攻击了。

比如：
一个登录接口，正常的请求是：
```
username=admin&password=123456
```

攻击者改成：
```
username=admin' or 1=1--&password=随便
```

如果后端没做防护，SQL就变成了：
```sql
SELECT * FROM users WHERE username='admin' or 1=1--' AND password='随便'
```

`or 1=1` 永远成立， `--` 是注释符，后面的都被注释了。
结果就是，不用密码也能登录。

这就是SQL注入。

**危害有多大？**
- 可以拖库（把整个数据库都偷了）
- 可以删库（把数据都删了）
- 可以提权（拿到服务器权限）
- 危害非常大

**怎么检测？**
- WAF检测：看请求里有没有SQL关键字（select、union、and、or、sleep、load_file...）
- 日志检测：Web日志里有没有SQL注入特征
- 数据库审计：数据库执行的SQL有没有异常的

**怎么防护？**
- 预编译（PreparedStatement）
- 输入校验
- WAF
- 最小权限原则（数据库账号别给太高权限）
- 定期做漏洞扫描

### 9.2.2 XSS（跨站脚本攻击）

XSS也是非常常见的Web攻击。

啥是XSS？

简单说就是：
攻击者把恶意脚本（一般是JavaScript）插到网页里，用户访问这个网页的时候，脚本就在用户的浏览器里执行了。

比如：
一个论坛，用户发帖的时候，在内容里插一段：
```html
<script>偷你cookie的代码</script>
```

如果论坛没做过滤，其他用户看这个帖子的时候，这段脚本就会执行，用户的cookie就被偷走了。

偷了cookie干啥？
可以冒充这个用户登录啊。

**XSS分三种：**
- **反射型**：脚本在URL里，服务器反射回来
- **存储型**：脚本存在数据库里，每个访问的用户都中招
- **DOM型**：纯前端的，不经过服务器

存储型最危险，因为每个访问的人都会中招。

**怎么检测？**
- WAF检测：请求里有没有 `<script`、`javascript:`、`onerror`、`onload` 这些关键字
- 看看输出的HTML里有没有用户可控的内容没转义

**怎么防护？**
- 输出编码（HTML转义、JS转义）
- 输入校验
- CSP（内容安全策略）
- HttpOnly Cookie（就算被偷了也用不了）
- WAF

### 9.2.3 文件上传漏洞

这个也是非常常见的，而且直接就能getshell，危害极大。

啥是文件上传漏洞？

很多Web应用都有上传功能：上传头像、上传附件、上传文档...
如果上传的时候没校验好，攻击者就能上传一个脚本文件（比如 .php、.jsp、.asp），然后访问这个文件，就能执行代码了。

比如：
一个网站有头像上传功能，只在前端校验了文件后缀名。
攻击者抓个包，把文件名改成 `shell.php`，就上传成功了。
然后访问 `/upload/shell.php`，就能执行命令了。

就这么简单。

**怎么检测？**
- WAF检测：上传请求里有没有可疑的文件后缀
- 看Web日志里有没有对上传目录里脚本文件的访问
- 服务器上找可疑的文件

**怎么防护？**
- 文件后缀白名单（只允许上传指定类型，黑名单不可靠）
- 文件内容检测（别光看后缀，看内容是不是真的是图片）
- 文件名随机化（别用用户传的文件名）
- 上传目录不可执行（把上传目录的执行权限去掉）
- 上传目录和域名分开（最好放CDN或者对象存储上）
- WAF

### 9.2.4 命令注入/代码执行

这个也很危险，直接就能执行系统命令。

啥是命令注入？

Web应用需要调用系统命令的时候，如果把用户输入拼到命令里了，没做过滤，攻击者就能注入恶意命令。

比如：
一个网站有个ping功能，用户输入IP，后台去ping一下，把结果返回给用户。
后台执行的命令大概是：`ping 用户输入的IP`

如果用户输入的是：`127.0.0.1; whoami`
那后台执行的就变成了：`ping 127.0.0.1; whoami`
`;` 是命令分隔符，两个命令都会执行。
于是 `whoami` 就被执行了，攻击者就能看到当前用户是谁。

这就是命令注入。

代码执行类似，只不过执行的是代码（PHP、Python、Java...），不是系统命令。
一样危险。

**怎么检测？**
- WAF检测：请求里有没有 `;`、`|`、`&&`、`$(` 这些命令分隔符
- 看系统日志里有没有异常的命令执行

**怎么防护？**
- 尽量不要调用系统命令
- 必须调用的话，用参数化的方式，别拼接字符串
- 输入校验（只允许输入合法的字符）
- 用安全的函数替代危险函数
- WAF

### 9.2.5 逻辑漏洞

前面几种都是"技术漏洞"，靠WAF、靠代码规范就能防住大部分。

还有一种漏洞，叫"逻辑漏洞"，这个就比较头疼了。

啥是逻辑漏洞？

简单说就是：业务逻辑设计有问题，被攻击者利用了。

比如：
- 越权访问：普通用户能看管理员的内容
- 密码重置：可以重置别人的密码
- 支付漏洞：1块钱买100块的东西
- 验证码绕过：可以无限尝试
- 短信轰炸：可以无限发验证码

这些漏洞，WAF根本检测不出来，因为请求看起来都是"正常"的。
正常的参数、正常的URL、正常的访问，只是业务逻辑有问题。

**怎么检测？**
- 这个WAF基本没用
- 得靠人工渗透测试、代码审计
- 业务安全测试

**怎么防护？**
- 开发的时候就要考虑安全
- 每个接口都要做权限校验
- 关键操作要二次验证
- 做业务安全测试
- 做代码审计

逻辑漏洞是最容易被忽视的，也是最容易出问题的。
很多时候，技术层面防护得很好，结果栽在了业务逻辑上。

### 9.2.6 其他常见攻击

还有一些常见的，简单提一下：

**CSRF（跨站请求伪造）：**
攻击者诱导用户访问一个恶意页面，这个页面里向目标网站发请求。
因为用户已经登录了目标网站，所以请求会带着用户的cookie，目标网站会以为是用户自己操作的。
比如：用户登录了网银，攻击者诱导用户访问他的网站，他的网站里有个图片，地址是网银的转账接口。
用户一访问，就转了一笔钱。

防护：加token、验证Referer、关键操作二次验证。

**SSRF（服务器端请求伪造）：**
攻击者让服务器去请求一个URL，服务器去请求了，结果就被利用了。
比如：攻击者让服务器去请求内网的地址，就能探测内网、攻击内网系统。

防护：限制能请求的地址、用白名单、禁用不需要的协议。

**反序列化漏洞：**
应用把用户输入反序列化了，如果反序列化的类有问题，就能执行代码。
Java、PHP、Python 都有这种问题。

防护：别反序列化不可信的数据、用安全的序列化方式、更新组件版本。

**目录遍历：**
攻击者通过 `../` 这样的方式，访问到web目录以外的文件。
比如：`download.php?file=../../etc/passwd`，就能下载到系统密码文件。

防护：输入校验、规范文件路径、限制在指定目录里。

## 9.3 Web攻击怎么检测？

讲完了攻击类型，再说说怎么检测。

### 9.3.1 WAF检测

最常用的就是WAF（Web应用防火墙）。

WAF干啥的？
专门检测和防护Web攻击的。

WAF怎么检测？
主要是特征匹配——
请求来了，看看里面有没有攻击特征。
有，就拦下来。

WAF能检测啥？
- SQL注入
- XSS
- 文件上传
- 命令注入
- 代码执行
- 目录遍历
- 常见的Web漏洞攻击

WAF的准确率还是挺高的，大部分已知的Web攻击都能拦下来。

但WAF不是万能的：
- 0day漏洞拦不住（特征库还没更新）
- 编码绕过可能拦不住（各种编码混淆）
- 逻辑漏洞拦不住（看起来是正常请求）
- 加密流量不好检测（HTTPS的话得解密）

WAF是第一道防线，很重要，但不是唯一的防线。

### 9.3.2 日志分析

除了WAF，还得自己看Web日志。

为啥？
因为WAF可能有漏网之鱼。
而且就算WAF拦住了，你也得知道谁在攻击你、用的什么手法。

Web日志看啥？

**看状态码：**
- 大量404 → 可能在扫路径、扫备份文件
- 大量403 → 可能在试探权限
- 大量500 → 可能在fuzz，触发了错误

**看URL：**
- URL里有SQL关键字 → SQL注入尝试
- URL里有 `<script` → XSS尝试
- URL里有 `../` → 目录遍历尝试
- URL里有 `.php`、`.jsp` 但正常网站不用这种后缀 → 可能在扫漏洞
- 访问了上传目录里的可疑文件 → 可能已经上传了WebShell

**看请求方法：**
- 大量POST请求到同一个URL → 可能在暴力破解、或者WebShell通信
- 不常见的请求方法（PUT、DELETE...）→ 可能在利用PUT上传文件

**看User-Agent：**
- 扫描器的UA（sqlmap、nmap、awvs...）→ 有人在扫描
- 没有UA、UA很奇怪 → 可疑
- 工具的UA → 不是正常人的浏览器

**看来源IP：**
- 同一个IP，短时间内大量请求 → 扫描、暴力破解
- 同一个IP，触发了很多WAF告警 → 攻击者
- 凌晨时间，不常见的IP → 可疑

日志分析很重要。
WAF可能会漏，但日志不会骗你。

### 9.3.3 流量分析

比日志更靠谱的，是流量分析。

为啥？
日志是应用记的，万一应用有问题，日志可能不全、不对。
流量是真实的通信，不会骗人。

流量分析看啥？
- 看HTTP请求的完整内容（URL、参数、POST数据）
- 看响应内容（有没有返回敏感信息、有没有WebShell的特征）
- 看异常的请求模式（比如WebShell的心跳包）
- 看上传的文件内容（是不是真的图片，还是脚本）

如果有全流量分析设备，Web攻击的检测能力会强很多。

### 9.3.4 WebShell检测

WebShell是Web攻击的"终极产物"。
攻击者打进来之后，肯定会上传一个WebShell。

WebShell怎么检测？

**静态检测：**
- 扫文件，找可疑的PHP/JSP/ASP文件
- 看文件里有没有危险函数（eval、assert、system、exec...）
- 看文件修改时间（突然出现的新文件？）
- 看文件权限和所有者（是不是Web服务器的权限？）

**动态检测：**
- 看Web进程有没有起异常的子进程（cmd、powershell、bash...）
- 看网络连接有没有异常的外联
- 看文件操作有没有异常（新建文件、修改系统文件）

**流量检测：**
- 同一个URL，大量POST请求
- 请求和响应的内容很可疑
- 冰蝎、哥斯拉这些有特征的流量模式

EDR、HIDS这些主机安全产品，一般都带WebShell检测功能。

发现WebShell了，就说明服务器已经被入侵了。
赶紧处理！

## 9.4 Web攻击怎么防护？

检测是一方面，防护更重要。

怎么防护Web攻击？

### 9.4.1 第一层：WAF

最外面一层，WAF。

WAF部署在Web应用前面，所有访问Web的流量都先经过WAF。
WAF检测到攻击，就直接拦下来。

WAF怎么选？
- 云WAF：阿里云、腾讯云的云WAF，部署简单，适合云环境
- 硬件WAF：物理设备，性能好，适合传统IDC
- 软件WAF：装在服务器上的，比如 ModSecurity

WAF怎么用？
- 别光用默认规则，要根据自己的业务调优
- 减少误报（别把正常业务给拦了）
- 及时更新规则库
- 关注最新的漏洞，及时加规则

WAF是第一道防线，能拦住大部分已知攻击。

### 9.4.2 第二层：代码安全

WAF是外面的防线，最根本的还是代码本身要安全。

代码安全怎么做？

**安全开发规范：**
- 输入校验：所有用户输入都要校验，长度、格式、类型...
- 输出编码：输出到HTML、JS、SQL里的内容，都要做对应的编码
- 不用危险函数：eval、system、exec 这些能不用就不用
- 参数化查询：SQL查询用预编译，别拼接字符串
- 最小权限：数据库账号、运行账号，都给最小权限

**安全测试：**
- 代码审计：写完代码，专人做安全审计
- 渗透测试：上线前，找专业的人做渗透测试
- 静态扫描（SAST）：用工具扫代码里的漏洞
- 动态扫描（DAST）：上线前用扫描器扫一遍

**安全开发流程（DevSecOps）：**
- 把安全嵌到开发流程里
- 需求阶段考虑安全
- 开发阶段按规范写
- 测试阶段做安全测试
- 上线前做安全检查

代码安全是根本。
代码写得安全，就算没有WAF，也没那么容易被攻破。

### 9.4.3 第三层：主机安全

Web服务器本身的安全也很重要。

就算Web应用被攻破了，如果服务器够安全，攻击者也折腾不了多久。

主机安全怎么做？
- 服务器加固（关端口、关服务、最小权限）
- 装EDR/HIDS（能检测WebShell、异常进程、异常操作）
- 定期打补丁
- Web服务降权运行（别用root、别用system权限跑Web）
- 上传目录不可执行（就算上传了WebShell，也执行不了）
- 重要目录做文件完整性监控（被改了立刻告警）

主机安全是Web服务器的最后一道防线。

### 9.4.4 第四层：数据安全

最后，数据安全。

万一前面所有防线都被突破了，数据安全就是最后一道防线。

怎么做？
- 敏感数据加密存储（密码、身份证、手机号...）
- 数据访问控制（谁能访问什么数据，最小权限）
- 数据库审计（谁查了什么数据，都记下来）
- 数据脱敏（测试环境别用真实数据）
- 防数据泄露（DLP）
- 备份（数据被删了还能恢复）

数据安全做好了，就算被攻破了，损失也不会太大。

## 9.5 WAF的使用和调优

WAF是Web防护的核心设备，很多人都用，但真正用好的不多。

这里专门聊聊WAF怎么用、怎么调优。

### 9.5.1 WAF的部署模式

WAF有几种常见的部署模式：

**1. 透明网桥模式**
WAF像网桥一样串在网络里，Web流量经过WAF。
优点：不用改配置，透明部署。
缺点：WAF挂了，整个Web就挂了。

**2. 反向代理模式**
WAF当反向代理，用户先访问WAF，WAF再转发给Web服务器。
优点：功能强，能做的事情多。
缺点：需要改DNS或者IP，WAF是单点。

**3. 旁路镜像模式**
WAF不串在链路里，流量镜像一份给WAF。
WAF只检测，不拦截（或者通过其他方式拦截）。
优点：不影响业务，WAF挂了也没事。
缺点：拦截不及时，或者没法实时拦截。

**4. 云WAF**
云服务商提供的WAF，域名解析过去就行。
优点：部署最简单，不用管硬件。
缺点：数据要经过云厂商，适合云环境。

根据自己的情况选。
一般来说，反向代理模式最常用，功能也最强。

### 9.5.2 WAF的常见问题

WAF用起来，经常遇到这些问题：

**问题一：误报太多**

WAF把正常的业务请求当成攻击给拦了。
用户正常发个帖，内容里有个 `<script>` 标签（比如讲技术的文章），结果被WAF拦了。

怎么办？
- 加白名单：这个URL、这个参数，不检测
- 调规则：把误报的规则关了或者调松
- 自定义规则：针对自己的业务写规则
- 不能一关了之，要找到平衡点

**问题二：漏报**

攻击来了，WAF没拦下来。
比如0day漏洞、编码绕过、逻辑漏洞...

怎么办？
- 及时更新规则库
- 关注新漏洞，及时加自定义规则
- 不能光靠WAF，还要有其他检测手段
- 定期做渗透测试，看看WAF能不能拦住

**问题三：性能问题**

WAF串在链路里，会增加延迟。
如果流量太大，WAF性能不够，可能会成为瓶颈。

怎么办？
- 选性能足够的WAF
- 做集群、做负载均衡
- 静态资源可以不经过WAF
- 缓存常用的内容

WAF不是买了就完事了，得调、得管、得优化。

### 9.5.3 WAF使用的最佳实践

给你几个WAF使用的最佳实践：

1. **先观察，再拦截**
   刚上线的时候，先开"观察模式"（只告警不拦截）。
   观察一段时间，把误报都调好了，再开"拦截模式"。
   别一上来就拦，把正常业务拦了就麻烦了。

2. **定期调优规则**
   规则不是一成不变的。
   业务在变，攻击也在变。
   定期（比如每个月）看看误报、漏报情况，调调规则。

3. **关注新漏洞**
   新的Web漏洞出来了，赶紧看看自己有没有受影响。
   受影响的话，赶紧加WAF规则先挡一下，然后再补代码。

4. **WAF不是万能的**
   别觉得有了WAF就万事大吉了。
   WAF能拦已知攻击，拦不住0day和逻辑漏洞。
   该做代码安全还得做，该做测试还得测。

5. **和其他设备联动**
   WAF发现攻击了，把IP告诉防火墙，防火墙直接封了。
   WAF和SIEM联动，告警统一管理。
   联动起来，效果更好。

## 9.6 Web安全防护的常见误区

最后，说说Web安全防护的几个常见误区。

### 误区一：有WAF就安全了

很多人觉得，装了WAF，Web安全就搞定了。

大错特错。

WAF只是第一道防线，只能拦已知攻击。
- 0day拦不住
- 逻辑漏洞拦不住
- 业务问题拦不住
- 社会工程拦不住

WAF是必要的，但不是万能的。

### 误区二：只重技术，不重业务

很多人技术防护做得很好，SQL注入、XSS都防住了。
结果栽在逻辑漏洞上。

- 越权访问
- 支付漏洞
- 密码重置漏洞
- 验证码绕过

这些漏洞，技术层面看起来都是正常的。
但业务逻辑有问题，一样能被利用。

技术和业务，两手都要抓。

### 误区三：只靠工具，不靠人

很多人觉得，有了扫描器、有了WAF，就不用人管了。

不对。

工具只是辅助。
- 扫描器扫不出逻辑漏洞
- WAF拦不住0day
- 工具不会思考，不会变通

人，才是最关键的。

定期做渗透测试、代码审计、安全评审。
这些都得靠人。

### 误区四：只防外面，不防里面

很多人防Web攻击，只防外部的攻击者。
内部的呢？

内部人员搞破坏怎么办？
内部人员偷数据怎么办？
内部账号被盗了怎么办？

别觉得内部就是安全的。
很多数据泄露事件，都是内部人干的。

内外都要防。

## 9.7 本章总结

这一章我们讲了Web攻击的检测与防护，内容很多，给你总结一下：

**常见的Web攻击：**
- SQL注入：最经典、最危险
- XSS：跨站脚本，偷用户cookie
- 文件上传：直接getshell
- 命令注入/代码执行：直接执行命令
- 逻辑漏洞：业务逻辑问题，WAF拦不住
- 其他：CSRF、SSRF、反序列化、目录遍历...

**Web攻击怎么检测：**
- WAF检测：最常用，特征匹配，拦已知攻击
- 日志分析：看状态码、URL、UA、IP
- 流量分析：更真实，能发现更多
- WebShell检测：静态、动态、流量

**Web防护四层体系：**
1. WAF：第一道防线，拦大部分已知攻击
2. 代码安全：根本，代码写得好才是真的好
3. 主机安全：Web服务器的最后一道防线
4. 数据安全：最后一道防线，就算被攻破了损失也不大

**WAF的使用：**
- 部署模式：透明网桥、反向代理、旁路镜像、云WAF
- 常见问题：误报、漏报、性能
- 最佳实践：先观察再拦截、定期调优、关注新漏洞、不是万能的、和其他设备联动

**常见误区：**
- 有WAF就安全了 → WAF不是万能的
- 只重技术不重业务 → 逻辑漏洞更危险
- 只靠工具不靠人 → 人是最关键的
- 只防外面不防里面 → 内外都要防

记住一句话：

> **Web应用是最大的攻击面，Web防护是边界防护的重中之重。**

Web这道门守好了，攻击者就难进多了。
