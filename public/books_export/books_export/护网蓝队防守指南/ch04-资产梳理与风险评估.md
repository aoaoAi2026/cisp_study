# 第四章 资产梳理与风险评估 —— 知己知彼

> 第4章 | 预计学习时间：4小时 | 难度：⭐⭐

---

## 📖 写在前面

孙子兵法曰：

> "知己知彼，百战不殆。"

打仗是这样，防守也是这样。

你连自己家里有啥都不知道，怎么防？

很多单位安全做得不好，根子上的原因就是——资产没摸清。

- "我们好像有台服务器在阿里云上？谁管的来着？"
- "这个域名是谁注册的？没人知道？"
- "这个系统还在用吗？好像没人维护了。"
- "这个数据库里存的啥数据？重要不重要？"

这些"没人管的资产"，就是红队最喜欢的突破口。

为什么？因为没人管 → 没人打补丁 → 漏洞一大堆 → 一打就进来了 → 进来了也没人发现。

完美。

所以，资产梳理是蓝队所有工作的基础。基础不牢，地动山摇。

这一章，我们就好好聊聊：怎么把资产摸清楚？怎么做风险评估？怎么给资产分级分类？

坐稳了，开唠。

## 4.1 先从一个真实的故事讲起

### 4.1.1 一台被遗忘的测试机

先给你讲个真实的故事。

有一年护网，我认识的一个蓝队队长，姓王。

护网开始前，王队长带着团队做了充分的准备：
- 资产梳理了一遍
- 漏洞扫了三遍
- 高危漏洞全补了
- 安全加固也做了
- 应急预案也写了

王队长觉得，这次护网应该没什么大问题。

护网开始了，前三天风平浪静。

第四天，出事了。

凌晨两点，监控告警：有一台服务器在对内网做端口扫描。

王队长一下子精神了，赶紧查。

一查，发现这台服务器是一台测试服务器，放在一个没人管的网段里。

"这台机器是谁的？"王队长问。

大家面面相觑，没人知道。

查了半天，才搞清楚——这是两年前一个项目的测试机，项目做完了，机器忘了关，也没人管了，就一直在那跑着。

系统是Windows Server 2008，好几年没打补丁了。
密码是弱口令：`test123456`。
还开了3389远程桌面，直接映射在公网上。

红队怎么打进来的？

很简单——扫到了这个公网IP，扫到了3389端口，弱口令试了一下，进去了。

然后就用这台机器当跳板，开始扫内网。

要不是这台机器扫描的动静太大，触发了告警，还不知道什么时候能发现呢。

王队长吓出一身冷汗：
"还好发现得早，不然内网就被打穿了。"

赶紧把机器断网了，赶紧查有没有其他机器被渗透。

忙活了一整夜，总算没造成太大损失。

护网结束后复盘，王队长说：
"我们把所有重要系统都查了三遍，没想到栽在一台没人管的测试机上。真是百密一疏啊。"

---

这个故事告诉我们什么？

**资产梳理，最怕的就是"漏"。**

你把重要资产都查得明明白白，没用。
只要漏掉一台，红队就能从那台打进来。

木桶效应——一个木桶能装多少水，取决于最短的那块木板。

防守也是一样——你的防线牢不牢，取决于最薄弱的那个资产。

所以，资产梳理，一定要全、要细、要准。

## 4.2 资产梳理到底梳理啥？

很多人一听到资产梳理，第一反应就是：
"不就是列个清单吗？把IP和服务器列出来不就行了？"

哪有那么简单。

资产梳理，梳理的不只是"有什么"，还要梳理"是什么、归谁管、重要不重要、有什么风险"。

我给你列个清单，你对照着看，看你们单位梳理全了没有。

### 4.2.1 网络资产

先从最底层的网络开始。

```
公网资产：
  • 公网IP段 / 公网IP地址
  • 域名（主域名、子域名、备用域名）
  • 公网开放的端口和服务
  • 公网应用系统
  • VPN/远程访问入口
  • 云资产（阿里云、腾讯云、AWS...）

内网资产：
  • 内网IP段 / VLAN划分
  • 网络设备（路由器、交换机、负载均衡...）
  • 安全设备（防火墙、IDS、IPS、WAF...）
  • 无线接入点（WiFi）
  • 网络拓扑图

网络配置：
  • 防火墙策略
  • NAT映射
  • 访问控制列表（ACL）
  • 路由配置
```

很多单位，内网有啥都搞不清楚。
- "这个网段是干啥的来着？"
- "这个VLAN是谁在用？"
- "这条防火墙规则谁加的？为啥加？"

这些都得搞清楚。

### 4.2.2 主机资产

网络上面跑的，就是各种主机/服务器。

```
物理服务器：
  • 位置（哪个机房、哪个机柜）
  • 配置（CPU、内存、硬盘）
  • 操作系统（Windows/Linux/其他、版本号）
  • 用途（Web服务器、数据库服务器、文件服务器...）
  • 责任人（谁管的）
  • 重要级别

虚拟服务器：
  • 在哪台物理机上
  • 配置、系统、用途、责任人...
  • （其他和物理机一样）

云服务器：
  • 哪个云厂商
  • 实例ID
  • 配置、系统、用途、责任人...
  • （其他和物理机一样）

终端/办公电脑：
  • 数量
  • 操作系统
  • 是否安装了EDR/杀毒软件
  • 补丁更新情况
```

别只记服务器，终端也很重要。

很多时候，红队就是从终端打进来的——
一封钓鱼邮件，员工点了，终端被黑了，然后从终端横向移动到服务器。

终端也是资产，不能忽视。

### 4.2.3 应用系统资产

服务器上面跑的，就是各种应用系统。

```
Web应用：
  • 应用名称
  • 访问地址
  • 开发语言/框架（Java、PHP、Python...）
  • 中间件（Tomcat、Nginx、Apache...）
  • 数据库（MySQL、Oracle、SQL Server...）
  • 版本号
  • 开发/运维负责人
  • 重要级别
  • 数据敏感级别

数据库：
  • 数据库类型和版本
  • 存了什么数据
  • 数据敏感级别
  • 访问权限
  • 备份情况
  • 负责人

其他应用：
  • 中间件
  • 消息队列
  • 缓存
  • FTP服务器
  • 文件服务器
  • ......
```

应用系统是红队最喜欢的攻击入口——
Web漏洞一打一个准。

所以应用系统的信息，一定要全。

### 4.2.4 数据资产

最后，也是最重要的——数据。

为什么黑客要攻击你？
不是为了黑你的服务器玩，是为了偷你的数据。

所以数据资产，才是最核心的资产。

```
数据分类：
  • 核心数据（比如：用户隐私数据、核心业务数据、商业机密...）
  • 重要数据（比如：内部管理数据、普通业务数据...）
  • 一般数据（比如：公开数据、测试数据...）

数据存储：
  • 存在哪？（哪些数据库、哪些文件服务器...）
  • 有多少？（量级）
  • 谁管？（数据责任人）
  • 备份情况？

数据安全：
  • 加密了吗？（传输加密、存储加密）
  • 访问控制怎么样？（谁能访问、权限多大）
  • 审计了吗？（谁访问过、干了啥）
  • 脱敏了吗？（测试环境有没有用真实数据）
```

数据资产梳理清楚了，你才知道：
- 哪些数据是命根子，得重点保护
- 万一被偷了，损失有多大
- 该怎么保护这些数据

很多单位，连自己有什么数据、数据存在哪、谁能访问，都搞不清楚。

那还怎么防？

### 4.2.5 人员与组织资产

最后，还有一类资产，很多人容易忽视——人。

```
人员：
  • 管理员账号（系统管理员、网络管理员、数据库管理员...）
  • 普通用户账号
  • 第三方人员（外包、运维、开发...）
  • 离职人员（账号删了吗？）

组织：
  • 安全组织架构
  • 安全管理制度
  • 安全流程
  • 应急预案
```

人是最薄弱的环节，也是最重要的资产。

人的问题不解决，技术再强也没用。

## 4.3 怎么把资产梳理全？

说起来容易，做起来难。

怎么才能把资产都梳理全？特别是那些"影子资产"——没人管、没人要、没人知道的资产。

我给你分享几个实用的方法。

### 4.3.1 方法一：自动化扫描

这是最基本的方法。

用扫描器扫，把能扫到的都扫出来。

```
网络扫描：
  • Nmap扫端口、扫服务
  • Masscan扫大网段
  • 各种资产测绘工具（比如FOFA、钟馗之眼...）

域名/子域名扫描：
  • 子域名爆破（subfinder、amass...）
  • 证书透明度（CT）查询
  • 搜索引擎查询（Google Hacking）
  • 第三方威胁情报

Web应用扫描：
  • AWVS、AppScan、Xray...
  • 爬虫爬取所有页面和接口
  • 被动扫描
```

自动化扫描的好处是快、全，能扫到很多你不知道的资产。

但坏处也有：
- 扫不到的资产（比如不在线的、网段没扫到的）
- 扫不准（比如服务识别错误）
- 需要人工确认和补充

所以，自动化扫描是第一步，不是最后一步。

### 4.3.2 方法二：人工盘点

光靠扫描不行，还得人工盘点。

怎么盘？

**第一步：找各个部门要**

- 找运维要服务器清单
- 找网络要网络设备清单
- 找开发要应用系统清单
- 找业务要数据清单
- 找行政要终端清单

各个部门都有自己的台账，先收上来，汇总一下。

**第二步：逐个确认**

收上来的清单，很多是不准的、过时的。

怎么办？
- 跟扫描结果对一对，对得上的，确认
- 对不上的，找相关的人问
- 没人认领的，标记出来，重点核查

**第三步：实地勘察**

有条件的话，最好去机房实地看看。

- 机柜里有多少台机器？
- 哪些是开着的？哪些是关着的？
- 网线怎么接的？
- 跟台账对不对得上？

很多时候，台账是一回事，实际是另一回事。

实地看一眼，比看十页台账都管用。

### 4.3.3 方法三：从流量反推

还有一个好办法——从流量反推。

你有全流量分析设备吗？或者防火墙日志、NetFlow数据？

有的话，从流量里就能看到：
- 哪些IP在通信？
- 哪些端口在开放？
- 哪些应用在跑？
- 哪些服务器是活跃的？

很多"影子资产"，台账里没有，但流量里有。

因为只要它在联网，就会有流量。

从流量反推，往往能发现很多惊喜。

### 4.3.4 方法四：从日志反推

跟流量类似，从日志里也能反推出资产。

比如：
- 从SIEM的日志里，看看有哪些设备在发日志
- 从防火墙日志里，看看有哪些IP在访问
- 从域名解析日志里，看看有哪些域名在解析

很多资产，你可能忘了它的存在，但它一直在默默地上报日志。

从日志里一找，就找到了。

### 4.3.5 方法五：从管理系统查

还有一个地方，往往藏着完整的资产清单——各种管理系统。

比如：
- CMDB（配置管理数据库）
- 云平台控制台（阿里云后台、腾讯云后台...）
- 虚拟化管理平台（VMware vCenter...）
- AD域控（里面有所有的计算机和用户）
- IT资产管理系统
- 财务系统（买服务器总要走财务吧？）

从这些系统里导出数据，跟你自己梳理的对一对。

往往能发现很多漏掉的。

### 4.3.6 怎么发现"影子资产"？

最后，专门说说"影子资产"——那些没人管、没人知道的资产。

这些资产最危险，因为：
- 没人打补丁 → 漏洞多
- 没人监控 → 被黑了也不知道
- 没人负责 → 出事了找不到人

怎么发现这些影子资产？

几个实用技巧：

1. **扫一遍所有公网IP段**，看看有哪些IP是通的、有端口开放的。然后去台账里找，找不到的，就是影子资产。

2. **扫一遍所有内网网段**，看看有哪些活跃主机。台账里没有的，重点查。

3. **查云平台**，看看有多少台云主机。很多单位，各个部门自己买云主机，公司根本不知道。

4. **查DNS记录**，看看有多少子域名。每个子域名背后都对应着一个系统，很多系统你可能都没听说过。

5. **查财务记录**，看看每年花在IT上的钱，花在哪了？买了哪些设备、哪些服务？跟台账对不对得上？

6. **问老员工**，特别是干了五六年以上的老运维。很多"历史遗留"资产，只有他们知道。

总之，影子资产不可怕，可怕的是你不知道它的存在。

只要找到了，就好办了。

## 4.4 资产分级分类

资产都梳理出来了，然后呢？

然后给它们分个级、分个类。

为什么要分级分类？

因为资源是有限的。
你不可能给所有资产同样的保护等级。

- 域控和一台测试服务器，防护等级能一样吗？
- 核心业务数据和公开数据，保护等级能一样吗？
- 生产环境和测试环境，防护等级能一样吗？

肯定不一样。

好钢要用在刀刃上。

### 4.4.1 怎么分级？

资产一般分几级？

常见的是分三级或四级。我推荐三级，简单好记。

```
核心资产（一级）：
  没了就要命的。
  比如：
  • 域控服务器
  • 核心业务系统
  • 核心数据库
  • 用户隐私数据
  • 商业机密数据
  • ......

重要资产（二级）：
  很重要，但不是致命的。
  比如：
  • 普通业务系统
  • 重要办公系统
  • 内部管理系统
  • 普通业务数据
  • ......

一般资产（三级）：
  一般般，被黑了影响也不大。
  比如：
  • 办公终端
  • 测试系统
  • 公开数据
  • 内部论坛
  • ......
```

怎么判定一个资产属于哪一级？

主要看三个维度：

1. **保密性**：如果数据泄露了，影响有多大？
2. **完整性**：如果数据被篡改了，影响有多大？
3. **可用性**：如果系统挂了，影响有多大？

三个维度综合评估，取最高的那个。

比如：
- 一个公开的网站，保密性低，完整性中，可用性高 → 重要资产
- 一个内部文档系统，保密性中，完整性中，可用性中 → 重要资产
- 一个核心数据库，保密性高，完整性高，可用性高 → 核心资产

大概就是这么个思路。

不用追求特别精确，大概分清楚就行。

### 4.4.2 分级之后干啥？

分级不是为了好看，是为了指导防护。

不同级别的资产，防护要求不一样。

```
核心资产：
  • 最高优先级的保护
  • 必须部署全流量监测
  • 必须部署EDR/HIDS
  • 日志必须全采集
  • 必须有备份，而且是异地备份
  • 访问控制最严格
  • 漏洞修复最优先
  • 专人负责，7×24小时监控

重要资产：
  • 较高优先级的保护
  • 重要的部署监测
  • 尽量部署EDR
  • 重要日志采集
  • 定期备份
  • 访问控制较严格
  • 高危漏洞优先修复
  • 有人负责，工作时间监控

一般资产：
  • 常规保护
  • 杀毒软件
  • 基本日志
  • 定期备份（有就更好）
  • 基本访问控制
  • 漏洞有空就补
  • 有人管就行
```

一句话：越重要的资产，保护越严。

把80%的资源，投到20%的核心资产上。

这才是高效的防守。

### 4.4.3 资产责任人

除了分级，还要定责任人。

每个资产，都得有对应的责任人。

- 出了问题找谁？
- 漏洞修复找谁？
- 应急处置找谁？

都得明确。

最怕的就是"三不管"资产——谁都不管，谁都不负责。

出事了，推来推去，没人管。

所以，每个资产，都得指定一个责任人。

出了问题，就找他。

## 4.5 风险评估怎么做？

资产梳理完了，分级也分完了，然后呢？

然后做风险评估。

啥叫风险评估？

简单说就是：
> 看看你的资产，面临哪些风险？这些风险有多大？先处理哪个？

### 4.5.1 风险怎么算？

风险是怎么算出来的？

一个简单的公式：

```
风险 = 资产价值 × 威胁 × 脆弱性
```

用人话说就是：

- **资产价值**：这个资产值多少钱？有多重要？
- **威胁**：这个资产被攻击的可能性有多大？
- **脆弱性**：这个资产有多少漏洞？有多脆弱？

三个乘起来，就是风险大小。

比如：
- 一个核心服务器（价值高），暴露在公网上（威胁高），还有很多高危漏洞（脆弱性高）→ 风险极高
- 一个测试系统（价值低），在内网里（威胁低），虽然有漏洞（脆弱性高）→ 风险一般
- 一个办公终端（价值中），经常点钓鱼邮件（威胁高），没装杀毒软件（脆弱性高）→ 风险高

大概就是这么个意思。

### 4.5.2 常见的风险有哪些？

我给你列一些常见的风险，你对照着看，你们单位有没有。

```
技术风险：
  • 系统漏洞（高危、中危、低危）
  • Web应用漏洞（SQL注入、XSS、文件上传...）
  • 弱口令/默认口令
  • 不安全的配置
  • 缺少安全设备（没有WAF、没有EDR、没有IDS...）
  • 日志不全/没有集中存储
  • 没有备份/备份不可用
  • 网络隔离不到位
  • ......

管理风险：
  • 没有安全制度
  • 没有安全责任人
  • 账号管理混乱（乱开账号、离职不删号）
  • 权限过大（人人都是管理员）
  • 没有安全培训
  • 没有应急预案
  • 没有定期演练
  • 第三方人员管理混乱
  • ......

人员风险：
  • 安全意识薄弱
  • 弱密码习惯
  • 容易被钓鱼
  • 管理员技术水平不足
  • 人员流动大
  • ......
```

风险无处不在。

你要做的，就是把这些风险都找出来，排个序，先处理最严重的。

### 4.5.3 风险怎么分级？

风险一般分四级：

```
极高风险（红色）：
  可能造成严重损失的，必须立即处理。
  比如：
  • 核心系统有远程代码执行漏洞
  • 公网服务器存在弱口令
  • 核心数据没有备份
  • ......

高风险（橙色）：
  可能造成较大损失的，尽快处理。
  比如：
  • 重要系统有高危漏洞
  • 内部网络隔离不到位
  • 重要数据加密不足
  • ......

中风险（黄色）：
  有一定影响，但不是特别紧急的，排期处理。
  比如：
  • 一般系统有中危漏洞
  • 安全配置不规范
  • 安全制度不完善
  • ......

低风险（蓝色）：
  影响很小的，有空再处理。
  比如：
  • 低危漏洞
  • 最佳实践建议
  • 安全意识有待提高
  • ......
```

怎么判定属于哪一级？

还是看那三个维度：
- 可能性有多大？（会不会发生）
- 影响有多大？（发生了会怎样）
- 资产有多重要？（发生在谁身上）

综合评估一下，大概定个级就行。

不用追求100%准确，八九不离十就可以。

### 4.5.4 风险处置策略

风险找到了，然后怎么办？

一般有四种处置策略：

```
1. 规避（Avoid）：
   把风险源去掉，彻底避免。
   比如：
   • 这个系统漏洞太多修不好？那就下线，不用了。
   • 这个端口太危险？那就关了，不开放了。
   • 这个第三方服务风险太高？那就不用了，换一个。

2. 转移（Transfer）：
   把风险转移给别人。
   比如：
   • 买保险（出事了保险公司赔）
   • 外包给专业公司（出事了他们负责）
   • 上云（云厂商负责基础设施安全）

3. 缓解（Mitigate）：
   采取措施，降低风险。
   这是最常用的。
   比如：
   • 有漏洞 → 补漏洞
   • 有弱口令 → 改强密码
   • 没有WAF → 装个WAF
   • 没有备份 → 做备份

4. 接受（Accept）：
   风险太小，或者处理成本太高，那就接受它。
   比如：
   • 一个低危漏洞，修的话要改很多代码，不值当，那就先接受
   • 一个测试系统，被黑了也没啥，那就接受
   • 概率极低、影响极小的风险，接受就好
```

具体用哪种策略，看情况。

- 风险极高的，能规避就规避，不能规避就缓解
- 风险高的，尽量缓解
- 风险中的，排期缓解
- 风险低的，接受就行

总之，把高风险的都处理掉，中低风险的慢慢处理，就差不多了。

## 4.6 资产梳理的常见误区

最后，再说说资产梳理的几个常见误区，别掉进去。

### 误区一：资产梳理是一次性工作

很多人觉得，资产梳理嘛，搞一次，列个清单，就完事了。

大错特错。

资产是动态变化的。
- 今天上了个新系统
- 明天下线了个老系统
- 后天买了几台云主机
- 大后天项目组加了几台测试机

你梳理完的清单，过三个月可能就不准了。

所以，资产梳理不是一次性工作，是持续性的工作。

- 定期更新（至少每季度更新一次）
- 有变更及时更新（上新系统、下线系统...）
- 定期复核（每年至少全面复核一次）

资产清单，得是活的，不是死的。

### 误区二：只管服务器，不管终端

很多单位的资产清单里，只有服务器，没有终端。

"终端有啥好管的？不就是员工办公电脑吗？"

错了。

终端是红队最喜欢的突破口之一。

- 钓鱼邮件 → 终端中招 → 横向移动 → 打穿内网

这条路径，现在是最常见的攻击路径之一。

终端不管，等于大门敞开。

所以，终端也得纳入资产管理。

- 有多少台终端？
- 都装了EDR/杀毒软件了吗？
- 补丁都打了吗？
- 系统版本是什么？

心里得有数。

### 误区三：只管技术，不管数据

很多人资产梳理，只梳理设备和系统，不梳理数据。

"数据？数据存在数据库里，数据库我们已经管了啊。"

不对。

数据库是资产，数据本身也是资产，而且是更重要的资产。

黑客攻击你，不是为了你的数据库，是为了库里的数据。

所以，数据资产也要梳理：
- 有哪些数据？
- 哪些是核心数据？
- 数据存在哪？
- 谁能访问？
- 加密了吗？
- 备份了吗？

这些都得搞清楚。

### 误区四：梳理完就完了，不用风险评估

很多人资产梳理完了，列了个长长的清单，然后呢？

然后就没有然后了。

清单放那吃灰。

那你梳理它干啥？

资产梳理是手段，不是目的。

目的是什么？
- 知道哪些资产重要 → 重点保护
- 知道哪些资产有风险 → 赶紧修复
- 知道资产分布 → 优化防护策略

梳理完了，要做风险评估，要排优先级，要整改，要落地。

不然，梳理了也白梳理。

## 4.7 本章总结

这一章我们讲了资产梳理和风险评估，内容挺多的，给你总结一下：

**资产梳理的范围：**
- 网络资产（公网、内网、网络设备、安全设备...）
- 主机资产（物理服务器、虚拟服务器、云服务器、终端...）
- 应用系统资产（Web应用、数据库、中间件...）
- 数据资产（核心数据、重要数据、一般数据...）
- 人员与组织资产（账号、制度、流程...）

**怎么把资产梳理全：**
- 自动化扫描（端口扫描、子域名扫描、漏洞扫描...）
- 人工盘点（找各部门要、逐个确认、实地勘察）
- 从流量反推（全流量分析、防火墙日志）
- 从日志反推（SIEM日志、DNS日志）
- 从管理系统查（CMDB、云平台、AD域）
- 发现"影子资产"（公网扫、内网扫、查云、查财务）

**资产分级分类：**
- 核心资产 → 最高优先级保护
- 重要资产 → 较高优先级保护
- 一般资产 → 常规保护
- 每个资产都要有责任人

**风险评估：**
- 风险 = 资产价值 × 威胁 × 脆弱性
- 风险分级：极高、高、中、低
- 风险处置：规避、转移、缓解、接受

**常见误区：**
- 资产梳理是一次性工作 → 要持续更新
- 只管服务器不管终端 → 终端也是重要资产
- 只管技术不管数据 → 数据才是核心
- 梳理完就完了 → 要做风险评估和整改

记住一句话：

> **知己知彼，百战不殆。**

先把自己搞清楚了，才能谈防守。

资产梳理，是蓝队一切工作的基础。

基础打牢了，后面的工作才能顺利开展。
