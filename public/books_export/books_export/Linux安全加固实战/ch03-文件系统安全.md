# 第3章 文件系统安全

## 3.1 文件权限基础

### 3.1.1 Linux权限模型概述

Linux文件系统安全建立在精心设计的权限模型之上。在Linux中，每个文件和目录都有一组权限属性，决定了哪些用户可以读取、写入或执行它们。这种权限模型被称为自主访问控制（Discretionary Access Control, DAC），因为文件的所有者可以自主决定谁有权访问其文件。

在Linux文件系统中，权限被分为三个类别：所有者（Owner）、所属组（Group）和其他用户（Others）。每种类别又定义了三种权限：读取（Read, r）、写入（Write, w）和执行（Execute, x）。这种9位权限模型构成了Linux安全的基础。

```
# 查看文件权限示例
$ ls -la /etc/passwd /etc/shadow /bin/ls
-rw-r--r-- 1 root root  3421 Jun 25 10:30 /etc/passwd
-rw-r----- 1 root root  4341 Jun 25 10:30 /etc/shadow
-rwxr-xr-x 1 root root 139920 Jan 15 2024 /bin/ls

# 权限字段解释：
# -rw-r--r--
# 第一个字符表示文件类型（-为普通文件，d为目录，l为符号链接）
# rw- 为所有者权限（root可读写）
# r-- 为所属组权限（root组成员可读）
# r-- 为其他用户权限（其他用户可读）
```

权限的数字化表示是另一种常用方式。每种权限对应一个数值：r=4, w=2, x=1。因此：

- rwx (7) = 读 + 写 + 执行
- rw- (6) = 读 + 写
- r-x (5) = 读 + 执行
- r-- (4) = 只读
- --- (0) = 无权限

```bash
# 权限的数字化表示示例
chmod 755 /path/to/file    # rwxr-xr-x (所有者可读写执行，组和其他可读执行)
chmod 644 /path/to/file    # rw-r--r-- (所有者可读写，组和其他只读)
chmod 600 /path/to/file    # rw------- (仅所有者可读写)
chmod 700 /path/to/dir     # rwx------ (仅所有者可读写执行)
```

### 3.1.2 chmod命令详解

`chmod`（change mode）命令用于修改文件或目录的权限。这是Linux系统中最常用的命令之一，掌握其各种用法对系统管理至关重要。

```bash
# 基本语法
chmod [选项] 模式 文件

# 常用选项
# -R: 递归处理（用于目录）
# -v: 显示每个文件的处理信息
# -c: 只显示有变更的信息
# -f: 不显示错误信息
# --reference=RFILE: 参考某个文件的权限

# 符号模式设置权限
# 格式：[who][operator][permission]

# who部分：
# u - 用户（所有者）
# g - 组
# o - 其他用户
# a - 所有（相当于ugo）

# operator部分：
# + - 添加权限
# - - 移除权限
# = - 设置权限

# permission部分：
# r - 读取
# w - 写入
# x - 执行
# s - SUID/SGID
# t - Sticky Bit

# 示例
chmod u+x file              # 为所有者添加执行权限
chmod g-w file              # 为组移除写权限
chmod o=r file              # 设置其他用户为只读
chmod a+rx file             # 为所有用户添加读和执行权限
chmod u=rw,g=r,o= file      # 所有者读写，组只读，其他无权限

# 数字模式设置权限
chmod 755 script.sh         # rwxr-xr-x
chmod 644 document.txt      # rw-r--r--
chmod 600 secret.key        # rw-------
chmod 700 .ssh              # rwx------
chmod 600 .ssh/authorized_keys  # 用户SSH密钥文件权限
```

```bash
# 目录权限的特殊性
# 目录的权限含义与文件不同：
# r (list) - 允许列出目录内容（如ls）
# w (create/delete) - 允许在目录中创建或删除文件
# x (access) - 允许进入目录或访问其中的文件

# 示例：安全设置Web目录
chmod -R 755 /var/www/html          # 允许所有人读取和执行
chmod -R 644 /var/www/html/*.html    # HTML文件只读
chmod -R 755 /var/www/cgi-bin        # CGI脚本可执行

# 示例：用户主目录安全设置
chmod 700 /home/username            # 只有所有者能访问
chmod 711 /home/username/.ssh       # 允许SSH配置访问但不列出文件
chmod 600 /home/username/.ssh/authorized_keys
chmod 600 /home/username/.ssh/id_rsa
```

### 3.1.3 chown和chgrp命令详解

`chown`（change owner）和`chgrp`（change group）命令分别用于修改文件的所有者和所属组。正确的文件所有权设置是系统安全的重要保障。

```bash
# chown基本语法
chown [选项] owner[:group] file

# 常用选项
# -R: 递归处理
# -v: 显示处理信息
# -c: 只显示有变更的信息
# --reference=RFILE: 参考某个文件的属主

# 示例：修改文件所有者
chown user1 file.txt               # 修改所有者为user1
chown user1:group1 file.txt        # 修改所有者为user1，组为group1
chown :group1 file.txt            # 只修改组为group1
chown user1: file.txt             # 修改所有者为user1，组为用户的主组

# 递归修改目录所有权
chown -R user1:group1 /home/user1/webroot

# 修改Web目录所有者
chown -R www-data:www-data /var/www/html

# 修改数据库文件所有者
chown -R mysql:mysql /var/lib/mysql

# 使用参考文件设置所有权
chown --reference=/path/to/reference /path/to/target
```

```bash
# chgrp基本语法（与chown类似，但只改组）
chgrp [选项] group file

# 常用选项
# -R: 递归处理
# -v: 显示处理信息
# -c: 只显示有变更的信息
# --reference=RFILE: 参考某个文件

# 示例
chgrp www-data file.txt            # 修改组为www-data
chgrp -R developers /project       # 递归修改组
```

```bash
# 实际应用场景示例

# 1. 设置Apache/Nginx Web目录
chown -R www-data:www-data /var/www/html
find /var/www/html -type d -exec chmod 755 {} \;
find /var/www/html -type f -exec chmod 644 {} \;

# 2. 设置用户主目录
chown -R username:username /home/username
chmod 700 /home/username                    # 主目录只有所有者可访问
chmod 711 /home/username/.ssh               # SSH目录可访问但不列出内容
chmod 600 /home/username/.ssh/authorized_keys
chmod 600 /home/username/.ssh/id_rsa

# 3. 设置系统日志目录
chown -R root:adm /var/log
chmod 755 /var/log
chmod 644 /var/log/*.log 2>/dev/null
chmod 750 /var/log/secure 2>/dev/null
chmod 640 /var/log/auth.log 2>/dev/null

# 4. 设置临时目录
chmod 1777 /tmp                             # Sticky bit确保只能删除自己的文件
chown root:root /tmp
```

## 3.2 特殊权限

### 3.2.1 SUID权限详解

SUID（Set User ID）是一种特殊权限，当应用于可执行文件时，允许该程序以文件所有者的权限运行，而不是以执行该程序的用户的权限运行。这是一个强大的功能，但如果使用不当，也可能造成严重的安全风险。

SUID权限的典型应用是`passwd`命令。普通用户需要修改自己的密码，这需要写入/etc/shadow文件（该文件只有root可写）。通过为passwd命令设置SUID，普通用户可以临时获得root权限来修改shadow文件。

```bash
# 查看SUID文件
find / -perm -4000 -type f 2>/dev/null

# 示例输出：
# -rwsr-xr-x 1 root root 35672 Jan 15 2024 /usr/bin/passwd
# -rwsr-xr-x 1 root root  6724 Jan 15 2024 /usr/bin/sudo
# -rwsr-xr-x 1 root root 48384 Jan 15 2024 /usr/bin/newgrp
# -rwsr-xr-x 1 root root 67336 Jan 15 2024 /usr/bin/gpasswd
# ---s--x--- 1 root root 0 Jan 15 2024 /usr/bin/crontab
# -rwsr-xr-x 1 root root 22384 Jan 15 2024 /bin/mount
# -rwsr-xr-x 1 root root 14624 Jan 15 2024 /bin/umount
# -rwsr-xr-x 1 root root 39168 Jan 15 2024 /bin/su

# 注意：s 表示有SUID权限且有执行权限
#       S (大写) 表示有SUID权限但没有执行权限（异常状态）
```

```bash
# 设置SUID权限
chmod u+s /path/to/file    # 符号方式
chmod 4755 /path/to/file   # 数字方式（4表示SUID）

# 移除SUID权限
chmod u-s /path/to/file
chmod 755 /path/to/file

# 查找所有SUID文件并记录
find / -perm -4000 -type f -exec ls -la {} \; 2>/dev/null > /root/suid-files.txt

# 检查SUID文件是否被篡改（与已知哈希比较）
sha256sum /usr/bin/passwd > /root/passwd.sha256
# 之后验证
sha256sum -c /root/passwd.sha256
```

**SUID安全风险**：SUID程序如果存在漏洞，攻击者可能利用它获得root shell。以下是一些已知的高风险SUID程序：

```bash
# 高风险SUID程序列表（通常应该避免设置SUID或考虑移除）
# /bin/ping (可用cap_net_raw替代)
# /bin/mount (某些系统)
# /bin/umount (某些系统)
# /usr/bin/at
# /usr/bin/gpasswd
# /usr/bin/passwd (必需但应监控)
# /usr/bin/sudo (必需但应监控)
# /usr/bin/su (必需但应监控)
# /usr/bin/crontab
# /usr/bin/xpdf (如果有)
# /usr/bin/mlocate (如果有)
```

### 3.2.2 SGID权限详解

SGID（Set Group ID）与SUID类似，但它设置的是GID而不是UID。当应用于目录时，SGID会使该目录中创建的新文件继承该目录的组，而不是创建者的主组。

```bash
# 查看SGID文件
find / -perm -2000 -type f 2>/dev/null

# 示例：
# -rwxr-sr-x 1 root root 6724 Jan 15 2024 /usr/bin/sudo

# 查看SGID目录
find / -perm -2000 -type d 2>/dev/null

# 示例：
# dr-xr-sr-x 2 root root 4096 Jun 25 10:30 /var/log/apache2
# drwxr-sr-x 2 root root 4096 Jun 25 10:30 /var/log/nginx
```

```bash
# 设置SGID权限
chmod g+s /path/to/directory    # 符号方式
chmod 2755 /path/to/directory    # 数字方式（2表示SGID）

# 移除SGID权限
chmod g-s /path/to/directory

# SGID目录的典型应用：共享目录
# 创建共享目录
mkdir /project/shared
chgrp developers /project/shared
chmod 2775 /project/shared    # 2775 = rwxrwsr-x

# 这样，任何在developers组中的用户在/shared中创建的文件
# 都会自动属于developers组
```

### 3.2.3 Sticky Bit权限详解

Sticky Bit是一种特殊的权限，主要用于目录。当一个目录设置了Sticky Bit后，只有文件的所有者、目录的所有者或root用户才能删除或重命名该目录中的文件。这对于/tmp或/var/shared等共享目录尤为重要。

```bash
# 查看Sticky Bit目录
ls -ld /tmp
# drwxrwxrwt 18 root root 4096 Jun 25 10:30 /tmp

# 注意最后的 t 表示Sticky Bit且有执行权限
# T (大写) 表示Sticky Bit但没有执行权限（异常状态）
```

```bash
# 设置Sticky Bit权限
chmod +t /path/to/directory    # 符号方式
chmod 1777 /path/to/directory  # 数字方式（1表示Sticky Bit）

# 移除Sticky Bit权限
chmod -t /path/to/directory

# 正确设置/tmp目录权限
chmod 1777 /tmp
chown root:root /tmp

# 安全设置公共写入目录
mkdir -p /var/shared
chown root:users /var/shared
chmod 1777 /var/shared
```

## 3.3 ACL访问控制列表

### 3.3.1 ACL基础概念

传统的Unix权限模型只有所有者、组和其他用户三种分类，这在很多场景下是不够用的。例如，可能需要让多个不相关的用户或组对同一文件拥有不同的权限。ACL（Access Control List，访问控制列表）扩展了这一模型，允许更精细的权限控制。

```bash
# 检查文件系统是否支持ACL
# 大多数现代Linux文件系统（ext4、xfs、btrfs）都支持ACL
tune2fs -l /dev/sda1 | grep "Default mount options"
# 或
mount | grep acl

# 临时启用ACL（如果文件系统支持但未启用）
mount -o remount,acl /

# 永久启用ACL（添加到/etc/fstab）
# /dev/sda1 / ext4 defaults,acl 0 1
```

### 3.3.2 getfacl和setfacl命令

```bash
# 查看文件的ACL
getfacl filename

# 示例输出：
# # file: test.txt
# # owner: user1
# # group: group1
# user::rw-
# user:user2:rw-         # 额外为user2设置读写权限
# group::r--             # 所有者组的权限
# group:developers:rw-   # 额外为developers组设置读写权限
# mask::rw-              # ACL掩码
# other::r--             # 其他用户的权限
```

```bash
# 设置文件的ACL

# 为单个用户设置权限
setfacl -m u:username:rx /path/to/file

# 为单个组设置权限
setfacl -m g:groupname:rx /path/to/file

# 为多个用户/组设置权限
setfacl -m u:user1:rw,u:user2:r,g:group1:rx /path/to/file

# 设置目录的默认ACL（该目录下新建的文件自动继承此ACL）
setfacl -m d:u:username:rx /path/to/directory
setfacl -m d:g:groupname:rx /path/to/directory

# 移除特定用户的ACL
setfacl -x u:username /path/to/file

# 移除所有ACL
setfacl -b /path/to/file

# 复制ACL到另一个文件
getfacl file1 | setfacl --set-file=- file2

# 递归设置ACL
setfacl -R -m u:username:rx /path/to/directory
setfacl -R -m d:u:username:rx /path/to/directory  # 默认ACL递归
```

```bash
# ACL应用示例

# 1. 允许特定用户访问某目录
setfacl -m u:developer:rx /project/confidential
setfacl -m u:developer:rx /project/confidential/*

# 2. 允许开发组读取，生产组不能访问
setfacl -m g:developers:rx /project
setfacl -m g:production:--- /project

# 3. 为Web目录设置灵活的权限
setfacl -R -m u:www-data:rx /var/www/html
setfacl -R -m d:u:www-data:rx /var/www/html

# 4. 备份ACL设置
getfacl -R /home > /root/home.acl.backup

# 5. 恢复ACL设置
setfacl --restore=/root/home.acl.backup
```

### 3.3.3 ACL掩码和有效性

ACL掩码（mask）限制了所有额外授予的权限（用户ACL和组ACL）的最大范围。如果掩码设置为r--，那么即使某个用户被授予了rwx权限，实际也只能获得r--权限。

```bash
# 查看ACL掩码
getfacl /path/to/file | grep mask

# 修改ACL掩码
setfacl -m m::r /path/to/file

# 有效权限与实际权限的关系
# 如果ACL掩码是r--，即使设置u:user:rwx，实际只有r--
setfacl -m u:user:rwx /path/to/file
getfacl /path/to/file
# 显示：user::rwx
#       user:user:rwx      # effective:r--
# 最后的effective:r--表示实际权限被掩码限制了
```

## 3.4 文件属性

### 3.4.1 chattr和lsattr命令

Linux提供了另一种安全机制——文件属性（File Attributes）。与权限不同，文件属性用于控制文件的一些特殊行为，如防止删除、修改或追加。

```bash
# 查看文件属性
lsattr filename

# 示例输出：
# -------------e-- /etc/passwd
# ----i---------e-- /etc/shadow
# -----a---------e-- /var/log/wtmp

# 属性说明：
# a - append only（只能追加，不能覆盖或删除）
# A - no atime updates（不更新访问时间）
# c - compressed（自动压缩）
# C - no copy on write（禁用写时复制）
# d - no dump（不被dump命令备份）
# e - extent format（使用扩展区格式）
# i - immutable（不可变，不能删除、修改、重命名）
# j - data journaling（数据日志）
# s - secure deletion（安全删除）
# S - synchronous updates（同步更新）
# T - top of directory hierarchy（目录层级顶部）
# u - undeletable（不可删除）
```

```bash
# 设置文件属性

# 防止文件被修改（非常有用但要小心！）
chattr +i /etc/passwd           # 不可变（即使root也不能修改，除非移除属性）
chattr +i /etc/shadow
chattr +i /etc/gshadow
chattr +i /etc/group

# 允许追加但不允许覆盖（适合日志文件）
chattr +a /var/log/auth.log     # 只能追加，不能覆盖或删除
chattr +a /var/log/syslog
chattr +a /var/log/wtmp

# 防止文件被删除
chattr +u /path/to/file

# 组合使用
chattr +i +A /etc重要文件

# 移除属性
chattr -i /path/to/file
chattr -a /path/to/file
```

```bash
# 安全加固中的实际应用

# 1. 保护系统密码和账户文件
chattr +i /etc/passwd
chattr +i /etc/shadow
chattr +i /etc/group
chattr +i /etc/gshadow
chattr +i /etc/crontab
chattr +i /etc/sudoers
chattr -i /etc/sudoers.d  # sudoers.d目录不能是immutable

# 2. 保护日志文件
chattr +a /var/log/auth.log
chattr +a /var/log/secure
chattr +a /var/log/syslog
chattr +a /var/log/wtmp
chattr +a /var/log/lastlog

# 3. 查看设置了特殊属性的文件
lsattr /etc/passwd
lsattr /etc/shadow
lsattr /var/log/*.log

# 4. 找出所有immutable文件
find / -attr +i 2>/dev/null

# 5. 找出所有append-only文件
find / -attr +a 2>/dev/null
```

**注意**：即使root用户也不能删除设置了`i`属性的文件，除非先移除该属性。这提供了一层额外的保护，但也可能导致合法操作失败。例如，系统更新时可能需要修改/etc/passwd，如果该文件被设为不可变，更新就会失败。

```bash
# 如果需要修改immutable文件
# 先移除immutable属性
chattr -i /etc/passwd

# 修改文件
vim /etc/passwd

# 重新设置为immutable
chattr +i /etc/passwd
```

## 3.5 敏感文件保护

### 3.5.1 系统关键文件权限标准

Linux系统中有一些文件对系统安全至关重要，必须严格控制其访问权限。

```bash
# 标准的系统关键文件权限配置

# /etc/passwd - 所有用户可读，但只有root可写
chmod 644 /etc/passwd
chown root:root /etc/passwd

# /etc/shadow - 只有root可读写
chmod 600 /etc/shadow
chown root:root /etc/shadow

# /etc/gshadow - 只有root可读写
chmod 600 /etc/gshadow
chown root:root /etc/gshadow

# /etc/group - 所有用户可读，但只有root可写
chmod 644 /etc/group
chown root:root /etc/group

# /etc/sudoers - 必须600，只能root和sudo组可读
chmod 600 /etc/sudoers
chown root:root /etc/sudoers

# /etc/sudoers.d/ - 目录权限700，内部文件权限600
chmod 700 /etc/sudoers.d
chmod 600 /etc/sudoers.d/*

# /etc/ssh - SSH配置目录权限700
chmod 700 /etc/ssh
chmod 600 /etc/ssh/sshd_config
chmod 600 /etc/ssh/ssh_config
chmod 600 /etc/ssh/*_key          # 私钥文件权限600
chmod 644 /etc/ssh/*_key.pub      # 公钥文件权限644

# /etc/pam.d/ - PAM配置目录
chmod 755 /etc/pam.d
chmod 644 /etc/pam.d/*

# /etc/shells - 用户可用Shell列表
chmod 644 /etc/shells
chown root:root /etc/shells
```

### 3.5.2 自动检查和修复脚本

```bash
#!/bin/bash
# filename: secure-permissions.sh
# 用于检查和修复系统关键文件权限

echo "检查系统文件权限..."

# 定义正确的权限
declare -A CORRECT_PERMS=(
    ["/etc/passwd"]="644:root:root"
    ["/etc/shadow"]="600:root:root"
    ["/etc/gshadow"]="600:root:root"
    ["/etc/group"]="644:root:root"
    ["/etc/sudoers"]="600:root:root"
    ["/etc/shells"]="644:root:root"
)

# 检查并修复文件权限
for file in "${!CORRECT_PERMS[@]}"; do
    if [ -f "$file" ]; then
        expected_perms="${CORRECT_PERMS[$file]}"
        expected_mode=$(echo "$expected_perms" | cut -d: -f1)
        expected_owner=$(echo "$expected_perms" | cut -d: -f2)
        expected_group=$(echo "$expected_perms" | cut -d: -f3)
        
        current=$(stat -c "%a:%U:%G" "$file")
        
        if [ "$current" != "${expected_mode}:${expected_owner}:${expected_group}" ]; then
            echo "修复: $file ($current -> $expected_perms)"
            chmod "$expected_mode" "$file"
            chown "$expected_owner:$expected_group" "$file"
        else
            echo "正常: $file"
        fi
    fi
done

# 检查SSH目录和配置
if [ -d "/etc/ssh" ]; then
    echo "检查SSH配置..."
    current_ssh_dir=$(stat -c "%a" /etc/ssh)
    if [ "$current_ssh_dir" != "700" ]; then
        echo "修复: /etc/ssh 权限 (${current_ssh_dir} -> 700)"
        chmod 700 /etc/ssh
    fi
fi

echo "权限检查完成。"
```

## 3.6 关键目录权限

### 3.6.1 /tmp目录安全

/tmp目录用于存储临时文件，几乎所有用户都需要在其中创建文件。因此，/tmp目录必须设置全世界可写的权限，但这也带来了安全风险。攻击者可能利用/tmp目录进行符号链接攻击或符号替换攻击。

```bash

# 推荐的/tmp挂载选项
# noexec - 禁止执行/tmp中的程序
# nodev - 禁止设备文件
# nosuid - 禁止SUID程序

# 3. 如果使用tmpfs（内存文件系统）
# 在/etc/fstab中添加：
# tmpfs /tmp tmpfs defaults,noexec,nodev,nosuid,mode=1777 0 0
```


### 3.6.2 /var目录安全

/var目录包含可变的系统数据，包括日志文件、 spool目录、临时文件和运行时数据。


### 3.6.3 /home目录安全

/home目录是普通用户的主目录存储位置。用户主目录中可能包含敏感的个人数据、SSH密钥、配置文件等。

```bash
chmod 600 /home/username/.ssh/id_rsa
chmod 644 /home/username/.ssh/id_rsa.pub
chmod 600 /home/username/.ssh/config

chmod 600 /home/username/.bashrc
chmod 600 /home/username/.profile
chmod 600 /home/username/.bash_history
chmod 600 /home/username/.mysql_history
```

## 3.7 文件系统挂载选项

### 3.7.1 常用安全挂载选项

Linux文件系统挂载时可以选择多种安全相关的选项，这些选项可以有效防止多种攻击。

## 3.9 加密文件系统

### 3.9.1 LUKS加密

LUKS（Linux Unified Key Setup）是Linux下标准的磁盘加密方案。它提供了透明的文件系统加密，可以加密整个分区或逻辑卷。

```bash
# 安装加密工具
apt-get install cryptsetup     # Debian/Ubuntu
yum install cryptsetup        # CentOS/RHEL

# 创建加密容器（文件而非分区）
dd if=/dev/urandom of=/secure/encrypted.img bs=1M count=1024
losetup /dev/loop0 /secure/encrypted.img

# 初始化LUKS容器
cryptsetup luksFormat /dev/loop0
# 警告：此操作会销毁现有数据

# 打开加密容器
cryptsetup luksOpen /dev/loop0 secure_volume

# 创建文件系统
mkfs.ext4 /dev/mapper/secure_volume

# 挂载使用
mkdir -p /mnt/secure
mount /dev/mapper/secure_volume /mnt/secure

# 使用完毕后关闭
umount /mnt/secure
cryptsetup luksClose secure_volume
losetup -d /dev/loop0
```

```bash
# 加密现有分区
# 警告：会丢失现有数据，需要先备份

# 1. 备份数据
cp -a /data /backup/data

# 2. 格式化加密分区
cryptsetup luksFormat /dev/sdb1

# 3. 打开加密分区
cryptsetup luksOpen /dev/sdb1 encrypted_data

# 4. 创建文件系统
mkfs.ext4 /dev/mapper/encrypted_data

# 5. 恢复数据
mount /dev/mapper/encrypted_data /data
cp -a /backup/data/* /data/

# 6. 配置自动解锁（使用密钥文件）
dd if=/dev/urandom of=/root/luks-keyfile bs=512 count=1
chmod 600 /root/luks-keyfile
cryptsetup luksAddKey /dev/sdb1 /root/luks-keyfile

# 7. 配置自动挂载（/etc/crypttab和/etc/fstab）
# /etc/crypttab
# encrypted_data /dev/sdb1 /root/luks-keyfile luks

# /etc/fstab
# /dev/mapper/encrypted_data /data ext4 defaults 0 0
```

## 3.10 文件完整性检查

### 3.10.1 AIDE配置

AIDE（Advanced Intrusion Detection Environment）是一个文件完整性检查工具。它通过建立文件系统的"快照"（数据库），然后定期比较当前状态与快照的差异来检测未授权的文件变更。

```bash

aide --init
# 通常需要几分钟时间扫描文件系统

# 重命名初始数据库（使其可写）
mv /var/lib/aide/aide.db.new /var/lib/aide/aide.db

# 首次检查
aide --check

# 更新数据库（确认正常变更后）
aide --update
mv /var/lib/aide/aide.db.new /var/lib/aide/aide.db
```

```bash

# n: number of links
```

```bash

#!/bin/bash
LOGFILE="/var/log/aide check.log"

if [ $RESULT -eq 0 ]; then
    echo "检查完成：无异常发现" >> $LOGFILE
    echo "No changes found - AIDE check passed"
elif [ $RESULT -eq 1 ]; then

else
    echo "AIDE检查失败" >> $LOGFILE
fi

exit $RESULT

# 2. 设置定时检查
# 编辑 /etc/cron.d/aide
# 每天凌晨3点执行检查
# 0 3 * * * root /usr/sbin/aide --check | /usr/bin/mail -s "AIDE Report" root@localhost

# 3. 将数据库和配置文件备份到安全位置
```

## 3.11 实战案例：敏感文件泄露的处理

### 3.11.1 发现泄露

敏感文件泄露可能通过多种途径发生：代码仓库意外提交、日志文件暴露、备份文件泄露等。

```bash
# 1. 检查Git仓库中的敏感文件
# 在仓库根目录执行
git log --all --full-history --source --remotes -- .env | head -20

# 使用git-secrets或similar工具扫描
git diff HEAD~10..HEAD --name-only | grep -E '\.pem|\.key|\.env|config.*prod|secret'

# 查找可能包含敏感信息的提交
git log --all -p --full-history -- "**/*.env"
git log --all -p --full-history -- "**/secrets.yml"
git log --all -p --full-history -- "*password*"

# 2. 检查GitHub/GitLab等平台的泄露告警
# GitHub Secret Scanning告警

# 3. 本地检查敏感文件
find /home -name ".env" -o -name "*.pem" -o -name "*.key" -o -name "*.sql" 2>/dev/null

# 4. Web服务器泄露检测
grep -r "password" /var/www/html/ --include="*.php" --include="*.html" 2>/dev/null | grep -v ".git"

# 5. 检查备份文件
find / -name "*.bak" -o -name "*.backup" -o -name "*.old" -o -name "*.sql" 2>/dev/null | grep -v proc
```

### 3.11.2 评估影响范围

一旦发现泄露，需要立即评估影响范围，确定泄露的信息可能被谁获取，以及可能被如何利用。

```bash
# 1. 分析泄露的内容类型
# 如果是密码/密钥
# - 确定是哪个系统/服务的凭证
# - 检查密码是否还在其他系统使用

# 如果是数据库备份
# - 确定备份中包含哪些数据
# - 检查数据是否涉及用户个人信息

# 如果是私钥/证书
# - 确定私钥对应的公钥/证书用途
# - 评估使用该私钥可以做什么

# 2. 检查访问日志
# 如果通过Web泄露
grep "sensitive_file" /var/log/nginx/access.log
grep "sensitive_file" /var/log/apache2/access.log

# 查看是否有异常IP在泄露前后的访问
tail -1000 /var/log/auth.log | grep IP_FROM_FINDING

# 3. 检查代码提交历史
git log --all --oneline --source --remotes | head -100
git log --all --oneline -p | grep -C5 "SENSITIVE_DATA"
```

### 3.11.3 紧急处置

```bash
# 1. 如果是Git仓库泄露
# 删除敏感文件的所有历史
git filter-branch --force --index-filter \
    'git rm --cached --ignore-unmatch path/to/sensitive/file' \
    --prune-empty --tag-name-filter cat -- --all

# 推送清理后的历史
git push origin --force --all
git push origin --force --tags

# 警告：这不会清除已经克隆了仓库的人的本地副本

# 2. 如果在GitHub上
# 立即删除仓库或使private
# 联系GitHub Support请求清除缓存

# 3. 轮换所有可能泄露的凭证
# 生成新密码
openssl rand -base64 16

# 为所有受影响的账户重置密码
# 数据库连接密码
# API密钥
# SSH密钥
# TLS证书

# 4. 吊销并重新颁发证书
# 如果私钥泄露
openssl req -new -newkey rsa:4096 -nodes -keyout new.key -out new.csr
# 提交CSR给证书颁发机构
```

### 3.11.4 后续改进

```bash
# 1. 实施Git hooks防止未来泄露
# 安装pre-commit钩子
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/bash
# 扫描敏感信息
echo "Running sensitive data check..."

for file in $(git diff --cached --name-only | grep -E '\.(env|key|pem|sql|json|xml|yml|yaml)$'); do
    if grep -qiE 'password|secret|key|token|api_key' "$file"; then
        echo "WARNING: $file may contain sensitive data"
        read -p "Commit anyway? (y/n) " -n 1 -r
        echo
        if [[ ! $REPLY =~ ^[Yy]$ ]]; then
            exit 1
        fi
    fi
done
```

## 3.12 小结

本章系统地介绍了Linux文件系统安全的各个方面。文件权限是Linux安全模型的基础，通过chmod、chown和chgrp命令可以管理文件和目录的访问权限。特殊权限SUID、SGID和Sticky Bit提供了额外的访问控制机制，但也带来了潜在的安全风险，需要谨慎使用和管理。

ACL访问控制列表扩展了传统的三权模型，允许更精细的权限控制。文件属性（chattr/lsattr）提供了一层额外的保护，特别是不可变属性（i）和只可追加属性（a），可以有效防止重要文件被篡改或删除。

敏感文件的保护是系统安全的重中之重。/etc/passwd、/etc/shadow、/etc/sudoers等文件都有其标准的安全权限配置，任何偏离都可能导致安全风险。

合理的磁盘分区方案和正确的挂载选项是纵深防御的重要组成部分。通过使用noexec、nosuid、nodev等挂载选项，可以显著降低系统被攻击的风险。

对于需要保护的敏感数据，LUKS等加密文件系统提供了透明的全盘加密功能。文件完整性监控工具如AIDE和Tripwire则可以帮助及时发现文件的未授权变更。

## 3.14 文件权限深度解析

### 3.14.1 数字权限与符号权限详解

Linux文件权限可以用数字或符号两种方式表示，理解它们的转换关系对于权限管理至关重要。

**数字权限详解**

数字权限使用三位八进制数表示，每一位对应一类用户（所有者、组、其他），每一位的值由读(4)、写(2)、执行(1)三个权限值相加得出。

```
权限计算示例：
rwx = 4+2+1 = 7  (读+写+执行)
rw- = 4+2+0 = 6  (读+写)
r-x = 4+0+1 = 5  (读+执行)
r-- = 4+0+0 = 4  (只读)
-wx = 0+2+1 = 3  (写+执行，较少使用)
-w- = 0+2+0 = 2  (只写，较少使用)
--x = 0+0+1 = 1  (只执行)
--- = 0+0+0 = 0  (无权限)

常见权限组合：
755 = rwxr-xr-x  所有者完全权限，组和其他用户可读可执行
644 = rw-r--r--  所有者可读写，组和其他用户只读
600 = rw-------  只有所有者可读写
700 = rwx------  只有所有者完全权限
775 = rwxrwxr-x  所有者和组完全权限，其他用户可读可执行
```

**符号权限详解**

符号权限使用字母表示权限类型，用+、-、=操作符来修改权限。

```bash
# 符号权限格式：ugoa[+-=]rwxXst

# u = user（所有者）
# g = group（组）
# o = other（其他用户）
# a = all（所有用户，默认）

# + = 添加权限
# - = 移除权限
# = = 设置精确权限

# r = read（读）
# w = write（写）
# x = execute（执行）
# X = 仅对目录或已有执行权限的文件设置执行权限
# s = SUID/SGID
# t = Sticky Bit

# 示例
chmod u+x file        # 给所有者添加执行权限
chmod g-w file        # 移除组的写权限
chmod o=r file        # 设置其他用户只读权限
chmod a+x file        # 给所有用户添加执行权限
chmod u=rwx,g=rx,o=r file  # 设置精确权限755
chmod +x file         # 默认对所有用户添加执行权限
```

### 3.14.2 umask默认权限详解

umask决定了新创建文件和目录的默认权限。理解umask的工作原理对于控制新文件的安全性非常重要。

```bash
# umask工作原理
# 文件默认权限 = 666 - umask
# 目录默认权限 = 777 - umask

# 常见umask值
umask 022  # 文件644，目录755（默认值）
umask 077  # 文件600，目录700（更安全）
umask 002  # 文件664，目录775（组协作）

# 查看当前umask
umask

# 设置umask（临时）
umask 077

# 永久设置umask
# 编辑 ~/.bashrc 或 /etc/profile
echo "umask 077" >> ~/.bashrc

# 系统级设置
# 编辑 /etc/login.defs
UMASK 077

# 创建测试文件验证
touch test_file
ls -l test_file  # 应显示600权限

mkdir test_dir
ls -ld test_dir  # 应显示700权限
```

### 3.14.3 SUID/SGID/Sticky Bit详解

这三种特殊权限在系统安全中扮演重要角色，但也可能带来安全风险。

**SUID（Set User ID）**

当可执行文件设置了SUID权限，执行该文件的用户将临时获得文件所有者的权限。

```bash
# SUID示例 - passwd命令
ls -l /usr/bin/passwd
# -rwsr-xr-x 1 root root ... passwd
# 注意's'代替了'x'，表示SUID已设置

# 查找所有SUID文件
find / -perm -4000 -type f 2>/dev/null

# 查找可疑的SUID文件（不在标准目录）
find / -perm -4000 -type f 2>/dev/null | grep -v -E "^/usr|^/bin|^/sbin|^/lib"

# SUID安全风险：
# 1. 如果SUID程序有漏洞，攻击者可以获得root权限
# 2. 自定义SUID程序必须非常谨慎编写
# 3. 定期审计SUID文件列表

# 移除不必要的SUID
chmod u-s /path/to/file
```

**SGID（Set Group ID）**

对于可执行文件，SGID使执行者临时获得文件所属组的权限。对于目录，SGID使在该目录中创建的文件继承目录的组。

```bash
# SGID在目录上的应用
mkdir /shared/project
chgrp developers /shared/project
chmod g+s /shared/project
# 在此目录创建的文件将自动属于developers组

# 查找SGID文件和目录
find / -perm -2000 2>/dev/null

# SGID示例 - locate命令
ls -l /usr/bin/locate
# -rwx--s--x 1 root locate ...
```

**Sticky Bit**

Sticky Bit主要用于共享目录，防止用户删除其他用户的文件。

```bash
# Sticky Bit示例 - /tmp目录
ls -ld /tmp
# drwxrwxrwt ... tmp
# 注意't'代替了'x'，表示Sticky Bit已设置

# 设置Sticky Bit
chmod +t /shared/uploads

# 查找Sticky Bit目录
find / -perm -1000 -type d 2>/dev/null

# Sticky Bit的作用：
# 在设置了Sticky Bit的目录中，只有文件所有者、目录所有者或root可以删除文件
# 即使其他用户有目录写权限，也不能删除其他用户的文件
```

## 3.15 ACL访问控制列表详解

### 3.15.1 ACL基础概念

ACL（Access Control List）是对传统Unix权限模型的扩展，允许为特定用户或组设置更精细的权限。

```bash
# 检查ACL支持
# 查看文件系统是否支持ACL
tune2fs -l /dev/sda1 | grep "Default mount options"
# 或
cat /etc/fstab | grep acl

# 安装ACL工具
apt-get install acl     # Debian/Ubuntu
yum install acl         # CentOS/RHEL

# 查看ACL
getfacl filename

# 设置ACL
setfacl -m u:username:rwx filename    # 给用户设置权限
setfacl -m g:groupname:rx filename    # 给组设置权限
setfacl -m o::r filename              # 设置其他用户权限

# 删除ACL
setfacl -x u:username filename        # 删除用户的ACL
setfacl -b filename                   # 删除所有ACL
```

### 3.15.2 ACL继承与默认ACL

默认ACL用于目录，设置后该目录中新创建的文件将继承这些ACL。

```bash
# 设置默认ACL（目录）
setfacl -d -m u:developer:rwx /shared/project

# 查看默认ACL
getfacl /shared/project
# 输出包含"default:"前缀的条目

# 默认ACL示例
mkdir /shared/team
setfacl -d -m u:alice:rwx /shared/team
setfacl -d -m u:bob:rx /shared/team
setfacl -d -m g:team:rwx /shared/team

# 在目录中创建文件测试
touch /shared/team/test.txt
getfacl /shared/team/test.txt
# 文件将继承目录的默认ACL
```

### 3.15.3 ACL与传统权限的关系

```bash
# ACL权限优先级
# 1. 如果用户是文件所有者，应用所有者权限
# 2. 如果用户在ACL用户列表中，应用对应ACL权限
# 3. 如果用户所属组在ACL组列表中，应用对应ACL组权限
# 4. 应用其他用户权限

# ACL mask限制
# mask是ACL权限的上限，所有ACL权限都不能超过mask
setfacl -m m::rx filename    # 设置mask为rx

# 查看mask
getfacl filename
# mask::rx 表示所有ACL权限上限为rx

# 有效权限计算
# 用户有效权限 = ACL权限 AND mask
# 例如：用户ACL为rwx，mask为rx，则有效权限为rx
```

## 3.16 文件系统加密详解

### 3.16.1 LUKS磁盘加密

LUKS（Linux Unified Key Setup）是Linux标准的磁盘加密方案。

```bash
# 安装LUKS工具
apt-get install cryptsetup    # Debian/Ubuntu
yum install cryptsetup        # CentOS/RHEL

# 创建加密分区
cryptsetup luksFormat /dev/sdb1
# 输入密码确认

# 打开加密分区
cryptsetup luksOpen /dev/sdb1 encrypted_disk
# 输入密码

# 创建文件系统
mkfs.ext4 /dev/mapper/encrypted_disk

# 挂载
mount /dev/mapper/encrypted_disk /mnt/encrypted

# 关闭加密分区
umount /mnt/encrypted
cryptsetup luksClose encrypted_disk

# 添加密钥
cryptsetup luksAddKey /dev/sdb1
# 可以添加多个密钥

# 查看加密信息
cryptsetup luksDump /dev/sdb1
```

### 3.16.2 eCryptfs加密目录

eCryptfs是堆叠式加密文件系统，可以加密单个目录。

```bash
# 安装eCryptfs
apt-get install ecryptfs-utils

# 加密目录
mount -t ecryptfs /home/user/private /home/user/private
# 馶次挂载时需要设置密码和选项

# 查看挂载
mount | grep ecryptfs

# 自动挂载配置
# 编辑 /etc/fstab
/home/user/private /home/user/private ecryptfs defaults,noauto 0 0

# 使用ecryptfs-setup-private创建加密私有目录
ecryptfs-setup-private
```

### 3.17.2 文件哈希校验脚本

```bash
#!/bin/bash
# 文件哈希校验脚本
# 保存为 /usr/local/bin/file_hash_check.sh

HASH_DIR="/var/lib/file_hashes"
CRITICAL_FILES=(
    "/etc/passwd"
    "/etc/shadow"
    "/etc/group"
    "/etc/sudoers"
    "/etc/ssh/sshd_config"
    "/etc/ssh/ssh_config"
    "/etc/crontab"
    "/etc/fstab"
    "/bin/ls"
    "/bin/ps"
    "/usr/bin/top"
    "/usr/bin/netstat"
    "/usr/sbin/sshd"
)

mkdir -p $HASH_DIR

echo "=== 文件哈希校验 ==="
echo "时间: $(date)"

for file in "${CRITICAL_FILES[@]}"; do
    if [ -f "$file" ]; then
        current_hash=$(sha256sum "$file" | awk '{print $1}')
        stored_hash_file="$HASH_DIR/$(echo $file | tr '/' '_').hash"
        
        if [ -f "$stored_hash_file" ]; then
            stored_hash=$(cat "$stored_hash_file")
            if [ "$current_hash" = "$stored_hash" ]; then
                echo "[OK] $file"
            else
                echo "[警告] $file 哈希已改变!"
                echo "  旧哈希: $stored_hash"
                echo "  新哈希: $current_hash"
            fi
        else
            echo "[新建] $file"
            echo "$current_hash" > "$stored_hash_file"
        fi
    else
        echo "[缺失] $file 不存在"
    fi
done

echo ""
echo "=== 检查完成 ==="
```
