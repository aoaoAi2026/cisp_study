## 10.10 MySQL高级安全配置

### 10.10.1 my.cnf完整安全配置详解

MySQL的安全配置是数据库防护的第一道防线。一个完善的my.cnf配置文件应该涵盖网络安全、认证安全、数据安全、日志审计和资源限制等多个维度。在生产环境中，绝不能使用默认配置，必须根据业务场景和安全要求进行深度定制。

安全配置的核心原则包括：最小化服务暴露面、强制安全认证、禁用危险功能、开启全面日志、限制资源滥用。以下是一个生产环境级别的完整安全配置示例，每个配置项都附带详细说明。

```ini
# /etc/my.cnf - 生产环境安全配置完整示例

[mysqld]
# ===== 基础安全设置 =====
user = mysql
port = 3306
basedir = /usr
datadir = /var/lib/mysql
socket = /var/lib/mysql/mysql.sock
pid-file = /var/run/mysqld/mysqld.pid

# ===== 网络安全配置 =====
# 只监听本地回环地址，如需远程访问请指定具体IP
bind-address = 127.0.0.1
# bind-address = 192.168.1.100

# 禁用DNS解析，提高连接速度同时防止主机名欺骗
skip-name-resolve

# 禁用符号链接，防止目录遍历攻击
symbolic-links = 0

# 禁用LOCAL INFILE，防止任意文件读取
local-infile = 0
loose-local-infile = 0

# ===== 认证安全配置 =====
# 启用密码验证插件
default-authentication-plugin = mysql_native_password

# 密码策略（需安装validate_password插件）
validate-password = FORCE_PLUS_PERMANENT
validate_password_length = 12
validate_password_mixed_case_count = 1
validate_password_number_count = 1
validate_password_special_char_count = 1
validate_password_policy = MEDIUM

# 登录失败锁定（需安装connection_control插件）

connection-control-failed-login-connections = FORCE_PLUS_PERMANENT
connection_control_failed_connections_threshold = 5
connection_control_min_connection_delay = 1000
connection_control_max_connection_delay = 86400

connect_timeout = 10
net_read_timeout = 30
net_write_timeout = 60

# 单用户最大连接数限制
max_user_connections = 50

# 通过权限管理控制，不在配置文件中全局禁用

# 限制LOAD DATA权限
```

### 10.10.2 SSL/TLS加密连接深度配置

SSL/TLS加密是保护数据库数据在网络传输中不被窃听和篡改的关键措施。在生产环境中，所有跨网络的数据库连接都应该使用加密传输。MySQL支持TLS 1.0到TLS 1.3多个版本，建议使用TLS 1.2或更高版本。

完整的SSL/TLS配置不仅包括服务器端证书配置，还涉及客户端证书验证、证书吊销、密码套件选择等多个方面。以下是详细的配置步骤和最佳实践。

```bash
# ===== 第一步：创建CA证书和服务器证书 =====

# 创建SSL目录
mkdir -p /etc/mysql/ssl
cd /etc/mysql/ssl

# 1. 生成CA私钥和自签名证书
openssl genrsa 4096 > ca-key.pem
openssl req -new -x509 -nodes -days 3650 \
    -key ca-key.pem \
    -out ca.pem \
    -subj "/C=CN/ST=Beijing/L=Beijing/O=DBCA/CN=MySQL CA"

# 2. 生成服务器私钥和证书签名请求(CSR)
openssl req -newkey rsa:2048 -days 730 -nodes \
    -keyout server-key.pem \
    -out server-req.pem \
    -subj "/C=CN/ST=Beijing/L=Beijing/O=DBCA/CN=mysql-server.example.com"

# 3. 使用CA签署服务器证书
openssl x509 -req -in server-req.pem \
    -days 730 \
    -CA ca.pem \
    -CAkey ca-key.pem \
    -set_serial 01 \
    -out server-cert.pem

# 4. 生成客户端证书（用于双向认证）
openssl req -newkey rsa:2048 -days 730 -nodes \
    -keyout client-key.pem \
    -out client-req.pem \
    -subj "/C=CN/ST=Beijing/L=Beijing/O=DBCA/CN=mysql-client"

openssl x509 -req -in client-req.pem \
    -days 730 \
    -CA ca.pem \
    -CAkey ca-key.pem \
    -set_serial 02 \
    -out client-cert.pem

# 5. 设置正确的文件权限
chown -R mysql:mysql /etc/mysql/ssl
chmod 600 /etc/mysql/ssl/*-key.pem
chmod 644 /etc/mysql/ssl/ca.pem /etc/mysql/ssl/*-cert.pem

# ===== 第二步：配置MySQL服务器 =====

# 编辑my.cnf添加SSL配置
cat >> /etc/my.cnf << 'EOF'
[mysqld]
# 强制所有连接使用SSL
require_secure_transport = ON

# SSL证书路径
ssl_ca = /etc/mysql/ssl/ca.pem
ssl_cert = /etc/mysql/ssl/server-cert.pem
ssl_key = /etc/mysql/ssl/server-key.pem

# TLS版本限制（禁用旧版本）
tls_version = TLSv1.2,TLSv1.3

# 加密套件（可选，根据安全需求调整）
# ssl_cipher = DHE-RSA-AES256-GCM-SHA384:DHE-RSA-AES128-GCM-SHA256:ECDHE-RSA-AES256-GCM-SHA384
EOF

# 重启MySQL
systemctl restart mysqld

# ===== 第三步：验证SSL配置 =====

# 检查SSL是否启用
mysql -u root -p -e "SHOW VARIABLES LIKE '%ssl%';"

# 查看当前连接是否使用SSL
mysql -u root -p -e "SHOW STATUS LIKE 'Ssl_cipher';"
mysql -u root -p -e "SHOW STATUS LIKE 'Ssl_version';"

# ===== 第四步：配置用户SSL要求 =====

# 强制用户使用SSL连接
mysql -u root -p << 'EOF'
-- 强制用户必须使用SSL
ALTER USER 'app_user'@'%' REQUIRE SSL;

-- 要求使用特定的客户端证书（双向认证）
ALTER USER 'app_user'@'%' REQUIRE SUBJECT '/CN=mysql-client';

-- 要求客户端证书由特定CA签发
ALTER USER 'app_user'@'%' REQUIRE ISSUER '/CN=MySQL CA';

-- 同时要求SSL和客户端证书
ALTER USER 'app_user'@'%' REQUIRE X509;

-- 组合使用多种认证要求
ALTER USER 'app_user'@'%' 
REQUIRE SUBJECT '/CN=mysql-client'
AND ISSUER '/CN=MySQL CA'
AND CIPHER 'DHE-RSA-AES256-GCM-SHA384';

FLUSH PRIVILEGES;
EOF

# ===== 第五步：客户端SSL连接测试 =====

# 使用SSL连接（单向认证）
mysql -u app_user -p \
    -h 127.0.0.1 \
    --ssl-ca=/etc/mysql/ssl/ca.pem \
    --ssl-mode=REQUIRED

# 使用客户端证书（双向认证）
mysql -u app_user -p \
    -h 127.0.0.1 \
    --ssl-ca=/etc/mysql/ssl/ca.pem \
    --ssl-cert=/etc/mysql/ssl/client-cert.pem \
    --ssl-key=/etc/mysql/ssl/client-key.pem \
    --ssl-mode=VERIFY_CA

# 验证连接状态
mysql -u app_user -p -e "\s" | grep -i ssl
```

### 10.10.3 MySQL审计日志高级配置

数据库审计是安全合规的重要组成部分，也是事后追溯的关键依据。完整的审计日志应该记录：登录事件（成功/失败）、权限变更、数据操作（增删改查）、DDL操作、数据库启停等关键事件。

MySQL社区版没有内置审计功能，但可以通过第三方插件（如MariaDB Audit Plugin、Percona Audit Plugin）实现。MySQL企业版提供了更强大的MySQL Enterprise Audit功能。以下是几种常见的审计方案配置。

```bash
# ===== 方案一：MySQL Enterprise Audit（企业版） =====

# 安装审计插件
mysql -u root -p << 'EOF'
INSTALL PLUGIN audit_log SONAME 'audit_log.so';
EOF

# 配置审计日志
cat >> /etc/my.cnf << 'EOF'
[mysqld]
# 启用审计日志
plugin-load-add = audit_log.so
audit_log = FORCE_PLUS_PERMANENT

# 审计日志文件
audit_log_file = /var/log/mysql/audit.log

# 审计格式：JSON / NEWJSON / XML / OLDXML
audit_log_format = NEWJSON

# 审计策略：ALL / NONE / LOGINS / QUERIES / LOGINS_AND_QUERIES
audit_log_policy = ALL

# 连接审计策略：ALL / FAILURES / SUCCESSES
audit_log_connection_policy = ALL

# 语句审计策略：ALL / DDL / DML / DCL / NONE
audit_log_statement_policy = ALL

# 日志轮转
audit_log_rotate_on_size = 100M
audit_log_rotations = 30

# 日志缓冲
audit_log_buffer_size = 4M
audit_log_flush = ON

# 加密审计日志（可选）
# audit_log_encryption = AES
# audit_log_encryption_key = /etc/mysql/audit.key
EOF

# ===== 方案二：MariaDB Audit Plugin =====

# 安装MariaDB审计插件
mysql -u root -p << 'EOF'
INSTALL PLUGIN server_audit SONAME 'server_audit.so';
EOF

# 配置审计
cat >> /etc/my.cnf << 'EOF'
[mysqld]
plugin-load-add = server_audit.so
server_audit_logging = ON
server_audit_events = 'CONNECT,QUERY,TABLE,QUERY_DDL,QUERY_DML'
server_audit_file_path = /var/log/mysql/audit/
server_audit_file_rotate_size = 100M
server_audit_file_rotations = 30
server_audit_excl_users = 'replication_user'
server_audit_incl_users = ''
server_audit_output_type = FILE
server_audit_query_log_limit = 1024
EOF

# ===== 方案三：Percona Audit Log Plugin =====

# 安装Percona审计插件
mysql -u root -p << 'EOF'
INSTALL PLUGIN audit_log SONAME 'audit_log.so';
EOF

# 配置
cat >> /etc/my.cnf << 'EOF'
[mysqld]
plugin-load-add = audit_log.so
audit_log_file = /var/log/mysql/audit.log
audit_log_policy = LOGINS_AND_QUERIES
audit_log_format = JSON
audit_log_rotate = ON
audit_log_rotations = 30
audit_log_rotate_on_size = 100M
audit_log_buffer_size = 4M
audit_log_strategy = ASYNCHRONOUS
audit_log_exclude_accounts = 'app_user@localhost'
audit_log_include_accounts = ''
EOF

# ===== 审计日志管理 =====

# 创建日志目录
mkdir -p /var/log/mysql/audit
chown mysql:mysql /var/log/mysql/audit
chmod 750 /var/log/mysql/audit

# 配置日志轮转（logrotate）
cat > /etc/logrotate.d/mysql-audit << 'EOF'
/var/log/mysql/audit*.log {
    daily
    rotate 30
    compress
    delaycompress
    missingok
    notifempty
    create 640 mysql mysql
    postrotate
        if systemctl is-active --quiet mysqld; then
            mysql -u root -p'YourPassword' -e "SET GLOBAL audit_log_rotate = ON;" 2>/dev/null || true
        fi
    endscript
}
EOF

# ===== 审计日志分析 =====

# 查看登录失败记录
grep -i "failed" /var/log/mysql/audit.log | head -20

# 查看DDL操作
grep -i "ALTER\|CREATE\|DROP\|TRUNCATE" /var/log/mysql/audit.log

# 查看权限变更
grep -i "GRANT\|REVOKE" /var/log/mysql/audit.log

# 统计登录IP分布
grep "CONNECT" /var/log/mysql/audit.log | awk '{print $NF}' | sort | uniq -c | sort -rn

# 使用MySQL查看审计状态
mysql -u root -p -e "SHOW STATUS LIKE 'audit%';"
mysql -u root -p -e "SHOW VARIABLES LIKE 'audit%';"
```

## 10.11 PostgreSQL安全加固进阶

### 10.11.1 PostgreSQL安装初始化安全

PostgreSQL以其强大的安全性、丰富的功能和出色的扩展性著称。与MySQL相比，PostgreSQL默认配置更加安全，但仍然需要进行一系列安全加固才能用于生产环境。

安装初始化阶段的安全配置至关重要，包括：选择安全的安装源、禁用不必要的模块、设置强密码、配置数据目录权限等。以下是完整的安全初始化流程。

```bash
# ===== 第一步：安装PostgreSQL =====

# CentOS/RHEL - 使用官方仓库
yum install -y https://download.postgresql.org/pub/repos/yum/reporpms/EL-7-x86_64/pgdg-redhat-repo-latest.noarch.rpm
yum install -y postgresql14-server postgresql14-contrib

# Debian/Ubuntu - 使用官方仓库
# sh -c 'echo "deb http://apt.postgresql.org/pub/repos/apt $(lsb_release -cs)-pgdg main" > /etc/apt/sources.list.d/pgdg.list'
# wget --quiet -O - https://www.postgresql.org/media/keys/ACCC4CF8.asc | apt-key add -
# apt-get update
# apt-get install -y postgresql-14 postgresql-contrib

# ===== 第二步：初始化数据库集群 =====

# CentOS/RHEL
/usr/pgsql-14/bin/postgresql-14-setup initdb

# 或手动初始化
su - postgres
/usr/pgsql-14/bin/initdb -D /var/lib/pgsql/14/data \
    --auth-local=peer \
    --auth-host=scram-sha-256 \
    --encoding=UTF8 \
    --locale=en_US.UTF-8

# ===== 第三步：配置数据目录权限 =====

# 确保数据目录权限安全
chmod 700 /var/lib/pgsql/14/data
chown -R postgres:postgres /var/lib/pgsql/14/data

# ===== 第四步：设置postgres用户密码 =====

# 切换到postgres用户
su - postgres

# 设置数据库密码
psql -c "ALTER USER postgres WITH PASSWORD 'StrongPostgresP@ss2024!';"

# ===== 第五步：安装安全相关扩展 =====

psql << 'EOF'
-- 安装密码强度检查扩展
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- 安装uuid-ossp（用于生成安全UUID）
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- 安装pg_stat_statements（用于监控）
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- 安装auto_explain（用于分析慢查询）
LOAD 'auto_explain';
EOF

# ===== 第六步：配置文件权限 =====

# 配置文件权限
chmod 600 /var/lib/pgsql/14/data/postgresql.conf
chmod 600 /var/lib/pgsql/14/data/pg_hba.conf
chmod 600 /var/lib/pgsql/14/data/pg_ident.conf
chown postgres:postgres /var/lib/pgsql/14/data/*.conf

# ===== 第七步：启用并启动服务 =====

systemctl enable postgresql-14
systemctl start postgresql-14

# 验证服务状态
systemctl status postgresql-14
su - postgres -c "psql -c 'SELECT version();'"
```

### 10.11.2 pg_hba.conf高级配置

pg_hba.conf（Host-Based Authentication）是PostgreSQL的访问控制配置文件，决定了哪些用户可以从哪些IP地址连接到哪个数据库，以及使用什么认证方式。正确配置pg_hba.conf是PostgreSQL安全的重中之重。

配置原则：默认拒绝、最小权限、分层授权。配置文件的读取顺序是从上到下，第一条匹配的规则生效，因此更严格的规则应该放在前面。

```bash
# /var/lib/pgsql/14/data/pg_hba.conf
# ==============================================================
# PostgreSQL Client Authentication Configuration File
# ==============================================================
# TYPE  DATABASE        USER            ADDRESS                 METHOD

# ===== 本地连接配置 =====

# 本地Unix套接字连接 - 超级用户使用peer认证
local   all             postgres                                peer

# 本地Unix套接字连接 - 普通用户使用peer认证
local   all             all                                     peer

# ===== 本地回环IPv4连接 =====

# 本地回环 - 所有数据库使用scram-sha-256认证
host    all             all             127.0.0.1/32            scram-sha-256

# ===== 本地回环IPv6连接 =====
host    all             all             ::1/128                 scram-sha-256

# ===== 复制专用连接 =====

# 复制连接 - 仅限内网
host    replication     replicator      192.168.1.10/32         scram-sha-256
host    replication     replicator      192.168.1.11/32         scram-sha-256

# ===== 应用用户连接 =====

# 主应用数据库 - 应用服务器网段
host    app_db          app_user        192.168.1.0/24          scram-sha-256

# 只读用户 - 报表服务器
host    app_db          app_reader      192.168.2.50/32         scram-sha-256

# ===== 管理员连接 =====

# 数据库管理员 - 仅限管理终端
host    all             db_admin        192.168.1.100/32        scram-sha-256

# ===== SSL强制连接 =====

# 所有外部连接必须使用SSL
hostssl all             all             10.0.0.0/8              scram-sha-256

# ===== 拒绝规则（放在最后） =====

# 拒绝所有其他连接
host    all             all             0.0.0.0/0               reject
host    all             all             ::/0                    reject

# ==============================================================
# 认证方法说明：
# ==============================================================
# trust          - 无条件允许连接（不安全，仅限测试）
# peer           - 使用操作系统用户名认证（本地连接）
# ident          - 使用Ident协议认证（TCP连接）
# scram-sha-256  - SCRAM-SHA-256密码认证（推荐，最安全）
# md5            - MD5密码认证（旧版，不推荐）
# password       - 明文密码认证（极不安全，不推荐）
# gss            - GSSAPI认证（Kerberos）
# sspi           - SSPI认证（Windows）
# ldap           - LDAP认证
# radius         - RADIUS认证
# cert           - 客户端SSL证书认证
# pam            - PAM认证
# reject         - 拒绝连接
# ==============================================================
```

### 10.11.3 PostgreSQL权限管理进阶

PostgreSQL的权限系统非常强大和灵活，支持细粒度的权限控制。理解角色（ROLE）、权限（PRIVILEGE）、模式（SCHEMA）之间的关系是做好权限管理的基础。

权限管理的最佳实践：使用角色组管理权限、遵循最小权限原则、定期审计权限、使用默认权限简化管理、分离读写权限。

```bash
# ===== 角色与权限管理 =====

su - postgres
psql

-- ===== 创建角色组 =====

-- 创建只读角色组
CREATE ROLE read_only_group NOINHERIT;

-- 创建读写角色组
CREATE ROLE read_write_group NOINHERIT;

-- 创建管理员角色组
CREATE ROLE db_admin_group NOINHERIT;

-- ===== 授予角色组权限 =====

-- 授予连接权限
```

### 10.11.4 PostgreSQL SSL配置详解

PostgreSQL的SSL配置比MySQL更加灵活，支持多种SSL模式和证书验证级别。在生产环境中，强烈建议启用SSL并使用双向认证（客户端证书验证）。

PostgreSQL支持的SSL模式包括：disable、allow、prefer、require、verify-ca、verify-full，安全性依次递增。对于生产环境，建议使用verify-full模式。

```bash
# ===== 第一步：生成SSL证书 =====

su - postgres
cd /var/lib/pgsql/14/data

# 创建证书目录
mkdir -p ssl
cd ssl

# 1. 生成CA私钥和证书
openssl genrsa -out rootCA.key 4096
openssl req -x509 -new -nodes \
    -key rootCA.key \
    -sha256 \
    -days 3650 \
    -out rootCA.crt \
    -subj "/C=CN/ST=Beijing/L=Beijing/O=DBCA/CN=PostgreSQL CA"

# 2. 生成服务器私钥
openssl genrsa -out server.key 2048

# 3. 生成服务器CSR
openssl req -new \
    -key server.key \
    -out server.csr \
    -subj "/C=CN/ST=Beijing/L=Beijing/O=DBCA/CN=pgsql.example.com"

# 4. 使用CA签署服务器证书
openssl x509 -req \
    -in server.csr \
    -CA rootCA.crt \
    -CAkey rootCA.key \
    -CAcreateserial \
    -out server.crt \
    -days 730 \
    -sha256

# 5. 生成客户端证书（双向认证用）
openssl genrsa -out client.key 2048
openssl req -new \
    -key client.key \
    -out client.csr \
    -subj "/C=CN/ST=Beijing/L=Beijing/O=DBCA/CN=dbadmin-client"

openssl x509 -req \
    -in client.csr \
    -CA rootCA.crt \
    -CAkey rootCA.key \
    -CAcreateserial \
    -out client.crt \
    -days 730 \
    -sha256

# 6. 设置正确权限
chmod 600 server.key client.key rootCA.key
chmod 644 server.crt client.crt rootCA.crt
chown postgres:postgres *

# ===== 第二步：配置postgresql.conf =====

cat >> /var/lib/pgsql/14/data/postgresql.conf << 'EOF'
# ===== SSL配置 =====
ssl = on
ssl_cert_file = 'ssl/server.crt'
ssl_key_file = 'ssl/server.key'
ssl_ca_file = 'ssl/rootCA.crt'

# TLS版本（PostgreSQL 12+支持配置）
# ssl_min_protocol_version = 'TLSv1.2'
# ssl_max_protocol_version = 'TLSv1.3'

# 密码套件
# ssl_ciphers = 'HIGH:!aNULL:!MD5:!3DES'

# 证书吊销列表（可选）
# ssl_crl_file = 'ssl/root.crl'

# 预共享密钥（可选，用于高安全场景）
# ssl_prefer_server_ciphers = on
EOF

# ===== 第三步：配置pg_hba.conf使用SSL =====

# 修改pg_hba.conf，将远程连接改为hostssl
cat >> /var/lib/pgsql/14/data/pg_hba.conf << 'EOF'
# 强制SSL连接（hostssl只接受SSL连接）
hostssl  all          all           192.168.1.0/24        scram-sha-256

# 使用客户端证书认证（cert模式）
# hostssl  all          db_admin      192.168.1.100/32      cert clientcert=1
EOF

# ===== 第四步：重启服务并验证 =====

systemctl restart postgresql-14

# 验证SSL是否启用
su - postgres -c "psql -c 'SHOW ssl;'"
su - postgres -c "psql -c 'SELECT version();'"

# ===== 第五步：客户端SSL连接测试 =====

# 使用SSL连接（require模式）
psql "host=192.168.1.100 port=5432 dbname=app_db user=app_user sslmode=require"

# 使用verify-ca模式（验证服务器证书）
psql "host=pgsql.example.com port=5432 dbname=app_db user=app_user \
    sslmode=verify-ca \
    sslrootcert=/path/to/rootCA.crt"

# 使用verify-full模式（验证服务器证书和主机名）
psql "host=pgsql.example.com port=5432 dbname=app_db user=app_user \
    sslmode=verify-full \
    sslrootcert=/path/to/rootCA.crt"

# 使用客户端证书（双向认证）
psql "host=pgsql.example.com port=5432 dbname=app_db user=db_admin \
    sslmode=verify-full \
    sslrootcert=/path/to/rootCA.crt \
    sslcert=/path/to/client.crt \
    sslkey=/path/to/client.key"

# 查看当前连接的SSL信息
psql -c "SELECT * FROM pg_stat_ssl WHERE pid = pg_backend_pid();"
```

## 10.12 SQL注入防护

### 10.12.1 SQL注入攻击原理与危害

SQL注入是最常见也是最危险的Web安全漏洞之一。攻击者通过在应用程序的输入框、URL参数等位置注入恶意SQL代码，从而欺骗数据库执行非授权操作。SQL注入可能导致数据泄露、数据篡改、权限提升，甚至完全控制数据库服务器。

SQL注入的根本原因是应用程序将用户输入直接拼接到SQL语句中，而没有做适当的转义或使用参数化查询。根据注入方式的不同，SQL注入可分为：基于错误的注入、基于时间的盲注、基于布尔的盲注、UNION查询注入、堆叠查询注入等。

```bash
# ===== SQL注入攻击示例 =====

# 1. 经典的登录绕过
# 应用代码（不安全）：
# query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'"
# 攻击者输入：
# username: admin' -- 
# password: anything
# 生成的SQL：
# SELECT * FROM users WHERE username = 'admin' -- ' AND password = 'anything'
# 结果：直接以admin身份登录

# 2. UNION查询注入
# 原始URL：
# http://example.com/products?id=1
# 攻击URL：
# http://example.com/products?id=1 UNION SELECT username,password FROM users--

# 3. 基于错误的注入
# http://example.com/products?id=1' AND (SELECT COUNT(*) FROM information_schema.tables) > 0--

# 4. 基于时间的盲注
# http://example.com/products?id=1' AND SLEEP(5)--

# ===== 数据库层面的防护措施 =====

# MySQL防护配置
mysql -u root -p << 'EOF'
-- 1. 启用SQL严格模式
SET GLOBAL sql_mode = 'STRICT_TRANS_TABLES,NO_ZERO_IN_DATE,NO_ZERO_DATE,ERROR_FOR_DIVISION_BY_ZERO,NO_AUTO_CREATE_USER,NO_ENGINE_SUBSTITUTION';

-- 2. 限制用户权限（最小权限原则）
-- 应用用户只授予必要的SELECT, INSERT, UPDATE, DELETE权限
-- 绝不授予FILE, SUPER, PROCESS等权限

-- 3. 创建专用的只读用户用于报表查询
CREATE USER 'report_user'@'localhost' IDENTIFIED BY 'ReportP@ss2024!';
GRANT SELECT ON app_db.* TO 'report_user'@'localhost';

-- 4. 禁用多语句执行（防止堆叠注入）
-- 在应用程序连接字符串中设置 multiStatements=false

-- 5. 定期检查是否有SQL注入痕迹
-- 查看慢查询日志中的可疑查询
EOF

# PostgreSQL防护配置
su - postgres
psql << 'EOF'

-- 2. 使用SECURITY DEFINER函数时要特别小心

-- 3. 限制函数执行权限

-- 4. 启用pgAudit监控可疑SQL
EOF

# ===== 检测SQL注入攻击的方法 =====
```

### 10.12.2 存储过程安全

存储过程是数据库中保存的可复用SQL代码集合。虽然存储过程可以提高性能和代码复用性，但如果设计不当，也可能成为安全隐患。特别是动态SQL存储过程，如果没有正确处理输入，可能导致SQL注入。

存储过程安全的关键点：最小权限原则、避免动态SQL、输入验证、使用SECURITY INVOKER而非SECURITY DEFINER、定期审计存储过程。

```sql
-- ===== MySQL存储过程安全 =====

-- 1. 安全的存储过程示例（使用参数）
DELIMITER //
CREATE PROCEDURE GetUserByEmail(IN p_email VARCHAR(255))
BEGIN
    -- 使用参数化查询，安全
    SELECT id, username, email, created_at
    FROM users
    WHERE email = p_email
    LIMIT 1;
END //
DELIMITER ;

-- 2. 危险的存储过程示例（动态SQL拼接）
DELIMITER //
CREATE PROCEDURE SearchUsers(IN p_search_term VARCHAR(255))
BEGIN
    -- 危险！直接拼接SQL，存在注入风险
    SET @sql = CONCAT('SELECT * FROM users WHERE username LIKE "%', p_search_term, '%"');
    PREPARE stmt FROM @sql;
    EXECUTE stmt;
    DEALLOCATE PREPARE stmt;
END //
DELIMITER ;

-- 3. 安全的动态SQL存储过程（使用预处理语句参数）
DELIMITER //
CREATE PROCEDURE SafeSearchUsers(IN p_search_term VARCHAR(255))
BEGIN
    -- 安全：使用参数化预处理语句
    SET @sql = 'SELECT * FROM users WHERE username LIKE ?';
    SET @search = CONCAT('%', p_search_term, '%');
    PREPARE stmt FROM @sql;
    EXECUTE stmt USING @search;
    DEALLOCATE PREPARE stmt;
END //
DELIMITER ;

-- 4. 存储过程权限控制
-- 执行存储过程的用户只需要EXECUTE权限
GRANT EXECUTE ON PROCEDURE GetUserByEmail TO 'app_user'@'localhost';

-- 5. 审计存储过程
-- 查看所有存储过程
SELECT db, name, type, definer, created, modified
FROM mysql.proc
WHERE db = 'app_db';

-- 查看存储过程内容
SHOW CREATE PROCEDURE GetUserByEmail;

-- ===== PostgreSQL存储过程/函数安全 =====

-- 1. 安全的函数（SECURITY INVOKER，默认）
CREATE OR REPLACE FUNCTION get_user_by_email(p_email TEXT)
RETURNS TABLE (
    id INT,
    username VARCHAR(50),
    email VARCHAR(255)
) AS $$
BEGIN
    RETURN QUERY
    SELECT id, username, email
    FROM users
    WHERE email = p_email
    LIMIT 1;
END;
$$ LANGUAGE plpgsql SECURITY INVOKER;

-- 2. SECURITY DEFINER函数（需谨慎使用）
-- SECURITY DEFINER函数以定义者的权限执行，权限提升风险
CREATE OR REPLACE FUNCTION update_user_password(
    p_user_id INT,
    p_new_password TEXT
) RETURNS BOOLEAN AS $$
BEGIN
    -- 只能修改自己的密码
    IF current_user = (SELECT username FROM users WHERE id = p_user_id) THEN
        UPDATE users SET password = p_new_password WHERE id = p_user_id;
        RETURN TRUE;
    END IF;
    RETURN FALSE;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- 3. 防止函数注入：使用quote_ident和quote_literal
CREATE OR REPLACE FUNCTION get_table_count(p_table_name TEXT)
RETURNS BIGINT AS $$
DECLARE
    v_count BIGINT;
BEGIN
    -- 使用quote_ident安全地转义标识符
    EXECUTE format('SELECT COUNT(*) FROM %I', p_table_name) INTO v_count;
    RETURN v_count;
END;
$$ LANGUAGE plpgsql SECURITY INVOKER;

-- 4. 函数权限管理
-- 撤销PUBLIC的函数执行权限
REVOKE ALL ON FUNCTION get_user_by_email(TEXT) FROM PUBLIC;

-- 授予特定用户执行权限
GRANT EXECUTE ON FUNCTION get_user_by_email(TEXT) TO app_user;

-- 5. 审计函数
SELECT proname, prosrc, prosecdef, proowner::regrole
FROM pg_proc
WHERE pronamespace = 'public'::regnamespace;
```

### 10.12.3 SQL注入防护最佳实践

SQL注入防护需要多层次、纵深防御的策略。单靠某一种防护措施是不够的，需要从代码层、数据库层、网络层、运维层多个角度综合防护。

最重要的防护措施是使用参数化查询（预编译语句），这是防止SQL注入最根本的方法。在此基础上，结合输入验证、最小权限、WAF、数据库审计等措施，构建完整的防护体系。

## 10.13 数据库加密

### 10.13.1 透明数据加密（TDE）

透明数据加密（Transparent Data Encryption, TDE）是一种在数据库文件层面进行加密的技术，对应用程序完全透明，无需修改代码。TDE加密的是数据库的物理文件（数据文件、日志文件、备份文件），当数据写入磁盘时加密，读取到内存时解密。

TDE的主要作用是防止数据文件被窃取后导致的数据泄露。但TDE不能防止已授权用户的数据访问，也不能防止SQL注入攻击。MySQL企业版和PostgreSQL（通过扩展或文件系统级加密）都支持TDE。

```bash
# ===== MySQL TDE配置（企业版） =====

# MySQL Enterprise Edition支持TDE

# 1. 安装Keyring插件（用于存储加密密钥）
mysql -u root -p << 'EOF'
-- 安装keyring_file插件（文件存储密钥）
INSTALL PLUGIN keyring_file SONAME 'keyring_file.so';

-- 或使用keyring_okv（Oracle Key Vault）
-- INSTALL PLUGIN keyring_okv SONAME 'keyring_okv.so';
EOF

# 2. 配置my.cnf
cat >> /etc/my.cnf << 'EOF'
[mysqld]
# 启用keyring插件
early-plugin-load = keyring_file.so
keyring_file_data = /var/lib/mysql-keyring/keyring

# 配置InnoDB表空间加密
innodb_encrypt_tables = ON
innodb_encrypt_log = ON
innodb_encryption_threads = 4
innodb_encryption_rotation_iops = 100
default_table_encryption = ON
EOF

# 3. 创建密钥文件目录
mkdir -p /var/lib/mysql-keyring
chown mysql:mysql /var/lib/mysql-keyring
```

### 10.13.2 列级加密

列级加密（也叫字段级加密）是对数据库表中的特定敏感字段进行加密，如密码、身份证号、银行卡号、手机号等。与TDE不同，列级加密是在数据层面进行的，即使攻击者获取了数据库访问权限，没有加密密钥也无法读取敏感数据。

列级加密的实现方式包括：应用层加密（应用程序加密后存入数据库）、数据库层加密（使用数据库加密函数）。推荐使用应用层加密，将加密密钥与数据库分离存储。

```sql
-- ===== MySQL列级加密 =====

-- 1. 使用AES_ENCRYPT/AES_DECRYPT函数
-- 首先设置加密密钥（在应用程序中设置，不要硬编码）
SET @encryption_key = 'YourSecretKey2024!';

-- 创建带加密字段的表
CREATE TABLE user_profiles (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    id_card_no VARBINARY(256) COMMENT '加密的身份证号',
    phone VARBINARY(256) COMMENT '加密的手机号',
    bank_card VARBINARY(256) COMMENT '加密的银行卡号',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 插入加密数据
INSERT INTO user_profiles (user_id, id_card_no, phone, bank_card)
VALUES (
    1,
    AES_ENCRYPT('110101199001011234', @encryption_key),
    AES_ENCRYPT('13800138000', @encryption_key),
    AES_ENCRYPT('6222021234567890123', @encryption_key)
);

-- 查询解密数据
SELECT 
    id,
    user_id,
    CAST(AES_DECRYPT(id_card_no, @encryption_key) AS CHAR) AS id_card_no,
    CAST(AES_DECRYPT(phone, @encryption_key) AS CHAR) AS phone,
    CAST(AES_DECRYPT(bank_card, @encryption_key) AS CHAR) AS bank_card
FROM user_profiles
WHERE user_id = 1;

-- 2. 密码存储（使用哈希，不可逆）
-- 推荐使用bcrypt或Argon2（在应用层实现）
-- MySQL 8.0支持：
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 存储密码（应用层哈希后存储）
-- 示例：使用SHA2（不推荐，应该用bcrypt/Argon2）
INSERT INTO users (username, password_hash)
VALUES ('testuser', SHA2('UserP@ss2024!', 256));

-- 验证密码
SELECT id, username 
FROM users 
WHERE username = 'testuser' 
AND password_hash = SHA2('UserP@ss2024!', 256);

-- ===== PostgreSQL列级加密（pgcrypto扩展） =====

-- 1. 启用pgcrypto扩展
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- 2. 对称加密（AES）
-- 设置加密密钥
SET app.encryption_key = 'YourSecretKey2024!';

-- 创建表
CREATE TABLE user_profiles (
    id SERIAL PRIMARY KEY,
    user_id INT NOT NULL,
    id_card_no BYTEA,
    phone BYTEA,
    bank_card BYTEA,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 插入加密数据
INSERT INTO user_profiles (user_id, id_card_no, phone, bank_card)
VALUES (
    1,
    pgp_sym_encrypt('110101199001011234', current_setting('app.encryption_key')),
    pgp_sym_encrypt('13800138000', current_setting('app.encryption_key')),
    pgp_sym_encrypt('6222021234567890123', current_setting('app.encryption_key'))
);

-- 查询解密数据
SELECT
    id,
    user_id,
    pgp_sym_decrypt(id_card_no, current_setting('app.encryption_key')) AS id_card_no,
    pgp_sym_decrypt(phone, current_setting('app.encryption_key')) AS phone,
    pgp_sym_decrypt(bank_card, current_setting('app.encryption_key')) AS bank_card
FROM user_profiles
WHERE user_id = 1;

-- 3. 密码哈希存储
-- 使用crypt()函数和gen_salt()
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 存储密码（使用bf算法 - Blowfish）
INSERT INTO users (username, password_hash)
VALUES ('testuser', crypt('UserP@ss2024!', gen_salt('bf', 10)));

-- 验证密码
SELECT id, username
FROM users
WHERE username = 'testuser'
AND password_hash = crypt('UserP@ss2024!', password_hash);

-- 4. 非对称加密（RSA）
-- 生成密钥对（在命令行执行）
-- gpg --gen-key
-- 或使用openssl
```

### 10.13.3 备份加密

数据库备份包含完整的数据，是攻击者的重要目标。备份加密是数据保护的重要环节，确保即使备份文件泄露，数据也不会被未授权访问。

备份加密可以在多个层面实现：数据库工具内置加密、文件系统加密、第三方加密工具。推荐在备份生成时就进行加密，而不是先存明文再加密，减少明文暴露窗口。

```bash
# ===== MySQL备份加密 =====

# 方案一：使用mysqldump + openssl加密
MYSQL_USER="root"
MYSQL_PASSWORD="YourPassword"
ENCRYPTION_KEY="YourBackupKey2024!"
BACKUP_DIR="/backup/mysql"
DATE=$(date +%Y%m%d_%H%M%S)

mkdir -p $BACKUP_DIR

# 备份并即时加密（不生成中间明文文件）
mysqldump -u $MYSQL_USER -p$MYSQL_PASSWORD \
    --all-databases \
    --single-transaction \
    --routines \
    --triggers \
    --events | gzip | openssl enc -aes-256-cbc -salt -pbkdf2 -pass pass:$ENCRYPTION_KEY \
    > $BACKUP_DIR/mysql_full_${DATE}.sql.gz.enc

# 解密恢复
openssl enc -d -aes-256-cbc -pbkdf2 \
    -in $BACKUP_DIR/mysql_full_${DATE}.sql.gz.enc \
    -pass pass:$ENCRYPTION_KEY | gunzip | mysql -u $MYSQL_USER -p$MYSQL_PASSWORD

# 方案二：MySQL Enterprise Backup（企业版）
# mysqlbackup --user=root --password --backup-dir=/backup --encrypt --key-file=/path/to/key backup

# 方案三：使用Percona XtraBackup加密
# xtrabackup --backup --target-dir=/backup --encrypt=AES256 --encrypt-key-file=/path/to/key

# ===== PostgreSQL备份加密 =====

# 方案一：pg_dump + openssl
PG_USER="postgres"
PG_DB="app_db"
ENCRYPTION_KEY="YourBackupKey2024!"
BACKUP_DIR="/backup/postgres"
DATE=$(date +%Y%m%d_%H%M%S)

mkdir -p $BACKUP_DIR

# 备份并加密
su - $PG_USER -c "pg_dump -Fc $PG_DB" | gzip | \
    openssl enc -aes-256-cbc -salt -pbkdf2 -pass pass:$ENCRYPTION_KEY \
    > $BACKUP_DIR/${PG_DB}_${DATE}.dump.gz.enc

# 解密恢复
openssl enc -d -aes-256-cbc -pbkdf2 \
    -in $BACKUP_DIR/${PG_DB}_${DATE}.dump.gz.enc \
    -pass pass:$ENCRYPTION_KEY | gunzip | \
    su - $PG_USER -c "pg_restore -d $PG_DB -Fc"

# 方案二：使用pgcrypto加密备份
# 见列级加密部分

# ===== MongoDB备份加密 =====

# 方案一：mongodump + openssl
ENCRYPTION_KEY="YourBackupKey2024!"
BACKUP_DIR="/backup/mongo"
DATE=$(date +%Y%m%d_%H%M%S)

mkdir -p $BACKUP_DIR

mongodump --archive --gzip | \
    openssl enc -aes-256-cbc -salt -pbkdf2 -pass pass:$ENCRYPTION_KEY \
    > $BACKUP_DIR/mongo_${DATE}.gz.enc

# 解密恢复
openssl enc -d -aes-256-cbc -pbkdf2 \
    -in $BACKUP_DIR/mongo_${DATE}.gz.enc \
    -pass pass:$ENCRYPTION_KEY | mongorestore --archive --gzip

# ===== 加密密钥管理最佳实践 =====

# 1. 不要将密钥硬编码在脚本中
# 2. 使用密钥管理服务（KMS）
# 3. 定期轮换密钥
# 4. 分离备份和密钥的存储位置

# 示例：从环境变量读取密钥
# export BACKUP_ENCRYPTION_KEY="your-secure-key"
# openssl ... -pass env:BACKUP_ENCRYPTION_KEY

# 示例：从文件读取密钥（文件权限600）
# echo "your-secure-key" > /etc/backup_key
# chmod 600 /etc/backup_key
# chown root:root /etc/backup_key
# openssl ... -pass file:/etc/backup_key

# ===== 备份加密验证脚本 =====

cat > /usr/local/bin/verify_backup.sh << 'EOF'
#!/bin/bash
# 验证备份文件完整性和可恢复性

BACKUP_FILE=$1
DECRYPTION_KEY=$2
TEMP_DIR=$(mktemp -d)

echo "正在验证备份: $BACKUP_FILE"

# 检查文件是否存在
if [ ! -f "$BACKUP_FILE" ]; then
    echo "错误：文件不存在"
    exit 1
fi

# 检查文件大小
SIZE=$(du -h "$BACKUP_FILE" | cut -f1)
echo "文件大小: $SIZE"

# 尝试验证解密
if echo "$DECRYPTION_KEY" | openssl enc -d -aes-256-cbc -pbkdf2 -in "$BACKUP_FILE" -pass stdin > /dev/null 2>&1; then
    echo "✓ 解密验证通过"
else
    echo "✗ 解密失败，密钥可能错误或文件损坏"
    rm -rf $TEMP_DIR
    exit 1
fi

# 验证SQL语法（MySQL示例）
if echo "$DECRYPTION_KEY" | openssl enc -d -aes-256-cbc -pbkdf2 -in "$BACKUP_FILE" -pass stdin | gunzip | head -100 > /dev/null 2>&1; then
    echo "✓ 格式验证通过"
else
    echo "✗ 格式验证失败"
    rm -rf $TEMP_DIR
    exit 1
fi

rm -rf $TEMP_DIR
echo "备份验证完成 ✓"
EOF

chmod +x /usr/local/bin/verify_backup.sh
```

### 10.14.1 MySQL审计插件详解

数据库审计是安全合规的基础要求，也是发现安全事件的重要手段。MySQL有多种审计方案，社区版可以使用MariaDB Audit Plugin或Percona Audit Plugin，企业版可以使用MySQL Enterprise Audit。

一个完善的审计系统应该能够记录：登录事件（成功/失败）、查询语句、DDL操作、DML操作、权限变更、配置变更等。审计日志本身也需要保护，防止被篡改或删除。

```bash
# ===== MariaDB Audit Plugin详细配置 =====

# 1. 安装插件（MySQL也可使用MariaDB的审计插件）

INSTALL PLUGIN server_audit SONAME 'server_audit.so';
EOF

# 2. 完整配置
cat >> /etc/my.cnf << 'EOF'
[mysqld]
plugin-load-add = server_audit.so

# 启用审计
server_audit_logging = ON

# 审计事件类型：
# CONNECT - 连接和断开
# QUERY - 查询语句和结果
# TABLE - 表级操作
# QUERY_DDL - DDL查询
# QUERY_DML - DML查询
# QUERY_DCL - DCL查询
server_audit_events = 'CONNECT,QUERY,QUERY_DDL,QUERY_DML,QUERY_DCL'

# 审计日志输出类型：FILE, SYSLOG, OFILE（滚动）
server_audit_output_type = FILE

# 日志文件路径
server_audit_file_path = /var/log/mysql/audit/audit.log

# 日志文件大小限制（字节）
server_audit_file_rotate_size = 1073741824  # 1GB

# 保留日志文件数
server_audit_file_rotations = 30

# 日志刷新间隔（秒）
server_audit_file_flush_interval = 1

# 查询日志长度限制
server_audit_query_log_limit = 4096

# 排除的用户（不审计这些用户的操作）
server_audit_excl_users = 'backup_user,replication_user'

# 包含的用户（只审计这些用户，留空表示全部）
server_audit_incl_users = ''

# 排除的数据库
# server_audit_excl_databases = 'test_db'

# 包含的数据库
# server_audit_incl_databases = 'app_db'

# 系统日志配置（如果使用SYSLOG）
# server_audit_syslog_ident = 'mysql-audit'
# server_audit_syslog_facility = LOG_USER
# server_audit_syslog_priority = LOG_INFO
EOF

# 3. 创建审计日志目录

# ===== 审计日志分析 =====

# 审计日志格式（文件输出）
# [timestamp],[serverhost],[username],[host],[connectionid],[queryid],[operation],[database],[object],[retcode]

# 1. 查看登录失败记录
grep ",FAILED," /var/log/mysql/audit/audit.log | head -20
```

### 10.14.2 PostgreSQL审计（pgaudit）

pgAudit是PostgreSQL最流行的审计扩展，提供了详细的会话审计和对象审计功能。pgAudit可以记录所有SQL语句，包括SELECT、INSERT、UPDATE、DELETE以及DDL语句，是PostgreSQL安全审计的标准解决方案。

pgAudit支持两种审计模式：会话审计（SESSION）记录会话执行的所有语句，对象审计（OBJECT）只记录特定对象上的操作。根据合规要求，可以选择适当的审计级别。

```bash
# ===== 安装pgAudit =====

# CentOS/RHEL
yum install -y pgaudit15_14

# Debian/Ubuntu
# apt-get install -y postgresql-14-pgaudit

# ===== 配置pgAudit =====

# 编辑postgresql.conf
cat >> /var/lib/pgsql/14/data/postgresql.conf << 'EOF'
# ===== pgaudit配置 =====
shared_preload_libraries = 'pgaudit, pg_stat_statements'

# 审计日志类别：
# READ - SELECT和COPY FROM
# WRITE - INSERT, UPDATE, DELETE, TRUNCATE, COPY TO
# FUNCTION - 函数和DO块
# ROLE - GRANT, REVOKE, CREATE/ALTER/DROP ROLE
# DDL - DDL语句（除ROLE相关）
# MISC - 其他语句
# ALL - 所有类别
pgaudit.log = 'ddl, role, write, misc'

# 日志参数（记录完整语句）
pgaudit.log_parameter = on

# 记录关系（表名等）
pgaudit.log_relation = on

# 记录语句详情
pgaudit.log_statement_once = off

# 审计日志级别（发送到标准日志）
pgaudit.log_level = log

# 审计日志格式：csv 或 stdlog
pgaudit.log_format = csv

# 会话级审计（所有会话）
pgaudit.role = 'auditor'
EOF

# ===== 在数据库中启用pgaudit =====

su - postgres
psql << 'EOF'

\c app_db

-- 创建审计角色

-- 配置特定对象审计（对象审计模式）
-- 对特定表启用对象审计
-- SELECT pgaudit.audit_table('public.users', 'r,w');
-- r=SELECT, w=INSERT/UPDATE/DELETE

-- 查看已审计的表
SELECT * FROM pgaudit.audit_tables;

-- 查看审计配置
SHOW pgaudit.log;
SHOW pgaudit.log_parameter;

-- 测试审计
CREATE TABLE test_audit (id INT, name TEXT);
INSERT INTO test_audit VALUES (1, 'test');
UPDATE test_audit SET name = 'test2' WHERE id = 1;
DELETE FROM test_audit WHERE id = 1;
DROP TABLE test_audit;
EOF

# ===== 查看审计日志 =====

# PostgreSQL日志目录
PG_LOG_DIR="/var/lib/pgsql/14/data/log"

# 查看审计条目
grep "AUDIT:" $PG_LOG_DIR/*.log | head -50

# 审计日志格式说明：
# AUDIT: SESSIONID,STATEMENTID,USER,DATABASE,RELATION,COMMAND,OBJECT_TYPE,OBJECT_NAME,STATEMENT,PARAMETER

# 1. 查看DDL操作
grep "AUDIT:.*,DDL," $PG_LOG_DIR/*.log

# 2. 查看权限变更

grep "AUDIT:.*,app_user," $PG_LOG_DIR/*.log

su - postgres
psql << 'EOF'

-- 启用会话级审计（对特定角色）

-- 启用对象级审计
-- 审计users表的所有操作
SELECT pgaudit.audit_table('public.users', 'rw');

-- 审计orders表的写操作
-- SELECT pgaudit.audit_table('public.orders', 'w');

# 已在postgresql.conf中配置

# 2. 日志备份
```

### 10.14.3 慢查询日志分析

慢查询日志不仅是性能优化的重要工具，也是安全分析的重要数据源。异常的慢查询可能暗示着SQL注入攻击、数据泄露尝试、或其他恶意活动。例如，包含UNION、information_schema、SLEEP等关键词的慢查询可能是攻击行为。

定期分析慢查询日志，结合性能优化和安全分析，可以同时提升系统性能和安全性。

```bash
# ===== MySQL慢查询日志分析 =====

# 1. 启用慢查询日志
mysql -u root -p << 'EOF'
```

## 10.15 数据库备份安全

### 10.15.1 备份策略设计

数据库备份是数据安全的最后一道防线。一个好的备份策略不仅要考虑数据恢复能力，还要考虑备份数据本身的安全性。备份策略需要根据业务的RTO（恢复时间目标）和RPO（恢复点目标）来设计。

备份策略的核心要素包括：备份类型（全量、增量、差异）、备份频率、保留周期、存储位置、加密方式、验证机制、灾难恢复预案。不同的业务系统有不同的备份需求，需要量身定制。

```bash
# ===== 备份策略设计原则 =====

# 1. 3-2-1备份原则
#    - 至少3份数据副本
#    - 存储在2种不同介质上
#    - 1份异地存放

# 2. RTO和RPO
#    RTO（Recovery Time Objective）：最大容忍恢复时间
#    RPO（Recovery Point Objective）：最大容忍数据丢失时间

# 3. 定期演练恢复
#    备份不可验证等于没有备份

# ===== MySQL备份策略示例 =====

# 场景：中等规模业务系统，RTO=4小时，RPO=1小时

# 备份策略：
# - 每周日全量备份
# - 周一至周六每天增量备份（使用binlog）
# - 保留30天的全量备份
# - 备份加密存储
# - 异地同步备份

# 实现脚本：
cat > /usr/local/bin/mysql_backup.sh << 'EOF'
#!/bin/bash
# MySQL备份脚本 - 全量+增量

MYSQL_USER="root"
MYSQL_PASSWORD="YourSecurePassword2024!"
BACKUP_DIR="/backup/mysql"
ENCRYPTION_KEY_FILE="/etc/mysql_backup.key"
DATE=$(date +%Y%m%d)
DAY_OF_WEEK=$(date +%u)  # 1=周一, 7=周日

mkdir -p $BACKUP_DIR/{full,binlog,logs}
LOG_FILE="$BACKUP_DIR/logs/backup_${DATE}.log"

echo "[$(date)] 开始MySQL备份..." > $LOG_FILE

# 全量备份（每周日）
if [ $DAY_OF_WEEK -eq 7 ]; then
    echo "[$(date)] 执行全量备份..." >> $LOG_FILE
    
    BACKUP_FILE="$BACKUP_DIR/full/mysql_full_${DATE}.sql.gz.enc"
    
    mysqldump -u $MYSQL_USER -p$MYSQL_PASSWORD \
        --all-databases \
        --single-transaction \
        --routines \
        --triggers \
        --events \
        --master-data=2 \
        --flush-logs 2>> $LOG_FILE | \
        gzip | \
        openssl enc -aes-256-cbc -salt -pbkdf2 \
            -pass file:$ENCRYPTION_KEY_FILE \
            -out $BACKUP_FILE
    
    if [ $? -eq 0 ]; then
        echo "[$(date)] 全量备份成功: $BACKUP_FILE" >> $LOG_FILE
        BACKUP_SIZE=$(du -h $BACKUP_FILE | cut -f1)
        echo "[$(date)] 备份大小: $BACKUP_SIZE" >> $LOG_FILE
    else
        echo "[$(date)] 全量备份失败!" >> $LOG_FILE
        exit 1
    fi
fi

# 增量备份（每天刷新binlog，备份binlog）
echo "[$(date)] 执行增量备份（刷新binlog）..." >> $LOG_FILE

# 刷新binlog
mysqladmin -u $MYSQL_USER -p$MYSQL_PASSWORD flush-logs 2>> $LOG_FILE

# 备份所有binlog（除了当前正在写的）
CURRENT_BINLOG=$(mysql -u $MYSQL_USER -p$MYSQL_PASSWORD -e "SHOW MASTER STATUS\G" 2>/dev/null | grep File | awk '{print $2}')
BINLOG_DIR="/var/log/mysql"

for binlog in $BINLOG_DIR/mysql-bin.*; do
    basename=$(basename $binlog)
    if [ "$basename" != "$CURRENT_BINLOG" ] && [ ! -f "$BACKUP_DIR/binlog/${basename}.enc" ]; then
        gzip -c $binlog | \
        openssl enc -aes-256-cbc -salt -pbkdf2 \
            -pass file:$ENCRYPTION_KEY_FILE \
            -out $BACKUP_DIR/binlog/${basename}.gz.enc
        echo "[$(date)] 备份binlog: $basename" >> $LOG_FILE
    fi
done

# 清理旧备份
echo "[$(date)] 清理30天前的备份..." >> $LOG_FILE
find $BACKUP_DIR/full -name "*.enc" -mtime +30 -delete
find $BACKUP_DIR/binlog -name "*.enc" -mtime +30 -delete
find $BACKUP_DIR/logs -name "*.log" -mtime +30 -delete

echo "[$(date)] 备份完成" >> $LOG_FILE
EOF

chmod +x /usr/local/bin/mysql_backup.sh

# 设置定时任务（每天凌晨2点执行）
echo "0 2 * * * root /usr/local/bin/mysql_backup.sh" > /etc/cron.d/mysql-backup

# ===== PostgreSQL备份策略示例 =====

cat > /usr/local/bin/pg_backup.sh << 'EOF'
#!/bin/bash
# PostgreSQL备份脚本 - 支持PITR

PG_USER="postgres"
BACKUP_DIR="/backup/postgres"

echo "[$(date)] 开始PostgreSQL备份..." > $LOG_FILE

    echo "[$(date)] 执行基础备份..." >> $LOG_FILE
    
    BACKUP_FILE="$BACKUP_DIR/base/base_${DATE}.tar.gz.enc"
    
    su - $PG_USER -c "pg_basebackup -Ft -z -P -v -D -" 2>> $LOG_FILE | \
        openssl enc -aes-256-cbc -salt -pbkdf2 \

            -out $BACKUP_FILE
    
    if [ $? -eq 0 ]; then
        echo "[$(date)] 基础备份成功: $BACKUP_FILE" >> $LOG_FILE
    else
        echo "[$(date)] 基础备份失败!" >> $LOG_FILE
        exit 1
    fi
fi

# 逻辑备份（每天）
echo "[$(date)] 执行逻辑备份..." >> $LOG_FILE
BACKUP_FILE="$BACKUP_DIR/dumps/app_db_${DATE}.dump.gz.enc"

su - $PG_USER -c "pg_dump -Fc app_db" 2>> $LOG_FILE | \
    gzip | \
    openssl enc -aes-256-cbc -salt -pbkdf2 \
        -pass file:$ENCRYPTION_KEY_FILE \
        -out $BACKUP_FILE

# WAL归档（实时，配置在postgresql.conf中）
# archive_mode = on
# archive_command = 'cp %p /backup/postgres/archive/%f'

# 清理旧备份
find $BACKUP_DIR/base -name "*.enc" -mtime +30 -delete
find $BACKUP_DIR/dumps -name "*.enc" -mtime +30 -delete
find $BACKUP_DIR/logs -name "*.log" -mtime +30 -delete

echo "[$(date)] 备份完成" >> $LOG_FILE
EOF

chmod +x /usr/local/bin/pg_backup.sh
```

### 10.15.2 加密备份与异地备份

备份数据的安全性与备份本身同样重要。未加密的备份文件是数据泄露的重大隐患。同时，只在本地备份也不够安全，当服务器发生物理故障或机房灾难时，本地备份可能一同丢失。

异地备份是灾难恢复的关键措施。可以选择的异地备份方案包括：对象存储（S3/OSS）、远程服务器、磁带库、云备份服务等。重要的是，异地备份也需要加密，并且密钥需要与备份数据分开管理。

```bash
# ===== 备份加密最佳实践 =====

# 1. 加密算法选择
#    - 对称加密：AES-256-CBC（推荐）
#    - 密钥派生：PBKDF2
#    - 加盐：防止彩虹表攻击

# 2. 密钥管理
#    - 密钥与备份数据分开存储
#    - 使用KMS（密钥管理服务）
#    - 定期轮换密钥
#    - 多副本保存密钥

# 生成GPG密钥对（用于非对称加密备份）
# gpg --gen-key
# gpg --export -a "Backup User" > backup_pub.key
# gpg --export-secret-key -a "Backup User" > backup_priv.key

# 使用公钥加密备份
# mysqldump ... | gzip | gpg --encrypt --recipient "Backup User" -o backup.sql.gz.gpg

# 使用私钥解密
# gpg --decrypt backup.sql.gz.gpg | gunzip | mysql ...

# ===== 异地备份方案 =====

# 方案一：使用rsync同步到远程服务器
# 在备份服务器上配置专用备份用户和SSH密钥认证
```

### 10.15.3 备份验证与恢复演练

备份的目的是恢复，无法恢复的备份没有任何价值。很多企业虽然做了备份，但从未验证过备份的可用性，真正需要恢复时才发现备份损坏或不完整，造成了不可挽回的损失。

备份验证需要定期进行，包括：备份完整性校验、备份解密测试、数据恢复演练、恢复时间测试。建议每月至少进行一次完整的恢复演练，确保备份策略有效。

```bash
# ===== 备份验证脚本 =====

cat > /usr/local/bin/verify_mysql_backup.sh << 'EOF'
#!/bin/bash
# MySQL备份验证脚本
# 功能：验证备份文件的完整性、可解密性、可恢复性

BACKUP_DIR="/backup/mysql/full"
ENCRYPTION_KEY_FILE="/etc/mysql_backup.key"
MYSQL_ROOT_PASS="TempR00tP@ss!"
TEMP_DIR="/tmp/mysql_backup_verify"
RESULT_FILE="/var/log/backup_verify_$(date +%Y%m%d).log"

echo "===== MySQL备份验证 $(date) =====" > $RESULT_FILE
echo "" >> $RESULT_FILE

# 获取最新的备份文件
LATEST_BACKUP=$(ls -t $BACKUP_DIR/*.enc 2>/dev/null | head -1)

if [ -z "$LATEST_BACKUP" ]; then
    echo "错误：未找到备份文件" >> $RESULT_FILE
    exit 1
fi

echo "备份文件: $LATEST_BACKUP" >> $RESULT_FILE

# 1. 文件大小检查
FILE_SIZE=$(du -h "$LATEST_BACKUP" | cut -f1)
echo "文件大小: $FILE_SIZE" >> $RESULT_FILE

if [ $(stat -c%s "$LATEST_BACKUP") -lt 1024 ]; then
    echo "✗ 文件过小，可能损坏" >> $RESULT_FILE
    exit 1
fi

# 2. 解密测试
echo "" >> $RESULT_FILE
echo "步骤1: 解密测试..." >> $RESULT_FILE
mkdir -p $TEMP_DIR

if openssl enc -d -aes-256-cbc -pbkdf2 \
    -in "$LATEST_BACKUP" \
    -pass file:$ENCRYPTION_KEY_FILE \
    -out $TEMP_DIR/backup.sql.gz 2>/dev/null; then
    echo "✓ 解密成功" >> $RESULT_FILE
else
    echo "✗ 解密失败" >> $RESULT_FILE
    rm -rf $TEMP_DIR
    exit 1
fi

# 3. 解压缩测试
echo "步骤2: 解压缩测试..." >> $RESULT_FILE
if gunzip -t $TEMP_DIR/backup.sql.gz 2>/dev/null; then
    echo "✓ 压缩文件正常" >> $RESULT_FILE
else
    echo "✗ 压缩文件损坏" >> $RESULT_FILE
    rm -rf $TEMP_DIR
    exit 1
fi

# 4. SQL语法检查（抽样）
echo "步骤3: SQL格式检查..." >> $RESULT_FILE
gunzip -c $TEMP_DIR/backup.sql.gz | head -100 > /dev/null
if [ $? -eq 0 ]; then
    # 检查是否包含CREATE TABLE语句
    TABLE_COUNT=$(gunzip -c $TEMP_DIR/backup.sql.gz | grep -c "CREATE TABLE")
    echo "✓ SQL格式正常，包含约 $TABLE_COUNT 个表" >> $RESULT_FILE
else
    echo "✗ SQL文件格式错误" >> $RESULT_FILE
    rm -rf $TEMP_DIR
    exit 1
fi

# 5. 完整恢复测试（可选，需要测试环境）
# echo "步骤4: 恢复测试..." >> $RESULT_FILE
# 这里可以启动一个临时MySQL实例进行完整恢复测试

# 清理
rm -rf $TEMP_DIR

echo "" >> $RESULT_FILE
echo "===== 验证通过 ✓ =====" >> $RESULT_FILE

exit 0
EOF

chmod +x /usr/local/bin/verify_mysql_backup.sh
```

## 10.16 数据库漏洞扫描

### 10.16.1 常见数据库漏洞

数据库漏洞是攻击者入侵数据库系统的重要途径。常见的数据库漏洞包括：弱密码、默认配置漏洞、SQL注入、权限提升、缓冲区溢出、认证绕过等。定期进行漏洞扫描和安全评估是保持数据库安全的重要手段。

根据CVE数据库统计，MySQL、PostgreSQL、MongoDB、Redis等主流数据库每年都会曝出多个安全漏洞。及时关注安全公告、升级到最新版本是防范已知漏洞的最有效方法。

## 10.17 MongoDB安全加固进阶

### 10.17.1 MongoDB认证与授权深入

MongoDB的安全机制在早期版本中比较薄弱，但从3.0版本开始有了显著改进。MongoDB使用基于角色的访问控制（RBAC），支持细粒度的权限管理。正确配置认证和授权是MongoDB安全的基础。

MongoDB内置了多种角色，包括数据库用户角色（read、readWrite）、数据库管理角色（dbAdmin、userAdmin）、集群管理角色（clusterAdmin、clusterManager）、备份恢复角色（backup、restore）、超级用户角色（root）等。

```javascript
// ===== MongoDB角色管理 =====

// 连接MongoDB
mongo -u admin -p --authenticationDatabase admin

// 1. 查看内置角色
use admin
db.runCommand({ rolesInfo: 1, showBuiltinRoles: true })

// 2. 自定义角色
// 创建一个只访问特定集合的只读角色
db.createRole({
    role: "collection_reader",
    privileges: [
        {
            resource: { db: "app_db", collection: "users" },
            actions: ["find"]
        },
        {
            resource: { db: "app_db", collection: "products" },
            actions: ["find"]
        }
    ],
    roles: []
})

// 3. 创建应用用户角色
db.createRole({
    role: "app_user_role",
    privileges: [
        {
            resource: { db: "app_db", collection: "" },
            actions: ["find", "insert", "update", "remove"]
        }
    ],
    roles: []
})

// 4. 查看角色信息
db.getRole("collection_reader", { showPrivileges: true })

// 5. 修改角色
db.updateRole("app_user_role", {
    privileges: [
        {
            resource: { db: "app_db", collection: "" },
            actions: ["find", "insert", "update"]
        }
    ]
})

// 6. 删除角色
db.dropRole("old_role")

// ===== 用户管理 =====

// 1. 创建具有多个角色的用户
use admin
db.createUser({
    user: "ops_user",
    pwd: "OpsUserP@ss2024!",
    roles: [
        { role: "read", db: "app_db" },
        { role: "dbAdmin", db: "app_db" },
        { role: "clusterMonitor", db: "admin" }
    ],
    authenticationRestrictions: [
        {
            clientSource: ["192.168.1.0/24"]
        }
    ],
    mechanisms: ["SCRAM-SHA-256"]
})

// 2. 查看用户
db.getUsers()
db.getUser("ops_user")

// 3. 修改用户密码
db.changeUserPassword("ops_user", "NewP@ssword2024!")

// 4. 修改用户角色
db.grantRolesToUser("ops_user", [
    { role: "backup", db: "admin" }
])

db.revokeRolesFromUser("ops_user", [
    { role: "dbAdmin", db: "app_db" }
])

// 5. 禁用用户
db.updateUser("ops_user", { roles: [] })

// 6. 删除用户
db.dropUser("old_user")

// ===== 认证配置 =====

// 1. 配置文件认证设置
// /etc/mongod.conf
security:
  authorization: enabled
  # 认证机制
  authenticationMechanisms: SCRAM-SHA-256
  
// 2. 查看当前认证方式
db.runCommand({ getCmdLineOpts: 1 })

// 3. 查看当前连接的认证信息
db.runCommand({ connectionStatus: 1 })
```

### 10.17.2 MongoDB网络安全与加密

MongoDB的网络安全配置包括：绑定地址、SSL/TLS加密、IP白名单、端口安全等。在生产环境中，MongoDB绝不应该暴露在公网上，必须有严格的网络访问控制。

MongoDB支持TLS/SSL加密连接，包括单向认证和双向认证。同时，MongoDB还支持字段级加密（Field-Level Encryption, FLE），可以在客户端对敏感字段进行加密，实现端到端的数据加密保护。

```bash
# ===== MongoDB网络安全配置 =====

# 1. 绑定地址配置
# /etc/mongod.conf
net:
  port: 27017
  bindIp: 127.0.0.1  # 只监听本地
  # bindIp: 127.0.0.1,192.168.1.100  # 监听指定IP

# 2. SSL/TLS配置
mkdir -p /etc/mongodb/ssl
cd /etc/mongodb/ssl

# 生成CA证书
openssl genrsa -out ca.key 4096
openssl req -x509 -new -nodes -key ca.key -sha256 -days 3650 -out ca.crt \
    -subj "/C=CN/ST=Beijing/L=Beijing/O=DBCA/CN=MongoDB CA"

# 生成服务器证书
openssl genrsa -out server.key 2048
openssl req -new -key server.key -out server.csr \
    -subj "/C=CN/ST=Beijing/L=Beijing/O=DBCA/CN=mongodb.example.com"

openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key \
    -CAcreateserial -out server.crt -days 730 -sha256

# 合并为PEM文件
cat server.key server.crt > server.pem
chmod 600 server.pem
chown mongod:mongod server.pem

# 配置MongoDB使用SSL
cat >> /etc/mongod.conf << 'EOF'
net:
  port: 27017
  bindIp: 127.0.0.1
  tls:
    mode: requireTLS
    certificateKeyFile: /etc/mongodb/ssl/server.pem
    CAFile: /etc/mongodb/ssl/ca.crt
    # 客户端证书认证（双向认证）
    # allowConnectionsWithoutCertificates: false
    # certificateSelector: subject=CN=client1
EOF

# 3. 重启MongoDB
systemctl restart mongod

# 4. 客户端使用SSL连接
mongo --tls --tlsCAFile /etc/mongodb/ssl/ca.crt \
    --host mongodb.example.com --port 27017 \
    -u admin -p --authenticationDatabase admin

# 使用客户端证书（双向认证）
# mongo --tls --tlsCAFile ca.crt --tlsCertificateKeyFile client.pem ...

# ===== MongoDB字段级加密 =====

# MongoDB 4.2+支持客户端字段级加密(FLE)
# 需要在应用程序中使用MongoDB驱动实现

# 主要概念：
# - Data Encryption Key (DEK)：数据加密密钥
# - Customer Master Key (CMK)：主密钥
# - Key Vault：密钥库集合

# Node.js示例（概念演示）
# const { MongoClient, ClientEncryption } = require('mongodb');
#
```

## 10.18 完整实战案例：数据库入侵应急响应

### 10.18.1 案例背景与入侵检测

某电商公司的生产环境MySQL数据库被入侵，攻击者通过SQL注入获取了应用数据库的用户名和密码，然后登录数据库窃取了大量用户数据，并创建了后门账户。安全团队接到告警后立即启动应急响应。

本案例模拟了真实的数据库入侵场景，从入侵检测、隔离、分析、清除、恢复到加固的完整流程，展示了数据库安全加固在实际工作中的应用。

```bash
# ===== 第一步：入侵检测与告警 =====

# 场景：
# - 监控系统发出数据库异常连接告警
# - 应用防火墙检测到大量SQL注入尝试
# - DBA发现数据库中有未知的用户

# 1. 检查异常连接
mysql -u root -p -e "SHOW PROCESSLIST;"
mysql -u root -p -e "SELECT * FROM information_schema.PROCESSLIST;"

# 查看当前连接数
mysql -u root -p -e "SHOW STATUS LIKE 'Threads_connected';"

# 2. 检查错误日志
tail -200 /var/log/mysql/error.log | grep -iE "access denied|error|warning"

# 3. 检查慢查询日志中的可疑操作
grep -iE "union|information_schema|drop|delete from|into outfile" /var/log/mysql/slow.log | tail -50

# 4. 检查审计日志
grep "CREATE USER\|GRANT\|DROP" /var/log/mysql/audit.log | tail -20

# 5. 检查异常用户
mysql -u root -p -e "SELECT user, host, create_time, password_last_changed FROM mysql.user ORDER BY create_time DESC;"

# 发现可疑用户：hacker@% （最近创建的）
# 发现：app_user的host被改为%（原先是localhost）

# ===== 第二步：紧急隔离 =====

# 1. 防火墙阻断数据库对外服务
iptables -I INPUT 1 -p tcp --dport 3306 -j DROP

# 2. 保留管理IP的访问
iptables -I INPUT 1 -p tcp -s 192.168.1.100 --dport 3306 -j ACCEPT

# 3. 杀死所有非本地连接
mysql -u root -p << 'EOF'
-- 杀死所有非本地连接的会话
KILL 12345;  -- 根据实际进程ID
-- 或者批量杀死
SELECT CONCAT('KILL ', id, ';') 
FROM information_schema.PROCESSLIST 
WHERE host NOT LIKE 'localhost%' AND user != 'root';
EOF

# 4. 关闭应用服务器的数据库连接
# 通知应用团队暂时关闭应用

# ===== 第三步：入侵范围评估 =====

# 1. 盘点可疑用户
mysql -u root -p << 'EOF'
-- 查看所有用户及其权限
SELECT user, host, Select_priv, Insert_priv, Update_priv, Delete_priv,
       Create_priv, Drop_priv, File_priv, Super_priv, Grant_priv
FROM mysql.user
ORDER BY create_time DESC;

-- 查看具有超级权限的用户
SELECT user, host FROM mysql.user WHERE Super_priv = 'Y';

-- 查看具有FILE权限的用户（危险）
SELECT user, host FROM mysql.user WHERE File_priv = 'Y';

-- 查看具有GRANT权限的用户
SELECT user, host FROM mysql.user WHERE Grant_priv = 'Y';
EOF

# 2. 检查数据库对象
mysql -u root -p << 'EOF'
-- 查看最近创建的存储过程
SELECT db, name, type, definer, created, modified
FROM mysql.proc
ORDER BY created DESC
LIMIT 20;

-- 查看最近创建的触发器
SELECT trigger_schema, trigger_name, event_object_table, created
FROM information_schema.TRIGGERS
ORDER BY created DESC
LIMIT 20;

-- 查看最近修改的表
SELECT table_schema, table_name, create_time, update_time
FROM information_schema.TABLES
WHERE table_schema NOT IN ('mysql', 'information_schema', 'performance_schema')
ORDER BY update_time DESC
LIMIT 20;
EOF

# 3. 检查数据篡改
# 对比最近的备份，检查关键表的数据变化
# 检查用户表、管理员表等关键数据

# 4. 检查服务器层面的入侵
# 检查系统日志
tail -200 /var/log/secure | grep -iE "failed|accepted|root"
# 检查新增的系统用户
cat /etc/passwd
# 检查定时任务
crontab -l
ls -la /etc/cron.d/
# 检查后门进程
ps aux | grep -E "nc|netcat|shell"
# 检查rootkit
```

### 10.18.2 后门清除与系统加固

在确认入侵范围后，下一步是清除后门和进行系统加固。这一步需要非常仔细，确保所有后门都被清除，同时加固系统防止再次被入侵。

清除后门的工作包括：删除可疑用户、回收过度权限、删除恶意存储过程和触发器、修复被篡改的数据、系统层面的后门清除等。加固工作包括：修改所有密码、配置防火墙、启用审计、加强认证等。

```bash
# ===== 第四步：清除后门 =====

# 1. 清除数据库后门用户
mysql -u root -p << 'EOF'
-- 列出所有需要删除的可疑用户
-- 假设发现以下可疑用户：
```

### 10.19.4 Redis安全检查清单

Redis因其高性能和丰富的数据结构被广泛使用，但其默认配置安全性较弱。以下检查清单帮助管理员全面加固Redis服务。

```markdown

- [ ] 禁用操作系统用户远程登录
```
