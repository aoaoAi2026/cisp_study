# 第13章 容器安全

> 第13章 | 容器安全

## 13.1 Docker安全基础

### 13.1.1 Docker安全架构概述

Docker作为目前最流行的容器化平台，其安全机制是整个容器生态安全的核心。理解Docker的安全架构对于保障容器环境安全至关重要。Docker容器虽然提供了进程隔离，但与虚拟机相比，共享宿主系统内核的特性使得容器的安全性需要特别关注。

Docker的安全模型建立在多个层面之上：内核命名空间（Namespaces）提供了基本的进程隔离，控制组（cgroups）实现了资源限制， Capabilities机制细粒度地管理特权操作，而Seccomp和AppArmor则限制了系统调用的访问权限。这些安全机制的协同工作，构建起了Docker容器的基本安全屏障。

在实际生产环境中，Docker安全需要考虑多个维度：镜像来源的可信性、容器运行时的权限控制、网络通信的隔离性、存储卷的数据保护、以及运行时资源的合理限制。任何一个环节的疏漏都可能导致安全风险，因此需要建立完整的安全防护体系。

### 13.1.2 Docker内核安全机制

Docker容器利用Linux内核的多种安全特性来实现隔离和限制。命名空间是最基础的安全机制，它将系统资源划分为互不干扰的视图，使得每个容器都拥有独立的进程ID、网络栈、挂载点、用户ID空间等资源。

进程命名空间（PID Namespace）确保每个容器拥有独立的进程树，容器内的进程无法看到或操作宿主机或其他容器的进程。用户命名空间（User Namespace）允许在容器内使用root用户，而在宿主系统上该用户映射为普通用户，从而降低了容器逃逸的风险。

网络命名空间（NET Namespace）为每个容器提供独立的网络栈，包括独立的网络接口、路由表、iptables规则等。挂载命名空间（Mount Namespace）实现了文件系统的隔离，而UTS命名空间则提供主机名和域名的隔离。

IPC命名空间（IPC Namespace）隔离System V IPC对象和POSIX消息队列，确保不同容器间的进程通信互不干扰。Cgroups（Control Groups）则是资源限制的核心机制，它能够精确控制容器可以使用的CPU、内存、磁盘I/O和网络带宽资源。

### 13.1.3 Docker安全配置最佳实践

Docker提供了多种配置选项来增强安全性。首先，在Docker守护进程配置层面，应该启用TLS加密的远程API访问，使用--tlsverify、--tlscacert、--tlscert和--tlskey参数配置TLS证书。守护进程应该以非root用户运行，并限制其监听范围。

编辑Docker守护进程配置文件 `/etc/docker/daemon.json`，可以启用多项安全特性：

```json
{
    "icc": false,
    "userns-remap": "default",
    "selinux-enabled": true,
    "userns-remap": "default",
    "live-restore": true,
    "no-new-privileges": true,
    "default-ulimits": {
        "nofile": {
            "Name": "nofile",
            "Hard": 64000,
            "Soft": 64000
        }
    },
    "storage-driver": "overlay2",
    "log-driver": "json-file",
    "log-opts": {
        "max-size": "10m",
        "max-file": "3"
    }
}
```

配置中的 `no-new-privileges` 选项尤为重要，它阻止容器内的进程通过setuid程序获取更高权限。`live-restore` 确保Docker守护进程更新时容器继续运行。存储驱动推荐使用overlay2，它相比aufs具有更好的性能和安全性。

## 13.2 Docker镜像安全

### 13.2.1 镜像来源管理与签名

镜像安全是容器安全的第一道防线。使用来自不可信来源的镜像可能导致恶意软件植入、后门程序或已知漏洞的传播。因此，建立严格的镜像来源管理机制至关重要。

生产环境中，应该搭建私有镜像仓库，如Harbor或GitLab Container Registry。私有仓库支持基于角色的访问控制（RBAC）、镜像签名和验证、以及漏洞扫描等功能。所有生产镜像必须从私有仓库获取，严禁使用未经审核的外部镜像。

镜像签名使用Docker Content Trust机制，它基于The Update Framework（TUF）来确保镜像的完整性和发布者身份。启用Content Trust后，只有经过签名的镜像才能被拉取：

```bash
# 启用Docker Content Trust
export DOCKER_CONTENT_TRUST=1
export DOCKER_CONTENT_TRUST_SERVER=https://your-registry:4443

# 拉取镜像时会自动验证签名
docker pull your-registry.com/app:latest
```

可以使用 `docker trust` 命令手动管理镜像签名：

```bash
# 为镜像添加签名
docker trust sign your-registry.com/app:v1

# 查看镜像的签名信息
docker trust inspect your-registry.com/app:v1

# 撤销镜像签名
docker trust revoke your-registry.com/app:v1
```

### 13.2.2 最小化镜像构建

最小化镜像是减少攻击面的重要手段。一个精简的镜像只包含应用运行所必需的软件包和依赖，减小了潜在漏洞的数量和攻击面。

首先，应该选择合适的基础镜像。官方维护的镜像通常经过安全审计，但体积较大。对于需要更小体积的场景，可以使用Alpine Linux镜像，它是一个专门为容器设计的轻量级发行版：

```dockerfile
# 不推荐：使用完整系统镜像
FROM ubuntu:22.04

# 推荐：使用Alpine镜像
FROM alpine:3.18

# 推荐：使用多阶段构建
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
RUN go build -o myapp

FROM alpine:3.18
COPY --from=builder /app/myapp /usr/local/bin/
CMD ["myapp"]
```

多阶段构建是创建小体积镜像的最佳实践。它允许在第一个阶段使用完整的构建环境来编译程序，然后在第二个阶段只将编译产物复制到精简的运行时镜像中：

```dockerfile
# 第一阶段：构建
FROM maven:3.9 AS maven-build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

# 第二阶段：运行
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=maven-build /app/target/myapp.jar .
EXPOSE 8080
USER nobody
ENTRYPOINT ["java", "-jar", "myapp.jar"]
```

在Dockerfile中，还应该遵循以下最佳实践：避免使用root用户运行应用、使用COPY替代ADD、合并多个RUN指令减少镜像层数、以及合理使用.dockerignore文件排除敏感文件。

### 13.2.3 镜像漏洞扫描

镜像漏洞扫描是发现和修复已知安全漏洞的重要手段。常用的镜像扫描工具包括Trivy、Clair和Anchore等。

Trivy是由Aqua Security开发的开源镜像扫描工具，使用简单且扫描速度快：

```bash
# 安装Trivy
apt-get install wget
wget https://github.com/aquasecurity/trivy/releases/download/v0.48.0/trivy_0.48.0_Linux-64bit.deb
dpkg -i trivy_0.48.0_Linux-64bit.deb

# 扫描镜像
trivy image nginx:1.24
trivy image --severity HIGH,CRITICAL your-registry.com/app:v1

# 扫描文件系统
trivy fs /path/to/project

# 生成HTML报告
trivy image --format html --output report.html nginx:1.24
```

Trivy支持多种输出格式，可以集成到CI/CD流水线中：

```bash
# JSON格式输出，便于程序解析
trivy image --format json --output results.json your-registry.com/app:v1

# 在CI中失败于发现高危漏洞
trivy image --exit-code 1 --severity CRITICAL your-registry.com/app:v1
```

Clair是CoreOS开发的深度镜像分析工具，支持自动更新漏洞数据库：

```bash
# 使用Docker Compose启动Clair
# docker-compose.yml
version: '3'
services:
  clair:
    image: quay.io/coreos/clair:v4.8.0
    volumes:
      - clair-db:/var/lib/postgresql/data
    environment:
      - POSTGRES_PASSWORD=password
    ports:
      - "6060:6060"
      - "5432:5432"
```

```bash
# 使用Clair API进行扫描
curl -X POST http://localhost:6060/api/v1/scan \
  -d '{"image": "your-registry.com/app:v1"}'
```

## 13.3 Docker容器加固

### 13.3.1 容器运行时安全配置

容器运行时的安全配置是防止容器被攻击和限制容器逃逸影响范围的关键。Docker提供了丰富的运行时安全选项，合理配置这些选项可以显著提升容器环境的安全性。

首先，强烈建议在运行容器时添加 `--read-only` 选项，将容器的根文件系统设置为只读模式：

```bash
docker run --read-only -d myapp
```

对于需要写入的目录，应该通过 volumes 显式挂载。对于临时文件，可以使用 tmpfs 挂载：

```bash
docker run --read-only --tmpfs /tmp:rw,noexec,nosuid,size=64m myapp
```

`--no-new-privileges` 选项阻止容器内的进程通过setuid程序获取更高权限：

```bash
docker run --no-new-privileges true myapp
```

用户命名空间重映射可以将容器内的root用户映射为宿主机的普通用户，增强容器逃逸难度：

```bash
# 在/etc/docker/daemon.json中配置
{
    "userns-remap": "default"
}

# 重启Docker后运行容器
docker run --userns=host myapp
```

### 13.3.2 Linux Capabilities精细控制

Linux的Capabilities机制将传统的超级用户权限分解为多个独立单元，容器运行时可以根据需要精确地授予或限制特定权限。默认情况下，Docker容器只拥有有限的 Capabilities。

查看容器默认拥有的 Capabilities：

```bash
docker run --rm alpine capsh --print
```

移除所有 Capabilities 后按需添加是最佳实践：

```bash
# 移除所有Capabilities
docker run --cap-drop ALL myapp

# 添加必要的Capabilities（如需要绑定低端口）
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE -p 80:80 myapp

# 仅保留网络CAP_NET_RAW和CAP_NET_ADMIN
docker run --cap-drop ALL --cap-add NET_RAW --cap-add NET_ADMIN myapp
```

常用Capabilities说明：

| Capability | 说明 | 使用场景 |
|------------|------|----------|
| CAP_NET_BIND_SERVICE | 绑定到低于1024的端口 | Web服务绑定80端口 |
| CAP_NET_RAW | 使用原始套接字 | 网络诊断工具 |
| CAP_NET_ADMIN | 网络管理操作 | 修改网络配置 |
| CAP_SYS_CHROOT | 使用chroot | 某些应用需要 |
| CAP_SYS_ADMIN | 系统管理权限 | 挂载文件系统等 |

### 13.3.3 Seccomp与AppArmor配置

Seccomp（Secure Computing Mode）是Linux内核提供的安全机制，用于限制进程可用的系统调用。Docker默认提供了一个默认的Seccomp配置文件，阻止了约44个不常用的系统调用。

查看Docker默认的Seccomp配置文件：

```bash
cat /etc/docker/seccomp.json
```

自定义Seccomp配置文件：

```json
{
    "defaultAction": "SCMP_ACT_ERRNO",
    "syscalls": [
        {
            "names": ["read", "write", "exit", "exit_group", "fstat"],
            "action": "SCMP_ACT_ALLOW"
        },
        {
            "names": ["mkdir", "rmdir"],
            "action": "SCMP_ACT_ALLOW",
            "args": [
                {
                    "index": 0,
                    "value": 0,
                    "op": "SCMP_CMP_EQ"
                }
            ]
        }
    ]
}
```

应用自定义Seccomp配置：

```bash
docker run --security-opt seccomp=/path/to/seccomp.json myapp
```

AppArmor是另一个重要的安全模块，它提供基于路径的强制访问控制。Docker会自动为每个容器生成AppArmor配置文件：

```bash
# 查看容器的AppArmor状态
docker inspect --format '{{ .AppArmorProfile }}' container_name

# 使用自定义AppArmor profile
docker run --security-opt apparmor=/etc/apparmor.d/your-profile myapp
```

创建自定义AppArmor profile：

```bash
# /etc/apparmor.d/containers/myapp
#include <tunables/global>
profile myapp flags=(attach_disconnected,mediate_deleted) {
  #include <abstractions/base>
  
  #允许读取系统库
  /usr/lib/** r,
  /lib/** r,
  
  #允许读取应用目录
  /opt/myapp/** r,
  
  #允许写入应用目录
  /opt/myapp/logs/** aw,
  
  #网络权限
  network tcp,
  bind 0.0.0.0:8080,
}
```

## 13.4 Docker网络配置

### 13.4.1 网络隔离与桥接

Docker提供了多种网络模式，每种模式都有不同的安全特性。默认情况下，Docker创建一个bridge网络，容器通过NAT连接到宿主机网络。生产环境中，应该根据应用需求选择合适的网络模式。

Docker网络模式：

- **bridge**：默认模式，容器连接到Docker创建的桥接网络
- **host**：容器直接使用宿主机网络，消除网络隔离
- **overlay**：跨Docker守护进程的网络，用于Docker Swarm
- **macvlan**：为容器分配MAC地址，使其成为网络上的真实设备
- **none**：禁用容器的网络功能

查看当前Docker网络：

```bash
docker network ls
docker network inspect bridge
```

创建自定义桥接网络并启用网络隔离：

```bash
# 创建内部网络
docker network create --driver bridge --internal myapp-network

# 运行容器连接到内部网络
docker run --network myapp-network --name database mysql:8

# 运行应用容器
docker run --network myapp-network --name app myapp:latest
```

配置网络访问控制，限制容器对外部网络的访问：

```bash
# 禁止容器访问外部网络（但允许内部通信）
docker network create --driver bridge \
  --gateway 172.20.0.1 \
  --subnet 172.20.0.0/16 \
  isolated-network

# 容器无法直接访问外网，需要通过NAT代理
```

### 13.4.2 网络流量加密

在分布式系统中，容器间的网络通信可能涉及敏感数据，应该使用TLS/SSL加密。Docker Swarm默认使用相互认证的TLS来保护管理平面通信。

对于使用overlay网络的服务，启用加密：

```bash
# 创建加密的overlay网络
docker network create -d overlay --opt encrypted my-encrypted-network

# 验证网络加密
docker network inspect my-encrypted-network
```

对于应用层的加密，可以使用Docker Secrets来管理证书：

```bash
# 创建TLS证书secret
docker secret create server-cert.pem /path/to/server-cert.pem
docker secret create server-key.pem /path/to/server-key.pem

# 在服务中使用secret
docker service create \
  --name secure-app \
  --secret server-cert.pem \
  --secret server-key.pem \
  -p 443:443 \
  myapp:latest
```

应用程序通过挂载的secret文件读取证书：

```python
# 示例：Python应用读取TLS证书
import os

cert_path = '/run/secrets/server-cert.pem'
key_path = '/run/secrets/server-key.pem'

if os.path.exists(cert_path):
    app.config['SSL_CERT'] = cert_path
    app.config['SSL_KEY'] = key_path
```

### 13.4.3 防火墙规则配置

合理的iptables规则可以增强Docker网络的安全性。Docker会自动生成一些iptables规则，但这些规则可能不够严格，需要根据实际需求进行优化。

查看Docker生成的iptables规则：

```bash
iptables -L -n -v | grep DOCKER
iptables -L DOCKER -n -v
```

创建自定义iptables规则限制容器网络访问：

```bash
# 允许已建立的连接
iptables -A FORWARD -i docker0 -m state --state RELATED,ESTABLISHED -j ACCEPT

# 禁止容器访问宿主机上的其他服务
iptables -A FORWARD -i docker0 -d 172.17.0.1/32 -j DROP

# 限制容器可以访问的外部IP范围
iptables -A FORWARD -i docker0 -d 10.0.0.0/8 -j ACCEPT
iptables -A FORWARD -i docker0 -d 0.0.0.0/0 -j DROP

# 记录被阻止的连接（用于审计）
iptables -A FORWARD -i docker0 -m limit --limit 5/min -j LOG --log-prefix "DOCKER_BLOCKED: "
```

创建 `/etc/systemd/system/docker.service.d/no-masquerade.conf` 禁用Docker的自动 masquerade：

```ini
[Service]
ExecStart=
ExecStart=/usr/bin/dockerd --iptables=false
```

然后手动添加更严格的规则：

```bash
# 刷新Docker的iptables规则
systemctl restart docker

# 添加自定义规则
iptables -I DOCKER-USER -s 10.0.0.0/8 -j ACCEPT
iptables -I DOCKER-USER -d 192.168.0.0/16 -j ACCEPT
iptables -I DOCKER-USER -j DROP
```

## 13.5 Docker存储安全

### 13.5.1 存储卷安全管理

Docker存储卷是容器持久化数据的推荐方式，但不当的配置可能导致数据泄露或被未授权访问。理解存储卷的安全机制对于保护敏感数据至关重要。

Docker支持多种存储卷类型：

- **bind mount**：挂载宿主文件系统路径
- **named volume**：Docker管理的命名卷
- **tmpfs**：存储在内存中的临时文件系统
- **npipe**：Windows命名管道

对于敏感数据，应使用只读挂载或限制访问权限：

```bash
# 只读挂载敏感配置
docker run -v /path/to/config:/etc/secrets:ro myapp

# 使用tmpfs存储敏感数据（不持久化到磁盘）
docker run --tmpfs /run/secrets:rw,noexec,nosuid,size=64m myapp
```

创建只读存储卷：

```bash
# 创建只读卷
docker volume create --label env=prod --label data=confidential secure-volume

# 使用只读卷
docker run -v secure-volume:/data:ro myapp
```

定期清理不再使用的卷：

```bash
# 列出未使用的卷
docker volume ls -f dangling=true

# 清理未使用的卷
docker volume prune

# 删除指定卷
docker volume rm volume_name
```

### 13.5.2 数据加密存储

对于高度敏感的数据，应该在存储层进行加密。Docker企业版提供了透明数据加密（DTR）功能，开源版本可以使用LUKS或其他加密方案。

使用LUKS加密存储卷：

```bash
# 创建加密容器
dd if=/dev/zero of=/secure_volume bs=1M count=100
cryptsetup luksFormat /secure_volume
cryptsetup open /secure_volume secure_volume
mkfs.ext4 /dev/mapper/secure_volume
mkdir -p /mnt/secure
mount /dev/mapper/secure_volume /mnt/secure

# 挂载到容器
docker run -v /mnt/secure:/data myapp
```

使用GPG加密文件：

```bash
# 加密敏感文件
gpg --symmetric --cipher-algo AES256 sensitive_data.json

# 在容器中解密使用
docker run -v $(pwd)/sensitive_data.json.gpg:/data/sensitive_data.gpg myapp \
  sh -c "gpg --decrypt /data/sensitive_data.gpg | app"
```

### 13.5.3 存储卷权限控制

合理设置存储卷的权限，防止容器越权访问：

```bash
# 创建专用用户和用户组
groupadd -g 1000 appgroup
useradd -r -u 1000 -g appgroup appuser

# 设置目录权限
chown -R appuser:appgroup /data/app
chmod -R 750 /data/app

# 运行容器使用指定用户
docker run -u 1000:1000 -v /data/app:/data myapp
```

在Dockerfile中创建非root用户：

```dockerfile
FROM alpine:3.18

RUN addgroup -g 1000 appgroup && \
    adduser -u 1000 -G appgroup -D -s /bin/sh appuser

COPY --chown=appuser:appgroup app /home/appuser/app

USER appuser
CMD ["./app"]
```

## 13.6 Docker资源限制

### 13.6.1 内存限制

内存限制是防止容器耗尽宿主机资源的关键措施。可以通过 `--memory` 和 `--memory-swap` 等参数进行配置。

```bash
# 设置内存限制为512MB
docker run -m 512m myapp

# 设置内存限制为512MB，swap限制为1GB
docker run -m 512m --memory-swap=1g myapp

# 设置软限制和硬限制
docker run \
  --memory-reservation=256m \
  --memory-swappiness=60 \
  -m 512m \
  myapp
```

常用内存限制参数说明：

| 参数 | 说明 |
|------|------|
| -m, --memory | 容器可以使用的最大内存 |
| --memory-swap | 允许使用的swap总量 |
| --memory-reservation | 软限制，低于此值时可能被回收 |
| --memory-swappiness | swap使用比例（0-100） |
| --oom-kill-disable | 禁用OOM killer |
| --oom-score-adj | OOM优先级调整 |

创建默认的内存限制配置，在 `/etc/docker/daemon.json` 中设置：

```json
        "memlock": {
            "Name": "memlock",
            "Hard": -1,
            "Soft": -1
        }
    },
    "default-memory": "512m",
    "default-memory-reservation": "256m"
}
```

## 13.8 Kubernetes安全基础

### 13.8.1 K8s安全架构

Kubernetes作为容器编排平台，其安全架构涉及多个层面：节点安全、网络策略、认证授权、Pod安全、Secrets管理、以及数据加密等。理解K8s的安全架构对于构建安全的容器平台至关重要。

K8s安全架构的核心组件：

- **API Server**：集群的中央管理平面，所有操作都通过API Server进行
- **etcd**：存储集群所有数据的键值存储，必须严格保护
- **Kubelet**：运行在每个节点上的代理，负责管理容器
- **Kube-proxy**：网络代理，负责服务发现和负载均衡
- **Container Runtime**：实际运行容器的软件（Docker/containerd等）

K8s安全加固的总体策略：

1. **最小权限原则**：只授予必要的权限
2. **纵深防御**：多层安全防护
3. **零信任网络**：不信任任何网络请求
4. **安全默认值**：使用最安全的默认配置
5. **持续监控**：实时监控安全事件

### 13.8.2 K8s认证与授权

Kubernetes支持多种认证机制，用于验证API请求的来源：

**静态令牌认证**：

```yaml
# 创建ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
  namespace: production

---
# 创建Secret用于存储令牌
apiVersion: v1
kind: Secret
metadata:
  name: app-sa-token
  namespace: production
  annotations:
    kubernetes.io/service-account.name: app-sa
type: kubernetes.io/service-account-token
```

**X.509证书认证**：

```bash
# 为用户创建私钥
openssl genrsa -out user.key 2048

# 创建证书签名请求
openssl req -new -key user.key -out user.csr -subj "/CN=user01/O=developers"

# 使用Kubernetes CA签署证书
openssl x509 -req -in user.csr -CA /etc/kubernetes/pki/ca.crt \
  -CAkey /etc/kubernetes/pki/ca.key -CAcreateserial \
  -out user.crt -days 365

# 在kubeconfig中添加用户
kubectl config set-credentials user01 \
  --client-certificate=user.crt \
  --client-key=user.key
```

**RBAC授权**：

```yaml
# 创建Role定义命名空间内的权限
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production
  name: app-reader
rules:
- apiGroups: [""]
  resources: ["pods", "services"]
  verbs: ["get", "list", "watch"]
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list"]

---
# 将Role绑定到用户
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-reader-binding
  namespace: production
subjects:
- kind: User
  name: user01
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: app-reader
  apiGroup: rbac.authorization.k8s.io
```

**ClusterRole和ClusterRoleBinding**（集群级别权限）：

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: node-reader
rules:
- apiGroups: [""]
  resources: ["nodes"]
  verbs: ["get", "list", "watch"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: node-reader-binding
subjects:
- kind: Group
  name: system:nodes

  kind: ClusterRole
```

## 13.9 K8s网络安全策略

### 13.9.1 NetworkPolicy基础

NetworkPolicy是Kubernetes提供的网络策略资源，用于控制Pod之间的网络流量。通过NetworkPolicy，可以实现微服务之间的网络隔离，阻止未授权的访问。

创建默认拒绝策略（先禁止所有流量，再按需允许）：

```yaml
# 命名空间级别的默认拒绝
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
```

允许特定Pod之间的通信：

```yaml
# 允许frontend访问backend
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080

---
# 允许backend访问数据库
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-backend-to-db
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: database
  policyTypes:
  - Egress
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: backend
    ports:
    - protocol: TCP
      port: 5432
```

### 13.9.2 服务网格安全

服务网格（Service Mesh）提供了更高级别的网络安全功能，如mTLS加密、智能流量管理和零信任网络。

**Istio** 是最流行的服务网格解决方案之一：

```yaml
# 启用mTLS的PeerAuthentication策略
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: production
spec:
  mtls:
    mode: STRICT
```

```yaml
# 授权策略
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: backend-auth
  namespace: production
spec:
  selector:
    matchLabels:
      app: backend
  rules:
  - from:
    - source:
        principals: ["cluster.local/ns/production/sa/frontend"]
    to:
    - operation:
        ports: ["8080"]
```

**Linkerd** 是另一个轻量级的服务网格选择：

```yaml
# Linkerd mTLS配置
apiVersion: linkerd.io/v1alpha2
kind: Server
metadata:
  name: backend-server
  namespace: production
spec:
  proxyProtocol: HTTP/2
  port: 8080
  targetRef:
    kind: Service
    name: backend
---
apiVersion: linkerd.io/v1alpha2
kind: ServerAuthorization
metadata:
  name: backend-auth
  namespace: production
spec:
  server:
    name: backend-server
  client:
    - serviceAccountRefs:
      - name: frontend
```

## 13.10 K8s Secrets管理

### 13.10.1 Secrets基本使用

Kubernetes Secrets用于存储敏感信息，如密码、OAuth令牌和SSH密钥。与ConfigMap类似，但专门用于敏感数据。

创建Secrets的几种方式：

```bash
# 从文件创建
kubectl create secret generic db-credentials \
  --from-file=username=/path/to/username.txt \
  --from-file=password=/path/to/password.txt

# 从字面值创建
kubectl create secret generic api-key \
  --from-literal=API_KEY=your-secret-key

# 从环境变量文件创建
kubectl create secret generic env-secrets \
  --from-env-file=/path/to/env-file

# 从TLS证书创建
kubectl create secret tls tls-secret \
  --cert=/path/to/tls.crt \
  --key=/path/to/tls.key
```

在Pod中使用Secrets：

```yaml
# 通过环境变量引用
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
  namespace: production
spec:
  containers:
  - name: app
    image: myapp:latest
    env:
    - name: DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: username
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: password
```

## 13.13 容器逃逸防护

### 13.13.1 容器逃逸风险类型

容器逃逸是指攻击者从容器内部突破隔离限制，访问宿主机或其他容器的资源。常见的逃逸手法包括：

**特权容器逃逸**：特权容器拥有近乎完整的宿主系统权限，攻击者可能通过设备访问实现逃逸：

```bash
# 检查容器是否为特权容器
docker inspect --format='{{.HostConfig.Privileged}}' container_name
```

防护措施：永远不要运行特权容器

```bash
# 使用--privileged=false明确禁止
docker run --privileged=false myapp
```

**共享命名空间逃逸**：错误配置导致容器共享宿主机的命名空间：

```bash
# 检查是否共享了宿主机网络
docker inspect --format='{{.HostConfig.NetworkMode}}' container_name

# 检查是否共享了宿主机PID
docker inspect --format='{{.HostConfig.PidMode}}' container_name
```

**危险挂载点逃逸**：将宿主系统目录直接挂载到容器：

```bash
# 检查危险挂载
docker inspect --format='{{range .Mounts}}{{.Source}}:{{.Destination}} {{end}}' container_name
```

### 13.13.2 容器逃逸防护措施

**启用用户命名空间**：

```bash
# /etc/docker/daemon.json
{
    "userns-remap": "default"
}
```

**限制容器能力**：

```bash
# 运行时强制限制
docker run --cap-drop ALL --read-only --no-new-privileges myapp
```

**配置Seccomp过滤**：

```bash
# 使用默认Seccomp配置
docker run --security-opt seccomp=default myapp
```

**AppArmor/SELinux强制访问控制**：

```bash
# 启用SELinux
docker run --security-opt label=type:container_file_t myapp
```

**定期安全扫描**：使用工具扫描容器镜像和配置中的安全风险：

```bash
# 使用Trivy扫描
trivy image --security-checks vuln,config myapp:latest

# 使用docker-bench-security检查
./docker-bench-security.sh
```

### 13.14.2 漏洞修复方案

**方案一：升级基础镜像**

最直接的修复方式是使用包含安全更新的新版本基础镜像：

```dockerfile
# 原始Dockerfile
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y nginx
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

# 修复后的Dockerfile
FROM ubuntu:24.04
RUN apt-get update && apt-get install -y nginx
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

**方案二：使用Alpine等更安全的基础镜像**

```dockerfile
FROM alpine:3.19
RUN apk add nginx
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

**方案三：在构建时修复已知漏洞**

```dockerfile
FROM ubuntu:22.04
RUN apt-get update && \
    apt-get install -y --only-upgrade \
        openssl \
        libssl3 \
        nginx \
        zlib && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```
