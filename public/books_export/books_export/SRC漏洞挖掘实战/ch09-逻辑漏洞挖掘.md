# 第九章 逻辑漏洞挖掘

## 9.1 逻辑漏洞概述

### 9.1.1 什么是逻辑漏洞

逻辑漏洞是指由于业务逻辑设计缺陷或实现错误导致的安全漏洞。与传统的技术漏洞（如SQL注入、XSS）不同，逻辑漏洞不依赖于代码层面的技术缺陷，而是源于业务流程、权限控制、数据处理等方面的设计问题。

**通俗理解**

想象一个电商网站的购物流程：用户选择商品、加入购物车、填写地址、支付、完成订单。如果这个流程中某个环节的逻辑设计有问题，比如支付时只验证了金额是否大于0，而没有验证是否等于商品价格，用户就可以用1元购买100元的商品。这就是逻辑漏洞。

**逻辑漏洞与技术漏洞的区别**

| 特征 | 逻辑漏洞 | 技术漏洞 |
|-----|---------|---------|
| 成因 | 业务设计缺陷 | 代码实现错误 |
| 检测方式 | 业务流程分析 | 技术测试工具 |
| 防护方式 | 业务逻辑完善 | 输入输出过滤 |
| 发现难度 | 需要理解业务 | 相对容易 |
| 危害程度 | 可能很严重 | 取决于漏洞类型 |

### 9.1.2 逻辑漏洞的危害

逻辑漏洞可能导致以下严重后果：

1. **经济损失**：低价购买高价商品、绕过支付、重复退款
2. **账户安全**：绕过密码验证、重置他人密码、账户接管
3. **信息泄露**：获取他人订单信息、查看敏感数据
4. **权限提升**：绕过权限检查、获取管理员权限
5. **业务破坏**：破坏业务流程、影响正常运营

**真实案例**

2019年，某电商平台存在支付逻辑漏洞，攻击者通过修改支付金额参数，以0.01元购买价值数千元的商品，造成平台直接经济损失超过100万元。

### 9.1.3 逻辑漏洞的分类

**按业务场景分类**

| 类型 | 说明 | 常见场景 |
|-----|------|---------|
| 支付逻辑漏洞 | 支付流程缺陷 | 金额篡改、绕过支付 |
| 订单逻辑漏洞 | 订单处理缺陷 | 订单篡改、重复下单 |
| 认证逻辑漏洞 | 身份验证缺陷 | 密码重置、绕过验证 |
| 权限逻辑漏洞 | 权限控制缺陷 | 越权访问、权限提升 |
| 数据逻辑漏洞 | 数据处理缺陷 | 数据篡改、信息泄露 |
| 流程逻辑漏洞 | 业务流程缺陷 | 条件竞争、流程绕过 |

**按漏洞成因分类**

```text
1. 参数篡改：修改请求参数绕过验证
2. 条件竞争：并发操作导致逻辑错误
3. 流程绕过：跳过必要步骤完成操作
4. 验证缺失：缺少必要的验证环节
5. 边界错误：边界条件处理不当
6. 状态错误：状态转换逻辑缺陷
```

## 9.2 支付逻辑漏洞

### 9.2.1 支付流程分析

**典型支付流程**

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 选择商品 │───▶│ 加入购物车│───▶│ 生成订单 │───▶│ 发起支付 │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
                                                    │
                    ┌───────────────────────────────┘
                    ▼
               ┌─────────┐    ┌─────────┐    ┌─────────┐
               │ 支付验证 │───▶│ 支付完成 │───▶│ 订单完成 │
               └─────────┘    └─────────┘    └─────────┘
```

**支付流程关键点**

```text
1. 商品价格验证：支付金额是否等于商品价格
2. 数量验证：购买数量是否合法
3. 库存验证：库存是否充足
4. 用户身份验证：是否为合法用户
5. 支付状态验证：支付是否真正完成
6. 优惠券验证：优惠券是否有效、是否过期
```

### 9.2.2 金额篡改漏洞

**漏洞原理**

支付时，服务器未正确验证支付金额，攻击者可以修改金额参数，以低价购买高价商品。

**漏洞场景**

```php
<?php
// 不安全的支付处理
$amount = $_POST['amount'];  // 直接使用用户提交的金额
$order_id = $_POST['order_id'];

// 发起支付
$payment_result = initiate_payment($amount, $order_id);

if ($payment_result['status'] == 'success') {
    // 标记订单为已支付
    update_order_status($order_id, 'paid');
}
?>
```

**攻击方式**

```http
POST /api/payment HTTP/1.1
Host: shop.example.com
Cookie: session=abc123
Content-Type: application/x-www-form-urlencoded

order_id=12345&amount=0.01

# 原订单金额为1000元，修改为0.01元
```

**检测方法**

```text
步骤1：正常下单，记录订单金额
步骤2：抓取支付请求
步骤3：修改金额参数为较小值
步骤4：观察是否支付成功
步骤5：检查订单状态是否变为已支付
```

**修复方案**

```php
<?php
// 安全的支付处理
$order_id = $_POST['order_id'];

// 从数据库获取订单金额，不信任用户输入
$order = get_order_from_db($order_id);
$amount = $order['amount'];

// 验证订单状态
if ($order['status'] != 'pending') {
    die('订单状态异常');
}

// 验证用户权限
if ($order['user_id'] != $_SESSION['user_id']) {
    die('无权操作此订单');
}

// 发起支付
$payment_result = initiate_payment($amount, $order_id);

// 验证支付结果
if ($payment_result['status'] == 'success') {
    // 验证支付金额
    if ($payment_result['paid_amount'] != $amount) {
        die('支付金额异常');
    }
    update_order_status($order_id, 'paid');
}
?>
```

### 9.2.3 优惠券滥用漏洞

**漏洞原理**

优惠券使用逻辑存在缺陷，攻击者可以重复使用优惠券、使用过期优惠券、使用他人优惠券。

**漏洞场景**

```php
<?php
// 不安全的优惠券使用
$coupon_code = $_POST['coupon_code'];
$order_id = $_POST['order_id'];

// 应用优惠券
apply_coupon($coupon_code, $order_id);

// 未验证优惠券是否已使用
// 未验证优惠券是否过期
// 未验证优惠券是否属于当前用户
?>
```

**攻击方式**

```text
攻击1：重复使用优惠券
- 同一优惠券在多个订单中使用

攻击2：使用过期优惠券
- 修改请求中的优惠券有效期参数

攻击3：使用他人优惠券
- 获取他人优惠券代码后使用

攻击4：优惠券叠加
- 同时使用多个本不应叠加的优惠券
```

**检测方法**

```text
步骤1：获取一个优惠券
步骤2：使用优惠券下单
步骤3：尝试再次使用同一优惠券
步骤4：观察是否成功应用

步骤5：获取过期优惠券
步骤6：尝试使用过期优惠券
步骤7：观察是否成功应用
```

**修复方案**

```php
<?php
// 安全的优惠券使用
$coupon_code = $_POST['coupon_code'];
$order_id = $_POST['order_id'];
$user_id = $_SESSION['user_id'];

// 获取优惠券信息
$coupon = get_coupon_from_db($coupon_code);

// 验证优惠券是否存在
if (!$coupon) {
    die('优惠券不存在');
}

// 验证优惠券是否过期
if (strtotime($coupon['expire_time']) < time()) {
    die('优惠券已过期');
}

// 验证优惠券是否已使用
if ($coupon['used'] == 1) {
    die('优惠券已使用');
}

// 验证优惠券归属
if ($coupon['user_id'] != $user_id) {
    die('无权使用此优惠券');
}

// 验证优惠券适用范围
if (!check_coupon_applicable($coupon, $order_id)) {
    die('优惠券不适用于此订单');
}

// 应用优惠券
apply_coupon($coupon_code, $order_id);

// 标记优惠券已使用
mark_coupon_used($coupon_code, $order_id);
?>
```

### 9.2.4 绕过支付漏洞

**漏洞原理**

支付流程中存在可跳过的环节，攻击者可以绕过实际支付直接完成订单。

**漏洞场景**

```php
<?php
// 不安全的订单完成处理
$order_id = $_GET['order_id'];

// 直接标记订单为已支付，未验证支付状态
update_order_status($order_id, 'paid');
?>
```

**攻击方式**

```text
攻击1：直接调用支付成功接口
- 支付成功后服务器会调用某个接口
- 攻击者直接调用该接口

攻击2：修改支付状态参数
- 请求中包含支付状态参数
- 攻击者修改为已支付状态

攻击3：利用支付回调漏洞
- 支付平台的回调验证不严
- 攻击者伪造回调请求
```

**检测方法**

```text
步骤1：正常下单，不进行支付
步骤2：分析支付成功后的请求
步骤3：尝试直接调用支付成功接口
步骤4：观察订单是否变为已支付状态
```

**支付回调伪造**

```http
POST /api/payment/callback HTTP/1.1
Host: shop.example.com
Content-Type: application/json

{
    "order_id": "12345",
    "payment_status": "success",
    "transaction_id": "fake_transaction_123"
}

# 伪造支付平台回调，未验证签名
```

**修复方案**

```php
<?php
// 安全的支付回调处理
$callback_data = json_decode(file_get_contents('php://input'), true);

// 验证签名
$signature = $callback_data['signature'];
$expected_signature = calculate_signature($callback_data, $secret_key);

if ($signature !== $expected_signature) {
    die('签名验证失败');
}

// 获取订单信息
$order_id = $callback_data['order_id'];
$order = get_order_from_db($order_id);

// 验证订单状态
if ($order['status'] == 'paid') {
    die('订单已支付');
}

// 验证支付金额
if ($callback_data['amount'] != $order['amount']) {
    die('支付金额不匹配');
}

// 验证交易ID唯一性
if (check_transaction_used($callback_data['transaction_id'])) {
    die('交易ID已使用');
}

// 更新订单状态
update_order_status($order_id, 'paid');
record_transaction($callback_data['transaction_id'], $order_id);
?>
```

### 9.2.5 负数金额漏洞

**漏洞原理**

支付金额验证不严格，允许负数金额，可能导致退款逻辑错误或账户余额异常。

**漏洞场景**

```php
<?php
// 不安全的金额处理
$amount = $_POST['amount'];

// 未验证金额是否为正数
if ($amount > 0) {  // 只验证大于0，负数可能绕过其他逻辑
    process_payment($amount);
}

// 充值逻辑
$balance = get_user_balance($user_id);
$new_balance = $balance + $amount;  // 负数会导致余额减少
update_user_balance($user_id, $new_balance);
?>
```

**攻击方式**

```text
攻击1：负数充值
- 充值金额设置为负数
- 导致账户余额减少而非增加

攻击2：负数购买
- 购买数量设置为负数
- 可能导致库存增加或账户余额增加

攻击3：负数退款
- 退款金额设置为负数
- 可能导致反向退款
```

**检测方法**

```text
步骤1：找到充值或支付功能
步骤2：修改金额为负数
步骤3：观察账户余额变化
步骤4：检查是否出现异常
```

**修复方案**

```php
<?php
// 安全的金额处理
$amount = $_POST['amount'];

// 验证金额为正数且在合理范围
if (!is_numeric($amount) || $amount <= 0 || $amount > MAX_AMOUNT) {
    die('金额参数异常');
}

// 验证金额精度（防止浮点数精度问题）
$amount = round($amount, 2);

// 使用整数处理金额（以分为单位）
$amount_in_cents = intval($amount * 100);
?>
```

### 9.2.6 支付逻辑漏洞实战案例

**案例：某电商平台支付金额篡改**

**漏洞发现**

```text
1. 正常下单：商品价格1000元
2. 抓取支付请求：
   POST /api/pay
   order_id=12345&amount=1000&payment_method=alipay
   
3. 修改金额参数：
   order_id=12345&amount=1&payment_method=alipay
   
4. 支付成功，订单状态变为已支付
5. 实际支付1元，获得1000元商品
```

**漏洞分析**

```php
// 后端代码
$amount = $_POST['amount'];  // 直接使用用户输入的金额
$order_id = $_POST['order_id'];

// 未从数据库验证订单金额
$payment = create_payment($order_id, $amount);
```

**PoC构造**

```python
import requests

def exploit_payment():
    session = requests.Session()
    session.cookies.set('session', 'victim_session')
    
    # 创建订单
    order_response = session.post('https://shop.example.com/api/create-order', data={
        'product_id': '1001',
        'quantity': '1'
    })
    order_id = order_response.json()['order_id']
    
    # 正常订单金额1000元
    print(f"订单ID: {order_id}, 正常金额: 1000元")
    
    # 修改支付金额为1元
    pay_response = session.post('https://shop.example.com/api/pay', data={
        'order_id': order_id,
        'amount': '1',  # 修改为1元
        'payment_method': 'alipay'
    })
    
    if pay_response.json()['status'] == 'success':
        print("支付成功！实际支付1元获得1000元商品")
```

## 9.3 订单逻辑漏洞

### 9.3.1 订单流程分析

**典型订单流程**

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 创建订单 │───▶│ 订单待支付│───▶│ 订单已支付│───▶│ 订单发货 │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
                    │              │              │
                    ▼              ▼              ▼
               ┌─────────┐    ┌─────────┐    ┌─────────┐
               │ 取消订单 │    │ 申请退款 │    │ 订单完成 │
               └─────────┘    └─────────┘    └─────────┘
```

**订单状态转换**

```text
pending    → 待支付
paid       → 已支付
shipping   → 发货中
completed  → 已完成
cancelled  → 已取消
refunded   → 已退款
```

### 9.3.2 订单篡改漏洞

**漏洞原理**

订单信息未正确验证归属，攻击者可以修改他人订单的信息。

**漏洞场景**

```php
<?php
// 不安全的订单修改
$order_id = $_POST['order_id'];
$new_address = $_POST['address'];

// 未验证订单归属
update_order_address($order_id, $new_address);
?>
```

**攻击方式**

```text
攻击1：修改他人订单地址
- 获取他人订单ID
- 修改收货地址为自己的地址
- 获取他人购买的商品

攻击2：修改订单商品
- 修改订单中的商品ID
- 可能获得更高价值的商品

攻击3：修改订单数量
- 修改购买数量
- 可能获得更多商品
```

**检测方法**

```text
步骤1：创建自己的订单，记录订单ID
步骤2：创建另一个用户账户
步骤3：用另一个账户尝试修改第一个订单
步骤4：观察是否成功修改
```

**修复方案**

```php
<?php
// 安全的订单修改
$order_id = $_POST['order_id'];
$user_id = $_SESSION['user_id'];

// 获取订单信息
$order = get_order_from_db($order_id);

// 验证订单归属
if ($order['user_id'] != $user_id) {
    die('无权操作此订单');
}

// 验证订单状态（只能修改待支付状态的订单）
if ($order['status'] != 'pending') {
    die('订单状态不允许修改');
}

// 更新订单地址
update_order_address($order_id, $_POST['address']);
?>
```

### 9.3.3 重复下单漏洞

**漏洞原理**

订单创建逻辑存在缺陷，攻击者可以重复下单获取更多商品或优惠。

**漏洞场景**

```php
<?php
// 不安全的订单创建
$product_id = $_POST['product_id'];
$quantity = $_POST['quantity'];

// 未限制下单次数
// 未验证是否已下单
create_order($product_id, $quantity);
?>
```

**攻击方式**

```text
攻击1：重复下单获取限量商品
- 限量商品每人限购1件
- 攻击者重复下单获取多件

攻击2：重复使用限时优惠
- 限时优惠活动每人限用一次
- 攻击者重复下单使用优惠

攻击3：并发下单绕过库存限制
- 库存仅剩1件
- 攻击者并发下单获取多件
```

**检测方法**

```text
步骤1：找到限购商品
步骤2：正常下单一次
步骤3：尝试再次下单同一商品
步骤4：观察是否成功下单
```

**修复方案**

```php
<?php
// 安全的订单创建
$product_id = $_POST['product_id'];
$quantity = $_POST['quantity'];
$user_id = $_SESSION['user_id'];

// 获取商品信息
$product = get_product_from_db($product_id);

// 验证限购数量
$user_orders = count_user_orders_for_product($user_id, $product_id);
if ($user_orders >= $product['max_per_user']) {
    die('已达到限购数量');
}

// 验证库存（使用事务和锁）
$db->beginTransaction();
$stock = get_product_stock($product_id);
if ($stock < $quantity) {
    $db->rollback();
    die('库存不足');
}
decrease_stock($product_id, $quantity);
$db->commit();

// 创建订单
create_order($user_id, $product_id, $quantity);
?>
```

### 9.3.4 订单状态篡改漏洞

**漏洞原理**

订单状态转换逻辑存在缺陷，攻击者可以直接修改订单状态绕过必要流程。

**漏洞场景**

```php
<?php
// 不安全的订单状态更新
$order_id = $_POST['order_id'];
$new_status = $_POST['status'];

// 未验证状态转换逻辑
update_order_status($order_id, $new_status);
?>
```

**攻击方式**

```text
攻击1：直接将订单标记为已完成
- 未支付订单直接标记为已完成
- 绕过支付流程

攻击2：将已取消订单恢复为待支付
- 已取消订单恢复状态
- 可能获取已释放的库存

攻击3：将已退款订单标记为已完成
- 已退款订单重新完成
- 可能重复获得商品和退款
```

**检测方法**

```text
步骤1：创建订单但不支付
步骤2：抓取订单状态更新请求
步骤3：修改状态参数为已完成
步骤4：观察订单状态是否变化
```

**修复方案**

```php
<?php
// 安全的订单状态更新
$order_id = $_POST['order_id'];
$new_status = $_POST['status'];
$user_id = $_SESSION['user_id'];

// 获取订单信息
$order = get_order_from_db($order_id);

// 验证订单归属
if ($order['user_id'] != $user_id && !is_admin($user_id)) {
    die('无权操作此订单');
}

// 定义允许的状态转换
$allowed_transitions = [
    'pending' => ['paid', 'cancelled'],
    'paid' => ['shipping', 'refunded'],
    'shipping' => ['completed'],
    'completed' => [],
    'cancelled' => [],
    'refunded' => []
];

// 验证状态转换是否合法
if (!in_array($new_status, $allowed_transitions[$order['status']])) {
    die('不允许的状态转换');
}

// 更新订单状态
update_order_status($order_id, $new_status);
?>
```

### 9.3.5 退款逻辑漏洞

**漏洞原理**

退款流程存在缺陷，攻击者可以重复退款、退款金额异常、退款后保留商品。

**漏洞场景**

```php
<?php
// 不安全的退款处理
$order_id = $_POST['order_id'];
$refund_amount = $_POST['refund_amount'];

// 未验证是否已退款
// 未验证退款金额
process_refund($order_id, $refund_amount);
?>
```

**攻击方式**

```text
攻击1：重复退款
- 同一订单申请多次退款
- 每次退款都成功

攻击2：退款金额篡改
- 修改退款金额大于实际支付金额
- 获得超额退款

攻击3：退款后保留商品
- 申请退款但保留已收货商品
- 获得退款和商品

攻击4：部分退款后再次全额退款
- 先申请部分退款
- 再申请全额退款
```

**检测方法**

```text
步骤1：下单并支付
步骤2：申请退款
步骤3：退款成功后再次申请退款
步骤4：观察是否成功退款
```

**修复方案**

```php
<?php
// 安全的退款处理
$order_id = $_POST['order_id'];
$user_id = $_SESSION['user_id'];

// 获取订单信息
$order = get_order_from_db($order_id);

// 验证订单归属
if ($order['user_id'] != $user_id) {
    die('无权操作此订单');
}

// 验证订单状态
if ($order['status'] != 'paid' && $order['status'] != 'shipping') {
    die('订单状态不允许退款');
}

// 验证是否已退款
if ($order['refund_status'] == 'refunded') {
    die('订单已退款');
}

// 计算退款金额（从数据库获取，不信任用户输入）
$refund_amount = $order['paid_amount'];

// 验证已退款金额
$already_refunded = get_refunded_amount($order_id);
if ($already_refunded > 0) {
    $refund_amount = $refund_amount - $already_refunded;
    if ($refund_amount <= 0) {
        die('无可退款金额');
    }
}

// 处理退款
process_refund($order_id, $refund_amount);

// 更新订单状态
update_order_status($order_id, 'refunded');
?>
```

## 9.4 密码重置漏洞

### 9.4.1 密码重置流程分析

**典型密码重置流程**

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 输入邮箱 │───▶│ 发送重置链接│───▶│ 点击链接 │───▶│ 设置新密码 │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
                    │              │              │
                    ▼              ▼              ▼
               ┌─────────┐    ┌─────────┐    ┌─────────┐
               │ 生成Token │    │ 验证Token │    │ 完成重置 │
               └─────────┘    └─────────┘    └─────────┘
```

**密码重置关键点**

```text
1. 身份验证：确认是账户所有者
2. Token生成：生成安全的重置Token
3. Token验证：验证Token的有效性
4. Token时效：设置Token过期时间
5. Token一次性：Token使用后失效
6. 新密码验证：新密码强度验证
```

### 9.4.2 Token泄露漏洞

**漏洞原理**

密码重置Token泄露或可预测，攻击者可以获取Token重置他人密码。

**漏洞场景**

```php
<?php
// 不安全的Token生成
$token = md5($email . time());  // 可预测的Token

// Token发送到邮箱，但可能泄露
$reset_link = "https://example.com/reset?token=" . $token;
send_email($email, $reset_link);

// Token未设置过期时间
// Token未绑定用户
?>
```

**攻击方式**

```text
攻击1：Token在响应中返回
- 重置请求响应中包含Token
- 攻击者直接获取Token

攻击2：Token可预测
- Token生成算法简单
- 攻击者可以预测Token值

攻击3：Token在Referer中泄露
- 用户点击重置链接后跳转
- Token出现在Referer中

攻击4：Token在URL中泄露
- Token出现在URL参数中
- 可能被日志记录或截图泄露
```

**检测方法**

```text
步骤1：发起密码重置请求
步骤2：检查响应是否包含Token
步骤3：分析Token生成规律
步骤4：尝试预测Token值
```

**修复方案**

```php
<?php
// 安全的Token生成
function generate_reset_token($user_id) {
    // 使用加密安全的随机数生成器
    $token = bin2hex(random_bytes(32));
    
    // 存储Token信息
    store_reset_token($user_id, $token, time() + 3600);  // 1小时有效
    
    return $token;
}

// Token验证
function validate_reset_token($token) {
    $token_info = get_token_info($token);
    
    // 验证Token是否存在
    if (!$token_info) {
        return false;
    }
    
    // 验证Token是否过期
    if ($token_info['expire_time'] < time()) {
        delete_token($token);
        return false;
    }
    
    // 验证Token是否已使用
    if ($token_info['used']) {
        return false;
    }
    
    return $token_info['user_id'];
}

// 密码重置处理
$token = $_POST['token'];
$new_password = $_POST['new_password'];

$user_id = validate_reset_token($token);
if (!$user_id) {
    die('Token无效或已过期');
}

// 更新密码
update_password($user_id, $new_password);

// 标记Token已使用
mark_token_used($token);
?>
```

### 9.4.3 重置链接绕过漏洞

**漏洞原理**

密码重置验证逻辑存在缺陷，攻击者可以绕过Token验证直接重置密码。

**漏洞场景**

```php
<?php
// 不安全的密码重置
$email = $_POST['email'];
$new_password = $_POST['new_password'];

// 未验证Token，直接根据邮箱重置密码
update_password_by_email($email, $new_password);
?>
```

**攻击方式**

```text
攻击1：删除Token参数
- 重置请求中删除Token参数
- 服务器可能默认通过验证

攻击2：使用空Token
- Token参数设置为空
- 服务器可能未验证空值

攻击3：修改邮箱参数
- 使用自己的Token重置他人密码
- 修改邮箱为受害者邮箱
```

**检测方法**

```text
步骤1：获取自己的重置Token
步骤2：修改请求中的邮箱为他人邮箱
步骤3：删除或修改Token参数
步骤4：观察是否成功重置他人密码
```

**攻击示例**

```http
POST /api/reset-password HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded

email=victim@example.com&new_password=hacked123

# 未提供Token，直接根据邮箱重置密码
```

**修复方案**

```php
<?php
// 安全的密码重置
$token = $_POST['token'] ?? '';
$new_password = $_POST['new_password'];

// 必须提供Token
if (empty($token)) {
    die('Token不能为空');
}

// 验证Token
$user_id = validate_reset_token($token);
if (!$user_id) {
    die('Token无效或已过期');
}

// 获取用户邮箱（从数据库，不信任用户输入）
$user = get_user_by_id($user_id);

// 更新密码
update_password($user_id, $new_password);

// 标记Token已使用
mark_token_used($token);

// 发送密码已修改通知
send_password_changed_notification($user['email']);
?>
```

### 9.4.4 Host头注入漏洞

**漏洞原理**

密码重置链接使用Host头构造URL，攻击者可以修改Host头使重置链接指向攻击者服务器。

**漏洞场景**

```php
<?php
```

**攻击方式**

```http

```

**检测方法**

## 9.5 验证码绕过漏洞

### 9.5.1 验证码类型分析

### 9.5.2 图形验证码绕过

**漏洞原理**

图形验证码生成或验证逻辑存在缺陷，攻击者可以绕过验证码验证。

**漏洞场景**

```php
<?php
// 不安全的验证码验证
$user_captcha = $_POST['captcha'];
$session_captcha = $_SESSION['captcha'];

// 未验证验证码是否为空
if ($user_captcha == $session_captcha) {
    // 验证通过
}

// 验证码未设置过期时间
// 验证码可重复使用
?>
```

**攻击方式**

```text
攻击1：验证码为空绕过
- 不提交验证码参数
- 服务器可能未验证空值

攻击2：验证码重复使用
- 同一验证码多次使用
- 服务器未标记已使用

攻击3：验证码爆破
- 简单验证码可识别
- 使用OCR工具识别

攻击4：验证码预测
- 验证码生成规律可预测
- 分析生成算法
```

**检测方法**

```text
步骤1：找到需要验证码的功能
步骤2：删除验证码参数
步骤3：观察是否成功绕过

步骤4：使用验证码后再次使用
步骤5：观察是否成功绕过
```

**修复方案**

```php
<?php
// 安全的验证码验证
$user_captcha = $_POST['captcha'] ?? '';

// 验证码不能为空
if (empty($user_captcha)) {
    die('请输入验证码');
}

// 验证码不存在
if (!isset($_SESSION['captcha'])) {
    die('验证码已过期');
}

// 验证码已使用
if ($_SESSION['captcha_used']) {
    die('验证码已使用');
}

// 验证码不匹配
if (strtolower($user_captcha) !== strtolower($_SESSION['captcha'])) {
    die('验证码错误');
}

// 标记验证码已使用
$_SESSION['captcha_used'] = true;

// 清除验证码
unset($_SESSION['captcha']);
?>
```

### 9.5.3 短信验证码绕过

**漏洞原理**

短信验证码验证逻辑存在缺陷，攻击者可以绕过验证或爆破验证码。

**漏洞场景**

```php
<?php
// 不安全的短信验证码验证
$phone = $_POST['phone'];
$code = $_POST['code'];

// 未验证验证码是否为空
// 未限制验证码尝试次数
// 未验证手机号归属
if ($code == $_SESSION['sms_code']) {
    // 验证通过
}
?>
```

**攻击方式**

```text
攻击1：验证码为空绕过
- 不提交验证码参数
- 服务器可能默认通过

攻击2：验证码爆破
- 短信验证码通常4-6位
- 尝试次数足够可爆破

攻击3：验证码回显
- 发送验证码后响应包含验证码
- 直接获取验证码

攻击4：修改手机号
- 使用自己的验证码验证他人手机
- 修改手机号参数
```

**检测方法**

```text
步骤1：发送短信验证码
步骤2：检查响应是否包含验证码
步骤3：删除验证码参数尝试绕过
步骤4：尝试爆破验证码
```

**验证码爆破**

```python
import requests

def brute_force_sms_code():
    session = requests.Session()
    
    # 发送验证码
    session.post('https://example.com/api/send-sms', data={
        'phone': '13800138000'
    })
    
    # 爆破验证码（4位数字）
    for code in range(1000, 9999):
        response = session.post('https://example.com/api/verify-sms', data={
            'phone': '13800138000',
            'code': str(code)
        })
        
        if response.json()['status'] == 'success':
            print(f"验证码破解成功: {code}")
            break
```

**修复方案**

```php
<?php
// 安全的短信验证码验证
$phone = $_POST['phone'];
$code = $_POST['code'] ?? '';

// 验证码不能为空
if (empty($code)) {
    die('请输入验证码');
}

// 验证码长度检查
if (strlen($code) != 6) {
    die('验证码格式错误');
}

// 验证码不存在
if (!isset($_SESSION['sms_code']) || !isset($_SESSION['sms_phone'])) {
    die('请先获取验证码');
}

// 验证码过期检查
if ($_SESSION['sms_expire'] < time()) {
    die('验证码已过期');
}

// 手机号匹配检查
if ($_SESSION['sms_phone'] != $phone) {
    die('手机号不匹配');
}

// 尝试次数限制
if ($_SESSION['sms_attempts'] >= 5) {
    die('验证码尝试次数过多');
}

// 验证码不匹配
if ($code != $_SESSION['sms_code']) {
    $_SESSION['sms_attempts']++;
    die('验证码错误');
}

// 验证成功，清除验证码
unset($_SESSION['sms_code']);
unset($_SESSION['sms_attempts']);
?>
```

**漏洞场景**

```php
    'code' => $code,  // 验证码泄露

]);
?>
```

**攻击方式**

```text
攻击1：直接获取验证码
- 发送验证码请求
- 从响应中获取验证码

攻击2：获取他人验证码
- 发送验证码到他人手机
- 从响应中获取验证码
- 使用验证码验证他人账户
```

**检测方法**

```text

步骤2：检查响应内容
步骤3：确认是否包含验证码
```

**修复方案**

```php
// 存储验证码
```

## 9.6 条件竞争漏洞

### 9.6.1 条件竞争原理

**什么是条件竞争**

条件竞争（Race Condition）是指多个并发操作同时访问共享资源时，由于执行顺序不确定导致的逻辑错误。

**通俗理解**

想象一个银行账户余额为100元。如果两个人同时发起取款50元的请求，正常情况下第二次取款应该失败（余额不足）。但如果两个请求几乎同时到达，服务器可能都读取到余额100元，都认为可以取款，最终余额变成-50元。

**条件竞争流程**

```
时间线：
T0: 余额 = 100
T1: 用户A读取余额 = 100
T2: 用户B读取余额 = 100
T3: 用户A计算新余额 = 100 - 50 = 50
T4: 用户B计算新余额 = 100 - 50 = 50
T5: 用户A写入余额 = 50
T6: 用户B写入余额 = 50
```

## 9.12 支付逻辑漏洞高级技巧

### 9.12.1 金额篡改进阶技巧

**浮点精度漏洞**

```text
漏洞原理：
- 使用浮点数处理金额可能导致精度丢失
- 0.1 + 0.2 != 0.3
- 攻击者可以利用精度误差获取优惠

攻击方式：
- 修改金额为0.1、0.2等特殊值
- 利用精度误差绕过金额验证
```

**攻击示例**

```python
import requests

def exploit_float_precision():
    session = requests.Session()
    
    # 创建订单，商品价格100.1元
    order_response = session.post('https://shop.example.com/api/create-order', data={
        'product_id': '1001',
        'quantity': '1'
    })
    order_id = order_response.json()['order_id']
    
    # 修改支付金额为100元（利用浮点精度误差）
    pay_response = session.post('https://shop.example.com/api/pay', data={
        'order_id': order_id,
        'amount': '100'  # 实际应该支付100.1元
    })
    
    if pay_response.json()['status'] == 'success':
        print("[!] 浮点精度漏洞：100.1元商品只支付了100元")
```

**修复方案**

```php
<?php
// 使用整数处理金额（以分为单位）
$amount_in_cents = intval($amount * 100);

// 比较金额时使用整数比较
if ($paid_amount_in_cents >= $order_amount_in_cents) {
    // 支付成功
}

// 或者使用BCMath扩展进行精确计算
if (bccomp($paid_amount, $order_amount, 2) >= 0) {
    // 支付成功
}
?>
```

**金额溢出漏洞**

```text
漏洞原理：
- 金额字段使用整数类型，存在溢出风险
- 攻击者可以输入超大金额导致溢出
- 溢出后金额可能变为负数或0

攻击方式：
- 修改金额为整数最大值+1
- 修改金额为负数（利用溢出）
```

**攻击示例**

**修复方案**

### 9.12.2 优惠券组合滥用

**优惠券叠加漏洞**

```text
漏洞原理：
- 系统允许多张优惠券叠加使用
- 但未验证叠加后的优惠金额是否合理
- 攻击者可以叠加多张优惠券实现0元购买

攻击方式：
- 获取多张优惠券
- 在同一订单中使用多张优惠券
- 叠加后的优惠金额超过订单金额
```

**攻击示例**

```python
import requests

def exploit_coupon_stack():
    session = requests.Session()
    
    # 获取多张优惠券
    coupons = ['SAVE100', 'SAVE50', 'SAVE30']
    
    # 创建订单，商品价格150元
    order_response = session.post('https://shop.example.com/api/create-order', data={
        'product_id': '1001',
        'quantity': '1'
    })
    order_id = order_response.json()['order_id']
    
    # 叠加使用三张优惠券（100+50+30=180元优惠）
    pay_response = session.post('https://shop.example.com/api/pay', data={
        'order_id': order_id,
        'amount': '0',  # 优惠后金额为0
        'coupons': ','.join(coupons)
    })
    
    if pay_response.json()['status'] == 'success':
        print("[!] 优惠券叠加漏洞：150元商品0元购买")
```

**修复方案**

```php
<?php
// 验证优惠券叠加规则
function validate_coupon_stack($coupons, $order_amount) {
    $total_discount = 0;
    
    foreach ($coupons as $coupon_code) {
        $coupon = get_coupon($coupon_code);
        
        // 验证优惠券有效性
        if (!$coupon || $coupon['used'] || $coupon['expired']) {
            return false;
        }
        
        $total_discount += $coupon['discount'];
    }
    
    // 优惠金额不能超过订单金额
    if ($total_discount > $order_amount) {
        return false;
    }
    
    return true;
}
?>
```

```php
// 验证支付回调签名
function validate_payment_signature($data, $signature) {
```

```text
   - 用户积分余额：100积分
   - 使用并发请求同时兑换10次
   - 成功兑换10次，获得10张礼品卡
   
3. 漏洞分析阶段：

   - 但由于积分不足，扣除后变成负数
```

```python
        'product_id': 'gift_card'
    })

        thread.start()
    
    for thread in threads:
        thread.join()
    
    success_count = sum(1 for r in results if r and r.get('status') == 'success')
    print(f"兑换成功次数: {success_count}")
    print(f"预期兑换次数: 1")
    
    # 查询最终积分余额
    balance = session.get('https://finance.example.com/api/points').json()['balance']
    print(f"最终积分余额: {balance}")
    
    if success_count > 1:
        print("[!] 存在条件竞争漏洞")
```

**漏洞发现过程**

```text
1. 信息收集阶段：
```

### 9.16.3 案例七：某社交平台密码重置漏洞

**漏洞描述**

某社交平台密码重置功能存在Host头注入漏洞，攻击者可以修改重置链接的域名，获取用户的重置Token。

**漏洞发现过程**

```text
1. 信息收集阶段：
   - 目标网站：social.example.com
   - 发现密码重置功能：POST /api/request-reset
   
2. 漏洞检测阶段：
   - 发起密码重置请求
   - 修改Host头为攻击者域名
   - 用户收到指向攻击者域名的重置链接
   - 用户点击链接，Token被发送到攻击者服务器
   
3. 漏洞分析阶段：
   - 后端使用Host头构造重置链接
   - 未验证Host头的合法性
   - 攻击者可以完全控制重置链接的域名
```

**PoC构造**

```python
import requests

def exploit_host_header_injection():
    # 发起密码重置请求，修改Host头
    response = requests.post('http://social.example.com/api/request-reset',
        data={'email': 'victim@example.com'},
        headers={'Host': 'attacker.com'}
    )
    
    print("[!] Host头注入攻击已发起")
    print("受害者将收到以下重置链接：")
    print("https://attacker.com/reset?token=xxx")
    print("点击后Token将发送到攻击者服务器")
```

**修复建议**

1. 使用配置的域名，不信任Host头
2. 禁用X-Forwarded-Host等不可信头
3. 验证重置链接的域名
4. 发送重置链接前验证邮箱归属

```php
<?php
// 验证输入参数
function validate_input($data) {
    // 验证金额
    if (!isset($data['amount']) || !is_numeric($data['amount'])) {
        die('金额参数无效');
    }
    
    $amount = floatval($data['amount']);
    if ($amount <= 0 || $amount > 1000000) {
        die('金额参数异常');
    }
    
    // 验证数量
    if (!isset($data['quantity']) || !is_numeric($data['quantity'])) {
        die('数量参数无效');
    }
    
    $quantity = intval($data['quantity']);
    if ($quantity <= 0 || $quantity > 1000) {
        die('数量参数异常');
    }
    
    return true;
}
?>
```

```php
// 验证业务数据
function validate_business_data($order_id) {
    $order = get_order_from_db($order_id);
```

```php
<?php
// 使用数据库事务
function process_order($order_id) {
    $db->beginTransaction();
    
    try {
        // 查询订单并锁定
        $order = $db->query("SELECT * FROM orders WHERE id = ? FOR UPDATE", [$order_id]);
        
        // 验证订单状态
        if ($order['status'] != 'pending') {
            throw new Exception('订单状态异常');
        }
        
        // 验证库存
        $stock = $db->query("SELECT stock FROM products WHERE id = ? FOR UPDATE", [$order['product_id']]);
        if ($stock['stock'] < $order['quantity']) {
            throw new Exception('库存不足');
        }
        
        // 更新库存
        $db->query("UPDATE products SET stock = stock - ? WHERE id = ?", [$order['quantity'], $order['product_id']]);
        
        // 更新订单状态
        $db->query("UPDATE orders SET status = 'paid' WHERE id = ?", [$order_id]);
        
        // 提交事务
        $db->commit();
        
        return true;
    } catch (Exception $e) {
        // 回滚事务
        $db->rollback();
        die($e->getMessage());
    }
}
?>
```

## 9.18 本章小结

### 9.18.1 逻辑漏洞挖掘要点

**核心概念**
- 逻辑漏洞源于业务设计缺陷，而非代码实现错误
- 逻辑漏洞需要理解业务流程才能发现
- 逻辑漏洞危害可能非常严重（经济损失、账户接管等）

**挖掘方法**
- 系统化分析业务流程
- 识别边界条件和异常场景
- 参数篡改测试
- 流程绕过测试
- 状态转换测试
- 并发测试

**防护措施**
- 输入验证和数据验证
- 使用事务和锁机制
- 实现状态机控制
- 添加审计日志
- 定期安全审计

### 9.18.2 关键漏洞类型要点

**支付逻辑漏洞**
- 金额必须从数据库获取
- 支付回调必须验证签名
- 优惠券使用必须有防重复机制
- 使用整数处理金额避免浮点精度问题

**订单逻辑漏洞**
- 订单归属必须验证
- 状态转换必须合法
- 退款金额必须验证
- 使用Redis原子操作实现限购

**密码重置漏洞**
- Token不能泄露给前端
- Token必须不可预测
- 重置链接使用固定域名
- Token使用后立即失效

**验证码漏洞**
- 验证码必须验证，不能为空
- 验证码一次性使用
- 尝试次数必须限制
- 使用复杂验证码防止OCR识别

### 9.18.3 条件竞争要点

**核心概念**
- 并发操作可能导致逻辑错误
- 时间窗口是漏洞关键
- 需要同步机制保护共享资源

**检测方法**
- 使用多线程并发测试
- 使用Burp Suite Intruder
- 分析并发结果

**防护措施**
- 使用数据库事务和行锁
- 使用Redis原子操作
- 使用分布式锁
- 验证最终状态
