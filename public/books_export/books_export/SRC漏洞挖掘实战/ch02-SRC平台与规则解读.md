# 第二章 SRC平台与规则解读

选对了平台，事半功倍。不同的SRC平台有不同的特点、规则和奖励标准。选择适合自己的平台，是SRC之路的第一步。本章我们将详细解读国内主流SRC平台，对比它们的规则、奖励、审核速度、活动，以及如何选择、如何提交、如何沟通、如何申诉，让你全面了解SRC平台的玩法。

## 2.1 国内主流SRC平台详解

国内SRC平台数量众多，各有特色。下面我们详细介绍各大主流平台的特点、业务范围、奖励标准和注意事项。

### 2.1.1 阿里巴巴ASRC

**平台地址**：https://security.alibaba.com

**平台简介**：
阿里巴巴安全响应中心（Alibaba Security Response Center），简称ASRC，是国内最早的企业级SRC之一，2014年正式上线。经过多年发展，ASRC已经成为国内规模最大、奖励最丰厚、最具影响力的企业SRC之一。

**业务范围**：
- 淘宝、天猫电商平台
- 支付宝金融业务（注意：支付宝部分业务属于蚂蚁集团，需单独注意）
- 阿里云云计算
- 钉钉办公软件
- 优酷视频平台
- 高德地图
- 菜鸟物流
- 饿了么本地生活
- 1688批发网
- 飞猪旅行
- 闲鱼二手交易
- 等等，阿里系几乎所有业务

**奖励标准（参考，以官方最新为准）**：

| 漏洞等级 | 奖励范围（现金） | 典型漏洞 |
|---------|---------------|---------|
| 严重 | 50000 - 100000+ 元 | 核心系统RCE、大规模数据泄露、重大支付漏洞 |
| 高危 | 10000 - 50000 元 | 核心业务SQL注入、任意用户登录、严重越权 |
| 中危 | 2000 - 10000 元 | 存储型XSS、普通越权、中等信息泄露 |
| 低危 | 200 - 2000 元 | 反射型XSS、轻微信息泄露、路径泄露 |

**特色活动**：
- **双11安全挑战赛**：每年双11期间举办，奖励翻倍甚至更多
- **月度漏洞排名奖励**：月度前10名有额外现金奖励
- **年度白帽子大会**：邀请年度优秀白帽子参加线下大会
- **内推机会**：排名靠前的白帽子可获得阿里安全团队内推
- **专项活动**：不定期举办云安全专项、IoT安全专项等
- **礼品周边**：经常有定制礼品、T恤、纪念章等

**测试规则**：
- **允许**：常规Web漏洞测试、逻辑漏洞挖掘、移动端测试
- **禁止**：DDoS攻击、社工员工、下载大量数据、暴力破解
- **特殊注意**：
  - 支付相关漏洞需特别谨慎，验证即可，不要造成实际资金流动
  - 阿里云相关业务测试前务必确认在测试范围内
  - 涉及用户数据的漏洞，仅用自己的测试账号验证
  - 禁止测试第三方商家的系统

**提交要求**：
- 使用官方报告模板
- 必须提供详细的复现步骤
- PoC代码要清晰可运行
- 截图要完整，关键信息标注
- 建议提供危害分析和修复建议

**审核速度**：
- 严重/高危：1-3个工作日
- 中危：3-5个工作日
- 低危：5-7个工作日
- 总体来说审核速度较快，响应及时

**友好度评分**：★★★★★
- 奖励丰厚，业界领先
- 审核专业，反馈详细
- 沟通顺畅，态度友好
- 活动丰富，福利多

**适合人群**：
- 有一定经验的白帽子
- 想挑战高难度、高奖励的白帽子
- 想进阿里的朋友

### 2.1.2 腾讯TSRC

**平台地址**：https://security.tencent.com

**平台简介**：
腾讯安全应急响应中心（Tencent Security Response Center），简称TSRC。腾讯作为国内最大的互联网公司之一，业务涵盖社交、游戏、金融、云等多个领域，TSRC也是国内最活跃的SRC平台之一。

**业务范围**：
- 微信生态（小程序、公众号、支付、视频号等）
- QQ社交平台
- 腾讯游戏（王者荣耀、和平精英、英雄联盟等上百款游戏）
- 腾讯视频
- 腾讯云
- 腾讯音乐
- 腾讯新闻
- 应用宝
- 等等，腾讯系全线产品

**奖励标准（参考）**：

| 漏洞等级 | 奖励范围（现金） | 典型漏洞 |
|---------|---------------|---------|
| 严重 | 30000 - 80000 元 | 核心系统RCE、重大支付漏洞、大规模数据泄露 |
| 高危 | 8000 - 30000 元 | SQL注入、越权获取敏感信息、存储型XSS(大范围) |
| 中危 | 1500 - 8000 元 | 存储型XSS、普通越权、CSRF(敏感操作) |
| 低危 | 100 - 1500 元 | 反射型XSS、轻微信息泄露 |

**特色奖励与活动**：
- **游戏安全专项奖励**：游戏漏洞奖励较高，部分游戏高危可达5万+
- **月度之星评选**：每月评选优秀白帽子，额外奖励
- **年度白帽子大会**：规模大，规格高
- **腾讯安全应急响应中心证书**：官方证书，含金量高
- **小程序安全专项**：微信小程序相关漏洞专项奖励
- **腾讯云专项**：云服务相关漏洞单独奖励
- **内推机会**：优秀白帽子可推荐腾讯安全团队

**测试规则**：
- 微信相关漏洞需特别注意影响范围，涉及用户隐私的要谨慎
- 游戏漏洞需说明具体影响和利用方式
- 禁止测试第三方接入的业务（如公众号、小程序主体是第三方的）
- 不接受纯理论性漏洞，必须有实际影响证明
- 企业微信、企业版产品需特别注意测试范围

**审核特点**：
- 审核团队专业，技术实力强
- 反馈比较详细，会告知漏洞原因和修复情况
- 游戏业务线多，测试范围广
- 等级评定相对严格

**友好度评分**：★★★★☆
- 奖励丰厚
- 审核专业
- 游戏业务机会多
- 等级评定略严

**适合人群**：
- 对游戏安全感兴趣的白帽子
- 想深耕社交、小程序安全的白帽子
- 有一定经验的中高级白帽子

## 2.3 漏洞等级评定标准对比

不同平台的漏洞等级评定标准有相似之处，但也各有特点。了解各平台的评定标准，有助于你准确判断漏洞等级，提高通过率。

### 2.3.1 通用评定框架

虽然各平台细节不同，但总体框架基本一致，都是从以下几个维度判断：

| 评定维度 | 说明 |
|---------|------|
| **影响范围** | 受影响的用户数量、业务重要性、数据敏感性 |
| **利用难度** | 是否需要认证、是否需要交互、利用条件是否苛刻 |
| **危害程度** | 直接危害、间接危害、可扩展性 |
| **漏洞类型** | 不同类型漏洞有基础等级不同 |

四个等级的通用定义：

- **严重**：核心系统被控制，大规模敏感数据泄露，重大经济损失
- **高危**：敏感数据泄露，部分功能被控制，一定经济损失
- **中危**：部分数据泄露，需要特定条件利用，中等危害
- **低危**：危害较小，利用难度高，轻微影响

### 2.3.2 各平台评定特点

**阿里ASRC评定特点：**
- 非常重视业务影响，影响大的漏洞等级高
- 支付、电商相关逻辑漏洞评级较高
- 严重漏洞奖励上限高，下限也高
- 对报告质量要求高
- 逻辑漏洞评级相对重视
- 新业务漏洞有加成

**腾讯TSRC评定特点：**
- 游戏业务漏洞评级较高
- 微信生态相关漏洞影响范围大的评级高
- 等级评定相对严格
- 对漏洞利用证明要求高
- 纯理论漏洞不接受
- 第三方业务要区分清楚

**百度BSRC评定特点：**
- 相对宽松一些
- 信息泄露类漏洞评级还可以
- AI、自动驾驶相关漏洞重视
- 等级评定弹性较大
- 新手友好

**字节BSRC评定特点：**
- 新业务漏洞评级高
- 严重上限高
- 响应快，评级也比较爽快
- 逻辑漏洞评级不错
- 重视报告质量和影响分析

**京东/美团等电商平台评定特点：**
- 支付、订单、支付漏洞评级高
- 逻辑漏洞比较受重视
- 金融相关敏感
- 用户数据敏感的评级高

**补天/漏洞盒子等第三方平台评定特点：**
- 不同企业标准不同
- 要看具体企业而定
- 一般来说等级相对宽松
- 企业自主评定，平台审核

注意：以上仅为大致参考，实际评级还要看具体场景、影响范围、利用难度等多种因素。

### 2.3.4 影响评级的其他因素

除了漏洞本身，还有一些因素会影响最终评级：

**会提升评级的因素：
- 影响范围特别大
- 利用难度特别低
- 造成的危害特别严重
- 漏洞质量高，报告写得好
- 首次发现的新类型漏洞
- 新业务、新功能的漏洞
- 活动期间提交的漏洞(活动加成)

会降低评级的因素：
- 影响范围小
- 利用条件苛刻
- 危害有限
- 报告质量差，复现困难
- 已知公开的漏洞
- 重复漏洞(重复提交的不奖励 |

## 2.4 新手如何选择第一个目标平台

新手入门，选对第一个平台很重要。选好了，事半功倍；选错了，可能打击信心。下面给新手一些建议。

### 2.4.1 新手选平台的几个原则

**原则一：从易到难，循序渐进**

不要一开始就盯着阿里、腾讯、字节这种大厂SRC，这些地方高手如云，新手很难挖到漏洞，容易打击信心。

建议从第三方平台或中小厂SRC开始，等有一定经验了再挑战大厂。

**原则二：匹配自己的兴趣和特长**

如果你擅长逻辑漏洞，可以选电商、本地生活类平台；如果你擅长移动安全，可以选APP多的平台；如果你擅长云安全，可以选云厂商。

选自己感兴趣、擅长的方向，更容易出成果。

**原则三：关注新平台、新业务**

新上线的SRC平台，新上线的业务，通常漏洞比较多，竞争也小，新手更容易挖到第一个漏洞。

**原则四：综合考虑奖励和难度**

奖励高的平台，通常难度也大，竞争也激烈。新手不要只看奖励，还要看自己能不能挖到。

先保证能挖到，再考虑奖励多少。

### 2.4.5 新手选平台的常见误区

**误区一：只看奖励，不看难度**

很多新手一上来就选奖励最高的平台，结果挖了半天一个漏洞都挖不到，信心受挫。

**正解：** 先看能不能挖到，再看奖励多少。

**误区二：同时挖太多平台**

有些新手同时注册十几个平台，每个都挖两下，结果哪个都没挖深。

**正解：** 一次专注1-3个平台，挖透了再换。

**误区三：只盯大厂，看不起小厂**

有些新手觉得只有大厂SRC才叫SRC，小厂的不屑一顾。

**正解：** 小厂虽然单个漏洞多，容易上手，是新手练手的好地方。

**误区四：跟风走极端**

听别人说哪个好就去哪个，没有自己的判断。

**正解：** 适合自己的才是最好的，根据自己的情况选择。

## 2.5 完整提交流程实战

了解了平台，下面我们走一遍完整的提交流程，从注册到拿奖励，一步一步详细讲解。

### 2.5.1 第一步：注册与实名认证

我们以补天平台为例，走一遍注册流程。

**1. 访问平台首页**

打开浏览器，访问补天平台官网：https://www.butian.net

**2. 点击注册**

点击右上角的"注册"按钮，进入注册页面。

**3. 填写基本信息**

- 用户名：起一个好记的ID，建议英文+数字组合
- 邮箱：常用邮箱，用于验证和找回密码
- 密码：设置一个强密码，大小写字母+数字+特殊字符
- 手机号：填写真实手机号，接收验证码

**4. 验证邮箱和手机**

- 点击邮箱验证链接
- 输入手机验证码

**5. 进入个人中心**

注册成功后，登录进入个人中心。

**6. 实名认证（重要！**

这是最重要的一步，必须完成实名认证才能提交漏洞和获得奖励。

需要准备的材料：
- 身份证正面照片
- 身份证反面照片
- 手持身份证照片（脸部清晰，身份证信息清晰）

**注意事项：**
- 照片要清晰，所有文字都能看清
- 手持身份证照片要露出整个脸部
- 不要用PS修改照片
- 一个身份证只能注册一个账号
- 审核通常1-3个工作日

**7. 绑定收款账户**

在个人中心找到"账户设置"或"奖励设置"，绑定你的收款方式：
- 支付宝（推荐，到账快）
- 银行卡
- 其他方式（看平台支持）

**8. 完善个人资料**

- 头像：可以不用真实头像，也可以用个网图
- 个人简介：简单介绍一下自己
- 技能标签：选自己擅长的方向
- 这些资料完善有助于提高账号可信度

### 2.5.2 第二步：阅读规则与测试范围

注册完别急着挖洞，先读规则！

**1. 找到规则页面**

一般在首页底部或导航栏找到"规则中心"、"测试范围"、"帮助中心"等入口。

**2. 重点阅读内容**

**测试范围：**
- 哪些域名/IP可以测
- 哪些业务可以测
- 子域名包含不包含
- 第三方业务能不能测
- 生产环境和测试环境区分

**禁止行为：**
- 禁止DDoS攻击
- 禁止暴力破解
- 禁止下载大量数据
- 禁止社工
- 禁止破坏性操作
- 等等

**奖励标准：**
- 各等级漏洞奖励范围
- 积分怎么算
- 怎么兑换
- 发放时间

**提交要求：**
- 报告格式要求
- 需要哪些证明材料
- 审核周期
- 申诉流程

**3. 选择目标企业**

在企业列表里选一个你感兴趣的企业，点进去看详细信息：
- 企业介绍
- 测试范围
- 奖励标准
- 活动信息
- 注意事项

**4. 保存规则**

把重要的规则内容保存下来，或者收藏页面，不确定的时候翻一翻。

### 2.5.3 第三步：信息收集与漏洞挖掘

选好目标企业，就可以开始信息收集和漏洞挖掘了。

详细的信息收集和漏洞挖掘方法我们第一章讲过了，这里简单回顾一下流程：

**1. 信息收集**
- 子域名枚举
- 端口扫描
- 目录扫描
- 指纹识别
- GitHub信息收集

**2. 漏洞挖掘**
- 手工测试常见漏洞
- 工具辅助扫描
- 重点测试核心功能
- 关注业务逻辑

**3. 漏洞验证**
- 确认漏洞存在
- 评估漏洞等级
- 收集证据截图

详细步骤参考第一章的内容。

### 2.5.4 第四步：撰写漏洞报告

发现漏洞后，写一份高质量的报告。

**报告结构：**

1. **漏洞标题**
   - 格式：[漏洞类型] 位置 + 影响
   - 例子：[SQL注入] 商品搜索接口id参数存在SQL注入
   - 简洁明了，一看就知道是什么漏洞

2. **漏洞类型**
   - 选择对应的漏洞类型
   - SQL注入、XSS、越权、逻辑漏洞等

3. **漏洞等级**
   - 你认为的等级
   - 严重/高危/中危/低危

4. **漏洞详情**
   - 详细描述漏洞是什么
   - 漏洞原理是什么
   - 会造成什么危害

5. **复现步骤**
   - 一步一步写清楚
   - 最好编号
   - 每一步做什么
   - 预期结果是什么
   - 实际结果是什么
   - 越详细越好

6. **PoC/数据包**
   - 完整的HTTP请求包
   - 或者Python脚本
   - 让审核员复制就能用

7. **修复建议**
   - 具体怎么修复
   - 不要空泛
   - 给出具体方案

8. **证明材料**
   - 截图：关键步骤
   - 视频：复杂的录屏
   - 其他证明

**写报告的小技巧：**
- 站在审核员角度想，他需要什么信息
- 步骤越详细越好
- 截图要清晰，标注重点
- 提交前自己再复现一遍
- 不要有错别字，语言通顺

### 2.5.5 第五步：提交报告

报告写好了，就可以提交了。

**提交步骤：**

1. **进入提交页面**
   - 在目标企业页面点击"提交漏洞"
   - 或者在个人中心点击"提交漏洞"

2. **选择目标企业**
   - 选择你测试的那个企业
   - 别选错了！

3. **填写报告内容**
   - 按照表单填写各项内容
   - 把写好的报告内容复制进去
   - 上传截图和证明材料

4. **检查一遍**
   - 提交前再检查一遍
   - 有没有漏填的
   - 截图上传了没有
   - 企业选对了没有

5. **点击提交**
   - 确认无误后点击提交
   - 提交成功后会有漏洞编号
   - 保存好这个编号，方便后续查询

### 2.5.6 第六步：等待审核与沟通

提交后，就是等待审核了。

**审核周期：**
- 一般1-7个工作日
- 严重/高危优先处理
- 不同平台速度不同

**审核过程中可能出现的情况：

**情况一：审核通过**
- 漏洞确认，给了等级和奖励
- 如果同意，就等奖励发放
- 如果不同意等级，可以申诉

**情况二：要求补充材料**
- 审核员有些地方没看懂，要你补充
- 及时补充材料
- 尽快回复，不要拖延

**情况三：无法复现**
- 审核员复现不了
- 可能是你操作有问题
- 也可能是环境差异
- 配合审核员找到原因，提供更多信息

**情况四：重复漏洞**
- 已经有人提交过了
- 很遗憾，没有奖励
- 下次再接再厉

**情况五：不予受理**
- 漏洞不在测试范围内
- 或者漏洞不符合要求
- 看清楚原因，下次注意

**沟通注意事项：**
- 及时查看消息通知
- 回复要礼貌专业
- 耐心配合审核员
- 有话好好说，不要吵架

### 2.5.7 第七步：奖励发放

漏洞确认后，就等着拿奖励了。

**奖励发放流程：**

1. **漏洞确认** → 计入奖励确定
2. **每月结算** → 一般月度结算
3. **申请兑换** → 积分的话需要自己兑换
4. **打款到账** → 现金奖励打你账户

**到账时间：**
- 不同平台不同
- 一般月度发放
- 有的次月10号、15号、月底等
- 具体看平台规则

**税务问题：**
- 有些平台会代扣个人所得税
- 奖励是税前还是税后
- 注意看平台说明

**注意查收：**
- 留意支付宝/银行卡到账通知
- 没到账及时联系客服
- 保存好记录

## 2.6 沟通技巧实战案例

与审核员沟通，也是一门学问。沟通得好，可能等级都可能高一级；沟通不好，本来能过的也可能给你打回。下面分享几个真实的沟通案例。

### 2.6.1 案例一：审核员说复现不了，怎么办？

**背景：**
小白提交了一个越权漏洞，自己测试了好几次都能复现。但审核员说复现不了，要打回。

**小白的第一反应：**
"怎么可能复现不了？明明我这边可以啊！审核员是不是没认真测？"

小白有点生气，想直接回复说"我这边明明可以，你再试试啊"。

但他忍住了，想了想，换了一种方式。

**小白的回复：**

"您好，辛苦了！麻烦再帮忙确认一下，我这边测试了好几次都是可以的。可能是我步骤写得不够清楚，我再补充一些细节：

1. 测试账号：
- 账号A：test123（普通用户）
- 账号B：test456（普通用户）

2. 具体步骤再详细说一下：
第一步：用账号A登录，进入个人中心，查看自己的订单，URL是 /order/detail?id=1001
第二步：用Burp抓这个请求包
第三步：把id参数改成1002（这是账号B的订单ID）
第四步：放行请求，就能看到订单信息了

3. 我再补充几张截图：
- 图1：账号A的个人中心截图
- 图2：抓包改参数截图
- 图3：成功看到账号B订单的截图
- 图4：两个账号的登录状态截图

另外，请问您那边是哪一步不行呢？可能是我哪里没写清楚，我再补充。

麻烦了，谢谢！"

**结果：**
审核员看了补充的材料，很快就复现成功了。漏洞确认中危，奖励1500元。

审核员还特意说："感谢详细，你补充得很清楚，辛苦了！"

**经验总结：**
1. 审核员说复现不了，先不要生气，先从自己找原因。
2. 可能是你步骤写得不够清楚，不是审核员没跟上。
3. 补充更详细的步骤、账号信息、更多截图。
4. 问问审核员卡在哪一步，针对性补充。
5. 态度要好，互相理解，大家都是工作。

### 2.6.2 案例二：等级被降级了，要不要申诉？

**背景：**
老周提交了一个漏洞，自己觉得是高危，结果审核给了中危。老周有点不甘心，想申诉，但又怕申诉了反而更糟。

**老周的分析：**
老周先冷静下来，仔细想了想：
- 这个漏洞是越权查看用户手机号
- 用户手机号算不算敏感信息？
- 影响范围有多大？
- 利用难度高不高？

老周觉得，手机号属于敏感个人信息，而且影响所有用户，利用难度低，应该是高危才对。

但老周也知道，申诉不是吵架，得有证据。

**老周的申诉邮件：**

"您好，关于漏洞编号XXXX的越权漏洞，我想申请复核一下等级，麻烦了。

我理解平台给出的中危评定，但我认为这个漏洞可以评为高危，理由如下：

1. **泄露的信息敏感度高**：
   - 泄露的是用户真实手机号，属于公民个人敏感信息
   - 根据《网络安全法》，个人信息保护要求很高
   - 手机号泄露可能导致诈骗、骚扰等后续危害

2. **影响范围大**：
   - 影响所有注册用户，数量巨大
   - 只要知道用户ID就能查，遍历用户ID是连续的，可以批量遍历

3. **利用难度低**：
   - 只需要普通用户登录
   - 不需要任何特殊条件
   - 构造简单，容易批量利用

4. **实际危害分析**：
   - 黑产可以利用这个漏洞批量爬取用户手机号
   - 可能造成大规模用户信息泄露
   - 对企业声誉和用户信任影响较大

我再补充一下危害证明：
- 我测试了3个不同的用户ID，都能成功获取手机号
- 附上批量获取的截图（已经打码处理了）
- 用户ID是自增的，可以批量遍历

以上是我的看法，可能有考虑不周的地方，麻烦审核团队再评估一下。

非常感谢！"

**结果：**
审核团队重新评估后，把等级从中危调整为高危，奖励从1500元升到了8000元。

审核员回复说："感谢的详细分析，我们重新评估后认为确实高危。报告质量很高，继续加油！"

**经验总结：**
1. 可以申诉，但要有理有据。
2. 申诉不是说"我觉得是高危"，而是给出为什么是高危的理由。
3. 从影响范围、利用难度、危害程度等角度分析。
4. 提供更多的危害证明。
5. 语气要礼貌，尊重审核员的工作。
6. 即使申诉失败也没关系，下次继续努力。

### 2.6.3 案例三：审核员问了很多问题，如何高效沟通

**背景：**
阿明提交了一个比较复杂的逻辑漏洞。审核员问了好几个问题，阿明有点不耐烦了，觉得审核员事儿多。

但阿明知道，审核员问问题说明对你感兴趣，是好事。说明在认真看你的漏洞。

**阿明的沟通原则：**

1. **及时回复**：审核员的问题，当天就回复，不要拖。
2. **耐心详细**：每个问题都认真回答，不嫌麻烦。
3. **主动补充**：主动提供更多信息，帮审核员理解。
4. **态度友好**：礼貌客气，营造良好关系。

**具体的沟通：**

审核员问："这个漏洞的影响范围有多大？所有用户都可以吗？"

阿明回复：
"您好，这个漏洞影响所有注册用户。只要是注册了的用户，都可以被越权查看。我测试了3个不同的账号，都可以成功。
补充一下测试账号：普通用户权限就可以，不需要特殊权限。"

审核员又问："那有没有什么限制？比如能不能批量利用？"

阿明回复：
"可以批量利用的。因为用户ID是连续的，可以写脚本批量遍历获取。
我写了个简单的Python脚本测试了一下，一秒钟可以查几十个。脚本我也可以提供您参考，不过是测试用的，没跑太多。"

审核员再问："修复的话，你有什么建议吗？"

阿明回复：
"修复建议的话，我觉得可以从这几个方面入手：
1. 服务端校验用户权限，查询订单的时候校验当前用户ID对应用户，不能只传ID就查
2. 增加对象级别的权限控制
3. 用户ID不要用连续的自增ID，可以用UUID或者加密后的ID

以上是我的一些建议，可能有考虑不周全的地方，您参考。"

**结果：**
漏洞很快确认了，而且因为报告质量高，还额外给了一点奖励加成。

后来，阿明和这个审核员熟了，有时候有问题还会请教，审核员也会给一些指点。阿明进步很快。

**经验总结：**
1. 审核员问问题是好事，说明认真看了。
2. 耐心回答每一个问题，不嫌麻烦。
3. 主动提供更多信息，帮审核员快速理解。
4. 良好的沟通关系，对你以后提交漏洞也有好处。
5. 和审核员做朋友，能学到不少东西。

### 2.6.4 沟通的通用技巧

总结一下和审核员沟通的通用技巧：

**1. 礼貌专业**
- 开头说"您好"，结尾说"谢谢"、"辛苦了"
- 语气客气，尊重对方
- 专业，就事论事

**2. 及时回复**
- 审核消息尽快回，不要拖
- 超过24小时内回复比较好
- 实在忙的话，可以先说一声

**3. 有理有据**
- 说什么都要有依据
- 不要空口说白话
- 提供证据和证明

**4. 换位思考**
- 站在审核员角度想问题
- 审核员每天看很多报告，很忙
- 尽量把报告写清楚，帮省时间

**5. 保持耐心**
- 审核需要时间，不要催
- 有问题慢慢沟通，不要急
- 互相理解，互相尊重

**6. 不要做的事：**
- 不要骂人、不要吵架
- 不要胡搅蛮缠
- 不要人身攻击
- 不要威胁
- 不要重复催审

## 2.7 申诉技巧：如何争取应得的奖励

对漏洞等级或奖励有异议，可以申诉。但申诉不是闹，要有技巧。

### 2.7.1 什么时候该申诉，什么时候不该申诉

**适合申诉的情况：**
1. 等级明显偏低，和标准差距大
2. 影响被低估了，可以证明影响更大
3. 等级判断有误，明显错判
4. 重复误判，明明不是重复
5. 奖励计算错误

**不适合申诉的情况：**
1. 本身就在两个等级之间，可上可下
2. 自己高估了漏洞危害
3. 没有充分的证据证明等级
4. 已经申诉过一次，维持了原判，还想申诉
5. 因为自己报告写得烂，导致等级低

申诉成功率高的申诉：
- 提供新的危害证明
- 更详细的利用场景
- 漏洞影响被低估的证据
- 等级判定有事实错误

### 2.7.2 申诉的正确姿势

**申诉前的准备：**

1. **先仔细想清楚**
   - 为什么觉得等级低？
   - 理由是什么？
   - 有没有证据支持？
   - 成功概率大不大？

2. **收集证据**
   - 更多的危害证明
   - 更大的影响范围证明
   - 更容易的利用方法
   - 其他平台类似案例参考

3. **组织语言**
   - 怎么说审核员更容易接受
   - 逻辑清晰，有理有据
   - 语气礼貌，不卑不亢

**申诉信的结构：**

1. **开头：礼貌开场**
   "您好，关于漏洞编号XXXX，我想申请复核一下等级，麻烦了。"

2. **表达理解**
   "我理解平台给出的XX级评定，也尊重审核团队的判断。"
   先表达理解，再讲理由，不要上来就质疑。

3. **陈述理由（重点！）**
   分点说明为什么觉得应该更高等级：
   - 理由一：影响范围更大
   - 理由二：利用难度更低
   - 理由三：危害程度更严重
   - 每条理由都要有证据支撑

4. **补充证据**
   附上更多的证明材料：
   - 更多截图
   - 更详细的利用过程
   - 危害分析
   - 数据支撑

5. **结尾：礼貌收尾**
   "以上是我的一些看法，可能有考虑不周的地方，麻烦审核团队再评估一下。非常感谢！"

### 2.7.3 申诉成功案例

**案例：一个中危升高危**

**漏洞情况：**
某白帽子提交了一个存储型XSS漏洞，在用户评论区。审核给了中危，理由是"需要管理员审核才能显示，影响范围有限。

白帽子觉得这个漏洞应该是高危，因为：
1. 评论量很大，每天很多人看
2. 虽然需要审核，但审核是机器审核，很容易绕过
3. 可以窃取用户Cookie，危害大

**申诉信：**

"您好，关于漏洞XXXX的存储型XSS漏洞，我想申请复核一下等级。

我理解平台给出中危的评定，因为评论需要审核。但我认为这个漏洞实际上影响还是比较大的，理由如下：

1. **审核机制可以绕过**：
   我测试了一下，评论的审核是关键词过滤，很容易绕过。我试了几种变形的XSS Payload都能通过审核，正常显示。
   我再补充了3种不同的绕过方式的截图。

2. **影响范围大**：
   这个评论区是热门商品的评论区，每天浏览量很大。
   存储型XSS可以窃取浏览用户的Cookie，进而登录用户账号。
   用户账号里有手机号、地址、订单等敏感信息。

3. **利用难度低**：
   只需要注册账号，发一条评论就行
   不需要用户任何交互，只要浏览就会触发
   危害较大。

我再补充一下完整的利用链：
发评论 → 通过审核 → 用户浏览 → 窃取Cookie → 登录用户账号 → 获取用户敏感信息

以上是我的看法，可能有考虑不周全的地方，麻烦团队再评估一下。

非常感谢！"

**结果：**
审核重新评估后，调整为高危，奖励从2000元升到8000元。

### 2.7.4 申诉失败案例

**案例：高危想升严重，失败**

**漏洞情况：**
某白帽子提交了一个SQL注入漏洞，自己觉得是严重，审核给了高危。

他觉得不服，申诉。但他的申诉信是这么写的：

"这个漏洞明明是严重，为什么给高危？我觉得就是严重！你们会不会评？这么大的影响才高危？"

结果：申诉被驳回，维持高危。而且还被备注了"态度恶劣，以后审核更严格。

**为什么失败：**
1. 没有任何理由，就是情绪发泄
2. 态度不好，不尊重审核员
3. 没有提供任何新的证据
4. 胡搅蛮缠

### 2.7.5 申诉的注意事项

1. **一次就好**
   申诉一次就够了，不要反复申诉。
   申诉一次被驳回，就接受吧。
   反复申诉只会让审核员烦。

2. **对事不对人**
   就事论事，说漏洞本身。
   不要人身攻击，不要质疑审核员。
   不要说"你们会不会"这种话。

3. **接受结果**
   申诉成功，恭喜。
   申诉失败，也没关系。
   吸取经验，下次写更好的报告。
   不要耿耿于怀。

4. **提升自己**
   与其花时间申诉，不如花时间挖更多更好的漏洞。
   提升自己挖洞能力，比什么都强。

## 2.8 平台活动攻略

平台活动是冲奖励的好机会，活动期间奖励通常有加成，还有排名奖励。做好活动，收入翻倍不是梦。

**节日活动：**
- 春节活动
- 五一活动
- 国庆活动
- 双11活动
- 618活动
- 圣诞、元旦等
- 一般奖励1.5倍、2倍，甚至更高

**专项活动：**
- 小程序安全专项
- 云安全专项
- AI安全专项
- 移动安全专项
- 业务安全专项
- 某个新产品上线专项

**排名赛：**
- 7天赛
- 月度赛
- 季度赛
- 年度赛
- 奖金池制，排名越高奖励越多
- 冠军可能拿几万甚至几十万

**新平台上线活动：**
- 新企业入驻补天
- 新SRC上线
- 通常前几个月有加成
- 第一个漏洞奖励更高
- 竞争小，漏洞多

### 2.8.2 活动参与攻略

**活动前：**

1. **关注公告**
   - 每天看平台公告
   - 关注公众号
   - 加入社群，第一时间知道消息
   - 提前知道活动，早做准备

2. **提前准备**
   - 活动开始前就选好目标
   - 提前做信息收集
   - 提前找好疑似漏洞点
   - 活动一开始就提交
   - 抢占先机

3. **研究规则**
   - 活动规则仔细看
   - 奖励怎么算
   - 排名怎么排
   - 有没有什么特殊要求
   - 别等活动结束了才发现理解错了

**活动中：**

1. **集中精力**
   - 活动期间全力以赴
   - 其他事情放一放
   - 集中冲活动
   - 时间有限，抓紧

2. **主攻高价值**
   - 优先高危、严重漏洞
   - 积分多，排名涨得快
   - 不要浪费时间在低危上
   - 质量比数量重要

3. **及时提交**
   - 发现了就提交
   - 不要攒着
   - 早提交早审核
   - 避免被别人抢了先

4. **关注排名**
   - 时不时看看自己排第几
   - 差距多少
   - 调整策略
   - 要不要再冲一冲

**活动后：**

1. **跟进审核**
   - 活动提交的漏洞跟进审核
   - 有问题及时回复
   - 争取都能确认
   - 该申诉申诉

2. **总结经验**
   - 这次活动收获怎么样
   - 哪些地方做得好
   - 哪些地方可以改进
   - 下次活动怎么更好

3. **休息调整**
   - 活动期间累了，休息休息
   - 劳逸结合
   - 准备下一次活动

### 2.8.3 排名赛攻略

排名赛竞争激烈，想拿好名次不容易。

**排名赛特点：**
- 时间短，竞争激烈
- 奖金池大，奖励丰厚

   - 提前信息收集
   - 提前找好漏洞点
   - 比赛一开始就提交

   - 先提交的积分高

   - 手快有手慢无

   - 不要浪费时间低危

   - 增加产出概率

   - 合理安排时间
   - 不要透支健康

## 2.9 多平台管理策略

白帽子做到一定程度，都会同时运营多个平台。多平台运营，能最大化收入，分散风险。但多平台也需要管理，不然会乱。

### 2.9.1 为什么要多平台运营

**好处：**

1. **收入最大化**
   - 一个漏洞选奖励最高的平台提交
   - 多个平台都有产出
   - 东边不亮西边亮
   - 总收入更高

2. **分散风险**
   - 不要把鸡蛋放一个篮子
   - 一个平台没漏洞了，还有其他的
   - 一个平台规则变了，还有其他的
   - 抗风险能力强

3. **更多机会**
   - 每个平台活动不一样
   - 每个平台业务不一样
   - 机会更多，选择更多
   - 锻炼能力

4. **积累人脉**
   - 认识更多的人
   - 更多的交流机会
   - 更多的职业机会

**坏处：**

1. **精力分散**
   - 平台多了，每个都要花时间
   - 容易每个都挖不深
   - 反而影响产出

2. **管理成本高**
   - 账号多了，管理麻烦
   - 每个平台规则不一样，要记的东西多
   - 跟进审核也麻烦

3. **质量可能下降**
   - 精力分散，每个都不精
   - 报告质量可能下降
   - 通过率可能下降

所以，多平台运营有利有弊，要根据自己的情况选择。

### 2.9.2 多平台运营的几种模式

**模式一：主辅结合**

- 1-2个主要平台，深耕
- 2-3个辅助平台，偶尔挖挖
- 主要平台贡献主要收入
- 辅助平台补充收入和机会

适合大多数人，比较平衡。

**模式二：全面撒网**

- 注册很多平台
- 每个都试试看
- 哪个有洞挖哪个
- 广撒网，多敛鱼

适合新手练手，找感觉。

**模式三：专项专攻**

- 专注某一类型漏洞
- 哪个平台有这类漏洞就去哪个
- 比如专门挖逻辑漏洞
- 比如专门挖云安全

适合有专长的白帽子。

**模式四：活动驱动**

- 平时挖1-2个平台
- 哪个平台搞活动就去哪个
- 活动期间冲一波
- 平时正常挖

性价比比较高的模式。

### 2.9.3 多平台账号管理

**账号信息管理：**

平台多了，账号密码容易忘。怎么办？

1. **统一管理工具**
   - 用密码管理器
   - 比如1Password、LastPass、Bitwarden等
   - 一个主密码，管理所有账号
   - 安全又方便

2. **表格记录**
   - 建一个Excel表格
   - 记录每个平台：
     - 平台名称
     - 用户名
     - 注册邮箱
     - 手机号
     - 绑定的收款方式
     - 账号等级/信誉
     - 备注

3. **用户名策略**
   - 不同平台可以用相似的用户名
   - 方便记忆
   - 但密码不要一样
   - 安全第一

**账号安全注意事项：**
- 每个平台密码不一样
- 开启二次验证（2FA）
- 不要在公共设备登录
- 定期改密码
- 注意钓鱼

### 2.9.4 漏洞提交管理

挖了漏洞，提交到哪个平台？

**选择原则：**

1. **奖励优先**
   - 哪个平台奖励高，提交哪个
   - 前提是在测试范围内
   - 收益最大化

2. **速度优先**
   - 哪个平台审核快，提交哪个
   - 想快速确认，快速拿奖励
   - 时间也是钱

3. **通过率优先**
   - 哪个平台容易通过，提交哪个
   - 新手期先保证通过
   - 建立信心

4. **活动优先**
   - 哪个平台在搞活动，提交哪个
   - 活动有加成，更划算
   - 活动期间奖励高

**注意事项：**
- 一个漏洞不要同时提交多个平台！
- 属于违规，会被封号的
- 选一个最合适的提交
- 不要贪多

### 2.9.5 时间与精力管理

多平台运营，时间怎么分配？

**时间分配建议：**

- **主要平台：60%-70%的时间
- 辅助平台：20%-30%的时间
- 学习充电：10%的时间

**根据活动调整：**
- 平时正常分配
- 哪个平台有活动，多分配时间
- 活动结束，回归正常

**效率提升技巧：**

1. **工具自动化**
   - 信息收集自动化
   - 批量扫描
   - 报告模板化
   - 节省时间

2. **方法论复用**
   - 挖洞方法论通用的
   - 换个平台也能用
   - 经验可以
   - 熟能生巧

3. **批量处理**
   - 信息收集一批做
   - 报告一批写
   - 提高效率

4. **定期复盘**
   - 每周复盘
   - 看看时间花得值不值
   - 哪些平台产出高
   - 调整分配

### 2.9.6 多平台运营的注意事项

1. **不要贪多**
   - 不要一下注册十几个平台
   - 顾不过来，反而都做不好
   - 慢慢加，一个一个来
   - 稳扎稳打

2. **重点突出**
   - 要有重点
   - 1-2个主要平台深耕
   - 其他辅助
   - 平均用力不行

3. **遵守规则**
   - 每个平台规则都不一样
   - 每个平台规则都要看
   - 不要A平台的习惯带到B平台
   - 避免违规

4. **注意信誉**
   - 每个平台账号信誉都要维护
   - 提交高质量漏洞
   - 良好沟通
   - 珍惜每个账号

## 2.10 各平台避坑指南

每个平台都有各自的坑，一不小心就可能踩。下面总结各平台容易踩的坑和注意事项。

### 2.10.1 通用避坑总原则

**所有平台通用的坑：**

**坑一：不看规则就开干**
- 后果：违规，封号
- 避坑：每个平台注册完先读规则
- 重点：测试范围、禁止行为、奖励标准

**坑二：超出测试范围**
- 后果：漏洞作废，甚至封号
- 避坑：测试前确认在范围内
- 不确定的先问

**坑三：违规操作**
- 后果：封号，法律责任
- 避坑：禁止的绝对做
- 点到为止，不要深入

**坑四：重复提交**
- 后果：浪费时间，影响信誉
- 避坑：提交前搜搜有没有人提过
- 新漏洞尽快提交

**坑五：报告质量差**
- 后果：通过率低，等级低
- 避坑：认真写报告
- 步骤详细，截图完整

### 2.10.2 大厂SRC避坑指南

**阿里ASRC容易踩的坑：**

1. **测试范围要仔细看**
   - 阿里业务线很多，不是所有阿里业务都在范围内
   - 蚂蚁集团（支付宝）部分是单独的
   - 注意区分
   - 不确定的先问

2. **支付相关要谨慎**
   - 支付漏洞验证即可
   - 不要真的造成资金流动
   - 用小额测试
   - 测试完及时退回

3. **阿里云单独注意**
   - 云服务相关测试范围
   - 不要测试其他用户的云资源
   - 只测自己的
   - 注意云安全规则要看

4. **报告质量要求高**
   - 阿里对报告质量要求高
   - 写得太敷衍容易被打回
   - 认真写，详细点
   - 提高通过率

**腾讯TSRC容易踩的坑：**

1. **第三方业务不算**
   - 公众号、小程序很多是第三方的
   - 这些不算腾讯的
   - 要看主体
   - 不要白忙活

2. **游戏业务规则多**
   - 游戏漏洞规则多
   - 注意不要影响其他玩家
   - 不要用漏洞牟利
   - 测试用测试服？不对，是正式服谨慎
   - 按规则来

3. **微信生态谨慎**
   - 微信相关影响大
   - 测试要格外小心
   - 不要涉及用户隐私
   - 不要造成大面积影响
   - 点到为止

4. **审核严格**
   - 腾讯审核相对严
   - 纯理论漏洞不收
   - 一定要有实际影响证明
   - 证明要充分

**字节BSRC容易踩的坑：**

1. **TikTok不要乱测**
   - 国际版TikTok是单独的
   - 需要单独申请权限
   - 不要擅自测试
   - 违规后果严重

2. **新业务变化快**
   - 字节业务变化快
   - 测试范围可能更新
   - 及时看最新公告
   - 不要凭老印象

3. **竞争激烈**
   - 字节高手多
   - 漏洞手慢无
   - 发现了尽快提交
   - 别磨磨蹭蹭

### 2.10.3 第三方平台避坑指南

**补天平台容易踩的坑：**

1. **企业规则不一样**
   - 每个入驻企业规则可能不一样
   - 不要以为所有企业都一样
   - 每个企业都要看单独的规则
   - 不要想当然

2. **积分兑换注意**
   - 积分兑换有手续费吗？
   - 兑换到账时间？
   - 有没有最低兑换额？
   - 都了解清楚

3. **企业质量参差不齐**
   - 有的企业好，有的差
   - 奖励高低不同
   - 审核速度不同
   - 先了解清楚再挖

4. **排名奖励规则**
   - 月度排名怎么算
   - 有没有什么额外要求
   - 看清楚规则
   - 别白忙活

**漏洞盒子容易踩的坑：**

1. **项目规则要看清**
   - 每个项目规则不同
   - 奖励方式不同
   - 时间不同
   - 都看清楚

2. **私有项目申请**
   - 私有项目需要资格
   - 提升技能等级
   - 提高通过率
   - 慢慢积累

3. **众测项目排名**
   - 排名赛制
   - 竞争激烈
   - 规则要看懂
   - 奖金怎么算

### 2.10.4 金融/银行类SRC避坑指南

银行、金融类SRC，要格外小心。

**特别注意：**

1. **规则更严**
   - 金融行业规则更严
   - 禁止的更多
   - 一定要仔细读规则
   - 不要越界

2. **数据更敏感**
   - 金融数据极其敏感
   - 绝对不要下载数据
   - 验证即可，点到为止
   - 只用自己的测试账号

3. **支付格外小心**
   - 支付相关漏洞要非常谨慎
   - 不要造成实际资金损失
   - 用最小金额测试
   - 测试完及时返还

4. **法律风险更高**
   - 金融行业法律风险更高
   - 违规后果更严重
   - 严格遵守规则
   - 拿不准先问

### 2.10.5 避坑终极心法

最后，总结一下避坑的心法：

1. **规则先行**
   做什么之前先看规则，规则。

2. **不确定就问**
   拿不准的，先问客服或审核员。
   不要猜，不要赌。

3. **点到为止**
   验证漏洞即可，不要深入利用。
   证明有危害就够了，不要真的造成危害。

4. **保守为了安全第一**
   安全第一，奖励第二。
   不要为了钱违规。
   保护好自己，比什么都重要。

5. **长期主义**
   做SRC是长期的事，
   不要为了一个漏洞毁了自己的账号和前途。
   细水长流，稳稳当当。

## 2.11 SRC平台最佳实践

最后，总结一些在SRC平台混的最佳实践，帮助你少走弯路。

### 2.11.1 提高漏洞通过率

**提交前检查清单：**

提交漏洞之前，对照检查一遍：

- [ ] 漏洞在测试范围内吗？
- [ ] 漏洞真实存在吗？多次复现了吗？
- [ ] 是误报吗？
- [ ] 漏洞等级判断合理吗？
- [ ] 报告写清楚了吗？
- [ ] 步骤详细吗？
- [ ] 截图都有了吗？
- [ ] PoC能直接运行吗？
- [ ] 修复建议具体吗？
- [ ] 有没有人提交过？
- [ ] 有没有违规操作？

**报告撰写黄金法则：**

1. **标题要抓眼球**
   一眼看懂是什么漏洞
   简洁明了，重点突出

2. **步骤要像菜谱**
   一步一步，编号
   审核员照着做就能复现
   越详细越好

3. **截图要说话**
   关键步骤都有截图
   重点信息红框标注
   清晰明了

4. **危害要讲透**
   不仅说有漏洞，还要说有什么危害
   影响多大，利用多容易
   帮审核员理解为什么这个等级

5. **建议要具体**
   修复建议要具体可行
   不要空泛
   体现专业度

### 2.11.2 与审核员相处之道

**尊重审核员**
- 审核员也是人，也会累
- 每天看很多报告，很辛苦
- 互相理解，互相尊重
- 你尊重别人，别人也尊重你

**及时回复**
- 审核员的消息尽快回
- 不要拖着不回
- 节省双方时间，效率高

**有理有据**
- 有不同意见好好说
- 拿出证据来
- 不要胡搅蛮缠

**建立良好关系**
- 和审核员保持良好沟通
- 以后提交多了，熟了
- 有时候会有意想不到的收获
- 比如技术交流，学习东西

### 2.11.3 提升排名的策略

想提升平台排名，可以从这几个方面入手：

**数量策略**
- 多挖洞，多提交
- 积少成多
- 低危也是积分
- 适合新手

**质量策略**
- 深耕高危严重
- 一个顶十个
- 质量取胜
- 适合有经验的

**活动策略**
- 活动期间冲一波
- 活动加成多
- 排名涨得快
- 性价比高

**新业务策略**
- 盯着新业务
- 新业务漏洞多
- 竞争小
- 容易出成果

### 2.11.4 长期发展建议

**打造个人品牌**
- 起个好记的ID
- 提交高质量漏洞
- 写技术文章分享
- 参加活动露脸
- 慢慢积累名气

**拓展人脉圈子**
- 加入社群
- 多和同行交流
- 认识平台的人
- 人脉就是资源

**规划职业发展**
- SRC不是一辈子挖洞
- 考虑未来发展
- 是去企业？还是自己干？
- 早做规划

**保持学习进步**
- 技术在更新
- 要一直学习
- 不进则退
- 保持竞争力
