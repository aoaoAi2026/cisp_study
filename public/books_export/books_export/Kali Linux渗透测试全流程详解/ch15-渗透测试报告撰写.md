## 15.1 先讲一个故事

### 15.1.1 "技术很强，但报告太烂"

我刚入行的时候，
认识一个技术很厉害的哥们。
打靶机、做CTF，
他都是队里最厉害的，
什么漏洞一找一个准。
但他有个毛病——
不会写报告。

有一次我们一起做一个项目，
他打进去了十几台机器，
发现了二十多个高危漏洞，
收获满满。
然后到了写报告的时候，
他写出来的东西，
我一看就懵了——
满篇都是技术术语，
漏洞就是一句话"存在SQL注入"，
然后贴了一堆截图，
连个修复建议都没有。
客户的非技术人员看了，
根本不知道说的是什么。

最后项目交付的时候，
客户很不满意，
说："你们技术行不行啊？
报告写得这么乱，
我们看不懂，
也不知道怎么改。"
我们老大赶紧让我把报告全部重写了一遍，
才交差。

你看，
技术再强，
报告写不好，
客户也不买账。
因为客户要的不是"你打进去了"，
而是"我们有什么问题、该怎么修"。

从那之后我就明白了：
**渗透测试，七分靠打，三分靠写。
报告写不好，
活干得再好也白搭。**

这是渗透测试工程师的必备技能。
写好报告，

## 15.2 报告的目的与受众

在讲怎么写之前，
先想清楚两个问题：
**报告是写给谁看的？
目的是什么？**

### 15.2.1 报告的受众

一份渗透测试报告，
看的人可能有几类：

```
  1. 高层领导 / 管理层
     • 关心的是：风险有多大？
       影响业务吗？要花多少钱修？
       优先级是什么？
     • 不关心技术细节，不关心你怎么打进去的
     • 要的是结论、是风险、是建议

  2. 安全负责人 / 安全团队
     • 关心的是：哪里有漏洞？
       是什么类型的漏洞？
       严重程度怎么样？怎么复现？
       修复建议是什么？
     • 需要比较详细的技术细节

  3. 开发 / 运维人员
     • 关心的是：
       这个漏洞在哪个地方？
       怎么复现？具体怎么修？
     • 需要非常详细的复现步骤和修复方案
     • 最好能照着步骤一步步修好
```

所以一份好的报告，
要兼顾不同的受众。
- 前面要有摘要、有结论，给管理层看
- 中间要有漏洞详情、风险分析，给安全团队看
- 后面要有详细的复现步骤、修复建议，给开发运维看

不同的人各取所需，
都能从报告里得到他想要的东西。

### 15.2.2 报告的目的

报告的目的是什么？
不是为了炫耀你技术多厉害，
也不是为了把客户骂一顿。
报告的目的是：

```
  1. 说明问题
     清楚地告诉客户，哪里有问题、是什么问题

  2. 评估风险
     这个漏洞有多严重？影响多大？
     让客户知道事情的轻重缓急

  3. 给出解决方案
     告诉客户该怎么修、怎么防
     这才是客户最想要的

  4. 体现专业
     报告专业不专业，
     直接影响客户对你、对公司的评价
     好的报告是最好的名片
```

记住：
**报告是给客户解决问题的，
不是秀技术的。**
写得越让客户容易理解、
越容易落地整改，
报告质量就越高。

## 15.3 报告的基本结构

一份标准的渗透测试报告，
一般都有哪些部分？
我给你一个通用的结构，
你可以根据实际情况调整。

```
一份典型的渗透测试报告结构：

  1. 封面
  2. 文档信息 / 版本历史
  3. 执行摘要（Executive Summary）
  4. 测试概述
     - 测试范围
     - 测试时间
     - 测试方法
     - 测试环境
  5. 总体风险评估 / 漏洞概览
     - 漏洞统计
     - 风险评级说明
  6. 漏洞详情
     - 按严重程度排序
     - 每个漏洞：描述、危害、复现步骤、修复建议
  7. 整体安全建议
  8. 附录
     - 术语表
     - 工具清单
     - 参考资料
     - 等等
```

我们一个一个说。

## 15.4 各部分怎么写

### 15.4.1 封面 & 文档信息

这部分比较简单。
封面写上：
- 项目名称
- 渗透测试报告
- 客户名称
- 测试单位
- 报告日期
- 版本号

然后是文档信息/版本历史：
| 版本 | 日期 | 作者 | 变更说明 |
|------|------|------|----------|
| v1.0 | 2024-xx-xx | xxx | 初稿 |
| v1.1 | 2024-xx-xx | xxx | 根据客户反馈修改 |

有个版本历史，
显得专业，
也方便追溯。

### 15.4.2 执行摘要（Executive Summary）

这部分非常重要，
是给管理层看的。
很多领导只看这一页。

执行摘要写什么？
```
  • 测试背景：为什么做这次测试
  • 测试范围：测了哪些东西
  • 测试结论：总体怎么样，安全状况如何
  • 发现的主要问题：最严重的几个问题，几句话说清楚
  • 风险等级：整体风险是高/中/低
  • 总体建议：大概该怎么做
```

要点：
- **简洁**：别太长，一两页就够了
- **通俗易懂**：别说太多技术术语，领导听不懂
- **结论先行**：先给结论，再说细节
- **突出重点**：最严重的问题、最大的风险先说

别写得又臭又长，
没人看。
领导能从这一页里明白：
"我们现在安全状况怎么样、
最大的风险是什么、
大概要做什么"，
就够了。

### 15.4.3 测试概述

这部分说明测试的基本情况。

包括：
```
  • 测试目标 / 测试范围
    - 测了哪些系统、哪些域名、哪些IP段
    - 哪些在范围内，哪些不在
    - 这个要写清楚，避免后面扯皮

  • 测试时间
    - 什么时候开始、什么时候结束

  • 测试方法 / 测试依据
    - 依据什么标准做的测试
    - 比如OWASP Top 10、PTES渗透测试执行标准...
    - 用的什么方法（黑盒/白盒/灰盒）

  • 测试环境 / 测试条件
    - 测试期间有没有什么特殊情况
    - 有没有什么限制

  • 测试人员
    - 谁参与了测试
```

这部分是交代背景，
让看报告的人知道这次测试是怎么做的、
测了哪些东西。

### 15.4.4 总体风险评估 & 漏洞概览

这部分是给安全团队和管理层看的，
一目了然看到有多少漏洞、
严重程度分布。

一般包括：
```
  • 漏洞数量统计
    - 高危多少个、中危多少、低危多少、信息泄露多少
    - 用图表展示更好看（饼图、柱状图）

  • 风险等级说明
    - 高危、中危、低危、信息泄露
      分别是怎么定义的
    - 评分标准是什么（比如用CVSS）

  • 漏洞分布
    - 按系统分、按类型分
    - 哪个系统问题最多
    - 哪类漏洞最常见

  • 整体安全评价
    - 整体安全状况怎么样
    - 主要风险点在哪里
```

用图表、用数据说话，
直观、清晰。

### 15.4.5 漏洞详情（核心部分！）

这是报告的核心，
也是篇幅最长的部分。
所有发现的漏洞，
一个一个详细写清楚。

漏洞怎么排序？
一般按严重程度从高到低排，
高危的放前面，
先看最严重的。

每个漏洞应该包含哪些内容？
我给你一个模板：

```
漏洞编号：VUL-001
漏洞名称：XXX系统登录处SQL注入漏洞
严重程度：高危
影响系统：xxx.xxx.com
发现时间：2024-xx-xx

一、漏洞描述
  一句话说清楚是什么漏洞、在哪里

二、漏洞危害
  这个漏洞能造成什么影响
  比如：可获取数据库所有数据、
        可直接获取服务器权限...

三、复现步骤
  一步一步写清楚怎么复现
  1. ...
  2. ...
  3. ...
  配上截图，标重点

四、风险等级/评分
  给个评分，比如CVSS分数
  为什么是高危/中危/低危

五、修复建议
  具体怎么修
  要详细、可操作
  开发拿到就能照着改
```

写漏洞详情，
有几个要点：

**1. 描述要清晰准确**
别含糊其辞，
"好像有个漏洞"、"可能有问题"，
不行。
要确定、要准确。

**2. 复现步骤要详细**
一步一步写，
最好别人照着你的步骤，
能复现出来。
该给的请求、该给的参数，
都写清楚。
别让人猜。

**3. 危害要讲清楚，但别夸大**
实事求是，
是什么危害就说什么危害。
别一个低危漏洞吹成高危，
也别把高危说轻了。
客观、准确。

**4. 修复建议要具体、可落地**
这是最重要的。
别只说"过滤用户输入"、"加强验证"，
太空了，开发不知道怎么改。
要具体，
比如：
"使用预编译SQL语句（PreparedStatement），
不要拼接SQL字符串。
参考代码示例：......"

最好给出具体的修复方案，
甚至代码示例。
开发拿到就能用，
这样的报告才有价值。

**5. 截图要清晰，重点要标注**
截图别糊成一团，
也别一整张大图啥都有。
要清晰，
关键地方用红框标出来，
让人一眼就能看到问题在哪。

### 15.4.6 整体安全建议

除了单个漏洞的修复建议，
还要有一个整体的、宏观的安全建议。
从体系上、从管理上，
给客户一些建议。

比如：
```
  • 建立安全开发流程（SDL）
  • 定期做安全培训
  • 部署WAF、IPS等防护设备
  • 建立漏洞管理流程
  • 定期做渗透测试
  • 加强安全运营
  • 等等
```

这部分是拔高的，
体现你的专业性。
告诉客户，
光修单个漏洞不够，
要从体系上提升安全能力。

### 15.4.7 附录

最后是附录，
放一些补充信息。

比如：
```
  • 术语表
    报告里用到的专业术语，解释一下
    方便非技术人员看

  • 工具清单
    测试用了哪些工具
    比如：Nmap、Burp Suite、SQLMap、Metasploit...

  • 参考资料 / 参考链接
    漏洞相关的参考文章、CVE编号...

  • 免责声明
    法律上的一些声明
    （这个很重要，一定要有）
```

## 15.5 写报告的几个原则

讲完结构，
再讲几个写报告的原则。

### 原则一：客观、准确、实事求是

报告一定要客观。
- 发现了就是发现了，没发现就是没发现
- 高危就是高危，低危就是低危
- 别夸大，也别隐瞒
- 不确定的东西，就说不确定，别瞎写

实事求是，
是专业的基本要求。

### 原则二：条理清晰，逻辑清楚

别写得东一榔头西一棒子。
结构要清晰，
层次要分明。
- 大标题、小标题、编号
- 分点说明，别一大段一大段的
- 重要的内容加粗、标红
- 能用图表的用图表

让人看着舒服，
找东西好找。

### 原则三：语言专业，避免口语化

写报告不是聊天，
别用口语化的表达。
别写"我点了一下，就弹出来了"、
"这个地方挺危险的"。
要专业一点：
"点击提交按钮后，系统返回了敏感信息。"

当然，
专业不代表晦涩难懂。
要专业，但也要让人看得懂。
别堆砌术语，
以为越难懂越厉害，
那是装逼，不是专业。

### 原则四：站在读者的角度写

写的时候想想，
读者想看什么？
他能看懂吗？
他拿到这个报告，
知道该怎么做吗？

- 给管理层看的部分，结论先行，少说技术
- 给开发看的修复建议，要具体、可操作
- 别自说自话，自己写嗨了，别人看不懂

### 原则五：注意保密

报告里全是敏感信息——
客户的漏洞、系统、数据...
一定要注意保密。
- 报告文件要加密
- 发送要走安全渠道
- 别随便乱放、随便发
- 测试完了该删的要删

保密是职业操守，
也是法律要求。

## 15.6 常见的坑

最后讲几个新手写报告常见的坑，
别踩。

### 坑一：只有截图，没有文字

很多新人写报告，
贴一堆截图，
啥文字说明都没有。
别人看了一脸懵，
不知道你想表达什么。

截图是辅助，
文字才是主要的。
截图要配合文字说明，
告诉别人这张图说明什么问题、
重点在哪里。

### 坑二：修复建议太笼统

这个前面说过了。
别只说"加强过滤"、"注意安全"，
没用。
要具体，
告诉人家怎么改、
改哪里、
为什么这么改。

### 坑三：漏洞描述含糊不清

"某个地方存在漏洞"、
"可能有安全隐患"，
这种话就别说了。
要说清楚：
- 哪个系统、哪个URL、哪个参数
- 是什么漏洞
- 怎么回事

别含糊。

### 坑四：错别字、格式混乱

这个是低级错误，
但很多人犯。
- 错别字一大堆
- 格式乱七八糟
- 字体不统一
- 编号错乱
- 截图糊

这些细节虽然不影响技术内容，
但很影响观感。
客户一看报告这么糙，
自然会觉得你干活也糙。

写完了自己检查一遍，
别着急交。
细节决定成败。

### 坑五：只说问题，不给方案

有的报告列了一堆漏洞，
每个都写得很详细，
但就是没有修复建议，
或者修复建议写得很敷衍。

客户要的不只是"你告诉我有问题"，
更是"你告诉我怎么修"。
修复建议写得好不好，
直接体现你的水平。

## 15.7 本章总结

这一章我们讲了渗透测试报告撰写，
内容相对轻松但很重要。
总结一下：

**报告的重要性：**
- 报告是最终交付物
- 七分靠打，三分靠写
- 好的报告体现专业度，也真正帮客户解决问题

**报告的受众：**
- 管理层 —— 看结论、看风险、看建议
- 安全团队 —— 看漏洞详情、看风险评估
- 开发运维 —— 看复现步骤、看修复方案
- 一份好报告要兼顾不同受众

**报告的基本结构：**
1. 封面 & 文档信息
2. 执行摘要（很重要！给领导看的）
3. 测试概述
4. 总体风险评估 & 漏洞概览
5. 漏洞详情（核心！）
   - 漏洞描述、危害、复现步骤、修复建议
6. 整体安全建议
7. 附录

**写报告的原则：**
- 客观、准确、实事求是
- 条理清晰、逻辑清楚
- 语言专业、通俗易懂
- 站在读者的角度写
- 注意保密

**常见的坑：**
- 只有截图没有文字
- 修复建议太笼统
- 漏洞描述含糊
- 错别字、格式乱
