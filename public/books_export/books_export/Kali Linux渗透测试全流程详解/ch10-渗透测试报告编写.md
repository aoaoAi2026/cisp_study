# 第十章 渗透测试报告编写——说得清才收得到钱

> 第10章 | 预计学习时间：3小时 | 难度：⭐⭐

---

## 📖 写在前面

很多新手觉得渗透测试就是打打打，
技术牛逼就行。

但实际上：
> **技术决定你能不能挖到洞，
> 报告决定你能不能拿到钱。**

你挖了10个高危漏洞，
但报告写得乱七八糟，
客户看不懂，
或者看不懂有多严重，
那人家就觉得你没干活，
钱都收不回来。

反过来，
你只挖到几个中危，
但报告写得专业、清楚、有说服力，
客户觉得你很靠谱，
还会找你做下一次。

这一章我们就来讲讲：
- 渗透测试报告怎么写
- 报告结构是什么样的
- 漏洞怎么描述
- 风险等级怎么评
- 怎么写才能显得专业

## 10.1 报告的重要性

### 10.1.1 报告是你的交付物

```
渗透测试是个服务，
服务是看不见摸不着的，
客户能拿到什么？
就是一份报告。

报告 = 你的工作成果
报告 = 你的专业水平
报告 = 客户付钱的依据

客户不懂技术，
他判断你干得好不好，
全靠看报告。
```

### 10.1.2 好报告和坏报告的区别

```
❌ 坏报告：
  • 错别字一大堆
  • 格式乱，排版丑
  • 漏洞描述就一句话
  • 没有复现步骤
  • 没有修复建议
  • 全是截图没有文字
  • 客户看完不知道啥意思

✅ 好报告：
  • 排版美观，格式统一
  • 逻辑清晰，结构完整
  • 漏洞描述清楚准确
  • 复现步骤详细，客户自己能跟着复现
  • 风险等级合理
  • 修复建议具体可操作
  • 技术和非技术人员都能看懂
```

## 10.2 渗透测试报告的一般结构

一份专业的渗透测试报告，
一般包含这些部分：

```
1. 封面
2. 目录
3. 修订历史
4. 执行摘要（Executive Summary）
5. 测试范围与方法
6. 整体风险评估
7. 漏洞详情
8. 修复建议
9. 附录
```

我们一个一个来讲。

### 10.2.1 封面

封面不用太复杂，
但基本信息要有：

```
┌─────────────────────────────────────┐
│                                     │
│                                     │
│        XX公司渗透测试报告            │
│                                     │
│                                     │
│                                     │
│  项目名称：XX系统渗透测试            │
│  测试时间：2024年X月X日 - X月X日     │
│  报告版本：V1.0                      │
│  测试团队：XX安全团队                │
│  客户名称：XX公司                    │
│  保密等级：机密                      │
│                                     │
└─────────────────────────────────────┘
```

### 10.2.2 目录

章节多的话要有目录，
方便客户找想看的部分。

```
目录

第一章 执行摘要 ....................................... 3
第二章 测试概述 ....................................... 5
  2.1 测试范围 ........................................ 5
  2.2 测试方法 ........................................ 6
  2.3 测试时间 ........................................ 7
第三章 整体风险评估 ................................... 8
  3.1 风险等级分布 .................................... 8
  3.2 总体结论 ........................................ 9
第四章 漏洞详情 ...................................... 10
  4.1 高危漏洞 ....................................... 10
    4.1.1 SQL注入漏洞（用户登录处） ................... 10
    4.1.2 任意文件上传 ................................ 15
  4.2 中危漏洞 ....................................... 20
    4.2.1 ......
  4.3 低危漏洞 ....................................... 30
  4.4 信息泄露 ....................................... 35
第五章 修复建议 ...................................... 40
第六章 附录 .......................................... 45
  附录A：漏洞风险等级说明 ............................ 45
  附录B：术语表 ..................................... 46
  附录C：测试工具清单 ............................... 47
```

### 10.2.3 修订历史

报告可能会改好几版，
要有修订历史记录。

```
修订历史

版本  日期         修改人   修改内容
V1.0  2024-01-01   张三     初始版本
V1.1  2024-01-05   李四     新增XX漏洞修复建议
V2.0  2024-01-10   张三     根据客户反馈调整风险等级
```

### 10.2.4 执行摘要（Executive Summary）

这是给领导看的！
领导没工夫看你几十个漏洞的细节，
他只想知道：
- 安不安全？
- 有多危险？
- 要花多少钱修？
- 先修哪个？

所以执行摘要要：
- 简洁，1-2页足够
- 非技术人员也能看懂
- 说清楚总体风险和优先级

执行摘要一般包含：

```
执行摘要

1. 项目背景
   简单说一下为什么做这次渗透测试，
   测试的是什么系统。

2. 测试范围
   测了哪些系统、哪些IP、哪些域名。
   （别写太长，一句话概括）

3. 测试结论
   总体风险等级：高/中/低
   共发现多少个漏洞：
     高危：X个
     中危：X个
     低危：X个
     信息泄露：X个

4. 最严重的几个问题
   列2-3个最严重的漏洞，
   用大白话讲清楚危害，
   别说技术术语。
   比如：
   "攻击者可以通过SQL注入漏洞获取所有用户的密码，
    包括管理员密码，从而控制整个系统。"

5. 修复优先级建议
   先修什么，后修什么。
   比如：
   "建议优先修复SQL注入和文件上传漏洞，
    这两个可以直接导致服务器被控制。"

6. 总体建议
   宏观层面的建议，
   比如加强安全培训、建立SDL流程等。
```

> 💡 **写执行摘要的诀窍**：
> 想象你在给老板汇报，
> 老板只有5分钟时间，
> 你怎么把事情说清楚？
> 就按这个思路写。

## 10.3 漏洞详情怎么写

这是报告的核心部分，
也是最考验功力的地方。

### 10.3.1 单个漏洞的标准结构

每个漏洞都应该包含这些部分：

```
【漏洞标题】
  一句话说清楚是什么漏洞、在哪

【风险等级】
  严重/高危/中危/低危/信息泄露

【影响范围】
  哪些URL、哪些功能受影响

【漏洞描述】
  详细解释这个漏洞是什么，怎么产生的

【复现步骤】
  一步步教别人怎么复现，
  要详细到客户的技术人员照着做就能复现出来

【验证截图/证据】
  截图或者抓包证据

【风险分析】
  这个漏洞能造成什么危害
  （技术危害 + 业务危害）

【修复建议】
  具体怎么修，越详细越好
  最好给代码示例

【参考链接】
  相关的CVE、官方文档、技术文章
```

### 10.3.2 漏洞标题怎么写

标题要清晰，
让人一看就知道是什么漏洞、在哪。

```
❌ 不好的标题：
  "SQL注入"
  （太笼统了，哪个地方的SQL注入？）
  
  "有漏洞"
  （废话，没漏洞写它干嘛）

✅ 好的标题：
  "用户登录接口存在SQL注入漏洞"
  "商品详情页id参数存在SQL注入"
  "头像上传处存在任意文件上传漏洞"
  "后台管理系统存在弱口令（admin/123456）"
```

公式：**位置 + 漏洞类型**

### 10.3.3 风险等级怎么评

一般用 **严重、高危、中危、低危、信息泄露** 五个等级。

怎么评呢？可以从这几个维度考虑：

```
评估维度：

1. 利用难度
   容易 → 风险更高
   很难 → 风险相对低

2. 影响范围
   全系统 → 高
   单个功能 → 中
   仅影响自己 → 低

3. 危害程度
   能直接拿服务器权限 → 严重/高危
   能偷数据 → 高危/中危
   只能弹个框 → 中危/低危
   只是泄露点信息 → 低危/信息泄露

4. 是否需要认证
   不用登录就能利用 → 风险更高
   需要管理员权限 → 风险相对低
```

一般可以参考 **CVSS评分系统**，
但不用太死板，
根据实际情况调整。

给个大致的参考：

| 等级 | 说明 | 例子 |
|------|------|------|
| 严重 | 直接能控制服务器，影响巨大 | 远程代码执行、SQL注入getshell |
| 高危 | 能获取敏感数据或较高权限 | SQL注入、文件上传getshell、越权访问 |
| 中危 | 有一定危害，但需要条件 | 存储型XSS、普通越权、CSRF |
| 低危 | 危害不大，或者很难利用 | 反射型XSS、信息泄露、弱验证码 |
| 信息泄露 | 只是泄露一些信息，没有直接危害 | 泄露服务器版本、路径泄露 |

### 10.3.4 复现步骤怎么写

复现步骤一定要详细，
详细到什么程度？

> 客户的技术人员，
> 按照你的步骤一步步做，
> 一定能复现出来。

```
❌ 不好的复现步骤：
  "访问url，输入payload，成功。"
  （太简略了，谁知道你说的啥）

✅ 好的复现步骤：

  1. 打开Burp Suite，配置浏览器代理
  2. 浏览器访问：http://target.com/login.php
  3. 随便输入用户名和密码，点击登录
  4. 在Burp里抓到登录请求，发送到Repeater
  5. 将username参数的值改为：
     admin' and 1=2 union select 1,version(),3 -- 
  6. 点击Send发送请求
  7. 在响应中可以看到数据库版本信息，
     证明存在SQL注入漏洞

  （附请求包和响应包的截图）
```

**要点**：
- 一步步来，编号清晰
- 写清楚每个参数怎么改
- payload完整贴出来
- 配截图（圈出重点）
- 说明怎么判断成功了

### 10.3.5 修复建议怎么写

修复建议不能太空，
要具体、可操作。

```
❌ 不好的修复建议：
  "加强过滤"
  "做好安全检查"
  "升级系统"
  （都是废话，谁不知道要加强？怎么加？）

✅ 好的修复建议：

  【修复方案】
  使用参数化查询（Prepared Statement），
  不要将用户输入直接拼接到SQL语句中。

  【代码示例】
  // 错误写法
  $sql = "SELECT * FROM users WHERE username = '" . $_POST['username'] . "'";
  $result = mysql_query($sql);

  // 正确写法（PDO）
  $stmt = $pdo->prepare('SELECT * FROM users WHERE username = :username');
  $stmt->bindParam(':username', $_POST['username']);
  $stmt->execute();

  【参考链接】
  PHP官方文档：https://www.php.net/manual/zh/pdo.prepared-statements.php
  OWASP SQL注入防御：https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html
```

**要点**：
- 说清楚原理（为什么这么修）
- 给代码示例（最好有错误写法和正确写法对比）
- 给参考链接（方便客户深入了解）
- 分短期修复和长期方案

## 10.4 报告写作技巧

### 10.4.1 排版和格式

```
排版注意事项：

1. 格式统一
   字体、字号、标题样式要统一
   别一会儿宋体一会儿黑体
   别有的地方标题大有的地方小

2. 编号清晰
   章节号、图号、表号要规范
   比如图4-1，表3-2

3. 截图规范
   • 截图要清晰，不能模糊
   • 重要的地方用红框圈出来
   • 截图大小适中，别太大也别太小
   • 每张图要有图号和图标题

4. 表格规范
   边框、对齐要整齐
   表头加粗

5. 有错别字？绝对不行！
   写完一定要检查错别字
   错别字形像非常差
   （建议写完自己读一遍，或者用工具查）
```

### 10.4.2 语言表达

```
写作注意事项：

1. 客观中立
   用事实说话，别带情绪
   别说"你们这个系统太烂了"
   要说"该系统存在XX漏洞，可能导致XX危害"

2. 专业但易懂
   技术细节要准确，
   但也要让非技术人员能看懂大概。
   第一次出现的术语可以加个括号解释一下。

3. 不说废话
   每句话都要有意义，
   别为了凑字数写一堆没用的。

4. 正面表达
   不说"你们没做XX"，
   说"建议增加XX功能/措施"

5. 中文标点
   中文用中文标点，
   英文用英文标点，
   别混着用。
```

### 10.4.3 一些容易犯的错误

```
常见错误：

1. 漏洞描述抄百度
   直接复制粘贴百度百科的定义，
   和目标系统一点关系都没有。
   → 要结合目标系统的具体情况来写

2. 只有截图没有文字
   甩一堆截图上去，
   啥说明都没有。
   → 截图是辅助，文字才是主体

3. 风险等级乱评
   把反射型XSS评成严重，
   把SQL注入评成低危。
   → 等级要合理，不然显得不专业

4. 修复建议不切实际
   "建议直接换系统"
   "建议全部重写"
   → 客户做不到的建议等于没说

5. 泄露测试方法
   把你是怎么绕过WAF的、
   用了什么0day，全写进去了。
   → 写清楚漏洞就行，
     没必要把你的"绝招"都告诉别人

6. 忘了打码
   截图里有客户的敏感信息
   （真实姓名、身份证、手机号...）
   直接放报告里了。
   → 敏感信息一定要打码！
```

## 10.5 漏洞详情示例

光说不练假把式，
我们来看一个完整的漏洞详情示例。

---

### 4.1.1 用户登录接口存在SQL注入漏洞

**【风险等级】** 高危

**【影响范围】**
- URL：http://target.com/api/login
- 参数：username
- 方法：POST

**【漏洞描述】**
用户登录接口的 `username` 参数未做严格的输入验证和过滤，直接拼接到SQL查询语句中，导致攻击者可以构造恶意SQL语句，获取数据库中的任意数据。

攻击者可以利用该漏洞：
- 获取所有用户的用户名和密码哈希
- 获取数据库中的所有业务数据
- 在特定情况下可以进一步提权，控制整个服务器

**【复现步骤】**

1. 配置浏览器代理，使用Burp Suite抓包
2. 访问登录页面：http://target.com/login
3. 用户名输入 `test`，密码随便输入，点击"登录"
4. 在Burp Suite中拦截到登录请求，将其发送到Repeater
5. 修改请求中的 `username` 参数为以下内容：
   ```
   test' UNION SELECT 1,2,3,4,version(),6,7-- -
   ```
6. 点击"Send"发送请求，在响应中可以看到数据库版本信息：
   ```
   "msg":"登录失败，用户不存在",
   "data": "5.7.26-log"
   ```
   其中 `5.7.26-log` 是数据库版本，说明SQL语句被成功执行

7. 进一步构造payload获取用户表数据：
   ```
   test' UNION SELECT 1,2,3,4,group_concat(username,0x3a,password),6,7 FROM users-- -
   ```
   响应中返回了所有用户的用户名和密码哈希。

**【验证截图】**

（此处放截图：Burp Repeater的请求和响应，红框圈出注入点和返回的数据）

图4-1 SQL注入漏洞复现截图

**【风险分析】**

技术风险：
- 攻击者可以读取数据库中的所有数据
- 包括用户信息、订单信息、业务数据等
- 如果数据库权限较高，可能写入文件或执行命令，导致服务器被控制

业务风险：
- 用户数据泄露，可能导致用户隐私泄露
- 管理员账号密码泄露，攻击者可以登录后台
- 可能造成业务数据被篡改或删除
- 存在合规风险（不符合等保、数据安全法等要求）

**【修复建议】**

1. 使用参数化查询（推荐）
   不要将用户输入直接拼接到SQL语句中，使用预编译语句（Prepared Statement）。

   错误写法（PHP示例）：
   ```php
   $username = $_POST['username'];
   $password = $_POST['password'];
   $sql = "SELECT * FROM users WHERE username = '$username' AND password = '$password'";
   $result = mysql_query($sql);
   ```

   正确写法（PDO示例）：
   ```php
   $username = $_POST['username'];
   $password = $_POST['password'];
   $stmt = $pdo->prepare('SELECT * FROM users WHERE username = ? AND password = ?');
   $stmt->execute([$username, $password]);
   $user = $stmt->fetch();
   ```

2. 输入验证
   对用户输入进行严格的格式校验，例如：
   - 用户名只允许字母、数字、下划线
   - 使用白名单验证，而不是黑名单过滤

3. 最小权限原则
   数据库连接账号使用普通权限账号，
   不要使用root或sa等管理员账号，
   只授予必要的权限。

4. 部署WAF
   在Web前端部署Web应用防火墙（WAF），
   对SQL注入等常见攻击进行拦截。

**【参考链接】**
- OWASP SQL注入防御速查表：https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html
- PHP PDO官方文档：https://www.php.net/manual/zh/book.pdo.php
- CWE-89: SQL注入：https://cwe.mitre.org/data/definitions/89.html

## 10.6 常用的报告模板和工具

### 10.6.1 报告模板

```
可以参考的模板：

1. OWASP Testing Report Template
   OWASP官方的测试报告模板

2. PTES（渗透测试执行标准）
   里面有报告的标准要求

3. 各种SRC的漏洞报告模板
   比如补天、漏洞盒子的提交模板，
   虽然短，但结构可以参考

4. 自己积累
   做的项目多了，
   把自己的报告整理成模板，
   下次直接改内容就行，
   效率高很多。
```

### 10.6.2 报告生成工具

手动写报告太麻烦？
有工具可以自动生成。

```
常见的报告生成工具：

1. Dradis
   → 渗透测试协作和报告生成工具
   → 可以导入各种扫描器的结果
   → 统一整理后导出报告

2. Serpico
   → 开源的渗透测试报告生成工具
   → 有模板，能生成Word报告

3. MagicTree
   → 数据整合和报告生成

4. PwnDoc
   → 专门写渗透测试报告的工具
   → 支持Markdown
   → 可以导出Word、PDF

5. 自己写脚本
   把常见漏洞写成模板，
   用Python脚本自动生成报告，
   用熟了效率很高。
```

> 💡 **工具是辅助**：
> 工具能提高效率，
> 但报告的质量还是取决于人。
> 别太依赖自动生成，
> 每个漏洞都要自己过一遍，
> 确保准确、合理。

## 10.7 本章总结

这一章我们讲了渗透测试报告怎么写，
虽然不如技术内容"酷"，
但非常非常重要。

```
✅ 报告的重要性
  ├── 报告是交付物
  └── 好报告 vs 坏报告

✅ 报告结构
  ├── 封面
  ├── 目录
  ├── 修订历史
  ├── 执行摘要（重点！给领导看的）
  ├── 测试范围与方法
  ├── 整体风险评估
  ├── 漏洞详情
  ├── 修复建议
  └── 附录

✅ 漏洞详情写法
  ├── 标题（位置+类型）
  ├── 风险等级怎么评
  ├── 复现步骤（要详细！）
  ├── 修复建议（要具体！）
  └── 完整示例

✅ 写作技巧
  ├── 排版格式
  ├── 语言表达
  └── 常见错误

✅ 工具和模板
  ├── 参考模板
  └── 报告生成工具
```

### 关键知识点回顾

| 要点 | 说明 |
|------|------|
| 执行摘要 | 给领导看的，要简洁易懂 |
| 漏洞标题 | 位置 + 漏洞类型 |
| 复现步骤 | 详细到别人照着做就能复现 |
| 修复建议 | 具体可操作，给代码示例 |
| 风险等级 | 合理，不能乱评 |
| 排版格式 | 统一、规范、无错别字 |
