## 5.2 OWASP Top 10

在讲具体漏洞之前，
先聊聊**OWASP Top 10**。
做Web安全的，
这个词你肯定经常听到。

什么是OWASP？
OWASP是一个开放式Web应用程序安全项目，
是一个国际非营利组织，
专门研究Web安全的。

什么是OWASP Top 10？
就是OWASP每隔几年，
会发布一个榜单，
列出**最危险的10种Web应用安全风险**。
相当于Web安全的"通缉令"。

这个榜单很权威，
基本上做Web安全的都以它为参考。
很多企业的安全标准，
也是基于OWASP Top 10来的。

2021年版的Top 10是这样的：

```
2021年 OWASP Top 10：

  1. 失效的访问控制（Broken Access Control）
  2. 加密机制失效（Cryptographic Failures）
  3. 注入（Injection）—— 包括SQL注入、命令注入等
  4. 不安全的设计（Insecure Design）
  5. 安全配置错误（Security Misconfiguration）
  6. 易受攻击和过时的组件（Vulnerable and Outdated Components）
  7. 识别和认证失败（Identification and Authentication Failures）
  8. 软件和数据完整性失败（Software and Data Integrity Failures）
  9. 安全日志和监控失败（Security Logging and Monitoring Failures）
  10. 服务端请求伪造（Server-Side Request Forgery - SSRF）
```

十年前的Top 10，
注入排第一，XSS排第三。
现在注入排第三，XSS甚至没进前十（归到别的类别里了）。
说明大家的安全意识在提高，
经典的漏洞在减少。
但这不代表这些漏洞没有了，
只是相对于其他风险来说，
占比下降了。

实际上，
SQL注入、XSS这些经典漏洞，
现在还是大量存在，
还是最常见的攻击手段。
学Web渗透，
这些经典漏洞还是得扎扎实实学好。

## 5.3 SQL注入

第一个，
也是最经典的——
**SQL注入（SQL Injection）**。

### 5.3.1 什么是SQL注入？

什么是SQL注入？
简单说就是：
**攻击者在输入的参数里注入了SQL语句，
后端的程序没做过滤，
直接把用户输入拼接到SQL语句里执行了，
导致攻击者可以操控SQL语句，
从而操作数据库。**

用人话说就是：
你输入的东西，
被当成SQL命令执行了。

举个最简单的例子：

假设一个登录功能，
后端的SQL语句是这么写的：

```sql
SELECT * FROM users WHERE username='用户输入的用户名' AND password='用户输入的密码'
```

正常情况下，
你输入用户名 admin，密码 123456，
SQL语句就是：

```sql
SELECT * FROM users WHERE username='admin' AND password='123456'
```

没问题，正常逻辑。

但是如果攻击者在用户名的地方输入：
`admin' OR '1'='1`
密码随便输点啥，
那SQL语句就变成了：

```sql
SELECT * FROM users WHERE username='admin' OR '1'='1' AND password='随便啥'
```

你看，
`OR '1'='1'` 这个条件永远成立，
所以不管密码对不对，
都能查到admin的用户信息，
直接就登录进去了。

这就是最经典的SQL注入——
万能密码。

就这么简单？
就这么简单。
原理就是这么个原理。
但就这么个简单的原理，
到现在还有大把的网站中招。

### 5.3.2 SQL注入有什么危害？

SQL注入的危害可大可小，
看是什么权限、什么数据库。
但总的来说，
危害非常大。

```
可能的危害：
  • 绕过登录（万能密码）
  • 获取数据库里的所有数据（拖库）
  • 增加、删除、修改数据
  • 执行系统命令（某些数据库、某些权限下）
  • 读取文件、写入文件
  • 提权、拿服务器权限
  • ......
```

严重的SQL注入，
可以直接从一个注入点，
拿到整个数据库，
甚至拿到服务器权限。
可以说是Web漏洞里危害最大的之一。

### 5.3.3 SQL注入的类型

SQL注入的分类方式很多，
常见的有这么几种分法：

```
按返回结果分：

  1. 回显注入（联合注入）
     • 执行的结果会直接显示在页面上
     • 最简单，最好利用
     • 可以用 UNION 查询直接把数据查出来

  2. 盲注
     • 执行结果不会直接显示
     • 只能通过页面差异、时间差异来判断
     • 分两种：
       - 布尔盲注：页面返回不一样（真和假两种状态）
       - 时间盲注：通过 sleep 等函数造成时间延迟来判断

  3. 报错注入
     • 页面会显示数据库的错误信息
     • 利用报错把数据带出来
     • 比盲注快，比回显注入麻烦一点
```

```
按注入位置分：

  • GET注入：参数在URL里
  • POST注入：参数在POST请求体里
  • Cookie注入：参数在Cookie里
  • User-Agent注入：在请求头里
  • ......
  只要是用户可控、拼接到SQL里的地方，都可能有注入
```

不管什么类型，
原理都是一样的——
用户输入被当成SQL执行了。
只是利用方式不同而已。

### 5.3.4 怎么判断有没有注入？

说了这么多，
那怎么判断一个参数有没有SQL注入呢？
新手最常问的就是这个。

简单说，
就是**加特殊字符，看反应**。

```
简单的测试步骤：

  1. 先加个单引号 '
     • 如果报错了，那可能有注入
     • 如果页面异常了，也可能有
     • 如果没变化，大概率没有（也不一定）

  2. 试试 and 1=1 和 and 1=2
     • and 1=1 页面正常
     • and 1=2 页面异常/不一样
     • 那很可能有布尔注入

  3. 试试 or 1=1
     • 看看能不能绕过登录
     • 或者页面返回更多数据

  4. 试试时间延迟
     • and sleep(5)
     • 如果页面明显变慢了，那就是时间盲注
```

当然，
这只是最基础的测试方法。
实际情况要复杂得多，
还要考虑各种绕过、各种类型。
但基本思路就是这样——
改变输入，看返回有没有差异，
从而判断是不是注入。

### 5.3.5 SQLMap——注入神器

讲SQL注入，
就不能不讲**SQLMap**。
这是SQL注入的神器，
Kali Linux里自带的。

SQLMap有多厉害？
只要你给它一个URL，
它能自动帮你：
- 判断有没有注入
- 是什么类型的注入
- 枚举数据库、表、字段
- 拖库
- 执行命令
- 提权
- ......

基本上，只要有注入点，
SQLMap就能帮你把能做的都做了。
新手的最爱，
也是渗透测试工程师的必备神器。

```
常用的SQLMap命令：

  基础用法：
    sqlmap -u "http://xxx.com/test.php?id=1"
    • 检测这个URL的参数有没有注入

  查数据库：
    sqlmap -u "URL" --dbs
    • 枚举所有数据库

  查表：
    sqlmap -u "URL" -D 数据库名 --tables
    • 查某个数据库里的所有表

  查字段：
    sqlmap -u "URL" -D 数据库名 -T 表名 --columns
    • 查某个表的所有字段

  拖数据：
    sqlmap -u "URL" -D 数据库名 -T 表名 -C "字段1,字段2" --dump
    • 把数据都导出来

  提权/拿shell：
    sqlmap -u "URL" --os-shell
    • 尝试拿系统shell
    • 看权限和数据库类型

  POST注入：
    sqlmap -u "URL" --data "post参数"
    • 测试POST参数

  Cookie注入：
    sqlmap -u "URL" --cookie "参数" --level 2

  更多用法自己看帮助：
    sqlmap -h
    sqlmap -hh （更详细的帮助）
```

SQLMap的参数非常多，
功能非常强大。
新手先把上面这几个常用的掌握好，
够用了。
更多高级用法，
慢慢研究。

但是，
**不要只会用SQLMap。**
要懂原理，
要知道SQL注入到底是怎么回事，
手工也要会注入。
不然哪天SQLMap跑不出来，
你就傻眼了。
工具是死的，人是活的。

### 5.3.6 SQL注入的防御

讲攻击也要讲防御。
SQL注入怎么防？

```
防御方法：
  1. 参数化查询（预编译）
     • 最有效的方法
     • 不要拼接SQL语句
     • 用参数化的方式，把用户输入当参数，不是当SQL的一部分
     • 比如Java的PreparedStatement，Python的参数化查询等

  2. 输入验证和过滤
     • 对用户输入做检查
     • 该是数字的就必须是数字
     • 该是字母的就不能有特殊字符
     • 白名单比黑名单好

  3. 权限控制
     • 数据库账号权限最小化
     • 普通业务账号不要给DBA权限
     • 就算被注入了，危害也小一点

  4. 错误信息屏蔽
     • 不要把数据库错误直接显示在页面上
     • 防止报错注入
     • 也防止泄露数据库信息

  5. WAF
     • 加个WAF，能挡掉大部分常规注入
     • 但不是万能的，高手还是能绕过
     • 纵深防御嘛
```

核心就是一句话：
**不要拼接SQL，用参数化查询。**
做到这一点，
99%的SQL注入就没了。

## 5.4 XSS跨站脚本攻击

第二个经典漏洞：
**XSS（Cross Site Scripting），跨站脚本攻击。**

为什么叫XSS不叫CSS？
因为CSS已经是层叠样式表了，
怕混淆，
所以叫XSS。

### 5.4.1 什么是XSS？

什么是XSS？
简单说就是：
**攻击者往Web页面里注入了恶意的HTML/JS代码，
其他用户浏览这个页面的时候，
代码就会在他们的浏览器里执行。**

用人话说就是：
你的页面把用户输入的内容直接显示出来了，
没转义，
攻击者输入一段JS代码，
别人看页面的时候，
这段代码就跑起来了。

举个例子：
有个留言板功能，
用户可以留言，
留言内容会显示在页面上。
如果后端没做转义，
攻击者输入：
```html
<script>alert('XSS')</script>
```
别人看这个留言的时候，
浏览器就会执行这段JS，
弹出个框。
这就是XSS。

当然，
弹个框只是演示，
真的攻击可比这厉害多了。

### 5.4.2 XSS有什么危害？

XSS的危害也很大，
而且往往被低估。
很多人觉得XSS就是弹个框，
没什么大不了的。
大错特错。

```
XSS能干嘛：
  • 盗取Cookie（最常见）—— 冒充用户登录
  • 钓鱼 —— 伪造登录框，骗密码
  • 挂马 —— 让用户下载病毒
  • 键盘记录 —— 记录用户敲了什么
  • 内网探测 —— 用受害者的浏览器扫内网
  • 篡改页面 —— 想显示啥就显示啥
  • CSRF配合 —— 做各种操作
  • ......
```

严重的XSS，
配合上管理员账号，
直接就能拿后台权限，
甚至进一步拿服务器。
危害一点都不小。

### 5.4.3 XSS的三种类型

XSS一般分三种类型：

```
第一种：反射型XSS（Reflected XSS）
  • 恶意代码在URL里
  • 需要诱导用户点这个链接
  • 非持久化，点一次触发一次
  • 比如搜索功能、错误页面
  • 危害相对小一点，因为要骗用户点击

第二种：存储型XSS（Stored XSS）
  • 恶意代码被存到数据库里了
  • 所有访问这个页面的用户都会中招
  • 持久化，危害大
  • 比如留言板、评论、个人资料
  • 最危险的一种

第三种：DOM型XSS（DOM-based XSS）
  • 不经过后端，是前端JS的问题
  • 前端JS把用户输入直接插到DOM里了
  • 也是反射型的一种，但原理不太一样
  • 现在越来越多了，因为前端框架多了
```

三种类型，
存储型最危险，
因为不用骗用户点链接，
用户正常浏览就中招了。

### 5.4.4 怎么找XSS？

怎么判断一个地方有没有XSS？
跟SQL注入类似，
输入特殊字符，
看有没有被执行。

```
简单测试方法：

  1. 输入一些特殊字符
     < > ' " / script
     看看源代码里这些字符是不是原样输出了
     如果是，那可能有XSS

  2. 试试最基础的Payload
     <script>alert(1)</script>
     能弹框就是有XSS

  3. 如果被过滤了，试试各种绕过
     • 大小写混写：<sCrIpT>
     • 用别的标签：<img src=1 onerror=alert(1)>
     • 各种编码
     • 绕过的姿势多了去了

  4. DOM型的要去看前端JS代码
     看它怎么处理用户输入的
     是不是直接插到页面里了
```

当然，
实际情况千变万化，
有各种过滤、各种编码、各种WAF。
XSS的绕过也是一门很深的学问，
这里就不展开讲了。

### 5.4.5 XSS利用平台：BeEF

讲XSS，
就不能不提**BeEF**（The Browser Exploitation Framework）。
这是专门的XSS利用框架，
Kali里也有。

BeEF能干啥？
只要用户的浏览器中招了（弹了个框那种级别的XSS就行），
你就能通过BeEF控制他的浏览器，
然后：
- 看他的浏览器信息、系统信息
- 记录键盘
- 截图
- 重定向
- 钓鱼
- 扫内网
- 拿shell（利用浏览器漏洞）
- ......
功能非常多。

有了BeEF，
一个简单的XSS也能玩出花来。
新手可以装来玩玩，
挺有意思的。

### 5.4.6 XSS的防御

XSS怎么防？

```
防御方法：
  1. 输入输出编码/转义
     • 最基本的，用户输出到页面的内容要转义
     • < 转成 &lt; > 转成 &gt; 之类的
     • 让浏览器把它当文本，不是代码

  2. CSP（内容安全策略）
     • 设置 Content-Security-Policy 响应头
     • 规定哪些资源能加载、哪些JS能执行
     • 就算有XSS，也很难执行恶意代码
     • 非常强力的防御手段

  3. HttpOnly Cookie
     • 给Cookie加上HttpOnly属性
     • JS就读不到Cookie了
     • 就算有XSS，也偷不到Cookie
     • 不能防XSS，但能降低危害

  4. 输入验证
     • 对用户输入做检查
     • 白名单验证
     • 该是数字的就不能有别的
```

XSS防御的核心是：
**该转义的转义，该验证的验证，再加上CSP。**
做到这些，
大部分XSS就都防住了。

## 5.5 文件上传漏洞

第三个经典漏洞：
**文件上传漏洞。**

这个也很常见，
而且危害也很大，
因为直接就能上传webshell，
拿服务器权限。

### 5.5.1 什么是文件上传漏洞？

什么是文件上传漏洞？
简单说就是：
**网站有上传功能，
但对上传的文件控制不严，
导致攻击者可以上传恶意的脚本文件
（比如php、asp、jsp的webshell），
并且能访问执行，
从而拿到服务器权限。**

用人话说就是：
你能上传一个后门脚本，
然后能在浏览器里访问它、执行它，
那你就拿到这台服务器的权限了。

比如网站有个头像上传功能，
你上传一个 `shell.php`，
然后访问 `http://xxx.com/upload/shell.php`，
如果能执行，
那就getshell了。

就这么直接，
就这么暴力。
所以文件上传漏洞也是危害最大的Web漏洞之一。

### 5.5.2 文件上传的各种校验

为什么会有文件上传漏洞？
因为开发者做了校验，
但校验做得不好，
被绕过了。

常见的校验方式有哪些？
以及怎么绕过？

```
常见校验与绕过：

  1. 前端JS校验
     • 只在前端用JS检查后缀名
     • 看起来选不了.php的文件
     • 绕过方法：
       - 把JS禁用了
       - 或者用Burp抓包，改文件名
       - 前端校验等于没校验，防君子不防小人

  2. 后缀名校验（黑名单）
     • 后端检查后缀名，不让上传 .php .asp 等
     • 但用的是黑名单
     • 绕过方法：
       - 换个不常见的后缀：.php5 .phtml .phar...
       - 大小写：.pHp .Php（Windows下不区分大小写）
       - 加点：shell.php. （Windows会自动去掉最后的点）
       - ::$DATA（Windows NTFS流的特性）
       - ..... 各种奇技淫巧

  3. 后缀名校验（白名单）
     • 只允许上传 jpg png gif 这些
     • 白名单比黑名单安全
     • 但也不是万无一失
     • 绕过方法：
       - %00截断（老版本PHP、老系统）
       - 解析漏洞（比如Apache的解析漏洞）
       - 配合文件包含漏洞
       - ......

  4. 文件头/MIME校验
     • 检查文件类型是不是图片
     • 比如检查Content-Type是不是image/jpeg
     • 或者检查文件头是不是GIF89a
     • 绕过方法：
       - 抓包改Content-Type
       • 在文件内容开头加上GIF89a
       • 制作图片马（图片+代码的合成文件）

  5. 文件内容校验
     • 检查文件内容是不是真的图片
     • 这个就比较难绕了
     • 绕过方法：
       - 图片马
       - 配合文件包含
       - 二次渲染绕过
       - ......
```

文件上传的绕过是门大学问，
各种姿势各种技巧。
不同的环境、不同的语言、不同的中间件，
绕过方法都不一样。
新手先知道个大概，
后面慢慢研究。

### 5.5.3 Webshell是什么？

上传了文件之后，
用什么呢？
用**WebShell**。

什么是WebShell？
就是一个网页形式的后门，
上传到服务器上，
通过浏览器访问，
就能执行命令、操作服务器。

WebShell有什么功能？
- 查看目录、浏览文件
- 上传、下载文件
- 执行系统命令
- 管理数据库
- 提权
- ......
基本上啥都能干。

常见的WebShell：
- 一句话木马：最小的webshell，就一行代码
  - PHP：`<?php eval($_POST['a']);?>`
  - ASP：`<%eval request("a")%>`
- 大马：功能很全的webshell，有界面，啥都能干
- 菜刀/蚁剑/冰蝎：WebShell管理工具，连接一句话木马用的

Kali里也有一些webshell样本，
可以学习研究，但别乱用。

```
Kali里的webshell位置参考：
  /usr/share/webshells/
  里面有各种语言的各种webshell
```

还是那句话，
学习研究可以，
别拿去干违法的事。

### 5.5.4 文件上传的防御

文件上传怎么防？

```
防御方法：
  1. 后缀名校验用白名单
     • 不要用黑名单，永远列不全
     • 用白名单，只允许指定的后缀
     • 更严格，更安全

  2. 文件内容校验
     • 不光看后缀，还要检查内容
     • 是图片就重新渲染一下
     • 让图片马失效

  3. 文件重命名
     • 上传的文件不要用原来的名字
     • 随机生成一个文件名
     • 攻击者就算上传了，也不知道路径
     • 也能防止覆盖现有文件

  4. 上传目录不让执行
     • 上传文件的目录，不给执行权限
     • 就算上传了脚本，也执行不了
     • 这个很重要

  5. 文件单独存放到单独的域名/服务器
     • 跟主站隔开
     • 就算出问题也影响不到主站

  6. 权限最小化
     • Web服务的账号权限要小
     • 就算被拿了webshell，也干不了太多事
```

文件上传的防御，
核心就是：
**白名单校验 + 文件内容检查 + 上传目录不给执行权限。**
做到这几点，
基本就稳了。

## 5.6 本章总结

这一章我们讲了三个最经典的Web漏洞，
内容挺多的，总结一下：

**OWASP Top 10：**
- Web安全最权威的风险榜单
- 注入、XSS这些经典漏洞至今仍然常见

**SQL注入：**
- 原理：用户输入被拼接到SQL里执行了
- 危害：拖库、绕过登录、执行命令、拿shell
- 类型：回显注入、报错注入、布尔盲注、时间盲注
- 神器：SQLMap，自动化注入工具
- 防御：参数化查询（最有效）+ 输入验证 + 权限控制

**XSS跨站脚本：**
- 原理：恶意JS代码被浏览器执行
- 危害：偷Cookie、钓鱼、挂马、键盘记录、内网探测
- 类型：反射型、存储型、DOM型
- 利用平台：BeEF
- 防御：输入输出转义 + CSP + HttpOnly

**文件上传：**
- 原理：上传校验不严，能上传恶意脚本并执行
- 危害：直接getshell，拿到服务器权限
- 各种校验与绕过：前端校验、黑名单、白名单、MIME、文件头...
- 工具：WebShell（一句话、大马）、菜刀/蚁剑/冰蝎
- 防御：白名单 + 内容校验 + 目录无执行权限 + 重命名

最后送你一句话：

> **经典漏洞永远不会过时，
> 把基础打牢，
> 比学十个高大上的新漏洞都管用。**
