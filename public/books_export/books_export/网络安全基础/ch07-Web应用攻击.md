## 7.1 先讲一个故事

### 7.1.1 "一个单引号引发的血案"

我给你讲个真实的故事。

很多年前，
某电商网站刚上线没多久，
有个安全研究员上去逛了逛，
在一个商品搜索框里，
随手输入了一个单引号 `'`，
然后点了搜索。

结果页面报错了，
直接把SQL错误信息吐了出来。
他一看，
心中一喜——
**有SQL注入。**

接下来的事情就简单了：
- 用union查询，爆出了数据库的表名和字段名
- 找到了管理员表，拿到了管理员账号密码的哈希
- 破解了哈希，拿到了明文密码
- 找到了后台地址，登录进去
- 后台有文件上传的地方，传了个Webshell上去
- 拿到了服务器权限
- 然后又从这台服务器内网横向，
  打到了数据库服务器
- 最后，整个用户数据库都被拖走了
  几百万用户的信息泄露了

你看，
就这么一个小小的单引号，
最后导致了整个网站被拖库，
几百万用户信息泄露。
这就是SQL注入的威力。

---

一个输入框、一个参数，
但利用起来，
危害可能非常大。
所以做Web安全的，
对每一个输入点都不能放过。

## 7.2 SQL 注入（SQL Injection）

第一个，
**SQL注入**。
可以说是最经典、最有名、
也是危害最大的Web漏洞之一。
常年霸占OWASP Top 10第一名。

### 7.2.1 什么是SQL注入？

什么是SQL注入？
简单说就是：
**攻击者在输入参数里，
注入恶意的SQL语句，
后端没做过滤，
直接把输入拼到了SQL语句里执行，
导致攻击者可以操控SQL语句，
获取、修改、删除数据，
甚至获取服务器权限。**

用人话说就是：
**你让用户输入数据，
结果用户输入了一段SQL代码，
你的程序没判断，
直接把这段代码当SQL命令执行了。**

举个最简单的例子：

比如有个登录功能，
后端SQL语句是这么写的：
```sql
SELECT * FROM users 
WHERE username='用户输入的用户名' 
AND password='用户输入的密码'
```

正常情况下，
用户输入用户名admin和密码123456，
SQL变成：
```sql
SELECT * FROM users 
WHERE username='admin' 
AND password='123456'
```
没问题。

但如果攻击者在用户名那里输入：
`admin' --`
那SQL就变成了：
```sql
SELECT * FROM users 
WHERE username='admin' --' 
AND password='123456'
```
`--` 是SQL的注释符，
后面的都被注释掉了。
于是就变成了只查用户名是admin的，
密码那部分直接被忽略了。
这样攻击者不用知道密码，
直接就登录进去了。

这就是最经典的SQL注入例子——
**万能密码。**

### 7.2.2 SQL注入的原理

SQL注入的根本原因是什么？
**把用户输入的数据，
和SQL代码拼接在一起，
没有做过滤和转义，
导致数据被当成了代码执行。**

本质就是：
**数据和代码没有分离。**
用户输入的本来应该是数据，
结果被当成SQL代码执行了。

这和后面要讲的命令注入、
XSS、代码执行...
原理都是一样的——
数据和代码没分开，
数据被当成代码执行了。
这是一类非常常见的漏洞。

### 7.2.3 SQL注入的分类

SQL注入怎么分类？
有几种分法：

```
  按提交方式分：
    • GET注入：参数在URL里
    • POST注入：参数在POST body里
    • Cookie注入：参数在Cookie里
    • HTTP头注入：参数在HTTP头里
      比如User-Agent、Referer、X-Forwarded-For...

  按返回结果分：
    • 回显注入（有回显）
      执行结果直接显示在页面上
      比如union注入、报错注入

    • 盲注（没有回显）
      页面不显示执行结果
      只能通过页面变化、时间延迟来判断
      又分：
        • 布尔盲注：根据页面返回不同判断
        • 时间盲注：根据响应时间判断

  按注入位置分：
    • 数字型注入
      参数是数字，比如?id=1
      没有引号包裹

    • 字符型注入
      参数是字符串，有引号包裹

    • 搜索型注入
      跟搜索相关的，like语句里
```

分类方式很多，
不用死记，
理解原理最重要。

### 7.2.4 SQL注入的危害

SQL注入危害有多大？
非常大。

```
  • 数据泄露
    拖库，把整个数据库都下载下来
    用户信息、订单、机密数据...
    这是最常见的危害

  • 数据篡改
    修改数据，比如改余额、改密码、
    把自己改成管理员

  • 数据删除
    删库、删表、Drop Table
    破坏性很大

  • 权限提升
    利用数据库的功能执行系统命令
    比如MySQL的into outfile写文件
    SQL Server的xp_cmdshell执行命令
    直接拿到服务器权限

  • 内网渗透
    拿下数据库服务器之后，
    从数据库服务器打内网
    危害更大
```

可以说，
一个严重的SQL注入，
足以毁掉整个网站。
所以SQL注入一直是Web安全的头号威胁。

### 7.2.5 怎么防御SQL注入？

SQL注入怎么防？
核心就一句话：
**永远不要相信用户的输入，
永远不要把用户输入直接拼到SQL语句里。**

具体怎么防？

```
  1. 预编译/参数化查询（最有效）
     用Prepared Statement
     SQL语句模板提前编译好，
     参数用占位符，
     用户输入永远只是参数，
     不会被当成SQL代码执行
     这是最根本、最有效的防御方法
     强烈推荐

  2. 输入过滤与转义
     对用户输入进行过滤和转义
     比如把单引号转义
     但这个方法不如预编译可靠，
     很容易被绕过
     只能作为辅助

  3. WAF
     Web应用防火墙
     可以拦截常见的SQL注入攻击
     但WAF也可以被绕过
     只能作为防御的一层，
     不能作为根本手段

  4. 最小权限原则
    数据库账号用最小权限
    不要用root、sa之类的高权限账号
    就算被注入了，危害也小一点

  5. 错误信息屏蔽
    不要把数据库的错误信息直接吐给前端
    攻击者可以通过报错信息获取很多东西
    前端只显示通用的错误提示
```

最根本的防御就是**预编译/参数化查询**。
只要你不用字符串拼接SQL，
SQL注入基本就不会有。

### 7.2.6 SQL注入工具

提到SQL注入，
就不得不提一个神器——
**sqlmap**。
这是最有名的SQL注入工具，
功能极其强大。
你只要给它一个URL，
它就能自动检测有没有注入，
是什么类型的注入，
然后自动帮你拖库、
提权、拿shell...
基本上全自动。

但是，
工具是工具，
原理你还是得懂。
不然用sqlmap跑出来了，
你也不知道为什么。
而且很多复杂的注入，
还是要手动调的。

## 7.3 XSS 跨站脚本攻击

第二个，
**XSS（Cross-Site Scripting）**，
跨站脚本攻击。
也是非常经典、非常常见的Web漏洞。

为什么叫XSS不叫CSS？
因为CSS已经被样式表用了，
所以叫XSS。

### 7.3.1 什么是XSS？

什么是XSS？
简单说就是：
**攻击者在网页中注入恶意的脚本代码（一般是JavaScript），
其他用户访问这个网页的时候，
恶意脚本就在他们的浏览器里执行了。**

和SQL注入有点像，
都是注入攻击，
只不过SQL注入是注入SQL语句，
在数据库里执行；
XSS是注入HTML/JavaScript代码，
在用户浏览器里执行。

举个简单的例子：
一个论坛网站，
用户可以发帖，
帖子内容会显示在页面上。
如果后端没过滤，
攻击者发了个帖子，
内容是：
```html
<script>alert('XSS')</script>
```
其他用户看这个帖子的时候，
这段脚本就会在他们的浏览器里执行，
弹出个框。
这就是XSS。

当然，
弹个框只是演示，
真实的XSS攻击可没这么简单。

### 7.3.2 XSS的分类

XSS一般分三类：

```
  1. 存储型XSS（Persistent）
     恶意脚本被存储到了服务器端（比如数据库）
     其他用户访问的时候，
     服务器从数据库里读出来，
     渲染到页面上，
     脚本就执行了
     危害最大，影响用户多
     比如论坛发帖、评论、留言板...

  2. 反射型XSS（Reflected）
     恶意脚本在URL里
     用户点击了带恶意脚本的URL，
     服务器把URL里的参数反射回页面，
     脚本就执行了
     需要诱导用户点击
     危害比存储型小一些
     比如搜索框、错误页面...

  3. DOM型XSS（DOM-based）
     不经过服务器，
     完全是前端JavaScript的问题
     JS直接从URL或者别的地方拿数据，
     没有过滤，
     直接插到DOM里或者用eval之类的执行
     这种是前端的锅，后端没责任
     也属于反射型的一种，
     但成因不太一样
```

三种类型，
存储型危害最大，
因为不需要诱导用户点链接，
用户正常访问就中招。

### 7.3.3 XSS的危害

XSS能干嘛？
很多人觉得XSS不就是弹个框吗，
有什么大不了的。
大错特错。
XSS的危害非常大，
甚至不比SQL注入小。

```
  • 窃取Cookie
    最经典的用法
    拿到用户的Cookie，
    就可以冒充用户登录，
    劫持用户会话
    你想想，如果是管理员的Cookie被偷了，
      会怎么样？

  • 钓鱼攻击
    在页面上插入一个假的登录框，
    用户输入账号密码，
    直接发到攻击者那里
    用户根本分不清真假

  • 挂马
    在网页里植入木马，
    利用浏览器漏洞，
    让访问的用户中毒

  • 键盘记录
    记录用户在页面上输入的所有内容
    比如密码、银行卡号

  • 篡改页面内容
    想显示什么就显示什么
    改价格、改信息...

  • CSRF攻击
    用XSS来做CSRF，更方便

  • 内网扫描
    用户浏览器就是个跳板
    可以用JS扫描用户的内网
    探测内网设备

  • ......
    能做的事情太多了
    基本上用户在这个网站上能干什么，
    攻击者就能干什么
```

所以千万别小看XSS，
它的危害非常大。

### 7.3.4 怎么防御XSS？

XSS怎么防？
核心还是：
**永远不要相信用户的输入，
永远不要把用户输入直接输出到HTML里。**

具体防御方法：

```
  1. 输入过滤
    对用户输入进行过滤，
    把危险的标签、字符过滤掉
    但是过滤很难做到完美，
    容易被绕过

  2. 输出转义/编码（最关键）
    把用户输入输出到HTML的时候，
    进行HTML转义
    把 < 变成 &lt;，> 变成 &gt;，
    " 变成 &quot;，' 变成 &#x27;
    这样就算有script标签，
    也会被当成文本显示，
    不会执行
    这是最基本、最有效的防御方法

  3. HttpOnly Cookie
    给Cookie加上HttpOnly属性
    这样JavaScript就读不到Cookie了
    就算有XSS，也偷不到Cookie
    能大大降低XSS的危害
    强烈建议开启

  4. CSP（内容安全策略）
    Content Security Policy
    设置白名单，
    只允许加载指定来源的脚本
    就算注入了脚本，
    如果不在白名单里也执行不了
    是非常强的防御手段
    现在越来越多网站在用

  5. 安全的前端框架
    现在的前端框架比如React、Vue，
    默认都会做HTML转义
    只要你不用v-html、dangerouslySetInnerHTML
    之类的东西，
    一般不会有XSS
    比手动写HTML安全多了
```

XSS的防御，
**输出转义是基础，
HttpOnly和CSP是重要辅助。**
多层防护，
才能防得住。

## 7.4 文件上传漏洞

第三个，
**文件上传漏洞**。
这个也非常常见，
而且往往危害很大，
直接就能拿webshell。

### 7.4.1 什么是文件上传漏洞？

很多网站都有文件上传功能，
比如头像上传、图片上传、
附件上传...
如果后端对上传的文件校验不严，
攻击者就可以上传一个脚本文件
（比如.php、.asp、.jsp），
然后访问这个文件，
脚本就在服务器上执行了，
拿到webshell，
控制整个服务器。

这就是**文件上传漏洞**。

简单说就是：
**你让用户上传文件，
结果用户上传了个可执行的脚本，
你没校验出来，
还把文件存到了Web可以访问的目录，
用户访问这个文件，
脚本就执行了。**

文件上传漏洞往往直接导致服务器被拿下，
危害非常大。

### 7.4.2 为什么会有文件上传漏洞？

根本原因：
**对上传的文件校验不严。**

常见的绕过方式：
```
  1. 前端校验，后端没校验
    很多网站只在前端用JS校验后缀名
    后端根本不校验
    攻击者直接抓包改后缀名，
    或者禁用JS，就绕过了
    （前端校验只能防普通用户，防不了攻击者）

  2. MIME类型校验不严
    只校验Content-Type
    攻击者抓包把Content-Type改成image/jpeg
    就绕过了

  3. 后缀名校验不严格
    • 黑名单绕过：
      黑名单里有php，但没禁止php3、php4、
      phtml、pHp（大小写）...
      这些也能执行
    • 白名单绕过：
      白名单也可能被绕过，
      比如%00截断、解析漏洞...

  4. 文件名可控制
    上传的文件名用户可以控制，
    可以目录穿越，
    把文件传到别的目录去

  5. 解析漏洞
    中间件本身的解析漏洞
    比如老版本的IIS、Apache、Nginx
    都有过解析漏洞
    比如 1.php.jpg 会被当成php执行

  6. 二次渲染/图片马
    就算校验图片内容，
    也可以做图片马——
    把PHP代码写到图片里，
    配合文件包含或者解析漏洞来利用
```

绕过的方式非常多，
一个不小心就中招了。

### 7.4.3 怎么防御文件上传漏洞？

文件上传怎么防？
要多管齐下，多层校验。

```
  1. 后端严格校验
    前端校验不算数，后端一定要校验
    而且要多重校验

  2. 文件后缀名校验——白名单
    用白名单，不要用黑名单
    白名单只允许上传你需要的类型
    比如只允许jpg、png、gif
    白名单比黑名单安全得多

  3. 文件内容校验
    不要只看后缀名，
    还要校验文件内容
    比如图片要检查文件头、
    甚至用图片处理库重新生成一下
    防止图片马

  4. 文件重命名
    上传的文件不要用原文件名
    重新生成一个随机的文件名
    防止攻击者猜到文件名
    也防止目录穿越之类的问题

  5. 文件存储目录
    上传的文件不要放在Web可访问的目录下
    或者放在独立的域名下
    就算上传了脚本，
    访问不到也没用
    这是最狠的一招

  6. 限制文件大小
    防止上传大文件搞事情

  7. 中间件安全配置
    升级中间件，打补丁
    避免解析漏洞
```

文件上传的防御，
核心就是：
**严格校验 + 随机文件名 + 不可执行目录。**
三层都做到了，
基本就安全了。

## 7.5 CSRF 跨站请求伪造

第四个，
**CSRF（Cross-Site Request Forgery）**，
跨站请求伪造。
这个也很经典。

### 7.5.1 什么是CSRF？

什么是CSRF？
简单说就是：
**攻击者诱导用户访问一个恶意网站，
恶意网站里发起一个对目标网站的请求，
因为用户已经在目标网站登录过了，
浏览器会自动带上Cookie，
所以目标网站会以为是用户自己发的请求，
就执行了。**

用人话说就是：
**攻击者借用户的手，
以用户的身份，
在用户不知情的情况下，
向目标网站发请求，
干一些用户不想干的事。**

举个例子：
你登录了银行网站，
然后又去访问了一个恶意网站。
恶意网站里有个隐藏的表单，
自动向银行网站的转账接口发了个POST请求，
转1万块钱到攻击者账户。
因为你已经登录了银行网站，
浏览器会自动带上你的Cookie，
银行网站收到请求，
一看Cookie是对的，
就以为是你自己转的，
就给转了。
这就是CSRF。

### 7.5.2 CSRF的原理

CSRF为什么能成功？
两个关键点：
```
  1. Cookie的自动携带
    浏览器的同源策略，
    对于请求的发送是不拦截的，
    只是拦截响应的读取。
    也就是跨站可以发请求，
    浏览器会自动带上目标网站的Cookie，
    只是发起请求的网站读不到响应。
    但很多操作不需要读响应，
    请求发出去就够了。

  2. 目标网站只靠Cookie验证身份
    目标网站只看Cookie对不对，
    不对请求做额外的验证，
    只要Cookie对就认为是用户本人。
```

两个条件加起来，
就有了CSRF。

### 7.5.3 CSRF的危害

CSRF能干嘛？
```
  凡是用户能做的操作，
  CSRF都能借用户的身份做：
    • 转账、改密码
    • 发帖子、发评论
    • 加好友、关注
    • 提交表单
    • ......
    只要是修改数据的操作，
    都可以被CSRF利用
```

CSRF攻击的是用户，
以用户的身份操作。
危害大小取决于被攻击用户的权限，
如果是管理员中招了，
那危害就大了。

### 7.5.4 怎么防御CSRF？

CSRF怎么防？
核心思想：
**证明这个请求确实是用户自己发的，
而不是别的网站伪造的。**

常见的防御方法：

```
  1. CSRF Token（最常用）
    在表单里加一个随机的Token，
    这个Token存在用户的Session里，
    提交的时候一起带上，
    后端验证Token对不对
    因为攻击者的网站拿不到这个Token，
    所以伪造的请求没有正确的Token，
    就通不过验证
    这是最常用、最有效的方法

  2. 验证Referer / Origin
    检查请求的来源
    看Referer是不是本网站的
    但是Referer可以被篡改，
    也可能被浏览器禁用，
    所以只能作为辅助

  3. 关键操作二次验证
    重要操作（比如转账、改密码）
    要求重新输入密码，
    或者短信验证码、人脸识别...
    就算有CSRF，
    攻击者也不知道密码和验证码
    这样最安全

  4. SameSite Cookie
    给Cookie设置SameSite属性
    限制跨站请求携带Cookie
    比如Strict、Lax模式
    从源头上防止CSRF
    现在浏览器都支持了，
    是未来的趋势
```

最常用的是**CSRF Token**，
配合**SameSite Cookie**，
基本就能防住CSRF了。
关键操作再加上二次验证，
就更安全了。

## 7.6 SSRF 服务端请求伪造

第五个，
**SSRF（Server-Side Request Forgery）**，
服务端请求伪造。
这几年越来越受重视，
也越来越常见。

### 7.6.1 什么是SSRF？

什么是SSRF？
简单说就是：
**攻击者让服务器端发起一个请求，
请求攻击者指定的地址。**

普通的攻击都是从外部打进来，
SSRF反而是让服务器自己往外发请求，
或者往内部发请求。
利用服务器的身份，
去访问它能访问的东西。

举个例子：
很多网站有个功能，
比如"从URL下载图片"、
"远程文件抓取"、
"网页截图"、
"在线翻译"...
需要后端去请求一个用户指定的URL。
如果后端没做限制，
用户就可以让服务器请求内网的地址，
比如 `http://127.0.0.1/`、
`http://内网IP/`，
来探测内网、攻击内网服务。
甚至用file://协议读本地文件。

这就是SSRF。

### 7.6.2 SSRF的危害

SSRF能干嘛？
```
  1. 内网探测
    让服务器去扫内网的IP和端口，
    看看内网有什么机器、开了什么服务
    相当于把服务器当代理，
    打内网

  2. 攻击内网服务
    内网的服务一般防护比较弱，
    甚至很多没有认证
    可以利用SSRF去攻击内网的
      Redis、MySQL、Memcached、
      Elasticsearch...
    很多服务如果能从localhost访问，
    危害很大

  3. 读取本地文件
    用file://协议
    让服务器读本地的文件
    比如/etc/passwd、配置文件...

  4. 攻击云服务元数据
    很多云服务商（比如AWS、阿里云）
    有元数据服务（169.254.169.254）
    如果服务器在云上，
    利用SSRF可以访问元数据，
    拿到临时凭证、AK/SK...
    直接接管整个云资源
    这个危害非常大

  5. DoS攻击
    让服务器请求大文件，
    或者大量请求，
    造成服务器资源耗尽
```

SSRF的危害很大，
尤其是在云时代，
很多企业都在云上，
SSRF打到元数据服务，
直接就崩了。
所以SSRF这几年越来越受重视。

### 7.6.3 怎么防御SSRF？

SSRF怎么防？
核心就是：
**限制服务器能请求的地址，
不让它随便请求内网和本地。**

具体防御方法：

```
  1. 白名单域名
    只允许请求指定白名单里的域名
    这是最安全的
    但可能不够灵活

  2. 禁止访问内网和本地
    解析域名的IP，
    如果是内网IP（10.x.x.x、
    172.16-31.x.x、192.168.x.x、
    127.x.x.x、169.254.x.x...）
    就拒绝请求
    注意：一定要先解析IP再校验，
    不能只校验域名，
    因为域名可以解析到内网IP

  3. 禁用危险协议
    只允许http和https协议
    禁用file://、gopher://、dict://、
    ftp://...这些危险协议
    这些协议经常被用来搞事情

  4. 统一错误信息
    不要把连接错误、超时错误、
    目标返回内容等信息返回给前端
    防止攻击者通过错误信息判断内网情况

  5. 不跟随跳转
    很多SSRF是通过302跳转绕过的
    跳到内网地址
    所以不要自动跟随302跳转，
    或者跳转的时候也校验目标地址

  6. 网络层隔离
    从网络层面做隔离，
    Web服务器所在的网段，
    不能随便访问内网的其他服务
    就算有SSRF，也打不到内网
    纵深防御
```

SSRF的防御要注意很多绕过的技巧，
比如DNS重绑定、
302跳转、
各种进制表示的IP、
短地址跳转...
防御的时候要考虑周全。

## 7.7 命令执行与代码执行

第六个，
**命令执行（Command Injection）
和代码执行（Code Injection）**。
这俩都是高危漏洞，
直接就能拿服务器权限。

### 7.7.1 命令执行

什么是命令执行？
简单说就是：
**攻击者可以在服务器上执行系统命令。**

怎么产生的？
后端代码里调用了系统命令，
并且把用户输入拼到了命令里，
没有做过滤。
比如PHP的system()、exec()、
shell_exec()...
Java的Runtime.exec()...
Python的os.system()、subprocess...
如果参数是用户可控的，
又没过滤，
就可能导致命令执行。

举个例子：
一个ping功能，
用户输入IP，
后端执行 `ping 用户输入的IP`。
如果用户输入：
`127.0.0.1 ; whoami`
那后端执行的就是：
`ping 127.0.0.1 ; whoami`
分号在Linux里是命令分隔符，
于是 whoami 也被执行了。
这就是命令执行。

命令执行的危害非常大，
直接就能控制服务器，
想干嘛干嘛。

### 7.7.2 代码执行

代码执行是什么？
和命令执行类似，
只不过执行的是代码（比如PHP代码、Python代码），
不是系统命令。

比如PHP的eval()、assert()，
Python的eval()、exec()，
如果参数用户可控，
就可以执行任意代码。

代码执行的危害也很大，
基本和命令执行差不多，
都是直接拿服务器权限。

### 7.7.3 怎么防御？

命令执行和代码执行怎么防？
```
  1. 尽量不要调用系统命令/eval
    能用代码实现的，
    就不要调用系统命令
    能用安全函数的，
    就不要用eval之类的
    从源头上避免

  2. 如果必须用，严格过滤输入
    用白名单，
    只允许输入特定的字符
    比如IP地址，就用正则严格校验
    是合法IP才允许执行

  3. 参数化 / 数组形式传参
    很多语言执行系统命令支持数组形式，
    而不是字符串拼接
    比如Python的subprocess用列表传参，
    PHP的escapeshellarg转义参数
    这样就不会有注入的问题

  4. 最小权限原则
    Web服务运行在低权限用户下，
    就算被执行了命令，
    危害也小一点
```

还是那句话：
**数据和代码分离，
永远不要把用户输入拼到命令或代码里执行。**

## 7.8 文件包含漏洞

第七个，
**文件包含漏洞（File Inclusion）**。
PHP里比较常见。

### 7.8.1 什么是文件包含？

很多语言有文件包含的功能，
比如PHP的include()、require()、
include_once()、require_once()。
可以把另一个文件的内容包含进来执行。

如果包含的文件名是用户可控的，
用户就可以指定包含任意文件，
比如：
- 包含恶意的PHP文件（webshell），执行代码
- 包含系统文件，读取敏感信息
- 包含图片马，执行里面的PHP代码
- 包含远程文件（如果allow_url_include开了的话），
  直接执行远程的恶意代码

这就是**文件包含漏洞**。

文件包含分两种：
```
  • 本地文件包含（LFI）
    只能包含服务器本地的文件

  • 远程文件包含（RFI）
    可以包含远程的文件（URL）
    危害更大，直接就能拿shell
    但需要php.ini里开allow_url_include
    现在一般都关了
```

### 7.8.2 文件包含的危害与防御

危害：
- 读取敏感文件（配置文件、密码文件...）
- 执行恶意代码，拿webshell
- 配合文件上传，上传图片马然后包含执行

防御：
```
  1. 尽量不要让用户控制包含的文件名
    白名单，只允许包含指定的文件
    这是最好的

  2. 关闭远程文件包含
    allow_url_include = Off
    防止RFI

  3. open_basedir限制
    限制PHP只能访问指定目录
    就算有包含，也跑不出这个目录

  4. 过滤输入
    过滤 ../ 之类的目录穿越字符
    但这个容易被绕过
```

## 7.9 逻辑漏洞

最后讲一个大类——
**逻辑漏洞**。
这个跟前面的都不一样，
前面的都是技术层面的注入、执行，
逻辑漏洞是**业务逻辑设计有问题**。

### 7.9.1 什么是逻辑漏洞？

什么是逻辑漏洞？
简单说就是：
**业务逻辑设计有缺陷，
攻击者可以利用这个缺陷，
做一些本不应该做的事情。**

比如：
- 越权访问：
  普通用户可以看/改管理员的数据
- 支付漏洞：
  支付的时候改价格，
  一块钱买一万块的东西
- 密码找回漏洞：
  可以重置任意用户的密码
- 验证码绕过：
  验证码可以重复用，
  或者可以绕过
- 短信轰炸：
  短信验证码接口没有限制，
  可以无限发
- 并发漏洞（条件竞争）：
  利用并发，
  一次抽奖抽好几次，
  或者一块钱充好几次值
- 等等...

逻辑漏洞非常多，
每个业务都不一样，
没有统一的模式，
全靠对业务的理解和细心。

### 7.9.2 为什么逻辑漏洞越来越受重视？

现在技术层面的漏洞（SQL注入、XSS...）
因为大家安全意识提高了，
也有各种框架和WAF保护，
越来越少了。
但逻辑漏洞，
每个业务都不一样，
WAF也防不了，
扫描器也扫不出来，
只能靠人工去测。
所以逻辑漏洞越来越多，
也越来越受重视。

很多大厂的SRC（安全应急响应中心），
收到的漏洞里，
逻辑漏洞占比越来越高。

### 7.9.3 怎么防御逻辑漏洞？

逻辑漏洞怎么防？
这个没有银弹，
只能靠：
```
  1. 设计阶段考虑安全
    做需求设计的时候，
    就要考虑安全，
    想清楚各种异常情况、边界情况
    有没有越权的可能？
    有没有逻辑上的漏洞？
    安全左移，从设计阶段就介入

  2. 后端不能相信前端传来的任何数据
    所有的校验都要在后端做
    价格、数量、权限...
    前端的校验只是给用户体验用的，
    后端一定要再校验一遍
    永远不要相信前端

  3. 权限控制要做好
    每个操作都要校验权限
    你有没有权限看这个数据？
    有没有权限改这个数据？
    不能只校验登录状态，
    还要校验数据归属

  4. 关键操作要有日志、有审计
    出问题了能追查

  5. 安全测试
    上线前做安全测试，
    专门测逻辑漏洞
```

逻辑漏洞的防御，
更多的是靠意识和流程，
而不是靠某个技术。

## 7.10 本章总结

这一章我们讲了常见的Web应用攻击，
内容非常多，
也非常重要。
给你总结一下：

**SQL注入：**
- 用户输入被当成SQL代码执行
- 危害极大，拖库、篡改、拿shell
- 防御：预编译/参数化查询（根本）、输入过滤、最小权限、WAF

**XSS跨站脚本：**
- 恶意脚本在用户浏览器执行
- 分存储型、反射型、DOM型
- 危害：偷Cookie、钓鱼、挂马、会话劫持...
- 防御：输出转义（根本）、HttpOnly、CSP、前端框架

**文件上传漏洞：**
- 上传恶意脚本，直接拿webshell
- 各种绕过方式：前端校验、MIME、后缀名、解析漏洞...
- 防御：后端多重校验、白名单、文件内容校验、随机文件名、不可执行目录

**CSRF跨站请求伪造：**
- 借用户的身份发请求
- 利用Cookie自动携带
- 防御：CSRF Token（常用）、SameSite Cookie、二次验证、Referer检查

**SSRF服务端请求伪造：**
- 让服务器发起请求，打内网、读文件
- 危害：内网探测、攻击内网服务、云元数据攻击
- 防御：白名单、禁止内网IP、禁用危险协议、网络隔离

**命令执行/代码执行：**
- 用户输入被拼到命令/代码里执行
- 高危，直接拿服务器权限
- 防御：避免调用、白名单过滤、参数化传参、最小权限

**文件包含漏洞：**
- 包含任意文件，执行代码或读文件
- 分LFI和RFI
- 防御：白名单、关闭远程包含、open_basedir限制

**逻辑漏洞：**
- 业务逻辑设计缺陷
- 种类多，每个业务都不一样
- 扫描器扫不出来，WAF防不住
- 防御：安全左移、后端校验、权限控制、安全测试

最后送你一句话：

> **Web安全的本质，
> 就是输入和输出的安全。
> 所有输入都不可信，
> 所有输出都要谨慎。**
