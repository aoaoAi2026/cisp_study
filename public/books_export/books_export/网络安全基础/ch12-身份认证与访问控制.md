**身份认证与访问控制。**

**你是谁？（认证）

不管是系统、应用还是网络，
首先得搞清楚你是谁，
然后再决定你能做什么、不能做什么。
这就是身份认证和访问控制。
这是安全的基础中的基础。

## 12.1 先讲一个故事

### 12.1.1 "一个通用账号引发的惨案"

我有个朋友在一家公司做运维。
刚去的时候，
发现公司有个坏习惯——
**大家共用一个root账号。**
所有运维都知道root密码，
大家都用root登录服务器操作。

他跟领导提过好多次，
说这样不安全，
每个人用自己的账号，
给不同的权限，
操作也能审计。
领导说：
"哎呀，麻烦死了，
大家都是自己人，
用一个账号多方便。
我们人少，没事的。"

朋友劝了好几次没用，
也就算了。

结果有一天，
公司出大事了。
核心数据库被人删了，
数据全没了。
想查是谁干的，
查不到——
因为大家都用root账号，
谁都可能操作，
根本查不出来是谁删的。
是内鬼？还是被黑客攻破了？
都不知道。

最后查了好久才搞清楚，
是一个新来的运维，
操作的时候手滑了，
写错了SQL语句，
把整个库都删了。
但因为是公共账号，
他一开始不敢承认，
耽误了很多恢复的时间。
最后公司损失惨重，
那个领导也被问责了。

---

你看，
这就是没有身份认证和访问控制的后果——
**谁都能进、
谁都能操作、
出了事谁也找不到。**

很多公司、很多系统，
都有这个问题：
权限过大、共用账号、
职责不清、审计不明。
不出事则已，
一出事就是大事。

这一章，
我们就来讲讲身份认证和访问控制，
看看怎么搞清楚"你是谁"
和"你能干什么"这两个问题。

## 12.2 身份认证概述

### 12.2.1 什么是身份认证？

什么是身份认证（Authentication）？
简单说就是：
**验证你是谁的过程。**
证明你就是你声称的那个人。

比如：
- 你登录系统，输入用户名密码，
  系统验证密码对不对，
  对了就认你是这个用户。
  这就是身份认证。
- 你刷身份证进站，
  机器验证身份证是真的，
  人跟身份证照片对得上，
  就认你。
  这也是身份认证。
- 你刷脸解锁手机，
  手机验证你的脸对不对，
  对了就解锁。
  这也是身份认证。

身份认证是安全的第一步——
连你是谁都搞不清楚，
后面的授权、审计都无从谈起。

### 12.2.2 认证的三要素

身份认证的方式很多，
但一般可以归为三大类，
叫**认证三要素**：

```
  1. 你知道什么（Something you know）
    只有你知道的信息
    比如：密码、PIN码、
      密保问题答案、
      手势密码...

  2. 你有什么（Something you have）
    只有你拥有的东西
    比如：手机、U盾、
      动态令牌、智能卡、
      硬件密钥...

  3. 你是谁（Something you are）
    你本身的生物特征
    比如：指纹、人脸、
      虹膜、声纹、
      掌纹、步态...
```

三种要素各有优缺点：
- 知道什么：方便，
  但容易忘、容易被偷、容易被猜
- 有什么：安全，
  但丢了就麻烦了，还要花钱买
- 你是谁：最方便，不用记也不用带，
  但有隐私问题，也可能被伪造

一般来说，
要素越多越安全。
比如只用密码（单因素），
安全性就差一些；
密码 + 手机验证码（双因素），
安全性就高很多。

### 12.2.3 什么是多因素认证？

**多因素认证（MFA，Multi-Factor Authentication）**，
也叫双因素认证（2FA），
就是同时用两种或以上不同类型的认证要素来验证身份。

比如：
- 密码 + 短信验证码
- 密码 + 指纹
- 密码 + 动态令牌（Google Authenticator）
- U盾 + PIN码
- 人脸 + 密码
- 等等...

为什么多因素更安全？
因为就算一个因素泄露了，
还有另一个。
比如密码被偷了，
但攻击者没有你的手机，
还是登不进去。
大大提升了安全性。

现在重要的系统，
比如银行、邮箱、云服务、
企业内部系统...
都强烈建议开启多因素认证。
很多安全事件，
如果开了MFA，
根本就不会发生。

**强烈建议：
所有重要账号，
一定要开启双因素认证！**
这是成本最低、
效果最好的安全措施之一。

## 12.3 常见的认证方式

常见的身份认证方式有哪些？
给你介绍几种。

### 12.3.1 用户名 + 密码

最经典、最常用的认证方式——
**用户名 + 密码。**

优点：
- 简单，不用额外的硬件
- 成本低，大家都会用
- 实现简单

缺点：
- 安全性差，容易被猜、被偷、被泄露
- 用户容易设弱密码
- 不同网站用同一个密码，
  一个网站泄露，全网遭殃
- 钓鱼攻击容易偷到密码

虽然密码有很多问题，
但目前来说，
它依然是最主流的认证方式。
短时间内还不会被淘汰。

关于密码安全的建议：
- 密码要足够长、足够复杂
- 不同网站用不同的密码
- 用密码管理器管理密码
- 开启双因素认证
- 定期更换重要密码

### 12.3.2 短信验证码 / 邮箱验证码

第二种很常见的——
**短信验证码**。
登录的时候，
给你手机发个短信，
里面有个验证码，
你输入进去就认证通过了。
邮箱验证码类似。

优点：
- 方便，大家都有手机
- 比纯密码安全一些，
  属于第二因素

缺点：
- 短信可能被劫持、被诈骗
- 手机没信号就收不到
- 换手机号麻烦
- 有成本（短信费）

短信验证码作为第二因素还是不错的，
但不建议作为唯一的认证方式。

### 12.3.3 动态令牌 / TOTP

第三种，
**动态令牌**，
也叫一次性密码（OTP，One-Time Password）。

最常见的是 **TOTP**（基于时间的一次性密码），
比如Google Authenticator、
微软身份验证器、
各种令牌App。
它每隔30秒生成一个6位数字的验证码，
登录的时候输入这个验证码。

优点：
- 比短信更安全，
  不会被短信劫持
- 不用联网也能用
- 免费，只要有手机就行

缺点：
- 手机丢了就麻烦了
  （可以有恢复码）
- 需要装个App

这是目前最推荐的双因素认证方式之一，
安全又方便。

### 12.3.4 硬件密钥 / U2F / WebAuthn

更高级的，
**硬件密钥**。
比如YubiKey，
一个小小的USB设备，
插在电脑上，
按一下就认证通过了。
基于U2F或者WebAuthn标准。

优点：
- 非常安全，几乎不会被破解
- 防钓鱼，
  就算点了钓鱼网站，
  硬件密钥也不会认证通过
- 不用装软件，
  插上去按一下就行

缺点：
- 要花钱买硬件
- 丢了要重新配置
- 有些旧网站不支持

这是目前最安全的认证方式之一，
对安全要求高的场景推荐使用。

### 12.3.5 生物特征认证

第五种，
**生物特征认证**。
指纹、人脸、虹膜、声纹...
用你的身体特征来认证。

优点：
- 方便，不用记也不用带
- 用户体验好

缺点：
- 隐私问题，生物特征信息很敏感
- 不是百分之百准确，有误识和拒识
- 生物特征泄露了没法改
  （密码泄露了可以改，指纹泄露了怎么办？）
- 可以被伪造（比如人脸用照片、模型）

生物识别用在解锁手机、
门禁这些场景很方便，
但用在高安全级别的场景，
还是要谨慎。

### 12.3.6 单点登录（SSO）

最后讲一个概念——
**单点登录（SSO，Single Sign-On）**。

什么是单点登录？
简单说就是：
**一次登录，到处通行。**
你只需要登录一次，
就可以访问所有互相信任的系统，
不用每个系统都登录一遍。

比如：
- 你登录了公司的OA系统，
  再打开邮件系统、CRM系统、
  财务系统...
  都自动登录了，不用再输密码。
- 你登录了Google账号，
  Gmail、YouTube、Google Drive...
  都不用再登录了。

这就是单点登录。

SSO有什么好处？
```
  • 用户体验好，不用记那么多密码，
    不用登录那么多次
  • 更安全，密码只用在一个地方输，
    减少泄露风险
  • 管理方便，
    统一管理账号和权限，
    离职了一个地方禁用就全禁用了
```

常见的SSO协议：
```
  • SAML
    比较老的协议，
    企业级用得很多

  • OAuth 2.0
    现在最流行的授权框架
    "用XX账号登录"基本都是这个
    注意：OAuth是授权框架，
      不是认证协议，
      但经常被用来做认证

  • OpenID Connect（OIDC）
    基于OAuth 2.0的身份层
    专门用来做认证的
    现在越来越流行

  • CAS
    耶鲁大学开发的，
    高校用得比较多

  • LDAP / AD域
    企业内网常用，
    域账号登录也是一种SSO
```

单点登录是企业身份管理的基础，
现在几乎所有稍大一点的企业，
都有自己的SSO系统。

## 12.4 AAA 框架

讲到身份认证，
就不得不提 **AAA**。
这是身份管理的一个经典框架。

### 12.4.1 什么是AAA？

AAA是什么？
三个A：
```
  Authentication  认证  —— 你是谁？
  Authorization   授权  —— 你能干什么？
  Accounting      计费/审计 —— 你干了什么？
```

三个A连在一起，
构成了一个完整的身份管理闭环：
1. 先认证你是谁
2. 再授权你能做什么
3. 最后记录你做了什么（审计/计费）

AAA最早是用在网络接入里的，
比如拨号上网、VPN接入，
现在已经扩展到各个领域了，
是身份与访问管理（IAM）的基础框架。

三个A缺一不可：
- 没有认证，不知道是谁
- 没有授权，随便干什么都行
- 没有审计，出了事查不到

### 12.4.2 认证（Authentication）

第一个A，
**认证**。
前面已经讲了很多了，
就是验证身份的过程。
证明你是谁。

### 12.4.3 授权（Authorization）

第二个A，
**授权**。
认证完了之后，
决定你能做什么、不能做什么。
你有什么权限，
能访问哪些资源，
能执行哪些操作。

比如：
- 普通用户只能看自己的数据，
  管理员可以看所有数据
- 运维能登录服务器，
  研发不能登录生产服务器
- 财务能看财务数据，
  其他人看不了

授权的方法很多，
后面访问控制部分会详细讲。

### 12.4.4 审计 / 计费（Accounting）

第三个A，
**Accounting**。
这个A有两种翻译，
计费或者审计。
在运营商场景叫计费（计算上网时长、流量，用来收费），
在企业安全场景更多叫**审计**——
记录用户的操作，
谁、在什么时间、做了什么操作，
都记下来，
出了问题可以追溯。

审计非常重要：
- 出了安全事件，可以溯源
- 合规要求（等保、ISO 27001都要求审计日志）
- 威慑作用，知道操作会被记录，
  就不敢乱来了

审计日志要注意：
- 要完整，关键操作都要记
- 要安全，不能被篡改、被删除
- 最好存在远程的日志服务器上，
  本地的日志不可信

## 12.5 访问控制

讲完了认证，
接下来讲**访问控制（Access Control）**。
也就是授权——
你能访问什么、不能访问什么。

### 12.5.1 什么是访问控制？

什么是访问控制？
简单说就是：
**对主体访问客体的行为进行控制，
允许授权的访问，
拒绝未授权的访问。**

用人话说就是：
**谁（主体），
能对什么东西（客体），
做什么操作（权限）。**

三个要素：
```
  主体（Subject）：
    发起访问的一方
    比如用户、进程、服务...

  客体（Object）：
    被访问的资源
    比如文件、数据、接口、
      服务器、打印机...

  权限（Permission / Access Right）：
    可以做什么操作
    比如读、写、执行、删除、
      修改、管理...
```

访问控制就是管理这三者之间的关系，
规定哪个主体对哪个客体有什么权限。

### 12.5.2 访问控制的原则

访问控制有几个重要的原则：

```
  1. 最小权限原则（Principle of Least Privilege）
    给主体刚好能完成工作的最小权限
    多一点都不给
    这是最重要的原则！
    比如运维只给他需要运维的那几台机器的权限，
      不要给所有服务器的root权限
    比如普通员工只给他需要的系统权限，
      不要给管理员权限
    最小权限能大大降低安全风险，
      就算账号被攻破了，
      危害也有限

  2. 最小泄露原则
    知道最少够用的信息就行
    跟最小权限类似

  3. 职责分离（Separation of Duties）
    一项重要的工作，
    不能一个人说了算
    要分成几步，
    由不同的人来完成
    互相监督，互相制约
    比如转账，
      一个人操作，一个人审核
    防止一个人作弊
    这也是前面讲的Clark-Wilson模型的核心思想

  4. 知其所需（Need to Know）
    只有需要知道的人，
    才能访问对应的信息
    不需要知道的就不让看
    跟最小权限类似

  5. 默认拒绝（Default Deny）
    默认情况下，
    什么都不让访问
    只允许明确授权的访问
    白名单思路
    比默认允许黑名单安全多了
```

这些原则都非常重要，
尤其是**最小权限原则**，
可以说是安全领域最重要的原则之一。
很多安全事件，
就是因为权限太大了，
一个普通用户账号被攻破，
直接就拿到了管理员权限，
一路横推。

### 12.5.3 常见的访问控制模型

访问控制有哪些常见的模型？
给你介绍几个经典的。

**1. 自主访问控制（DAC）**

DAC，Discretionary Access Control。
什么意思？
**资源的所有者（创建者）
自己决定谁能访问这个资源，
有什么权限。**

比如：
- 你建了一个文件，
  你可以设置谁能读、谁能写、谁能改
- 你是项目负责人，
  你可以把项目权限分给团队成员

Linux/Unix的文件权限就是典型的DAC，
文件的所有者可以改权限。

优点：
- 灵活，用户自己控制
- 适合小团队、个人场景

缺点：
- 安全性不够高，
  权限容易被乱传
- 不好统一管理
- 信息容易泄露

**2. 强制访问控制（MAC）**

MAC，Mandatory Access Control。
和DAC相反，
**访问控制是系统强制的，
用户自己说了不算。
由系统（或者管理员）
统一分配安全标签，
主体和客体都有安全级别，
按照规则来决定能不能访问。**

比如：
- 文件的安全级别是"机密"，
  你的安全级别是"秘密"，
  那你就不能看这个文件
  （就是前面讲的BLP模型那种思路）

优点：
- 安全性高，强制管控
- 适合保密性要求高的场景
  （比如政府、军队、涉密系统）

缺点：
- 不灵活，管理成本高
- 太严格，用户体验差
- 一般企业用不上

**3. 基于角色的访问控制（RBAC）**

RBAC，Role-Based Access Control。
这是目前企业里用得最多的访问控制模型。

什么意思？
**不直接给用户分配权限，
而是给角色分配权限，
用户属于什么角色，
就有什么角色的权限。**

比如：
- 角色：管理员、运维、研发、
  财务、HR、普通员工...
- 每个角色有对应的权限
- 用户加入某个角色，
  就有了这个角色的所有权限

一个用户可以有多个角色。

优点：
```
  • 管理方便，
    人员变动了只要改角色就行，
    不用一个个改权限
  • 权限和角色对应，
    清晰易懂
  • 适合企业这种组织结构比较稳定的场景
  • 大大降低了权限管理的复杂度
```

缺点：
- 有时候不够细，
  角色粒度不好把握
- 复杂场景可能不够用

RBAC是目前最主流的访问控制模型，
几乎所有企业系统、各种管理后台，
用的都是RBAC或者它的变种。

**4. 基于属性的访问控制（ABAC）**

ABAC，Attribute-Based Access Control。
比RBAC更灵活的模型。

什么意思？
**根据各种属性来动态判断有没有权限。**
属性可以是：
- 用户属性（部门、职位、级别...）
- 资源属性（文件级别、所属部门...）
- 环境属性（时间、地点、IP...）
- 操作属性（读、写...）

比如：
- "工作时间内，研发部的员工
  可以访问研发部的项目文档"
- "只有在公司内网，
  才能访问生产环境"
- "财务数据只有财务部门的人，
  在上班时间，用公司电脑才能访问"

优点：
- 非常灵活，
  可以实现很复杂的权限策略
- 动态判断，更智能

缺点：
- 复杂，实现和管理成本高
- 策略写不好容易出问题

ABAC是未来的趋势，
零信任架构里经常用到ABAC。

还有一些其他模型，
比如基于规则的访问控制（RuBAC）、
基于组织的访问控制（OrBAC）...
就不一一讲了。

---

## 12.6 特权账号管理（PAM）

讲访问控制，
不得不提 **特权账号管理（PAM，Privileged Access Management）**。

### 12.6.1 什么是特权账号？

什么是特权账号？
简单说就是**权限特别大的账号**。
比如：
- 服务器的root、administrator
- 数据库的DBA账号
- 网络设备的管理员账号
- 系统的超级管理员账号
- 云服务的管理员账号
- 等等...

这些账号权限极大，
一旦泄露或者被滥用，
危害非常大。
很多安全事件，
最后都是因为特权账号没管好。

### 12.6.2 特权账号怎么管？

特权账号怎么管理？
核心思路就是：
**最小权限 + 集中管理 + 全程审计。**

常见的措施：
```
  1. 减少特权账号
    能不用特权账号的就不用
    普通账号够用就用普通账号
    用的时候再申请临时提权

  2. 集中管理
    特权账号统一管理，
    密码存在密码保险箱里，
    不用的时候谁也不知道密码
    要用的时候申请，
    系统自动分配，用完改密码

  3. 堡垒机 / 运维审计
    所有特权操作都通过堡垒机
    统一入口、统一认证、统一授权
    全程录像、全程审计
    谁操作了什么都清清楚楚

  4. 最小权限
    给刚好够用的权限，
    不要什么都给root
    比如有些操作用sudo给特定命令的权限就行

  5. 多因素认证
    特权账号必须开MFA
    多一层保障

  6. 定期轮换密码
    特权账号密码定期更换
    防止密码泄露了很久才发现

  7. 审计与监控
    特权操作全程监控
    异常操作及时告警
```

特权账号是安全的重中之重，
一定要管好。
很多企业等保三级、
ISO 27001认证，
都要求有特权账号管理。

## 12.7 零信任（Zero Trust）

最后讲一个现在很火的概念——
**零信任（Zero Trust）。**

### 12.7.1 什么是零信任？

零信任是什么？
核心思想就一句话：
**永远不要信任，
永远要验证。
（Never Trust, Always Verify）**

传统的安全思路是：
**内网都是可信的，
外网都是不可信的。**
所以我们在边界堆防火墙，
把坏人挡在外面，
里面的就都信任。

但现在这种思路不行了，
为什么？
- 攻击者可以通过钓鱼、
  社工等手段，
  从内部突破
- 内网扁平化，
  一台机器被攻破，
  整个内网都危险
- 现在很多业务都在云上，
  边界越来越模糊
- 远程办公越来越多，
  人不一定在公司内网

所以就有了零信任——
**不管你在内网还是外网，
不管你是谁，
每次访问都要验证身份和权限，
从不自动信任。**

零信任的几个核心原则：
```
  1. 永不信任，始终验证
    没有什么"内部可信网络"
    任何访问都要验证

  2. 最小权限原则
    只给刚好够用的权限
    刚好够用的时间

  3. 假设 breach
    假设已经被攻破了
    从最坏情况出发设计安全
    处处设防，层层防护

  4. 持续验证、持续评估
    不是登录一次就万事大吉了
    持续监控用户行为，
    发现异常及时调整权限
    动态访问控制
```

### 12.7.2 零信任怎么落地？

零信任怎么落地？
它不是一个产品，
而是一种理念、一种架构。
落地零信任需要很多东西配合：

```
  • 身份认证：
    强认证、MFA、统一身份管理
    身份是零信任的核心

  • 访问控制：
    细粒度的访问控制，
    基于属性的动态授权（ABAC）
    最小权限

  • 微隔离：
    网络微分段，
    每个应用、每个服务之间都隔离
    就算一台被攻破，
      也横向移动不了
    相当于内网里也到处是防火墙

  • 持续监控与分析：
    持续监控用户行为和流量
    用UEBA（用户与实体行为分析）
    发现异常

  • 数据安全：
    数据分类分级，
    数据防泄漏
    数据本身也要保护

  • ......
    很多很多
```

零信任是现在安全的大趋势，
很多企业都在往零信任方向走。
当然，
零信任不是一步就能到位的，
是个逐步建设的过程。

## 12.8 本章总结

这一章我们讲了身份认证与访问控制，
内容不少。
总结一下：

**身份认证：**
- 验证你是谁的过程
- 认证三要素：你知道什么、你有什么、你是谁
- 多因素认证（MFA）：用两种以上要素，更安全
  强烈建议重要账号都开MFA

**常见认证方式：**
- 用户名密码（最常用，但安全性一般）
- 短信/邮箱验证码
- TOTP动态令牌（推荐）
- 硬件密钥（最安全）
- 生物特征（方便但有隐私问题）
- 单点登录（SSO）：一次登录，到处通行
  协议：SAML、OAuth 2.0、OIDC...

**AAA 框架：**
- Authentication 认证：你是谁？
- Authorization 授权：你能干什么？
- Accounting 审计/计费：你干了什么？
- 三个A构成完整的身份管理闭环

**访问控制：**
- 控制谁对什么资源能做什么操作
- 三要素：主体、客体、权限
- 重要原则：
  最小权限（最重要！）、
  职责分离、默认拒绝、知其所需...
- 常见模型：
  DAC（自主）、MAC（强制）、
  RBAC（基于角色，最常用）、
  ABAC（基于属性，更灵活）

**特权账号管理（PAM）：**
- 特权账号权限大，风险高
- 集中管理、堡垒机、最小权限、
  MFA、全程审计

**零信任（Zero Trust）：**
- 核心思想：永不信任，始终验证
- 传统边界安全思路不行了
- 原则：永不信任、最小权限、
  假设突破、持续验证
- 不是一个产品，是一种架构和理念
- 身份是核心，微隔离、动态授权...

最后送你一句话：

> **身份是新的边界，
> 权限是新的防线。
> 守好身份这道门，
> 比堆十层防火墙都有用。**
