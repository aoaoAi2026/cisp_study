**灾难恢复与业务连续性。**

- 机房着火了
- 地震、洪水等自然灾害
- 电力中断
- 光纤被挖断
- 服务器硬件全部损坏
- 甚至整栋楼都没了

这些极端情况，
虽然发生概率低，
但一旦发生，
对企业的打击可能是毁灭性的。
没有准备的话，
可能直接就倒闭了。

所以，
企业不光要考虑安全事件，
还要考虑各种灾难场景，
要有灾难恢复的能力，
要保证业务连续性——
**就算天塌下来，
业务也能继续跑，
至少能尽快恢复。**

这一章我们就来讲讲：
什么是灾难恢复、

RTO和RPO是什么、
备份怎么做、

## 17.1 先讲一个故事

### 17.1.1 "一场大火烧掉的公司"

我给你讲个真实的故事。

某公司，
做电商的，
发展得还不错，
一年几个亿的营收。
他们的机房在办公楼的地下室，
服务器什么的都在那。
公司觉得自己有备份——
每天晚上备份到同一机房的另一台存储上。
觉得这样就万无一失了。

结果某天晚上，
大楼电路老化，
地下室着火了。
等消防员把火扑灭，
机房里的服务器、存储、网络设备...
全烧没了。
包括那台备份存储，
也一起烧成了焦炭。

第二天，
公司业务全停了，
网站打不开，
APP用不了，
用户的数据全没了。
老板这才慌了——
**所有数据都在那栋楼里，
楼烧没了，数据也没了。**
没有异地备份，
没有容灾机房，
什么都没有。

最后，
公司花了好大功夫，
从用户手机端、
从各种历史记录里，
一点点恢复数据，
但还是损失了大量数据。
业务停了大半个月，
用户跑了一大半，
公司元气大伤，
差点就倒闭了。

后来老板说：
"以前觉得容灾是大公司才搞的东西，
我们小公司用不着。
现在才知道，
没有容灾，
一场火就能把公司烧没了。
早知道花点钱搞个异地备份，
也不至于这样。"

---

你看，
很多人都有侥幸心理：
"这种倒霉事怎么会轮到我？"
但概率再低，
万一赶上了呢？
对企业来说，
可能就是灭顶之灾。

**灾难恢复和业务连续性，
就是为这种"万一"准备的。
平时可能用不上，
但一旦用上，
就是救命的。**

这一章，
我们就来讲讲灾难恢复和业务连续性那些事。

## 17.2 基本概念

先讲几个基本概念。

### 17.2.1 什么是灾难恢复？

什么是灾难恢复
（Disaster Recovery，简称DR）？
简单说就是：
**灾难发生之后，
把IT系统和数据恢复过来，
让业务重新跑起来的能力。**

什么算灾难？
大的小的都算：
- 服务器坏了
- 存储坏了
- 机房断电了
- 光纤断了
- 机房着火了
- 地震、洪水、台风...
- 重大安全事件（比如勒索软件把全公司数据都加密了）

凡是能导致业务中断的大的故障，
都可以算灾难。
灾难恢复，
就是出了这些事之后，
怎么把系统和数据恢复回来。

### 17.2.2 什么是业务连续性？

什么是业务连续性
（Business Continuity，简称BC）？
简单说就是：
**保证业务在各种情况下
都能持续运行，
不中断，或者只中断很短时间。**

灾难恢复是"出了事之后恢复"，
业务连续性是更高层面的——
**尽量不要断，
就算出了事也要能继续跑，
或者断了也能很快恢复，
对业务影响最小。**

灾难恢复偏技术，
业务连续性偏业务、偏管理。
业务连续性不光考虑IT，
还考虑人、流程、办公场地、
供应链、客户沟通...
是个更宏观的概念。

一般合起来叫 **BCDR**
（Business Continuity and Disaster Recovery），
业务连续性与灾难恢复。

### 17.2.3 RTO 和 RPO

讲灾难恢复，
两个最重要的指标必须知道——
**RTO** 和 **RPO**。

这两个指标是衡量
灾难恢复能力的核心标准。

**RTO（Recovery Time Objective，
恢复时间目标）：**
> 灾难发生之后，
> 业务从中断到恢复正常，
> 最多能接受多长时间？
> 也就是**最长能断多久**。

比如RTO = 4小时，
意思就是：
灾难发生后，
4小时之内必须恢复业务。
超过4小时就不能接受了。

RTO越短，
说明对恢复速度要求越高，
当然成本也越高。

**RPO（Recovery Point Objective，
恢复点目标）：**
> 灾难发生之后，
> 最多能容忍丢失多长时间的数据？
> 也就是**最多能丢多久的数据**。

比如RPO = 1小时，
意思就是：
恢复之后，
最多丢1小时的数据。
也就是至少每小时要备份一次。

RPO越短，
说明对数据完整性要求越高，
丢的数据越少，
成本也越高。

举个例子帮助理解：
- 如果每天凌晨备份一次，
  那RPO大概是24小时。
  白天的数据如果没备份，
  灾难发生了就丢了。
- 如果每小时备份一次，
  那RPO就是1小时。
  最多丢1小时的数据。
- 如果是实时同步（双活），
  那RPO接近于0，
  基本不丢数据。

RTO和RPO是容灾建设的核心指标，
搞容灾首先要确定这两个指标，
然后根据指标来设计方案。
指标越高，
成本越高，
要根据业务的重要性和预算来定。

## 17.3 备份

讲灾难恢复，
首先得讲**备份**。
备份是最基础、
也是最重要的灾难恢复手段。

### 17.3.1 备份的重要性

备份有多重要？
这么说吧：
**备份是数据安全的最后一道防线。**
- 中了勒索软件？有备份就能恢复。
- 服务器坏了？有备份就能恢复。
- 误删了数据？有备份就能恢复。
- 机房烧了？有异地备份就能恢复。

没有备份，
数据丢了就是真没了，
神仙也找不回来。

**所以，重要数据一定要备份！
一定要备份！一定要备份！**
重要的事情说三遍。

### 17.3.2 备份的类型

备份有哪些类型？
常见的分法：

```
  1. 全量备份（Full Backup）
    把所有数据完整备份一份
    优点：恢复快，完整
    缺点：占空间大，备份时间长

  2. 增量备份（Incremental Backup）
    只备份上次备份之后变化的数据
    优点：备份快，占空间小
    缺点：恢复麻烦，
      要先恢复全量，再一个个恢复增量
      恢复时间长

  3. 差异备份（Differential Backup）
    只备份上次全量备份之后变化的数据
    和增量的区别：
      增量是跟上一次备份（不管什么类型）比
      差异是跟上一次全量备份比
    介于全量和增量之间
    恢复比增量快，
      只需要全量 + 最后一次差异
    备份速度和空间也介于两者之间
```

实际中一般是组合着用，
比如：
- 每周日做一次全量备份
- 周一到周六每天做增量备份
  （或者差异备份）
这样兼顾备份速度和恢复速度。

### 17.3.3 备份的"3-2-1"原则

备份有个很经典的原则，
叫 **3-2-1 原则**。
非常实用，建议都记住。

什么意思？
```
  3 份数据副本：
    除了原始数据，还有至少2份备份，
    加起来一共3份

  2 种存储介质：
    备份存在至少两种不同的介质上
    比如硬盘 + 磁带，
      或者本地硬盘 + 云存储
    防止一种介质坏了全完蛋

  1 份异地备份：
    至少有一份备份在异地
    不在同一个机房、同一个城市
    防止本地灾难（火灾、地震...）
    把所有备份都毁了
```

就像前面故事里的公司，
备份和原始数据都在同一个机房，
机房一烧，全没了。
这就违反了3-2-1原则的
"1份异地备份"。

记住3-2-1原则，
你的备份策略就不会差到哪去。

### 17.3.4 备份的注意事项

备份还有几个注意事项：

```
  1. 定期测试恢复
    很多人备份了就不管了，
      真要恢复的时候发现备份坏了，
      或者恢复不了
    那就悲剧了
    备份不是目的，能恢复才是目的
    一定要定期测试备份能不能正常恢复
    （非常重要！）

  2. 备份要加密
    备份里有重要数据，
      要加密存储
    防止备份泄露

  3. 备份访问控制
    不是谁都能看备份、删备份
    权限要控制好
    防止攻击者进来把备份也删了
    （勒索软件就经常干这个）

  4. 离线备份 / 冷备份
    至少有一份备份是离线的，
      或者不可写的
    比如磁带、光盘、
      或者对象存储的归档存储
    防止勒索软件把在线备份也加密了

  5. 备份的保留策略
    备份保留多久？
    每天的备份保留多久？
    每周的、每月的、每年的？
    要有个策略
    不能无限存，也不能存太少
```

备份说起来简单，
但真的做好也不容易。
很多公司的备份都是摆设，
真出事了才发现不好使。
**备份好不好，
拉出来恢复一下就知道了。**

## 17.4 容灾架构

备份是基础，
但如果对RTO和RPO要求高，
光靠备份就不够了。
这时候就需要**容灾架构**。

什么是容灾？
就是在异地再建一套系统，
平时主用，
灾难的时候切换到备用的，
保证业务快速恢复，
甚至不中断。

容灾有不同的级别，
我们从低到高讲几种常见的。

### 17.4.1 冷备

最低级的容灾——
**冷备（Cold Backup）**。

什么意思？
异地有一套备用的环境，
但平时不开机，
或者只有基础环境，
没有数据，也不跑业务。
灾难发生之后，
再把机器开起来，
把数据恢复过去，
再把业务切过去。

```
  平时：
    主站点跑业务
    备站点关机 / 空跑

  灾难时：
    启动备站点机器
    恢复数据（从备份恢复）
    切流量过去
    业务恢复
```

优点：
- 成本低
  备站点平时不用跑，
  不用那么多资源

缺点：
- RTO长，
  可能几小时甚至几天
- RPO也长，
  取决于备份频率
- 恢复过程复杂，容易出问题

适合那些对RTO要求不高的
非核心业务。

### 17.4.2 温备

比冷备高级一点——
**温备（Warm Backup）**。

什么意思？
备站点平时是开机的，
系统也装好了，
数据也定期同步过去，
但平时不跑业务。
灾难发生了，
把服务启动起来，
切流量过去就行。

```
  平时：
    主站点跑业务
    备站点开着机，
      数据定期同步过去，
      但不跑业务

  灾难时：
    启动备站点的服务
    切流量过去
    业务恢复
```

优点：
- RTO比冷备短，
  几小时到几十分钟
- RPO也短一些，
  数据定期同步

缺点：
- 成本比冷备高
- 还是有中断时间

大部分对容灾有要求但
要求不是特别高的业务，
温备就够用了。

### 17.4.3 热备 / 主备

再高级一点——
**热备（Hot Backup）**，
也叫主备模式。

什么意思？
备站点和主站点同时运行，
数据实时同步，
备站点随时可以接管业务。
但平时流量不走备站点，
只有主站点出问题了，
才切到备站点。

```
  平时：
    主站点跑业务，所有流量都在主
    备站点数据实时同步，待命

  灾难时：
    检测到主站点挂了
    自动/手动切换到备站点
    业务很快恢复
```

优点：
- RTO很短，
  几分钟到几十分钟
  （取决于切换速度）
- RPO也很短，
  基本不丢数据或者只丢一点点

缺点：
- 成本高，
  备站点平时不用但也要一直开着
- 资源利用率低，
  备站点平时闲着

适合核心业务，
对RTO和RPO要求比较高的。

### 17.4.4 双活 / 多活

最高级的——
**双活 / 多活（Active-Active）**。

什么意思？
两个或多个站点同时跑业务，
流量同时走多个站点，
同时对外服务。
一个站点挂了，
流量自动切到其他站点，
用户基本感觉不到。

```
  平时：
    多个站点同时跑，
      同时提供服务
    数据实时同步
    负载均衡分配流量

  灾难时：
    一个站点挂了
    流量自动切到其他站点
    用户基本无感知
```

优点：
- RTO几乎为0，
  基本不中断
- RPO几乎为0，
  基本不丢数据
- 资源利用率高，
  所有站点都在干活

缺点：
- 成本非常高
  （两倍甚至多倍的资源）
- 技术复杂，
  数据同步、流量调度、
  一致性...都很麻烦
- 不是所有业务都能做多活

适合非常核心的业务，
对可用性要求极高的。
比如金融、电商核心交易系统...

### 17.4.5 两地三中心

最后讲一个国内很常见的
容灾架构——
**两地三中心**。

什么意思？
```
  两地：两个城市
  三中心：
    • 同城双中心（同一个城市两个机房）
      平时双活或者主备，
      一个机房挂了另一个顶上
      因为同城，延迟低，好做同步

    • 异地灾备中心（另一个城市）
      防止整个城市出问题
      （比如地震、大停电...）
      平时是冷备或者温备
```

同城的两个中心，
距离近（几十公里），
网络延迟低，
可以做双活或者同步复制，
RTO和RPO都很好。
但如果整个城市都出事了
（比如大地震），
两个中心可能都完蛋。
所以再搞个异地的，
距离远一点（几百公里以上），
城市级灾难的时候，
还能从异地恢复。

这样兼顾了
日常的高可用
和极端灾难的恢复，
是国内很多金融、政府、
大型企业常用的容灾架构。

## 17.5 高可用（HA）

最后简单提一下**高可用（HA）**。
很多人容易把高可用和容灾搞混，
简单说说区别。

### 17.5.1 什么是高可用？

什么是高可用
（High Availability，简称HA）？
简单说就是：
**系统尽量不停机，
一直能提供服务。**

高可用和容灾的区别：
```
  高可用（HA）：
    主要应对的是单个组件的故障
    比如一台服务器坏了、
      一块硬盘坏了、
      一个进程挂了
    范围比较小，
      是本地的、局部的
    目标是尽量不中断，
      或者只中断很短时间

  灾难恢复（DR）：
    应对的是更大范围的灾难
    比如整个机房挂了、
      整个城市出问题
    范围更大，
      是异地的、全局的
    目标是灾难后能恢复
```

简单说：
HA是小故障不宕机，
容灾是大灾难能恢复。
两者是互补的，
不是一回事。

### 17.5.2 常见的高可用技术

常见的高可用技术：
```
  • 集群
    多台服务器组成集群，
      一台挂了另一台顶上
    比如LVS集群、
      Keepalived、
      数据库集群...

  • 负载均衡
    流量分发到多台服务器，
      一台挂了切到其他的
    同时提高性能和可用性

  • 冗余
    关键组件都有备份
    双电源、双网卡、双交换机、
      双链路...
    一个坏了另一个顶上

  • RAID
    磁盘阵列，
      一块硬盘坏了不丢数据，
      不影响使用

  • 进程守护
    进程挂了自动拉起来
    比如systemd、supervisor...
```

高可用和容灾一起，
共同保证业务的连续性。

## 17.6 业务连续性管理（BCM）

前面讲的都是偏技术的，
最后讲一下偏管理的——
**业务连续性管理
（Business Continuity Management，
简称BCM）**。

什么是业务连续性管理？
简单说就是：
**一整套管理体系，
来识别潜在的威胁，
制定预案和措施，
确保重大事件发生时，
业务能够持续运行，
或者快速恢复。**

业务连续性管理不光是IT的事，
是整个公司的事。
比如：
- 办公楼不能用了，
  员工去哪里办公？
  （远程办公、备用办公场地）
- 关键岗位的人出事了，
  谁来顶？
- 供应链断了怎么办？
- 客户怎么通知？
- 媒体怎么应对？
- ......

这些都是业务连续性管理要考虑的。
技术只是其中一部分。

BCM的大致步骤：
```
  1. 业务影响分析（BIA）
    分析各个业务的重要性
    如果中断了会有什么影响
    能容忍中断多久
    确定RTO、RPO等指标

  2. 风险评估
    有哪些潜在的风险和威胁？
    发生的概率多大？
    影响多大？

  3. 制定策略和预案
    针对不同的风险，
      制定应对策略和预案
    技术的、管理的、人的...

  4. 建设和实施
    按照策略去建设
    容灾系统、备份、备用场地...

  5. 培训和演练
    员工培训，
    定期演练
    检验预案行不行

  6. 持续改进
    定期审计、评估、改进
    业务变了也要跟着变
```

业务连续性管理是个系统工程，
涉及公司的方方面面。
大公司一般都有专门的人或者团队来管。

## 17.7 本章总结

这一章我们讲了灾难恢复与业务连续性。
总结一下：

**基本概念：**
- **灾难恢复（DR）**：灾难后恢复IT系统和数据
- **业务连续性（BC）**：保证业务持续运行，更宏观
- **RTO（恢复时间目标）**：最多能中断多久
- **RPO（恢复点目标）**：最多能丢多久的数据
- 指标越高，成本越高

**备份：**
- 备份是数据安全的最后一道防线
- 类型：全量备份、增量备份、差异备份
- **3-2-1原则**：3份副本、2种介质、1份异地
- 注意：定期测试恢复、加密、权限控制、离线备份

**容灾架构：**
- **冷备**：成本低，RTO/RPO长，适合非核心业务
- **温备**：平时开机，定期同步，RTO几小时
- **热备（主备）**：实时同步，随时切换，RTO短
- **双活/多活**：同时跑，几乎不中断，成本最高
- **两地三中心**：同城双中心 + 异地灾备，国内常用

**高可用（HA）：**
- 应对单个组件故障，尽量不宕机
- 和容灾的区别：范围不同，局部 vs 全局
- 常见技术：集群、负载均衡、冗余、RAID...

**业务连续性管理（BCM）：**
- 全公司层面的管理体系
- 不只是IT，还有人、流程、场地...
- BIA、风险评估、预案、演练、持续改进

> **天有不测风云，
> 人有旦夕祸福。
> 未雨绸缪，
> 才能有备无患。**
