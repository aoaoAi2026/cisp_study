## 16.1 先讲一个故事

### 16.1.1 "一次手忙脚乱的勒索软件事件"

某公司某天早上上班，
员工一打开电脑，
发现桌面上的文件都打不开了，
后缀名变成了奇怪的字母，
桌面上还有个勒索信，
说文件被加密了，
要付比特币才能解密。

不好，中勒索软件了！

然后整个公司就乱套了：
- 员工慌了，不知道怎么办
- IT部门到处救火，手忙脚乱
- 老板大发雷霆，质问怎么回事
- 有人说赶紧拔网线，
  结果拔晚了，已经传了好多台
- 有人说赶紧付钱，
  不然数据就没了
- 有人说不能付钱，
  付钱也不一定能解密
- 乱哄哄闹了一整天，
  才想起来找专业的应急响应团队

最后花了好大功夫才搞定，
数据虽然大部分恢复了
（幸好有备份），
但业务停了好几天，
损失惨重。

事后复盘的时候发现：
- 公司根本没有应急响应预案
- 出了事谁都不知道该干嘛
- 也不知道找谁、走什么流程
- 第一时间没有断网隔离，
  导致感染范围扩大
- 没有保留现场，
  很多日志被覆盖了，
  根本查不出来是怎么进来的
- 备份虽然有，
  但从来没试过恢复，
  恢复的时候出了好多问题

---

你看，
很多企业平时不重视应急响应，
觉得"我们不会出事的"。
结果真出事了，
手忙脚乱，
小事变大，大事变炸。
本来可以很快搞定的事情，
因为应对不当，
损失翻了好几倍。

**应急响应，
就是平时准备好，
出事了不慌，
按流程来，
把损失降到最低。**

我们就来讲讲应急响应那些事。

## 16.2 应急响应概述

### 16.2.1 什么是应急响应？

什么是应急响应
（Incident Response / Emergency Response）？
简单说就是：
**安全事件发生之后，
为了应对事件、
控制损失、
查明原因、
恢复业务、
防止再次发生，
而采取的一系列措施和行动。**

用人话说就是：
**出安全事件了，
怎么办？
怎么把事摆平，
怎么把损失降到最低，
怎么防止下次再出。**

应急响应的目标：
- **控制事态**：不让事件继续扩大
- **降低损失**：把损失降到最低
- **查明原因**：搞清楚怎么回事
- **恢复业务**：尽快恢复正常
- **总结改进**：吸取教训，防止再犯

应急响应能力，
是衡量一个企业安全水平的重要指标。
防御做得再好，
出事了不会处理，
也是白搭。

### 16.2.2 为什么需要应急响应？

为什么需要应急响应？
```
  1. 没有百分之百的安全
    再强的防御也可能被攻破
    出事是正常的，不出事是偶然的
    关键是出事了能不能扛得住

  2. 降低事件损失
    同样的事件，
    应对得好损失小，
    应对不好损失大
    应急响应做得好，
      能把损失降到最低

  3. 满足合规要求
    等保、ISO 27001、
      各个行业的监管要求
    都要求有应急预案和应急响应能力

  4. 维护企业声誉和客户信任
    出事了响应得快、处理得好，
    能挽回声誉
    响应得慢、捂着盖着，
    反而更糟
```

一句话：
**常在河边走，哪有不湿鞋。
既然湿鞋是难免的，
就得准备好擦鞋的布。**

## 16.3 应急响应的流程

应急响应不是瞎忙活，
要有流程、有步骤。
不同的组织、不同的事件，
流程可能不太一样，
但大致的思路是差不多的。

比较经典的有PDCERF模型，
六个阶段：
```
  P - Preparation        准备
  D - Detection          检测
  C - Containment        遏制
  E - Eradication        根除
  R - Recovery           恢复
  F - Follow-up          跟踪 / 复盘
```

我们一个一个讲。

### 16.3.1 准备阶段（Preparation）

第一个阶段，
**准备阶段**。
这个是在事件发生之前做的。
不是出事了才开始准备，
而是平时就要准备好。

准备什么？
```
  1. 应急预案
    制定各种场景的应急预案
    出事了按预案来，不用瞎想
    不同类型的事件有不同的预案
    比如勒索软件预案、
      数据泄露预案、
      DDoS攻击预案...

  2. 组织与人员
    明确应急响应团队的组成
    谁负责什么、什么级别找谁
    联系方式是什么
    要有7x24小时的联系人

  3. 工具与资源
    应急响应需要的工具
    比如取证工具、分析工具、
      隔离工具、恢复工具...
    提前准备好，
      出事了不用到处找

  4. 备份
    重要数据定期备份
    而且要测试备份能不能恢复
    备份是很多事件的最后一道防线

  5. 演练
    定期做应急演练
    检验预案行不行、
      团队熟不熟
    演练过和没演练过，
      出事了差别巨大

  6. 日志与监控
    平时就要有日志和监控
    出事了才有东西可查
    不然两眼一抹黑
```

**准备阶段是应急响应的基础。
平时准备得越充分，
出事了就越从容。**

### 16.3.2 检测与发现（Detection）

第二个阶段，
**检测与发现**。
就是发现安全事件。

怎么发现？
```
  1. 自动告警
    SIEM、IDS、IPS、WAF、
      EDR、杀毒软件...
    安全设备的告警
    这是最常见的发现方式

  2. 用户报告
    员工发现电脑有问题、
      数据不对了、
      收到可疑邮件...
    报告给安全团队
    很多事件是用户先发现的

  3. 主动发现
    威胁狩猎、漏洞扫描、
      安全评估、渗透测试...
    主动去找问题
    在变成事件之前发现

  4. 外部通报
    监管部门通报、
      合作方通知、
      第三方情报...
    别人告诉你出事了
```

发现事件之后，
首先要做的是**确认和定级**：
- 是不是真的安全事件？
  还是误报？
- 是什么类型的事件？
- 严重程度怎么样？
  一般、较大、重大、特别重大？
- 影响范围有多大？
- 要不要启动应急预案？
- 要不要升级？

先搞清楚状况，
再决定怎么应对。

### 16.3.3 遏制阶段（Containment）

第三个阶段，
**遏制阶段**，
也叫控制阶段。
这是出事之后最紧急的一步。

什么是遏制？
就是**把事件控制住，
不让它继续扩散、
不让损失继续扩大。**

就像着火了，
第一步不是查谁放的火，
而是先把火控制住，
别让它烧得更大。

遏制的措施根据事件类型不同而不同：
```
  比如勒索软件：
    • 立刻隔离被感染的机器
      （拔网线、断网）
    • 关闭共享、关闭不必要的服务
    • 阻止横向移动
    • 保护备份不被加密

  比如网页篡改：
    • 先把页面恢复成正常的
    • 暂时下线有漏洞的功能
    • 阻止攻击者继续篡改

  比如数据泄露：
    • 先封住泄露的口子
    • 停用被泄露的账号、密钥
    • 阻止攻击者继续偷数据

  比如DDoS攻击：
    • 开流量清洗
    • 切高防
    • 把攻击扛住
```

**遏制阶段的核心目标：
止损。**
先把损失控制住，
再考虑别的。
很多人出事了第一反应是查原因，
查半天原因没查到，
事件扩散得更大了，
这是不对的。
先止损，再查因。

遏制的时候要注意：
- 尽量保留现场，
  不要破坏证据
  （比如不要直接重装系统）
- 操作要记录，
  做了什么、什么时候做的
- 不要盲目操作，
  避免把事情搞更大

### 16.3.4 根除阶段（Eradication）

第四个阶段，
**根除阶段**。
把威胁彻底清除掉。

遏制是控制住，
根除是彻底消灭。
比如：
- 病毒木马彻底清除
- 漏洞补上
- 后门删掉
- 被篡改的文件修复
- 被盗的账号密码重置
- 等等...

根除阶段要做什么？
```
  1. 查明原因
    搞清楚攻击是怎么进来的
    漏洞在哪里？
    是钓鱼邮件？还是漏洞？
    还是内部人员？

  2. 清除威胁
    把所有的恶意代码、
      后门、木马都清除掉
    所有被攻陷的系统都要清理干净
    不能有遗漏，
      不然春风吹又生

  3. 修补漏洞
    把被利用的漏洞补上
    相关的漏洞也要一起修
    防止再从同一个地方进来

  4. 加强防护
    根据攻击的手法，
      加强对应的防护措施
    比如这次是钓鱼进来的，
      那就加强邮件安全和安全意识
    这次是利用某个漏洞，
      那就加强补丁管理
```

根除要彻底，
不能留尾巴。
不然清理不干净，
过几天又复发了，
那就麻烦了。

### 16.3.5 恢复阶段（Recovery）

第五个阶段，
**恢复阶段**。
把业务和系统恢复正常。

恢复不是一下子全开，
一般是**逐步恢复**：
```
  1. 先验证系统是干净的
    确认威胁已经清除了
    漏洞已经补上了
    确认安全了再恢复

  2. 恢复数据
    用备份恢复数据
    （当然备份要是干净的）
    要验证数据的完整性

  3. 逐步恢复业务
    先恢复最重要的业务
    再恢复次要的
    一台一台、一个系统一个系统地恢复
    恢复一台验证一台
    不要一下全开，
      万一还有问题就麻烦了

  4. 加强监控
    恢复之后加强监控
    密切观察有没有异常
    防止事件复发
```

恢复阶段最关键的是：
**确保安全了再恢复，
而且是逐步恢复。**
急急忙忙恢复，
结果系统里还有后门，
刚恢复又被攻破了，
那就尴尬了。

### 16.3.6 跟踪与复盘阶段（Follow-up）

第六个阶段，
**跟踪与复盘**。
事件处理完了，
不能就这么算了，
要总结经验教训，
防止下次再出。

复盘要做什么？
```
  1. 事件调查与溯源
    彻底搞清楚：
      事件是怎么发生的？
      攻击链是什么样的？
      攻击者是谁？（如果能查到的话）
      损失有多大？
      影响范围有多大？
    形成完整的调查报告

  2. 总结经验教训
    哪里做得好？
    哪里做得不好？
    为什么会发生这个事件？
    哪个环节出了问题？
    是技术问题还是管理问题？

  3. 制定改进措施
    根据发现的问题，
    制定改进计划
    该补的漏洞补上，
    该加的防护加上，
    该改的流程改了，
    该培训的培训
    一项一项落实

  4. 更新预案和流程
    根据这次事件的经验，
      更新应急预案
    把这次的教训加进去
    下次遇到类似事件就能应对得更好

  5. 跟进改进措施的落地
    改进措施不能只写在报告里
    要真的落地执行
    不然下次还是同样的问题
```

**复盘是应急响应最重要的环节之一。
吃一堑，长一智。
同样的错误不能犯第二次。**

很多企业出了事，
忙活一阵恢复了就完事了，
不总结不改进，
过段时间同样的事情又发生了，
这就很不应该了。

## 16.4 常见安全事件的处置

应急响应流程是通用的，
但不同类型的事件，
具体的处置方法不一样。
我们讲几种常见的。

### 16.4.1 勒索软件事件处置

勒索软件现在最常见，
也最让人头疼。

大致的处置流程：
```
  1. 遏制（第一时间！）
    • 立刻隔离被感染的机器（断网）
    • 不要让感染者在内网里继续传播
    • 隔离共享文件夹
    • 保护备份系统，别让备份也被加密了
    • 禁用被攻陷的账号

  2. 评估影响
    • 哪些机器被感染了？
    • 哪些数据被加密了？
    • 业务影响有多大？
    • 有没有备份？备份能不能用？

  3. 找原因
    • 勒索软件是怎么进来的？
      钓鱼邮件？漏洞？弱口令？
    • 是哪个家族的勒索软件？
      有没有解密工具？

  4. 决策
    • 付不付赎金？
      （一般不建议付，
        付了也不一定能解密，
        还会助长攻击者）
    • 是恢复备份还是想别的办法？

  5. 根除与恢复
    • 清除所有被感染机器上的恶意代码
    • 修补被利用的漏洞
    • 用干净的备份恢复数据
    • 逐步恢复业务
    • 加强监控，防止复发

  6. 复盘改进
    • 为什么会中招？
    • 哪里没防住？
    • 怎么加强？
    • 更新预案
```

**对付勒索软件，备份是王道。**
有干净的离线备份，
什么都不怕。

### 16.4.2 网页篡改 / 被黑事件处置

网站被黑了、
页面被篡改了，
怎么办？

```
  1. 遏制
    • 先把页面恢复成正常的
      （用备份或者干净的版本）
    • 暂时下线有问题的功能
    • 如果有webshell，先把webshell删掉
      （但要先留证据，备份一下）

  2. 找原因
    • 怎么进来的？
      SQL注入？文件上传？
      弱口令？社工？
    • 攻击者改了什么？
      除了改页面，
        有没有偷数据、留后门？
    • 拿到了什么权限？
      只是web权限还是服务器权限？

  3. 根除
    • 清除所有后门和webshell
    • 修补漏洞
    • 重置相关的账号密码
    • 检查有没有其他地方被改了

  4. 恢复
    • 确认安全了再恢复业务
    • 加强监控

  5. 复盘
    • 为什么会被黑？
    • 怎么加强？
```

### 16.4.3 数据泄露事件处置

数据泄露是比较严重的事件，
处理起来也更复杂。

```
  1. 遏制
    • 立刻封住泄露的口子
    • 阻止数据继续泄露
    • 停用被盗的账号、密钥、API Key
    • 切断攻击者的访问

  2. 评估影响
    • 哪些数据泄露了？
      多少条？什么类型？
      有没有敏感信息、个人信息？
    • 影响多少人、多少客户？
    • 泄露的原因是什么？

  3. 调查与溯源
    • 攻击者是谁？
    • 攻击时间线是什么样的？
    • 怎么进来的？偷了什么？
    • 还有没有留后门？

  4. 处置
    • 修补漏洞、清除后门
    • 重置所有相关凭证
    • 通知受影响的用户和相关方
      （法律要求的话必须通知）
    • 向监管部门报告
      （如果达到报告标准）

  5. 恢复
    • 修复系统，恢复正常
    • 加强数据安全防护

  6. 复盘与改进
    • 为什么会发生数据泄露？
    • 哪个环节出了问题？
    • 怎么加强数据安全？
```

数据泄露事件往往涉及法律和合规问题，
比如《个人信息保护法》规定，
发生个人信息泄露要通知用户、
要向监管部门报告。
处理的时候要法务、公关一起参与，
不能光靠技术部门。

### 16.4.4 DDoS 攻击事件处置

DDoS攻击也很常见。

```
  1. 确认
    • 是不是真的DDoS攻击？
      还是业务量突增？
    • 是什么类型的DDoS？
      流量型？CC攻击？

  2. 缓解
    • 开流量清洗
    • 切高防IP/高防CDN
    • 封禁攻击IP
    • 增加带宽
    • 下线非必要服务
    • 想尽办法把攻击扛住

  3. 溯源
    • 攻击来源是哪里？
    • （溯源比较难，一般只能大概定位）

  4. 恢复
    • 攻击过去了，确认正常了
    • 逐步切回正常流量

  5. 复盘
    • 为什么被DDoS？
    • 防护够不够？
    • 要不要加强抗D能力？
```

DDoS攻击主要是扛，
溯源比较难。
平时就要做好抗D的准备，
不能等攻击来了才想办法。

## 16.5 事件溯源与取证

应急响应中很重要的一块是
**溯源与取证**。
搞清楚攻击是怎么发生的、
谁干的、干了什么，
不仅是为了复盘改进，
有时候还要作为证据，
走法律程序。

### 16.5.1 什么是电子取证？

电子取证是什么？
简单说就是：
**用符合法律规范的方式，
收集、保存、分析电子证据，
还原事件的真相。**

如果事件要走法律程序、
要报警抓人，
那取证就非常重要。
证据不符合规范，
法庭不认，
就没用了。

取证的基本原则：
```
  1. 保护现场
    尽量不破坏原始证据
    不要随便动被攻破的机器
    不要直接在原硬盘上操作

  2. 完整镜像
    取证的时候先做完整的磁盘镜像
    分析在镜像上分析，
      不动原始证据
    保证原始证据的完整性

  3. 哈希校验
    镜像前后都要算哈希
    证明证据没有被篡改

  4. 全程记录
    谁、什么时候、做了什么操作
    都要有记录
    证据的流转链条要完整
    （ Chain of Custody，保管链）

  5. 符合规范
    操作要符合法律要求和技术规范
    不然证据无效
```

如果是重大事件，
可能需要找专业的取证机构来做，
他们更专业，
也有资质。

### 16.5.2 溯源分析

溯源是干嘛的？
就是**还原攻击的整个过程**：
- 攻击者怎么进来的？（入口点）
- 进来之后干了什么？
- 横向移动到了哪些机器？
- 偷了什么数据？
- 留了什么后门？
- 攻击时间线是什么样的？

怎么溯源？
靠日志。
各种日志：
- 防火墙日志
- IDS/IPS日志
- Web服务器日志
- 系统日志、安全日志
- 数据库日志
- 终端EDR日志
- 网络流量日志
- 等等...

把各种日志拼起来，
还原出完整的攻击链。
这就是为什么平时要做好日志收集，
出事了全靠它。

## 16.6 应急预案与演练

最后讲一下应急预案和演练。

### 16.6.1 应急预案

什么是应急预案？
简单说就是：
**提前写好的，
出事了该怎么办的操作手册。**

预案里一般有什么？
```
  • 应急响应组织架构
    谁负责、各部门职责
  • 事件分级标准
    什么级别的事件对应什么响应
  • 各种事件的处置流程
    一步一步写清楚
  • 联系人列表
    各相关人员的联系方式
    7x24小时能找到人
  • 外部资源联系方式
    厂商、服务商、警方、
      应急响应团队...
  • 上报流程
    什么情况要上报、报给谁
  • 各种操作指南
  • 等等...
```

预案不是写在那就完事了，
要定期更新，
根据实际情况调整。

### 16.6.2 应急演练

光有预案还不行，
还要**演练**。
不演练的预案等于废纸。

演练有什么用？
```
  • 检验预案行不行
    写的时候觉得没问题，
      一演练就发现好多漏洞
  • 让团队熟悉流程
    真出事了不慌，
      知道自己该干嘛
  • 发现不足，持续改进
    哪里有问题改哪里
  • 提升整体应急响应能力
```

演练的方式：
```
  • 桌面演练
    大家坐在一起，
      假设发生了某个事件，
      讨论该怎么办
    成本低，容易组织

  • 模拟演练
    模拟一个真实的攻击场景，
      让团队真刀真枪地干
    更接近真实，效果更好
    但成本高，组织复杂

  • 红蓝对抗
    红队攻，蓝队防
    最真实的演练
    效果最好，但成本最高
```

建议企业至少每年做一两次应急演练，
重要的场景多练几次。
练过和没练过，
出事的时候差别巨大。

## 16.7 本章总结

这一章我们讲了应急响应与事件处置。
总结一下：

**应急响应概述：**
- 安全事件发生后的应对措施
- 目标：控制事态、降低损失、
  查明原因、恢复业务、总结改进
- 没有百分之百的安全，应急响应很重要

**PDCERF 六阶段模型：**
1. **准备（Preparation）**：平时就准备好，预案、人员、工具、备份、演练
2. **检测（Detection）**：发现事件，确认定级
3. **遏制（Containment）**：先止损，控制事态，不让继续扩散
4. **根除（Eradication）**：彻底清除威胁，修补漏洞
5. **恢复（Recovery）**：逐步恢复业务，确保安全
6. **复盘（Follow-up）**：总结经验教训，改进，防止再犯

**常见事件处置：**
- 勒索软件：先隔离，保护备份，找原因，恢复，复盘
- 网页篡改：先恢复页面，找入口，清后门，补漏洞
- 数据泄露：封口子，评估影响，调查，通知，合规报告
- DDoS攻击：确认类型，扛攻击，溯源，恢复

**溯源与取证：**
- 电子取证：符合法律规范，保护现场，完整镜像，哈希校验，全程记录
- 溯源分析：靠日志还原攻击链，平时要做好日志收集

**应急预案与演练：**
- 预案：提前写好出事怎么办
- 演练：桌面演练、模拟演练、红蓝对抗
- 练过和没练过，差别巨大

最后送你一句话：

> **平时多流汗，
> 战时少流血。
> 应急响应也是一样，
> 准备得越充分，
> 出事了就越从容。**
