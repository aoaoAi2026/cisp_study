## 5.1 先讲一个故事

### 5.1.1 "到底什么级别的数据，能给什么级别的人看？"

我有个朋友，
在一家公司做安全负责人。
刚上任的时候，
公司的权限管理一塌糊涂——
- 开发能直接看生产数据库
- 运营能改财务数据
- 实习生能访问核心代码
- 谁都能随便看用户信息

老板让他整改，
说"你把权限收一收，
该看的看，不该看的别看。"

他就问老板：
"那什么人能看什么数据？
标准是什么？"

老板想了半天，
说：
"嗯...你看着办吧，
合理就行。"

我朋友当时就懵了。
什么叫"合理就行"？
每个人对"合理"的理解不一样啊。
开发觉得我看生产数据是为了排错，合理；
财务觉得我看所有数据是工作需要，合理；
老板觉得除了我自己，别人都不该看，合理。

没有一个统一的模型和标准，
安全工作就没法做。
公说公有理，婆说婆有理，
最后什么也推不动。

后来他参照了一些安全模型和标准，
给公司做了数据分类分级，
制定了访问控制策略，
什么级别的人能访问什么级别的数据，
清清楚楚写下来，
老板审批通过，
然后照着执行。
虽然推进过程中也有阻力，
但至少有标准可依，
不用再"看着办"了。

---

你看，
安全模型和标准，
就是干这个的——
**给你一个框架、一个参考、一个依据，
让你做安全工作的时候，
不是拍脑袋，
而是有标准可依。**

这一章，
我们就来讲讲这些模型和标准。
不用死记硬背，

## 5.2 常见安全模型

先讲几个经典的**安全模型**。
什么是安全模型？
简单说就是：
**用数学或者形式化的方法，
描述安全策略的模型。**
说人话就是：
**把安全的规则，
用抽象的模型表达出来。**

安全模型有很多，
我给你讲几个最经典的，
也是考试、面试常考的。

### 5.2.1 BLP 模型（Bell-LaPadula模型）

第一个，
**BLP模型**，
全名叫Bell-LaPadula模型，
以两个发明者的名字命名的。
这是最经典的安全模型之一，
主要关注**保密性**。

BLP模型是干嘛的？
简单说就是：
**规定了不同安全级别的主体（用户/进程）
和不同安全级别的客体（文件/数据）之间，
能不能读、能不能写。**

BLP模型有两个核心规则：
```
  1. 简单安全规则（不上读）
     主体不能读安全级别比自己高的客体
     简单说：低级别不能读高级别的
     （不能往上读）

  2. *特性（星特性，不下写）
     主体不能写安全级别比自己低的客体
     简单说：高级别不能往低级别的写
     （不能往下写）
```

用人话说就是：
**只能往上写，往下读。**
- 你的级别是"机密"，
  那你可以读"机密"和"秘密"的文件（往下读），
  但不能读"绝密"的（不能往上读）。
- 你可以写"绝密"和"机密"的文件（往上写），
  但不能写"秘密"的（不能往下写）。

为什么要这样？
为了保证**保密性**——
高密级的信息，
不会流到低密级的地方去。
高权限的人，
不能把高密级的文件，
写到低密级的地方泄露出去。

BLP模型是第一个数学上严格证明的安全模型，
也是很多军事、政府安全系统的理论基础。
它的核心目标是**保密性**。

### 5.2.2 Biba 模型

第二个，
**Biba模型**。
BLP关注的是保密性，
Biba模型关注的是**完整性**。

Biba模型也有两个核心规则，
跟BLP刚好反过来：
```
  1. 简单完整性规则（不下读）
     主体不能读完整性级别比自己低的客体
     （不能往下读）

  2. *完整性特性（不上写）
     主体不能写完整性级别比自己高的客体
     （不能往上写）
```

用人话说就是：
**只能往上读，往下写。**
- 高完整性的人，
  不能读低完整性的东西（怕被污染）
- 低完整性的人，
  不能往高完整性的地方写（怕搞破坏）

为什么要这样？
为了保证**完整性**——
高完整性的数据，
不会被低完整性的东西污染。
低级别的人，
不能改高级别的数据。
高级别的人，
也别去读低级别的脏数据，
免得被带坏了。

你看，
BLP管保密性（别泄露），
Biba管完整性（别篡改），
刚好一对。

### 5.2.3 Clark-Wilson 模型

第三个，
**Clark-Wilson模型**。
这个模型也是关注完整性的，
但跟Biba不一样，
更贴近商业应用。

Clark-Wilson模型的核心思想是：
**通过完善的事务处理和职责分离，
来保证数据的完整性。**

它有几个核心概念：
```
  • 约束数据项（CDI）：
    需要保证完整性的数据

  • 转换过程（TP）：
    操作CDI的程序，
    必须是经过验证的、可信的
    （只能通过TP来修改CDI，不能直接改）

  • 完整性验证过程（IVP）：
    验证CDI的完整性是不是好的
```

还有几个重要的原则：
- **良形事务**：数据只能通过可信的程序来修改，
  程序本身是经过验证的，保证操作是合规的
- **职责分离**：一项操作要由多个人/角色共同完成，
  互相监督，防止一个人作弊

这个模型更适合商业场景，
比如银行、财务系统。
比如转账，
不能让一个人想转多少转多少，
得有审批、有复核、有记录，
互相制约。
这就是Clark-Wilson模型的思路。

### 5.2.4 其他安全模型

还有一些其他的安全模型，
简单了解一下就行：

```
  • 中国墙模型（Chinese Wall）
    主要用于利益冲突的场景
    比如咨询公司，
      给A公司做了项目，
      就不能再给A的竞争对手做
      防止利益冲突
    同一个人不能同时访问有利益冲突的两个客户的数据
    中间像隔了一堵墙一样，所以叫中国墙

  • 最小权限模型
    每个主体只拥有完成其工作所必需的最小权限
    多一点都不给
    这个我们后面还会讲

  • 信息流模型
    关注信息的流动
    防止信息从高安全级流向低安全级
    BLP其实就是一种信息流模型
```

不用全记住，
知道有这些模型，
各自大概是干嘛的，
就够了。

## 5.3 信息安全管理体系（ISMS）

讲完安全模型，
我们来讲讲**信息安全管理体系（ISMS）**。

### 5.3.1 什么是ISMS？

ISMS是什么？
全称是Information Security Management System，
信息安全管理体系。

简单说就是：
**一整套管理信息安全的体系，
包括方针、策略、流程、组织、
技术、人员、资源等等。**

用人话说就是：
**企业做信息安全，
不是买几个防火墙、装几个杀毒软件就完事了。
而是要建立一套完整的体系，
从制度、流程、人员、技术等各个方面，
系统化地管理安全。**

ISMS是一个系统性的方法，
不是单点的防护。
它强调的是**持续改进**，
PDCA循环（计划-执行-检查-行动），
不断优化，越来越好。

### 5.3.2 为什么需要ISMS？

为什么要搞ISMS？
买一堆设备不行吗？
还真不行。

```
  1. 安全是系统工程，不是单点问题
     只靠技术设备防不住
     人的问题、流程的问题、管理的问题
     都要管起来

  2. 合规要求
     很多行业有合规要求
     比如等保、比如ISO 27001
     建立ISMS是合规的基础

  3. 持续改进
     安全不是一劳永逸的
     需要持续运营、持续改进
     ISMS提供了一套方法论

  4. 降低风险
     系统化地识别风险、管理风险
     把风险降到可接受的水平
```

企业做大了，
安全不能靠几个牛人撑着，
得靠体系。
体系建好了，
就算人换了，
安全水平也不会掉太多。

## 5.4 ISO 27001 标准

说到ISMS，
就不得不提 **ISO 27001**。
这是目前国际上最主流、
最权威的信息安全管理体系标准。

### 5.4.1 什么是ISO 27001？

ISO 27001是什么？
它是国际标准化组织（ISO）
发布的一个信息安全管理体系标准。
全称是：
**ISO/IEC 27001 —— 信息安全管理体系要求。**

简单说就是：
**国际上公认的，
做信息安全管理体系的标准模板。**
你照着这个标准来建你的安全体系，
建好了通过认证，
就说明你的信息安全管理达到了国际水平。

ISO 27001最新版本是2022版的，
不过很多企业还是用的2013版。

### 5.4.2 ISO 27001的核心内容

ISO 27001大致包含什么内容？

```
  • 4 组织环境
    理解组织及其环境、理解相关方的需求
    确定ISMS的范围

  • 5 领导作用
    领导的重视和承诺
    安全方针
    角色职责和权限

  • 6 策划
    应对风险和机遇的措施
    信息安全目标

  • 7 支持
    资源、能力、意识、沟通、文件化信息

  • 8 运行
    运行的策划和控制
    信息安全风险评估
    信息安全风险处置
    附录A的114项控制措施

  • 9 绩效评价
    监视测量分析评价
    内部审核
    管理评审

  • 10 改进
    不符合和纠正措施
    持续改进
```

整体是基于PDCA循环的：
计划（Plan）→ 执行（Do）→ 检查（Check）→ 行动（Act）
持续改进，
不断优化。

ISO 27001的附录A里，
有114条控制措施（2022版），
覆盖了安全的方方面面：
- 安全策略
- 信息安全组织
- 人力资源安全
- 资产管理
- 访问控制
- 密码学
- 物理和环境安全
- 操作安全
- 通信安全
- 系统获取、开发和维护
- 供应商关系
- 信息安全事件管理
- 业务连续性
- 合规性

非常全面。
你照着这114条一条一条对照着做，
安全体系就建起来了。

### 5.4.3 ISO 27001 认证

很多企业会做ISO 27001认证。
什么意思？
就是请第三方认证机构来审核，
审核通过了，
给你发个证书，
证明你的ISMS符合ISO 27001标准。

有什么用？
```
  • 证明企业的安全管理水平
  • 满足客户的合规要求
  • 提升企业形象和竞争力
  • 系统性地提升企业安全水平
```

现在很多甲方招标，
都要求供应商有ISO 27001认证。
这个证书还是挺有用的。

## 5.5 等级保护2.0

ISO 27001是国际标准，
那我们国内呢？
我们国内有自己的标准——
**网络安全等级保护制度**，
简称**等保**。

### 5.5.1 什么是等级保护？

等级保护是什么？
简单说就是：
**国家对网络和信息系统，
按重要程度分成不同的等级，
不同等级的系统，
有不同的安全保护要求，
分级进行监管和保护。**

用人话说就是：
**越重要的系统，
安全要求越高，
管得越严。**

等保是我们国家网络安全的基本制度，
是法律要求的。
《网络安全法》里明确规定了，
国家实行网络安全等级保护制度。
不是你想做就做，不想做就不做的——
这是法律义务。

### 5.5.2 等级保护的五个等级

等级保护分几个等级？
五个等级，从低到高：

```
  第一级（自主保护级）
    一般的信息系统，
    受到破坏后对公民、法人有影响，
    但不损害国家安全、社会秩序。
    由运营者自主保护。

  第二级（指导保护级）
    受到破坏后会对社会秩序和公共利益
    造成一定损害，
    但不损害国家安全。
    由相关部门指导保护。

  第三级（监督保护级）
    受到破坏后会对社会秩序和公共利益
    造成严重损害，
    或者对国家安全造成损害。
    由公安机关监督保护。
    （最常见的就是三级，
      很多企业的业务系统都是三级）

  第四级（强制保护级）
    受到破坏后会对社会秩序和公共利益
    造成特别严重损害，
    或者对国家安全造成严重损害。
    由公安机关强制监督保护。
    （比如电力、金融等重要行业的核心系统）

  第五级（专控保护级）
    受到破坏后会对国家安全
    造成特别严重损害。
    由国家指定的部门专门监督保护。
    （最高级，一般涉及国家核心机密）
```

一般企业的业务系统，
做等保二级或者三级的比较多。
金融、电力、交通这些重要行业的，
可能有四级。
五级一般企业接触不到。

### 5.5.3 等级保护2.0

等保2.0又是什么？
简单说就是等级保护的升级版。
以前的等保是1.0时代，
主要是传统的信息系统。
现在云、大数据、物联网、工控都出来了，
老标准跟不上了，
所以升级到了2.0。

等保2.0有什么变化？
```
  • 覆盖范围扩大了
    不光是传统的信息系统
    还包括云计算、移动互联、物联网、
      工业控制系统...
    全覆盖

  • "一个中心，三重防护"
    安全管理中心 +
    安全计算环境、安全区域边界、安全通信网络

  • 更强调主动防御、动态感知、持续监测
    以前是静态防护为主
    现在强调持续监测、动态调整

  • 强化了可信验证
  • 强化了个人信息保护
  • 等等...
```

等保2.0的标准体系包括：
- 《网络安全等级保护基本要求》
- 《网络安全等级保护测评要求》
- 《网络安全等级保护安全设计技术要求》
- 还有各个扩展标准：云计算、移动互联、物联网、工控...

### 5.5.4 等保工作流程

做等保大致是个什么流程？
```
  1. 定级
     先确定你的系统是第几级
     运营者自己初步定级，然后专家评审，
     最后公安机关备案审核

  2. 备案
     到公安机关备案
     拿到备案证明

  3. 建设整改
     按照等保的要求，
     建设安全技术体系和管理体系
     缺什么补什么

  4. 等级测评
     找有资质的测评机构，
     对你的系统做测评
     看看符合不符合等保要求
     测评通过了发测评报告

  5. 监督检查
     公安机关会定期监督检查
     一般三级系统每年测一次
     二级每两年测一次
```

简单说就是：
**定级 → 备案 → 建设整改 → 测评 → 持续运营**

等保是国内企业安全的基础，
做安全的一定要了解。

## 5.6 网络安全法律法规

讲完标准，
再讲讲法律法规。
这个也很重要。
做安全的，
不能连相关的法律都不知道。

我们国家现在主要有几部核心的法律：

### 5.6.1 《网络安全法》

第一部，
**《中华人民共和国网络安全法》**，
简称《网安法》。
2017年6月1日起施行。

这是我国网络安全领域的**基础性、综合性法律**。
是网络安全领域的"母法"。

主要内容包括：
```
  • 网络安全等级保护制度
  • 关键信息基础设施保护
  • 网络运营者的安全义务
  • 个人信息保护
  • 网络安全监测预警与应急处置
  • 法律责任
  • 等等
```

它确立了网络安全的基本制度，
比如等级保护、关键信息基础设施保护、
数据安全、个人信息保护、
网络安全审查...
都是从这里来的。

### 5.6.2 《数据安全法》

第二部，
**《中华人民共和国数据安全法》**，
简称《数安法》。
2021年9月1日起施行。

顾名思义，
这部法律是管**数据安全**的。
现在数据越来越重要，
成了重要的生产要素，
所以专门立法来管。

主要内容：
```
  • 建立数据分类分级保护制度
  • 重要数据和核心数据的特殊保护
  • 数据安全风险评估、监测预警
  • 数据交易管理
  • 跨境数据流动管理
  • 政务数据安全
  • 法律责任
  • 等等
```

做数据安全的，
这部法律是根本依据。

### 5.6.3 《个人信息保护法》

第三部，
**《中华人民共和国个人信息保护法》**，
简称《个保法》。
2021年11月1日起施行。

这部法律是管**个人信息保护**的。
相当于中国版的GDPR。

主要内容：
```
  • 个人信息处理的基本原则
    合法、正当、必要、诚信
    告知同意原则
    最小必要原则
    公开透明原则
    ......

  • 个人在个人信息处理中的权利
    知情权、决定权、查阅复制权、
    更正补充权、删除权、可携带权...

  • 个人信息处理者的义务
    安全保护义务、告知义务...

  • 敏感个人信息的特殊保护
    生物识别、宗教信仰、特定身份、
    医疗健康、金融账户、行踪轨迹...
    处理敏感个人信息需要单独同意

  • 个人信息跨境提供的规则

  • 法律责任
    最高可处上一年度营业额百分之五以下罚款
    很严厉
```

现在个人信息保护越来越受重视，
企业做用户数据相关的业务，
这部法律一定要了解。

### 5.6.4 其他相关法律法规

除了这三部核心的，
还有很多相关的法律法规：

```
  • 《关键信息基础设施安全保护条例》
    专门管关键信息基础设施的

  • 《网络安全审查办法》
    网络安全审查制度
    影响或者可能影响国家安全的
      网络平台运营者赴国外上市，
      要申报网络安全审查

  • 《生成式人工智能服务管理暂行办法》
    管AI的，最新的

  • 《网络数据安全管理条例》
    还在制定中，不过快了

  • 各个行业的规定
    金融的、医疗的、教育的...
    各行业有各自的监管要求
```

法律法规这东西，
不用死记硬背，
但要有个印象，
知道有这么回事，
知道大概管什么的。
碰到具体问题了再去查。

## 5.7 NIST 网络安全框架

最后再介绍一个国际上很有名的框架——
**NIST网络安全框架（NIST CSF）**。

### 5.7.1 什么是NIST CSF？

NIST是什么？
美国国家标准与技术研究院（National Institute of Standards and Technology）。
它发布的网络安全框架（Cybersecurity Framework），
是国际上很流行的一个安全框架。

NIST CSF是干嘛的？
**帮助企业管理和降低网络安全风险的一套框架。**
它把网络安全活动分成了五大核心功能：

```
  1. 识别（Identify）
     搞清楚你有什么资产、有什么风险
     资产识别、风险评估、治理、业务环境...

  2. 保护（Protect）
     采取防护措施，
     确保关键基础设施和业务的安全
     访问控制、意识培训、数据安全、
     维护、防护技术...

  3. 检测（Detect）
     及时发现安全事件
     异常检测、持续监测、检测流程...

  4. 响应（Respond）
     发现事件之后怎么响应
     响应计划、沟通、分析、缓解、改进...

  5. 恢复（Recover）
     事件之后怎么恢复
     恢复计划、改进、沟通...
```

这五大功能，
覆盖了安全的全生命周期：
**识别 → 保护 → 检测 → 响应 → 恢复**

是不是很有道理？
先搞清楚有什么，
然后做防护，
然后要能检测到攻击，
检测到了要能响应，
出了事要能恢复。
形成一个闭环。

NIST CSF很受欢迎，
很多企业都用这个框架来建设自己的安全体系。
因为它逻辑清晰，
也比较通用，
适合各种规模的企业。

## 5.8 安全成熟度模型

最后简单提一下**安全成熟度模型**。

什么是安全成熟度？
就是**衡量一个企业安全水平的模型**。
从初级到高级，分几个等级，
看看你在哪个级别，
下一步该怎么走。

比如常见的成熟度分级：
```
  1级：初始级（混乱、被动）
     没有什么安全措施，
     出了事才救火
     完全靠人

  2级：可重复级（基础）
     有一些基础的安全措施
     但主要是单点的、零散的
     靠制度和流程

  3级：已定义级（体系化）
     建立了完整的安全体系
     有制度、有流程、有技术
     系统化管理

  4级：可管理级（量化）
     安全可以量化度量
     有数据、有指标
     精细化运营

  5级：优化级（持续改进）
     持续优化，自适应
     不断进步
     安全和业务融合
```

不同的模型分级不太一样，
但大致的思路是一样的——
从混乱到规范，
从被动到主动，
从单点到体系，
从经验到数据。

企业可以对照成熟度模型，
看看自己在哪个级别，
然后规划下一步怎么建设。

## 5.9 本章总结

这一章我们讲了安全模型和标准规范，
内容比较杂，
但都是了解性的，
有个印象就行。
总结一下：

**经典安全模型：**
- **BLP模型** —— 关注保密性，不上读，不下写
- **Biba模型** —— 关注完整性，不下读，不上写
- **Clark-Wilson模型** —— 商业完整性，良形事务、职责分离
- 中国墙模型、最小权限、信息流模型...

**ISMS 信息安全管理体系：**
- 系统化管理信息安全的一整套体系
- 基于PDCA循环，持续改进
- 不是买设备，而是建体系

**ISO 27001：**
- 国际主流的信息安全管理体系标准
- 114项控制措施，覆盖全面
- 可以做认证，提升企业安全水平和竞争力

**等级保护2.0：**
- 我国网络安全基本制度，法律要求
- 分五级：自主、指导、监督、强制、专控
- 2.0时代：全覆盖，一个中心三重防护
- 流程：定级 → 备案 → 建设整改 → 测评 → 监督检查

**国内网络安全法律法规：**
- 《网络安全法》—— 基础综合性法律
- 《数据安全法》—— 数据安全
- 《个人信息保护法》—— 个人信息保护
- 关键信息基础设施保护、网络安全审查...

**NIST 网络安全框架：**
- 五大核心功能：识别、保护、检测、响应、恢复
- 全生命周期闭环
- 国际通用，逻辑清晰

**安全成熟度模型：**
- 衡量安全水平
- 从初始级到优化级，逐步提升

最后送你一句话：

> **技术是点，
> 体系是面。
> 点动成线，
> 线动成面。
> 真正的安全，
> 是体系化的安全。**
