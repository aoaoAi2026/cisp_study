## 4.1 先讲一个故事

### 4.1.1 "凯撒密码"和"月光宝盒"

先给你讲个古老的故事。

两千多年前，
古罗马有个大将军叫凯撒，
就是那个很有名的凯撒大帝。
他带兵打仗的时候，
要给前线的将军送信，
但又怕信被敌人截获，
泄露军情。

怎么办呢？
他想了个办法——
把信里的每个字母，
都往后移3位。
比如A变成D，B变成E，
C变成F...
这样就算信被敌人截获了，
敌人也看不懂，
不知道写的是什么。
只有知道这个"移3位"规则的自己人，
才能把信还原回来。

这就是著名的**凯撒密码**，
也是最早的加密方法之一。

你看，
密码学的思路其实很古老了——
**把明文变成别人看不懂的密文，
只有有钥匙的人才能还原。**

当然，
凯撒密码太简单了，
随便就能破解。
现在的密码学可比这个复杂多了，
但核心思想是一样的。

## 4.2 密码学概述

### 4.2.1 什么是密码学？

什么是密码学（Cryptography）？
简单说就是：
**研究怎么把信息藏起来，
以及怎么把藏起来的信息还原回来的学问。**

专业一点说：
密码学是研究编制密码和破译密码的技术科学。
编制密码（加密），
保护信息不被未授权的人知道；
破译密码（密码分析），
就是破解加密，获取信息。
两个方面互相对立，
又互相促进，
道高一尺魔高一丈。

### 4.2.2 基本概念

先搞清楚几个基本概念，
不然容易晕。

```
  明文（Plaintext）：
    原始的、能看懂的信息
    就是你想保护的内容

  密文（Ciphertext）：
    加密之后的、看不懂的信息
    别人拿到也没用

  加密（Encryption）：
    把明文变成密文的过程

  解密（Decryption）：
    把密文还原成明文的过程

  密钥（Key）：
    加密和解密用的"钥匙"
    有了密钥才能加密和解密
    密钥是密码学的核心
```

一般的加密过程就是：
```
明文 + 密钥 → 加密算法 → 密文

密文 + 密钥 → 解密算法 → 明文
```

密码学里，
算法一般是公开的，
大家都知道算法是什么。
保密的是**密钥**。
只要密钥够安全，
就算别人知道算法，
也破解不了。

这是现代密码学的一个基本原则——
**柯克霍夫原则（Kerckhoffs's principle）**：
系统的安全性不依赖于算法的保密，
只依赖于密钥的保密。

用人话说就是：
**算法就算公开了也没事，
只要密钥是秘密的，就安全。**

### 4.2.3 密码学的分类

密码学怎么分类？
大的方向可以分成两类：

```
  1. 对称加密（Symmetric Cryptography）
     加密和解密用同一个密钥
     就像一把钥匙锁门，同一把钥匙开门
     比如：AES、DES、3DES

  2. 非对称加密（Asymmetric Cryptography）
     也叫公钥加密
     有一对密钥：公钥和私钥
     公钥加密，私钥解密
     或者私钥签名，公钥验证
     比如：RSA、ECC

还有一类很常用：
  3. 哈希函数（Hash Function）
     严格说不算加密，因为不可逆
     把任意长度的数据，
     变成固定长度的哈希值
     而且不可逆，推不回原文
     比如：MD5、SHA-1、SHA-256
```

这三类是最核心的，
我们一个一个讲。

## 4.3 对称加密

先讲**对称加密**，
这个最好理解。

### 4.3.1 什么是对称加密？

对称加密，
就是**加密和解密用的是同一个密钥**。
就像你家门锁，
锁门用这把钥匙，
开门也用这把钥匙。
对称的，所以叫对称加密。

```
明文 + 密钥 → 加密 → 密文
密文 + 密钥 → 解密 → 明文
```

对称加密的特点：
```
优点：
  • 速度快，加密解密效率高
  • 适合加密大量数据
  • 算法相对简单

缺点：
  • 密钥分发问题：
    双方怎么安全地交换密钥？
    密钥在传输过程中被截获了怎么办？
  • 密钥管理问题：
    N个人互相通信，需要N*(N-1)/2个密钥
    人多了密钥管理很麻烦
```

对称加密速度快，
所以实际应用中，
大量数据的加密，
基本都是用对称加密。
但是它有个致命的问题——
**密钥怎么安全地传给对方？**
直接传的话，
被中间人截获了，
加密就白加了。
这个问题怎么解决？
后面讲非对称加密的时候你就知道了。

### 4.3.2 常见的对称加密算法

常见的对称加密算法有哪些？

```
  • DES（Data Encryption Standard）
    数据加密标准，很老了
    密钥太短，只有56位
    现在已经不安全了，
    暴力就能破解
    基本淘汰了

  • 3DES（Triple DES）
    DES的加强版，加密三次
    比DES安全，但速度慢
    也在逐渐淘汰

  • AES（Advanced Encryption Standard）
    高级加密标准
    现在最主流、最常用的对称加密算法
    密钥长度有128位、192位、256位
    速度快，安全性高
    现在基本都用AES
    除非被量子计算机攻破，
      否则AES-256基本是无敌的

  • 其他：
    Blowfish、Twofish、RC4、SM4（国密）...
    各有各的应用场景
```

现在用的最多的就是**AES**。
可以说，
AES是对称加密的王者，
安全、高效、应用广泛。
记住它就对了。

### 4.3.3 加密模式

对称加密还有个**加密模式**的概念，
就是怎么把明文切块加密的方式。
这个稍微有点复杂，
但也很重要，
因为不同的模式安全性不一样。

常见的加密模式：
```
  • ECB（电子密码本）
    最简单的，每块独立加密
    问题：相同的明文块会得到相同的密文块
    容易被分析，不安全
    不推荐用

  • CBC（密码分组链接）
    每一块加密前，先跟前一块的密文异或一下
    这样相同的明文块，密文也不一样
    安全性比ECB好很多
    用得比较多

  • GCM（伽罗瓦/计数器模式）
    既能加密又能认证（提供完整性校验）
    流式加密，速度快
    现在越来越流行
    HTTPS里常用
    推荐用这个

  还有CFB、OFB、CTR...
  各有各的特点
```

新手不用搞太细，
知道有这么回事就行。
记住：
**ECB不安全，CBC和GCM比较常用，GCM更好。**

还有个东西叫**填充（Padding）**，
因为对称加密是分块的，
最后一块不够长度怎么办？
就要填充一下。
填充也有不同的方式，
比如PKCS5、PKCS7。
而且填充也可能有安全问题，
比如著名的Padding Oracle攻击。
这里就不展开了。

## 4.4 非对称加密

接下来是**非对称加密**，
也叫公钥加密。
这个稍微抽象一点，
但也很重要。

### 4.4.1 什么是非对称加密？

对称加密有个大问题——
密钥怎么安全交换？
非对称加密就是来解决这个问题的。

非对称加密有**一对密钥**：
- 公钥（Public Key）：可以公开给所有人
- 私钥（Private Key）：自己秘密保存，绝不能给别人

这两个密钥是配对的，
- **用公钥加密的，只能用对应的私钥解密**
- **用私钥加密的（签名），只能用对应的公钥验证**

而且，
从公钥推不出私钥，
所以公钥公开了也没事。

```
最经典的用法：
  你要给我发保密消息
    ↓
  我把我的公钥给你（公钥可以公开，谁拿都没事）
    ↓
  你用我的公钥加密消息
    ↓
  加密后的消息，只有我的私钥能解密
    ↓
  我用我的私钥解密，看到消息
    ↓
  （中间人就算截到了密文和公钥，也解密不了）
```

神奇吧？
这样就完美解决了密钥交换的问题。
不用提前约定密钥，
你把公钥公开，
谁想给你发消息，
就用公钥加密，
只有你能解密。

非对称加密的特点：
```
优点：
  • 解决了密钥分发的问题
    不用安全信道交换密钥
  • 可以用来做数字签名
    身份认证、防抵赖

缺点：
  • 速度慢，非常慢
    比对称加密慢几百上千倍
  • 不适合加密大量数据
```

所以实际应用中，
一般是**非对称加密 + 对称加密 混合使用**：
- 用非对称加密来交换对称密钥
- 然后用对称密钥加密实际的数据

这样既解决了密钥分发的问题，
又保证了加密速度。
HTTPS就是这么干的。

### 4.4.2 常见的非对称加密算法

常见的非对称加密算法有哪些？

```
  • RSA
    最有名、最常用的非对称加密算法
    名字来自三个发明者的首字母
    基于大整数分解的数学难题
    应用非常广泛
    密钥长度一般用2048位、4096位
    1024位的已经不安全了

  • ECC（椭圆曲线密码学）
    Elliptic Curve Cryptography
    基于椭圆曲线离散对数问题
    相同安全强度下，密钥比RSA短很多
    比如256位ECC ≈ 3072位RSA的安全性
    密钥短，运算快
    现在越来越流行，
      移动设备、区块链都用得很多

  • DSA / ECDSA
    数字签名算法
    专门用来做签名的
    DSA是老的，ECDSA是椭圆曲线版本的

  • Diffie-Hellman（DH）密钥交换
    严格说不是加密算法，
    是密钥交换算法
    双方在不安全的信道上，
    协商出一个共同的密钥
    中间人就算看到所有通信，
      也算不出这个密钥
    HTTPS里经常用

  还有很多其他的...
```

最经典、最常用的就是 **RSA**，
这个一定要知道。
ECC是后起之秀，
也越来越重要。

### 4.4.3 Diffie-Hellman 密钥交换

前面提到了DH密钥交换，
这个很有意思，
也很重要，
我给你简单讲讲原理。

DH算法解决的问题是：
**在不安全的信道上，
双方怎么协商出一个共同的密钥，
而且中间人就算截获了所有通信，
也算不出这个密钥？**

听起来很神奇是不是？
说的好像"两个人当着第三个人的面，
商量出一个秘密，
第三个人还不知道"一样。
但数学上确实可以做到。

我给你用颜色的例子打个比方：

```
  两个人要商量一个共同的秘密颜色。

  第一步：
    两人先选一个共同的颜色（比如黄色），
    这个颜色是公开的，谁都知道。

  第二步：
    两人各自选一个自己的秘密颜色（不告诉对方）
    A选红色，B选蓝色。

  第三步：
    两人把自己的秘密颜色和公开颜色混在一起，
    得到一个混合色，发给对方。
    A发给B：黄+红=橙色
    B发给A：黄+蓝=绿色

  第四步：
    两人把收到的混合色，
    再和自己的秘密颜色混在一起。
    A：绿色 + 红色 = ？
    B：橙色 + 蓝色 = ？

  结果：
    两人得到的最终颜色是一样的！
    （都是黄+红+蓝的混合色）

  而中间人呢？
    中间人只看到了黄色（公开色）、橙色、绿色，
    不知道A的红色和B的蓝色，
    所以混不出最终的那个颜色。
```

当然，
实际DH算法用的是数学（模幂运算），
不是颜色。
但原理就是这么个意思——
**双方各自有个私有的秘密，
交换公开的信息，
然后各自算出一个相同的密钥，
中间人拿不到。**

是不是很巧妙？
HTTPS里经常用ECDH（椭圆曲线版本的DH）
来做密钥交换。

## 4.5 哈希函数

接下来讲**哈希函数**，
也叫散列函数、杂凑函数。
这个非常常用，
但很多人搞不清楚它和加密的区别。

### 4.5.1 什么是哈希函数？

哈希函数是干嘛的？
**把任意长度的输入数据，
通过哈希算法，
变成一个固定长度的输出，
这个输出就叫哈希值（或者摘要、散列值）。**

```
任意长度的数据 → 哈希函数 → 固定长度的哈希值
```

哈希函数有几个重要的特性：

```
  1. 单向性（不可逆）
     只能从明文算哈希值，
     不能从哈希值反推出明文
     （至少理论上很难）
     这是和加密最大的区别——
       加密是可逆的，哈希是不可逆的

  2. 定长输出
     不管输入多长，
     输出长度都是固定的
     比如MD5输出128位（32位十六进制）
     SHA-256输出256位（64位十六进制）

  3. 雪崩效应
     输入哪怕只改一点点，
     输出的哈希值就会完全不一样
     比如"abc"和"abd"，
       哈希值天差地别

  4. 抗碰撞性
     理论上很难找到两个不同的输入，
     得到相同的哈希值
     （当然不是完全不可能，是很难）
```

因为有这些特性，
哈希函数的用途非常广。

### 4.5.2 哈希函数有什么用？

哈希函数能干嘛？
太多了。

```
  1. 密码存储
     网站存密码，不能存明文
     存密码的哈希值
     用户登录时，把输入的密码算哈希，
       和数据库里的比
     这样就算数据库泄露了，
       也拿不到明文密码
     （当然现在还要加盐，后面讲）

  2. 数据完整性校验
     下载文件的时候，
     网站会给个MD5或者SHA值
     你下载完算一下哈希，
       跟给的对比一下
     一样就说明文件没被篡改、没损坏

  3. 数字签名
     非对称加密签名太慢了
     一般先对数据算哈希，
       再对哈希值签名
     既保证了完整性，又高效

  4. 消息认证码（HMAC）
     后面讲

  5. 区块链
     区块链里大量用哈希
     区块哈希、Merkle树...

  还有很多其他用途...
```

哈希函数是安全里用得最多的东西之一，
一定要搞懂。

### 4.5.3 常见的哈希算法

常见的哈希算法有哪些？

```
  • MD5
    最有名的哈希算法之一
    输出128位（32位十六进制）
    但是已经不安全了！
      碰撞攻击已经很成熟了
      可以很容易构造出相同哈希的不同文件
    现在只能用来做不重要的校验，
      绝对不能用来存密码、做签名

  • SHA-1
    安全哈希算法第1版
    输出160位
    也已经不安全了！
      碰撞攻击也被攻破了
    也不推荐用了

  • SHA-2 家族
    SHA-224、SHA-256、SHA-384、SHA-512...
    现在最主流、最常用的
    SHA-256和SHA-512用得最多
    目前还是安全的
    推荐用这个

  • SHA-3
    最新一代的SHA
    采用了完全不同的设计（Keccak算法）
    更安全
    也在慢慢普及

  • SM3
    国密哈希算法
    中国自己的标准
    输出256位
    国内政府、国企常用

  还有很多其他的...
```

记住：
**MD5和SHA-1已经不安全了，
重要的场合用SHA-2或者SHA-3。**

### 4.5.4 加盐哈希 —— 密码安全存储

前面说哈希可以用来存密码，
但直接存密码的哈希值就够了吗？
不够。

为什么？
因为有**彩虹表**。
什么是彩虹表？
就是提前算好的，
一大堆常用密码和它们的哈希值的对应表。
拿到哈希之后，
去表里一查，
就知道明文是什么了。
简单的密码基本一查一个准。

怎么防？
**加盐（Salt）。**

什么是加盐？
就是在密码哈希之前，
先拼上一串随机的字符串（盐），
再算哈希。
```
哈希值 = SHA256(密码 + 盐)
```
每个用户的盐都不一样，
是随机生成的。
数据库里存：盐 + 哈希值。

这样有什么好处？
- 就算两个用户密码一样，
  因为盐不一样，
  哈希值也不一样
- 彩虹表没用了，
  因为你不知道盐是什么，
  没法提前算
- 破解难度大大增加

当然，
加盐不是绝对安全的，
如果密码太简单，
还是可以通过暴力破解跑出来。
所以现在更推荐用**慢哈希**，
比如bcrypt、scrypt、Argon2，
这些算法故意设计得很慢，
让暴力破解的成本变得非常高。

记住：
**存密码不要用普通的MD5/SHA，
要用加盐的慢哈希，
比如bcrypt、Argon2。**

---

## 4.6 消息认证码（HMAC）

接下来讲一个组合的——
**消息认证码（MAC，Message Authentication Code）**，
最常用的就是 **HMAC**。

### 4.6.1 HMAC是干嘛的？

HMAC是干嘛的？
它结合了哈希函数和密钥，
用来**同时验证消息的完整性和来源**。

用人话说就是：
**这个消息是真的，是对方发的，没被篡改过。**

怎么实现的？
```
HMAC = 哈希函数(密钥 + 消息)
```
（实际算法有点不一样，是两次哈希，
但大致是这么个意思）

发送方：
- 用一个双方约定的密钥 + 消息
- 算出一个HMAC值
- 把消息和HMAC一起发给对方

接收方：
- 收到消息和HMAC
- 用同样的密钥 + 收到的消息，再算一次HMAC
- 跟收到的HMAC对比
- 一样的话，说明消息没被篡改，而且确实是对方发的

因为密钥只有双方知道，
中间人没有密钥，
改了消息就没法算出正确的HMAC。
这样就保证了消息的完整性和来源真实性。

### 4.6.2 HMAC和普通哈希有什么区别？

普通的哈希也能校验完整性啊，
为什么还要HMAC？

因为普通的哈希只能防意外损坏，
防不了恶意篡改。
如果中间人改了消息，
同时把哈希也改成改后消息的哈希，
你就发现不了。

但HMAC不一样，
因为有密钥。
中间人没有密钥，
改了消息也算不出正确的HMAC。
所以HMAC可以防恶意篡改。

HMAC在很多地方都用，
比如JWT、API签名、
各种协议的认证...
很常用。

## 4.7 数字签名

接下来讲**数字签名**。
这个也非常重要。

### 4.7.1 什么是数字签名？

现实生活中，
我们签合同、签文件，
要手写签名或者盖公章。
签名有什么用？
- 证明是你签的（身份认证）
- 你签了就不能抵赖（防抵赖）
- 文件内容不能随便改（完整性）

数字签名也是一样的作用，
只不过是在数字世界里。

数字签名是怎么做的？
用非对称加密。
```
签名过程：
  发送方
  1. 先对消息算哈希
  2. 用自己的私钥对哈希值加密（签名）
  3. 把消息 + 数字签名一起发出去

验证过程：
  接收方
  1. 用发送方的公钥解密签名，得到哈希值A
  2. 对收到的消息自己算哈希，得到哈希值B
  3. 对比A和B
     一样的话，说明：
       • 消息没被篡改（完整性）
       • 确实是对方发的（身份认证）
       • 对方不能抵赖（不可否认）
```

为什么能防抵赖？
因为私钥只有发送方自己有，
只有他能生成这个签名。
他签了就不能说不是他签的。

数字签名是PKI、HTTPS、
区块链、电子合同...
很多东西的基础。
非常重要。

### 4.7.2 常见的数字签名算法

```
  • RSA签名
    用RSA算法做签名
    最经典的

  • DSA
    数字签名算法
    专门用来签名的，不能加密

  • ECDSA
    椭圆曲线数字签名算法
    基于ECC
    密钥更短，速度更快
    现在用得越来越多
    区块链里也用这个

  • SM2
    国密的非对称算法，
    也可以做签名
```

## 4.8 数字证书与PKI

非对称加密和数字签名有个问题：
**我怎么知道这个公钥真的是对方的？**
万一公钥是中间人伪造的怎么办？
这就要讲到**数字证书**和**PKI**了。

### 4.8.1 什么是数字证书？

数字证书是什么？
简单说就是：
**由权威机构颁发的，
证明"这个公钥确实属于某个人/某个网站"
的电子文件。**

就像你的身份证，
由公安局颁发，
证明"这个身份证号的人确实是你"。
数字证书就是网络世界的身份证。

颁发数字证书的权威机构，
叫 **CA（Certificate Authority，证书颁发机构）**。
比如DigiCert、Let's Encrypt、
国内的CFCA等等。

数字证书里有什么？
```
  • 证书持有者的信息（域名、公司名...）
  • 持有者的公钥
  • 证书的有效期
  • 证书颁发机构（CA）
  • 证书的数字签名（CA用自己的私钥签的）
  • 等等
```

怎么验证证书是真的？
用CA的公钥去验证证书上的签名。
CA的公钥又由更高一级的CA证明，
一级一级往上，
直到根CA。
根CA的公钥是预置在操作系统和浏览器里的，
我们信任它。

这就形成了一条信任链：
```
你信任根CA
  → 根CA给二级CA签名，证明它可信
    → 二级CA给网站证书签名，证明网站可信
      → 你就相信网站的公钥是真的
```

这样就解决了公钥的信任问题。

### 4.8.2 什么是PKI？

**PKI（Public Key Infrastructure，
公钥基础设施）**是什么？
简单说就是：
**一整套管理密钥和证书的体系。**
包括：
- CA（证书颁发机构）
- 注册机构（RA）
- 证书库
- 密钥管理
- 证书生命周期管理（颁发、吊销、更新...）
- 等等

PKI是非对称加密体系的基础设施，
有了PKI，
非对称加密和数字签名才能真正用起来。
HTTPS、电子签名、VPN、
身份认证...
很多东西都依赖PKI。

## 4.9 HTTPS 的工作原理

讲了这么多密码学的知识，
最后我们用一个实际的例子——
**HTTPS**，
把这些知识串起来。
看看密码学在实际中是怎么应用的。

### 4.9.1 HTTPS是什么？

HTTPS = HTTP + SSL/TLS
就是在HTTP外面加了一层SSL/TLS加密，
保证HTTP传输的内容是加密的，
中间人看不到也改不了。

TLS（Transport Layer Security）
是SSL的升级版，
现在用的都是TLS（SSL是老的，不安全了）。
不过习惯上还是经常叫SSL。

### 4.9.2 TLS握手过程

HTTPS在传实际数据之前，
要先做一次**TLS握手**，
协商加密算法、交换密钥、验证身份。
等握手完了，
后面的HTTP数据就都是加密的了。

TLS握手的大致过程（简化版）：

```
  客户端                                   服务端
    │                                       │
    │         1. ClientHello                │
    │  （我支持哪些加密套件、随机数A...）     │
    │──────────────────────────────────────>│
    │                                       │
    │         2. ServerHello                │
    │  （选好加密套件、随机数B...）          │
    │                                       │
    │         3. 证书                       │
    │  （把我的证书发给你）                 │
    │<──────────────────────────────────────│
    │                                       │
    │  验证证书（用CA公钥验证签名）          │
    │  证书合法，确认是服务器的              │
    │  取出服务器的公钥                     │
    │                                       │
    │         4. 密钥交换                   │
    │  （用服务器公钥加密预主密钥，发过去）  │
    │──────────────────────────────────────>│
    │                                       │
    │                        用私钥解密，    │
    │                        得到预主密钥    │
    │                                       │
    │  双方根据预主密钥+两个随机数，         │
    │  各自算出会话密钥（对称密钥）          │
    │                                       │
    │         5. Finished（加密的）         │
    │  （我这边准备好了）                   │
    │──────────────────────────────────────>│
    │         6. Finished（加密的）         │
    │  （我也好了）                         │
    │<──────────────────────────────────────│
    │                                       │
    │  握手完成，后面所有数据都用会话密钥加密 │
    │                                       │
```

实际的TLS握手比这个复杂，
不同的密钥交换算法流程也不一样。
但大致就是这么个意思：

1. 双方打招呼，协商用什么加密算法
2. 服务器给证书，客户端验证身份
3. 交换密钥（用非对称加密）
4. 双方算出共同的会话密钥（对称密钥）
5. 后面的数据都用这个对称密钥加密传输

你看，
这里面用到了：
- 非对称加密（RSA/ECDH）—— 交换密钥、验证证书
- 对称加密（AES等）—— 加密实际数据
- 哈希函数（SHA-2等）—— 完整性校验
- 数字签名 —— 证书验证
- PKI —— 信任链

把我们这一章讲的大部分内容都用上了。
是不是很巧妙？

## 4.10 本章总结

这一章我们讲了密码学基础，
内容比较多，也比较抽象。
给你总结一下：

**基本概念：**
- 明文、密文、加密、解密、密钥
- 柯克霍夫原则：安全靠密钥，不靠算法保密

**对称加密：**
- 加密解密用同一个密钥
- 速度快，适合大数据量
- 缺点：密钥分发难、管理难
- 代表算法：DES（已过时）、3DES、AES（主流）
- 加密模式：ECB不安全、CBC、GCM推荐

**非对称加密：**
- 一对密钥：公钥（公开）、私钥（保密）
- 公钥加密，私钥解密
- 私钥签名，公钥验证
- 解决了密钥分发问题，还能做签名
- 速度慢，不适合大数据
- 代表算法：RSA（经典）、ECC（椭圆曲线，趋势）
- DH密钥交换：不安全信道协商共同密钥

**哈希函数：**
- 任意输入 → 固定长度输出
- 单向不可逆、雪崩效应、抗碰撞
- 用途：密码存储、完整性校验、数字签名...
- 算法：MD5（不安全）、SHA-1（不安全）、SHA-2/SHA-3（推荐）
- 密码存储：加盐 → 慢哈希（bcrypt、Argon2）

**消息认证码（HMAC）：**
- 哈希 + 密钥
- 同时验证完整性和来源真实性
- 防恶意篡改

**数字签名：**
- 用私钥对哈希签名，公钥验证
- 作用：身份认证、完整性、不可否认
- 算法：RSA签名、DSA、ECDSA

**数字证书与PKI：**
- 数字证书：CA颁发，证明公钥属于谁
- CA：证书颁发机构
- 信任链：根CA → 二级CA → 网站证书
- PKI：公钥基础设施，一整套管理体系

**HTTPS / TLS：**
- HTTP + TLS加密
- TLS握手：协商算法、验证证书、交换密钥、生成会话密钥
- 混合加密：非对称交换密钥，对称加密数据
- 综合应用了各种密码学技术

最后送你一句话：

> **密码学是网络安全的基石，
> 没有牢不可破的加密，
> 只有相对安全的设计。**
