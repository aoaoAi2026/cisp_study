## 📖 写在前面

上一章我们讲了终端安全与主机加固，
这一章我们来讲防御体系篇的最后一章——
**安全运营与威胁检测。**

很多人对安全的理解，
可能就是买一堆安全设备，
防火墙、IDS、WAF、杀毒软件...
然后就觉得安全了。
但实际上，
安全设备买得再多，
没人运营、没人管，
跟废铁没什么区别。

**安全是三分技术，七分运营。**
设备和技术只是工具，
真正让安全发挥作用的，
是持续的运营。
设备买了不调策略，
告警多了没人看，
出了事没人响应，
那买再多设备也白搭。

这一章我们就来讲讲安全运营：
什么是安全运营、
安全运营都做什么、
什么是SIEM、什么是SOC、
什么是威胁情报、
什么是态势感知、
什么是威胁狩猎...
都是现在很火的概念。

## 15.1 先讲一个故事

### 15.1.1 "买了一堆设备，还是被黑了"

某公司老板很重视安全，
一上来就买了一大堆安全设备：
防火墙、IPS、WAF、IDS、
漏扫、堡垒机、数据库审计、
上网行为管理...
能买的都买了，
花了好几百万。
老板觉得这下安全了，
"这么多设备，
黑客肯定攻不进来。"

结果呢？
公司还是被黑客入侵了，
内网被横向移动，
数据被偷了一大堆。
直到黑客走了好久，
他们才发现。

老板很生气，
把安全负责人叫来问：
"我花那么多钱买了那么多设备，
怎么还是被黑了？
你们是干什么吃的？"

安全负责人很委屈：
"老板，设备是买了，
但没人管啊。
防火墙规则是默认的，
基本什么都放行；
IDS每天几千条告警，
没人看，也分不清哪个是真的；
WAF上线之后就没调过规则，
误报一大堆，
业务部门天天抱怨，
最后就把规则调松了，
跟没开差不多；
漏扫扫出一堆漏洞，
没人修，
报告出了就扔那了...
设备是买了，
但没人运营，
真的跟没装差不多。"

老板："..."

---

你看，
这就是很多企业安全的现状——
**重产品，轻运营；
重建设，轻维护。**
以为买了设备就安全了，
实际上设备只是工具，
用得好不好，
全靠人运营。

安全不是一劳永逸的，
买完设备就完事了？
不存在的。
安全是个持续的过程，
需要不断地运营、
不断地优化、
不断地改进。

这一章，
我们就来讲讲安全运营那些事。

## 15.2 安全运营概述

### 15.2.1 什么是安全运营？

什么是安全运营（Security Operations）？
简单说就是：
**通过持续的运营工作，
让安全体系持续有效地运行，
不断改进，
以应对不断变化的安全威胁。**

用人话说就是：
**安全不是一次性项目，
是天天都要做的日常工作。
设备要管、策略要调、
漏洞要修、告警要处理、
出了事要响应...
这些日常工作，
就是安全运营。**

为什么需要安全运营？
```
  1. 威胁在不断变化
    新的漏洞、新的攻击手法、
      新的恶意代码...
    天天都有新东西出来
    你不跟着变，
      防护就失效了

  2. 安全设备需要调优
    设备买回来是默认配置，
      不一定适合你的环境
    策略要调、规则要优化、
      误报要处理
    不然要么误报多到没人看，
      要么什么都拦不住

  3. 漏洞需要持续修复
    新漏洞天天出
    系统在变、业务在变
    漏洞永远修不完
    需要持续地发现、修复、验证

  4. 安全是个持续改进的过程
    没有百分之百的安全
    出了事情要总结，
      要改进，要提高
    一次比一次好
```

安全运营的目标是什么？
就是**让企业的安全水平
持续保持在一个足够的高度，
能应对当前的主要威胁。**
不是追求完美，
而是持续有效。

### 15.2.2 安全运营都做什么？

安全运营具体都干些什么？
内容很多，大致可以分成几类：

```
  1. 监控与检测
    • 7x24小时安全监控
    • 安全告警分析与处置
    • 威胁检测与发现
    • 日志收集与分析

  2. 漏洞管理
    • 漏洞扫描
    • 漏洞验证
    • 漏洞修复跟进
    • 漏洞复测验证

  3. 安全设备运维
    • 防火墙、IDS/IPS、WAF等
      安全设备的日常运维
    • 策略管理与优化
    • 规则更新
    • 版本升级

  4. 事件响应
    • 安全事件的应急响应
    • 事件分析与溯源
    • 事件处置与恢复
    • 事件复盘与改进

  5. 威胁情报
    • 收集和分析威胁情报
    • 把情报用到检测和防御中
    • 了解最新的攻击手法和威胁趋势

  6. 合规与审计
    • 等保、ISO 27001等合规工作
    • 安全审计
    • 自查与整改

  7. 安全意识培训
    • 员工安全意识培训
    • 钓鱼演练

  8. 其他
    • 渗透测试、红队演练
    • 安全评估
    • 配置基线管理
    • ......
```

安全运营的内容非常杂，
什么都要管一点。
安全运营团队就像是企业安全的"大管家"，
什么事都找你。

## 15.3 SIEM —— 安全信息与事件管理

讲安全运营，
就不得不提 **SIEM**。
这是安全运营最核心的工具之一。

### 15.3.1 什么是SIEM？

SIEM是什么？
Security Information and Event Management，
安全信息与事件管理。

什么意思？
简单说就是：
**把企业里所有安全设备、
系统、应用的日志，
都收集到一个平台上，
集中存储、集中分析、
集中告警、集中展示。**

为什么需要SIEM？
因为企业里安全设备太多了——
防火墙、IDS、IPS、WAF、
杀毒软件、EDR、
服务器、数据库、应用...
每个设备都有自己的日志，
各看各的，
效率极低。
而且很多攻击是跨多个系统的，
单个设备的日志看不出什么，
但把所有日志放一起关联分析，
就能发现问题。

SIEM就是干这个的——
**集中日志 + 关联分析 + 统一告警。**

### 15.3.2 SIEM的核心功能

SIEM一般有这些核心功能：

```
  1. 日志收集
    收集各种来源的日志：
      网络设备、安全设备、
      服务器、数据库、应用、
      操作系统、中间件...
    各种格式的日志都能接

  2. 日志存储与检索
    把日志存起来，
    方便查询和检索
    一般存几个月到一年
    合规要求一般是6个月以上

  3. 实时监控与告警
    实时分析日志，
    发现异常和攻击就告警
    统一的告警平台

  4. 关联分析
    SIEM的核心价值
    把不同来源的日志关联起来分析
    比如：
      防火墙检测到扫描
      + IDS检测到SQL注入尝试
      + WAF检测到攻击被拦截
      + 服务器上发现异常登录
    单个看可能都不严重，
      但关联起来看，
      就是一次完整的攻击
    关联分析能发现单个设备发现不了的攻击

  5. 仪表盘与报表
    各种可视化仪表盘
    安全态势一目了然
    各种合规报表、统计报表

  6. 事件管理
    告警的工单管理
    分派、处理、闭环
    事件跟踪和统计

  7. 威胁情报集成
    集成威胁情报，
    有情报命中的IP/域名/哈希
      直接告警
    提高检测能力
```

SIEM是安全运营中心（SOC）的核心平台，
做安全运营基本都离不开SIEM。

### 15.3.3 常见的SIEM产品

常见的SIEM产品有哪些？
```
  商业产品：
    • Splunk：最有名的SIEM之一，
      功能强大，生态好
      就是贵
    • IBM QRadar：IBM的，
      大企业用得多
    • LogRhythm、ArcSight...
    • 国内的：
      绿盟、启明星辰、奇安信、
      深信服、安恒...
      基本每个安全厂商都有自己的SIEM
      很多叫"态势感知平台"

  开源产品：
    • ELK Stack（Elasticsearch + Logstash + Kibana）
      最流行的开源日志平台
      自己搭可以当轻量级SIEM用
      免费、灵活，但需要自己开发
    • Graylog
    • OSSIM
    • Wazuh
```

SIEM产品很多，
价格也差很多，
从几十万到几百万都有。
企业根据自己的规模和预算选。

## 15.4 SOC —— 安全运营中心

讲了SIEM，
再讲讲 **SOC**。

### 15.4.1 什么是SOC？

SOC是什么？
Security Operations Center，
安全运营中心。

简单说就是：
**专门负责安全运营的团队 + 平台 + 流程。**
一个专门的组织，
专门的人，
用专门的工具（SIEM等），
按照专门的流程，
7x24小时做安全监控、
检测、响应等工作。

SOC不是一个产品，
也不是一个部门，
而是**人 + 技术 + 流程**的组合。

### 15.4.2 SOC的运作模式

SOC一般怎么运作？

```
  人员：
    • L1（一线）：
      7x24小时监控告警
      处理低级别的告警
      处理不了的升级给L2
    • L2（二线）：
      经验更丰富的分析师
      处理复杂的告警和事件
      做深度分析
      处理不了的升级给L3
    • L3（三线）：
      资深专家
      处理重大事件、
        高级威胁、溯源分析
      做威胁狩猎
    • 管理层：
      管团队、管流程、
      跟其他部门沟通

  技术：
    • SIEM平台
    • 各种安全设备
    • 威胁情报
    • 分析工具
    • 响应工具
    • ......

  流程：
    • 告警处理流程
    • 事件响应流程
    • 漏洞管理流程
    • 升级流程
    • 交接班流程
    • 汇报流程
    • ......
```

成熟的SOC，
人、技术、流程三者缺一不可。

### 15.4.3 SOC的建设模式

企业想建SOC，
一般有几种模式：

```
  1. 自建SOC
    自己买设备、自己招人、
      自己建团队、自己运营
    优点：自己掌控，数据安全
    缺点：贵，招人难，运营成本高
    适合大型企业、政府、金融等

  2. 外包SOC / MSSP
    托管安全服务提供商
    把安全运营外包出去
    专业的安全公司帮你做
    优点：省心，专业，成本相对低
    缺点：数据给了第三方，
      响应可能没那么及时
    适合中小企业，
      或者自己建不起团队的

  3. 混合模式
    自己有一部分人，
      复杂的、核心的自己做
    基础的、人力密集的外包
    或者白天自己管，
      晚上和周末外包
    很多企业是这种模式
```

建不建SOC，
怎么建SOC，
要看企业的规模、
安全需求和预算。
不是每个企业都需要建SOC，
但安全运营的工作是要做的。

## 15.5 威胁情报

接下来讲**威胁情报（Threat Intelligence）**。
这几年也特别火。

### 15.5.1 什么是威胁情报？

什么是威胁情报？
简单说就是：
**关于威胁的信息和知识。
比如最新的攻击手法、
恶意IP/域名/URL、
恶意样本哈希、
攻击者的TTPs
（战术、技术、流程）、
漏洞信息、威胁趋势...
这些都是威胁情报。**

用人话说就是：
**知道敌人是谁、
敌人会怎么打、
敌人用什么工具、
敌人有什么特征...
知道了这些，
就能更好地防御和检测。**

威胁情报有什么用？
```
  1. 提升检测能力
    把情报的IOC
      （IP、域名、哈希、URL）
      放到防火墙、IDS、SIEM里
    匹配上了直接告警或拦截
    发现已知威胁快多了

  2. 辅助应急响应
    出了安全事件，
      查一下威胁情报，
    知道这个攻击是什么、
      是谁干的、目的是什么、
      怎么处置、怎么防御

  3. 帮助决策
    了解最新的威胁趋势
    知道现在什么攻击最流行
    安全建设和投入有侧重点

  4. 支撑威胁狩猎
    根据情报的线索，
      主动去搜内网里有没有
    主动发现潜藏的威胁
```

### 15.5.2 威胁情报的分类

威胁情报怎么分类？
按类型分：
```
  • IOC情报：
    最基础的情报
    就是各种指标：
      IP、域名、URL、文件哈希...
    用来匹配和检测

  • 技战术情报（TTPs）：
    比IOC高级
    攻击者的战术、技术、流程
    比如用什么工具、怎么钓鱼、
      怎么横向移动、用什么手法...
    基于ATT&CK框架的情报就是这类
    用来检测和防御高级威胁

  • 战略情报：
    更高层面的
    威胁趋势、攻击组织、
      行业威胁态势...
    给管理层做决策用的
```

按来源分：
```
  • 开源情报（OSINT）：
    公开渠道能拿到的
    比如各种漏洞库、安全博客、
      威胁情报社区、公开的IOC...
    免费，但质量参差不齐

  • 商业情报：
    花钱买的商业威胁情报
    质量更高，更及时
    比如各种威胁情报厂商的服务

  • 自建情报：
    自己收集、分析、生产的情报
    最贴合自己的环境
    但成本高，需要有能力的团队

  • 行业共享：
    同行业之间共享威胁情报
    比如金融行业、运营商行业
    行业内的情报最有针对性
```

威胁情报这东西，
不是买得多就好，
关键是要用起来，
能落地产生价值。
不然买了一堆情报，
放在那没人用，
就是浪费钱。

## 15.6 态势感知

再讲一个国内很火的概念——
**态势感知**。

### 15.6.1 什么是态势感知？

态势感知（Situation Awareness），
最早是军事上的概念，
后来被用到网络安全领域。

什么意思？
简单说就是：
**在一定的时空范围内，
感知环境中的元素，
理解它们的含义，
并预测它们未来的状态。**

用人话说就是：
**知道现在网络里发生了什么、
知道这意味着什么、
知道接下来会怎么样。**

三级态势感知：
```
  第一级：感知（Perception）
    知道发生了什么
    能看到现在的状态
    （有哪些告警、哪些攻击、流量怎么样...）

  第二级：理解（Comprehension）
    理解这意味着什么
    能分析出是什么攻击、
      严重程度怎么样、
      影响范围有多大

  第三级：预测（Projection）
    预测接下来会怎么样
    能预判攻击的下一步，
      提前防御
```

听起来是不是很厉害？
但说实话，
现在市面上很多叫"态势感知平台"的产品，
其实就是个SIEM加个好看的大屏，
能做到第一级就不错了，
第二级很多都做不好，
第三级基本还在概念阶段。

当然，
概念归概念，
实际归实际。
不管叫什么名字，
能真正帮企业发现问题、
解决问题的，
就是好平台。

### 15.6.2 态势感知和SIEM的关系

很多人搞不清态势感知和SIEM的区别。
简单说：
- SIEM是个工具平台，
  主要是日志收集、关联分析、告警
- 态势感知是更高一层的概念，
  强调的是对整体安全状态的感知和理解
- 态势感知平台一般会以SIEM为基础，
  加上更多的分析、可视化、
  威胁情报、溯源、响应等功能
- 国内厂商喜欢叫"态势感知"，
  国外一般叫SIEM或者安全分析平台

本质上都是差不多的东西，
都是安全运营的支撑平台。
叫什么名字不重要，
好用就行。

## 15.7 威胁狩猎（Threat Hunting）

再讲一个高级一点的概念——
**威胁狩猎（Threat Hunting）**。

### 15.7.1 什么是威胁狩猎？

什么是威胁狩猎？
简单说就是：
**主动地、有假设地
在网络和终端里
寻找潜藏的威胁。**

和传统的被动等待告警不一样，
威胁狩猎是**主动出击**——
我猜可能有什么什么威胁，
然后我主动去搜，
看看是不是真的有。

为什么需要威胁狩猎？
因为很多高级威胁（比如APT），
潜伏性很强，
不会触发告警，
你等着告警是等不到的。
你不去主动找，
它可能在你内网里潜伏半年你都发现不了。

威胁狩猎就是为了发现这些
潜藏的、告警发现不了的威胁。

### 15.7.2 威胁狩猎的过程

威胁狩猎一般怎么干？
```
  1. 提出假设
    基于威胁情报、攻击趋势、
      历史事件等，
    提出一个假设：
      "我们内网可能存在XX类型的攻击"
    比如：
      "最近勒索软件很流行，
        我们内网会不会已经有中招的了？"
      "最近很多攻击用XX手法横向移动，
        我们内网会不会也有？"

  2. 收集数据
    收集需要的数据：
      日志、流量、终端数据、
        威胁情报...

  3. 调查分析
    根据假设，
      去数据里找线索
    用各种工具和分析手法
    看看假设是不是成立

  4. 发现与响应
    如果真的发现了威胁，
      就启动应急响应
    该处置处置，该溯源溯源

  5. 总结与改进
    狩猎完了总结经验
    把发现的攻击特征加到检测规则里
    以后就能自动检测了
    持续提升检测能力
```

威胁狩猎是比较高级的安全运营工作，
需要有经验的分析师，
也需要有足够的数据和工具支撑。
不是每个企业都需要做威胁狩猎，
但安全成熟度比较高的企业，
都会做。

## 15.8 安全运营的关键指标

最后，
讲一下安全运营做得好不好，
怎么衡量？
也就是**安全运营的KPI**。

常见的指标有这些：

```
  告警相关：
    • 日均告警数量
    • 告警误报率
    • 告警平均响应时间（MTTA）
      从告警产生到有人开始处理的时间
    • 告警平均处置时间（MTTR）
      从告警产生到处置完成的时间
    • 告警闭环率
      有多少告警被真正处理了

  漏洞相关：
    • 漏洞总数、高危漏洞数
    • 漏洞修复率
    • 高危漏洞平均修复时间
    • 漏洞复发率

  事件相关：
    • 安全事件数量
    • 重大事件数量
    • 事件平均响应时间
    • 事件平均恢复时间（MTTR）
    • 事件造成的损失

  其他：
    • 安全设备在线率
    • 补丁安装率
    • 安全培训覆盖率
    • 钓鱼演练点击/中招率
    • 合规达标率
    • ......
```

指标不用太多，
选几个核心的、
能真正反映安全运营水平的就行。
关键是要有指标，
能衡量，能持续改进。

## 15.9 本章总结

这一章我们讲了安全运营与威胁检测。
总结一下：

**安全运营概述：**
- 安全是三分技术，七分运营
- 安全不是一次性项目，是持续的日常工作
- 目标：让安全体系持续有效运行
- 内容：监控检测、漏洞管理、设备运维、
  事件响应、威胁情报、合规、培训...

**SIEM：**
- 安全信息与事件管理
- 集中日志收集、关联分析、统一告警
- 安全运营的核心平台
- 产品：商业（Splunk、QRadar等）、
  开源（ELK等）

**SOC 安全运营中心：**
- 人 + 技术 + 流程
- 分级：L1一线、L2二线、L3三线
- 建设模式：自建、外包、混合

**威胁情报：**
- 关于威胁的信息和知识
- IOC情报、TTPs情报、战略情报
- 作用：提升检测、辅助响应、帮助决策
- 关键：要用起来，能落地

**态势感知：**
- 感知 → 理解 → 预测
- 国内很多态势感知平台≈SIEM + 大屏
- 概念很美好，落地要务实

**威胁狩猎：**
- 主动出击，寻找潜藏的威胁
- 不是等告警，而是主动搜
- 对付高级、潜伏的威胁
- 过程：假设 → 收集 → 调查 → 响应 → 改进

**安全运营指标：**
- 告警相关：响应时间、处置时间、误报率
- 漏洞相关：修复率、修复时间
- 事件相关：事件数、响应时间、恢复时间

最后送你一句话：

> **安全不是建设出来的，
> 是运营出来的。
> 设备买得再多，
> 没人运营也是白搭。**
