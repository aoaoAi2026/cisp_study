## Pass-16：二次渲染绕过 🎨

### 什么是二次渲染？——服务器给你的图片"整容"了

在开始这一关之前，我先问大家一个问题：你们有没有用过那种美图秀秀、醒图之类的P图软件？拍完照之后，软件会给你磨皮、瘦脸、加滤镜，最后出来的照片和原图已经不一样了对吧？

**二次渲染**就差不多是这个意思！🤯

啥叫二次渲染呢？简单说就是：你上传了一张图片，服务器收到之后，不会原封不动地保存，而是会用PHP的GD库（或者Imagick库）把图片**重新处理一遍**，然后再保存处理后的图片。

服务器为什么要这么做呢？通常有几个原因：

1. 📐 **压缩图片**：把大图变小，节省空间和流量
2. ✂️ **裁剪图片**：比如头像要裁成正方形
3. 💧 **加水印**：防止图片被盗用
4. 🧹 **清理元数据**：去掉图片里的EXIF信息（比如拍摄时间、地点等）
5. 🛡️ **安全考虑**：防止图片里藏了恶意代码

哎，最后这个"安全考虑"就和我们有关了！服务器心想：你不是喜欢往图片里藏PHP代码吗？我把图片重新生成一遍，你藏的代码不就没了吗？

就像什么呢？举个生活例子 🏠：
> 你想给朋友送一封信，但是信里藏了个小纸条。结果你朋友的妈妈收到信之后，把信的内容重新抄了一遍，然后把抄好的信给你朋友。你藏在信纸缝隙里的小纸条，当然就被"过滤"掉了！
>
> 在这里：
> - 原图 = 你写的信（藏了小纸条）
> - 二次渲染 = 妈妈重新抄信
> - 处理后的图 = 抄好的信（小纸条没了）
> - 你的PHP代码 = 藏的小纸条

哎，这可怎么办？我们往图片里插的代码，被服务器重新"抄"了一遍之后就没了，那图片马不就失效了吗？

### 怎么绕过？——找个"抄不掉"的地方藏代码

别着急！俗话说得好：上有政策，下有对策！😏

虽然服务器会重新渲染图片，但是图片的**某些部分**，渲染前后是**不会变**的！如果我们把PHP代码藏在这些不变的地方，那渲染之后代码不就还在吗？

就像刚才那个抄信的例子：
> 妈妈虽然会重新抄信的内容，但是信封上的地址、邮票、邮戳这些东西，她是不会动的。如果你把小纸条藏在邮票背面，妈妈抄完信之后，邮票还在，小纸条也就还在！

对！就是这个思路！我们要在图片文件里找到那些**二次渲染后不会改变的位置**，把PHP代码插在那里。

那什么图片格式比较容易找到这种位置呢？答案是——**GIF图片**！🎞️

为什么是GIF呢？因为GIF图片的格式比较特殊，它有很多"帧"（就是动图的每一帧），还有各种文件头、文件尾的标记。有些位置的数据，GD库在渲染的时候是不会修改的。

### GIF图片马制作思路

那具体怎么制作GIF图片马呢？我给大家讲一下思路，新手朋友能理解概念就行，不用太纠结细节。

大概步骤是这样的：

1. 🖼️ **找一张正常的GIF图片**
   - 随便找个小GIF动图就行
   - 越小越好，方便分析

2. 🔍 **用十六进制编辑器打开GIF**
   - 什么是十六进制编辑器？就是能看到文件原始二进制内容的工具
   - 推荐工具：WinHex、010 Editor、HxD（免费的）
   - 打开之后，你会看到一堆0和1组成的二进制数据，用十六进制表示

3. 🎯 **找合适的位置插入代码**
   - 这一步是关键！你需要找到渲染前后不变的位置
   - 怎么找呢？教大家一个笨办法：
     - 先把原图上传一次，下载渲染后的图片
     - 用十六进制编辑器对比两张图，看看哪些地方是一样的
     - 在一样的地方插入PHP代码就行啦

4. 💉 **插入PHP代码**
   - 在找到的位置插入 `<?php phpinfo();?>` 或者一句话木马
   - 注意：插入的代码不能破坏GIF的文件结构，否则图片就坏了

5. ✅ **验证效果**
   - 把修改后的GIF再上传一次
   - 下载渲染后的图片，看看代码还在不在
   - 如果还在，那就成功啦！

### 给新手的小提示 💡

说到这里，可能有小伙伴会问：这也太麻烦了吧？有没有简单点的办法？

哈哈，确实有点麻烦。不过没关系，我给大家几个实用小提示：

1. **GIF相对容易**：一般来说，GIF图片比JPG、PNG更容易绕过二次渲染。因为GIF的文件结构更复杂，GD库处理的时候不会动所有地方。

2. **可以用现成的工具**：网上有一些现成的图片马生成工具，可以直接生成能绕过二次渲染的图片马。不过我建议大家先理解原理，再用工具，不然知其然不知其所以然。

3. **思路比工具重要**：大家要记住，二次渲染绕过的核心思路是——**找渲染前后不变的地方藏代码**。这个思路比具体怎么操作更重要。

4. **实战中很少用**：说实话，在真实的渗透测试中，二次渲染绕过用得不多。但是理解这个思路很重要，它能帮你建立"**找漏洞就要找不变的东西**"这种思维方式。

### 源码分析 🔍

好了，讲完思路，让我们看看Pass-16的源码，加深一下理解：

```php
$is_upload = false;
$msg = null;
if (isset($_POST['submit'])){
    // 获得上传文件的基本信息，文件名，类型，大小，临时文件路径
    $filename = $_FILES['upload_file']['name'];
    $filetype = $_FILES['upload_file']['type'];
    $tmpname = $_FILES['upload_file']['tmp_name'];

    $target_path=UPLOAD_PATH.'/'.basename($filename);

    // 获得上传文件的扩展名
    $fileext= substr(strrchr($filename,"."),1);

    //判断文件后缀与类型，合法才进行上传操作
    if(($fileext == "jpg") && ($filetype=="image/jpeg")){
        if(move_uploaded_file($tmpname,$target_path)){
            //使用上传的图片生成新的图片（二次渲染）
            $im = imagecreatefromjpeg($target_path);
            if($im == false){
                $msg = "该文件不是jpg格式的图片！";
                @unlink($target_path);
            }else{
                //给新图片指定文件名
                srand(time());
                $newfilename = strval(rand()).".jpg";
                //显示二次渲染后的图片（使用用户上传图片生成的新图片）
                $img_path = UPLOAD_PATH.'/'.$newfilename;
                imagejpeg($im,$img_path);
                @unlink($target_path);
                $is_upload = true;
            }
        } else {
            $msg = "上传出错！";
        }

    }else if(($fileext == "png") && ($filetype=="image/png")){
        // 类似上面的PNG处理逻辑...
        // 使用imagecreatefrompng二次渲染
    }else if(($fileext == "gif") && ($filetype=="image/gif")){
        // 类似上面的GIF处理逻辑...
        // 使用imagecreatefromgif二次渲染
    }else{
        $msg = "只允许上传.jpg|.png|.gif类型图像文件！";
    }
}
```

大家看这段源码，发现关键点了吗？

1. 程序先检查后缀和MIME类型，没问题就把文件移到目标路径
2. **关键步骤**：然后用 `imagecreatefromjpeg()`、`imagecreatefrompng()`、`imagecreatefromgif()` 这些函数，把图片重新创建一遍
3. 最后把重新创建的图片保存下来，删掉原来的文件

这就是**二次渲染**！你上传的原图会被删掉，保存的是服务器重新生成的新图。所以你在原图里插的PHP代码，如果在渲染过程中被"画"掉了，那就没用了。

但是！就像我们刚才说的，GIF图片有些地方渲染后是不变的。所以这一关一般用GIF图片马来绕过。

## Pass-17：条件竞争漏洞（移动前访问）🏃‍♂️💨

### 什么是条件竞争？——和服务器比谁手快！

接下来这一关就有意思了！这一关的漏洞叫做**条件竞争漏洞**，也叫**竞争条件漏洞**（Race Condition）。

这个名字听起来很高大上对吧？但其实原理超级简单！用大白话讲就是：**和服务器比速度！看谁更快！** 🚀

怎么个比法呢？我先给大家讲一下服务器处理文件上传的两种常见逻辑：

**第一种（安全的做法）：**
1. 先检查文件合不合法
2. 合法才移动到目标目录
3. 不合法就直接拒绝

就像小区门口的保安，先看你是不是业主，是业主才让你进。

**第二种（不安全的做法）：**
1. 先把文件移动到目标目录
2. 然后检查文件合不合法
3. 不合法再删掉

哎？这就有问题了！发现了吗？🤔

文件先被移到目标目录，然后才检查，检查不通过再删掉。那在**移动之后、删除之前**这段时间里，文件是**真实存在**的啊！

虽然这段时间可能很短很短，可能只有几毫秒、几微秒，但它确实存在！只要存在，我们就有机会！

举个生活例子 📦：
> 想象一下，快递员给你送快递。正常的流程应该是：快递员先核对你的身份，确认是你本人，然后把快递给你。
>
> 但是有个迷糊的快递员，他是这么干的：先把快递扔你家门口，然后再看收件人名字对不对。如果不对，他再把快递拿回去。
>
> 那会发生什么？在快递员核对名字的这几秒钟里，快递就在你家门口放着！如果你动作够快，趁快递员还在核对名字的时候，冲出去把快递拆了、拿走了，那快递员就拿不回去了！
>
> 这就是条件竞争！
> - 先放快递 = 先移动文件
> - 再核对名字 = 再检查文件
> - 不对就拿回去 = 不合法就删除
> - 你趁他核对的时候拿走 = 你趁删除前访问文件

哈哈，是不是一下子就懂了？条件竞争就是钻了"**先放东西，后检查**"这个逻辑漏洞的空子！

### 原理详解：时间差里的机会

让我再用更技术一点的语言给大家解释一下。

假设服务器的代码是这样写的：

```php
// 1. 先把文件移到upload目录
move_uploaded_file($_FILES['file']['tmp_name'], 'upload/shell.php');

// 2. 然后检查文件是不是合法的PHP文件
if (check_if_php_file('upload/shell.php')) {
    // 3. 如果是PHP文件，就删掉
    unlink('upload/shell.php');
    echo '不允许上传PHP文件！';
}
```

大家看，第1步和第3步之间，是不是有个时间差？

虽然这个时间差非常非常短，可能只有0.001秒，甚至更短。但从理论上来说，在这个时间窗口里，`shell.php` 是真实存在于服务器上的！

如果我们能在这个时间窗口里访问到 `shell.php`，那PHP代码就会被执行！一旦代码执行了，就算服务器后面把文件删了也没关系了——因为代码已经跑过了！

这就像什么呢？🏃‍♂️
> 就像你往一个房间里扔一颗手雷，然后门会自动关上。虽然门很快就关上了，但只要手雷在门关上之前就爆炸了，那效果是一样的！

太形象了有没有！💥

### 怎么利用？——多线程狂轰滥炸！

道理我们都懂了，但是怎么实际利用呢？

因为那个时间窗口实在太短了，可能只有几微秒。我们手动刷新浏览器肯定是赶不上的，手速再快也没用。那怎么办呢？

答案是：**用多线程！一边疯狂上传，一边疯狂访问！** 🤯

具体思路是这样的：

1. 📤 **上传线程**：开很多很多线程，不停地上传 `shell.php`
2. 📥 **访问线程**：同时开很多很多线程，不停地访问 `upload/shell.php`

只要其中有一次，上传的文件还没被删掉，访问请求就刚好到了，那我们就成功了！

这就像什么呢？🎯
> 就像打地鼠游戏！地鼠会随机冒出来一下，然后马上缩回去。你打一次可能打不中，但如果你拿着100把锤子同时乱打，总有一锤能打中！
>
> - 地鼠冒出来 = 文件存在的时间窗口
> - 地鼠缩回去 = 文件被删除
> - 100把锤子乱打 = 多线程不停访问

只要尝试的次数足够多，概率再小的事情也会发生！这就是传说中的"**大力出奇迹**"！💪

### 实战演示思路

具体怎么操作呢？我给大家讲两种常用方法：

#### 方法一：Burp Suite Intruder（推荐新手用）

Burp Suite是渗透测试的神器，它的Intruder模块可以用来做这种暴力请求。

大概步骤：

1. 准备好一句话木马 `shell.php`
2. 用Burp抓上传包，发送到Intruder
3. 把Intruder的payload类型设为Null payload（就是什么都不改，一直发同样的包）
4. 线程数设高一点（比如20个线程）
5. 开始攻击，让它不停地发上传请求
6. 同时，浏览器或者另一个Intruder，不停地访问 `upload/shell.php`
7. 只要有一次访问成功了，就OK了

#### 方法二：Python脚本（进阶玩法）

如果你会写Python，也可以自己写个小脚本，多线程上传+访问。

大概思路：

```python
import requests
import threading

def upload():
    # 上传文件的代码
    while True:
        files = {'file': open('shell.php', 'rb')}
        requests.post('http://xxx/upload.php', files=files)

def access():
    # 访问文件的代码
    while True:
        r = requests.get('http://xxx/upload/shell.php')
        if r.status_code == 200:
            print('成功了！')
            break

# 开很多线程同时跑
for i in range(10):
    threading.Thread(target=upload).start()
    threading.Thread(target=access).start()
```

当然，上面只是简化的思路，实际写的时候还要考虑更多细节。

### 给新手的小提示 💡

说到条件竞争，可能有小伙伴会问：这玩意儿靠谱吗？会不会跑半天都不成功？

哈哈，这个问题问得好！我给大家说几个要点：

1. **成功是概率性的**：条件竞争不是每次都能成功，它是概率性的。就像买彩票，买得多了中奖概率就大。

2. **线程越多越快**：理论上线程越多，成功得越快。但也不是越多越好，太多了服务器可能扛不住，反而变慢了。

3. **要有耐心**：有时候可能要跑几十秒甚至几分钟才成功，耐心等待就好。

4. **思路最重要**：还是那句话，思路比具体操作重要。条件竞争这个思路在很多地方都能用，不只是文件上传。比如购物网站的"领取优惠券"、"抽奖"等等，都可能存在条件竞争漏洞。

### 源码分析 🔍

好了，让我们看看Pass-17的源码，验证一下我们的思路：

```php
$is_upload = false;
$msg = null;

if(isset($_POST['submit'])){
    $ext_arr = array('jpg','png','gif');
    $file_name = $_FILES['upload_file']['name'];
    $temp_file = $_FILES['upload_file']['tmp_name'];
    $file_ext = substr($file_name,strrpos($file_name,".")+1);
    $upload_file = UPLOAD_PATH . '/' . $file_name;

    // 关键点！先把文件移上去再说！
    if(move_uploaded_file($temp_file, $upload_file)){
        // 移上去之后才检查后缀！
        if(in_array($file_ext,$ext_arr)){
             $img_path = UPLOAD_PATH . '/'. rand(10, 99).date("YmdHis").".".$file_ext;
             rename($upload_file, $img_path);
             $is_upload = true;
        }else{
            $msg = "只允许上传.jpg|.png|.gif类型图像文件！";
            // 检查不通过才删除！
            unlink($upload_file);
        }
    }else{
        $msg = '上传出错！';
    }
}
```

大家仔细看这段代码的逻辑顺序：

1. **第1步**：`move_uploaded_file($temp_file, $upload_file)` —— 先把文件移动到目标目录！
2. **第2步**：`if(in_array($file_ext,$ext_arr))` —— 然后才检查后缀合不合法！
3. **第3步**：如果合法，就重命名；如果不合法，就 `unlink($upload_file)` 删除！

完美符合我们说的"**先移动，后检查，不合法再删除**"的模式！

那时间窗口在哪里呢？就在第1步和第3步之间！虽然只有短短几行代码的执行时间，但它确实存在！

只要我们在文件被删掉之前访问到它，就能执行PHP代码！

## Pass-18：条件竞争+图片马 🖼️🏃‍♂️

### 升级版挑战：检查更严了

好的，接下来我们看Pass-18。这一关可以看作是Pass-17的"**升级版**"。

为什么说是升级版呢？因为Pass-17的代码逻辑是：先移动文件，然后只检查一下文件后缀。那如果我们传的是图片马呢？后缀是.jpg，检查是能通过的，对吧？

但是Pass-18不一样，它的检查更严格了！它不仅检查后缀，还会**检查文件内容是不是真的图片**！

也就是说，如果你直接传个 `shell.php`，虽然能利用条件竞争短暂存在一下，但因为后缀不对，马上就被删了，时间窗口太短，可能还没等访问到就没了。

那怎么办呢？既然要检查图片，那我们就传个真的图片马呗！😏

### 组合技：图片马 + 条件竞争 + 文件包含

这一关的思路是这样的：

1. 🖼️ **传图片马**：我们传一个后缀是.jpg的图片马，因为后缀是白名单里的，所以不会被删掉
2. 🏃‍♂️ **条件竞争**：等等，那还要条件竞争干嘛？直接传图片马不就行了？
3. 📂 **文件包含**：哎，别着急！图片马虽然能传上去，但如果直接访问的话，图片马是不会执行的（除非有解析漏洞）。我们需要配合**文件包含漏洞**来用。

哎？不对啊，Upload-Labs里没有文件包含漏洞啊？那怎么利用图片马？

哈哈，问得好！我换个思路给大家讲：

其实这一关还有另一种玩法——**在条件竞争的时间窗口里，用图片马写一个shell进去！** 🤯

什么意思呢？我给大家捋一捋：

1. 我们先做一个特殊的图片马，这个图片马的功能不是 `phpinfo()`，而是**写文件**！
   - 比如：`<?php fputs(fopen('shell.php','w'),'<?php eval($_POST[1]);?>');?>`
   - 这个代码的意思是：在当前目录下创建一个 `shell.php`，里面写一句话木马

2. 然后我们用条件竞争的方法，一边上传这个图片马，一边访问它
   - 虽然图片马最终会被删掉（因为内容检查可能通不过，或者其他原因）
   - 但只要在被删掉之前被访问到一次，它就会执行写文件的代码
   - 一旦 `shell.php` 被写出来了，就算图片马被删掉也没关系了！

3. 最后我们直接访问写出来的 `shell.php` 就行了

哇！这个思路是不是很巧妙？图片马就像一个敢死队，它自己活不长久没关系，只要它在牺牲之前把"战友"（shell.php）召唤出来就行了！

举个生活例子 💣：
> 就像打仗的时候，你派一个突击队员去炸敌人的碉堡。突击队员自己可能会牺牲，但只要他在牺牲之前把碉堡炸掉了，任务就完成了！
>
> - 图片马 = 突击队员
> - 写shell = 炸碉堡
> - 图片马被删掉 = 突击队员牺牲
> - 任务完成 = 我们拿到shell

太形象了有没有！🫡

### 另一种思路：配合文件包含

当然，如果目标网站同时存在文件包含漏洞，那就更简单了：

1. 传图片马（后缀是.jpg，能通过检查，不会被删）
2. 然后用文件包含漏洞包含这个图片马
3. 图片马里面的PHP代码就会被执行

这种情况下就不需要条件竞争了，因为图片马是合法的，会一直存在。

但是在Upload-Labs的Pass-18里，主要还是考察**条件竞争+图片马写shell**这种思路。

### 给新手的小提示 💡

这一关可能稍微有点绕，我给新手朋友再梳理一下：

1. **为什么不能直接传php？** 因为后缀不对，虽然能利用条件竞争短暂存在，但时间太短，不容易成功。

2. **为什么传图片马？** 因为图片马后缀是.jpg，在白名单里，存在的时间更长，更容易被访问到。

3. **图片马怎么执行？** 要么配合文件包含，要么让图片马自己写一个shell出来。

4. **核心思路是什么？** 把一个"短命"的东西，变成一个"长命"的东西。图片马虽然可能活不长，但它可以在活着的时候创造出一个能一直活下去的shell.php。

大家好好体会一下这个思路，很有意思的！😉

### 源码分析 🔍

让我们看看Pass-18的源码，看看它的检查逻辑：

```php

    $filename = date("YmdHis").rand(1000, 9999).".jpg";
    $temp_file = $_FILES['upload_file']['tmp_name'];
```

## Pass-19：00截断（POST版）✂️📮

### 老朋友回来了！

看到"00截断"这几个字，是不是觉得很眼熟？哈哈，没错！我们在Pass-11、Pass-12已经见过%00截断了！

那Pass-19又有什么不一样呢？为什么叫"POST版"？

别急，听我慢慢道来！😉

### 回顾一下：%00截断是什么

在讲Pass-19之前，我们先快速回顾一下什么是%00截断。

%00截断的原理是：字符串在处理的时候，遇到 `\x00`（也就是ASCII码为0的字符）就会认为字符串结束了，后面的内容都被忽略掉了。

就像快递员看到"区"字就不往后看了一样。

之前我们学的%00截断，是把 `%00` 放在**URL参数**里（也就是GET方式传参）。比如：

```
save_path=shell.php%00
```

因为URL里的%00会被自动解码成 `\x00`，所以能产生截断效果。

那如果 `save_path` 是用POST方式传的呢？直接在POST参数里写 `shell.php%00` 行不行？

答案是：不行！🙅‍♂️

因为POST参数里的 `%00` 不会被自动解码，它就是字面量的%00，不会被当成 `\x00` 来处理。

就像你在快递单上写"北京市朝阳区%00某某小区"，快递员不会把%00当成截断符，他只会觉得这地址怎么这么奇怪。

### 那怎么办？——POST里怎么传0x00

那如果save_path是POST参数，我们怎么实现00截断呢？

方法很简单：**直接传二进制的 `\x00` 字符！**

怎么传呢？用Burp Suite抓包，然后在Hex（十六进制）视图里修改！

具体操作思路：

1. 抓包，找到POST参数里的 `save_path`
2. 比如你想让它截断在 `.php` 后面，那就在 `.php` 后面插入一个 `\x00`
3. 在Burp的Hex视图里，找到对应的位置，把字节改成 `00`
4. 这样POST参数里就有了真正的 `\x00` 字符
5. 程序处理的时候，遇到这个 `\x00` 就会截断！

举个例子 🌰：

假设我们POST参数里写的是：
```
save_path=shell.php.jpg
```

我们想让它在 `.php` 后面截断，那我们就把 `shell.php` 后面那个 `.` 的位置，改成 `0x00`。

改完之后，字符串变成了：
```
shell.php\x00.jpg
```

程序处理的时候，一遇到 `\x00` 就停了，实际保存的文件名就是 `shell.php`。完美！🎉

### 原理都是一样的

其实不管是GET版还是POST版，原理都是一样的——都是利用 `\x00` 字符来截断字符串。

区别只是在于：
- **GET方式**：URL里的 `%00` 会被自动解码成 `\x00`，所以直接写 `%00` 就行
- **POST方式**：需要手动插入二进制的 `\x00` 字符，不能直接写 `%00`

就像同样是一把刀：
- 在超市买，得放在包装盒里（%00编码）
- 拿回家用，就得拆了包装（直接\x00）

刀还是那把刀，只是包装不一样而已！🔪

### 前提条件 ⚠️

和之前的%00截断一样，POST版的00截断也有前提条件：

1. **PHP版本 < 5.3.4**
   - 高版本PHP修复了这个问题
2. **magic_quotes_gpc = Off**
   - 如果开了，`\x00` 会被转义掉

这两个条件和GET版是一样的，大家记一下就好。

### 给新手的小提示 💡

关于00截断，我再给新手朋友几个小提示：

1. **思路是核心**：不管是GET还是POST，不管是%00还是直接\x00，核心思路都是——**利用空字节截断字符串**。记住这个核心，不管形式怎么变，你都能认出来。

2. **实战中越来越少见**：说实话，因为PHP 5.3.4已经是很久以前的版本了，现在实战中00截断已经很少见了。但是这个思路还是很重要的，它能帮你理解"截断"这种攻击方式。

3. **截断不止于文件上传**：00截断不只能用在文件上传里，很多地方只要是字符串处理，都可能存在截断漏洞。比如文件包含、SQL注入等等。

4. **多动手试试**：光看没用，一定要自己动手试试，用Burp改改十六进制数据，感受一下截断的效果。

## Pass-20：文件名数组绕过 📦🤯

### 什么？文件名还能是数组？！

接下来这一关就更有意思了！它利用的是一个很多人都想不到的点——**文件名可以用数组的方式传递！**

啥意思？就是说，正常情况下我们上传文件，文件名是一个字符串，比如 `shell.php`。但是在PHP里，我们也可以把文件名当成一个**数组**来传！

比如：
```
$_FILES['upload_file']['name'][0] = 'shell.php'
$_FILES['upload_file']['name'][1] = 'jpg'
```

哎？这能有什么用呢？别急，听我慢慢给你讲！😏

### 原理：函数处理数组会出问题

为什么文件名用数组传就能绕过呢？

因为很多程序在处理文件名的时候，会用一些函数来获取文件后缀。比如：
- `end(explode('.', $filename))` —— 用点分割，取最后一个
- `strrchr($filename, '.')` —— 找最后一个点的位置
- `pathinfo($filename, PATHINFO_EXTENSION)` —— 获取扩展名

这些函数有一个共同特点：它们都是用来处理**字符串**的。

那如果传给它们的不是字符串，而是一个**数组**呢？会发生什么？

答案是：**会出错！或者返回不正常的值！** 🤯

这就像什么呢？举个生活例子 🍎：
> 你让一个人帮你数苹果，你给他一筐苹果，他数完告诉你有多少个。这很正常对吧？
>
> 但是如果你给他的不是一筐苹果，而是一筐装着苹果的箱子呢？他可能就懵了——这玩意儿怎么数？是数箱子还是数苹果？
>
> 在这里：
> - 字符串 = 一筐苹果
> - 数组 = 一筐箱子
> - 函数懵了 = 程序逻辑出错

程序逻辑一出错，我们的机会就来了！

### 具体怎么玩？——利用end()函数的特性

让我给大家举个具体的例子。假设程序是这样获取文件后缀的：

```php
$file_name = $_FILES['upload_file']['name'];
$temp = explode('.', $file_name);
$file_ext = end($temp);
```

这段代码的意思是：
1. 把文件名用点分割成数组
2. 用 `end()` 函数取数组的最后一个元素，就是后缀

正常情况下，比如文件名是 `shell.php`：
- `explode('.', 'shell.php')` → `['shell', 'php']`
- `end(['shell', 'php'])` → `'php'`
- 后缀是php，被黑名单拦下来 ❌

那如果我们把文件名传成一个数组呢？

我们构造一个这样的请求：
```
name[0] = shell.php
name[1] = （空的）
```

或者更巧妙一点：
```
name = shell.php/.
```

哎？怎么回事？让我解释一下。

在PHP里，如果我们把 `name` 参数传成一个数组，比如：
```
POST /upload.php
Content-Type: multipart/form-data; boundary=----

------
Content-Disposition: form-data; name="upload_file[name][]"

shell.php
```

这样的话，`$_FILES['upload_file']['name']` 就不是一个字符串了，而是一个数组！

那 `explode('.', $file_name)` 会怎么样？`explode` 函数期望第二个参数是字符串，你给它一个数组，它会返回什么？

在PHP里，`explode` 传入数组的话，会返回 `false` 或者空数组之类的（不同版本行为可能不一样）。然后 `end(false)` 又会怎么样？

总之，这一套下来，后缀检查很可能就失效了！

### 另一种思路：利用pathinfo的特性

还有一种情况，如果程序用 `pathinfo` 函数来获取后缀：

```php
$file_ext = pathinfo($file_name, PATHINFO_EXTENSION);
```

如果 `$file_name` 是一个数组，那 `pathinfo` 也会出错，返回不正常的值。

或者，我们还可以利用 `pathinfo` 的另一个特性。比如：

```
文件名：shell.php/
```

对，就是在 `.php` 后面加一个斜杠。`pathinfo` 处理这种文件名的时候，可能会认为没有后缀，或者后缀是空的。然后检查就绕过了。

但是保存的时候呢？因为后面有个斜杠，系统可能会把斜杠去掉，最后保存成 `shell.php`。

哎？这就很有意思了！检查的时候说没后缀，放行；保存的时候变成.php了。完美绕过！🎉

### 给新手的小提示 💡

数组绕过这一块可能稍微有点抽象，我给新手朋友梳理一下：

1. **核心思路是什么？** 给期望字符串的地方传一个数组，让程序逻辑出错，从而绕过检查。

2. **为什么能成功？** 因为程序员写代码的时候，默认文件名就是个字符串，根本没想到有人会传个数组进来。考虑不周 = 漏洞。

3. **怎么构造数组？** 用Burp抓包，修改参数名，加上 `[]` 就行。比如把 `name="upload_file"` 改成 `name="upload_file[]"`。

4. **多试几种方法**：数组绕过的具体姿势有很多种，不同的代码有不同的绕过方法。不要死记硬背，要理解思路，多尝试。

### 源码分析 🔍

让我们看看Pass-20的源码，看看它是怎么处理的：

```php
$is_upload = false;
$msg = null;
if (isset($_POST['submit'])) {
    if (file_exists(UPLOAD_PATH)) {
        $deny_ext = array("php","php5","php4","php3","php2","html","htm","phtml","pht","jsp","jspa","jspx","jsw","jsv","jspf","jtml","asp","aspx","asa","asax","ascx","ashx","asmx","cer","shtml","htaccess");
        $file_name = $_POST['save_name'];
        $file_ext = pathinfo($file_name,PATHINFO_EXTENSION);

        if(!in_array($file_ext,$deny_ext)) {
            $temp_file = $_FILES['upload_file']['tmp_name'];
            $img_path = UPLOAD_PATH . '/' .$file_name;
            if (move_uploaded_file($temp_file, $img_path)) { 
                $is_upload = true;
            }else{
                $msg = '上传出错！';
            }
        }else{
            $msg = '禁止保存为该类型文件！';
        }
    } else {
        $msg = UPLOAD_PATH . '文件夹不存在,请手工创建！';
    }
}
```

大家看这段代码：

1. 文件名是从 `$_POST['save_name']` 取的，也就是用户可控的
2. 用 `pathinfo($file_name, PATHINFO_EXTENSION)` 获取后缀
3. 检查后缀不在黑名单里就放行

那怎么绕过呢？

方法一：传数组！
- 把 `save_name` 传成一个数组，`pathinfo` 处理数组会出错，返回空或者false
- 空的后缀当然不在黑名单里，于是就放行啦！

方法二：利用pathinfo的特性
- 比如文件名写成 `shell.php/.` 或者 `shell.php/`
- `pathinfo` 可能认为后缀是空的，或者认为后缀不存在
- 然后 `move_uploaded_file` 保存的时候，斜杠可能被处理掉，最后保存成 `shell.php`

具体哪种方法能行，大家可以自己动手试试！实践出真知嘛！🔬

## Pass-21：数组+00截断 🎭✂️

### 终极组合技！

好了，终于到了最后一关——Pass-21！这一关是Upload-Labs的"**终极BOSS**"，它把我们之前学的好几种技巧都结合起来了。

这一关的名字叫"数组+00截断"，光听名字就知道不简单对吧？😎

### 思路：数组绕过 + 00截断 = 双重绕过

Pass-21的防护比Pass-20更严格一点。它可能不仅检查了后缀，还做了其他的一些防护。只用数组绕不过去，只用00截断也绕不过去。

那怎么办呢？**两个一起用！** 就像格斗游戏里的组合技，一招不行就两招连起来用！

具体思路是这样的：

1. 📦 **第一步：用数组绕过第一道检查**
   - 把文件名传成数组
   - 让某些检查函数失效

2. ✂️ **第二步：用00截断绕过第二道检查**
   - 在数组的某个元素里加入0x00字符
   - 利用截断来控制最终的文件名

3. 🎯 **最终效果**
   - 所有检查都通过了
   - 但实际保存的文件名还是 `shell.php`

就像打游戏打BOSS，BOSS有好几管血，你得一套连招下去才能打死。一技能破防，二技能输出，三技能收割！🎮

### 具体怎么操作？

我给大家讲一下大概的操作思路（因为不同版本的Upload-Labs，Pass-21的具体代码可能略有不同，所以重点是理解思路）：

假设程序是这样处理的：

```php
$file_name = $_POST['save_name'];
// 第一道检查：用某种方式检查后缀
if (check_ext($file_name)) {
    // 第二道检查：其他检查
    if (other_check($file_name)) {
        // 保存文件
        move_uploaded_file($tmp_name, $file_name);
    }
}
```

我们的目标是让两道检查都通过，但最后保存的是 `.php` 文件。

怎么构造呢？我们可以构造这样一个数组：

```
save_name[0] = shell.php\x00.jpg
save_name[1] = jpg
```

或者类似的构造方式。

第一道检查可能只看数组的最后一个元素，看到是 `jpg`，就通过了。
第二道检查可能看整个字符串，或者用另一种方式处理，也通过了。
最后保存的时候，因为有 `\x00` 截断，实际保存成了 `shell.php`。

哇！是不是很巧妙？每一道检查都被不同的技巧绕过了，最后组合起来就成功了！

### 核心思想：分而治之

其实这种组合绕过的核心思想就是八个字：**分而治之，各个击破**。

程序有好几层检查，每一层检查的逻辑可能都不一样。我们就针对每一层检查的特点，用对应的技巧去绕过它。

就像一个保险柜有好几道锁，每道锁的类型都不一样。有的是密码锁，有的是钥匙锁，有的是指纹锁。你得用不同的方法打开每一道锁，最后才能打开保险柜。🔐

### 绕过姿势大汇总

接下来，让我们一起回顾一下，我们都学了哪些文件上传绕过的姿势。大家可以对照着看看，哪些你已经掌握了，哪些还需要再复习复习。

#### 一、客户端验证绕过 🖱️

这类漏洞是最菜的，防护在前端，等于没防护。

1. **JS验证绕过**
   - 原理：前端JS检查后缀，后端不检查
   - 绕过：禁用JS、抓包改后缀、浏览器改HTML
   - 难度：⭐（有手就行）

#### 二、服务端MIME验证绕过 📝

检查Content-Type字段，这个也是纸老虎。

2. **Content-Type绕过**
   - 原理：只检查请求头里的Content-Type
   - 绕过：抓包把Content-Type改成 `image/jpeg` 之类的
   - 难度：⭐

#### 三、黑名单绕过 📋

用黑名单禁止某些后缀，但是黑名单总有漏掉的。

3. **特殊后缀绕过**
   - 原理：黑名单没列全，比如只禁了.php，没禁.php5、.phtml
   - 绕过：用.php5、.php3、.phtml、.pht等后缀
   - 难度：⭐⭐

4. **大小写绕过**
   - 原理：Windows系统大小写不敏感，黑名单没转小写
   - 绕过：用.pHp、.PhP之类的大小写混合
   - 难度：⭐⭐

5. **空格绕过**
   - 原理：Windows系统文件名末尾的空格会被自动去掉
   - 绕过：文件名改成 `shell.php `（后面加个空格）
   - 难度：⭐⭐

6. **点号绕过**
   - 原理：Windows系统文件名末尾的点会被自动去掉
   - 绕过：文件名改成 `shell.php.`（后面加个点）
   - 难度：⭐⭐

7. **::$DATA绕过**
   - 原理：Windows NTFS文件系统的特性
   - 绕过：文件名改成 `shell.php::$DATA`
   - 难度：⭐⭐⭐

8. **双写绕过**
   - 原理：程序把.php替换成空，但只替换一次
   - 绕过：文件名改成 `shell.pphphp`（双写php）
   - 难度：⭐⭐

9. **.htaccess绕过**
   - 原理：Apache的配置文件，可以自定义解析规则
   - 绕过：先传.htaccess，让.jpg文件按PHP解析
   - 难度：⭐⭐⭐

#### 四、白名单绕过 ✅

白名单比黑名单安全，但也不是绝对安全。

10. **%00截断（GET版）**
    - 原理：低版本PHP的字符串截断漏洞
    - 绕过：路径参数设为 `shell.php%00`
    - 前提：PHP < 5.3.4，magic_quotes_gpc = Off
    - 难度：⭐⭐⭐

11. **00截断（POST版）**
    - 原理：和GET版一样，只是0x00要手动插
    - 绕过：Burp Hex视图里插入00字节
    - 难度：⭐⭐⭐

12. **解析漏洞绕过**
    - 原理：Web服务器的解析漏洞
    - 比如：Apache从右往左解析、IIS6.0分号解析、Nginx畸形解析
    - 绕过：利用解析漏洞让图片马被当成PHP执行
    - 难度：⭐⭐⭐⭐

#### 五、文件内容检查绕过 🖼️

检查文件内容是不是图片，这就要用到图片马了。

13. **图片马绕过**
    - 原理：只检查文件头，不检查文件内容
    - 绕过：把PHP代码插在图片里，制作图片马
    - 配合：文件包含漏洞或者解析漏洞
    - 难度：⭐⭐⭐

14. **二次渲染绕过**
    - 原理：服务器会重新生成图片，但有些部分不变
    - 绕过：找渲染前后不变的位置插入代码
    - 常用：GIF图片相对容易绕过
    - 难度：⭐⭐⭐⭐

#### 六、逻辑漏洞绕过 🧠

这类漏洞最考验思维，是逻辑上的问题。

15. **条件竞争漏洞**
    - 原理：先移动文件，后检查，检查不通过再删除
    - 绕过：多线程同时上传+访问，趁删除前执行代码
    - 思路：和服务器比速度
    - 难度：⭐⭐⭐⭐

#### 七、数组绕过 📦

利用PHP数组的特性，让处理函数出错。

16. **文件名数组绕过**
    - 原理：给期望字符串的函数传数组，逻辑出错
    - 绕过：把文件名参数传成数组
    - 常用：配合end()、pathinfo()等函数的特性
    - 难度：⭐⭐⭐⭐

17. **数组+00截断组合绕过**
    - 原理：组合多种技巧，绕过多层检查
    - 绕过：数组绕过+00截断一起上
    - 难度：⭐⭐⭐⭐⭐

## 文件上传防御总结 🛡️

讲了这么多攻击方法，可能有小伙伴会问：那如果我是开发，我该怎么防御文件上传漏洞呢？

问得好！知攻善守嘛！只会攻击不会防御，不是好的安全工程师。接下来我就给大家讲一讲，怎么防御文件上传漏洞。

### 防御1：白名单验证后缀 ✅

**不要用黑名单！要用白名单！**

这是最重要的一点！黑名单永远列不全，总会有漏掉的后缀。而白名单只允许指定的后缀，其他的都不让传，这样就安全多了。

比如你只想让用户传图片，那白名单就设成 `['jpg', 'jpeg', 'png', 'gif']`，只有这几个后缀能传，其他的一律拒绝。

这就像什么呢？🏠
> 就像小区的门禁系统。黑名单是"不让张三李四进"，但你永远不知道还有哪些坏人。白名单是"只让业主进"，其他人都不让进，显然白名单更安全。

### 防御2：验证文件内容（二次渲染）🎨

光验证后缀还不够，万一有人传图片马呢？所以还要验证文件内容。

最安全的方法就是**二次渲染**！用GD库或者ImageMagick把图片重新生成一遍。这样的话，图片里藏的任何代码都会被清掉。

虽然二次渲染会牺牲一点性能，但是为了安全，这点牺牲是值得的。

当然，二次渲染也不是100%安全（就像我们Pass-16学的，GIF有些地方可能绕过去），但它已经是目前最有效的图片内容检查方法了。

### 防御3：随机重命名 🎲

上传的文件，不要用用户传的文件名！一定要**随机重命名**！

比如用时间戳+随机数+白名单后缀：`20250101120000_abc123.jpg`

随机重命名有什么好处呢？

1. **防止00截断之类的攻击**：你想控制文件名？门都没有，文件名我给你随机生成！
2. **防止猜文件名**：就算传上去了，别人也不知道文件名是什么，访问不到。
3. **防止重名覆盖**：大家都传 `头像.jpg`，不会互相覆盖。

这就像快递柜取件码，快递放进去之后，给你一个随机取件码，只有你知道，别人拿不走。📦🔢

### 防御4：文件存储到非Web目录 📂

如果可以的话，把上传的文件存到**Web根目录之外**的地方。

比如你的网站在 `C:\www\`，那上传的文件就存到 `D:\upload\`。这样就算有人传了木马，也没法通过URL直接访问到。

那用户怎么下载文件呢？用PHP脚本去读文件，然后输出给用户。这样所有的访问都经过PHP，安全可控。

这就像什么呢？🏦
> 就像银行的保险柜，不直接放在大厅里（Web目录），而是放在金库（非Web目录）里。你要取东西，得经过柜台（PHP脚本）办理手续才行。

### 防御5：设置文件权限为只读 🔒

上传目录的文件权限，一定要设置成**只读**，绝对不能有执行权限。

在Linux下，可以把上传目录的权限设为 `755`，文件设为 `644`。
在Windows下，可以设置上传目录只有读取权限，没有执行权限。

这样就算有人传了木马上去，也没法执行，因为没有执行权限。

当然，这是"最后一道防线"，不能只靠这个，但是加上它会更安全。

### 防御6：先检查再移动（避免条件竞争）🏃‍♂️

文件上传的逻辑顺序一定要对！**先检查，再移动**！

正确的顺序是：
1. 检查文件名后缀
2. 检查文件内容
3. 检查通过了，再移动文件
4. 移动失败就报错

错误的顺序（就是条件竞争那种）是：
1. 先移动文件
2. 再检查
3. 检查不通过再删除

先检查再移动，就不存在时间窗口，也就没有条件竞争的问题了。

### 防御总结：多层防御体系

最后跟大家说一句：**没有银弹！** 没有任何一种单一的防御方法是100%安全的。

最好的防御是**多层防御体系**，把上面说的几种方法都用上：

白名单 + 二次渲染 + 随机重命名 + 非Web目录存储 + 只读权限 + 先检查后移动

一层一层叠起来，就算某一层被绕过了，还有下一层挡着。这样的防御才是最坚固的。

就像我们穿衣服，一件单衣可能不保暖，但是秋衣+毛衣+外套+羽绒服，四层叠起来，再冷的天也不怕了！🧥🧣🧤

## 新手学习建议 📚💡

Upload-Labs打完了，接下来该怎么学习呢？我以过来人的身份，给新手朋友们几条建议。

### 建议1：每一关都要自己动手打 🎮

这是最重要的一条！不要光看教程，不要光看视频，一定要**自己动手打**！

看别人打和自己打，完全是两码事。看的时候你觉得"哦，这么简单，我会了"，等你自己动手的时候就会发现，到处都是坑，怎么都不成功。

只有自己动手踩过坑，你才能真正理解。失败的经验比成功的经验更宝贵。

就像学游泳，你看100遍游泳教学视频，不下水永远也学不会。🏊‍♂️

### 建议2：一定要看源码，理解为什么能绕过 🔍

很多新手打靶场，就是网上找个payload，粘进去，成功了，然后下一关。这样是没用的，过几天就忘了。

**一定要看源码！一定要理解为什么能绕过！**

每过一关，都去看看这一关的源码，思考几个问题：
- 这个漏洞出在哪里？
- 程序员哪行代码写错了？
- 为什么我的payload能绕过？
- 如果是我来写，我会怎么修复？

想清楚这几个问题，这一关才算真正过了。不然你就是记了个payload，换个场景就不会了。

### 建议3：不是为了记payload，是为了学思路 🧠

我再强调一遍：**不要死记硬背payload！要学思路！**

payload是死的，思路是活的。今天你记住了Pass-17的payload，明天换个网站，代码不一样了，你就不会了？那有什么用？

真正重要的是思路：
- 条件竞争的思路是"找时间差"
- 00截断的思路是"字符串截断"
- 数组绕过的思路是"类型混淆"
- 黑名单绕过的思路是"找漏掉的"

把这些思路学会了，不管遇到什么新场景，你都能自己想出绕过方法。这才是真本事。

### 建议4：多总结归纳 📝

打靶场的过程中，要多总结多归纳。

比如：
- 哪些漏洞是前端的，哪些是后端的？
- 哪些漏洞和操作系统有关？（比如::$DATA只有Windows有）
- 哪些漏洞和Web服务器有关？（比如解析漏洞）
- 哪些漏洞和PHP版本有关？（比如%00截断）
- 哪些漏洞需要什么前提条件？

把这些东西整理出来，形成自己的知识体系。这样以后遇到问题，你就能快速定位，知道该往哪个方向想。

我自己打靶场的时候，就喜欢画思维导图，把每一类漏洞的原理、利用方式、防御方法都整理出来。这样复习起来也方便。

### 建议5：保持好奇心和热情 ❤️

最后一条，但可能也是最重要的一条：**保持好奇心和热情**。

网络安全这个领域，更新换代很快，新的漏洞、新的技术层出不穷。如果你没有好奇心，没有热情，很快就会跟不上。

但如果你真的热爱这个领域，享受那种破解谜题的成就感，那你会学得很快，而且学得很开心。

希望大家都能在安全这条路上走得长远，走得开心！💪
