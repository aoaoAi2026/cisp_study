## 💡 CSRF vs SSRF：用一句话把它们彻底分开

这两个名字太像了，只差一个字母，初学者经常搞混。
这里用一个超级简单的记忆法：

> **第一个字母说了算：**
> - **C**SRF → **C**lient（客户端）→ 借用户的身份
> - **S**SRF → **S**erver（服务端）→ 借服务器的身份
>
> C → 用户帮他干活；S → 服务器帮他访问。

**更直观的对比：**

```
CSRF（借刀杀人）：
攻击者 → 伪造请求 → 用户浏览器 → 目标网站
                             ↑
                        带着用户的Cookie
                        
SSRF（找内鬼）：
攻击者 → 传URL → 目标服务器 → 内网资源
                              ↑
                        用服务器的身份
```

**CSRF 关心的是：** 用户的浏览器向哪里发了请求
**SSRF 关心的是：** 服务器向哪里发了请求

**CSRF 的攻击效果：** 以用户的身份执行操作（改密码、转账、发帖...）
**SSRF 的攻击效果：** 以服务器的身份访问资源（内网扫描、读文件、打内网...）

有了这个区分框架，接下来看详细的复习内容就清晰多了。

## 📊 CSRF知识体系回顾

### 1.1 CSRF核心概念

**CSRF（Cross-Site Request Forgery，跨站请求伪造）**

> 攻击者诱导用户在已登录的Web应用中，
> 执行非预期的操作。

**核心：借刀杀人，借用用户的身份干坏事。**

**原理：**
- 浏览器自动携带Cookie
- 跨站请求可以发出去（同源策略不阻止发送，只阻止读取）
- 服务器只认Cookie，不认请求来源

**本质：身份冒用**

### 1.2 CSRF攻击条件

1. **用户已登录目标网站**（有Cookie）
2. **目标网站存在敏感操作**（改密码、转账等）
3. **操作只依赖Cookie认证**（没有Token等其他验证）
4. **攻击者能构造有效请求**（知道URL和参数）

### 1.3 CSRF的类型

| 类型 | 说明 | 实现方式 | 难度 |
|------|------|----------|------|
| **GET型** | 最简单 | img、script、iframe等标签 | 简单 |
| **POST型** | 最常见 | 表单自动提交 | 中等 |
| **JSON型** | 较少见 | fetch/XHR，可能受CORS限制 | 较难 |

### 1.4 CSRF能做什么？

用户能做什么，CSRF就能做什么：
- 修改密码、修改个人信息
- 转账、支付
- 添加管理员、删除用户
- 发微博、加关注、点赞
- 添加购物车、下单
- 等等...

**危害等级：取决于被攻击的功能**
- 低危：点赞、关注
- 中危：修改信息、发帖
- 高危：改密码、转账
- 严重：管理员操作

### 1.5 CSRF防御措施

| 防御方法 | 安全性 | 推荐度 | 说明 |
|----------|--------|--------|------|
| **Token验证** | 高 | ⭐⭐⭐⭐⭐ | 最推荐，随机不可预测 |
| **验证码** | 高 | ⭐⭐⭐⭐ | 敏感操作用，影响体验 |
| **SameSite Cookie** | 中高 | ⭐⭐⭐⭐ | 浏览器层面，老浏览器不支持 |
| **二次确认** | 高 | ⭐⭐⭐⭐ | 输入密码等，影响体验 |
| **Referer验证** | 中 | ⭐⭐ | 不推荐单独用，可绕过 |

**最佳实践：多层防御，纵深防御**
- 首选Token验证
- 辅助SameSite Cookie
- 敏感操作加验证码/二次确认

### 1.6 CSRF绕过技巧

- **Token绕过**：校验不严格、可预测、泄露（XSS、Referer）
- **Referer绕过**：不发Referer、检查不严谨、利用跳转
- **SameSite绕过**：浏览器bug、子域名、特殊场景
- **终极武器：XSS**：有XSS的话，CSRF防御形同虚设

### 1.7 CSRF与XSS的组合

- **XSS打CSRF**：用JS发请求，能拿Token，绕过防御
- **CSRF辅助XSS**：CSRF加管理员，然后找XSS/上传Shell
- **CSRF蠕虫**：存储型XSS + CSRF，指数级传播

## 📊 SSRF知识体系回顾

### 2.1 SSRF核心概念

**SSRF（Server-Side Request Forgery，服务端请求伪造）**

> 攻击者能够让服务器发起伪造的请求，
> 请求的目标地址由攻击者控制。

**核心：让服务器替你发请求，访问你访问不到的资源。**

**原理：**
- 服务器有访问URL的功能
- URL参数用户可控
- 服务器会请求攻击者指定的地址

**本质：服务器作为代理/跳板**

### 2.2 SSRF常见场景

只要是服务器会去访问URL的地方，
就可能有SSRF：
- 图片远程加载/下载
- 文章采集/爬虫
- 在线翻译/转码/截图
- URL检测/可用性检查
- 远程头像上传
- 第三方服务回调
- 等等...

**常见参数名：**
url、src、source、target、link、image、img、pic、path、file、callback、feed、uri、dest、redirect...

### 2.3 SSRF能做什么？

#### 信息收集类
- 扫描内网IP存活
- 扫描内网端口
- 识别内网服务
- 探测内网Web应用
- 读取本地文件

#### 攻击利用类
- 攻击内网Redis → 写Shell、写公钥、写计划任务
- 攻击内网MySQL → 读文件、写Shell
- 攻击内网FastCGI → 命令执行
- 攻击内网MongoDB/Elasticsearch/Memcached等
- 打内网Web漏洞
- 云环境获取元数据

#### 进阶利用
- 作为内网渗透的跳板
- 从SSRF到GetShell
- 从SSRF到内网全域

### 2.4 SSRF协议利用

| 协议 | 用途 | 重要性 |
|------|------|--------|
| **http/https** | 扫端口、打内网Web | ⭐⭐⭐ 基础 |
| **file** | 读取本地文件 | ⭐⭐⭐⭐ 重要 |
| **dict** | 探测服务、简单命令 | ⭐⭐⭐ 有用 |
| **gopher** | 发送任意TCP数据，打各种服务 | ⭐⭐⭐⭐⭐ 神器 |
| **ftp/sftp** | 探测、可能有其他利用 | ⭐⭐ 了解 |
| **ldap** | 探测、LDAP注入 | ⭐⭐ 了解 |

### 2.5 Gopher协议详解

**格式：** `gopher://host:port/_数据`

**特点：** 可以发送任意TCP数据包

**能做什么：**
- 打Redis（最常用）
- 打MySQL
- 打FastCGI
- 打SMTP
- 几乎所有TCP服务都能试试

**常用工具：** Gopherus

**Redis利用方式：**
1. 写WebShell（知道Web目录）
2. 写SSH公钥（root + SSH）
3. 写计划任务反弹Shell（Linux）
4. 主从复制RCE（Redis 4.x+）
5. ...

### 2.6 SSRF绕过技巧

#### 绕过localhost/127.0.0.1
- 进制转换：十进制、八进制、十六进制
- 特殊表示：127.1、0x7f000001
- DNS解析：localhost、lvh.me、nip.io等
- IPv6：[::1]

#### 绕过内网IP过滤
- 进制转换
- 302跳转
- DNS重绑定
- URL解析差异
- 短链接

#### 绕过协议限制
- 试试各种协议（dict、gopher等）
- 利用URL解析问题
- 利用@符号

### 2.7 SSRF防御措施

| 防御方法 | 安全性 | 推荐度 | 说明 |
|----------|--------|--------|------|
| **协议限制** | 中 | ⭐⭐⭐ | 只允许http/https |
| **域名白名单** | 高 | ⭐⭐⭐⭐ | 只允许指定域名 |
| **内网IP过滤** | 中高 | ⭐⭐⭐⭐ | 禁止访问内网 |
| **不返回响应** | 中 | ⭐⭐⭐ | 不让用户看到结果 |
| **统一错误信息** | 中 | ⭐⭐⭐ | 增加扫描难度 |
| **权限最小化** | 高 | ⭐⭐⭐⭐⭐ | 降低被攻破的影响 |

**最佳实践：多层防御**
- 协议限制 + 域名白名单（能加就加）
- 内网IP过滤
- 不返回原始响应
- 权限最小化

## ⚖️ CSRF vs SSRF 对比

### 3.1 核心对比表

| 对比项 | CSRF | SSRF |
|--------|------|------|
| **全称** | Cross-Site Request Forgery | Server-Side Request Forgery |
| **中文** | 跨站请求伪造 | 服务端请求伪造 |
| **问题在哪** | 客户端（浏览器） | 服务端（服务器） |
| **核心** | 借用用户身份 | 让服务器当代理 |
| **攻击者身份** | 冒充用户 | 站在服务器的位置 |
| **能访问什么** | 用户能访问的 | 服务器能访问的 |
| **攻击者能否看到响应** | 一般不能 | 一般能 |
| **是否需要用户登录** | 是 | 否 |
| **是否需要用户交互** | 是（访问恶意页面） | 否（攻击者直接构造） |
| **危害方向** | 操作用户账号 | 探测/攻击内网 |
| **常见场景** | 改密码、转账 | 图片加载、爬虫 |
| **最有效防御** | Token验证 | 内网IP过滤 + 协议限制 |
| **神器/技巧** | XSS组合拳 | Gopher协议 |

### 3.2 一个比喻来理解

想象你要进一个小区：

- **CSRF**：你不进去，你找了一个小区里的住户，让他帮你把东西带进去。你借用了住户的身份。
- **SSRF**：你不进去，但你能指挥小区里的保安帮你去各个楼送信、取东西。保安能去的地方，你都能通过他去到。

### 3.3 两者的联系

虽然原理不同，
但在实战中它们经常一起出现：

1. **都是"请求伪造"家族**
   - 都是伪造请求
   - 一个是让浏览器发，一个是让服务器发

2. **可以组合使用**
   - 用CSRF打SSRF？（比较绕）
   - 用SSRF打内网的CSRF？（可以）
   - 都可以和XSS组合

3. **防御思路相似**
   - 都是"不信任用户的输入"
   - 都是要做白名单/过滤
   - 都需要纵深防御

4. **在护网中都很重要**
   - CSRF：打用户、打管理员
   - SSRF：打内网、横向移动

## ❓ 常见问题答疑

### Q1：CSRF和XSS有什么区别？

**A：**

| 对比项 | XSS | CSRF |
|--------|-----|------|
| **核心** | 注入脚本，执行JS | 伪造请求，借身份 |
| **攻击者能做什么** | 偷Cookie、偷数据、挂马 | 执行用户有权限的操作 |
| **能否看到数据** | 能 | 不能（只是借身份执行） |
| **是否需要JS** | 是 | 否（可以用img等） |
| **危害方向** | 客户端 | 服务端操作 |

简单说：
- XSS是"我替你执行"
- CSRF是"我让你自己执行"

而且，**有XSS的话，CSRF防御基本没用**，
因为XSS可以拿到Token，然后发请求。

---

### Q2：SSRF和XXE有什么区别？

**A：**

这个问题很好，
XXE（XML外部实体注入）也能读文件、发起请求，
和SSRF有点像。

| 对比项 | SSRF | XXE |
|--------|------|-----|
| **原理** | 服务器请求用户指定的URL | XML解析时加载外部实体 |
| **入口** | URL参数 | XML输入 |
| **读文件** | file协议 | SYSTEM实体 |
| **发起请求** | 各种协议 | 也是各种协议 |
| **危害** | 更广（能打各种服务） | 相对局限一些 |

XXE可以看成是SSRF的一种特殊形式，
或者说XXE可以导致SSRF。

---

### Q3：CSRF能拿到数据吗？

**A：**

一般情况下**不能**。

因为CSRF的原理是：
- 攻击者构造请求
- 用户浏览器发送请求
- 服务器处理请求
- 响应返回给用户浏览器

攻击者**看不到响应内容**，
因为同源策略限制了跨域读取响应。

所以CSRF只能"做事"（执行操作），
不能"看事"（获取数据）。

但是！
如果配合XSS的话，
那就可以拿到数据了，
因为XSS是在同一个域执行的，
可以读取响应。

---

### Q4：SSRF一定能看到响应吗？

**A：**

不一定。

要看具体场景：
- **能看到响应**：比如图片加载、文章采集，服务器会把响应返回给用户
- **看不到响应**：比如只是后台异步请求，不返回结果

如果看不到响应，
就是**盲SSRF**，
利用难度会大一些，
但还是能做很多事：
- 扫端口（根据响应时间判断）
- 打Redis（不需要看响应，发命令就行）
- 写Shell（不需要看响应，只要写进去了就行）

所以即使是盲SSRF，
危害也很大。

---

### Q5：怎么快速判断一个参数有没有SSRF？

**A：**

最简单的方法：
**让它访问你能控制的地址，看有没有请求过来。**

**方法1：用DNSLog**
```
http://target.com/ssrf.php?url=http://xxx.dnslog.cn/test
```
然后去DNSLog平台看有没有解析记录。

**方法2：用自己的公网服务器**
在自己服务器上开个http服务，
然后让目标访问，
看access.log有没有记录。

**方法3：用Burp Collaborator**
Burp专业版自带的功能，
非常好用。

---

### Q6：SSRF只能打内网吗？

**A：**

不是。

SSRF当然也能打外网，
但打外网没什么意义啊，
攻击者自己就能访问外网，
何必通过SSRF呢？

SSRF的价值在于：
- 访问内网（攻击者访问不到的）
- 访问本地服务（127.0.0.1的）
- 访问云元数据服务
- 借服务器的身份访问某些只允许白名单IP的服务

所以大家关注的都是打内网。

---

### Q7：CSRF现在还多吗？

**A：**

比以前少多了，
因为现在框架基本都自带CSRF Token了。

但还是有：
- 老系统
- 开发不注意的
- 接口没加Token的
- Token校验不严的
- 有XSS的话，Token就没用了

而且在护网中，
CSRF经常和XSS组合使用，
威力巨大。

所以还是要掌握。

---

### Q8：SSRF在护网中常用吗？

**A：**

**非常常用！**

可以说是红队最喜欢的漏洞之一，
因为：
1. 外网入口少，SSRF能直接进内网
2. 内网易打，很多弱口令、未授权
3. 相对隐蔽，流量是服务器发的
4. 危害大，经常能直接打穿

很多护网行动，
都是靠一个SSRF打开局面的。

## 💼 面试题精选

### 基础题

**1. 什么是CSRF？原理是什么？**

> CSRF（跨站请求伪造）是指攻击者诱导用户在已登录的Web应用中执行非预期操作的攻击。
> 
> 原理：浏览器自动携带Cookie + 跨站请求可以发送 + 服务器只认Cookie = 身份被冒用。

**2. 什么是SSRF？原理是什么？**

> SSRF（服务端请求伪造）是指攻击者能让服务器发起伪造的请求，请求地址由攻击者控制。
> 
> 原理：服务器有访问URL的功能 + URL用户可控 + 服务器去访问攻击者指定的地址。

**3. CSRF和SSRF有什么区别？**

> - CSRF是客户端问题，让用户浏览器发请求，借用户身份
> - SSRF是服务端问题，让服务器发请求，访问内网
> - 一个是冒用用户身份，一个是把服务器当代理

**4. CSRF和XSS有什么区别？**

> - XSS是注入脚本，执行JS，偷数据
> - CSRF是伪造请求，借身份，执行操作
> - XSS能看到数据，CSRF看不到
> - 有XSS的话，CSRF防御基本没用

**5. CSRF怎么防御？**

> 最推荐的是Token验证（随机不可预测，每个用户一个）。
> 其他还有：验证码、SameSite Cookie、二次确认、Referer验证等。
> 最好是多层防御，纵深防御。

**6. SSRF怎么防御？**

> - 协议限制：只允许http/https
> - 域名白名单：能加就加
> - 内网IP过滤：禁止访问内网
> - 不返回响应内容
> - 统一错误信息
> - 权限最小化
> 
> 同样是多层防御。

### 进阶题

**7. SSRF有哪些利用方式？**

> - 扫描内网端口和存活主机
> - 读取本地文件（file协议）
> - 攻击内网服务（Redis、MySQL、FastCGI等）
> - 云环境获取元数据
> - 打内网Web漏洞
> - 作为内网渗透跳板
> - ...等等

**8. Gopher协议是什么？为什么这么强大？**

> Gopher是一个老协议，特点是可以发送任意TCP数据。
> 
> 为什么强大：因为只要是TCP的服务，理论上都能用gopher协议打，比如Redis、MySQL、FastCGI、SMTP等。
> 
> 格式：gopher://host:port/_数据（数据需要URL编码）

**9. SSRF怎么绕过IP过滤？**

> - 进制转换：十进制、八进制、十六进制
> - 特殊域名：解析到127.0.0.1的域名
> - IPv6：[::1]
> - 302跳转
> - DNS重绑定
> - URL解析差异
> - 短链接
> - ...等等

**10. CSRF的Token怎么绕过？**

> - Token校验不严（只检查存在，不检查值）
> - Token可预测（用时间戳、用户名等生成）
> - Token泄露（XSS、Referer、URL中）
> - 有XSS的话，直接拿Token
> - 子域名上的XSS也可能偷到Token

---

### 高级题

**11. SSRF + Redis能做什么？**

> 1. 写WebShell（知道Web目录的话）
> 2. 写SSH公钥（root且开SSH的话）
> 3. 写计划任务反弹Shell（Linux）
> 4. 写启动项（Windows）
> 5. 主从复制RCE（Redis 4.x/5.x）
> 6. Lua沙箱绕过
> 7. 读取Redis中的数据

**12. 什么是盲SSRF？怎么利用？**

> 盲SSRF就是看不到响应内容的SSRF。
> 
> 利用方式：
> - 扫端口（根据响应时间判断）
> - 打Redis（不需要看响应，发命令就行）
> - 写Shell（只要写进去了就行）
> - DNSLog外带数据（有些情况）
> 
> 虽然看不到响应，但危害依然很大。

**13. CSRF蠕虫是什么？**

> CSRF蠕虫 = 存储型XSS + CSRF
> 
> 原理：
> - 用户A访问有XSS的页面
> - XSS代码自动发CSRF请求，让用户A发帖
> - 帖子内容又包含XSS代码
> - 用户B看了A的帖子，又中招
> - 这样一传十，十传百...指数级传播
> 
> 历史上著名的案例：MySpace的Samy蠕虫、Twitter的CSRF蠕虫等。

## 📚 综合案例分析

### 案例1：护网行动中的SSRF打点

**背景：**
某红队参加护网行动，
目标单位外网只有一个官网，
漏洞很少，
硬攻很难打进去。

**过程：**

**第一步：信息收集**
- 对外网官网进行全面扫描
- 发现一个"图片分享"功能，可以输入图片URL

**第二步：漏洞挖掘**
- 测试这个功能，发现存在SSRF
- 验证：让它访问DNSLog，有记录

**第三步：内网探测**
- 用file协议读/etc/hosts、/proc/net/arp
- 发现内网网段：10.0.0.0/24
- 扫内网端口，发现大量存活主机

**第四步：发现突破口**
- 扫到10.0.0.50有Redis未授权
- 用gopher协议打Redis，写计划任务反弹Shell
- 成功拿到第一台内网机器

**第五步：内网渗透**
- 在这台机器上收集信息，发现域环境
- 抓哈希、找凭据
- 横向移动，一台一台打过去
- 最终拿下域控

**结果：**
一个SSRF漏洞，
从外网直接打穿到域控，
红队大获全胜。

**启示：**
SSRF是护网中的神器，
一个漏洞可能就打开了整个内网的大门。

### 案例2：CSRF + XSS组合拳打管理员

**背景：**
某论坛有存储型XSS漏洞，
但只有普通用户能发帖，
管理员权限很高，
但XSS打不到管理员（管理员不看普通帖子）。

**攻击过程：**

**第一步：发现XSS**
- 普通用户发帖的地方有存储型XSS

**第二步：发现CSRF**
- 管理员后台有"添加管理员"功能
- 存在CSRF漏洞（或者Token可以通过XSS拿）
- 但问题是怎么让管理员中招？

**第三步：社工 + 诱导**
- 攻击者注册一个账号
- 给管理员发举报/私信，内容里带XSS
- 或者发一个很诱人的帖子标题，吸引管理员点进去

**第四步：管理员中招**
- 管理员打开了私信/帖子
- XSS执行
- 用JS发CSRF请求，添加一个新管理员

**第五步：登录后台**
- 攻击者用新添加的管理员账号登录
- 上传WebShell
- 提权
- 内网渗透
- ...

**启示：**
XSS + CSRF组合拳威力巨大，
再配上社工，
防不胜防。

### 案例3：绕过SSRF防护的各种姿势

**背景：**
某网站做了SSRF防护，
过滤了127.0.0.1和内网IP，
只允许http协议。

**绕过过程：**

**第一轮：基础测试**
```
http://target.com/ssrf?url=http://127.0.0.1/
→ 被拦截
```

**第二轮：进制转换**
```
http://target.com/ssrf?url=http://2130706433/
→ 成功访问127.0.0.1！
```
（127.0.0.1的十进制）

**第三轮：试试内网IP**
```
http://target.com/ssrf?url=http://3232235777/
→ 成功访问192.168.1.1！
```
（192.168.1.1的十进制）

**第四轮：试试其他协议**
```
dict://127.0.0.1:6379/INFO
→ 成功！dict协议没过滤！
```

**第五轮：试试gopher**
```
gopher://127.0.0.1:6379/_INFO%0d%0a
→ 成功！gopher也没过滤！
```

**第六轮：打Redis拿Shell**
- 用gopher协议打Redis
- 写计划任务反弹Shell
- 成功GetShell！

**启示：**
SSRF防护很容易被绕过，
不能只靠简单的字符串过滤，
要正确解析URL，
要做全面的防护。

## 📝 本章小结

这一章我们对CSRF和SSRF做了全面的总结，
让我们再回顾一下重点：

### CSRF核心
- **是什么**：跨站请求伪造，借用户身份干坏事
- **原理**：浏览器自动带Cookie + 跨站能发请求
- **类型**：GET型、POST型、JSON型
- **防御**：Token验证（最推荐）、验证码、SameSite、二次确认
- **绕过**：Token不严、Referer绕过、XSS组合
- **神器**：XSS + CSRF组合拳

### SSRF核心
- **是什么**：服务端请求伪造，让服务器替你发请求
- **原理**：URL可控 + 服务器去访问
- **场景**：图片加载、爬虫、翻译、截图...
- **协议**：http、file、dict、gopher（神器）
- **利用**：扫内网、读文件、打Redis、打内网服务
- **绕过**：进制转换、302跳转、DNS重绑定...
- **防御**：协议限制、白名单、内网IP过滤、权限最小化

### 两者对比
- CSRF是客户端问题，SSRF是服务端问题
- CSRF借用户身份，SSRF借服务器位置
- CSRF操作用户账号，SSRF探测攻击内网
- 都很重要，都要掌握
