**重点掌握：**

**系统调用：**
- 用户态 → 内核态的桥梁
- ntdll.dll里的Native API
- syscall指令（x64）
- 系统调用号

**直接系统调用：**
- 原理：自己直接发syscall
- 优势：绕过用户态Hook
- 劣势：返回地址可疑
- 工具：SysWhispers2、Hell's Gate

**间接系统调用：**
- 原理：借ntdll的syscall指令
- 优势：返回地址在ntdll，更隐蔽
- 劣势：实现复杂

**Unhooking：**
- 原理：从磁盘加载干净代码覆盖
- 方法：.text段覆盖、单个函数Unhook
- 检测：周期性检查、硬件断点

**AMSI绕过：**
- 原理：Patch AmsiScanBuffer
- 方法：内存补丁、初始化失败、混淆

**ETW绕过：**
- 原理：Patch EtwEventWrite或停止追踪
- 方法：内存补丁、停止会话

**EDR对抗体系：**
- 多层检测，多层绕过
- 综合运用多种技术
- 没有银弹，组合使用

### 3.2 不同场景的技术选型

**场景1：只需要过静态扫描**
- 推荐：加壳 + 花指令 + 字符串加密
- 理由：实现简单，效果好，稳定

**场景2：需要过基础杀软（360、火绒等）**
- 推荐：Shellcode加密 + 进程注入 + 反沙箱
- 理由：对付行为检测和启发式扫描

**场景3：需要过EDR**
- 推荐：间接系统调用 + Unhooking + 无文件
- 理由：绕过用户态Hook和行为检测

**场景4：红队实战，要求最高**
- 推荐：综合方案（间接系统调用 + Unhooking + 无文件 + 反调试 + 分离免杀）
- 理由：多层绕过，确保成功率

### 3.3 免杀方案设计原则

1. **分层原则**
   - 不要只靠一种技术
   - 多种技术叠加使用
   - 一层被拦还有下一层

2. **够用原则**
   - 不是越复杂越好
   - 根据目标选择合适的方案
   - 过度复杂反而容易出问题

3. **稳定原则**
   - 免杀效果重要，稳定性更重要
   - 不能为了免杀导致程序崩溃
   - 兼容性要考虑

4. **测试原则**
   - 没有测试的免杀都是耍流氓
   - 多环境、多引擎测试
   - 持续测试，及时更新

### 问题2：有没有万能的免杀？

**答：**
**没有。**

为什么？
因为：
1. 不同杀软检测机制不一样
2. 杀软在不断更新升级
3. 今天能过，明天可能就不行了

### 问题4：为什么我做的免杀总是被杀？

3. **用的技术太老**

   - 比如简单的XOR加密加个壳

4. **测试不充分**
   - 只测了一个引擎
   - 没在真实环境测试

**解决方法：**
- 多分析，找出到底是哪里触发的
- 一种一种技术地试，定位问题
- 多测试，多调整

### 三、简答题参考答案

1. **杀软工作原理：**
   杀软通过多种方式检测恶意软件，包括：
   - 特征码检测：比对已知病毒的特征码
   - 启发式检测：根据行为特征判断
   - 云查杀：上传文件到云端分析
   - 行为检测：监控程序运行时的行为
   - AI检测：用机器学习模型判断

2. **Shellcode免杀思路：**
   核心思路是"加密 + 自定义加载器"：
   - 把Shellcode加密后存储
   - 运行时在内存中解密
   - 自己写加载器执行，不用现成工具
   常见加密方式：XOR加密、AES加密、Base64编码、自定义加密算法等。

3. **进程注入原理：**
   把代码注入到其他进程的内存空间中执行。
   常见方式：远程线程注入、进程掏空、DLL注入、反射型DLL注入、APC注入、线程劫持、映射注入、原子Bomb注入、进程空洞变种等。

4. **EDR检测与免杀：**
   EDR检测手段包括：用户态Hook、行为检测、内存扫描、ETW、内核回调、AMSI等。
   对应免杀思路：直接系统调用/间接系统调用绕过Hook、模拟正常行为对付行为检测、加密分段对付内存扫描、Patch ETW/AMSI等。
