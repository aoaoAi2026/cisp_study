## 💥 坑一：把目标扫挂了

### 1.1 怎么踩的坑？

这个坑，新手最容易踩。
我刚入行的时候，就踩过。

那是我第一次参加护网。
拿到目标，我特别激动。
打开nmap，直接扫全端口全网段。

```bash
nmap -p- -T4 -A -v 192.168.1.0/24
```

扫了没几分钟，目标那边的运维就发现了。
不是因为发现了攻击，是因为——
**网站卡了，访问不了了。**

原来目标的服务器性能不太行，
我这一通猛扫，把带宽占满了，
把服务器也扫崩了。

结果呢？
目标的运维一顿排查，
发现是被人扫的，
然后就把我们的IP封了。

第一天刚上场，就被封了IP。
别提多尴尬了。

### 1.2 后果有多严重？

把目标扫挂了，后果可大可小：

```
小的后果：
- 被封IP
- 被发现攻击
- 目标加强防护，更难打了

大的后果：
- 目标业务受影响
- 造成经济损失
- 搞不好还要担责任
- 严重的话，可能违法
```

> 记住：
> **我们是做渗透测试的，不是搞破坏的。**
> 影响目标正常业务，是大忌。

### 1.3 怎么避免？

怎么避免把目标扫挂了？

```
🛡️ 避坑指南：

1. 控制扫描速度
   - 不要用-T4、-T5这种猛的
   - 用-T2、-T3，慢一点
   - 或者用--rate参数限制速率
   - 慢一点总没错

2. 不要全端口全网段一起扫
   - 先扫常见端口
   - 先扫重点IP
   - 慢慢扩大范围
   - 不要一上来就猛干

3. 分批扫描
   - 不要一次扫完
   - 分批次来
   - 扫一会儿停一会儿
   - 给目标留点喘息的时间

4. 准备好备用IP
   - 被封了一个还有另一个
   - 不要一个IP用到黑

5. 先轻后重
   - 先轻轻摸一下
   - 确认目标扛得住，再加大力度
   - 不要一上来就用全力
```

> 老K的名言：
> **"做红队，要像个幽灵。
> 来无影去无踪，目标都不知道你来过。
> 你这倒好，跟拆迁队似的，哐哐哐一通砸，
> 人还没进去，先把人吵醒了。"**

## 💥 坑二：SQL注入把数据库跑挂了

### 2.1 怎么踩的坑？

这个坑也挺经典的。
有一次打一个目标，有个SQL注入点。
我很开心，直接上sqlmap开跑。

```bash
sqlmap -u "http://target.com/news?id=1" --dbs --batch
```

跑着跑着，突然网站打不开了。
怎么回事？
数据库挂了！

原来那个注入点是个查询很频繁的页面，
sqlmap跑的时候发了大量请求，
把数据库干崩了。

结果呢？
网站500错误，
运维紧急重启数据库，
然后就发现了攻击，
把漏洞修了，把我们IP封了。

好好的一个注入点，就这么没了。

### 2.2 为什么会这样？

```
原因分析：
1. sqlmap默认发请求很快
   - 没加延迟，一秒钟发好几个请求
   - 数据库扛不住

2. 注入点在热门页面
   - 这个页面平时访问量就大
   - 再加我们的攻击流量
   - 雪上加霜

3. 数据库性能差
   - 目标数据库配置不高
   - 慢查询一多就崩了

4. 我们太心急了
   - 想快点跑出结果
   - 没考虑目标的承受能力
```

### 2.3 怎么避免？

```
🛡️ 避坑指南：

1. 给sqlmap加延迟
   - --delay=1 （每个请求延迟1秒）
   - 慢一点，稳一点
   - 不要急着出结果

2. 先手工测试，确认没问题再上工具
   - 手工试试注入点
   - 看看页面响应速度
   - 心里有数了再用工具跑

3. 选择访问量小的时间点
   - 比如深夜、凌晨
   - 这时候人少，影响小
   - 不容易被发现

4. 控制好请求数量
   - 不要一下子把所有payload都扔过去
   - 慢慢试
   - 跑一会儿停一会儿

5. 随时观察目标状态
   - 跑着跑着看看网站还能不能访问
   - 发现不对赶紧停
   - 不要一条道走到黑
```

> 记住：
> **渗透测试不是比谁快，是比谁稳。**
> 慢一点没关系，把目标搞崩了就麻烦了。

## 💥 坑三：提权把系统提崩了

### 3.1 怎么踩的坑？

这个坑我印象特别深。
有一次拿到一个Windows服务器的webshell，
权限很低，想提权。

找了个内核漏洞的exp，
编译好了，传上去，运行。

然后...
蓝屏了。

对，直接蓝屏了。
系统崩了。

我当时人都傻了。
这怎么办？
还好是测试环境，目标也知道我们在做测试。
不然就出大事了。

后来分析原因：
exp有bug，在那个特定版本的系统上会蓝屏。
我没仔细看exp的适用范围，
拿过来就用，结果把系统搞崩了。

### 3.2 后果有多严重？

```
提权把系统搞崩了的后果：

1. 目标业务中断
   - 如果是重要服务器，损失可大了
   - 比如：电商网站崩一小时，损失多少万？

2. 打草惊蛇
   - 目标肯定发现被攻击了
   - 接下来肯定会加强防护
   - 后面更难打了

3. 担责任
   - 如果是正式项目
   - 搞崩了客户的服务器
   - 赔钱是小事，搞不好还要吃官司

4. 丢人
   - 这个就不用多说了
   - 太尴尬了
```

### 3.3 怎么避免？

```
🛡️ 避坑指南：

1. 用exp之前先看清楚
   - 适用于什么系统？什么版本？
   - 有没有蓝屏的风险？
   - 别人用了反馈怎么样？
   - 不要拿过来就用

2. 先在本地测试
   - 找个一样版本的虚拟机
   - 先在本地试试
   - 没问题再往目标上用
   - （虽然麻烦，但是安全）

3. 选择风险小的提权方法
   - 能不用内核exp就不用
   - 比如：sudo提权、SUID提权、定时任务提权...
   - 这些方法安全得多，不会崩系统
   - 内核exp是最后没办法了才用

4. 做好心理准备
   - 用内核exp就有可能蓝屏
   - 要有心理准备
   - 想好万一崩了怎么办
   - （比如：有没有办法远程重启？）

5. 重要目标慎之又慎
   - 如果是特别重要的服务器
   - 宁可不提权，也别把系统搞崩了
   - 慢慢想别的办法
   - 安全第一
```

> 老K说：
> **"提权就像开车。
> 老司机开车，稳得一批，乘客都睡着了都没感觉。
> 新手开车，猛踩油门猛刹车，动不动就翻车。
> 别急，慢慢来，稳比快重要。"**

## 💥 坑四：留后门把自己锁外面了

### 4.1 怎么踩的坑？

这个坑，说出来有点丢人。
有一次拿下一台Linux服务器，
想留个后门，方便以后进来。

我想了想，加个root用户吧。

```bash
useradd -o -u 0 backdoor
echo "backdoor:password" | chpasswd
```

加完之后，我想试试能不能登。
先退出当前shell，然后用新用户登。

结果...
登不进去。

为什么？
因为sshd配置里禁止了UID为0的用户远程登录。
我加的这个用户UID是0，跟root一样，
所以SSH登不上。

这也就算了。
更坑的是，我把原来的shell退了，
结果...
webshell也断了。

对，我把自己锁外面了。

刚拿下的服务器，
因为留后门，
把自己锁外面了。

你说尴不尴尬？

### 4.2 为什么会这样？

```
原因分析：
1. 对目标系统不了解
   - 不知道sshd有这个配置
   - 想当然地以为加个root用户就能登

2. 操作顺序不对
   - 应该先试新用户能不能登
   - 没问题了再退旧的
   - 我倒好，先退了再试

3. 后门方式太单一
   - 只留了一个后门
   - 这个不行就全完了
   - 应该多留几个不同类型的后门

4. 心急
   - 急着留后门
   - 没仔细想清楚
   - 结果弄巧成拙
```

### 4.3 怎么避免？

```
🛡️ 避坑指南：

1. 留后门之前先了解清楚环境
   - 有哪些登录方式？
   - 有哪些限制？
   - 防火墙规则是什么样的？
   - 搞清楚了再留

2. 先测试，再退出
   - 新后门一定要先测试
   - 确认能用了，再退旧的
   - 不要着急

3. 多留几个不同的后门
   - 不要只留一种
   - SSH密钥、计划任务、webshell、启动项...
   - 东方不亮西方亮
   - 这个不行还有那个

4. 不要留太明显的后门
   - 加个uid=0的用户，太明显了
   - 管理员一查就发现了
   - 要留就留点隐蔽的

5. 留后门也要讲规矩
   - 护网/测试结束之后要清掉
   - 不要给人家留隐患
   - 这是职业操守
```

> 记住一句话：
> **留后门没成功不可怕，
> 可怕的是把原来的权限也搞丢了。**
> 煮熟的鸭子飞了，最气人了。

## 💥 坑五：打错目标了

### 5.1 怎么踩的坑？

这个坑，说出来你们可能不信。
但是真的发生过。

有一次护网，目标给了一堆IP。
我们分工，一人打几个。

我拿到我的那几个IP，就开始打。
打得挺顺利，没一会儿就拿下一个。
我还挺开心，今天手气不错。

打了半天，老K过来问我：
> "你打得怎么样了？"

我得意地说：
> "拿下一个了，还有两个有眉目了。"

老K看了看我屏幕，脸色一变：
> "等会儿，你这个IP...
> 好像不在目标列表里啊？"

我："啊？"

赶紧一核对。
还真不在。
我把隔壁组的目标给打了。

我当时人都傻了。
这怎么办？
赶紧撤，赶紧清后门，赶紧跟人道歉。

还好都是友军，
不然就出大事了。

### 5.2 怎么会打错呢？

```
原因分析：
1. 目标太多，看花眼了
   - 几十个IP，抄错了一个
   - 就差一个数字，没注意

2. 没仔细核对
   - 拿到IP就开始打
   - 也没核对一下是不是自己的目标
   - 太心急了

3. 网段太像
   - 192.168.1.0/24 和 192.168.2.0/24
   - 就差一位，很容易搞混

4. 分工没搞清楚
   - 谁打哪几个，没明确
   - 或者中间变了，没通知到
```

### 5.3 怎么避免？

```
🛡️ 避坑指南：

1. 开打之前先核对
   - 目标IP对不对？
   - 是不是在授权范围内？
   - 确认了再动手

2. 把目标列表整理好
   - 用表格列出来
   - 哪些打了，哪些没打
   - 标记清楚
   - 不要凭记忆

3. 重要目标反复确认
   - 特别是要搞大动作之前
   - 再核对一遍IP
   - 确认没错再继续

4. 分工明确
   - 谁打哪几个，说清楚
   - 不要抢别人的目标
   - 也不要漏了自己的

5. 发现打错了立刻停
   - 不要心存侥幸
   - 马上停手，马上道歉
   - 该清理的清理
   - 越拖越麻烦
```

> 这个事说起来搞笑，但是真的很重要。
> **打错目标，轻则尴尬，重则违法。**
> 动手之前，一定要再三确认。
> 这是红线，不能碰。

## 💥 坑六：工具没做免杀，一上去就被杀

### 6.1 怎么踩的坑？

这个坑，新手经常踩。
我也踩过。

有一次拿到一个Windows的shell，
想传个mimikatz上去抓密码。

直接把mimikatz.exe传上去了。
刚传完，还没来得及运行，
就被删了。

被杀软干掉了。

这还不算完。
杀软还报警了，
目标的安全团队马上就发现了。
然后就是全网排查，
我们好不容易拿到的shell，也没了。

就因为一个mimikatz，
整条线都断了。
亏不亏？

### 6.2 为什么会这样？

```
原因分析：
1. 工具太出名了
   - mimikatz这种神器，杀软都盯着呢
   - 特征码都写烂了
   - 传上去就是送人头

2. 没做免杀
   - 直接拿原版就用
   - 不杀你杀谁？
   - 好歹加个壳啊

3. 太心急了
   - 拿到shell就想传工具
   - 也不先看看有没有杀软
   - 也不先试试免杀

4. 操作太明显
   - 直接传exe文件
   - 还是传到敏感目录
   - 不报警才怪
```

### 6.3 怎么避免？

```
🛡️ 避坑指南：

1. 先看看有什么杀软
   - 拿到shell先摸清楚环境
   - 有没有杀软？是什么杀软？
   - 心里有数再动手

2. 工具做免杀
   - 不要直接用原版
   - 至少加个壳、改改特征
   - 有能力的话自己写
   - 或者用powershell、C#之类的，免杀容易点

3. 先用系统自带的工具
   - 能不用第三方工具就不用
   - Windows自带的命令很多的
   - net、wmic、powershell...
   - 系统工具杀软不会乱杀

4. 上传要隐蔽
   - 不要直接传exe
   - 可以先传base64编码的
   - 传上去再解码
   - 或者直接在内存里加载
   - 尽量不落地

5. 从小动作开始
   - 不要一上来就搞大动作
   - 先试试小的，看看杀软反应
   - 没问题再加大力度
   - 稳一点
```

> 老K说：
> **"免杀是红队的基本功。
> 工具都过不了杀软，还怎么内网渗透？
> 当然，最好的免杀是不用工具，
> 用系统自带的东西就能搞定。
> 那才是高手。"**
>
> （详见第52章：免杀技术入门与进阶）

## 💥 坑七：忘了清日志，被溯源了

### 7.1 怎么踩的坑？

有一次打一个目标，
打得挺顺利，从外网打到内网，
还拿了好几台服务器。

我们觉得打得差不多了，就撤了。
结果过了几天，目标那边的安全团队找过来了。
把我们的攻击路径说得一清二楚。
连我们用了什么工具、执行了什么命令，都知道。

怎么回事？
我们忘了清日志了。

目标的服务器上，日志都好好的。
人家一查日志，
我们干了什么，一目了然。

虽然是护网行动，本来就是授权的，
但是被人完整溯源出来，
还是挺丢人的。

而且，要是真的攻击，
不就是被人逮住了吗？

### 7.2 哪些日志会暴露你？

```
常见的日志类型：

1. Web日志
   - access.log、error.log
   - 你访问了什么页面，传了什么参数
   - 一清二楚

2. 系统日志
   - Windows的事件日志
   - Linux的/var/log/下面的各种日志
   - 你什么时候登录的，执行了什么命令
   - 都有记录

3. 命令历史
   - .bash_history
   - 你敲过的命令，全在里面
   - 跟看直播似的

4. 中间件/数据库日志
   - Apache、Nginx、MySQL...
   - 也都有日志

5. 安全设备日志
   - 防火墙、WAF、IDS、EDR...
   - 这些日志更多
   - 你的一举一动都被记录着
```

### 7.3 怎么避免？

```
🛡️ 避坑指南：

1. 时刻想着擦屁股
   - 每做一步，都想想会不会留下痕迹
   - 有没有日志？有没有记录？
   - 养成随手清日志的习惯

2. 尽量不留痕迹
   - 能不落地的就不落地
   - 能内存执行的就内存执行
   - 留下的痕迹越少越好

3. 撤退前彻底清理
   - 走之前一定要把尾巴擦干净
   - 日志清了，文件删了，后门（如果允许的话）藏好
   - 不要拍拍屁股就走

4. 伪造/混淆日志
   - 高手还会伪造日志
   - 或者混入正常流量里
   - 让对方分不清哪些是攻击的，哪些是正常的

5. 用跳板/代理
   - 不要用真实IP直接打
   - 多跳几次
   - 就算被发现了，也溯源不到你
```

> 记住：
> **一个好的红队，打完就走，片叶不沾身。
> 目标甚至都不知道自己被打过。**
> 留下一堆日志让人逮个正着，那是新手干的事。

## 💥 坑八：密码太简单，被对方反打了

### 8.1 怎么踩的坑？

这个坑就更丢人了。
有一次我们搭了个C2服务器，
用来做远控。

为了方便，
我们给C2服务器设了个很简单的密码：
`123456`

我们当时想：
"谁会来打我们的服务器啊？"

结果呢？
还真有人来打。
而且还真打进来了。

有一天我们发现C2服务器不对劲，
多了几个不是我们的账号。
查了一下，被人扫到了，
弱口令，直接就进来了。

还好对方也是做安全的，
发现是我们的C2，就跟我们说了。
不然的话，
我们的C2被人端了都不知道。

你说丢不丢人？
我们是做渗透的，
结果自己的服务器被人渗透了。

### 8.2 为什么会犯这种低级错误？

```
原因分析：
1. 侥幸心理
   - "谁会来打我啊？"
   - "不会这么巧吧？"
   - 结果就是这么巧

2. 图方便
   - 复杂密码记不住
   - 简单的多省事
   - 反正只有自己知道

3. 灯下黑
   - 天天挖别人的漏洞
   - 自己的安全反而不注意
   - 理发师的头发总是最乱的

4. 安全意识不够
   - 觉得自己是做安全的，不会出事
   - 结果偏偏出事
   - 淹死的都是会水的
```

### 8.3 怎么避免？

```
🛡️ 避坑指南：

1. 自己的安全先做好
   - 不要灯下黑
   - 打别人之前先把自己的屁股擦干净
   - 自己都不安全，还打什么别人？

2. 密码要复杂
   - 强密码，越长越好
   - 不同服务器不同密码
   - 用密码管理器

3. 不要用密码登录
   - 用SSH密钥
   - 或者双因素认证
   - 比密码安全多了

4. 做好防护
   - 防火墙开起来
   - 只允许必要的端口
   - 只允许可信的IP访问
   - 能不暴露在公网就不暴露

5. 经常自查
   - 定期看看自己的服务器
   - 有没有异常登录？
   - 有没有可疑文件？
   - 不要等被人打进来了才发现
```

> 老K说：
> **"做安全的，先把自己的安全做好。
> 自己的服务器都守不住，还怎么打别人？
> 这不叫灯下黑，这叫丢人。"**
>
> 话糙理不糙。

## 💥 坑九：只顾着打，忘了做记录

### 9.1 怎么踩的坑？

这个坑，很多人都踩过。
包括我。

有一次打一个目标，打得特别顺。
一路从外打到内，还拿了域控。
打得兴起，光顾着爽了，忘了记录。

等打完了，要写报告了。
我傻眼了。

"等会儿，我是怎么打进来的来着？
第一个漏洞是什么来着？
我在哪台机器上提的权来着？
域管理员哈希是从哪台机器抓的来着？"

全忘了。
打得太投入，没记笔记。
结果写报告的时候，
全靠回忆，好多细节都记不清了。
报告写得一塌糊涂。

### 9.2 为什么记录很重要？

```
为什么一定要做记录？

1. 写报告要用
   - 这是最直接的
   - 没有记录怎么写报告？
   - 总不能全靠编吧？

2. 复盘要用
   - 打完了要复盘
   - 哪里打得好，哪里打得不好
   - 没有记录怎么复盘？

3. 留作证据
   - 如果是正式项目
   - 你得证明你确实打进去了
   - 截图、日志、记录... 都是证据

4. 积累经验
   - 这次打的过程记录下来
   - 下次遇到类似的目标
   - 直接参考，省得重新摸索

5. 团队协作要用
   - 如果是团队作战
   - 你不记录，队友不知道你干了什么
   - 很容易重复工作，甚至互相干扰
```

### 9.3 怎么做好记录？

```
🛡️ 避坑指南：

1. 养成随手记录的习惯
   - 每做一步都记下来
   - 哪怕只是简单写几句
   - 也比什么都不记强

2. 记录什么？
   - 时间：什么时候做的
   - 目标：打的哪个IP/域名
   - 操作：做了什么
   - 结果：成功了还是失败了
   - 截图：关键步骤截个图
   - 账号密码：拿到的账号密码记下来
   - 思路：当时是怎么想的，为什么这么做

3. 用什么记录？
   - 最简单的：txt、markdown
   - 专业点的：XMind、思维导图
   - 团队协作的话：飞书、Notion、语雀...
   - 适合自己的就是最好的

4. 打完之后整理
   - 打的时候可以记乱一点
   - 打完了一定要整理
   - 梳理出完整的攻击路径
   - 该补的截图补上
   - 形成完整的文档

5. 注意保密
   - 记录里有很多敏感信息
   - 要加密保存
   - 不要随便乱放
   - 项目结束了该删的删
```

> 老K说：
> **"好记性不如烂笔头。
> 打得再漂亮，没记录等于白打。
> 写报告的时候你就知道了，
> 记录有多重要。"**

## 💥 坑十：贪多嚼不烂，战线拉太长

### 10.1 怎么踩的坑？

有一次护网，
我们队运气不错，
第一天就拿下了好几个目标。

我当时特别兴奋，
觉得自己无敌了。
这个也想打，那个也想打。
同时开了好几个目标，
每个都打到一半。

结果呢？
- A目标打到一半，B目标有新发现，去打B了
- B目标打到一半，C目标有新进展，去打C了
- C目标打到一半，又想起A目标还没打完...

到最后，
哪个都没打完。
都是半拉子。

时间全浪费了，
战果却没多少。

### 10.2 为什么会这样？

```
原因分析：
1. 贪心
   - 什么都想要
   - 这个也想打，那个也想拿
   - 结果哪个都没打好

2. 没有规划
   - 打到哪算哪
   - 没有优先级
   - 想到什么做什么

3. 注意力不集中
   - 容易被新东西吸引
   - 做着做着就跑偏了
   - 缺乏定力

4. 高估自己的能力
   - 觉得自己能同时打好几个
   - 实际上根本顾不过来
   - 贪多嚼不烂
```

### 10.3 怎么避免？

```
🛡️ 避坑指南：

1. 制定计划
   - 开打之前先规划好
   - 先打哪个，后打哪个
   - 哪些是重点，哪些可以放一放
   - 有计划就不会乱

2. 优先级排序
   - 按价值排序
   - 先打价值高的
   - 先打容易打的
   - 不要眉毛胡子一把抓

3. 打完一个再打一个
   - 集中精力打一个
   - 打下来了，再换下一个
   - 不要同时开太多线
   - （当然，团队作战的话可以分工）

4. 懂得放弃
   - 这个目标打了半天没进展
   - 该放弃就放弃
   - 换一个继续
   - 不要死磕一个，浪费时间
   - （但是也不能轻易放弃，要把握好度）

5. 定期复盘进度
   - 每隔一段时间看看
   - 现在进展怎么样？
   - 有没有偏离方向？
   - 需不需要调整计划？
```

> 老K说：
> **"做红队，不是比谁打得多，是比谁打得深。
> 你打了10个目标，每个都只打了皮毛，
> 不如集中精力把一个目标打穿。
> 质量比数量重要。"**

### 案例2：一个标点符号引发的血案

这个坑，说出来你可能不信。
但是真事。

有一次，我们队的一个新人，
在目标服务器上执行删除命令。
他想删一个临时目录。

命令是这样的：
```bash
rm -rf /tmp/test
```

但是呢，他不小心在 `/` 和 `tmp` 之间多打了个空格。
变成了：
```bash
rm -rf / tmp/test
```

看到问题了吗？
`rm -rf /` 啊！
删根目录啊！

还好，
他当时不是root权限，
删了半天没删掉多少东西，
系统还能跑。
但是也删了不少文件，
搞得一团糟。

后来我们花了好长时间才恢复。
那个新人差点吓死。

> 🎯 这个故事告诉我们：
> 1. **敲命令一定要仔细**
> 2. **特别是rm -rf这种危险命令**
> 3. **输完了先看一遍再回车**
> 4. **一个空格，就能毁掉一切**
> 5. **能用相对路径就不要用绝对路径**
> 6. **能不用rm -rf就不用rm -rf**

### 实操题（5道）

1. 检查一下你自己常用的密码：
   - 是不是弱口令？
   - 有没有密码复用？
   - 赶紧改一改，从自己做起。

2. 找一个你之前打靶的经历，复盘一下：
   - 哪些地方做得好？
   - 哪些地方踩了坑？
   - 如果再打一次，你会怎么改进？
   - 写个复盘笔记。

3. 养成记录的习惯：
   - 下次打靶/做实验的时候，试着做记录
   - 时间、目标、操作、结果、思路... 都记下来
   - 打完之后整理成一篇完整的writeup
   - 你会发现，记录真的很有用

4. 练习"稳"：
   - 找一个你已经打过的靶场
   - 这次用最慢、最稳的方式打
   - 每一步都想清楚，不着急
   - 看看跟你之前快打有什么不一样的感受

5. 想一想：
   - 你现在最容易踩什么坑？
   - 为什么会踩这个坑？
   - 怎么避免？
   - 给自己列一个"避坑清单"，时刻提醒自己。

## 📝 本章小结

- 没踩过坑的红队不是好红队，踩坑是正常的，重要的是吸取教训
- 做红队，稳比快重要——不要像拆迁队，要像幽灵
- 扫描要控制速度，不要把目标扫挂了
- sqlmap加延迟，不要把数据库跑崩了
- 内核提权有风险，用之前先看清楚、先测试
- 留后门要先测试再退出，别把自己锁外面了
- 开打之前一定要核对目标，打错目标是红线
- 工具做免杀，能不用第三方工具就不用
- 随手清日志，擦干净屁股再走
- 自己的安全先做好，不要灯下黑
- 好记性不如烂笔头，做记录很重要
- 不要贪多，集中精力打好一个比什么都强
- 踩坑不可怕，可怕的是不长记性
