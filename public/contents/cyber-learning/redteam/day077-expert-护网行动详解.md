## 70.1 什么是护网行动？护网的历史与发展

> **最核心的类比：网络世界的"军事演习"**
>
> 每年国家会组织一场大规模实战对抗演练——让红队（攻击方）对真实目标发起网络攻击，蓝队（防守方）进行防御。这就是护网。
>
> | 概念 | 现实世界类比 | 解释 |
> |------|-------------|------|
> | **护网行动** | 朱日和军事演习 | 真刀真枪，但打的是空包弹(不造成真实破坏) |
> | **红队** | 蓝军(演习中的假想敌) | 模拟真实APT攻击者的手法 |
> | **蓝队** | 红军(演习中的防守方) | 防守、监控、应急响应 |
> | **裁判组** | 演习导演部 | 判定得分、确认战果、监督规则 |
> | **规则/红线** | 演习安全守则 | 不能攻击的范围(如核心业务系统) |
>
> **为什么要搞护网？一句话：平时不练，战时挨打。**
> 安全设备买了一堆，到底管不管用？安服团队号称7x24值守，遇到真攻击能响应吗？
> 护网就是最好的检验——用真攻击测试你的真防御。

### 70.1.1 护网行动的定义

**护网行动**（简称"护网"）是指由国家、行业或企业组织的网络安全攻防演练活动。简单来说就是：

> **在规定的时间内，红队（攻击方）对目标发起网络攻击，蓝队（防守方）进行防御，通过实战对抗来检验网络安全防护能力和应急响应能力。**

可以把它理解成网络世界的"军事演习"——双方真刀真枪地对抗，但不会造成真实的破坏。

### 70.1.2 为什么要搞护网？

**核心目的：以攻促防，以练代战。**

1. **发现真实弱点**：平时安全检查发现不了的问题，在实战对抗中会暴露
2. **检验防御能力**：安全设备买了不少，到底好不好用？护网来检验
3. **锻炼安全队伍**：让安全人员在实战中成长，提升应急响应能力
4. **推动安全建设**：护网发现的问题，推动了安全投入和建设
5. **满足合规要求**：国家相关法规要求关键信息基础设施单位定期开展演练

### 70.1.3 护网的历史发展

```
时间线：护网行动发展历程

2016年
  ↓ 第一届国家级护网行动开始
  ↓ 规模较小，参与单位有限
  ↓ 主要是金融、电信等关键行业

2017年
  ↓ 《网络安全法》正式实施
  ↓ 护网行动有了法律依据
  ↓ 参与范围扩大

2018-2019年
  ↓ 护网规模逐年扩大
  ↓ 攻防对抗强度增加
  ↓ 行业级护网开始兴起

2020-2021年
  ↓ 护网成为年度例行活动
  ↓ 参与单位数百家
  ↓ 攻防技术明显升级
  ↓ 开始引入供应链攻击场景

2022-2023年
  ↓ 护网进入常态化
  ↓ 企业级护网大量涌现
  ↓ 攻防对抗更加激烈
  ↓ 蓝队防御能力显著提升

2024年至今
  ↓ 护网行动持续深化
  ↓ 国际化趋势显现
  ↓ AI等新技术引入攻防
  ↓ 供应链安全和数据安全成为重点
```

### 70.1.4 护网与渗透测试的区别

很多人容易把护网和渗透测试混为一谈，其实它们有很大的区别：

| 维度 | 渗透测试 | 护网行动 |
|------|----------|----------|
| **范围** | 通常针对特定系统 | 全面的网络对抗 |
| **时间** | 通常1-2周 | 通常2-3周 |
| **方式** | 按计划逐步测试 | 实时攻防对抗 |
| **目标** | 找漏洞、出报告 | 检验整体安全能力 |
| **攻防** | 只有攻击方 | 红蓝双方实时对抗 |
| **场景** | 模拟单一攻击场景 | 模拟真实APT攻击 |
| **评估** | 技术层面为主 | 技术+管理+人员 |
| **结果** | 漏洞报告 | 攻防战绩+改进建议 |
| **参与者** | 测试团队+客户 | 红队+蓝队+裁判+组织方 |

## 70.2 护网的组织形式（国家级/行业级/企业级）

### 70.2.1 国家级护网

**定义：** 由国家相关主管部门（如公安部、网信办等）组织的全国性网络安全攻防演练。

**特点：**
- 规模最大，参与单位最多（数百家）
- 持续时间最长（通常3周左右）
- 攻防强度最高
- 涉及关键信息基础设施单位
- 有严格的规则和红线
- 有统一的裁判组和指挥中心

**参与方：**
```
组织方（公安部/网信办等）
    ↓
裁判组（第三方安全公司或专家组）
    ↓
┌──────────────┬──────────────┐
│    红队       │    蓝队       │
│ （攻击方）    │ （防守方）    │
│              │              │
│ 多支红队队伍  │ 被检查单位    │
│ 专业安全公司  │ 关键基础设施  │
│ 安全研究员    │ 重要行业企业  │
└──────────────┴──────────────┘
```

**典型参与单位类型：**
- 金融行业：银行、证券、保险
- 电信行业：三大运营商
- 能源行业：电力、石油、天然气
- 交通行业：铁路、民航、港口
- 政府部门：各委办局
- 医疗卫生：大型医院
- 教育行业：重点高校
- 重要企业：央企、国企

### 70.2.2 行业级护网

**定义：** 由行业主管部门或行业联盟组织的行业性网络安全攻防演练。

**特点：**
- 针对特定行业
- 规模适中
- 更关注行业特有风险
- 行业标准引导

**常见行业护网：**
- 金融行业护网（银保监会组织）
- 电力行业护网（能源局组织）
- 电信行业护网（工信部组织）
- 医疗行业护网（卫健委组织）
- 教育行业护网（教育部组织）

### 70.2.3 企业级护网

**定义：** 由企业自行组织或委托安全公司组织的内部网络安全攻防演练。

**特点：**
- 规模灵活可控
- 更贴近企业实际
- 可以定制化场景
- 成本相对较低
- 可常态化开展

**组织方式：**
1. **自主组织**：企业安全团队自己组织
2. **委托第三方**：请专业安全公司来组织
3. **联合演练**：多家企业联合组织

**企业级护网的常见场景：**
- 新系统上线前安全检验
- 安全整改效果验证
- 年度安全能力评估
- 安全意识提升活动
- 应急响应能力检验

### 70.2.4 三种护网形式对比

| 维度 | 国家级 | 行业级 | 企业级 |
|------|--------|--------|--------|
| **组织方** | 国家部委 | 行业主管 | 企业自身 |
| **规模** | 数百家单位 | 几十家单位 | 单个/少数企业 |
| **持续时间** | 2-3周 | 1-2周 | 3-7天 |
| **攻击强度** | 极高 | 高 | 中高 |
| **参与门槛** | 很高 | 较高 | 灵活 |
| **权威性** | 最高 | 高 | 内部认可 |
| **成本** | 极高 | 较高 | 可控 |
| **适用范围** | 关键基础设施 | 行业重点单位 | 各类企业 |

## 70.3 红队的角色与任务（侦察/攻击/钓鱼/后渗透）

### 70.3.1 红队的定位

在护网行动中，**红队**扮演"攻击者"的角色，模拟真实的黑客组织对蓝队目标发起网络攻击。

**红队的核心使命：**
- 模拟真实威胁行为者（APT组织）的攻击手法
- 发现目标网络的安全弱点
- 验证蓝队的安全防护是否有效
- 检验蓝队的检测和响应能力

### 70.3.2 红队的组织架构

一支完整的护网红队通常由以下角色组成：

```
红队队长（总指挥）
    │
    ├── 侦察组（情报收集）
    │   ├── OSINT情报分析师
    │   ├── 外部资产测绘员
    │   └── 社会工程学专家
    │
    ├── 攻击组（漏洞利用）
    │   ├── Web渗透工程师
    │   ├── 二进制安全研究员
    │   ├── 漏洞挖掘/利用专家
    │   └── 边界突破专家
    │
    ├── 钓鱼组（社会工程学攻击）
    │   ├── 钓鱼邮件制作
    │   ├── 钓鱼网站搭建
    │   └── 社会工程学实施
    │
    ├── 后渗透组（内网渗透）
    │   ├── 内网渗透工程师
    │   ├── 域渗透专家
    │   ├── 权限提升专家
    │   └── 横向移动专家
    │
    └── 支撑组（后勤保障）
        ├── C2运维
        ├── 免杀工程师
        ├── 工具开发
        └── 数据分析/报告
```

### 70.3.3 各组任务详解

#### 侦察组

**任务：** 全面收集目标信息，为攻击提供情报支持。

**工作内容：**
```bash
# 外部资产测绘
# 收集目标的IP段、域名、子域名
# 识别开放的端口和服务
# 发现暴露在互联网上的系统

# OSINT情报收集
# 企业信息、组织架构
# 员工信息（邮箱、电话、职位）
# 技术栈和工具使用情况
# 历史泄露数据

# 攻击面分析
# 识别高价值目标
# 分析可能的攻击路径
# 评估各目标的防护强度
# 为攻击组提供目标建议
```

**输出：**
- 目标资产清单
- 情报分析报告
- 攻击面评估
- 高价值目标推荐

#### 攻击组

**任务：** 从外部突破目标防线，获取初始访问权限。

**工作内容：**
```bash
# Web系统渗透
# 外部网站漏洞挖掘和利用
# SQL注入、XSS、文件上传等漏洞
# CMS漏洞、中间件漏洞利用

# 边界设备攻击
# 防火墙、VPN、路由器等设备
# 弱口令、配置不当
# 设备漏洞利用

# 漏洞研究
# 研究最新漏洞和利用方法
# 漏洞武器化
# Exp开发和调试
```

**输出：**
- 初始访问权限（Shell/账号）
- 漏洞利用报告
- 攻击路径记录

#### 钓鱼组

**任务：** 通过社会工程学获取初始访问权限。

**工作内容：**
```bash
# 钓鱼邮件制作和投递
# 钓鱼网站搭建
# 社会工程学实施
# 附件免杀处理
# 钓鱼数据分析
```

**输出：**
- 钓鱼获得的账号和权限
- 钓鱼成功率统计
- 社工攻击路径

#### 后渗透组

**任务：** 从已获取的权限出发，进行内网渗透。

**工作内容：**
```bash
# 内网信息收集
# 权限提升
# 横向移动
# 域渗透
# 权限维持
# 数据获取
# 痕迹清理
```

**输出：**
- 内网渗透路径
- 获取的系统权限
- 战果记录

#### 支撑组

**任务：** 为前线攻击提供技术支持和后勤保障。

**工作内容：**
```bash
# C2服务器部署和维护
# 免杀Payload制作
# 工具开发和定制
# 流量隐匿配置
# 数据记录和分析
# 报告撰写
```

### 70.3.4 红队成员的能力要求

| 角色 | 核心能力 | 经验要求 |
|------|----------|----------|
| 红队队长 | 统筹规划、团队管理、技术全面 | 5年以上，多次护网经验 |
| 侦察组 | OSINT、资产测绘、情报分析 | 2年以上 |
| 攻击组 | Web渗透、漏洞利用、编程 | 3年以上 |
| 钓鱼组 | 社会工程学、心理学、文案 | 2年以上 |
| 后渗透组 | 内网渗透、域渗透、提权 | 3年以上 |
| 支撑组 | 免杀、C2运维、工具开发 | 2年以上 |

### 70.3.5 红队的工作节奏

护网期间红队的工作节奏通常是这样的：

```
一天的工作安排（示例）：

07:00-08:00  早会，通报昨晚进展，安排今日任务
08:00-12:00  上午攻击时段，各组按计划行动
12:00-13:00  午餐+简短碰头
13:00-18:00  下午攻击时段
18:00-19:00  晚餐
19:00-23:00  晚间攻击时段（通常效果最好）
23:00-01:00  数据整理、日志记录、工具准备
01:00-07:00  轮班值守（通常留人值守）

注意：
- 护网期间作息不规律，体力消耗大
- 要合理安排轮班，避免疲劳作战
- 关键时段集中力量攻坚
```

## 70.4 护网的评分规则（得分项/扣分项/加分项）

### 70.4.1 评分体系概述

护网行动有严格的评分规则，红蓝双方都有明确的得分和扣分项。

**评分目的：**
- 量化评估双方的攻防能力
- 提供公平公正的评判标准
- 激励双方积极对抗
- 为改进提供数据依据

### 70.4.2 红队评分规则

#### 得分项

| 攻击成果 | 分值（参考） | 说明 |
|----------|------------|------|
| 获取普通服务器权限 | 5-10分/台 | 拿到Shell或同等权限 |
| 获取核心服务器权限 | 20-50分/台 | 数据库、应用服务器等 |
| 获取域控权限 | 100分 | 拿下域控制器 |
| 获取核心业务系统权限 | 50-100分 | 关键业务系统 |
| 获取工控系统权限 | 100分+ | 工业控制系统 |
| 窃取核心数据 | 50-100分 | 敏感数据、机密信息 |
| 获取管理员账号 | 20-50分 | 域管、系统管理员 |
| 钓鱼成功 | 10-30分/人 | 获取员工账号或主机权限 |
| 发现重大漏洞 | 10-30分 | 0day或高危漏洞 |
| 供应链攻击成功 | 50-100分 | 通过供应商渗透目标 |

#### 扣分项

| 违规行为 | 扣分 | 说明 |
|----------|------|------|
| 超出授权范围 | 严重扣分/取消资格 | 攻击非授权目标 |
| 使用禁止手段 | 严重扣分 | 使用DDoS等禁用手段 |
| 造成业务影响 | 严重扣分 | 导致业务中断或数据损坏 |
| 暴露攻击基础设施 | 扣分 | C2被发现且溯源到身份 |
| 违反操作规范 | 扣分 | 不按流程操作 |
| 提前泄露攻击信息 | 严重扣分 | 向蓝队透露攻击计划 |

### 70.4.3 蓝队评分规则

#### 得分项

| 防御成果 | 分值（参考） | 说明 |
|----------|------------|------|
| 发现并拦截攻击 | 5-20分 | 成功检测和阻断攻击 |
| 发现红队基础设施 | 20-50分 | 找到红队C2、钓鱼站等 |
| 溯源到红队身份 | 50-100分 | 识别出红队具体信息 |
| 修复漏洞 | 5-20分 | 及时修复发现的漏洞 |
| 应急响应有效 | 10-30分 | 快速响应和处理安全事件 |
| 报送有效情报 | 5-20分 | 向指挥中心报送有价值的情报 |
| 成功反制攻击 | 20-50分 | 对红队实施有效反制 |

#### 扣分项

| 防御失误 | 扣分 | 说明 |
|----------|------|------|
| 被拿权限 | 对应红队得分 | 被红队获取系统权限 |
| 未发现攻击 | 扣分 | 攻击发生但未检测到 |
| 响应不及时 | 扣分 | 发现攻击但处置缓慢 |
| 瞒报漏报 | 严重扣分 | 隐瞒安全事件 |
| 业务中断 | 严重扣分 | 防御操作导致业务受影响 |

### 70.4.4 加分项

**红队加分项：**
- 使用创新攻击手法
- 发现0day漏洞
- 完成复杂的攻击链
- 供应链攻击
- 社会工程学攻击成功

**蓝队加分项：**
- 成功溯源红队身份
- 有效反制攻击
- 创新防御手段
- 建立有效的威胁情报共享
- 全程无失分

### 70.4.5 评分争议处理

- 裁判组负责评分的公平公正
- 有争议的成果需要提交证据
- 红队需要提供攻击证据（截图、日志、权限证明）
- 蓝队可以申诉，但需要有依据
- 裁判组最终裁定

## 70.8 紫队/裁判组的角色

### 70.8.1 裁判组

**角色定位：** 护网行动的"裁判"，负责规则的执行和评分。

**主要职责：**
1. **规则解释**：对规则有争议时进行解释和裁定
2. **成果认定**：审核红蓝双方的成果，确认是否有效
3. **评分计算**：按照评分规则计算双方得分
4. **争议处理**：处理双方提出的争议和申诉
5. **违规判定**：判定是否有违规行为
6. **报告编写**：编写最终评估报告

**裁判组的组成：**
- 主裁判：总体协调和最终裁定
- 技术裁判：技术层面的成果认定
- 规则裁判：规则解释和争议处理
- 观察员：监督过程公平公正

### 70.8.2 紫队

**角色定位：** 紫队不是红队也不是蓝队，而是两者的"桥梁"。

**什么是紫队：**
紫队（Purple Team）是红队和蓝队的融合体，其目标不是对抗，而是**协同**——红队提供攻击视角，蓝队提供防御视角，共同提升安全能力。

**紫队的工作方式：**
```
传统红蓝对抗：
  红队 → 攻击 → 蓝队 → 防御
  （双方各自为战，信息不共享）

紫队协同：
  红队 ←→ 蓝队
  （信息共享，协同改进）
  
  1. 红队演示攻击手法
  2. 蓝队分析检测能力
  3. 双方共同分析差距
  4. 制定改进方案
  5. 验证改进效果
```

**紫队的价值：**
- 加速红蓝双方的知识传递
- 快速验证防御改进效果
- 建立攻防双方的理解
- 持续提升整体安全能力

### 70.8.3 其他角色

**组织方：**
- 制定总体方案
- 协调各方资源

## 70.9 护网中的规则与红线

> **军事演习为什么用空包弹？——规则不是为了限制你，而是为了不造成真实伤害**
>
> 护网中的"红线"就像军事演习中用**空包弹**代替实弹——你展现真实战斗力，但不允许造成真实的伤亡。
> 
> **为什么不能DDoS？** 因为DDoS可能导致业务瘫痪，就像演习中你不能真的炸掉一栋楼。测试的是防御方的反应能力，而不是真的破坏。
> 
> **核心原则：证明了你能做到就行，不需要真的破坏。**
> 比如你拿到了数据库管理员权限，截图证明即可，不需要真的删数据。

### 70.9.1 必须遵守的规则

#### 红队规则

**可以做：**
- ✅ 在授权范围内进行攻击
- ✅ 使用常规渗透测试技术
- ✅ 社会工程学攻击（在授权范围内）
- ✅ 漏洞利用
- ✅ 内网渗透
- ✅ 获取数据和权限

**不可以做：**
- ❌ 攻击授权范围外的目标
- ❌ 使用DDoS/DoS攻击
- ❌ 导致业务系统中断
- ❌ 破坏或篡改数据
- ❌ 攻击公共设施和民用系统
- ❌ 进行物理入侵（除非特别授权）
- ❌ 对蓝队人员的人身攻击
- ❌ 散布恶意软件到互联网
- ❌ 使用真实恶意勒索软件

#### 蓝队规则

**可以做：**
- ✅ 检测和拦截攻击
- ✅ 溯源分析
- ✅ 加固防御
- ✅ 修复漏洞
- ✅ 反制措施（在规则内）

**不可以做：**
- ❌ 攻击红队的基础设施（除非授权）
- ❌ 关闭业务系统来"防御"
- ❌ 瞒报安全事件
- ❌ 修改规则来获得优势
- ❌ 与红队串通

### 70.9.2 红线（绝对不能碰）

```
┌─────────────────────────────────────────┐
│            护网红线（绝对禁止）           │
├─────────────────────────────────────────┤
│                                         │

│  🔴 破坏、篡改、删除重要数据             │

│  🔴 在互联网散播恶意软件                 │
│  🔴 进行物理入侵和人身威胁               │
│  🔴 泄露护网相关信息到外部               │
```

### 70.9.3 数据安全要求

护网过程中会接触大量敏感数据，必须严格遵守数据安全要求：

1. **数据最小化**：只收集必要的数据
2. **数据加密**：敏感数据加密存储和传输
3. **数据隔离**：护网数据与生产数据隔离
4. **数据销毁**：护网结束后按规范销毁数据
5. **保密义务**：所有参与人员签署保密协议
6. **报告脱敏**：报告中对敏感信息进行脱敏处理

3. **内网横向以凭据为主**：Pass-the-Hash最常用
4. **EDR和流量监控是主要威胁**：红队需要更隐蔽
5. **行业差异明显**：金融最强，教育和医疗较弱
6. **蓝队越来越强**：攻防对抗在升级
