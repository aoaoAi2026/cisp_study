## 💡 核心原理：静态免杀——让文件"长得不像坏人"

> 基础免杀主要对付的是**静态检测**（文件落盘时的特征码扫描）。

### 一个贯穿始终的比喻：你的Payload是一封"逮捕令"

想象你的Payload是一张**绘有通缉犯面部纹身的通缉令照片**。杀软手里也有一张一模一样的通缉令照片，它会拿过来比对——像素级比对！

基础免杀的所有技术，本质上都是**改变这张照片的特征，让算法认不出来**：

| 技术 | 怎么做 | 类比 |
|------|--------|------|
| **加壳** | 把图放进一个信封，警察只能看到信封看不到图 | 压缩/加密整个程序 |
| **花指令** | 在图周围画很多花边，让自动比对系统困惑 | 加无意义代码干扰分析 |
| **代码混淆** | 把笔画拆散重组，人认得，但机器比不对 | 改变代码结构保留功能 |
| **字符串加密** | 把"通缉"两个字用密码写，运行时再翻译 | 加密敏感字符串 |
| **入口点修改** | 改掉照片的编号，让比对系统找不到照片 | 改OEP指向 |
| **等价指令替换** | 把"move a to b"改成用三种方式表达同一个意思 | 用不同指令实现相同功能 |

**这些技术的共同本质：**
```
文件在磁盘上的形态 ≠ 文件在内存中执行的形态
（杀软扫描到的）        （实际运行的）
```

## 62.1 加壳与脱壳基础

### 62.1.1 什么是加壳？

加壳，简单说就是
给程序"穿一层衣服"。

加壳工具会把
原始的程序代码
进行压缩或加密，
然后加上一段
"脱壳代码"。

程序运行时，
先运行脱壳代码，
把原始程序
在内存中还原出来，
然后再执行。

```
加壳前：
┌─────────────────┐
│   原始程序代码   │  ← 直接能看到
└─────────────────┘

加壳后：
┌─────────┬─────────────────┐
│ 脱壳代码 │ 压缩/加密的代码 │  ← 看不到原始代码
└─────────┴─────────────────┘

运行时：
1. 执行脱壳代码
2. 在内存中解密/解压原始代码
3. 跳转到原始代码执行（OEP）
```

### 62.1.2 加壳的原理

**加壳过程：**
1. 读取原始PE文件
2. 压缩/加密各个节区（.text、.data等）
3. 添加一个新的节区，存放脱壳代码
4. 修改入口点，指向脱壳代码
5. 修改PE头信息
6. 保存加壳后的文件

**脱壳过程（运行时）：**
1. 程序从脱壳代码开始执行
2. 申请内存空间
3. 解密/解压各个节区
4. 还原导入表、重定位表等
5. 跳转到原始入口点（OEP）
6. 原始程序开始执行

### 62.1.3 壳的分类

壳可以分为几大类：

```
壳的分类
│
├── 压缩壳（Compression Packer）
│   ├── 原理：压缩代码和数据
│   ├── 目的：减小文件体积
│   ├── 特点：简单，容易脱
│   ├── 代表：UPX、ASPack、PECompact
│   └── 免杀效果：一般
│
├── 加密壳（Encryption Packer）
│   ├── 原理：加密代码和数据
│   ├── 目的：保护代码、防逆向
│   ├── 特点：有加密算法，较难脱
│   ├── 代表：ASProtect、Armadillo、PELock
│   └── 免杀效果：较好
│
├── 虚拟机壳（VM Protector）
│   ├── 原理：把代码变成虚拟机指令
│   ├── 目的：高强度保护
│   ├── 特点：非常难逆向，运行慢
│   ├── 代表：VMProtect、Themida、Obsidium
│   └── 免杀效果：很好
│
└── 其他
    ├── 混淆壳
    ├── 反调试壳
    ├── 授权壳（带注册功能）
    └── ...
```

### 62.1.4 加壳为什么能免杀？

1. **隐藏原始代码** - 杀软扫描的是加壳后的文件，看不到原始代码
2. **改变特征码** - 加壳后文件结构变了，特征码就失效了
3. **增加分析难度** - 杀软需要先脱壳才能分析，增加了成本
4. **启发式干扰** - 壳的代码可能干扰启发式检测

但是要注意：
**加壳不是万能的。**

- 常见的壳（如UPX），杀软都能自动脱壳
- 强壳虽然免杀效果好，但也有对应的脱壳方法
- 加壳只是免杀的手段之一，通常需要配合其他方法

## 62.3 花指令免杀

### 62.3.1 什么是花指令？

花指令（Junk Code），
就是一些没用的、
不影响程序功能的
垃圾代码。

在程序中加入花指令，
可以：
- 改变程序的特征码
- 增加静态分析难度
- 干扰反汇编
- 干扰杀软的特征匹配

```
原始代码：
  mov eax, 1
  add eax, ebx
  ret

加入花指令后：
  mov eax, 1
  nop                ← 花指令（没用的指令）
  push ecx
  pop ecx             ← 花指令（压栈又弹出，等于没做）
  add eax, ebx
  jmp skip
    xor eax, eax      ← 花指令（永远不会执行到）
  skip:
  ret
```

### 62.3.2 花指令的原理

花指令的核心是：
**加了等于没加，不影响功能，但改变了代码结构。**

常见的花指令类型：

**1. 无效指令**
```asm
nop          ; 空操作
xchg eax, eax ; 自己和自己交换，等于没做
mov eax, eax ; 自己赋值给自己
```

**2. 互补指令**
```asm
push ecx
pop ecx      ; 压栈又弹出，等于没做

inc eax
dec eax      ; 加1又减1，等于没做
```

**3. 永远不会执行的代码**
```asm
jmp skip
  ; 这里的代码永远不会执行
  xor eax, eax
  mov ebx, ecx
  ...
skip:
```

**4. 垃圾数据**
在代码段中插入一些
随机的字节数据，
但用跳转指令绕过去。

### 62.3.3 花指令的效果

花指令为什么能免杀？

1. **改变特征码** - 插入花指令后，代码的字节序列变了，特征码就不匹配了
2. **干扰反汇编** - 花指令可能让反汇编器出错，影响静态分析
3. **增加代码大小** - 代码变多了，杀软分析起来更费劲
4. **干扰启发式** - 花指令可能干扰启发式检测的判断

### 62.3.4 花指令的局限

花指令虽然简单，
但也有局限性：

1. **效果有限** - 只能对付简单的特征码检测
2. **容易被识别** - 有经验的分析者一眼就能看出来
3. **行为检测无效** - 对行为检测、沙箱检测没用
4. **不能加太多** - 加太多会影响程序性能，反而更可疑

### 62.3.5 花指令生成

有一些工具可以
自动生成花指令：

- **花指令生成器** - 各种小工具
- **OllyDbg插件** - 调试器插件
- **编译器选项** - 某些编译器可以加花

但更靠谱的是
自己写汇编代码，
手动插入花指令。

## 62.4 代码混淆技术

### 62.4.1 什么是代码混淆？

代码混淆（Code Obfuscation）
就是把代码变得
难以理解、难以分析，
但功能保持不变。

```
原始代码（清晰易懂）：
  int sum = 0;
  for (int i = 0; i < 100; i++) {
      sum += i;
  }
  return sum;

混淆后（乱七八糟，但功能一样）：
  int v1 = 0x1234;
  int v2 = v1 ^ 0x1234;
  int v3 = v2 + (0xff & ~0xff);
  goto label3;
label1:
  v3 += (v2++ ^ 0) + 0;
  goto label4;
label2:
  v2 = v2 * 2 / 2;
  goto label1;
label3:
  if (v2 < 100 + (5 - 5)) goto label2;
label4:
  return v3;
```

### 62.4.2 常见混淆技术

**1. 控制流混淆**
- 插入多余的跳转
- 把简单的逻辑变复杂
- 扁平化控制流
- 不透明谓词（永远为真/假的条件）

**2. 数据混淆**
- 变量拆分和合并
- 数组混淆
- 字符串加密
- 编码变换

**3. 名字混淆**
- 把有意义的变量名、函数名
  变成无意义的名字（a、b、c、func1、func2...）
- 常见于高级语言的混淆

**4. 指令混淆**
- 等价指令替换
- 指令拆分
- 插入花指令
- 寄存器替换

### 62.4.3 常见混淆工具

**针对.NET程序：**
- **ConfuserEx** - 开源的.NET混淆器，功能强大
- **dotfuscator** - 商业.NET混淆器
- **Obfuscar** - 开源.NET混淆器

**针对Java程序：**
- **ProGuard** - Java混淆器
- **Allatori** - Java混淆器
- **DashO** - Java混淆器

**针对Native程序：**
- **VMProtect** - 虚拟机保护，也算混淆
- **Themida** - 虚拟化保护
- **Code Virtualizer** - 代码虚拟化

**针对脚本：**
- JavaScript混淆器
- PowerShell混淆器
- Python混淆器
- ...

### 62.4.4 混淆和免杀的关系

代码混淆主要目的是
**防逆向工程**，
但也有一定的免杀效果：

1. **改变特征码** - 混淆后代码结构变了
2. **干扰静态分析** - 杀软分析起来更困难
3. **增加启发式难度** - 奇怪的代码可能干扰启发式判断

但混淆的免杀效果
不如专门的免杀技术。
混淆更多是
增加逆向分析的难度。

## 62.5 字符串加密

### 62.5.1 什么是字符串加密？

很多恶意程序
会有一些特征字符串，
比如：
- "cmd.exe"
- "powershell.exe"
- 恶意域名
- 注册表路径
- 等等...

杀软经常把这些字符串
作为特征码。

字符串加密就是
把这些敏感字符串
加密存储，
运行时再解密使用。

```
原始代码：
  char* cmd = "cmd.exe";      ← 明文存储，容易被匹配
  WinExec(cmd, 0);

加密后：
  // 加密后的字符串（比如异或加密）
  char encrypted[] = {0x63, 0x5f, 0x64, 0x5e, 0x27, 0x65, 0x78};
  char key = 0x02;

  // 运行时解密
  for (int i = 0; i < 7; i++) {
      encrypted[i] ^= key;       ← 运行时才解密
  }

  WinExec(encrypted, 0);
```

### 62.5.2 为什么字符串加密有效？

1. **消除字符串特征** - 明文字符串是很重要的特征来源
2. **增加静态分析难度** - 杀软静态扫描看不到原始字符串
3. **干扰启发式** - 没有明显的恶意字符串，启发式可能不报警

### 62.5.3 常见加密方式

**1. 异或加密（XOR）**
最简单的加密方式，
用一个密钥和每个字节异或。

```c
// 加密
for (int i = 0; i < len; i++) {
    data[i] ^= key;
}

// 解密（异或是对称的，加密和解密一样）
for (int i = 0; i < len; i++) {
    data[i] ^= key;
}
```

优点：简单、代码少、速度快
缺点：强度低，容易被识别

**2. 加法/减法加密**
```c
// 加密
for (int i = 0; i < len; i++) {
    data[i] += key;
}

// 解密
for (int i = 0; i < len; i++) {
    data[i] -= key;
}
```

**3. 自定义算法**
自己设计加密算法，
比如：
- 多轮异或
- 位运算组合
- 查表替换
- 等等...

**4. AES等标准算法**
用标准的加密算法，
强度高但代码量大。

### 62.5.4 使用建议

1. **加密所有敏感字符串** - 不要漏掉任何一个
2. **解密时机要晚** - 用的时候再解密，用完可以再加密回去
3. **密钥不要太明显** - 不要用简单的0xFF、0xAA这种
4. **解密函数要混淆** - 不要让杀软一眼看出是字符串解密
5. **配合其他方法** - 字符串加密只是免杀的一部分

## 62.6 入口点修改（OEP）

### 62.6.1 什么是入口点？

入口点（Entry Point），
就是程序开始执行的地方。

PE文件头里
有一个字段叫AddressOfEntryPoint，
记录了入口点的RVA
（相对虚拟地址）。

```
PE文件结构：
┌─────────────┐
│   PE头      │
│  + 入口点地址  │ ← 记录了程序从哪里开始执行
├─────────────┤
│   .text     │ ← 代码段
│  入口点代码 →│
│   ...       │
├─────────────┤
│   .data     │ ← 数据段
└─────────────┘
```

### 62.6.2 为什么修改入口点？

1. **改变特征码** - 入口点的代码常常是特征码的位置
2. **干扰分析** - 奇怪的入口点可能干扰分析工具
3. **配合其他免杀** - 比如加壳、SMC等

### 62.6.3 怎么修改入口点？

**方法一：添加新节区**
1. 添加一个新的节区
2. 把入口代码复制到新节区
3. 在新节区最后跳转到原入口
4. 修改PE头的入口点为新节区

```
修改前：
  入口 → .text段开头

修改后：
  入口 → 新节区（一些代码） → 跳转到原.text段
```

**方法二：代码移动**
1. 把入口处的代码移到别的地方
2. 入口点改成跳转指令
3. 跳转到移动后的代码

**方法三：加壳**
加壳工具本质上
也是修改了入口点，
让程序从脱壳代码开始执行。

### 62.6.4 注意事项

1. **不要破坏程序功能** - 修改后要确保程序能正常运行
2. **注意重定位** - 如果有重定位，要处理好
3. **注意导入表** - 确保导入的函数能正确解析
4. **不要太刻意** - 改得太奇怪反而更可疑

## 62.7 资源加密与修改

### 62.7.1 什么是资源？

PE文件里有一个
专门的节区叫.rsrc（资源节），
里面存放各种资源：

### 62.7.2 为什么修改资源？

1. **改变文件特征** - 资源也是文件特征的一部分
2. **伪装成正常软件** - 改成正常软件的图标、版本信息
3. **消除资源特征** - 某些杀软会检查资源中的特征

### 62.7.3 常见修改方法

**1. 修改图标**
把木马的图标
改成正常软件的图标
（比如Word、Excel、PDF的图标）。

工具：
- Resource Hacker
- Restorator
- PE Resource Editor

**2. 修改版本信息**

比如：
- 公司名：Microsoft Corporation

- ...

**3. 修改其他资源**
- 菜单
- 对话框
- 字符串表
- 等等...

**4. 资源加密**
把资源加密，
运行时再解密加载。

### 62.7.4 注意事项

1. **不要改得太假** - 比如图标是Word，但文件名是xxx.exe，就很假
2. **配合社会工程学** - 图标和文件名、扩展名要配合
3. **版本信息要合理** - 不要写得太离谱
4. **只是辅助手段** - 资源修改的免杀效果有限，主要是为了社会工程学

## 62.8 分段执行/代码分块

### 62.8.1 什么是分段执行？

分段执行，就是把
连续的代码拆成很多小块，
分散到不同的地方，
用跳转指令连起来。

```
原始代码（连续的）：
  指令1
  指令2
  指令3
  指令4
  指令5
  ...

分段后（分散的）：
  指令1
  jmp block2

  ...（其他代码或垃圾数据）

block3:
  指令3
  jmp block4

  ...

block2:
  指令2
  jmp block3

  ...

block4:
  指令4
  jmp block5
  ...
```

### 62.8.2 为什么要分段执行？

1. **破坏特征码** - 特征码通常是连续的字节，分段后就不连续了
2. **增加分析难度** - 代码跳来跳去，分析起来很费劲
3. **干扰反汇编** - 可能让反汇编工具出错

### 62.8.3 实现方法

**1. 手动拆分**
- 把代码分成很多小块
- 每块末尾加一个跳转
- 跳到下一块
- 块之间插入垃圾数据

**2. 函数级拆分**
- 把一个函数拆成多个小函数
- 互相调用
- 增加调用链的复杂度

**3. 控制流扁平化**
- 更高级的混淆技术
- 把所有代码块放在一个大switch里
- 用状态变量控制执行顺序
- 控制流变得非常复杂

### 62.8.4 注意事项

1. **不要影响性能** - 跳太多会影响运行速度
2. **不要太明显** - 跳来跳去的代码反而更可疑
3. **配合其他技术** - 单独使用效果有限

## 62.9 自修改代码（SMC）

> **生活类比：信纸上的隐形墨水**

### 62.9.1 什么是SMC？

SMC（Self-Modifying Code，自修改代码），
就是程序在运行过程中
修改自己的代码。

```
运行前（磁盘上）：
  代码段：加密的代码

运行时：
  1. 程序开始执行
  2. 解密/修改代码段的内容
  3. 执行修改后的代码
```

### 62.9.2 SMC的原理

**基本流程：**
1. 代码在磁盘上是加密的
2. 程序运行时，先解密代码
3. 然后执行解密后的代码

**为什么能免杀？**
1. **静态扫描看不到原始代码** - 磁盘上的代码是加密的
2. **特征码失效** - 加密后的代码和原始代码完全不同
3. **增加分析难度** - 杀软需要模拟执行才能看到真实代码

### 62.9.3 简单的SMC示例

```asm
; 原始代码（假设是这样）：
  mov eax, 1234
  ret

; SMC处理后（磁盘上）：
  ; 先解密代码
  mov ecx, 6          ; 代码长度
  mov esi, offset target_code
decrypt_loop:
  xor byte ptr [esi], 0xAA  ; 异或解密
  inc esi
  loop decrypt_loop

  ; 跳转到解密后的代码
  jmp target_code

  ; 加密后的代码
target_code:
  db 0x80 ^ 0xAA, 0xC0 ^ 0xAA, ...  ; 加密的字节
```

### 62.9.4 SMC的注意事项

1. **内存属性** - 代码段默认是只读的，修改前要先改成可写
2. **缓存问题** - CPU有指令缓存，自修改代码要注意缓存一致性
3. **不要太明显** - 典型的SMC模式可能被启发式检测
4. **配合其他技术** - SMC常和其他免杀技术配合使用

## 62.10 等价指令替换

### 62.10.1 什么是等价指令替换？

等价指令替换，就是
把一条指令换成
功能相同但字节不同的
另一条指令（或几条指令）。

这样可以
改变代码的字节序列，
从而绕过特征码检测，
但功能完全不变。

### 62.10.2 常见等价替换

**示例1：mov eax, 0**
```asm
; 原始
mov eax, 0

; 等价替换1
xor eax, eax

; 等价替换2
sub eax, eax

; 等价替换3
and eax, 0
```

**示例2：inc eax**
```asm
; 原始
inc eax

; 等价替换1
add eax, 1

; 等价替换2
lea eax, [eax+1]
```

**示例3：mov eax, ebx**
```asm
; 原始
mov eax, ebx

; 等价替换1
push ebx
pop eax

; 等价替换2
lea eax, [ebx]

; 等价替换3
xchg eax, ebx
xchg eax, ebx  ; 交换两次等于没换，但字节变了
```

**示例4：跳转指令**
```asm
; 原始
jz target

; 等价替换
jnz skip
jmp target
skip:
```

**示例5：寄存器替换**
```asm
; 原始（用eax）
mov eax, 100
add eax, 200

; 等价替换（换成ebx）
mov ebx, 100
add ebx, 200
; （前提是后面不用eax，或者调整好）
```

### 62.10.3 为什么有效？

1. **改变特征码** - 指令的字节变了，特征码就不匹配了
2. **保持功能不变** - 程序还能正常运行
3. **不影响性能** - 大部分等价指令性能差不多

### 62.10.4 注意事项

1. **确保真的等价** - 不要改出bug来
2. **注意标志位** - 有些指令会影响标志位，有些不会
3. **注意寄存器** - 替换后不要影响其他代码
4. **系统调用号** - 如果是Linux的int 0x80，注意eax里的调用号

## 62.11 反调试与反沙箱

> **生活类比：发现有人在监视你**

### 62.11.1 什么是反调试？

反调试（Anti-Debugging），
就是检测程序是不是
被调试器（如OllyDbg、x64dbg、GDB）
附加了。

如果检测到调试器，
程序就：
- 退出
- 崩溃
- 执行假代码
- 等等...

### 62.11.2 常见反调试技术

**1. IsDebuggerPresent**
Windows API，
直接检测有没有调试器。

```c
#include <windows.h>
if (IsDebuggerPresent()) {
    // 检测到调试器，退出
    ExitProcess(0);
}
```

**2. CheckRemoteDebuggerPresent**
另一个API，
功能类似。

```c
BOOL isDebugger = FALSE;
CheckRemoteDebuggerPresent(GetCurrentProcess(), &isDebugger);
if (isDebugger) {
    ExitProcess(0);
}
```

**3. PEB BeingDebugged标志**
PEB（进程环境块）里
有一个BeingDebugged标志，
可以直接检查。

```asm
mov eax, fs:[30h]   ; PEB地址
mov al, [eax+2]     ; BeingDebugged
test al, al
jne debug_detected
```

**4. NtQueryInformationProcess**
用Native API
查询进程信息，
检测调试端口。

**5. 时间差检测**
正常执行一段代码
用的时间很短，
但如果是单步调试，
时间会很长。

```c
DWORD start = GetTickCount();
// 执行一些代码
DWORD end = GetTickCount();
if (end - start > 1000) {  // 超过1秒，可能在调试
    ExitProcess(0);
}
```

**6. 硬件断点检测**
检查有没有硬件断点。

**...还有很多很多**

### 62.11.3 反沙箱

沙箱（Sandbox）是
杀软用来分析可疑程序的
虚拟环境。

反沙箱就是
检测程序是不是在
沙箱环境中运行。

如果是沙箱，
就不执行恶意代码。

**常见反沙箱技术：**

**1. 检测内存大小**
沙箱的内存通常比较小。

```c
MEMORYSTATUSEX mem;
mem.dwLength = sizeof(mem);
GlobalMemoryStatusEx(&mem);
if (mem.ullTotalPhys < 2 * 1024 * 1024 * 1024) {
    // 内存小于2G，可能是沙箱
    ExitProcess(0);
}
```

**2. 检测CPU核心数**
沙箱通常CPU核心数少。

```c
SYSTEM_INFO si;
GetSystemInfo(&si);
if (si.dwNumberOfProcessors < 2) {
    // CPU少于2核，可能是沙箱
    ExitProcess(0);
}
```

**3. 检测磁盘大小**
沙箱的磁盘通常比较小。

**4. 检测鼠标移动**
沙箱里可能没有真实的鼠标操作。

**5. 延迟执行**
沙箱通常不会等太久，
如果程序延迟几分钟再执行，
沙箱可能已经停止分析了。

```c
// 睡5分钟再执行
Sleep(5 * 60 * 1000);
// 然后执行恶意代码
```

**6. 检测常见沙箱特征**
- 沙箱的文件名
- 沙箱的窗口名
- 沙箱的进程名
- 沙箱的DLL
- 等等...

### 62.11.4 反调试/反沙箱和免杀的关系

很多杀软的分析引擎
就是调试器或沙箱。
如果程序能
**检测到分析环境并停止恶意行为**，
杀软就会认为它是安全的。

但这是一把双刃剑：
- 如果做得太明显，杀软可能反而认为可疑
- 高级的沙箱会对抗反沙箱技术
- 需要平衡隐蔽性和有效性

## 62.12 PE文件结构基础（免杀必备）

> **核心类比：PE结构 = Windows程序的"身份证"**
>
> 每个Windows程序（.exe/.dll）都有一个"身份证"——PE结构。这张身份证上写明了：这个程序是谁（DOS头）、入口在哪（EP）、代码放在哪（.text段）、数据放在哪（.data段）等等。
>
> 免杀为什么必须懂PE结构？因为你要**改身份证**：
> - 改出生地（入口点修改）→ 杀软查不到"原籍"
> - 加层信封（加壳）→ 杀软看不到身份证内容
> - 改身份证号（修改PE头字段）→ 杀软的特征库里找不到匹配
>
> ```
> PE文件 = 信封(PE头) + 身份证(各节区信息) + 实际内容(.text代码/.data数据)
>       = DOS头 → NT头 → 节表 → 各节区数据
> ```

### 62.12.1 为什么要学PE结构？

做免杀，
必须懂PE文件结构。
因为：
- 加壳要改PE结构
- 修改资源要懂PE结构
- 修改入口点要懂PE结构
- 添加节区要懂PE结构
- 很多免杀技术都要操作PE文件

PE（Portable Executable）
是Windows平台的
可执行文件格式。
.exe、.dll、.sys、.ocx
都是PE格式。

### 62.12.2 PE结构概览

```
PE文件结构
│
├── DOS头（IMAGE_DOS_HEADER）
│   └── e_lfanew → 指向PE头
│
├── DOS存根（DOS Stub）
│   └── "This program cannot be run in DOS mode"
│
├── PE头（IMAGE_NT_HEADERS）
│   ├── Signature（"PE\0\0"）
│   ├── 文件头（IMAGE_FILE_HEADER）
│   │   ├── Machine（机器类型）
│   │   ├── NumberOfSections（节区数量）
│   │   └── Characteristics（特征）
│   └── 可选头（IMAGE_OPTIONAL_HEADER）
│       ├── AddressOfEntryPoint（入口点）
│       ├── ImageBase（基地址）
│       ├── SectionAlignment（节区对齐）
│       ├── FileAlignment（文件对齐）
│       ├── DataDirectory（数据目录）
│       └── ...
│
├── 节区表（Section Table）
│   ├── .text（代码段）
│   ├── .data（数据段）
│   ├── .rdata（只读数据）
│   ├── .rsrc（资源段）
│   ├── .reloc（重定位表）
│   └── ...（其他节区）
│
└── 各个节区的实际数据
```

### 62.12.3 重要字段

**DOS头：**
- `e_magic` - MZ标志（"MZ"）
- `e_lfanew` - PE头的偏移

**文件头：**
- `Machine` - 机器类型（x86是0x14c，x64是0x8664）
- `NumberOfSections` - 节区数量
- `Characteristics` - 文件属性

**可选头：**
- `AddressOfEntryPoint` - 入口点RVA
- `ImageBase` - 建议加载地址
- `SectionAlignment` - 内存中的节区对齐
- `FileAlignment` - 文件中的节区对齐
- `SizeOfImage` - 内存映像大小
- `DataDirectory` - 数据目录表（导入表、导出表、资源表等）

**节区表：**
- `Name` - 节区名（8字节）
- `VirtualSize` - 内存中的大小
- `VirtualAddress` - 内存中的RVA
- `SizeOfRawData` - 文件中的大小
- `PointerToRawData` - 文件中的偏移
- `Characteristics` - 节区属性（可读/可写/可执行）

### 62.12.4 RVA和偏移

免杀中经常要用到
RVA和文件偏移的转换。

**RVA（Relative Virtual Address）**
- 相对虚拟地址
- 相对于ImageBase的偏移
- 内存中用RVA

**文件偏移（File Offset）**
- 在文件中的位置
- 磁盘上用文件偏移

**转换公式：**
```
文件偏移 = RVA - 节区VirtualAddress + 节区PointerToRawData
```

简单说就是：
找到这个RVA属于哪个节区，
然后用RVA减去节区的起始RVA，
再加上节区在文件中的起始偏移。

### 62.12.5 常见操作

做免杀经常要做的
PE操作：

1. **添加节区** - 添加一个新的节区，放代码或数据
2. **修改入口点** - 改AddressOfEntryPoint
3. **修改节区属性** - 给节区加可写、可执行属性
4. **修改资源** - 改图标、版本信息等
5. **修改导入表** - 隐藏导入的函数
6. **添加重定位** - 让程序能在任意地址加载

学好PE结构，
是免杀的基本功。
虽然一开始有点复杂，
但多看看、多练练，
很快就能掌握。

## 📚 案例1：UPX加壳与脱壳演示

### 场景描述
用UPX给一个程序加壳，
然后再脱壳，
理解加壳的原理。

### 准备工作

```bash
# 1. 准备一个测试程序（比如notepad.exe，或者自己写的小程序）
# 2. 下载UPX工具
# 3. 准备调试器（x32dbg/x64dbg或OllyDbg）
```

### 加壳操作

```bash
# 查看UPX帮助
upx --help

# 加壳（默认压缩级别）
upx test.exe -o test_packed.exe

# 最高压缩比加壳
upx -9 test.exe -o test_packed.exe

# 查看压缩信息
upx -l test_packed.exe
```

加壳后，
文件大小会明显变小。

### 测试加壳后的程序

```
1. 运行 test_packed.exe
2. 确认程序功能正常
3. 用杀软扫描一下
4. 对比加壳前后的检出率
```

### 脱壳操作（命令行）

```bash
# UPX自带脱壳功能
upx -d test_packed.exe -o test_unpacked.exe

# 脱壳后对比原始文件
# 应该基本一样（可能有细微差别）
```

### 手动脱壳（学习用）

手动脱壳更能理解原理：

```
1. 用调试器加载加壳后的程序
2. 在入口点停下
3. 单步跟踪，找到OEP（原始入口点）
4. 到达OEP后，用工具dump内存
5. 修复导入表
6. 得到脱壳后的程序
```

UPX的壳比较简单，
适合用来学习脱壳。

### 经验总结
1. UPX是最基础的加壳工具，适合入门
2. 加壳后文件变小，但免杀效果有限
3. 学习加壳原理对理解免杀很有帮助
4. 手动脱壳是逆向和免杀的基本功
5. 加壳只是免杀的第一步，还要配合其他技术

## 📚 案例2：花指令免杀实战

### 场景描述
给一段代码加入花指令，
看看能不能绕过特征码检测。

### 原始代码

假设我们有一段
有特征码的代码：

```asm
; 原始代码（有特征码）
mov eax, 0x12345678
add eax, ebx
sub eax, ecx
ret
```

杀软的特征码
可能就是这一段的字节。

### 加入花指令

我们来加一些花指令：

```asm
; 加入花指令后的代码
nop                  ; 花指令1：空操作
push ebx
pop ebx              ; 花指令2：压栈又弹出

mov eax, 0x12345678

jmp skip1
  xor eax, eax       ; 花指令3：永远不会执行
  mov ebx, ecx       ; 花指令3：永远不会执行
skip1:

add eax, ebx

xchg eax, eax        ; 花指令4：自己和自己交换

sub eax, ecx

push edx
pop edx              ; 花指令5：压栈又弹出

ret
```

### 效果分析

加了花指令后：
- 代码的字节序列完全变了
- 原来的特征码不匹配了
- 但功能完全一样
- 体积稍微变大了一点

### 注意事项

1. **花指令不能影响功能** - 加之前想清楚，加完要测试
2. **不要太规律** - 全是nop的话，杀软也能识别
3. **分散插入** - 不要集中在一个地方
4. **配合其他方法** - 花指令+字符串加密+加壳效果更好

### 进阶：更复杂的花指令

```asm
; 看起来像真的，但实际没用的代码
call $+5
pop eax
sub eax, 5
; （获取当前指令地址，但后面没用）

; 条件永远为真/假的跳转
xor eax, eax
jz real_code
  ; 假代码
  ...
real_code:

; 用宏定义生成随机花指令
; 每次编译都不一样
```

### 经验总结
1. 花指令是最简单的免杀技术之一
2. 原理简单，容易上手
3. 效果有限，主要对付特征码检测
4. 可以用工具自动生成，也可以手动加
5. 配合其他免杀技术效果更好

## 📚 案例3：字符串加密绕过特征码

### 场景描述
用字符串加密
隐藏敏感字符串，
绕过基于字符串的特征检测。

### 原始代码

```c
#include <windows.h>

int main() {
    // 明文字符串，很容易被特征匹配
    char* cmd = "cmd.exe";
    char* args = "/c calc.exe";

    WinExec(cmd, SW_SHOW);
    // 或者执行其他恶意操作...

    return 0;
}
```

杀软可能会匹配
"cmd.exe"、"calc.exe"
这些字符串。

### 异或加密版本

```c
#include <windows.h>

// 加密函数
void xor_encrypt(char* data, int len, char key) {
    for (int i = 0; i < len; i++) {
        data[i] ^= key;
    }
}

int main() {
    // 加密后的字符串（实际中可以预先生成）
    char cmd[] = {'e', 'o', 'f', '0', 'g', 'y', 'e', 0};  // "cmd.exe" ^ 0x02
    char args[] = {'.', 'a', '}', ' ', 'e', 'q', 'l', 'd', '.', 'h', 'z', 'g', 0};  // "/c calc.exe" ^ 0x02
    char key = 0x02;

    // 运行时解密
    xor_encrypt(cmd, 7, key);
    xor_encrypt(args, 11, key);

    // 使用解密后的字符串
    WinExec(cmd, SW_SHOW);

    // 用完可以再加密回去（可选）
    xor_encrypt(cmd, 7, key);
    xor_encrypt(args, 11, key);

    return 0;
}
```

### 如何生成加密字符串？

写一个小工具：

```c
#include <stdio.h>
#include <string.h>

int main(int argc, char* argv[]) {
    if (argc < 3) {
        printf("Usage: %s <string> <key>\n", argv[0]);
        return 1;
    }

    char* str = argv[1];
    char key = (char)strtol(argv[2], NULL, 16);
    int len = strlen(str);

    printf("char encrypted[] = {");
    for (int i = 0; i < len; i++) {
        printf("0x%02X, ", str[i] ^ key);
    }
    printf("0};\n");  // 结束符

    return 0;
}
```

使用：
```bash
./encrypt "cmd.exe" 02
# 输出：char encrypted[] = {0x65, 0x6F, 0x66, 0x30, 0x67, 0x79, 0x65, 0};
```

### 进阶：更复杂的加密

```c
// 多轮加密
void complex_encrypt(char* data, int len, char* key, int key_len) {
    for (int i = 0; i < len; i++) {
        data[i] ^= key[i % key_len];  // 循环密钥
        data[i] += 0x10;              // 再加一步
        data[i] = ~data[i];           // 取反
    }
}
```

### 经验总结
1. 字符串加密是非常实用的免杀技术
2. 实现简单，效果明显
3. 异或加密最常用，代码少、速度快
4. 要加密所有敏感字符串，不要漏掉
5. 解密时机要注意，最好用的时候再解密

## 📚 案例4：代码混淆工具使用

### 场景描述
使用代码混淆工具
混淆一个.NET程序，
观察混淆前后的变化。

### 工具准备

我们用ConfuserEx，
一个开源的.NET混淆器。

```
1. 下载ConfuserEx
2. 准备一个.NET程序（比如C#写的简单程序）
3. 准备反编译工具（如dnSpy、ILSpy）
```

### 混淆前

先看看混淆前的代码：

```csharp
// 原始C#代码
using System;

namespace TestApp
{
    class Program
    {
        static void Main(string[] args)
        {
            int result = CalculateSum(1, 100);
            Console.WriteLine("Sum: " + result);
        }

        static int CalculateSum(int start, int end)
        {
            int sum = 0;
            for (int i = start; i <= end; i++)
            {
                sum += i;
            }
            return sum;
        }
    }
}
```

用反编译工具看，
代码清清楚楚。

### 使用ConfuserEx混淆

**方法一：图形界面**
1. 打开ConfuserEx
2. 拖入要混淆的程序
3. 选择混淆级别
4. 点击保护

**方法二：命令行**
```bash
Confuser.CLI.exe -n project.crproj
```

**常见混淆选项：**
- `rename` - 重命名（类名、方法名、变量名）
- `ctrl flow` - 控制流混淆
- `ref proxy` - 引用代理
- `constants` - 常量加密
- `resources` - 资源保护
- `anti debug` - 反调试
- `anti dump` - 反dump
- ...

### 混淆后效果

**重命名后：**
```csharp
using System;

namespace a
{
    internal class b
    {
        private static void c(string[] A_0)
        {
            int d = b.d(1, 100);
            Console.WriteLine("Sum: " + d);
        }

        private static int d(int A_0, int A_1)
        {
            int num = 0;
            for (int i = A_0; i <= A_1; i++)
            {
                num += i;
            }
            return num;
        }
    }
}
```

类名、方法名、变量名
全变成无意义的名字了。

**控制流混淆后：**
控制流变得乱七八糟，
很难看懂逻辑，但功能一样。

**常量加密后：**
字符串、数字常量
都被加密了，
运行时再解密。

### 免杀效果

混淆对免杀有一定效果：
- 改变了代码结构
- 加密了字符串常量
- 干扰了静态分析

但对于.NET程序，
因为是托管代码，
杀软还是有很多方法检测。
通常还要配合其他技术。

### 经验总结
1. 代码混淆主要是防逆向，附带一定免杀效果
2. 不同语言有不同的混淆工具
3. 混淆级别越高，性能影响越大
4. 混淆不是万能的，要配合其他免杀技术
5. 学习混淆原理对理解免杀很有帮助

## 📚 案例5：基础免杀组合拳

### 场景描述
把前面学的各种
基础免杀技术组合起来，
做一个简单的免杀马。

### 原始Payload

假设我们有一个
MSF生成的Meterpreter：
```
msfvenom -p windows/meterpreter/reverse_tcp ...
```

检出率大概95%，
几乎全中。

### 第一步：字符串加密

```
1. 找出Payload里的敏感字符串
2. 全部用异或加密
3. 运行时解密
```

效果：检出率降到70%左右。

### 第二步：加入花指令

```
1. 在代码的关键位置插入花指令
2. 用不同类型的花指令
3. 不要太规律
```

效果：检出率降到60%左右。

### 第三步：等价指令替换

```
1. 找出特征码所在位置
2. 把特征码里的指令换成等价指令
3. 或者用二分法定位特征码，然后修改
```

效果：检出率降到50%左右。

### 第四步：加壳

```
1. 先用ASProtect加密壳
2. 或者用VMProtect虚拟机壳
3. 选择合适的保护级别
```

效果：
- ASProtect：检出率降到30%左右
- VMProtect：检出率降到15%左右

### 第五步：修改资源

```
1. 改成正常软件的图标
2. 添加版本信息（伪装成正常软件）
3. 修改其他资源
```

效果：检出率再降5-10%。

### 最终效果

```
原始：    95% 检出
字符串加密：70%
+花指令：  60%
+等价替换：50%
+加壳：    15-30%
+资源修改：10-25%

最终检出率：10-25%
（具体数值取决于杀软和方法）
```

### 还能怎么提升？

基础免杀的极限
大概就是这样了。
要更好的效果，
需要更高级的技术：

- Shellcode免杀
- 进程注入
- 内存加载
- 无文件落地
- 绕过行为检测
- 对抗EDR
- ...

这些我们后面的章节会学。

### 经验总结
1. 单一免杀技术效果有限，组合拳效果更好
2. 基础免杀能对付一些简单的杀软
3. 对付高级杀软和EDR需要更高级的技术
4. 免杀是一个持续优化的过程
5. 没有最好的方法，只有最适合的方法
