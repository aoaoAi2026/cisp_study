## 💰 支付漏洞

### 1.1 什么是支付漏洞？

**支付漏洞，
是指在支付相关的业务逻辑中存在缺陷，
导致攻击者可以以不正常的价格购买商品，
甚至不花钱就能买到东西。**

简单说就是：
**一分钱买iPhone，
0元逛商场。**

这是电商、支付类系统最严重的漏洞之一，
直接造成经济损失。

> 💰 **大白话说支付漏洞**
>
> 支付漏洞的本质是什么？
> 用大白话说就是把"价格标签"交到了用户手里。
>
> **生活例子：超市收银台的故事**
>
> 你去超市买东西，推着一车东西到收银台。
> 正常的流程是：
> - 收银员扫描商品条码 → 系统自动显示价格 → 你付钱
>
> 但如果这家超市的流程是：
> - 你自己告诉你收银员"这个东西1分钱" → 收银员就真收1分钱
>
> 这听起来很荒谬对吧？
> 但在网站里，这种荒谬的事情每天都在发生！
>
> 因为很多网站的逻辑是这样的：
> - 商品价格在页面上显示的是999元
> - 点击"下单"时，浏览器把"价格=999"一起发给服务器
> - 服务器收到后，直接就用这个价格创建订单
>
> **问题在哪？** 浏览器发来的数据是可以随便改的！
> 我明明看到价格是999元，但我可以让浏览器告诉服务器"价格是0.01元"。
> 服务器如果不验证，就按0.01元让我付钱。
>
> 这就像超市收银员不扫描，而是直接问你"多少钱"。
> 你当然会回答"1分钱"对吧？
>
> **支付漏洞的根本原因**：服务器把"决定价格"的权力交给了用户。

### 1.2 支付漏洞的常见类型

#### 类型1：修改价格/金额

最常见的支付漏洞。

**场景：**
下单的时候，
请求里带着商品价格，
后端直接用这个价格来计算。

比如：
```
POST /api/order/create
{
  "product_id": 1001,
  "price": 999.00,
  "quantity": 1
}
```

攻击者把price改成0.01：
```json
{
  "product_id": 1001,
  "price": 0.01,
  "quantity": 1
}
```

如果后端直接用前端传过来的price，
那就能用一分钱买999元的东西！

**为什么会有这种问题？**
开发觉得：
"前端已经算好价格了，
后端直接用就行。"

大错特错！
**前端传过来的一切都不可信！**
价格应该从数据库里查，
不能用前端传的。

#### 类型2：修改数量

和修改价格类似，
但是改数量。

比如运费是按件算的，
买一件运费10元，
买十件运费100元。

但如果数量可以改成负数呢？
```
quantity: -1
```

有些系统会计算：
总价 = 单价 × 数量 = 999 × (-1) = -999元

结果不但不花钱，
还倒找你钱！

当然这种比较少见，
但确实存在。

#### 类型3：修改优惠金额

有些系统优惠券的金额是前端传的：
```
coupon_discount: 100.00
```

攻击者改成9999.00，
直接把商品价格减成负数。

#### 类型4：修改订单状态

有些系统支付是靠前端通知后端的：
```
POST /api/order/pay_callback
{
  "order_id": 2001,
  "status": "paid",
  "amount": 999.00
}
```

如果后端不验证支付平台的签名，
直接相信前端传的status，
那攻击者直接构造一个"已支付"的请求，
订单就变成已支付了！

#### 类型5：支付逻辑缺陷

有些系统的支付流程有问题，
比如：
- 可以无限使用优惠券
- 可以重复使用一个支付凭据
- 支付回调可以重放
- 等等

### 1.3 支付漏洞的挖掘方法

#### 第一步：抓包，找金额相关参数

下单、支付的过程中抓包，
看看哪些参数和钱有关：
- price、total_price、amount
- discount、coupon_amount
- quantity、num
- pay_amount、real_pay
- ...

#### 第二步：修改参数试试

把这些参数改一改：
- 价格改小（比如改成0.01、0、-1）
- 数量改大改小改负数
- 优惠金额改大
- 运费改小改成0

看看后端有没有验证。

#### 第三步：看支付回调

支付完成后，
支付平台会回调商家的接口。
看看这个回调：
- 有没有验签？
- 能不能重放？
- 能不能修改金额和状态？

#### 第四步：测试各种异常情况

- 价格为0会怎么样？
- 价格为负数会怎么样？
- 数量为0会怎么样？
- 数量很大会怎么样？
- 优惠券用了一次再用一次会怎么样？

### 1.4 支付漏洞的防御

1. **价格从数据库查**：绝对不能用前端传的价格
2. **验证所有金额相关参数**：价格、数量、优惠都要验证
3. **支付回调必须验签**：验证支付平台的签名，确保是真的
4. **订单状态由支付回调驱动**：不能由前端直接修改
5. **防止重放攻击**：每个订单只能支付一次，回调只能处理一次
6. **金额计算放在后端**：所有计算都在后端做，前端只负责展示

## 🔢 验证码漏洞

### 2.1 什么是验证码漏洞？

**验证码漏洞，
是指验证码机制存在缺陷，
导致攻击者可以绕过验证码、
无限次尝试验证码、
或者重复使用验证码。**

验证码是用来防暴力破解、
防机器人的，
如果验证码有问题，
那防护就失效了。

### 2.2 验证码漏洞的常见类型

#### 类型1：验证码可爆破（没有次数限制）

最常见的问题。

比如短信验证码，
4位数字，
总共10000种可能。
如果可以无限次尝试，
那攻击者写个脚本跑一会儿就能跑出来。

**为什么会有这个问题？**
开发忘了加次数限制，
或者限制是前端做的。

#### 类型2：验证码不过期

验证码发了之后，
10分钟内有效很正常。
但如果永远不过期呢？
或者有效期特别长？

那攻击者可以慢慢猜，
或者今天发的验证码明天还能用。

#### 类型3：验证码可重复使用

一个验证码用了一次之后，
应该就失效了。
但如果用了一次还能用第二次、第三次...

那攻击者拿到一个验证码，
就可以一直用。

#### 类型4：验证码回显在响应里

有些系统更离谱，
发送验证码的接口，
响应里直接把验证码返回了！

比如：
```
POST /api/send_sms
{
  "phone": "13800138000"
}
```

响应：
```json
{
  "code": 0,
  "msg": "发送成功",
  "sms_code": "123456"
}
```

那还测什么？
直接从响应里拿验证码就行了。

#### 类型5：验证码太简单/有规律

- 验证码是固定的（比如永远是1234）
- 验证码是手机号后几位
- 验证码和时间有关（当前时间戳的后几位）
- 验证码可以预测

#### 类型6：万能验证码

有些系统为了测试方便，
留了个万能验证码，
比如000000、888888，
上线忘了删。

#### 类型7：图形验证码可绕过

图形验证码（就是那种歪歪扭扭的图片），
可以用OCR识别，
或者用机器学习打码平台绕过。

现在的AI识别准确率很高，
简单的图形验证码已经防不住了。

#### 类型8：验证码绕过（不填也能过）

有些系统：
- 不填验证码也能通过
- 验证码为空也能过
- 把验证码参数删掉就绕过了
- 或者验证码错了也能继续

#### 类型9：短信验证码轰炸

这个不算严格的"漏洞"，
但也是常见问题：
- 发送短信验证码没有频率限制
- 可以无限给同一个手机号发
- 可以给任意手机号发（用来骚扰）

### 2.3 验证码漏洞的挖掘方法

1. **看响应里有没有验证码**：发送验证码的接口，看看响应里有没有回显
2. **测试次数限制**：故意输错N次，看看会不会被锁
3. **测试有效期**：发个验证码，等很久再用，看看还能不能用
4. **测试重复使用**：一个验证码用两次，看看第二次能不能过
5. **测试绕过**：不填验证码、删掉验证码参数，看看能不能过
6. **测试万能验证码**：试试000000、123456、888888之类的
7. **测试频率限制**：短时间内发很多次，看看会不会被限制

### 2.4 验证码漏洞的防御

1. **加次数限制**：同一手机号/IP，每天最多发几次，最多错几次
2. **设置有效期**：验证码5-10分钟内有效
3. **用一次就失效**：验证成功后立即作废
4. **不要回显验证码**：绝对不能在响应里返回验证码
5. **用复杂的验证码**：图形+短信，或者行为验证码（滑块、点选）
6. **频率限制**：发送短信要有频率限制，防轰炸
7. **不要留后门**：上线前删掉万能验证码、测试账号之类的

## 🔑 密码找回漏洞

### 3.1 什么是密码找回漏洞？

**密码找回漏洞，
是指在"忘记密码"、"重置密码"的流程中存在逻辑缺陷，
导致攻击者可以重置任意用户的密码。**

这是非常严重的漏洞，
直接导致账号被盗。

### 3.2 密码找回漏洞的常见类型

#### 类型1：验证码可爆破/绕过

和上面的验证码漏洞类似，
但发生在密码找回的场景。

如果短信验证码可以爆破，
那攻击者就能重置任意用户的密码。

#### 类型2：用户ID可控

密码找回的最后一步，
提交新密码的时候，
请求里带着user_id。

比如：
```
POST /api/reset_password
{
  "user_id": 1001,
  "new_password": "123456",
  "sms_code": "123456"
}
```

如果后端只验证了sms_code对不对，
但没验证这个user_id和sms_code是不是对应的，
那攻击者就可以：
1. 用自己的手机发验证码
2. 提交的时候把user_id改成别人的
3. 用自己的验证码改别人的密码

#### 类型3：验证后跳转到设置密码页面，没有绑定用户

有些流程是这样的：
1. 输入手机号 → 发送验证码
2. 输入验证码 → 验证通过
3. 跳转到设置新密码的页面
4. 输入新密码 → 提交

问题可能出在第4步：
提交新密码的时候，
后端不知道这是哪个用户的重置请求，
或者用户身份是前端传的。

比如：
```
POST /api/set_new_password
{
  "phone": "13800138000",
  "new_password": "123456"
}
```

只要验证通过了（第2步过了），
第4步传谁的phone就能改谁的密码。

#### 类型4：验证码可以重复使用/不会失效

如果密码找回的验证码用了一次还能用，
那攻击者拿到一个验证码就能反复用。

或者，
验证通过后，
重置密码的Token长期有效。

#### 类型5：邮箱验证码/链接问题

如果是通过邮箱找回密码：
- 重置链接里的Token可预测
- 重置链接不会过期
- 重置链接可以改用户
- 重置链接可以重复使用

#### 类型6：回答密保问题

密保问题的答案：
- 可以在用户资料里找到
- 答案太简单（比如生日、家乡）
- 可以无限次尝试

### 3.3 密码找回漏洞的挖掘方法

1. **走完整个密码找回流程**：抓包，看每一步的请求和参数
2. **找用户身份参数**：看哪一步传了user_id、phone、username之类的
3. **修改用户参数试试**：把user_id/phone改成别人的，看看能不能改别人的密码
4. **测试验证码**：能不能爆破、能不能绕过、能不能重复用
5. **测试重置链接/Token**：能不能预测、会不会过期、能不能改用户
6. **测试多账号交叉**：A的验证码能不能用来改B的密码

### 3.4 密码找回漏洞的防御

1. **验证码和用户绑定**：每个验证码只能对应用户，验证和重置必须是同一个用户
2. **用Token关联整个流程**：生成一个唯一的Token，绑定用户，后续步骤都用Token识别用户
3. **Token一次性且有时效**：用了就失效，15分钟内有效
4. **验证身份后再重置**：确保验证的人和重置密码的人是同一个
5. **次数限制**：每天最多找回几次，防止爆破
6. **敏感操作通知**：密码找回后，给用户发邮件/短信通知

## 🏃 并发竞争漏洞

### 4.1 什么是并发竞争漏洞？

**并发竞争漏洞（Race Condition），
是指在多线程/多进程环境下，
多个请求同时操作同一个资源，
由于时序问题导致逻辑错误。**

简单说就是：
**一个东西用了两次，
或者1块钱花了两次。**

举个最经典的例子：
**积分兑换。**

你有100积分，
兑换一个商品需要100积分。
正常流程：
1. 查询积分 → 100，够
2. 扣减积分 → 变成0
3. 发放商品

但如果同时发两个请求呢？
请求1：查询积分 → 100，够
请求2：查询积分 → 100，够
请求1：扣减积分 → 变成0
请求2：扣减积分 → 变成-100（或者也扣成0）
请求1：发放商品
请求2：发放商品

结果：
**100积分兑换了两个商品！**

这就是并发竞争漏洞。

### 4.2 并发竞争漏洞的常见场景

#### 场景1：积分/余额扣减

- 积分兑换
- 余额支付
- 优惠券使用
- 提现

#### 场景2：库存扣减

- 抢购商品
- 限量商品
- 秒杀

如果库存是1，
但同时来了100个请求，
可能会卖出去好几个。

#### 场景3：抽奖/签到

- 每日签到：能不能重复签，领多次奖励
- 抽奖：能不能抽多次只扣一次钱
- 每日任务：能不能重复完成

#### 场景4：验证码/优惠券领取

- 优惠券能不能重复领取
- 新人礼包能不能领多次
- 验证码能不能重复用

### 4.3 并发竞争漏洞的原理

核心原因是：
**查询和修改之间有时间差，
而且没有加锁。**

```
查询余额 → 判断够不够 → 扣减余额
    ↑                      ↑
    |-------时间窗口-------|
```

在这个时间窗口里，
如果有另一个请求也来查询，
它看到的余额还是扣减之前的，
于是判断"够"，
也执行扣减。

结果就是扣了两次，
但只判断了一次。

> 🏃 **大白话说并发竞争漏洞**
>
> 并发竞争，用两个生活场景来理解：
>
> **场景一：抢车位**
> 小区只有一个空车位（只有100积分），
> 你和邻居同时开车回来（两个请求同时到），
> 你们都看到了空车位（都查询到100积分），
> 你们都停了进去（都兑换了商品）。
>
> 结果：一个车位停了两个车！逻辑就崩了。
>
> 为什么会这样？
> 因为在"看到车位"和"停车"之间，有一个时间差。
> 系统在这个时间差里没有加"锁"，导致两个人都以为车位是自己的。
>
> **场景二：自动取款机**
> 你银行卡里有100元。
> 你同时在两台取款机上操作，都取100元。
> 两台取款机都检查余额→都有100元→都吐钱。
> 结果：你取走了200元，但卡里只有100元。
>
> **并发竞争的数学本质**：
> ```
> 请求1：读余额(=100) → 判断(够) → 扣(-100) → 写余额(0)
> 请求2：读余额(=100) → 判断(够) → 扣(-100) → 写余额(-100)
> ```
> 两个请求读到的都是"扣减之前"的值，所以判断都通过了。
>
> 这就是经典的 **"读-判断-写"竞态条件**。
> 防御的关键是让"读+判断+写"变成不可分割的原子操作。

### 4.4 怎么测试并发竞争漏洞？

#### 方法1：Burp Intruder并发发送

用Burp Suite：
1. 抓一个请求
2. 发到Intruder
3. payload用同一个值（比如同一个优惠券码）
4. 设20个线程，同时发
5. 看看成功了多少次

如果成功了不止一次，
说明有并发竞争。

#### 方法2：自己写脚本

用Python、Go等写并发脚本，
同时发N个请求，
看看结果。

Python例子：
```python
import requests
import threading

url = "http://target.com/api/exchange"
cookies = {"session": "xxx"}

def exchange():
    r = requests.post(url, cookies=cookies, json={"coupon_id": 1001})
    print(r.status_code, r.text)

# 同时发20个请求
threads = []
for i in range(20):
    t = threading.Thread(target=exchange)
    threads.append(t)

for t in threads:
    t.start()

for t in threads:
    t.join()
```

#### 方法3：常见的测试点

- 积分兑换：同时兑换多次
- 优惠券使用：同一个优惠券同时用多次
- 签到：同时签到多次
- 提现：同时提现多次
- 支付：同时支付多次
- 库存只有1的商品：同时下单多次

### 4.5 并发竞争漏洞的防御

1. **数据库事务 + 行锁**：用SELECT ... FOR UPDATE加行锁
2. **乐观锁**：加version字段，更新的时候带上version
3. **分布式锁**：Redis、ZooKeeper等实现分布式锁
4. **唯一约束**：数据库加唯一索引，防止重复插入
5. **幂等性设计**：同一个请求发多次和发一次效果一样

## 📦 批量操作漏洞

### 5.1 什么是批量操作漏洞？

**批量操作漏洞，
是指系统没有对批量操作进行限制，
导致攻击者可以批量遍历、批量操作、批量获取数据。**

比如：
- 批量遍历用户ID，获取所有用户信息
- 批量注册账号
- 批量发送短信
- 批量下载文件

### 5.2 常见类型

#### 类型1：批量遍历（水平越权的批量版）

如果单个查询有水平越权，
那批量遍历就是把所有数据都扒下来。

比如：
```
/api/user/info?user_id=1001
```
改成1001、1002、1003...
批量跑一遍，
所有用户信息都拿到了。

#### 类型2：批量注册/批量发送

- 注册没有频率限制 → 可以批量注册小号
- 短信验证码没有频率限制 → 可以短信轰炸
- 邮箱验证没有限制 → 可以发垃圾邮件

#### 类型3：批量导出/下载

- 导出数据没有限制 → 一次性导出所有数据
- 下载文件没有限制 → 批量下载所有文件

#### 类型4：批量爆破

- 登录没有次数限制 → 批量爆破密码
- 验证码没有次数限制 → 批量爆破验证码

### 5.3 防御

1. **频率限制**：每个用户/IP每分钟最多请求多少次
2. **分页限制**：列表页每次最多返回多少条
3. **验证码**：批量操作前需要验证码
4. **次数限制**：每天最多注册几个号、最多发几条短信
5. **监控告警**：发现异常流量及时封禁

## 🔄 接口重放漏洞

### 6.1 什么是接口重放漏洞？

**接口重放漏洞，
是指同一个请求可以被重复发送多次，
每次都执行同样的操作。**

比如：
- 点赞接口：点一次+1，发100次就+100
- 签到接口：发一次签一次，发100次签100次
- 优惠券领取：发一次领一张，发100次领100张

### 6.2 和并发竞争的区别

- **并发竞争**：同时发多个请求，利用时序问题
- **接口重放**：一个一个发，或者隔一段时间发，每次都有效

并发竞争是"时间差"的问题，
接口重放是"唯一性"的问题。

### 6.3 常见场景

- 点赞/投票
- 签到领积分
- 领取优惠券
- 提交订单（重复支付？）
- 表单重复提交

### 6.4 防御

1. **幂等Token**：每个操作生成一个唯一Token，用一次就失效
2. **业务唯一约束**：比如每天只能签到一次，数据库加唯一索引
3. **频率限制**：短时间内不能重复操作
4. **请求签名+时间戳**：防止篡改和重放（有时间窗口）

## 🚶 业务流程绕过

### 7.1 什么是业务流程绕过？

**业务流程绕过，
是指攻击者跳过正常的业务步骤，
直接执行后面的步骤。**

比如正常流程是：
1. 浏览商品
2. 加入购物车
3. 结算
4. 支付
5. 完成

但如果攻击者跳过支付那一步，
直接访问"完成"页面呢？
如果后端没验证，
那就直接下单成功了！

> 🚶 **大白话说业务流程绕过**
>
> 业务流程绕过，就像迷宫里的"翻墙"：
>
> **正常路线**：入口 → 走廊A → 走廊B → 走廊C → 出口
> 但迷宫里有矮墙，你直接翻过去，从入口跳到出口。
>
> 开发以为所有人都会按着迷宫路线走，
> 所以只在入口处查了票，
> 走廊里用的都是矮墙（没验证），
> 攻击者直接翻墙就行了。
>
> **典型的支付流程绕过**：
> ```
> 正常：选商品 → 确认订单 → 支付页面 → 支付 → 支付成功页
> 绕过：选商品 → 确认订单 → 直接访问"支付成功"页的接口
> ```
>
> 如果后端在"支付成功页"没有验证"你真的付了钱吗？"，
> 那攻击者就能跳过支付，直接"白嫖"。
>
> **为什么开发会犯这个错误？**
> 因为他们在设计流程时，假设用户会"按规矩来"。
> 但攻击者不会按规矩来，他们会跳、会翻、会绕。
> 所以记住：**服务端不能假设用户按你的流程走，每一步都要独立验证！**

### 7.2 常见场景

#### 场景1：跳过支付步骤

刚才说的，
直接访问支付成功页面，
或者直接调用支付成功的接口。

#### 场景2：跳过验证步骤

比如注册需要邮箱验证，
但如果直接访问注册成功页面，
或者直接调用设置密码的接口，
跳过验证呢？

#### 场景3：跳过前置条件

比如：
- 抽奖需要先分享，但是直接调用抽奖接口呢？
- 领优惠券需要先关注公众号，但是直接调用领取接口呢？
- 某些功能需要VIP，但直接调用VIP接口呢？

#### 场景4：修改流程状态

有些系统用参数表示流程到了哪一步：
```
?step=1
?step=2
?step=3
```

直接把step改成最后一步，
看看能不能跳过前面的步骤。

### 7.3 挖掘方法

1. **梳理业务流程**：先搞清楚正常流程有几步
2. **跳过步骤试试**：直接访问/调用后面步骤的接口
3. **修改状态参数**：看看有没有step、status、stage之类的参数，改了试试
4. **看每个接口的依赖**：每个接口是不是真的验证了前置条件

### 7.4 防御

1. **每个步骤都验证前置条件**：不能假设用户是按步骤来的
2. **用状态机管理流程**：每个状态只能流转到特定的下一个状态
3. **服务端记录进度**：不要靠前端传step之类的参数

## 🎫 优惠券/积分漏洞

### 8.1 优惠券漏洞

优惠券是薅羊毛的重灾区。

#### 常见问题：
1. **优惠券可以无限领取**：没有次数限制
2. **优惠券可以叠加使用**：本来只能用一张，结果能用多张
3. **优惠券可以重复使用**：用了一次还能用
4. **优惠券金额可以改**：前端传优惠金额
5. **优惠券可以和其他优惠叠加**：导致价格特别低
6. **并发竞争**：同时用多次
7. **优惠券码可以爆破**：如果是随机码但不够随机

### 8.2 积分漏洞

#### 常见问题：
1. **积分可以刷**：比如评论得积分，那刷1000条评论
2. **积分兑换有并发竞争**：100积分换两次
3. **积分可以无限签到**：并发签到
4. **积分充值有漏洞**：充100得1000
5. **积分可以转让/交易**：刷了积分转走

### 8.3 防御

1. **次数限制**：每人最多领几张、每天最多用几张
2. **唯一性**：每个优惠券只能用一次，数据库加唯一约束
3. **叠加规则**：明确规定哪些可以叠加、哪些不行
4. **并发控制**：加锁，防止并发竞争
5. **积分获取要有上限**：每天最多得多少积分

## 🎯 更多逻辑漏洞

除了上面讲的这些，
还有很多很多...

### 9.1 上传相关的逻辑漏洞

- 文件重命名可以控制路径
- 上传数量没有限制（塞满磁盘）
- 可以覆盖任意文件
- 上传的文件不会被清理（越堆越多）

### 9.2 搜索/查询相关的

- 没有权限控制（能搜所有用户的数据）
- SQL注入（这个算技术漏洞了）
- 没有分页限制（一次查全部，拖库）
- 排序参数有问题（order by注入）

### 9.3 分享/邀请相关的

- 邀请奖励可以刷（自己邀请自己）
- 分享链接泄露敏感信息
- 邀请码可以预测

### 9.4 评论/留言相关的

- XSS（技术漏洞）
- 没有审核就发布
- 可以无限发（刷屏）
- 可以发任意内容（垃圾广告）

### 9.5 消息通知相关的

- 短信/邮件轰炸
- 消息内容可以注入（比如注入链接钓鱼）
- 可以给任意用户发消息（骚扰）

### 9.6 导出/报表相关的

- 越权导出别人的数据
- 没有限制，一次性导出全部（拖库）
- 导出的文件里有敏感信息
- 导出的路径可以控制（路径遍历）

...

**逻辑漏洞的种类实在太多了，
每个业务都不一样，
不可能一一列举。**

关键是要培养**业务思维**，
站在产品和开发的角度想：
"这个功能是怎么设计的？
可能哪里会出问题？"

## 🕵️ 逻辑漏洞挖掘思路总结

### 10.1 通用挖掘步骤

#### 第一步：理解业务
先把系统的功能、流程、角色都搞清楚。
不理解业务，
根本不知道哪里可能有问题。

#### 第二步：找关键参数
找所有和业务相关的参数：
- 用户身份类：user_id、uid、phone
- 金额价格类：price、amount、total
- 状态类：status、step、state
- ID类：order_id、product_id、coupon_id
- 数量类：quantity、num、count

#### 第三步：改参数试试
把这些参数改一改：
- 改大改小
- 改成0、改成负数
- 改成别人的ID
- 删掉参数
- 加特殊参数

看看后端有没有验证。

#### 第四步：测异常情况
- 边界值：0、最大值、负数
- 重复操作：一个请求发两次
- 并发操作：同时发多个请求
- 跳过步骤：直接走后面的流程
- 交叉测试：A的东西用B的账号试

#### 第五步：看前后端是否一致
- 前端显示没权限，后端接口呢？
- 前端有验证，后端有没有？
- 前端算的价格，后端有没有重新算？

### 10.2 核心思维

- **永远不要相信前端**：前端传的一切都可能被篡改
- **每个接口都要独立验证**：不要假设用户是按步骤来的
- **凡是涉及钱和数据的都要重点测**
- **业务越复杂，漏洞越多**

> 💡 **大白话说逻辑漏洞挖掘的底层思维**
>
> 学完了所有逻辑漏洞类型，你有没有发现一个共同点？
>
> **所有逻辑漏洞的本质都是一句话：开发假设你会按规矩来，但你偏不。**
>
> 用生活中的场景来理解：
>
> | 逻辑漏洞类型 | 开发的假设 | 你的"不按规矩" |
> |-------------|-----------|--------------|
> | 支付漏洞 | "价格是前端算好传过来的，不会有问题" | 改价格参数为0.01 |
> | 验证码漏洞 | "验证码是短信发的，不可能被猜到" | 写脚本跑完所有4位数字 |
> | 密码找回漏洞 | "验证码对了，那说明就是本人在操作" | 用自己的验证码+改user_id |
> | 并发竞争 | "用户只会点一次按钮" | 同时发20个请求 |
> | 业务流程绕过 | "用户会按照1→2→3→4的步骤走" | 直接从1跳到4 |
>
> **关键思维转变**：
> - 不要想"这个功能应该怎么用" —— 那是产品经理的事
> - 要问"这个功能还能怎么用" —— 这才是渗透测试人员的思维
>
> 在每一个业务环节问自己三个问题：
> 1. **钱在哪里？** 涉及金额、价格、优惠券的地方 → 必然有逻辑漏洞
> 2. **权限在哪里？** 涉及用户身份、数据归属的地方 → 必然有越权
> 3. **流程在哪里？** 涉及多步骤、顺序操作的地方 → 必然有跳过机会
>
> 逻辑漏洞不是靠工具扫出来的，是靠脑子想出来的。
- **多站在开发的角度想**：他可能会忘了什么？

## 📚 案例讲解

### 案例1：0元购买iPhone（支付漏洞）

**场景描述：**
某电商平台存在支付漏洞，
用户可以修改订单价格，
用0.01元购买价值8000元的iPhone。

**漏洞详情：**

用户下单的时候，
前端会计算订单金额，
然后把价格传给后端：
```json
{
  "product_id": 2001,
  "product_name": "iPhone 15 Pro",
  "price": 8999.00,
  "quantity": 1,
  "total_price": 8999.00
}
```

后端直接用前端传的total_price来创建订单，
然后去支付。

**利用过程：**

**第一步：正常下单**
把iPhone加入购物车，去结算，抓包。

**第二步：修改价格**
把total_price改成0.01：
```json
{
  "product_id": 2001,
  "product_name": "iPhone 15 Pro",
  "price": 8999.00,
  "quantity": 1,
  "total_price": 0.01
}
```

**第三步：支付**
用支付宝/微信支付0.01元。

**第四步：订单变成已支付**
支付回调后，订单状态变成已支付。
然后真的发货了！

**为什么会这样？**
- 后端没有验证价格，直接用前端传的
- 觉得"前端已经算好了，后端直接用"
- 商品价格没有从数据库查

**修复方案：**
- 价格必须从数据库查询，绝对不能用前端传的
- 所有金额计算都在后端做
- 创建订单时，根据product_id去数据库查价格，重新计算
- 支付回调验证订单金额和支付金额是否一致

**启示：**
**涉及钱的地方，
绝对不能相信前端！
价格必须从数据库查！**

---

### 案例2：短信验证码爆破，登录任意账号

**场景描述：**
某网站登录支持短信验证码登录，
但验证码没有次数限制，
可以无限次尝试，
导致可以爆破任意用户的验证码。

**漏洞详情：**

登录接口：
```
POST /api/login_by_sms
{
  "phone": "13800138000",
  "sms_code": "123456"
}
```

响应：
- 验证码正确 → 登录成功
- 验证码错误 → 返回"验证码错误"

问题是：
**没有次数限制！**

输错100次也不会锁，
输错10000次也不会锁。

4位数字验证码只有10000种可能，
6位数字有100万种可能，
只要时间够，总能跑出来。

**利用过程：**

**第一步：获取目标手机号**
比如13800138000。

**第二步：发送验证码**
给这个手机号发一个验证码（或者不用发，直接跑历史验证码？）。
其实更简单的是：
用密码找回的验证码，
或者直接在登录页点"发送验证码"。

**第三步：爆破验证码**
用Burp Intruder，
sms_code从0000到9999（4位的话），
开10个线程跑。

运气好的话几分钟就跑出来了。

**第四步：登录成功**
跑出来正确的验证码，
用这个验证码登录，
成功进入对方账号。

**修复方案：**
- 同一手机号，每天最多发送5次验证码
- 同一手机号，验证码连续错误5次就锁定1小时
- 同一IP，每天最多请求20次
- 验证码5分钟内有效
- 用图形验证码/行为验证码防机器

**启示：**
**验证码必须有次数限制，
不然和没有一样。**

---

### 案例3：任意用户密码重置（密码找回漏洞）

**场景描述：**
某网站密码找回功能存在逻辑漏洞，
可以重置任意用户的密码。

**漏洞详情：**

密码找回流程：
1. 输入手机号 → 发送短信验证码
2. 输入验证码 → 验证通过，跳转到设置密码页
3. 输入新密码 → 提交，重置成功

问题出在第3步：
提交新密码的请求是这样的：
```
POST /api/reset_password
{
  "phone": "13800138000",
  "new_password": "123456",
  "sms_code": "123456"
}
```

后端验证了sms_code是对的，
**但没验证sms_code是不是对应这个phone的！**

或者说，
第2步验证通过后，
第3步就不验证了，
你传哪个phone就改哪个phone的密码。

**利用过程：**

**第一步：用自己的手机号发验证码**
自己的手机号：13900139000
点击发送验证码，收到验证码：654321

**第二步：验证自己的验证码**
输入验证码654321，验证通过。
进入设置密码页面。

**第三步：提交的时候改手机号**
抓包，把phone改成目标手机号（比如13800138000）：
```json
{
  "phone": "13800138000",
  "new_password": "hacked123",
  "sms_code": "654321"
}
```

发送...

**第四步：重置成功！**
响应返回"重置成功"。
用13800138000 + hacked123登录，
真的登进去了！

**为什么会这样？**
- 验证验证码和重置密码是两个独立的接口
- 验证通过后没有绑定用户身份
- 重置密码时用户身份是前端传的，后端直接信了

**修复方案：**
- 用Token串联整个流程：验证通过后生成一个Token，绑定用户和过期时间
- 重置密码时用Token识别用户，而不是前端传phone
- Token只能用一次，15分钟有效
- 每个验证码只能对应用户，验证和重置必须是同一个手机号

**启示：**
**身份认证不能靠前端传参数，
要用服务端的Session/Token。**

### 案例4：并发竞争，100积分换两个商品

**场景描述：**
某积分商城存在并发竞争漏洞，
100积分可以兑换两个价值100积分的商品。

**漏洞详情：**

兑换商品的流程：
1. 查询用户积分 → 够不够
2. 如果够，扣减积分
3. 生成订单，发放商品

代码大概是这样的：
```php
// 伪代码
$points = 查询积分($user_id); // 100
if ($points >= 100) {
    扣减积分($user_id, 100); // 变成0
    发放商品($user_id, $goods_id); // 发商品
}
```

看起来没问题对吧？
但如果同时来两个请求呢？

**时序：**
```
时间点 | 请求1                    | 请求2
-------|--------------------------|-------------------------
  T1   | 查询积分 → 100，够       |
  T2   |                          | 查询积分 → 100，够
  T3   | 扣减积分 → 0             |
  T4   |                          | 扣减积分 → -100（或0）
  T5   | 发放商品                 |
  T6   |                          | 发放商品
```

结果：
**扣了100（或者200）积分，
但发了两个商品！**

如果扣减是用`update user set points = points - 100`，
那扣减是对的（扣了200），
但判断的时候都认为够，
所以发了两个商品。
如果用户只有100积分，
扣200就变成负数了。

**利用过程：**

**第一步：准备100积分**
确认账号里有100积分。

**第二步：写并发脚本**
```python
import requests
import threading

url = "http://target.com/api/points/exchange"
cookies = {"session": "xxx"}
data = {"goods_id": 1001}

def exchange():
    r = requests.post(url, cookies=cookies, json=data)
    print(r.text)

# 同时发20个请求
threads = [threading.Thread(target=exchange) for _ in range(20)]
for t in threads:
    t.start()
for t in threads:
    t.join()
```

**第三步：运行脚本**
20个请求同时发出去。

**第四步：查看结果**
去"我的订单"里看，
发现有2个兑换成功的订单！
再看积分，
变成了-100（或者只扣了100，看怎么实现的）。

**修复方案：**
- 用数据库事务 + SELECT ... FOR UPDATE行锁
- 或者用乐观锁：update ... where points >= 100 and version = xxx
- 或者用Redis分布式锁
- 数据库加约束：积分不能为负数

**启示：**
**凡是涉及"判断-修改"模式的，
都要考虑并发问题。
查询和修改必须是原子操作。**

### 案例5：跳过支付步骤，直接"购买成功"

**场景描述：**
某在线课程网站存在业务流程绕过漏洞，
可以跳过支付步骤，直接观看付费课程。

**漏洞详情：**

购买课程的流程：
1. 选择课程
2. 去支付
3. 支付成功 → 跳转到课程页面
4. 观看课程

开发觉得：
"用户只有支付成功了，
才会跳转到课程页面，
所以课程页面不用判断有没有购买。"

或者说，
"用户访问课程页面时，
只要是从支付成功页过来的，
就有权限看。"（靠Referer判断）

**利用过程：**

**第一步：找到课程页面的URL**
比如：
```
http://target.com/course/play?course_id=1001
```

**第二步：直接访问**
登录自己的账号（没买过这个课程），
直接在浏览器输入这个URL。

**第三步：居然能看！**
视频直接播放了，
完全没验证有没有购买。

或者，
需要加个Referer：
```
Referer: http://target.com/pay/success
```
就绕过了。

**为什么会这样？**
- 开发想当然地认为用户会按流程走
- 没有在课程播放页面验证购买状态
- 或者只靠Referer判断，不可靠

**修复方案：**
- 每个需要权限的页面/接口，都要独立验证权限
- 观看课程前，查询用户是否购买了该课程
- 不要相信Referer、不要相信前端跳转
- 服务端记录购买状态，每个请求都验证

**启示：**
**不要假设用户会按你的流程走，
每个接口都要独立验证权限。**

## 🎤 面试官视角：逻辑漏洞考点精讲

逻辑漏洞也是安全面试的高频考点，以下是常见的面试题和最佳回答思路：

> **面试题1："你挖过哪些印象深刻的逻辑漏洞？"**
>
> 用"STAR法则"回答：
> - **S（场景）**：什么系统、什么功能模块
> - **T（目标）**：你测试时关注的点是什么
> - **A（行动）**：你用了什么方法和工具发现的
> - **R（结果）**：漏洞的影响和危害
>
> 比如："我在某电商平台的积分兑换功能发现并发竞争漏洞，用Burp Intruder同时发20个请求，发现100积分可以兑换两个100积分的商品。"
>
> 加分项：顺带提一下你的修复建议。

> **面试题2："支付漏洞，后端怎么防御？"**
>
> 标准回答：
> 1. **价格从数据库查**，绝对不能取前端传的价格
> 2. **服务端计算总价**，商品单价 × 数量，优惠扣减，都在后端算
> 3. **支付回调验签**，不能信任前端通知的支付结果
> 4. **金额一致性校验**：订单金额必须等于支付金额
> 5. **幂等性**：同一个订单不能被重复支付

> **面试题3："并发竞争漏洞怎么解决？"**
>
> 回答层次递进：
> - **应用层**：数据库悲观锁（`SELECT ... FOR UPDATE`）或乐观锁（version字段）
> - **中间件层**：Redis分布式锁 / ZooKeeper
> - **数据库层**：唯一索引约束、金额非负约束
> - **架构层**：队列化，串行处理同一用户的请求
>
> 面试官想听的不只是一个答案，而是你知道不同场景用不同方案。

> **面试题4："逻辑漏洞和技术漏洞有什么区别？"**
>
> 经典回答：
> | 维度 | 技术漏洞 | 逻辑漏洞 |
> |------|---------|---------|
> | 原因 | 代码实现问题（拼接SQL、没转义） | 业务设计问题（假设用户按流程走） |
> | 检测 | 工具可检测（SQLMap、扫描器） | 需要理解业务，难以自动化 |
> | 修复 | 修复代码实现（参数化查询、转义） | 修改业务逻辑、增加验证步骤 |
> | 例子 | SQL注入、XSS、命令注入 | 越权、支付漏洞、流程绕过 |

> 🎯 **记忆口诀（逻辑漏洞七字诀）**
>
> ```
> 支付别信前端价，价格必须后端查
> 验证次数要限制，反复爆破要防住
> 找回密码绑用户，张三的码不改李四
> 并发竞争要加锁，读写改是原子步
> 业务流程步步验，跳过一步拦一步
> 接口重放加Token，重复请求不生效
> 批量操作限频率，薅羊毛的人白忙活
> ```

## 📝 本章小结

这一章我们学习了各种常见的逻辑漏洞，
内容非常多，
让我们来总结一下：

### 支付漏洞
- **核心**：前端传的价格/金额不可信
- **常见类型**：改价格、改数量、改优惠、改订单状态
- **防御**：价格从数据库查，后端计算，支付回调验签

### 验证码漏洞
- **核心**：没有次数限制、可绕过、可回显
- **常见类型**：可爆破、不过期、可重复用、回显、万能验证码
- **防御**：次数限制、有效期、用一次就失效、不回显

### 密码找回漏洞
- **核心**：身份验证和重置密码不绑定
- **常见类型**：用户ID可控、验证码可爆破、Token可预测
- **防御**：Token串联流程，绑定用户，一次性且有时效

### 并发竞争漏洞
- **核心**：查询和修改不是原子操作，多请求同时执行
- **常见场景**：积分兑换、余额支付、库存扣减、签到
- **防御**：行锁、乐观锁、分布式锁、唯一约束

### 其他逻辑漏洞
- **批量操作漏洞**：没有频率/次数限制
- **接口重放漏洞**：同一个请求可以重复生效
- **业务流程绕过**：跳过步骤直接访问后面的接口
- **优惠券/积分漏洞**：薅羊毛各种姿势

### 挖掘思路
1. 先理解业务
2. 找关键参数，改了试试
3. 测异常情况（边界、重复、并发、跳过）
4. 永远不要相信前端

逻辑漏洞种类繁多，
关键是培养业务思维，
多站在开发和产品的角度想问题。

以及整个进阶篇的内容。
