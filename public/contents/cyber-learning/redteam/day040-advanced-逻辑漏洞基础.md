## 📖 本章概述

::: tip 写在前面
很多新手会问：
"逻辑漏洞是不是很简单？
不就是改改参数吗？"

在正式开始之前，先和大家分享一个重要的认知：

**逻辑漏洞是Web安全里最"特别"的一类漏洞。**

前面学的SQL注入、XSS、命令执行、文件包含、CSRF、SSRF——
这些都属于**技术漏洞**。有固定的技术原理、标准的测试方法。

但逻辑漏洞不一样。它不是"技术错误"，而是**"设计错误"**。

打个比方：
- 技术漏洞 = 你家门锁太差，一踹就开（锁的问题）
- 逻辑漏洞 = 你家门锁很好，但隔壁有个没锁的窗户（设计问题）

技术漏洞的攻击者需要技术能力（会写SQL、会HTML注入...），
但逻辑漏洞的攻击者只需要**理解业务流程**和**有一点想象力**。

这也是为什么说：
> **WAF和防火墙挡不住逻辑漏洞。**
> 因为逻辑漏洞的每一个操作都是"合法的"——
> 只是组合起来的结果不合法。

在这一章，你会学到逻辑漏洞的思维方式。
这可能是进阶篇里最有用的一章，
因为**逻辑漏洞在真实渗透中的占比极高，而且WAF拦不住**。

## 🔍 什么是逻辑漏洞？

### 1.1 概念

**逻辑漏洞（Logic Vulnerability），
也叫业务逻辑漏洞，
是指由于程序的业务逻辑设计不当或实现有缺陷，
导致攻击者可以通过操纵业务流程、
修改参数、绕过验证等方式，
完成非预期的操作。**

说人话就是：
> 程序的功能本身是正常的，
> 代码也没什么"注入"之类的问题，
> 但是**设计的时候没想周全**，
> 被人钻了空子。

举个最简单的例子：

**修改密码功能：**
正常流程应该是：
1. 输入旧密码
2. 验证旧密码是否正确
3. 输入新密码
4. 确认新密码
5. 修改成功

但如果程序员偷懒，
把第1、2步给忘了，
直接就是输入新密码就能改，
那会怎么样？

攻击者只要知道你的用户名（或者ID），
就能直接把你密码改了！

这就是逻辑漏洞——
**不是代码写错了，
是逻辑设计错了。**

### 1.2 一个经典的例子：0元购买

我们来看一个最经典的逻辑漏洞：
**支付金额篡改。**

假设某电商网站的下单流程：

**第一步：选商品，加入购物车**
- 商品A，价格1000元
- 数量1件

**第二步：确认订单**
- 提交订单，总金额1000元

**第三步：支付**
- 跳转到支付页面
- 支付1000元

看起来挺正常的对吧？

但是！
如果在第二步提交订单的时候，
把价格参数给改了呢？

比如本来是：
```
price=1000&quantity=1
```

攻击者改成：
```
price=0.01&quantity=1
```

如果服务器没有验证价格，
直接就用前端传过来的价格生成订单，
那会怎么样？

**一分钱买1000块的东西！**

这就是典型的逻辑漏洞——
**服务器太信任前端传过来的数据了。**

### 1.3 逻辑漏洞的本质

逻辑漏洞的本质可以用一句话概括：

> **服务器没有严格验证用户的输入和操作，
> 太信任前端了，
> 导致业务流程被篡改。**

记住一句话：
**永远不要相信前端传过来的任何数据！**

前端的验证只是为了用户体验，
真正的安全验证必须在后端做！

## ⚠️ 逻辑漏洞为什么难防？为什么危害大？

### 2.1 为什么难防？

#### 原因1：WAF防不住

WAF（Web应用防火墙）能防什么？
- SQL注入
- XSS
- 命令执行
- 文件上传
- ...这些有特征的攻击

但逻辑漏洞呢？
它就是正常的业务请求，
参数也都是正常的，
只是值被改了一下。
WAF根本分不清：
- 这个1000元是正常用户传的
- 还是攻击者改的

WAF："我太难了..."

#### 原因2：每个系统的业务都不一样

技术漏洞有通用的Pattern，
比如SQL注入就是单引号报错，
XSS就是<script>标签。

但逻辑漏洞呢？
每个系统的业务逻辑都不一样，
电商系统和社交系统完全不同，
金融系统和OA系统更是天差地别。

没有通用的扫描工具，
只能靠人去理解业务，
去思考，去测试。

#### 原因3：开发不重视

很多开发觉得：
"功能实现了就行，
哪有那么多乱七八糟的攻击。"

或者：
"用户不会这么操作的，
正常用户谁会改价格啊？"

但攻击者不是正常用户啊！

#### 原因4：测试不充分

测试人员一般只测正常流程：
- 正常下单 → 正常支付 → 没问题
- 正常改密码 → 没问题

但很少测异常流程：
- 价格改成负数会怎么样？
- 数量改成0会怎么样？
- 不按顺序走流程会怎么样？
- ...

### 2.2 为什么危害大？

#### 1. 直接触碰核心利益

- **支付漏洞** → 直接损失钱
- **越权漏洞** → 泄露大量用户数据
- **密码找回漏洞** → 直接登录任意用户
- **短信轰炸** → 营销资源被滥用
- **刷积分/刷优惠券** → 经济损失

#### 2. 隐蔽性强

技术漏洞可能还有日志、告警，
逻辑漏洞呢？
看起来都是正常操作，
很难发现。

比如：
- 用户A买了东西，只付了0.01元
- 从日志上看，就是一笔正常订单
- 如果不是财务对账，根本发现不了

#### 3. 利用门槛低

很多逻辑漏洞不需要高深的技术，
只要改个参数就行，
新手也能利用。

比如：
- 改个订单ID就能看别人订单
- 改个价格就能0元买东西
- 改个用户ID就能改别人密码

#### 4. 在护网中很吃香

为什么？
因为：
- 很多单位WAF、防火墙都配好了
- 技术漏洞很难打
- 但业务逻辑一塌糊涂
- 一个逻辑漏洞直接打穿

护网中很多高分漏洞，
都是逻辑漏洞。

## 📋 逻辑漏洞的常见类型

逻辑漏洞种类繁多，
这里我们列举20种最常见的，
让大家有个大概的了解。
后面的章节会详细讲重点的几种。

### 3.1 越权类（最常见）

#### 1. 水平越权
- 用户A能看/操作用户B的数据
- 比如改一下用户ID，就能看别人的订单
- 最常见的逻辑漏洞

#### 2. 垂直越权
- 普通用户能做管理员才能做的事
- 比如普通用户直接访问管理员接口
- 也叫权限绕过

#### 3. 未授权访问
- 不需要登录就能访问需要登录的页面
- 比如直接输入后台URL就能进
- 很多系统都有这个问题

### 3.2 认证/授权类

#### 4. 密码找回漏洞
- 可以重置任意用户的密码
- 比如验证码可以爆破、可以绕过
- 危害极大

#### 5. 验证码漏洞
- 验证码可以重复使用
- 验证码可以无限试错（爆破）
- 验证码太简单（纯数字、4位）
- 验证码可以在前端拿到

#### 6. 登录绕过
- 不需要密码就能登录
- 比如只要用户名对就行
- 或者改个参数就登录成功

#### 7. 会话固定
- 攻击者给用户一个SessionID
- 用户登录后，这个SessionID就变成登录状态
- 攻击者用这个SessionID就能登录

### 3.3 支付/交易类（危害最大）

#### 8. 支付金额篡改
- 修改价格、数量、运费等
- 0元购买、负数购买
- 最经典的逻辑漏洞

#### 9. 订单篡改
- 修改订单状态、订单金额
- 比如把未支付改成已支付
- 或者改订单的商品

#### 10. 优惠券/积分漏洞
- 优惠券可以无限领
- 积分可以无限刷
- 一个优惠券能用多次

#### 11. 并发竞争条件
- 同时发多个请求，利用时间差
- 比如充值100，同时发10个请求，可能到账1000
- 也叫竞态条件漏洞

### 3.4 输入/输出类

#### 12. 短信/邮件轰炸
- 短信验证码可以无限发
- 用来骚扰用户、消耗资源
- 虽然不直接偷东西，但也很烦人

#### 13. 接口重放
- 同一个请求可以重复执行
- 比如发送短信的请求，重放一次就发一次
- 或者下单的请求，重放一次就下一次单

#### 14. 批量操作漏洞
- 可以批量刷、批量注册
- 比如批量注册账号、批量抽奖

### 3.5 业务流程类

#### 15. 流程绕过
- 不按正常顺序走流程
- 比如跳过支付直接到成功页
- 或者直接访问下一步的URL

#### 16. 密码修改漏洞
- 修改密码不需要旧密码
- 或者可以修改任意用户的密码
- 危害很大

#### 17. 注册逻辑漏洞
- 可以注册管理员账号
- 或者用户名可以重复
- 或者注册后直接就是管理员权限

### 3.6 其他类

#### 18. 文件下载/读取漏洞
- 改个文件名就能下载任意文件
- 比如../../etc/passwd
- 也算逻辑漏洞的一种（路径遍历）

#### 19. 数据篡改
- 可以修改任意数据
- 比如修改别人的个人信息、修改订单地址

#### 20. 业务功能滥用
- 正常功能被恶意使用
- 比如邀请好友得奖励，被批量刷
- 或者投票功能被刷票

### 3.3 按危害程度排序

| 危害等级 | 漏洞类型 | 说明 |
|----------|----------|------|
| 🔴 严重 | 支付漏洞 | 直接经济损失 |
| 🔴 严重 | 密码找回漏洞 | 直接控制账号 |
| 🔴 严重 | 越权访问（敏感数据） | 大量数据泄露 |
| 🟠 高危 | 登录绕过 | 直接登录 |
| 🟠 高危 | 垂直越权 | 拿到管理员权限 |
| 🟠 高危 | 订单篡改 | 经济损失 |
| 🟡 中危 | 水平越权 | 信息泄露 |
| 🟡 中危 | 验证码绕过 | 可被爆破 |
| 🟡 中危 | 并发竞争条件 | 刷钱、刷积分 |
| 🟢 低危 | 短信轰炸 | 资源滥用 |
| 🟢 低危 | 接口重放 | 根据场景而定 |

当然，
危害程度也要看具体场景，
不能一概而论。

## 🕵️ 逻辑漏洞挖掘思路

### 4.1 核心思想

挖逻辑漏洞的核心是什么？

> **永远不要相信前端！
> 永远不要假设用户会按正常流程操作！
> 永远从攻击者的角度思考问题！**

记住这三个"永远"。

### 4.2 挖掘步骤

#### 第一步：理解业务流程

先搞清楚：
- 这个系统是干什么的？
- 有哪些功能模块？
- 正常的操作流程是什么？
- 每个步骤都做了什么？
- 传了哪些参数？

**理解业务是挖逻辑漏洞的前提。**

你都不知道这个系统干嘛的，
怎么找它的逻辑漏洞？

所以第一步：
**老老实实走一遍正常流程，
把每个请求都记下来。**

#### 第二步：找出关键参数

每个请求有哪些参数？
哪些参数是关键的？

比如：
- 用户ID（user_id、uid、id...）
- 订单ID（order_id、oid...）
- 金额（price、amount、money...）
- 数量（quantity、num、count...）
- 角色（role、level、is_admin...）
- 状态（status、state...）

这些参数，
都是重点关注对象。

#### 第三步：尝试篡改参数

找到关键参数后，
开始改！

怎么改？
- **改数值**：改大、改小、改成0、改成负数
- **改ID**：改成别人的ID、改成管理员的ID
- **改角色**：普通用户改成管理员
- **改状态**：未支付改成已支付
- **改数量**：改成0、改成负数
- ...

**只有你想不到，没有你改不了的。**

然后看服务器怎么响应，
有没有问题。

#### 第四步：尝试绕过流程

正常流程是：
步骤1 → 步骤2 → 步骤3 → ...

那如果：
- 跳过步骤1，直接到步骤2会怎么样？
- 从步骤3直接回到步骤1会怎么样？
- 步骤2做完，再做一遍步骤2会怎么样？
- 不按顺序来会怎么样？

很多漏洞就是这么出来的。

#### 第五步：尝试并发请求

正常都是一个一个请求发，
那如果同时发很多个请求呢？

比如：
- 同时发10个提现请求
- 同时发10个领优惠券请求
- 同时发10个充值请求

会不会因为时间差，
导致数据不一致？

这就是**并发竞争条件漏洞**。

#### 第六步：尝试边界情况

正常情况都是正常值，
那边界情况呢？

比如：
- 数量是0会怎么样？
- 数量是负数会怎么样？
- 金额是0会怎么样？
- 金额是负数会怎么样？
- 字符串特别长会怎么样？
- 特殊字符会怎么样？

很多系统边界情况处理得不好。

### 4.3 常用工具

挖逻辑漏洞需要什么工具？

其实工具很简单：
- **Burp Suite**（最常用，抓包、改包、重放）
- **浏览器F12**（看请求、改参数）
- **Postman**（接口测试）
- **自己写脚本**（批量测试、并发测试）

核心是Burp Suite，
尤其是Repeater（重放）和 Intruder（爆破）功能，
用得非常多。

### 4.4 挖逻辑漏洞的正确姿势

#### 1. 多问"为什么"

- 为什么这个参数要传？
- 为什么这个接口要这样设计？
- 为什么要走这一步？
- 为什么不能跳过？

多问几个为什么，
可能就发现问题了。

#### 2. 换位思考

如果你是攻击者，
你会怎么搞这个系统？
你最想干什么？
怎么才能干成？

站在攻击者的角度想问题。

#### 3. 不走寻常路

正常人怎么操作，
你就偏不怎么操作。

- 正常人从前往后走，你从后往前走
- 正常人输入正数，你输入负数
- 正常人输对的，你输错的
- ...

**不正常的操作，
往往能发现不正常的漏洞。**

#### 4. 细心+耐心

逻辑漏洞不像SQL注入，
一测就有反应。
很多时候需要耐心地测，
细心地观察响应。

可能改了几十个参数，
才发现一个漏洞。

但只要找到一个，
可能就是高危！

## 🏢 业务安全测试方法论

### 5.1 什么是业务安全测试？

业务安全测试，
就是对系统的业务逻辑进行全面的安全测试，
发现逻辑漏洞和业务风险。

和常规的Web渗透测试不同，
业务安全测试更关注：
- 业务流程有没有问题
- 权限控制严不严
- 交易逻辑有没有漏洞
- 数据会不会被篡改
- ...

简单说：
Web渗透测试偏向技术漏洞，
业务安全测试偏向逻辑漏洞。

### 5.2 业务安全测试流程

#### 第一步：业务调研

先了解清楚：
- 系统是干什么的？
- 有哪些角色？（普通用户、商家、管理员...）
- 有哪些核心业务流程？
- 哪些是敏感操作？
- 数据流向是怎样的？

**越了解业务，越容易发现漏洞。**

#### 第二步：梳理业务流程

把所有的业务流程都列出来：
- 用户注册/登录/找回密码
- 个人信息管理
- 商品浏览/搜索/下单/支付
- 订单管理
- 优惠券/积分
- 后台管理
- ...

每个流程都画出来，
一步一步的。

#### 第三步：识别风险点

每个流程、每个步骤，
都想一想：
- 这里可能有什么漏洞？
- 攻击者会怎么攻击这里？
- 如果参数被改了会怎么样？
- 如果跳过这一步会怎么样？

列出所有可能的风险点。

#### 第四步：测试验证

对每个风险点进行测试，
看看是不是真的有漏洞。

测试的时候要：
- 改参数
- 绕流程
- 测并发
- 测边界
- 测越权

#### 第五步：输出报告

把发现的漏洞整理成报告，
包括：
- 漏洞描述
- 复现步骤
- 危害程度
- 修复建议

### 5.3 业务安全测试的核心原则

#### 原则1：所有用户输入都不可信
- 前端传的任何参数都可能被篡改
- 所有关键数据都要在后端验证

#### 原则2：最小权限原则
- 用户只能访问自己的数据
- 只能做自己有权限做的事
- 接口也要做权限校验

#### 原则3：关键操作多重验证
- 支付要验证密码、验证码
- 修改密码要验证旧密码
- 敏感操作要有二次确认

#### 原则4：流程不可跳过
- 必须按顺序走流程
- 不能跳过中间步骤
- 每一步都要验证上一步是否完成

#### 原则5：数据一致性
- 并发操作也要保证数据正确
- 不能因为并发出现数据错乱

### 5.4 常见的测试维度

| 测试维度 | 说明 | 例子 |
|----------|------|------|
| **越权测试** | 测试能不能访问别人的数据、做别人的操作 | 改user_id看别人订单 |
| **参数篡改** | 测试修改关键参数的影响 | 改price、改quantity |
| **流程绕过** | 测试能不能跳过某些步骤 | 跳过支付直接成功 |
| **并发测试** | 测试同时多个请求会不会有问题 | 并发提现、并发领券 |
| **边界测试** | 测试边界值、异常值 | 0、负数、超大值 |
| **重放测试** | 测试请求能不能重复执行 | 重放短信接口、重放下单 |
| **验证码测试** | 测试验证码能不能绕过、爆破 | 验证码无限试、验证码复用 |
| **密码找回测试** | 测试能不能重置别人密码 | 验证码绕过、Token泄露 |

## ⚖️ 逻辑漏洞 vs 技术漏洞

很多新手分不清，
我们来对比一下。

### 7.1 核心区别

| 对比项 | 技术漏洞 | 逻辑漏洞 |
|--------|----------|----------|
| **核心** | 代码实现有问题 | 业务逻辑设计有问题 |
| **例子** | SQL注入、XSS、命令执行 | 越权、支付漏洞、密码找回 |
| **利用方式** | 注入恶意代码、特殊字符 | 改参数、绕流程、并发 |
| **WAF能不能防** | 大多能防 | 防不住 |
| **扫描器能不能扫** | 大多能扫 | 很难扫（需要理解业务） |
| **有没有固定模式** | 有（相对固定的payload） | 没有（每个系统不一样） |
| **危害方向** | 服务器权限、数据泄露 | 业务损失、资金损失 |
| **学习难度** | 入门难，进阶有套路 | 入门易，进阶靠脑洞 |
| **需要的知识** | 编程语言、系统原理、协议 | 业务理解、脑洞、思路 |

### 7.2 一个比喻

想象一个保险柜：

- **技术漏洞**：保险柜的锁有问题，用铁丝就能捅开 → 锁本身的技术问题
- **逻辑漏洞**：保险柜是好的，但密码写在保险柜门上 → 设计逻辑有问题

### 7.3 两者的关系

不是非此即彼的，
很多时候它们会组合出现：

比如：
- 有SQL注入（技术漏洞），拿到管理员密码 → 然后就有了各种逻辑漏洞的利用条件
- 有越权（逻辑漏洞），拿到管理员权限 → 然后找技术漏洞拿Shell

**技术漏洞偏"打进去"，
逻辑漏洞偏"搞事情"。**

### 7.4 哪个更重要？

都重要！

但从护网的角度来说：
- 外网打点 → 技术漏洞用得多（SQL注入、XSS、SSRF等）
- 深入利用 → 逻辑漏洞用得多（越权、改数据、业务操作）

从危害的角度来说：
- 技术漏洞可能拿服务器权限
- 逻辑漏洞可能直接搞垮业务、损失钱

**都要学，都要会！**

## 📚 案例讲解

### 案例1：0.01元买iPhone（支付漏洞）

**场景描述：**
某电商网站存在支付金额篡改漏洞，
攻击者可以用一分钱买任意商品。

**漏洞详情：**

用户下单流程：
1. 选择商品，加入购物车
2. 确认订单，提交订单请求
3. 生成订单，跳转到支付页
4. 支付完成

**问题出在哪？**

在第2步提交订单的时候，
前端会把商品价格、数量等参数传给后端，
后端直接就用了，
**没有验证价格对不对！**

**利用过程：**

**第一步：选商品**
选一个iPhone，价格5999元，加入购物车。

**第二步：抓包改价格**
到确认订单页面，
点击提交订单的时候用Burp抓包，
把请求里的价格改掉：

原来的：
```
price=5999&quantity=1
```

改成：
```
price=0.01&quantity=1
```

然后放行。

**第三步：订单生成**
后端收到请求，
一看price是0.01，
就生成了一个0.01元的订单。

**第四步：支付**
用支付宝/微信支付0.01元。

**第五步：发货**
订单显示已支付，
商家真的发货了...

**为什么会这样？**
- 后端太信任前端传过来的价格了
- 没有从数据库里查商品的真实价格
- 直接就用前端传的值

**修复方案：**
- 订单的价格不能信前端传的
- 后端要从数据库里查商品的真实价格
- 计算总金额也要后端算
- 支付的时候还要再验证一遍金额

**启示：**
**任何和钱有关的参数，
都不能信前端！
必须后端验证！**

---

### 案例2：改个ID就能看别人订单（水平越权）

**场景描述：**
某电商网站存在水平越权漏洞，
用户可以查看任意用户的订单。

**漏洞详情：**

查看订单详情的URL：
```
http://target.com/order/detail?order_id=1001
```

order_id是订单ID。

问题是：
后端只验证了用户登录了，
**没有验证这个订单是不是当前用户的！**

**利用过程：**

**第一步：登录自己的账号**
正常登录，看自己的订单。

**第二步：改order_id**
把order_id从1001改成1002：
```
http://target.com/order/detail?order_id=1002
```

**第三步：看到别人的订单了！**
页面返回了订单1002的详情，
包括：
- 收货人姓名、电话、地址
- 买了什么东西
- 支付了多少钱
- ...

**第四步：批量遍历**
写个脚本，
从1遍历到10000，
就能把所有用户的订单都扒下来！

**危害：**
- 大量用户信息泄露
- 收货地址、电话泄露 → 诈骗风险
- 购买记录泄露 → 隐私问题

**修复方案：**
- 查询订单的时候，要带上用户ID
- SQL类似：`SELECT * FROM orders WHERE id=? AND user_id=?`
- 只返回当前用户的订单
- 不是自己的订单，直接返回403

**启示：**
**任何涉及用户数据的查询，
都要验证数据归属！
确保用户只能看自己的数据。**

---

### 案例3：验证码可以无限试（爆破登录）

**场景描述：**
某网站登录有图形验证码，
但验证码可以无限次尝试，
导致可以爆破密码。

**漏洞详情：**

登录接口：
```
POST /login.php
参数：username=admin&password=123456&captcha=abcd
```

验证码的问题：
- 验证码用了之后没有过期
- 同一个验证码可以用很多次
- 或者验证码错误后，不会刷新

**利用过程：**

**第一步：获取一个验证码**
正常访问登录页，
拿到一个验证码，比如是 `abcd`。

**第二步：爆破密码**
用Burp的Intruder，
password字段跑字典，
captcha字段固定填 `abcd`。

然后开始跑...

因为验证码不会失效，
所以可以一直试，
直到试出密码为止。

**为什么会这样？**
- 验证码验证后没有立即销毁
- 或者验证码和Session绑定，但Session不会变
- 或者前端传验证码，后端不验证

**修复方案：**
- 验证码用一次就销毁
- 验证失败也要刷新验证码
- 限制登录失败次数（比如5次就锁账号）
- 加短信验证码（敏感操作）

**启示：**
**验证码的生命周期管理很重要，
用一次就必须作废！**

---

### 案例4：并发抢优惠券（竞态条件）

**场景描述：**
某电商搞活动，
每人限领一张100元优惠券，
但可以通过并发请求，
一个人领很多张。

**漏洞详情：**

领优惠券的逻辑（伪代码）：
```php
// 1. 查用户领了多少张
$count = 查询用户领券数量;

// 2. 判断是否达到上限
if ($count >= 1) {
    die("每人限领一张");
}

// 3. 给用户发券
插入一条领券记录;
```

看起来没问题对吧？
先查，再判断，再插入。

但是！
在高并发的情况下，
可能有问题。

**利用过程：**

**第一步：准备工具**
用Burp的Intruder，
或者自己写脚本，
同时发100个领券请求。

**第二步：同时发送**
100个请求几乎同时到达服务器。

**第三步：发生了什么？**

想象一下这个时序：
```
请求1：查数量 → 0张
请求2：查数量 → 0张
请求3：查数量 → 0张
...
请求100：查数量 → 0张

请求1：判断 < 1 → 满足，发券
请求2：判断 < 1 → 满足，发券
请求3：判断 < 1 → 满足，发券
...
请求100：判断 < 1 → 满足，发券
```

因为所有请求都先查询，
都查到是0张，
然后都判断通过，
然后都发了券。

结果：
用户领了100张优惠券！

**这就是并发竞争条件漏洞，
也叫竞态条件（Race Condition）。**

**修复方案：**
- 用数据库事务 + 锁
- 或者用Redis的原子操作
- 或者加唯一索引（user_id + coupon_id）
- 用乐观锁/悲观锁

**启示：**
**涉及数量、库存、金额的操作，
一定要考虑并发场景！**

---

### 案例5：密码找回漏洞（验证码绕过）

**场景描述：**
某网站密码找回功能存在漏洞，
可以重置任意用户的密码。

**漏洞详情：**

密码找回流程：
1. 输入手机号/邮箱
2. 发送验证码
3. 输入验证码
4. 设置新密码

看起来挺正常的，
问题出在哪呢？

**问题1：验证码太简单**
- 4位纯数字
- 而且没有尝试次数限制
- 可以爆破

**问题2：验证码可以在响应中拿到**
- 发送验证码后，接口的响应里居然把验证码返回了！
- 或者验证码在页面的HTML里
- 或者存在Cookie里

**问题3：第二步和第三步没有关联**
- 就是说，A用户的验证码可以给B用户用
- 或者只要验证码对就行，不管是谁的

我们来看一种经典的情况：

**利用过程：**

**第一步：用自己的手机号获取验证码**
输入自己的手机号，
点击发送验证码，
收到验证码：123456。

**第二步：到第三步的时候改手机号**
到输入验证码、设置新密码的页面，
抓包，
把手机号改成目标用户的手机号，
验证码用自己收到的123456。

然后提交...

**第三步：密码重置成功！**
为什么？
因为后端只验证了验证码对不对，
**没有验证这个验证码是不是这个手机号的！**

用自己手机号的验证码，
就能改别人的密码...

**修复方案：**
- 验证码和手机号/账号要绑定
- 验证码有有效期（5分钟）
- 验证次数有限制（错5次就作废）
- 每个步骤都要验证前面步骤的状态
- 用Session或者Token把整个流程串起来

**启示：**
**密码找回是高危功能，
每个步骤都要严格验证，
流程一定要串起来！**

## 📝 本章小结

这一章我们学习了逻辑漏洞的基础知识，
内容很多，也很有意思，
让我们来总结一下：

### 核心概念
- **逻辑漏洞**：业务逻辑设计不当导致的漏洞
- **本质**：服务器太信任前端，没有严格验证
- **特点**：WAF防不住、扫描器难扫、每个系统不一样
- **危害**：直接触碰核心业务利益（钱、数据）

### 常见类型（20种）
- **越权类**：水平越权、垂直越权、未授权访问
- **认证类**：密码找回、验证码漏洞、登录绕过、会话固定
- **支付类**：金额篡改、订单篡改、优惠券、并发竞争
- **输入类**：短信轰炸、接口重放、批量操作
- **流程类**：流程绕过、密码修改、注册漏洞
- **其他**：文件下载、数据篡改、功能滥用

### 挖掘思路
1. 理解业务流程
2. 找出关键参数
3. 篡改参数测试
4. 尝试绕过流程
5. 并发请求测试
6. 边界情况测试

### 核心原则
- 永远不要相信前端
- 最小权限原则
- 关键操作多重验证
- 流程不可跳过
- 保证数据一致性

### 逻辑漏洞 vs 技术漏洞
- 技术漏洞：代码实现问题，有固定模式，WAF可防
- 逻辑漏洞：业务设计问题，无固定模式，WAF难防
- 两者都重要，经常组合使用

逻辑漏洞是一个非常有意思的领域，
它考验的不是你的技术有多高深，
而是你的思路有多开阔，
你的脑洞有多大。

下一章我们会深入学习越权漏洞，
这是最常见、也是最重要的逻辑漏洞之一，
敬请期待！
