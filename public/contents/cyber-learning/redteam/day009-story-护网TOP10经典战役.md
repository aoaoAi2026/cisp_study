## 🏆 战役一：24小时拿下某省级政务外网

### 1.1 背景

这是我参加的比较早的一次护网。
目标是某省的政务外网。

对方的防守很强，
防守团队是国内知名的安全厂商，
设备也很齐全：
防火墙、WAF、IDS、IPS、EDR、态势感知...
该有的都有。

我们这边呢？
一支5人的小队，
装备一般，
经验也不算特别丰富。

赛前大家都觉得，
我们能拿到几个低分的目标就不错了。

结果呢？
**我们用了不到24小时，就拿下了域控。**

怎么做到的？
听我慢慢讲。

### 1.2 攻防过程

#### 第一天上午：信息收集

拿到目标之后，
我们没有急着开打。
先做信息收集。

```
我们收集到的信息：
├──── 目标有一个政务门户网站
├──── 有一堆业务系统（OA、邮箱、政务服务平台...）
├──── 外网暴露的面不算大
├──── WAF很严，扫描器一跑就被封
└──── 邮箱系统用的是某知名厂商的产品
```

试了几个常见的系统，
都没找到明显的漏洞。
WAF确实挺厉害。

这时候，
我们队长老K说话了：
> "正面刚不行，
> 咱们换个思路。
> 既然是政务系统，
> 那供应商呢？
> 第三方服务商呢？
> 从供应链入手试试。"

一语惊醒梦中人。

#### 第一天下午：供应链突破

我们开始收集目标的供应商信息。

```
怎么找供应商？
1. 看网站底部的"技术支持"
2. 看政务采购公告
3. 看系统的版权信息
4. 看招聘信息里的技术栈
5. ...
```

找来找去，
我们发现目标有好几个业务系统，
都是同一家软件公司开发的。

这家软件公司不大，
是本地的一个小厂商。

老K说：
> "就打这家软件公司。
> 拿下他们，
> 就等于拿到了目标的钥匙。"

于是我们调转枪口，
开始打这家软件公司。

小公司的防护，
跟政务系统比起来，
差了不是一星半点。

```
我们的攻击路径：
1. 找到软件公司的官网
2. 官网有个后台，弱口令进去了
3. 后台有文件上传漏洞，拿了webshell
4. 提权，拿到服务器权限
5. 服务器上有源代码，有数据库连接信息
6. 数据库里有他们员工的邮箱账号密码
7. 用员工邮箱，给目标单位的人发钓鱼邮件
8. ...
```

对，你没看错。
**我们是通过第三方供应商，
间接拿下目标的。**

#### 第一天晚上：钓鱼+内网渗透

拿到了软件公司的员工邮箱，
我们就开始发钓鱼邮件。

发给谁？
发给目标单位里跟这家软件公司有对接的人。

怎么知道谁是对接人？
从邮件往来记录里找啊。
软件公司的邮箱里，
有大量跟目标单位的邮件往来。

我们伪装成软件公司的项目经理，
给目标单位的对接人发邮件：
> "张工您好，
> 咱们那个XX系统要升级，
> 需要您配合测试一下。
> 这是升级包和测试说明，
> 您帮忙部署测一下，
> 有问题随时找我。"

附件里是个木马。

对方没多想，就点开了。
毕竟是"长期合作的供应商"嘛。

就这样，
我们拿到了目标内网的一个入口。

后面的事情就顺理成章了：
- 信息收集
- 横向移动
- 域渗透
- 拿下域控

从开始到拿下域控，
不到24小时。

### 1.3 战役点评

这个案例，
是典型的**"供应链攻击"**。

```
为什么这个案例经典？

1. 以弱胜强
   - 正面打不过，就绕着打
   - 不跟你硬刚，找你的软肋

2. 供应链思维
   - 目标本身防护很强
   - 但是第三方供应商防护弱
   - 从供应商入手，迂回突破

3. 社会工程学的运用
   - 利用信任关系
   - 伪装成熟人，降低警惕
   - 钓鱼邮件的成功率大大提高

4. 思路比工具重要
   - 工具大家都有
   - 但不是谁都能想到这个思路
   - 红队的核心，是思路
```

> 老K说：
> **"护网就像打仗。
> 正面硬碰硬，那是莽夫。
> 真正的高手，
> 都是避实击虚，
> 找敌人最薄弱的地方下手。
> 你的防守再强，
> 只要有一个口子，
> 就够了。"**

## 🏆 战役二：一封邮件搞定某金融企业内网

### 2.1 背景

这个案例，
是我听朋友讲的。
某大型金融企业的护网演练。

金融企业的防护，
你懂的，
那叫一个严。

- 内外网物理隔离
- 各种安全设备堆成山
- 管理制度非常严格
- 安全意识培训也到位

按说应该很难打吧？

结果呢？
**红队只用了一封邮件，就进去了。**

### 2.2 攻防过程

#### 信息收集阶段

红队先做了大量的信息收集。

```
收集到的信息：
├──── 目标是某银行
├──── 内外网物理隔离
├──── 外网没什么可打的
├──── 员工的安全意识很高
├──── 但是...
└──── 他们有个客服邮箱，对外公开的
```

外网确实没什么突破口。
那怎么办？

红队想了个招：
**"物理隔离是吧？
那咱们就从人身上突破。"**

#### 钓鱼邮件设计

红队设计了一封非常巧妙的钓鱼邮件。

不是发给普通员工的，
是发给**HR部门**的。

邮件内容是什么呢？
是一份**简历**。

```
📧 钓鱼邮件内容：

发件人：张三 <zhangsan@163.com>
收件人：hr@bank.com
主题：应聘-信息安全岗-张三-10年工作经验

HR您好：

我叫张三，有10年信息安全工作经验，
之前在某知名安全公司做过技术总监，
看到贵行在招聘信息安全岗位，
特此投递简历。

我的简历在附件里，
里面有我详细的工作经历和项目经验。
我的期望薪资是XX万/年。

如果您觉得合适，
请随时跟我联系。
我的电话是：13XXXXXXXXX

谢谢！

张三
2024年X月X日

附件：张三_简历.pdf.exe
```

等等，
附件名是`张三_简历.pdf.exe`？
这也太明显了吧？
HR会点开吗？

别急，
人家有后手。

#### 为什么能成功？

你可能会问：
"后缀是exe，HR能看不见？"

哎，
还真看不见。

为什么？
因为**Windows默认是隐藏已知文件类型的扩展名的**。

也就是说，
`张三_简历.pdf.exe`这个文件，
在HR的电脑上显示的是：
`张三_简历.pdf`

后面的`.exe`被藏起来了。
而且图标也是PDF的图标。

HR一看，
"哦，PDF简历，看看。"
双击，
中招了。

就这么简单？
就这么简单。

但是你以为这就完了？
没有。

金融企业内网有EDR啊，
木马一上去就被杀了怎么办？

红队早就想到了。
他们的木马做了免杀，
而且是专门针对这家银行用的EDR做的免杀。
成功率很高。

就这样，
HR的电脑成了红队在内网的跳板。

后面的事情，
就不用我多说了吧？
- 内网信息收集
- 横向移动
- 找域管
- 拿域控

#### 更绝的

你以为这就够了？
还有更绝的。

红队拿到HR的邮箱之后，
还用HR的邮箱，
给IT部门的人发了一封邮件：

```
📧 第二封钓鱼邮件：

发件人：李XX（HR） <lixx@bank.com>
收件人：it-admin@bank.com
主题：新入职员工IT账号申请

IT部门的同事您好：

这是本周新入职的3名员工的信息，
麻烦帮忙开通一下域账号和邮箱，
谢谢！

附件里是员工信息表。

人力资源部
李XX
```

IT部门一看是HR发的，
想都没想就点开了。
然后...
IT部门的电脑也中招了。

IT部门的权限，
你懂的。
有了IT部门的账号，
内网基本就横着走了。

### 2.3 战役点评

这个案例，
是**社会工程学**的经典运用。

```
为什么这个案例经典？

1. 突破物理隔离
   - 技术上搞不定，就从人入手
   - 人是最大的安全漏洞
   - 物理隔离也挡不住"人肉摆渡"

2. 细节决定成败
   - 利用Windows默认隐藏扩展名的特性
   - 伪装成PDF文件
   - 这些细节，决定了钓鱼的成功率

3. 精准打击
   - 不撒网，精准定位关键岗位
   - HR → IT部门
   - 一步一步渗透到核心

4. 链式攻击
   - 拿下一个，用这个身份攻击下一个
   - 利用内部信任关系
   - 一环扣一环
```

> 老K常说：
> **"技术再强，
> 也架不住人点一下。
> 最好的漏洞，
> 从来都不是技术漏洞，

## 🏆 战役三：Redis未授权引发的"血案"

### 3.1 背景

这个案例，
说出来有点搞笑。
某大型互联网公司的护网演练。

这家公司技术很强，
安全团队也很牛。
按说应该很难打吧？

结果呢？
**红队通过一个Redis未授权访问漏洞，
一路打到了核心数据库。**

怎么回事呢？
听我给你讲。

### 3.2 攻防过程

#### 初战不利

红队一开始也是正面打。
Web系统、API接口、各种业务...
试了一圈，
没找到什么像样的漏洞。

这家公司的安全做得确实不错。

就在大家一筹莫展的时候，
一个队员说：
> "我扫端口的时候，
> 发现有几台服务器开了6379端口，
> 而且是直接暴露在公网的。
> 会不会是Redis？"

大家一听，
马上来了精神。

#### Redis未授权

我们赶紧去看。
还真是Redis。
而且——
**没有设置密码！**

```bash
# 连接Redis
$ redis-cli -h x.x.x.x
x.x.x.x:6379> ping
PONG
```

就这么连上了。

大家都懵了。
这么大的公司，
Redis居然直接暴露公网，还没密码？

后来才知道，
这几台Redis是测试环境的，
运维人员图方便，
直接映射到公网了，
还忘了加密码。
本来测试完要收回去的，
结果忘了。

#### 从Redis到服务器

连上Redis之后，
接下来就是常规操作了：
**写SSH公钥，拿服务器权限。**

```bash
# 生成公钥
$ ssh-keygen -t rsa

# 将公钥写入Redis
$ (echo -e "\n\n\n"; cat ~/.ssh/id_rsa.pub; echo -e "\n\n\n") > key.txt

# 连接Redis，写入公钥
$ redis-cli -h x.x.x.x
x.x.x.x:6379> config set dir /root/.ssh/
OK
x.x.x.x:6379> config set dbfilename authorized_keys
OK
x.x.x.x:6379> set x "\n\n\nssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQ...\n\n\n"
OK
x.x.x.x:6379> save
OK

# 用SSH登录
$ ssh root@x.x.x.x
# 登录成功！
```

就这样，
我们拿到了这台服务器的root权限。

#### 从一台到一片

拿到这台服务器之后，
我们发现：
**这台服务器在内网里。**

而且，
这是测试环境的服务器，
权限很大，
能访问很多东西。

```
内网里有什么？
├──── 一堆测试服务器
├──── 测试环境的数据库（有很多真实数据的备份）
├──── 各种源代码
├──── 配置文件（有生产环境的密码！）
└──── ...
```

你没看错，
**测试环境的配置文件里，有生产环境的密码。**

为什么？
因为开发人员图方便，
把生产环境的配置也放在测试环境了。
说是"方便测试"。

于是我们拿着这些密码，
去登生产环境。
还真登进去了。

后面的事情就不用多说了。
从测试环境打到生产环境，
一路高歌猛进，
最后打到了核心数据库。

### 3.3 战役点评

这个案例，
是典型的**"千里之堤，溃于蚁穴"**。

```
为什么这个案例经典？

1. 低级漏洞，严重后果
   - Redis未授权，说穿了就是个低级错误
   - 但是造成的后果极其严重
   - 真是"一个没密码的Redis毁了所有"

2. 测试环境是重灾区
   - 很多公司不重视测试环境的安全
   - 觉得"反正是测试环境，无所谓"
   - 但是测试环境和生产环境往往是通的
   - 测试环境一丢，生产环境也危险了

3. 密码复用/明文存储
   - 配置文件里存明文密码
   - 测试环境放生产环境的密码
   - 这些都是大忌

4. 运维不规范
   - 测试环境随便映射公网
   - 用完不回收
   - 这些不规范的操作，都是安全隐患
```

> 老K说：
> **"很多时候，
> 打败你的不是什么高深的0day，
> 而是一个低级到不能再低级的错误。
> 安全这件事，
> 细节决定成败。
> 一个小小的疏忽，
> 就可能导致满盘皆输。"**

（Redis漏洞利用的详细内容，详见第28章：Redis漏洞利用实战）

## 🏆 战役四：0day真的那么神吗？

### 4.1 背景

这个案例，
是关于0day的。

很多人觉得，
有0day就无敌了。
护网的时候，
扔一个0day过去，
直接就拿下了。

真的是这样吗？

我给你讲个真实的案例。

### 4.2 攻防过程

#### 赛前：豪言壮语

有一年护网，
某支队伍财大气粗，
赛前放话：
> "我们准备了好几个0day，
> 这次肯定拿第一。"

大家一听，
都觉得这支队伍稳了。
0day啊，那可是大杀器。

我们也挺紧张的，
觉得这次冠军悬了。

#### 赛中：0day哑火了

结果比赛开始之后，
戏剧性的一幕发生了。

那支队伍的0day，
**一个都没用上。**

为什么？

```
原因1：目标环境不对
   - 0day是针对某个特定版本的
   - 目标的版本不一样，用不了

原因2：目标有防护
   - 虽然0day能打进去
   - 但是后续操作被EDR拦了
   - 木马上去就被杀
   - 等于白打

原因3：没用对地方
   - 0day打了一个不重要的系统
   - 打完之后发现，
   - 这个系统跟内网不通
   - 拿了也白拿

原因4：被发现了
   - 0day一用就被蓝队发现了
   - 虽然没拦住
   - 但是蓝队马上就告警了
   - 然后全网排查、修补漏洞
   - 后面想再用就难了
```

最后那支队伍的成绩怎么样？
很一般。
别说冠军了，
前十都没进。

#### 反观我们

我们呢？
没有0day，
用的都是Nday，
甚至有些是几年前的老漏洞。

但是我们成绩还不错，
拿了第三。

为什么？
因为我们**思路灵活，打法多样**。

```
我们的打法：
├──── Web打点不行就钓鱼
├──── 正面打不行就走供应链
├──── 技术不行就搞社工
├──── 一个方法不行就换另一个
```

## 🏆 战役六：一台打印机引发的内网漫游

### 6.1 背景

打印机？

大家凑过去一看，
是一个IP，
开放了9100端口。

9100端口是什么？
是打印机的端口啊！

#### 打印机暴露公网

这是一台打印机，
不知道什么原因，
直接暴露在公网上了。

可能是运维人员图方便，
把打印机映射到公网了，
方便远程打印。
也可能是配置错误。
反正就是——
一台打印机，裸奔在公网上。

大家一开始都没当回事。
打印机嘛，
能有什么用？
总不能用打印机打内网吧？

但是红队队长不这么想。
他说：
> "打印机也是设备，
> 只要是设备，就可能有漏洞。
> 先研究研究。"

于是大家开始研究这台打印机。

#### 打印机漏洞

这一研究，
还真研究出东西来了。

这台打印机是某知名品牌的，
有一个**未授权访问漏洞**。
不用密码，就能直接登录管理后台。

登录后台能干嘛？
能看打印历史，能改配置，
还能——
**查看打印机的网络配置。**

网络配置里有什么？
有IP地址，有子网掩码，
还有——
**SMB服务器的地址。**

对，
打印机配置了SMB共享，
用来扫描文件什么的。
SMB服务器的地址，
就是内网的一台文件服务器。

不仅如此，
打印机里还存了**SMB的用户名和密码**。
（因为要访问SMB共享嘛，
打印机的配置里存了凭据）

虽然是个低权限的账号，
但是——
**这是内网的账号啊！**

#### 从打印机到内网

有了内网的账号密码，
还有SMB服务器的地址，
接下来怎么办？

很简单，
**VPN啊。**

不对，没有VPN。
那怎么进内网？

哎，
打印机本身就在内网啊！
打印机能访问内网的SMB服务器，
说明打印机在内网里。

那——
**能不能把打印机当成跳板？**

还真能。

这台打印机支持**SSH登录**。
（很多打印机都支持SSH，方便运维）
而且，
因为有未授权访问漏洞，
我们可以直接在后台开启SSH，
设置个密码。

然后呢？
然后就可以用SSH连打印机了啊！
连上打印机，
不就等于进了内网吗？

虽然打印机的系统很精简，
很多命令都没有，
但是——
**能ping通内网，能做端口转发啊！**

于是，
我们通过打印机的SSH做端口转发，
把内网的SMB服务器的端口转发出来，
然后用刚才拿到的账号密码登录。

就这样，
**我们通过一台打印机，
成功进入了目标的内网。**

后面的事情就不用多说了，
有了内网的一个立足点，
剩下的就是时间问题。

- 哪些漏洞会触发告警？
- 蓝队的响应速度有多快？
- 蓝队一般什么时候人最少？

摸清楚这些之后，
心里就有数了。

#### 第三周：突破

第三周，
才开始真正的攻击。

但是也不是猛打猛冲，
而是选了一个最隐蔽的突破口，
悄悄地打进去。

他们选的是什么突破口？
**一个非常冷门的第三方系统的漏洞。**

这个系统用的人不多，
漏洞也不怎么出名，
所以蓝队没怎么关注。

他们用这个漏洞，
悄悄地拿了一台服务器。
然后马上把所有痕迹都清掉了，
然后就不动了。

为什么不动了？
**先观察观察，看看有没有被发现。**

观察了三天，
没什么动静。
说明没被发现。

这才开始下一步。

#### 第四周：内网渗透+横向移动

第四周，
开始内网渗透。

但是速度依然很慢。
今天移一台，
明天移两台，
移完就清日志，
然后歇半天。

为什么这么慢？
**怕动静太大，被发现。**

就这么慢慢悠悠地，
用了整整一周，
从外网的一台服务器，
一步步移到了核心区，
最后拿下了域控。

直到护网结束，
蓝队都没发现他们是怎么进来的。

### 8.3 战役点评

这个案例，
是**"稳扎稳打，以慢打快"**的经典。

```
为什么这个案例经典？

1. 慢工出细活
   - 不是越快越好
   - 有时候慢一点，反而更稳
   - 稳，才能走得远

2. 信息收集的重要性
   - 用一周时间做信息收集
   - 看起来很慢，其实是在磨刀
   - 磨刀不误砍柴工

3. 隐蔽性第一
   - 红队作战，隐蔽是第一位的
   - 一旦被发现，就难打了
   - 宁可慢一点，也不要暴露
   - 悄悄的进村，打枪的不要

4. 耐心很重要
   - 很多人沉不住气，急于求成
   - 结果就是打草惊蛇，前功尽弃
   - 真正的高手，都有耐心
   - 像猎人一样，等待最好的时机
```

> **"红队就像幽灵。
> 来无影去无踪，
> 目标都不知道你来过。

> 目标还没发现你，
> 那你才算真的厉害。

> 打得隐蔽才是本事。"**

## 🏆 战役九：功亏一篑——最可惜的失败案例

### 9.1 背景

讲了这么多成功的案例，
再给你讲一个失败的案例。

为什么要讲失败的？
因为——
**失败的教训，比成功的经验更宝贵。**

这个案例，
是我亲身经历的。
那是我参加的第二次还是第三次护网，
因为一个低级错误，
功亏一篑。

现在想起来，
还觉得可惜。

### 9.2 攻防过程

#### 一路高歌猛进

那次护网，
我们打得特别顺。

```
我们的进度：
├──── 第一天：找到SQL注入，拿了webshell
├──── 第二天：提权，拿到服务器权限
├──── 第三天：内网信息收集，找到了域管
├──── 第四天：横向移动，拿到了域管权限
└──── 第五天：准备拿下域控，结束战斗
```

才五天，
就拿到了域管权限。
照这个速度，
一周就能结束战斗。
我们都觉得稳了。

拿到域管权限之后，
我们特别开心，
觉得冠军稳了。
甚至都开始讨论赢了去哪吃饭了。

#### 乐极生悲

拿到域管权限的当天晚上，
我们队里有个小伙子，
叫小李。
他觉得反正都赢了，
不如"玩点花的"。

他干了什么呢？
他用域管权限，
**给所有域用户发了一封"恐吓信"。**

邮件内容大概是：
> "你们已经被我黑了！
> 所有数据都在我手里！
> 想知道怎么被黑的吗？
> 来找我啊~ —— 红队留"

你没看错，
就是这么中二的一封邮件。

他发这封邮件干嘛？
就是想装个X，
吓唬吓唬对方，
证明我们厉害。

结果呢？
结果就是——
**整个蓝队都炸了。**

本来我们藏得好好的，
蓝队都没怎么发现我们。
这封邮件一发，
相当于直接告诉蓝队：
"我在这呢，快来抓我啊！"

#### 全线溃败

蓝队收到邮件之后，
全员紧急动员。
- 全网断网排查
- 所有密码强制更换
- 所有服务器排查
- 所有账号审计
- ...

我们的木马，
一个接一个被清掉。
我们的后门，
一个接一个被堵上。

本来我们有好多据点，
一夜之间，
全没了。

就剩下一两个藏得比较深的，
也不敢动了。
一动就可能被发现。

之后的十几天，
我们就只能猫着，
什么都干不了。
眼睁睁看着到手的胜利，
就这么没了。

最后成绩怎么样？
勉勉强强进了前十。
本来稳拿冠军的。

### 9.3 战役点评

这个案例，
每次想起来我都心疼。
太可惜了。

## 🏆 战役十：最强的防守，最强的进攻

### 10.1 背景

最后一个案例，
我想讲一个最精彩的。

这是我见过的，
水平最高的一次护网对抗。
红队强，蓝队也强。
两边打得有来有回，
跟看大片似的。

### 10.2 攻防过程

#### 第一阶段：红队猛攻，蓝队死守

一开始，
红队攻势很猛。
各种漏洞、各种钓鱼、各种手法，
一波接一波。

蓝队也不是吃素的，
见招拆招，
守得滴水不漏。

```
前三天的攻防：
├──── 红队发了10种钓鱼邮件，被拦了9种
├──── 红队试了20多个漏洞，成了1个
├──── 那个成了的，刚进去就被发现了
├──── 红队扫端口，被封了20多个IP
├──── 红队换IP，继续扫，又被封
└──── 僵持阶段
```

前三天，
两边打了个平手。
红队没占到什么便宜，
蓝队也没出什么大问题。

#### 第二阶段：红队迂回，蓝队补漏

正面打不动，
红队开始迂回。
供应链、第三方、员工个人账号...
各种旁敲侧击。

蓝队也很警觉，
红队刚从供应链那边有点苗头，
蓝队马上就发现了，
然后把供应链的口子也堵上了。

```
中间一周的攻防：
├──── 红队打供应商，被蓝队发现，未果
├──── 红队搞社工，被蓝队反将一军
├──── 红队找了个0day，刚用就被蓝队捕获
├──── 蓝队甚至还溯源到了红队的C2服务器
├──── 红队被迫换C2
└──── 蓝队防守得密不透风
```

当时我们都觉得，
红队这次悬了。
蓝队太强了。

#### 第三阶段：转折点

就在大家以为红队要输了的时候，
转折点来了。

什么转折点？
**蓝队的安全设备，出bug了。**

不是被攻击的bug，
是设备本身的bug。
设备厂商的一个新版本，
有个性能问题，
导致全网的安全设备——
**集体卡顿。**

就跟Windows更新完蓝屏似的。

安全设备一卡，
检测能力就下降了。
很多告警都出不来了。

蓝队急得团团转，
赶紧联系厂商排查。
但是哪那么快能修好？

红队抓住这个机会，
发起了总攻。

```
红队的总攻：
├──── 所有入口一起上
├──── 钓鱼邮件批量发
├──── 漏洞批量打
├──── 各种手法齐上阵
├──── 趁你病，要你命
└──── 全面进攻
```

蓝队那边，
安全设备卡得要死，
根本顾不过来。
眼睁睁看着红队打进来，
拦都拦不住。

几天之内，
红队就打进了内网，
一路高歌猛进，
拿下了很多服务器。

#### 第四阶段：蓝队反击

但是蓝队毕竟是蓝队，
反应也快。
一看拦不住，
马上调整策略。

```
蓝队的应对：
├──── 安全设备先不管了
├──── 把核心系统断网保护
├──── 人工排查，手动封堵
├──── 组织人手，一台一台服务器查
├──── 跟红队拼人力
└──── 能保多少保多少
```

核心系统被断网了，
红队打不进去。
非核心的服务器，
红队拿了不少。

两边就这么僵持着，
直到护网结束。

#### 最终结果

最后谁赢了？
**红队小胜。**

红队拿到了不少权限，
但是核心系统没拿下来。
蓝队虽然丢了不少服务器，
但是核心资产保住了。

怎么说呢，
红队赢了，但是赢的不多；
蓝队输了，但是输得不难看。
两边都打出了水平。

### 10.3 战役点评

这个案例，
是我见过的最精彩的攻防对抗。

```
为什么这个案例经典？

1. 棋逢对手
   - 红队强，蓝队也强
   - 有来有回，势均力敌
   - 这样的对抗才有意思
   - 才能互相促进，共同提高

2. 不确定性
   - 谁能想到最后是安全设备出bug了？
   - 计划赶不上变化
   - 很多时候，决定胜负的不是技术
   - 而是运气，或者说，是应变能力

3. 没有永远的胜者
   - 没有攻不破的系统
   - 也没有防得住的防守
   - 都是相对的
   - 看谁犯的错少，看谁抓得住机会

4. 核心资产保护
   - 蓝队虽然丢了很多服务器
   - 但是核心系统保住了
   - 这就不算输
   - 防守的目标，不是守住所有东西
   - 而是守住最重要的东西
```

> 老K说：
> **"护网就像下棋。
> 你厉害，对手也厉害。
> 有赢有输，才有意思。
> 赢了别骄傲，
> 输了别气馁。
> 每一次对抗，
> 都是一次成长的机会。"**

## 📚 案例总结

讲了10个案例，
我们来总结一下。

### 11.1 红队获胜的共同特点

```
1. 思路灵活
   - 正面打不动就迂回
   - 技术不行就搞社工
   - 不钻牛角尖，懂得变通

2. 信息收集充分
   - 知己知彼，百战不殆
   - 动手之前先摸清楚情况
   - 不打无准备之仗

3. 重视隐蔽
   - 悄悄的进村，打枪的不要
   - 一旦暴露，难度就会指数级上升
   - 隐蔽是红队的生命线

4. 有耐心
   - 不急于求成
   - 稳扎稳打，步步为营
   - 等待最好的时机

5. 出其不意
   - 不走寻常路
   - 专找对手想不到的地方下手
   - 攻其不备，出其不意
```

### 11.2 红队失败的常见原因

```
1. 得意忘形
   - 有点成绩就飘了
   - 装逼、炫耀、搞事情
   - 最后把自己作死了

2. 操之过急
   - 急于求成
   - 还没摸清情况就猛打猛冲
   - 打草惊蛇

3. 思路单一
   - 只会一种打法
   - 正面打不动就不知道怎么办了
   - 不会变通

4. 忽视细节
   - 一个小失误，导致满盘皆输
   - 比如忘了清日志，比如工具没做免杀
   - 细节决定成败

5. 不尊重对手
   - 觉得对手菜
   - 掉以轻心
   - 最后阴沟里翻船
```

### 11.3 给你的启示

看完这10个案例，
你有什么感想？

我给你总结几条启示：

```
1. 技术重要，但思路更重要
   - 工具大家都有，漏洞大家都知道
   - 但是有人能打进去，有人就打不进去
   - 差别就在思路上

2. 不要低估任何对手
   - 哪怕对方看起来很菜
   - 也不要掉以轻心
   - 阴沟里翻船的事，太多了

3. 不要忽视任何细节
   - 一个小小的疏忽
   - 可能导致全盘皆输
   - 小心驶得万年船

4. 保持隐蔽，保持耐心
   - 红队作战，隐蔽第一，耐心第二
   - 宁可慢一点，也不要暴露
   - 稳，才能走到最后

5. 善于从失败中学习
   - 失败不可怕
   - 可怕的是失败了还不长记性
   - 每一次失败，都是成长的机会
```

6. 第六个案例（打印机）给你什么启发？为什么说"万物皆可攻破"？

7. 第七个案例（帮蓝队修漏洞）中，红队是怎么一步步建立信任的？这种手法为什么有效？

8. 第八个案例（潜伏一个月）中，"慢工出细活"体现在哪里？为什么有时候慢反而更快？

9. 第九个案例（功亏一篑）中，红队犯了什么错误？你觉得应该吸取什么教训？

10. 看完这10个案例，你最大的收获是什么？你觉得红队最重要的素质是什么？

### 实操题

1. 复盘一个你打过的靶场或CTF题：
   - 你是怎么打的？用了什么思路？
   - 中间遇到了什么困难？怎么解决的？
   - 打完之后有什么收获？
   - 如果让你重新打，你会怎么打？

2. 换位思考：如果你是蓝队，你会怎么防住这些攻击？
   - 从10个案例中任选一个
   - 站在蓝队的角度思考
   - 怎么才能防住红队的攻击？
   - 写一篇你的防守方案。

3. 脑洞大开：设计一个你觉得最"秀"的攻击方案
   - 可以是任何场景、任何手法
   - 越有创意越好
   - 不用考虑可行性，尽情开脑洞
   - 看看你的想象力有多丰富

4. 找一个真实的安全事件，做一次案例分析：
   - 事件的背景是什么？
   - 攻击方是怎么攻击的？
   - 防守方哪里没做好？
   - 你从中学到了什么？

5. 给自己定一个小目标：
   - 看完了这么多案例，你是不是更有动力了？
   - 给自己定一个本周的学习目标
   - 比如：打完10个SQLi-Labs的关卡
   - 比如：看完某本书的前3章
   - 把目标写下来，然后去实现它。

## 📝 本章小结

这一章，
我们讲了10个护网经典战役。

总结一下：

1. **战役一：24小时拿下某省级政务外网**
   - 供应链攻击，避实击虚
   - 思路比工具重要

2. **战役二：一封邮件搞定某金融企业内网**
   - 社会工程学的经典运用
   - 人是最大的安全漏洞

3. **战役三：Redis未授权引发的"血案"**
   - 低级漏洞，严重后果
   - 千里之堤，溃于蚁穴

4. **战役四：0day真的那么神吗？**
   - 0day很重要，但不是决定性的
   - 思路比漏洞重要

5. **战役五：最危险的地方最安全**
   - 出其不意，攻其不备
   - 藏在对手眼皮子底下

6. **战役六：一台打印机引发的内网漫游**
   - 万物皆可攻破
   - 边缘设备的安全不可忽视

7. **战役七：逆向思维——从防守者的角度思考**
   - 逆向思维，攻心为上
   - 让对方自己把你请进去

8. **战役八：慢工出细活——潜伏一个月的红队**
   - 稳扎稳打，以慢打快
   - 隐蔽和耐心很重要

9. **战役九：功亏一篑——最可惜的失败案例**
   - 得意忘形，功败垂成
   - 永远不要飘，稳字当头

10. **战役十：最强的防守，最强的进攻**
    - 棋逢对手，将遇良才
    - 在对抗中共同成长
