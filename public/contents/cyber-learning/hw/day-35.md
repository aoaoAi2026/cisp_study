## 一、漏洞扫描——蓝队的主动防御之眼

### 1.1 为什么要做漏洞扫描？

想象一下：攻击者就像小偷，在你的房子外面转悠，寻找没锁的门窗。
漏洞扫描就是你自己先把所有的门窗检查一遍，在坏人发现之前修好。

蓝队做漏洞扫描的三个核心理由：
1. 知己知彼：知道自己系统有哪些弱点，才能在攻击者利用之前修复
2. 合规要求：等保 2.0、ISO 27001 等标准都要求定期漏洞扫描
3. 持续改进：每次扫描的结果对比，能看到安全建设的进展

关键认知：漏洞扫描不能保证 100% 发现所有漏洞，但不做漏洞扫描
就等于蒙着眼睛防守——你不知道自己在防什么。

### 1.2 主流漏洞扫描器对比

| 扫描器 | 类型 | 费用 | 优势 | 劣势 |
|:---|:---|:---|:---|:---|
| OpenVAS | 开源 | 免费 | NVT 规则多（50000+）、无授权限制 | 部署复杂、扫描慢 |
| Nessus | 商业 | $3000+/年 | 扫描快、误报低、报告美观 | 贵！ |
| Nexpose | 商业 | 收费 | 与 Metasploit 集成好 | 资源占用大 |
| AWVS | 商业 | 收费 | Web 漏洞扫描专用 | 只能扫 Web |
| Xray | 开源 | 社区版免费 | 被动扫描、POC 自定义 | 社区版有限 |
| Nuclei | 开源 | 免费 | 模板丰富、社区活跃 | 需手动写模板 |

结论：预算有限选 OpenVAS + Nuclei，预算充足选 Nessus。

## 二、OpenVAS 架构——四层结构

```
┌─────────────────────────────────────┐
│         GSA (Greenbone Security      │
│         Assistant) - Web 管理界面    │
│         https://localhost:9392       │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│         GVMD (Greenbone              │
│         Vulnerability Manager)       │
│         - 任务调度、用户管理         │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│         OpenVAS Scanner (OSP)        │
│         - 实际执行扫描的引擎         │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│         NVT (Network Vulnerability   │
│         Tests) - 漏洞检测规则库      │
│         50000+ 条 .nasl 规则         │
└─────────────────────────────────────┘
```

简单理解：
- GSA = 你操作的网页界面
- GVMD = 中间调度员（接收你的指令，安排扫描任务）
- Scanner = 干活的工人（真正去探测漏洞）
- NVT = 工人手上的工具和检查清单（50000+ 条规则）

## 三、OpenVAS 部署实战

### 3.1 Docker 方式（推荐，最简单）

Docker 部署只需要一条命令：

```bash
# 启动 OpenVAS 容器
docker run -d -p 443:443 --name openvas \
  -v openvas_data:/data \
  --restart unless-stopped \
  mikesplain/openvas

# 查看初始化日志（首次启动需要 5-15 分钟下载规则库）
docker logs -f openvas

# 等待看到类似这行表示初始化完成：
# "Rebuilding NVT cache... done"

# 访问 Web 界面
# URL: https://localhost
# 用户名: admin
# 密码: admin（首次登录后需修改）
```

注意事项：
- 初始化时容器会下载约 2GB 的 NVT 规则库，请确保网络畅通
- 容器重启后 NVT 规则不会丢失（因为映射了 volume）
- 生产环境建议绑定固定 IP 和端口

### 3.2 NVT 规则库管理

NVT 规则库是 OpenVAS 的核心——没有规则就检测不了漏洞。

更新规则的命令：
```bash
# 进入容器更新规则
docker exec -it openvas greenbone-nvt-sync

# 确认规则数量（应该超过 50000 条）
docker exec -it openvas gvmd --get-scanners
```

规则更新频率：
- 绿骨头（Greenbone）每天发布新的 NVT 规则
- 建议每周更新一次规则库
- 重大漏洞（如 Log4j）发布后，24 小时内会有对应规则

## 四、创建扫描任务——从零到扫描完成

完整操作流程：

Step 1：创建扫描目标
- 菜单：Configuration -> Targets
- 点击新建按钮（星形图标）
- Name: 给目标起个名字（如 内网 Web 服务器）
- Hosts: 输入目标 IP 或域名（如 192.168.1.10）
- Port List: 选择 All TCP（扫描所有端口）

Step 2：选择扫描配置
- 菜单：Scans -> Tasks
- 点击新建任务
- Scan Targets: 选择刚才创建的目标
- Scan Config: 选择扫描强度
  - Base: 基础扫描（5-15 分钟）
  - Full and fast: 全面快速（30 分-2 小时）<- 最常用
  - Full and very deep: 深度扫描（2-8 小时）

Step 3：执行扫描
- 点击开始按钮（播放图标）
- 状态变化：Requested -> Running -> Done

Step 4：查看报告
- 扫描完成后点击任务名
- Results 标签页：按严重级别排序的漏洞列表
- 导出：可以导出 PDF/HTML 格式的报告

## 五、解读扫描报告——看懂了才是有效扫描

### 5.1 CVSS 评分体系

CVSS（通用漏洞评分系统）是全球统一的漏洞严重性评估标准：

| CVSS 分值 | 严重级别 | 符号 | 处置时限 | 例子 |
|:---|:---|:---|:---|:---|
| 9.0-10.0 | Critical（严重） | 🔴 | 24 小时内 | 远程代码执行 RCE |
| 7.0-8.9 | High（高危） | 🟠 | 72 小时内 | SQL 注入、文件上传 |
| 4.0-6.9 | Medium（中危） | 🟡 | 1 周内 | XSS、CSRF |
| 0.1-3.9 | Low（低危） | 🟢 | 1 月内 | 版本信息泄露 |
| 0.0 | Info（信息） | 🔵 | 记录即可 | 开放端口、服务识别 |

蓝队处置原则：
- Critical/High = 立即启动漏洞修复流程
- Medium = 纳入本周修复计划
- Low = 纳入月度修复计划
- Info = 记录在资产台账中

### 5.2 误报识别与处理

常见误报场景：
1. 服务版本误判：扫描器根据 Banner 识别版本可能出错
   验证方法：手动连接服务确认版本（如 nc -v target 22 看 SSH Banner）

2. 补丁后误报：系统打了补丁但版本号未变（Backport 情况）
   验证方法：查看变更记录，确认补丁已正确应用

3. 环境差异：扫描器通用判断不适用于特定配置
   验证方法：在测试环境部署相同版本，尝试 POC 验证

误报处理流程：
确认漏洞 -> 标记为有效 -> 纳入修复计划
确认为误报 -> 标记为误报（False Positive）-> 记录原因
无法确认 -> 标记为待验证 -> 安排人工验证

## 六、漏洞管理五步闭环

漏洞扫描不是一次性动作，而是持续循环的过程：

第一步：发现（Discover）
- OpenVAS 定期扫描 -> 生成漏洞清单
- 记录：扫描时间、目标、发现的漏洞数量

第二步：评估（Assess）
- 去误报 -> CVSS 定级 -> 分配责任人和修复时限
- 记录：每个漏洞的评估结果和责任人

第三步：修复（Remediate）
- 打补丁 / 改配置 / 加 WAF 规则
- 记录：修复时间、修复方法、变更单号

第四步：验证（Verify）
- 复扫 -> 确认漏洞已修复 -> 回归测试业务功能正常
- 记录：复扫时间和结果

第五步：关闭（Close）
- 关闭漏洞工单 -> 记录完整的处置时间线
- 将处置经验沉淀到知识库

这个流程要形成制度，每次都要严格执行。
漏洞管理做得好的团队，高危漏洞平均修复时间 < 72 小时。

## 面试高频问答

Q: OpenVAS 和 Nessus 怎么选？

A: 
- 预算有限 / 需要定制规则 / 学习研究 -> 选 OpenVAS
- 企业生产环境 / 需要商业支持 / 需要美观报告 -> 选 Nessus
- 很多企业两者都用：OpenVAS 日常扫描 + Nessus 合规审计

Q: 漏洞扫描会对生产系统造成影响吗？

A: 会，需要注意：
1. 大量探测可能消耗目标 CPU/内存 -> 限制并发、选择低峰时段
2. 某些探测 payload 可能触发应用崩溃 -> 先做端口扫描再针对漏洞检测
3. 扫描流量会触发 IDS/IPS 告警 -> 提前通知安全值班团队
4. 目标系统日志被扫描记录填满 -> 事后清理相关日志

最佳实践：在业务低峰时段（凌晨 2-5 点）、限制并发连接数、
排除关键系统、先在测试环境验证扫描配置。

Q: 发现高危漏洞后的标准处置流程是什么？

A:
1. 核实确认（30 分钟）：在测试环境复现，排除误报
2. 风险评估（15 分钟）：评估影响范围、利用难度、数据敏感度
3. 应急止血（1 小时）：WAF 临时阻断 / 关端口 / 限访问
4. 根治修复（根据复杂度）：打补丁 / 改配置 / 代码修复
5. 验证闭环（1 小时）：复扫确认 + 回归测试 + 报告归档

## 案例分析：一次 OpenVAS 发现的高危漏洞处置

背景：某公司使用 OpenVAS 对内网 Web 服务器做月度扫描。

扫描结果：
目标: 订单系统 192.168.1.50
高危: 3 个 | 中危: 8 个 | 低危: 15 个

重点高危漏洞：Apache Struts2 S2-045 (CVE-2017-5638)
- CVSS: 10.0（最高危）-> 远程代码执行！
- 影响：攻击者可上传恶意 Content-Type 实现 RCE

处置过程：
1. 确认阶段：在测试环境部署相同 Struts2 版本，POC 验证 -> 确认存在
2. 应急止血：WAF 加入 S2-045 临时拦截规则
3. 根治修复：升级 Struts2 至最新安全版本（非工作时间）
4. 验证闭环：OpenVAS 复扫确认修复 -> 业务回归测试 -> 报告归档

经验教训：
- S2-045 是 2017 年的漏洞！说明系统长期无人维护
- 必须建立周期性扫描机制，不能部署即遗忘
- 老系统是高危漏洞的重灾区

## 补充专题：OpenVAS 实战中的常见问题与解决方案

### 问题一：扫描速度太慢怎么办？
OpenVAS 的 Full and very deep 扫描可能耗时 4-8 小时，
在生产环境中不现实。

优化策略：
1. 分阶段扫描：先做端口扫描（5 分钟）-> 再做漏洞检测（1-2 小时）
2. 限制扫描范围：只扫关键端口（80/443/22/3306/6379 等），不全量
3. 分批扫描：不要一次扫描 /24 网段，分成 4 批次
4. 设置并发上限：减少同时进行的探测数
5. 对非关键资产使用 Base 或 Full and fast 配置

### 问题二：扫描结果太多看不过来
一个中小型企业内网扫描一次可能产生几千个漏洞。

筛选策略：
1. 按严重级别：只看 Critical + High（通常只占总数的 10-20%）
2. 按可远程利用：优先看不需要认证的远程漏洞
3. 按漏洞年龄：新发现的漏洞 > 老漏洞（老漏洞可能已经被修复但报告残留）
4. 按资产重要性：核心业务服务器 > 边缘测试服务器

### 问题三：扫描被防火墙/IDS 拦截
这是正常的——大量端口探测必然触发安全设备。

解决方案：
1. 将扫描器 IP 加入防火墙/IDS 白名单
2. 提前通知安全值班团队：「今晚 2-5 点进行漏洞扫描，预计有告警风暴」
3. 降低扫描速率：减少并发数，延长扫描间隔
4. 从内网进行扫描（不需要经过外网防火墙）

### 问题四：修复后复扫还是不通过
打补丁后 OpenVAS 仍然报告漏洞存在。

可能原因：
1. 补丁没生效：需要重启服务或系统
2. Backport 补丁：厂商只修复了漏洞不改版本号 -> 标记为误报
3. 多个应用实例：只修了一个，还有其他实例
4. 残留配置文件：虽然是同一漏洞但修复不完整

### 漏洞扫描工具链建议

除了 OpenVAS，建议蓝队同时使用以下工具形成完整的扫描体系：

| 工具 | 用途 | 扫描频率 |
|:---|:---|:---|
| OpenVAS | 系统和网络漏洞扫描 | 每月全量 + 每周增量 |
| Nuclei | Web 应用快速 POC 验证 | 每周 |
| Trivy | 容器镜像漏洞扫描 | 每次构建 / 每天 |
| Dependency-Check | 第三方依赖漏洞 | 每次代码提交 |
| Nmap | 端口和资产发现 | 每周 |

## 蓝队实战心得：漏洞优先级排序的四象限法则

面对几十上百个漏洞，不可能全都马上修复。
按以下标准给漏洞排序优先级：

第一象限（立即修复 - 24 小时内）：
- CVSS >= 9.0
- 可远程利用、无需认证
- 影响核心业务系统
- 存在公开的 POC/EXP

第二象限（尽快修复 - 72 小时内）：
- CVSS 7.0-8.9
- 需要认证但认证门槛低
- 影响核心业务

第三象限（计划修复 - 1 周内）：
- CVSS 4.0-6.9
- 利用条件苛刻
- 影响非关键系统

第四象限（跟踪观察 - 1 月内）：
- CVSS < 4.0
- 需要物理接触或极高权限
- 已被其他防护措施缓解

记住一个原则：不要试图修复所有漏洞。
专注修复那些攻击者最可能利用的、影响最大的 20% 漏洞，
能消除 80% 的风险。这就是漏洞管理的二八法则。

## OpenVAS 扫描配置详解

### 三种扫描配置的选择策略

什么时候用 Base？
- 日常快速检查
- 新上线的系统首次检查
- 扫描频率超过每天一次时
- 只是想快速了解资产暴露面

什么时候用 Full and fast？
- 常规月度安全评估（最常用）
- 等保测评前的自检
- 系统变更后的安全检查
- 新漏洞爆发后全网排查

什么时候用 Full and very deep？
- 年度安全审计
- 合规检查（如 PCI-DSS）
- 发生安全事件后的深度排查
- 对核心资产做最全面的评估

### 扫描调度最佳实践

推荐的三层扫描制度：

第一层 - 每日快速扫（凌晨 3 点，Base配置，30 分钟）
只扫关键资产 + 常见端口，发现当天新增的高危漏洞

第二层 - 每周深度扫（周末凌晨，Full and fast，2 小时）
对所有生产资产做全面扫描

第三层 - 每月全量扫（每月第一个周末，Full and very deep，6-8 小时）
对所有资产（含测试/开发环境）做最全面的扫描

### 报告解读进阶

看报告的正确顺序（不要一上来就看漏洞列表）：
1. 先看概览页 -> 了解整体风险状况（多少高/中/低危）
2. 再看高危漏洞详细列表 -> 按 CVSS 排序，从高到低
3. 对每个高危漏洞 -> 看 Solution 字段（修复建议）
4. 看中危漏洞 -> 评估是否需要立即修复还是可以排期
5. 看低危和信息 -> 记录但不紧急处理

### 漏洞扫描与其他安全工具的关系

漏洞扫描不是孤立的，需要和其他工具配合：
- OpenVAS（发现漏洞）+ AWVS/Xray（Web专项）-> 互补覆盖
- OpenVAS（发现漏洞）+ Metasploit（验证利用）-> 确认漏洞真实性
- OpenVAS（发现漏洞）+ Wazuh/ELK（日志分析）-> 判断是否已被利用
- OpenVAS（发现漏洞）+ Jira（工单系统）-> 自动化修复流程
