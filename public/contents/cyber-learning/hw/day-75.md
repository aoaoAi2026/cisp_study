## 📖 核心知识讲解



### 一、Webshell的本质


Webshell = 一段放在Web服务器上的恶意脚本，攻击者通过浏览器访问它来执行服务器端命令。

通俗比喻：攻击者在你的服务器上装了一个"遥控器"，通过HTTP协议远程控制你的服务器。




### 二、Webshell隐藏手法大全


| 隐藏类别 | 具体手法 | 绕过效果 |
|----|----|----|
| 文件名欺骗 | shell.php伪装成config.inc.php | 看起来像合法文件 |
| 文件后缀 | .phtml/.pht/.php5/.php7等非标准后缀 | 绕过.php黑名单 |
| 图片马 | GIF89a头部+PHP代码，改后缀为.jpg | 绕过文件类型检查 |
| .htaccess注入 | 利用.htaccess将图片解析为PHP | 让合法文件变成后门 |
| 无文件Webshell | 将代码注入内存，不落地文件 | 文件扫描完全失效 |
| 内存马 | Java/PHP通过修改运行时加载恶意类 | 需要内存检测 |
| 日志写马 | 将代码注入到Nginx/Apache访问日志中 | 避开文件上传检测 |
| 编码混淆 | base64/gzdeflate/自定义加密 | 绕过特征库匹配 |
| 混淆变量名 | $a=$_POST['x'];@eval($a) | 绕过eval($_POST)特征 |
| 替换函数 | assert()替代eval()、preg_replace的/e | 绕过函数黑名单 |
| 二次拆分 | $_POST拆成多个变量再拼接 | 绕过变量名黑名单 |
| 隐藏目录 | 放在.开头的隐藏目录或深山目录中 | 普通ls命令不易发现 |
| 修改时间戳 | touch -r 伪装创建时间 | 按时间排查找不到 |




### 三、Webshell检测方法（多层防御）




**第一层：流量检测（网络层）**

- 检测HTTP流量中的恶意函数特征：`eval`、`assert`、`system`、`exec`

- 检测异常User-Agent（Webshell客户端通常有特征UA）

- 检测心跳包：Webshell客户端经常定时连接维持会话

- 检测异常请求URL：如/shell.php?cmd=whoami

- 检测通信特征：蚁剑/冰蝎/哥斯拉各有独特的加密通信模式



**第二层：文件检测（主机层）**

- 静态特征：正则匹配已知Webshell代码特征

- 文件熵值：正常PHP文件熵值低，混淆Webshell熵值高

- 文件修改时间：Web目录文件的异常修改时间

- 文件属性：上传Webshell的创建者/权限可能异常

- 文件对比：和之前的文件列表做diff，发现新增/修改



**第三层：行为检测（运行时）**

- Web进程发起的异常网络连接（连接外网）

- Web进程执行的系统命令（fork出的子进程）

- 异常进程：www-data用户执行了/bin/bash

- 文件操作：PHP进程写入了.php文件以外的文件




### 四、常见Webshell管理工具特征


| 工具 | 流量特征 | 检测方法 |
|--|----|----|
| 蚁剑(AntSword) | POST载荷base64编码，默认UA: antsword/v2.x | 检测base64内容解码后的eval/system |
| 冰蝎(Behinder) | AES加密通信，密钥协商阶段特征明显 | 检测AES密钥交换特征包 |
| 哥斯拉(Godzilla) | 高强度加密，流量随机化 | 检测请求体长度异常+编码特征 |
| 中国菜刀(Chopper) | Payload较短，eval+base64_decode | 检测eval(base64_decode)特征 |

## 💎 蓝队实战精华：从日志中发现攻击者

今天的主题在实际工作中可能不会每天遇到，但一旦遇到就是大事。
作为蓝队，你要培养一种"直觉"——当你看着日志的时候，哪些是正常的、哪些是异常的。

### 培养"安全直觉"的方法
1. **先看100个小时的正常日志**。是的，就是纯看。知道"正常"长什么样，才能识别"异常"。
2. **搭建实验环境复现攻击**。看到了攻击的真实面貌，下次在日志中遇到就不会错过。
3. **关注"不和谐"**。比如：平时这个时间没人登录，今天凌晨3点有登录→可疑。
4. **关注"新东西"**。新进程、新服务、新计划任务、新网络连接→第一反应应该是怀疑。
5. **关注"量变"**。平时的日志量是100条/小时，突然变成1000条/小时→出事了。

### 分析未知攻击的思路
当你遇到一条看不懂的告警时：
1. 先确认基本事实：时间、来源IP、目标IP、涉及进程/服务
2. 搜索关键信息：把告警中的关键字放到Google/安全社区搜索
3. 找同类对比：搜索今天的主题+case study
4. 搭建复现：在隔离环境中复现看到的异常行为
5. 总结提炼：把这次的经验变成一条新的检测规则

> 🌟 记住：蓝队不需要比所有攻击者都聪明，你只需要比"昨天的自己"更聪明。

## 🔍 深入剖析：为什么安全基线总是不达标？

做了这么多年的安全基线检查，你会发现一个"奇怪"的现象：
同样的问题反复出现——空口令、默认配置、多余服务、过期补丁。

**根因分析：**
1. **运维和安全脱节**：运维装系统→安全去查→查出问题→运维说"改不了，会影响业务"→不了了之
2. **没有自动化**：每次检查靠手工，查完就忘了，下次继续同样的坑
3. **基线"一刀切"**：所有系统用同一套基线，导致很多项不适用，合规率"虚假"偏低
4. **新增资产无管控**：新上的系统没人做安全配置，自由生长

**解决思路：**
- 安全基线应该是上线流程的一部分（安全左移），不是上线后来挑刺的
- 用配置管理工具（Ansible/Salt/Puppet/GPO）自动化基线部署
- 分级管理：核心系统严格基线、一般系统标准基线、测试环境基本基线
- 定期自动扫描+不合规告警，形成"扫描→整改→验证→固化"的闭环

## 🎯 实战经验分享：一次主机安全加固的正确姿势

场景：你接到任务，要给50台Linux服务器做安全加固，怎么做？

**错误做法：** 一台一台登上去手工改配置。（做完最后一台，第一台的配置可能已经漂移了）

**正确做法：**
1. 先在1台模板机上做好所有加固
2. 用ansible/salt把配置写成playbook
3. 在3台测试机上跑一遍，验证业务不受影响
4. 分批滚动推送到50台（每批5台，观察无异常再继续）
5. 加固完成后，立刻用基线扫描工具验证
6. 每周自动重新扫描，防止配置漂移
7. 新增机器默认走加固playbook

**关键原则：**
- 可重复 > 一次性
- 自动化 > 手工
- 验证 > 信任
