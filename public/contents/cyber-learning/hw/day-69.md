## 📖 核心知识讲解



### 一、SQLMap是什么？


SQLMap是开源世界最强大的SQL注入自动化工具，用Python写成。

它能自动检测SQL注入点、获取数据库信息、导出数据、甚至执行系统命令（在有条件时）。

作为蓝队，理解SQLMap不是因为你要用，而是因为**攻击者都在用**，你必须知道它的特征。




### 二、SQLMap常用参数速查


| 参数 | 作用 | 示例 |
|--|--|--|
| -u | 指定目标URL | -u 'http://test.com/page.php?id=1' |
| --dbs | 列出所有数据库 | --dbs |
| -D | 指定数据库 | -D dbname --tables |
| --tables | 列出表 | -D dbname --tables |
| -T | 指定表 | -T users --columns |
| --columns | 列出列 | -T users --columns |
| --dump | 导出数据 | -T users --dump |
| --os-shell | 获取系统shell | --os-shell（需要dba权限） |
| --level | 测试等级(1-5) | --level=3（更全面的测试） |
| --risk | 风险等级(1-3) | --risk=2（更高风险的Payload） |
| --tamper | 使用绕过脚本 | --tamper=space2comment（绕过WAF） |
| --random-agent | 随机User-Agent | 避免被固定UA检测 |
| --delay | 请求间隔(秒) | --delay=2（降低频率避检） |
| --proxy | 使用代理 | --proxy='http://127.0.0.1:8080' |




### 三、SQLMap的流量特征（蓝队重点）


**SQLMap发起的请求有一些"指纹"特征，可以用来检测：**



**1. 默认User-Agent**

SQLMap默认UA包含 `sqlmap/` 字符串，如：

但攻击者可以用 `--random-agent` 或 `--user-agent` 修改，所以不能只靠UA。



**2. 请求参数特征**

- 大量SQL关键字：UNION、SELECT、SLEEP、BENCHMARK、WAITFOR DELAY等

- 明显的SQL注入测试字符串：`'`、`"`、`AND 1=1`、`OR 1=1`

- 时间盲注的延时参数：`SLEEP(5)`、`pg_sleep(5)`、`WAITFOR DELAY '0:0:5'`

- 布尔盲注的数学表达式：`AND 7175=7175`、`AND 1=2`



**3. URI路径特征**

- 大量重复请求同一URL（注入测试需要多发请求）

- 请求间隔非常规律（特别是加了--delay的情况）

- 参数值变化模式固定（如id=1, id=2, id=3...) 但夹杂了SQL片段



**4. 响应特征**

- 响应时间明显延长（时间盲注中SLEEP导致）

- 响应长度规律性变化（布尔盲注中不同条件的返回不同）

- 数据库错误信息泄露（语法错误被返回到前端）




### 四、SQLMap绕过WAF的tamper脚本


SQLMap自带大量tamper脚本用于绕过WAF，蓝队需要了解这些绕过手法：



| Tamper脚本 | 绕过手法 | 检测对抗思路 |
|--------|----|------|
| space2comment | 空格替换为/**/注释 | 检测SQL关键字间有/**/ |
| between | >替换为BETWEEN | 扩展BETWEEN关键字的检测 |
| equaltolike | =替换为LIKE | 检测LIKE的注入模式 |
| randomcase | 关键字大小写随机 | 大小写不敏感的检测规则 |
| space2plus | 空格替换为+ | 检测URL中+号分隔的SQL片段 |
| charunicodeencode | 字符Unicode编码 | 解码后再检测 |
| commalesslimit | LIMIT改为OFFSET | 检测OFFSET注入 |
| percentage | 每个字符前加% | 解码后检测 |
| charencode | URL全编码 | 多次递归解码后检测 |




### 五、蓝队如何检测SQLMap攻击


**多层检测策略：**



**第一层：WAF规则**（实时拦截）

- 检测明显的SQL注入关键字组合

- 检测异常时间响应（单个请求响应时间>5秒）

- 频率限制：同一IP短时间内大量请求同一接口



**第二层：SIEM/SOC分析**（近实时/准实时）

- 关联分析：同一IP对多个参数进行注入测试

- 行为分析：某个IP的请求模式突然变化

- 告警聚合：同一源IP的多条SQL注入告警归并为一个事件



**第三层：日志审计**（事后分析）

- Web访问日志中搜索SQLMap的特征字符串

- 分析高延时请求的分布和规律

- 回溯攻击时间窗口内的所有请求

## 🔬 SQLMap流量深度分析

**SQLMap请求的生命周期：**

第一轮：指纹识别
- 发几个正常请求，分析响应内容和响应时间
- 确定注入点是否存在（基于响应差异）

第二轮：技术选择
- 逐个尝试注入技术：Boolean盲注→Error-based→Union→Time-based
- 每种技术发几个测试请求来验证可行性

第三轮：数据获取
- 一旦确定可行技术，开始大量请求获取数据
- 这是流量最密集的阶段

**用Wireshark分析SQLMap流量的关键步骤：**
1. 过滤HTTP流量：`http.request`
2. 统计请求频率：Statistics→HTTP→Requests
3. 导出HTTP对象：File→Export Objects→HTTP
4. 关注请求中出现的SQL关键字变化模式
5. 观察响应时间的规律性（时间盲注）

**检测盲区：**
- 如果攻击者用了 `--delay=5 --random-agent --tamper=space2comment`
- 流量看起来变"慢"和"普通"了
- 这时候要靠：较长时间窗口内的行为分析和SQL语法模式识别

> 💡 SQLMap检测的核心：不是在单次请求中发现SQLMap，而是在一段时间的行为中识别出"自动化SQL注入扫描"的模式。
