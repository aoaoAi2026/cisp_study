## 📖 核心知识讲解



### 一、横向移动是什么？


攻击者拿到第一台机器的权限后，不会停下——他们会像"病毒"一样在内网扩散，寻找更有价值的目标。

这个过程就是"横向移动"（Lateral Movement）。



类比：小偷进了小区，撬开第一户人家的门，发现没啥值钱的，于是从阳台翻进隔壁。




### 二、横向移动核心技术


| 技术 | 原理 | 所需权限 | 典型工具 |
|--|--|----|----|
| PTH (Pass The Hash) | 使用NTLM Hash替代明文密码进行认证 | 管理员Hash | mimikatz/impacket |
| PTT (Pass The Ticket) | 使用Kerberos票据进行认证 | 有效票据 | mimikatz/Rubeus |
| PsExec | 通过SMB共享写入服务文件并执行 | 目标管理员权限 | Sysinternals PsExec |
| WMI | 通过Windows WMI执行远程命令 | 目标管理员权限 | wmic/impacket-wmiexec |
| WinRM | 通过PowerShell Remoting远程管理 | 目标管理员权限 | Enter-PSSession |
| SMB/RDP | 直接通过文件共享或远程桌面 | 目标有效凭据 | net use / mstsc |
| DCOM | 通过COM对象执行远程操作 | 特定COM权限 | MMC20.Application等 |
| SSH | 通过SSH跳转（Linux环境） | SSH凭据 | ssh / scp |




### 三、横向移动的日志检测（重点！）


**关键Event ID速查表：**

| Event ID | 含义 | 攻击场景 | 检测要点 |
|--------|--|----|----|
| 4624 | 账号登录成功 | 任何横向移动成功后都会产生 | 关注登录类型（3=网络,10=远程交互） |
| 4625 | 账号登录失败 | 密码爆破/PTH失败 | 短时间内大量4625+同一源IP |
| 4648 | 使用显式凭据登录 | runas/PsExec指定不同账号 | 主体账号≠新登录账号 |
| 4672 | 特殊权限登录 | 获取了管理员权限 | 普通用户触发此事件 |
| 4768 | Kerberos TGT请求 | PTT攻击前奏 | 异常加密类型(0x17=RC4可能是mimikatz) |
| 4769 | Kerberos服务票据请求 | PTT获取服务票据 | 异常的服务名/IP |
| 4776 | NTLM认证尝试 | PTH攻击 | 源工作站名称异常 |
| 4688 | 新进程创建 | 执行命令/工具 | 可疑进程：cmd.exe/powershell.exe以SYSTEM运行 |
| 7045 | 新服务安装 | PsExec创建服务 | 服务名随机或无意义 |
| 5140 | 文件共享访问 | 通过SMB访问C$等共享 | 对C$/ADMIN$的异常访问 |




### 四、PTH攻击深度检测


PTH攻击的检测是蓝队的内网基本功。

**PTH的日志特征：**

1. 事件4624（登录成功）+ 登录类型3（网络登录）

2. 认证包名称为NTLM（而非Kerberos）→ 正常域内登录应该用Kerberos

3. 源IP不是该用户常用的IP

4. 短时间内同一用户从多个源IP登录






### 五、权限提升检测


**Linux提权检测：**

- 命令历史中的提权操作：`sudo`、`su`、SUID文件

- 进程的UID/EUID不一致（UID=1000但EUID=0）

- 新增的SUID文件：`find / -perm -4000 -newer /tmp/reference`



**Windows提权检测：**

- 事件4672：非管理员用户获取了特殊权限

- 新进程以SYSTEM权限运行且父进程异常

- 令牌权限变化：SeDebugPrivilege被赋予不该有的进程

## 💎 蓝队实战精华：从日志中发现攻击者

今天的主题在实际工作中可能不会每天遇到，但一旦遇到就是大事。
作为蓝队，你要培养一种"直觉"——当你看着日志的时候，哪些是正常的、哪些是异常的。

### 培养"安全直觉"的方法
1. **先看100个小时的正常日志**。是的，就是纯看。知道"正常"长什么样，才能识别"异常"。
2. **搭建实验环境复现攻击**。看到了攻击的真实面貌，下次在日志中遇到就不会错过。
3. **关注"不和谐"**。比如：平时这个时间没人登录，今天凌晨3点有登录→可疑。
4. **关注"新东西"**。新进程、新服务、新计划任务、新网络连接→第一反应应该是怀疑。
5. **关注"量变"**。平时的日志量是100条/小时，突然变成1000条/小时→出事了。

### 分析未知攻击的思路
当你遇到一条看不懂的告警时：
1. 先确认基本事实：时间、来源IP、目标IP、涉及进程/服务
2. 搜索关键信息：把告警中的关键字放到Google/安全社区搜索
3. 找同类对比：搜索今天的主题+case study
4. 搭建复现：在隔离环境中复现看到的异常行为
5. 总结提炼：把这次的经验变成一条新的检测规则

> 🌟 记住：蓝队不需要比所有攻击者都聪明，你只需要比"昨天的自己"更聪明。

## 🔍 深入剖析：为什么安全基线总是不达标？

做了这么多年的安全基线检查，你会发现一个"奇怪"的现象：
同样的问题反复出现——空口令、默认配置、多余服务、过期补丁。

**根因分析：**
1. **运维和安全脱节**：运维装系统→安全去查→查出问题→运维说"改不了，会影响业务"→不了了之
2. **没有自动化**：每次检查靠手工，查完就忘了，下次继续同样的坑
3. **基线"一刀切"**：所有系统用同一套基线，导致很多项不适用，合规率"虚假"偏低
4. **新增资产无管控**：新上的系统没人做安全配置，自由生长

**解决思路：**
- 安全基线应该是上线流程的一部分（安全左移），不是上线后来挑刺的
- 用配置管理工具（Ansible/Salt/Puppet/GPO）自动化基线部署
- 分级管理：核心系统严格基线、一般系统标准基线、测试环境基本基线
- 定期自动扫描+不合规告警，形成"扫描→整改→验证→固化"的闭环

## 🎯 实战经验分享：一次主机安全加固的正确姿势

场景：你接到任务，要给50台Linux服务器做安全加固，怎么做？

**错误做法：** 一台一台登上去手工改配置。（做完最后一台，第一台的配置可能已经漂移了）

**正确做法：**
1. 先在1台模板机上做好所有加固
2. 用ansible/salt把配置写成playbook
3. 在3台测试机上跑一遍，验证业务不受影响
4. 分批滚动推送到50台（每批5台，观察无异常再继续）
5. 加固完成后，立刻用基线扫描工具验证
6. 每周自动重新扫描，防止配置漂移
7. 新增机器默认走加固playbook

**关键原则：**
- 可重复 > 一次性
- 自动化 > 手工
- 验证 > 信任
