## 什么是VulHub？

VulHub是一个开源的漏洞复现环境集合，由国内安全团队维护。
它的设计理念是：**一个漏洞 = 一个Docker Compose文件 = 一键启动**。

截止目前VulHub收录了数百个真实漏洞环境，涵盖：
- Web中间件漏洞（Tomcat、Weblogic、JBoss、Nginx...）
- Web应用漏洞（Struts2、Shiro、Fastjson、ThinkPHP...）
- 数据库漏洞（Redis、MySQL、PostgreSQL...）
- 系统服务漏洞（Samba、Docker、Kubernetes...）
- 常见CMS漏洞（WordPress、Drupal、Discuz...）

## VulHub环境搭建（三步完成）

### 第一步：安装Docker和Docker Compose

**Ubuntu/Debian：**
```
sudo apt update
sudo apt install docker.io docker-compose -y
sudo systemctl start docker
sudo usermod -aG docker $USER  # 让自己不用sudo也能用docker
```

**CentOS/RHEL：**
```
sudo yum install docker docker-compose -y
sudo systemctl start docker
sudo usermod -aG docker $USER
```

**Windows：**
安装 Docker Desktop for Windows：
1. 从 docker.com 下载 Docker Desktop
2. 安装后重启电脑
3. 在PowerShell中运行 docker --version 确认安装成功

### 第二步：下载VulHub

```
git clone https://github.com/vulhub/vulhub.git
cd vulhub
```

如果下载慢，可以用国内镜像：
```
git clone https://gitee.com/puier/vulhub.git
```

### 第三步：启动第一个漏洞环境

以Tomcat漏洞为例：
```
cd vulhub/tomcat/CVE-2017-12615
docker-compose up -d
```

打开浏览器访问 http://localhost:8080 ，看到Tomcat页面就说明成功了。

## VulHub核心操作命令

| 命令 | 作用 | 使用场景 |
|:---|:---|:---|
| docker-compose up -d | 启动漏洞环境 | 开始复现漏洞 |
| docker-compose down | 关闭并清理环境 | 漏洞复现完毕 |

| docker-compose logs | 查看服务日志 | 排查启动问题 |
| docker ps | 查看所有运行容器 | 确认端口和状态 |

## 实战思维训练：从理论到肌肉记忆

### 为什么要做实战思维训练？

理论学完了，不等于实战中用得上。蓝队工作中最常见的问题是：
"我知道SQL注入是什么，但真看到日志时反应不过来。"

原因：知识和行动之间缺少"条件反射"训练。就像学开车——
知道刹车在哪儿和紧急情况能踩刹车是两回事。

### 蓝队条件反射训练法

**方法一：5秒判断法**

给自己看一条日志，5秒内回答三个问题：
1. 这是攻击吗？（是/否/不确定）
2. 如果是攻击，是什么类型？
3. 我应该做什么？（忽略/记录/告警/升级）

每天练10条，坚持一周，判断速度会明显提升。

**方法二：攻击路径推演法**

看到一条可疑日志后，在心里推演攻击者的下一步：
- SQL注入检测到 -> 攻击者下一步会尝试 UNION SELECT 查数据
- Webshell上传成功 -> 下一步会访问shell执行命令
- 端口扫描发现 -> 下一步会针对开放的服务做漏洞利用

提前想到攻击者的下一步，你就能先发制人。

**方法三：误报记忆库法**

每当你发现一个误报，记录下来：
- 误报的类型（SQL注入/XSS/目录遍历等）
- 触发误报的正常业务场景
- 你如何判断它是误报的

积累100个误报案例后，你的判断准确率会翻倍。

## 蓝队面试高频自测

**1. 请描述一次完整的Web攻击检测流程**
参考答案：收到告警 -> 查看日志确认 -> 分析Payload -> 判断类型 -> 确认是否成功 -> 处置 -> 写报告。

**2. 什么是IDS/IPS？区别是什么？**
IDS（入侵检测系统）只检测不阻断，IPS（入侵防御系统）检测+阻断。IDS像摄像头（记录），IPS像门禁（拦截）。

**3. ATT&CK框架对蓝队有什么用？**
ATT&CK告诉我们攻击者"会做什么"和"怎么做的"，蓝队据此在对应的战术阶段设置检测点。

**4. 告警很多怎么处理？**
先分级（高危立即处理、中危今日内、低危周汇总），再优化规则减少误报。

## VulHub常见踩坑指南

### Docker拉取镜像太慢
配置国内镜像加速器（阿里云、中科大等）：
编辑 /etc/docker/daemon.json：
```
{
  "registry-mirrors": ["https://your-id.mirror.aliyuncs.com"]
}
```
然后 sudo systemctl restart docker。

### 端口冲突
如果8080端口被占用，修改docker-compose.yml中的端口映射：
```
ports:
  - "8081:8080"  # 改成8081
```

### 环境启动后打不开
1. 检查防火墙是否开放了端口
2. 检查 docker-compose ps 中各服务是否都是 Up 状态
3. 查看日志 docker-compose logs

### Windows Docker Desktop 问题
确保开启了虚拟化（BIOS -> VT-x/AMD-V -> Enabled）
确保WSL2已安装（Windows 10/11的新版Docker依赖WSL2）

## VulHub深入使用指南

### 如何搜索特定的漏洞环境

VulHub收录的漏洞很多，找到你需要的：

```
cd vulhub
ls -d */   # 列出所有组件
ls tomcat/   # 列出Tomcat的所有漏洞
```

按组件名搜索：VulHub按组件名组织目录（tomcat/、weblogic/、redis/、struts2/、shiro/...）

按CVE编号搜索：
```
find . -name "CVE-2017-*" -type d
```

### VulHub目录结构详解

以 tomcat/CVE-2017-12615 为例：

```
tomcat/CVE-2017-12615/
├── docker-compose.yml   # 环境配置（端口、镜像、依赖）
├── README.md            # 漏洞说明文档（中文！非常详细）
├── Dockerfile           # 漏洞环境的Docker构建文件
└── index.jsp            # 有时会包含示例文件
```

docker-compose.yml 是核心，定义了：
- 端口映射（ports）
- 漏洞镜像版本（image）
- 如需数据库等依赖也在此定义

### 蓝队学习VulHub的正确姿势

不要急于"一键启动->打PoC->结束"。按照这个流程：

1. **先读README.md**：理解漏洞原理、影响版本、利用条件
2. **启动环境，正常访问**：先看正常的服务是什么样
3. **用公开PoC验证**：观察攻击请求的每一个参数
4. **用Wireshark抓包**：记录完整的攻击流量
5. **分析流量特征**：提取可检测的规则
6. **写检测规则**：在WAF/IDS中编写规则
7. **验证规则**：再次攻击，看规则是否能检测到
8. **写学习笔记**：记录漏洞原理、检测特征、加固方法

### VulHub环境管理技巧

批量关闭所有环境：
```
cd vulhub
for dir in */; do
    cd $dir
    docker-compose down 2>/dev/null
    cd ..
done
```

不用时清理无用镜像（节省磁盘）：
```
docker system prune -a
```

## Docker安全须知

在学VulHub时顺便了解Docker的安全问题（你自己可能也会遇到）：

1. 不要把Docker socket暴露到公网（/var/run/docker.sock）
2. 不要在容器里挂载宿主机的敏感目录
3. 用完的容器及时 docker-compose down
4. 不要用 --privileged 模式运行测试容器
5. 关注Docker逃逸漏洞（如CVE-2019-5736 runc漏洞）

蓝队视角：如果发现内部有未授权的Docker容器在运行，这是严重的安全事件。
Docker容器逃逸 = 宿主机权限。

## 面试题

**Q: VulHub和DVWA有什么区别？**
A: VulHub是单个漏洞的独立环境集合（每个漏洞一个Docker容器），适合深入学习特定漏洞。
DVWA是一个包含多种漏洞的单一Web应用，适合入门学习多种漏洞类型。
简单说：VulHub是漏洞"专科医院"，DVWA是"全科诊所"。

**Q: Docker在安全学习中的价值是什么？**
A: (1)环境隔离——不会污染本机 (2)快速部署——一行命令启动漏洞环境
(3)可复现——同样的docker-compose.yml在任何机器上结果一致 (4)用完即删——docker-compose down后不留痕迹。

## 蓝队日常演练：模拟真实护网场景

### 场景一：凌晨3点收到Redis告警

**场景描述：**
你值夜班，凌晨3点监控平台弹出告警：一台服务器（10.0.1.50）的Redis端口（6379）
收到了来自境外IP（45.x.x.x）的连接请求，并且成功执行了 CONFIG SET 命令。

**你的处置流程：**

1. **第一反应（0-30秒）**：P0告警，立即确认
   - 打开ELK，搜索源IP 45.x.x.x 的所有日志
   - 发现该IP在过去10分钟内一直在对内网做端口扫描

2. **分析阶段（30秒-2分钟）**：
   - 查看Redis日志：发现执行了 CONFIG SET dir /root/.ssh/ 和 CONFIG SET dbfilename authorized_keys
   - 判断：攻击者正在尝试写入SSH公钥，获取服务器控制权！

3. **处置阶段（2-5分钟）**：
   - 立即在防火墙上封禁 45.x.x.x
   - SSH登录 10.0.1.50，检查 /root/.ssh/authorized_keys 是否有异常公钥
   - 给Redis设置临时密码：CONFIG SET requirepass "emergency_pass_2026"
   - 通知组长和运维

4. **善后阶段（5-30分钟）**：
   - 排查同一网段其他Redis是否也有类似配置问题
   - 检查是否有后续的SSH登录记录（如果有，说明已被入侵）
   - 写应急处置报告

**这个场景的启示：**
- Redis未授权访问不是"小问题"，它可以导致服务器被完全控制
- 凌晨的攻击需要特别重视，因为攻击者知道这个时候防守最薄弱
- 处置要快：从发现到封禁最好在5分钟内完成

### 场景二：办公网发现Weblogic扫描流量

**场景描述：**
IDS检测到某内网IP（192.168.1.100）在扫描多个服务器的7001端口，
并发送了带有 ProcessBuilder 关键字的SOAP请求。

**判断思路：**
内网IP发起Weblogic攻击 -> 两台可能：
1. 安全团队在做渗透测试（查一下有没有授权的测试计划）
2. 这台机器已经被黑客控制，正在内网横向移动

查询后发现当天没有测试计划 -> 选项2成立！

**处置流程：**
1. 立即断网这台机器（但不关机，保留内存证据）
2. 检查该机器的进程列表、网络连接、计划任务
3. 发现了一个可疑的进程：java -jar reverser_shell.jar
4. 追踪攻击入口：该机器上运行着一个未修复的Shiro应用

**教训：**
- 内网的异常扫描比外网扫描危险100倍，因为说明边界已被突破
- 不看来源IP的"态度"（内网=信任）是安全的大忌
- 一旦确认内网机器被控，必须立即隔离

## VulHub快速上手补充

### 效率技巧

**技巧1：给常用的docker-compose命令设别名**
```bash
alias vup='docker-compose up -d'
alias vdown='docker-compose down'
alias vlog='docker-compose logs -f'
```
在 .bashrc 或 .zshrc 中加入这些别名，操作VulHub快很多。

**技巧2：预下载常用镜像**
第一次启动环境时会下载镜像，比较慢。可以提前下载常用镜像：
```bash
docker pull vulhub/tomcat:7.0.81
docker pull vulhub/weblogic:10.3.6.0
```
