## 📖 核心知识讲解

### 一、XSS跨站脚本攻击——三种类型深度解析

XSS（Cross-Site Scripting）是OWASP Top 10的常客，攻击者在网页中注入恶意脚本，当其他用户访问时脚本被执行——窃取Cookie、劫持会话、钓鱼、键盘记录等。

**类型1：反射型XSS（非持久型）**
- 攻击方式：恶意脚本通过URL参数/表单提交→服务器"反射"回响应页面→浏览器执行
- 示例URL：`/search.php?q=<script>alert(1)</script>` → 搜索词被直接输出到页面HTML中
- 蓝队检测：URL参数中包含<script>、javascript:、onerror=等标签。同一参数在响应HTML中直接出现（反射型标志）

**类型2：存储型XSS（持久型）**
- 攻击方式：恶意脚本被存储到数据库（评论/昵称/文章内容）→其他用户访问时从数据库读取并执行
- 危害最大：每个访问该页面的用户都会被攻击——"一颗炸弹炸所有人"
- 蓝队检测：检测用户提交到服务器的内容中是否包含脚本标签，数据库中存储的数据是否包含可执行脚本

**类型3：DOM型XSS**
- 攻击方式：恶意脚本不经过服务器，直接在浏览器端通过修改DOM执行
- 特殊之处：服务器的响应HTML中看不到恶意脚本（纯前端问题）→传统WAF日志检测不到
- 蓝队检测：检查前端JS代码中是否使用了不安全的DOM操作（innerHTML/document.write/eval）

**检测命令**：
```bash
# 从日志中检测XSS攻击（反射型+存储型的投递阶段）
grep -iE '(<script|javascript:|onerror=|onload=|onclick=|alert\\(|document\\.cookie|img src=x)' access.log

# 检测编码绕过的XSS（URL编码/十六进制编码）
grep -iE '(%3Cscript|%3Cimg|\\x3cscript|\\u003cscript)' access.log
```

### 二、CSRF跨站请求伪造——"借刀杀人"的攻击

CSRF（Cross-Site Request Forgery）的核心原理：攻击者诱导用户点击一个恶意链接→该链接以用户的身份向目标网站发送非预期的请求→"借"用户的"刀"执行操作。

**攻击必要条件**：
1. 用户已在目标网站登录（浏览器携带有合法的Cookie/Session）
2. 目标网站的关键操作可以通过简单的GET/POST请求完成（如转账/修改密码/删除数据）
3. 目标网站没有验证请求来源（Referer/Origin头）和CSRF Token

**典型攻击场景**：
- 用户在bank.com已登录→收到一封邮件包含`<img src="https://bank.com/transfer?to=attacker&amount=10000">`→用户打开邮件→浏览器自动加载图片→以用户身份发起转账→钱被转走

**蓝队检测思路**：
- 检查敏感操作是否都有CSRF Token验证（burp/浏览器F12查看请求参数）
- 监控Referer/Origin头异常（正常请求的Referer应该来自本站）
- 关注同一用户短时间内执行关键操作的频率异常（正常用户不会1秒内转账10次）

**防御方案**：
| 方案 | 原理 | 蓝队检查要点 |
| --- | --- | --- |
| CSRF Token | 每次请求携带随机Token | Token是否存在、是否可预测 |
| SameSite Cookie | Cookie附带SameSite属性 | 检查Set-Cookie头是否有SameSite |
| Referer验证 | 验证请求来源 | 检查是否可被伪造（空Referer绕过） |
| 二次验证 | 敏感操作需要密码/验证码 | 所有敏感操作是否有二次确认 |

### 三、SSRF服务端请求伪造——"让服务器当跳板"

SSRF（Server-Side Request Forgery）让攻击者以服务器为跳板，请求内网资源。因为请求是从"服务器"发出的（而非攻击者IP），可以绕过防火墙访问内网受保护的资源。

**SSRF的危害分级**：
- Level 1：扫描内网（探测开放端口和服务）→ 获取内网拓扑信息
- Level 2：读取服务器本地文件（file:///etc/passwd）→ 信息泄露
- Level 3：访问云元数据服务（169.254.169.254）→ 获取云服务器凭据——"终极危害"
- Level 4：攻击内网Web应用（如内网的Redis/MySQL未授权访问）→ 内网横向

**SSRF的URL参数特征**：
- 参数值为URL格式（如`url=http://xxx`、`image=http://xxx`、`path=http://xxx`）
- 攻击payload包含内网地址：127.0.0.1 / 10.x.x.x / 172.16-31.x.x / 192.168.x.x
- 包含非HTTP协议：file:///、gopher://、dict://、ftp://
- 包含云元数据地址：169.254.169.254（AWS/阿里云/腾讯云等）

**检测命令**：
```bash
# 检测SSRF攻击
grep -iE '(127\\.0\\.0\\.1|10\\.[0-9]+\\.[0-9]+\\.[0-9]+|172\\.(1[6-9]|2[0-9]|3[0-1])\\.|192\\.168\\.|169\\.254\\.169\\.254|file:///|gopher://|dict://)' access.log
```

### 四、XSS/CSRF/SSRF的综合检测矩阵

将三种攻击的检测特征统一到一个矩阵中，建立蓝队综合分析能力：

| 攻击类型 | WAF检测点 | 日志特征 | 响应特征 | 误报陷阱 |
| --- | --- | --- | --- | --- |
| XSS反射型 | <script>等标签 | URL参数含脚本+响应中回显 | 响应HTML中包含payload | URL中含正常script关键词的业务 |
| XSS存储型 | 提交内容含脚本 | 提交表单含脚本 | 后续访问页面含脚本 | 富文本编辑器允许HTML |
| CSRF | 无可见payload | 敏感操作缺少Token/Referer异常 | 操作成功（攻击者角度） | 正常跨域请求（CORS） |
| SSRF | URL参数指向内网 | 参数值为内网/特殊协议URL | 响应中包含内网资源 | 正常的代理/回调功能 |
| DOM XSS | WAF无法检测 | 服务器日志中无特征 | 前端JS异常行为 | 很难从服务器侧检测 |

---

### 五、蓝队防御Web应用漏洞的最佳实践

**从被动检测到主动防御**：

1. **开发阶段**（安全左移）：
   - XSS防御：输出编码（HTML实体编码/JS编码/URL编码）+ CSP头
   - CSRF防御：所有状态变更操作使用CSRF Token + SameSite Cookie
   - SSRF防御：URL白名单+禁用非HTTP协议+内网地址黑名单

2. **测试阶段**：
   - 用BurpSuite/Xray做自动化扫描（SAST+DAST）
   - 人工渗透测试覆盖自动化工具盲区（DOM XSS/逻辑CSRF/SSRF盲打）

3. **运行阶段**（蓝队核心）：
   - WAF配置XSS/SQL注入通用规则 + 针对业务的专用规则
   - 日志监控关键接口（登录/支付/文件上传/URL跳转）的异常访问
   - 定期做攻击模拟验证检测规则是否生效

4. **应急阶段**：
   - XSS攻击确认→清除恶意脚本+修复输出编码+清除受影响用户Session
   - SSRF攻击确认→检查是否已访问内网/云元数据→隔离服务器→修复URL白名单

## 💡 面试高频题

**Q: XSS三种类型的区别是什么？检测重点有什么不同？**

A: 反射型XSS：payload在URL参数中→服务器反射回响应→检测URL参数中的脚本标签。存储型XSS：payload存储在数据库→其他用户访问触发→检测用户提交内容中的脚本。DOM型XSS：纯前端问题→服务器日志看不到→需要前端代码审计。最关键区别：反射型看请求和响应的对应关系，存储型看数据库和输出的对应关系，DOM型只能靠前端代码扫描。

**Q: CSRF和XSS有什么区别和联系？**

A: 区别：XSS需要注入脚本（利用网站对用户的信任），CSRF不需要注入脚本（利用网站对用户浏览器的信任）。联系：如果存在XSS漏洞，可以绕过CSRF的所有防御（因为XSS可以读取CSRF Token/修改请求头/读取响应）——"XSS是万能钥匙，能开CSRF这把锁"。所以XSS比CSRF优先级更高。

**Q: SSRF如何检测内网？为什么蓝队要特别关注？**

A: 攻击者通过SSRF可以让目标服务器访问内网IP段（10.x/172.16.x/192.168.x），防火墙看到的是"服务器在访问内网"而非"攻击者在访问内网"——所以边界防火墙完全看不到。蓝队需要：①在应用层检测URL参数中的内网地址；②对URL跳转/文件下载/图片加载等SSRF高发功能做专项监控；③监控服务器对云元数据服务(169.254.169.254)的访问。

## 📊 真实案例

**2019年Capital One数据泄露——SSRF的实际破坏力**

2019年7月，美国Capital One银行遭遇数据泄露，1.06亿用户的个人信息被窃取。攻击者Paige Thompson利用了一个配置错误的AWS WAF中的SSRF漏洞——通过SSRF访问了AWS的元数据服务（169.254.169.254），获取了临时凭证→进而访问S3存储桶→窃取了大量客户数据。

这是一个教科书级的SSRF攻击案例：1个SSRF漏洞→访问云元数据→获取凭据→访问云存储→大规模数据泄露。Capital One最终被罚款1.9亿美元。

蓝队教训：①永远不要让服务器可以自由访问169.254.169.254；②URL跳转/代理功能必须有严格的白名单；③云环境的IMDSv2可以防御此类元数据访问攻击。

## XSS 攻击三部曲：从原理到防御（通俗易懂版）

### 什么是 XSS？一个生活中的比喻

想象一个场景：你在学校门口的门卫处留下一张纸条，上面写着
【请把这位同学的课本交给他】。门卫（服务器）忠实地把这张纸条
交给了之后进来的每个同学（用户）。

但如果你的纸条上写的是【把书包里的钱放到门卫桌上】，
天真的门卫照样传递——这就是 XSS 的核心原理。

用技术语言说：攻击者在网站里贴了一段恶意脚本，
之后访问这个网站的用户，浏览器会忠实地执行这段脚本——
因为它分不清哪些是网站原本的代码，哪些是攻击者塞进来的。

### 反射型 XSS —— 一次性陷阱

通俗理解：攻击者给你发了一个特制的链接，你点开之后，
链接里的恶意代码在页面上执行了。就像一次性陷阱——
你不点那个链接就不会中招。

实际例子：
- 正常搜索: https://shop.com/search?q=手机
- 恶意链接: https://shop.com/search?q=<script>偷Cookie代码</script>

为什么危险：如果这行代码不是弹窗，而是把你的登录Cookie
发送到攻击者的服务器，攻击者就能用你的身份登录了。
整个过程你完全不知道！

### 存储型 XSS —— 埋在花园里的地雷

通俗理解：攻击者在网站的评论区、个人简介等地方留下恶意脚本，
这个脚本被存到数据库里。之后任何人打开这个页面都会中招——
不需要点击任何链接。

类比：就像在公园的沙地里埋了一颗地雷，之后每个路过的人都会踩到。
相比反射型的一次性陷阱，存储型是持久化威胁。

最危险的场景：
- 社交网站的个人签名被嵌入恶意代码 -> 所有看你主页的人中招
- 论坛帖子内容包含恶意代码 -> 所有浏览这个帖子的人中招
- 商品评价里藏着恶意脚本 -> 所有看评价的买家账号被盗

### DOM 型 XSS —— 不用经过服务器的秘密行动

通俗理解：前面两种 XSS 都需要服务器配合（把恶意代码返回给受害者），
但 DOM 型 XSS 完全不同——恶意代码的注入和执行都在浏览器里完成，
服务器全程不知情。

类比：就像快递员（服务器）只是把包裹（页面 HTML+JS）送给收件人，
但收件人自己拆包裹的方式有问题（JS 代码不安全地处理了 URL 参数），
导致炸弹爆炸了。

蓝队检测难点：因为流量中没有明显的攻击 payload（只是正常的 URL 参数），
传统 WAF 很难发现 DOM 型 XSS。需要通过代码审计或浏览器端的动态检测工具。

### XSS 攻击怎么检测？蓝队三板斧

**第一板斧：看请求参数里有没有脚本标签**
从日志中搜索这些关键字，它们都是 XSS 攻击的指纹：
```
- <script>     最经典的 XSS 标签
- <img         图片标签可以嵌入 onerror 等事件来执行脚本
- <svg         SVG 标签也可以嵌入脚本
- onerror=     当加载失败时执行代码
- onload=      当加载成功时执行代码
- javascript:  伪协议执行 JS
- <iframe      内嵌框架可加载恶意页面
```

实际日志分析命令：
```bash
# 在 Nginx 日志中搜索常见的 XSS 特征
grep -iE '<script|<img|<svg|onerror=|onload=|javascript:' access.log
```

**第二板斧：看输出过滤是否到位**
在代码层面，检查以下几点：
1. 用户输入的内容在输出到页面时，有没有把特殊字符转义？
   - < 转为 &lt;     > 转为 &gt;     「转为 &quot;     ' 转为 &#x27;
2. 是否设置了 HttpOnly Cookie？（这样 JS 就偷不走 Cookie）
3. 是否启用了 CSP（内容安全策略）？
   CSP 相当于告诉浏览器只允许执行来自白名单的脚本

**第三板斧：用工具辅助检测**
- BurpSuite 的 DOM Invader 可以自动检测 DOM 型 XSS
- 浏览器 F12 控制台可以看到哪些 JS 执行了、来自哪里
- 使用 XSStrike 等工具在测试环境进行自动化 XSS 检测

## CSRF 跨站请求伪造 —— 借刀杀人的艺术

### 一句话理解 CSRF

通俗解释：你登录了银行网站（Cookie 保存在浏览器里），
然后不小心点开了攻击者的恶意网页。这个恶意网页偷偷用你的身份（Cookie）
向银行发送了一个转账请求——银行看到请求里有你的 Cookie，
就以为是你在操作，转账成功了。整个过程你毫不知情。

核心问题：浏览器的自动携带 Cookie 机制——对哪个网站发请求，
就自动带上那个网站的 Cookie。这是为了方便用户（不用每次都登录），
但也成了 CSRF 攻击的温床。

### CSRF 攻击的三个必要条件（缺一不可！）
条件1：Cookie 自动携带——浏览器会自动带上目标网站的 Cookie
> 如果目标网站用了 SameSite=Strict Cookie，浏览器就不会
> 在跨站请求中携带 Cookie，攻击就失败了。

条件2：请求参数可预测——攻击者能猜出请求长什么样
> 比如转账接口是 POST /transfer，参数是 to=xxx&amount=xxx，格式固定。
> 如果每次请求需要一个随机 Token，攻击者猜不到 Token 值就无法攻击。

条件3：没有额外验证——服务器只看 Cookie 就执行操作
> 如果转账需要输入短信验证码，即使 CSRF 请求发出去了也转不走钱。

### CSRF 的防护方案（从简单到完善）
| 方案 | 原理 | 通俗解释 | 效果 |
|:---|:---|:---|:---|
| SameSite Cookie | 限制 Cookie 只在本站发送 | 给 Cookie 贴个禁止外借标签 | 五星推荐 |
| CSRF Token | 每次请求带一个随机数 | 像验证码一样，攻击者猜不到 | 五星推荐 |
| Referer 检查 | 看请求是从哪个页面来的 | 只允许从自家页面发起的请求 | 三星 |
| 验证码 | 人机验证 | 机器伪造不了人的操作 | 四星 |
| 二次确认 | 敏感操作弹窗确认 | 多一道防线 | 四星 |

## SSRF 服务端请求伪造 —— 让服务器当间谍

### 一句话理解 SSRF

通俗解释：攻击者让服务器去替他访问一些正常情况下
访问不到的东西——内网系统、云服务后台等。就像一个外人进不了公司大楼，
但他骗公司前台说帮我查一下这个房间号里有什么，前台（服务器）因为是
自己人所以能进内网，就成了攻击者的内应。

最经典的例子：很多网站有通过 URL 上传图片的功能，比如头像设置。
你输入 https://example.com/photo.jpg，服务器去下载这张图片。
但如果攻击者输入 http://192.168.1.1/admin，服务器就会去访问内网的管理后台！

### SSRF 能做什么坏事？
1. 探测内网：让服务器扫描内网有哪些机器、开了哪些端口
2. 读取敏感文件：file:///etc/passwd —— 直接读服务器上的文件
3. 攻击内网服务：内网的 Redis、MySQL 本来不对外开放，通过 SSRF 就能打进去
4. 偷云服务凭证：云服务器上有一个特殊地址 169.254.169.254，
   里面存着云账号的密钥，攻击者通过 SSRF 拿到密钥就能控制整个云账户
5. 绕过防火墙：攻击者在外网，以服务器为跳板攻击内网

### SSRF 怎么检测？蓝队三步法
第一步：看 URL 参数里有没有内网地址
```bash
# 检测请求参数中的内网 IP
grep -E '10\.[0-9]+\.[0-9]+\.[0-9]+' access.log
grep -E '172\.(1[6-9]|2[0-9]|3[01])\.' access.log
grep -E '192\.168\.' access.log
grep '127\.0\.0\.1' access.log
```

第二步：看有没有访问云元数据地址
```bash
# 云服务的元数据地址是最重要的 SSRF 目标
grep '169\.254\.169\.254' access.log
```

第三步：看有没有用奇怪的协议
```bash
# 正常情况下 Web 请求只用 http/https，出现以下协议就是异常
grep -E 'file://|gopher://|dict://|ftp://' access.log
```

## 面试高频问答

Q: XSS、CSRF、SSRF 三者的本质区别是什么？

用最简单的语言区分：
- XSS：在别人的浏览器里执行代码（攻击用户的浏览器）
- CSRF：用别人的身份发请求（冒充用户操作）
- SSRF：让服务器去访问不该访问的东西（攻击服务器/内网）

XSS 和 CSRF 经常被混淆，记住：
XSS 是执行脚本，CSRF 是伪造请求。
一个利用了代码执行，一个利用了身份冒用。

Q: 如果网站已经做了 XSS 防护（输出编码），还需要做 CSRF 防护吗？

需要！两者是完全独立的防护维度。即使 XSS 防住了（恶意脚本不能执行），
CSRF 攻击依然可以奏效——因为 CSRF 根本不依赖脚本执行，
它靠的是浏览器的自动 Cookie 机制。

相反，如果 XSS 没防住，CSRF Token 防护也可能被绕过
（XSS 可以读取页面上的 Token）。所以两种防护都要做，缺一不可。

Q: SSRF 为什么被称为最被低估的漏洞？

因为很多开发者不理解 SSRF 的危害。表面上服务器去访问一个 URL
没什么大不了的，但实际上：
1. 服务器通常在内网，有权限访问外网访问不到的资源
2. 云环境的凭证就在 SSRF 可触及的地方（169.254.169.254）
3. 一个 SSRF 可能直接导致整个内网沦陷
SSRF 的 CVSS 评分常常能达到 9.0+。

## 案例分析：一个真实的 XSS+CSRF 组合攻击

场景：某电商网站同时存在两个漏洞——
搜索页有反射型 XSS，修改绑定手机号的接口没有 CSRF 防护。

攻击者做了什么（分步讲解）：

第 1 步：攻击者发现搜索页会把搜索关键词原样输出到页面
（没有做输出编码），这就是反射型 XSS。

第 2 步：攻击者构造了一个特殊的搜索链接，搜索词不是普通文字，
而是一段 JavaScript 代码：
```
<script>
// 读取页面上的 CSRF Token（如果有的话）
var token = document.getElementById('csrf_token').value;
// 利用 XSS 的执行权限，向修改手机号的接口发送请求
fetch('/api/change_phone', {
  method: 'POST',
  body: 'phone=13800000000&token=' + token
});
</script>
```

第 3 步：攻击者把这个链接通过钓鱼邮件发给受害者——
邮件主题写着【您的订单异常，请点击确认】。
受害者已经登录了电商网站，点击链接后：
- 浏览器向电商网站发送搜索请求（自动带上了 Cookie）
- 服务器返回搜索结果页面，其中包含了攻击者的恶意脚本
- 恶意脚本在受害者浏览器中执行
- 脚本读取页面上的 CSRF Token，然后以受害者的身份发送修改手机号的请求
- 手机号被改为攻击者的号码

第 4 步：攻击者用新手机号进行忘记密码，收到验证码，
重置密码，完全控制了账户。

蓝队教训：
1. XSS 和 CSRF 经常被组合利用——修复一个漏洞不等于安全
2. 敏感操作（改手机/改密码）一定要二次验证（短信验证码/邮箱验证）
3. CSP（内容安全策略）可以在浏览器层面阻止 XSS 脚本的执行
4. SameSite Cookie + CSRF Token 双重防护才是正解

## 蓝队实战心得：XSS/CSRF/SSRF 快速判断口诀

在实际工作中，当你面对一个告警时需要快速判断这是什么类型的攻击。以下口诀可以帮助你：

### 三秒判断法

```
告警中出现了 <script> 或 onerror= 或 <img src=x onerror=...>
  -> 这是 XSS 攻击！

告警中的请求缺少 Referer 或 Referer 来自外部域名，且是敏感操作（改密码/转账）
  -> 怀疑 CSRF！

告警中的 URL 参数包含内网 IP（10.x.x.x / 192.168.x.x）或 file:/// 协议
  -> 这是 SSRF 攻击！
```

### 蓝队告警研判优先级

| 告警类型 | 紧急度 | 原因 | 处置动作 |
|:---|:---|:---|:---|
| SSRF + 访问 169.254.169.254 | 最高 | 云凭证可能泄露 | 立即阻断 IP + 检查云账户操作日志 |
| 存储型 XSS | 极高 | 所有用户受影响 | 先下线功能 + 清理恶意数据 + 修复代码 |
| CSRF（涉及资金） | 极高 | 可能造成资金损失 | 临时关闭接口 + 加验证码 + 追查损失 |
| 反射型 XSS | 高 | 需要用户点击 | 加 CSP 头 + 输出编码修复 |
| CSRF（非敏感操作） | 中 | 影响有限 | 加 CSRF Token |
| DOM 型 XSS | 中 | 利用条件苛刻 | 前端代码修复 |

### 常见误区提醒

1. 误区：「用了 HTTPS 就不会被 XSS」—— XSS 和 HTTP/HTTPS 无关，即使 HTTPS 也可能被 XSS
2. 误区：「加了 CSRF Token 就绝对安全」—— 如果同时存在 XSS 漏洞，Token 可以被读取
3. 误区：「SSRF 只是信息泄露」—— SSRF 可以执行命令（gopher 协议攻击 Redis 等）
4. 误区：「内网不需要防护 CSRF」—— 内网同样存在 CSRF 风险
5. 误区：「WAF 能拦截所有 XSS」—— DOM 型 XSS 不经过服务器，WAF 无法拦截

### 防护优先级建议

如果你负责一个 Web 应用的安全加固，建议按以下顺序推进：

第一步（当天完成）：
- 全局添加 CSP 头（default-src 'self'）
- 关键 Cookie 设置 HttpOnly + SameSite=Strict
- 文件上传接口加固（白名单 + 重命名 + 隔离存储）

第二步（本周完成）：
- 所有输出点增加上下文相关的编码（HTML 编码 / JS 编码 / URL 编码）
- 所有敏感操作接口加入 CSRF Token
- SSRF 入口点加入 URL 白名单 + 协议限制

第三步（本月完成）：
- 部署 WAF + 自定义规则
- 建立 XSS/CSRF/SSRF 告警监控 Dashboard
