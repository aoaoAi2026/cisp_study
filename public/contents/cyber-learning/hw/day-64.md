## 命令注入与代码注入高级检测

命令注入和代码注入看起来相似，但有本质区别。

### 命令注入 vs 代码注入

命令注入：攻击者输入被拼接到系统命令中执行（shell命令层面）
代码注入：攻击者输入被作为代码执行（应用代码层面，如PHP eval()）
检测区别：命令注入关注系统命令关键字(whoami/id/ls/cat)，
代码注入关注语言内置函数的调用(eval/assert/system/exec)。

### 命令注入高级绕过

1. 命令分隔符替代：\n、%0a、$(cmd)、`cmd`
2. 空格绕过：${IFS}、<、<>、{cmd,args}
3. 关键字绕过：ca$1t、c\at、/bin/c?t
4. 编码绕过：echo base64encoded | base64 -d | sh
5. 通配符绕过：/???/c?t /???/p?sswd

### 蓝队检测思路

不只检测已知的恶意关键字，还要检测'命令执行的行为模式'：
- 请求后出现了新的网络连接（反弹Shell）
- 请求后文件系统发生了变化（写入/修改文件）
- 请求的响应中包含系统信息（/etc/passwd内容、whoami输出）
## 检测命令注入的WAF规则思路

```
# 检测Shell连接符
SecRule ARGS "[;|&`$()]"

# 检测系统命令
SecRule ARGS "(?:id|whoami|uname|ifconfig|cat|ls|pwd|wget|curl)"

# 检测编码后的命令执行
SecRule ARGS "base64.*-d|\\x[0-9a-f]{2}"
```

## 高级蓝队场景演练

### 场景：发现内网横向移动

凌晨3点，IDS告警：内网服务器192.168.1.50正在扫描同网段的445(SMB)和3389(RDP)端口。

你的处置：
1. **立即确认(P0)**：内网告警 > 外网告警，边界已被突破
2. **检查日志**：回溯192.168.1.50的入侵路径，发现Weblogic反序列化漏洞被利用
3. **隔离主机**：断网不关机，保留证据
4. **排查扩散**：检查同网段其他服务器是否被扫描
5. **启动应急**：通知组长，启动完整应急响应流程

### 场景：SQL注入绕过WAF

WAF日志：某IP发送了包含URL编码的SQL注入Payload绕过WAF。
你的分析：
1. 原始Payload：%55%4E%49%4F%4E%20%53%45%4C%45%43%54 (UNION SELECT的URL编码)
2. 判断：攻击者在尝试绕过WAF的字符串匹配
3. 应对：在WAF中加入解码后再匹配的规则，或升级为语义检测

### 场景：文件上传绕过

WAF发现文件上传，filename为 shell.php.jpg，Content-Type为 image/jpeg，但文件头为 `<?php`。
分析：双扩展名 + MIME伪造 + 真实文件内容是PHP。三层检测全部通过才能发现攻击。
教训：文件上传安全必须多层检测，单靠一层必然有疏漏。

## 蓝队面试深度题库（10题）

**Q1: WAF部署在什么位置？有哪些部署模式？**
A: 反向代理模式(串联)、透明代理模式(旁路/桥接)、Agent模式。
反向代理最常用：流量先到WAF -> WAF检查 -> 转发到后端。优点是能阻断攻击，缺点是增加延迟。

**Q2: 什么是纵深防御？举例说明。**
A: 纵深防御=多层防护。Internet -> 边界防火墙 -> WAF -> 应用服务器(主机安全) -> 数据库(审计)。
每一层都是独立防线，攻击者突破一层还有下一层。经典案例：边界防火墙被绕过->WAF拦截了SQL注入。

**Q3: 你如何判断一条告警是误报还是真攻击？**
A: 五步法：看源IP(是否已知恶意)、看Payload(是否典型攻击特征)、看响应(是否攻击成功)、
看上下文(同一IP之前的行为)、看资产(受影响的资产重要程度)。五者结合判断。

**Q4: 护网中攻击者常用的5种初始入侵手法？**
A: 1.Web漏洞(SQL注入/XSS/文件上传等) 2.弱口令(SSH/RDP/数据库默认密码)
3.中间件漏洞(Weblogic/Tomcat/Shiro) 4.钓鱼邮件 5.第三方组件漏洞(Fastjson/Log4j)

**Q5: 什么是SOAR？为什么蓝队需要它？**
A: SOAR=安全编排自动化与响应。将重复性工作(告警初筛、IP封禁、日报生成)自动化，
让分析师专注于需要人类判断的高价值告警。是解决告警疲劳的关键工具。

**Q6: 护网中发现一台服务器CPU 100%，你怎么排查？**
A: 1. top/htop看哪个进程占用CPU 2. netstat看是否有异常网络连接
3. crontab检查是否有异常计划任务 4. history查看最近执行的命令 5. /tmp目录检查是否有可疑文件
思路：先确认是否被入侵 -> 然后找入侵入口 -> 处置 -> 恢复。

**Q7: 什么是TTPs？和IOC有什么区别？**
A: TTPs=Tactics, Techniques, Procedures(战术、技术、过程)，
描述攻击者的行为模式("他们怎么做的")。IOC是攻击留下的具体痕迹("他们留下了什么")。
IOC像指纹(TTL=64)，TTPs像作案手法(喜欢从窗户进来)。TTPs更难改变，更有长期防御价值。

**Q8: 如何检测加密流量中的恶意行为？**
A: 虽然看不到内容，但可以分析：1.元数据(连接时间、包大小、频率)
2.JA3/JA3S指纹(TLS握手的客户端特征) 3.证书分析(自签名/过期/异常) 4.DNS前置查询 5.流量熵值

**Q9: 你是如何持续学习安全知识的？**
A: 关注安全社区(FreeBuf、安全客、先知社区)、阅读CVE公告、在VulHub上复现新漏洞、
分析公开的PCAP样本、参加CTF练习、关注Twitter/X上的安全研究者。

**Q10: 如果遇到没见过的新型攻击，你会怎么做？**
A: 1.保持冷静 2.收集证据(日志/流量/内存) 3.搜索公开信息(CVE/安全社区)
4.寻求团队帮助 5.隔离受影响系统 6.分析攻击特征，即使不知道具体漏洞也能通过特征检测
7.事后学习该攻击的原理，防止再次发生。

## 蓝队日常巡检脚本模板

### 每日安全巡检checklist

```bash
# 1. 检查最近1小时内登录失败记录
grep "Failed password" /var/log/auth.log | wc -l

# 2. 检查最近1小时内sudo使用记录
grep "sudo" /var/log/auth.log | tail -20

# 3. 检查异常的大文件
find / -type f -size +100M -mtime -1 2>/dev/null

# 4. 检查异常的网络连接
netstat -tunap | grep ESTABLISHED | grep -v "127.0.0.1"

# 5. 检查最近修改的文件
find /etc /var/www -type f -mtime -1 2>/dev/null

# 6. 检查crontab是否有新增
crontab -l | grep -v "^#"

# 7. 检查正在运行的进程
ps aux | grep -E "(nc|bash.*-i|python.*socket|perl.*socket)"

# 8. 检查最近1小时的Web访问错误
tail -1000 /var/log/nginx/access.log | grep -c " 4[0-9][0-9] "
tail -1000 /var/log/nginx/access.log | grep -c " 5[0-9][0-9] "
```

### SOC关键指标(KPI)

1. MTTD(Mean Time to Detect)：平均检测时间，目标<15分钟
2. MTTR(Mean Time to Respond)：平均响应时间，目标<30分钟
3. 误报率：告警中误报的比例，目标<30%
4. 漏报率：漏过的真实攻击比例，目标<1%
5. 告警处理量：每人每天处理的告警数量

## 高级检测规则编写实战

### 手写一条WAF规则（ModSecurity格式）

检测SQL注入（包含编码绕过检测）：
```
SecRule ARGS "@rx (?:UNION\s+SELECT|SELECT\s+.*\s+FROM|\d+\s*=\s*\d+|'\s*OR\s+'\d+'\s*=\s*'\d+|SLEEP\(|BENCHMARK\(|INFORMATION_SCHEMA)" \
  "id:100001,msg:'SQL Injection Detected',phase:2,deny,status:403"
```

检测XSS攻击：
```
SecRule ARGS "@rx (<script[^>]*>|on(?:load|error|click|mouse)\s*=|javascript\s*:|<iframe|<object|<embed|<link)" \
  "id:100002,msg:'XSS Attack Detected',phase:2,deny,status:403"
```

### 手写一条Snort/Suricata IDS规则

检测Webshell上传后的访问行为：
```
alert tcp $EXTERNAL_NET any -> $HOME_NET $HTTP_PORTS
  (msg:"Possible Webshell Access";
   flow:to_server,established;
   content:"GET"; http_method;
   pcre:"/\/(shell|cmd|upload|assert)\d*\.(php|jsp|asp)/iU";
   sid:1000001; rev:1;)
```

## 安全测试与验证方法

### 如何验证你的检测规则是否有效

1. 部署规则（先在测试环境）
2. 在测试环境执行对应的攻击（用DVWA/VulHub）
3. 观察规则是否被触发
4. 执行正常的业务操作，确认没有误报
5. 用编码/绕过后重新测试，确认规则能检测到变形攻击
6. 调优规则直到：检出率>95% + 误报率<5%

## 💡 知识卡片：企业安全经验分享——一个真实漏洞的故事

> 某次护网，一个不起眼的文件上传接口被找到。上传白名单配置了：jpg、png、gif、bmp。
> 攻击者上传了一个 .pht 文件（PHP的备用后缀），成功拿到Webshell。
> 
> **根因**：运维在配置Nginx时，发现某些PHP版本也解析 .pht，但上传白名单里没写。

**从这个案例中学到的：**
1. 文件上传防御不能只靠前端白名单，后端要校验文件内容(MIME/魔术字节)
2. Nginx/Apache配置中的解析规则是安全配置的一部分，不是纯运维的事
3. 上传目录必须禁止脚本执行权限
4. 即使上传成功，文件也不能放在Web可访问的路径

**防御三层：**
- 第一层（前端）：文件类型+大小校验 → 拦截低水平攻击
- 第二层（后端）：魔术字节校验+重命名+内容安全检查 → 核心防线
- 第三层（架构）：上传目录禁止脚本执行+文件访问做权限控制 → 纵深防御

> 🔑 **安全不是做完就完了，要持续验证**。今天安全的配置，明天可能就成了漏洞。
