## 模拟值守第四天：值班交接与协同作战

护网不是一个人的战斗。今天的重点是：团队协作和信息共享。

### 值班交接规范

交接做不好，等于前面8小时白干了。护网中很多事故就是因为
"上一班发现了异常但没交接清楚，下一班没有继续跟进而导致的。"

**标准交接流程**：
1. 交接时间：每班提前15分钟开始交接
2. 交接内容：
   - 本班重点告警和处理情况
   - 当前仍在监控中的可疑IP/行为
   - 已封禁的IP列表和封禁原因
   - 待处理的事项（如某服务器待打补丁）
   - 设备/工具的运行状态
3. 交接方式：口头交接 + 书面日报
4. 交接确认：接班人员确认理解并签字

### 护网团队的协同机制

大型护网中，防守团队分为几个协同工作组：

| 组别 | 职责 | 协同对象 |
|:-|:-|:-|
| 监控组 | 7x24监控告警，第一时间发现攻击 | -> 研判组 |
| 研判组 | 深度分析告警，判断真伪和危害 | -> 处置组/监控组 |
| 处置组 | 执行封IP/隔离/恢复等处置操作 | -> 研判组/运维 |
| 溯源组 | 攻击链路还原，IOC提取，情报共享 | -> 所有组 |
| 情报组 | 收集外部威胁情报，下发IOC | -> 监控组 |
| 运维组 | 配合安全团队做服务器加固和修复 | -> 处置组 |

### 信息共享工具

护网期间的信息共享常用工具：
- 企业微信/钉钉/飞书群：实时沟通
- 共享文档（语雀/飞书文档/腾讯文档）：实时更新告警处理状态
- TAPD/Jira：正式的问题跟踪
- 电话/VOIP：紧急情况直接电话（不依赖文字沟通）

> 护网的黄金法则：看到威胁先喊出来，不要自己默默判断。
> 多一个人看到就少一分漏过的可能。

### 模拟协同演练

今天的模拟练习：找1-2个学习伙伴（或自己在群里模拟），演练以下场景：

**演练场景**：监控组发现X服务器出现Webshell上传告警

角色A（监控组）-> 角色B（研判组）：
"发现P0告警：10.0.1.50疑似被上传Webshell，已发你详情。"

## 蓝队面试自测

**Q1: 描述一次完整的Web攻击检测流程**
A: 收到告警 -> ELK日志确认 -> 分析Payload -> 判断类型 -> 确认是否成功 -> 处置 -> 写报告。

**Q2: IDS和IPS的区别？**
A: IDS只检测不阻断（摄像头），IPS检测并阻断（门禁）。

**Q3: ATT&CK框架对蓝队的价值？**
A: 告诉我们攻击者"会做什么"和"怎么做"，蓝队据此在对应战术阶段设置检测点。

**Q4: 告警太多怎么处理？**
A: 分级处理——P0立即、P1今日内、P2周汇总，然后优化规则减少误报。

**Q5: WAF和IPS的区别？**
A: WAF专用于Web应用层（HTTP/HTTPS），IPS用于网络层（TCP/IP）。WAF理解SQL注入和XSS，IPS理解端口扫描和DDoS。

处置步骤：
1. **立即确认**（30秒）：查看ELK中192.168.1.50的最近1小时日志

4. **深入排查**（30分钟）：检查入侵路径（Weblogic漏洞？弱口令SSH？）

### 蓝队的日常：从告警到处置的完整心智模型

每天早上打开监控平台，你的流程应该是：
1. 扫一眼大盘——有无P0/P1未处理告警
2. 浏览昨晚的告警汇总——凌晨的攻击最需要关注
3. 处理积压告警——从高危到低危逐条研判

5. 持续监控——新告警随时可能弹出

这个流程看似简单，但做到位需要长期训练。

### 5秒判断法实战演练

以下每条告警请在5秒内做出判断：

1. WAF: request contains ' UNION SELECT ' -> [  ]
2. IDS: SYN packets to 100+ ports -> [  ]
3. Server log: /etc/passwd in URL -> [  ]
4. Firewall: outbound to 45.x.x.x:4444 -> [  ]

答案：1-SQL注入攻击、2-端口扫描、3-文件包含攻击、4-疑似C2通信、5-暴力破解

### 攻击者心理画像

攻击者通常分为三类：
- **脚本小子**：用现成工具扫描，技术含量低但数量多，占告警的80%

- **红队/渗透测试**：合法的"攻击"，需要白名单排除

蓝队的精力分配：70%对付脚本小子，25%追踪可疑高级威胁，5%处理误报。

## 蓝队面试题库

**Q1: TCP三次握手是什么？为什么理解它对蓝队很重要？**

A: SYN -> SYN-ACK -> ACK。理解三次握手就能理解端口扫描（大量SYN到不同端口）、
DDoS（SYN Flood耗尽半连接队列）、C2通信（异常的TCP连接模式）。

**Q2: 你怎么理解纵深防御？**

A: 纵深防御 = 多层防护。边界防火墙是第一层、WAF是第二层、主机安全是第三层、
数据库审计是第四层。攻击者突破一层还有下一层——这就是纵深防御的价值。

**Q3: 护网期间发现一台服务器被黑了，你的第一步是什么？**

A: 隔离（断网不关机保留证据）> 通知组长 > 排查入侵路径 > 清除后门 >
修复漏洞 > 恢复上线 > 复盘报告。注意：千万不要关机！内存证据会丢失。

**Q4: 怎样区分自动化扫描和定向攻击？**

A: 自动化扫描特征：Payload是字典式（SQLMap的常见Payload）、
扫描速度快（<1秒一个请求）、同时扫描多个不相关目标。
定向攻击：Payload有针对性（针对特定应用）、
攻击节奏有策略（先探测再利用）、
关注特定高价值目标。

**Q5: 日志分析中最常用的Linux命令？**

A: grep（搜索关键字）、awk（字段提取）、sed（文本替换）、
sort + uniq -c（统计）、tail -f（实时监控）。
组合示例：cat access.log | grep "UNION SELECT" | awk '{print $1}' | sort | uniq -c | sort -nr

### 等保合规的常见落地难点

1. **日志保存6个月**：日志量巨大，存储成本高。建议分级存储：热数据(最近7天SSD) → 温数据(30天普通硬盘) → 冷数据(6个月对象存储)

## 蓝队实战：经典案例分析

### 案例：一次成功的护网防守

某金融企业在护网期间成功抵御了所有红队攻击，经验总结：

**防守策略**：
1. 提前1个月做资产梳理（发现了12个遗忘的测试服务器，及时下线或加固）
2. 对所有对外服务做了漏洞扫描和修复

- 资产梳理是护网的第一道防线

- 双人值守避免了单人疲劳导致的漏报
- 情报共享（与其他防守单位互通攻击IP）事半功倍

**教训**（尽管防守成功，仍有改进空间）：
- 每天日报花了太多时间（2小时），应该自动化日报生成
- 第一天的值守效率低（手忙脚乱），应该提前做模拟演练
- 有些告警处置慢了（因为需要层层上报），应该赋予一线分析师更多权限

## 工具进阶技巧

### Shell命令效率技巧

统计攻击IP排行榜：
```
cat access.log | grep -E "(UNION SELECT|<script>|\.\./etc/)" | awk '{print $1}' | sort | uniq -c | sort -nr | head -20
```

实时监控新出现的攻击：
```
tail -f access.log | grep --line-buffered -E "(SQL injection|XSS|command injection)"
```

批量封禁IP：
```
for ip in $(cat blocklist.txt); do
  iptables -A INPUT -s $ip -j DROP
done
```

### Python安全脚本速查

快速端口扫描：
```python
import socket
for port in range(1, 1025):
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.settimeout(0.1)
    if s.connect_ex(('127.0.0.1', port)) == 0:
        print(f"Port {port}: OPEN")
    s.close()
```

检查IP是否为恶意：
```python
import requests
# 使用AbuseIPDB等免费威胁情报API
resp = requests.get(f"https://api.abuseipdb.com/api/v2/check?ipAddress={ip}", 
    headers={"Key": "YOUR_API_KEY", "Accept": "application/json"})
if resp.json()["data"]["abuseConfidenceScore"] > 80:
    print(f"{ip} is malicious!")
```

3. 看到异常先喊出来，不要独自判断
