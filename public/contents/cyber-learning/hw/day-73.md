## 📖 核心知识讲解



### 一、内网渗透的典型攻击路径


内网渗透一般不是"一步到位"的，而是分多个阶段：





**详细步骤：**

1. **信息收集**：端口扫描、服务版本、Web目录扫描、whois/子域名

2. **初始打点**：利用Web漏洞拿到Webshell/利用弱口令登录

3. **权限维持**：上传后门、添加计划任务、创建隐藏账号

4. **内网探测**：arp扫描、端口扫描内网存活主机和服务

5. **横向移动**：利用Hash传递(PTH)、票据传递(PTT)、漏洞利用

6. **权限提升**：本地提权→获取SYSTEM/root权限

7. **目标达成**：拿下域控、窃取数据、破坏系统




### 二、攻击步骤 → 检测点映射


| 攻击阶段 | 攻击手法 | 数据源 | 检测方法 |
|----|----|---|----|
| 信息收集 | 端口扫描(Nmap/Masscan) | 网络流量/防火墙日志 | 短时间大量SYN包到不同端口 |
| 信息收集 | 目录扫描(dirb/gobuster) | Web访问日志 | 短时间内大量404响应 |
| 初始打点 | SQL注入/文件上传/命令注入 | WAF日志/Web访问日志 | SQL关键字、异常文件上传、命令执行痕迹 |
| 初始打点 | 弱口令爆破 | 登录日志 | 同一账号多次失败→成功 |
| 权限维持 | 创建计划任务/服务/启动项 | Windows事件日志/进程监控 | 新增计划任务、异常服务创建 |
| 权限维持 | Webshell上传 | Web访问日志/文件监控 | .php/.jsp文件出现在上传目录 |
| 内网探测 | 内网端口扫描 | 网络流量 | 单主机对内网大量IP的SYN扫描 |
| 内网探测 | net view/net group | Windows事件日志(4688) | 可疑进程执行net相关命令 |
| 横向移动 | PTH(Pass The Hash) | Windows事件日志(4624/4625) | 登录类型3(NTLM)无Kerberos预认证 |
| 横向移动 | PsExec/WMI横向 | Windows事件日志(4688/7045) | psexec.exe执行,新服务创建 |
| 权限提升 | 内核漏洞提权/令牌窃取 | Windows安全日志(4672) | 非SYSTEM进程获取高权限 |
| 目标达成 | DCSync获取域控Hash | Windows安全日志(4662) | 非域控机器发起目录复制请求 |




### 三、ATT&CK框架简介


MITRE ATT&CK是一个"攻击行为知识库"，它把攻击者的行为分解成一个个"战术→技术"。

**战术(Tactic) = 攻击者想达到什么目的**（如：初始访问、横向移动、数据窃取）

**技术(Technique) = 攻击者用什么方法达到目的**（如：T1078有效账号、T1059命令执行）

## 🚀 蓝队进阶之路：从"能检测"到"能防御"

蓝队能力成长一般经历三个阶段：

**第一阶段：能识别** — 看到告警知道大概是什么攻击
**第二阶段：能分析** — 能判断攻击的严重程度、攻击者的意图、攻击到了哪一步
**第三阶段：能防御** — 不仅知道怎么检测，还知道怎么从根本上防御这类攻击

问问自己：对于今天学的这类攻击/技术，你目前在哪个阶段？

目标很清晰：把这个主题学透到能给别人讲清楚的程度。

### 如何加速成长

## 💎 蓝队实战精华：从日志中发现攻击者

今天的主题在实际工作中可能不会每天遇到，但一旦遇到就是大事。
作为蓝队，你要培养一种"直觉"——当你看着日志的时候，哪些是正常的、哪些是异常的。

### 培养"安全直觉"的方法
1. **先看100个小时的正常日志**。是的，就是纯看。知道"正常"长什么样，才能识别"异常"。
2. **搭建实验环境复现攻击**。看到了攻击的真实面貌，下次在日志中遇到就不会错过。
3. **关注"不和谐"**。比如：平时这个时间没人登录，今天凌晨3点有登录→可疑。
4. **关注"新东西"**。新进程、新服务、新计划任务、新网络连接→第一反应应该是怀疑。
5. **关注"量变"**。平时的日志量是100条/小时，突然变成1000条/小时→出事了。

### 分析未知攻击的思路
当你遇到一条看不懂的告警时：
1. 先确认基本事实：时间、来源IP、目标IP、涉及进程/服务
2. 搜索关键信息：把告警中的关键字放到Google/安全社区搜索
3. 找同类对比：搜索今天的主题+case study
4. 搭建复现：在隔离环境中复现看到的异常行为
5. 总结提炼：把这次的经验变成一条新的检测规则

> 🌟 记住：蓝队不需要比所有攻击者都聪明，你只需要比"昨天的自己"更聪明。

## 🔍 深入剖析：为什么安全基线总是不达标？

做了这么多年的安全基线检查，你会发现一个"奇怪"的现象：
同样的问题反复出现——空口令、默认配置、多余服务、过期补丁。

**根因分析：**
1. **运维和安全脱节**：运维装系统→安全去查→查出问题→运维说"改不了，会影响业务"→不了了之
2. **没有自动化**：每次检查靠手工，查完就忘了，下次继续同样的坑
3. **基线"一刀切"**：所有系统用同一套基线，导致很多项不适用，合规率"虚假"偏低
4. **新增资产无管控**：新上的系统没人做安全配置，自由生长

**解决思路：**
- 安全基线应该是上线流程的一部分（安全左移），不是上线后来挑刺的
- 用配置管理工具（Ansible/Salt/Puppet/GPO）自动化基线部署
- 分级管理：核心系统严格基线、一般系统标准基线、测试环境基本基线
- 定期自动扫描+不合规告警，形成"扫描→整改→验证→固化"的闭环

## 🎯 实战经验分享：一次主机安全加固的正确姿势

场景：你接到任务，要给50台Linux服务器做安全加固，怎么做？

**错误做法：** 一台一台登上去手工改配置。（做完最后一台，第一台的配置可能已经漂移了）

**正确做法：**
1. 先在1台模板机上做好所有加固
2. 用ansible/salt把配置写成playbook
3. 在3台测试机上跑一遍，验证业务不受影响
4. 分批滚动推送到50台（每批5台，观察无异常再继续）
5. 加固完成后，立刻用基线扫描工具验证
6. 每周自动重新扫描，防止配置漂移
7. 新增机器默认走加固playbook

**关键原则：**
- 可重复 > 一次性
- 自动化 > 手工
- 验证 > 信任
