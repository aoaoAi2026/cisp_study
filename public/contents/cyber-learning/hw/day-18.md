## 📖 核心知识讲解

### 一、告警研判四步法

收到一条告警后，按以下4步逐级进行：

```
第一步：核查（Verify）   → 这条告警是真的吗？
第二步：研判（Analyze）   → 攻击成功了吗？危害多大？
第三步：处置（Respond）   → 该怎么办？
第四步：闭环（Close）     → 记录什么？后续跟什么？
```

### 二、第一步：核查 —— 判断告警真实性

**核查清单（逐条过）：**

| 核查项 | 确认方法 | 判断标准 |
|:---|:---|:---|
| 源IP情报 | 查VirusTotal/微步在线 | 红色→真实攻击嫌疑大；绿色→可能误报 |
| 攻击时间 | 看告警时间戳 | 凌晨/非工作日→更可疑 |
| 告警触发频率 | 同一告警出现了多少次 | 大量重复→自动化扫描；偶尔一次→可能误报 |
| 关联证据 | 查对应的Web日志/系统日志 | 日志里真有攻击载荷→真实；日志里没异常→可能误报 |
| 目标价值 | 被攻击的是什么系统 | 核心数据库→高危；公共测试页→低危 |

**误报常见原因：**
- 业务正常行为触发了规则（如CMS后台编辑器使用`<script>`标签）
- 安全测试团队的扫描
- 爬虫/搜索引擎的请求被误判为扫描
- IDS规则写得太宽泛

### 三、第二步：研判 —— 分析攻击结果

确认是真实攻击后，最关键的问题是：**攻击成功了吗？**

**判断攻击是否成功的核心方法：**

#### 方法1：看HTTP状态码
```
攻击请求 → 200 → ⚠️ 可能成功！（需要进一步确认）
攻击请求 → 403/401/404 → 🟢 大概率没成功
攻击请求 → 500 → ⚠️ 可能触发漏洞导致服务器崩溃
```

#### 方法2：看响应内容
```
Wireshark Follow TCP Stream：
  请求：GET /product.php?id=1' OR '1'='1
  响应：包含了数据库中的用户名列表 → 🔴 注入成功！
  响应：报错"SQL syntax error" → 🟡 注入尝试中（有一定信息泄露）
  响应：返回正常产品页面 → 🟢 注入未成功
```

#### 方法3：关联系统日志
```
Web攻击告警 + secure日志同一时间出现异常登录 = 🔴 攻击成功+权限获取
Web攻击告警 + 无其他异常 = 🟡 攻击尝试中
```

### 四、第三步：处置 —— 根据不同情况采取行动

| 情况 | 处置措施 | 紧急程度 |
|:---|:---|:---|
| **攻击成功，系统被控** | ①立即隔离被控服务器 ②封禁攻击IP ③启动应急响应 ④保留证据 | 🔴 立即 |
| **攻击尝试中，尚未成功** | ①封禁攻击IP ②加固目标 ③持续监控 | 🟡 24h内 |
| **攻击失败（403/404）** | ①记录攻击IP ②确认防护有效 ③加入监控列表 | 🟢 48h内 |
| **确认误报** | ①标注误报原因 ②考虑优化告警规则 | 🟢 定期处理 |

**典型处置动作：**
```bash
# 封禁攻击IP（iptables）
sudo iptables -A INPUT -s 攻击IP -j DROP

# 封禁攻击IP（nginx层面）
# 在nginx配置中添加 deny 攻击IP;

# 拉入黑名单并在SOC平台标记
# 编写工单记录
```

### 五、第四步：闭环 —— 记录与跟踪

**一条合格的研判记录应该包含：**

```markdown
## 告警研判工单

**告警编号**：ALERT-20260610-001
**告警时间**：2026-06-10 03:15:22
**告警来源**：IDS设备
**告警类型**：SQL注入攻击
**来源IP**：45.33.32.156
**目标资产**：Web服务器 (192.168.1.100:80)

**研判过程**：
1. 情报核验：微步判定为恶意IP(红色)，标签为"Web攻击"
2. 日志分析：查看nginx访问日志，该IP在凌晨3点发起了多次请求
3. 攻击特征：URL中包含"UNION SELECT"，确认为SQL注入尝试
4. 攻击结果：请求返回状态码200，但后续未发现数据泄露迹象

**研判结论**：真实攻击，攻击尝试中。SQL注入payload被服务器正常响应，但尚未发现数据泄露。

**处置措施**：
1. [已完成] 在iptables中封禁IP 45.33.32.156
2. [进行中] 排查目标系统是否存在SQL注入漏洞
3. [待跟进] 通知开发团队修复漏洞

**处置人**：张三
**处置时间**：2026-06-10 03:45:00
**工单状态**：已闭环，待漏洞修复
```

### 六、告警研判SOP速查手册

**快速决策流程图：**

```
收到告警
  │
  ├─→ 查源IP情报
  │     ├─ 恶意 → 真实性+1分
  │     └─ 安全 → 可能是误报
  │
  ├─→ 看告警频率
  │     ├─ 短时间大量 → 自动化攻击+1分
  │     └─ 就一条 → 继续观察
  │
  ├─→ 查关联日志
  │     ├─ 日志对齐且攻击特征明显 → 确认攻击
  │     └─ 日志无异常 → 可能误报
  │
  └─→ 判断攻击结果
        ├─ 有成功迹象(200+数据外传+后续异常) → P0/P1
        ├─ 攻击尝试中(200但无异常) → P1/P2
        ├─ 攻击失败(403/404) → P3
        └─ 确认误报 → P4
```

## 📝 今日小结

告警研判是蓝队分析师最核心的技能。今天你学会了四步法——核查、研判、处置、闭环。记住，研判的关键不在于"看到了告警"，而在于判断"攻击成功了吗"和"应该怎么处理"。

**记住今天的核心**：
- 四步法：核查→研判→处置→闭环
- 关键判断：攻击成功的标志（200+后续异常≠只是告警）
- 误报不可怕，只要标注清楚原因
- 每条告警都要有记录（好记性不如烂工单）
