## 📖 核心知识讲解

### 一、护网行动到底是什么？——一场国家级的「网络消防演习」

先来一个比喻，保证你忘不了：

> 🧯 每年消防队都会搞大型消防演习。假设市中心某栋大楼「失火」，110、119、120 全出动，检验各部门的应急响应能力。结束后复盘：哪个环节慢了？谁该到没到？预案哪里有问题？下一年改进。

**护网行动就是网络安全界的消防演习。** 只不过：
- 「着火」变成了「黑客攻破系统」
- 「大楼」变成了政府网站、银行系统、电力调度系统
- 「消防员」变成了网络安全工程师（也就是你）

---

**护网行动的核心事实清单：**

| 维度 | 真实情况 |
|:---|:---|
| **谁组织的？** | 国家公安部、国家网信办等主管部门牵头组织 |
| **谁参加？** | 政府部门、央企、金融机构、大型互联网公司、关键基础设施运营单位（水电气交通等） |
| **多长时间？** | 通常 2-4 周，7×24 小时不间断 |
| **怎么打？** | 红队使用真实攻击手法（不事先通知），蓝队进行真实防守 |
| **输了会怎样？** | 全行业通报、扣分、影响单位的网络安全考核排名 |
| **赢了有什么？** | 荣誉、排名、证明自己的安全体系经得起考验 |

**护网行动的根本目的——不是为了争输赢，而是为了：**

1. **暴露问题**：你平时觉得「应该没问题」的防火墙策略、口令强度、日志覆盖度，在真实攻击下到底能不能扛住？
2. **锻炼队伍**：安全团队有没有实战经验差距？能不能在规定时间内从海量告警中发现真正的攻击？
3. **推动建设**：护网中暴露的短板，就是下一步安全建设的优先级清单
4. **形成威慑**：对外展示国家网络安全防护能力

> 💡 **一句话记住**：护网是安全行业的「高考」——一年一次，考的是真实的防御能力，不是书面上的「我们有防火墙」「我们有 IPS」。

---

**一个真实的护网场景缩影（凌晨 3:17）：**

```
SOC 大屏上，告警声突然响起——
「红标：外网 IP 45.33.xx.xx 对 OA 系统 /api/login 发起 SQL 注入攻击」

蓝队值班员小王：
├─ 0 分钟：点击告警，查看详情（源 IP、攻击 payload、目标 URL）
├─ 2 分钟：确认不是误报——请求中包含 ' OR 1=1 -- 的注入特征
├─ 3 分钟：查阅威胁情报平台，确认该 IP 属于某 APT 组织惯用基础设施
├─ 5 分钟：在 WAF 上对该 IP 执行永久封禁
├─ 8 分钟：排查 OA 系统 access.log，逐一检查该 IP 的所有历史请求
├─ 10 分钟：发现该 IP 在一周前就曾用别的 payload 探测过——但当时未触发告警
├─ 12 分钟：升级事件等级，通知值班组长
├─ 15 分钟：提交事件处置工单，详细记录攻击过程、处置动作、后续建议
└─ 全程不超过 15 分钟——蓝队值守的铁律
```

这 15 分钟里浓缩了蓝队所有的核心能力：**快速研判 → 精准处置 → 完整溯源 → 规范记录**。后面的 119 天，你要学

的就是这四项能力。

### 二、红队 🆚 蓝队 🆚 紫队——三支队伍的分工

护网行动中有明确的队伍划分，每一支都有特定的使命。用最容易理解的方式类比：

| 队伍 | 角色 | 通俗比喻 | 核心任务 | 你会加入吗？ |
|:---|:---|:---|:---|:---|
| 🔴 **红队** | 攻击方 | 假扮的小偷/盗贼 | 模拟 APT 组织，用各种手段突破防线，拿下靶标系统 | 不是（你守） |
| 🔵 **蓝队** | 防守方 | 保安+侦察组 | 监控全局，发现攻击、研判告警、拦截入侵、应急处置 | **就是你了！** |
| 🟣 **紫队** | 裁判/协调组 | 比赛裁判+导演 | 定规则、监督流程、判定得失分、协调红蓝矛盾 | 不是 |

---

**蓝队的核心职责（你的工作内容）：**

```
蓝队 = 监控者 + 分析者 + 处置者 + 记录者

                    ┌──────────────┐
                    │  蓝队大本营   │
                    │  (SOC/NOC)   │
                    └──────┬───────┘
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
   ┌────▼─────┐     ┌──────▼──────┐    ┌─────▼──────┐
   │ 实时监控  │     │  告警研判   │    │  事件处置   │
   │ (眼睛)   │     │  (大脑)     │    │  (手脚)     │
   └──────────┘     └─────────────┘    └────────────┘
   看大屏+收告警      判断真假+严重性     封IP+杀进程+隔离
        │                  │                  │
        └──────────────────┼──────────────────┘
                           │
                    ┌──────▼──────┐
                    │  报告记录   │
                    │  (笔杆子)   │
                    └─────────────┘
                    日报+处置单+事件报告
```

**一个口诀记住蓝队四项基本功**：**眼（监控）→ 脑（研判）→ 手（处置）→ 笔（记录）**

---

**为什么选择蓝队？——三大职业优势：**

1. **需求量大**：每个护网参演单位都需要蓝队成员，缺口巨大。红队顶尖人才少但竞争激烈，蓝队入门门槛更友好
2. **成长路径清晰**：初级值守 → 中级分析师 → 高级应急专家 → 安全运营经理 → CSO
3. **能力可迁移**：蓝队学到的日志分析、应急响应、系统运维等技能，在任何 IT 安全岗位都通用

### 三、蓝队一天上班到底干什么？——五类日常核心工作

很多新人以为蓝队就是整天跟黑客对打，实际上大部分时间在做这些：

---

#### 工作一：告警监控与研判（占工作时长 50%）

这是蓝队**最核心、最日常**的工作。你坐在 SOC 大屏或 SIEM 平台前，屏幕上的每一条告警都需要你来判断。

**告警从哪来？**
- 防火墙日志中的异常连接请求
- WAF（Web 应用防火墙）拦截的可疑 Web 请求
- IDS/IPS（入侵检测系统）匹配的攻击特征
- 终端 EDR（端点检测与响应）发现的异常行为
- 威胁情报平台推送的 IOC（入侵指标）匹配告警

**你需要对每条告警做三件事：**

```
第一步：看——这条告警在说什么？
├─ 来源 IP 是谁？（外部攻击者？内部机器？）
├─ 目标是什么？（Web 服务器？数据库？域控？）
├─ 攻击类型是什么？（SQL 注入？暴力破解？端口扫描？）

第二步：判——这是真攻击还是误报？
├─ 误报：正常业务行为被安全设备误判（不处理，标记后关闭）
├─ 试探：攻击者在扫描，但未成功突破（封 IP + 记录）
├─ 入侵：攻击已成功打入系统（升级！启动应急响应！）

第三步：处——根据研判结果执行动作
├─ 误报 → 标记关闭，必要时调整告警规则减少误报
├─ 试探 → 封禁源 IP，记录到威胁情报库
├─ 入侵 → 立即升级给组长，启动应急响应流程
```

> ⏱️ **黄金 15 分钟法则**：从告警产生到完成研判处置，护网期间要求不超过 15 分钟。超时 = 失分。

---

#### 工作二：日志分析（占工作时长 20%）

一个中型企业的 Web 服务器每天产生几十 GB 的访问日志。里面藏着大量的信息：

| 你在找什么 | 日志中的特征 | 用什么命令 |
|:---|:---|:---|
| 有人在扫目录？ | 同一个 IP 大量 404 状态码 | `grep "404" log \| awk '{print $1}' \| sort \| uniq -c \| sort -rn` |
| 有人在 SQL 注入？ | 请求参数中有 `' OR`、`UNION SELECT` 等 SQL 关键字 | `grep -iE "(union|select|or 1=1)" log` |
| 有人在暴力破解？ | 同一 IP 短时间内大量 POST 到登录接口 | `grep "POST /login" log \| awk '{print $1}' \| sort \| uniq -c \| sort -rn` |

**日志分析就像破案——你翻阅案发现场的所有监控录像来找线索。** 区别在于，你要翻的不是几个小时的录像，而是数十 GB 的文本日志，工具是你的 grep/awk/sed 三剑客。

---

#### 工作三：安全设备运维（占工作时长 10%）

安全设备是蓝队的「武器」。你需要确保：

- **防火墙**规则正确加载，策略没有遗漏
- **WAF** 检测引擎正常，没有因为性能问题降级
- **IDS/IPS** 特征库是最新的（至少每天更新一次）
- **SIEM** 日志源都在正常上报，没有因为磁盘满而丢数据
- **EDR**（端点检测与响应）所有终端的 Agent 都在线

**检查方法**：每天早上用 15 分钟，逐一登录每台设备的管理界面，看状态指示灯是不是全绿。红一个都不行。

---

#### 工作四：事件响应与应急处置（占工作时长 10%）

当告警研判确认发生了真实的入侵事件，你要立刻切换到「应急模式」：

```
应急处置六步法（PDCERF）：
1. 准备（Prepare）——平时已建好应急预案、工具库、通讯录
2. 检测（Detect）——确认事件真实性和影响范围
3. 遏制（Contain）——隔离受感染机器，断网但不断电
4. 根除（Eradicate）——清除后门、恶意进程、Webshell
5. 恢复（Recover）——从干净备份恢复系统，逐步上线
6. 复盘（Follow-up）——写事件报告，改进防护策略
```

> ⚠️ **关键原则**：遏制优先于根除——先止血，再动手术。如果看到可疑进程就直接 kill，攻击者可能在你 kill 之前已经完成了数据窃取。正确的做法是先隔离网络（断网），保留现场证据，再做分析。

---

#### 工作五：报告与文档（占工作时长 10%）

这可能是最「不酷」但最重要的工作。在安全行业有一个铁律：

> **没有记录 = 没有发生**

护网期间每天需要写三类报告：

1. **值守日报**：当日告警总数、处置告警数、封禁 IP 数、重大事件摘要、当前风险状态
2. **事件处置工单**：每条真实告警的完整处置记录（时间、动作、结果、操作人）
3. **事件分析报告**：对重大事件的深度分析（攻击路径还原、影响评估、改进建议）

**为什么文档这么重要？**
- 护网结束后，裁判组通过你的文档来评判你的防守质量
- 如果你的 WAF 拦截了 100 次攻击但没有任何处置记录，裁判会认为你的防守是盲目的
- 如果你的系统被攻破了但你有完整的事件响应记录，反而可能因为「处置规范」而加分

### 四、护网值守的完整标准流程——从接班到交班

假设你是白班（08:50-18:00）的值班人员，这是你一天的标准动作：

```
============================
 08:50  接班签到
============================
- 到达 SOC 值班室（提前 10 分钟）
- 和夜班同事做口头交接：
  ├─ 昨晚有无重大事件？（有无入侵告警未闭环？）
  ├─ 当前封禁列表里有哪些 IP？（避免重复处理）
  ├─ 有什么需要持续监控的目标？（某台服务器出现异常？）
  └─ 有什么上级指令？（领导要求重点监控某系统？）

============================
 09:00-09:30  全量告警扫描
============================
- 打开 SIEM 平台，查看「待处置告警」列表
- 把昨晚产生的所有未处理告警逐条过一遍
- 按优先级处理：红色（高）→ 橙色（中）→ 黄色（低）
- 每条告警都要有结论：误报/已处置/升级处理

============================
 09:30-12:00  持续监控（上午班）
============================
- 实时盯告警大屏 + SIEM 界面
- 新告警弹出 → 研判 → 处置 → 记录
- 每 2 小时做一次态势汇总：
  ├─ 过去 2 小时内有多少条告警？
  ├─ 有没有新的攻击源出现？
  └─ 当前封禁 IP 列表变化？

============================
 12:00-13:00  午休（必须找人替班！）
============================
- 绝对不能让监控席位空着！
- 你去吃饭/上厕所之前，必须找同事暂代你的席位
- 代班同事也要在你的 SIEM 账号下操作（责任可追溯）

============================
 13:00-17:30  持续监控（下午班）
============================
- 同上午，持续告警监控 + 研判处置
- 14:00 左右：第二次态势汇总
- 16:00 左右：第三次态势汇总

============================
 17:30-17:50  撰写值班日报
============================
日报模板：
┌────────────────────────────────────┐
│ 护网值守日报 - 第X天白班           │
│ 值班人：XXX                        │
│ 时间：08:50 - 18:00                │
├────────────────────────────────────┤
│ 一、告警概况                        │
│   总告警数：XXX                     │
│   已处置：XXX（其中误报 XXX）        │
│   待处置：XXX                       │
│   已升级事件：XXX                    │
├────────────────────────────────────┤
│ 二、处置动作                        │
│   封禁 IP：XXX 个                   │
│   IP 列表：xxx, xxx, xxx           │
│   隔离主机：XXX 台                   │
├────────────────────────────────────┤
│ 三、重点关注                        │
│   （有异常的资产/持续的攻击源/      │
│    未闭环的事件）                    │
├────────────────────────────────────┤
│ 四、交接事项                        │
│   （需要夜班继续跟进的工单号）        │
└────────────────────────────────────┘

============================
 17:50-18:00  交班
============================
- 把日报发给值班组长和夜班同事
- 口头交接重点事项
- 确保你的 SIEM 账号没有未处理告警
```

> ⚠️ **值守铁律**：
> 1. 7×24 小时席位不空——上厕所找替班，吃饭轮流去
> 2. 告警不超过 15 分钟响应——慢一秒就可能失分
> 3. 任何操作都要留痕——截图 + 文字说明，不能事后补
> 4. 不确定的事立即升级——不要自己猜，不要侥幸

### 五、蓝队新人必须掌握的 15 个核心术语

新入行最怕的不是技术难，是开会听不懂别人在说什么。下面这张表，15 分钟看懂，以后开会不再懵逼：

| # | 术语 | 全称 | 用大白话解释 | 一句话记忆 |
|:---|:---|:---|:---|:---|
| 1 | **SOC** | Security Operations Center | 安全运营中心，蓝队的大本营，所有告警的汇聚地 | 蓝队指挥部 |
| 2 | **SIEM** | Security Information and Event Management | 日志集中管理+分析平台，把全公司日志收到一处 | 蓝队的中央情报局 |
| 3 | **告警（Alert）** | — | 安全设备发现可疑行为时发出的警告 | 门铃响了——有人敲门 |
| 4 | **误报（False Positive）** | — | 正常行为被设备判定成了攻击 | 风吹动了门，门铃也响了 |
| 5 | **漏报（False Negative）** | — | 真实攻击发生了但设备没发现 | 小偷翻窗进来，门铃没响 |
| 6 | **IOC** | Indicator of Compromise | 入侵指标：恶意IP、恶意域名、恶意文件哈希等 | 罪犯的指纹档案 |
| 7 | **研判（Triage）** | — | 分析告警、判定真假、定级的完整过程 | 急诊分诊台 |
| 8 | **闭环（Close Loop）** | — | 从发现到处置完毕的全流程走完 | 有始有终，不烂尾 |
| 9 | **CVE** | Common Vulnerabilities and Exposures | 漏洞身份证号，国际通用 | 漏洞的身份证 |
| 10 | **ATT&CK** | Adversarial Tactics, Techniques, and Common Knowledge | MITRE 发布的攻击行为知识库 | 黑客战术百科全书 |
| 11 | **PDCERF** | Prepare-Detect-Contain-Eradicate-Recover-Follow-up | 事件应急响应的标准六步法 | 应急六步诀 |
| 12 | **EDR** | Endpoint Detection and Response | 终端检测与响应，监控每台电脑上的异常行为 | 每台电脑上的保安 |
| 13 | **WAF** | Web Application Firewall | Web 应用防火墙，专防针对网站的 HTTP 攻击 | 网站专属保安 |
| 14 | **TTP** | Tactics, Techniques, Procedures | 攻击者的战术、技术和流程，也是 ATT&CK 的核心分类维度 | 攻击者的行为模式 |
| 15 | **SLA** | Service Level Agreement | 服务水平协议，规定了「多快响应、多快处理」 | 护网的绩效 KPI |

**自测方法**：拿张纸挡住「大白话解释」列，看到术语能不能自己说出解释。15 个全说对 → 你已经是半个老手了。
