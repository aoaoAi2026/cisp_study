## 📖 核心知识讲解



### 一、越权漏洞的本质


越权（Broken Access Control）是OWASP Top 10中排名第一的漏洞类型。它的本质是：

**应用程序信任了用户提供的数据来决定"谁能看什么"，而不是在服务端做严格校验。**



通俗比喻：你去银行柜台，柜员问你"你想看谁的账户？"，你说"张三的"，柜员就给你看了。

正常的流程应该是：柜员让你出示身份证，然后只给你看你自己的账户。

越权漏洞就是——应用程序没检查"你到底是谁"，直接相信了你的请求。




### 二、水平越权（IDOR）


**定义**：同权限级别的用户之间，通过修改资源ID访问了别人的数据。



**典型场景**：

- 修改URL中的user_id参数：`/api/order?user_id=1001` → `user_id=1002` 看到了别人的订单

- 修改请求体中的ID字段：`{"order_id": 12345}` → `{"order_id": 12346}`

- 遍历资源ID：`/user/profile/1001` → `/user/profile/1002` 挨个枚举






### 三、垂直越权


**定义**：低权限用户执行了需要高权限才能做的操作。



**典型场景**：

- 普通用户访问管理员页面：`/admin/user_manage` 没有做权限校验

- 普通用户调用管理员接口：`/api/admin/delete_user` 只用前端隐藏了按钮

- Cookie/Token角色篡改：`role=user` 改成 `role=admin`






### 四、水平越权 vs 垂直越权对比


| 维度 | 水平越权 | 垂直越权 |
|--|----|----|
| 攻击者身份 | 已登录的普通用户 | 已登录的低权限用户 |
| 攻击目标 | 同级其他用户的数据 | 管理员才能执行的操作 |
| 关键特征 | 修改资源ID | 绕过角色/权限检查 |
| 危害等级 | 数据泄露，影响面看数据量 | 系统控制，通常危害更高 |
| 检测难度 | 日志看起来完全正常 | 可能有异常API调用记录 |
| 典型接口 | 查询/读取类接口 | 增删改/管理类接口 |




### 五、越权漏洞的检测方法


**方法一：代码审计**

- 检查每个API是否在入口处做权限校验

- 关注直接使用用户输入作为资源ID的地方

- 搜索权限注解/装饰器的使用覆盖度



**方法二：黑盒测试（手动）**

- 登录用户A，抓包，替换Cookie/Token用用户B的身份

- 遍历URL中的数字ID参数

- 用低权限账号访问高权限URL路径



**方法三：日志分析（蓝队视角）**

- 同一用户短时间内访问多个不同用户的数据 → 水平越权特征

- 低权限角色调用了管理类API → 垂直越权特征

- 某IP在一段时间内遍历了大量用户ID → 枚举攻击




### 六、蓝队如何检测越权攻击


越权攻击从日志角度看"完全正常"，因为请求格式、参数都合法。关键是：

**建立"正常行为基线"，找出偏离基线的行为。**



**检测规则思路：**

1. **ID遍历检测**：同一用户短时间内请求了超过N个不同资源ID → 告警

2. **角色-接口映射**：建立"角色→允许的API列表"，命中不允许的API → 告警

3. **数据量异常**：某用户突然拉取了大量数据（远超其正常使用量）→ 告警

4. **时间异常**：非工作时间有管理操作 → 告警

5. **跨用户数据访问**：用户A访问了用户B的数据，且A和B没有组织关联 → 告警




### 七、防御越权的最佳实践


1. **服务端校验（最重要）**：永远不要在前端做权限控制，服务端每个接口都要验权

2. **用户身份绑定**：资源查询时用当前登录用户的身份，不要信任请求参数中的用户标识

3. **使用UUID替代自增ID**：让攻击者无法通过遍历数字来猜资源ID

4. **最小权限原则**：用户/角色只有完成其职责所需的最小权限

5. **操作审计日志**：记录"谁在什么时间操作了什么资源"，用于事后追溯






## 🔧 实操任务





### 任务一：手工越权测试（30分钟）


1. 在DVWA或Pikachu靶场中找到越权漏洞练习模块

2. 用Burp Suite抓包，尝试修改请求中的资源ID参数

3. 观察响应，确认是否成功访问了其他用户的数据

4. 记录你的测试步骤和结果




### 任务二：编写越权检测规则（30分钟）


假设你的SIEM中有以下日志字段：user_id、api_path、resource_id、timestamp、role

设计一条检测水平越权的规则：

1. 定义"正常访问模式"：一个用户通常访问哪些resource_id？

2. 定义"异常阈值"：短时间内访问超过N个不同的resource_id

3. 写出伪代码或ELK查询语句




### 任务三：日志分析实战（30分钟）


如果有条件，找一些包含越权攻击的Web访问日志样本

用grep/awk/python分析，找出其中的越权行为

关键分析维度：用户→时间→资源ID→API路径

## 📝 今日小结


越权漏洞之所以排在OWASP Top 10的第一位，因为它是"最容易被忽略但影响面最广"的安全问题。

攻击者不需要写复杂的Payload，不需要绕过WAF，只需要改一个数字——这就是越权的可怕之处。

作为蓝队，你要理解：好的越权检测一定是和业务紧密结合的。多跟业务方沟通，了解每个接口的正常用法，才能发现异常用法。

## 🚀 蓝队进阶之路：从"能检测"到"能防御"

**第一阶段：能识别** — 看到告警知道大概是什么攻击
**第二阶段：能分析** — 能判断攻击的严重程度、攻击者的意图、攻击到了哪一步
**第三阶段：能防御** — 不仅知道怎么检测，还知道怎么从根本上防御这类攻击

问问自己：对于今天学的这类攻击/技术，你目前在哪个阶段？

### 如何加速成长
1. **刻意练习**：不是泛泛地看，而是针对一个漏洞反复演练检测和防御

## 🔬 深入案例：一起真实越权攻击的分析

**背景**：某电商平台被曝大量用户订单数据泄露。

**攻击过程还原：**
1. 攻击者注册了一个普通账号，查看自己的订单详情
2. 在Burp Suite中拦截了 `/api/order/detail?order_id=20230601001` 请求
3. 发现 order_id 是有规律的递增数字
4. 用Intruder遍历 order_id 从 20230601001 到 20230602000
5. 成功获取了1000+个其他用户的订单（包含姓名、地址、电话）

**为什么日志看起来正常？**
- 每个请求都是合法的HTTP GET
- 使用了有效的Session Token
- 返回状态码都是200 OK
- 没有触发任何WAF规则

**日志中唯一能发现异常的线索：**
- 同一IP/Token在10分钟内请求了1000+个不同的order_id
- 如果是正常用户购物，怎么会1分钟看100个不同的订单？

> 💡 **教训**：越权检测最有效的手段不是看"请求长什么样"，而是看"请求的行为模式"。
