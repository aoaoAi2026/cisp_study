## 📖 核心知识讲解

### 一、HTTP是什么？——网页世界的「快递单」

HTTP（超文本传输协议）就是网页浏览器和Web服务器之间的通信规则。你每次打开网页，背后都发生了HTTP请求和响应。

**一个形象比喻**：你去餐厅吃饭

| HTTP流程 | 餐厅比喻 |
|:---|:---|
| URL | 餐厅地址（比如「王府井大街88号2楼川菜馆」） |
| 请求方法(GET/POST) | 你是说「给我菜单」（GET）还是「我要点菜」（POST） |
| 请求头 | 你的要求：「不放香菜」「加急」「打包带走」 |
| 请求体 | 你填的点菜单 |
| 状态码 | 服务员回复：「好的马上」（200）「卖完了」（404）「后厨炸了」（500） |
| 响应体 | 端上来的菜（HTML页面） |

---

### 二、解剖一个HTTP请求——每个部分都干

当你访问 `http://example.com/login?user=admin` 时，浏览器发出的请求是这样的：

```
GET /login?user=admin HTTP/1.1              ← 请求行
Host: example.com                            ← 请求头开始
User-Agent: Mozilla/5.0 Chrome/120
Accept: text/html
Accept-Language: zh-CN
Cookie: session_id=abc123
Referer: http://example.com/home
                                             ← 空行，头结束
                                             ← 请求体（GET请求通常为空）
```

**请求行三个部分：**

```
GET          /login?user=admin     HTTP/1.1
↑             ↑                      ↑
请求方法      请求路径+参数           协议版本
```

**蓝队关注的重点请求头：**

| 请求头 | 含义 | 蓝队为什么关注 |
|:---|:---|:---|
| **User-Agent** | 浏览器标识 | 攻击工具（sqlmap/python-requests）的UA和正常浏览器不一样 |
| **Cookie** | 登录凭证 | 被窃取的Cookie可以让攻击者冒充你 |
| **Referer** | 从哪个页面跳转来 | 如果Referer是空的或奇怪的URL，可能是直接构造的请求 |
| **Host** | 目标主机 | 检查是不是在访问钓鱼域名 |
| **X-Forwarded-For** | 原始IP | 攻击者可能伪造这个头来隐藏真实IP |

### 三、解剖一个HTTP响应

服务器收到请求后，会返回这样的响应：

```
HTTP/1.1 200 OK                              ← 状态行
Server: nginx/1.18.0                          ← 响应头开始
Content-Type: text/html; charset=utf-8
Content-Length: 1234
Set-Cookie: session_id=xyz789; HttpOnly
                                              ← 空行
<!DOCTYPE html>                               ← 响应体开始
<html>
<head><title>Welcome</title></head>
<body>...</body>
</html>
```

**蓝队关注的响应头：**

| 响应头 | 含义 | 蓝队为什么关注 |
|:---|:---|:---|
| **Server** | Web服务器软件和版本 | nginx/1.18.0——如果这个版本有已知漏洞，你就要排查 |
| **Set-Cookie** | 设置Cookie | 检查HttpOnly和Secure标志是否正确设置 |
| **Content-Type** | 响应内容类型 | 如果请求的是图片但返回了text/html→可能有文件上传漏洞 |
| **X-Powered-By** | 后端语言和版本 | PHP/7.4.33——同样，版本可能暴露漏洞信息 |

---

### 四、五大最重要的HTTP状态码

状态码就像服务员给你的回复：

| 状态码 | 含义 | 餐厅比喻 | 蓝队视角 |
|:---|:---|:---|:---|
| **200 OK** | 成功 | 「菜来了，请慢用」 | 正常，也可能是webshell连接成功 |
| **301/302** | 重定向 | 「我们换新店了，请移步」 | 检查重定向目标是不是钓鱼网站 |
| **403 Forbidden** | 禁止访问 | 「VIP区域，你不能进」 | 有人试图访问没权限的目录——可能是在探测 |
| **404 Not Found** | 找不到 | 「这道菜我们不做」 | 大量404说明有人在扫目录 |
| **500** | 服务器内部错误 | 「后厨炸了！」 | 可能是SQL注入攻击导致了数据库报错 |

**蓝队日志分析的关键模式：**

```bash
# 找出所有非200的响应，按状态码统计
awk '{print $9}' access.log | sort | uniq -c | sort -rn

# 结果可能类似：
# 8542 200    ← 正常请求
# 1204 404    ← 这么多404！有人在扫描不存在的路径
#  523 302    ← 重定向
#  156 500    ← 服务器错误，要排查有没有SQL注入
#   89 403    ← 被拒绝的请求
#   12 502    ← 网关错误
```

**一个真实的攻击模式：大量404 → 说明有人在目录扫描 → 接下来可能就有攻击payload出现在200响应中。**

### 五、GET vs POST——两种请求方式的天壤之别

| 对比维度 | GET | POST |
|:---|:---|:---|
| **用途** | 获取数据（查） | 提交数据（增删改） |
| **参数位置** | URL里（网址上能看到） | 请求体里（网址上看不到） |
| **比喻** | 在图书馆查书（图书证上没秘密） | 在银行取款（密码不能写在取款单上） |
| **安全性** | 参数暴露在URL中，日志会记录 | 参数在body中，但并非加密 |
| **长度限制** | URL长度有限制 | 理论无限制 |

**蓝队检测要点**：
- 正常用户在URL里看到`?user=admin' OR '1'='1`是不可能的→这是SQL注入攻击
- 登录操作必须用POST（密码不能出现在URL里），如果某系统用GET传密码→严重安全漏洞
- 攻击者常用POST请求提交webshell命令

### 六、HTTPS是怎么保护你的？——SSL/TLS握手

HTTP是明文传输的——就像寄明信片，经过的每个人都看得到内容。HTTPS就像把信装进了保险箱。

**SSL/TLS握手过程（简化版）：**

```
客户端（浏览器）                        服务器（网站）
    │                                      │
    │ ① ClientHello: "我支持这些加密算法"    │
    │─────────────────────────────────────>│
    │                                      │
    │ ② ServerHello: "我们用这种算法"        │
    │    证书: "这是我的身份证，CA签过名的"    │
    │<─────────────────────────────────────│
    │                                      │
    │ ③ 验证证书（看身份证是不是假的）        │
    │    生成随机密钥，用服务器公钥加密发给它    │
    │─────────────────────────────────────>│
    │                                      │
    │ ④ 双方用协商好的密钥开始加密通信         │
    │<══════════ 加密通道 ═══════════════>│
```

**蓝队需要知道的HTTPS安全问题：**
1. **证书过期**：浏览器会报警，用户可能忽略→攻击者可利用
2. **自签名证书**：没有CA认证的证书，容易被中间人攻击
3. **旧版本TLS**：TLS 1.0/1.1已不安全，应使用TLS 1.2+
4. **HTTPS不能防所有攻击**：HTTPS只保证传输过程加密，不防SQL注入、XSS等应用层攻击

## 🔧 实操任务

1. **浏览器F12实战**：打开任意网站，按F12→Network→刷新页面→点开第一个请求→查看Request Headers和Response Headers
2. **识别状态码**：故意访问一个不存在的页面（如`xxx.com/abcdefg`），看返回什么状态码
3. **对比GET/POST**：在登录页面输入账号密码，点登录，在F12 Network中找到登录请求，确认它是POST请求，查看请求体
4. **Wireshark抓HTTP**：用Wireshark过滤`http`，访问一个HTTP网站（不要用HTTPS），查看明文传输的HTTP请求和响应
5. **日志分析练习**：找一段nginx日志，用awk统计状态码分布、找出所有返回500的请求
6. **对比HTTPS**：用Wireshark抓HTTPS流量，和HTTP对比——你只能看到TLS加密后的乱码

## 📝 今日小结

HTTP/HTTPS是Web安全的基石。绝大多数攻击（SQL注入、XSS、文件上传、暴力破解）都是通过HTTP请求发起的。看懂HTTP请求 = 看懂攻击者在对你的网站说什么。今天的内容是后面学习所有Web攻击检测的基础，务必把每个状态码、每个请求头的作用都搞清楚。
