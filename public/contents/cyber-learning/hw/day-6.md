## 一、应急响应到底是怎么回事？

### 1.1 用大白话说

**应急响应** = 攻击发生了，你怎么快速、正确地把攻击者赶走，把损失降到最低，同时保留证据以便秋后算账。

类比：你家着火了——
- 🔥 第一步不是去追究谁放的火，而是**灭火**（遏制）
- 🧹 然后清理现场，**修复损坏**（根除和恢复）
- 📝 最后总结**为什么着火**，防止再犯（复盘）

### 1.2 应急响应失败的典型场景

| 失败案例 | 问题出在哪 |
|:---|:---|
| 发现Webshell直接删了 | 没保留证据，溯源无从谈起 |
| 中毒断网后立即重装系统 | 破坏了取证现场 |
| 勒索病毒来了立马付赎金 | 可能助长犯罪且不一定能恢复 |
| 发现入侵后一个人闷头处理 | 没有按规定上报，团队不知道 |

## 二、PDCERF 六步应急法详解

### 2.1 全貌速览

```
P — Prepare    准备    🎒 平时准备好应急预案和工具
D — Detection  检测    🔍 发现安全事件
C — Containment 遏制   🛑 控制事态，防止扩散
E — Eradication 根除   🧹 清除攻击者所有痕迹
R — Recovery   恢复    🔄 恢复系统正常运行
F — Follow-up  总结    📝 复盘，避免再犯
```

### 2.2 逐步深入

#### P：准备（Prepare）——仗怎么打，兵就怎么练

| 准备工作 | 具体内容 |
|:---|:---|
| 应急预案 | 各类事件的标准处置流程文档 |
| 应急工具包 | 取证U盘、抓包工具、日志收集脚本 |
| 应急团队 | 明确每个角色和备份人员 |
| 沟通机制 | 上报流程、通报渠道、升级条件 |
| 资源准备 | 备用服务器、备用带宽、隔离网络环境 |
| 定期演练 | 至少每季度一次应急演练 |

> **💡 金句**：应急响应的质量，90%取决于平时准备了什么。

#### D：检测（Detection）——发现敌情

| 检测渠道 | 能发现什么 |
|:---|:---|
| SIEM告警 | 关联分析后的高危事件 |
| 用户报告 | 用户发现异常（网站打不开、文件加密等） |
| 外部通报 | CERT、监管机构、同行通报 |
| 主动巡检 | 日常威胁狩猎主动发现 |

**确认事件必须回答的三个问题**：
1. 是真实攻击还是误报？
2. 影响范围多大？
3. 攻击还在持续吗？

#### C：遏制（Containment）——先止血！

| 遏制措施 | 适用场景 | 注意 |
|:---|:---|:---|
| 断网隔离 | 确认已感染的主机 | **先收集关键证据再断网！** |
| 封禁IP | 确定攻击来源IP | 注意不要误封CDN/代理 |
| 关闭服务 | 受影响的业务服务 | 评估业务影响，通知相关人员 |
| 修改凭据 | 密码已泄露 | 所有相关账号一起改 |
| 启用WAF拦截规则 | Web攻击 | 快速部署虚拟补丁 |

> ⚠️ **最重要的原则**：遏制前先保留证据！断网之前，先保存进程列表、网络连接信息和可疑文件。

#### E：根除（Eradication）——连根拔起

| 操作 | 做什么 |
|:---|:---|
| 删除恶意文件 | WebShell、木马、后门程序 |
| 清除持久化机制 | 计划任务、注册表启动项、服务 |
| 修复漏洞 | 打补丁、改配置、升级版本 |
| 重置凭据 | 修改所有可能泄露的账号密码 |
| 全面杀毒 | 全盘扫描确保没有残留 |

> ⚠️ **别只删除文件**！攻击者通常留了多个后门，必须按攻击链逐一排查。

#### R：恢复（Recovery）——重新上线

| 步骤 | 说明 |
|:---|:---|
| 从可信备份恢复 | 确保备份没有被感染 |
| 验证系统完整性 | 文件校验、服务测试 |
| 加强监控 | 恢复后短期内重点监控 |
| 分阶段上线 | 先非核心系统，再核心系统 |

#### F：总结（Follow-up）——吃一堑长一智

复盘要回答：
- 攻击者怎么进来的？（根因）
- 为什么没及时发现？（检测盲区）
- 处置过程有什么可以优化的？（流程）
- 如何防止同类事件再次发生？（改进）

## 三、护网最常见的四种攻击事件处置

### 3.1 Webshell入侵事件

```
发现Webshell
  ↓
【遏制】确认文件路径 → 记录文件属性和内容(HASH) → 备份样本
  ↓
【分析】分析Webshell类型和功能 → 检查访问日志 → 确定上传时间
  ↓
【溯源】查Web访问日志 → 找到上传请求 → 定位攻击者IP和手法
  ↓
【根除】删除Webshell → 修复上传漏洞 → 检查是否有其他Webshell
  ↓
【恢复】从安全备份恢复被篡改的页面 → 加强Web监控
  ↓
【总结】为什么上传漏洞没修？有没有类似漏洞？
```

### 3.2 勒索软件事件

```
发现文件被加密/勒索信
  ↓
【遏制】立即断网隔离感染主机！→ 检查其他主机是否感染 → 阻断横向传播
  ↓
【分析】确定勒索家族 → 检查备份可用性 → 评估影响范围
  ↓
【决策】❌ 不建议支付赎金！→ ✅ 从备份恢复 → 使用解密工具（如有）
  ↓
【根除】全盘杀毒 → 修复感染途径（通常是钓鱼/弱口令/RDP暴露）
  ↓
【恢复】从备份恢复数据 → 验证数据完整性
```

### 3.3 数据泄露事件

```
发现数据外传
  ↓
【遏制】阻断外传通道 → 隔离受影响系统 → 修改泄露的凭据
  ↓
【评估】哪些数据泄露了？→ 涉及多少用户？→ 是否需要依法上报？
  ↓
【溯源】怎么泄露的？→ 谁拿的？→ 传到了哪里？
  ↓
【通知】按规定上报监管 → 通知受影响用户
```

### 3.4 钓鱼邮件事件

```
员工点击了钓鱼邮件
  ↓
【遏制】检查该员工电脑是否有异常 → 重置该员工所有密码 → 隔离可疑邮件
  ↓
【排查】同主题邮件还发给了谁？→ 还有谁点击了？→ 全量排查
  ↓
【根除】删除所有同主题钓鱼邮件 → 封禁钓鱼域名和IP
  ↓
【培训】对点击者进行再教育 → 全员通报提醒
```

## 四、证据保护——护网取证的铁律

### 4.1 电子取证第一原则

> **不破坏原始证据！不破坏原始证据！不破坏原始证据！**

| 正确做法 | 错误做法 |
|:---|:---|
| 先做磁盘/内存镜像再分析 | 直接在原始磁盘上翻找文件 |
| 使用只读工具取证 | 双击打开可疑文件"看看是什么" |
| 记录每一步操作 | 做了什么都不记 |
| 建立证据保管链 | 谁拿了证据不知道 |

### 4.2 证据保管链（Chain of Custody）

记录证据的每一段"旅程"：

```
谁取的 → 什么时候取的 → 从哪取的 → 怎么保管的 → 交给了谁
```

**意义**：确保证据在法庭上有法律效力，不会被质疑被篡改。

### 4.3 取证工具推荐

| 工具 | 用途 | 免费？ |
|:---|:---|:---|
| FTK Imager | 磁盘镜像 | ✅ 免费 |
| Autopsy | 磁盘取证分析 | ✅ 开源 |
| Volatility | 内存取证 | ✅ 开源 |
| Wireshark | 网络数据包分析 | ✅ 开源 |
| RegRipper | 注册表分析 | ✅ 开源 |

## 五、应急响应记录与报告

### 5.1 应急响应 PDCERF 自动化引擎

```python
from datetime import datetime

class IncidentResponder:
    def __init__(self, incident_name):
        self.name = incident_name
        self.timeline = []
        self.status = '待响应'
    
    def execute_phase(self, phase, actions):
        """执行一个应急阶段"""
        timestamp = datetime.now().strftime("%Y-%m-%d %H:%M")
        self.timeline.append(f'[{timestamp}] === {phase}阶段 ===')
        for action in actions:
            self.timeline.append(f'  → {action}')
        print(f'✅ {phase}阶段完成 ({len(actions)}项操作)')
    
    def report(self):
        """输出应急响应报告"""
        print(f'\n{"="*50}')
        print(f'  应急响应报告：{self.name}')
        print(f'{"="*50}')
        for entry in self.timeline:
            print(entry)

# 模拟一次Webshell应急响应
ir = IncidentResponder('Webshell入侵事件 - SRV-WEB01')

ir.execute_phase('准备', [
    '激活应急响应小组（张三、李四、王五）',
    '确认分工：张三-分析、李四-处置、王五-协调',
    '准备取证工具：FTK Imager、Wireshark'
])

ir.execute_phase('检测', [
    '确认Webshell文件路径：/var/www/html/shell.php',
    '分析Webshell类型：PHP一句话木马',
    '评估影响范围：仅Web服务器SRV-WEB01'
])

ir.execute_phase('遏制', [
    '备份Webshell样本用于分析',
    '隔离受影响Web服务器（限制出站流量）',
    '收集系统进程列表和网络连接信息',
    '收集Web访问日志'
])

ir.execute_phase('根除', [
    '删除Webshell文件 shell.php',
    '修复文件上传漏洞',
    '修改相关账号密码',
    '检查是否有其他可疑文件'
])

ir.execute_phase('恢复', [
    '恢复Web服务访问',
    '验证网站功能正常',
    '通知业务方确认',
    '加强Web监控规则'
])

ir.execute_phase('总结', [
    '编写事件报告',
    '复盘上传漏洞根因',
    '制定同类漏洞排查计划',
    '更新应急预案'
])

ir.report()
```

## 六、应急响应中的常见错误

| 错误 | 后果 | 正确做法 |
|:---|:---|:---|
| 发现后门直接删除 | 丢失取证线索 | 先备份留样再删除 |
| 自己一个人处理 | 缺乏协作，处置不全面 | 按预案分工协作 |
| 急于重启/重装 | 丢失内存证据 | 先做内存镜像 |
| 只清理明面上的恶意文件 | 攻击者隐藏得更深 | 按攻击链接力排查 |
| 不写应急报告 | 无法复盘改进 | 边处置边记录 |

## 七、护网应急响应的"4要4不要"

### 4要 ✅

1. **要立即启动预案**：别犹豫，按预案行事
2. **要保留原始证据**：再急也先留样本
3. **要同步通知领导**：别怕被批评，隐瞒才是大问题
4. **要做好过程记录**：每一步都记下来

### 4不要 ❌

1. **不要急于删除样本**：先取证后清理
2. **不要单独操作**：双人检查，互相监督
3. **不要随意重启系统**：内存中的数据是宝贵证据
4. **不要对外透露细节**：保密纪律
