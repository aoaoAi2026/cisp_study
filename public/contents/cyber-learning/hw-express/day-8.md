### 二、攻击链路（Kill Chain）——把散落的告警串成"故事"

单条告警像散落的拼图碎片。攻击链路分析就是**把碎片拼成完整的犯罪现场**。

#### 2.1 七阶段 Kill Chain 模型

#### 2.2 蓝队视角的"简化四阶段"

实战中，我们更关注**能检测到的环节**：

```
① 边界突破阶段（Breach）
   ├─ 攻击方式：SQL注入、文件上传、命令执行、弱口令
   ├─ 检测手段：WAF告警、Web日志、IDS特征匹配
   └─ 关键日志：access.log（异常请求）、数据库日志（异常查询）

② 权限提升阶段（Privilege Escalation）
   ├─ 攻击方式：SUID提权、内核漏洞、sudo滥用
   ├─ 检测手段：系统日志、进程审计
   └─ 关键日志：/var/log/secure（sudo操作）、EventID 4672（特权分配）

③ 横向移动阶段（Lateral Movement）
   ├─ 攻击方式：SSH密钥、RDP、SMB共享、WMI
   ├─ 检测手段：网络流量、登录日志
   └─ 关键日志：EventID 4624（网络登录）、SSH auth日志

④ 数据窃取阶段（Exfiltration）
   ├─ 攻击方式：HTTP POST大文件、DNS隧道、FTP外传
   ├─ 检测手段：出站流量监控、DLP
   └─ 关键日志：出站流量异常（大文件、非标准端口、外连境外IP）
```

### 三、WAF绕过手法——攻击者怎么"骗过安检"？

WAF就像机场安检，它会检查HTTP请求包的"行李"里有没有违禁品。但聪明的攻击者总想方设法把"炸弹"伪装成"面包"。

#### 3.1 WAF绕过的八种经典手法

**① 大小写混写（Case Toggle）**

```sql
-- WAF正则会匹配：union select
-- 攻击者写成：
UnIoN SeLeCt 1,2,3-- 

-- WAF可能把大小写不同的词当作不同的东西（取决于WAF配置）
```

**防御**：WAF规则应该使用大小写不敏感匹配（`/union select/i`）

---

**② URL编码（URL Encoding）**

```sql
-- 正常：union select
-- 编码后：%75%6e%69%6f%6e%20%73%65%6c%65%63%74
-- 双重编码：%25%37%35%25%36%65...（先%→%25）
```

**防御**：WAF应先做URL解码再匹配规则

---

**③ 注释符分割（Comment Injection）**

```sql
-- 正常：union select
-- 攻击者写成：
/**/union/**/select
/*!50000union*/ select   -- MySQL版本条件注释
uni/**/on sel/**/ect
```

**防御**：WAF应在匹配前先去除注释

---

**④ 内联注释（Inline Comment）**

```sql
/*!50000union*/ select 1,2,3
-- MySQL中 /*!50000...*/ 相当于执行其中的语句
-- WAF可能认为这是注释而放行
```

---

**⑤ 等价函数替换（Function Equivalence）**

```sql
-- 正常：SELECT @@version
-- 等价：SELECT VERSION()
-- 正常：SELECT DATABASE()
-- 等价：SELECT SCHEMA()
-- 正常：sleep(5)
-- 等价：benchmark(50000000,md5(1))
```

**防御**：WAF需要对已知的等价函数建立映射，不仅是关键词黑名单

---

**⑥ 参数污染（HPP / Parameter Pollution）**

```
正常：?id=1 union select 1,2,3
污染：?id=1&id=union&id=select&id=1,2,3

不同的Web服务器处理方式不同：
- Apache/PHP：取最后一个参数值
- IIS/ASP：取第一个参数值
- Tomcat/JSP：合并所有值
```

**防御**：理解后端平台的参数处理逻辑，针对性检测

---

**⑦ 分块传输（Chunked Transfer Encoding）**

```http
POST /login.php HTTP/1.1
Transfer-Encoding: chunked

5
union
6
 select
3
 1,2
```

WAF可能不重组分块数据，导致匹配不到完整的攻击payload。

---

**⑧ 编码绕过（Base64 / Hex / Char）**

```sql
-- SQL Server中：
-- 正常：SELECT * FROM users WHERE id = 1
-- Hex编码绕过：
DECLARE @s VARCHAR(8000)
SET @s=CAST(0x73656C656374202A2066726F6D207573657273 AS VARCHAR(8000))
EXEC(@s)
```

---

#### 3.2 WAF绕过检测的蓝队思路

当你在日志中看到**看似"正常"但又有微妙不同的请求**时，要警惕：

```
正常请求：GET /search.php?q=正常搜索词 HTTP/1.1
绕过请求：GET /search.php?q=%2f%2a%2a%2f%75%6e%69%6f%6e%2f%2a%2a%2f%73%65%6c%65%63%74 HTTP/1.1
                                               ↑↑↑ URL解码后是：/**/union/**/select
```

**检测思路**：
1. 对URL参数做URL解码后再分析
2. 统计参数中特殊字符的比例（正常业务参数不会有大比例的%编码或特殊字符）
3. 检查User-Agent（正常浏览器不会手动构造这种请求）
4. 关联同一IP的其他请求，看整体行为模式

## 🎯 蓝队面试高频题（Day 8 主题）

**Q1：如何判断一条告警是扫描行为还是真实入侵？**

> 回答：从三个维度判断。①是否有"成功证据"：扫描通常只有探测行为（端口开放与否、路径是否存在），入侵则有明确的攻击成功痕迹——如SQL注入返回了数据库内容、文件上传返回了200且有新文件落地、暴力破解出现了4624登录成功。②攻击的"深度"：扫描停留在"探"的阶段，入侵则涉及"利用"和"维持"——如果日志中出现了命令执行、权限提升、横向移动等后续动作，一定不是扫描。③Payload的"目的性"：扫描payload通常简短通用（如探测端口或目录是否存在），入侵payload则复杂具体（如完整的恶意上传文件内容、带CSRF token的表单提交）。

**Q2：什么是Kill Chain？在蓝队防守中怎么用？**

> 回答：Kill Chain（杀伤链）是Lockheed Martin提出的网络攻击七阶段模型：侦察→武器化→投递→利用→安装→C2→行动。蓝队的核心用法是"纵深防御"——在每个阶段都设置检测和阻断手段。越早期发现攻击，损失越小。比如在"侦察"阶段就封禁扫描IP，在"投递"阶段拦截钓鱼邮件，在"利用"阶段通过WAF检测攻击payload。面试时我会强调：蓝队的目标不是在某一层做到100%防御，而是在Kill Chain的每一层都增加攻击者的成本和被发现的风险。

**Q3：常见的WAF绕过手法有哪些？蓝队如何检测？**

> 回答：常见绕过手法包括：大小写混写（UnIoN SeLeCt）、注释分割（/**/union/**/select）、URL编码、分块传输、等价函数替换、内联注释等。蓝队的检测思路不是"堵住每一种绕过"，而是建立多层检测：①WAF定期更新规则，支持大小写不敏感和注释去除；②在应用日志层面做二次检测——即使WAF放过了，应用日志里仍然能看出异常；③建立行为基线——正常用户不会发送高度编码的参数、不会在参数中使用SQL关键字；④针对成功的绕过做"特征回填"——一旦发现新的绕过手法，立刻更新检测规则。

## 📖 深度补充内容

### Kill Chain 各阶段的"蓝队干预点"

| 阶段 | 攻击者行为 | 蓝队检测点 | 蓝队阻断点 |
|------|-----------|-----------|-----------|
| ①侦察 | 端口扫描、目录探测 | NIDS/日志分析发现扫描特征 | 封禁扫描IP、限制请求频率 |
| ②武器化 | 制作木马/构造Payload | ❌ 不可检测（攻击者本地进行） | ❌ 无法干预 |
| ③投递 | 发送钓鱼邮件/注入恶意链接 | 邮件网关/Web应用防火墙 | 拦截恶意邮件/WAF阻断 |
| ④利用 | 漏洞触发、代码执行 | WAF/IDS签名匹配 | WAF阻断/虚拟补丁 |
| ⑤安装 | 植入后门/Webshell | 文件完整性监控/进程审计 | 阻断文件写入/隔离主机 |
| ⑥C2 | 反弹Shell/DNS隧道 | 异常出站流量检测/DNS审计 | 封禁出站连接/拦截DNS |
| ⑦行动 | 窃取数据/破坏系统 | DLP/数据流量分析 | 切断网络/终止进程 |

## ⚠️ 新手常见误区纠正

1. **误区**："WAF报警就是被攻击了，WAF没报警就是安全的"
   - **真相**：WAF有漏报（False Negative）——攻击者可能用绕过手法绕过了WAF。永远要在应用日志层面做二次验证
   - **正确做法**：WAF告警+应用日志+系统日志三者交叉验证

2. **误区**："攻击链分析就是高级分析师的事情，初级岗不用管"
   - **真相**：你现在看到的每条P2扫描告警，可能是未来P0入侵事件的前奏。建立"链的思维"让你不会漏掉关键线索
   - **正确做法**：即使你现在只做单条告警研判，也要养成"同一IP还做了什么"的关联思维

3. **误区**："封禁了攻击IP就万事大吉"
   - **真相**：攻击者可以换IP。更重要的是你封禁之前，攻击有没有已经成功过？有没有已经装了后门？
   - **正确做法**：封禁前先排查攻击是否已成功，封禁后持续监控该目标是否有异常

### "如果是你，你怎么防？"

**场景**：你的Web应用每天遭受大量SQL注入攻击，有些绕过了WAF。

- **预防层**：应用层使用参数化查询（Prepared Statement），从根源上杜绝SQL注入；输入做白名单校验
- **检测层**：数据库增加查询审计日志；Web日志对所有含SQL关键字的请求做二次记录
- **响应层**：发现绕过WAF的注入成功后，除了封IP，还要更新WAF规则、分析绕过手法
- **复盘层**：定期做WAF规则有效性测试；用已知绕过手法做"离线测试"，确保WAF生效

## 🛠️ WAF绕过深度实践指南

### 绕过手法检测实战手册

每种绕过手法的具体检测命令和日志特征：

```bash
# 检测1：URL编码绕过
# 正常请求的参数中%比例通常<10%
cat access.log | awk -F'"' '{print $2}' | grep -oP '%[0-9A-Fa-f]{2}' | wc -l
# 如果某请求参数中%比例>30% → 可能在尝试URL编码绕过

# 检测2：注释分割绕过
grep -E '/\*.*\*/' access.log  # 查找含/**/注释的请求
grep -E '/\*![0-9]+.+\*/' access.log  # 查找含内联注释的请求

# 检测3：分块传输绕过
# WAF日志中如果同一IP的连续请求体都很小(1-10字节)
# 且拼接起来符合攻击特征 → 分块传输绕过

# 检测4：大小写混写绕过
# 如果参数中大小写交替出现异常
grep -iE 's(e|E)(l|L)(e|E)(c|C)(t|T)' access.log  # 各种大小写组合的select

# 检测5：双重编码绕过
# 参数中%25开头 → 可能是双重URL编码
grep '%25[0-9a-fA-F][0-9a-fA-F]' access.log

# 检测6：等价函数绕过
# 搜索WAF规则中没有覆盖的等价函数
grep -iE 'benchmark|char\(|hex\(|unhex\(|0x[0-9a-f]+' access.log
```

### 绕过检测的综合判定方法

单靠一条grep命令是不够的。真正的绕过检测需要多维度综合判断：

```
维度1：字符分布异常
  正常参数 → 字母数字为主，少量特殊字符
  编码绕过 → 大量%xx编码或hex字符
  判断方法：计算参数中"异常字符"的比例

维度2：参数结构异常  
  正常参数 → 键值对清晰
  绕过参数 → 拼接碎片、嵌套编码
  判断方法：URL解码后检查是否包含SQL/XSS关键词

维度3：行为关联异常
  正常用户 → 偶尔触发一个WAF规则
  绕过攻击者 → 反复修改参数尝试不同绕过手法
  判断方法：同一IP在一定时间窗口内触发了多种不同类型的WAF规则

维度4：时间模式异常
  正常用户 → 随机时间间隔
  绕过测试 → 连续密集的请求（在尝试不同payload）
  判断方法：计算同一IP的请求间隔分布
```

## 🎬 真实WAF绕过案例复盘

### 案例1：某电商平台的SQL注入绕过全过程

```markdown
背景：某电商搜索接口 /search?keyword=xxx 被WAF保护，规则严格检测SQL关键字。

攻击者的绕过尝试（耗时约45分钟，12次尝试）：

第1次：keyword=' OR 1=1--
  → WAF拦截（检测到 OR 和 1=1）

第2次：keyword='/**/OR/**/1=1--
  → WAF拦截（注释分割没骗过）

第3次：keyword=' oR 1=1#
  → WAF拦截（大小写混写也没骗过）

第4次：keyword='%20OR%201=1--
  → WAF拦截（URL编码同样被检测）

第5次：keyword=1' AND '1'='1
  → 请求通过WAF！但返回500（应用层报错）
  → 关键发现：WAF没有拦截 AND 关键字的组合！

第6次：keyword=1' AND 1=1 AND '1'='1
  → 请求通过WAF！返回200！
  → 攻击者确认：可以用 AND 构造条件！

第7次：keyword=1' AND SLEEP(5) AND '1'='1
  → 请求通过WAF！响应延迟5秒！
  → SQL注入确认！WAF不知道 SLEEP 是攻击函数！

第8次：keyword=1' AND SUBSTRING((SELECT GROUP_CONCAT(table_name) 
        FROM information_schema.tables WHERE table_schema=database()),1,50)='a
  → 请求通过WAF！盲注成功！
  → 攻击者正在逐字节枚举所有表名！

蓝队教训：
  ✅ WAF规则通常覆盖最常见的攻击模式（OR/UNION），
     但不一定能覆盖所有的检测向量（AND盲注）
  ✅ 这就是为什么需要"纵深防御"——
     WAF没拦截的，应用日志+数据库审计日志可以发现
  ✅ 如果第5次尝试产生了500错误，而应用日志中出现了SQL语法错误
     → 即使WAF没拦截，SIEM中的日志关联也能发现异常
```

### 案例2：内网WAF被"跳过"——HTTPS证书钉扎绕过

```markdown
很多企业只在外网入口部署了WAF，内网之间的流量完全不受WAF保护。
这意味着：

场景：
  攻击者通过钓鱼获得了内网一台普通PC的控制权 →
  从这台PC直接访问内网的Web服务器（DB管理后台）→
  没有任何WAF保护！（因为WAF只部署在互联网边界）

攻击者在内网可以随心所欲地：
  - 直接对内网Web应用发动SQL注入（没有WAF拦）
  - 扫描所有内网Web应用的漏洞（没有IDS检测内网扫描）
  - 利用发现的漏洞横向移动

教训：
  ✅ WAF不仅要部署在"外→内"的方向上
  ✅ 关键应用之间"内→内"的流量也需要WAF或RASP保护
  ✅ 零信任的核心思想——不要因为流量是从内网来的就信任
```

## 🧠 WAF绕过的"思维模式"训练

作为一名蓝队人员，理解攻击者的绕过思维比记住具体的绕过手法更重要。以下是攻击者的5种核心思维模式：

```markdown
思维模式1：变换形式（Morphing）
  "我不能用 SELECT，那我用 SeLeCt 行不行？用 \x53\x45\x4C\x45\x43\x54 行不行？"
  → 防御：WAF规则配置不区分大小写，解码后再检测
  
思维模式2：拆散重组（Fragmentation）
  "WAF检测 'UNION SELECT' 这个完整字符串，那我用 UN/**/ION SEL/**/ECT 绕过去"
  → 防御：WAF在检测前先标准化（去注释、去空格）

思维模式3：借道而行（Proxy/Protocol Switching）
  "WAF只检测HTTP流量，那我用WebSocket传payload？用gRPC？"
  → 防御：WAF需要覆盖所有应用层协议

思维模式4：参数污染（Parameter Pollution）
  "WAF检测 ?id=1 UNION SELECT，那我传 ?id=1&id=UNION&id=SELECT"
  → 防御：WAF需要理解后端语言对重复参数的处理逻辑

思维模式5：利用逻辑漏洞（Logic Exploit）
  "WAF检测 SELECT 关键字，但我只需要知道 id 参数存在注入就够了——
   我用 ?id=2-1 测试——如果返回 id=1 的页面，说明注入存在"
  → 防御：不要只靠关键字匹配，要结合行为分析
```

## 🎯 红蓝对抗中的"检测盲区"——攻击者专门利用的5个缝隙

红蓝对抗（护网演练）中，红队专门找蓝队的检测盲区下手。了解这些盲区，你才能堵上它们：

```markdown
【盲区1：非工作时间（Nights & Weekends Gap）】
  为什么是盲区：
    → 值班分析师可能只有1-2人（白天可能有5-8人）
    → 夜间人手不足 → 告警响应速度慢 → 给了红队更多操作窗口
    → 红队最喜欢的攻击时间：周五下午5点（接近下班，周末2天无人发现）
  
  堵上盲区：
    → SOAR自动化处置P1告警（夜间不需要人工确认）
    → 红队如果在非工作时间活动 → 告警级别自动提高一级
    → 建立"周末安全简报"——自动化系统在周五17:00前自动生成当前安全态势

【盲区2：非标准协议和端口（Protocol Gap）】
  为什么是盲区：
    → 安全设备主要检测HTTP/HTTPS/SSH/SMB等标准协议
    → 红队使用DNS隧道、ICMP隧道、自定义TCP协议 → 绕过大部分检测
    → 红队使用高位端口（>10000）→ 不在WAF和IDS的默认检测范围
  
  堵上盲区：
    → 配置NTA（网络流量分析）对所有协议做行为分析，不只是已知协议
    → 关注"流量模式"：周期性心跳+固定大小包=可能是C2，无论什么端口
    → 对DNS查询做频率分析和熵值检测（DGA域名/隧道检测）

【盲区3：加密流量内部（Encryption Gap）】
  为什么是盲区：
    → 约90%的互联网流量已加密（HTTPS），WAF/IDS看不到内容
    → 红队的C2通信全部走HTTPS → 在检测设备看来就是"正常的加密流量"
    → CobaltStrike Beacon的HTTPS通信和普通浏览器的HTTPS通信→对WAF来说没有区别
  
  堵上盲区：
    → JA3/JA3S指纹检测：不同HTTPS客户端的TLS握手指纹不同
    → 流量行为分析：正常HTTPS浏览=间歇性、不规律；C2 Beacon=周期性、有规律
    → 出站HTTPS连接的"目标异常":正常访问是CDN/云服务/CMS等，C2是境外未知VPS

【盲区4：合法工具滥用（Living-off-the-Land Gap）】
  为什么是盲区：
    → 红队使用系统自带工具（PowerShell、WMI、PsExec、certutil）做恶意操作
    → 这些工具在正常运维中也大量使用 → 签名白名单失效
    → EDR不能直接阻拦合法的签名工具 → 需要行为分析而非文件签名
    → 案例：用certutil下载恶意文件 → certutil是Windows自带的合法工具
  
  堵上盲区：
    → 关注合法工具的"异常使用上下文"
      正常：管理员在域控上执行PowerShell查看AD用户 → 正常
      异常：Web服务器的www-data用户执行PowerShell → 异常！
    → 关注进程链：是谁启动了这些合法工具？
      explorer.exe→powershell.exe = 正常（用户手动打开）
      w3wp.exe→cmd.exe→powershell.exe = 异常（Web进程不应该启动shell）

【盲区5：云服务和SaaS的API盲区（Cloud API Gap）】
  为什么是盲区：
    → 传统安全设备监控"网络流量"，但云API调走的是HTTPS
    → 红队如果通过OAuth令牌访问你的云资源→看起来就是正常的API调用
    → 你在AWS/GCP/Azure上的安全日志可能不在主SIEM的监控范围内
  
  堵上盲区：
    → 把云平台的CloudTrail/AuditLog接入SIEM
    → 为云API的异常行为建检测规则（异地登录、非工作时间操作、批量导出）
    → 云资源的配置变更（如S3桶公开、安全组开放）→ 必须实时告警
```
