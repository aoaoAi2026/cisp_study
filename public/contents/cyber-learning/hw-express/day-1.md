## 📖 核心知识讲解

### 一、护网行动到底是什么？——一场国家级的「网络消防演习」

先来一个比喻，保证你一辈子忘不了：

> 🧯 每年消防部门都会搞大型消防演习。假设市中心某栋大楼「失火」了，110、119、120 全部出动，检验各部门之间的协同应急响应能力。演习结束后，全体复盘：哪个环节响应慢了？谁该到却没有到？应急预案哪里有漏洞？下一年怎么改进？

**护网行动就是网络安全界的消防演习。** 只不过：

- 「着火的大楼」变成了「政府网站、银行交易系统、电力调度系统」
- 「消防员」变成了「网络安全工程师」——也就是你
- 「灭火」变成了「发现攻击、拦截入侵、应急响应」

**护网行动的核心事实清单：**

| 维度 | 真实情况 |
|:---|:---|
| **谁组织的？** | 国家公安部、国家网信办等主管部门牵头组织 |
| **谁参加？** | 政府部门、央企、金融机构、大型互联网企业、关键信息基础设施运营单位 |
| **多长时间？** | 通常 2-4 周，7×24 小时不间断 |
| **怎么打？** | 红队在真实规则下发起实战攻击（不事先通知蓝队具体时间点），蓝队进行真实防守 |
| **输了会怎样？** | 全行业通报、扣分、直接影响该单位的网络安全年度考核排名 |
| **赢了有什么？** | 荣誉、排名、证明自己单位的安全体系真正经得起考验 |
| **打分标准** | 红队拿到靶标系统的控制权 → 蓝队失分；蓝队及时发现并有效拦截 → 蓝队得分 |

**护网行动参与的行业覆盖（仅供参考，非官方名单）：**

| 行业 | 典型防守对象 | 蓝队关注重点 |
|:---|:---|:---|
| 政府 | 政务服务平台、公文系统 | 数据泄露、篡改 |
| 金融 | 网银系统、支付网关 | 资金安全、交易篡改 |
| 能源 | 电力调度系统、油气管道控制 | 生产安全、物理破坏 |
| 通信 | 运营商核心网、短信网关 | 通信中断、数据窃听 |
| 医疗 | HIS系统、电子病历 | 患者隐私、诊疗数据 |
| 教育 | 招生系统、学籍管理 | 成绩篡改、数据泄露 |

**护网行动的根本目的**——不是为了争输赢，而是为了四个目标：

1. **暴露问题**：你平时觉得「应该没问题」的防火墙策略、口令强度、日志覆盖度，在真实攻击下到底能不能扛住？
2. **锻炼队伍**：安全团队到底有没有真正的实战经验？能不能在规定时间内从海量告警中发现真正的攻击？
3. **推动建设**：护网中暴露的每一个短板，就是下一步安全建设投资的优先级清单
4. **形成威慑**：对外展示国家关键基础设施的网络安全防护能力，让潜在攻击者知难而退

> 💡 **一句话记住**：护网是网络安全行业的「高考」——一年一次，考的是真实防御能力，不是书面上的「我们有防火墙」「我们有IPS」。

### 二、红队 🆚 蓝队 🆚 紫队——三支队伍的精确分工

护网行动中的队伍划分非常明确，每支队伍都有特定的使命：

| 队伍 | 角色定位 | 通俗比喻 | 核心任务 | 你的位置 |
|:---|:---|:---|:---|:---|
| 🔴 **红队** | 攻击方 | 假扮的盗贼团伙 | 模拟APT组织，使用真实攻击手法，想方设法突破防线、拿下靶标系统 | 不是（你防） |
| 🔵 **蓝队** | 防守方 | 保安+监控中心侦察组 | 监控全局态势，发现攻击行为、研判每条告警、拦截入侵、应急处置、溯源加固 | **就是你了！** |
| 🟣 **紫队** | 裁判/协调组 | 比赛裁判长+导演 | 制定比赛规则、监督攻防流程、判定得分失分、协调红蓝矛盾、确保公平 | 不是 |

**蓝队的四项基本功——一个口诀记住：**

```
眼 → 监控（看大屏、收告警，时刻感知全局）
脑 → 研判（分析每一条告警是真攻击还是误报，判断威胁等级）
手 → 处置（封IP、杀进程、隔离主机、下发策略，立刻行动）
笔 → 记录（日报、处置工单、事件报告，没有记录等于没有发生）
```

**蓝队核心职责的详细拆解：**

### 三、蓝队核心日常工作——五类任务逐一拆解

#### 工作一：告警监控与研判（占工作时长 50%）

这是蓝队**最核心、最日常**的工作。告警来源有五个渠道：

- 防火墙日志中的异常连接请求（如非工作时间的大量外连）
- WAF（Web应用防火墙）拦截的可疑Web请求（如SQL注入payload）
- IDS/IPS（入侵检测/防御系统）匹配的攻击特征（如端口扫描签名）
- 终端EDR（端点检测与响应）发现的异常行为（如powershell下载执行）
- 威胁情报平台推送的IOC（入侵指标）匹配告警（如发现C2通信）

**研判三步法——蓝队分析师的"肌肉记忆"：**

```
第一步：看 —— 这条告警在说什么？
├─ 来源IP是谁？（外部攻击者？内部机器中招后的C2回连？）
├─ 目标是什么？（Web服务器？数据库服务器？域控制器？）
├─ 攻击类型是什么？（SQL注入？暴力破解？端口扫描？Webshell连接？）
├─ 时间点异常吗？（凌晨3点有人登录OA？周末有大量数据传输？）

第二步：判 —— 这是真攻击还是误报？
├─ 误报（占70-80%）：正常业务行为被安全设备误判为攻击
│  └→ 动作：标记关闭，必要时调整告警规则减少误报
├─ 试探（占15-20%）：攻击者在扫描探测，但尚未成功突破防线
│  └→ 动作：封禁源IP、记录到威胁情报库、纳入重点监控
├─ 入侵（占1-5%）：攻击已经成功打入系统内部！🆘
│  └→ 动作：立即升级给组长、启动应急响应流程！

第三步：处 —— 根据研判结果执行对应的处置动作
├─ 封禁IP：在WAF/防火墙上添加封禁规则
├─ 隔离主机：将被入侵的主机从网络中断开
├─ 清除文件：删除Webshell后门文件
├─ 修改策略：增加防护规则、加强认证策略
└─ 升级上报：重大事件通报值班组长和管理层
```

> ⏱️ **黄金15分钟法则**：护网期间，每一条告警从产生到完成研判处置，规定时间不超过15分钟。超时 = 失分。

#### 工作二：日志分析（占 20%）

日志是蓝队的"监控录像"——你必须会看：

| 你在找什么 | 日志中的特征 | 常用命令示例 |
|:---|:---|:---|
| 有人在扫目录？ | 同一个IP短时间内大量404状态码 | `grep "404" access.log \| awk '{print $1}' \| sort \| uniq -c \| sort -rn` |
| SQL注入攻击？ | 请求参数中有 `' OR`、`UNION SELECT` 等SQL关键字 | `grep -iE "(union.*select\|or.*1=1\|'--)" access.log` |
| 暴力破解登录？ | 同一IP短时间内大量POST到登录接口 | `grep "POST /login" access.log \| awk '{print $1}' \| sort \| uniq -c \| sort -rn` |
| 后门连接？ | 异常时间段的非常规URL访问 | `grep "03:" access.log \| grep -vE "(js$\|css$\|png$)"` |

#### 工作三：安全设备运维（占 10%）

日常要保证四件事：
- ✅ 防火墙规则正确且生效
- ✅ WAF检测引擎正常运行，规则库已更新
- ✅ IDS/IPS特征库是最新版本
- ✅ SIEM平台所有日志源都在正常上报

#### 工作四：事件响应与应急处置（占 10%）

当研判确认发生了真实入侵，立刻切换到「应急模式」——PDCERF六步法（Day5详解）：
- **P**repare（准备）→ **D**etect（检测）→ **C**ontain（遏制）→ **E**radicate（根除）→ **R**ecover（恢复）→ **F**ollow-up（复盘）

#### 工作五：报告与文档（占 10%）

> ⚠️ **"没有记录 = 没有发生"** ——这是蓝队的铁律。

| 文档类型 | 内容要点 | 频率 |
|:---|:---|:---|
| 值守日报 | 当日告警总数、处置数、封禁IP数、重大事件摘要 | 每日交班前 |
| 事件处置工单 | 每条真实告警的完整研判+处置过程记录 | 每次处置 |
| 事件分析报告 | 对重大事件的深度分析（攻击链、影响范围、改进建议） | 重大事件后 |
| 态势周报 | 一周攻击趋势、TOP攻击类型、防守措施效果评估 | 每周 |

### 四、TCP/IP协议栈——网络世界的「高速公路+快递体系」

如果把一次网络通信比喻为「你在网上下单买了一双鞋」：

```
┌─────────────────────────────────────────────────────┐
│  OSI参考模型    TCP/IP模型      网购比喻             │
├─────────────────────────────────────────────────────┤
│  7 应用层  ┐                                       │
│  6 表示层  ├── 应用层     "我要买一双42码白色运动鞋" │
│  5 会话层  ┘             （HTTP/HTTPS/DNS/FTP）     │
├─────────────────────────────────────────────────────┤
│  4 传输层  ─── 传输层     "顺丰快递，单号SF12345678"│
│                          （TCP：可靠快递/UDP：普通信）│
├─────────────────────────────────────────────────────┤
│  3 网络层  ─── 网络层     "发往广东省深圳市南山区"  │
│                          （IP地址：门牌号/路由：导航）│
├─────────────────────────────────────────────────────┤
│  2 数据链路层── 网络接口层  "装进快递车，走京港澳高速"│
│  1 物理层  ──            （以太网/WiFi/光纤/网线）   │
└─────────────────────────────────────────────────────┘
```

**每层通俗解释：**

| 层级 | 核心问题 | 代表协议 | 蓝队关注点 |
|:---|:---|:---|:---|
| 应用层 | "你要做什么？" | HTTP、HTTPS、DNS、FTP、SSH | 攻击payload在这层！SQL注入、XSS都在HTTP请求中 |
| 传输层 | "怎么保证送到？" | TCP（可靠）、UDP（快速） | 三次握手异常（SYN Flood攻击）、异常端口连接 |
| 网络层 | "往哪儿送？" | IP、ICMP、ARP | 恶意IP、DDoS流量、ARP欺骗 |
| 链路层 | "走哪条路？用哪辆车？" | 以太网、WiFi | MAC地址欺骗、无线攻击 |

#### TCP三次握手——建立连接的「打电话」

```
客户端（你）                        服务器（朋友）
    │                                    │
    │── SYN (Seq=x) ───────────────────▶│  ① "喂？能听到我说话吗？"
    │   标志位：SYN=1, ACK=0              │     （发送同步请求）
    │                                    │
    │◀─ SYN+ACK (Seq=y, Ack=x+1) ───────│  ② "能听到！你能听到我吗？"
    │   标志位：SYN=1, ACK=1              │     （确认收到+反向同步）
    │                                    │
    │── ACK (Ack=y+1) ─────────────────▶│  ③ "能听到！我们开始聊天吧！"
    │   标志位：SYN=0, ACK=1              │     （确认连接）
    │                                    │
    │═══════ TCP连接建立，开始传输数据 ══════│
```

**Wireshark抓包过滤器——如何在真实流量中找到三次握手：**

```bash
# 只看SYN包（第一次握手）
tcp.flags.syn == 1 && tcp.flags.ack == 0

# 只看SYN+ACK包（第二次握手）
tcp.flags.syn == 1 && tcp.flags.ack == 1

# 只看三次握手的三个包（完整握手）
tcp.flags.syn == 1 || (tcp.seq == 1 && tcp.ack == 1 && tcp.len == 0)
```

#### TCP四次挥手——断开连接的「礼貌告别」

```
客户端                              服务器
   │                                   │
   │── FIN ──────────────────────────▶│  ① "我说完了，准备挂电话了"
   │                                   │
   │◀── ACK ──────────────────────────│  ② "好的，我知道你说完了"
   │                                   │     （但服务器可能还有数据要发）
   │                                   │
   │◀── FIN ──────────────────────────│  ③ "我也说完了，挂电话吧"
   │                                   │
   │── ACK ──────────────────────────▶│  ④ "好的，再见！"
   │                                   │
   │═══════ TCP连接完全关闭 ════════════│
```

> 💡 **为什么挥手比握手多一次？** 因为TCP是全双工的（双方都可以同时发数据）。当客户端说"我说完了"只代表客户端不发数据了，但服务器可能还有数据要发。所以需要服务器单独说"我也说完了"才算彻底结束。

#### 常用协议端口速查表（蓝队必备）

| 端口 | 协议 | 服务 | 蓝队安全关注点 |
|:---|:---|:---|:---|
| 20/21 | FTP | 文件传输 | 匿名登录、明文密码传输 |
| 22 | SSH | 安全远程管理 | 暴力破解高频目标！弱口令 |
| 23 | Telnet | 远程登录（明文） | 🚫 高危！必须禁止使用 |
| 25 | SMTP | 邮件发送 | 垃圾邮件中继、钓鱼邮件 |
| 53 | DNS | 域名解析 | DNS隧道（隐蔽C2通信）、DNS劫持 |
| 80 | HTTP | Web服务（明文） | SQL注入、XSS等所有Web攻击 |

## 🎯 蓝队面试高频题（Day 1 主题）

**Q1：请用一句话概括护网行动是什么？**
> 护网行动是由公安部等主管部门组织的国家级网络安全实战演习，红队以真实攻击手法模拟APT组织渗透，蓝队负责监控、检测、响应和加固，紫队担任裁判，旨在全面检验和提升关键信息基础设施的真实防御能力。

**Q2：蓝队和红队的核心区别是什么？红队厉害还是蓝队厉害？**
> 红队的核心目标是"找到一条路攻进来"——发现漏洞并利用。蓝队的核心目标是"封住所有的路不让攻进来"——监控、检测、响应、加固整个攻击面。没有谁更厉害，这是两种截然不同的思维模式。优秀的蓝队工程师既需要防守思维（纵深防御），也需要懂攻击（知道对手会怎么打，才能提前布防）。

**Q3：什么是误报和漏报？为什么说漏报比误报更危险，但误报太多也是一种致命问题？**
> 误报（False Positive）：正常行为被判定为攻击——"狼来了"太多次。漏报（False Negative）：真实攻击没有被检测到——"狼真的来了但警报没响"。漏报更危险因为攻击者在眼皮底下活动而你完全不知情。但误报太多会导致"告警疲劳"——蓝队分析师对告警变得麻木后，真实攻击被忽略的概率会直线上升。优秀的安全运营要在两者之间找到黄金平衡。

**Q4：TCP三次握手的具体过程是什么？如果第三次ACK包丢失会发生什么？**
> 第一次：客户端发送SYN（Seq=x）→ 服务器。第二次：服务器回复SYN+ACK（Seq=y, Ack=x+1）→ 客户端。第三次：客户端回复ACK（Ack=y+1）→ 服务器，连接建立。如果第三次ACK丢失，服务器会认为自己的SYN+ACK没被收到，会重传SYN+ACK（默认重试5次，间隔时间递增）。如果始终收不到ACK，服务器最终会放弃并释放半连接资源。这也是SYN Flood攻击的原理——攻击者只发SYN不完成握手，耗尽服务器的半连接队列。

**Q5：SOC（安全运营中心）的核心职能是什么？一个标准的SOC团队有哪些角色？**
> SOC的核心职能：7×24实时安全监控、告警研判与分类、安全事件响应与处置、威胁情报消费与运营、安全态势感知与报告。标准SOC团队通常包含：一线分析师（Tier 1，告警初筛）、二线分析师（Tier 2，深度调查）、三线分析师（Tier 3，高级威胁狩猎与取证）、SOC经理（流程与人员管理）、安全架构师（策略与工具链设计）。

### 告警分类的四象限法则

蓝队分析师在处理告警时，可以用这个四象限快速分类：

```
                高影响
                  │
      ┌───────────┼───────────┐
      │           │           │
      │  B区     │  A区      │
      │ 重要不紧急│ 紧急且重要 │
      │           │           │
低概率 ──────────┼────────── 高概率
      │           │           │
      │  D区     │  C区      │
      │ 可忽略   │ 紧急不重要│
      │           │           │
      └───────────┼───────────┘
                  │
                低影响
```

- **A区（紧急且重要）**：如确认的Webshell上传、数据外泄 → 立即启动应急响应
- **B区（重要不紧急）**：如安全策略待优化、漏洞待修复 → 排入计划，限时完成
- **C区（紧急不重要）**：如大量低危扫描告警 → 批量处理，调用标准SOP
- **D区（可忽略）**：如已被拦截的无效探测 → 标记关闭即可

## ⚠️ 新手常见误区纠正

1. **误区**："护网就是来抓黑客的，每天都很刺激很燃"
   - **真相**：80%的时间是在看告警+研判误报，15%的时间在写报告填工单，只有5%的时间在真正处置攻击。但这关键的5%需要你用95%的日常积累来应对。
   - **正确心态**：耐心是蓝队的第一美德。能坐得住冷板凳，才能在关键时刻不掉链子。

2. **误区**："蓝队不需要懂攻击，把设备配好就能守住"
   - **真相**：不懂攻击原理，你连告警都看不懂。你会看到一条"UNION SELECT"告警，但不知道这意味着什么、攻击者下一步会做什么。蓝队 = 70%防守技能 + 30%攻击理解。
   - **正确心态**：知己知彼，方能百战不殆。了解攻击者是蓝队的基本素养。

3. **误区**："HTTPS就是加密版的HTTP，所以HTTPS网站不会有安全问题"
   - **真相**：HTTPS只加密传输过程（防止中间人窃听），但不防护应用层攻击。攻击者完全可以合法使用HTTPS发送SQL注入payload——加密隧道里跑的攻击数据，外面是看不到的。
   - **正确心态**：HTTPS保护的是传输安全，不是应用安全。两者是完全不同的维度。

4. **误区**："我学完28天就能独立参加护网了"
   - **真相**：28天让你具备初级值守上岗能力——能看懂告警、会用工具、能按照SOP执行标准处置。但真正的护网战场远比课程复杂，你会遇到各种意想不到的攻击手法。
   - **正确心态**：28天是高质量起步，不是终点。学习是永远在路上的。

5. **误区**："漏报无所谓，反正不是我的责任"
   - **真相**：如果攻击者通过你漏掉的告警打进了系统，整条防线都会受影响。蓝队是一个团队，每个人的疏漏都可能成为整条防线的破绽。
   - **正确心态**：每一名蓝队成员都是最后一道防线。

### "如果是你，你怎么防？" 四层思考

**场景：你发现有一个外网IP正在对你单位的OA系统进行信息收集探测**

| 层面 | 思考与动作 |
|:---|:---|
| 🛡️ **预防层** | 这个IP是怎么绕过边界防火墙的？防火墙策略是否需要收紧？有无CDN/WAF做第一层防护？ |

| 🚨 **响应层** | 如何快速封禁这个IP？封禁后需要通知哪些人？是否需要在全网排查该IP的历史访问记录？ |

## 🔰 蓝队新人的"生存法则"——前30天最重要的事

如果你是第一天做护网蓝队，以下不是"建议"，是"生存法则"：

```
法则1：搞清"谁是谁"——建立联系人网络（第一天就要做）
  安全主管（你的直属上级）：________ 电话：________
  网络运维值班：________ 电话：________
  系统运维值班：________ 电话：________
  应用开发值班：________ 电话：________
  云服务商技术支持：________ 电话：________
  运营商网络安全：________ 电话：________
  
  为什么要第一天就做这个？
  → 凌晨3点告警时，你不会想用30分钟去找"防火墙是谁管的"
  → 把上面的空格填完，存到手机通讯录里，设置"安全-紧急"标签

法则2：搞清楚"哪是哪"——资产认知（第一周内完成）
  你的防护范围包括哪些？
  - 公网资产：哪些IP/域名对外提供服务？
  - 核心系统：数据库、支付系统、域控、文件服务器在哪？
  - 安全设备：WAF/防火墙/IDS部署在什么位置？
  - "皇冠资产"（Crown Jewel）：如果只能保护一个系统，保护哪个？
  
  关键问题：如果有人问你"你们公司最重要的数据在哪？"，你能回答吗？

法则3：搞清楚"什么是正常"——建立基线
  新人最容易犯的错误是"把所有东西都当异常"。
  → 你得先知道"凌晨3点运维巡检登录服务器是正常的"（因为有夜班运维）
  → 你才能辨别"凌晨3点一个从未出现过的IP登录服务器是异常的"
  
  建立基线的方法：
  ① 花1小时看最近3天的SIEM日志（只看不处理）
  ② 标记出"规律性的行为"（每天凌晨2点的备份任务、每小时的心跳监控）
  ③ 以后看到重复事件时对照基线判断

法则4：新人不要怕"问"
  老员工最怕的不是"新人问问题"，而是"新人犯错不吭声"。
  
  该问的：
  ① "这个告警我判断是误报，但我不确定，你能帮我看一下吗？"
  ② "我把这个IP封了，但业务好像出了点问题，我该怎么回滚？"
  ③ "这个攻击手法我没见过，该从哪个方向排查？"
  
  不该问的（自己能查到的）：
  ① "SQL注入是什么？" → 去看第一周的笔记
  ② "iptables命令怎么写？" → man iptables 或查速查表
  ③ "微步在线的网址是什么？" → 收藏夹里有

法则5：学会"分层汇报"
  向上汇报是一门技术活，护网中尤其重要：
  
  第一句话（结论先行）："我发现了一个情况，判断为疑似入侵"
  第二句话（基本信息）："时间X，IP是Y，目标是Z，现象是A/B/C"
  第三句话（当前状态）："我已经做了D和E，正在做F，需要你帮忙G"
  第四句话（预期结果）："预计H分钟后会有更明确的结论，到时候再同步"
  
  千万不要这样汇报：
  ❌ "额，是这样的，我刚才在看日志，然后发现有个IP，它好像有点问题，
      它访问了那个，就是那个目录，然后返回了200，我也不确定是不是有问题，
      你觉得呢？"  → 领导听到这里已经失去耐心了
```

## 🗣️ 护网中的"人与人的协作"——沟通是最好的安全工具

护网中最容易被忽视但也最容易出问题的，不是技术，是沟通。

**场景1：你发现了一个可能的入侵，需要通知运维重启服务器**
```
❌ 坏沟通："那个，web服务器好像有问题，你能重启一下吗？"
   → 运维不知道重启哪台、不知道什么原因、不知道风险
    
✅ 好沟通："Web-03（10.0.1.103）确认被上传Webshell到/var/www/uploads/，
   攻击者执行了whoami和wget命令，我需要你帮忙做两件事：
   ① 先把这台机器从负载均衡中摘掉（不要影响线上用户）
   ② 等我做了内存取证后，配合重启清理
   预计需要15分钟，有变化我随时同步。"
   → 运维知道自己要做什么、为什么做、什么时候做
```

**场景2：你需要在凌晨3点拨通安全主管的电话**
```
❌ "喂，老板，出事了…（沉默5秒）…好像有webshell…我不确定…"
   → 主管半梦半醒，不知道事态严重程度

✅ "老板抱歉打扰，Web-03服务器确认被getshell，我已经做了网络隔离和取证， 
   目前没发现横向移动，但需要你帮忙确认是否需要扩大排查范围。
   具体情况我已经发到企业微信了，你可以先看一下再决定。"
   → 主管立刻知道：事情已经控制住了、需要做什么决策、信息已经备好
```

**沟通三原则速记：**
1. **先说结论，再说细节**——不要让人猜你想表达什么
2. **给出选项，不是只抛出问题**——"情况是这样的，我们可以A或B，我建议A，你觉得呢？"
3. **告知进展，主动同步**——不要等别人来问"怎么样了"，主动报告"进展到哪一步了"

  学习方式：找一个人（学伴/导师）给你做模拟面试
