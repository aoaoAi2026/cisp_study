## 📖 核心知识讲解

### 一、域环境是什么？——"把1000台电脑管成1台"的管理体系

#### 1.1 先用一个比喻彻底搞懂"域"

> 🏢 Windows域 = 一栋大型写字楼的中央门禁系统

```
没有域的环境（Workgroup）：           有域的环境（Domain）：
┌─────────────────────┐              ┌──────────────────────────────┐
│ 每台电脑自己管自己    │              │         域控制器 (DC)          │
│                      │              │         ┌─────────┐          │
│ PC1: 账号张三 密码123│              │         │ AD数据库 │          │
│ PC2: 账号张三 密码456│              │         │ (所有账号│          │
│ PC3: 没有张三的账号   │              │         │  和权限) │          │
│                      │              │         └────┬────┘          │
│ 问题：               │              │              │               │
│ · 每台机器单独管理    │              │    ┌─────────┼─────────┐     │
│ · 密码不同容易忘      │              │    ▼         ▼         ▼     │
│ · 权限无法统一控制    │              │  PC1      PC2      PC3      │
│ · 出了事说不清谁干的   │              │  ↑ 统一身份认证、统一策略 ↓  │
└─────────────────────┘              │   张三登录任何一台都是同一个密码│
                                     │   管理员可远程管理所有机器     │
                                     │   所有操作日志汇总到DC        │
                                     └──────────────────────────────┘
```

**域的核心价值：**
1. **统一身份认证**：一个账号，全网通用（Single Sign-On）
2. **集中权限管理**：在DC上改一个设置，1000台电脑同时生效
3. **统一安全策略**：密码复杂度、锁屏时间、禁止USB——一条GPO搞定
4. **集中审计**：谁登录了哪台机器、做了什么操作——全部记录在DC

### 二、域内7条黄金攻击链——攻击者控域的"七种武器"

> ⚠️ 蓝队学攻击是为了防守——知道敌人怎么打，才能建好防线。

#### 攻击链全景图

```
攻击目标：从一台普通域成员机器 → 最终拿到域管理员权限

┌──────────────────────────────────────────────────────────────┐
│          域内攻击的7条黄金攻击链（按难度从低到高）              │
├──────────────────────────────────────────────────────────────┤
│                                                                │
│  链1：LLMNR/NBNS投毒 → 拿到NetNTLM哈希 → 离线破解 → 横向移动  │
│  链2：Kerberoasting → 请求服务票据 → 离线破解 → 横向移动      │
│  链3：AS-REP Roasting → 不需要Kerberos预认证 → 破解 → 横移    │
│  链4：Pass-the-Hash → 拿到本地管理员哈希 → 横向到其他机器      │
│  链5：Pass-the-Ticket → 拿到内存中的票据 → 假冒用户身份        │
│  链6：DCSync → 伪装成DC → 请求所有用户哈希 → 域控完全沦陷     │
│  链7：GPO滥用 → 修改组策略 → 所有域机器执行恶意命令            │
│                                                                │
└──────────────────────────────────────────────────────────────┘
```

#### 链1：LLMNR/NBNS投毒（最简单、最常被忽视）

```
攻击原理：
当你在资源管理器输入 \\fileserver 时，Windows会尝试解析这个名称。
解析顺序：DNS → LLMNR(广播) → NBNS(广播)
如果DNS没有这条记录，Windows会向局域网广播"谁是fileserver？"
攻击者监听广播，抢在真实服务器之前回复"我就是！请给我你的密码哈希来认证！"
→ 攻击者获得你的NetNTLMv2哈希 → 离线破解或中继攻击

蓝队防御：
1. 在域控上禁用LLMNR（组策略 → 计算机配置 → 管理模板 → 网络 → DNS客户端 → 关闭多播名称解析）✓
2. 禁用NBNS（禁用NetBIOS over TCP/IP）✓
3. 确保所有内部服务器都在DNS中有A记录 ✓
4. 部署WPAD的DNS记录（防止WPAD协议同样被投毒）✓
```

#### 链2：Kerberoasting（中等难度，面试高频）

```
攻击原理：
1. 攻击者拿到一个普通域用户的shell
2. 执行：请求该用户能访问的所有服务票据（ST）
3. 服务票据中包含了服务账户密码的哈希
4. 服务票据离线下载 → 用hashcat/john破解 → 明文密码
5. 服务账户通常权限很高（如SQL服务账户属于Domain Admins）
6. 拿到服务账户密码 → 横向到DC → 最终控域

蓝队防御：
- 服务账户使用超长强密码（25+字符，随机生成）——让破解成本无限高
- 使用组托管服务账户（gMSA）——密码由AD自动管理，定期轮换，人不知道密码
- 定期审计：哪些服务账户密码超过90天没改？
- 监控事件ID 4769（Kerberos服务票据请求）的异常频率
```

#### 链4：Pass-the-Hash（PTH，最经典的内网横移手段）

```
攻击原理：
1. 攻击者在机器A上拿到本地管理员权限
2. 用Mimikatz导出内存中的NTLM哈希：
   mimikatz # sekurlsa::logonpasswords
   可以拿到所有登录过机器A的用户的NTLM哈希
3. 如果机器B的本地管理员密码和机器A相同（常见于没部署LAPS的环境）
4. 用机器A的本地管理员哈希去认证机器B：
   mimikatz # sekurlsa::pth /user:Administrator /domain:WORKGROUP /ntlm:[HASH]
5. 不需要知道明文密码，只需要哈希！
6. 这样就可以从A跳到B、B跳到C……直到跳到DC

蓝队防御：
- LAPS（Local Administrator Password Solution）——每台机器的本地管理员密码是随机且唯一的
- 限制本地管理员的使用范围——只有应急情况下才用
- 监控事件ID 4624（登录类型3：网络登录），关注异常的登录模式
- 使用Credential Guard（Windows 10/Server 2016+）保护LSASS中的凭证
```

#### 链6：DCSync（最危险，域控级别权限）

```
攻击原理：
DCSync不是"攻击DC"，而是"伪装成DC"向其他DC请求同步数据。
攻击者只要在域中有"复制目录更改"权限，就能伪造成一个DC，
向真正的DC发送DCSync请求，获取任意用户的密码哈希（包括域管理员）。

mimikatz # lsadump::dcsync /domain:corp.local /user:Administrator
→ 返回域管理员的NTLM哈希（不需要接触DC！不需要在DC上执行！）

蓝队防御：
- 严格权限管理：只有Domain Admins和Enterprise Admins有"复制目录更改"权限
- 监控事件ID 4662（目录服务访问），关注DCSync相关的GUID
- 监控谁有"复制目录更改"权限 — 这个权限不应该给任何人除了域管组本身
- 网络分段：DC在独立的管理网段，普通服务器访问不到
```

### 三、域控5大加固措施

#### 加固1：补丁管理（攻击链的前置条件防御）

```
域控为什么需要严格的补丁管理？
1. 域控承载了整个域的认证体系，域控被打穿 = 整个公司的IT基础崩塌
2. 很多提权漏洞（如Zerologon CVE-2020-1472）可以直接让攻击者成为域管
3. 一个打了补丁的DC vs 一个有Zerologon漏洞的DC：前者需要30天攻破，后者30分钟

补丁管理的最佳实践（面试答案）：

补丁节奏：
- 关键安全补丁：微软发布后48小时内评估，72小时内部署 ✓
- 常规安全补丁：月度补丁星期二后一周内部署
- 功能更新：在测试环境验证2周后再部署

补丁测试流程：
1. 在测试DC上部署补丁（不要在生产DC上直接打）
2. 验证：域认证正常？组策略应用正常？DNS解析正常？复制正常？
3. 测试通过→在生产DC上分批部署（先部署一台，观察24小时再部署其他的）
4. ⚠️ 绝对禁止：在生产DC上直接打补丁不测试

面试加分点：
"我们使用WSUS或SCCM来统一管理补丁部署，DC和关键服务器放在单独的补丁组，
审批流程：安全分析师评估 → 变更管理委员会批准 → 窗口期部署 → 回滚预案就绪。"
```

#### 加固2：凭证保护（攻击链的核心阻断点）

```
凭证保护 = 防止攻击者获取和利用密码/哈希

┌─────────────────────────────────────────────────────────────┐
│                    凭证保护的分层防御                          │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│  第一层：密码策略                                              │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ - 密码长度≥14位（不只是8位！）                          │  │
│  │ - 密码复杂度：大小写+数字+特殊字符                       │  │
│  │ - 密码过期：90天更换（服务账户例外，用gMSA）             │  │
│  │ - 密码历史：记住最近24个密码（防止用户循环使用）         │  │
│  │ - 账户锁定：5次错误锁定30分钟                           │  │
│  │ 组策略路径：计算机配置 → Windows设置 → 安全设置         │  │
│  │            → 账户策略 → 密码策略                        │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                               │
│  第二层：LAPS（本地管理员密码管理）                              │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 没有LAPS：100台机器的本地Administrator密码都一样       │  │
│  │ → 拿下1台 = 拿下100台                                   │  │
│  │                                                          │  │
│  │ 有了LAPS：每台机器的本地Administrator密码随机+唯一      │  │
│  │ → 密码存在AD中（ms-MCS-AdmPwd属性）                     │  │
│  │ → 密码每30天自动轮换                                    │  │
│  │ → 只有授权管理员能查看                                  │  │
│  │ → Pass-the-Hash直接被废掉                               │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                               │
│  第三层：Credential Guard（凭证守卫）                           │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 问题：Mimikatz可以直接从LSASS进程读取密码哈希          │  │
│  │ 解决：Credential Guard把凭证放在虚拟化安全模式(VSM)中   │  │
│  │ → 即使拿到SYSTEM权限也读不到哈希                        │  │
│  │ → Mimikatz等工具无法获取凭证                            │  │
│  │ → 限制：需要Windows 10 Enterprise或Server 2016+         │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                               │
│  第四层：Protected Users组                                      │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 把高权限用户加入Protected Users组后：                    │  │
│  │ - 只能使用Kerberos认证，不能用NTLM                      │  │
│  │ - 不缓存明文密码（WDigest被禁用）                       │  │
│  │ - 不缓存Kerberos长期密钥                                │  │
│  │ - TGT生命周期从10小时降到4小时                           │  │
│  │ → Pass-the-Hash和Pass-the-Ticket的生存时间大幅缩短     │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                               │
└─────────────────────────────────────────────────────────────┘
```

#### 加固3：GPO安全基线（组策略全员加固）

```powershell
# 域控核心GPO安全基线检查清单

# 1. 禁用LM哈希存储（防止哈希被轻易破解）
# GPO: 计算机配置 → Windows设置 → 安全设置 → 本地策略 → 安全选项
# "网络安全: 在下一次更改密码时不存储LAN Manager哈希值" → 已启用

# 2. 强制SMB签名（防止SMB中继攻击）

# 3. 禁用匿名枚举（防止列出域用户）
# "网络访问: 不允许匿名SAM账户和共享的枚举" → 已启用

# 4. UAC最高级别（用户账户控制，提权不能静默）
# "用户账户控制: 以管理员批准模式运行所有管理员" → 已启用

# 5. 禁用PowerShell v2（老版本没有日志记录功能）
# Windows组件 → Windows PowerShell → 禁用

# → 启用脚本块日志记录

#### 加固4：管理跳板机（PAW - Privileged Access Workstation）

```
PAW = 专用安全管理工作站，只用于管理域控和关键服务器

核心原则：管理凭据绝不触碰非管理设备

┌──────────────────────────────────────────────────────────────┐
│                    三层设备分离策略                             │
├──────────────────────────────────────────────────────────────┤
│                                                                │
│  日常办公电脑（Normal Workstation）                             │
│  ├─ 收发邮件、浏览网页、写文档、上外网                          │
│  ├─ 最容易中招（钓鱼邮件、恶意网站、宏病毒）                     │
│  └─ ⚠️ 绝对不能用这台电脑管理DC！                               │
│                                                                │
│  安全运维电脑（SAW - Secure Admin Workstation）                  │
│  ├─ 可以访问跳板机和DevOps平台                                  │
│  ├─ 不能访问DC！不能访问生产数据库！                             │
│  └─ 安全级别：中级                                              │
│                                                                │
│  专用管理电脑（PAW - Privileged Access Workstation）             │
│  ├─ 只能管理DC、生产服务器、安全设备                              │
│  ├─ 不能访问外网！不能收发邮件！不能安装任何非管理软件！          │
│  ├─ 物理隔离或专用的VDI                                        │
│  ├─ 从网络层限制：PAW只能访问运维管理网段                        │

#### 加固5：AD回收站与备份（最后防线）

```
AD回收站（Active Directory Recycle Bin）可不是Windows回收站！

AD回收站的作用：
- 普通删除：对象（用户/组/OU）被逻辑删除，存留180天（墓碑期），需要工具恢复
- 启用了AD回收站：被删除的对象可以直接从AD管理中心恢复到删除前的状态
  → 所有属性完整保留（组成员关系、GPO链接等）

如何启用AD回收站：
1. 域功能级别必须≥2008 R2
2. 在AD管理中心(ADAC)中启用
3. ⚠️ 启用后无法撤销（这是一次性操作，启用前确认）

注意：不启用AD回收站的后果：
如果攻击者删除了所有域管理员账户 → 没有回收站 → 需要从备份还原 → 停机时间以小时计
如果启用了回收站 → 5分钟内恢复 → 比备份恢复快100倍

AD备份要点：
- 至少保留3份系统状态备份（Windows Server Backup → 系统状态备份）
- 备份不应存在同一台DC上（攻击者拿到DC就能删备份）
- 定期验证备份可恢复（不要等到灾难发生时才测试备份）
- 备份也必须加密存储
```

## 📊 面试模拟：域安全

**Q1： "什么是Kerberos认证？简单解释一下TGT和ST的区别。"**

> **标准回答**：Kerberos是Windows域的核心认证协议，名字来源于希腊神话的三头地狱犬，三个头分别代表协议中的三个角色：客户端、KDC（密钥分发中心）、服务端。TGT是Ticket Granting Ticket——门票授予票据，在用户第一次登录时由KDC发放，有效期通常8-10小时。你可以理解为入境时海关给你盖的"临时通行证"。ST是Service Ticket——服务票据，当你拿着TGT去访问某个具体服务（如文件服务器）时，KDC会根据TGT给你发一个ST，这个ST只能访问这一个服务，用完即失效。可以理解为拿着通行证去每个房间门口换的"进门卡"。这种"两段式"设计的好处是：即使攻击者截获了一个ST，也只能访问一个服务，危害有限。

**Q2： "攻防演练中，蓝队如何发现攻击者在做DCSync？"**

> **标准回答**：DCSync的核心特征是攻击者伪装成DC向其他DC请求目录复制数据。在Windows安全日志中，事件ID 4662是关键——当一个账户对目录服务对象执行操作，并且操作对象的GUID包含 `1131f6aa-9c07-11d1-f79f-00c04fc2dcd2`（这是"目录复制"权限的GUID）时，就是DCSync的强烈特征。此外，如果执行DCSync的账户不是正常的DC计算机账户（以$结尾的），那就是确定的攻击行为。监控策略上，我会在SIEM中建立规则：非DC计算机账户触发4662事件→P0告警→立即启动应急响应。面试官，补充一点：保护DCSync攻击面的最好方法是权限最小化——确保"复制目录更改"权限只授予Domain Admins和域控计算机账户，并定期审计这个权限的成员列表。

**Q3： "如果你管理的域控被攻破了，你的应急处置步骤是什么？"**

> **标准回答**（这是面试中的"压力测试"题，回答要体现冷静和系统性）：

> 第一步：隔离——立即将失陷的DC从网络中断开（拔网线或隔离VLAN），但不要关机！关机 = 丢失内存中的证据。
> 第二步：切换——如果有其他DC，确认它们未被攻破，然后把FSMO角色转移到安全的DC上。
> 第三步：取证——对失陷DC做内存dump和磁盘镜像（取证备份），这是后续分析和法律维权的依据。
> 第四步：重置——域内所有账户密码强制重置（包括域管、服务账户、所有用户）。因为攻击者可能已经拿到了所有哈希。
> 第五步：重装——失陷DC做数据库恢复或完全重装，不要尝试"修复"，不安全。
> 第六步：排查——查清攻击入口和影响范围，修复漏洞。如果LAPS没部署，现在马上部署。如果域管没加Protected Users组，现在加入。
> 第七步：报告——完整的事件报告和法律合规报告。
> 面试官，总结一下核心原则：域控失陷 = 整个AD数据库可能已泄露 = 全公司的身份体系需要重建。这不是一个"装个补丁就完"的事件，这是一个最高级别的安全事故。

## ⚠️ 常见误区

| 误区 | 真相 |
|:---|:---|
| ❌ "域控只是装了AD的Windows Server，和普通服务器一样管就行" | 域控承载整个身份认证体系，它的安全级别应该和核心数据库一样高。域控应该是最少人知道密码、最少人登录、最少安装软件的"三最"服务器。 |
| ❌ "我们域控打了补丁，很安全" | 补丁只能防已知漏洞。攻击者可能通过密码喷洒拿到一个用户名/密码，然后正常登录进来——这些都不需要利用漏洞。纵深防御不能靠一个措施。 |
| ❌ "LAPS部署太麻烦，我们每台机器手动设不同密码就行" | 手动管理500台机器的不同密码 = 不可能。这就是为什么需要LAPS——自动化是安全的必要条件，不可持续的手工操作=安全风险。 |
| ❌ "域管理员账号有20个人，方便" | 域管理员组成员越少越好，5人以内是目标。每多一个域管，攻击面就多一个——攻击者只需要拿到其中一个人的凭证就赢了。日常运维用委派权限，不要给域管。 |
| ❌ "域控的GPO审计太细了，告警太多" | 告警多是因为规则没做好。有效的审计不是"什么都记录"，而是"只记录关键变更"。GPO变更、域管组变更、密码重置——这三类事件必须有实时告警。 |

## 📝 今日总结

> **Day 15 核心收获：**
>
> 1. 域 = 集中身份管理 + 统一安全策略。它的核心价值（统一管理）同时也是它的核心风险（单点失陷=全局沦陷）
> 2. 攻击者从一台普通域成员到域控，可能只需要经过1-3次跳转。降低横向移动的可能——LAPS是第一优先级
> 3. 凭证保护是域安全的基石：长密码+LAPS+Credential Guard+Protected Users，层层叠加
> 4. AD审计不是"开了就行"，关键事件ID必须配置实时告警——DCSync（4662+GUID）、域管组变更（4728）、Kerberoasting（4769+加密类型）
> 5. 域控加固的"五件套"：补丁管理、凭证保护、GPO基线、PAW、AD备份（缺一不可）
> 6. 域控失陷 = 灾难级事件。如果你的域控被攻破，你需要重置所有密码、重建所有凭证、修复所有漏洞——成本极高

## 🎤 域安全面试中的"深度追问"——面试官真正想听的是什么

域安全是面试中拉开差距的关键话题。以下是常见的"追问链"和高质量回答示范：

```markdown
【追问链1：从表面到深层】
面试官：你怎么防御Pass-the-Hash？
你（基础回答）：部署LAPS，确保每台机器本地管理员密码不同。
面试官（追问）：如果LAPS因为某些原因暂时不能部署呢？
你（进阶回答）：短期替代方案——①用GPO把本地Administrator账户重命名为非默认名称
  (如改为LocalAdmin_%COMPUTERNAME%)；②通过GPO禁止本地管理员通过网络登录
  (拒绝访问来自网络的本地账户)；③部署Credential Guard(需要Win10+/Server2016+)。
  但这些只是缓解措施，最终还是要上LAPS。
面试官（二次追问）：即使部署了LAPS，还有什么途径可以拿到域管凭证？
你（深度回答）：即使LAPS防护了横向移动，攻击者还可以通过以下途径：
  ① Kerberoasting——请求高权限SPN的服务票据并离线破解；
  ② DCSync——如果拿到了具有"复制目录更改"权限的账户；
  ③ 在DC本地提取——如果攻击者攻陷了域控本身或能物理访问；
  ④ DPAPI提取——从域管的本地凭据存储中提取；
  ⑤ 中间人攻击——如LLMNR/NBNS投毒→捕获NetNTLMv2哈希→中继或破解。
  所以Pass-the-Hash只是凭据窃取的其中一环，域安全需要纵深防御。

【追问链2：从工具到思维】
面试官：域控上需要开启哪些审计策略？
你（基础回答）：登录事件(4624/4625)、账户管理(4720/4728/4738)、
  目录服务访问(4662)、安全组变更(4728/4756)、GPO变更(5136/5137)。
面试官（追问）：这么多审计事件，SIEM会不会告警爆炸？
你（进阶回答）：确实会。所以我不会全量告警，而是：
  ① 设置基线——过去30天的正常行为模式；
  ② 只告警"偏离基线"——如域管组增加了从未有过的新成员；
  ③ 区分告警优先级——DCSync相关的4662事件是P0，普通用户的4738是P3；
  ④ 用ATT&CK技战术映射来关联审计事件，降噪聚合。
面试官（二次追问）：如果你的审计策略被攻击者关闭了怎么办？
你（深度回答）：这是"防御者假设"问题。我的回答分三层：
  ① 防护层：用GPO强制开启审计策略，并通过"禁止覆盖"设置防止篡改。
     攻击者想关审计→需要域管权限修改GPO→但这又会触发GPO变更审计(5136)。
  ② 检测层：独立于AD审计，在SIEM/Sentinel中配置"审计策略关闭"的告警——
     当任何DC上EventID 4719(系统审计策略被修改)出现→P0告警。
  ③ 冗余层：不要只依赖AD审计。在DC上也部署Sysmon/EDR/NTA——
     即使AD审计被关，还有主机日志(进程创建/网络连接)和流量日志作为补充证据源。
```

**域安全面试高分心法：**
面试官问域安全不是为了考你"记不记得住事件ID"，而是看你能不能从"单点防御"思维升级到"多层防御"思维。所以任何回答都遵循这个公式：
具体措施 → 为什么这个措施有效 → 如果这个措施被绕过怎么办 → 补充措施是什么

## 🏗️ Active Directory分层管理模型——域安全的"地基"

域安全不只是配置几条GPO，而是要从架构上设计安全。微软推荐的分层模型（Tier Model）是域安全的地基：

```markdown
═══════════════════════════════════════════════════
  Active Directory 分层管理模型
═══════════════════════════════════════════════════

【Tier 0 — 域控和核心身份基础设施】（最高权限层）
  资产：域控服务器、AD FS服务器、Azure AD Connect、PKI/CA服务器
  管理员：Domain Admins、Enterprise Admins、Schema Admins
  安全要求：
    → Tier 0账户只能用Tier 0的PAW（特权访问工作站）登录
    → PAW本身必须在Tier 0的OU中，和其他层物理隔离
    → Tier 0的PAW不能用于上网、收邮件、处理文档
    → 登录Tier 0必须用独立的、不在其他层使用的账户
    → 必须启用多因素认证（最好是FIDO2硬件密钥）

【Tier 1 — 服务器和应用】（中间层）
  资产：所有成员服务器（Web/DB/App/文件服务器）
  管理员：Server Admins、应用管理员
  安全要求：
    → Tier 1管理员不能登录Tier 0资产
    → Tier 1管理员不能管理Tier 2工作站
    → Tier 1的远程管理必须通过堡垒机（不能直接RDP）
    → 所有Tier 1管理操作必须有审计日志

【Tier 2 — 用户工作站和设备】（标准层）
  资产：所有员工的电脑、笔记本、手机
  管理员：Helpdesk、桌面支持
  安全要求：
    → Tier 2管理员只能管理Tier 2资产
    → 日常操作（上网/邮件/文档）只在Tier 2设备上进行
    → 即使你同时是Domain Admin，你必须在Tier 0的PAW上做域管操作，
      用Tier 2的电脑处理日常工作

【跨层"不能做"的铁律】
  Tier 2的用户 → 不能登录Tier 0或Tier 1的服务器
  Tier 0的管理员 → 日常上网/邮件操作不能在Tier 0的PAW上做
  不同的Tier → 不同的账户、不同的设备、不同的密码
  
  为什么？因为如果域管在Tier 2电脑上登录并上网，
  一旦浏览器被钓鱼/水坑攻击 → 域管的凭证就从Tier 2泄露了 →
  攻击者用这个凭证跳到了Tier 1 → 再跳到Tier 0 → 全军覆没。
```

**给面试的回答——"什么是Active Directory分层管理？"**

```markdown
"AD分层管理就是把你的域环境分成三个层级：Tier 0是域控等核心资产（最敏感），
Tier 1是成员服务器，Tier 2是用户工作电脑。核心原则是：
① 不同层级使用不同的管理账户（域管账户绝不能在Tier 2电脑上使用）；
② 高权限操作在专用的PAW（特权访问工作站）上进行，这台机器不能做日常工作；
③ 凭证不能跨层流动——即使攻击者控制了Tier 2的一台普通电脑，
   他手中的凭证也不能在Tier 1或Tier 0使用。
这套模型是微软推荐的AD安全最佳实践，能有效防止凭证窃取导致的'全层沦陷'。"
```

## 🛡️ 域环境中"最容易忽视但最致命"的5个安全问题

域控安全不只有Golden Ticket和DCSync这些"高大上"的攻击。很多域被拿下是因为以下"低级但常见"的问题：

```markdown
【问题1：域管账户在非域控上登录过】
危害：域管在Tier 2电脑上登录 → 域管密码Hash留在了那台电脑的LSASS内存中 →
      攻击者如果控制了那台电脑 → Mimikatz导出Hash → 域控沦陷
检测方法：
  → 在所有Windows安全日志中搜索EventID 4624（登录成功）+ 域管账户名
  → 如果域管在非DC上的登录记录 → 立即排查该电脑的安全状态
  → 用GPO禁止域管登录到Tier 2工作站
  → 部署PAW（特权访问工作站）——域管只能从PAW登录
检查命令（在DC上PowerShell）：
  Get-WinEvent -FilterHashtable @{LogName='Security'; ID=4624} -MaxEvents 1000 |
    Where-Object { $_.Properties[5].Value -like '*Domain Admin*' } |
    Select-Object TimeCreated, @{N='User';E={$_.Properties[5].Value}},
                  @{N='SourceIP';E={$_.Properties[18].Value}}

【问题2：默认的Administrator账户没有重命名或禁用】
危害：攻击者不需要知道你的域管叫什么——"Administrator"这个默认名全世界都知道
检测方法：
  → 检查Administrator账户是否被重命名（不要用"Admin"/"root"也很容易被猜）
  → 检查Administrator账户是否被禁用（最佳实践：禁用默认Administrator，
    为每个域管创建个人命名的账户）
  → 创建"蜜罐Administrator"——一个名字叫Administrator但没有任何权限的假账户，
    任何对这个账户的登录尝试都是攻击信号
检查命令（PowerShell）：
  Get-ADUser -Identity Administrator | Select-Object Name, Enabled, SamAccountName

【问题3：服务账户（Service Account）使用弱密码且密码永不过期】
危害：服务账户通常有较高权限（如备份服务需要访问所有文件），但密码可能是
      "Service@123"且5年没改 → 攻击者破解一个服务账户 = 拿到高权限
检测方法：
  → 扫描域内所有密码超过1年没有更改的服务账户
  → 检查服务账户是否被加入了高权限组（Domain Admins / Enterprise Admins）
  → 服务账户密码复杂度是否满足要求（14位+随机密码）
  → 部署组托管服务账户（gMSA）——系统自动管理密码且定期轮换
检查命令（PowerShell）：
  Get-ADUser -Filter {PasswordLastSet -lt (Get-Date).AddDays(-365)} -Properties PasswordLastSet |
    Select-Object Name, PasswordLastSet, Enabled

【问题4：域信任关系（Domain Trust）被遗忘的安全风险】
危害：A域信任B域 → 如果B域被攻破 → 攻击者可以通过信任关系入侵A域
检测方法：
  → 列出所有域信任关系，检查是否有：
    - 与已废弃/已解约的合作方的信任关系还在
    - 信任方向是不是"双向"的（双向信任 = 相互完全信任 = 风险最大）
    - 测试/开发域和生产域的信任关系（测试域安全通常松散→是攻击者进入生产域的跳板）
检查命令（PowerShell）：
  Get-ADTrust -Filter * | Select-Object Name, Direction, TrustType, Target

【问题5：AD回收站没有启用】
危害：攻击者删除了关键AD对象（如所有域管账户）→ 没有回收站就需要从备份恢复 → 
      恢复时间以小时计 → 期间整个域无法正常工作
检测方法：
  → 检查AD回收站是否启用（Windows Server 2008 R2+支持）
  → 测试：创建一个测试用户 → 删除 → 能否从回收站恢复？
  → 即使启用了回收站，也要有独立的AD备份（不要只依赖回收站）
检查命令（PowerShell）：
  Get-ADOptionalFeature -Filter {Name -eq 'Recycle Bin Feature'}
```
  
**以上5个问题的检查可以合并成一个脚本，每月跑一次。大部分域被拿下不是因为APT的高级技术，而是因为这些"基础"没做好。**
