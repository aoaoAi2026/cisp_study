## 📖 核心知识讲解

### 一、先讲清楚——攻击链思维的两个版本

#### 1.1 Lockheed Martin Kill Chain（老版本，7步线性链）

```
Kill Chain = "杀死"一个攻击的七个步骤

┌──────────────────────────────────────────────────────────────┐
│                                                               │
│  1. 侦察        2. 武器化      3. 投递        4. 利用         │
│  Reconnaissance Weaponization  Delivery      Exploitation    │
│  踩点搜集信息   制作恶意程序   把武器送进去   利用漏洞激活   │
│                                                               │
│  5. 安装        6. 命令控制     7. 目标达成                    │
│  Installation   C2             Actions on Objectives          │
│  安装后门       建立控制通道    执行最终目标                  │
│                                                               │
└──────────────────────────────────────────────────────────────┘

防御策略（基于Kill Chain）：
  → 在每一步阻止攻击者，就打断了他的链
  → 阻止侦察 = 隐藏信息
  → 阻止投递 = 防火墙+邮件过滤
  → 阻止利用 = 打补丁
  → 阻止C2 = 封禁IP/域名
```

**Kill Chain的局限（面试加分点）：**

> Kill Chain的问题在于它假设攻击是"线性的"——侦察→武器化→投递→利用→安装→C2→目标达成。但真实攻击不是线性的！攻击者可能在利用阶段失败，换一个漏洞再试；可能在横向移动后又回来侦察新的目标。Kill Chain无法描述这种"非线性"的攻击过程。这就是为什么ATT&CK诞生了——它是"矩阵式"的，不是"链式"的。

#### 1.2 MITRE ATT&CK（新版本，14维战术矩阵）

```
ATT&CK = Adversarial Tactics, Techniques, and Common Knowledge
      = 对抗战术、技术和通用知识库

大白话：
  ATT&CK = 把全球已知的200+种攻击技术，按照"攻击者要完成的14个目标"分类整理
  → 你想知道"攻击者怎么横向移动的？" → 查"横向移动"战术下的技术列表
  → 你想知道"你的WAF能防哪些投递技术？" → 查"初始访问"战术下的技术，看覆盖率
  → 你想讨论一个事件 → 所有人用同一套编号沟通（T1059.001=PowerShell脚本执行）
```

**ATT&CK的14个战术（像"攻击者要做的事的checklist"）：**

```
战术ID  战术名称          大白话解释
────────────────────────────────────────────────────────────
TA0043  Reconnaissance     侦察 —— "踩点，了解目标是什么系统"
TA0042  Resource Development 资源开发 —— "准备武器弹药"
TA0001  Initial Access     初始访问 —— "想办法进大门"
TA0002  Execution          执行 —— "在目标上运行恶意代码"
TA0003  Persistence        持久化 —— "留把备用钥匙，下次还能来"
TA0004  Privilege Escalation 权限提升 —— "从普通员工变成总经理"
TA0005  Defense Evasion    防御规避 —— "躲监控摄像头"
TA0006  Credential Access  凭证访问 —— "偷别人的工牌和密码"
TA0007  Discovery          发现 —— "进大楼后找哪个房间有钱"
TA0008  Lateral Movement   横向移动 —— "从一个房间走进另一个房间"
TA0009  Collection         收集 —— "把值钱东西装进口袋"
TA0011  Command and Control 命令与控制 —— "和外面的同伙保持联系"
TA0010  Exfiltration       数据渗出 —— "把东西运出去"
TA0040  Impact             影响 —— "搞破坏、删数据、加密文件"
```

#### 1.3 ATT&CK技术编号规则（面试必须会读）

```
技术编号格式：TXXXX[.YYY]

T1003.001 = T1003(OS Credential Dumping) + .001(LSASS Memory)
            ↑ 父技术：操作系统凭证转储    ↑ 子技术：LSASS内存方式

常见子技术举例：
T1003.001 = LSASS Memory → Mimikatz用的就是这个
T1003.002 = Security Account Manager → SAM文件提取
T1003.003 = NTDS → DCSync攻击
T1003.004 = LSA Secrets
T1003.005 = Cached Domain Credentials

T1059.001 = PowerShell
T1059.003 = Windows Command Shell
T1059.005 = Visual Basic

读法：
"攻击者使用了 T1003.001（LSASS内存凭证转储）获取了域管哈希，
然后通过 T1550.002（Pass-the-Hash）横向移动到域控。"
→ 用ATT&CK语言描述一件事，所有安全人员都能秒懂
```

### 二、用ATT&CK框架做三件事（护网蓝队的核心能力）

#### 能力1：攻击模拟（红队思维，蓝队也要精通）

```
目标：模拟一个APT组织的攻击手法，检验你的防御体系

操作流程：
1. 选择一个APT组织（如APT29）
2. 在ATT&CK官网查该组织使用的技术清单
3. 用ATT&CK Navigator画出来，看技术覆盖
4. 按照这些技术，一步步模拟攻击
5. 用蓝队监测能力检验每一步是否被检测到
6. 没被检测到的技术 = 你的"防御盲区"

实战例子：模拟APT29（Cozy Bear）的攻击

APT29的标志性技术（根据公开情报）：
→ T1566.001 鱼叉钓鱼附件（初始访问）
→ T1059.001 PowerShell（执行）
→ T1053.005 计划任务（持久化）
→ T1003.001 LSASS内存转储（凭证访问）
→ T1021.002 SMB/Windows管理共享（横向移动）
→ T1071.001 HTTPS（C2通信）

模拟步骤（在你的测试环境中！）：
1. 制作并投递钓鱼邮件 → 看邮件网关能不能拦
2. 执行PowerShell脚本 → 看EDR能不能检测PowerShell下载行为
3. 创建计划任务 → 看SIEM能不能检测计划任务的创建
4. 用Mimikatz dump LSASS → 看EDR能不能拦截
5. 用PsExec横向移动 → 看网络检测能不能发现SMB异常
6. 建立HTTPS Beacon → 看流量分析能不能发现周期性心跳
```

#### 能力2：防御评估（蓝队核心，面试必考）

```
目标：量化评估你现有的安全防御能力，"知道自己的短板在哪"

评估方法（5步法）：

步骤1：列出你已有的安全能力
  例如：我们有WAF、EDR、SIEM、防火墙、邮件网关、IDS

步骤2：将每个能力映射到ATT&CK矩阵
┌──────────────────┬───────────────────────────────────────┐
│ 安全能力          │ 能覆盖的ATT&CK技术                     │
├──────────────────┼───────────────────────────────────────┤
│ 邮件网关          │ T1566.001(钓鱼附件)                   │
│                  │ T1566.002(钓鱼链接)                   │
├──────────────────┼───────────────────────────────────────┤
│ WAF              │ T1190(漏洞利用Web应用)                │
│                  │ T1505.003(Webshell)                   │
├──────────────────┼───────────────────────────────────────┤
│ EDR              │ T1059(脚本执行) T1003(凭证转储)       │
│                  │ T1547(启动项) T1053(计划任务)         │
├──────────────────┼───────────────────────────────────────┤
│ 防火墙            │ T1071(C2通信) T1048(数据外传)        │
│                  │ T1090(代理)                           │
├──────────────────┼───────────────────────────────────────┤
│ SIEM/日志审计     │ T1078(有效账户登录)                   │
│                  │ T1021(远程服务横向移动)               │
├──────────────────┼───────────────────────────────────────┤
│ IDS/流量分析      │ T1040(网络嗅探) T1573(加密通道C2)    │
│                  │ T1572(协议隧道)                       │
└──────────────────┴───────────────────────────────────────┘

步骤3：找出"空白格子"
  哪些ATT&CK技术没有被任何安全能力覆盖？
  → 例如：T1056(输入捕获/键盘记录) — 邮件网关拦不住、WAF管不着、EDR可能漏
  → 这就是你的"优先买装备"清单

步骤4：用ATT&CK Navigator可视化
  红色格子 = 有覆盖  空白格子 = 没有覆盖  黄色格子 = 部分覆盖
  你可以一眼看出你的防御覆盖率是多少

步骤5：制定改进计划
  为每个空白格子分配优先级和预算，逐步补齐
```

### 三、ATT&CK Navigator实战——防御覆盖度可视化

#### 3.1 什么是ATT&CK Navigator？

```
ATT&CK Navigator = 一个交互式的热力图，
把一个组织的防御能力映射到 ATT&CK 矩阵上

在线版：https://mitre-attack.github.io/attack-navigator/
离线版：下载源码在本地部署

你可以做：
- 选择一个APT组织的技术集（如APT29使用的所有技术）
- 标出你的每个安全设备能检测/阻止哪些技术
- 自动计算出"未被任何设备覆盖的技术"（你的防御盲区）
- 按颜色分层：红色=已阻止、黄色=已检测、白色=无覆盖
```

#### 3.2 实战：建立一个"护网蓝队检测能力矩阵"

```
让我们用ATT&CK Navigator建立一个典型的护网蓝队检测矩阵

假设你们团队有这些工具：
├── WAF (Web应用防火墙)
├── NIDS (网络入侵检测系统 — Suricata)
├── EDR (端点检测和响应 — 如某厂商EDR)
├── SIEM (安全信息和事件管理 — Splunk/ELK)
├── 邮件安全网关
└── 全流量分析平台 (Zeek + Arkime)

现在，逐项标注这些工具能检测的ATT&CK技术：

WAF 检测能力 → 标记为黄色（检测但不能完全阻止）
  ✅ T1190 — 利用面向公众的应用程序（检测SQL注入/XSS payload）
  ✅ T1505.003 — Web Shell（检测异常文件上传和Webshell通信模式）
  ❌ T1003 — 凭证转储（WAF看不到终端操作）

NIDS 检测能力 → 标记为黄色
  ✅ T1046 — 网络服务扫描（检测端口扫描模式）
  ✅ T1071.001 — Web协议C2（检测HTTP/HTTPS Beacon心跳）
  ✅ T1048.003 — 非标准协议渗出（检测非标准端口的大流量外传）
  ❌ T1059.001 — PowerShell（加密的WinRM流量看不出来）

EDR 检测能力 → 标记为绿色（这是最强的检测层）
  ✅ T1059.001 — PowerShell脚本执行
  ✅ T1003.001 — LSASS内存访问（检测进程读取LSASS）
  ✅ T1053.005 — 计划任务创建
  ✅ T1547.001 — 注册表Run键修改
  ❌ T1021.004 — SSH横向移动（EDR在Linux上的覆盖较差）

SIEM 检测能力 → 标记为黄色（依赖日志源完整性）
  ✅ T1078 — 有效账户（检测异常登录：时间/地点/频率）
  ✅ T1021.002 — SMB/Windows管理共享（检测登录事件4624）
  ✅ T1136 — 创建账户（检测事件ID 4720）
  ❌ T1564 — 隐藏文件和目录（如果没有文件完整性监控）

最后结果：
→ 你可以在Navigator中一目了然地看到：
  - 哪些高级威胁技术完全没有覆盖（需要追加预算）
  - 哪些技术被多个工具覆盖了（可以精简）
  - 各工具的责任边界是否清晰
```

### 四、两个真实APT案例分析（用ATT&CK拆解）

#### 案例1：某金融企业遭受的钓鱼攻击（中级复杂度）

```
背景：HR部门员工收到一封"年终奖调整通知"邮件，打开附件后中招

ATT&CK技术拆解（按时间顺序）：

【阶段1：初始访问】
T1566.001 鱼叉钓鱼附件
  → 邮件附件是带宏的Excel文件
  → 员工启用宏后，下载了后续payload

【阶段2：执行】
T1204.002 恶意文件（用户执行）
  → 用户手动启用了宏（这在HR部门很常见，年终奖通知嘛）
T1059.001 PowerShell
  → 宏调用PowerShell下载了CobaltStrike Beacon

【阶段3：持久化】
T1053.005 计划任务
  → 创建了名为"OfficeUpdate"的计划任务，每30分钟执行一次Beacon

【阶段4：防御规避】
T1027 混淆的文件或信息
  → Beacon payload经过Base64编码和简单XOR加密
T1070.004 文件删除
  → 执行完宏后，自动删除了原始Excel文件的副本
T1140 解码/解密文件或信息
  → PowerShell解码后执行真正的payload

【阶段5：凭证访问】
T1003.001 LSASS内存
  → Mimikatz从LSASS转储凭证 → 拿到了HR员工的密码哈希
T1003.003 NTDS
  → 从域控DCSync获取所有用户哈希（发现HR员工有域管权限！）

【阶段6：发现】
T1082 系统信息发现
  → 执行 systeminfo, ipconfig, netstat 收集信息
T1018 远程系统发现
  → net view /domain 列举域内所有计算机
T1087.002 域账户发现
  → net user /domain 列出所有域用户

【阶段7：横向移动】
T1021.002 SMB/Windows管理共享
  → 使用域管哈希通过PsExec横向移动到域控

【阶段8：收集与渗出】
T1005 来自本地系统的数据
  → 在文件服务器上找到财务数据共享文件夹
T1560.001 归档收集的数据
  → 用7-zip打包数据为 encrypted.7z
T1041 通过C2通道渗出
  → 通过HTTPS Beacon将数据分段传输到C2服务器

【整体评估】
攻击者水平：L2-L3（熟练攻击者）
使用的工具：CobaltStrike + Mimikatz + PsExec
攻击者弱点：钓鱼邮件有拼写错误（如果员工受到过安全意识培训可能被发现）
成功原因：HR员工缺少安全意识 + HR账户有域管权限（严重权限过度分配）
从入侵到数据泄露：约72小时
```

**面试回答模板**："这个案例体现了典型的以人为入口的攻击链：T1566.001鱼叉钓鱼 → T1204.002用户执行 → T1059.001 PowerShell → T1053.005持久化 → T1003.001凭证转储 → T1021.002横向移动 → T1041数据渗出。这个链条的防御可以在至少5个环节打断：邮件网关拦钓鱼、安全意识防点击、EDR拦PowerShell、LAPS防PTH横向、DLP拦数据外传。任何一个环节做得好，事件都不会到这个地步。"

#### 案例2：某科技公司的供应链攻击（高级复杂度）

```
背景：公司使用了某第三方开源组件，该组件被投毒

ATT&CK技术拆解：

【阶段1：资源开发】
T1584.006 攻陷基础设施（Web服务）
  → 攻击者先攻陷了组件的开发者账户

T1608.002 上传恶意软件
  → 攻击者在组件新版本中植入了恶意代码

【阶段2：初始访问】
T1195.001 供应链攻陷（软件依赖项和开发工具）
  → 公司通过npm更新了该组件到"带毒"版本
  → 恶意代码在CI/CD构建过程中被执行

【阶段3：执行+防御规避】
T1059.001 PowerShell
  → 植入的代码在构建服务器上执行PowerShell
T1036.005 匹配合法名称
  → 恶意进程伪装成 node.exe（正常的Node.js进程）
T1070.004 文件删除
  → 执行完初始脚本后自删除

【阶段4：持久化】
T1543.003 Windows服务
  → 在构建服务器上安装了伪装成"Node.js Update Service"的服务
T1098 账户操纵
  → 在Azure AD中创建了名为"cicd-pipeline"的新用户

【阶段5：凭证访问】
T1552.005 云实例元数据API

## ⚠️ 常见误区

| 误区 | 真相 |
|:---|:---|
| ❌ "ATT&CK就是背技术ID，考个证就行" | ATT&CK的精髓不是背ID，是用它的结构化思维来分析攻击。T1003.001不是考试答案，是让你理解"攻击者通过访问LSASS进程内存来窃取凭证"这种攻击模式及其防御方法。 |
| ❌ "把所有技术都覆盖了就是安全的" | 覆盖所有技术是不可能的。正确的做法是按"攻击者最可能用的技术"排优先级，聚焦检测覆盖率而非技术覆盖率。按APT组织的TTPs而非所有技术来做检测。 |
| ❌ "用了ATT&CK就是高级安全团队了" | ATT&CK是工具，不是护身符。关键是你怎么用：有没有用它做月度复盘？有没有用它指导预算分配？有没有用它统一团队的沟通语言？ |

  → 使用SolarWinds自己的数字签名的DLL作为攻击载体
  → 所有AV/EDR都信任这个签名的DLL
  → 所以全世界没有一家安全公司能在第一时间检测到

（命令执行）阶段就发现，而不是等到T1505.003（Webshell）阶段。"

→ 结构化、有深度、有改进思路，面试官一听就知道你"懂行"
```

## 🎓 ATT&CK在护网中的"终极应用"——用它来设计你的检测体系

很多蓝队把ATT&CK当成"事后分析工具"——事件发生了才用它来还原。但ATT&CK最大的价值是"事前设计"——用它来规划你的检测体系：

```markdown
【用ATT&CK设计检测体系的四步法】

Step 1：选择你的威胁场景
  不是"所有场景"，而是"你最害怕的3个场景"。
  例如：① 外部攻击者通过Web漏洞入侵；② 钓鱼邮件导致办公网沦陷；③ 内部员工数据窃取

Step 2：为每个场景列出技术链
  场景①的技术链（Web入侵）：
  → T1595 扫描 → T1190 漏洞利用 → T1059 命令执行 → T1505 Webshell → T1071 C2 → T1041 数据外传

Step 3：评估你当前的检测覆盖
  逐项问自己：这个技术我们有检测吗？检测是实时的还是批量的？
  → T1595扫描：IDS有检测（实时）✅
  → T1190漏洞利用：WAF有检测（实时）✅
  → T1059命令执行：EDR有检测（实时）✅
  → T1505 Webshell：只有定时扫描（每天1次）⚠️ → 需要升级到实时FIM
  → T1071 C2：只有SIEM规则（15分钟延迟）⚠️ → 需要NTA实时检测
  → T1041数据外传：没有检测 ❌ → 需要部署DLP

Step 4：按优先级填补缺口
  高优先级（没有检测→可能完全错过攻击）：T1041数据外传
  中优先级（有检测但时效差）：T1505 Webshell、T1071 C2
  低优先级（已有检测但可优化）：T1595扫描

  然后制定改进计划：Q1上DLP（最高优先级），Q2优化EDR+NTA（中优先级）
```

**给面试用的"ATT&CK防守一句话"：**

```markdown
"我们团队不是等攻击发生后才用ATT&CK做复盘，
而是主动用ATT&CK来设计检测体系——我们识别了针对我们行业
最可能的3个攻击场景，为每个场景建立了ATT&CK技术链，
然后评估我们的检测覆盖率，按优先级填补缺口。
目前我们的检测覆盖率约67%，下一季度目标是提升到75%。"

→ 这句话展示了：你不会被动防守，你会主动规划。
  你不是只会用工具，你懂得用框架指导安全建设。
  你有数据思维——"67%→75%"说明你在衡量和迭代。
```

## 📊 威胁情报在护网中的"消费"vs"生产"——蓝队不只要"查情报"，还要"产情报"

大多数蓝队只把威胁情报当成"查一下这个IP是不是恶意的"。但更高阶的蓝队会从自己处理的安全事件中"生产"情报，反哺整个安全社区和企业内部：

```markdown
【威胁情报消费（Consuming）——查别人的情报】

Level 1（基础消费）：查IP/域名/哈希的恶意标签
  → VirusTotal、微步在线、AbuseIPDB → 输入IOC → 看有没有被标记
  → 局限：很多攻击使用全新基础设施（0检测率），查不到不代表安全

Level 2（进阶消费）：查IOC的关联信息
  → 通过被动DNS查这个IP绑定了哪些域名
  → 通过SSL证书关联查同一证书还被哪些IP使用
  → 通过Whois查域名的注册人和注册邮件地址
  → 通过Shodan/Fofa/Censys查这个IP上还开放了什么服务
  → 这些"关联信息"往往比"恶意标签"更有价值

Level 3（高级消费）：情报驱动的威胁狩猎
  → 收到行业ISAC（信息共享与分析中心）的情报：APT29最近针对金融行业使用新型钓鱼邮件
  → 不被动等告警，而是主动搜索：
    - 邮件网关日志中有没有来自情报中提到的发件人域名？
    - SIEM中有没有PowerShell下载情报中提到的C2域名？
    - EDR中有没有检测到情报中描述的恶意文件行为？

【威胁情报生产（Producing）——从你的事件中提取情报】

你处理的每一个安全事件都包含宝贵的威胁情报。大多数蓝队浪费了这些情报：

可以从每个事件中提取的情报类型：
  ① 网络IOC：攻击者使用的IP、域名、URL
  ② 主机IOC：恶意文件的Hash（MD5/SHA1/SHA256）、文件路径、注册表键值
  ③ 行为IOC：攻击者使用的命令行参数、PowerShell脚本特征
  ④ TTP：攻击者的技战术过程（如先用OAuth钓鱼→再用PowerShell Empire建立C2）

情报生产流程：
  Step 1：事件处置完成后，从分析报告中提取所有IOC
  Step 2：去敏处理（脱敏内部IP、内部域名、员工姓名等）
  Step 3：标注每个IOC的"置信度"和"生命周期"
    置信度：这个IOC有多大把握是恶意的？（确认/很可能/可能）
    生命周期：这个IP可能是临时的（几小时）还是长期的（数月）？
  Step 4：同步到内部威胁情报平台（TIP），让所有安全设备自动消费
  Step 5：如果合适（不涉及商业机密），共享到行业ISAC或威胁情报社区

为什么蓝队需要"生产"情报：
  → 外部的威胁情报永远有滞后性（别人发现了才会出现在情报平台）
  → 你的企业被攻击时使用的IOC可能还没有被任何平台标记
  → 你生产的情报可能是"0day情报"——全球第一个发现这个IOC的
  → 内部情报平台可以让你的WAF/防火墙/SIEM在"第一个请求"就拦截，而不是等第三方标记后才拦截
```

**威胁情报分析的"黄金法则"：**

```markdown
法则1：一个IOC不能定罪
  一个IP被标记为"恶意" → 不能直接断定"这是攻击"
  → 必须结合你的环境中的行为日志交叉验证
  → 情报告诉你"这个IP不好"，日志告诉你"这个IP在你的环境中做了什么"

法则2：没有情报标签 ≠ 安全
  VirusTotal检出率0/89 → 不代表这个IP是干净的
  → 可能只是还没有被标记的新基础设施
  → 看行为！这个IP在你的环境中做了什么？

法则3：情报质量 > 情报数量
  每天收到10000条威胁情报 → 不代表你安全
  → 其中9990条可能和你的环境完全无关
  → 聚焦"对你有用的情报"——针对你行业、你技术栈的攻击情报

法则4：情报的时效性决定价值
  一个恶意IP的情报在攻击发生24小时后才收到 → 价值接近于零
  → 自动化情报集成（TIP → WAF/Firewall → 自动封禁）是缩短"情报到防护"时间的关键
```
