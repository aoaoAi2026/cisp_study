## 📖 核心知识讲解

### 一、威胁情报是什么？——"黑名单的进化版"

**一句话定义**：威胁情报 = 关于"谁是坏人、他们怎么干坏事"的结构化知识

```
传统方式：                威胁情报方式：
                          ┌─────────────────┐
IP: 1.2.3.4 攻击我了      │ IOC: 1.2.3.4     │
    ↓                     │ 标签：扫描器       │
手动封禁这个IP             │ 关联组织：APT28   │ ← 知道是谁
    ↓                     │ 攻击手法：T1595   │ ← 知道怎么干的
明天 5.6.7.8 又来攻击      │ 关联域名：evil.com│ ← 知道他的基础设施
    ↓                     │ 历史活动：活跃中   │ ← 知道最近还在干
再次手动封...              │ 处置建议：立即封禁 │ ← 知道该怎么办
                          └─────────────────┘
                          → 自动封禁 + 关联分析 + 提前防御
```

### 二、主要威胁情报平台使用

#### 2.1 VirusTotal（国际）

```
网址：https://www.virustotal.com

查询类型：
├── IP地址：看多少安全厂商标记了该IP为恶意
├── 域名：看域名解析历史、关联的恶意活动
├── 文件Hash：上传Hash看文件是否被识别为恶意
└── URL：看URL是否被标记为钓鱼/恶意

解读技巧：
- 标记数/总引擎数 = 1/90 → 大概率误报
- 标记数/总引擎数 = 30/90 → 高度可疑
- 标记数/总引擎数 = 60/90 → 基本确定恶意
- 看"Relations"栏目 → 该IP关联了哪些域名和文件（扩展IOC）
```

#### 2.2 微步在线（国内）

```
网址：https://x.threatbook.com

优势：对中国本土的攻击IP和组织有更好的覆盖

查询类型：
├── IP：威胁标签、地理位置、ASN、关联域名
├── 域名：whois信息、解析记录、关联样本
├── 文件Hash：是否是已知恶意样本
└── URL：是否是钓鱼/恶意链接

特色功能（面试常问）：
- 威胁标签体系：扫描器、暴力破解、C2、钓鱼、挖矿...
- 关联分析：IP → 注册邮箱 → 注册的其他域名（拓线）
```

### 三、攻击分级标准——从Day 5的P0-P3深化

| 等级 | 名称 | 升级触发条件 | 响应要求 | 上报层级 |
|------|------|-------------|---------|---------|
| 🔴 P0 | 紧急 | ① 已确认入侵（获权限/装后门）<br>② 核心系统受影响<br>③ 数据泄露确认 | 5分钟启动 | 立刻上报管理层+监管 |
| 🟠 P1 | 高危 | ① 攻击正在进行<br>② 目标为核心/重要系统<br>③ 存在已知可利用漏洞<br>④ 高强度攻击(如密集暴破) | 15分钟启动 | 上报安全负责人 |
| 🟡 P2 | 中危 | ① 攻击尝试但未确认成功<br>② 攻击手法常见(扫描/普通注入)<br>③ 无数据影响证据 | 1小时内处理 | 记录在日报中 |
| 🟢 P3 | 低危 | ① 明确误报<br>② 低风险扫描(爬虫/内扫)<br>③ 信息收集行为 | 当班内记录 | 归档即可 |

**升级决策"三看"**：
```
一看入侵成功证据：    有没有 200返回 / 数据回显 / 登录成功 / 文件落地？
二看影响范围：        核心系统 vs 边缘系统？涉及用户数据吗？
三看攻击烈度：        定向手工攻击 vs 自动扫描器？
```

## 🎯 蓝队面试高频题（Day 12 主题）

**Q1：什么是威胁情报？蓝队怎么用？**

> 回答：威胁情报是关于网络威胁的经过分析和处理的结构化信息。蓝队主要用运营层面的威胁情报——具体IOC（IP/域名/Hash/URL）来快速判断告警中的攻击源是否已知恶意、该采取什么级别的处置。比如一个IP触发了告警，先查IP情报，如果在多个平台上都被标记为"C2服务器"且置信度高，那就不是简单的"封禁+观察"，而是应该立刻升级响应级别、排查内部主机是否有连接到该IP的行为（C2通信）。

**Q2：IP情报查询发现该IP是CDN节点，但告警确实是这个IP发起的攻击，你怎么办？**

> 回答：CDN节点本身不是攻击源，它只是转发了攻击者的请求。应该检查HTTP请求头中的X-Forwarded-For或X-Real-IP字段，找到真实攻击IP再封禁。绝对不能封禁CDN节点的IP，否则会导致大量正常用户无法访问网站。如果无法获取真实IP（如CDN配置不正确），则优先考虑在应用层做限制（如对该URL增加验证码、限制单IP请求频率），而非在网络层封禁IP。

**Q3：P0和P1告警最核心的区别是什么？**

> 回答：P0是"已经着火了"，P1是"电线冒烟了"。P0有明确的入侵成功证据——已经获取了系统权限、已经安装了后门、已经泄露了数据。P1是攻击正在发生、很有可能会成功，但目前还没有确认成功的证据。处置上，P0除了要封禁攻击源，还要立刻启动PDCERF完整流程做根除和恢复；P1的核心是遏制——掐灭攻击以免升级为P0。

## ⚠️ 新手常见误区纠正

1. **误区**："情报平台显示恶意就是恶意，没显示就是干净"
   - **真相**：情报平台有滞后性——新出现的攻击IP可能还没被任何平台标记。没标记 ≠ 安全
   - **正确做法**：情报查询是参考，不是唯一判断依据。最终还是要看攻击payload和响应证据

2. **误区**："所有被标记恶意的IP都该封禁"
   - **真相**：标记可能是历史行为，当前这次请求可能是正常的。还要看当前告警的上下文和证据
   - **正确做法**：情报标签+当前行为+攻击证据，三者综合判断

3. **误区**："CDN的IP发起攻击，那我就封了这个CDN IP"
   - **真相**：CDN IP背后可能有成千上万个正常网站。你一封，这些网站用户全断了
   - **正确做法**：从X-Forwarded-For提取真实IP后封禁

## 🔬 深度案例：一次真实的威胁情报驱动的应急响应

### 背景

某金融公司在护网第一天凌晨3点，SIEM告警：内网一台OA服务器向境外IP 45.33.32.156 的4444端口发起TCP连接。流量不大（每5分钟一个心跳包约2KB），但持续了3天没人发现。你是新接手的安全分析师，现在需要做威胁情报驱动的调查。

### 第一步：情报查询

```
查询 IOC：45.33.32.156

微步在线：
  ├─ 威胁标签：C2通信、APT28关联、窃密木马
  ├─ 置信度：高（90%+）
  ├─ 关联域名：update-win.com（伪装成Windows更新）

## 🔧 高级实操：MISP威胁情报平台入门

MISP（Malware Information Sharing Platform）是开源的情报共享平台。很多护网团队用它做情报共享和协作。

### MISP的核心概念

```
MISP中的核心对象：

事件（Event）
  ├─ 一次威胁活动的完整描述
  ├─ 包含：描述、时间、威胁级别、关联组织
  └─ 示例："来自45.33.32.156的APT28 C2活动"

属性（Attribute）
  ├─ 事件的组成元素，通常是一个IOC
  ├─ 类型：IP、域名、URL、文件Hash、邮箱、CVE...
  └─ 示例：45.33.32.156（类型=ip-dst）

对象（Object）
  ├─ 多个属性的组合，形成有意义的"实体"
  ├─ 如：一个"文件"对象包含 filename + md5 + sha1 + size
  └─ 如：一个"网络连接"对象包含 src-ip + dst-ip + src-port + dst-port

标签（Tag）
  ├─ 给事件/属性打标签，方便分类检索
  ├─ 示例：tlp:amber（只共享给信任的合作伙伴）
  └─ 示例：APT28、C2、windows-malware

星系（Galaxy）
  ├─ 高级分类系统，如ATT&CK技战术
  └─ 示例：给一个事件关联 ATT&CK T1071.001
```

### MISP的实际使用场景

```
场景1：情报导入
  → 从外部情报源（微步/VirusTotal/合作伙伴）导入IOC
  → MISP自动去重+丰富上下文
  → 为所有IOC打上标签和分类

场景2：情报共享
  → 你发现了一个新的攻击者C2基础设施
  → 你在MISP中创建一个Event
  → 设置TLP级别（决定谁能看到）
  → 联盟内的其他成员自动同步到该情报

场景3：SIEM联动
  → MISP输出IOC列表（CSV/JSON/STIX）
  → SIEM读取IOC列表，自动更新告警规则
  → 发现新告警 → SIEM自动查询MISP确认是否为已知威胁

场景4：历史情报回溯
  → 新发现的IOC写入MISP
  → 自动在历史日志中回溯搜索
  → "这个IP在3天前就出现过！我们当时以为是误报！"
```

## 🌐 威胁情报共享生态——TLP协议与STIX/TAXII

### TLP（Traffic Light Protocol）——情报的"交通灯"

```markdown
TLP是情报共享中的分级标签，决定了情报可以分享给谁：

┌──────────────────────────────────────────────────────────────┐
│ TLP分级    │ 颜色  │ 含义              │ 可以分享给           │
├────────────┼───────┼───────────────────┼──────────────────────┤
│ TLP:CLEAR  │ ⬜ 白色 │ 无限制公开         │ 任何人，可以公开发布  │
│ TLP:GREEN  │ 🟩 绿色 │ 限社区内共享       │ 安全社区内，不可公开发布│
│ TLP:AMBER  │ 🟨 黄色 │ 限组织内共享       │ 仅限参与方组织内部    │
│ TLP:AMBER+ │ 🟧 橙色 │ 限组织内+严格限制   │ 仅在"需要知道"的基础上 │
│            │        │ Strict            │ 向特定个人分享        │
│ TLP:RED    │ 🟥 红色 │ 仅限指定接收者     │ 只能给指定的个人      │
│            │        │                   │ 不能转发给任何人      │
└────────────┴───────┴───────────────────┴──────────────────────┘

实战中的用法：
  IP 45.33.32.156 TLP:GREEN — 可以和同行分享这个恶意IP
  APT29最新TTP报告 TLP:AMBER — 只能在你所在的组织内部分享
  正在进行的攻击活动 TLP:RED — 只能和相关应急人员沟通
```

### STIX/TAXII——情报共享的"世界语"

```markdown
STIX（Structured Threat Information eXpression）：
  威胁情报的结构化表达语言，用JSON描述威胁。

一个简化的STIX 2.1 Indicator示例：
{
  "type": "indicator",
  "id": "indicator--a2b3c4d5-e6f7-8901-2345-678901234567",
  "created": "2024-06-15T10:30:00.000Z",
  "modified": "2024-06-15T10:30:00.000Z",
  "name": "APT28 C2 IP Address",
  "pattern": "[ipv4-addr:value = '45.33.32.156']",
  "pattern_type": "stix",
  "valid_from": "2024-06-15T10:30:00.000Z",
  "labels": ["malicious-activity"],
  "indicator_types": ["malicious-activity"],
  "confidence": 85
}
→ 这个JSON可以被任何支持STIX的平台自动理解！

TAXII（Trusted Automated eXchange of Indicator Information）：
  STIX的传输协议——相当于"情报的HTTPS"。
  → TAXII Server：情报提供方的服务器
  → TAXII Client：情报消费方的客户端
  → 支持"推"和"拉"两种模式
```

### 练习D：编写一个IOC自动化处理脚本

```python
#!/usr/bin/env python3
"""
ioc_enricher.py — 威胁情报IOC批量查询与富化脚本
从SIEM导出的IP列表中自动查询威胁情报并生成报告
"""

import requests
import csv
import time
import sys
from datetime import datetime

# 注意：需要先申请API Key
VT_API_KEY = "your_virustotal_api_key"  
ABUSEIPDB_KEY = "your_abuseipdb_api_key"

def query_virustotal(ip):
    """查询VirusTotal"""
    url = f"https://www.virustotal.com/api/v3/ip_addresses/{ip}"
    headers = {"x-apikey": VT_API_KEY}
    try:
        resp = requests.get(url, headers=headers, timeout=10)
        if resp.status_code == 200:
            data = resp.json()
            stats = data.get('data', {}).get('attributes', {}).get('last_analysis_stats', {})
            return {
                'malicious': stats.get('malicious', 0),
                'suspicious': stats.get('suspicious', 0),
                'harmless': stats.get('harmless', 0),
                'total': sum(stats.values()),
                'country': data.get('data', {}).get('attributes', {}).get('country', ''),
            }
    except Exception as e:
        print(f"  [!] VT查询失败: {e}")
    return None

def query_abuseipdb(ip):
    """查询AbuseIPDB"""
    url = "https://api.abuseipdb.com/api/v2/check"
    headers = {"Key": ABUSEIPDB_KEY, "Accept": "application/json"}
    params = {"ipAddress": ip, "maxAgeInDays": 90}
    try:
        resp = requests.get(url, headers=headers, params=params, timeout=10)
        if resp.status_code == 200:
            data = resp.json().get('data', {})
            return {
                'abuse_score': data.get('abuseConfidenceScore', 0),
                'total_reports': data.get('totalReports', 0),
                'country': data.get('countryCode', ''),
                'isp': data.get('isp', ''),
                'usage_type': data.get('usageType', ''),
                'last_reported': data.get('lastReportedAt', ''),
            }
    except Exception as e:
        print(f"  [!] AbuseIPDB查询失败: {e}")
    return None

def main():
    if len(sys.argv) < 2:
        print("用法: python3 ioc_enricher.py ips.txt")
        sys.exit(1)
    
    report_file = f"ioc_report_{datetime.now().strftime('%Y%m%d_%H%M%S')}.csv"
    
    with open(sys.argv[1], 'r') as f_in, open(report_file, 'w', newline='') as f_out:
        writer = csv.writer(f_out)
        writer.writerow(['IP', 'VT_Malicious', 'VT_Suspicious', 'VT_Total', 
                        'VT_Country', 'ABIPDB_Score', 'ABIPDB_Reports',
                        'ABIPDB_Country', 'ISP', 'Usage_Type', 'Verdict'])
        
        for line in f_in:
            ip = line.strip()
            if not ip or ip.startswith('#'):
                continue
            
            print(f"\n🔍 查询: {ip}")
            
            vt_result = query_virustotal(ip)
            if vt_result:
                print(f"  VirusTotal: {vt_result['malicious']}/{vt_result['total']} 恶意检出")
            
            time.sleep(0.5)  # 控制API频率
            
            ab_result = query_abuseipdb(ip)
            if ab_result:
                print(f"  AbuseIPDB: 滥用分数 {ab_result['abuse_score']}/100")
            
            time.sleep(0.5)
            
            # 综合判定
            verdict = "UNKNOWN"
            if vt_result and ab_result:
                if vt_result['malicious'] >= 5 or ab_result['abuse_score'] >= 50:
                    verdict = "BLOCK"   # 建议封禁
                elif vt_result['malicious'] >= 2 or ab_result['abuse_score'] >= 20:
                    verdict = "WATCH"   # 建议监控
                else:
                    verdict = "SAFE"    # 暂未发现恶意
            
            writer.writerow([
                ip,
                vt_result['malicious'] if vt_result else 'N/A',
                vt_result['suspicious'] if vt_result else 'N/A',
                vt_result['total'] if vt_result else 'N/A',
                vt_result['country'] if vt_result else 'N/A',
                ab_result['abuse_score'] if ab_result else 'N/A',
                ab_result['total_reports'] if ab_result else 'N/A',
                ab_result['country'] if ab_result else 'N/A',
                ab_result['isp'] if ab_result else 'N/A',
                ab_result['usage_type'] if ab_result else 'N/A',
                verdict
            ])
    
    print(f"\n📄 报告已保存到: {report_file}")

if __name__ == '__main__':
    main()
```

### 练习E：威胁情报消费的最佳实践

```markdown
护网值守中的情报消费原则：

原则1：优先消费"高置信度+可操作"的情报
  ✅ 高置信度（>70%）+ 具体IOC（IP/域名/Hash） → 直接用于自动封禁
  ⚠️ 中置信度（40-70%）+ 具体IOC → 标记为"需人工确认"
  ❌ 低置信度（<40%）或只有"描述"没有IOC → 仅参考，不自动处置

原则2：情报时效性管理
  运营级IOC（IP/域名）：TTL 7天（攻击者经常更换基础设施）
  战术级情报（TTP/工具）：TTL 6个月（攻击手法变化较慢）
  战略级情报（组织画像）：TTL 1年（APT组织特征相对稳定）

原则3：情报源多样化
  不要只依赖一个情报源！至少使用：
    → 1个商业情报源（如微步、Recorded Future）
    → 1个开源情报源（如MISP社区、AlienVault OTX）
    → 1个内部生产的情报（你自己从事件中提取的IOC）
  三个来源交叉验证 → 降低被单一情报源误导的风险
```

## 🌍 威胁情报在护网中的"三道防线"——从被动消费到主动生产

威胁情报的价值不是"查一下某IP是不是恶意"，而是建立三道防线：

```markdown
【第一道防线：阻断已知威胁】（战术级情报）
情报类型：IP黑名单、域名黑名单、文件Hash黑名单
消费方式：自动同步到防火墙/WAF/DNS/EDR → 自动阻断
时效要求：分钟级（新IOC出现后5分钟内必须同步到所有设备）
数据来源：商业威胁情报(微步/Recorded Future) + 开源情报(MISP/OTX)
你的角色：情报消费者

真实场景：
→ 威胁情报平台推送了一条"APT28新C2 IP: 45.33.32.156"（置信度95%）
→ SOAR在30秒内自动将该IP同步到全网防火墙+WAF+DNS黑名单
→ 如果内网有任何机器试图连接这个IP → 在TCP握手阶段就被防火墙丢弃
→ 这是最基础但最有效的防线——挡住"已知的坏人"

【第二道防线：识别未知威胁】（战术→运营级情报）
情报类型：TTP（Tactics, Techniques, Procedures）、攻击模式
消费方式：将TTP转化为SIEM检测规则（Sigma规则/Elastic规则）
时效要求：小时级（新TTP公开后24小时内完成规则部署）

### 场景1：凌晨3点，一个陌生IP在SSH暴力破解——要不要封？

```markdown
你的SIEM弹了一堆SSH登录失败告警。攻击源IP是 45.33.32.156。
你刚来值夜班，不认识这个IP。情报怎么帮你？

【无情报的情况下的决策】
  凭感觉猜→可能是某个扫描器在扫→不太像真正的攻击→先观察→
  结果真的是APT攻击者的C2服务器在做横向移动→
  等你"观察"完，攻击者已经爆破成功了一台服务器→损失已经发生。

【有情报的情况下的决策——5分钟内完成】
  Step 1（30秒）：查VirusTotal → 45.33.32.156被12/89个引擎标记为恶意
  Step 2（30秒）：查AbuseIPDB → 该IP在过去90天内被报告了47次，滥用分数96/100
  Step 3（20秒）：查威胁情报平台 → 该IP被标记为"APT28组常用C2基础设施"（置信度90%）
  Step 4（20秒）：关联你们自己的情报库 → 上周护网同学封禁过45.33.32.x网段的IP
  Step 5（10秒）：做出决策！
  
  决策：自动封禁 45.33.32.0/24 整个/24网段（因为APT28常在同一C段部署多个C2）
  通知：在班组群里发一条："封禁45.33.32.0/24——疑似APT28 C2基础设施，已关联VT和AbuseIPDB确认"
  → 从发现到封禁，总共不到5分钟。这就是情报的价值：把"决策时间"从不确定变成明确。
```

### 场景2：EDR发现一个陌生进程，要不要杀？

```markdown
EDR告警：服务器上出现了一个名叫"svchost.exe"的进程，
它在C:\Windows\Temp\目录下运行，连接了外网IP 103.x.x.x:8080。

【用Hash情报判断】
  拿到进程文件的SHA256：
  查询VirusTotal → Hash'abc123...'被 47/71 个引擎标记为恶意
  标记包括："Trojan.Agent"、"Backdoor.Cobalt"、"Beacon"（CobaltStrike）
  
  决策：这绝对是Beacon！不要犹豫，立即隔离！
  
  如果没有Hash情报 → 你可能花20分钟手动分析这个进程，
  期间它一直在回传数据。Hash查询帮你节省的20分钟=攻击者少了20分钟。

【用"应用情报"补全攻击全貌】
  Hash情报告诉你"这是CobaltStrike"，但应用情报（ATT&CK映射情报）能告诉你：
  
  这个CobaltStrike Beacon使用的是：
  → TA0005（Defense Evasion）：通过注入到svchost.exe来实现进程伪装
  → TA0011（Command and Control）：使用HTTPS over 8080端口，JA3指纹为'6734...'
  → 常见后渗透行为：执行whoami→net view→net use→mimikatz→PsExec横向移动
  
  所以你知道下一步该查什么了：
  → 查net view → 攻击者看了哪些域内主机
  → 查PsExec → 攻击者是否横向移动了
  → 查mimikatz → 攻击者是否dump了凭据
  → 情报告诉你了攻击者的"剧本"，你按剧本去找他在你们网络里的"演出"痕迹
```

### 场景4："这个告警我见过很多次了，都是误报吧？"——情报帮你戳破"狼来了"

```markdown
护网中最危险的思维定势："这个告警每天几百次，都是误报，不用管。"

真实事件：某公司SIEM每天产生"境外IP访问443端口"告警约3000+条，
SOC团队建立了一种"默认忽略境外443访问"的习惯。
直到有一天，一个真正的CobaltStrike HTTPS Beacon以每天10次的心跳
隐藏在3000条"正常"境外访问中——因为频率太低，没有人注意到。
攻击者在这台服务器上潜伏了3周，完成了内部侦察和信息收集。

【情报帮你区分"正常境外访问"和"C2 Beacon"】

正常境外访问（可以忽略）的特征：
  → 目的IP是已知的CDN/云服务IP（如Cloudflare、AWS的IP段）
  → 目的域名是知名商业网站（*.google.com、*.microsoft.com）
  → 连接是间歇性的，没有固定心跳间隔
  → TLS证书是由公共CA签发的（Let's Encrypt除外——因为攻击者也用）

C2 Beacon（不能忽略）的特征（核查）：
  → 目的IP的情报标签：被标记为"C2"、"Malware"、"APT"中的任何一个
  → 目的域名是3个月内注册的（新域名=高频C2特征！）
  → 连接有心跳规律（每X秒一个固定大小的数据包）
  → TLS证书是自签名的或由Let's Encrypt签发给可疑域名

情报过滤器方案：
  在SIEM里加一条"情报增强"规则——
  → 对每一个境外443访问的目标IP/域名，自动查询威胁情报API
  → 如果情报显示"干净"→标记为"低优先级"（不产生告警）
  → 如果情报显示"恶意/可疑"→告警+自动升级
  → 这样能把3000条/天变成3条/天——这3条才是你真正该关注的
```
