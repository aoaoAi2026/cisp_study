## 📖 核心知识讲解

### 一、DDoS到底是什么？——"三个臭皮匠打一个诸葛亮"

#### 1.1 用比喻彻底搞懂

> 🚰 DDoS = 你们公司只有一个水龙头，突然来了1000个人同时拧开水龙头接水，你的正常员工一滴水都接不到了

```
正常情况：                     DDoS攻击：
┌──────────┐                  ┌──────────┐
│ 真实用户1 │──→              │ 攻击者控制│
│ 真实用户2 │──→  服务器      │ 的僵尸网络│──→
│ 真实用户3 │──→              │ 1000台肉鸡│──→  服务器 💀 瘫了
└──────────┘                  │ 同时发送  │──→
                               │ 海量请求  │──→
                               └──────────┘

核心：DDoS不是"一个人特别厉害"，而是"一群人一起挤"。
      攻击者控制了分布在各地的被入侵设备（肉鸡/僵尸），
      指挥它们同时向你的服务器发送请求，
      直到你的带宽/连接/计算资源被耗尽。
```

#### 1.2 DDoS攻击的三种类型——攻击者打的是不同的"资源"

```
┌──────────────────────────────────────────────────────────────┐
│           DDoS攻击三层分类法（面试必答框架）                    │
├──────────────────────────────────────────────────────────────┤
│                                                                │
│  🔴 第一层：带宽耗尽型（Volume-based）                          │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 攻击目标：你的网络带宽                                  │    │
│  │                                                        │    │
│  │ 攻击手法：                                              │    │
│  │ - UDP Flood：发送海量UDP包占满带宽                     │    │
│  │ - ICMP Flood：发送海量Ping包                          │    │
│  │ - DNS Amplification：利用DNS服务器的放大效应            │    │
│  │   （发送60字节请求，DNS返回3000字节响应，放大50倍！）     │    │
│  │ - NTP Amplification：利用NTP服务器放大（最高556倍）     │    │
│  │                                                        │    │
│  │ 典型规模：几十Gbps ~ 几百Gbps甚至Tbps级别              │    │
│  │                                                        │    │
│  │ 大白话：就像有人用消防水管对着你家门缝灌水，             │    │
│  │         你家门再结实也挡不住水量                         │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                                │
│  🟡 第二层：协议耗尽型（Protocol-based）                        │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 攻击目标：网络设备（防火墙/负载均衡器）的连接表           │    │
│  │                                                        │    │
│  │ 攻击手法：                                              │    │
│  │ - SYN Flood：发送海量SYN包但不完成三次握手              │    │
│  │   → 服务器为每个"半开连接"分配内存等待                  │    │
│  │   → 连接表满了 → 正常用户连不进来                       │    │
│  │                                                        │    │
│  │ - ACK Flood：发送大量随机ACK包                          │    │
│  │ - RST Flood：发送大量RST包                              │    │
│  │ - Ping of Death：发送超大ICMP包（已过时但概念重要）     │    │
│  │                                                        │    │
│  │ 关键区别：不需要很大的带宽，只需要大量的连接            │    │
│  │ 大白话：1000个人同时在餐厅门口排队但不进去吃饭，         │    │
│  │         门外的队排满了，真正的客人进不去                 │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                                │
│  🟢 第三层：应用层耗尽型（Application Layer - Layer 7）         │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 攻击目标：应用服务器资源（CPU/内存/数据库连接/磁盘IO）    │    │
│  │                                                        │    │
│  │ 攻击手法：                                              │    │
│  │ - HTTP Flood：发送大量HTTP GET/POST请求                │    │
│  │   → 每次都请求最耗时的页面（如搜索功能、报表页面）       │    │
│  │                                                        │    │
│  │ - Slowloris：慢慢发送HTTP请求头部，永远不发送完整请求   │    │
│  │   → 服务器一直保持连接等待，直到连接池耗尽              │    │
│  │                                                        │    │
│  │ - Slow Read：正常请求页面但以极慢速度读取响应            │    │
│  │   → 服务器为慢速客户端保持连接，耗尽线程池              │    │
│  │                                                        │    │
│  │ - DNS Query Flood：发送大量DNS查询                     │    │
│  │                                                        │    │
│  │ 大白话：1000个人每人点了一道需要厨师花30分钟做的菜，     │    │
│  │         厨房忙不过来了，正常客人等不了就走了             │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                                │
└──────────────────────────────────────────────────────────────┘
```

**面试必考：三种DDoS的区别和对应防御方法**

| 攻击类型 | 攻击层 | 典型规模 | 特征 | 防御方法 |
|:---|:---|:---|:---|:---|
| 带宽耗尽 | L3/L4 | Gbps-Tbps | 流量曲线直线上升，带宽占满 | 黑洞路由 + 云清洗 |
| 协议耗尽 | L4 | 百万PPS | 连接表耗尽，CPU高但没有大流量 | SYN Cookie + 连接限制 |
| 应用层 | L7 | 正常带宽但CPU/内存爆 | 特定URL被大量请求，响应变慢 | WAF + 速率限制 + JS挑战 |

### 二、DDoS检测的5个关键指标

#### 2.1 流量基线——"正常"是你判断"异常"的唯一依据

```
没有基线就没有告警。DDoS检测的第一步永远是搞清楚"正常是什么样"。

你需要建立每条业务线的流量基线：

周一~周五 工作时间：
  ├── 入站流量：200-500 Mbps
  ├── 出站流量：100-300 Mbps
  ├── 新建连接数：5000-10000 个/秒
  ├── HTTP请求数：5000-20000 个/秒
  └── DNS查询数：1000-3000 个/秒

周末：
  ├── 入站流量：100-200 Mbps
  └── ...（普遍降低50%）

凌晨（02:00-06:00）：
  ├── 入站流量：50-100 Mbps
  └── ...（业务最低谷时段）

DDoS判定阈值（经验值，根据业务调整）：
  ⚠️ 黄色告警：流量超过基线的200%
  🚨 红色告警：流量超过基线的500%
  🔴 P0告警：带宽使用率达到100%且持续上升
```

#### 2.2 5个DDoS关键指标及检测方法

```bash
# 指标1：网络接口流量（最基础的DDoS指示器）
# 正常：200-500 Mbps  →  DDoS：突然飙到 5 Gbps

# Linux查看实时流量
sar -n DEV 1 5
# 或
iftop -i eth0
# 或
nload eth0

# 指标2：新建连接速率（SYN Flood的核心指标）
# 正常：5000-10000 conn/s  →  DDoS：突然 500000 conn/s

# Linux查看TCP连接统计
ss -s
# 输出中的 "synrecv" 数量如果突然暴增 → SYN Flood！

netstat -n | awk '/^tcp/ {print $6}' | sort | uniq -c | sort -rn
# SYN_RECV 状态大量出现 → SYN Flood！

# 指标3：并发连接数（Slowloris等慢速攻击的指标）
# 正常：10000-50000 并发  →  DDoS：突然 500000 并发

ss -tan | wc -l  # 总TCP连接数
ss -tan state established | wc -l  # 已建立连接数

# 指标4：HTTP请求速率（HTTP Flood的指标）
# 正常：5000-20000 req/s  →  DDoS：突然 200000 req/s

# 从nginx日志统计请求速率
tail -f /var/log/nginx/access.log | pv -l -i 1 > /dev/null
# 或使用 ngxtop
ngxtop -l /var/log/nginx/access.log

# 指标5：DNS查询速率（DNS Flood/Amplification的指标）
# 正常：1000-3000 qps  →  DDoS：突然 50000 qps

# 从网络流量统计DNS查询
tshark -i eth0 -Y "dns" -q 2>/dev/null | grep "packets"
```

### 三、DDoS防护的4种核心策略

#### 策略1：黑洞路由（BGP Blackhole）——"核选项"

```
原理：
  把你的目标IP从互联网路由表中"删除"
  → 运营商在骨干网路由器上添加一条路由：
    "所有去往 203.0.113.1 的流量 → 丢进黑洞"
  → 攻击流量和正常流量一起被丢掉
  → 你的IP在互联网上"不存在了"

优点：快（分钟级部署）、彻底（运营商层面丢弃）、成本低
缺点：杀敌一千自损八百（正常用户也访问不了）、只能用于"不重要的IP"

适用场景：
  ✅ 非关键业务IP遭到大流量DDoS
  ✅ 你的IP已经是"弃子"（准备切到备用IP）
  ❌ 你的主要对外服务IP（黑洞=你自己把自己从互联网搞没了）

面试话术：
"黑洞路由是DDoS应急响应的'核选项'。当攻击流量超过你的带宽上限、
云清洗服务也扛不住的时候，黑洞路由可以保护上游网络不瘫痪。
但代价是这个IP完全不可达——所有正常用户也会被挡在门外。
所以黑洞路由一般用在多IP部署的场景：黑洞被攻击的IP，
把流量切换到备用IP，让用户无感知切换。"
```

#### 策略2：流量清洗（Traffic Scrubbing）——"净水器模式"

```
原理：
  正常流量 + 攻击流量 ──→ [清洗中心] ──→ 只有正常流量
                            │
                         分析流量
                         过滤恶意包
                         转发正常包

工作流程：
  1. 检测到DDoS → 触发清洗
  2. 通过BGP宣告：把目标IP的流量"劫持"到清洗中心
  3. 清洗中心分析流量 → 识别并丢弃攻击包 → 转发正常流量你的服务器
  4. 正常用户几乎无感知（延迟可能增加10-50ms）

清洗技术：
  - 基于签名的过滤（黑名单IP/特征包）
  - 基于行为的过滤（正常用户不会每秒发10000个SYN）
  - 基于挑战的过滤（SYN Cookie、JS Challenge）
  - 基于机器学习的流量分类

清洗中心的局限：
  - 只能清洗"明确可识别"的攻击流量
  - 对"和正常流量一模一样"的应用层攻击（如HTTP Flood请求真实URL）难区分
  - 引入额外延迟（清洗中心需要时间分析流量）
```

#### 策略3：速率限制（Rate Limiting）——"限流阀"

```nginx
# 在nginx上配置速率限制（防御HTTP Flood）

# 第一步：定义限制区域
http {
    # 限制每个IP每秒最多10个请求
    limit_req_zone $binary_remote_addr zone=perip:10m rate=10r/s;

    # 限制每个服务器每秒最多1000个请求
    limit_req_zone $server_name zone=perserver:10m rate=1000r/s;
}

# 第二步：在需要保护的位置应用限制
server {
    location / {
        limit_req zone=perip burst=20 nodelay;
        # burst=20：允许瞬间20个请求的突发
        # nodelay：超过速率立即返回503，不等排队
    }

    # 对搜索API更严格的限制（因为搜索比静态页面更耗资源）
    location /api/search {
        limit_req zone=perip burst=5 nodelay;
    }
}

# 第三步：限制并发连接数（防御Slowloris）
server {
    # 每个IP最多10个并发连接
    limit_conn perip 10;

    # 每个虚拟服务器最多5000个并发连接
    limit_conn perserver 5000;
}
```

```bash
# 在iptables层面限制连接速率（防御SYN Flood）

# 限制每个IP每秒最多10个新的SYN包
iptables -A INPUT -p tcp --syn -m limit --limit 10/s --limit-burst 20 -j ACCEPT
iptables -A INPUT -p tcp --syn -j DROP

# 限制每个IP的并发连接数
iptables -A INPUT -p tcp --dport 80 -m connlimit --connlimit-above 50 -j DROP

# 启用SYN Cookie（内核层面防御SYN Flood）
echo 1 > /proc/sys/net/ipv4/tcp_syncookies

# 减少SYN_RECV超时时间
echo 5 > /proc/sys/net/ipv4/tcp_synack_retries
```

#### 策略4：Anycast分散——"把鸡蛋放在多个篮子里"

```
Anycast = 同一个IP地址广播到全球多个节点

传统架构（Unicast）：             Anycast架构：
┌──────────┐                     ┌──────────────────────┐
│ 攻击者    │                     │ 攻击者在欧洲发起      │
│ 所有流量   │                     │ DDoS攻击             │
└─────┬────┘                     └──────────┬───────────┘
      │                                     │
      ▼                              ┌──────▼──────┐
┌──────────┐                         │ 欧洲节点     │ ← 消化欧洲流量
│ 唯一服务器│ 💀 被打爆               │ (Anycast IP) │
└──────────┘                         └──────────────┘
                                     ┌──────────────┐
                                     │ 美洲节点     │ ← 不受影响
                                     │ (Same IP)    │
                                     └──────────────┘
                                     ┌──────────────┐
                                     │ 亚洲节点     │ ← 不受影响
                                     │ (Same IP)    │
                                     └──────────────┘

优点：
- BGP会自动把用户流量路由到最近的节点（最低延迟）
- 如果一个节点遭到DDoS，其他节点不受影响
- 攻击流量被分散到多个节点，每个节点承受的压力大大降低

局限：
- 需要多个数据中心/云节点，成本高
- 只对地域性DDoS有效（如果攻击来自全球，每个节点都会受影响）
- 有状态服务的部署复杂（数据库同步等问题）

云防护/CDN本质上就是基于Anycast的：
CloudFlare、AWS Shield、阿里云高防、腾讯云大禹 ——
它们的核心是：全球几十上百个节点共享同一个IP，
攻击流量被分散到全球，清洗中心清除恶意流量后
才转发到你的源站。
```

### 四、DDoS应急响应标准流程

```
DDoS应急响应六步法：

┌──────────────────────────────────────────────────────────────┐
│                                                               │
│  Step 1: 发现 (0-1分钟)                                       │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ · 监控告警：带宽使用率>90%、连接数异常飙升          │    │
│  │ · 业务反馈：用户报告网站打不开、响应慢              │    │
│  │ · 初步判断：是带宽被打满？还是连接数被打满？         │    │
│  └─────────────────────────────────────────────────────┘    │
│                         ↓                                      │
│  Step 2: 确认 (1-3分钟)                                       │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ · 查看流量图：是突发还是持续增长？                    │    │
│  │ · 查看攻击来源：IP/地区/ASN分布？是集中还是分散？    │    │
│  │ · 查看业务日志：是否伴随其他安全事件？                │    │
│  │ · 确认攻击类型：带宽/协议/应用层？                    │    │
│  └─────────────────────────────────────────────────────┘    │
│                         ↓                                      │
│  Step 3: 上报 (3-5分钟)                                       │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ · 通知值班主管和运维团队                              │    │
│  │ · 通知运营商/云防护服务商（如果是大流量攻击）         │    │
│  │ · 通知业务方：告知影响范围和预计恢复时间              │    │
│  │ · 发起应急响应流程（拉群、建War Room）                │    │
│  └─────────────────────────────────────────────────────┘    │
│                         ↓                                      │
│  Step 4: 防护 (5-30分钟)                                      │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 按攻击类型分层防护：                                  │    │
│  │                                                       │    │
│  │  带宽耗尽 → 联系运营商启用黑洞路由/流量清洗           │    │
│  │            → 或切换到云防护的高防IP                   │    │
│  │                                                       │    │
│  │  协议耗尽 → 启用SYN Cookie                             │    │
│  │            → 配置连接速率限制                         │    │
│  │            → 云防护开启协议清洗                       │    │
│  │                                                       │    │
│  │  应用层   → WAF开启CC防护规则                         │    │
│  │            → 配置JS Challenge人机验证                  │    │
│  │            → 对攻击IP/UA进行精准阻断                   │    │
│  │            → 启用静态页面缓存（让CDN扛住）             │    │
│  └─────────────────────────────────────────────────────┘    │
│                         ↓                                      │
│  Step 5: 恢复 (30分钟-2小时)                                  │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ · 观察流量是否恢复正常                               │    │
│  │ · 逐步取消临时防护措施（不要一次全撤，可能再攻击）    │    │
│  │ · 确认业务恢复正常                                   │    │
│  │ · 持续监控24小时                                     │    │
│  └─────────────────────────────────────────────────────┘    │
│                         ↓                                      │
│  Step 6: 复盘 (事件后24-48小时)                               │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ · 攻击源分析：IP/地区/攻击工具/攻击手法               │    │
```

### 练习1：SYN Flood检测与防御

```bash
# 场景：你的Web服务器突然响应变慢

# 第一步：查连接状态
ss -s
# 输出中重点看 synrecv 的数量（半开连接数）

# 第二步：查SYN_RECV状态的连接来源IP
ss -tan state syn-recv | awk '{print $5}' | cut -d: -f1 | sort | uniq -c | sort -rn | head -10
# → 如果某个IP有成千上万个半开连接 → SYN Flood！

# 第三步：启用SYN Cookie（内核防护）
echo 1 > /proc/sys/net/ipv4/tcp_syncookies

# 第四步：在iptables额外限制SYN速率
iptables -A INPUT -p tcp --syn -m limit --limit 50/s -j ACCEPT
iptables -A INPUT -p tcp --syn -j DROP

# 第五步：减少SYN重试次数（释放资源更快）
echo 2 > /proc/sys/net/ipv4/tcp_syn_retries
echo 5 > /proc/sys/net/ipv4/tcp_synack_retries
```

## 📊 面试模拟

**Q1："什么是DNS放大攻击？为什么它的攻击效果如此显著？"**

> **标准回答**：DNS放大攻击利用了DNS服务器的"不对称响应"特性。攻击者伪造了受害者的IP地址，向互联网上开放递归查询的DNS服务器发送一个约60字节的DNS查询请求（如查询ANY类型的记录），DNS服务器会返回一个约3000-4000字节的响应。这样一来，攻击者用60字节的请求换来了3000+字节的响应——放大倍数高达50倍。攻击者还同时使用多台DNS服务器，乘以肉鸡数量，一个1Gbps出入口的肉鸡可以打出50Gbps的攻击流量。面试官补充一点：NTP服务的放大倍数更高——最高可达556倍。所以DDoS防护中，禁止公网开放DNS递归查询和NTP服务是最基本的。

## ⚠️ 常见误区

| 误区 | 真相 |
|:---|:---|
| ❌ "有CDN就不怕DDoS了" | CDN主要防护应用层DDoS（CC攻击）。超大流量DDoS（超过500Gbps）仍然需要运营商级别的清洗。CDN能解决70%的DDoS问题，剩下30%需要多层防护。 |
| ❌ "买个抗DDoS设备放在机房就行了" | 大流量DDoS会把运营商给你的带宽先打满——流量还没到你的设备就已经堵死了。必须把防护前置到运营商/云平台层面。本地的抗D设备只能防护小于你带宽的攻击。 |
| ❌ "DDoS只是攻击手段，不会窃取数据" | 大多数情况下是的，但DDoS经常被用作"烟雾弹"——大量噪音掩盖了同时进行的入侵行为。安全团队在处理DDoS时注意力被转移，真正的数据窃取可能正在进行。 |

## 🔬 深度案例：三次真实的DDoS攻击事件

### 案例1：游戏公司的CC攻击——"我们不是被DDoS，是被模拟用户打死了"

```
某游戏公司在2024年护网期间遭遇了持续3天的CC攻击（HTTP Flood挑战Collapsar）。

攻击特征：
  → 请求频率：从正常的2,000 req/s突然飙到150,000 req/s
  → 请求来源：分布在全球150+个国家（典型的Botnet特征）
  → 请求特征：GET请求的都是真实的游戏API（/api/login, /api/join_game, /api/rank）
  → 流量特征：每个请求都很小（约2KB），看起来和真实用户一模一样
  
为什么传统的防护失效：
  ❌ 黑洞路由：不能用——这是主要业务IP
  ❌ IP封禁：IP太多（10万+），而且每天都在变
  ❌ SYN Cookie：没用——因为这走的是完成三次握手的正常HTTP请求
  ❌ 带宽清洗：没用——总带宽并没有打满，只是HTTP并发数爆了
  
当时怎么扛住的：
  ① WAF启用JS Challenge：要求所有/api/请求先通过JS挑战
     → 正常浏览器可以执行JS（真人用户可以），但Bot做不到
  ② 启用人机验证（Captcha）：对高并发IP弹出验证码
  ③ 基于用户行为的流量识别：
     → 正常的游戏玩家：登录→进入大厅→加入房间→开始游戏（有序列）
     → Bot：直接大量请求 /api/login 和 /api/rank（无序列）
     → 对"无行为序列的高频访问"进行限速
  ④ API速率限制：同一IP对 /api/rank 每分钟最多请求5次
  ⑤ CDN分散：将静态资源全部切到CDN，只保留动态API在源站
  
教训：
  - 应用层DDoS（CC攻击）最难防御——因为它模仿正常用户行为
  - 行为分析 > 流量分析
  - JS Challenge是最有效的应用层DDoS对策之一
```

### 案例2：DNS放大攻击——60字节怎么变成300MB的灾难

```
某金融机构遭遇了DNS放大攻击，过程如下：

攻击前：
  该机构对外DNS服务器（ns.company.com）开放了递归查询
  → 任何人都可以向这台DNS服务器发起查询
  → 这是被利用的前提！

攻击时：
  Step 1：攻击者控制了1万台被感染的IoT设备（摄像头/路由器）
  Step 2：每台僵尸设备发送请求到互联网上约5000台开放的DNS递归服务器
  Step 3：每台设备伪造源IP为：该金融机构的官网IP 203.0.113.10
  Step 4：查询类型为 "ANY"（要求返回所有记录类型）
          查询域名为攻击者自己注册的大记录域名（如 huge.attacker.com）
  Step 5：每台开放DNS服务器的响应：
          请求包 = 60字节
          响应包 = 3000-4000字节
          放大倍数 = 50倍
  Step 6：1万台僵尸 × 5000台DNS × 4000字节响应 × 50倍放大
          = 约10 Tbps的攻击流量涌向该机构
          
后果：
  - 该机构的官网入口带宽10Gbps瞬间被塞满
  - 所有人的正常请求都丢失在攻击流量中
  - 耗时4小时才完成清洗和恢复
  
为什么会发生：
  - 该机构的DNS服务器开放了递归查询（本应只对内部用户开放）
  - 没有在防火墙/路由器上限制UDP 53端口的入站流量速率
  - 没有和上游运营商建立DDoS应急联动机制
  
事后整改：
 ① 关闭DNS递归查询（只保留权威解析）
 ② 配置BGP Flowspec和上游运营商建立自动清洗通道
 ③ 启用RTBH（远程触发黑洞路由）——一旦检测到超大流量，
    可以通过BGP路由宣告将攻击流量牵引到清洗中心
 ④ 定期对DNS配置做安全审计
```

### 案例3：DDoS作为烟雾弹——攻击者真正的目标是数据库

```
时间线：
10:00 — 公司官网开始遭到HTTP Flood，首页响应时间从200ms变成10s
10:05 — 安全团队全员投入到DDoS应急响应中
10:10 — 启动了云清洗 + WAF JS Challenge
10:20 — DDoS开始减弱（可能是攻击者主动停止）
10:25 — 安全团队松一口气，开始写报告
10:30 — ⚠️ SIEM告警：核心数据库的审计日志显示，有一条异常的SELECT * FROM customers
        并且这条查询的时间...是10:12，正好在DDoS最激烈的时候！

调查还原：
  攻击者的真实目的不是DDoS，是窃取客户数据。
  DDoS只是用来"占线"——让所有人的注意力都在救火上，
  而真正的"小偷"在DDoS的掩护下已经悄悄进入了数据库。

  攻击链：
  ① Day -3：攻击者通过钓鱼邮件获得了一名DBA（数据库管理员）的凭证
  ② Day -2：用DBA权限在数据库服务器上创建了一个隐蔽的定时任务
  ③ Day 0 10:00：触发DDoS攻击，吸引全部安全资源
  ④ Day 0 10:12：定时任务执行 → dump了客户表 → 压缩 → 通过HTTPS外传
  ⑤ Day 0 10:30：数据已经在外网了
  
  如果安全团队没有在10:30留意SIEM的那条异常告警...
  → 可能要等到几天后暗网上出现公司客户数据，才知道"真正发生了什么"
  
重要教训：
  ✅ DDoS应急响应的时候，一定要有人在同时监控数据安全
  ✅ "排除了DDoS这个噪音之后"要对核心资产做一轮快速安全扫描
  ✅ 任何非工作时间的数据库操作都需要告警（哪怕是DBA）
```

## 🛠️ DDoS工具与命令速查

### 服务端检测命令

```bash
# 实时流量监控
iftop -i eth0 -n -P       # 按连接查看实时流量
nload eth0                 # 简洁的流量仪表盘
bmon                       # 带宽监控（支持历史数据）
vnstat -l -i eth0          # 实时带宽统计

# SYN Flood检测
ss -s | grep synrecv       # 半开连接数量
watch -n 1 'ss -s'         # 每秒刷新连接状态
# 如果synrecv > 1000 → 极可能正在遭受SYN Flood

# HTTP Flood检测（从Web日志）
# 每秒请求数统计
tail -10000 access.log | wc -l  # 最近10000条日志覆盖了多少秒？
# 如果10000条日志在10秒内产生 → 1000 req/s → 异常

# 查看Top请求URL（CC攻击通常集中在少数URL）
tail -10000 access.log | awk '{print $7}' | sort | uniq -c | sort -rn | head -10

# 查看Top来源IP
tail -10000 access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head -20
```

### 内核参数加固

```bash
# ===== 防御SYN Flood的内核参数 =====

# 启用SYN Cookie（必开！）
echo 1 > /proc/sys/net/ipv4/tcp_syncookies

# 减少SYN重试次数（更快释放半开连接资源）
echo 2 > /proc/sys/net/ipv4/tcp_syn_retries
echo 5 > /proc/sys/net/ipv4/tcp_synack_retries

# 增加半开连接队列大小
echo 8192 > /proc/sys/net/ipv4/tcp_max_syn_backlog

# 启用TIME_WAIT复用（减少连接表的压力）
echo 1 > /proc/sys/net/ipv4/tcp_tw_reuse

# ===== 防御带宽耗尽 =====

# 限制ICMP速率（防Ping Flood）
echo 1 > /proc/sys/net/ipv4/icmp_echo_ignore_broadcasts
iptables -A INPUT -p icmp --icmp-type echo-request -m limit --limit 1/s -j ACCEPT
iptables -A INPUT -p icmp --icmp-type echo-request -j DROP

# ===== 持久化（重启不丢失）=====
# 把以上配置写入 /etc/sysctl.conf
cat >> /etc/sysctl.conf << EOF
net.ipv4.tcp_syncookies = 1
net.ipv4.tcp_syn_retries = 2
net.ipv4.tcp_synack_retries = 5
net.ipv4.tcp_max_syn_backlog = 8192
net.ipv4.tcp_tw_reuse = 1
EOF
sysctl -p
```

## 🛡️ DDoS防护中"自研防护"vs"云清洗"——选错方案=白花钱

护网蓝队面对DDoS，最关键的决策是：用自己的设备扛还是上云清洗？以下是决策指南：

```markdown
═══════════════════════════════════════════════════════
  DDoS防护方案选择决策矩阵
═══════════════════════════════════════════════════════

【自研防护】（用防火墙/WAF/Nginx等自有设备硬扛）
适用场景：
  ✅ 攻击流量 < 你的出口带宽（如攻击5Gbps，你的带宽10Gbps）
  ✅ 攻击类型简单（SYN Flood/CC攻击），你有调优经验
  ✅ 你有专门的网络工程师可以7x24待命
  ✅ 业务不能接受"流量清洗可能误拦正常用户"（如金融交易）

不适用场景：
  ❌ 攻击流量 > 你的带宽 → 直接打满，自研防护无效
  ❌ 攻击类型复杂（混合攻击：SYN+UDP+DNS+HTTP同时来）
  ❌ 攻击持续时间长（>2小时）→ 你的团队撑不住
  ❌ 你没有大流量防护经验

【云清洗】（购买阿里云/AWS/Cloudflare等云端DDoS防护）
适用场景：
  ✅ 攻击流量 >> 你的带宽（如攻击50Gbps，你的带宽只有1Gbps）
  ✅ 攻击类型复杂多变，你不想自己研究防护策略
  ✅ 你没有专门的网络工程师
  ✅ 业务可以接受"清洗中心可能增加10-50ms延迟"

不适用场景：
  ❌ 强合规要求（数据不能经过第三方网络）
  ❌ 低延迟业务（如高频交易，不能接受额外延迟）
  ❌ 预算有限（云端防护按月收费，大流量清洗费用不菲）

【混合方案】（自研+云清洗联动）★ 护网推荐方案
  自研防护处理日常和中小规模攻击（挡住90%的日常攻击）
  + 云端清洗作为"兜底"（当攻击超过自研能力时自动牵引到云端）
  → 日常成本低 + 极端情况有保障
  → 要求：自研和云端之间需要BGP Flowspec/RTBH联动
```

**真实决策案例：**

```markdown
场景A：某电商公司的DDoS防护决策
  业务峰值：双11期间约5Gbps正常流量
  出口带宽：10Gbps
  攻击历史：去年遭遇过3次DDoS，最大一次30Gbps SYN Flood
  团队：2个网络工程师（非7x24）

  决策：购买云清洗作为"兜底"+ 自研防护处理日常
  理由：
  → 10Gbps带宽对付5Gbps正常流量+小攻击足够了
  → 30Gbps攻击超出带宽→需要云清洗
  → 2个人的团队无法7x24值守→云端自动清洗减轻人力负担
  
  实际效果：
  → 日常自研防护成功率：92%
  → 遇到超带宽攻击时：云端30秒内检测→1分钟内清洗流量→业务几乎无感知
  → 年成本：云清洗约15万/年 + 自研设备折旧5万/年 = 20万/年
  → ROI评估：避免了至少3次数小时的业务中断 = 挽回了约300万的损失

场景B：某游戏公司的错误决策（教训）
  决策：只依赖自研防护（省钱）
  原因：云清洗太贵（30万/年），自研设备一次性15万就搞定
  
  结果：
  → 第一年平安无事（运气好）
  → 第二年遭遇80Gbps UDP Flood → 10Gbps带宽瞬间打满
  → 游戏全部掉线，持续6小时 → 玩家大量流失 → 估算损失500万+
  → 事后紧急购买云清洗 → 但已经失去了大量用户
  
  教训：DDoS防护和保险一样——你买的时候觉得贵，
       但真正出事时你会觉得"当初怎么不买"。不要用"赌运气"来做安全决策。
```

## ⚡ DDoS应急响应的"3分钟黄金窗口"——在攻击打满带宽前的操作

DDoS和普通入侵的应急不一样——你必须在带宽被打满之前完成关键操作。以下是3分钟内的操作清单：

```markdown
【第0-30秒：确认攻击（不要误判正常流量为DDoS）】
□ 看监控大盘：入站流量是否突然飙升？（≥平时的3倍）
□ 看连接类型：是否大量SYN包/SYN_RECV状态？
□ 看来源IP：是否来自大量不同的IP？（DDoS特征）
□ 如果以上三项全部确认 → 判定为DDoS攻击，立即启动应急响应

【第30-60秒：启动基础防护（不需要批准的操作）】
□ 如果服务器有管理面板/SSH → 立即开启SYN Cookie
  echo 1 > /proc/sys/net/ipv4/tcp_syncookies
□ 如果CPU还扛得住 → 开启Nginx rate limiting
  limit_req_zone $binary_remote_addr zone=ddos:10m rate=10r/s;
□ 通知团队（一句话版）："正在遭受DDoS，峰值X Gbps，正在处置"
□ ⚠️ 不要做的事：不要重启服务器（网卡初始化需要时间，这期间完全无法访问）

【第60-120秒：攻击分类——决定下一步怎么防】
□ 分析攻击类型：
  → SYN大量=SYN Flood → 聚焦内核参数+SYN Cookie+上游清洗
  → UDP大量=UDP Flood → 聚焦上游过滤+云清洗（本地扛不住UDP Flood）
  → HTTP大量=CC攻击 → 聚焦WAF+验证码+逆向代理过滤
  → DNS大量=DNS Amplification → 聚焦DNS服务器配置+上游过滤
□ 分析攻击目标：是打某个具体IP/端口还是全站？
□ 估算攻击规模：攻击流量/正常流量 = 倍数

【第120-180秒：决定"自扛还是上云"】
  → 如果攻击流量 < 你带宽的80% → 自扛+调优+准备封禁
  → 如果攻击流量 > 你带宽的80% → 立即启动云端清洗（不要犹豫！）
  → 通知ISP/上游运营商：请求黑洞路由/流量清洗（如果你有合同）
  → 如果使用了CDN → 确认CDN的DDoS防护是否生效

【3分钟后的持续操作】
□ 持续监控流量趋势（攻击会不会继续增长？）
□ 收集攻击特征（源IP/端口/包大小/频率）→ 用于后续策略调优
□ 和业务部门沟通：当前哪些服务受影响？预计什么时候恢复？
□ 如果攻击持续>30分钟 → 考虑切换到静态页面/维护模式（至少让用户知道"我们在维护"而不是"网站打不开"）
```

**关键心态：DDoS应急和入侵应急不一样**

```markdown
入侵应急 → 你控制的是你自己的系统 → 你掌握主动权
DDoS应急 → 攻击流量来自外部，你无法"关掉"攻击 → 你只能"防御"和"等待"

所以：
  → 不要责备自己"为什么防不住"——没人能完全防住DDoS
  → 做好你能做的（调优内核参数、封禁特征IP、上云清洗）
  → 如果攻击超出你的能力范围 → 及时求助（ISP/云服务商/安全公司）
  → 你的角色是"减少伤害"而不是"消灭攻击"
```

## 🎯 应用层DDoS（CC攻击）的深度防御——最难防的DDoS类型

应用层DDoS（通常叫CC攻击——Challenge Collapsar，即挑战黑洞）是护网中最让人头疼的攻击类型，因为它不需要很大的带宽就能打垮一个网站。以下是蓝队必须掌握的CC攻击防御体系：

### CC攻击为什么"以小博大"？

```markdown
【带宽型DDoS vs 应用层DDoS的对比】

带宽型DDoS（SYN Flood/UDP Flood）：
  攻击原理：用海量数据包撑爆你的带宽
  攻击规模：Gbps级别（如50Gbps）
  防御思路：扩容带宽 + 上游清洗 + SYN Cookie
  → "粗暴"但好防——有流量清洗就基本能防住

应用层DDoS（CC攻击/HTTP Flood）：
  攻击原理：用"合法"的HTTP请求耗光你服务器的处理能力（CPU/内存/数据库连接）
  攻击规模：可能只有几十Mbps！
  但效果可能更大——因为每个请求都可能是"慢查询"、"复杂搜索"、"登录尝试"
  → "温柔"但难防——流量看起来是正常的HTTP请求，
    但每个请求都在后端执行了昂贵的数据库操作
```

### CC攻击的5种"高级玩法"

```markdown
类型1：慢速HTTP攻击（Slowloris/Slow POST）
  原理：攻击者发送不完整的HTTP请求，让服务器一直等→
        所有连接槽位被占满→正常用户无法连接
  特征：
    → 每个请求只有部分HTTP头，没有结束标记
    → 每个连接保持很长时间（分钟级）但几乎没有数据
  防御：
    → Nginx: client_header_timeout 5s（请求头必须在5秒内发完）
    → Nginx: client_body_timeout 5s（请求体必须在5秒内发完）
    → Nginx: limit_conn（限制每个IP的连接数，如limit_conn perip 10）
  
  真实场景：某电商网站在双11凌晨遭遇Slowloris攻击，
  攻击者只需要在每个VPS上开100个连接——10台VPS就占满了1000个连接。
  正常用户在浏览器里看到的是"连接中..."—转圈—放弃—去竞品网站。

类型2："重查询"CC攻击（针对搜索/报表功能）
  原理：攻击者反复请求服务器最耗资源的页面（如全站搜索、
        大报表导出、复杂筛选功能），每次都让服务器做昂贵的计算
  特征：
    → 请求的都是同一个或几个"重量级"URL
    → 每个请求的后端处理时间特别长（>3秒）
    → 攻击带宽不高，但服务器负载接近100%
  防御：
    → 对"重查询"URL单独限速：limit_req zone=heavy_search rate=1r/m（每分钟1次！）
    → 给"重查询"添加异步队列——不直接执行，先排队
    → 给"重查询"加缓存——相同参数的查询5分钟内只执行一次
  
  真实场景：某数据分析平台的"全量数据导出"功能被攻击者发现——
  攻击者写了个脚本，每10秒请求一次全量导出。每次导出会锁住数据库30秒。
  结果：数据库一直被锁→所有用户在页面上看到"加载中..."→
  实际攻击流量只有50Kbps，但整个平台瘫痪了。

类型3："缓存穿透"CC攻击（Bypass CDN/缓存）
  原理：攻击者精心构造URL参数，让每个请求都绕过缓存而打到源站
  特征：
    → 同一个页面，但每次请求加不同的随机参数
      /product?id=1&_rand=12345 → 绕过缓存
      /product?id=1&_rand=12346 → 绕过缓存
      ...
    → CDN的缓存命中率从95%降到0%
    → 所有请求都回源→源站被打挂
  防御：
    → CDN配置：忽略特定query参数（如_rand、_t、_nocache）
    → WAF规则：检测URL中的随机参数模式（连续多位数随机参数）
    → Nginx配置：对带特定参数的请求返回301跳转到无参数版本
  
  真实场景：某媒体网站在发布一篇重要新闻后遭到CC攻击——
  攻击者在新闻URL后拼接随机数，让CDN的每一层缓存都失效，
  所有流量都回源到源站→源站宕机→CDN返回504→用户打不开文章。

类型4："登录爆破型"CC攻击（模拟正常登录但大量尝试）
  原理：攻击者用大量代理IP，以正常频率（如每IP每5秒一次）尝试登录，
        从流量上看不像攻击，但后端数据库每5秒都在验证用户名密码——
        数据库连接池被耗光
  特征：
    → 登录接口（POST /login）的请求量突增（平时100/分钟→现在5000/分钟）
    → 每个请求都是不同的用户名（猜测攻击者在使用用户名+密码库）
    → 登录失败率很高（>90%）
  防御：
    → 对登录接口限流：limit_req zone=login rate=3r/m（每个IP每分钟最多3次登录尝试）
    → 对登录失败的IP累积封禁：5分钟内错误5次→临时封禁30分钟
    → 引入验证码：对登录失败率高的IP段弹出验证码
    → WAF规则：检测User-Agent的多样性（正常登录用户Agent相对集中，
      攻击者可能使用大量不同的Agent）

类型5："API滥用型"CC攻击（针对移动APP后端API）
  原理：攻击者逆向APP后发现了未做频率限制的API接口→
        用自动化脚本高频调用这些API
  特征：
    → 某个API端点（如/api/user/profile）的调用量突增100倍
    → 请求格式完全合法（因为是从APP逆向来的）
    → User-Agent、Token、签名都正确→看起来像"正常用户"→WAF难拦截
  防御：
    → API网关层做频率限制：每个认证用户每分钟最多X次
    → 对高频用户降级返回缓存数据（而不是实时查数据库）
    → 引入"用户行为分析"：如果同一个用户1分钟内查看了1000个不同商品→
      正常用户不可能这样做→标记为机器人
    → 监控"每个API端点的QPS曲线"：如果某个API的QPS突然跃升→自动告警
```

### CC攻击防御的"分层体系"——单靠一种手段防不住

```markdown
【CC攻击的5层防御体系】

第1层：CDN/边缘节点层（距离用户最近）
  防御手段：
  → 静态资源缓存（JS/CSS/图片直接在CDN返回，不打源站）
  → 边缘IP频率限制（每个IP的请求频率超过阈值→返回429 Too Many Requests）
  → 边缘JS Challenge（让浏览器执行一段JS计算才放行→区分真浏览器和脚本）
  → 边缘CAPTCHA（为可疑请求弹出验证码）
  → 效果：挡掉80%的低级CC攻击
  产品举例：Cloudflare DDoS Protection、阿里云DDoS高防、腾讯云EdgeOne

第2层：WAF/反向代理层（防止恶意请求到达后端）
  防御手段：
  → 请求特征过滤（User-Agent为空/过于常见→拦截）
  → IP黑名单/灰名单（基于威胁情报和实时行为判定）
  → URI频率限制（特定URL的访问频率上限）
  → GEO拦截（如果你们的业务只在中国→拦截所有境外IP）
  → 效果：挡掉15%的中级CC攻击

第3层：应用层/业务层（在代码层面保护核心功能）
  防御手段：
  → "重查询"接口限流（如全量导出功能每分钟每个用户1次）
  → 验证码保护（登录/注册/表单提交页面）
  → 业务逻辑限流（如每个用户每天最多查看1000个商品详情页）
  → 异步处理（把"重操作"从同步改为异步+队列）
  → 效果：挡掉4%的高级CC攻击（这些攻击绕过了CDN和WAF）

第4层：数据库/缓存层（保护最核心的资源）
  防御手段：
  → 读写分离（读请求走从库，写请求走主库——即使从库被打挂，写操作仍正常）
  → 查询结果缓存（相同查询5分钟内不重复执行）
  → 数据库连接池限制（最多X个并发连接→超出排队或拒绝）
  → 慢查询熔断（如果某个SQL执行>3秒→自动熔断返回默认数据）
  → 效果：保护数据库不被最后一层攻击打挂

第5层：监控告警层（"守门员"——什么时候该启动人工介入）
  防御手段：
  → QPS异常检测（当前QPS超过历史均值3倍→告警）
  → 错误率监控（5XX错误率>5%→告警）
  → 响应时间监控（P95响应时间>3秒→告警）
  → 效果：当自动化防御不够用时，人工介入做决策（如临时切换为静态页面）
```

### DDoS/CC攻击应急中的"保底方案"——当所有防线都失效时

```markdown
准备一个"静态降级页"——这是你的最后底牌：

降级页要求：
□ 纯HTML（不依赖任何后端、数据库、动态语言）
□ 用CDN托管（全球加速）
□ 页面包含：
  - Logo + "系统维护中，预计XX分钟内恢复"
  - 客服联系方式（电话/微信）——让用户知道"你们还在"！
  - 关键的静态通知（如"订单不受影响，恢复后可正常查看"）

降级决策触发条件：
□ 核心业务响应时间>10秒持续5分钟以上
□ 错误率>30%持续3分钟以上
□ 数据库连接池耗尽/数据库不可用

降级操作的黄金流程：
① 确认触发条件满足→通知团队"准备降级"（30秒）
② 切换DNS/负载均衡→将所有流量导向静态降级页（1分钟）
③ 在降级页运行期间：修复后端、扩容资源、加CC规则（可能需要10-30分钟）
④ 切回正常服务（分批切回——先切10%流量看是否正常→再全量切回）

面试加分点：当面试官问"如果所有防御都失效了你怎么办"→
答案=降级方案。这展示了你有"灾难恢复思维"而不只是"理想状态防御思维"。
```
