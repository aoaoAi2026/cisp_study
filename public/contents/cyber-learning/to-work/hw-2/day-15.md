## 📖 核心知识讲解

### 一、防火墙是什么？（门卫比喻）

**防火墙 = 大楼的门卫**

门卫的工作很简单：
1. 有人要进来 → 看身份证（检查IP和端口）
2. 不在白名单上 → 不让进
3. 在黑名单上 → 直接拦住

防火墙管的是**第3-4层**（IP和端口层面），它不关心你进门后干了什么。

```
防火墙判断逻辑：
  来源IP是白名单吗？→ 放行
  目标端口是允许的吗？→ 放行
  以上都不是？→ 拒绝
```

**防火墙典型规则示例：**
| 规则 | 含义 |
|:---|:---|
| 允许 192.168.1.0/24 访问 22端口 | 只允许内网SSH登录 |
| 拒绝 45.33.32.156 的所有访问 | 封禁恶意IP |
| 允许 0.0.0.0/0 访问 80,443端口 | 允许全世界访问网站 |
| 拒绝 0.0.0.0/0 访问 3306端口 | 禁止外网访问数据库 |

### 二、WAF是什么？（安检机比喻）

**WAF = 机场的安检机**

安检机不是看你的身份证（那是门卫的事），而是检查你**带了什么**：
- 包里有没有刀？（请求里有没有SQL注入）
- 瓶子里是不是水？（参数里有没有XSS代码）
- 行李中有没有违禁品？（上传的文件是不是webshell）

WAF管的是**第7层**（HTTP应用层面），它能看到具体的请求内容。

```
WAF判断逻辑：
  HTTP请求进来了 → 检查URL参数有没有SQL关键字 → 有就拦截
                → 检查POST内容有没有script标签 → 有就拦截
                → 检查上传文件是不是.php → 是就拦截
                → 都通过了 → 放行
```

### 三、防火墙 vs WAF —— 核心区别

| 对比维度 | 防火墙 | WAF |
|:---|:---|:---|
| 工作层级 | 第3-4层（网络/传输层） | 第7层（应用层） |
| 比喻 | 门卫 | 安检机 |
| 看什么 | IP地址、端口、协议 | HTTP请求内容、参数、文件 |
| 能挡什么 | 非法IP访问、端口扫描 | SQL注入、XSS、文件上传、CSRF |
| 能防端口扫描吗 | ✅ | ❌ |
| 能防SQL注入吗 | ❌ | ✅ |
| 能防DDoS吗 | ✅（一定程度上） | ✅（HTTP层面） |
| 典型产品 | iptables、深信服FW | 阿里云WAF、Cloudflare WAF、ModSecurity |

### 四、iptables基础实操（Linux自带防火墙）

iptables是Linux系统自带的防火墙，在蓝队应急响应中经常用来**紧急封禁攻击IP**。

**iptables核心概念：**
```
iptables有三张"表"和五条"链"：

三张表（按功能分）：
  filter表 → 过滤数据包（最常用）
  nat表   → 地址转换
  mangle表→ 修改数据包

五条链（按时机分）：
  INPUT   → 进入本机的包（最常用）
  OUTPUT  → 从本机发出的包
  FORWARD → 经过本机转发的包
  PREROUTING  → 路由前
  POSTROUTING → 路由后
```

**蓝队最常用的iptables命令：**

```bash
# 查看当前规则
sudo iptables -L -n -v

# 封禁一个IP（阻止所有来自该IP的访问）
sudo iptables -A INPUT -s 45.33.32.156 -j DROP

# 只允许指定IP访问SSH
sudo iptables -A INPUT -p tcp --dport 22 -s 192.168.1.100 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 22 -j DROP

# 允许Web端口
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# 删除一条规则
sudo iptables -D INPUT -s 45.33.32.156 -j DROP

# 保存规则（重启后生效）
sudo iptables-save > /etc/iptables/rules.v4     # Debian/Ubuntu
sudo service iptables save                        # CentOS/RHEL
```

### 五、WAF的检测与绕过

WAF主要通过**规则匹配**来检测攻击，但它也有"天花板"：

**WAF能检测的攻击模式：**
```bash
# 这些请求WAF一般能拦截：
GET /page?id=1 OR 1=1              → 规则匹配：发现SQL关键字
GET /search?q=<script>alert(1)     → 规则匹配：发现XSS标签
POST /upload (文件是shell.php)      → 文件名黑名单
```

**WAF绕过的常见手法（了解即可，理解WAF局限）：**
- 大小写混写：`SeLeCt` 绕过区分大小写的规则
- 编码绕过：URL编码/Unicode编码
- 注释插入：`SEL/**/ECT` 在关键字中插注释
- 分块传输：把payload拆成多块发送

> 🎯 **蓝队需要理解**：WAF不是万能的，它只是第一道防线。被WAF拦住不代表没风险（可能已经被绕过），没被拦住的也要看是不是新攻击方式。

### 六、蓝队视角：防火墙和WAF的日志怎么看

**防火墙日志示例：**
```
Jun 12 14:20:00 kernel: [iptables] DROP IN=eth0 OUT= MAC=... SRC=45.33.32.156 DST=192.168.1.100 LEN=60 TOS=0x00 PROTO=TCP SPT=12345 DPT=22
```
解读：来自45.33.32.156的连接被防火墙丢弃，目标是SSH端口(22)。

**WAF日志示例：**
```
[2026-06-12 14:20:00] [error] [client 45.33.32.156] ModSecurity: Access denied. [msg "SQL Injection Attack"] [uri "/product.php?id=1' OR '1'='1"]
```
解读：WAF拦截了来自45.33.32.156的SQL注入攻击。

## 🎯 蓝队面试高频题（Day 15 主题）

**Q1：防火墙和WAF的核心区别是什么？**

> | 维度 | 防火墙 | WAF |
> |:---|:---|:---|
> | 工作层 | 网络层/传输层（第3-4层） | 应用层（第7层） |
> | 看什么 | IP地址、端口、协议 | HTTP请求内容、参数、URL |
> | 防什么 | 端口扫描、DDoS、IP黑名单 | SQL注入、XSS、CSRF、文件上传 |
> | 决策依据 | 五元组（源IP/端口/目的IP/端口/协议） | HTTP语义（URL/参数/Cookie/请求体） |
> | 典型产品 | iptables、pfSense、山石/华为防火墙 | ModSecurity、阿里云WAF、CloudFlare |
>
> 比喻：防火墙 = 小区的门禁（看你住不住这栋楼）；WAF = 快递安检机（看你寄的东西里有没有违禁品）

**Q2：iptables的INPUT/OUTPUT/FORWARD链分别有什么作用？**

> iptables有三条默认链（链 = 规则链条）：
> - **INPUT**：目标是本机的包 → 判断是否放行进本机。比如：外部IP访问本机的80端口。
> - **OUTPUT**：本机发出的包 → 判断是否放行出站。比如：本机程序连接外部服务器。
> - **FORWARD**：经过本机转发给其他主机的包 → 判断是否允许转发。比如：本机做路由器/NAT时用的。
>
> 蓝队日常最常用的：INPUT链封禁恶意IP、OUTPUT链阻止被入侵服务器外连C2、FORWARD链控制内网流量转发。

**Q3：WAF是怎么检测SQL注入的？什么情况下WAF会被绕过？**

> WAF检测SQL注入的三种方式：
> 1. 正则匹配：匹配 `' OR 1=1`、`UNION SELECT` 等SQL注入特征字符串 → 最基础但易绕过
> 2. 语义分析：解析SQL语法，判断是正常查询还是恶意查询 → 更难绕过
> 3. 行为分析：学习正常访问模式，异常请求自动拦截 → 最智能但可能误报
>
> 绕过WAF的常见手段：
> - 编码绕过：把 `'` 写成 `%27`（URL编码）或 `&#39;`（HTML实体编码）
> - 注释混淆：`UN/**/ION SEL/**/ECT`（在关键字中插入注释）
> - 大小写混用：`uNiOn SeLeCt` → 如果WAF大小写敏感就会被绕过
> - 等价替换：用 `||` 代替 `OR`、用 `&&` 代替 `AND`

## 📖 深度阅读：防火墙规则的"最小权限原则"

蓝队写防火墙规则最容易犯的错误是**太宽松**。最小权限原则要求："只放行必须放行的，其余全部拒绝。"

```bash
# ❌ 坏的规则：全放行然后逐一拒绝
iptables -P INPUT ACCEPT       # 默认放行所有入站
iptables -A INPUT -s 1.2.3.4 -j DROP  # 只拒绝特定IP

# ✅ 好的规则：全拒绝然后逐一放行
iptables -P INPUT DROP         # 默认拒绝所有入站
iptables -A INPUT -p tcp --dport 80 -j ACCEPT   # 只开放80端口
iptables -A INPUT -p tcp --dport 443 -j ACCEPT  # 只开放443端口
iptables -A INPUT -s 192.168.1.0/24 -p tcp --dport 22 -j ACCEPT # 只允许内网SSH
```

**规则的优先级（匹配顺序很重要）：**
```
规则1 → 规则2 → 规则3 → ... → 默认策略
遵循"最先匹配"原则——第一个匹配的规则生效，后续不再检查。
```

## 💡 面试高频题：防火墙与WAF

**Q: 防火墙和WAF的核心区别是什么？**
A: 防火墙工作在第三/四层（网络/传输层），基于IP/端口/协议做访问控制——允许哪些IP访问哪些端口。WAF工作在第七层（应用层），基于HTTP请求内容做检测——拦截SQL注入/XSS等Web攻击。简单记忆：防火墙管"谁能进门"，WAF管"进门之后有没有带武器"。

**Q: WAF的三种部署模式是什么？**
A: ①反向代理模式（最常用）：WAF作为Web服务器的"门卫"，所有流量先经过WAF再转发到后端——可以对HTTPS流量做解密检测；②透明桥模式：WAF接入网络链路，对流经的HTTP流量做旁路检测，不改变网络拓扑；③插件模式：WAF作为Web服务器的一个模块安装（如ModSecurity），与Web服务器共享进程。蓝队需要了解各种模式的特点以正确解读WAF告警。

**Q: WAF的拦截逻辑是什么？**
A: 收到HTTP请求→逐条匹配规则库→如果命中恶意特征→返回403/自定义错误页面并记录日志→未命中→转发到后端服务器。关键在于"规则库"——规则太严=误报多（正常业务被拦截），规则太松=漏报多（攻击穿透）。蓝队的一项重要工作就是持续优化WAF规则，在误报和漏报之间找到平衡。

**Q: iptables在蓝队工作中有哪些实际用途？**
A: ①临时封禁攻击IP：`iptables -A INPUT -s 攻击IP -j DROP`；②限制SSH爆破：`iptables -A INPUT -p tcp --dport 22 -m recent --update --seconds 60 --hitcount 5 -j DROP`；③端口白名单（除业务端口外全封）：先把22/80/443加入白名单，再DROP所有其他端口的入站流量。注意：iptables规则重启后丢失，需要`iptables-save`持久化。

**Q: 蓝队如何评估现有WAF的防护能力？**
A: ①规则覆盖测试：用常见攻击payload（SQL注入/XSS/文件上传/命令注入）逐条测试WAF是否拦截；②绕过测试：尝试常见绕过手法（大小写/编码/注释/分片）看WAF是否仍能检测；③误报测试：用正常业务流量测试，看WAF是否误拦。评估结果应整理成WAF能力矩阵表格。


## 📊 实操：iptables常用规则速查

```bash
# 查看当前规则
iptables -L -n -v

# 封禁单个IP
iptables -A INPUT -s 1.2.3.4 -j DROP

# 封禁IP段
iptables -A INPUT -s 1.2.3.0/24 -j DROP

# 只允许特定IP访问SSH
iptables -A INPUT -p tcp --dport 22 -s 192.168.1.100 -j ACCEPT
iptables -A INPUT -p tcp --dport 22 -j DROP

# 限制SSH连接频率（防爆破）
iptables -A INPUT -p tcp --dport 22 -m recent --set --name ssh_brute
iptables -A INPUT -p tcp --dport 22 -m recent --update --seconds 60 --hitcount 3 --name ssh_brute -j DROP

# 保存规则（持久化）
iptables-save > /etc/iptables/rules.v4
```

## 🔬 深度专题：防火墙与WAF深度对比——蓝队防护体系基石

### 防火墙规则优化实战

**1. 最小权限原则的iptables实现**
```bash
#!/bin/bash
# 默认拒绝所有入站流量
iptables -P INPUT DROP
# 允许已建立的连接
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
# 允许SSH（仅限内网）
iptables -A INPUT -p tcp -s 192.168.0.0/16 --dport 22 -j ACCEPT
# 允许HTTP/HTTPS
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT
# 记录被拒绝的连接
iptables -A INPUT -j LOG --log-prefix "IPTABLES-DROP: " --log-level 4
```

**2. 防暴力破解规则**
```bash
# SSH防爆破（60秒内超过3次新连接则封禁）
iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --set --name SSH
iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --update --seconds 60 --hitcount 3 --name SSH -j DROP
```

**3. WAF日志分析——发现规则优化点**
```bash
# 统计WAF拦截最多的规则（TOP10）
grep "blocked" waf.log | awk -F'rule_id=' '{print $2}' | awk '{print $1}' | sort | uniq -c | sort -rn | head -10

# 统计WAF误报（返回403但实为正常业务的请求）
grep "403" nginx_access.log | grep -v "waf"
```

### 下一代防火墙（NGFW）的核心能力

现代护网环境中，传统防火墙已升级为NGFW（下一代防火墙），蓝队需要了解其核心差异：

| 能力 | 传统防火墙 | NGFW |
| --- | --- | --- |
| 工作层 | L3/L4 | L3-L7 |
| 规则粒度 | IP+端口 | IP+端口+应用+用户+内容 |
| 应用识别 | 仅端口 | 深度包检测（DPI），识别端口上的实际应用 |
| 用户识别 | 无 | 集成AD/LDAP，用户级别策略 |
| IPS集成 | 无（需单独部署） | 内置IPS引擎 |
| 威胁情报 | 无 | 集成实时威胁情报 |
| SSL检测 | 无法 | SSL解密后检测 |

**蓝队NGFW实操要点**：
- 启用应用识别——不要仅依赖端口做策略
- 开启SSL解密（对核心业务段的流量）
- 配置用户级别策略——"研发部门不能访问外部网盘"
- 联动威胁情报——自动封禁已知恶意IP

网络安全值守中，很多判断需要在几秒内完成。以下是本日主题相关的"条件反射"训练：

**看到以下现象 → 立即联想到 → 采取动作**：

1. 短时间内同一IP大量不同URL请求 → 目录/漏洞扫描 → 检查是否返回了不该返回的内容
2. WAF告警+同IP的Web日志中有500错误 → 攻击可能在尝试绕过WAF → 查看完整请求体
3. 非工作时间的管理员登录 → 凭据泄露/后门 → 确认是否为合法运维操作
4. 同一文件被频繁POST请求 → Webshell心跳 → 检查文件内容和创建时间
5. 出站流量突增到非标准端口 → 数据渗出/C2通信 → 追踪目标IP并阻断

### "如果是你，你怎么防？"

假设你是护网蓝队负责人，面对今天学习的安全威胁，请设计你的防御方案：

- 响应层：确认攻击后如何处置？（隔离/封禁/取证/恢复的标准动作）
- 复盘层：事后如何防止再次发生？（规则优化/流程改进/培训加固）

## 🔬 进阶专题：本日知识在真实护网中的应用

### 护网场景还原

想象你正在护网值守中，突然收到一条与本日主题相关的告警。以下是标准的思维和处理流程：

**1. 第一反应（0-30秒）**：确认告警来源设备是否正常 → 查看告警级别 → 判断是否为"狼来了"

**2. 快速研判（30秒-2分钟）**：提取关键信息（源IP/目标资产/攻击类型/时间）→ 查威胁情报 → 看同一IP的其他告警

**3. 深度分析（2-10分钟）**：回放原始流量/日志 → 还原攻击payload → 判断攻击是否成功

**4. 处置决策（10-15分钟）**：确认是攻击→定级→封IP/隔离/升级事件；确认是误报→标记→优化规则

**5. 记录闭环（15-20分钟）**：填写工单（研判依据+处置动作+证据截图）→ 标记闭环


### 高手与新手的关键区别

| 维度 | 新手 | 高手 |
| --- | --- | --- |
| 看告警 | 只看这一条告警的内容 | 同时关联其他设备/其他时间的同类告警 |
| 判断依据 | "这个payload看起来像攻击" | "这个payload+这个User-Agent+这个时间点+这个频率=SQLMap自动化扫描" |
| 处置方式 | 封IP→完事 | 封IP+排查漏洞+检查是否已有数据泄露+通知业务方+更新WAF规则 |
| 记录质量 | "已处置" | "10:05收到WAF SQL注入告警→核查源IP 45.x为境外IP+威胁情报标记恶意→查看该IP完整请求链→确认SQLMap自动化扫描→已在WAF永久封禁+通知开发修复参数化查询→附件：攻击流量pcap+威胁情报截图" |

### 你今天学的知识在ATT&CK中的映射

花5分钟思考以下问题（有助于建立安全思维框架）：

- 今天的攻击手法在ATT&CK中属于哪个战术？对应的技术编号是什么？

- 用Kill Chain模型分析：这种攻击通常发生在哪几个阶段？

- 你的检测手段应该部署在Kill Chain的哪一环？为什么？

- 如果你负责设计针对这种攻击的防御方案，你会从哪几个层面入手？


> **记住**：蓝队不是"见招拆招"的被动防守，而是"知己知彼"的主动防御。每天学完一个知识点后，立刻思考"如果我是攻击者我会怎么用"+"如果我是防御者我该怎么挡"——这种双向思维是蓝队核心竞争力的来源。
