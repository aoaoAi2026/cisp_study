## 📖 核心知识讲解

### 一、什么是Webshell

Webshell是一段恶意脚本，攻击者上传到Web服务器后，可以通过浏览器远程控制服务器。

核心危害：拿到Webshell = 拿到了服务器的操作权限。攻击者可以执行命令、查看文件、窃取数据、横向移动。

常见webshell类型：

| 语言 | 典型一句话木马 |
|:---|:---|
| PHP | `<?php @eval($_POST['cmd']);?>` |
| ASP | `<%eval request("cmd")%>` |
| JSP | `<%Runtime.getRuntime().exec(request.getParameter("cmd"));%>` |
| ASPX | `<%@ Page Language="Jscript"%><%eval(Request.Item["cmd"],"unsafe");%>` |

### 二、常见一句话木马识别

PHP一句话木马危险函数特征库：

```
eval()          - 执行PHP代码
assert()        - 断言执行
system()        - 执行系统命令
exec()          - 执行外部程序
shell_exec()    - 通过shell执行命令
passthru()      - 执行外部程序并输出
popen()         - 打开进程文件指针
proc_open()     - 执行命令打开文件指针
create_function() - 创建匿名函数执行
call_user_func()  - 回调函数执行
```

一句话木马的伪装手法：

```php
// 正常：直接的eval
<?php @eval($_POST['cmd']);?>

// 伪装1：变量拆分
<?php $a = 'e'.'v'.'a'.'l'; $a($_POST['cmd']);?>

// 伪装2：字符串反转
<?php $a = strrev('lave'); $a($_POST['cmd']);?>

// 伪装3：base64解码
<?php $a = base64_decode('ZXZhbA=='); $a($_POST['cmd']);?>

// 伪装4：藏在图片里（图片马）
// GIF89a<?php @eval($_POST['cmd']);?>
```

### 三、文件特征检测法

时间异常检测：

```bash
# 查找最近24小时内修改的.php文件
find /www -name "*.php" -mtime -1 -ls

# 查找与站点大部分文件时间不一致的孤立时间
find /www -name "*.php" -newer /www/index.php -ls
```

文件权限异常：

```bash
# webshell通常由www-data/nobody写入
find /www -name "*.php" -perm /o+w
```

文件名异常：

```
- index.php.bak        (伪装成备份)
- images.php           (藏在图片目录)
- user.php5            (利用解析绕过)
- config.inc.php       (伪装成配置文件)
```

### 四、日志检测法

Web访问日志关键特征：

```bash
# 查找对.php文件的POST请求
grep "POST.*\.php" access.log | awk '{print $1, $7}' | sort | uniq -c | sort -rn

# 查找返回200但极少正常访问的可疑文件
grep "\.php" access.log | awk '{print $7}' | sort | uniq -c | sort -n
# 访问次数为1-3次的.php文件需要重点关注
```

特征流量检测：

webshell通信通常有这些特征：
- POST请求body包含 eval、base64_decode、cmd=、exec
- User-Agent异常（非浏览器UA，如python-requests）
- 请求时间异常（凌晨3-5点）
- Referer为空（直接访问，非正常浏览路径）

```bash
# 查找凌晨时段的.php POST请求
grep "POST.*\.php" access.log | awk '$4 ~ /0[3-5]:/ {print}'

# 查找User-Agent异常的请求
grep "POST.*\.php" access.log | grep -v "Mozilla" | grep -v "Chrome"
```

### 五、实战检测流程

完整检测5步法：

```
第1步：文件搜索
- 查近N天新增/修改文件
- 查非正常目录下的脚本文件

第2步：内容扫描
- 搜索危险函数：eval|assert|system|exec|shell_exec|passthru
- 搜索编码特征：base64_decode|strrev|gzinflate|str_rot13

第3步：日志关联
- 找到可疑文件路径，查该文件的访问记录
- 关注POST请求、异常UA、异常时间

第4步：工具确认
- D盾、河马查杀、WebShellKiller
- 在线检测：Virustotal、微步在线

第5步：应急响应
- 确认后立即备份证据（文件+日志）
- 隔离文件（修改权限400禁止执行）
- 排查入侵路径（上传点、漏洞入口）
```

## 🎯 高频面试题

**Q1：WebShell是什么？常见的一句话木马长什么样？**

> WebShell = 以网页脚本形式存在的后门，攻击者通过浏览器访问它来远程控制服务器。
> 
> PHP一句话木马（最经典）：
> ```php
> <?php @eval($_POST['cmd']); ?>
> ```
> 解释：通过POST参数cmd传入任意PHP代码→eval执行→攻击者可以执行任何系统命令。
>
> 变种（混淆版）：
> ```php
> <?php $a = 'asse'; $b = 'rt'; $c = $a.$b; $c($_POST['x']); ?>
> // 等价于 assert($_POST['x']) —— 把关键字拆开绕过简单的关键词检测
> ```

**Q2：怎么从Web目录中找出WebShell？**

> 6种检测方法：
> 1. **最近修改的文件**：`find /var/www -name "*.php" -mtime -7` → 最近7天修改的PHP文件
> 2. **可疑文件名**：不属于正常Web应用的文件（随机字符串.php、1.php这类）
> 3. **关键字搜索**：`grep -r "eval\|assert\|base64_decode\|system\|exec\|shell_exec" /var/www --include="*.php"`
> 4. **文件时间异常**：PHP文件修改时间和网站部署时间不一致
> 5. **文件大小异常**：一句话木马通常只有几十字节
> 6. **通过访问日志反向查找**：访问了奇怪路径的IP → 顺着路径找到文件

**Q3：WebShell的访问日志特征是什么？**

> - 低频访问（攻击者不会频繁来访，以免被注意）
> - 访问路径不常见（/images/avatar.php? 这种伪装在正常目录下的）
> - 只有POST请求，几乎没有GET请求（攻击者通过POST发送命令）
> - User-Agent可能异常（攻击者可能用自定义UA或脚本访问）
> - Referer为空或来自外部（正常Web页面间跳转应该有Referer）
> - 请求参数包含编码后的命令（base64、URL编码）

### 面试高频题

## 💡 面试高频题：Webshell检测

**Q: Webshell是什么？有哪些常见类型？**
A: Webshell是一种以Web脚本（php/jsp/asp/aspx）形式存在的后门程序，攻击者将其上传到Web服务器后，可以通过HTTP请求远程执行系统命令、管理文件、操作数据库。常见类型：①一句话木马（最简洁，如`<?php @eval($_POST['cmd']);?>`）；②大马（功能丰富的Webshell，提供图形化管理界面）；③内存马（不留文件，仅驻留在应用进程内存中，极难检测）；④冰蝎/哥斯拉等加密Webshell（通信内容加密，传统特征匹配失效）。

**Q: 蓝队如何检测Webshell？**
A: 多层次检测方案：①文件层面检测——检查Web目录中是否有最近新增/修改的脚本文件（`find /var/www -name "*.php" -mtime -1`）；搜索文件内容中的危险函数（`grep -rE "eval\(|assert\(|system\(|exec\(|shell_exec\(|popen\(" /var/www/`）；与已知Webshell样本库做哈希比对；注意检测文件的时间戳异常（创建时间晚于修改时间=克隆的文件）。②流量层面检测——Webshell通信流量通常具有以下特征：请求体/响应体极小（心跳包）、请求间隔固定、请求中携带加密或编码的payload、User-Agent异常。③行为层面检测——Web进程（如www-data/apache用户）不应该执行netstat/ifconfig等系统命令→如果监控到异常行为关联Web进程→可能是Webshell在活动。

**Q: 内存马（无文件Webshell）怎么检测？**
A: 内存马不落盘，传统文件扫描完全无效。检测思路：①Java内存马——通过检查JVM中加载的Filter/Servlet/Listener列表，发现未在web.xml中注册的组件（用arthas/jmap工具dump内存分析）；②PHP内存马——检查PHP-FPM进程中是否加载了异常扩展；③通用方法——EDR/HIDS监控Web进程的系统调用行为，如果www-data进程执行了netstat/curl/wget等命令=高度可疑。内存马检测是蓝队的高级技能，需要深入理解应用运行机制。

**Q: 发现Webshell后的应急处理流程是什么？**
A: ①立即断网隔离受感染服务器（不关机，保留内存证据）；②提取Webshell文件样本和访问日志（用于溯源分析）；③排查Webshell的上传入口（文件上传漏洞？SQL注入写文件？弱口令登录后台？）；④检查该服务器是否被用作跳板攻击了其他内网系统（检查~/.ssh/known_hosts、bash_history等）；⑤检查同Web目录下是否还有其他Webshell（攻击者通常会留多个后门）；⑥修复上传漏洞、修改所有相关密码、重装或从干净备份恢复系统。

## 🔬 深度专题：Webshell深度检测——从文件到内存到流量

### 常见Webshell混淆手法与反混淆

攻击者通过各种手段让Webshell逃过检测，蓝队需要了解这些手法才能反制：

**1. 字符串拼接混淆**
```php
# 明文：<?php eval($_POST["cmd"]); ?>
# 混淆：<?php $a="ev"."al"; $a($_POST["cmd"]); ?>
# 更强混淆：<?php ${"GLOBALS"}["x"."x"]="as"; $xx(${"_PO"."ST"}["c"]); ?>
```
检测思路：不仅匹配eval/system等关键字，还要检测字符串拼接模式。

**2. 编码/加密混淆**
```php
# Base64编码：<?php eval(base64_decode("ZmlsZV9nZXRfY29udGVudHMoJF9HRVRbJ2YnXSk7")); ?>
# XOR加密
# ROT13编码
```
检测思路：检测base64_decode/gzinflate/str_rot13等解码函数与危险函数的组合使用。

**3. 图片马**
攻击者将Webshell代码隐藏在图片文件的EXIF信息中，后端通过include方式执行。
检测思路：对所有上传的"图片"做文件格式验证+内容扫描。

**4. .htaccess配合绕过**
```
# 将.jpg文件当作PHP执行
AddType application/x-httpd-php .jpg
```
检测思路：检查Web目录中的.htaccess是否有异常配置。

### Webshell流量特征——不管怎么变形，通信模式是变不了的

即使Webshell代码做了深度混淆，它的通信模式仍然有规律可循：

**特征1：心跳包**
- Webshell客户端（如蚁剑/冰蝎/哥斯拉）定期发送小请求确认后门存活
- 特征：对同一个PHP文件的POST请求，间隔固定（如每60秒一次），请求体极短

**特征2：加密通信**
- 冰蝎/哥斯拉等现代Webshell使用AES加密通信
- 特征：POST请求体为Base64字符串，解密后是序列化的命令数据
- 检测：请求体熵值（entropy）异常高——正常表单数据的熵值低，加密数据的熵值接近随机

**特征3：User-Agent异常**
- 蚁剑默认UA：`antSword/v2.1`
- 冰蝎默认UA：可自定义但默认有规律
- 检测：建立正常UA白名单，排除非白名单UA的请求

**检测命令**（流量层面）**：**
```bash
# 检测同一文件的高频POST请求（心跳包特征）
grep "POST" access.log | awk '{print $7}' | sort | uniq -c | sort -rn | head -10

# 检测请求体熵值（需要Python脚本）
python3 -c "import sys,math; [print(f'{l}: entropy={sum(-p*math.log2(p) for p in [l.count(c)/len(l) for c in set(l)])}') for l in sys.stdin]" < post_bodies.txt
```

## 🔬 进阶专题（续2）

### 护网场景还原

想象你正在护网值守中，突然收到一条与本日主题相关的告警。以下是标准的思维和处理流程：

**1. 第一反应（0-30秒）**：确认告警来源设备是否正常 → 查看告警级别 → 判断是否为"狼来了"

**2. 快速研判（30秒-2分钟）**：提取关键信息（源IP/目标资产/攻击类型/时间）→ 查威胁情报 → 看同一IP的其他告警

**3. 深度分析（2-10分钟）**：回放原始流量/日志 → 还原攻击payload → 判断攻击是否成功

**4. 处置决策（10-15分钟）**：确认是攻击→定级→封IP/隔离/升级事件；确认是误报→标记→优化规则

**5. 记录闭环（15-20分钟）**：填写工单（研判依据+处置动作+证据截图）→ 标记闭环


### 高手与新手的关键区别

| 维度 | 新手 | 高手 |
| --- | --- | --- |
| 看告警 | 只看这一条告警的内容 | 同时关联其他设备/其他时间的同类告警 |
| 判断依据 | "这个payload看起来像攻击" | "这个payload+这个User-Agent+这个时间点+这个频率=SQLMap自动化扫描" |
| 处置方式 | 封IP→完事 | 封IP+排查漏洞+检查是否已有数据泄露+通知业务方+更新WAF规则 |
| 记录质量 | "已处置" | "10:05收到WAF SQL注入告警→核查源IP 45.x为境外IP+威胁情报标记恶意→查看该IP完整请求链→确认SQLMap自动化扫描→已在WAF永久封禁+通知开发修复参数化查询→附件：攻击流量pcap+威胁情报截图" |

### 你今天学的知识在ATT&CK中的映射

花5分钟思考以下问题（有助于建立安全思维框架）：

- 今天的攻击手法在ATT&CK中属于哪个战术？对应的技术编号是什么？

- 用Kill Chain模型分析：这种攻击通常发生在哪几个阶段？

- 你的检测手段应该部署在Kill Chain的哪一环？为什么？

- 如果你负责设计针对这种攻击的防御方案，你会从哪几个层面入手？


> **记住**：蓝队不是"见招拆招"的被动防守，而是"知己知彼"的主动防御。每天学完一个知识点后，立刻思考"如果我是攻击者我会怎么用"+"如果我是防御者我该怎么挡"——这种双向思维是蓝队核心竞争力的来源。
