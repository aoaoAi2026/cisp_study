## 📖 核心知识讲解

### 一、给「寄快递」就能秒懂的 OSI 七层模型

OSI 七层模型是网络通信的理论基础。很多教程把它讲得很抽象，但用「寄快递」来类比，谁都听得懂。

---

**完整类比：你从北京给上海的朋友寄一箱苹果**

| OSI层 | 层名 | 寄快递时对应什么 | 蓝队关心的攻击 |
|:---|:---|:---|:---|
| 第7层 | **应用层** | 你写好信/选好苹果品种（生成数据） | SQL注入、XSS、文件上传 |
| 第6层 | **表示层** | 用泡沫纸把苹果包好/给信加密（格式化） | SSL/TLS配置错误 |
| 第5层 | **会话层** | 打电话说「我要寄快递了，注意收」（建立联系） | 会话劫持、Cookie窃取 |
| 第4层 | **传输层** | 贴快递单号+选保价（保证送到+不丢件） | SYN Flood、端口扫描 |
| 第3层 | **网络层** | 快递公司规划路线「北京→南京→上海」（寻路） | IP欺骗、路由劫持 |
| 第2层 | **数据链路层** | 快递小哥骑三轮车送最后一公里（局域网） | ARP欺骗、MAC泛洪 |
| 第1层 | **物理层** | 三轮车、马路、快递箱（物理介质） | 网线被拔、光纤被剪 |

---

**核心理解——每一层只干自己的事：**

```
你在第7层写好了信→第6层加密包装→第5层确认会话→第4层贴单号→第3层规划路线→第2层装车→第1层上路

关键：每一层只关心自己的活，不关心其他层。
     第3层（网络层）只管「这个包裹走哪条路到上海」，
     不关心「包裹里是苹果还是炸弹」（那是第7层的事）。

     攻击者就可以在任何一层动手脚：
     ├─ 第7层：把苹果换成毒药（SQL注入）
     ├─ 第4层：伪造单号骗保（SYN Flood）
     ├─ 第2层：冒充快递小哥（ARP欺骗）
     └─ 蓝队的工作：每一层都要盯着！
```

---

**蓝队为什么主要关注第3、4、7层？**

| 层 | 关注的协议 | 攻击手法 | 检测方法 |
|:---|:---|:---|:---|
| 第7层（应用层） | HTTP/HTTPS/DNS/FTP | SQL注入/XSS/Webshell/钓鱼DNS | WAF检测/日志分析 |
| 第4层（传输层） | TCP/UDP | 端口扫描/SYN Flood/DDoS | 流量监控/连接数告警 |
| 第3层（网络层） | IP/ICMP | IP欺骗/ICMP隧道/Smurf攻击 | 源IP校验/流量基线 |

实际上，这三层覆盖了 90% 以上的网络攻击，所以蓝队入门的重点也在这三层。

### 二、TCP 三次握手——网络中最重要的「礼仪」

如果把一次网络通信比作打电话，TCP 三次握手就是拨通电话前的「确认对方在线」的过程。

---

**图解三次握手：**

```
客户端（你）                                    服务器（朋友）
    │                                               │
    │  ① SYN = 1, Seq = x                           │
    │  「喂？在吗？我的编号是 x」                       │
    │──────────────────────────────────────────────>│
    │                                               │
    │  ② SYN = 1, ACK = 1, Seq = y, Ack = x+1      │
    │  「在的！我的编号是 y，我收到你的 x 了」            │
    │<──────────────────────────────────────────────│
    │                                               │
    │  ③ ACK = 1, Seq = x+1, Ack = y+1             │
    │  「好的，我收到你的 y 了，开始聊吧」               │
    │──────────────────────────────────────────────>│
    │                                               │
    │        连接建立 ✓  开始传输数据                   │
```

---

**三个关键标志位——永远记住：**

| 标志位 | 含义 | 记忆口诀 |
|:---|:---|:---|
| SYN | 同步（Synchronize） | **敲门**——我想建立连接 |
| ACK | 确认（Acknowledgment） | **点头**——我收到了 |
| FIN | 结束（Finish） | **道别**——我说完了 |
| RST | 重置（Reset） | **摔门**——强制断开，不解释 |

---

**为什么必须是三次，不是两次也不是四次？**

> 两次握手的问题：如果客户端发了一个 SYN，网络延迟，很久才到服务器。客户端以为丢包了，又发了一个新的 SYN。结果两个 SYN 都到了服务器，服务器以为别人要连两次。两次握手的话，服务器收到第一个 SYN 就认为连接建立了 — 但客户端早就不等了，这个连接就成了「幽灵连接」。
>
> 三次握手就解决了这个问题：服务器回复 SYN+ACK → 客户端必须回复 ACK → 服务器才确认连接。如果客户端不回复 ACK，服务器就知道「哦，这个请求过期了」。

---

**蓝队视角：三次握手的三个重要检测场景**

**场景一：SYN Flood 攻击检测**

```
攻击者：发海量 SYN 包 → 服务器回 SYN+ACK → 攻击者不回 ACK
        服务器为每个半连接保留资源 → 很快资源耗尽 → 正常用户连不上

检测方法：
├─ 监控「半连接队列」大小
├─ 观察 SYN 包速率是否异常（正常网站每秒几十个，攻击时几千个）
└─ netstat -s | grep "SYN_RECV"
```

**场景二：端口扫描检测**

```
攻击者：向目标 IP 的 1-65535 端口逐一发 SYN 包
        如果端口开放 → 回 SYN+ACK → 攻击者就知道这个端口开着

检测方法：同一个源 IP 短时间内连接大量不同端口 → 端口扫描
```

**场景三：连接方向判断**

```
SYN 包的方向告诉你「谁主动连的谁」
├─ 内网 IP → 外网 IP 有大流量上传 → 可能在窃取数据
├─ 外网 IP → 内网 22 端口有 SYN → 有人在尝试 SSH 登录
└─ 凌晨 4 点的大量外→内连接 → 非常可疑
```

### 三、TCP 四次挥手——「说再见也要讲礼貌」

连接完了要断开，TCP 要求双方都点头同意才能断。但因为双方可能都有话说，所以要四次：

```
客户端                                         服务器
    │                                               │
    │  ① FIN = 1, Seq = u                          │
    │  「我说完了，准备挂电话了」                       │
    │──────────────────────────────────────────────>│
    │                                               │
    │  ② ACK = 1, Ack = u+1                        │
    │  「好的，我知道你说完了」                          │
    │<──────────────────────────────────────────────│
    │                                               │
    │  （服务器可能还有话要说，不能说断就断）              │
    │                                               │
    │  ③ FIN = 1, ACK = 1, Seq = v, Ack = u+1      │
    │  「那我也说完了，挂吧」                           │
    │<──────────────────────────────────────────────│
    │                                               │
    │  ④ ACK = 1, Ack = v+1                        │
    │  「好的，拜拜」                                  │
    │──────────────────────────────────────────────>│
    │                                               │
    │  客户端等 2MSL（约60秒）后彻底关闭                │
    │  服务器收到 ACK 后立即关闭                       │
```

**为什么不是三次？** 因为 TCP 是全双工的（双方可以同时说话）。你闭了你的嘴，对方还有话没说完，得等他也闭了嘴才算完。不能两个 FIN 合并成一个。

**蓝队关注的异常挥手场景：**
- 大量 RST 包 → 连接被强制断开（攻击者在测试，或者防火墙在掐连接）
- FIN 扫描 → 攻击者用 FIN 包探测端口（隐蔽扫描，不发 SYN）
- 半开连接大量存在 → 可能是 SYN Flood 攻击或者服务配置问题

### 四、16 个你必须秒答的端口号

蓝队工作中，看到端口要立刻反应出对应服务。这些是最高频的：

**基础服务端口（最常被攻击）：**

| 端口 | 服务 | 一句话记忆 | 蓝队特别注意事项 |
|:---|:---|:---|:---|
| **21** | FTP | 二姨(21)传文件 | 弱口令重灾区，传输文件不加密 |
| **22** | SSH | 双双(22)远程连 | **暴力破解第一目标！** |
| **23** | Telnet | 乔丹(23)打篮球 | 明文密码！必需禁用，换 SSH |
| **25** | SMTP | 二五(25)发邮件 | 可能被利用发钓鱼邮件/垃圾邮件 |
| **53** | DNS | 巫山(53)域名解析 | DNS 隧道可能被用来偷数据 |
| **80** | HTTP | 八零(80)看网页 | Web 攻击主入口，每个请求都要过 |
| **110** | POP3 | 妖妖灵(110)收邮件 | 收邮件协议，不加密有风险 |
| **443** | HTTPS | 四四三加密浏览 | 加密流量更难检测，但仍是攻击入口 |

**数据库与中间件端口（内网最爱打）：**

| 端口 | 服务 | 蓝队特别注意事项 |
|:---|:---|:---|
| **1433** | MSSQL | SQL Server，Windows 环境下常被攻击 |
| **1521** | Oracle | 企业级数据库，版本旧可能有漏洞 |
| **3306** | MySQL | **暴破第二目标**，root 弱口令 = 数据库被拖走 |
| **3389** | RDP | **远程桌面暴破头号目标**，千万不能暴露到公网 |
| **5432** | PostgreSQL | 开源数据库，配置漏洞集中在 pg_hba.conf |
| **6379** | Redis | **默认无密码**，攻破后写入 SSH 公钥即拿 shell |
| **27017** | MongoDB | NoSQL 数据库，默认无认证 |
| **9200** | Elasticsearch | 搜索引擎，配置暴露可能导致数据泄露 |

**蓝队端口速查命令：**

```bash
# Windows：查看本机监听的端口
netstat -ano | findstr "LISTENING"

# Linux：查看本机监听的端口
ss -tlnp
# 或传统命令
netstat -tlnp

# 查看特定端口的连接（如22端口）
ss -tnp | grep ':22 '

# 查看所有已建立的TCP连接
ss -tn state established
```

**真实案例记忆**：某次护网中，蓝队发现一台 Web 服务器的 6379 端口暴露在公网且无密码，红队利用 Redis 写入了 SSH 公钥，直接拿到服务器 root 权限。端口暴露 + 无认证 = 被拿 shell 只是时间问题。

### 五、Wireshark 实战——亲手抓一次三次握手

理论看完，马上动手验证。这是加深理解最有效的方法。

**步骤：**

```
1. 打开 Wireshark，选择你正在上网的网卡（通常是 Wi-Fi 或 Ethernet）

2. 在过滤器栏输入：tcp.port == 80
   （只显示 HTTP 相关的 TCP 流量）

3. 打开浏览器，访问 http://example.com
   （注意：用 HTTP 而不是 HTTPS，这样能看到明文流量）

4. 在 Wireshark 中，你应该能看到：
   你的IP → 93.184.216.34  [SYN]      ← 第1步：敲门
   93.184.216.34 → 你的IP  [SYN, ACK] ← 第2步：应答
   你的IP → 93.184.216.34  [ACK]      ← 第3步：确认

5. 展开第一个包，查看 TCP 段的详细信息：
   ├─ Source Port: 54321（你的随机端口）
   ├─ Destination Port: 80（HTTP端口）
   ├─ Flags: 0x002 (SYN)
   └─ Sequence Number: 随机值（用于后续确认）
```

**更高级的过滤器语法（记下来，天天用）：**

```bash
# 只看某个 IP 的流量
ip.addr == 192.168.1.100

# 只看某个端口的流量
tcp.port == 80

# 只看 SYN 包（只看敲门动作）
tcp.flags.syn == 1 && tcp.flags.ack == 0

# 组合：某 IP 的 SYN 包
ip.src == 10.0.0.5 && tcp.flags.syn == 1 && tcp.flags.ack == 0

# 只看 RST 包（摔门——不正常断连）
tcp.flags.reset == 1

# 跟踪一个完整的 TCP 流（看完整对话）
右键某个包 → Follow → TCP Stream
```

### 六、8 种常见网络攻击——分别发生在哪一层？

| 攻击 | OSI层 | 攻击原理（大白话） | 蓝队检测方法 |
|:---|:---|:---|:---|
| **ARP 欺骗** | 第2层 | 冒充网关的 MAC 地址，截获流量 | 监控 ARP 表变化、MAC 与 IP 绑定 |
| **IP 欺骗** | 第3层 | 伪造源 IP 地址，隐藏真实身份 | uRPF 反向路径转发检查 |
| **ICMP 隧道** | 第3层 | 把数据藏在 ping 包里偷运出去 | 监控异常 ICMP 包大小（正常<100字节） |
| **SYN Flood** | 第4层 | 海量 SYN 包撑爆连接池 | 监控半连接数 + SYN 速率 + SYN Cookie |
| **UDP Flood** | 第4层 | 海量 UDP 包占满带宽 | 流量基线对比、异常 UDP 比例检测 |
| **DNS 隧道** | 第7层 | 用 DNS 查询偷运数据（TXT 记录塞数据） | DNS 查询频率异常、请求域名长度异常 |
| **HTTP 慢速攻击** | 第7层 | 极慢地发 HTTP 请求头，耗尽连接 | 连接超时设置、反向代理限速 |
| **SSL/TLS 降级** | 第6层 | 强迫使用弱加密，便于窃听 | 强制 TLS 1.2+、禁用弱加密套件 |

### 七、新手最容易犯的五个错误

1. **分不清 TCP 和 UDP 的区别**
   TCP 像打电话——先接通再说话，确保对方收到。UDP 像发传单——扔出去不管别人接没接到。端口扫描主要用 TCP（UDP 扫描慢且不准），但 DNS、视频流等用 UDP。

2. **不知道「端口号」可以同时用于 TCP 和 UDP**
   TCP 53 和 UDP 53 是两个不同的端口！DNS 通常用 UDP 53，但大响应或区域传输会切到 TCP 53。

3. **以为封了端口就安全了**
   80 端口开着就有风险，443 端口开着也有风险。安全是纵深防御，不是封端口那么简单。

4. **不会用 Wireshark 过滤器**
   面对海量包不知道怎么筛选。「不会过滤 = 不会用 Wireshark」。上面给的过滤语法，背过它。

5. **只看不练**
   理论看三遍不如亲手抓一次包。打开 Wireshark，现在就抓一个 TCP 握手看看。

## 🎯 蓝队面试高频题（Day 2 主题）

**Q1：TCP为什么是三次握手而不是两次或四次？**

> 两次握手的致命缺陷：客户端发SYN，服务器回SYN+ACK后就认为连接已建立→但客户的第一个SYN可能因网络延迟滞留→服务器为"幽灵请求"保留了资源→浪费服务器连接池。 三次握手让双方都确认对方在线：客户发SYN(我活着)→服务器回SYN+ACK(我活着且收到了你)→客户回ACK(我收到了你的确认)→双向确认完毕。

> 四次没必要的原因：三步已经完成了双向确认，多了浪费。但**断开连接必须四次**，因为TCP是双向独立的——你闭了嘴不代表对方也说完了。

**Q2：SYN Flood攻击怎么防御？**

> 核心思路：不等三次握手完成就分配资源，而是采用SYN Cookie技术——服务器不保存半连接状态，而是把连接信息编码到SYN+ACK的序列号里，只有收到客户端的最终ACK时才重建连接状态。另外还有：SYN Proxy（防火墙代答）、限流（同一IP限制SYN速率）、清洗（流量过DDoS设备）。

**Q3：如何从Wireshark中快速识别端口扫描行为？**

> 过滤条件：`tcp.flags.syn == 1 && tcp.flags.ack == 0` → 只看SYN包。 特征：同一源IP短时间内向目标IP的大量不同端口（1-65535）发SYN包。如果在10秒内同一源IP连接了超过50个不同端口→高度可疑。 另外还可以看：大量RST包（端口关闭回的拒绝）+ 被拒绝端口的连续性（1,2,3,4...连续扫描）。

## 📖 深度阅读：网络协议中的"蓝队直觉"

蓝队看网络流量就像医生看化验单——外行看一堆数字，内行一眼看出异常。以下是培养这种直觉的方法：

**1. 先建立"正常基线"**
- 你的网络里，正常工作日每小时有多少TCP连接？
- 正常的DNS查询都是什么域名？（baidu.com、google.com 等正常，随机字符串域名异常）
- 正常的外连目标IP在什么地区？（中国境内正常，突然大量连境外要警觉）

**2. 再识别"异常模式"**
- 连接数突然翻10倍→有人在扫描或DDoS
- 外连突然增多且目标IP奇怪→可能是后门在回连C2
- DNS查询出现`.xyz` `.top` `.tk` 等可疑后缀→可能是DGA域名生成算法产生的恶意域名

**3. 最后形成"条件反射"**
- 看到大量SYN→第一反应：端口扫描还是SYN Flood？
- 看到内网IP连外网稀有端口→第一反应：C2通信？
- 看到DNS查询极长的域名→第一反应：DNS隧道？

## ⚠️ 新手常见误区纠正

1. **误区**："TCP和UDP的区别就是TCP可靠，UDP不可靠"
   - **更准确**：TCP有连接、有序、确认机制、流量控制；UDP无连接、不确认、不排序、不控速。DNS用UDP是因为一次查询就几十字节，不值得建TCP连接。但DNS区域传输用TCP因为数据量大。可靠≠好，不可靠≠差——看场景。
   
2. **误区**："端口号只是数字，记不记得无所谓"
   - **真相**：在蓝队工作中，端口号就是你识别服务的"车牌号"。看到22就是SSH→可能被暴力破解。看到3389就是RDP→绝不能暴露公网。看到6379就是Redis→默认无密码。不认识端口=看不懂网络在干什么。

3. **误区**："OSI七层太理论了，实战用不上"
   - **真相**：分层思维是你诊断问题的框架。攻击发生在第4层（SYN Flood）→在防火墙/流量层排查，和第7层没关系。攻击发生在第7层（SQL注入）→看WAF和应用日志，和网络层没关系。**不懂分层，你就在错误的地方找问题。**

## 🎓 进阶理解：UDP协议——TCP的"弟弟"也要认识

虽然TCP是蓝队最关注的协议，但UDP同样重要：

| 维度 | TCP | UDP |
|:---|:---|:---|
| 连接 | 有连接（三次握手） | 无连接（直接发） |
| 可靠性 | 确认重传机制 | 不确认，丢了就丢了 |
| 速度 | 较慢（握手开销） | 快（无握手开销） |
| 数据顺序 | 严格有序 | 不保证顺序 |
| 典型应用 | HTTP/SSH/邮件 | DNS/视频/游戏/VoIP |
| 蓝队关注 | 暴破、扫描、C2回连 | DNS隧道、UDP Flood |

**蓝队必须知道的UDP攻击场景：**

1. **DNS隧道**：攻击者把数据编码在DNS查询中通过UDP 53端口传出。因为大多数防火墙放行UDP 53（DNS必需），所以这是最常见的C2隐蔽通道。
   检测：DNS查询中出现异常长的域名（>52字符）、同一域名高频查询、TXT记录查询暴增。

2. **UDP Flood DDoS**：攻击者发送海量UDP包（通常伪造源IP），占满目标带宽。因为UDP无连接，源IP可以任意伪造。
   检测：入站UDP流量突增超过基线5倍以上 → 可能是UDP Flood。

3. **NTP/DNS放大攻击**：攻击者伪造受害者IP向NTP/DNS服务器发小请求，服务器回几十倍大的响应→间接DDoS。
   检测：来自53端口（DNS）或123端口（NTP）的大量入站流量（正常应该是出站请求入站响应很快）。

## 📖 深度补充内容

### 💡 面试高频题：OSI七层与TCP/IP

**Q1: 为什么蓝队必须理解OSI七层模型？**
A: 因为攻击发生在每一层。DDoS打第三/四层（网络/传输层），Web攻击打第七层（应用层），ARP欺骗打第二层（数据链路层）。不理解攻击发生在哪一层，就不知道该用什么手段防御。

**Q2: TCP三次握手如果只完成两次会怎样？**
A: 这就是SYN Flood攻击的原理。攻击者发送大量SYN包但不回应服务端的SYN-ACK，导致服务端半连接队列被占满，正常用户无法建立连接。蓝队通过SYN Cookie、连接超时调整、流量清洗来防御。

**Q3: 如何从Wireshark抓包中判断是否有端口扫描？**
A: 典型特征——同一源IP短时间内向大量目标端口发送SYN包，且多数收到RST（端口关闭）或无响应（被过滤）。过滤语句：`tcp.flags.syn==1 and tcp.flags.ack==0`。

**Q4: UDP协议在安全分析中有什么特殊意义？**
A: UDP无连接特性使其成为DNS放大攻击、NTP放大攻击的载体。蓝队需特别关注：53端口的异常大流量UDP（DNS隧道）、123端口的大响应包（NTP放大）、以及非标准端口的UDP流量（可能是C2通信）。

**Q5: 为什么说"没多少人真正理解MTU和分片"？**
A: 因为很多网络攻击利用分片来绕过检测——将恶意payload分散在多个小分片中，防火墙/WAF如果只检查单个分片就会漏过。蓝队需要在IDS/IPS中开启分片重组功能。

### 📊 实操扩展：用tcpdump抓包分析TCP状态机

在蓝队日常工作中，tcpdump比Wireshark更常用（服务器通常没有图形界面）：

```bash
# 抓取某IP的所有TCP流量
tcpdump -i eth0 host 192.168.1.100 -w capture.pcap

# 只看SYN包（发现扫描行为）
tcpdump -i eth0 'tcp[tcpflags] & tcp-syn != 0 and tcp[tcpflags] & tcp-ack == 0'

# 统计每个源IP的连接数（发现异常连接）
tcpdump -r capture.pcap 'tcp[tcpflags] & tcp-syn != 0' | awk '{print $3}' | cut -d. -f1-4 | sort | uniq -c | sort -rn | head -20

# 查看HTTP请求（排查Web攻击）
tcpdump -i eth0 -A 'tcp port 80 and (tcp[((tcp[12:1] & 0xf0) >> 2):4] = 0x47455420)'
```
