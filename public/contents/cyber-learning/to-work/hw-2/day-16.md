## 📖 核心知识讲解

### 一、IDS vs IPS —— 摄像头 vs 自动门禁

这是很多新手容易搞混的概念，用一个比喻彻底讲清楚：

| | IDS（入侵检测系统） | IPS（入侵防御系统） |
|:---|:---|:---|
| 通俗比喻 | 📹 监控摄像头 | 🚪 自动门禁 |
| 发现攻击时 | 发出警报（滴滴滴！） | 自动阻止（门直接锁上） |
| 部署方式 | 旁路（在旁边看着，不影响流量） | 串联（流量必须经过它） |
| 对业务的影响 | 无影响（只看不碰） | 可能误拦正常流量 |
| 响应方式 | 通知人去处理 | 自动拦截 |

> 记忆口诀：**IDS只会叫，IPS可以拦。** IDS = Detection（检测），IPS = Prevention（阻止）。

**实际工作中的部署组合：**
```
外网 → IPS（第一道自动拦截）→ 防火墙 → IDS（旁路监听+告警）→ 服务器
                                            ↓
                                        SOC平台展示告警
```

### 二、IDS怎么发现攻击？（三种检测方法）

#### 1. 基于特征（Signature-based）—— 最常用
就像一个"通缉犯名单"，把已知攻击的特征写成规则，匹配到就告警。

```
规则示例（通俗版）：
如果 数据包中包含 "OR '1'='1" → 告警：SQL注入攻击
如果 同一IP每秒发超过10个SYN包 → 告警：端口扫描
如果 HTTP请求中包含 "/etc/passwd" → 告警：路径遍历攻击
```

**优点：** 准确率高，误报低
**缺点：** 新攻击（没在名单上）检测不到

#### 2. 基于异常（Anomaly-based）
先学习"正常情况"是什么样，发现偏离正常的行为就告警。

```
比喻：你室友每天作息是 7点起 → 12点吃饭 → 11点睡
      突然有一天凌晨3点出门 → 异常！值得注意
```

**优点：** 能发现未知攻击
**缺点：** 误报率高（加班到凌晨也会报警）

#### 3. 基于协议状态（Protocol State Analysis）
检查协议交互是否符合标准流程。

```
正常HTTP：请求 → 响应
异常情况：在握手阶段就发数据 → 可疑
```

### 三、Snort —— 最经典的开源IDS

Snort是IDS领域的"活化石"，免费开源，理解它就能理解所有IDS产品的核心逻辑。

**Snort的工作模式：**
```
                     ┌──────────────┐
网络流量 → 解码器 → 预处理器 → 检测引擎 → 告警输出
           │          │          │
        拆包还原    整理分析    规则匹配
```

**一条Snort规则长什么样？**

```
alert tcp $EXTERNAL_NET any -> $HTTP_SERVERS 80 (msg:"SQL Injection"; content:"' OR '1'='1"; sid:1000001;)
│      │  │               │    │               │                          │                    │
告警   协议 来源(外网)   端口  目标(Web服务器)  端口  告警信息                匹配内容            规则ID
```

> 蓝队不需要精通写规则，但需要能**读懂规则**，知道它在检测什么。

### 四、常见IDS告警类型

蓝队在SOC平台上看到的告警，按攻击类型分类：

| 告警类型 | 典型示例 | 紧急程度 |
|:---|:---|:---|
| **扫描/探测** | 端口扫描、目录扫描、漏洞扫描 | 🟡 中（通常是攻击前奏） |
| **漏洞利用** | SQL注入、命令注入、文件包含 | 🔴 高（正在攻击） |
| **Webshell** | 中国菜刀、蚁剑通信流量 | 🔴 严重（已被控制） |
| **恶意软件** | 木马回连C2、挖矿通信 | 🔴 严重（已被植入） |
| **异常流量** | DDoS攻击、DNS隧道 | 🔴 高 |
| **认证攻击** | 暴力破解、密码喷洒 | 🟡 中（需确认是否成功） |
| **策略违规** | 访问敏感网站、使用禁止协议 | 🟢 低 |

### 五、IDS告警的蓝队处理优先级

```
收到IDS告警 → 第1步：看告警等级
                 ├── Critical/High → 立即处理（5分钟内响应）
                 ├── Medium        → 30分钟内处理
                 └── Low/Info      → 记录在案，定期清理

             → 第2步：确认是否为真实攻击
                 ├── 查源IP情报（VirusTotal/微步）
                 ├── 查看关联日志
                 └── 判断攻击是否成功

             → 第3步：处置
                 ├── 真实攻击 → 封IP、排查受害系统、记录工单
                 └── 误报 → 标注原因、优化规则
```

**Q2：Snort规则的基本格式是什么？**

> Snort规则由"规则头"和"规则选项"组成：
> ```
> alert tcp any any -> 192.168.1.0/24 80 (msg:"SQL Injection"; content:"union select"; nocase; sid:1000001;)
>  │     │   │   │        │            │    │                  │
>  │     │   │   │        │            │    │                  └─ 具体匹配内容
>  │     │   │   │        │            │    └─ 规则选项（检测逻辑）
>  │     │   │   │        │            └─ 规则头（什么流量）
>  │     │   │   │        └─ 目标网络和端口

>  │     │   └─ 源IP（any=任意）
>  │     └─ 协议（tcp/udp/icmp）
>  └─ 动作（alert=告警, drop=丢弃, pass=放行）
> ```

## 📖 深度阅读：IDS的"漏报"和"误报"——蓝队永恒的难题

IDS有两大难题困扰所有蓝队：

**漏报（False Negative）——攻击发生了但没检测到：**
- 原因1：规则库没覆盖新型攻击（0day）
- 原因2：攻击者用绕过技术（编码、分片、加密流量）
- 原因3：IDS部署位置看不到部分流量（比如东西向流量）
- 解决：定期更新规则库 + 结合全流量分析 + 不依赖单一设备

**误报（False Positive）——正常行为被当攻击告警：**
- 原因1：规则太宽泛（比如"URL含单引号就告警"→正常输入也有单引号）
- 原因2：业务特殊行为（比如内网IP大量访问数据库→正常的数据处理）
- 原因3：设备调试/测试行为
- 解决：调优规则（白名单+阈值）+ 告警分级 + 关联分析

> **核心认知**：不存在"0误报0漏报"的IDS。蓝队的价值不是"没有误报"，而是"在可接受的误报率下尽可能多地发现真实威胁"。这需要持续调优。

## 💡 面试高频题：IDS/IPS入侵检测

**Q: IDS和IPS的区别是什么？**
A: IDS（入侵检测系统）=旁路部署，只检测不阻断，像"监控摄像头"——看到坏人会报警但不拦人。IPS（入侵防御系统）=串联部署，检测到攻击后直接阻断，像"安检门"——检测到违禁品就把人拦在外面。实际部署时：内网核心区域用IPS（实时阻断），非核心区域用IDS（降低对业务的影响），两者通常是同一设备的不同工作模式。

**Q: IDS/IPS的检测原理是什么？**
A: ①特征匹配（Signature-based）：与已知攻击特征库逐条比对，类似于杀毒软件的病毒库——优点是准确率高（低误报），缺点是无法检测未知攻击（0day）；②异常检测（Anomaly-based）：先学习"正常流量基线"，偏离基线的流量就是可疑的——优点是可检测未知攻击，缺点是误报率高（正常业务波动也可能触发告警）；③协议分析（Protocol Analysis）：检查协议是否符合RFC标准——如HTTP请求头格式异常、DNS报文构造不规范等。

**Q: Snort的规则格式是什么样的？**
A: Snort规则格式：`action protocol src_ip src_port direction dst_ip dst_port (options)`。例如：`alert tcp any any -> 192.168.1.100 80 (msg:"SQL Injection Detected"; content:"union select"; nocase; sid:1000001;)`。这条规则的意思是：任何源IP/端口→192.168.1.100的80端口→TCP流量→如果内容中包含"union select"（不区分大小写）→产生告警。蓝队需要具备阅读和编写Snort规则的能力。

**Q: 蓝队如何优化IDS/IPS规则以降低误报？**
A: ①添加白名单：明确排除已知的正常流量源（如CDN节点IP）；②细化匹配条件：不只是匹配关键字，还要限定出现的位置（如仅在URL参数中匹配，不在Cookie中匹配）；③关联分析：单条告警可能是误报，但多条低置信度告警同时出现=高置信度攻击；④定期review无效规则：如果某条规则一个月没触发过真实攻击，评估是否需要调整或删除。

**Q: 常见的IDS告警类型有哪些？**
A: ①扫描探测类：端口扫描、主机发现、服务识别；②Web攻击类：SQL注入、XSS、文件包含、命令注入；③恶意软件类：已知恶意域名解析、恶意软件通信特征；④暴力破解类：SSH/FTP/RDP登录失败集中出现；⑤异常协议类：DNS隧道、ICMP隧道、非标准端口HTTP；⑥漏洞利用类：已知CVE的攻击流量特征。蓝队需要能快速判断每条告警的真实威胁等级。

## 🔬 进阶专题：本日知识在真实护网中的应用

### 护网场景还原

想象你正在护网值守中，突然收到一条与本日主题相关的告警。以下是标准的思维和处理流程：

**1. 第一反应（0-30秒）**：确认告警来源设备是否正常 → 查看告警级别 → 判断是否为"狼来了"

**2. 快速研判（30秒-2分钟）**：提取关键信息（源IP/目标资产/攻击类型/时间）→ 查威胁情报 → 看同一IP的其他告警

**3. 深度分析（2-10分钟）**：回放原始流量/日志 → 还原攻击payload → 判断攻击是否成功

**4. 处置决策（10-15分钟）**：确认是攻击→定级→封IP/隔离/升级事件；确认是误报→标记→优化规则
