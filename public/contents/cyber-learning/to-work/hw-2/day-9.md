## 📖 核心知识讲解

### 一、Web访问日志长什么样？

Web服务器（Nginx/Apache）会把每一次访问都记录下来。一条典型的日志长这样：

```
192.168.1.100 - - [10/Jun/2026:14:35:22 +0800] "GET /admin/login.php HTTP/1.1" 200 4521 "https://www.example.com/" "Mozilla/5.0 (Windows NT 10.0; Win64; x64)"
```

看起来很乱？我们把它拆开来：

| 字段 | 示例值 | 含义 | 通俗解释 |
|:---|:---|:---|:---|
| `$remote_addr` | `192.168.1.100` | 客户端IP | 谁访问的 |
| `-` | `-` | 远程用户（一般没有） | 不用管 |
| `-` | `-` | 认证用户（一般没有） | 不用管 |
| `$time_local` | `[10/Jun/2026:14:35:22]` | 访问时间 | 什么时候来的 |
| `$request` | `"GET /admin/login.php HTTP/1.1"` | 请求行 | 要干什么 |
| `$status` | `200` | HTTP状态码 | 结果如何 |
| `$body_bytes_sent` | `4521` | 响应大小(字节) | 回了多少数据 |
| `$http_referer` | `"https://www.example.com/"` | 从哪个页面跳过来的 | 上一站在哪 |
| `$http_user_agent` | `"Mozilla/5.0..."` | 浏览器/爬虫标识 | 用什么浏览器 |

> 🎯 **蓝队分析时最看重的3个字段**：客户端IP、请求内容、状态码。就这三个字段配合起来，能发现90%的攻击行为。

### 二、日志分析的"黄金三步法"

不管给你多大的日志文件，分析思路都一样：

```
第一步：宏观统计（把握全局）
  → 总共有多少条请求？
  → 什么时间段最活跃？
  → 哪些IP访问最多？

第二步：异常发现（找可疑点）
  → 谁在产生大量404？
  → 谁的请求参数很奇怪？（引号、SQL关键字等）
  → 谁在深夜大量访问？

第三步：深度钻取（聚焦攻击）
  → 可疑IP都干了什么？
  → 攻击成功了吗（状态码200还是403）？
  → 需要封禁和处理吗？
```

### 三、用Linux命令分析日志的"标准工具箱"

假设日志文件叫 `access.log`，以下是你最常用的分析命令：

#### 1. 宏观统计

```bash
# 总共有多少条日志
wc -l access.log

# 访问最多的前10个IP
cat access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head -10

# 状态码分布统计
cat access.log | awk '{print $9}' | sort | uniq -c | sort -rn

# 访问最多的前10个页面
cat access.log | awk '{print $7}' | sort | uniq -c | sort -rn | head -10

# 每小时的访问量分布（发现异常时间窗口）
cat access.log | awk '{print $4}' | cut -d: -f2 | sort | uniq -c
```

#### 2. 发现异常

```bash
# 产生最多404的IP（可能在扫目录）
cat access.log | grep " 404 " | awk '{print $1}' | sort | uniq -c | sort -rn | head

# 产生最多403的IP（可能在越权访问）
cat access.log | grep " 403 " | awk '{print $1}' | sort | uniq -c | sort -rn | head

# 产生最多500的请求（可能攻击导致服务器崩溃）
cat access.log | grep " 500 " | awk '{print $7}' | sort | uniq -c | sort -rn | head

# 使用非标准User-Agent的请求（可能是扫描器）
cat access.log | grep -v "Mozilla" | awk '{print $1, $NF}'
```

#### 3. 聚焦单个可疑IP

```bash
# 看看这个IP都访问了什么（汇总）
grep "192.168.1.100" access.log | awk '{print $7}' | sort | uniq -c

# 看看这个IP的访问时间范围
grep "192.168.1.100" access.log | head -1
grep "192.168.1.100" access.log | tail -1

# 看看这个IP产生了哪些状态码
grep "192.168.1.100" access.log | awk '{print $9}' | sort | uniq -c

# 导出这个IP的所有访问（方便进一步分析）
grep "192.168.1.100" access.log > suspect_ip.txt
```

### 四、攻击特征的日志识别

#### 1. SQL注入特征

攻击者在URL参数里插入SQL语句，日志里能看到明显的SQL关键字：

```
# 可疑日志示例：
192.168.1.100 - - [10/Jun/2026:15:00:00] "GET /product.php?id=1' OR '1'='1 HTTP/1.1" 200 1234
192.168.1.100 - - [10/Jun/2026:15:00:05] "GET /product.php?id=1 UNION SELECT 1,2,3 HTTP/1.1" 200 2345
192.168.1.100 - - [10/Jun/2026:15:00:10] "GET /product.php?id=1' AND 1=2 HTTP/1.1" 200 567
```

**识别特征：**
- URL中包含 `'`、`"`、`--` 
- URL中包含 `UNION`、`SELECT`、`OR 1=1`、`AND 1=2`
- 请求中带有 `' OR '1'='1`

**用grep快速排查：**
```bash
grep -iE "(union.*select|or.*1=1|and.*1=2|select.*from)" access.log
```

#### 2. 目录/文件扫描特征

攻击者用工具暴破网站目录，特点是短时间内大量404：

```
192.168.1.100 - - [10/Jun/2026:15:00:01] "GET /admin HTTP/1.1" 404 -
192.168.1.100 - - [10/Jun/2026:15:00:02] "GET /wp-admin HTTP/1.1" 404 -
192.168.1.100 - - [10/Jun/2026:15:00:03] "GET /phpmyadmin HTTP/1.1" 404 -
192.168.1.100 - - [10/Jun/2026:15:00:04] "GET /backup.zip HTTP/1.1" 404 -
...（几百条类似的404连续出现）
```

**识别特征：**
- 同一IP连续大量404
- 请求路径包含常见敏感目录名（admin/backup/test/phpmyadmin等）
- 请求间隔极短（零点几秒）

**排查方法：**
```bash
# 找出404最多的IP
grep " 404 " access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head

# 看这个IP都访问了什么目录
grep "可疑IP" access.log | awk '{print $7}' | sort -u
```

#### 3. XSS（跨站脚本）特征

```
GET /search?q=<script>alert(1)</script> HTTP/1.1
GET /comment?msg=<img src=x onerror=alert(1)> HTTP/1.1
```

**识别特征：**
- URL中包含 `<script>`
- URL中包含 `javascript:`
- URL中包含 `onerror=`、`onload=` 等事件处理

#### 4. 暴力破解特征

```
192.168.1.100 - - [10/Jun/2026:15:00:00] "POST /login HTTP/1.1" 401 -
192.168.1.100 - - [10/Jun/2026:15:00:01] "POST /login HTTP/1.1" 401 -
192.168.1.100 - - [10/Jun/2026:15:00:02] "POST /login HTTP/1.1" 401 -
...（POST请求 + 连续401 + 同一IP）
```

## 🎯 蓝队面试高频题（Day 9 主题）

**Q1：Nginx access log 的每一列代表什么？**

> 以 Combined Log Format 为例：
> ```
> 192.168.1.1 - admin [10/Jun/2026:14:30:00 +0800] "GET /admin HTTP/1.1" 403 1234 "https://example.com" "Mozilla/5.0"
>   $1=IP    $2 $3=$user $4=时间              $5=请求行               $6=状态码 $7=大小 $8=Referer         $9=User-Agent
> ```
> 注意：$2 和 $3 如果用 `-` 表示，是 HTTP Basic Auth 的用户名（现在几乎不用了）。

**Q2：从access.log中如何快速发现SQL注入攻击？**

> 三个特征：
> 1. URL参数中出现单引号 `'` → SQL注入的"起手式"
> 2. URL参数中出现 SQL 关键字：`UNION` `SELECT` `OR 1=1` `--` `#`
> 3. 请求返回了异常多的数据（响应大小比正常大几十倍→可能UNION SELECT拖了整张表）
> 
> 检测命令：
> ```bash
> grep -iE "union|select|' or |--" access.log
> # -i 忽略大小写，-E 启用扩展正则，| 表示"或"
> ```

**Q3：如何区分"正常爬虫"和"恶意扫描"？**

> | 特征 | 正常爬虫（如百度蜘蛛） | 恶意扫描 |
> |:---|:---|:---|
> | User-Agent | 明确标识（Baiduspider/Googlebot） | 可能伪装浏览器或无UA |
> | 来源IP | 可以反向DNS验证（属于百度/Google网段） | 随机IP/VPN/代理 |
> | 访问频率 | 均匀、有节制 | 突发、极高频率 |
> | 访问路径 | 按链接爬取 | 固定路径列表（/admin/ /wp-admin/ /.env ..） |
> | robots.txt | 遵守（会先读robots.txt） | 完全不遵守 |
> | 状态码分布 | 以200/301为主 | 大量404（探测不存在的路径） |
> 
> **验证方法**：对可疑爬虫IP做反向DNS查询（`host IP`）或查whois信息——真实的搜索引擎爬虫IP可以验证。

## 📖 深度阅读：日志分析——蓝队的第一核心能力

日志分析不是你工作中"偶尔做"的事，而是蓝队工作中频率最高的活动。以下是建立日志分析能力的系统框架：

**日志分析的三个层次：**

**Level 1：事件定位（"有什么异常？"）**
- 收到告警→去对应日志里找具体记录
- 比如：告警说"SQL注入"→到access.log里找对应的请求行
- 能力要求：知道去哪里找哪种日志、用grep/awk快速定位

**Level 2：关联分析（"这个异常和别的有关系吗？"）**
- 不只盯一条日志，而是串联多源日志
- 比如：access.log出现可疑请求→secure.log是否有同IP的SSH登录→cron是否有新增任务
- 能力要求：理解攻击链（Kill Chain）、跨日志源的关联思维

**Level 3：趋势洞察（"最近攻击趋势是什么？"）**
- 从长期日志中总结攻击趋势
- 比如：最近SQL注入减少、但XSS和CSRF攻击增多→建议WAF规则调整
- 能力要求：数据统计思维、报告撰写能力

> **你现在在Level 1**。Day 9-14的目标是熟练掌握Level 1，Day 15-28逐步进入Level 2。Level 3是中级蓝队（Day 61-90）的能力。

## ⚠️ 新手常见误区纠正

1. **误区**："分析日志就是grep关键词"
   - **补充**：grep是第一步。完整流程是：grep找到可疑记录→awk提取关键字段→sort|uniq -c统计→结合业务上下文判断→写出结论。grep只是"找到"，"看懂"才是核心能力。
   
2. **误区**："日志文件太大了根本分析不完"
   - **真相**：你不是要"看完"日志，而是要用统计手段"找到异常"。就好比你不需要看完整个城市的监控录像，而是先拉统计报表——哪条街异常人多？哪个时间段流量高？统计工具帮你缩小范围。

3. **误区**："Web日志分析就是看access.log"
   - **补充**：access.log是"请求记录"，error.log是"错误记录"——攻击往往在error.log中留下更明显的痕迹（PHP报错、权限错误等）。两者必须一起看。access.log告诉你"攻击者做了什么"，error.log告诉你"攻击造成了什么影响"。

### 面试高频题与实战分析

## 💡 面试高频题：Web访问日志分析

**Q: Nginx和Apache的日志格式有什么区别？**
A: Nginx默认combined格式：`$remote_addr - $remote_user [$time_local] "$request" $status $body_bytes_sent "$http_referer" "$http_user_agent"`。Apache默认common格式少Referer和User-Agent。蓝队分析时重点关注：$remote_addr（源IP）、$request（请求方法和路径，含攻击payload）、$status（状态码）、$body_bytes_sent（响应大小，拖库时异常大）。

**Q: 如何在日志中识别SQL注入攻击？**
A: 关键词检测——`union`, `select`, `from`, `sleep(`, `benchmark(`, `information_schema`, `concat(`, `char(`, `0x`（十六进制编码）等。用grep: `grep -iE "(union.*select|information_schema|sleep\(|benchmark\()" access.log`。但要排除误报（如URL中正常包含"select"的业务词）。

**Q: 如何在日志中识别目录扫描/爬虫？**
A: 特征——同一IP短时间内请求大量不同路径，且多数返回404/403/301。统计命令：`awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -10`，然后对TOP IP做进一步分析：`grep "TOP_IP" access.log | awk '{print $7}' | sort | uniq -c | sort -rn`——如果路径种类极多且杂乱，基本确定是扫描。

**Q: 如何判断一次攻击是否成功？只靠状态码够吗？**
A: 不够。需要三个维度交叉验证：①状态码（200可能成功，但403说明被拦截）；②响应体大小（拖库成功后响应体可能是正常请求的几十倍）；③后续行为（攻击成功后攻击者通常会发起下一阶段请求）。例如SQL注入成功→紧接着出现大量数据导出请求→确认攻击成功。

**Q: Web日志分析中，"基线"是什么意思？为什么重要？**
A: 基线就是"正常情况下是什么样子"——比如平时每小时1000个请求、TOP IP是负载均衡器、404占比约5%。有了基线，任何偏离都能被快速识别：突然每小时5000个请求→可能在被DDoS或扫描；突然出现新IP的大量404→可能在扫描；某接口的响应体突然变大→可能在拖库。没有基线的分析就像没有参照物的导航。

```
192.168.1.100 - - [18/Jun/2026:10:01:02 +0800] "GET /index.php?id=1' HTTP/1.1" 200 567
192.168.1.100 - - [18/Jun/2026:10:01:03 +0800] "GET /index.php?id=1' AND 1=1-- HTTP/1.1" 200 1234
192.168.1.100 - - [18/Jun/2026:10:01:04 +0800] "GET /index.php?id=1' AND 1=2-- HTTP/1.1" 200 567
192.168.1.100 - - [18/Jun/2026:10:01:05 +0800] "GET /index.php?id=1' UNION SELECT 1,2,3-- HTTP/1.1" 200 890
192.168.1.100 - - [18/Jun/2026:10:01:06 +0800] "GET /index.php?id=1' UNION SELECT user(),database(),version()-- HTTP/1.1" 200 1024
192.168.1.100 - - [18/Jun/2026:10:01:10 +0800] "GET /index.php?id=1' UNION SELECT table_name,2,3 FROM information_schema.tables-- HTTP/1.1" 200 5678
```

**分析结论**：
1. 攻击者在1秒内测试了该参数是否存在注入（单引号测试）
2. AND 1=1/1=2是经典的布尔盲注探测——验证注入是否存在
3. UNION SELECT 1,2,3确认了可联合查询的列数
4. 成功获取数据库版本和用户信息（步骤5）
5. 开始查询表名（information_schema.tables）——这是拖库的前奏
6. 响应体从1234增长到5678→可能已成功获取数据

**判定**：SQL注入攻击成功，攻击者正在进行信息收集，即将拖库。蓝队应立即封禁该IP，通知业务方排查漏洞，导出完整日志保留证据。

## 🔬 深度专题：Web日志分析进阶——从单条日志到攻击行为画像

### 构建攻击者行为画像

蓝队不只是"看到攻击"，更要"理解攻击者"。通过日志分析构建攻击者行为画像：

**攻击者画像维度**：
1. **技术水平**：手工构造payload（高）vs 使用自动化工具默认payload（低）→ User-Agent和payload复杂度判断
2. **攻击目标**：是广撒网式扫描还是针对特定业务接口？→ 通过请求路径的集中度判断
3. **攻击阶段**：探测阶段（大量404）→漏洞验证（少量200/500）→漏洞利用（出现shell上传）→后渗透（异常文件访问）
4. **持久性**：攻击者是单次尝试还是持续攻击？→ 分析攻击时间跨度和频率
5. **使用的工具**：通过User-Agent、payload结构、请求频率推断（Nmap/sqlmap/BurpSuite/自写脚本）

**画像输出模板**：
```
【攻击者画像报告】
源IP: xxx.xxx.xxx.xxx | 地理位置: 境外/境内
技术等级: 高级/中级/初级
攻击类型: SQL注入 + 目录扫描
攻击阶段: 漏洞利用阶段（已成功获取数据库名和表名）
使用工具: sqlmap（根据payload结构、时间盲注特征判断）
首次发现: 2026-06-18 10:05 | 最近活动: 2026-06-18 14:30
攻击频次: 共发出230+次恶意请求 | 峰值: 10:15-10:30（40次/分钟）
当前状态: 已被WAF封禁 | 建议: 排查漏铜并修复
```

### 日志基线建立方法

没有基线的日志分析就像没有参考系的导航。以下是建立日志基线的实操方法：

**第一步：收集一周的正常日志（非护网期、无已知攻击时段）**
```bash
# 确保日志覆盖所有业务时段（工作日/周末/白天/夜间）
ls -lh /var/log/nginx/access.log*
```

**第二步：统计基线指标**
```bash
# 每小时平均请求量
awk -F':' '{print $2}' access.log | cut -c1-2 | sort | uniq -c

# 正常状态码分布
awk '{print $9}' access.log | sort | uniq -c | sort -rn

# 正常请求路径TOP20
awk '{print $7}' access.log | sort | uniq -c | sort -rn | head -20

# 正常源IP TOP20
awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -20
```

**第三步：设定告警阈值**
```
# 基于基线设定偏离阈值（以标准差或百分比）
请求量 > 基线的2倍 → 告警
404比例 > 基线的3倍 → 扫描告警
500比例 > 基线的5倍 → 攻击告警
出现基线中不存在的IP且高频访问 → 可疑
```

### 蓝队"条件反射"训练

网络安全值守中，很多判断需要在几秒内完成。以下是本日主题相关的"条件反射"训练：

**看到以下现象 → 立即联想到 → 采取动作**：

1. 短时间内同一IP大量不同URL请求 → 目录/漏洞扫描 → 检查是否返回了不该返回的内容

5. 出站流量突增到非标准端口 → 数据渗出/C2通信 → 追踪目标IP并阻断
