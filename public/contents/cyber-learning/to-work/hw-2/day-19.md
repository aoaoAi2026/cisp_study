## 📖 核心知识讲解

### 一、探测类攻击总览

攻击者在动手之前，通常会先"踩点"——扫描你的系统，了解开放了什么端口、运行了什么服务、有哪些页面。所谓"知己知彼，百战不殆"。

```
攻击者的探测流程：
  第一步：网络扫描  → 哪些IP活着？
  第二步：端口扫描  → 开了哪些端口？
  第三步：服务识别  → 跑的是什么软件？什么版本？
  第四步：漏洞扫描  → 有没有已知漏洞？
  第五步：目录扫描  → 网站有哪些隐藏页面？
```

> 🎯 **蓝队要点**：扫描本身不造成直接伤害，但扫描是攻击的"前奏"。发现扫描行为后要提高警惕，因为真正的攻击可能紧随其后。

### 二、端口扫描识别

#### 流量特征（如何在Wireshark中识别）

**Nmap SYN扫描的特征：**
```
同一源IP → 在极短时间内 → 向大量不同目标端口 → 发送SYN包

tcp.flags.syn == 1 && tcp.flags.ack == 0
```

**在Wireshark中看到的样子：**
```
15:00:00.001  192.168.1.100 → 目标IP:22   [SYN]
15:00:00.002  192.168.1.100 → 目标IP:23   [SYN]  
15:00:00.003  192.168.1.100 → 目标IP:25   [SYN]
15:00:00.004  192.168.1.100 → 目标IP:80   [SYN]
...（连续数千个SYN包，间隔极短）
```

**Nmap的特征签名：**
- 默认SYN扫描时，源端口固定，Window Size通常是1024
- Nmap扫描结束后，通常不会完成三次握手（收到SYN+ACK后发RST）
- 不常用的端口被探测（如23/25/110等老旧服务端口）

#### 日志特征
```
# 防火墙日志中的扫描痕迹
Jun 12 15:00:00 kernel: DROP IN=eth0 SRC=192.168.1.100 DST=目标IP PROTO=TCP DPT=22
Jun 12 15:00:01 kernel: DROP IN=eth0 SRC=192.168.1.100 DST=目标IP PROTO=TCP DPT=23
Jun 12 15:00:02 kernel: DROP IN=eth0 SRC=192.168.1.100 DST=目标IP PROTO=TCP DPT=25
...
# 特征：同一源IP、密集TCP连接、不同目的端口
```

#### 常见扫描工具识别

| 工具 | 特征 | 日志/Wireshark中怎么辨认 |
|:---|:---|:---|
| **Nmap** | 有序扫描、TCP Window=1024 | 端口顺序递增（21,22,23,25...） |
| **Masscan** | 极快、随机端口顺序 | 端口是乱序的，每秒数千个SYN包 |
| **Zmap** | 只发SYN不等回复，单包扫描 | 几乎看不到回包 |
| **Rustscan** | 很快，Nmap风格 | 速度快但没有明显指纹 |

### 三、目录/文件扫描识别

#### Web日志中的目录扫描特征

```bash
# 目录扫描在Web访问日志中的经典特征：
192.168.1.100 - - [12/Jun/2026:15:00:00] "GET /admin HTTP/1.1" 404
192.168.1.100 - - [12/Jun/2026:15:00:01] "GET /wp-admin HTTP/1.1" 404
192.168.1.100 - - [12/Jun/2026:15:00:01] "GET /phpmyadmin HTTP/1.1" 404
192.168.1.100 - - [12/Jun/2026:15:00:02] "GET /backup.zip HTTP/1.1" 404
192.168.1.100 - - [12/Jun/2026:15:00:02] "GET /.git/config HTTP/1.1" 404
192.168.1.100 - - [12/Jun/2026:15:00:03] "GET /.env HTTP/1.1" 404

# 三大特征：
# 1. 同一IP密集请求
# 2. 大部分返回404（因为目录不存在）
# 3. 请求的都是敏感路径（admin/backup/.git/.env等）
```

#### 快速发现目录扫描的命令
```bash
# 统计产生最多404的IP
grep " 404 " access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head

# 查看某IP访问了哪些路径（判断是否在扫目录）
grep "可疑IP" access.log | awk '{print $7}' | sort -u

# 如果在扫目录，你会看到：
# /admin, /wp-admin, /phpmyadmin, /backup, /.git, /.env, /test, /tmp...
# 这些都不是正常用户会访问的路径
```

#### 常用扫描工具的User-Agent特征
```bash
# 目录扫描器的User-Agent常包含以下关键字：
DirBuster       → "DirBuster"
Gobuster        → "gobuster"
Dirb            → 通常没有独特UA，但会改
sqlmap          → "sqlmap"
Nikto           → "Nikto"
Nessus          → "Nessus"
AWVS            → "acunetix"
```

### 四、资产测绘/指纹识别

攻击者还会探测你的Web服务器类型、版本、框架：

```bash
# 资产测绘在日志中的表现：

# 1. 探测不存在的路径（看404页面的特征）
GET /wp-content/plugins/test.php → 确认是否为WordPress

# 2. 探测默认文件（判断服务器类型）
GET /phpinfo.php → 判断PHP版本
GET /manager/html → 判断是否为Tomcat
GET /console → 判断是否为Spring Boot

# 3. 探测特定header（识别框架和CDN）
# 通过HTTP响应中的Server/X-Powered-By/Src-Cookie判断
```

### 五、蓝队处置建议

| 扫描类型 | 威胁等级 | 推荐处置 |
|:---|:---|:---|
| 端口扫描（来自境外IP） | 🟡 P2 | 封禁IP，加入黑名单 |
| 端口扫描（来自境内扫描平台） | 🟢 P4 | 记录即可（很多是安全公司的常规扫描） |
| 目录扫描（大量404） | 🟡 P2 | 封禁IP，同时检查是否扫描到了敏感信息 |
| 目录扫描+发现敏感文件(200) | 🔴 P1 | 立即处置，排查信息泄露 |
| 资产测绘（探测框架版本） | 🟡 P3 | 记录，可能是后续定向攻击的前奏 |

## 📝 今日小结

扫描探测是攻击者的"侦察兵"。今天你学会了从流量和日志中识别它们——端口扫描看密集SYN，目录扫描看密集404。扫描本身不致命，但每一个严重的入侵，都是从一次扫描开始的。发现扫描后，你的雷达就要调高灵敏度了。

**记住今天的核心**：
- 端口扫描 = 同一IP + 密集SYN + 不同端口
- 目录扫描 = 同一IP + 密集404 + 敏感路径
- Nmap特征 = 端口有序、Window=1024、SYN→收到SYN+ACK→RST
- 扫描 = 攻击前的侦察，值得警惕

## 🎯 蓝队面试高频题（Day 19 主题）

**Q1：如何从流量/日志中识别Nmap端口扫描？**

> Nmap扫描的4个核心特征（满足其中2条即可判断为扫描）：
> 1. **SYN包大量+不完成握手**：同一源IP在短时间内发送大量SYN包，且很少完成三次握手
> 2. **目标端口递增/跳跃**：目标端口从1→2→3...（连续扫描）或22,80,443,3306,3389...（常见端口列表）
> 3. **固定源端口或随机化**：Nmap可用固定源端口（-g参数）或随机源端口
> 4. **TCP选项指纹**：Nmap的TCP选项组合（MSS/Window Size等）有独特指纹
>
> Wireshark检测：`tcp.flags.syn==1 && tcp.flags.ack==0` → Statistics → Conversations → 看哪个IP连接了最多不同端口

**Q2：目录扫描和端口扫描的日志特征有什么区别？**

> | 特征 | 端口扫描 | 目录扫描 |
> |:---|:---|:---|
> | 日志位置 | 防火墙/IDS日志 | Web服务器access.log |
> | 目标 | IP:端口 | URL路径 |
> | 频率 | 几千到几万个SYN包 | 几百到几万个HTTP请求 |
> | 失败标志 | RST包(端口关闭) | 404状态码 |
> | 常见工具 | Nmap/Masscan/Zmap | DirBuster/Gobuster/Dirsearch |
> | 识别特征 | 同一IP→大量不同端口 | 同一IP→大量不同URL→产生大量404 |

**Q3：资产测绘和端口扫描有什么区别？**

> 资产测绘（如FOFA/Shodan）：
> - 目的：发现"整个互联网上所有运行特定服务的设备"
> - 方法：对整个IP段进行单个或少个关键端口扫描
> - 合法用途：安全研究、企业自查公网暴露面
> - 识别：扫描来自搜索引擎/测绘平台的已知IP段
>
> 端口扫描（如Nmap）：
> - 目的：发现"某个具体IP开放了哪些端口"
> - 方法：对单个IP的所有端口进行扫描
> - 更可能是攻击前奏
>
> 蓝队判断：扫描来自Shodan的已知爬虫IP → 测绘扫描 → 低优先级。来自不知名VPS的IP → 攻击性扫描 → 高优先级。

## 💡 面试高频题：攻击识别-探测类

**Q: 端口扫描在流量中有哪些可识别的特征？**
A: ①SYN扫描：`SYN→SYN-ACK→RST`（收到响应后立即RST断开，Wireshark中大量TCP Retransmission）；②Connect扫描：完整的三次握手建立连接后FIN/RST断开；③NULL/FIN/Xmas扫描：发送异常标志位的包（无标志/仅FIN/FIN+URG+PSH），基于响应差异判断端口状态；④UDP扫描：发送UDP包到各端口，通过ICMP Port Unreachable判断端口关闭。蓝队检测方法：流量中短时间内大量不同的目标端口=扫描行为。

**Q: 目录扫描在Web日志中有什么特征？**
A: ①同一IP短时间内大量请求不同路径（路径种类>100种）；②多数请求返回404/403（被扫描的目录大多不存在）；③请求的路径有明显的爆破特征（如admin、backup、test、dev、config等敏感词）；④User-Agent可能是已知的扫描器（如dirbuster/gobuster/dirsearch等标识）；⑤请求间隔极短（自动化工具通常是毫秒级间隔）。

**Q: 资产测绘扫描是什么？蓝队需要担心吗？**
A: 资产测绘是指攻击者使用fofa/shodan/zoomeye等平台搜索暴露在公网的服务。这些平台会持续扫描全互联网的IP和端口。蓝队应对策略：①先去这些平台搜一下自己的域名/IP，看看暴露了哪些信息——这就是"攻击者眼中的你"；②关闭不必要的服务暴露面（如数据库端口不应开放在公网）；③在安全设备中为这些平台扫描行为建立白名单（它们不是恶意攻击，但会产生大量噪音告警）。


## 📊 实操：如何从pcap中识别扫描行为

```bash
# 用tshark统计每个源IP尝试了哪些目标端口
tshark -r capture.pcap -T fields -e ip.src -e tcp.dstport | sort | uniq -c | sort -rn | head -20

# 输出示例：
#   1500  192.168.1.100  80,443,22,21,3306,8080,8443,9090...
#   → 源IP 192.168.1.100在短时间内尝试了1500+不同端口 = Nmap扫描

# 用tshark看SYN包的频率
tshark -r capture.pcap -Y "tcp.flags.syn==1 and tcp.flags.ack==0" -T fields -e frame.time_relative -e ip.dst | head -30
# 如果时间间隔极短（<0.1s），且目标IP分散 → 扫描无疑

# 辅助判断：看User-Agent
tshark -r capture.pcap -Y "http.request" -T fields -e http.user_agent | sort -u
# 如果看到"Nmap Scripting Engine"、"sqlmap"等工具标识 → 确定是定向攻击
```

## 🔬 进阶专题：本日知识在真实护网中的应用

### 护网场景还原

想象你正在护网值守中，突然收到一条与本日主题相关的告警。以下是标准的思维和处理流程：

**1. 第一反应（0-30秒）**：确认告警来源设备是否正常 → 查看告警级别 → 判断是否为"狼来了"

**2. 快速研判（30秒-2分钟）**：提取关键信息（源IP/目标资产/攻击类型/时间）→ 查威胁情报 → 看同一IP的其他告警

**3. 深度分析（2-10分钟）**：回放原始流量/日志 → 还原攻击payload → 判断攻击是否成功

**4. 处置决策（10-15分钟）**：确认是攻击→定级→封IP/隔离/升级事件；确认是误报→标记→优化规则

**5. 记录闭环（15-20分钟）**：填写工单（研判依据+处置动作+证据截图）→ 标记闭环


### 高手与新手的关键区别

| 维度 | 新手 | 高手 |
| --- | --- | --- |
| 看告警 | 只看这一条告警的内容 | 同时关联其他设备/其他时间的同类告警 |
| 判断依据 | "这个payload看起来像攻击" | "这个payload+这个User-Agent+这个时间点+这个频率=SQLMap自动化扫描" |
| 处置方式 | 封IP→完事 | 封IP+排查漏洞+检查是否已有数据泄露+通知业务方+更新WAF规则 |
| 记录质量 | "已处置" | "10:05收到WAF SQL注入告警→核查源IP 45.x为境外IP+威胁情报标记恶意→查看该IP完整请求链→确认SQLMap自动化扫描→已在WAF永久封禁+通知开发修复参数化查询→附件：攻击流量pcap+威胁情报截图" |

### 护网场景还原

想象你正在护网值守中，突然收到一条与本日主题相关的告警。以下是标准的思维和处理流程：

**1. 第一反应（0-30秒）**：确认告警来源设备是否正常 → 查看告警级别 → 判断是否为"狼来了"

**2. 快速研判（30秒-2分钟）**：提取关键信息（源IP/目标资产/攻击类型/时间）→ 查威胁情报 → 看同一IP的其他告警

**3. 深度分析（2-10分钟）**：回放原始流量/日志 → 还原攻击payload → 判断攻击是否成功

**4. 处置决策（10-15分钟）**：确认是攻击→定级→封IP/隔离/升级事件；确认是误报→标记→优化规则

**5. 记录闭环（15-20分钟）**：填写工单（研判依据+处置动作+证据截图）→ 标记闭环


### 高手与新手的关键区别

| 维度 | 新手 | 高手 |
| --- | --- | --- |
| 看告警 | 只看这一条告警的内容 | 同时关联其他设备/其他时间的同类告警 |
| 判断依据 | "这个payload看起来像攻击" | "这个payload+这个User-Agent+这个时间点+这个频率=SQLMap自动化扫描" |
| 处置方式 | 封IP→完事 | 封IP+排查漏洞+检查是否已有数据泄露+通知业务方+更新WAF规则 |
| 记录质量 | "已处置" | "10:05收到WAF SQL注入告警→核查源IP 45.x为境外IP+威胁情报标记恶意→查看该IP完整请求链→确认SQLMap自动化扫描→已在WAF永久封禁+通知开发修复参数化查询→附件：攻击流量pcap+威胁情报截图" |
