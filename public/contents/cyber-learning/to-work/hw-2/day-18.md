## 📖 核心知识讲解

### 一、告警研判四步法

收到一条告警后，按以下4步逐级进行：

```
第一步：核查（Verify）   → 这条告警是真的吗？
第二步：研判（Analyze）   → 攻击成功了吗？危害多大？
第三步：处置（Respond）   → 该怎么办？
第四步：闭环（Close）     → 记录什么？后续跟什么？
```

### 二、第一步：核查 —— 判断告警真实性

**核查清单（逐条过）：**

| 核查项 | 确认方法 | 判断标准 |
|:---|:---|:---|
| 源IP情报 | 查VirusTotal/微步在线 | 红色→真实攻击嫌疑大；绿色→可能误报 |
| 攻击时间 | 看告警时间戳 | 凌晨/非工作日→更可疑 |
| 告警触发频率 | 同一告警出现了多少次 | 大量重复→自动化扫描；偶尔一次→可能误报 |
| 关联证据 | 查对应的Web日志/系统日志 | 日志里真有攻击载荷→真实；日志里没异常→可能误报 |
| 目标价值 | 被攻击的是什么系统 | 核心数据库→高危；公共测试页→低危 |

**误报常见原因：**
- 业务正常行为触发了规则（如CMS后台编辑器使用`<script>`标签）
- 安全测试团队的扫描
- 爬虫/搜索引擎的请求被误判为扫描
- IDS规则写得太宽泛

### 三、第二步：研判 —— 分析攻击结果

确认是真实攻击后，最关键的问题是：**攻击成功了吗？**

**判断攻击是否成功的核心方法：**

#### 方法1：看HTTP状态码
```
攻击请求 → 200 → ⚠️ 可能成功！（需要进一步确认）
攻击请求 → 403/401/404 → 🟢 大概率没成功
攻击请求 → 500 → ⚠️ 可能触发漏洞导致服务器崩溃
```

#### 方法2：看响应内容
```
Wireshark Follow TCP Stream：
  请求：GET /product.php?id=1' OR '1'='1
  响应：包含了数据库中的用户名列表 → 🔴 注入成功！
  响应：报错"SQL syntax error" → 🟡 注入尝试中（有一定信息泄露）
  响应：返回正常产品页面 → 🟢 注入未成功
```

#### 方法3：关联系统日志
```
Web攻击告警 + secure日志同一时间出现异常登录 = 🔴 攻击成功+权限获取
Web攻击告警 + 无其他异常 = 🟡 攻击尝试中
```

### 四、第三步：处置 —— 根据不同情况采取行动

| 情况 | 处置措施 | 紧急程度 |
|:---|:---|:---|
| **攻击成功，系统被控** | ①立即隔离被控服务器 ②封禁攻击IP ③启动应急响应 ④保留证据 | 🔴 立即 |
| **攻击尝试中，尚未成功** | ①封禁攻击IP ②加固目标 ③持续监控 | 🟡 24h内 |
| **攻击失败（403/404）** | ①记录攻击IP ②确认防护有效 ③加入监控列表 | 🟢 48h内 |
| **确认误报** | ①标注误报原因 ②考虑优化告警规则 | 🟢 定期处理 |

**典型处置动作：**
```bash
# 封禁攻击IP（iptables）
sudo iptables -A INPUT -s 攻击IP -j DROP

# 封禁攻击IP（nginx层面）
# 在nginx配置中添加 deny 攻击IP;

# 拉入黑名单并在SOC平台标记
# 编写工单记录
```

### 五、第四步：闭环 —— 记录与跟踪


### 六、告警研判SOP速查手册

**快速决策流程图：**

```
收到告警
  │
  ├─→ 查源IP情报
  │     ├─ 恶意 → 真实性+1分
  │     └─ 安全 → 可能是误报
  │
  ├─→ 看告警频率
  │     ├─ 短时间大量 → 自动化攻击+1分
  │     └─ 就一条 → 继续观察
  │
  ├─→ 查关联日志
  │     ├─ 日志对齐且攻击特征明显 → 确认攻击
  │     └─ 日志无异常 → 可能误报
  │
  └─→ 判断攻击结果
        ├─ 有成功迹象(200+数据外传+后续异常) → P0/P1
        ├─ 攻击尝试中(200但无异常) → P1/P2
        ├─ 攻击失败(403/404) → P3
        └─ 确认误报 → P4
```

## 🎯 蓝队面试高频题（Day 18 主题）

**Q1：告警研判的标准步骤"SOP"是什么？**

> 5步法（必须能背出来）：
> 1. **核查**：先确认告警的基本信息——来源设备、告警时间、源IP/目的IP、告警类型——这是真的告警还是设备误报？
> 2. **验证**：到原始日志/流量中验证——告警描述的攻击行为真实发生没有？找到对应日志记录
> 3. **研判**：结合上下文做判断——源IP是否可信（内网/白名单）、时间段是否合理、业务是否特殊
> 4. **处置**：根据研判结果决定动作——误报→标记关闭、低风险→记录关注、真攻击→按预案处置
> 5. **闭环**：记录处置过程和结论、更新工单状态、如果是新攻击手法则升级规则

**Q2：什么情况下你会判定一条告警是"误报"？**

> 误报的5类常见原因：
> 1. 正常运维操作：运维人员凌晨打补丁触发了异常登录告警→查证后确认是计划内操作→误报
> 2. 业务特殊性：爬虫流量触发了扫描告警→确认是合作方数据同步→误报
> 3. 规则太宽泛：URL中正常参数含单引号触发了SQL注入告警→实际是正常用户名→误报
> 4. 扫描器自身：安全扫描器的探测被自己的IDS告警了→狗咬狗→误报
> 5. 已失效攻击：针对已修复漏洞的攻击尝试→告警但无实际危害→降级处理
>
> 判定标准口诀：**有没有受害？有没有成功？有没有影响业务？三个都没有 → 大概率误报。**

**Q3：告警研判中"上下文"指的是什么？为什么重要？**

> 上下文 = 围绕这条告警的所有相关信息：
> - 这个源IP之前有没有过其他告警？（惯犯还是初犯）
> - 这个时间点这个系统正常在做什么？（凌晨3点备份 ≠ 凌晨3点入侵）
> - 这条告警的"前后"还有没有相关告警？（孤立告警 vs 攻击序列）
> - 告警中涉及的用户/系统是谁的？（普通测试环境 vs 核心生产系统）
>
> 无上下文的告警研判 = 盲人摸象。上下文越丰富，判断越准确。

### 面试高频题与实战演练

## 💡 面试高频题：告警研判标准流程

**Q: 告警研判的第一步"核查确认"具体怎么做？**
A: 收到告警后不要立即处置，先做以下核查：①核查告警源IP——是外网还是内网？是已知CDN/搜索引擎爬虫吗？查威胁情报确认是否为已知恶意IP；②核查目标资产——告警攻击的目标是什么系统？是否是核心业务？该资产最近是否有变更？③核查告警时间——是孤立事件还是持续性攻击？如果同一IP在1小时内触发了100条告警，很可能是真实攻击；④核查告警上下文——同一时间段内该IP还触发了哪些其他类型的告警？关联分析才能看全貌。

**Q: 如何判断一条告警是误报？**
A: 典型误报场景：①正常业务触发（如URL中包含"select"的业务查询接口）；②安全扫描器自查（企业自己的漏洞扫描器在例行扫描）；③CDN/搜索引擎的正常访问（User-Agent是Googlebot/Bingbot）；④WAF规则过于敏感（如检测到"drop"就告警，但实际是网页上的正常单词）；⑤开发人员测试（内网开发环境的正常测试行为）。判断方法：先溯源（谁发的请求？目的是什么？），而不是只看告警内容。

**Q: 告警研判中"关联分析"的价值是什么？**
A: 单条告警单独看不严重，但多条关联可能就是大型攻击。举例：①WAF告警"SQL注入尝试" + IDS告警"内网SMB扫描" = 攻击者可能已经突破外网正在横向移动；②威胁情报告警"连接到已知C2域名" + EDR告警"异常进程启动" = 主机已失陷正在与C2通信；③漏洞扫描告警"发现OpenSSH漏洞" + SSH登录告警"异常IP登录成功" = 漏洞可能已被利用。蓝队要养成"不只看单条告警，要看整个攻击链"的习惯。

**Q: 写出你整理的个人《告警研判操作SOP》应该包含哪些内容？**
A: ①研判流程：接收→核查→判定→处置→闭环（每一步的具体操作和判断标准）；②告警来源清单：列出环境中的所有安全设备和每类设备产生的告警类型；③常见误报清单：经验的积累——哪些告警在你环境中大概率是误报、哪些需要重点排查；④升级标准：什么情况下需要把告警升级为安全事件、通知谁；⑤处置模板：常见攻击类型（扫描/注入/爆破/webshell）的标准处置动作。

研判：计划任务后门！攻击者可能在建立持久化。P0级别，立即隔离服务器、提取脚本做恶意代码分析、排查同一攻击者的其他失陷资产。

## 🔬 深度专题：告警研判高级技巧——从"看一条告警"到"看攻击全貌"

### 告警关联分析——"串珠成链"

单条告警看不出全貌，关联分析才能还原攻击链：

**场景1：扫描+注入+上传 = 完整攻击链**
```
10:00 - IDS: 端口扫描 from IP-X（目录扫描探测）
10:05 - WAF: SQL注入尝试 from IP-X（利用找到的接口注入）
10:08 - WAF: SQL注入成功（响应异常大）
10:10 - WAF: 文件上传 .php from IP-X（写入webshell）
10:11 - Proxy: 访问 /uploads/shell.php（验证webshell）
10:15 - EDR: 异常进程启动（webshell执行系统命令）
```
→ 关联结论：完成了一次完整的SQL注入→Webshell上传→命令执行的攻击链！

**场景2：看似无关的告警其实是一个攻击者的多线作战**
```
09:00 - WAF1: SQL注入 to 业务A
09:02 - WAF2: XSS to 业务B
09:05 - NIDS: SSH爆破 to 服务器C
→ 源IP都是同一个 → 攻击者在同时对多个目标发起不同类型攻击
```

### 告警研判决策树（Decision Tree）

将研判经验体系化，建立决策树可以加速新人培训：

```
收到告警 →
  ├── 告警源IP是CDN/代理IP？ → YES → 大概率误报，记录观察
  │   └── NO → 继续 →
  ├── 告警目标是核心业务资产？ → YES → 提升优先级为P1+
  │   └── NO → 继续 →
  ├── 攻击payload被WAF成功拦截？ → YES → P3级别，记录IP加入监控
  │   └── NO → 继续 →
  ├── 同一IP在5分钟内触发了>10条告警？ → YES → P2级别，疑似持续性攻击
  │   └── NO → 继续 →
  ├── 告警包含webshell/后门特征？ → YES → P0级别，立即隔离
  │   └── NO → 继续 →
  ├── 响应状态码200且响应体异常大？ → YES → P0级别，可能数据泄露中
  │   └── NO → P3级别，常规记录
```

## 🔬 进阶专题：本日知识在真实护网中的应用

### 护网场景还原

想象你正在护网值守中，突然收到一条与本日主题相关的告警。以下是标准的思维和处理流程：

**1. 第一反应（0-30秒）**：确认告警来源设备是否正常 → 查看告警级别 → 判断是否为"狼来了"

**2. 快速研判（30秒-2分钟）**：提取关键信息（源IP/目标资产/攻击类型/时间）→ 查威胁情报 → 看同一IP的其他告警

**3. 深度分析（2-10分钟）**：回放原始流量/日志 → 还原攻击payload → 判断攻击是否成功

**4. 处置决策（10-15分钟）**：确认是攻击→定级→封IP/隔离/升级事件；确认是误报→标记→优化规则

**5. 记录闭环（15-20分钟）**：填写工单（研判依据+处置动作+证据截图）→ 标记闭环


### 高手与新手的关键区别

| 维度 | 新手 | 高手 |
| --- | --- | --- |
| 看告警 | 只看这一条告警的内容 | 同时关联其他设备/其他时间的同类告警 |
| 判断依据 | "这个payload看起来像攻击" | "这个payload+这个User-Agent+这个时间点+这个频率=SQLMap自动化扫描" |
| 处置方式 | 封IP→完事 | 封IP+排查漏洞+检查是否已有数据泄露+通知业务方+更新WAF规则 |
| 记录质量 | "已处置" | "10:05收到WAF SQL注入告警→核查源IP 45.x为境外IP+威胁情报标记恶意→查看该IP完整请求链→确认SQLMap自动化扫描→已在WAF永久封禁+通知开发修复参数化查询→附件：攻击流量pcap+威胁情报截图" |
