# 等保 2.0 密码模块要求与产品选型指南

## 1. 密码模块定义与分级体系

在等保 2.0 中，"密码模块"是指实现密码功能、提供密码服务的硬件、软件、固件或其组合。典型形态包括：服务器密码机、智能密码钥匙（USB Key）、密码卡、VPN 网关、数据库加密网关、国密 CA 系统、手机安全元件（SE）、可信平台模块（TPM/TCM）等。

依据《商用密码产品认证目录》和 GM/T 0039《密码模块安全技术要求》，密码模块按安全等级分为 **一 ~ 四级**（一级最低，四级最高）。与等保级别对应关系：

| 等保等级 | 推荐密码模块等级 | 典型使用场景 |
|---------|-----------------|-------------|
| 等保二级 | 一级密码模块（或二级） | 中小型企业系统、门户网站 |
| 等保三级 | 二级密码模块（推荐三级） | 政府、金融、关键业务系统 |
| 等保四级 | 三级 / 四级密码模块 | 关键信息基础设施、金融核心 |

## 2. 密码模块安全要求分层（GM/T 0039）

GM/T 0039 将密码模块安全要求拆分为 11 个安全域：

| 安全域 | 核心要求 |
|--------|---------|
| 1. 密码模块规格 | 明确模块的安全边界、接口、端口 |
| 2. 密码模块接口 | 逻辑/物理接口安全划分、数据输入/输出隔离 |
| 3. 角色、服务与认证 | 多角色模型、认证失败处理、锁定策略 |
| 4. 软件/固件安全 | 代码签名、完整性校验、安全启动 |
| 5. 运行环境 | 操作系统/芯片/虚拟机的安全保障 |
| 6. 物理安全 | 防拆、防窃听、防侧信道攻击（DPA/CPA） |
| 7. 非入侵式安全 | 电源、时序、电磁、温度等侧信道防护 |
| 8. 敏感安全参数管理 | 密钥/口令/种子的生成、分发、存储、备份、销毁 |
| 9. 自测试 | 上电自检、条件自检、密码算法已知答案测试 |
| 10. 设计保障 | 开发流程安全、供应链安全、文档 |
| 11. 生命周期保障 | 部署、维护、退役、销毁 |

### 2.1 敏感安全参数（SSP）生命周期

敏感安全参数（如密钥、口令、种子值）生命周期必须符合以下模型：

```
生成（模块内部硬件随机数） → 分发（加密通道 / 面对面）
      │
      ▼
  存储（不可导出明文） → 使用（模块内部运算）
      │
      ▼
  备份（加密后导出） → 归档（加密 + 离线） → 销毁（物理 / 覆盖）
```

### 2.2 随机数质量是根基

密码模块的随机数发生器（RNG / DRBG）必须通过 GM/T 0042 随机性检测。常见问题：

- 使用弱随机源（`rand()` / `Math.random()`）替代 CSPRNG
- 种子固定或可预测（常见于未初始化的 IoT 设备）
- 随机数质量差（熵不足）导致密钥可被分析

## 3. 等保 2.0 对密码模块的刚性要求

依据 GB/T 22239-2019 和《商用密码应用安全性评估管理办法》，等保三级及以上系统必须满足：

| 要求项 | 等级 | 说明 |
|--------|------|------|
| 使用国密算法 | 强制 | 至少使用 SM2/SM3/SM4，必要时使用 SM1/SM9/ZUC |
| 使用商用密码认证产品 | 强制 | 必须选购已获得**商用密码产品认证证书**的产品 |
| 密钥不可导出明文 | 强制 | 私钥/主密钥不得导出，必须由密码模块内部保护 |
| 双证书体系 | 推荐 | 国密 CA 签发签名证书 + 加密证书（用于 TLS） |
| 密码模块安全等级 ≥ 一级 | 强制 | 等保三级建议 ≥ 二级 |
| 密钥管理体系 | 强制 | 建立密钥全生命周期管理制度 |
| 密码模块审计 | 强制 | 密码模块操作须独立审计 |

## 4. 典型密码模块与应用场景选型

### 4.1 产品选型对照

| 密码模块 | 用途 | 典型部署位置 |
|----------|------|-------------|
| 服务器密码机（SJJ1618 等） | 高频签名/验证、SM2/SM4 运算 | 数据中心机房核心柜 |
| 智能密码钥匙（USB Key） | 用户身份认证、数字签名 | 员工桌面 / 移动办公 |
| 密码卡（PCI-e 形态） | 嵌入式密码加速 | 服务器内部 |
| 国密 VPN 网关 | 分支互联、移动办公接入 | 总部/分支边界 |
| 数据库加密网关 / TDE | 数据库字段加密 | 数据库前置 |
| 国密 CA 系统 | 证书签发与管理 | 企业内网 |
| 可信密码模块（TCM / TPCM） | 可信启动、硬件级密钥存储 | 服务器主板 |

### 4.2 国密协议栈使用示例

```bash
# 使用 GmSSL / 国密版 OpenSSL
# 生成 SM2 密钥对
gmssl sm2 -genkey -out sm2.key
# 生成 SM2 证书签名请求
gmssl req -new -key sm2.key -out sm2.csr -sm3
# 使用 CA 签发 SM2 证书（国密 CA）
gmssl ca -in sm2.csr -out sm2.crt -cert ca.crt -keyfile ca.key

# 配置 Nginx 使用国密双证书
#   ssl_certificate       sm2-rsa.crt;    # RSA 兼容证书
#   ssl_certificate       sm2-sign.crt;   # SM2 签名证书
#   ssl_certificate_key   sm2-sign.key;
#   ssl_protocols         TLSv1.2 TLSv1.3 GMTLS;
#   ssl_ciphers           ECDHE-SM2-WITH-SM4-CBC-SM3;
```

## 5. 密码模块常见风险与整改

### 5.1 常见高风险问题

1. **使用自实现/开源密码模块**：未取得商用密码认证证书，即使算法"正确"也不可用于等保合规
2. **密钥硬编码**：`config.properties` 中出现 `AES_KEY=xxxx`
3. **随机数来源弱**：使用 `rand()`、`clock()`、用户时间戳等
4. **私钥导出**：为了"备份"把私钥明文导出成文件
5. **弱口令保护密钥**：密钥文件的保护口令强度不够
6. **TLS 配置不当**：SSLv3、TLS 1.0/1.1 未禁用；RC4/DES/3DES 仍在使用
7. **无密钥应急/销毁流程**：故障时直接更换设备，密钥被遗留

### 5.2 整改参考

```bash
# 1) 密钥硬编码扫描
grep -rnE "(password|key|secret|token|aes_key|sm4_key)\s*=|-----BEGIN" .

# 2) 弱 TLS 协议/套件检测（线上）
testssl.sh www.example.com

# 3) 证书指纹/链检查
openssl s_client -connect www.example.com:443 2>&1 | openssl x509 -text -noout | head -30

# 4) 随机数质量检测（dieharder / NIST SP800-90B）
#    dieharder -a -g 201 -f /dev/random
```

### 5.3 一个"从自实现升级到密码模块"的改造模式

```
改造前：
  业务代码 ── self_crypto.py（SM4 自实现 + 密钥写死） ── 数据库明文

改造后：
  业务代码 ── 密码机 SDK（JCE/PKCS#11/SDF） ── 服务器密码机
                    │ (密钥不可导出，仅获取运算接口)
                    ▼
                数据库字段级加密 (SM4)
```

## 6. 采购与部署注意事项

1. **认证证书有效性**：采购前在国家密码管理局网站查询商用密码认证证书是否有效
2. **性能与高可用**：考虑性能冗余（如 2+1 冗余部署、负载均衡）
3. **密钥分片**：关键主密钥使用 Shamir 秘密共享，由多人分别保管
4. **应急密钥**：必须建立密钥应急恢复机制，业务中断时可在受控条件下恢复
5. **运维与审计**：密码机操作需由密码安全员执行，操作行为独立审计

> 密码模块等保合规的核心是：**算法合规 + 产品合规 + 使用合规**。三方面缺一不可，任何一处短板都可能导致等保测评 / 密评结论为"不通过"。
