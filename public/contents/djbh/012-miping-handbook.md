# 密码应用安全性评估（密评）实操手册

## 1. 密评适用范围与启动时机

### 1.1 哪些系统必须做密评

依据《商用密码应用安全性评估管理办法》和等保 2.0 相关要求，以下系统**应当**开展密评：

- **等保三级及以上系统**：每年一次密评，与等保测评协同
- **关键信息基础设施（CII）**：由运营者按规定组织
- **政务信息系统**：按《政务信息系统密码应用与安全性评估工作指南》要求
- **重点行业**：金融、能源、交通、水利、卫生健康、教育、通信、公共服务等

### 1.2 启动时机

| 场景 | 要求 |
|------|------|
| 新建系统 | 规划阶段即引入密码方案，上线前开展首次密评 |
| 存量系统 | 2021 年 7 月 1 日前完成首次密评，之后每年一次 |
| 重大变更 | 密码产品 / 算法 / 架构变更时重新评估 |
| 密码事件 | 发生密码安全事件后立即重新评估 |

## 2. 密评参考标准体系

| 标准 | 作用 |
|------|------|
| GB/T 39786-2021《信息安全技术 信息系统密码应用基本要求》 | 密码应用"基本要求"，对应 GB/T 22239-2019 的密码维度 |
| GM/T 0054《信息系统密码应用基本要求》 | 商用密码行业配套标准 |
| GM/T 0115-2022《商用密码应用安全性评估规范》（系列） | 测评方法与判定规则 |
| GM/T 0039《密码模块安全技术要求》 | 密码产品自身的安全等级 |
| GM/T 0041《密码模块检测规范》 | 密码模块实验室检测 |
| 《政务信息系统密码应用与安全性评估工作指南》 | 政务系统专项 |

## 3. 密评实施全流程

密评通常由具有相应密评资质的机构（商密应用与安全性评估机构）承担，企业配合开展。典型流程如下：

### 3.1 评估实施流程总图

```
Step 1  需求与准备（业务方）
  ├─ 梳理系统清单、业务场景、密码应用现状
  ├─ 确定责任方、牵头人、测评窗口
  └─ 准备文档：系统架构图、拓扑图、密码产品清单、密钥管理方案
           │
Step 2  测评启动（测评机构）
  ├─ 召开启动会，明确范围、边界、角色
  └─ 制定密评实施计划与保密协议
           │
Step 3  技术测评
  ├─ 文档审查：密码应用方案、产品证书、密钥管理制度
  ├─ 现场核查：机房、密码设备、网络拓扑、密钥管理
  ├─ 工具测试：协议分析、密钥强度、证书链验证、算法一致性测试
  └─ 访谈：系统管理、安全管理、密码管理角色
           │
Step 4  问题整改（业务方）
  ├─ 针对测评发现问题制定整改计划
  ├─ 按优先级完成整改
  └─ 提交整改证据（截图 / 报告 / 配置）
           │
Step 5  复测与报告
  ├─ 测评机构对整改项进行复测
  └─ 出具《商用密码应用安全性评估报告》（结论：通过 / 基本通过 / 不通过）
           │
Step 6  报送与归档
  ├─ 运营者向本级密码管理部门或指定渠道报送
  └─ 相关文档归档保存 ≥ 3 年
```

## 4. 密码应用"七维度"评估方法

根据 GB/T 39786，密码应用评估通常从七个维度展开：

| 维度 | 典型评估内容 |
|------|-------------|
| 身份鉴别 | 是否使用 SM2 证书 / USB Key / 密码 Key 实现双因素认证 |
| 数据传输加密 | 是否使用国密 TLS（SM2 签名 + SM4 加密） |
| 数据存储加密 | 敏感数据是否 SM4 加密存储，密钥是否在密码模块内部保护 |
| 数据完整性 | 关键数据是否使用 SM3 校验 |
| 不可否认性 | 关键操作是否使用 SM2 数字签名 |
| 密钥管理 | 密钥生成、分发、存储、备份、归档、销毁全流程 |
| 密码产品与密码服务 | 产品是否具备商用密码产品认证证书，是否正确配置 |

### 4.1 常用测试工具清单

```bash
# 1) 国密协议 / 证书分析
gmssl s_client -connect api.example.gov.cn:443 -sm2   # 国密握手
openssl s_client -connect api.example.gov.cn:443 -showcerts  # 证书链

# 2) 证书签名算法、有效期、链完整性
openssl x509 -in cert.pem -text -noout

# 3) 传输协议与套件分析
testssl.sh api.example.gov.cn                       # 第三方脚本，自动化扫描

# 4) 随机数质量检测
#    生成 1GB 随机数文件，使用 NIST SP 800-90B / dieharder 分析
#    dd if=/dev/random of=sample.bin bs=1M count=1024

# 5) 密钥格式检查
openssl pkey -in key.pem -text -noout                # 私钥结构检查（长度/曲线）

# 6) 数据库敏感字段加密检查（手工/内部开发工具）
#    SELECT phone, id_card FROM users LIMIT 1;
#    判断值是否为 SM4 密文（base64 + 定长结构识别）
```

## 5. 典型风险场景与整改建议

### 5.1 典型问题清单

| 风险场景 | 等级 | 整改方法 |
|---------|------|---------|
| 仍使用 MD5 做密码哈希 | 高 | 切换为 bcrypt/Argon2id + SM3；存量密码平滑迁移 |
| 数据库敏感字段全部明文 | 高 | SM4 字段级加密 + 字段存储 KMS 管理的密钥 |
| 密钥硬编码 / 明文存储于配置文件 | 高 | 改用 KMS / HSM，运行时通过 SDK 获取 |
| HTTPS 仍用 RSA 2048 + SHA-256 | 中 | 升级国密 SM2 双证书；过渡期保留 RSA |
| 自实现加密算法 / 自制密码工具 | 高 | 替换为商用密码认证产品 |
| 无密码应急 / 销毁流程 | 中 | 制定密钥应急与销毁 SOP 并演练 |
| 无密钥审计 | 中 | 密码机操作日志统一收集并独立审计 |
| TLS 仍允许 SSLv3 / TLS 1.0 | 中 | 禁用弱协议，仅保留 TLS 1.2 / TLS 1.3 |

### 5.2 从明文到国密：一个升级路线示例

```
现状：HTTP + 数据库明文
  阶段 1（紧急）    HTTPS（RSA + AES）+ 敏感字段加密（应用层 AES）
  阶段 2（合规）    国密 SM2 双证书 + SM4 字段级加密 + 密码机
  阶段 3（完善）    全链路国密（TLS SM2 + 数据库 SM4 + 身份 SM2）
  阶段 4（持续）    定期密评 + 持续密钥轮换 + 应急演练
```

## 6. 密钥管理操作规范模板

以下是一份可直接用于企业密钥管理的操作规范模板（节选）：

```
1. 密钥生成
   - 使用密码机（HSM）内部硬件随机数生成器（CSPRNG）
   - 禁止由外部程序传入密钥
2. 密钥分发
   - 密钥分片由 ≥ 3 名密码管理员分别保管，使用 Shamir (k,n) 秘密共享
   - 禁止明文 U 盘分发密钥
3. 密钥存储
   - 主密钥必须存储在密码模块内部，不可导出明文
   - 会话密钥 / 工作密钥由主密钥派生
4. 密钥轮换
   - 工作密钥至少每 12 个月轮换一次
   - 主密钥每 3 年轮换一次，或在怀疑泄露时立即轮换
5. 密钥销毁
   - 由密码管理员执行密码模块内部销毁命令
   - 记录销毁时间、操作人员、销毁对象
6. 密钥审计
   - 所有密钥管理操作须独立审计日志
   - 日志保存 ≥ 3 年
```

## 7. 整改通过要点（实操建议）

```
Step 1  完成一次系统的"密码资产盘点"：产品 + 算法 + 密钥 + 证书
Step 2  解决高风险项：硬编码密钥、明文字段、弱算法（MD5/SHA-1/DES/3DES）
Step 3  引入服务器密码机 + KMS，替代自实现加密
Step 4  升级 HTTPS 证书为 SM2 双证书（国密 CA）
Step 5  完善密码管理制度与密钥管理 SOP
Step 6  选择具备密评资质的机构开展正式密评
Step 7  按建议整改并通过复测，归档报告与证据
```

> 密评不是"一考定终身"，而是密码应用的"体检"。每年一次密评，让密码应用"持续合规、持续安全、持续可审计"。
