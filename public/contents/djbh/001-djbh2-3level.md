## 一、等保 2.0 基础

### 1.1 法律依据

```
《网络安全法》第二十一条：
  国家实行网络安全等级保护制度

《网络安全等级保护条例》（征求意见稿）：
  明确了定级、备案、测评、整改的完整流程

关键标准：
  GB/T 22239-2019 — 安全通用要求（211个控制点，三级）
  GB/T 28448-2019 — 测评要求
  GB/T 22240-2020 — 定级指南
  GB/T 25070-2019 — 安全设计技术要求
```

### 1.2 五个等级

| 等级 | 侵害对象 | 监管要求 | 测评频率 | 典型系统 |
|------|---------|---------|---------|---------|
| 一级 | 个人/组织，一般损害 | 自主保护 | 无强制 | 小型办公系统 |
| 二级 | 个人/组织，严重损害 | 指导保护 | 建议2年1次 | 企业内部OA |
| 三级 | 社会秩序/公共利益，严重损害 | 监督保护 | 强制 ≥1次/年 | 政府网站/金融系统 |
| 四级 | 社会秩序，特别严重损害 | 强制保护 | 强制 ≥1次/半年 | 国家重要系统 |
| 五级 | 国家安全，特别严重损害 | 专门保护 | 特殊要求 | 核心涉密系统 |

## 二、全流程概览

```
等保三级全流程时间线：

  定级(2周) → 备案(2周) → 差距分析(2周) → 建设整改(2-4个月) →
  测评(2周) → 获取测评报告(1周) → 持续监督(年度)

总时长：3-8 个月（取决于整改复杂度）

关键节点：
  T+0:     项目启动，确定范围
  T+2W:    定级报告完成 + 专家评审
  T+4W:    公安机关备案完成
  T+6W:    差距分析报告完成
  T+14W:   整改建设完成
  T+16W:   正式测评
  T+18W:   获取测评报告
```

## 三、阶段一：定级

### 3.1 定级方法

```
定级 = max(业务信息安全等级 S, 系统服务安全等级 A)

S 判定：信息被破坏后对客体的侵害程度
  第一级 — 对公民/组织造成一般损害
  第二级 — 对公民/组织造成严重损害 / 对社会造成一般损害
  第三级 — 对社会造成严重损害 / 对国家安全造成一般损害
  第四级 — 对社会造成特别严重损害 / 对国家安全造成严重损害
  第五级 — 对国家安全造成特别严重损害

A 判定：服务中断后对客体的侵害程度（同上五级）

最终等级 = max(S, A)
```

### 3.2 定级报告模板

```markdown
# 信息系统安全等级保护定级报告

## 一、信息系统描述
- 系统名称：XXX公司核心业务系统
- 责任单位：XXX公司信息技术部
- 系统简介：承载公司日常OA审批、合同管理、财务报销等业务流程，
  服务全体员工（约2000人），日均处理审批单5000+。
  系统包含用户个人信息（姓名/部门/手机/身份证号）、合同信息、
  财务数据等。

## 二、业务信息安全等级（S）确定

| 业务信息 | 侵害客体 | 侵害程度 | 依据 |
|---------|---------|---------|------|
| 用户个人信息 | 公民合法权益 | 严重损害 | 含身份证号/手机号等，泄露可能导致诈骗 |
| 合同信息 | 组织合法权益 | 严重损害 | 泄露可导致商业机密外泄 |
| 财务数据 | 组织合法权益 | 严重损害 | 泄露造成竞争劣势 |

业务信息安全等级：S = 3（社会秩序/公共利益严重损害）

## 三、系统服务安全等级（A）确定

| 服务类型 | 中断影响 | 侵害客体 | 侵害程度 |
|---------|---------|---------|---------|
| OA审批 | 全员无法办公 | 组织合法权益 | 严重损害 |
| 合同管理 | 业务停滞 | 组织合法权益 | 严重损害 |

系统服务安全等级：A = 3

## 四、安全保护等级确定
S = 3, A = 3 → 第三级 (S3A3G3)

## 五、审批
定级责任人：________    日期：________
单位负责人：________    日期：________
```

## 四、阶段二：备案

### 4.1 备案材料清单

```
三级系统备案所需材料（提交给公安机关网安部门）：

1. 《信息系统安全等级保护备案表》
   - 共 4 页 16 项
   - 包含：备案单位信息、系统基本情况、定级信息、安全责任单位

2. 定级报告
   - 含专家评审意见（专家签字）
   - 必须S/A等级分别论证

3. 系统拓扑结构图
   - 标注安全区域划分
   - 标注网络边界安全设备

4. 安全设施清单
   - 防火墙/IPS/WAF/堡垒机/日志审计等
   - 设备型号和部署位置

5. 营业执照复印件
   - 加盖公章

6. 等保证书（测评通过后补交）
   - 首次备案时不需，测评通过后补交
```

### 4.2 专家评审组织

```
专家评审会要点：

专家组成：≥3 人（通常 5 人）
  ✓ 至少 1 名高级职称专家
  ✓ 与系统开发/运维方无利益关系
  ✓ 来自测评机构/高校/行业协会

评审材料：
  ① 定级报告（草案）
  ② 系统拓扑图 + 数据流图
  ③ 业务功能介绍

评审意见模板：
  经专家组评审，一致认为：
  1. 定级报告内容完整，符合 GB/T 22240-2020 要求
  2. 业务信息安全等级 S=3，系统服务安全等级 A=3
  3. 定级依据充分，结论可靠
  评审结论：☑ 通过
```

## 五、阶段三：差距分析

### 5.1 差距分析方法

```
对照 GB/T 22239-2019 等保三级 211 个控制点，逐项检查：

检查方法：
  ① 文档审查 — 安全管理制度是否齐全？
  ② 现场检查 — 机房门禁/温湿度/消防是否达标？
  ③ 配置核查 — 防火墙策略/服务器配置/审计日志
  ④ 工具测试 — 漏洞扫描/基线核查/弱口令检测
  ⑤ 人员访谈 — 询问管理员/操作员的实际流程

判定标准：
  ✓ 符合   — 完全满足（100%）
  △ 部分符合 — 部分满足（50-80%）
  ✗ 不符合  — 不满足（0-50%）
  N/A      — 不适用（需说明理由）
```

## 六、阶段四：建设整改

### 6.1 整改优先级

```
高风险项（一票否决）：
  → 必须整改，否则测评不通过

Top 5 高风险项：
  ① 弱口令（系统/数据库/网络设备存在默认口令或弱口令）
  ② 远程运维无加密（Telnet明文 / 无堡垒机）
  ③ 互联网直接可访问核心数据库
  ④ 无审计日志 或 日志留存<6个月
  ⑤ 无备份 或 备份从不测试恢复

中风险项（扣分但不否决）：
  → 影响最终得分，需尽量整改

低风险项：
  → 可在测评后持续改进
```

### 6.2 关键整改方案

```bash
# 1. 弱口令整改
# Linux 密码策略
sed -i 's/^PASS_MAX_DAYS.*/PASS_MAX_DAYS 90/' /etc/login.defs
sed -i 's/^PASS_MIN_LEN.*/PASS_MIN_LEN 8/' /etc/login.defs

# 登录失败锁定
echo "auth required pam_faillock.so deny=5 unlock_time=1800" \
  >> /etc/pam.d/system-auth

# 2. SSH 安全配置
cat >> /etc/ssh/sshd_config << EOF
PermitRootLogin no
PasswordAuthentication yes
PubkeyAuthentication yes
ClientAliveInterval 300
MaxAuthTries 5
Protocol 2
EOF
systemctl reload sshd

# 3. 开启审计日志
systemctl enable auditd && systemctl start auditd
# 配置审计规则
cat >> /etc/audit/rules.d/audit.rules << EOF
-a always,exit -F arch=b64 -S execve -k exec
-w /etc/passwd -p wa -k identity
-w /etc/shadow -p wa -k identity
EOF
augenrules --load

# 4. 防火墙策略
firewall-cmd --permanent --add-service=ssh
firewall-cmd --permanent --add-service=https
firewall-cmd --permanent --remove-service=ftp
firewall-cmd --permanent --remove-service=telnet
firewall-cmd --reload
```

### 6.3 安全产品部署清单

```
等保三级必须部署的安全产品：

☑ 下一代防火墙（NGFW）— 边界防护
☑ IPS/IDS — 入侵检测与防御
☑ WAF — Web 应用防护（如有 Web 应用）
☑ 堡垒机 — 运维操作审计（双因素认证）
☑ 日志审计系统 — 集中采集+留存≥6个月
☑ 漏洞扫描系统 — ≥1次/季度
☑ 数据库审计 — 数据库访问审计
☑ 杀毒软件/EDR — 恶意代码防范
☑ 备份系统 — 本地+异地备份

推荐部署架构：
  Internet → DDoS → 边界FW → IPS → WAF → 核心FW → 内网
                                    ↓
                                 DMZ（Web/反向代理）
```

## 七、阶段五：测评配合

### 7.1 测评前准备

```
测评前1周必须完成：

硬件准备：
  □ 安全产品全部运行（绿灯，无告警）
  □ 服务器/网络设备正常运行
  □ 日志系统正常运行

文档准备：
  □ 管理制度文件装订成册（8大类别）
  □ 定级报告+备案证明 复印件
  □ 拓扑图+数据流图（更新到最新）
  □ 安全配置记录（防火墙策略/ACL/安全基线）

人员准备：
  □ 测评期间至少1名系统管理员+1名安全管理员在位
  □ 机房/网络/数据库管理员联系方式
  □ 测评师的陪同人员安排

环境准备：
  □ 机房整洁（线缆规整/温湿度正常/消防器材在位）
  □ 测评用测试机准备（安装测评工具）
```

### 7.2 测评当天配合

```
Day 1-2: 文档审查 + 访谈
  ✓ 测评师查看制度文件 → 准备好所有文件
  ✓ 访谈安全管理员/系统管理员/网络管理员
  ✓ 可能的问题：
    "密码策略是什么？" → 现场登录展示
    "备份恢复测试做过吗？" → 展示恢复测试记录
    "审计日志保留多久？" → 展示日志系统

Day 2-4: 技术检测
  ✓ 漏洞扫描（通常夜间执行）
  ✓ 配置核查（基线检查）
  ✓ 渗透测试（授权范围内的安全测试）
  
Day 4-5: 结果确认
  ✓ 测评师输出初步问题清单
  ✓ 双方确认问题 + 签字
```

### 7.3 测评结果判定

```
等保三级通过标准：

  综合得分 ≥ 70 分    → 基本通过
  综合得分 < 70 分    → 不通过
  存在高风险项         → 一票不通过

如果测评不通过：
  → 90天内完成整改
  → 申请复测（费用约首次的50-70%）
  → 复测合格 → 获取报告
```

## 八、阶段六：持续监督

### 8.1 年度合规要求

```
等保三级系统年度持续合规：

1. 年度测评（≥1次/年）
   每年委托测评机构进行定期测评
   测评报告报送公安机关

2. 年度自查（≥1次/年）
   安全状况自评估
   12月31日前提交年度自查报告

3. 变更管理
   系统重大变更后可能需要重新定级备案：
   - 系统架构重大变化
   - 安全保护等级变化
   - 系统迁移到云环境

4. 安全事件报告
   发生安全事件后 24 小时内报告公安机关
   重大事件即时报告
```

## 九、完整案例：某企业 OA 等保三级

```
项目概况：
  系统：企业 OA 系统（员工 2000 人）
  目标：通过等保三级测评
  周期：3.5 个月
  预算：50 万元（产品+服务+测评）

时间线：
  Week 1-2:  定级 S3A3 → 专家评审 → 通过
  Week 3-4:  公安机关备案 → 备案证出示
  Week 5-6:  差距分析 → 合规率 45%
  
  关键差距：
    ① 无堡垒机（远程运维明文SSH → 高风险）
    ② 无日志审计系统（日志仅保留7天 → 高风险）
    ③ 弱口令（几个测试账户密码123456 → 高风险）
    ④ 防火墙策略过于宽松（any→any → 高风险）
    ⑤ 管理制度文件不完整（缺5类文件）

  Week 7-12: 建设整改
    ✓ 部署堡垒机（齐治，支持双因素+审计录像）
    ✓ 部署日志审计（日志易，接入20+日志源）
    ✓ 修改所有弱口令+配置密码策略
    ✓ 收紧防火墙策略（白名单模式）
    ✓ 编写管理制度文件（8大类全部补齐）

  Week 13-14: 自评 → 合规率 92%
    仍有8项部分符合（不涉及高风险）

  Week 15: 正式测评
    综合得分: 86.5 → 通过 ✅

  Week 16: 获取测评报告 + 公安局备案完成

关键经验：
  ✓ 高风险项必须清零（一票否决制）
  ✓ 管理制度文件是常见短板（提前准备）
  ✓ 测评期间每天沟通（有问题即时处理）
  ✗ 不要临时抱佛脚（设备采购周期可能需要4-8周）
```

## 十、合规排错指南

```
问题1: 定级被退回
原因：S/A等级论证不充分 / 格式不规范
解决：
  ✓ 按三要素（侵害客体/侵害程度/影响范围）逐条分析
  ✓ 提供量化数据支撑（如：日均处理XX万笔交易）
  ✓ 参照行业案例（金融企业→不低于三级）

问题2: 测评不通过
常见原因：
  ✗ 高风险项未清零
  ✗ 日志审计未开启（或留存<6个月）
  ✗ 管理制度文件缺失
解决：
  → 优先整改高风险项
  → 重新申请复测（90天内）

问题3: 安全产品选型困惑
原则：
  ✓ 必须有国密局认证证书（密码产品）
  ✓ 必须有公安部销售许可证
  ✓ 优先选择有等保三级案例的厂商
  ✓ 国产化优先（信创适配）

问题4: 预算不足
分阶段实施：
  优先级1: 堡垒机+日志审计（高风险直接相关）
  优先级2: 漏洞扫描+数据库审计
  优先级3: WAF+EDR（增强防护）
  
  注意：测评费（8-15万）必须单独预算
```
