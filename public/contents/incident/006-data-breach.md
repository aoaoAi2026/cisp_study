## 一、发现与确认

### 常见发现途径

```
内部发现:
  ✦ DLP告警(大量文件外传)
  ✦ SIEM告警(异常流量出站)
  ✦ 数据库审计(异常查询/大量导出)
  ✦ 员工报告(收到可疑邮件)

外部发现:
  ✦ 安全研究员报告
  ✦ 暗网发现兜售数据
  ✦ 媒体曝光
  ✦ 客户投诉
```

### 初步确认

```sql
-- 确认数据是否真正外泄
-- 1. 查询数据库审计日志
SELECT * FROM audit_log 
WHERE timestamp > '2026-06-15 00:00:00' 
  AND action = 'SELECT' 
  AND row_count > 1000;

-- 2. 检查防火墙出站流量(NetFlow)
-- 某IP大流量出站 → 可能正在外传数据
```

## 二、遏制与止损

```
Step 1: 阻断通道(15分钟)
  □ 防火墙封禁异常出站IP
  □ 安全组隔离受影响主机
  □ 禁用泄露的API Key/AK/SK
  □ 重置受影响用户密码

Step 2: 阻止扩散(1小时)
  □ 排查是否还在持续泄露
  □ 检查其他主机是否有类似外传
  □ 关闭不必要的对外端口
```

## 三、影响评估

```
评估维度:

1. 泄露数据量级
   ✦ 影响用户数
   ✦ 数据敏感度(姓名/手机/身份证/银行卡/密码)
   ✦ 数据是否可关联(组合后危害更大)

2. 泄露时间窗口
   ✦ 最早异常时间 → 当前时间
   ✦ 总持续时长

3. 泄露途径
   ✦ 直接数据库导出
   ✦ API接口批量调用
   ✦ 文件服务器下载
   ✦ 邮件转发

4. 受影响系统
   ✦ 生产数据库?
   ✦ 测试环境(也有真实数据?)
```

## 四、通知与合规

```
中国法律要求:
  ✦ 个人信息泄露 → 通知个人信息主体 + 网信部门
  ✦ 《个人信息保护法》第57条:
    "发生个人信息泄露的，个人信息处理者应当立即采取补救措施，
     并通知履行个人信息保护职责的部门和个人"

通知时限:
  无明确时限规定，但应当"及时"
  行业惯例: 确认后72小时内通知

通知内容(给用户):
  ✦ 泄露了哪些信息
  ✦ 可能的后果
  ✦ 已采取的措施
  ✦ 用户可以做什么(修改密码/警惕钓鱼)
  ✦ 联系方式

通知内容(给监管部门):
  ✦ 泄露详情(类型/数量/时间/原因)
  ✦ 可能造成的影响
  ✦ 已采取的补救措施
  ✦ 下一步整改计划
```

## 五、完整案例

```
案例: 某电商平台 50万用户数据泄露

Day 1: 发现
  安全研究员在暗网发现兜售该平台用户数据
  样本验证: 50条数据 → 真实用户的姓名+手机+地址

Day 1-2: 应急
  ✓ 排查导出源: 数据库审计 → 运维账号 6月10日导出50万条
  ✓ 确认: 该运维账号被盗(弱口令)
  ✓ 遏制: 封禁该账号 + 修改所有运维密码
  ✓ 评估: 50万用户(姓名/手机/地址/部分含身份证)

Day 2-3: 通知
  ✓ 向网信办报告
  ✓ 向受影响用户发送短信提醒
  ✓ 在官网发布公告

Day 3-7: 修复
  ✓ 数据库访问最小权限(运维不应有用户表读权限)
  ✓ 数据库操作审计+告警(大查询自动告警)
  ✓ 数据库运维→堡垒机+审批
  ✓ 加强运维账号MFA

教训:
  ✦ 运维账号权限过大(应只读必要数据)
  ✦ 无大查询告警(50万条查询无感知)
  ✦ 无DLP监控(批量导出无拦截)
```

## ✅ 数据泄露处置清单

- [ ] 确认泄露真实性
- [ ] 阻断泄露通道(封IP/禁AK/重置密码)
- [ ] 评估泄露范围(数据/用户/时间)
- [ ] 通知管理层+法务
- [ ] 通知监管(网信办/公安)
- [ ] 通知受影响用户
- [ ] 根因分析+修复
- [ ] 复盘+制度改进
