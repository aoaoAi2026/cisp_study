# 数据泄露事件处置流程实战

---

## 一、数据泄露事件分类

| 分类 | 典型场景 | 主要风险 |
|------|---------|---------|
| **内部人员泄露** | 员工导出 / 下载 / 邮件外传敏感数据 | 量大、目标精准、常绕过技术防护 |
| **外部入侵泄露** | 勒索双重勒索 / Web 拖库 / 数据库被攻破 | 攻击者主动、针对性强 |
| **合作伙伴泄露** | 云厂商 / 外包 / 渠道商数据泄露 | 责任归属复杂 |
| **公开信息泄露** | GitHub / 内网 Wiki / 配置文件 / 云存储桶公开 | 量大难发现、常被爬虫抓取 |
| **应用层泄露** | SQL 注入、越权、文件包含导致数据被读取 | 攻击者逐步渗透获取数据 |
| **邮件 / IM 泄露** | 钓鱼 + 社工获取账号后批量下载邮件 | 历史邮件可能包含大量秘密信息 |

## 二、数据分类分级（响应前必须完成的基础工作

```
  P1 公开信息 (Public)              = 无风险
  P2 内部信息 (Internal)            = 不对外, 但泄露不违法
  P3 敏感信息 (Confidential)     = 员工 PII / 合同未公开版
  P4 机密信息 (Restricted)        = 财务核心 / 源代码 / 客户 PII
  P5 受保护健康信息 (PHI)          = 健康数据 / 医疗数据
  P6 支付卡数据 (PCI)               = 信用卡号 / CVV

  针对 P3~P6 数据必须:
    - 加密存储 (AES-256 + KMS)
    - 加密传输 (TLS 1.2+)
    - DLP 监控
    - 数据库审计 + 文件审计
    - 最小权限原则 + 审批
    - 定期备份并加密
```

## 三、发现信号 (Detection)

```
技术监控 (必须实时):
  1. DLP 告警: 邮件外发含 PII / 外网下载 > N MB
  2. SIEM 规则:
     - 同一账号 1h 内下载 > N 份文档
     - 非工作时间 / 非办公 IP 访问核心系统
     - 数据库批量 SELECT (行数 > 阈值)
     - 大量 Export API 被调用
  3. UEBA: 用户行为基线偏离
  4. 云桶访问审计: S3 / OSS 访问日志
  5. Git 扫描: git-secrets / trufflehog (内部仓库出现公网泄露)

业务信号 (需业务配合):
  1. 用户举报: 收到陌生账号登录 / 钓鱼邮件
  2. 客户投诉: 被冒用身份 / 信用卡异常消费
  3. 执法机构通报
  4. 暗网情报: 数据在暗网售卖
```

## 四、黄金响应流程 (PIPL / GDPR 下 72h 内可能需上报

```
Phase 1. 初步评估 (< 1 小时)
  - 确认是否真实泄露 (样本验证 / hash 比对 / 是否为内部数据
  - 初步识别: 泄露范围、敏感级别
  - 泄露方式: 主动外泄 / 被拖库 / 系统漏洞?
  - 启动应急指挥: 成立 IR 小组 (IC + 技术 + 法务 + 公关 + HR + 业务线)

Phase 2. 遏制 (< 4 小时)
  - 关闭泄露通道 (修复漏洞接口 / 关闭公开桶 / 强制账号下线
  - 暂停可疑账号 (锁定 + 强制改密
  - 阻断可疑 IP / 代理 / Tor 出口节点
  - 保留取证 (磁盘 / 内存 / 日志, 不要直接删除原始数据)
  - 冻结相关系统变更 (freeze code / freeze 配置, 避免破坏证据)

Phase 3. 范围评估 (< 24 小时)
  - 数据库: 访问日志 (慢查询 / 审计日志) 排查异常大查询
  - Web 层: access.log 查可疑 POST / 下载接口
  - 文件层: 文件变更 + DLP 日志
  - 邮件: 邮件网关 / DLP 邮件日志 (含附件 hash)
  - 账号层: 谁在何时从哪里登录
  - 影响: 被泄数据字段 (身份证? 手机号? 邮箱? 支付?)、记录数、时间跨度

Phase 4. 根源分析 (< 48 小时)
  - 技术根因: 漏洞 / 弱密码 / 缺失鉴权 / 内部违规 / 供应链攻击
  - 流程根因: 缺失审批 / 缺失监控 / 缺失培训
  - 组织根因: 安全人力不足 / 资源投入不足 / 优先级过低

Phase 5. 补救 (48~72h 起, 长期)
  - 立即补救: 漏洞修复、密码重置、AK/SK 轮换、补丁上线
  - 中期补救: DLP 新规则、新增监控、权限收敛
  - 长期补救: 数据分类治理、零信任改造、数据脱敏系统、数据库审计全覆盖

Phase 6. 通知 (< 72h 或按监管要求)
  - 内部通知: 高管 + 相关业务线
  - 外部通知: 受影响用户 / 客户 / 监管机构 / 媒体
  - 法律通知: 律师 + 合规, 根据 PIPL / GDPR 72h 内向主管部门报告

Phase 7. 长期改进 (持续)
  - 整改清单: 每个 Action Item 明确责任人、截止日、验收标准
  - 培训: 全员安全意识 + 开发安全编码 + 管理层安全责任
  - 技术加固: DLP / DAP / 数据库审计 / 脱敏 / 加密 / 零信任
  - 合规: 隐私政策更新、用户同意流程重构、DPIA
```

## 五、数据泄露量级快速评估

```
Step 1. 识别被泄字段:
  - 姓名 / 手机号 / 身份证 / 邮箱 / 住址 / 银行卡 / 密码 / CVV

Step 2. 估算记录数:
  - 数据库: 相关表行数 / 最近 dump 时间
  - 文件: 文件大小 / 行数
  - 邮件: 附件大小 / 导出 CSV 行数

Step 3. 识别敏感级别:
  - 身份证号 + 姓名 + 手机号 = 高敏感 (PIPL 下为"敏感个人信息")

Step 4. 影响用户数 / 客户数 / 区域
  - 按地区 (是否有 EU 用户 → GDPR)

Step 5. 风险矩阵 (Likelihood × Impact
```

## 六、具体操作要点

### 6.1 保留证据 (法律 + 合规)

```bash
# 1. 立即冻结相关账号, 但不删除数据
# 2. 对被访问的数据库立即开启审计 (如未开启)
# 3. 拉取最近 90 天全量日志
ls -la /var/log/
journalctl --since "90 days ago" --no-pager > /evidence/journalctl.log
# 4. 保存访问日志 (Web / API / 数据库 / 文件)
cp -a /var/log/nginx /var/log/mysql /evidence/
# 5. 云审计
#    AWS: aws cloudtrail lookup-events
#    阿里云: aliyun actiontrail LookupEvents
# 6. git log / GitLab Audit Events
git log -p --all -- source/secret.txt > /evidence/git.log
# 7. Snapshots (VMware / 云快照)
#    立即对关键数据库 / 应用服务器打快照
# 8. 屏幕截图: 泄露源网站 / 论坛 / 暗网帖子截图
```

### 6.2 快速识别是否真实泄露

```bash
# 1. 验证泄露样本 hash 是否真的来自我方
#    取我方数据库样本行, 与被泄文件比对
echo "样本行内容" | sha256sum
# 2. 看数据结构是否为我方格式 (手机号段、用户 ID 段、特殊格式)

# 3. 查看最近导出 / 下载行为
grep -E "(ExportToExcel|Export|Download|export.csv|download|csv|excel" /var/log/nginx/access.log
# 数据库层
grep -E "(SELECT|into outfile|into dumpfile|LOAD DATA)" /var/log/mysql/slow.log

# 4. 检查批量查询
#    MySQL: show processlist; 看长事务、大查询
#    PostgreSQL: SELECT * FROM pg_stat_activity;

# 5. 看最近的备份文件、dump 文件
find / -name "*.sql" -o -name "*.bak" -o -name "*.csv" -o -name "*.gz" 2>/dev/null | head -30
```

### 6.3 大规模密码处理

```bash
# 1. 立即强制重置所有可能被泄的密码
#    - 数据库密码 (root / 应用账号 / DBA 账号)
#    - 应用层密码
#    - SSH 堡垒机密码
#    - LDAP / AD 密码
#    - API Key / AccessKey / Access Key

# 2. 立即失效所有 session / JWT
#    - Redis: FLUSHALL
#    - 短 token (重启应用)
#    - 清理所有在线登录 session

# 3. 通知所有用户强制改密 (发邮件 / SMS)
#    - 建议: 被泄用户在 72h 内完成密码重置, 并开启 MFA
```

## 七、通知与合规要点

### 7.1 通知时间要求

| 法规 | 通知时限 | 监管机构 |
|------|---------|---------|
| **PIPL (中国个人信息保护法) | "及时" 告知, 实际建议 72h 内 | 网信部门 / 行业监管 |
| **GDPR (欧盟) | 72h | DPA (各国数据保护机构) |
| **等保 2.0 | 发生重大安全事件立即报告 | 公安部网安 |
| **PCI DSS** | 4h 内通知 acquiring bank | 收单行 / 卡组织 |
| **内部政策** | 根据内部 SLA | CISO / CIO |

### 7.2 用户通知模板

```
尊敬的 [用户姓名]:

我们最近检测到 [简述事件]。我们已采取以下措施:

   - 登录后强制改密
   - 开启双因素认证
   - 检查最近登录记录
如需帮助请联系: support@company.com
```

## 八、长期整改技术加固要点

```
技术 (TLS everywhere
1. 数据加密:
   - 静态加密: AES-256 + KMS 密钥管理
   - 传输加密: TLS 1.2+
   - 字段级加密: PII 字段独立密钥

2. 权限与审计:
   - 数据库审计: 数据库审计插件开启, 保留 ≥ 180 天
   - DLP: 邮件 / Web / 终端三层覆盖 PII 规则齐全
   - 数据分类: 自动打标, 高敏感自动告警
   - UEBA: 行为基线 + 异常检测

3. 身份与访问:
   - 零信任: 按身份认证
   - MFA: 所有管理账号、远程访问
   - 权限最小化: 定期 access review

4. 监控与响应:
   - 数据库访问: 专人监控数据库审计日志
   - DLP 监控: 专人监控 DLP 告警
   - 暗网监控: 定期搜索公司域名/关键词
   - 安全运营中心 (SOC) 7×24h
```

## 九、数据泄露响应 Playbook (速查)

```
发现 → 确认 → 启动应急 → 隔离 → 取证 → 评估影响 → 根源分析 → 补救 → 通知 → 长期整改

Checklist:
- [ ] 确认泄露真实性 (样本验证 / hash 比对)
- [ ] 冻结并保留证据链 (磁盘镜像 / 日志 / 屏幕截图 / hash)
- [ ] 立即遏制: 修复漏洞 / 关闭桶 / 锁定账号 / IP 阻断
- [ ] 范围评估: 被泄表 / 字段 / 记录数 / 用户数 / 区域
- [ ] 敏感级别评估 (PII? PHI? PCI? 源代码?)
- [ ] 根源分析: 漏洞 / 弱密码 / 内部违规 / 供应链
- [ ] 立即补救: 密码重置、补丁、AK/SK 轮换、权限收敛
- [ ] 通知: 内部 / 用户 / 客户 / 监管 / 媒体 (法务)
- [ ] 报告: 时间轴、攻击链、影响面、证据、整改计划
- [ ] 7×24h 持续监控, 观察是否二次入侵
- [ ] Post-Mortem 复盘会议, Action Items 跟踪
- [ ] 技术加固: DLP / 加密 / 审计 / 权限收敛
- [ ] 培训: 全员安全意识 + 开发安全编码
```
