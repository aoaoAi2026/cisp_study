# APT 事件应急响应全流程

---

## 一、APT 攻击识别信号

```
APT 与普通攻击的区别：
  普通攻击：快进快出，目的明确（勒索/窃取数据）
  APT攻击：长期潜伏（数月至数年），多阶段，隐蔽性强

APT 识别信号：

  网络层：
  ✦ 异常DNS查询（DGA域名/长域名/低频域名）
  ✦ 非标准端口HTTPS出站（443外的TLS通信 → 隐藏C2）
  ✦ 心跳式通信（每隔固定时间（如60秒）发送小包）
  ✦ 数据在非工作时间外传

  主机层：
  ✦ 异常计划任务（名称伪装成系统任务）
  ✦ WMI事件订阅（无文件持久化）
  ✦ 未知服务（名称仿冒系统服务）
  ✦ 非系统路径的DLL注入

  账户层：
  ✦ 非工作时间域管登录
  ✦ 服务账户被用于交互式登录
  ✦ 大量Kerberos TGS请求（Kerberoasting）
```

## 二、六阶段应急响应

```
NIST SP 800-61 应急响应生命周期：

Phase 1: 准备 (Preparation)
  → APT应急工具箱就绪（取证工具/威胁情报/通信加密）
  → 建立与CERT/安全厂商/监管机构的联络通道

Phase 2: 检测与分析 (Detection & Analysis)
  → 确定入侵时间窗口（最早异常事件时间）
  → 识别攻击向量（钓鱼/漏洞利用/第三方？）
  → 评估影响范围（多少主机/哪些系统/什么数据）

Phase 3: 遏制 (Containment)
  → 短期遏制：隔离受感染主机（VLAN隔离/断网）
  → 长期遏制：修补入口、重置凭据、加固系统

Phase 4: 根除 (Eradication)
  → 清除恶意软件+后门+持久化机制
  → 重置所有受影响账户密码
  → 修补漏洞

Phase 5: 恢复 (Recovery)
  → 从已知干净的备份恢复系统
  → 增强监控后重新上线
  → 持续观察（至少2周强化监控）

Phase 6: 复盘 (Lessons Learned)
  → APT TTPs提取 → 补充检测规则
  → 安全架构改进
  → 应急响应流程优化
```

## 三、APT 取证特殊考虑

```
与普通事件取证的区别：

1. 时间跨度大 → 需要几个月甚至几年的日志
   → SIEM必须有长期存储能力（≥1年）
   → 云日志（CloudTrail/Azure Monitor）需开启+长期保留

2. 攻击者可能仍在系统中 → 取证过程不能惊动攻击者
   → 优先离线分析（从备份/镜像取证）
   → 网络层监控优于主机层（不被攻击者感知）

3. 多阶段攻击链 → 需要完整还原攻击路径
   → 使用时间线工具(Plaso/log2timeline)重建
   → 关联网络日志+主机日志+应用日志

4. 可能有数据已外泄 → 需要评估泄露范围
   → 分析出站流量(DNS日志/NetFlow/Firewall日志)
   → 评估泄露数据的类型和量级
```

## 四、Checklist

- [ ] APT应急工具箱就绪
- [ ] SIEM日志长期存储(≥1年)
- [ ] 威胁情报源接入（MISP/商业情报）
- [ ] 应急响应外部联络通道（CERT/监管/供应商）
- [ ] 入侵时间窗口确定
- [ ] 攻击链还原（MITRE ATT&CK映射）
- [ ] 遏制措施执行+根除+恢复
- [ ] 复盘+TTP提取+规则更新
