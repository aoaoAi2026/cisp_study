# 威胁情报分析报告模板与写作实战

> **📘 文档定位**：CISP 考试威胁情报实践内容 | 难度：⭐⭐⭐ | 预计阅读：15 分钟
> 威胁情报报告是情报工作的最终产出。本文从报告类型、写作框架到分发策略，系统梳理威胁情报报告撰写规范。

## 导航目录
- [一、威胁情报报告的类型与用途](#一威胁情报报告的类型与用途)

| 类型 | 频率 | 读者 | 长度 | 核心价值 |
|------|------|------|------|---------|
| **实时告警 (Alert)** | 即时 (5 分钟内) | SOC L1/L2 | 1 页 | 快速识别+阻断 |
| **IOC 清单 (Indicator Feed)** | 即时 / 每日 | SOC / WAF / EDR | N/A (机器可读) | 直接消费 |
| **事件报告 (Incident Report)** | 事件发生后 24h | CISO / 法务 / 业务 | 5-20 页 | 说明发生了什么 / 影响 / 整改 |
| **威胁简报 (Threat Brief)** | 每周 / 每月 | CISO / 管理层 | 2-5 页 | 行业威胁态势总览 |
| **APT 报告 (APT Report)** | 按需 | 安全团队 / 行业 ISAC | 10-50 页 | 详细 TTP + 样本分析 |
| **年度威胁报告** | 年度 | CEO / 董事会 / 监管 | 50-200 页 | 年度安全态势 + 来年建议 |

## 二、通用威胁情报报告模板

```
  报告标题: [威胁/事件名] 威胁情报报告 - v1.0
  报告编号: INC-2025-00xx
  作者: 网安百宝箱编辑部
  发布日期: YYYY-MM-DD
  机密等级: TLP:WHITE / GREEN / AMBER / RED
  版本历史: v1.0 (YYYY-MM-DD) 初稿

  ┌─────────────────────────────────────────────────────────────┐
  │ 1. 执行摘要 (Executive Summary)                             │
  │    1.1 一句话描述:                                        │
  │         本报告描述 [威胁/事件] 的技术细节、影响范围和建议。│
  │    1.2 关键发现 (Bullet Points, 3-5 条):                   │
  │         • 攻击始于 YYYY-MM-DD, 初始入口为 [方式]。        │
  │         • 被利用的漏洞为 CVE-xxxx-xxxx, 影响版本: xxx。    │
  │         • 攻击者窃取了 [数据类型], 但未发生进一步外泄。    │
  │         • 已识别 IOC 共 N 条, 建议立即阻断。              │
  │         • 建议企业在 [时间] 前完成 [整改]。                │
  │    1.3 风险评级:                                          │
  │         严重度: CRITICAL / HIGH / MEDIUM / LOW            │

  │         潜在损失: [估算]                                   │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ 2. 攻击时间轴 (Timeline - 按时间顺序)                      │

  │   YYYY-MM-DD HH:MM UTC - 攻击者首次成功登录               │
  │   YYYY-MM-DD HH:MM UTC - 横向移动开始                      │
  │   YYYY-MM-DD HH:MM UTC - 数据外泄                          │

  │   YYYY-MM-DD HH:MM UTC - 隔离 / 阻断                       │

  ┌─────────────────────────────────────────────────────────────┐
  │ 3. 攻击链详解 (Attack Chain - 按 MITRE ATT&CK 阶段)         │
  │   3.1 Initial Access (Txxxx)                               │
  │       详细描述: [技术细节 / 证据截图 / 日志片段]           │
  │   3.2 Execution (Txxxx)                                    │
  │       ...                                                  │
  │   3.x ... (每个阶段都应包含: 行为描述 + 证据 + 影响)       │
  │                                                            │
  │ 建议附图:                                                   │
  │   • 攻击链流程图 (Flow Chart)                               │
  │   • ATT&CK 矩阵热力图 (我们检测到了哪些技术)               │
  │   • 网络拓扑图 (攻击路径)                                   │
  │   • 关键日志 / 样本截图 (脱敏)                              │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ 4. IOC 清单 (Indicators of Compromise)                     │
  │   4.1 IP 地址 (C2 / 攻击源)                                  │
  │   4.2 域名 / URL                                            │
  │   4.3 文件 Hash (MD5 / SHA1 / SHA256)                       │
  │   4.4 YARA 规则                                             │
  │   4.5 文件名 / 路径 / 注册表键                               │
  │   4.6 JA3 / JA3S 指纹                                       │
  │                                                            │
  │ (以 CSV/JSON 机器可读方式提供, 便于自动化消费)               │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ 5. 建议 (Recommendations)                                   │
  │   5.1 紧急 (立即执行 - 24h 内)                               │

  │     • 重置相关账号密码                                      │
  │     • 部署相关 YARA 规则                                    │
  │   5.2 短期 (1 周内)                                          │
  │     • 补丁 / 升级受影响组件                                │
  │     • 修复漏洞 (漏洞利用路径)                                │
  │     • 完善日志记录 / 监测点                                  │
  │   5.3 中期 (1 月内)                                          │
  │     • 部署 EDR / XDR                                        │
  │     • 强化身份验证 (MFA)                                    │
  │     • 蓝队 / 红队演练                                       │
  │   5.4 长期 (季度)                                            │
  │     • 建立威胁情报订阅                                       │
  │     • 制定事件响应 Playbook                                 │
  │     • 安全意识培训                                          │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ 6. 参考与致谢 (References)                                  │
  │   6.1 外部引用链接 (厂商公告 / CVE 详情 / MITRE)             │
  │   6.2 内部引用 (内部报告 / 会议纪要)                         │
  │   6.3 致谢 (CERT / ISAC / 合作伙伴)                          │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ 7. 附录 (Appendix)                                         │
  │   7.1 术语表 (Glossary)                                    │
  │   7.2 版本历史                                              │
  │   7.3 作者 / 联系人信息                                      │
  │   7.4 Legal Disclaimer (法律声明)                            │

```

## 三、写作要点与最佳实践

```
  1. 读者优先:
     - 给高管看: 关注 1 页摘要 + 5 条建议, 不要写技术细节
     - 给 SOC 看: IOC 要机器可读 (CSV/JSON/STIX), TTP 要详细
     - 给技术团队看: 需要完整攻击链 + 日志片段 + 漏洞 PoC

  2. 数据说话:
     - 所有主张必须有证据 (日志 / 截图 / 样本哈希)
     - 不要写 "可能 / 也许", 要写 "根据证据 xxx, 我们判断 yyy"

  3. 结构化:
     - 段落短, 标题清晰, 使用 bullet points
     - 善用图表 / 表格, 降低阅读成本

  4. 可操作:
     - 每条建议要有明确 action + 负责人 + 截止时间
     - 不要写 "加强安全意识", 而要写 "在 2025-10-01 前完成对全公司 80%员工的钓鱼模拟测试"

  5. 脱敏:
     - 真实 IP / 用户名 / 服务器名 / 客户数据必须打码
     - 用 "服务器 A" / "用户 X" / "业务系统 Y" 替代
     - 对外报告尤其要规避泄露客户信息

  6. 版本控制:
     - 每次修改都 increment version (v1.0 → v1.1 → v2.0)
     - 记录修改内容, 便于追溯

  7. 法律合规:
     - 声明报告仅内部使用
     - 避免使用 "攻击源来自 XX 国" 等政治敏感表述 (除非有 100%证据)
     - 有法务团队协助审查对外报告
```

## 五、威胁简报示例 (每月给管理层)

```
  [公司 Logo]
  2025 年 9 月威胁简报

  本月关键发现:
    • 全球勒索软件活动: LockBit / BlackCat / Play 家族持续活跃, 针对本行业攻击 12 起
    • 关键漏洞: CVE-2025-xxxx (CVSS 10.0), 影响 [某组件], 本公司已修补 98%
    • 钓鱼邮件: 本月收到 3 波鱼叉钓鱼邮件, 来自冒充 [合作伙伴] 的邮箱
    • 内部事件: 检测到 2 起疑似暴力破解 RDP, 已自动阻断, 无进一步影响

  本月防御成果:
    • 新增 IOC 1,245 条, 自动阻断 320 次
    • YARA 规则命中 15 次 (7 次为真实恶意, 8 次误报, 已优化)
    • 员工钓鱼模拟点击率: 8% (上个月 12%, 持续改善)

  下月重点:
    • 部署 EDR 到剩余的 10%服务器
    • 开展勒索软件专项应急演练
    • 订阅 [X] 商业情报源, 提升 APT 检测能力

  重点 IOC 摘要 (完整清单见 TIP 平台):
    [表格: 10 条高置信度 IOC]

  APT 关注:
    • [APT 组织名] 近期针对 [行业] 有活动, 建议加强邮件网关规则与 MFA
    • TTP: T1566 Phishing → T1059 PowerShell → T1003 Credential Dumping

  [报告人 / 日期 / 版本]
```

## 六、报告工具与发布方式

```
  1. 纯文本 / Markdown: 轻量, Git 管理, 适合内部
  2. PDF: 最适合对外/管理层阅读 (可从 Markdown 自动生成: pandoc / weasyprint)
  3. MISP Event: 结构化情报, 便于 STIX 消费
  4. Confluence / Wiki: 团队内部知识库, 便于协作
  5. 邮件: 摘要版邮件发送管理层 / 全员安全通告
  6. 自动化推送: 通过 Webhook / API 自动推送到 Slack/飞书/钉钉群
```

## 七、CheckList

- [ ] 报告模板已标准化, 所有分析师按统一格式写作
- [ ] IOC 部分提供机器可读格式 (CSV / JSON / STIX)
- [ ] 每个技术主张都有证据支持 (截图 / 日志片段 / hash)
- [ ] 有明确的建议部分 (立即 / 短期 / 中期 / 长期)
- [ ] 有版本号 / 作者 / 日期 / 法律声明
- [ ] 所有敏感信息已脱敏, 不泄露客户 / 员工隐私
- [ ] 对外报告经过法务 / CISO 双审查
- [ ] IOC 已同步推送到 TIP 平台与防护设备
- [ ] 有明确的更新机制, 过期信息会被标注
- [ ] 定期 (每月 / 每季度) 发送威胁简报到管理层
- [ ] 报告接受反馈, 根据读者反馈持续优化模板
- [ ] 报告存档时间 ≥ 3 年 (合规需要)

## 高分考点与知识巧记

> 🔑 **高分考点**：威胁报告考点集中在报告分级、TLP 标记、关键要素。

| 考点 | 频次 | 核心记忆点 |
|:---|:---:|:---|
| 三级报告 | ⭐⭐⭐⭐ | 战略级(高管)、作战级(经理)、战术级(分析师) |
| TLP 标记 | ⭐⭐⭐⭐ | RED(个人)→AMBER(组织)→GREEN(社区)→WHITE(公开) |
| 关键要素 | ⭐⭐⭐ | 执行摘要 + 技术分析 + IOC 附录 + 防护建议 |

> 💡 **知识巧记**：TLP 四级记"红黄绿白"，报告要素记"摘分录防"。

### 高分考点速查表

| 考察维度 | 关键结论 | 常见干扰项 |
|:---|:---|:---|
| TLP:RED | 仅指定个人，不得分发 | "TLP:RED 可内部转发" ❌ |
| 报告受众 | 不同级别不同内容深度 | "一份报告适用所有人" ❌ |

### 知识巧记口诀

> **威胁报告口诀**：
> 三级受众战作战术分，TLP 红黄绿白标记清。
> 摘要分析 IOC 加建议，战术级给分析师看分明。
