# 等保三级与护网的衔接差异：从合规到实战

**等保 2.0（《网络安全等级保护 2.0》）** 的三级要求，本质上是国家对"关键信息基础设施之外的重要系统"给出的**最低合规基线**。而护网行动则是**实战化、对抗性**的网络攻防演习。许多单位投入大量人力通过了等保三级测评，却在护网中依然被轻松攻破 —— 原因在于：**合规 ≠ 安全**。本文讲两者差异，并给出把"等保成果"转化为"护网能力"的落地思路。

## 1. 核心差异一览

| 维度 | 等保三级 | 护网实战 |
|------|----------|----------|
| 评价方式 | 现场测评、材料检查、访谈、抽样 | 红队真实攻击、看是否能被攻破、看被攻破时间 |
| 关注重点 | 制度、文档、策略覆盖、合规痕迹 | 攻击链完整性、边界是否可突破、内网是否可横向、核心数据是否可获取 |
| 时间周期 | 一次测评管 2 年 | 每年集中 1~2 周，外加持续性"护网思维"建设 |
| 典型关注点 | 是否有制度、是否有审批、是否有日志 | 日志是否**被使用**、策略是否**真生效**、是否有**应急能力** |
| 扣分代表 | 缺少某项管理文档 / 策略未配置 | 被成功登录、上传 WebShell、数据外泄、内网横向 |

一句话：**等保关心"做没做"，护网关心"有没有用"。**

## 2. 等保常见"纸面合规"与实战失效点

### 2.1 制度多、落地少

- 有"应急响应预案"，但从未演练；护网时手忙脚乱。
- 有"人员安全教育要求"，但员工仍相信钓鱼邮件中的"人事通知"。
- 有"第三方安全协议"，但外包账号权限比内部员工还大。

### 2.2 日志留了、没人看

等保要求"至少 6 个月日志留存"，许多单位因此买了日志系统。但：

- 日志字段**不统一**，无法检索；
- 日志存储**过载**，关键事件被淹没；
- 没有 SIEM / 规则 / 人值守，日志只是"为了查的时候能查到"。

### 2.3 策略配置"有就行"，不管实际效果

- 防火墙有策略，但**全通 ANY-ANY** 的白规则最优先。
- 堡垒机号称统一入口，但核心服务器依然允许 SSH 密码直连。
- WAF 部署了，但运行在"观察模式"。
- 有备份策略，但从未做过**恢复验证**。

### 2.4 漏洞"修过就行"，缺乏生命周期管理

- 高危漏洞被"延期处理"或"业务繁忙搁置"。
- 修复后没有复测，也没有补丁回滚预案。
- 大量 0day/Nday 漏洞不在等保测评项目内，护网一上场就中招。

## 3. 把等保成果转化为护网能力的 5 个抓手

### 3.1 抓手一：把"制度"升级为"剧本 + 演练"

等保要求的《安全事件处置预案》需要升级为**可执行 Playbook**：

```yaml
# 示例：Web 系统被植入 WebShell 的 Playbook 简化版
playbook: webshell_incident
trigger:
  - WAF 报告 WebShell 上传成功
  - EDR 报告可疑进程（cmd.exe / bash 从 Web 目录 spawn）
  - 人工确认文件 Web 目录有可疑 jsp/php/aspx
step_1:
  action: 保留现场
  detail: 不删除文件、保留内存与磁盘镜像（如可能）、留一份 PCAP
step_2:
  action: 隔离主机
  detail: 在 EDR/防火墙隔离该主机，防止横向
step_3:
  action: 研判攻击路径
  detail: 回溯 Web 访问日志，定位上传点 / 漏洞点
step_4:
  action: 修补漏洞
  detail: 关相关接口 / 打补丁 / 改配置
step_5:
  action: 改密 + 审计
  detail: 重置该系统相关账号密码，审计近 72 小时登录
step_6:
  action: 复盘 + 上报
  detail: 24 小时内输出事件报告，更新检测规则
```

> **护网动作**：在护网前至少完成 1~2 次**实战演练**（桌面推演 + 红队小范围测试），把 Playbook 跑通。

### 3.2 抓手二：把"留日志"升级为"用日志"

从等保视角：**有日志 × 有留存** → 合规。
从护网视角：**能检索 + 有关联 + 有告警 + 有人看** → 有用。

| 等保基础 | 护网升级 |
|----------|----------|
| 有日志采集 | 统一字段（IP / 时间 / 用户 / 动作 / 结果） |
| 日志留存 ≥ 6 个月 | 重点日志留存 ≥ 1 年 + PCAP 7~30 天 |
| 手工审计 | SIEM 关联分析 + 7x24 值守 |
| 抽样抽查 | 全量接入、基于规则 + UEBA 主动检测 |

### 3.3 抓手三：把"策略文件"升级为"最小权限 + 基线核查"

等保三级对访问控制的典型要求是"按用户分配权限、按角色审计"。护网需要更进一步：

- 边界防火墙：**默认拒绝 + 白名单端口**，定期核查开放端口清单。
- 服务器：禁用 root 远程登录、改用 key、启用 MFA；高危端口（22/3389/6379/3306）只允许特定源。
- 应用：统一走 SSO，废除本地账号，接口鉴权最小化。
- 数据库：限制访问源 IP + 限制执行权限 + 审计高危语句（`select * from` 大表、`drop`、`outfile`）。

```bash
# 示例：一台 Linux 主机的"基线加固"基本命令（等保 + 护网叠加）
passwd -l root                            # 锁定本地 root（通过 sudo 管理）
sed -i 's/PermitRootLogin yes/PermitRootLogin no/' /etc/ssh/sshd_config
sed -i 's/#PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
systemctl restart sshd
echo 'install cramfs /bin/true' >> /etc/modprobe.d/CIS.conf  # 禁用不必要文件系统
apt-get install -y fail2ban               # 爆破自动封禁
systemctl enable --now fail2ban
```

### 3.4 抓手四：把"漏洞整改报告"升级为"漏洞闭环系统"

- 等保只要求"对发现的问题进行整改"。
- 护网要求"发现 → 定级 → 分派 → 修复 → 复测 → 豁免 → 可视化"全闭环。

详细做法可参考《漏洞管理闭环：发现 - 修复 - 验证全流程》（hw-014）。

### 3.5 抓手五：把"等保测评"升级为"常态化红蓝对抗"

- **一年一次等保测评** 是合规体检。
- **一月一次小攻防、半年一次大攻防** 才是实战能力养成。

在等保三级基础上，再叠加：

- 护网前 1 个月由**内部红队**或外部专业团队做一次模拟攻击；
- 所有发现问题作为"高优先级整改项"；
- 整改后复测，直到核心系统**在限定时间内未被攻陷**。

## 4. 一份"等保 → 护网"自查清单

在护网开打前，对着等保三级清单检查一次，但**换一种问法**：

| 等保问题 | 护网问法 |
|----------|----------|
| 是否有访问控制策略？ | 边界防火墙是否存在 ANY-ANY 规则？是否定期核查开放端口？ |
| 是否有日志留存？ | 关键系统日志是否入 SIEM？是否有 L1 人员盯？ |
| 是否有应急响应预案？ | 最近一次演练是什么时候？Playbook 是否能 10 分钟内启动？ |
| 是否有漏洞管理？ | 近 3 个月 P0/P1 漏洞是否 100% 修复？是否有自动复测？ |
| 是否做了安全培训？ | 员工最近一次钓鱼测试的点击率是多少？是否有改进课程？ |
| 是否有身份鉴别？ | 核心系统是否启用 MFA？是否仍存在默认口令 / 弱口令？ |
| 是否有备份？ | 最近一次备份恢复演练是什么时候？恢复耗时多久？ |

## 5. 总结：从"过关"到"能打"

- 等保三级给了**基础框架**：制度、日志、基线、访问控制、安全管理。
- 护网把框架变成**实战能力**：Playbook、SIEM、红蓝对抗、漏洞闭环、持续运营。
- 正确姿势不是"先做等保，再补护网"，而是在等保建设时**用护网思维倒推**每一个控制点。

把等保测评报告里的"符合项"再问一句 **"攻击者能绕过去吗？"**，就是迈向"实战能力"的第一步。
