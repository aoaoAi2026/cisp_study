## 一、实战场景总览

```
六个典型事件场景：

场景一：勒索软件攻击
├── 触发：用户报告文件被加密，出现勒索信息
├── 难点：终止加密扩散 vs 保留加密样本
└── 优先级：遏制扩散 > 清除 > 恢复

场景二：凭证窃取与横向移动
├── 触发：SIEM 告警 → 同用户短时间内登录多台主机
├── 难点：确认哪些主机被控、哪些凭证已泄露
└── 优先级：禁用被盗账户 > 遏制被控主机

场景三：WebShell 后门
├── 触发：WAF/安全扫描发现可疑 PHP/JSP/ASPX 文件
├── 难点：一个 WebShell 往往意味着还有更多
└── 优先级：隔离 → 全面搜索 → 清除后门

场景四：数据泄露
├── 触发：DLP 告警/暗网发现数据/外部通报
├── 难点：确定泄露范围、满足通知义务
└── 优先级：止损 > 取证 > 通知

场景五：DDoS 攻击
├── 触发：业务监控发现服务不可用，流量暴涨
├── 难点：区分攻击流量 vs 正常流量激增
└── 优先级：恢复可用性 > 分析攻击源

场景六：内部威胁
├── 触发：审计发现离职员工异常数据访问
├── 难点：区分正常操作 vs 恶意行为
└── 优先级：权限回收 > 取证 > 法务介入
```

## 二、场景一：勒索软件攻击

### 2.1 事件场景

```
时间：周一 9:30 AM
告警：财务部 5 名员工报告文件无法打开，出现加密扩展名
发现：共享驱动器中的文件被大量加密
      桌面出现勒索信 README.txt
      要求支付比特币赎金
```

### 2.2 响应步骤

```
Phase 1: 识别 (0-30分钟)

□ 确认告警真实性 → 核实文件确实被加密
□ 识别勒索软件类型：
  ├── 文件扩展名是什么？
  ├── 勒索信内容和风格？
  └── 使用 ID Ransomware 网站上传加密样本识别
□ 确定感染范围：
  ├── 哪些机器被感染？（EDR 查询）
  ├── 共享驱动器影响范围？
  └── 备份是否受影响？（这是恢复的关键！）

Phase 2: 遏制 (30分钟-2小时)

□ 立即操作：
  ├── 切断受影响服务器与网络的连接（但不断电！）
  │   内存中的加密密钥/样本信息断电会丢失
  ├── 禁用受影响用户账户
  ├── 切断共享驱动器的访问权限
  └── 通知全员停止访问共享驱动器
□ 短期遏制：
  ├── 防火墙阻断已知 C2 地址
  └── 增强全网 EDR 监控

Phase 3: 清除 (2-8小时)

□ 保留取证样本：
  ├── 内存 dump (Volatility)
  ├── 加密文件样本 + 勒索信
  └── 磁盘镜像 (加密前的文件可能可恢复)
□ 清除：
  ├── 无解密工具 → 重装系统
  ├── 如有解密工具 → 解密后重装
  └── 所有凭证重置

Phase 4: 恢复 (取决于备份状况)

□ 最佳情况：离线备份完好 → 恢复
□ 次佳情况：备份被加密但有解密工具
□ 最差情况：离线备份也受影响了
□ 恢复后：加强监控至少2周
```

### 2.3 是否支付赎金

```
⚠️ 专业建议：不支付赎金

理由：
├── 付了不一定给解密工具（~40% 付了没拿到）
├── 拿到工具不一定完美解密
├── 付款会资助犯罪组织
├── 第一次付款 → 你会再次成为目标
└── 即使解密了，后门可能还在 → 建议重装

例外：
└── 涉及生命安全（医院系统被加密影响病人）→ 可考虑

CISP 考试立场：推荐不支付赎金
```

## 三、场景二：凭证窃取与横向移动

### 3.1 事件场景

```
时间：凌晨 3:15 AM
告警：SIEM 关联规则触发
  用户 "wang.li" 在30分钟内从主机 A 登录到
  主机 B、C、D（分别是 HR、财务、研发服务器）
  使用 WMI / PSExec 远程执行
```

### 3.2 响应步骤

```
Phase 1: 识别

□ 确认事件：
  ├── 用户 "wang.li" 是什么角色？是否有正常业务需要访问这些服务器？
  ├── 凌晨3点登录是否正常？(大概率异常)
  └── 联系人确认是否本人操作？
□ 确定初始感染点：
  ├── 检查主机 A 的 EDR 记录
  ├── 查找 wang.li 的初始登录时间
  └── 是否有钓鱼邮件/漏洞利用的证据？
□ 确定横向移动路径：
  ├── A → B (通过 PSExec)
  ├── B → C (通过 WMI)
  └── C → D (通过 RDP)
□ 确定被访问/窃取的数据

Phase 2: 遏制

□ 立即操作：
  ├── 禁用用户 wang.li 的账户
  ├── 隔离主机 A、B、C、D (保留取证)
  ├── 重置所有可能接触的凭证
  └── 全局强制 MFA（如之前未开启）
□ 阻断 C2 通信：
  └── 根据 EDR 发现的 C2 IP/域名配置防火墙规则

Phase 3: 清除

□ 彻底清除所有受影响系统上的：
  ├── 恶意软件/Tools (Mimikatz, PSExec, BloodHound)
  ├── 后门账户/计划任务/启动项
  └── 如有疑问 → 重装系统
□ 重置所有凭证（不仅仅是直接受影响的）
  └── "假定所有域内凭证都已泄露"→ 域密码策略重置

Phase 4: 恢复

□ 从备份恢复数据（先验证备份未被污染）
□ 逐步恢复系统上线
□ 部署额外安全控制：
  ├── LAPS (本地管理员密码管理)
  ├── 特权访问管理 (PAM)
  └── 网络分段（限制横向移动）
```

### 3.3 凭证窃取的常见手段

```
攻击者窃取凭证的技术：

Mimikatz
├── 从 LSASS 内存 dump 中提取明文密码和 NTLM hash
└── 防御：启用 Credential Guard、LSASS 保护

Kerberoasting
├── 请求 TGS 票据 → 离线暴力破解
└── 防御：服务账号密码强复杂度 + 定期更换

Pass-the-Hash
├── 用 NTLM hash 直接认证（不需明文密码）
└── 防御：网络分段、限制本地管理员

DCSync
├── 伪装成域控请求复制目录数据
└── 防御：限制域复制的权限
```

## 四、场景三：WebShell 后门攻击

### 4.1 事件场景

```
时间：正常工作时间
告警：安全扫描工具/WAF 发现
  /upload/avatar/2024/connect.php
  文件内容包含 eval(base64_decode(...))
```

### 4.2 响应步骤

```
Phase 1: 识别

□ 确认 WebShell：
  ├── 分析该文件内容 → 确定是 WebShell
  ├── 查看文件创建时间和修改时间
  └── 查看创建该文件的进程/用户（Web 服务器日志）
□ 查找更多 WebShell：
  ├── 搜索所有最近创建/修改的 PHP/JSP/ASPX 文件
  ├── 搜索包含 eval/shell_exec/system/exec 的文件
  ├── 搜索异常大小的 Web 目录文件
  └── 检查是否有 webshell 管理工具的特征
□ 追踪攻击源：
  ├── Web 日志中该文件被上传的请求
  └── 上传漏洞是什么？（文件上传/文件包含/SQL注入）

Phase 2: 遏制

□ 隔离该 Web 服务器
□ 阻断该服务器向外的 C2 通信
□ 检查 Web 应用后台 → 是否有管理员被添加？
□ 检查数据库 → 是否有异常账号/操作？

Phase 3: 清除

□ 删除所有发现的 WebShell 文件
□ 修复上传漏洞：
  ├── 文件类型白名单
  ├── 上传目录禁止执行脚本
  └── 文件名随机化
□ 重置所有 Web 应用凭证
□ 检查代码 → 是否有后门账号被写入代码

Phase 4: 恢复

□ 从干净源码重新部署（避免只删除 WebShell）
□ 全量扫描 → 确保无遗漏
□ 加固 Web 服务器配置
```

## 五、场景四：数据泄露事件

### 5.1 事件场景

```
告警来源（三种常见情况）：

情形A：DLP 告警
├── 某员工在过去1小时内通过云存储上传了 10GB 数据
└── 包含客户 PII (个人身份信息)

情形B：暗网发现
├── 威胁情报团队在暗网论坛发现正在出售的数据
└── 包括客户姓名、电话、地址（格式与公司数据库完全匹配）

情形C：外部通报
├── 客户/媒体/监管机构联系公司
└── 声称公司数据被公开
```

### 5.2 响应步骤

```
Phase 1: 识别与确认

□ 确认泄露是否真实：
  ├── 比对泄露样本与自有数据库
  └── 确认泄露的时间窗口
□ 确定泄露范围：
  ├── 涉及多少条记录？多少用户？
  ├── 涉及哪些数据（姓名/身份证/银行卡/密码）？
  ├── 是否包括敏感个人信息？
  └── 是否包括未成年人信息？
□ 确定泄露途径：
  ├── 内部人员？
  ├── 外部攻击？
  └── 第三方合作伙伴？

Phase 2: 遏制

□ 立即封堵泄露渠道
□ 回收相关账户权限
□ 阻止进一步的数据外传
□ 如服务器被入侵 → 进入常规应急响应流程

Phase 3: 评估与通知

□ 法律评估：
  ├── 是否触发《个人信息保护法》的报告义务？
  │   └── 是 → 必须在事件发生后立即通知
  │          → 同时通知主管部门（网信办）
  ├── 是否涉及跨境数据？（涉及境外用户？）
  └── 是否有合同约定的通知时限？
□ 通知策略：
  ├── 谁需要被通知？（监管机构/受影响用户/合作伙伴/媒体）
  ├── 通知内容？（发生了什么/泄漏了什么/我们是么做的/你该怎么做）
  └── 通知时机？（尽快但需先确认事实）
□ 个人通知的义务内容（个保法第57条）：
  ├── 事件原因和可能造成的危害
  ├── 已采取的补救措施
  ├── 个人可以采取的减轻危害的措施
  └── 个人信息处理者的联系方式

Phase 4: 恢复与补救

□ 为客户/受影响用户提供帮助：
  ├── 信用监控服务
  └── 身份盗用保护
□ 加强数据保护措施
□ 安全体系全面审查
□ 如涉及监管处罚 → 配合调查
```

### 5.3 数据泄露通知时限

| 法规 | 时限要求 |
|------|----------|
| 个保法 | 立即通知 + 通知主管部门 |
| GDPR | 72小时内通知监管机构 |
| CCPA | 无明确时限但要求"不得无故延误" |
| 网安法 | "立即"报告 |

## 七、CISP 考试要点速查

### 关键优先级

```
应急响应的优先级顺序：
1. 保护人的生命安全 > 2. 遏制事件 > 3. 保留证据 > 4. 恢复业务

应急处置的"第一步"：
├── 勒索软件 → 隔离感染源（断网）
├── 凭证窃取 → 禁用被盗账户
├── WebShell → 隔离 Web 服务器
├── 数据泄露 → 封堵泄露渠道
└── DDoS → 启用清洗服务/黑洞路由
```

### 常见考试陷阱

1. ~~安全事件发生后应立即重启系统~~ → 应该**先取证内存**再重启
2. ~~数据泄露可以先不通知等调查完~~ → 个保法要求**立即通知**
3. ~~所有被勒索的都应该支付赎金~~ → CISP 立场是**不推荐支付**
4. ~~WebShell 删除就结束了~~ → 需要追溯到上传漏洞并修复

> 💡 **本章小结**：应急响应实战是 PICERL 框架的具体应用。四个场景中最常考的是勒索软件和数据泄露。记住：遏制永远优先于清除（防止扩散是第一位），取证优先于重启（内存证据珍贵）。个保法的"立即通知"义务是近年法规+应急响应的交叉考点。
