## 一、哈希碰撞攻击实战

### 1.1 MD5碰撞：从理论到实践

```
MD5碰撞时间线：
2004 - 王小云团队：理论碰撞（2^39次哈希运算）
2007 - 选定的前缀碰撞（Chosen-Prefix Collision）
2008 - SSL证书碰撞：伪造CA证书
2012 - Flame恶意软件：使用MD5碰撞伪造Microsoft签名
2017 - SHAttered：SHA-1首次实际碰撞（2^63.1次SHA-1压缩函数）

关键影响：
  Flame使用MD5碰撞 → 微软Windows Update签名被伪造
  → 恶意软件以Windows Update身份传播
```

### 1.2 选定的前缀碰撞

```
攻击场景：两份内容不同但哈希相同的合同

合同A（良性）：                          合同B（恶意）：
"I agree to pay $500"                  "I agree to pay $50000"
        ↓                                       ↓
    哈希 H(A)                               哈希 H(B)
        ↓                                       ↓
        ╰──────────── H(A) = H(B) ─────────────╯

技术原理：
1. 固定前缀使两个文件哈希"轨迹"收敛
2. 选择合适的碰撞块（near-collision blocks）
3. 最终产生完全相同的哈希值

实际利用：
  → 两封内容不同的PDF，MD5相同
  → 数字签名可互换
  → 一个签名可验证两个不同文档
```

### 1.3 SHAttered攻击（SHA-1首次碰撞）

```
2017年2月23日，Google和CWI宣布SHA-1首次碰撞：

计算量：2^63.1次SHA-1压缩函数 ≈ 6500 CPU年 + 110 GPU年
两个不同PDF文件产生完全相同的SHA-1哈希

技术细节：
  → 使用"相同前缀碰撞"（Identical-Prefix Collision）
  → 基于Joint Zhao的《扰动向量的加速寻找》
  → 两个PDF通过JPEG数据中的扰动块实现碰撞

行业反应：
  → Git宣布迁移到SHA-256
  → 浏览器停止信任SHA-1证书
  → NIST 2013年已建议2020年前淘汰SHA-1
```

## 二、长度扩展攻击

### 2.1 攻击原理

```
Merkle-Damgård结构哈希（MD5、SHA-1、SHA-256）的共同弱点：

原始哈希过程：
H = H(secret || message)

攻击者可计算：
H' = H(secret || message || padding || appended_data)

...而不需要知道secret！

┌─────────────────────────────────────────────────┐
│ 内部状态 = H(secret || message)                  │
│                                                  │
│ 攻击者把 内部状态 作为新的起始状态                │
│ 继续计算 padded_data 的哈希                       │
│ → 得到 H' = H(secret || message || !! || extra) │
│                                                  │
│ 利用条件：需要知道secret的长度                    │
└─────────────────────────────────────────────────┘
```

### 2.2 实际攻击场景

```
场景：URL签名验证
  URL: /download?file=report.pdf&mac=H(secret||file=report.pdf)
  
  攻击者可以：
  1. 截获原URL和MAC
  2. 计算 new_MAC = H(secret || "file=report.pdf" || padding || "&file=passwords.txt")
  3. 构造新URL: /download?file=report.pdf<padding>&file=passwords.txt&mac=new_MAC
  4. 服务器验证通过 → 越权下载！

防御：
  ✅ 使用 HMAC（H(K⊕opad || H(K⊕ipad || message))）
  ✅ 切换到 SHA-3 / BLAKE2（海绵结构，天然免疫）
```

### 2.3 受影响算法

| 算法 | 是否受影响 | 说明 |
|------|-----------|------|
| MD5 | ✅ 受影响 | Merkle-Damgård结构 |
| SHA-1 | ✅ 受影响 | 同上 |
| SHA-256 | ✅ 受影响 | 同上（但实际利用意义有限） |
| SHA-512 | ✅ 受影响 | 同上 |
| SHA-3 | ❌ 不受影响 | 海绵结构 |
| BLAKE2 | ❌ 不受影响 | HAIFA结构 |
| BLAKE3 | ❌ 不受影响 | 二叉树结构 |

## 三、HMAC与密钥哈希

### 3.1 HMAC标准结构

```
HMAC(K, m) = H((K' ⊕ opad) || H((K' ⊕ ipad) || m))

其中：
  K' = K 填充到分组大小（如SHA-256为512位）
  opad = 0x5c5c...5c（512位）
  ipad = 0x3636...36（512位）
```

### 3.2 HMAC vs 简单Hash-MAC

| 特性 | 简单H(K\|\|m) | HMAC |
|------|---------------|------|
| **长度扩展攻击** | 脆弱 | 免疫 |
| **密钥隔离** | 弱（密钥和消息混合） | 强（两次哈希隔离） |
| **可证明安全** | 无 | ✅ 有安全证明 |
| **标准** | 非标准 | RFC 2104, NIST FIPS 198 |

### 3.3 HMAC常见应用

| 应用场景 | 算法 | 用途 |
|----------|------|------|
| JWT签名 | HMAC-SHA256 | API认证令牌 |
| AWS SigV4 | HMAC-SHA256 | API请求签名 |
| OAuth 1.0 | HMAC-SHA1 | 请求授权 |
| IPsec | HMAC-SHA256/384 | 数据完整性 |
| TOTP | HMAC-SHA1 | 一次性密码 |

> **⚠️ 记忆要点**：只要是需要"带密钥的哈希"，就应使用HMAC，不要直接用Hash(Key||Message)。

## 四、密码哈希破解技术

### 4.1 攻击方法全景

```
密码哈希破解技术栈：

┌────────────┬──────────────────────────────────────────┐
│  攻击方式   │                  描述                    │
├────────────┼──────────────────────────────────────────┤
│  暴力破解   │ 遍历所有可能的密码组合（全空间）          │
│  字典攻击   │ 使用常见密码列表（高效）                  │
│  彩虹表     │ 时间-空间权衡，预计算哈希链（需要盐无效） │
│  规则攻击   │ 字典 + 变形规则（p@ssword → P@$$w0rd）   │
│  掩码攻击   │ 指定字符集和位置模式                     │
│  Markov攻击 │ 基于字符转移概率的智能猜测               │
│  组合攻击   │ 两个字典的笛卡尔积组合                   │
│  PRINCE     │ 概率预测密码生成                         │
└────────────┴──────────────────────────────────────────┘
```

### 4.2 加盐的重要性

```
不加盐（错误做法）：
  Alice: password123 → MD5 → 482c811da5d5b4bc6d497ffa98491e38
  Bob:   password123 → MD5 → 482c811da5d5b4bc6d497ffa98491e38
  → 相同密码 → 相同哈希 → 彩虹表一击即中

加盐（正确做法）：
  Alice: password123 + X7kQ2 → Hash → 唯一值
  Bob:   password123 + Z9mN5 → Hash → 另一个唯一值
  → 相同密码 → 不同哈希 → 彩虹表无效
  → 攻击者必须逐个破解
```

### 4.3 现代密码哈希算法基准

| 算法 | 诞生年份 | 可调参数 | 抗GPU/FPGA | 推荐程度 |
|------|----------|----------|-----------|----------|
| bcrypt | 1999 | 成本因子(2^n) | ⚠️ 较弱 | 可接受 |
| scrypt | 2009 | N,r,p (内存+CPU) | ✅ 良好 | 良好 |
| Argon2id | 2015 | 内存+时间+并行 | ✅ 最优 | 强烈推荐 |
| PBKDF2 | 2000 | 迭代次数 | ❌ GPU友好 | 不如上三 |

```
Argon2id参数建议（OWASP 2023）：
  内存: ≥ 46 MB (m=47104 KiB)
  时间: 1次迭代 (t=1)
  并行: 1线程 (p=1)
  盐: ≥ 16字节随机值
  输出: ≥ 32字节
  用于: 密码存储、密钥派生
```

## 五、哈希在数字取证中的应用

### 5.1 证据完整性

```
取证中的哈希使用：

1. 镜像哈希：
   取证镜像 → SHA-256 → 记录哈希值
   分析过程 → 不改变原始镜像
   验证 → 重新哈希 → 与记录对比

2. 文件分类：
   已知恶意哈希集 → 快速匹配 → 识别恶意文件
   已知良性哈希集 → 过滤排除 → 缩小分析范围

3. 证据链维护：
   Chain of Custody 文档 + 每个步骤的哈希记录
   → 可在法庭证明证据未被篡改
```

### 5.2 NSRL与哈希集

| 哈希数据库 | 内容 | 用途 |
|-----------|------|------|
| NIST NSRL | 已知良性文件哈希 | 排除已知文件 |
| VirusTotal | 恶意文件哈希 | 威胁检测 |
| Project VIC | CSAM内容哈希 | 犯罪调查 |
| HashKeeper | 执法数据库 | 证据匹配 |

## 六、Merkle树与区块链

### 6.1 Merkle树结构

```
Merkle树通过哈希构建防篡改数据结构：

                    Root Hash
                 H(H01 || H23)
                /              \
          H01=H(H0||H1)    H23=H(H2||H3)
          /       \         /       \
    H0=H(Tx0)  H1=H(Tx1)  H2=H(Tx2)  H3=H(Tx3)
       |          |          |          |
      Tx0        Tx1        Tx2        Tx3

    优势：
    → O(log n)复杂度验证单个交易
    → 任何交易改变都会导致Root Hash变化
    → 支持轻量级验证（SPV节点）
```

### 6.2 Merkle证明

```
要证明Tx0在Merkle树中：
  需要提供：Tx0, H1, H23
  验证者计算：H(H(H(Tx0)||H1)||H23) == Root Hash
  → 3次哈希运算验证，而非全部4条交易

应用：
  → Bitcoin区块头（Merkle Root）
  → Certificate Transparency（Merkle Tree Hash）
  → Git对象存储
  → BitTorrent分片验证
```

## 八、CISP考试速查

### 考试易考点

| 考点 | 关键记忆 |
|------|---------|
| MD5碰撞第一人 | "王小云，2004年，2^39" |
| SHA-1首次碰撞 | "SHAttered，2017年，2^63" |
| 长度扩展攻击 | "Merkle-Damgård的通病" |
| HMAC公式 | "H(K⊕opad ∥ H(K⊕ipad ∥ m))" |
| 当前推荐密码Hash | "Argon2id" |
| 国密哈希 | "SM3，256位输出" |
| 区块链结构 | "Merkle树，O(log n)验证" |

### 常见混淆

1. **"SHA-256不受长度扩展攻击"** → ❌，它是Merkle-Damgård结构，仍受影响
2. **"加盐就是加已知字符串"** → ⚠️，盐必须是随机且每用户唯一的
3. **"bcrypt就是最好的"** → 已不是，Argon2id更优
4. **"哈希加密"是不可逆** → 不能说成加密，哈希是单向的，加密可解密
