## 一、IDS vs IPS：核心区别

### 1.1 关键差异

```
┌──────────────┬─────────────────────┬─────────────────────┐
│    特性       │     IDS (检测)       │     IPS (防御)       │
├──────────────┼─────────────────────┼─────────────────────┤
│ 部署方式      │ 旁路 (镜像流量/SPAN) │ 串联 (inline)        │
│ 对网络影响    │ 无影响（被动监听）    │ 引入延迟             │
│ 阻断能力      │ ❌ 只能告警           │ ✅ 可以主动阻断       │
│ 故障影响      │ 故障不影响业务        │ 故障可导致网络中断    │
│ 签名模式      │ 检出→告警            │ 检出→阻断           │
│ 性能要求      │ 较低                 │ 较高                 │
│ 常见缩写      │ NIDS (网络)/HIDS(主机)│ NIPS/HIPS            │
└──────────────┴─────────────────────┴─────────────────────┘
```

### 1.2 IDS告警 vs IPS阻断

```
IDS模式（旁路部署）：                 IPS模式（串联部署）：

  Internet                             Internet
     │                                    │
     ▼                                    ▼
  ┌─────┐     正常流量               ┌─────┐
  │交换机│←───────────→              │ IPS │←─ 所有流量必须通过
  └──┬──┘                            └──┬──┘
     │ SPAN/镜像                         │
     ▼                                   ▼
  ┌─────┐                            正常流量/阻断恶意流量
  │ IDS │←─ 被动复制流量              ┌─────┐
  └──┬──┘                            │交换机│
     │                                └──┬──┘
     ▼                                    ▼
  告警（不影响流量）                   正常业务流量

IDS部署优势：不影响现有网络，无单点故障风险
IPS部署优势：实时阻断攻击，但引入延迟和单点故障风险
```

## 二、检测方法深度分析

### 2.1 四种检测方法

```
① 基于签名的检测 (Signature-based)：
  匹配已知攻击模式的规则
  ✅ 低误报率、精确的攻击识别
  ❌ 无法检测0-day攻击、变种攻击

  示例规则：检测 "SELECT * FROM users WHERE id = ' OR 1=1"

② 基于异常的检测 (Anomaly-based)：
  建立正常基线→检测偏离
  ✅ 可检测未知攻击、0-day
  ❌ 高误报率、需训练期

  基线示例：某服务器通常每天1000次MySQL查询
        今天突然100000次 → 异常告警

③ 基于协议的检测 (Protocol-based)：
  检查是否符合RFC规范
  ✅ 检测协议滥用、畸形报文
  ❌ 特定场景可能误报

  示例：HTTP请求中包含二进制数据（可能是缓冲区溢出）

④ 基于信誉的检测 (Reputation-based)：
  使用威胁情报数据库
  ✅ 快速阻断已知恶意源
  ❌ 依赖情报更新及时性

  来源：已知C2 IP列表、恶意域名、Tor出口节点
```

### 2.2 网络安全厂商对比

```
┌──────────────┬──────────┬──────────────┬──────────────┐
│    工具       │   类型    │   检测方法    │    特点       │
├──────────────┼──────────┼──────────────┼──────────────┤
│ Snort        │ NIDS/IPS │ 签名+协议     │ 开源鼻祖      │
│ Suricata     │ NIDS/IPS │ 签名+协议     │ 多线程高性能  │
│ Zeek (Bro)   │ NSM      │ 协议+行为     │ 深度网络取证  │
│ SecurityOnion│ 集成平台 │ 混合          │ 完整的NSM方案 │
│ OSSEC/Wazuh  │ HIDS     │ 日志+完整性   │ 主机层检测    │
│ Snort/Suricata│ NIDS/IPS│ 签名+协议     │ 主流开源      │
└──────────────┴──────────┴──────────────┴──────────────┘
```

## 三、部署架构与模式

### 3.1 传感器位置选择

```
┌──────────────────────────────────────────────────┐
│               IDS/IPS 部署位置指南                  │
├──────────────┬───────────────────────────────────┤
│   部署位置    │         优势/检测内容              │
├──────────────┼───────────────────────────────────┤
│ 互联网边界前  │ 检测所有外部攻击，最高告警量        │
│ 防火墙之后    │ 过滤后的攻击，告警更有价值          │
│ DMZ前/后     │ 检测对DMZ服务器的攻击               │
│ 核心交换侧    │ 检测横向移动、内部威胁              │
│ 关键服务器前  │ 针对性保护高价值资产                │
│ VPN集中器后   │ 检测通过VPN进入的威胁              │
└──────────────┴───────────────────────────────────┘
```

### 3.2 TAP vs SPAN

| 方式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **SPAN端口** | 交换机镜像端口 | 零成本，灵活 | 负载高时丢包 |
| **网络TAP** | 物理流量复制设备 | 不丢包，全双工 | 需要硬件成本 |
| **分光器** | 光纤信号分光 | 无电气干扰 | 信号衰减 |

## 四、Snort规则编写实战

### 4.1 Snort规则语法

```
规则结构：
[动作] [协议] [源IP] [源端口] -> [目标IP] [目标端口] ([规则选项])

alert tcp $EXTERNAL_NET any -> $HOME_NET 80 (
    msg:"SQL Injection Attempt";       # 告警信息
    flow:to_server,established;        # 流量方向
    content:"%27";                     # 单引号 URL编码
    content:"OR"; nocase;             # OR关键字 忽略大小写
    metadata:policy security-ips;     # 元数据
    classtype:web-application-attack;  # 攻击分类
    sid:1000001;                       # 规则ID
    rev:1;                             # 版本
)

关键选项：
  · content：匹配载荷内容（支持二进制和正则）
  · pcre：Perl兼容正则表达式匹配
  · flow：限定流量方向（to_server, to_client）
  · within/distance：content匹配位置限定
  · threshold：告警阈值控制
```

### 4.2 实战规则示例

```
# 1. 检测反向Shell（常见命令执行后）
alert tcp $HOME_NET any -> $EXTERNAL_NET any (
    msg:"Possible Reverse Shell";
    flow:to_server,established;
    content:"/bin/sh"; 
    content:"/bin/bash";
    metadata:policy balanced-ips;
    classtype:trojan-activity;
    sid:1000002;
)

# 2. 检测WebShell上传
alert tcp $EXTERNAL_NET any -> $HOME_NET 80 (
    msg:"WebShell Upload Detected";
    flow:to_server,established;
    content:"eval"; nocase;
    content:"base64_decode"; nocase;
    metadata:policy security-ips;
    classtype:web-application-attack;
    sid:1000003;
)

# 3. 检测暴力登录（频率控制）
alert tcp $EXTERNAL_NET any -> $HOME_NET 22 (
    msg:"SSH Brute Force Login";
    flow:to_server,established;
    content:"Failed password";
    threshold:type both, track by_src, count 5, seconds 60;
    metadata:policy security-ips;
    classtype:attempted-admin;
    sid:1000004;
)
```

## 五、HIDS：主机入侵检测

### 5.1 HIDS vs NIDS

| 维度 | HIDS | NIDS |
|------|------|------|
| 检测范围 | 单台主机 | 整个网络段 |
| 数据来源 | 系统日志、文件完整性、进程 | 网络流量 |
| 能检测加密流量 | ✅ (解密后的本地数据) | ❌ (除非做TLS解密) |
| 部署成本 | 每台主机需要agent | 集中部署 |
| 内核级威胁 | ✅ 可见 | ❌ 不可见 |

### 5.2 HIDS核心功能

```
HIDS (如Wazuh/OSSEC) 检测内容：

① 文件完整性监控 (FIM)：
  监控 /etc/passwd, /etc/shadow, 关键配置
  文件哈希变化 → 告警

② 日志分析：
  /var/log/auth.log → 登录失败
  /var/log/apache2/access.log → Web攻击
  系统审计日志 → Root行为

③ Rootkit检测：
  扫描异常内核模块、隐藏进程

④ 基线合规检查：
  CIS Benchmark检查
  系统加固状态审计

⑤ 进程监控：
  异常进程行为、CPU使用率突增
```

## 六、NDR与网络流量分析

### 6.1 NDR概念

```
NDR (Network Detection and Response)：

超越传统IDS，提供：
  · 全流量记录（PCAP存储）
  · AI/ML行为分析
  · 攻击链重建
  · 威胁狩猎能力

与传统IDS/SIEM的区别：
  IDS → 基于签名，已知攻击
  SIEM → 规则关联，多源日志
  NDR → 行为分析+全流量，未知威胁
```

### 6.2 流量基线

```
建立行为基线进行分析：

  ┌──────────────────────────────────────┐
  │ 指标            │ 正常基线           │
  ├──────────────────────────────────────┤
  │ 每分钟DNS查询    │ 50-200            │
  │ 每分钟新建连接   │ 100-500           │
  │ 流量数据量比例   │ 出:入 ≈ 4:1       │
  │ 非标准端口连接   │ <5%               │
  │ 与外国IP的连接   │ <10个/小时        │
  │ DNS TXT查询     │ 0（一般不正常）    │
  │ 长时间无数据连接 │ <1%               │
  └──────────────────────────────────────┘

  异常检测示例：
  · DNS TXT查询突增 → 可能DNS隧道
  · 深夜大量出站流量 → 数据泄露
  · 连接数突然100倍 → 蠕虫传播
```

## 七、SIEM与IDS的协作

### 7.1 分工协作

```
SIEM = 集中日志管理 + 关联分析 + 可视化

IDS → 网络层告警
  │
  ▼
SIEM接收和处理：
  · IDS告警
  · 防火墙日志
  · 服务器系统日志
  · 应用日志
  · 认证日志(AD/LDAP)
  · 终端安全(EDR)日志
  
  关联规则示例：
  IDS告警: "SQL注入尝试" 
  + 
  应用日志: "数据库错误"
  +
  WAF日志: "被绕过"
  =
  高优先级事件: "可能的数据库已被入侵"
```

### 7.2 攻击链视角

```
Cyber Kill Chain × 告警来源：

┌─────────────────┬──────────────────────────────┐
│  Kill Chain阶段  │        告警来源              │
├─────────────────┼──────────────────────────────┤
│ 侦察             │ 端口扫描(IDS), 目录爆破(WAF) │
│ 武器化           │ 威胁情报                       │
│ 投递             │ 恶意邮件附件(邮件网关)         │
│ 利用             │ SQL注入/WAF, 恶意文件(EDR)    │
│ 安装             │ 异常进程(HIDS), 注册表(EDR)   │
│ C2通信           │ 异常DNS(NDR), Beacon(IDS)    │
│ 目标行动         │ 数据外传(NDR), 横向移动(HIDS) │
└─────────────────┴──────────────────────────────┘
```

## 八、告警优化与误报管理

### 8.1 告警分级

```
告警优先级矩阵：

              准确性→
              高           低
        ┌──────────┬──────────┐
  影    │ P0 (紧急) │ P2 (监控) │
  响 高 │ 立即响应   │ 调查后降级 │
  力    ├──────────┼──────────┤
  ↓     │ P1 (重要) │ P3 (低优) │
  响 低 │ 计划处理   │ 可忽略     │
  力    └──────────┴──────────┘

典型告警处理比例：
  真实攻击：<1% (大海捞针)
  误报：50-70%
  扫描/侦察：10-20%
  无效/噪音：10-20%
```

### 8.2 减少误报策略

```
✅ 调优策略：
  ① 针对性调优（每个传感器独立调整）
  ② 排除已知安全测试流量
  ③ 使用threshold限制重复告警
  ④ suppression抑制特定主机/IP组合
  ⑤ 业务上下文关联（数据库服务器预期大量DB流量）

❌ 错误做法：
  · 直接关闭所有告警规则
  · 将所有流量加入白名单
  · 不分析就直接抑制
```

## 九、CISP考试速查

### 关键考点

| 考点 | 记忆要点 |
|------|---------|
| IDS vs IPS | "IDS旁路只告警，IPS串联能阻断" |
| 检测方法 | "签名(已知)+异常(未知)+协议(规范)+信誉(威胁情报)" |
| Snort三类规则 | "alert(告警), drop(阻断), pass(放行)" |
| HIDS部署 | "每台主机一个Agent，监控文件/日志/进程" |
| SPAN vs TAP | "SPAN零成本但可能丢包，TAP硬件但全双工无丢包" |
| SIEM关联 | "多源日志关联=IDS+FW+AD+服务器" |

### 常见陷阱

1. **"IPS不会产生误报"** → IPS同样有误报，阻断误报可能导致业务中断
2. **"部署IDS就安全了"** → IDS只告警不防护，需要人工或SOAR响应
3. **"开源的比商业的差"** → Snort/Suricata规则社区活跃，效果不输商业
4. **"只靠签名即可"** → 需要签名+异常+行为多维度检测
