## 一、漏洞基础概念

### 1.1 漏洞的定义

```
漏洞 (Vulnerability) =

信息系统在设计、实现、配置或运行管理过程中
存在的缺陷或弱点，可能被威胁利用，
从而破坏系统的安全策略。

漏洞 ≠ 攻击
├── 漏洞是被动的（存在弱点）
├── 攻击是主动的（利用弱点）
└── 关系：漏洞 + 威胁 → 风险

漏洞的三要素：
├── 存在性：缺陷确实存在
├── 可访问性：攻击者能够接触到该缺陷
└── 可利用性：攻击者能够利用该缺陷达成其目的
```

### 1.2 漏洞的产生来源

```
漏洞的五大来源：

1. 设计缺陷 (Design Flaw)
   ├── 协议设计缺陷 (如 WEP 加密缺陷)
   ├── 架构设计缺陷 (如缺乏认证机制)
   └── 最难修复（需要重新设计）

2. 编码实现错误 (Implementation Bug)
   ├── 缓冲区溢出 (Buffer Overflow)
   ├── 输入验证不足
   ├── 竞态条件 (Race Condition)
   └── 最常见

3. 配置不当 (Misconfiguration)
   ├── 默认密码未修改
   ├── 过度的权限设置
   ├── 不必要的服务开启
   └── 最容易修复

4. 运维疏忽 (Operational Weakness)
   ├── 未及时打补丁
   ├── 密码策略过弱
   └── 缺乏安全监控

5. 第三方组件漏洞 (Supply Chain)
   ├── Log4j (Java 日志框架远程代码执行)
   ├── OpenSSL Heartbleed
   └── 开源库漏洞影响广泛
```

## 二、漏洞分类体系

### 2.1 按漏洞类型分类

```
OWASP Top 10 / CWE Top 25 分类视角：

注入类 (Injection)
├── SQL 注入
├── 命令注入 (OS Command Injection)
├── LDAP 注入
└── XXE (XML 外部实体注入)

认证与授权类
├── 认证绕过 (Broken Authentication)
├── 越权访问 (Broken Access Control)
│   ├── 水平越权 (IDOR)：用户A访问用户B的数据
│   └── 垂直越权：普通用户执行管理员操作
└── 会话管理缺陷

数据安全类
├── 敏感数据泄露
├── 加密缺失/不足
├── 密钥管理不当
└── 不安全的直接对象引用 (IDOR)

逻辑漏洞类
├── 竞态条件 (Race Condition)
├── 业务逻辑缺陷
├── 整数溢出 (Integer Overflow)
└── 时序攻击 (Timing Attack)

配置安全类
├── 安全配置错误
├── 默认凭证未更改
├── 目录列表暴露
└── 详细错误信息泄露

客户端类
├── XSS (跨站脚本)
│   ├── 反射型 (Reflected)
│   ├── 存储型 (Stored)
│   └── DOM型
├── CSRF (跨站请求伪造)
└── 点击劫持 (Clickjacking)

内存安全类
├── 缓冲区溢出 (Buffer Overflow)
├── 栈溢出 (Stack Overflow)
├── 堆溢出 (Heap Overflow)
├── 格式化字符串 (Format String)
├── 释放后使用 (Use After Free)
└── 双重释放 (Double Free)
```

### 2.2 按影响分类

| 影响类型 | 描述 | CVE 示例 |
|----------|------|----------|
| RCE (远程代码执行) | 攻击者可在目标系统执行任意代码 | Log4Shell (CVE-2021-44228) |
| Privilege Escalation | 提权：从低权限到高权限 | Dirty Pipe (CVE-2022-0847) |
| DoS (拒绝服务) | 使系统服务不可用 | HTTP/2 Rapid Reset |
| 信息泄露 | 未授权获取系统/用户数据 | Heartbleed (CVE-2014-0160) |
| 身份伪造 | 冒充合法身份 | Golden SAML Attack |
| 数据篡改 | 未经授权修改数据 | — |

## 三、CVE、CWE 与 CVSS

### 3.1 CVE (通用漏洞披露)

```
CVE (Common Vulnerabilities and Exposures) =

全球统一的漏洞标识体系

格式：CVE-YYYY-NNNNN
示例：CVE-2021-44228 (Log4Shell)

CVE 是什么：
├── 一个唯一编号，用于标识特定漏洞
├── 不是漏洞数据库（只是标识符）
├── 不包含修复方案
└── 由 MITRE 维护

CVE 条目包含：
├── CVE ID
├── 简要描述
├── 参考链接
└── 发布日期

关键平台：
├── NVD (National Vulnerability Database)：NIST 维护，补充CVSS
├── CVE.org：MITRE 维护的官方CVE列表
└── 各厂商自有漏洞库
```

### 3.2 CWE (通用弱点枚举)

```
CWE (Common Weakness Enumeration) =

对漏洞的"类型"进行分类，而非具体实例

CVE vs CWE：
├── CVE：具体漏洞实例 → "这个版本的 Log4j 有远程代码执行"
└── CWE：漏洞类型 → "不安全的反序列化 (CWE-502)"

CWE Top 25 (2023) 中的代表性弱点：
├── CWE-787：越界写入 (Out-of-bounds Write)
├── CWE-79：XSS (Cross-Site Scripting)
├── CWE-89：SQL 注入
├── CWE-416：释放后使用 (Use After Free)
├── CWE-78：OS 命令注入
└── CWE-20：输入验证不当

考试重点：
└── CVE = 具体漏洞 (What)，CWE = 漏洞类型 (Why)
```

### 3.3 CVSS (通用漏洞评分系统)

```
CVSS (Common Vulnerability Scoring System) =

标准化漏洞严重性评分 (0-10分)

CVSS 的三大指标组：

基本指标组 (Base Score) ← 最常用
├── 攻击向量 (AV)：N(远程)/A(相邻)/L(本地)/P(物理)
├── 攻击复杂度 (AC)：L(低)/H(高)
├── 权限要求 (PR)：N(无)/L(低)/H(高)
├── 用户交互 (UI)：N(无)/R(需要)
├── 影响范围 (S)：U(不变)/C(改变)
├── 机密性影响 (C)：N(无)/L(低)/H(高)
├── 完整性影响 (I)：N(无)/L(低)/H(高)
└── 可用性影响 (A)：N(无)/L(低)/H(高)

时间指标组 (Temporal)
├── 漏洞利用代码成熟度 (E)
├── 修复级别 (RL)
└── 报告置信度 (RC)

环境指标组 (Environmental)
├── 根据实际部署环境影响调整
└── 企业更关注的评分！

CVSS 分数与严重级别：
├── 0.0      无
├── 0.1-3.9  低 (Low)
├── 4.0-6.9  中 (Medium)
├── 7.0-8.9  高 (High)
└── 9.0-10.0 严重 (Critical)

CISP 考试的 CVSS 考点：
├── CVSS 分数范围 0-10
├── 9.0+ = Critical
├── Base Score 是最常用的评分
└── AV:N + AC:L + PR:N = 最危险 (远程 + 简单 + 无权限)
```

## 四、漏洞生命周期

### 4.1 漏洞管理生命周期

```
漏洞发现的六个时间点：

        发现              公开               修复
        │                  │                  │
    ────●──────────────────●──────────────────●───
        │                  │                  │
        │←── 漏洞窗口 ────→│← 补丁窗口 ──────→│
        │  (未公开但可能    │  (已公开但尚未    │
        │   已被利用)       │   打补丁)         │

1. T0: 漏洞被引入（编码时产生）
2. T1: 漏洞被发现（安全研究员/攻击者）
3. T2: 漏洞被披露（公开CVE/私下报告厂商）
4. T3: 补丁发布（厂商提供修复）
5. T4: 补丁部署（用户安装补丁）
6. T5: 漏洞被彻底消除
```

### 4.2 漏洞管理流程

```
企业漏洞管理 PDCA：

发现 (Discovery)
├── 漏洞扫描器 (Nessus, Qualys, OpenVAS)
├── 渗透测试
├── 代码审计 (SAST/DAST)
└── 外部通告 (CVE / 威胁情报)

评估 (Assessment)
├── CVSS 评分
├── 资产重要性加权
│   核心系统上的 Low → 升级为 Medium
│   测试环境上的 Critical → 可降级
└── 确定修复优先级 (P1-P4)

修复 (Remediation)
├── 打补丁（首选）
├── 配置变更（补偿控制）
├── 虚拟补丁 (WAF/IPS)
├── 隔离/下线（临时）
└── 接受风险（无法修复或修复成本过高）

验证 (Verification)
├── 重新扫描确认修复
├── 验证补偿控制有效
└── 更新漏洞管理台账

报告 (Reporting)
├── 漏洞修复状态报告
├── 未修复风险清单
└── 趋势分析（修复率、平均修复时间等）
```

### 4.3 漏洞修复 SLA

```
漏洞修复时间窗口 (参考)：

严重级别   修复SLA      紧急处置
────────────────────────────
Critical   ≤ 72小时     ≤ 24小时
High       ≤ 2周        ≤ 72小时
Medium     ≤ 30天       ≤ 1周
Low        ≤ 90天       按需

注：以上为通用建议，实际 SLA 应根据组织风险偏好调整
```

## 五、漏洞披露机制

### 5.1 三种披露策略

```
1. 完全披露 (Full Disclosure)
├── 发现漏洞后立即公开所有细节
├── 优点：用户立即知道风险，推动厂商火速修复
├── 缺点：攻击者也立即知道怎么利用
└── 典型：某些安全研究员

2. 负责披露 (Responsible Disclosure) ← 推荐
├── 先通知厂商 → 给修复时间(通常90天) → 再公开
├── 优点：平衡安全和透明度
├── 缺点：厂商可能拖延
└── 典型：Google Project Zero (90天策略)

3. 不披露
├── 发现后不公开
├── 部分卖给政府/军方（如 Zerodium）
└── 部分用于犯罪（黑市 0-day交易）

0-day 漏洞（零日漏洞）：
├── 厂商尚未发布补丁的漏洞
├── 或者补丁发布了但用户还没装
└── 0-day 的市场价格：
    ├── iOS 零点击 RCE：$1M-$2M
    ├── Android RCE：$500K-$1M
    └── Chrome RCE：$100K-$500K
```

## 七、CISP 考试要点速查

### 核心概念

| 概念 | 要点 |
|------|------|
| 漏洞 | 系统中可能被利用的缺陷或弱点 |
| CVE | 具体漏洞的唯一编号 (YYYY-NNNNN) |
| CWE | 漏洞类型分类体系 |
| CVSS | 漏洞严重性评分 (0-10) |
| 0-day | 厂商未发布补丁的漏洞 |
| 负责披露 | 先通知厂商（90天），再公开 |

### CVSS 严重级别

| 分数 | 级别 |
|------|------|
| 0.0 | None |
| 0.1-3.9 | Low |
| 4.0-6.9 | Medium |
| 7.0-8.9 | High |
| **9.0-10.0** | **Critical** |

### 常见考试陷阱

1. ~~CVE 提供修复方案~~ → CVE 只是标识符，不包含修复
2. ~~所有漏洞都必须立即修复~~ → 根据 CVSS+资产重要性排优先级
3. ~~漏洞扫描=渗透测试~~ → 扫描是自动化检查，渗透测试是人工深入攻击
4. ~~漏洞管理就是打补丁~~ → 还包括评估、验证、接受风险等

### 自检清单

- [ ] 了解漏洞的五大来源
- [ ] 区分 CVE (具体漏洞)、CWE (漏洞类型)、CVSS (评分)
- [ ] 记住 CVSS 的分数范围和严重级别
- [ ] 理解漏洞生命周期（发现→公开→修复）
- [ ] 知道负责披露的原则（先通知厂商，后公开）
- [ ] 掌握漏洞修复的优先排序方法

---

> 💡 **本章小结**：漏洞管理是安全运营的基础工作。CVE/CWE/CVSS 三者的区别是经典考点：CVE 编号具体漏洞、CWE 分类漏洞类型、CVSS 量化漏洞严重性。CVSS 9.0+ = Critical 是考试常考察的分界线。负责披露（Responsible Disclosure）的 90 天窗口是漏洞管理的行业共识。
