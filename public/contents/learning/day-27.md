## 一、BCP vs DRP 辨析

### 1.1 概念区分

```
BCP (Business Continuity Plan) 业务连续性计划
┌──────────────────────────────────────────┐
│                                          │
│  整体框架——确保"核心业务"在中断后能继续    │
│                                          │
│  关注：人、流程、场所、供应商、通信        │
│                                          │
│  范围：全组织（业务 + IT + 设施 + 人员）   │
│                                          │
├──────────────────────────────────────────┤
│                                          │
│  DRP (Disaster Recovery Plan) 灾难恢复计划│
│                                          │
│  BCP的子集——专注于"IT系统和数据"的恢复     │
│                                          │
│  关注：服务器、网络、存储、应用、数据      │
│                                          │
│  范围：IT 基础设施                        │
│                                          │
└──────────────────────────────────────────┘

BCP ⊃ DRP  (DRP 是 BCP 的 IT 子集)
```

### 1.2 完整对比

| 维度 | BCP | DRP |
|------|-----|-----|
| **全称** | Business Continuity Plan | Disaster Recovery Plan |
| **关注** | 保持**业务运行** | 恢复 **IT 系统** |
| **范围** | 全组织 | IT 基础设施 |
| **未涵盖** | 纯技术中断 | 人员/场所/设备中断 |
| **目标** | 继续为客户提供服务 | 恢复 IT 服务至可用状态 |
| **参与者** | 业务 + IT + 管理层 | IT 技术团队 |
| **优先级** | 关键业务流程的继续 | 关键 IT 系统的恢复 |

### 1.3 相关术语

```
灾难恢复相关术语群：

BIA (Business Impact Analysis) 业务影响分析
└── 识别关键业务流程 → 确定中断的影响 → 设定恢复优先级
→ 这是制定 BCP/DRP 的前提和基础！

COOP (Continuity of Operations Plan) 运营连续性计划
└── 政府/军事对 BCP 的另一种称呼

BRP (Business Recovery Plan) 业务恢复计划
└── 业务层面的恢复方案（场所、人员、设备、供应商）

Crisis Management Plan 危机管理计划
└── 宏观层面的危机响应（包括沟通/公关/领导决策）

两者的依存关系：
    BIA → 喂给 → BCP (决定恢复什么)
    BIA → 喂给 → DRP (决定怎么恢复 IT)
```

## 二、业务影响分析 (BIA)

### 2.1 BIA 的核心目的

```
BIA 回答的五个核心问题：

1. 哪些业务流程是不可中断的？
   → 识别关键业务功能

2. 每个流程中断的影响是什么？
   → 财务影响（每小时损失多少钱？）
   → 合规影响（违反什么法规？）
   → 声誉影响（客户信任损失？）
   → 运营影响（多少员工无法工作？）

3. 每个流程能容忍多久的中断？
   → RTO：服务恢复的目标时间
   → RPO：数据恢复的目标时间点

4. 每个流程依赖什么资源？
   → IT系统、应用程序、数据
   → 人员、场所、供应商、设备

5. 恢复的优先级是什么？
   → 哪个先恢复？哪个可以后恢复？
```

### 2.2 关键业务流程识别

```
典型的 BIA 流程清单：

关键业务功能            最大可容忍中断时间    恢复优先级
────────────────────────────────────────────
在线支付系统            5分钟                P1 (最高)
客户服务平台            30分钟               P1
订单处理系统            4小时                P2
仓储管理系统            8小时                P2
HR 薪资系统             24小时               P3
财务报表系统            72小时               P3
内部培训平台            1周                  P4

MTD (Maximum Tolerable Downtime) 最大可容忍中断时间：
├── 超过此时间将造成不可接受的损害
└── 这是 RTO 的上限！（RTO ≤ MTD）
```

### 2.3 BIA 的四步法

```
1. 收集信息
   ├── 访谈业务部门负责人
   ├── 问卷调查
   └── 审查流程文档

2. 分析影响
   ├── 量化财务损失
   ├── 定性评估非财务影响
   └── 确定 MTD

3. 识别依赖关系
   ├── 流程 → IT 系统依赖矩阵
   ├── 流程 → 供应商依赖
   └── 流程 → 关键人员依赖

4. 制定恢复优先级
   ├── P1: ≤ 4小时
   ├── P2: 4-24小时
   ├── P3: 24-72小时
   └── P4: ≥ 72小时
```

## 三、RTO 与 RPO 深度解析

### 3.1 可视化理解

```
          RPO                             RTO
   数据丢失的窗口                   服务恢复时间
   ←─────→                        ←─────────→


    上次备份          灾难发生       服务恢复
       │                │              │
   ────●────────────────●──────────────●────→ 时间
       │←  数据丢失  →│              │
       │   (RPO窗口)   │              │
       │               │←─ 恢复时间 ─→│
       │               │   (RTO窗口)   │
```

### 3.2 深度对比

```
RTO (Recovery Time Objective) 恢复时间目标
├── 问题：服务中断后，多久必须恢复？
├── 定义：从灾难发生到服务恢复到可用状态的目标时间
├── 单位：分钟/小时
├── 示例：在线支付系统 RTO = 15分钟
├── 驱动因素：业务损失的承受能力
├── 技术要求：热备、自动故障切换、镜像
└── 缩短 RTO 的手段：高可用架构、自动切换

RPO (Recovery Point Objective) 恢复点目标
├── 问题：最多能容忍丢失多少数据？
├── 定义：灾难发生时允许的数据丢失量（以时间为单位）
├── 单位：分钟/小时
├── 示例：银行交易系统 RPO = 0（零数据丢失）
├── 驱动因素：数据的价值和合规要求
├── 技术要求：实时同步、持续数据保护
└── 缩短 RPO 的手段：同步复制、实时备份
```

### 3.3 RTO/RPO 指标矩阵

```
不同行业的 RTO/RPO 参考值：

行业/系统          RTO          RPO        灾备等级
────────────────────────────────────────────
银行核心交易       <5分钟        0 (零丢失)   Tier 4
证券交易系统       <5分钟        0           Tier 4
电商支付系统       <15分钟       <5分钟      Tier 3-4
医院 HIS 系统      <30分钟       <15分钟     Tier 3
企业内部 ERP       <4小时        <1小时      Tier 2
内部OA系统         <24小时       <24小时     Tier 1
归档系统           <72小时       <24小时     Tier 0-1
```

## 四、灾备策略与站点类型

### 4.1 三大灾备站点

```
冷站 (Cold Site)
├── 描述：只有场地（建筑物、电力、制冷），没有设备
├── 启动时间：数天到数周
├── 成本：低
├── 数据同步：无，需要从备份恢复
├── 适合：非关键系统，长恢复时间可接受
└── 考试要点：恢复时间最长、成本最低、不需要实时数据

温站 (Warm Site)
├── 描述：场地 + 部分设备（服务器、存储、网络），但需要启用和配置
├── 启动时间：数小时到1-2天
├── 成本：中
├── 数据同步：定期复制（如每晚）
├── 适合：重要但不是核心的业务系统
└── 考试要点：平衡成本与恢复速度的最常见选择

热站 (Hot Site)
├── 描述：完全配置、实时复制、随时可用的备份站点
├── 启动时间：分钟级（甚至自动切换）
├── 成本：高（接近主站点的投入）
├── 数据同步：实时/准实时
├── 适合：核心业务系统（必须秒/分钟级恢复）
└── 考试要点：恢复时间最短、成本最高、实时数据

镜像站 (Mirrored Site)
├── 描述：100% 完全相同的双活站点
├── 两个站点同时处理流量
├── 单个站点故障 → 流量自动切换到另一站点
├── RTO = 0, RPO = 0
└── 适合：绝不能中断的核心业务
```

### 4.2 站点对比表

| 特点 | 冷站 | 温站 | 热站 | 镜像站 |
|------|------|------|------|--------|
| 设备 | 无 | 部分 | 完备 | 完备+双活 |
| 数据 | 无 | 非实时 | 准实时 | 实时 |
| 恢复时间 | 天-周 | 小时-天 | 分钟-小时 | 秒-分钟 |
| 成本 | ⭐ | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |

### 4.3 其他灾备方式

```
冗余站点 (Redundant Site)
└── 组织自己拥有和运营的多余站点

互惠协议 (Reciprocal Agreement)
├── 两个组织互相约定在对方场地恢复系统
├── 优点：成本低
├── 缺点：对方也有灾难时无法使用、容量不确定
└── 不推荐用于关键系统

移动站点 (Mobile Site)
├── 拖车/集装箱中的移动数据中心
├── 可运输到任何地点
└── 适合：区域性灾难

云灾备 (Cloud DR)
├── 利用云服务快速启动备份站点
├── DRaaS (Disaster Recovery as a Service)
├── 按需付费，无需硬件投资
├── 启动时间：分钟-小时
└── 新兴趋势
```

## 五、备份策略设计与验证

### 5.1 三种备份类型

```
全量备份 (Full Backup)
├── 备份所有选中的数据
├── 恢复最简单（只需一套备份）
├── 耗时长、空间大
└── 周末一次 + 平时增量

增量备份 (Incremental Backup)
├── 备份自上次"任何备份"以来变化的数据
├── 备份最快、空间最小
├── 恢复最复杂（需要全量 + 所有增量链）
└── 每日一次

差异备份 (Differential Backup)
├── 备份自上次"全量备份"以来变化的数据
├── 备份速度介于两者之间
├── 恢复较简单（全量 + 最近一次差异）
└── 每日一次

恢复复杂度：
全量 < 差异 < 增量

备份空间需求：
增量 < 差异 < 全量
```

### 5.2 备份黄金法则

```
3-2-1 备份规则：

3 份数据副本
├── 1 份生产数据
└── 2 份备份数据

2 种不同的存储介质
├── 例如：本地磁盘 + 磁带/云
└── 防止同一介质类型故障

1 份异地备份
├── 地理上独立的副本
└── 防止站点级灾难（火灾/洪水/地震）

进阶版：3-2-1-1-0
├── 3 份副本
├── 2 种介质
├── 1 份异地
├── 1 份离线/不可变 (Air-gap/Immutable)
│   └── 勒索软件无法加密离线/不可变备份
└── 0 验证错误
    └── 定期验证备份可恢复性
```

### 5.3 备份存储介质

| 介质 | 速度 | 容量 | 耐久性 | 成本 |
|------|------|------|--------|------|
| SSD | ⭐⭐⭐ | ⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| HDD | ⭐⭐ | ⭐⭐⭐ | ⭐⭐ | ⭐⭐ |
| 磁带 (LTO) | ⭐ | ⭐ | ⭐⭐⭐ | ⭐ |
| 云对象存储 | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |

> **磁带未死**：大规模归档和长期保留仍推荐磁带（成本极低、离线隔离、长期可靠）。

### 5.4 备份验证

```
备份验证的四级测试：

Level 1: 文件存在性验证
└── "备份文件就在那儿" → 最低限度

Level 2: 校验和验证
└── 验证备份文件的哈希 → 确认未损坏

Level 3: 部分恢复测试
└── 随机抽取部分文件恢复并验证

Level 4: 全恢复测试
└── 在隔离环境中完整恢复系统 → 验证可正常运行
→ 至少每年做一次全恢复演练！
```

## 六、演练与测试

### 6.1 测试类型

```
BCP/DRP 的五种测试方法：

1. 检查清单测试 (Checklist Review)
   ├── 逐条检查计划文档的完整性
   ├── 成本最低
   └── 无法验证实际可用性

2. 桌面推演 (Tabletop Exercise)
   ├── 围绕假设场景讨论应对方案
   ├── 所有关键人员参与
   └── 验证沟通和决策流程

3. 模拟演练 (Simulation)
   ├── 真实场景模拟（但不影响生产系统）
   ├── "假装备"上操作
   └── 验证操作流程和协调

4. 并行测试 (Parallel Test)
   ├── 在备份站点实际恢复系统
   ├── 与生产系统并行运行
   ├── 不影响生产
   └── 验证恢复技术和RTO

5. 全中断测试 (Full-Interruption Test)
   ├── 实际关闭生产系统 → 从灾备恢复
   ├── 风险最高、最真实
   ├── 成本最高
   └── 通常仅在非业务时间
```

### 6.2 测试频率建议

| 测试类型 | 频率 |
|----------|------|
| 检查清单审查 | 每季度 |
| 桌面推演 | 每半年 |
| 模拟演练 | 每年 |
| 并行测试 | 每年 (关键系统) |
| 全中断测试 | 每2-3年 (非强制，按需) |

## 七、CISP 考试要点速查

### 核心概念

| 概念 | 一句话定义 |
|------|-----------|
| BCP | 确保业务在中断后继续运行的**总体框架** |
| DRP | BCP 的子集——专注于 **IT 恢复** |
| BIA | **前提分析**——识别关键业务流程和影响 |
| RTO | 服务恢复的**目标时间**（多久恢复） |
| RPO | 数据恢复的**目标时间点**（丢多少数据） |
| MTD | 最大可容忍中断时间（RTO 的上限） |

### 灾备站点

| 站点 | 恢复时间 | 成本 | 数据 |
|------|----------|------|------|
| 冷站 | 天-周 | 低 | 无/离线 |
| 温站 | 小时-天 | 中 | 定期 |
| 热站 | 分钟-小时 | 高 | 准实时 |
| 镜像站 | 秒-分钟 | 极高 | 实时 |

### 常见考试陷阱

1. ~~BCP = DRP~~ → BCP 范围更广，DRP 是 BCP 的 IT 子集
2. ~~RTO=恢复数据的时间~~ → RTO 是恢复**服务**的时间；RPO 是关于**数据**的
3. ~~热站不需要配置~~ → 热站仍需少量配置/切换（非自动）
4. ~~只做全量备份就够了~~ → 应遵循 3-2-1 规则
5. ~~BIA 是可选的分析步骤~~ → BIA 是 BCP/DRP 的**前提和基础**
