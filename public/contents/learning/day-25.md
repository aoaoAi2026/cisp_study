> 安全事件发生时，响应团队最怕的不是技术难题，而是"不知道下一步该做什么"。标准化的事件响应框架正是为解决这一问题而生。本章深入解析两大国际主流框架——SANS PICERL 和 NIST SP 800-61——帮你掌握安全事件响应的完整方法论。

## 一、事件响应的核心理念

### 1.1 什么是安全事件

```
安全事件 (Security Incident) 的定义：

事件 (Event)
├── 任何可观察到的系统/网络中发生的事情
├── 不一定是恶意的
└── 例：用户登录、文件修改、网络连接

安全事件 (Security Incident)
├── 违反了安全策略或对系统安全造成了实际或潜在威胁的事件
├── 需要采取响应措施
└── 例：恶意软件感染、DDoS攻击、数据泄露、未授权访问

事件响应 (Incident Response)
└── 对安全事件采取的有组织、有方法的处理流程
```

### 1.2 事件响应的黄金法则

```
事件响应的六条黄金法则：

1. 先取证，再行动
   "不要慌，先记录现状"
   被篡改前先截图、保存日志、导出内存

2. 先隔离，再清除
   "先防止扩散，再治疗感染"
   拔网线 → 取证 → 清除 → 恢复

3. 永远不要在被感染的系统上信任任何东西
   被黑的机器上的"正常"文件 → 可能已经被替换
   从干净的系统取证

4. 保持通信（但通过安全渠道）
   "攻击者可能在监控我们的通信"
   使用加密的带外通信渠道（手机/Signal/独立Teams）

5. 做任何事之前先保存证据
   "你只有一次机会查看现场状态"
   一旦重启/清除，很多证据永远消失

6. 记录一切操作
   每个操作的：时间、操作人、操作内容、原因、结果
   → 事后复盘和法律证据
```

## 二、SANS PICERL 六步模型

### 2.1 模型总览

```
SANS PICERL 事件响应模型：

P - Preparation     (准备)
I - Identification  (识别)
C - Containment     (遏制)
E - Eradication     (清除)
R - Recovery        (恢复)
L - Lessons Learned (经验教训)

    准备 ←─────────────────── 经验教训
     │                          ↑
     ▼                          │
    识别                         │
     │                          │
     ▼                          │
    遏制 ──→ 清除 ──→ 恢复 ───┘
```

### 2.2 各阶段详解

#### 准备 (Preparation)

```
准备好应对事件的"基础设施"：

人员准备
├── 组建 CSIRT (计算机安全事件响应团队)
├── 明确角色和职责
├── 24×7 联系方式 (手机 + 应急通信渠道)
└── 定期培训和演练

流程准备
├── 制定事件响应计划 (IRP)
├── 定义事件分类和升级路径
├── 制定沟通计划 (对内/对外/监管/媒体)
└── Playbook（针对常见事件类型的操作手册）

技术准备
├── 跳板机 (Jump Box)：干净的工具集
├── 取证工具包 (FTK Imager, Volatility, Wireshark)
├── 网络隔离方案 (VLAN 隔离/临时防火墙规则)
├── 备用通信渠道（独立于企业网络的）
└── 外网接入方案（不受内部攻击影响）

演练类型：
├── Tabletop Exercise (桌面推演)：讨论场景，验证流程
├── Simulation (模拟演练)：红蓝对抗，实战操作
├── Full-Scale (全规模演练)：真实环境模拟攻击
└── 频率：桌面推演每季度，全规模每年
```

#### 识别 (Identification)

```
确定是否真的发生了安全事件：

信息来源
├── SIEM 告警
├── EDR 检测
├── IDS/IPS 告警
├── 用户报告
├── 第三方通报 (CERT/ISAC/厂商)
└── 威胁情报

识别步骤：
1. 确认告警真实性 → 排除误报
2. 确定事件范围 → 多少系统受影响？
3. 评估影响 → 数据泄露？服务中断？
4. 初定严重级别 → Critical/High/Medium/Low
5. 记录初始发现 → 为后续调查留档

严重性的判断维度：
├── 受影响数据的敏感程度
├── 受影响系统的重要性
├── 攻击者的能力水平
├── 事件扩散的可能
└── 对业务的实际影响
```

#### 遏制 (Containment)

```
防止事件继续扩散：

短期遏制 (Short-term Containment)
├── 目标：立即止血，阻止扩散
├── 操作：
│   ├── 隔离受影响主机（拔网线/网络隔离）
│   ├── 禁用被攻破的用户账户
│   ├── 阻断 C2 通信 (防火墙/Proxy)
│   ├── 关闭受影响的服务
│   └── 修改被泄露的凭证
└── 通常在识别后 1 小时内完成

长期遏制 (Long-term Containment)
├── 目标：在取证完成前保持隔离状态
├── 操作：
│   ├── 将系统切换到隔离网段（仍可访问但不可出外网）
│   ├── 应用临时补丁
│   └── 加强临时监控
└── 同时进行：深入分析和取证

遏制策略的抉择：
├── 激进遏制：立即断网 → 取证受影响但业务损失小
├── 观察遏制：监控但不断网 → 取证信息多但风险大
└── 选择依据：事件的严重性和业务影响
```

#### 清除 (Eradication)

```
彻底清除攻击者的存在：

清除步骤：
1. 识别所有被修改/植入的组件
   ├── 恶意文件/进程
   ├── 后门账户
   ├── 计划任务/启动项
   ├── 注册表修改
   └── Rootkit/内核模块

2. 删除恶意组件
   ├── 杀毒/EDR 清除
   ├── 手动删除
   └── 终极方案：重装系统

3. 修复根本原因
   ├── 修补被利用的漏洞
   ├── 修复配置错误
   ├── 重置所有凭证
   └── 加强安全控制

4. 验证清除效果
   ├── 扫描确认无残留
   └── 确认防御措施生效

终极问题：重建还是重装？
├── 重装：最彻底，但可能丢失取证证据
├── 重建：保留证据，但可能遗留后门（需高度谨慎）
└── 规则：如果不够确定 → 重装
```

#### 恢复 (Recovery)

```
将系统恢复正常运行：

恢复步骤：
1. 从已知干净的备份恢复数据和系统
2. 部署修复后的系统
3. 恢复网络连接（去除隔离）
4. 恢复正常用户访问
5. 加强监控（恢复后至少 2 周密切关注）
6. 逐步恢复正常业务流程

恢复的注意事项：
├── 先验证备份的完整性（备份可能也被污染）
├── 在恢复前打好所有补丁（不要恢复后再打）
├── 恢复后保持高频率监控
├── 分批恢复，不要一次全部上线
└── 准备好回滚方案（恢复不成功时）

RTO / RPO：
├── RTO (恢复时间目标)：多久内恢复服务
└── RPO (恢复点目标)：最多丢失多久的数据
```

#### 经验教训 (Lessons Learned)

```
事后复盘，防止再次发生：

复盘会参与人员：
├── CSIRT 核心成员
├── 受影响业务部门
├── IT 运维
├── 安全架构师
├── 管理层代表
└── 外部顾问（如涉及）

复盘会的核心问题：
1. 攻击者是怎么进来的？(根因分析)
2. 我们是什么时候发现的？是否本可以更早？
3. 响应过程中哪些做得好？哪些需要改进？
4. 现有的控制措施哪些失效了？
5. 我们需要增加/改变什么来防止类似事件？

复盘后交付物：
├── 事件报告 (Incident Report)
├── 改进计划 (Remediation Plan) → 包含负责人和截止日
├── 更新的事件响应 Playbook
└── 安全控制增强方案
```

## 三、NIST 事件响应框架

### 3.1 NIST SP 800-61 Rev.2

```
NIST 事件响应四阶段 + 准备：

    准备 (贯穿始终)
         │
    ┌────▼────┐
    │1.检测与分析│
    └────┬────┘
         │
    ┌────▼────┐
    │2.遏制清除恢复│  ← 三步合并
    └────┬────┘
         │
    ┌────▼────┐
    │3.事后活动 │
    └─────────┘

与 SANS 的主要区别：
├── SANS：6步，步骤更细化
├── NIST：4步，将遏制/清除/恢复合并为一个阶段
└── 本质相同：都是 PDCA 循环思维
```

### 3.2 NIST 各阶段

| 阶段 | NIST 名称 | 对应 SANS |
|------|-----------|-----------|
| 0 | Preparation | Preparation |
| 1 | Detection and Analysis | Identification |
| 2 | Containment, Eradication, Recovery | Containment + Eradication + Recovery |
| 3 | Post-Incident Activity | Lessons Learned |

## 四、两大框架对比

| 维度 | SANS PICERL | NIST SP 800-61 |
|------|-------------|----------------|
| 步骤数 | 6步 | 4步 |
| 步骤粒度 | 更细化 | 更宏观 |
| 适用 | 企业 IR 实战 | 政府/合规导向 |
| 遏制清除恢复 | 分三步 | 合并一步 |
| CISP 考察频率 | ⭐⭐⭐ | ⭐⭐ |

## 五、事件响应团队与角色

### 5.1 CSIRT 构成

```
CSIRT (Computer Security Incident Response Team)：

核心角色：

事件响应经理 (Incident Response Manager)
├── 总指挥：协调整个响应过程
├── 决策升级/降级
└── 与管理层和外部沟通

安全分析师 (Security Analyst)
├── 技术调查：分析日志、恶意软件
├── 时间线重建
└── IOC 提取

取证专家 (Forensic Analyst)
├── 磁盘/内存取证
├── 证据收集和保管
└── 链式证据 (Chain of Custody)

网络分析师 (Network Analyst)
├── 网络流量分析
├── C2 通信追踪
└── 防火墙/交换机日志分析

法律顾问 (Legal Counsel)
├── 法律法规合规
├── 是否需上报监管机构
└── 证据的法律有效性

公关/沟通 (Communication)
├── 内部沟通
├── 客户通知
└── 媒体声明 (如有必要)
```

### 5.2 链式证据 (Chain of Custody)

```
链式证据 = 证明证据从收集到呈堂的完整性

一条完整的证据链记录：

证据编号：EVID-2024-001
证据描述：服务器 SVR01 的硬盘镜像 (SATA, 500GB, SN:ABC123)
收集人：张三 (取证工程师)
收集时间：2024-01-15 10:30 CST
收集方法：FTK Imager 4.7 → E01 格式
校验值：MD5 = abc..., SHA256 = def...

转移记录：
│ 时间 | 从谁 | 给谁 | 目的 | 确认签名 |
│ 10:30 | 张三(收集) | 李四(证据库) | 保管 | 张三、李四 |
│ 14:00 | 李四 | 王五(分析) | 取证分析 | 李四、王五 |
│ 18:00 | 王五 | 李四 | 分析完毕 | 王五、李四 |

关键原则：证据的每一次转移都必须有记录和签名
         → 保证在任何时间点都能追查谁接触过证据
```

## 六、事件分类与优先级

### 6.1 事件分类

```
安全事件分类体系：

类别一：恶意代码 (Malware)
├── 病毒/蠕虫/木马/勒索软件/挖矿程序
└── 例：WannaCry 勒索软件感染

类别二：入侵/未授权访问 (Intrusion)
├── 外部攻击者入侵
├── 内部越权访问
└── 账户密码泄露

类别三：拒绝服务 (DoS/DDoS)
├── 洪水攻击 (SYN/HTTP/DNS)
└── 资源耗尽

类别四：数据泄露 (Data Breach)
├── 敏感数据外传
└── 数据库被窃取

类别五：社会工程 (Social Engineering)
├── 钓鱼/鱼叉式钓鱼/BEC
└── 电话诈骗

类别六：策略违规 (Policy Violation)
├── 内部人员违反安全策略
└── 不当使用公司资源

类别七：物理安全 (Physical Security)
├── 未授权物理访问
└── 设备丢失/被盗
```

### 6.2 事件响应时间目标

| 严重级别 | 初始响应 | 遏制 | 恢复 | 复盘 |
|----------|----------|------|------|------|
| Critical | ≤ 15分钟 | ≤ 1小时 | ≤ 4小时 | 24小时内 |
| High | ≤ 1小时 | ≤ 4小时 | ≤ 8小时 | 48小时内 |
| Medium | ≤ 4小时 | ≤ 24小时 | ≤ 3天 | 1周内 |
| Low | ≤ 24小时 | ≤ 48小时 | ≤ 1周 | 2周内 |

## 七、CISP 考试要点速查

### SANS PICERL 六步

| 步骤 | 英文 | 核心任务 |
|------|------|----------|
| P | Preparation | 准备（人员/流程/工具/演练） |
| I | Identification | 识别（确认事件/评估范围） |
| C | Containment | 遏制（隔离/阻断/短期+长期） |
| E | Eradication | 清除（删恶意软件/修漏洞） |
| R | Recovery | 恢复（重建系统/准备回滚） |
| L | Lessons Learned | 复盘（根因分析/改进计划） |

### 常见考试陷阱

1. ~~事件的顺序是检测→分析→准备~~ → **准备在先**（还没出事就准备好了）
2. ~~应急响应的第一步是修复漏洞~~ → 第一步是**遏制（防止扩散）**
3. ~~清除=杀毒~~ → 清除还包括修复根因、重置凭证
4. ~~事件响应只是技术活~~ → 还需要法律、沟通、管理层参与
5. ~~恢复后工作就结束了~~ → 必须进行复盘和经验教训
