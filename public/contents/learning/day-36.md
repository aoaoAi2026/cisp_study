## 一、分组密码工作模式深度回顾

### 1.1 为什么需要工作模式？

分组密码（如AES）只能加密**固定长度**的数据块（16字节/128位），而实际数据往往更大。工作模式定义了如何将多个分组连接起来加密。

```
┌──────────────────────────────────────────────────────────────┐
│              工作模式核心选择维度                              │
├──────────────┬───────────────┬───────────────┬───────────────┤
│   模式       │   并行能力     │  是否需要IV   │   安全性      │
├──────────────┼───────────────┼───────────────┼───────────────┤
│   ECB        │   加密/解密   │   不需要      │   ⚠ 极不安全  │
│   CBC        │   解密可并行  │   需要，随机   │   ✅ 良好     │
│   CTR        │   加密/解密   │   需要，nonce  │   ✅ 良好     │
│   GCM        │   加密/解密   │   需要，nonce  │   ✅ 推荐     │
│   CFB        │   解密可并行  │   需要，随机   │   ✅ 良好     │
│   OFB        │   不可并行    │   需要，随机   │   ✅ 良好     │
└──────────────┴───────────────┴───────────────┴───────────────┘
```

### 1.2 安全红线的演变

```
历史轨迹：

1990s: ECB 广泛使用 → 2000s: 因"企鹅问题"被弃用
2000s: CBC 成为主流 → 2014: POODLE攻击揭示填充Oracle风险
2010s: GCM 崛起 → 当前: TLS 1.3 仅支持 AEAD 模式
```

## 二、ECB模式的致命缺陷演示

### 2.1 经典的"企鹅问题"

ECB模式最形象的安全缺陷展示：

```
原始图片（企鹅）：         ECB加密后（轮廓可见）：

■ ■ ■ ■ ■ ■ ■ ■ ■       ■ □ □ ■ ■ ■ □ □ □
■ □ □ □ □ □ □ □ ■       ■ □ □ □ □ □ □ □ ■
■ □ □ □ □ □ □ □ ■  →    ■ □ □ □ □ □ □ □ ■
■ ■ ■ ■ ■ ■ ■ ■ ■       ■ □ □ □ ■ • • • •
                         相同的明文块→相同的密文块
```

**核心问题**：相同的明文分组加密后产生相同的密文分组。

### 2.2 真实攻击示例：Cookie钓鱼

```
攻击场景：银行网站使用ECB加密Cookie中的用户角色

明文（16字节分组）：
Block 1: user=normaluser
Block 2: &role=user&mone
Block 3: y=1000&session=

攻击者通过调整输入位置，可以：
1. 将"&role=user"的密文替换为"&role=admin"的密文
2. 构造有效的管理员Cookie

原因：ECB下每个分组独立加密，块之间可以任意重排！
```

### 2.3 考试要点

| 知识点 | 记忆关键词 |
|--------|-----------|
| ECB缺陷 | "相同明文→相同密文"、"企鹅效应"、"分块独立" |
| 不建议使用 | 任何生产环境都不应使用ECB |
| 例外 | 仅加密随机数据且单次使用、KMS内部部分场景 |

## 三、CBC模式的填充Oracle攻击

### 3.1 攻击原理

CBC模式解密公式：
```
P_i = D_K(C_i) ⊕ C_{i-1}
```

如果攻击者能知道"解密后填充是否合法"，就能逐字节恢复明文——**这就是POODLE攻击的核心**。

```
攻击流程（Padding Oracle Attack）：
┌─────────────────────────────────────────────────────┐
│ 假设最后一个字节需要被猜测（padding值）               │
│                                                     │
│ C_{n-1}[15] ⊕ 0x01 = ?  (填充应为 0x01)            │
│                                                     │
│ 遍历修改C_{n-1}[15] 的256个可能值                    │
│   → 观察服务器返回："填充错误" 或 "MAC错误"?         │
│   → "MAC错误" = 填充通过检查 → 猜对！                │
│   → 由此可恢复 P_n[15] = 猜对的值 ⊕ 0x01            │
│                                                     │
│ 继续猜下一个字节...最终恢复全部明文                   │
└─────────────────────────────────────────────────────┘
```

### 3.2 防御方法

| 方法 | 说明 |
|------|------|
| **Encrypt-then-MAC** | 先加密，再对密文计算MAC（正确顺序） |
| **统一错误响应** | 填充错误和MAC错误返回相同错误信息 |
| **恒定时间比较** | 避免时序侧信道泄露 |
| **迁移到AEAD** | 直接使用GCM等认证加密模式 |

> **⚠️ 关键考试记忆**：CBC模式需要防范Padding Oracle攻击，迁移到GCM等AEAD模式是根本解决方案。

## 四、CTR模式的nonce重用问题

### 4.1 CTR模式原理

```
CTR模式将分组密码转为流密码：

Counter[0] → E_K → ⊕ P[0] = C[0]
Counter[1] → E_K → ⊕ P[1] = C[1]
Counter[2] → E_K → ⊕ P[2] = C[2]
...
```

### 4.2 Nonce重用的灾难

```
如果同一个Key和Nonce加密了两个不同的明文：

C1 = E_K(Nonce||Counter) ⊕ P1
C2 = E_K(Nonce||Counter) ⊕ P2

攻击者计算：
C1 ⊕ C2 = P1 ⊕ P2  （密钥流抵消！）

如果P1是已知头部（如HTTP请求行 "GET /"），
则可以恢复P2的全部内容！

这是流密码/CTR模式的通病："Two-time pad"
```

### 4.3 现实案例

| 案例 | 漏洞详情 |
|------|----------|
| **Adobe 2013** | ECB模式加密密码提示，泄露1.53亿用户 |
| **Sony PS3 2010** | ECDSA签名中随机数k被重用，私钥被恢复 |
| **WEP协议** | IV（24位）过短导致复用，密钥可恢复 |
| **Microsoft Office** | RC4流密钥重用，文档可解密 |

## 五、认证加密模式（GCM/CCM）

### 5.1 为什么需要AEAD

传统加密只保证**机密性**，不保证**完整性**。攻击者可以修改密文而不被检测。

```
传统加密：E_K(P) = C   → 攻击者修改C → C' → D_K(C') = P'（虽然P'可能无意义）
AEAD加密：E_K(P, AAD) = (C, Tag)  → 攻击者修改C → Tag验证失败 → 拒绝

AEAD = Authenticated Encryption with Associated Data
     = 加密 + 认证，一举两得
```

### 5.2 GCM vs CCM 对比

| 特性 | GCM | CCM |
|------|-----|-----|
| **基础原语** | AES-CTR + GHASH | AES-CBC-MAC + AES-CTR |
| **并行性** | ✅ 加密和解密均可并行 | 加密不能并行，解密可以 |
| **硬件加速** | ✅ 广泛支持（Intel AES-NI + CLMUL） | 较少 |
| **专利** | 无 | 无 |
| **Nonce长度** | 推荐96位（12字节） | 可变 |
| **标签长度** | 推荐128位 | 可变 |
| **TLS支持** | TLS 1.2 + TLS 1.3 | TLS 1.2 |

### 5.3 GCM Nonce管理铁律

> **⚠️ 对于同一密钥，GCM模式下Nonce绝对不能重复！** Nonce重用会导致认证密钥被恢复，所有安全属性全部丧失。

```
推荐方案：
1. 使用96位随机Nonce（碰撞概率极低）
2. 使用单调递增计数器（需状态管理）
3. 使用RBG-based deterministic construction（NIST SP 800-38D）
```

## 六、初始向量（IV）管理

### 6.1 IV vs Nonce

| 属性 | IV（Initialization Vector） | Nonce（Number used Once） |
|------|---------------------------|--------------------------|
| **唯一性要求** | 不可预测（对CBC） | 只需唯一，不需要随机 |
| **保密性** | 不需要保密 | 不需要保密 |
| **典型长度** | 分组大小（AES=128位） | 取决于算法（GCM推荐96位） |
| **使用模式** | CBC、CFB、OFB | CTR、GCM、CCM |

### 6.2 常见IV使用错误

```
错误1: 使用固定IV
  → CBC模式：泄露明文关系
  → 第一个分组可能被推断

错误2: 使用可预测的IV（如时间戳）
  → BEAST攻击（TLS 1.0）：利用CBC中可预测的IV

错误3: GCM使用短Nonce（<96位）
  → 需要确定性构造，增加实现复杂度

错误4: 加密后不传输IV
  → 接收方无法正确解密
```

## 七、现实世界的加密配置错误案例

### 7.1 案例汇总

| 案例 | 错误配置 | 后果 |
|------|---------|------|
| Adobe 2013 | ECB模式 + 密码提示明文存储 | 1.53亿用户泄露 |
| Zoom 2020 | ECB模式传输视频音频 | 非端到端加密 |
| PHP rand() | ECB + 弱随机数生成IV | 会话劫持 |
| JWT none算法 | 允许"none"算法 | 签名绕过 |
| WhatsApp | 信号协议实现bug | 中间人攻击 |
| WEP 2001 | 24位短IV + RC4 | 分钟级破解 |

### 7.2 安全加密检查清单

```
□ 永远不使用 ECB 模式
□ 使用 AEAD 模式（GCM 或 ChaCha20-Poly1305）
□ CBC 使用 Encrypt-then-MAC 结构
□ Nonce/IV 绝不重复使用
□ 密钥通过安全随机数生成器产生
□ 使用经过审计的密码库（OpenSSL, libsodium, BouncyCastle）
□ 密钥管理使用HSM或KMS
□ 密码算法支持协商和降级保护
```

## 八、CISP考试速查

### 考试高频考点

| 考点 | 记忆口诀 |
|------|---------|
| ECB为什么不安全 | "相同明文→相同密文，企鹅图" |
| CBC填充Oracle防御 | "先用GCM再说" |
| CTR Nonce重用危害 | "Two-time pad → XOR抵消" |
| GCM核心优势 | "有家有爱有认证（AEAD）" |
| 推荐加密模式 | "GCM当道，ChaCha20也是宝" |
| IV vs Nonce | "IV要随机，Nonce要唯一" |

### 常见陷阱

1. **"ECB用在什么地方都可以"** → ❌ 错！任何地方都不应该用ECB
2. **"CBC模式天然安全"** → ❌ 需要配合Encrypt-then-MAC和恒定时间实现
3. **"CTR Nonce可以随便用"** → ❌ Nonce重用是灾难性的
4. **"GCM只要Nonce不重复就行"** → ⚠️ 同时还要注意Tag长度和Nonce长度
