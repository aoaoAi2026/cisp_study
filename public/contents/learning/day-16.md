## 一、密码安全的核心挑战

### 1.1 密码的现实困境

```
密码悖论：

安全视角：密码应该
├── 长度长（≥12位，最好16位+）
├── 复杂度高（大小写+数字+符号）
├── 每个网站不同（避免撞库）
├── 定期更换
└── 不被记录在任何地方

人类视角：
├── 长密码记不住 → 写下来
├── 复杂密码记不住 → 用简单模式 (Password123!)
├── 每个网站不同 → 做不到 → 重复使用
└── 定期更换 → 只改最后一个数字 (Password1→Password2)

结论：单纯依赖密码的认证体系已经难以应对现代威胁
```

## 二、密码存储的进化史

### 2.1 五个时代的密码存储

```
Stage 1: 明文存储 ← 🔴 自杀式存储
├── 数据库里直接存密码
├── 黑客入侵 → 拿到所有密码
└── 后果：用户在其他网站的同密码全泄露

Stage 2: 简单 Hash ← 🟠 聊胜于无
├── 存储 MD5/SHA-1(password)
├── 相同密码 → 相同哈希（凭此可以判断密码相同）
├── 彩虹表攻击 → 反查哈希得到明文
└── 问题：没有加盐 → 批量破解

Stage 3: Hash + Salt ← 🟡 基本及格
├── 每个用户一个随机 Salt
├── 存储 Salt + Hash(password + salt)
├── 相同密码 → 不同哈希 → 破解成本增大
└── 问题：SHA 太快，GPU 仍可暴力

Stage 4: 慢哈希 + Salt ← 🟢 良好
├── bcrypt / scrypt / PBKDF2 + Salt
├── 故意让哈希运算很慢 + 耗内存
├── GPU 暴力破解速度从 10亿次/秒 降到 1万次/秒
└── 问题：参数需要与时俱进调整

Stage 5: Argon2 + Salt ← 🟢🟢 目前最佳
├── 2015 年密码哈希竞赛冠军
├── 抗 GPU（内存困难）+ 抗 ASIC（可调并行度）
├── 三个变体：Argon2d(抗GPU) / Argon2i(抗侧信道) / Argon2id(混合)
└── 推荐使用 Argon2id
```

### 2.2 加盐(Salt)的工作原理

```
为什么需要 Salt？

没有 Salt：
用户A密码: "123456" → MD5 → "e10adc3949ba59abbe56e057f20f883e"
用户B密码: "123456" → MD5 → "e10adc3949ba59abbe56e057f20f883e"
                           ↑ 一模一样 → 一次破解，全部沦陷

加了 Salt：
用户A: "123456" + "aB3xK9mQ" → Hash → "8f3a..."
用户B: "123456" + "wQ7zL2pR" → Hash → "c2d1..."
                           ↑ 完全不同 → 必须逐个破解

Salt 的安全要求：
├── 每个用户使用独立的随机 Salt
├── Salt 不需要保密（可以和 Hash 一起存储）
├── Salt 长度至少 16 字节（128 位）
└── Salt 必须使用密码学安全的随机数生成器 (CSPRNG)
```

### 2.3 慢哈希算法对比

| 算法 | 年份 | 抗 GPU | 抗 ASIC | 内存消耗 | 推荐度 |
|------|------|--------|---------|----------|--------|
| PBKDF2 | 2000 | ⚠️ 弱 | ❌ 弱 | 可配置 | ⭐⭐ |
| bcrypt | 1999 | ✅ | ⚠️ 中等 | 4KB固定 | ⭐⭐⭐ |
| scrypt | 2009 | ✅ | ✅ | 可配置(大) | ⭐⭐⭐ |
| **Argon2id** | 2015 | ✅✅ | ✅✅ | 可配置(大) | ⭐⭐⭐⭐⭐ |

## 三、密码攻击类型全览

### 3.1 在线攻击 vs 离线攻击

```
在线攻击 (Online Attack)：
├── 攻击者向登录接口提交密码尝试
├── 受网络延迟、速率限制、锁定策略限制
├── 速度：每秒几千次尝试（受限于网络）
└── 防御：速率限制、账户锁定、MFA、CAPTCHA

离线攻击 (Offline Attack)：
├── 攻击者已获得密码数据库（泄露/入侵）
├── 在自己的机器上本地爆破
├── 速度：GPU 每秒数十亿次尝试
└── 防御：慢哈希 + Salt → 强制降低破解速度
```

### 3.2 暴力破解与字典攻击

```
暴力破解 (Brute Force)：
├── 尝试所有可能的字符组合
├── a, b, c, ..., aa, ab, ..., aaa, ...
├── 时间复杂度 (按密码长度)：
│   ├── 8位纯数字：10^8 ≈ 1亿（秒级破解）
│   ├── 8位小写字母：26^8 ≈ 2000亿（分钟级）
│   ├── 8位大小写+数字：62^8 ≈ 218万亿（天级）
│   └── 12位大小写+数字+符号：95^12 ≈ 5.4×10^23（不可能）

字典攻击 (Dictionary)：
├── 使用常见密码列表尝试
├── 最常见的密码（2023年统计）：
│   ├── 123456
│   ├── password
│   ├── 123456789
│   ├── 12345678
│   ├── admin
│   └── qwerty
└── 增强：字典 + 规则变异 (password→P@ssw0rd)
```

### 3.3 彩虹表攻击

```
彩虹表 (Rainbow Table) = 预计算的 Hash→明文 映射表

原理：
├── 事先计算所有可能明文的哈希值，存入查找表
├── 拿到哈希后直接查表 → 秒级还原
└── 典型的 MD5 彩虹表可达 100+ GB

防御：
└── Salt → 每个用户的 Salt 不同 → 需要为每个用户单独建彩虹表
         → 成本从 O(1) 变为 O(n) → 彩虹表攻击失效
```

### 3.4 撞库攻击 (Credential Stuffing)

```
撞库攻击的流程：

1. 攻击者从 A 网站的泄露数据中获取大量用户名+密码
2. 用同样的凭证去尝试登录 B、C、D... 网站
3. 因为大量用户在不同网站使用相同密码，成功率 ~1-2%

防御：
├── MFA → 即使密码对了，也需要第二因素
├── 异常登录检测（异地、新设备、异常时间）
├── 密码泄露检测（对比 Have I Been Pwned 数据库）
└── 密码管理器推广（帮助用户使用不同密码）
```

### 3.5 密码喷洒攻击 (Password Spraying)

```
区别于暴力破解：
├── 暴力破解：对一个账户尝试大量密码
│   → 容易触发账户锁定
├── 密码喷洒：用少量常见密码（如 Spring2024!）
│             尝试大量账户
│   → 不容易触发锁定（每个账户只试几次）

典型喷洒密码列表：
├── 公司名+年份+!  (如 Cisp2024!)
├── 季节+年份+! (Spring2024!)
├── 城市名+数字 (Beijing123)
└── Password123

防御：
├── 检测：短时间内大量账户登录失败（不同账户）
├── 对所有失败登录进行速率限制（不限单个账户）
└── 密码复杂度要求 + MFA
```

## 四、密码策略最佳实践

### 4.1 NIST SP 800-63B 密码指南

```
NIST (美国国家标准与技术研究院) 2023 年最新密码指南：

✅ 应该做：
├── 密码长度至少 8 位（推荐 12-16 位）
├── 支持所有可打印字符（包括空格和 Unicode）
├── 接受最长 64+ 位密码
├── 检查密码是否在已知泄露密码列表中
├── 仅在怀疑泄露时强制更换
└── 使用密码强度指示器

❌ 不应该做：
├── 要求复杂的组合规则（大小写+数字+符号）
│   → 用户会找规律规避：Password123! → Passw0rd123!
├── 频繁强制更换密码（如每90天）
│   → 用户会做最小改动：Password1→Password2
├── 使用密保问题（母亲的姓氏、第一只宠物）
│   → 这些信息在社交媒体上大量公开
├── 使用密码提示
└── 截断密码（限制密码不影响安全）
```

### 4.2 企业密码策略建议

```
平衡安全性与可用性的企业密码策略：

┌──────────────────────────────────────┐
│ 策略项            推荐值               │
├──────────────────────────────────────┤
│ 最小长度          12 位               │
│ 最大长度          128 位              │
│ 必须检查弱密码    Have I Been Pwned  API │
│ 强制更换条件      密码泄露时            │
│ 账户锁定          5次失败→15分钟锁定    │
│ 密码历史          禁止重用最近5个密码   │
│ MFA              全员强制（特别是管理员）│
│ 密码管理器        建议使用（Bitwarden等）│
└──────────────────────────────────────┘

关键转变：
├── 从"定期更换"转向"泄露时更换"
├── 从"复杂度要求"转向"长度优先"
├── 从"密码信仰"转向"MFA 必需"
└── 从"记忆负担"转向"密码管理器"
```

### 4.3 特权账户密码管理

```
特权账户 (Privileged Account) 额外要求：

├── 密码长度 ≥ 16 位
├── 必须强制 MFA（不能妥协）
├── 自动密码轮换（PAM 工具自动定期更换）
├── 密码保险库 (Vault) 存储
│   └── 一次使用后自动更换 (Check-out/Check-in)
├── 会话录像和审计
├── 个人特权账户与日常账户分离
│   └── Admin-张三 ≠ 张三
└── 可以临时提权，到期自动回收 (JIT Access)
```

## 五、无密码认证的未来

### 5.1 FIDO2 / WebAuthn 标准

```
无密码认证的核心理念：

传统认证：共享秘密 (密码) → 服务器知道你的密码
FIDO2 认证：公私钥对 → 服务器只存公钥

FIDO2 流程：
1. 注册：设备生成公私钥对 → 公钥发给服务器，私钥存设备
2. 认证：服务器发 Challenge → 设备用私钥签名 → 服务器用公钥验证

关键安全特性：
├── 私钥永不离设备 → 服务器无法泄露私钥
├── Challenge-Response → 防重放
├── 域名绑定 → 防钓鱼（验证器检查 RP ID）
│   └── fake-bank.com ≠ real-bank.com → 拒绝签名
└── 支持平台验证器 (Touch ID/Windows Hello) 或漫游验证器 (YubiKey)
```

### 5.2 无密码的多种形态

| 方案 | 体验 | 安全 | 防钓鱼 |
|------|------|------|--------|
| 硬件密钥 (FIDO2) | ⚠️ 需携带 | ✅✅ | ✅✅ |
| 平台生物 (指纹/人脸) | ✅ 方便 | ✅ | ✅ 域绑定 |
| 手机推送 (Push) | ✅ 方便 | ✅ | ⚠️ MFA疲劳攻击 |
| 短信/OTP | ✅ 熟悉 | ⚠️ SIM卡劫持 | ❌ 无域绑定 |
| 密码+OTP | ⚠️ 组合 | ✅ | ⚠️ 密码可钓鱼 |

> **趋势**：FIDO2 是密码认证的未来方向。Google 内部部署 FIDO2 后，员工账户盗用事件降为零。

## 六、CISP 考试要点速查

### 核心概念

| 概念 | 要点 |
|------|------|
| Salt | 每个用户独立的随机值 → 抵抗彩虹表 |
| 慢哈希 | bcrypt/scrypt/Argon2 → 抗 GPU 暴力 |
| 在线攻击 | 受限于网络速率和锁定策略 |
| 离线攻击 | 拿到密码库后本地爆破 → 需慢哈希防御 |
| 撞库 | 一处泄露的密码尝试登录其他网站 |
| 密码喷洒 | 少量常见密码尝试大量账户 |

### 密码存储对比

| 方案 | 安全性 | 说明 |
|------|--------|------|
| 明文 | 🔴 自杀 | 绝对不可用 |
| Hash(密码) | 🟠 极弱 | 彩虹表秒破 |
| Hash(密码+盐) | 🟡 勉强 | SHA 太快，GPU 仍可爆破 |
| bcrypt(密码+盐) | 🟢 良好 | 成本因子 ≥ 12 |
| **Argon2id** | 🟢🟢 最佳 | 密码哈希竞赛冠军 |

### 常见考试陷阱

1. ~~MD5+Salt 可以安全存储密码~~ → MD5 太快，GPU 可以百亿次/秒暴力破解
2. ~~定期改密码一定更安全~~ → NIST 已不再推荐，泄漏时更换更好
3. ~~复杂组合规则提升安全~~ → 长度才是密码强度的第一要素
4. ~~密保问题增强了安全~~ → 密保问题答案常在社交媒体上可查
5. ~~Hash 可以"解密"~~ → Hash 是单向不可逆的，不存在"解密"

> 💡 **本章小结**：密码认证的核心矛盾是"安全"与"可用"的平衡。技术上，密码存储的黄金标准是 Argon2id + 独立随机 Salt + 高成本参数。策略上，NIST 提出的"长度优先、泄露更换、MFA 必须"三条原则是现代密码管理的基本指南。考试重点：Salt 抗彩虹表、慢哈希抗暴力、在线vs离线攻击的差异。
