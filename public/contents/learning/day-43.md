## 一、TCP/IP协议栈安全全景

### 1.1 TCP/IP四层模型与安全威胁

```
    OSI          TCP/IP           安全威胁示例
┌─────────┐  ┌──────────┐  ┌──────────────────────────────┐
│  应用层  │  │          │  │ HTTP劫持、DNS投毒、SMTP伪造   │
│  表示层  │  │  应用层   │  │ SSL/TLS降级、SSH中间人        │
│  会话层  │  │          │  │ Cookie窃取、会话劫持          │
├─────────┤  ├──────────┤  ├──────────────────────────────┤
│  传输层  │  │  传输层   │  │ TCP劫持、SYN Flood、UDP Flood │
├─────────┤  ├──────────┤  ├──────────────────────────────┤
│  网络层  │  │ 互联网层  │  │ IP欺骗、Smurf攻击、分片攻击   │
├─────────┤  ├──────────┤  ├──────────────────────────────┤
│ 数据链路层│ │ 网络接口层 │  │ ARP欺骗、MAC泛洪、VLAN跳跃   │
│  物理层  │  │          │  │ 线缆窃听、电磁泄露            │
└─────────┘  └──────────┘  └──────────────────────────────┘
```

### 1.2 协议安全基本原则

```
协议设计安全的四个原则（RFC 3552）：

① 加密优先：能加密就加密，避免明文传输
② 认证必选：通信双方身份必须经过认证
③ 完整性保护：数据在传输中不可被篡改
④ 抗重放：每条消息应包含防止重放的机制（时间戳/序号）
```

## 二、ARP协议攻击与防御

### 2.1 ARP欺骗原理

```
正常ARP流程：                        ARP欺骗攻击：

A想知道B的MAC                        攻击者C冒充B
  │                                    │
  │  ARP Request                      │  ARP Reply (伪造)
  │  "谁是192.168.1.2？"              │  "192.168.1.2 的MAC
  │  ─────────→ 广播                  │   是 CC:CC:CC:CC:CC:CC"
  │           │                       │  ─────────→ A
  │  ARP Reply                        │
  │  "我是，MAC=BB:BB"                │  结果：
  │  ←───────── B                     │  A将发往B的流量全部发给C
  │                                    │  → MITM中间人攻击！
  
ARP协议的根本问题：无状态、无认证、无条件接受ARP Reply
```

### 2.2 攻击影响

| 攻击类型 | 描述 | 影响 |
|----------|------|------|
| **ARP欺骗** | 伪造ARP响应，篡改IP-MAC映射 | 中间人攻击、流量窃听 |
| **ARP泛洪** | 发送大量伪造ARP请求 | 交换机CAM表溢出→广播所有帧 |
| **免费ARP欺骗** | 发送声称是网关的免费ARP | 局域网全部流量经攻击者 |
| **ARP DoS** | 将目标IP映射到不存在的MAC | 目标网络完全不可达 |

### 2.3 防御措施

```
✅ 静态ARP表（小网络适用）
✅ DAI (Dynamic ARP Inspection) — 交换机验证ARP包
✅ DHCP Snooping + IP Source Guard — 绑定信任的IP-MAC
✅ ARP监控工具（arpwatch, XArp）
✅ 网络分段隔离敏感区域
✅ 使用IPv6（理论上移除了ARP，但使用NDP有类似攻击）
```

## 三、IP协议安全

### 3.1 IP欺骗攻击

```
IP欺骗：伪造IP报文中的源IP地址

攻击类型：

① 盲欺骗 (Blind Spoofing)：
  攻击者无法看到目标响应
  需要猜测TCP序列号

② 非盲欺骗：
  攻击者在同一网段，可嗅探流量
  → TCP会话劫持

③ 反射攻击：
  源IP＝目标IP → 服务器响应发向目标
  → DNS Amplification, NTP Amplification

④ Smurf攻击：
  发送ICMP Echo Request到广播地址
  源IP＝目标IP → 所有主机回复目标
```

### 3.2 IP分片攻击

```
IP分片攻击类型：

① Tear Drop：
  构造重叠的分片偏移量
  → 旧系统重组时崩溃或死机

② Tiny Fragment Attack：
  将TCP头信息分散到极小的分片中
  → 绕过仅检查第一个分片的防火墙

③ Fragment Overlap：
  第二个分片覆盖第一个分片的端口信息
  → 防火墙放过第一个分片，实际到达的是恶意分片

防御：
  ✅ 防火墙重组分片后再检查
  ✅ 拒绝畸形分片（偏移量重叠、过小）
  ✅ 限制分片大小和数量
```

## 四、TCP协议安全

### 4.1 TCP三次握手与攻击

```
正常三次握手：                 SYN Flood攻击：

C→S: SYN (seq=x)             C→S: SYN (seq=rand₁)
S→C: SYN+ACK (seq=y, ack=x+1) C→S: SYN (seq=rand₂)
C→S: ACK (ack=y+1)           C→S: SYN (seq=rand₃)
                               C→S: SYN (seq=rand₄)
                               ... (不断发送，从不完成)
                               
                               S为每个SYN分配资源，
                               半连接队列被填满
                               → 合法连接无法建立！

防御方法：
  ① SYN Cookie：不在服务端存储半连接状态
  ② SYN Proxy：中间代理验证客户端后才转发
  ③ 增加半连接队列大小
  ④ 缩短SYN超时时间
```

### 4.2 TCP会话劫持

```
TCP劫持条件：
  攻击者需要：
  1. 能够嗅探通信
  2. 能预测TCP序列号
  3. 能发送伪造的数据包

攻击流程：
  1. 观察合法客户端和服务器之间的TCP会话
  2. 获取当前序列号
  3. 向服务器发送伪造的数据包（使用正确序列号）
  4. 客户端发送的数据被服务器忽略（序列号不匹配）
  5. 攻击者接管会话

防御：
  ✅ TLS加密（即使劫持也无法解密）
  ✅ TCP MD5签名（BGP使用）
  ✅ 随机初始序列号（现代OS已实施）
  ✅ TCP-AO (RFC 5925)
```

### 4.3 TCP Reset攻击

```
RST攻击：发送伪造的TCP RST包终止连接

需要信息：
  · 源IP、目标IP
  · 源端口、目标端口
  · 有效序列号（窗口范围内）

案例：Great Firewall的TCP RST
  → 检测敏感关键词 → 向两端发送RST → 中断连接

防御：TLS加密（RST需在加密层内，无法通过明文RST影响加密连接）
```

## 五、DNS安全深度分析

### 5.1 DNS攻击全景

```
┌───────────────┬───────────────────────────────────────┐
│   攻击类型     │                描述                    │
├───────────────┼───────────────────────────────────────┤
│ DNS缓存投毒    │ 向DNS服务器注入伪造记录，污染缓存       │
│ DNS劫持        │ 修改DNS服务器配置或域名注册信息          │
│ DNS隧道        │ 通过DNS查询/响应传输非DNS数据（C2）     │
│ DNS放大攻击    │ 小查询→大响应，反射型DDoS               │
│ DNS重绑定      │ 短TTL+切换IP→绕过同源策略              │
│ 域前向劫持      │ 子域名接管：过期CNAME指向可控资源       │
│ Phantom Domain │ 慢速DNS攻击：大量不存在的域名查询       │
└───────────────┴───────────────────────────────────────┘
```

### 5.2 DNS缓存投毒（Kaminsky攻击）

```
2008年Dan Kaminsky发现：
  DNS查询仅使用16位Transaction ID
  + 源端口固定（大多数DNS服务器）
  → 攻击者仅需猜测 2^16 次

攻击步骤：
  1. 向受害DNS发送对"xxx.google.com"的查询
  2. 洪水般发送伪造DNS响应
     → 每个伪造响应尝试不同的Transaction ID
     → 平均2^15次即可猜中
  3. 伪造响应中包含NS记录的额外信息：
     google.com NS ns.attacker.com
  4. DNS缓存被污染，整个google.com域名被劫持

防御：
  ✅ DNSSEC（数字签名验证DNS数据）
  ✅ 随机源端口（2^16×端口范围）
  ✅ 0x20编码（大小写随机化，增加熵）
  ✅ DNS over TLS/HTTPS
```

### 5.3 DNSSEC

```
DNSSEC提供：
  · 数据来源认证（DNS数据来自权威服务器）
  · 数据完整性（数据在传输中未被篡改）
  · 否定存在证明（证明某域名不存在）

DNSSEC不提供：
  · 机密性（DNS数据仍是明文的）
  · DDoS防护
  · 最后一英里保护（stub到递归的链路）

信任链：
  Root (.) → .com → example.com → www.example.com
  每个区域用私钥签名，上级区域包含下级公钥的DS记录
```

## 六、ICMP攻击与利用

### 6.1 常见ICMP攻击

| 攻击类型 | ICMP类型 | 描述 |
|----------|---------|------|
| Ping of Death | Echo (8) | 超大ICMP包(>65535B)→系统崩溃 |
| Smurf | Echo (8) | 广播地址+伪造源IP→放大攻击 |
| ICMP Flood | Echo (8) | 大量Ping→带宽耗尽 |
| ICMP重定向 | Redirect (5) | 伪造路由→流量劫持 |
| ICMP不可达 | Dest Unreach (3) | 伪造→阻断合法连接 |

### 6.2 ICMP隧道

```
攻击者利用ICMP的数据载荷传输任意数据：

  C2服务器 ←── ICMP Echo Reply [加密命令] ────── 被控主机
  C2服务器 ──── ICMP Echo Request [命令结果] → 被控主机

优势：大多数防火墙不深度检查ICMP载荷

防御：
  · 限制ICMP包大小
  · 检查ICMP载荷异常
  · 严格防火墙ICMP策略
```

## 七、DHCP安全

### 7.1 DHCP攻击类型

```
① DHCP Starvation：
   伪造大量MAC地址，耗尽DHCP地址池
   → 合法用户无法获取IP

② DHCP Spoofing/Rogue DHCP：
   部署恶意DHCP服务器
   → 分配攻击者控制的DNS/网关
   → 中间人攻击

③ DHCP ACK Flood：
   伪造多个IP的DHCP ACK
   → 目标IP地址冲突

防御：
  ✅ DHCP Snooping（交换机区分信任/非信任端口）
  ✅ 限制每个端口的MAC地址数量
  ✅ 802.1X认证
```

## 八、网络层安全协议IPsec概述

### 8.1 IPsec核心组成

```
IPsec两个主要协议：

┌───────────────┬────────────────────┬──────────────────┐
│    协议       │       模式         │      保护的       │
├───────────────┼────────────────────┼──────────────────┤
│ AH(协议51)    │ 传输/隧道          │ IP头+数据 完整性  │
│ ESP(协议50)   │ 传输/隧道          │ 数据 机密性+完整性 │
└───────────────┴────────────────────┴──────────────────┘

安全关联 (SA)：
  · 单向安全连接的定义
  · 通过SPI (Security Parameter Index)标识
  · IKE (Internet Key Exchange)协商SA

IKEv2密钥交换：
  Phase 1: IKE_SA_INIT → 建立加密通道(DH密钥交换)
  Phase 2: IKE_AUTH → 身份认证和子SA建立
```

### 8.2 AH vs ESP

| 特性 | AH | ESP |
|------|----|----|
| 机密性 | ❌ | ✅ |
| 完整性 | ✅（含IP头可变字段） | ✅（仅ESP数据） |
| 抗重放 | ✅ | ✅ |
| NAT穿透 | ❌（IP头被保护） | ✅（NAT-T） |
| 协议号 | 51 | 50 |

## 九、CISP考试速查

### 高频考点

| 考点 | 关键记忆 |
|------|---------|
| ARP欺骗本质 | "无状态、无认证、无条件接受Reply" |
| SYN Flood防御 | "SYN Cookie = 无状态握手" |
| DNS缓存投毒 | "16位TXID + 固定端口 = 可猜测" |
| DNSSEC三功能 | "来源认证 + 完整性 + 存在性证明" |
| DHCP安全 | "DHCP Snooping区分信任/非信任端口" |
| IPsec两协议 | "AH = 完整性，ESP = 机密性+完整性" |

### 常见陷阱

1. **"IPsec自动加密所有IP流量"** → 仅加密SA中定义的流量
2. **"DNSSEC加密了DNS查询"** → DNSSEC只签名，不加密。加密需DNS over TLS/HTTPS
3. **"交换机天然免疫ARP欺骗"** → 交换机只转发，不验证ARP内容
4. **"HTTPS不受ARP欺骗影响"** → 正确，但攻击者可以做SSL/TLS降级
