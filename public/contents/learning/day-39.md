## 一、证书透明化（Certificate Transparency）

### 1.1 为什么需要CT？

```
传统PKI的致命缺陷：
  任何受信任的CA都能为任何域名签发证书
  → 如果CA被攻破或恶意行为
  → 可签发伪造的google.com证书
  → 浏览器完全信任它！

历史事件：
  2011 - DigiNotar CA被攻破：
    攻击者签发了google.com等531个域名的伪造证书
    被用于对伊朗用户进行大规模MITM
    直接导致DigiNotar破产
    
  2012 - TurkTrust错误签发中级CA：
    错误签发可用于签发任何证书的中间CA
    被用于Google域名的MITM
```

### 1.2 CT三组件

```
┌─────────────────────────────────────────────────────┐
│              Certificate Transparency 架构            │
│                                                      │
│  ① 证书日志 (Log)                                     │
│     - 只追加的Merkle树                                │
│     - 公开可审计                                      │
│     - 提交证书 → 获得SCT (Signed Certificate Timestamp)│
│                                                      │
│  ② 监控器 (Monitor)                                    │
│     - 持续扫描日志                                    │
│     - 检测可疑证书（如非预期的域名）                  │
│     - 告警域名所有者                                  │
│                                                      │
│  ③ 审计器 (Auditor)                                   │
│     - 验证日志行为一致性                              │
│     - 确保日志没有"分叉"                              │
│     - 检测日志是否合规                                │
└─────────────────────────────────────────────────────┘
```

### 1.3 SCT交付方式

| 方式 | 说明 | 适用场景 |
|------|------|----------|
| **X.509 v3扩展** | CA提前获取SCT嵌入证书 | 推荐方式 |
| **TLS扩展** | 服务器在握手时发送SCT | 兼容旧CA |
| **OCSP Stapling** | OCSP响应中包含SCT | 补充方式 |

> **Chrome要求**：2021年起，所有新签发的证书必须提供至少2个SCT才能被信任。

## 二、证书钉扎（Certificate Pinning）

### 2.1 钉扎原理

```
证书钉扎：客户端只信任预置的特定证书或公钥

类型对比：

┌────────────────┬──────────────────────┬──────────────────┐
│    钉扎类型    │       钉扎内容       │      优缺点      │
├────────────────┼──────────────────────┼──────────────────┤
│ 证书钉扎       │ 完整的证书           │ 精确但需频繁更新 │
│ 公钥钉扎       │ SPKI Hash            │ 证书更换不失效   │
│ CA钉扎         │ 只信任特定CA签发的   │ 较宽松但有限制   │
│ 叶证书+CA      │ 组合策略             │ 平衡安全与灵活   │
└────────────────┴──────────────────────┴──────────────────┘

HTTP公钥钉扎（HPKP）- 已废弃
  原因：配置错误导致站点永久无法访问（"HPKP自杀"）
  替代：Certificate Transparency + Expect-CT
```

### 2.2 Android/iOS证书钉扎

```java
// Android Network Security Config (网络配置文件)
<network-security-config>
    <domain-config>
        <domain includeSubdomains="true">api.example.com</domain>
        <pin-set expiration="2026-01-01">
            <!-- 主证书备份密钥 -->
            <pin digest="SHA-256">7HIpactkIAq2Y49orFOOQKurWxmmSFZhBCoQYcRhJ3Y=</pin>
            <!-- 备份证书密钥（CA发生变化时使用） -->
            <pin digest="SHA-256">fwza0LRMXouZHRC8Ei+4PyuldPDcf3UKgO/04cDM1oE=</pin>
        </pin-set>
    </domain-config>
</network-security-config>
```

### 2.3 钉扎最佳实践

```
Do：
  ✅ 始终包含备用密钥（"backup pin"）
  ✅ 设置合理的过期时间
  ✅ 公钥钉扎优于证书钉扎
  ✅ 在请求中实现不中断降级策略

Don't：
  ❌ 只钉扎单个证书/密钥（单点故障）
  ❌ 钉扎过期时间设置过长（已废弃的HPKP曾导致此问题）
  ❌ 在生产环境直接启用前没做充分测试
```

## 三、PKI信任模型深度对比

### 3.1 三种信任模型

```
① 层级信任模型（Hierarchical）- 当前主流

         Root CA
           │
    ┌──────┼──────┐
    │      │      │
  ICA1   ICA2   ICA3
    │      │      │
  Cert   Cert   Cert

  优点：结构清晰，容易管理
  缺点：Root CA是单点故障


② 网状信任模型（Mesh/Web of Trust）- PGP/GPG

  Alice ←→ Bob
    ↕       ↕
  Carol ←→ Dave

  优点：去中心化
  缺点：信任度难以量化，可扩展性差


③ 桥接信任模型（Bridge CA）

  Org A ----\          /---- Org C
             Bridge CA
  Org B ----/          \---- Org D

  优点：连接不同PKI孤岛
  缺点：桥CA成为关键节点
```

### 3.2 各模型实战对比

| 维度 | 层级模型 | 网状模型(WoT) | 桥接模型 |
|------|---------|---------------|---------|
| **使用场景** | Web PKI, 企业PKI | PGP, 个人密钥 | 政府跨部门, 行业联盟 |
| **信任建立** | 通过CA链 | 通过签名/信任关系 | 通过桥CA |
| **可扩展性** | ✅ 优秀 | ❌ 差 | ⚠️ 中等 |
| **撤销效率** | CRL/OCSP | 密钥服务器 | 取决于实现 |
| **用户友好度** | ✅ 透明 | ❌ 需要用户判断 | ⚠️ 需要协调 |

## 四、ACME协议与自动化证书管理

### 4.1 ACME协议流程

```
ACME (Automatic Certificate Management Environment)
— Let's Encrypt使用的证书自动化协议

完整流程：

  客户端                         ACME服务器
    │                               │
    │──── 1. 创建账户 ─────────────→│
    │←─── 2. 账户URL+密钥 ─────────│
    │                               │
    │──── 3. 提交订单(域名列表) ───→│
    │←─── 4. 授权挑战列表 ──────────│
    │                               │
    │  5. 完成挑战(HTTP-01/DNS-01)  │
    │     └→ 放文件/.well-known/    │
    │     └→ 或 设置DNS TXT记录    │
    │                               │
    │──── 6. 请求验证 ─────────────→│
    │←─── 7. 验证结果 ──────────────│
    │                               │
    │──── 8. 发送CSR ──────────────→│
    │←─── 9. 签发证书 ─────────────│
    │                               │
    │    证书有效期：90天            │
    │    推荐续期：60天后            │
```

### 4.2 两种挑战方式

| 挑战类型 | 验证方式 | 适用场景 | 注意事项 |
|----------|---------|----------|----------|
| **HTTP-01** | 在 `http://domain/.well-known/acme-challenge/<token>` 放文件 | 公网可访问的Web服务 | 需要开放80端口 |
| **DNS-01** | 在DNS中设置TXT记录 `_acme-challenge.domain` | 通配符证书、内网服务 | 需要DNS API权限 |
| **TLS-ALPN-01** | 在TLS握手时验证 | 仅需要443端口 | 较少使用 |

## 五、PKI攻击面与防御

### 5.1 主要攻击向量

```
① CA妥协攻击
   → DigiNotar (2011), Comodo (2011)
   → 防御: CT, CAA记录, 多方签名CA密钥

② 中间人攻击（MITM）
   → 伪造证书 + 代理流量
   → 防御: 证书钉扎, CT监控, HSTS预加载

③ 降级攻击
   → 强制使用弱加密套件或过时TLS版本
   → 防御: 禁用过时协议, TLS_FALLBACK_SCSV

④ 域名验证绕过
   → BGP劫持、DNS投毒获取验证Token
   → 防御: 多路验证（Multi-VA）, CAA记录

⑤ 私钥泄露
   → 服务器被攻破、备份泄露
   → 防御: HSM, 密钥轮换, 访问控制
```

### 5.2 DNS CAA记录

```
CAA (Certification Authority Authorization)：
指定哪些CA可以为特定域名签发证书

示例DNS记录：
  example.com.  CAA  0  issue  "letsencrypt.org"
  example.com.  CAA  0  issue  "digicert.com"
  example.com.  CAA  0  issuewild  ";"   （禁止通配符证书）
  example.com.  CAA  0  iodef  "mailto:security@example.com"

作用：
  ✅ 限制可签发证书的CA
  ✅ CA在签发前必须检查CAA记录
  ✅ 违反CAA → 拒绝签发
```

## 六、国密证书体系

### 6.1 国密算法与标准

| 算法 | 对标国际 | 用途 | 标准 |
|------|---------|------|------|
| SM2 | ECDSA (P-256) | 数字签名、密钥交换 | GB/T 32918 |
| SM3 | SHA-256 | 哈希 | GB/T 32905 |
| SM4 | AES-128 | 对称加密 | GB/T 32907 |
| SM9 | IBE | 基于标识的密码 | GB/T 38635 |

### 6.2 国密证书与RSA/ECC证书对比

```
普通SSL证书：RSA + SHA-256 或 ECDSA + SHA-256
国密SSL证书：SM2 + SM3 + SM4

双证书方案（过渡期）：
  ┌──────────────────────────────────────┐
  │  同时部署国际证书和国密证书            │
  │                                      │
  │  国际浏览器 → RSA/ECDSA证书 → TLS    │
  │  国密浏览器 → SM2证书 → 国密TLS      │
  │                                      │
  │  Nginx/Apache需要同时配置两套证书     │
  └──────────────────────────────────────┘
```

## 七、代码签名与时间戳

### 7.1 代码签名的意义

```
代码签名提供：
  ① 来源认证：代码来自声称的发布者
  ② 完整性：代码在发布后未被篡改
  ③ 时间戳：证明签名时证书有效（即使证书后来过期/吊销）

时间戳的重要性：
  没有时间戳：
    签名时证书有效 → "证书过期" → 签名"失效"
  有时间戳：
    签名时证书有效 → 时间戳证明当时有效 → 签名永久有效
```

### 7.2 代码签名安全实践

```
最佳实践：
  ✅ 使用硬件安全模块（HSM）存储签名密钥
  ✅ 私钥离线存储，签名在气隙环境中进行
  ✅ 实施双人控制（需要两人同时授权）
  ✅ 记录签名操作日志
  ✅ 验证签名后再分发软件
  ✅ 始终包含时间戳
```

### 7.3 Stuxnet案例分析

```
Stuxnet（震网病毒，2010）：
  使用了两个被盗的数字证书：
  - JMicron Technology 的代码签名证书
  - Realtek Semiconductor 的代码签名证书
  
  证书来路：
  → 可能是物理入侵办公场所窃取
  → 或通过社会工程学获得
  
  影响：
  → 驱动文件被Windows信任并加载
  → 攻击伊朗核设施的西门子PLC
  → 伊朗纳坦兹核设施约1000台离心机受损
  
  教训：
  → 代码签名证书是最被垂涎的攻击目标之一
  → 需要极端严格的密钥保护措施
  → 证书吊销机制需要快速响应
```

## 八、CISP考试速查

### 关键考点

| 考点 | 记忆要点 |
|------|---------|
| Certificate Transparency | "三组件：Log + Monitor + Auditor" |
| 证书钉扎类型 | "证书/公钥/CA钉扎，别忘了backup pin" |
| PKI信任模型 | "层级→主流，网状→PGP，桥接→跨域" |
| ACME挑战方式 | "HTTP-01放文件，DNS-01加记录" |
| CAA记录 | "限制谁能签发我的域名证书" |
| 代码签名三作用 | "来源+完整性+时间戳" |
| 国密证书 | "SM2+SM3+SM4双证书过渡" |

### 常见陷阱

1. **"证书钉扎绝对安全"** → 配置错误可能导致服务不可用
2. **"有了CT就万事大吉"** → CT是事后检测，不能阻止证书签发
3. **"代码签名证书泄露没问题"** → 可以造成巨大影响（如Stuxnet）
4. **"CAA可以完全防止伪造证书"** → CAA依赖DNS安全，DNS可能被攻击
