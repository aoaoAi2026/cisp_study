## 一、对称加密架构

### 核心运维模型

```
对称加密 = 同一密钥加密和解密

    发送方                               接收方
    密钥 K    plaintext                  密钥 K
       │         │                         │
       ▼         ▼                         ▼
   [加密算法] → ciphertext → [不安全信道] → [解密算法]
                                           │
                                           ▼
                                       plaintext

速度：AES-256-GCM 可达 1-10 GB/s（硬件加速）
密钥分发：需要安全信道预先协商密钥
n个用户需 n(n-1)/2 个密钥用于两两通信
```

## 二、分组密码与流密码

| 特性 | 分组密码 (Block Cipher) | 流密码 (Stream Cipher) |
|------|------------------------|------------------------|
| 处理方式 | 固定大小块（AES 128位） | 逐位/逐字节 |
| 代表算法 | AES, SM4, DES | ChaCha20, RC4(已弃用) |
| 速度 | 快 | 极快 |
| 适合场景 | 文件/数据库加密 | 实时通信流加密 |
| 需要填充 | 是（如 AES-CBC） | 否 |
| 典型场景 | TLS 1.3 数据加密 | 移动端/低功耗设备 |

## 三、AES 深度剖析

```
AES (Advanced Encryption Standard)

起源：Rijndael 算法 → 2001年 NIST 标准化

结构：SPN (Substitution-Permutation Network)

参数：
├── 密钥长度：128 / 192 / 256 位
├── 块大小：固定 128 位 (16 字节)
├── 轮数：
│   ├── AES-128：10 轮
│   ├── AES-192：12 轮
│   └── AES-256：14 轮
└── 每轮四个步骤：
    1. SubBytes   (字节替换，S-Box)
    2. ShiftRows  (行移位)
    3. MixColumns (列混合)
    4. AddRoundKey(轮密钥加)

安全性：
AES-128 已经足够安全（2^128 密钥空间）
AES-256 提供额外安全余量（抗量子 + 未来证明）

硬件加速：
├── x86：AES-NI 指令集 → 10x+ 加速
└── ARM：ARMv8 Crypto Extensions
```

## 四、五种工作模式对比

| 模式 | 全称 | 安全性 | 并行 | 需要IV | 推荐 |
|------|------|--------|------|--------|------|
| ECB | 电子密码本 | 🔴 不安全 | ✅ | — | ❌ |
| CBC | 密码分组链接 | 🟡 一般 | ❌ | ✅ | ⚠️ 遗留 |
| CTR | 计数器模式 | 🟢 好 | ✅ | ✅ | ✅ |
| **GCM** | Galois/Counter | 🟢🟢 推荐 | ✅ | ✅ | ✅✅ |
| CFB | 密码反馈 | 🟡 | ❌ | ✅ | ⚠️ |

### ECB 的问题

```
ECB 模式 = 每块独立加密 → 相同明文块 = 相同密文块

原始图片经 AES-ECB 加密后：
→ 图片轮廓清晰可见！
→ 企鹅图片的轮廓在密文中依然可见

结论：永远不要用 ECB 模式！
```

### GCM — 推荐模式

```
GCM (Galois/Counter Mode) =

CTR 模式 (加密) + GMAC (认证标签)

= AEAD (Authenticated Encryption with Associated Data)
  ↓
  同时提供：
  ├── 机密性 (加密)
  ├── 完整性 (认证标签)
  └── 认证性 (数据源验证)

优势：
├── 一个模式解决两个需求 (无需额外 MAC)
├── 硬件加速支持好
└── TLS 1.3 的首选加密套件

TLS 1.3 推荐套件：
├── TLS_AES_128_GCM_SHA256
├── TLS_AES_256_GCM_SHA384
└── TLS_CHACHA20_POLY1305_SHA256 (移动端)
```

## 五、DES 到 3DES 的演进

```
DES (Data Encryption Standard)
├── 1977年发布
├── 密钥：56 位有效 (64位减8位校验)
├── 块大小：64 位 ← 过小
├── 暴力破解：1998年 EFF Deep Crack 56小时破解
└── 现状：已完全淘汰 ❌

3DES (Triple DES)
├── 三次 DES 加密/解密：E(K1)→D(K2)→E(K3)
├── 有效安全性：112 位
├── 性能：极慢（AES的1/3~1/6）
├── 块大小仍为64位 → Sweet32 生日攻击
└── 现状：正在淘汰 ⚠️

为什么 3DES = E-D-E 而非 E-E-E？
└── 兼容性：当 K1=K2=K3 时退化为单 DES
   单 DES 硬件可运行 3DES
```

## 六、国密 SM4 与流密码

### SM4

```
SM4 (国密对称加密标准)

├── 分组密码，128位块，128位密钥
├── 32轮非线性迭代
├── 与 AES-128 同级别安全
├── GB/T 32907-2016 国家标准
└── 应用于：金融IC卡、电子政务、VPN
```

### 流密码：ChaCha20

```
ChaCha20-Poly1305

ChaCha20 (流密码加密)
├── 256位密钥 + 96位 Nonce
├── 软件实现极快（不依赖AES-NI）
├── 移动端/ARM 平台的 AES 替代
└── Google 推荐用于 Android HTTPS

Poly1305 (认证标签)
└── 提供 AEAD (类似 GCM 的 GMAC)

→ TLS 1.3 的移动端首选
```

## 七、CISP 考试要点速查

| 概念 | 要点 |
|------|------|
| AES-128 | 10轮，128位块，当前对称加密标准 |
| ECB | 不安全！相同明文=相同密文 |
| GCM | 推荐模式：加密+认证 (AEAD) |
| DES | 56位密钥，已淘汰 |
| 3DES | 112位有效安全，正在淘汰 |
| SM4 | 国密分组密码，128位 |
| ChaCha20 | 流密码，移动端首选 |

### 工作模式速查

```
❌ ECB — 不安全，不要用
⚠️ CBC — 遗留系统可用，需小心 padding oracle
✅ CTR — 好，并行化
✅ GCM — 最佳推荐 (AEAD)
```

### 常见考试陷阱

1. ~~ECB 是安全的~~ → 相同明文 = 相同密文，信息泄露严重
2. ~~3DES 是现代标准~~ → 正在淘汰，64位块太小
3. ~~AES-256 比 AES-128 安全得多~~ → 都足够安全，只是安全余量不同
4. ~~GCM 只做加密~~ → GCM=加密+认证 (AEAD)

> 💡 **本章小结**：对称加密的核心知识点是 AES 各个工作模式的对比。ECB 要能识别为什么不安全，GCM 要理解 AEAD 的双重价值。DES 56位、3DES 的有效 112 位和 AES 128/256 位是考试中的数字考点。
