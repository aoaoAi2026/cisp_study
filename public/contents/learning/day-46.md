## 一、VPN基础概念

### 1.1 VPN分类

```
VPN技术全景：

  ┌─────────────────────────────────────────┐
  │              VPN技术分类                  │
  ├─────────────────────────────────────────┤
  │                                          │
  │  按OSI层：                               │
  │    · L2 VPN：PPTP, L2TP                 │
  │    · L3 VPN：IPsec, GRE                 │
  │    · L4/L7 VPN：SSL VPN, WireGuard       │
  │                                          │
  │  按部署模式：                             │
  │    · Site-to-Site：站点到站点（网关到网关） │
  │    · Remote Access：远程接入（客户端到网关）│
  │    · Client-to-Client：端到端             │
  │                                          │
  │  按实现方式：                             │
  │    · 硬件VPN网关                          │
  │    · 软件VPN客户端                        │
  │    · 云VPN服务                            │
  └─────────────────────────────────────────┘
```

### 1.2 VPN核心安全目标

```
VPN必须保证：
  ✅ 机密性 — 数据加密（DES→3DES→AES→ChaCha20）
  ✅ 完整性 — 防篡改（HMAC、AEAD）
  ✅ 认证 — 身份验证（Pre-shared Key、证书、EAP）
  ✅ 抗重放 — 序列号/时间戳防重放
  ✅ 密钥管理 — 安全密钥交换和更新
```

## 二、IPsec VPN深度解析

### 2.1 协议栈回顾

```
IPsec协议栈：

    ┌─────────────┐
    │  IKEv2      │ ← 密钥协商（UDP 500/4500）
    ├─────────────┤
    │  ESP (50)   │ ← 加密+认证（主要使用）
    │  AH  (51)   │ ← 仅认证（几乎不用）
    ├─────────────┤
    │  IP          │
    └─────────────┘

IKEv2流程（4条消息）：
  IKE_SA_INIT    (2条) → DH密钥交换 + 协商算法
  IKE_AUTH       (2条) → 证书认证 + 建立子SA
```

### 2.2 传输模式 vs 隧道模式

```
传输模式 (Transport Mode)：

  原IP头 │ TCP/UDP │ 数据
        ↓
  原IP头 │ ESP头 │ TCP/UDP │ 数据 │ ESP尾 │ ESP认证
  ←──── 加密范围 ────→
  ←────────── 认证范围 ──────────→

  适用：主机到主机的端到端通信


隧道模式 (Tunnel Mode)：

  原IP头 │ TCP/UDP │ 数据
        ↓
  新IP头 │ ESP头 │ 原IP头 │ TCP/UDP │ 数据 │ ESP尾 │ ESP认证
  ←───────────── 加密范围 ─────────────→
  ←──────────────── 认证范围 ────────────────→

  适用：Site-to-Site VPN，整个原始包被封装保护
```

### 2.3 IPsec安全关联(SA)

```
SA (Security Association) — IPsec核心概念：

SA是一个单向安全连接定义，包含：
  · SPI (Security Parameter Index) — 32位标识符
  · 加密算法和密钥
  · 认证算法和密钥
  · SA生存期（时间/字节数）

每个IPsec连接需要2个SA（入方向+出方向）

IKE SA (ISAKMP SA)：
  · 保护IKE通信本身
  · 双向，控制通道

IPsec SA (Child SA)：
  · 保护实际数据流量
  · 单向，数据通道
  · 一个IKE SA可以创建多个IPsec SA
```

## 三、SSL/TLS VPN

### 3.1 SSL VPN vs IPsec VPN

| 维度 | IPsec VPN | SSL VPN |
|------|-----------|---------|
| 客户端 | 需要安装VPN客户端 | 浏览器即可（无客户端） |
| 部署 | 复杂（需要证书或PSK） | 相对简单 |
| 防火墙友好 | ❌ (ESP协议可能被阻) | ✅ (走443端口) |
| 适用场景 | Site-to-Site, 全网络访问 | 远程访问特定应用 |
| 粒度 | 网络层（IP级别） | 应用层（URL/端口级别） |
| 性能 | 高效（硬件加速） | 略低（TLS开销） |
| NAT穿越 | 需要NAT-T | 天然支持 |

### 3.2 SSL VPN部署模式

```
① 无客户端模式 (Clientless)：
  用户 → 浏览器 → HTTPS → SSL VPN网关 → 内部Web应用
  仅支持Web应用

② 瘦客户端模式 (Thin Client)：
  用户 → 浏览器下载Java/ActiveX → 端口转发
  支持TCP应用

③ 隧道客户端模式 (Full Tunnel)：
  用户 → SSL VPN客户端 → 虚拟网卡 → 全网络访问
  功能类似传统IPsec VPN但基于TLS

④ 反向代理模式：
  外部用户 → SSL VPN (反向代理) → 内部应用
  浏览器到应用，透明转发
```

### 3.3 主流SSL VPN方案

| 方案 | 类型 | 特点 |
|------|------|------|
| OpenVPN | 开源 | 支持TCP/UDP，TLS+自定义协议 |
| OpenConnect | 开源 | Cisco AnyConnect兼容客户端 |
| SoftEther | 开源 | 多协议支持（L2TP/IPsec/OpenVPN/SSTP） |
| WireGuard | 开源内核 | 极简设计，Linux内核主线 |
| 深信服SSL VPN | 商业 | 国内主流，国密支持 |

## 四、WireGuard：新一代VPN

### 4.1 WireGuard设计哲学

```
WireGuard vs IPsec/OpenVPN：

代码量对比：
  IPsec (StrongSwan): ~400,000 行
  OpenVPN:            ~100,000 行
  WireGuard:           ~4,000 行

设计特点：
  ✅ 极简内核：代码量极小，易于审计
  ✅ 无状态：不维护连接状态（"无连接"理念）
  ✅ 现代密码原语：ChaCha20-Poly1305, Curve25519, BLAKE2s, SipHash24
  ✅ 密钥路由：公钥=身份，简单配置
  ✅ 内核态：作为Linux内核模块运行（高性能）
  ✅ 漫游支持：IP变化无缝切换

WireGuard接口：
  wg0: 虚拟网络接口
  每个Peer通过公钥识别
  允许的IP列表定义路由
```

### 4.2 WireGuard密码原语

```
WireGuard密码原语选择：

┌──────────────────┬─────────────────────────┐
│   用途            │       算法              │
├──────────────────┼─────────────────────────┤
│ 密钥交换          │ Curve25519 (ECDH)       │
│ 对称加密          │ ChaCha20 (256位)        │
│ 认证              │ Poly1305                │
│ 哈希/HKDF         │ BLAKE2s                │
│ 哈希表键          │ SipHash24               │
│ 身份              │ Curve25519 公钥(32字节) │
└──────────────────┴─────────────────────────┘

为什么不用AES-GCM？
  → ChaCha20-Poly1305在无硬件AES加速的设备上更快
  → 恒定时间实现更容易
  → 更适合移动/嵌入式场景
```

### 5.1 协议大比拼

```
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│   特性    │  IPsec   │ OpenVPN  │ WireGuard│  L2TP/IPsec│
├──────────┼──────────┼──────────┼──────────┼──────────┤
│ OSI层     │    3     │   3/7    │    3     │    2     │
│ 加密算法   │ 可配置    │ 可配置    │ 固定现代  │ 依赖IPsec │
│ 前向保密   │ ✅ PFS   │ ✅ PFS   │ ✅ 内置   │ ✅ PFS   │
│ NAT穿透    │ NAT-T    │ ✅ 天然   │ ❌ 困难   │ ✅ 天然  │
│ 移动漫游   │ IKEv2 MOBIKE│ ✅    │ ✅ 无缝   │ ❌       │
│ 配置复杂度 │ 高        │ 中        │ 极低      │ 中       │
│ 代码审计   │ 难(40万行) │ 较难(10万行)│ 极简(4千行)│ 较难   │
│ TCP支持    │ ❌        │ ✅       │ ❌        │ ❌       │
│ 内核支持   │ ✅        │ ❌ 用户态  │ ✅ 内核态 │ ✅(l2tp)+✅│
│ 适用场景   │ Site-to-Site│ 远程接入  │ 现代化替代 │ 兼容旧设备 │
└──────────┴──────────┴──────────┴──────────┴──────────┘
```

### 5.2 PPTP：一个不该再用的协议

```
PPTP (Point-to-Point Tunneling Protocol)：
  · 1999年发布
  · 使用 MPPE (Microsoft Point-to-Point Encryption)
  · 基于 MS-CHAPv2 认证
  · 使用 RC4 加密（40-128位）

安全问题：
  ❌ MS-CHAPv2 可被破解（2012年100%成功）
  ❌ RC4 流密码已知弱点
  ❌ 缺乏完美前向保密
  ❌ 2012年起 NSA 可以解密

结论：永远不要使用PPTP！
```

## 六、VPN安全风险与加固

### 6.1 常见VPN风险

```
① VPN账户泄露：
   → 弱密码/重用密码 → 攻击者合法接入内网
   → 防御：MFA强制（TOTP/证书/生物识别）

② VPN设备漏洞：
   → CVE漏洞未修补 (Pulse Secure, Fortinet, Citrix多次)
   → 防御：及时更新、关注安全公告

③ 分流隧道 (Split Tunneling)：
   → 部分流量走VPN，部分直连互联网
   → 风险：直连流量不受保护，可能引入恶意软件
   → 防御：强制全隧道或严格控制分流策略

④ VPN日志缺失：
   → 无法追溯谁、何时、做了什么
   → 防御：日志集中到SIEM

⑤ 弱加密配置：
   → 使用DES/3DES/MD5/SHA-1
   → 防御：强制AES-256/ChaCha20，禁用弱算法
```

### 6.2 VPN加固检查清单

```
✅ 强制多因素认证
✅ 加密套件白名单（仅AES-256-GCM/ChaCha20）
✅ 证书认证优先于PSK
✅ 部署设备健康检查（接入前终端检查）
✅ 最小权限：仅开通必要的网络资源
✅ 会话超时设置
✅ VPN设备定期安全评估
✅ VPN日志接入SIEM
✅ 禁用PPTP、L2TP（使用IPsec/SSL-VPN/WireGuard替代）
✅ 部署客户端自动更新机制
```

## 七、零信任与VPN的关系

### 7.1 VPN的局限性

```
传统VPN的"过度信任"问题：

VPN连接建立后：
  ✓ 用户被认证
  → 用户获得了"整个网络"的访问权限
  → 即使用户只需要访问一个Web应用

问题：
  · 一旦VPN被攻破 → 攻击者横跨整个内网
  · 无法区分正常用户和被劫持的用户
  · 一次认证，永久信任

零信任方案 (ZTNA)：
  · 每次请求都认证 + 授权
  · 不基于网络位置，基于身份
  · 最小权限原则（仅授予需要的资源）
  · 代理访问，不暴露内网
```

### 7.2 VPN vs ZTNA

| 维度 | 传统VPN | ZTNA |
|------|---------|------|
| 信任模型 | 网络边界信任 | 永不信任，始终验证 |
| 访问粒度 | 网络级（IP/子网） | 应用级（具体应用） |
| 用户可见 | 整个网络 | 仅授权应用 |
| 横向移动 | 容易 | 极其困难 |
| 部署 | 网关/VPN客户端 | Cloud/代理网关 |

## 八、国密VPN方案

```
国密IPsec VPN：

算法替换：
  国际标准            国密标准
  ─────────          ─────────
  DH/ECDH      →     SM2密钥交换
  RSA/ECDSA    →     SM2数字签名
  AES          →     SM4对称加密
  SHA-256      →     SM3哈希

认证方式：
  国际：RSA/ECDSA证书
  国密：SM2双证书（签名证书+加密证书）

部署注意事项：
  · 需要支持国密算法的VPN硬件/软件
  · 双证书管理复杂度增加
  · 速度可能低于国际算法（优化不足）
```

## 九、CISP考试速查

### 关键考点

| 考点 | 记忆要点 |
|------|---------|
| IPsec两协议 | "ESP加密+认证，AH仅认证" |
| 传输vs隧道 | "传输：端到端，隧道：网关到网关" |
| SSL VPN优势 | "无客户端 + 防火墙友好(443) + 细粒度" |
| WireGuard特点 | "4千行代码 + 现代密码 + 内核态" |
| PPTP安全 | "MS-CHAPv2可破解 → 永远不要用" |
| SA含义 | "单向安全连接，SPI标识" |

### 常见混淆

1. **"IPsec自动具有前向保密"** → 只有使用IKEv2+DH/ECDH密钥交换才有PFS
2. **"SSL VPN就是OpenVPN"** → OpenVPN只是SSL VPN的一种实现
3. **"VPN加密了就是安全的"** → 还需要认证（防MITM）+ 完整性
4. **"WireGuard因为是新的所以不如传统VPN安全"** → 极简代码反而更容易审计
