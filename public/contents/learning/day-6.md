## 一、密码学的核心目标

### 1.1 CIA + N

密码学服务于信息安全的五个核心目标：

```
      密码学五大安全目标
┌────────────────────────────────┐
│                               │
│   🔒 机密性 (Confidentiality)  │ → 加密算法
│   ✓  完整性 (Integrity)        │ → 哈希函数
│   🖊️ 认证性 (Authentication)    │ → 消息认证码
│   🚫 不可否认性 (Non-repudiation)│ → 数字签名
│   🔑 密钥管理 (Key Management)  │ → PKI/密钥交换
│                               │
└────────────────────────────────┘
```

**关键对应关系（CISP 必考）**：
- 加密 → 机密性
- 哈希 → 完整性
- 数字签名 → 完整性 + 认证性 + 不可否认性
- 消息认证码 (MAC) → 完整性 + 认证性

### 1.2 密码学简史

```
古典密码时代 (公元前 → 19世纪)
├── 凯撒密码：字母移位 (shift cipher)
├── 维吉尼亚密码：多表替换
└── Enigma：二战德国电机密码机

现代密码学 (1949 → 至今)
├── 1949: Shannon《保密系统的通信理论》→ 密码学成为科学
├── 1976: Diffie-Hellman 公钥交换 → 密码学革命
├── 1977: RSA 算法 → 第一个实用公钥体系
├── 1977: DES 标准 → 对称加密标准化
├── 2001: AES 标准 → 取代 DES
└── 2013: 斯诺登事件 → 密码学去后门化运动
```

## 二、密码学术语速查表

| 术语 | English | 含义 |
|------|---------|------|
| 明文 | Plaintext | 原始的、可读的消息 |
| 密文 | Ciphertext | 加密后的、不可读的消息 |
| 加密 | Encryption | Plaintext → Ciphertext |
| 解密 | Decryption | Ciphertext → Plaintext |
| 密钥 | Key | 控制加解密过程的参数 |
| 对称密钥 | Symmetric Key | 加解密使用相同密钥 |
| 公钥 | Public Key | 可公开分发的密钥（非对称） |
| 私钥 | Private Key | 必须保密的密钥（非对称） |
| 密码分析 | Cryptanalysis | 不通过密钥破解密码 |
| 暴力破解 | Brute Force | 尝试所有可能密钥 |
| 密钥空间 | Key Space | 所有可能密钥的集合 |
| 混淆 | Confusion | 密文与密钥的关系必须复杂 |
| 扩散 | Diffusion | 明文的微小变化应扩散到整个密文 |

**Kerckhoffs 原则**：密码系统的安全性应**仅依赖于密钥的保密**，而不依赖于算法的保密。也就是说，算法可以公开，只要密钥保密即可。

## 三、对称加密：一把钥匙的世界

### 3.1 原理模型

```
        对称加密模型

    明文 M        密钥 K        密文 C
  ┌────────┐    ┌────────┐    ┌────────┐
  │ HELLO  │ → │  AES   │ →  │ A3F9B1 │
  └────────┘    └────────┘    └────────┘
                    │
                    ↓  同样密钥 K
               ┌────────┐
               │  AES   │
               └────────┘
                    │
                    ↓
               ┌────────┐
               │ HELLO  │
               └────────┘
```

**特点**：
- ⚡ 速度快（适合大量数据加密）
- 🔑 加解密使用同一密钥
- ⚠️ 密钥分发是一大难题
- 🔢 n 个用户需要 n(n-1)/2 个密钥进行两两通信

### 3.2 两大流派

#### 流密码 (Stream Cipher)

```
机制：逐位/逐字节加密
密钥流 ← 密钥 → 伪随机发生器
明文  ⊕ 密钥流 = 密文
密文  ⊕ 密钥流 = 明文  (XOR的可逆性)

典型算法：RC4 (已弃用)、ChaCha20
适用场景：实时通信、流媒体
```

**核心要求**：密钥流必须只使用一次（One-Time Pad 原理）。同一密钥流重复使用 = 灾难。

#### 分组密码 (Block Cipher)

```
机制：将明文分成固定大小的块（如 128 位），逐块加密

工作模式 (Modes of Operation)：
├── ECB (电子密码本) 🔴 不安全！
│   相同明文 → 相同密文，模式可见
├── CBC (密码分组链接) 🟡 常用但需 IV
│   前一块密文与当前明文的 XOR 参与加密
├── CTR (计数器模式) 🟢 推荐
│   将分组密码变为流密码模式
└── GCM (Galois/Counter) 🟢 推荐
    CTR + 认证 = AEAD (认证加密)
```

### 3.3 核心算法对比

| 算法 | 密钥长度 | 块大小 | 安全性 | 现状 |
|------|----------|--------|--------|------|
| DES | 56 位 | 64 位 | ❌ 已破解 | 历史遗迹 |
| 3DES | 168 位 (有效 112) | 64 位 | ⚠️ 弱 | 逐步淘汰 |
| **AES** | **128/192/256** | **128 位** | ✅ **安全** | **当前标准** |
| SM4 | 128 位 | 128 位 | ✅ 安全 | 中国国密 |
| Blowfish | 32-448 位 | 64 位 | ⚠️ 块太小 | 不推荐新系统 |
| ChaCha20 | 256 位 | （流密码） | ✅ 安全 | 移动端首选 |

### 3.4 AES 深度剖析

```
AES 结构：Substitution-Permutation Network (SPN)

轮数：
├── AES-128: 10 轮
├── AES-192: 12 轮
└── AES-256: 14 轮

每轮四个步骤 ─── 用于混淆和扩散：
1. SubBytes    → 替换（S-Box 查表）
2. ShiftRows   → 行移位
3. MixColumns  → 列混合
4. AddRoundKey → 与轮密钥 XOR

最后一轮省略 MixColumns
```

**暴力破解的时间估算**：
```
AES-128 密钥空间: 2^128 ≈ 3.4 × 10^38
假设每秒尝试 10^12 次（超级计算机）：
需要 ≈ 10^19 年  (宇宙年龄约 10^10 年) → 实际上不可行
```

### 3.5 XOR 加密演示（教学用，非安全）

对称加密的核心数学工具是 **XOR**（异或）运算：

- `A ⊕ B ⊕ B = A`（自反性：用同一密钥 XOR 两次恢复明文）
- 这解释了为什么对称加密中加密和解密使用同一密钥

## 四、非对称加密：公钥的革命

### 4.1 为什么要非对称加密

对称加密的**密钥分发问题**：

> Alice 和 Bob 从未见过面。Alice 需要给 Bob 发送一条加密消息。如果她用对称加密，她怎么安全地把密钥交给 Bob？

这就是 Diffie-Hellman 在 1976 年要解决的问题——也是公钥密码学的诞生。

### 4.2 原理模型

```
      非对称加密模型

        Bob 的公钥         Bob 的私钥
       (公开发布)         (Bob 保密)
           │                  │
           ↓                  ↓
明文 M → [加密] → 密文 C → [解密] → 明文 M
       (RSA/ECC)         (RSA/ECC)

任何人都可以用公开的公钥加密 → 只有 Bob 能用私钥解密
```

**数学基础**：
- RSA：大整数分解的困难性
- ECC (椭圆曲线)：椭圆曲线离散对数问题
- DSA：离散对数问题

### 4.3 核心算法对比

| 算法 | 原理 | 密钥长度(等效安全) | 性能 | 典型用途 |
|------|------|-------------------|------|----------|
| **RSA** | 大整数分解 | 2048/3072/4096 | 慢 | 密钥交换、签名 |
| **ECC** | 椭圆曲线 | 256/384/521 | 快 | 移动端、TLS |
| **ECDSA** | ECC+签名 | 256 | 快 | 比特币、TLS |
| **Ed25519** | Edwards曲线 | 256 | 极快 | 现代签名标准 |
| SM2 | 中国国密ECC | 256 | 中 | 国内合规 |

**安全强度等效表（CISP 常考）**：

```
RSA-1024  ≈ 80位 安全 (⚠️ 已不够)
RSA-2048  ≈ 112位安全 (✅ 当前标准)
RSA-3072  ≈ 128位安全
ECC-256   ≈ 128位安全 (与 RSA-3072 等效!)
ECC-384   ≈ 192位安全
AES-128   ≈ 128位安全
```

**关键认知**：ECC 用更短的密钥实现同等级别的安全。256位的 ECC 密钥 ≈ 3072位的 RSA 密钥。

> **⚠️ 量子计算威胁**：Shor 算法可以在多项式时间内分解大整数和求解离散对数。一旦大规模量子计算机出现，RSA 和 ECC 都将被破解。后量子密码学 (PQC) 正在标准化中。

### 4.4 公钥与私钥的角色（CISP 必考重点）

| 场景 | 使用哪个密钥 | 为什么 |
|------|-------------|--------|
| **发送机密消息给 Bob** | Bob 的**公钥** | 只用 Bob 的私钥能解密 |
| **Bob 解密收到的消息** | Bob 的**私钥** | 私钥才能解密公钥加密的内容 |
| **Bob 进行数字签名** | Bob 的**私钥** | 签名 = 用私钥加密哈希 |
| **验证 Bob 的签名** | Bob 的**公钥** | 验证 = 用公钥解密签名 |
| **密钥交换** | 对方的公钥 + 我的私钥 | DH/ECDH 协议 |

> **记忆口诀**：加密用**对方的公钥**，解密用**自己的私钥**；签名用**自己的私钥**，验证用**对方的公钥**。

## 五、哈希函数：数字世界的指纹

### 5.1 核心特性

```
哈希函数的五大特性：

1. ✅ 确定性 (Deterministic)
   同一输入永远产生同一输出

2. ✅ 快速计算 (Efficiency)
   对任意长度的输入，快速生成固定长度输出

3. ❌ 抗原像/单向性 (Pre-image Resistance)
   给定 Hash(M)，找不回原始 M
   "不可逆"

4. ❌ 抗第二原像 (Second Pre-image Resistance)
   给定 M，找不到 M' ≠ M 且 Hash(M') = Hash(M)
   "很难伪造相同哈希的另一个输入"

5. ❌ 抗碰撞 (Collision Resistance)
   很难找到任意两个不同输入 M, M' 使 Hash(M) = Hash(M')
   "任意两段数据很难哈希相同"
```

**特性 3-5 的区别（考点）**：

| 特性 | 攻击者要做什么 | 难度 |
|------|---------------|------|
| 抗原像 | 给定 Hash，找 M | 2^n |
| 抗第二原像 | 给定 M，找 M' (Hash相同) | 2^n |
| 抗碰撞 | 找任意 M, M' (Hash相同) | 2^(n/2) ← 生日攻击 |

> **生日悖论**：在一个 23 人的房间中，两个人生日相同的概率 > 50%。同理，在 2^(n/2) 次尝试后，找到哈希碰撞的概率 > 50%。这就是为什么 SHA-256 理论安全强度是 128 位（而非 256 位）。

### 5.2 核心算法对比

| 算法 | 输出长度 | 安全性 | 现状 |
|------|----------|--------|------|
| MD5 | 128 位 | ❌ 已破解 | **禁止使用** |
| SHA-1 | 160 位 | ❌ 已破解 | **禁止使用** |
| **SHA-256** | **256 位** | ✅ **安全** | **当前标准** |
| SHA-384 | 384 位 | ✅ 安全 | 高安全场景 |
| SHA-512 | 512 位 | ✅ 安全 | 高安全场景 |
| **SHA-3** | 224/256/384/512 | ✅ 安全 | 新一代标准 |
| BLAKE3 | 任意长度 | ✅ 安全 | 极速替代品 |
| SM3 | 256 位 | ✅ 安全 | 中国国密 |

**⚠️ SHA-1 的"临终"时间线**：
- 2017 年 2 月：Google 发布首个 SHA-1 碰撞 (SHAttered)
- 2020 年：Chosen-prefix 碰撞只需要 $45K 的计算资源
- 结论：SHA-1 任何新系统都不能用

### 5.3 哈希的实际应用

```
哈希的六大应用场景：

1. 📁 文件完整性校验
   MD5/SHA256 常用于下载文件校验
   
2. 🔐 密码存储 (Password Hashing)
   存储 Hash(password + salt)，不存储明文密码
   专用算法：bcrypt, scrypt, Argon2id (比SHA更好，更慢)
   
3. ✍️ 数字签名
   Sign(Hash(Message))，而非 Sign(Message)
   原因：签名算法慢，先哈希再签名效率更高
   
4. 🔗 区块链
   每个区块包含前一区块的哈希 → 链式结构
   工作量证明：寻找 nonce 使 Hash(区块) < 目标值
   
5. 📊 数据去重
   相同内容 → 相同哈希 → 检测重复
   
6. 🔬 数字取证
   哈希用于"证据固定"：证明数据未被篡改
```

### 5.4 密码存储最佳实践

| 方案 | 安全性 | 建议 |
|------|--------|------|
| `Hash(password)` | ❌ 极不安全 | 彩虹表直接破解 |
| `Hash(password + salt)` | ⚠️ 不够 | SHA 太快，GPU 可暴力 |
| `bcrypt(password, cost)` | ✅ 较好 | cost=12+ 增加计算成本 |
| `scrypt(password, salt, N)` | ✅ 好 | 内存困难，抗 GPU |
| **`Argon2id`** | ✅✅ **最佳** | 2015 年 PHC 冠军 |

## 六、混合加密：取长补短的智慧

### 6.1 为什么需要混合加密

| 加密类型 | 优势 | 劣势 |
|----------|------|------|
| 对称加密 | 速度快，适合大量数据 | 密钥分发难 |
| 非对称加密 | 密钥分发方便 | 速度慢 100-1000 倍 |

**解决方案**：把两者的优势结合起来！

### 6.2 TLS/HTTPS 的混合加密过程

```
TLS 1.3 握手简化流程：

客户端                              服务器
  │                                   │
  │──① ClientHello ────────────────→│
  │  (支持的密码套件、密钥共享)        │
  │                                   │
  │←─② ServerHello ─────────────────│
  │  (选定的密码套件、密钥共享、证书)   │
  │                                   │
  │③ 验证服务器证书（CA链）            │
  │④ 双方计算会话密钥（ECDHE）         │
  │                                   │
  │═══⑤ 对称加密通信开始 ════════════│
  │  (AES-GCM / ChaCha20-Poly1305)   │
  │                                   │
  
  关键步骤：
  - 证书验证使用 RSA/ECDSA (非对称)
  - 密钥交换使用 ECDHE (非对称)
  - 数据传输使用 AES/ChaCha20 (对称)
```

**一句话总结**：非对称加密负责"安全地商量出对称密钥"，对称加密负责"高效地加密所有数据"。

### 6.3 数字信封 (Digital Envelope)

```
发送方：
1. 生成随机会话密钥 K_session
2. 用 K_session (对称) 加密消息 → 密文
3. 用接收方公钥 (非对称) 加密 K_session → 数字信封
4. 发送：密文 + 数字信封

接收方：
1. 用自己私钥解开数字信封 → 得到 K_session
2. 用 K_session 解密密文 → 得到明文
```

这正是 PGP、S/MIME 等邮件加密标准的核心理念。

## 七、密钥管理的挑战

### 7.1 密钥生命周期

```
密钥管理生命周期：
生成 → 分发 → 存储 → 使用 → 轮换 → 归档 → 销毁

每个阶段的威胁：
├── 生成：弱随机数 → 可预测密钥 (CVE-2008-0166: Debian SSH 漏洞)
├── 分发：中间人攻击 → 密钥被替换
├── 存储：明文存储 → 服务器被入侵后密钥泄露
├── 使用：侧信道攻击 → 时序分析泄露密钥
├── 轮换：不轮换 → 旧密钥可解密历史数据
├── 归档：密钥丢失 → 历史数据永久无法解密
└── 销毁：不彻底销毁 → 密钥被恢复
```

### 7.2 密钥管理的黄金法则

1. **从不硬编码密钥**：代码中的密钥 = 公开的秘密
2. **职责分离**：管理密钥的人 ≠ 使用密钥的人
3. **双人控制**：主密钥需要 2+ 人协作才能解锁 (分片)
4. **最小暴露**：密钥只在需要时的最小范围内存在
5. **定期轮换**：即使没泄露，也要定期更换密钥
6. **用 HSM/KMS**：硬件安全模块/云密钥管理服务

## 八、CISP 考试要点速查

### 核心概念对照

| 概念 | 要点 | 考试出现形式 |
|------|------|-------------|
| AES | 对称加密标准，128/192/256 位 | 选哪个是对称加密 |
| RSA | 非对称，基于大整数分解 | 数字签名用哪个密钥 |
| ECC | 非对称，短密钥高安全性 | 与 RSA 的安全强度对比 |
| SHA-256 | 哈希，256位输出 | 完整性校验用什么 |
| 数字签名 | 私钥签名，公钥验证 | 不可否认性靠什么 |
| 混合加密 | 非对称交换密钥 + 对称加密数据 | HTTPS 的工作原理 |
| Kerckhoffs | 安全性在密钥，不在算法保密 | 密码学基本原则 |
| 生日攻击 | 碰撞复杂度约 2^(n/2) | 哈希算法安全强度计算 |

### 常见考试陷阱

1. ~~数字签名用公钥进行签名~~ → 用**私钥**签名，**公钥**验证
2. ~~MD5 仍可用于完整性校验~~ → MD5 已破解，存在碰撞攻击
3. ~~对称加密比非对称加密更安全~~ → 安全性取决于密钥长度和算法，对称加密的劣势是密钥分发
4. ~~AES-128 比 AES-256 差在只有一半的安全性~~ → 实际上 AES-128 也是安全的，128位已足够抗暴力破解
5. ~~加密可以保证完整性~~ → 加密只保**机密性**，完整性需要**哈希/MAC**
