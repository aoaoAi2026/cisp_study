## 一、数字签名原理

```
数字签名 = 用私钥加密消息的哈希

为什么这样设计？

1. 先哈希再签名（不是直接签名消息）
   ├── 签名算法（RSA/ECDSA）很慢
   ├── 消息可能很大（GB级文件）
   └── → 先哈希 → 得到固定长度的摘要 → 签名摘要

2. 签名 = 用签名者的私钥加密哈希值
   Sign(M) = Encrypt(Hash(M), PrivateKey)

   验证 = 用签名者的公钥解密签名 → 得到哈希
         = 重新计算 Hash(M)
         → 两个哈希是否相同？
```

### 数字签名的三个属性

```
数字签名提供的安全属性：

1. 完整性 (Integrity)
   签名的文档被篡改 → 哈希改变 → 验证失败

2. 认证性 (Authentication)
   签名用私钥生成 → 只有私钥持有者能产生有效签名
   验证通过 = 确认签名者的身份

3. 不可否认性 (Non-repudiation)
   签名者不能否认自己签过名
   → 因为签名需要签名者的私钥（只有他有）
   → 法律证据

数字签名 vs 纸质签名：
├── 纸质签名：签名每次都相同 → 容易被模仿
└── 数字签名：签名 = 对消息哈希的私钥加密
    → 不同消息的签名不同！无法从一个签名伪造另一个
```

## 二、签名流程与验证

```
数字签名的完整流程：

签名方 (Alice)：                     验证方 (Bob)：
                                     
文档 M                               
  │                                  
  ▼                                  
Hash(M) → 256位摘要                  
  │                                  
  ▼                                  
用 Alice私钥 加密摘要                 
  │                                  
  ▼                                  
数字签名 S                          收到 M + S
  │                                    │
  ├─────────────────────────────→     │
  │ M + S                             │
  │                                    ▼
  │                              验证过程：
  │                              1. 用 Alice公钥 解密 S → 得到 摘要1
  │                              2. 计算 Hash(M) → 得到 摘要2
  │                              3. 比对 摘要1 == 摘要2？
  │                                 │          │
  │                               相同≠       不同≠
  │                                 │          │
  │                               ✅通过       ❌拒绝
  │                              (完整性OK)  (被篡改/假签名)
```

## 三、PKI 体系架构

### PKI 组成部分

```
PKI (Public Key Infrastructure) =

一个由策略、程序、硬件、软件和人员组成的体系，
用于创建、管理、分发、使用、存储和撤销数字证书。

PKI 的五个核心组件：

1. CA (Certificate Authority) 证书认证中心
   ├── 签发证书
   ├── 撤销证书 (CRL/OCSP)
   └── 信任的起点 (Root CA)

2. RA (Registration Authority) 注册机构
   ├── 验证证书申请者的身份
   └── 不签发证书，只做身份核实

3. 证书库 (Certificate Repository)
   ├── 存储已签发的证书
   └── LDAP/HTTP 公开可查

4. 证书撤销 (CRL/OCSP)
   ├── CRL：证书撤销列表 (定期下载)
   └── OCSP：在线证书状态协议 (实时查询)

5. 密钥管理
   ├── 密钥生成（安全环境下）
   ├── 密钥备份和恢复（CA 备份私钥？）
   └── 密钥销毁
```

### PKI 层次结构

```
        Root CA (根CA)
        (自签名证书，离线存储)
              │
              │
    ┌─────────┼─────────┐
    │         │         │
 Sub CA 1  Sub CA 2  Sub CA 3
 (签发用户证书)
    │
    │
 终端实体证书
 (服务器/个人)

信任传递：
├── 浏览器信任 Root CA（根证书预装）
├── Root CA 信任 Sub CA（签发 Sub CA 的证书）
├── Sub CA 签发终端证书
└── → 信任链：Root → Sub → End-Entity
```

## 四、数字证书详解

### X.509 证书结构

```
X.509 v3 证书格式：

版本号 (Version)：v3
序列号 (Serial Number)：CA 分配的唯一编号
签名算法 (Signature Algorithm)：Sha256WithRSAEncryption
签发者 (Issuer)：CN=DigiCert Global Root CA
有效期 (Validity)：
├── 生效时间 (Not Before)
└── 过期时间 (Not After)
主题 (Subject)：CN=www.example.com
主题公钥 (Subject Public Key)：
├── 算法：RSA (2048 bits)
└── 公钥：----BEGIN PUBLIC KEY----...
扩展 (Extensions)：
├── SAN (Subject Alternative Name)：DNS 备用名列表
├── Key Usage：数字签名、密钥加密
├── Extended Key Usage：服务器认证、客户端认证
├── Basic Constraints：CA/非CA
└── CRL Distribution Point：CRL 下载地址

签名 (Signature)：
└── CA 用其私钥对该证书的签名
```

### 证书类型

| 类型 | 验证级别 | 适用场景 |
|------|----------|----------|
| DV (域验证) | 仅验证域名控制权 | 个人网站、博客 |
| OV (组织验证) | 验证域名 + 组织真实存在 | 企业网站 |
| EV (扩展验证) | 最严格验证 | 银行/金融/电商(地址栏变绿) |
| 代码签名 | 验证软件发布者 | 软件分发 |
| 客户端证书 | 验证用户身份 | VPN/内部系统认证 |

## 五、PKI 信任模型

### 三种信任模型

```
1. 层次信任模型 (Hierarchical Trust Model)
   ├── 树状结构：Root CA → Sub CA → End-Entity
   ├── 优点：结构清晰，证书链短
   ├── 缺点：Root CA 是单点信任
   └── Web PKI 的主要模型

2. 网状信任模型 (Mesh/Web of Trust)
   ├── 参与者互相签发证书
   ├── 优点：无单点信任
   ├── 缺点：扩展性差，信任度难以量化
   └── PGP 使用的模型

3. 桥接信任模型 (Bridge Trust Model)
   ├── 通过桥CA连接多个信任域
   ├── 优点：跨组织/跨国互信
   └── 政府/军事跨域互操作
```

### 证书撤销机制

```
CRL (Certificate Revocation List)
├── CA 定期发布的被撤销证书列表
├── 客户端下载后检查
├── 延迟：可能几小时/几天
└── 体积：可能很大

OCSP (Online Certificate Status Protocol)
├── 实时查询单个证书状态
├── 响应：Good / Revoked / Unknown
├── 问题：OCSP 响应可能被拦截
└── OCSP Stapling：服务器主动获取OCSP响应并附在TLS握手中
                    → 隐私 + 速度

现代方案 → CRLite / Merkle Town
```

## 六、CISP 考试要点速查

### 数字签名核心

| 操作 | 密钥 | 原因 |
|------|------|------|
| 签名 | 签名者的**私钥** | 证明身份 |
| 验签 | 签名者的**公钥** | 验证来源 |

### PKI 核心

| 组件 | 职责 |
|------|------|
| CA | 签发和管理证书 |
| RA | 验证身份 |
| CRL | 被撤销证书列表 |
| OCSP | 实时查询证书状态 |

### 常见考试陷阱

1. ~~数字签名用公钥签名~~ → 用**私钥**签名，**公钥**验证
2. ~~PKI=CA~~ → CA 是 PKI 的核心但PKI还包括RA/CRL/OCSP/策略
3. ~~证书过期了还能用~~ → 过期后不可信，需及时续签
4. ~~自签名证书和CA签发的安全性相同~~ → 自签名无第三方信任
