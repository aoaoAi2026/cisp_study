## 一、AAA 框架全景

### 1.1 AAA 是什么

```
访问控制的核心框架 AAA：

┌──────────────────────────────────────────┐
│                                          │
│   Authentication (认证) → "你是谁？"       │
│        ↓                                 │
│   Authorization  (授权) → "你能做什么？"    │
│        ↓                                 │
│   Accounting     (审计) → "你做了什么？"    │
│                                          │
└──────────────────────────────────────────┘

三者缺一不可：
├── 没有认证 → 任何人都能声称是任何人
├── 没有授权 → 认证通过后什么都能做（权限过大）
└── 没有审计 → 出了事不知道谁干的、怎么干的
```

### 1.2 认证 vs 授权 vs 审计

| 维度 | 认证 (Authentication) | 授权 (Authorization) | 审计 (Accounting) |
|------|----------------------|---------------------|-------------------|
| 核心问题 | 你是谁？ | 你能做什么？ | 你做了什么？ |
| 时机 | 登录/会话开始时 | 每次资源访问时 | 持续进行 |
| 输出 | 身份确认 | 访问权限 | 日志记录 |
| 失败后果 | 拒绝访问 | 操作被阻止 | 无直接后果（但无记录） |
| 典型技术 | 密码/MFA/证书 | ACL/RBAC/策略 | 日志审计/SIEM |

### 1.3 认证的五个基本原则

```
认证的基本原则：

1. 身份声明 (Identification)
   └── 用户声称自己是谁（用户名、ID、卡号）
   → "我是 Alice"

2. 身份验证 (Verification)
   └── 提供证据证明自己是所声称的人
   → 输入密码/刷指纹/插卡

3. 认证通过 → 授予令牌 (Token/Session)
   └── 后续操作不再需要反复认证

4. 认证的强度取决于所使用认证因素的类型和数量
   └── 单因素 < 双因素 < 多因素

5. 认证机制应该保护凭证的安全
   └── 密码不传输明文 → Hash/Challenge-Response
```

## 二、认证的三大因素

### 2.1 三大认证因素

```
认证三因素 (Something you...)

因素一：你知道什么 (Knowledge Factor)
├── 密码、PIN码
├── 密保问题
└── 助记词
    ↳ 特点：记忆负担重，可被社会工程获取

因素二：你有什么 (Possession Factor)
├── 智能卡、U-Key
├── 手机（短信验证码/软令牌）
└── 硬件令牌 (RSA SecurID / YubiKey)
    ↳ 特点：物理物品可丢失或被窃取

因素三：你是谁 (Inherence Factor) — 生物特征
├── 指纹、掌纹
├── 人脸、虹膜、声纹
├── 签名动态特征
└── 击键模式
    ↳ 特点：无法"忘记"，但泄露后无法更换
```

### 2.2 多因素认证 (MFA) 的定义

```
双因素认证 (2FA) = 至少使用两种不同因素的组合

✅ 正确组合（来自不同因素）：
├── 密码 + 短信验证码  (Knowledge + Possession)
├── PIN + 指纹        (Knowledge + Inherence)
├── 智能卡 + PIN      (Possession + Knowledge)

❌ 不构成真正的双因素（来自同一因素）：
├── 密码 + PIN码      (都是 Knowledge)
├── 智能卡 + U-Key    (都是 Possession)
└── 指纹 + 人脸        (都是 Inherence)
```

### 2.3 认证因素对比

| 因素类型 | 优势 | 劣势 | 成本 |
|----------|------|------|------|
| **密码** | 实现简单，零硬件成本 | 记忆负担，易被社工/暴力破解 | ⭐ 低 |
| **短信/OTP** | 使用方便，用户熟悉 | SIM 卡劫持风险，延迟 | ⭐⭐ |
| **软令牌(TOTP)** | 离线可用，安全 | 手机丢失风险 | ⭐ 低 |
| **硬件令牌(U2F/FIDO2)** | 防钓鱼，最高安全 | 成本高，携带不便 | ⭐⭐⭐ |
| **指纹** | 方便快速 | 不可更换，容易残留 | ⭐⭐ |
| **人脸** | 无接触，用户体验好 | 光照敏感，3D 面具可绕过 | ⭐⭐⭐ |
| **证书** | PKI 体系，强安全 | 部署复杂，证书管理 | ⭐⭐⭐ |

## 三、五类访问控制模型

### 3.1 模型全景

```
访问控制模型的演进：

DAC (自主访问控制) ─── 1970s
     ↓
MAC (强制访问控制) ─── 1980s
     ↓
RBAC (基于角色) ──── 1992
     ↓
RBAC + 约束 ─────── 2000s
     ↓
ABAC (基于属性) ──── 2010s
```

### 3.2 五大模型详解

#### DAC (Discretionary Access Control) 自主访问控制

```
特点：资源所有者自主决定谁可以访问自己的资源

实现：ACL (访问控制列表)
├── 每个文件/对象都有拥有者
├── 拥有者可以设置读(R)、写(W)、执行(X)权限
└── 典型实现：Linux/Windows 文件权限

优点：灵活，用户自主管理
缺点：权限可能被恶意传播（用户 A 授权用户 B，B 又授权 C）
     无法统一策略管理

CISP 考点：
├── "资源的拥有者决定访问权限" ← DAC 的定义特征
└── 适用场景：个人文件共享、小型团队协作
```

#### MAC (Mandatory Access Control) 强制访问控制

```
特点：系统根据安全标签强制执行访问控制，用户无权修改

核心概念：
├── 主体 (Subject)：用户/进程 → 有安全许可级别
│   (Top Secret > Secret > Confidential > Unclassified)
├── 客体 (Object)：文件/资源 → 有安全分级标签
│   (Top Secret > Secret > Confidential > Unclassified)
└── 规则：主体必须 ≥ 客体的安全级别才能访问

两大经典模型：
├── Bell-LaPadula (BLP) 模型 — 保护机密性
│   ├── 不上读 (No Read Up)：低级不能读高级
│   └── 不下写 (No Write Down)：高级不能写低级
│
└── Biba 模型 — 保护完整性
    ├── 不下读 (No Read Down)：高级不读低级（怕被污染）
    └── 不上写 (No Write Up)：低级不写高级（怕污染高级）

CISP 考点：
├── MAC = 系统强制，用户无法绕过
├── BLP → 机密性
├── Biba → 完整性
├── "不上读/不下写" 分别属于哪个模型 → 选择题高频
└── SELinux / AppArmor 是现代 MAC 实现
```

#### RBAC (Role-Based Access Control) 基于角色的访问控制

```
特点：权限不直接授予用户，而是授予角色，用户通过角色获得权限

核心结构：

   用户 (Users)          角色 (Roles)         权限 (Permissions)
   ┌─────────┐           ┌─────────┐           ┌─────────────┐
   │  Alice   │────→      │  经理    │────→      │ 读工资单     │
   ├─────────┤           ├─────────┤           ├─────────────┤
   │  Bob     │────→      │ 分析师   │────→      │ 查询报告     │
   ├─────────┤     ↘     ├─────────┤     ↘     ├─────────────┤
   │  Carol   │──→ 角色1 →│  员工    │──→ 权限3 →│ 提交报销     │
   └─────────┘           └─────────┘           └─────────────┘

RBAC 的核心原则：
├── 最小权限 (Least Privilege)：只给完成任务所需的最小权限
├── 职责分离 (SoD)：关键操作需要多人完成
├── 权限抽象：用户＜角色＜权限 的间接映射

RBAC 的层级：
├── RBAC0：基础模型（用户↔角色↔权限）
├── RBAC1：角色继承（高角色继承低角色的权限）
├── RBAC2：角色约束（互斥角色、基数限制）
└── RBAC3：合并 RBAC1 + RBAC2（最完整）

CISP 考点：
├── RBAC 是目前企业最常用的访问控制模型
├── "通过角色间接授权" ← RBAC 的核心特征
└── "最小权限" + "职责分离" 常与 RBAC 关联出现
```

#### ABAC (Attribute-Based Access Control) 基于属性的访问控制

```
特点：访问决策基于主体属性、客体属性、环境属性等多维度组合

决策逻辑：
IF (用户.部门 == 资源.所属部门)
   AND (用户.安全级别 ≥ 资源.敏感级别)
   AND (时间 in 工作时间)
   AND (位置 == 公司内网)
THEN 允许访问

三个维度的属性：
├── 主体属性：角色、部门、安全级别、职位
├── 客体属性：敏感等级、类型、所属部门、创建时间
└── 环境属性：时间、地点、设备类型、网络状态

优点：
├── 细粒度：可以非常精确地控制
├── 动态：根据实时上下文决策
└── 灵活：新增规则不需要修改角色

缺点：
└── 复杂度高：策略管理难度大

CISP 考点：
├── ABAC 是最灵活的访问控制模型
├── 考虑"谁、什么、何时、何地、如何"多个维度
└── 适合云计算和零信任环境
```

#### 基于规则的访问控制 (Rule-Based)

```
特点：基于预设的规则（而非基于角色/属性标签）进行访问控制

实现：防火墙 ACL、路由策略

规则通常基于：
├── 源 IP / 目标 IP
├── 源端口 / 目标端口
├── 协议类型
└── 时间窗口

典型设备：防火墙、路由器、IPS
```

### 3.3 访问控制模型对比

| 模型 | 决策依据 | 灵活性 | 管理复杂度 | 典型场景 |
|------|----------|--------|-----------|----------|
| DAC | 资源拥有者决定 | ⭐⭐⭐ | ⭐ (低) | 个人文件共享 |
| MAC | 系统安全标签 | ⭐ (低) | ⭐⭐ | 军事/政府系统 |
| RBAC | 用户角色 | ⭐⭐ | ⭐⭐ | 企业应用（最常用） |
| ABAC | 多维属性 | ⭐⭐⭐ | ⭐⭐⭐ | 云计算、零信任 |

## 四、认证协议与机制

### 4.1 密码认证协议演进

```
密码认证的进化：

1. 明文传输密码 ← 🌑 黑暗时代
   问题：网络嗅探 → 直接获取密码

2. Hash 后传输
   问题：重放攻击 → 截获 Hash 直接重放

3. Challenge-Response (挑战-应答)
   服务器发随机数(Challenge)
   → 客户端: Hash(Challenge + Password) → Response
   → 服务器验证
   优势：每次 Challenge 不同 → 无法重放

4. Kerberos
   基于对称密钥的可信第三方认证协议
   ├── KDC (Key Distribution Center) = AS + TGS
   ├── 票据 (Ticket) 而非密码传输
   └── Windows Active Directory 的核心认证协议

5. 公钥认证 (SSH / TLS 客户端证书)
   使用公私钥对进行认证
   └── 私钥不离开客户端
```

### 4.2 Kerberos 协议（核心考点）

```
Kerberos 认证流程：

   客户端                     KDC                      服务端
     │                        │                         │
     │──① AS_REQ ──────────→│                         │
     │ (用户ID, TGT请求)     │                         │
     │                        │                         │
     │←─② AS_REP ───────────│                         │
     │ (TGT: 用KDC密钥加密)  │                          │
     │                        │                         │
     │──③ TGS_REQ ─────────→│                         │
     │ (TGT + 服务名)        │                          │
     │                        │                         │
     │←─④ TGS_REP ──────────│                         │
     │ (服务票据: 用服务密钥加密)│                        │
     │                        │                         │
     │──⑤ AP_REQ ─────────────────────────────────→│
     │ (服务票据)                                       │
     │                        │                         │
     │←─⑥ AP_REP ──────────────────────────────────│
     │ (认证确认)              │                         │

Kerberos 的三大组件：
├── AS (Authentication Server)：验证身份 → 发放 TGT
├── TGS (Ticket Granting Server)：发放服务票据
└── KDC = AS + TGS
```

## 五、身份管理全生命周期

### 5.1 身份生命周期

```
          创建/注册 (Provisioning)
                │
                ▼
          认证和访问 (Active)
                │
                ├── 权限变更 (Role Change)
                │       │
                │       ▼
                │   继续使用
                │
                ├── 冻结 (Suspended)
                │       │
                │       ├── 恢复 (Reactivation)
                │       │
                │       └── 注销 (Deprovisioning)
                │              │
                │              ▼
                │           归档/删除
                ▼
          密码重置循环
```

### 5.2 IAM (Identity and Access Management) 核心功能

```
IAM 的四大核心：

1. 身份治理 (Identity Governance)
   ├── 用户身份创建/修改/删除
   ├── 身份生命周期管理
   └── 身份合规审计

2. 访问管理 (Access Management)
   ├── 认证（登录）
   ├── 授权（权限控制）
   └── SSO（单点登录）

3. 权限管理 (Privileged Access Management - PAM)
   ├── 特权账号(管理员)的额外安全保护
   ├── 会话录像和控制
   └── 密码检查和轮换

4. 目录服务 (Directory Services)
   ├── 用户和资源的统一存储
   ├── LDAP / Active Directory
   └── 身份信息的单一来源 (Source of Truth)
```

## 六、审计与日志

### 6.1 审计的核心目标

```
安全审计的目的：

├── 可追溯 (Accountability)
│   └── "谁在什么时间做了什么" 可以被还原
│
├── 可检测 (Detection)
│   └── 从日志中发现异常行为和潜在入侵
│
├── 可取证 (Forensics)
│   └── 安全事件发生后可溯源
│
└── 合规 (Compliance)
    └── 满足法规要求（如网安法→日志留存6个月）
```

### 6.2 审计日志的要素

```
一条合格的审计日志应包含：

├── 时间戳 (Timestamp)：精确到毫秒的时钟同步(NTP)
├── 事件类型 (Event Type)：登录/登出/文件访问/权限变更
├── 主体标识 (Subject)：用户ID/进程ID/IP地址
├── 客体标识 (Object)：访问的资源/文件/系统
├── 操作 (Action)：读/写/删除/执行
├── 结果 (Result)：成功/失败/拒绝
└── 上下文 (Context)：设备信息/地理位置/会话标识
```

### 6.3 审计日志的安全保护

```
日志本身的保护措施：

├── 完整性保护 → 日志签名/哈希链/区块链
├── 防篡改 → WORM (Write Once Read Many) 存储
├── 集中存储 → 独立于业务系统的日志服务器
├── 访问控制 → 审计管理员与系统管理员分离
│               → 审计日志不能被系统管理员删除
├── 加密传输 → TLS 传输到集中日志服务器
└── 备份归档 → 异地备份，满足留存时间要求
```

## 七、CISP 考试要点速查

### 核心概念速查

| 概念 | 要点 |
|------|------|
| AAA | Authentication + Authorization + Accounting |
| 认证三因素 | 你知道的/你有的/你是的 |
| DAC | 资源拥有者决定权限 |
| MAC | 系统强制，基于安全标签 |
| RBAC | 通过角色间接授权（最常用） |
| ABAC | 基于多维属性动态决策 |
| BLP 模型 | 不上读、不下写 → 机密性 |
| Biba 模型 | 不下读、不上写 → 完整性 |
| Kerberos | 基于票据的对称密钥认证（KDC/AS/TGS） |
| 最小权限 | 只给完成任务所需的最小权限 |
| 职责分离 | 关键操作需多人完成 |

### 常见考试陷阱

1. ~~认证 (Authentication) = 授权 (Authorization)~~ → 先认证身份，再授予权限
2. ~~密码+密保问题是双因素认证~~ → 两者都是 Knowledge Factor，不构成双因素
3. ~~RBAC = MAC~~ → RBAC 是基于角色的自主定义，MAC 是系统强制标签
4. ~~BLP 模型保护完整性~~ → BLP 保护**机密性**，Biba 保护**完整性**
5. ~~Kerberos 使用非对称加密~~ → Kerberos 基于**对称密钥**，使用 KDC 作为可信第三方
