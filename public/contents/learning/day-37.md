## 一、RSA的数学脆弱面

### 1.1 RSA安全性基石

```
RSA安全性依赖于大数分解难题：
  给定 n = p × q（p、q为大素数）
  从 n 恢复 p 和 q 在计算上是不可行的

关键约束：
┌──────────────────────────────────────────────────────────┐
│  1. p 和 q 必须是随机生成的大素数（≥512位）               │
│  2. p 和 q 的差值必须足够大（|p-q| > 2^(nlen/2-100)）  │
│  3. p-1 和 q-1 必须有大的素因子                          │
│  4. 公钥 e 不应太小（常见 65537）                         │
│  5. 私钥 d 必须足够大（d > n^0.25/3）                    │
└──────────────────────────────────────────────────────────┘
```

### 1.2 模数n共享攻击

```
致命错误：两个用户共享相同的 n（即相同的 p,q）但使用不同的密钥对

设 Alice: (n, e_A, d_A)  Bob: (n, e_B, d_B)

Alice 用自己的私钥 d_A 可以计算：
  φ(n) = (p-1)(q-1)
  Bob的私钥 d_B = e_B^(-1) mod φ(n)

结论：任何知道自己密钥对的用户都能计算同一n下所有其他用户的私钥！
```

### 1.3 小指数攻击

| 攻击类型 | 条件 | 危害 |
|----------|------|------|
| **低加密指数** | e=3, 相同明文加密给3个人 | 中国剩余定理恢复明文 |
| **低解密指数** | d < n^0.25 | Wiener攻击可恢复d |
| **共模攻击** | 同一m用(e1,e2)加密，gcd(e1,e2)=1 | 无需私钥恢复明文 |
| **Fermat分解** | \|p-q\| 过小 | 快速分解n |

## 二、常见RSA实现攻击

### 2.1 Bletchley攻击（ROCA漏洞，CVE-2017-15361）

```
2017年发现的Infineon芯片RSA密钥生成漏洞：

原因：RSALib使用FastPrime算法加速密钥生成，但素数结构存在可预测性
影响：爱沙尼亚76万电子身份证需更换、大量TPM芯片受影响
攻击成本：约$40-80计算资源，每密钥约140 CPU小时
关键指标：密钥位数×2 + 密钥位数/2 次Miller-Rabin测试即可恢复私钥

教训：密码实现中任何"优化"都可能引入致命漏洞
```

### 2.2 Bleichenbacher攻击（PKCS#1 v1.5）

```
攻击目标：使用PKCS#1 v1.5填充的RSA加密

攻击原理：
1. 攻击者截获密文 C
2. 构造变形密文 C' 发送给服务器
3. 服务器尝试解密 → PKCS#1 v1.5填充验证
4. 根据服务器是否返回"填充错误"泄漏信息
   → "正确的PKCS#1结构" = 1位信息
5. 重复~2^20次查询 → 恢复明文

出现场景：TLS中RSA密钥交换、XML加密

防御：使用OAEP填充替代PKCS#1 v1.5
```

### 2.3 随机数生成失败

```
索尼PS3 ECDSA密钥泄露（2010）：
  ECDSA签名需要随机数 k
  索尼使用了固定的 k
  → 两个签名使用相同 k
  → 私钥可从两个签名中直接计算

公式推导：
  s1 = k^(-1)(z1 + r·d) mod n
  s2 = k^(-1)(z2 + r·d) mod n
  → k = (z1 - z2)/(s1 - s2) mod n
  → d = (s1·k - z1)/r mod n

教训：签名中随机数必须真正随机且唯一！
```

## 三、ECC的实现陷阱

### 3.1 曲线选择的安全考虑

| 曲线类型 | 示例 | 安全评估 |
|----------|------|----------|
| **NIST曲线** | P-256, P-384, P-521 | 主流，但存在未知种子质疑 |
| **Curve25519** | X25519, Ed25519 | 现代推荐，安全设计 |
| **国密曲线** | SM2 | 中国国家标准 |
| **Brainpool** | brainpoolP256r1 | 欧洲标准，可验证参数 |
| **自定义曲线** | - | ⚠️ 极易出错，不要使用 |

### 3.2 椭圆曲线关键参数验证

```
使用他人提供的曲线参数时必须验证：

1. 曲线方程是否正确？E: y² = x³ + ax + b (mod p)
2. 基点G是否在曲线上？
3. 阶n是否为素数？
4. 余因子h是否合理？
5. 曲线的嵌入度是否足够大？（防MOV攻击）

典型攻击：没有验证基点在曲线上
  → 攻击者提供不在曲线上但阶很小的"假基点"
  → 每次DH交换泄漏私钥的部分信息
  → 多次交互后恢复完整私钥

这就是"Invalid Curve Attack"
```

### 3.3 扭曲曲线安全性

```
X25519的优点之一：
  X25519使用Montgomery阶梯算法，不要求输入点验证
  → 输入任何32字节都能安全处理
  → 避免Invalid Curve Attack

对比传统ECDH：
  必须验证对方公钥在正确曲线上
  否则可能遭受Invalid Curve或Small Subgroup攻击
```

## 四、侧信道攻击

### 4.1 攻击分类

```
侧信道攻击：通过物理实现特性推断密钥

┌───────────────┬──────────────────────────────────┐
│   攻击类型    │           利用信息                │
├───────────────┼──────────────────────────────────┤
│ 计时攻击      │ 不同操作执行时间的差异            │
│ 功耗分析(SPA) │ 单个密码运算的功耗曲线            │
│ 功耗分析(DPA) │ 多个运算的统计功耗差异            │
│ 电磁分析      │ CPU/RAM发出的电磁辐射             │
│ 缓存攻击      │ CPU缓存的访问模式                 │
│ 故障注入      │ 电压/时钟/激光干扰导致错误计算    │
└───────────────┴──────────────────────────────────┘
```

### 4.2 RSA计时攻击示例

```
朴素RSA解密使用平方-乘算法：

def pow_mod(base, exp, mod):
    result = 1
    for bit in bits(exp):        # 从左到右扫描指数位
        result = result² mod mod  # 总是执行：平方
        if bit == 1:             # 仅当该位为1：
            result = result * base mod mod  # 乘操作（额外时间！）

攻击：测量解密时间
  → 位1：平方+乘 → 时间较长
  → 位0：仅平方   → 时间较短
  → 可逐位恢复私钥！

防御：恒定时间实现（总是执行乘操作，结果条件选择）
```

### 4.3 故障注入

```
Bellcore攻击（1996）：
  目标：RSA-CRT签名（中国剩余定理优化）

  正常：S = CRT(S_p, S_q)
    其中 S_p = m^d mod p, S_q = m^d mod q
  
  攻击：注入故障使S_q计算出错 → S'
  → gcd((S'^e - m) mod n, n) = p
  → 分解n，暴露私钥！

防御：
  - 验证签名：S^e mod n == m
  - 硬件防护：屏蔽、滤波、传感器
```

## 五、量子计算威胁

### 5.1 Shor算法的威胁

```
Shor算法（1994）能在多项式时间内分解大整数：

算法            经典复杂度      量子复杂度
─────────────────────────────────────────────
大数分解       次指数 O(2^n)   多项式 O(n³)
离散对数       次指数 O(2^n)   多项式 O(n³)

这意味着：
  → RSA、DSA、ECDSA、DH、ECDH 全部不安全
  → 当前所有公钥密码体系的基础将崩塌
```

### 5.2 量子计算发展现状

| 年份 | 里程碑 | 量子比特数 |
|------|--------|-----------|
| 2019 | Google "量子霸权" | 53 qubits |
| 2022 | IBM Osprey | 433 qubits |
| 2023 | IBM Condor | 1121 qubits |
| 预计2030+ | 破解RSA-2048需要 | ~4000 逻辑qubits |

> ⚠️ 虽然当前量子计算机还很原始，但"先存储后解密"（Harvest Now, Decrypt Later）已经是现实威胁。

## 六、后量子密码概述

### 6.1 NIST后量子密码标准

NIST于2024年发布了首批后量子密码标准：

| 算法 | 类型 | 用途 | 密钥大小 |
|------|------|------|----------|
| **CRYSTALS-Kyber (ML-KEM)** | 格密码 | 密钥封装 | ~800B 公钥 |
| **CRYSTALS-Dilithium (ML-DSA)** | 格密码 | 数字签名 | ~1.3KB 公钥 |
| **FALCON** | 格密码 | 数字签名 | ~897B 公钥 |
| **SPHINCS+** | 哈希签名 | 数字签名 | ~32B 公钥，17KB 签名 |

### 6.2 迁移策略

```
混合方案（当前推荐）：
  ┌─────────────────────────────────────┐
  │  经典密钥交换  +  后量子密钥交换    │
  │     (ECDHE)        (Kyber)          │
  │         ↓              ↓            │
  │         ⊕ 组合密钥材料 ⊕            │
  │              ↓                      │
  │         最终会话密钥                 │
  └─────────────────────────────────────┘
  优势：即使一方被攻破，整体仍然安全
```

### 7.1 重大事故总结

| 时间 | 事件 | 漏洞类型 | 影响 |
|------|------|----------|------|
| 2010 | Sony PS3 ECDSA | 随机数k重用 | 私钥泄露，系统被完全破解 |

| 2018 | ROBOT攻击 | Bleichenbacher变种 | Facebook、PayPal等受影响 |
| 2022 | CVE-2022-21449 (Java ECDSA) | 空(r,s)绕过签名验证 | Java 15/16/17关键漏洞 |

### 7.2 Java ECDSA漏洞深度分析

```java
// CVE-2022-21449: Java 15+ ECDSA签名验证漏洞

// 攻击者构造的"签名"：
byte[] r = new byte[32];  // 全0
byte[] s = new byte[32];  // 全0

// Java 15-17的实现问题：
// ECDSA验证时需要将r,s转为BigInteger
// getField()返回了错误的值，导致 (0,0) 被视为
// 等同于无限远点，从而绕过验证
Signature sig = Signature.getInstance("SHA256withECDSA");
sig.initVerify(publicKey);
sig.update(data);
sig.verify(signatureBytes);  // (0,0)签名 → 返回 true！

// 任何ECDSA公钥都能接受 (0,0) 作为有效签名！
```

## 八、CISP考试速查

### 高频考点

| 考点 | 记忆口诀 |
|------|---------|
| RSA安全基础 | "大数分解是前提" |
| 共模攻击条件 | "同n不同e，gcd(e1,e2)=1" |
| ECC优势 | "短密钥高强度，256≈RSA3072" |
| 量子威胁算法 | "Shor解RSA，Grover破哈希" |
| 后量子方向 | "格密码主力(Kyber)，哈希补位(SPHINCS)" |
| 签名随机数 | "k要随机k要随机k要随机（重要说三遍）" |

### 常见陷阱

1. **"RSA-2048永远安全"** → 量子计算突破后不再安全
2. **"ECC实现简单"** → 实际实现中陷阱很多，需要仔细验证
3. **"侧信道只在理论上存在"** → 大量实际攻击已证明其有效性
4. **"私钥只要保密就安全"** → 需要考虑密钥生命周期的完整管理
