# SRC 漏洞提交与奖金申请实战

## 1. SRC 平台介绍

SRC（Security Response Center / Security Research Center）是企业自建的漏洞接收与奖励平台，向外部安全研究者购买漏洞信息，以保障自身业务安全。国内主流 SRC：

| 平台 | 官方地址 | 典型奖金范围 |
|------|---------|------------|
| 阿里 SRC | https://security.alibaba.com/ | 500 - 50,000+ |
| 腾讯安全响应中心 | https://security.tencent.com/ | 500 - 100,000+ |
| 百度安全响应中心 | https://sec.baidu.com/ | 500 - 50,000+ |
| 京东 SRC | https://security.jd.com/ | 500 - 30,000+ |
| 字节跳动 SRC | https://src.bytedance.com/ | 1,000 - 50,000+ |
| 美团 SRC | https://security.meituan.com/ | 500 - 30,000+ |
| 小米 SRC | https://sec.xiaomi.com/ | 500 - 20,000+ |
| 360 SRC | https://security.360.net/ | 500 - 30,000+ |
| 银联安全应急响应中心 | https://sec.unionpay.com/ | 1,000 - 50,000+ |
| 国家信息安全漏洞共享平台 CNVD | https://www.cnvd.org.cn/ | 部分奖金 / 荣誉证书 |
| CTF 众测平台（补天 / 漏洞盒子 / HackerOne） | https://www.butian.net / https://www.vulbox.com / https://www.hackerone.com/ | 按项目结算 |

## 2. 注册前准备

### 2.1 身份与收款信息

```
- 身份证号（实名认证必填）
- 身份证扫描件 / 手持身份证照片（部分平台要求）
- 邮箱 / 手机号 / 真实姓名
- 收款银行账户（境内银行卡为主，部分支持支付宝）
- 税务信息（超过个税起征点需要扣税）
```

### 2.2 账号安全

```
- 使用独立邮箱 / 手机号，避免与公开身份混同
- 启用 2FA / 短信验证
- 不使用公司邮箱提交（否则归属可能被认定为职务漏洞）
- 不与他人共用账号
```

## 3. 典型奖金等级

各 SRC 一般按 CVSS 或自定标准划分等级，以阿里 SRC 为例（示意）：

| 等级 | 典型漏洞 | 奖金（示意） |
|------|---------|-------------|
| 严重 | 线上系统 RCE、核心数据库拖库、核心业务未授权访问 | 10,000 - 100,000 |
| 高危 | 后台 SQL 注入、可批量获取用户信息、支付业务漏洞 | 3,000 - 30,000 |
| 中危 | 反射型 XSS、存储型 XSS（非核心）、普通越权、文件读取 | 500 - 5,000 |
| 低危 | 信息暴露、弱密码、目录浏览 | 100 - 1,000 |
| 一般 | 信息类 / 误报 / 重复 | 0 |

## 4. 漏洞挖掘策略

### 4.1 目标选择

```
- 资产多、SRC 规则完善、奖金透明的大厂；
- 目标产品有最新版本发布、新功能上线的时间段；
- 有公开 API 的产品（移动端、小程序、开放平台）；
- 自身熟悉的技术栈（PHP / Java / Node.js / Go ...）。
```

### 4.2 资产收集

```bash
# 子域枚举
amass enum -passive -d target.com
subfinder -d target.com -silent | httpx -title -tech-detect

# 端口扫描
masscan 10.0.0.0/8 -p 1-65535 --rate 20000
nmap -sV -sC -T4 <ip>

# 指纹识别
whatweb https://target.com
nuclei -u https://target.com -t ~/nuclei-templates/technologies/
```

### 4.3 常见突破口

```
1. 登录 / 注册 / 密码重置 / 短信验证码（账号接管、暴力破解、回显）
2. /api /admin /internal /debug /actuator /swagger-ui（未鉴权接口）
3. 文件上传 / 下载 / 导出（WebShell、任意文件读取）
4. 回调地址 / 第三方 OAuth（SSRF、账号绑定劫持）
5. 订单 / 钱包 / 支付（金额篡改、负数、并发）
6. 移动端 App 反编译后的接口（Retrofit / OkHttp / 签名绕过）
7. CORS 配置不当导致敏感信息泄漏
8. 管理后台弱口令 / 短信验证码绕过
9. SSRF 穿透内网（Redis、Elasticsearch、MySQL）
10. Java 反序列化（Shiro、WebLogic、Fastjson、Dubbo）
```

## 5. 提交报告的标准结构

```
【漏洞标题】
<厂商> <产品> <模块> <漏洞类型>（示例："某电商 订单导出接口 存在 越权下载他人发票 漏洞"）

【漏洞概述】
简要说明漏洞成因、影响范围、危害程度。

【复现步骤】
Step 1: 登录普通用户账号 user_a / pass_a，访问 https://...
Step 2: 抓取到请求 GET /api/order/export?order_id=ORD12345 HTTP/1.1
Step 3: 修改 order_id 为他人订单 ORD12346，成功下载

【PoC / 请求与响应】
```
GET /api/order/export?order_id=ORD12346 HTTP/1.1
Host: api.target.com
Authorization: Bearer <user_a_token>

HTTP/1.1 200 OK
Content-Disposition: attachment; filename="invoice_ORD12346.pdf"
...
```

【危害分析】
- 影响所有历史订单用户
- 可遍历导出大量用户发票，泄露姓名、手机号、收货地址

【修复建议】
1. 在服务端校验订单所属用户是否等于当前登录用户
2. 增加导出频率限制（每小时最多 3 次）
3. 记录每次导出日志，用于审计
4. 敏感数据脱敏（手机号中间 4 位加 *）

【附件】
- 复现演示视频
- 完整请求/响应 HAR 文件
- PoC 脚本
```

## 6. 提交流程速查

1. 登录目标 SRC 官网，阅读提交规则与边界说明；
2. 选择提交类型：通用漏洞 / 业务逻辑漏洞 / 移动端漏洞 / 信息泄露；
3. 填写漏洞表单，粘贴 PoC、上传附件；
4. 选择影响等级（厂商可能在审核中调整）；
5. 提交后系统自动派发工单号，等待 3-15 工作日内审核；
6. 审核通过：奖金到账 / 积分累计 / 公示上榜；
7. 未通过：可在 SRC 站内信沟通，或补充信息申诉。

## 7. 常见争议与应对

| 问题 | 可能原因 | 应对方法 |
|------|---------|---------|
| 被判定为重复 | 他人已先提交 | 提供更深入分析作为补充；或转向新资产 |
| 被降级（如高危→中危） | 实际影响有限、修复困难或需特殊条件 | 提供更强危害论证、真实场景演示 |
| 被判定 "已知问题" | 厂商已发现但未公开 | 寻找其它模块、新版本或不同产品 |
| 奖金长时间未到账 | 财务流程 / 税务问题 | 站内信 / 邮件联系 SRC 运营方 |
| 同一漏洞在多家 SRC 均可提交 | 资产归属判断不清 | 先确认影响归属哪一家主业务，按规则提交 |

## 8. 合规与伦理

- **授权范围**：只在 SRC 明确声明的测试范围内活动；
- **不做破坏性操作**：不得删除数据、修改数据、影响线上用户；
- **最小数据量**：获取敏感信息时仅演示最小样例，不得批量拖库；
- **不披露未修复漏洞**：厂商未修复前不得公开详情；
- **不与他人共享未提交漏洞**：防止被抢注 / 违规；
- **尊重法律**：不得对未授权系统进行扫描 / 攻击，尤其是政府、事业单位等。

## 9. 长期经营建议

1. **建立资产清单**：维护你关注的目标列表，定期重测；
2. **工具自动化**：脚本化资产收集、指纹识别、初步 PoC；
3. **提交规范化**：形成标准化模板，提高审核通过率；
4. **关注官方公告**：各 SRC 会发布重点关注资产、双倍奖金活动、新产品发布等公告；
5. **社区交流**：关注安全社区（先知 / 小密圈 / 安全会议），了解新漏洞模式；
6. **提升专业度**：关注代码审计、IoT、逆向、工控等高级方向，避开红海漏洞。

## 10. 税务与结算小贴士

- **个人所得税**：多数 SRC 按"劳务报酬所得"代扣代缴个税（税率约 20%）；
- **发票 / 合同**：部分平台要求研究者提供发票或签订合同；
- **跨年结算**：部分 SRC 有年度累计、排名奖励等机制；
- **对公合作**：若以公司形式合作，提供增值税专用发票可抵扣；
- **保留记录**：妥善保留 SRC 邮件、工单号、银行流水，便于年度申报。
