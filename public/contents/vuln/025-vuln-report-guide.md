## 一、报告原则

一份优秀的漏洞报告应具备以下特征：

| 原则 | 说明 |
|------|------|
| **可复现** | 审核人员按步骤可完整复现漏洞 |
| **可理解** | 非原作者也能理解漏洞成因与危害 |
| **最小化** | 不引入与漏洞无关的攻击链；不做破坏性操作 |
| **结构化** | 使用标准章节（标题 / 概述 / 复现 / PoC / 危害 / 修复） |
| **合规** | 不泄漏他人真实数据，敏感信息打码 |
| **友好** | 语言客观专业，避免情绪化表达 |

## 2. 报告结构模板

```
==========================================================
【漏洞标题】
<厂商> <产品/网站/系统> <模块> 存在 <漏洞类型>（可选：影响 <版本/环境>）

【提交者】
<你的署名 / 团队>   日期：YYYY-MM-DD

【漏洞概述】
<一段 200-500 字中文描述：说明漏洞在哪、如何触发、可以干什么、影响哪些业务>

【影响范围】
- 影响产品 / URL：https://target.com/xxx
- 影响版本：v1.0 到 v1.2.3
- 受影响用户 / 数据：所有注册用户的手机号、收货地址
- 业务影响：可横向越权查看他人订单

==========================================================
【复现环境】
- 操作系统：Windows 11 / macOS 13 / iOS 16 / Android 13
- 浏览器：Chrome 115.0.5790.110
- 账号角色：普通用户（user_test / 138****0001）
- 测试时间：2024-08-18 14:30:00 (UTC+8)

==========================================================
【复现步骤】
Step 1. 注册普通用户账号（手机号 138****0001），并登录
        URL：https://target.com/login
        Credential：138****0001 / <登录密码>

Step 2. 进入"个人中心 - 我的订单"
        URL：https://target.com/user/orders
        点击"导出"按钮，抓包获得请求：

Step 3. 把抓包请求发送到 Burp Repeater，观察请求：

        POST /api/order/export HTTP/1.1
        Host: api.target.com
        Authorization: Bearer eyJhbGciOi...
        Content-Type: application/json

        {"order_id":"ORD10001"}

Step 4. 修改 order_id 为他人订单号 ORD10002，发送请求：

        {"order_id":"ORD10002"}

Step 5. 服务端返回 200 OK，并在响应中返回他人订单的 PDF 文件下载链接
        或在响应体中直接返回他人收货地址、手机号、商品信息

==========================================================
【PoC - 完整请求与响应】
```
--- Request ---
POST /api/order/export HTTP/1.1
Host: api.target.com
User-Agent: Mozilla/5.0
Accept: application/json
Authorization: Bearer eyJhbGciOi...
Content-Type: application/json
Content-Length: 25

{"order_id":"ORD10002"}

--- Response (关键片段) ---
HTTP/1.1 200 OK
Content-Type: application/pdf
Content-Disposition: attachment; filename="invoice_ORD10002.pdf"
X-Order-Owner: user_b                  <- 返回他人用户名 / 手机号
...
```

==========================================================
【漏洞原理分析】
目标服务的后端代码（示意）未在服务端进行数据归属校验：

```javascript
// /api/order/export
app.post('/api/order/export', auth, (req, res) => {
  const { order_id } = req.body;
  // ❌ 缺少 "订单是否属于当前登录用户" 的判断
  OrderModel.findOne({ _id: order_id }, (err, doc) => {
    if (err) return res.status(500).end();
    res.pdf(generate_invoice(doc));      // 直接返回订单信息 PDF
  });
});
```

攻击者只需变更 order_id 参数即可横向遍历 / 导出任意订单。

==========================================================
【危害等级与影响评估】
CVSS v3.1 Vector：AV:N/AC:L/PR:L/UI:N/S:U/C:H/I:N/A:N  → 7.5 (High)
- 机密性：高（可读取他人姓名、手机号、收货地址、商品明细）
- 完整性：无
- 可用性：无
- 影响规模：历史存在的所有订单（估计数万至数十万用户）

==========================================================
【修复建议】
1. 【强校验】在服务端对 order_id 强制追加 `owner_id = current_user_id` 查询条件
2. 【鉴权统一】抽象出通用鉴权中间件，在所有涉及个人数据的读写接口自动附加
3. 【资源节流】对订单导出接口做频率限制（如每小时 3 次）、时间窗口限制
4. 【日志与告警】对越权尝试记录 `user_id、target_order_id、timestamp、source_ip`
5. 【数据脱敏】即使在合法场景下，手机号 / 身份证号等 PII 也按 `138****0000` 规则脱敏

==========================================================
【临时缓解方案】
在线上修复上线前，临时在 WAF / Nginx 层对 /api/order/export 做：
- 登录态强制校验
- 每 IP 100 QPS 速率限制
- 监控高频率导出异常行为并告警

==========================================================
【附件清单】
1. 复现视频：replay_20240818_1430.mp4（已对手机号、Token 打码）
2. 浏览器 HAR：export_other_user_order.har
3. PoC 脚本：poc.py（仅实现读取一条他人订单用于演示，未批量）
4. 对比截图：
   - 正常用户自己的订单：img1_own_order.png
   - 越权访问他人订单：img2_other_user_order.png

==========================================================
【测试者声明】
本人已在授权范围内测试，并未对生产数据进行批量操作或造成实际业务影响；
报告中出现的用户信息均为本人注册的测试账号 / 或已做脱敏处理。
==========================================================
```

## 3. 规范要点

### 3.1 标题规范

- 简洁明确：避免"一个神奇的漏洞""严重漏洞"等模糊词
- 推荐格式：`厂商 产品 模块 漏洞类型`
  - 例 1：`XXCMS 后台 admin_user.php SQL 注入漏洞`
  - 例 2：`XX 商城 订单导出接口 水平越权漏洞`
  - 例 3：`XX APP v2.3.4 注册接口 短信验证码暴力破解漏洞`

### 3.2 复现步骤必须可执行

- 给出具体 URL / 接口路径
- 提供必要的 Cookie / Token / Header 示例（可脱敏）
- 提供完整 HTTP 请求与响应（在报告中以代码块形式粘贴）
- 复现视频 / HAR 文件 / 截图作为附件辅助

### 3.3 PoC 最小化

- 不要附加多余功能；
- 避免使用"自动化工具"一次性跑出大量数据；
- 只需展示影响最小样例即可；
- 敏感数据（手机号、Token、真实姓名）要打码（例如 `138****0001`）。

### 3.4 原理说明要有技术深度

- 定位到具体文件 / 代码行（有源码审计场景时）
- 对 Java 反序列化 / XSS / CSRF / SSRF 等，明确 payload 构造思路
- 对业务逻辑漏洞，分析设计上的问题（缺少鉴权、可预测、并发竞争等）

### 3.5 修复建议要具体

- 避免"加强安全检查"等空话；
- 建议给出具体代码片段（如参数化查询 / 中间件）
- 提供临时缓解方案（WAF 规则、Nginx 配置、接口下线）便于快速响应

## 4. 常见错误与改进

| 常见错误 | 改进 |
|---------|------|
| "越权成功了，自己看吧" | 明确说明哪个参数、如何构造 payload、响应中哪一行反映越权 |
| 直接贴 screenshot 不附请求文本 | 同时提供 raw 请求/响应文本 + 截图 |
| 只贴 payload 不说明前置条件 | 提供完整复现步骤（注册、登录、角色、环境） |
| 不区分"演示危害"与"真实危害" | 说明漏洞在真实业务场景下对用户/厂商的实际损害 |
| 报告中出现未经脱敏的他人信息 | 打码或只使用自己注册的测试账号 |
| 写大量情绪化抱怨或指责 | 保持客观专业，专注漏洞本身与技术细节 |

## 5. 可复用的检查清单

提交前请确认：

- [ ] 标题符合规范
- [ ] 包含厂商 / 产品 / 版本 / 官方 URL
- [ ] 复现步骤逐条可执行
- [ ] 附有完整的 HTTP 请求 / 响应文本
- [ ] 包含复现视频 / 截图 / HAR 文件
- [ ] 原理分析到位，引用了具体代码或配置
- [ ] 提供了修复建议与临时缓解方案
- [ ] 所有他人隐私信息已脱敏
- [ ] 未在报告中暴露内部凭证 / 密钥 / Token
- [ ] 已阅读对应 SRC / CNVD / CVE 平台的提交流程文档

## 6. 语言与格式建议

- 使用中文撰写（部分国际平台可英文）
- 代码 / 请求 / JSON 使用 Markdown 代码块
- 表格描述版本影响、CVSS 评分等信息
- 图片使用 PNG / JPG，大小控制在单张 2-5 MB 内
- 视频使用 MP4 / H.264，控制在 50 MB 以内，清晰度 720p 即可
- 附件命名规范：`厂商_漏洞类型_日期_序号_描述.ext`

## 7. 最后建议

- 先在本地靶场 / 授权测试环境中复现并截图；
- 使用独立的测试账号（避免用真实个人常用账号）；
- 对关键请求 / 响应保持一份本地备份，便于后续审核跟进；
- 遇到审核争议时提供补充材料或重新录制演示，不情绪化；
- 以专业、友好、负责任的态度提交，长期经营研究者声誉。
