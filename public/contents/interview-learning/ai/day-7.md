# Day 7：第1周总结 — 数据采集管道

> 🎯 面试目标：梳理第一周知识体系，构建安全数据采集管道的系统性认知

## 知识速览

### 核心概念
- **安全数据管道**：采集(Collection) → 传输(Transport) → 存储(Storage) → 处理(Processing) → 分析(Analysis) → 响应(Response)
- **数据源全景**：网络层(NetFlow/pcap)、主机层(syslog/eventlog)、应用层(WAF日志/API日志)、安全设备层(IDS/EDR告警)
- **数据质量三原则**：完整性（无缺失关键字段）、时效性（从事件发生到可查询的延迟）、一致性（统一时间格式和字段命名）

### 必问考点
| 面试官可能怎么问 | 你应该怎么答 |
|------------------|--------------|
| 你怎么设计一个安全大数据平台 | 分四层架构：1)采集层(Filebeat/Fluentd)→2)消息队列(Kafka缓冲)→3)处理层(Logstash/Flink)→4)存储分析层(Elasticsearch/MongoDB+分析引擎)。重点：Kafka解决生产-消费速率不匹配问题 |
| 安全日志量太大怎么办 | 1)分级采集（核心系统全量+一般系统采样）2)冷热数据分离（热数据ES/冷数据HDFS）3)生命周期管理（P0事件日志保留1年，一般日志保留90天）4)压缩存储 5)边缘预处理（只传异常日志） |
| 如何处理多源日志的时间对齐问题 | 1)统一用UTC时间 2)所有日志采集时强制添加@timestamp和timezone字段 3)利用NTP确保源系统时间准确 4)采集端做时间纠偏（已知偏移量自动修正） |

### 技术细节
日志采集Pipeline架构示例：
```
安全设备 → syslog → Logstash → Kafka → Flink → ES
主机日志 → Filebeat → Kafka → Logstash → ES
云API日志 → Lambda → S3 → Spark → ES
```

Elastic Common Schema(ECS)是所有日志标准化的关键——面试中提ECS展示你对标准化的理解。

## 常见陷阱
- ⚠️ 直接往ES灌数据而没有任何缓冲层——高并发时ES会掉数据，Kafka缓冲是必备的基础设施
- ⚠️ 日志采了但不用——采集成本高但分析师用不上等于浪费，先定义分析需求再确定采集范围
