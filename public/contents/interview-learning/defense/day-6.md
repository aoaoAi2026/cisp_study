# Day 6：漏洞管理生命周期

> 🎯 面试目标：掌握漏洞管理的完整生命周期、CVSS评分体系、漏洞优先级排序方法以及修补策略

## 知识速览
### 核心概念
- **漏洞管理生命周期**：发现（资产发现+漏洞扫描）→ 评估（CVSS评分+业务影响分析）→ 修复（补丁管理+配置加固）→ 验证（复扫确认）→ 报告与度量。这是一个持续循环的过程，不是一次性项目。
- **CVSS（通用漏洞评分系统）**：从三个维度评估漏洞严重性——基础指标（攻击向量、复杂度、权限要求、用户交互、影响范围、机密性/完整性/可用性影响）、时间指标（利用代码成熟度、补丁级别、报告置信度）、环境指标（受影响资产的重要性、安全控制措施）。得分0-10分。
- **漏洞优先级排序**：不能仅依赖CVSS分数，还需考虑：①是否有在野利用（KEV目录）②资产重要性③漏洞是否可远程利用④是否有公开EXP⑤业务影响范围⑥是否在DMZ或核心区域。
- **攻击面管理（ASM）**：从攻击者视角持续发现和监控组织的数字资产（包括影子IT、遗忘的云服务、泄露的凭证等），识别这些资产上的漏洞和暴露面。

### 必问考点
| 面试官可能怎么问 | 你应该怎么答 |
|------------------|--------------|
| 描述一个完整的漏洞管理流程 | ①资产发现与分类——知道你有什么，哪些是关键资产；②漏洞扫描——定期自动扫描+临时专项扫描；③评估与分级——CVSS打分+资产重要性加权+威胁情报；④修复计划——关键漏洞24h/高危7天/中危30天/低危90天；⑤修复实施——补丁/配置变更/补偿控制；⑥验证——复扫确认修复效果；⑦度量——漏洞修复率、平均修复时间等KPI |
| CVSS 3.1的评分公式是什么样的？ | CVSS v3.1评分分两部分：①利用子分数（Exploitability）基于攻击向量(AV)、攻击复杂度(AC)、权限要求(PR)、用户交互(UI)；②影响子分数（Impact）基于机密性(C)、完整性(I)、可用性(A)。影响范围(Scope)决定最终公式。基础分数通过查表或公式计算得出，再根据时间指标和环境指标修正 |
| 如何对待"无法修复"的漏洞？ | "无法修复"的场景：①老旧系统无补丁（厂商停止支持）②业务不允许中断③修复可能导致兼容性问题。应对策略：①补偿控制——WAF虚拟补丁、网络隔离、增强监控②风险接受——高层审批签字③退役计划——制定时间表逐步替换④入侵检测规则——针对该漏洞定制检测签名 |
| 漏洞扫描对生产系统有影响吗？应该注意什么？ | 有影响：①扫描流量可能触发IPS/IDS告警②某些扫描会导致服务崩溃（如旧版SSL协议探测）③大量请求可能造成性能下降。注意事项：①首次扫描先做小范围测试②关键业务系统选择非业务高峰期③配置安全扫描策略（仅探测不利用）④提前通知相关团队⑤准备好应急回滚方案 |

### 技术细节
漏洞管理KPI仪表盘指标：
```
| 指标                    | 目标值        | 说明                     |
|------------------------|--------------|--------------------------|
| 资产覆盖率              | ≥95%         | 已纳入漏洞管理的资产占比  |
| 高危漏洞修复SLA达成率   | ≥90%         | 规定时间内修复的高危漏洞  |
| 平均漏洞修复时间(MTTR)  | 关键<24h     | 按严重级别分级统计       |
| 漏洞复发率              | ≤5%          | 同一漏洞再次出现的比例    |
| 扫描频率覆盖率          | 关键系统每周  | 不同级别资产不同扫描频率  |
```

## 常见陷阱
- ⚠️ **陷阱："只关注CVE和CVSS高分漏洞"**——CVSS高分不等于高风险。一个CVSS 9.8但只能从本地网络利用且目标系统在隔离区，实际风险可能远低于一个CVSS 6.5但能从互联网利用且在核心业务系统上的漏洞。必须结合环境和威胁情报评估。
- ⚠️ **陷阱："扫描完了就完事了"**——很多企业的漏洞管理止步于"扫出来、发报告"，修复环节断层。漏洞管理的核心是修复和验证，否则扫描只是徒增焦虑。
- ⚠️ **陷阱："补丁管理=漏洞管理"**——补丁管理只是漏洞管理的一部分。漏洞还包括配置缺陷、弱口令、逻辑漏洞等非补丁类问题，需要配置加固、安全基线等多种修复手段。
