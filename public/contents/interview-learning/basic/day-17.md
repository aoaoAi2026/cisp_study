# Day 17：安全评估与审计

> 🎯 面试目标：掌握安全评估方法论(漏洞评估/渗透测试/红蓝对抗)、等保测评、SOC2/ISO27001审计要点

## 知识速览

### 核心概念
- **漏洞评估 vs 渗透测试**：漏洞评估是'地毯式扫描'(用工具找所有已知漏洞，广度优先)；渗透测试是'目标导向的深度利用'(人工验证漏洞的实际可利用性，深度优先)
- **红蓝对抗(Red/Blue/Purple Teaming)**：红队模拟APT攻击(目标导向)，蓝队防御+检测+响应，紫队协调双方确保演习质量和学习闭环
- **等保2.0五大规定动作**：定级(确定等级1-5)→备案(向公安机关备案)→建设整改→等级测评(第三方测评机构)→监督检查(定期检查)
- **SOC2 Type I vs Type II**：Type I是快照式(某时间点的控制设计有效性)，Type II是持续式(一段时间(通常6-12月)的控制运行有效性)，客户更看重Type II
- **漏洞管理生命周期**：发现(扫描+渗透+众测+威胁情报)→评估(CVSS评分+业务影响)→修复(补丁/配置/补偿控制)→验证(复扫)→度量(MTTR/修复率趋势)

### 必问考点
| 面试官可能怎么问 | 你应该怎么答 |
|------------------|--------------|
| 如何规划一次有效的渗透测试？ | 五阶段：1)范围定义(哪些IP/域名/应用/排除项+授权书签署) 2)信息收集(被动+主动，DNS/子域名/Shodan/WHOIS) 3)漏洞扫描+人工验证 4)利用与横向移动(按目标分级：获得Webshell/提权/域控) 5)报告与修复(按风险分级、给出POC和修复建议)。面试加分：提及PTES(Penetration Testing Execution Standard)标准。 |
| 等保2.0和1.0的核心变化？ | 五大变化：1)保护对象扩展——新增云计算/物联网/工控/移动互联四个扩展要求 2)安全要求从10大类变为8大类(安全物理环境、安全通信网络等) 3)可信验证三级起强制要求 4)个人信息保护成为独立控制点 5)密码技术要求更具体。面试强调：等保2.0真正的变化是'从被动合规到主动防御'。 |
| CVSS v3.1 评分的三个维度？ | 基础分(Exploitability:攻击向量/复杂度/权限/用户交互 + Impact:CIA三性影响)→环境分(修改基础分以适应目标环境)→时间分(漏洞利用代码成熟度/修复级别/报告置信度)。面试中能说出CVSS不是绝对的风险值(未考虑资产价值)，需要结合业务重要性做最终风险判定。 |
| ISO27001和SOC2的主要区别？为什么互联网公司倾向SOC2？ | ISO27001是全球性标准(ISMS建立+运行+持续改进)，SOC2是AICPA的美国审计标准(只关注五原则：安全/可用/处理完整性/机密/隐私)。互联网公司偏向SOC2原因：1)美国客户的合同要求 2)报告更详细(描述+控制测试结果) 3)有Type II持续审计的高商业认可度 4)不需要建立完整的ISMS体系。 |
| 如何处理扫描器报告的大量低危漏洞？ | 不可能全部修复。策略：1)按实际风险重新定级(内网低危可能不紧急、外网低危优先级提升) 2)业务影响分析(影响核心业务的低危可能是高危) 3)漏洞聚合(多个低危组合可能构成攻击链) 4)补偿控制(不能打补丁的系统通过WAF/IPS规则补偿)。面试核心：展示风险优先级判断能力而非机械修复。 |

### 技术细节
**自动化漏洞扫描流水线**：
```yaml
# CI/CD集成安全扫描
stages:
  - build
  - sast      # 静态代码分析(SonarQube/Semgrep)
  - sca       # 供应链扫描(Dependency-Check/Trivy)
  - dast      # 动态应用测试(ZAP/ Burp Suite API)
  - container # 容器镜像扫描(Trivy/Grype)
  - deploy
```
**等保测评时间线**：二级-每两年一次 → 三级-每年一次(含每年至少一次自查) → 四级-每半年一次。三级系统必须通过等保测评后才能上线运行。

## 常见陷阱
- ⚠️ 把合规当安全——过了ISO27001/等保不代表真的安全，合规是最低基线而非终点
- ⚠️ 扫描器依赖——只用自动化工具做评估，忽视人工逻辑漏洞(越权/业务逻辑/流程绕过)
- ⚠️ 评估报告写太技术——管理层只看风险和成本，用'可能造成XX万元损失'比'存在XX漏洞CVSS 7.5'更有说服力
