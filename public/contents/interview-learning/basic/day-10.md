# Day 10：入侵检测与防御系统

> 🎯 面试目标：掌握IDS/IPS的工作原理、部署策略、告警分析方法和面试中的常见追问

## 知识速览

### 核心概念
- **检测方法**：特征匹配（Signature-based）基于已知攻击特征库，准确率高但无法检测未知威胁；异常检测（Anomaly-based）基于基线偏离告警，可发现未知威胁但误报率高。现代IDS/IPS通常结合两者
- **HIDS vs NIDS**：基于主机的IDS（HIDS）部署在服务器上，监控文件完整性、注册表、进程、日志；基于网络的IDS（NIDS）部署在网络关键节点，分析流量数据包。面试重点：两者互补，企业应同时部署
- **误报与漏报的平衡**：调高检测灵敏度→误报增多→安全团队告警疲劳；调低检测灵敏度→漏报增多→威胁被忽略。这是IDS运维的核心难题，面试中展示对"平衡"的理解是加分项

### 必问考点
| 面试官可能怎么问 | 你应该怎么答 |
|------------------|--------------|
| IDS告警太多怎么办？ | 分三步优化：1)告警分级，按严重程度和资产重要性分P0-P3，优先处理P0 2)规则调优，关闭与业务无关的规则，调整阈值减少误报 3)引入SOAR自动化处理低级别告警，减少人工介入。长期看，需要结合威胁情报提高告警质量 |
| Snort和Suricata有什么区别？ | 两者都是开源IDS/IPS引擎，规则语法兼容。Suricata支持多线程（性能更好）、文件提取、TLS证书日志等高级特性。小规模用Snort够用，大规模流量建议Suricata。面试可以补充：两者都可以和ELK集成做日志分析 |
| 如何检测到C2通信（命令与控制）？ | 多维度检测：1)DNS Beaconing检测（固定时间间隔的DNS请求）2)异常外联（非工作时间、非常用端口、非标准协议）3)流量特征（心跳包大小固定）4)威胁情报匹配（已知C2域名/IP）5)JA3/JARM指纹检测TLS通信异常 |

### 技术细节

**Snort规则语法解析：**
```
alert tcp $EXTERNAL_NET any -> $HOME_NET 80 (
    msg:"WEB-IIS CodeRed v2 root.exe access";
    flow:to_server,established;
    uricontent:"/root.exe"; nocase;
    reference:url,www.cert.org/advisories/CA-2001-19.html;
    classtype:web-application-attack;
    sid:1256; rev:8;
)
```
- `alert`：动作（告警），还有log/pass/drop
- `tcp`：协议
- `$EXTERNAL_NET any -> $HOME_NET 80`：源→目的（方向箭头很关键）
- `msg`：告警描述信息
- `uricontent`：URL内容匹配
- `sid`：规则唯一标识，<100为保留，>=1000000为用户自定义

**IDS部署位置与检测范围：**
```
部署位置          能检测到              检测不到
─────────────────────────────────────────────────
互联网出口        外部攻击、C2通信      内网横向移动
核心交换机旁路    内网流量、横向移动    加密流量内容
服务器区前端      针对服务器的攻击      客户端攻击
终端HIDS          主机层面活动         网络层面攻击
```

**告警分级示例：**
| 等级 | 描述 | 举例 | 响应时间 |
|------|------|------|---------|
| P0 | 正在进行的成功攻击 | Webshell上传成功 | 15分钟 |
| P1 | 高可能性攻击 | SQL注入探测+数据库异常查询 | 1小时 |
| P2 | 可疑活动 | 异常端口扫描 | 4小时 |
| P3 | 一般事件 | 登录失败 | 24小时 |

## 常见陷阱
- ⚠️ "IDS部署了就可以高枕无忧"——IDS是"检测"工具不是"防御"工具。没有IPS联动或SOC人员响应，IDS只是记录攻击的日志系统
- ⚠️ "我们调高了IDS灵敏度所以更安全"——高灵敏度导致大量误报，安全团队容易产生告警疲劳，反而可能错过真正的攻击。需要持续优化规则找到平衡点
