# 逻辑漏洞与越权访问测试清单

## 1. 水平越权

**定义**：同角色用户可访问他人的数据（如用户 A 看到用户 B 的订单）。

**常用测试**：

- 修改 `?user_id=`、`?order_id=`、`?uid=`、`?file_id=` 等 ID 参数
- 替换 Cookie / JWT 中的用户标识字段（保持签名不变）
- `?username=admin` / `?role=user` → 改为 `?role=admin`
- 对比两位测试账号分别登录后，请求相同资源 ID 的返回

```
GET /api/order?order_id=123456 (本属于 B 的订单)
Authorization: Bearer token-of-A
```

若返回成功，即为水平越权。

## 2. 垂直越权

**定义**：低权限用户访问高权限功能（普通用户访问 admin 后台）。

**常用测试**：

- 直接访问 `/admin/xxx`、`/manager/xxx`
- 抓取管理员请求 URL/方法，换用普通用户 token 重放
- 修改隐藏字段：`?is_admin=0` → `1`
- 前端按钮被隐藏（CSS/JS）并不代表后端接口有权限校验
- `403 / 401` 响应上尝试：
  - 改 HTTP 方法：`GET` → `POST` / `PUT`
  - 加 Header：`X-Original-URL: /admin`、`X-Forwarded-For: 127.0.0.1`
  - 改 Referer 为管理后台 referer

## 3. IDOR（Insecure Direct Object Reference）

- `?file=../../etc/passwd`（路径遍历）
- `?download_id=123` → 121、122、123 依次读取（可批量遍历）
- 接口响应中包含用户其他信息（手机号、邮箱、身份证号）
- WebSocket / 长连接推送：未对订阅资源做权限校验

## 4. 业务逻辑漏洞

### 4.1 支付 / 金额 / 优惠券

- 改价格：`price=9999` → `price=0.01` 或负数
- 改数量：`count=1` → `count=-1`（积分返还）
- 多次使用同一优惠券 / 同一邀请奖励
- 支付回调篡改：将 `?state=fail` 改为 `?state=success`
- 重复提交订单造成多次奖励（竞态条件 / 并发扣减）

### 4.2 登录 / 注册 / 找回密码

- 用户名枚举：「用户不存在」/「密码错误」差异提示
- 注册已存在的用户名或特殊字符（`admin--`、`admin%00`、` admin `）
- 找回密码：验证码爆破、跨账号重置（`?user=admin` 重置他人密码）
- 登录验证码复用、无次数限制、可在前端判断
- 记住我 / Token 泄漏 / 会话永不过期

### 4.3 短信 / 邮件轰炸

- 无频率限制的发送接口：可批量轰炸他人
- 绕过频率限制：加 `X-Forwarded-For` / 不同邮箱、在手机号后加空格/0

### 4.4 验证码绕过

- 验证码在响应中明文下发
- 验证码可复用（不失效）
- 验证逻辑在前端 JavaScript 判断
- 改空验证码 `""` 或 `0000`

### 4.5 接口并发 / 竞态

- 并发提交同一提现请求 → 多次出款
- 并发领取优惠券 → 超库存
- 并发投票 / 抽奖 → 违规刷票

## 5. 建议的自动化手段

- **Burp Autorize / Authz** 插件：抓取高权限请求，用低权限会话重放，对比响应长度
- **Param Miner**：探测隐藏参数
- **自定义脚本**：批量替换 ID 字段并验证响应是否包含他人数据

## 6. 测试 Checklist

- [ ] 所有 ID 参数是否校验归属
- [ ] 管理接口是否有权限中间件
- [ ] 金额、数量、价格等字段是否校验合理范围
- [ ] 支付状态、订单状态是否由回调的参数决定
- [ ] 验证码是否服务端校验且一次性
- [ ] 用户枚举、短信轰炸是否有速率限制
- [ ] 并发提交是否加锁
- [ ] 前端隐藏的功能按钮，后端是否真的拦截
- [ ] 上传的文件名、路径、内容类型是否有严格校验
- [ ] 找回密码 / 邀请 / 分享链接是否绑定用户与时效

---

> 逻辑漏洞往往无法被自动化扫描器发现，必须由经验丰富的测试人员人工验证。在报告中对逻辑漏洞给出清晰的业务影响说明非常关键。
