## 一、古典 vs 现代密码学

### 1.1 核心区别

| 对比维度 | 古典密码学 | 现代密码学 |
| :--- | :--- | :--- |
| **代表算法** | 凯撒、维吉尼亚、栅栏、单表/多表替换 | AES、RSA、ECC、SHA-2 |
| **处理单位** | 以字符为单位 | 以比特/字节为单位 |
| **安全基础** | 依赖算法保密 | Kerckhoffs 原则：算法公开，密钥保密 |
| **数学基础** | 简单替换/置换 | 数论、代数、信息论 |
| **密钥管理** | 简单或无密钥 | 严格的密钥生成、分发、存储、销毁 |

> **🔑 高分考点**：Kerckhoffs 原则——"密码系统的安全性应仅依赖于密钥的保密，而不应依赖于算法的保密"。这是现代密码学的核心设计原则。

### 1.2 现代密码学三大分支

| 功能 | 典型算法 | 核心目标 | 密钥特点 |
| :--- | :--- | :--- | :--- |
| **对称加密** | AES、ChaCha20、SM4 | 机密性（加解密） | 加解密使用同一密钥 |
| **非对称加密** | RSA、ECC、ElGamal | 密钥交换 / 公钥加密 / 签名 | 公钥+私钥配对 |
| **哈希函数** | SHA-2、SHA-3、BLAKE3 | 完整性、摘要、指纹 | 无密钥 |

加上 **MAC/HMAC**（消息认证）与 **数字签名**，共同提供机密性、完整性、认证与不可否认性。

## 二、对称加密深度解析

### 2.1 分组密码 vs 流密码

```
分组密码（Block Cipher）：
  - 按固定长度（如 128-bit / 16 字节）处理数据
  - 代表：AES（128-bit 块）、DES（64-bit 块）、SM4（128-bit 块）
  - 需要工作模式（ECB/CBC/CTR/GCM 等）处理多块数据

流密码（Stream Cipher）：
  - 按位/字节加密，逐位处理
  - 代表：ChaCha20、RC4（已废弃）
  - AES-CTR 本质是用分组密码构造的流密码
```

### 2.2 工作模式详解（以 AES 为例）

| 模式 | 全称 | 原理 | 安全性 | 并行性 | 推荐度 |
| :--- | :--- | :--- | :--- | :---: | :---: |
| **ECB** | Electronic Codebook | 每块独立加密 | ❌ 不安全：相同明文→相同密文 | ✅ | 🚫 禁用 |
| **CBC** | Cipher Block Chaining | 前一块密文与当前明文 XOR 后加密 | ⚠️ 需安全随机 IV，padding oracle 风险 | ❌ 解密可并行 | ⚠️ 谨慎 |
| **CFB** | Cipher Feedback | 流密码模式，前密文块加密后与明文 XOR | ⚠️ 类似 CBC | ❌ | ⚠️ 谨慎 |
| **OFB** | Output Feedback | 流密码模式，连续加密 IV 产生密钥流 | ⚠️ IV 不可重用 | ❌ | ⚠️ 谨慎 |
| **CTR** | Counter | 计数器加密后与明文 XOR | ✅ IV/nonce 不可重用 | ✅ | ✅ 推荐 |
| **GCM** | Galois/Counter Mode | CTR + GHASH 认证 | ✅ AEAD（认证加密） | ✅ | ⭐ 首选 |
| **CCM** | CBC-MAC + CTR | 类似 GCM，但非流水线 | ✅ AEAD | ❌ | ✅ 可用 |

> **🔑 高分考点**：
> - ECB 模式**绝对不能用于生产**——相同明文块产生相同密文块，模式可见（企鹅图例）
> - AEAD（Authenticated Encryption with Associated Data）= 加密 + 认证 + 完整性校验，一次完成
> - GCM 的 nonce **绝对不能重用**——同一密钥+同一 nonce 会泄露认证密钥与明文异或

### 2.3 填充（Padding）

```
为什么需要填充：
  分组密码要求明文长度是块长度的整数倍
  AES 块 = 128 bits = 16 bytes
  如果明文长度不是 16 的倍数，需要填充

PKCS#7 填充（最常用）：
  需要 k 个字节 → 填充 k 个值为 k 的字节
  
  示例（AES 16 字节块）：
    明文 13 字节 → 填充 3 字节：0x03 0x03 0x03
    明文 16 字节 → 需要填充整个块：16 个 0x10
    明文 15 字节 → 填充 1 字节：0x01

Padding Oracle 攻击：
  利用 CBC 模式下服务器返回的"填充错误"信息
  → 可逐字节解密密文（不需要密钥！）
  → 防御：使用 AEAD（GCM/ChaCha20-Poly1305）避免此问题
```

### 2.4 密钥长度与算法选择

| 算法 | 密钥长度 | 块大小 | 状态 | 推荐场景 |
| :--- | :--- | :--- | :--- | :--- |
| **AES-128** | 128 bit | 128 bit | ✅ 安全 | 通用首选 |
| **AES-256** | 256 bit | 128 bit | ✅ 安全 | 合规/高安全需求 |
| **ChaCha20** | 256 bit | 流密码 | ✅ 安全 | 移动设备/无 AES 硬件加速 |
| **DES** | 56 bit | 64 bit | ❌ 已破解 | 禁止使用 |
| **3DES** | 168/112 bit | 64 bit | ❌ 不推荐 | 禁止新系统使用 |
| **RC4** | 可变 | 流密码 | ❌ 已废弃 | 禁止使用 |
| **SM4** | 128 bit | 128 bit | ✅ 安全 | 中国商用密码标准 |

> **💡 知识巧记**：**"AES-128 够安全、256 为合规、GCM 是首选、ECB 永不用"**

## 三、非对称加密 / 公钥密码学

### 3.1 核心概念

```
密钥对（公钥 + 私钥）：
  - 公钥：可公开发布，用于加密或验证签名
  - 私钥：必须保密，用于解密或生成签名
  - 数学关系：从公钥推导私钥在计算上不可行

三大应用场景：
  1. 公钥加密：用接收方公钥加密 → 仅接收方私钥能解密
     算法：RSA-OAEP、ECIES（Elliptic Curve Integrated Encryption Scheme）
  
  2. 数字签名：用发送方私钥签名 → 任何人用发送方公钥验证
     算法：RSA-PSS、ECDSA、EdDSA（Ed25519）
  
  3. 密钥协商：双方在不安全信道上协商共享密钥
     算法：DH（Diffie-Hellman）、ECDH
```

### 3.2 RSA 详解

```
数学基础：大整数分解难题（Factoring Problem）
  - 选两个大素数 p, q → n = p × q
  - 公钥：(n, e)，私钥：(n, d)
  - 加密：c = m^e mod n
  - 解密：m = c^d mod n
  - 安全假设：已知 n，难以分解出 p 和 q

密钥长度与安全级别：
  1024 bit → 约 80 bit 安全（已不安全）
  2048 bit → 约 112 bit 安全（当前最低推荐）
  3072 bit → 约 128 bit 安全
  4096 bit → 约 140 bit 安全（高安全需求）

RSA 使用要点：
  ✗ 不能直接加密长明文（慢 + 长度限制）
  ✓ 加密随机会话密钥 → 用对称密钥加密数据（混合加密）
  ✗ 教科书式 RSA（无 padding）不安全
  ✓ 必须使用 OAEP（加密）或 PSS（签名）padding
```

### 3.3 ECC（椭圆曲线密码学）

```
数学基础：椭圆曲线上离散对数问题（ECDLP）
  - 给定点 P 和 Q = k×P，求 k 在计算上不可行
  - 更短的密钥达到同等安全级别

安全级别对比：
  对称加密    RSA        ECC
  80 bit     1024 bit   160 bit
  128 bit    3072 bit   256 bit  ← 当前推荐
  256 bit    15360 bit  512 bit

常用曲线：
  NIST P-256 / P-384 / P-521  → 美国政府标准
  Curve25519（X25519）        → ECDH 密钥交换，广泛使用
  Ed25519                     → EdDSA 签名，SSH/TLS 1.3 首选
  
EdDSA（Ed25519）优势：
  - 确定性签名（不需要随机数）
  - 实现简单，不易出错
  - 高性能，抗侧信道
  - TLS 1.3、SSH、Signal 协议广泛采用
```

### 3.4 Diffie-Hellman 密钥协商

```
DH 原理（简化版）：
  公开参数：大素数 p，生成元 g
  
  Alice                           Bob
  选私钥 a                        选私钥 b
  计算 A = g^a mod p             计算 B = g^b mod p
          --- A --->
          <--- B ---
  共享密钥 K = B^a mod p         共享密钥 K = A^b mod p
  = g^(ab) mod p                 = g^(ab) mod p

安全属性：
  - 窃听者只能看到 g^a 和 g^b，无法计算 g^(ab)（离散对数难题）
  - 但不提供身份认证 → 中间人可替换公钥 → 需配合数字签名

ECDHE（Ephemeral ECDH）：
  - 使用椭圆曲线的 DH
  - 每次会话生成新的临时密钥对（Ephemeral）
  - 提供前向安全（Forward Secrecy）
  - TLS 1.3 仅支持 ECDHE/DHE，移除了 RSA 密钥交换
```

> **🔑 高分考点**：前向安全（Forward Secrecy）——即使服务器长期私钥泄露，历史会话的加密数据也无法被解密。关键是**每次握手使用临时密钥**（ECDHE/DHE），会话结束后销毁。RSA 密钥交换**不提供**前向安全。

## 四、哈希函数详解

### 4.1 哈希函数的三大安全属性

| 属性 | 英文 | 含义 | 破坏后果 |
| :--- | :--- | :--- | :--- |
| **抗原像** | Preimage Resistance | 给定哈希值 h，找不到 m 使 H(m)=h | 密码哈希被破解 |
| **抗第二原像** | Second Preimage Resistance | 给定 m1，找不到 m2 使 H(m1)=H(m2) | 文件签名伪造 |
| **抗碰撞** | Collision Resistance | 找不到任意 m1≠m2 使 H(m1)=H(m2) | 数字证书伪造 |

### 4.2 常见哈希算法

| 算法 | 输出长度 | 安全状态 | 说明 |
| :--- | :--- | :--- | :--- |
| **MD5** | 128 bit | ❌ 已破解（碰撞） | 仅可用于非安全场景（如校验和） |
| **SHA-1** | 160 bit | ❌ 已破解（选择前缀碰撞） | SHAttered 攻击，2020 年后不推荐 |
| **SHA-256** | 256 bit | ✅ 安全 | SHA-2 家族，最常用 |
| **SHA-384** | 384 bit | ✅ 安全 | SHA-2 家族，兼容 128 bit 安全 |
| **SHA-512** | 512 bit | ✅ 安全 | SHA-2 家族，64 位平台性能好 |
| **SHA-3-256** | 256 bit | ✅ 安全 | Keccak 结构，与 SHA-2 数学基础不同 |
| **BLAKE3** | 可变 | ✅ 安全 | 高性能，现代替代方案 |

> **🔑 高分考点**：MD5 和 SHA-1 **已被破解**，存在碰撞攻击。SHA-1 的 SHAttered 攻击（2017 年）证明了选择前缀碰撞的可行性。任何安全场景都应使用 SHA-256 及以上。

### 4.3 哈希的应用场景

```
应用场景：
  1. 文件完整性校验：下载文件后比对哈希值
  2. 密码存储：加盐 + 慢速哈希（bcrypt/Argon2），非 SHA-256！
  3. 数字签名：对消息哈希值签名（而非原始消息）
  4. 区块链：工作量证明、Merkle Tree
  5. HMAC：基于哈希的消息认证码
  6. 去重：通过哈希值判断内容是否重复
  7. Git：commit hash、对象标识
```

## 五、消息认证与数字签名

### 5.1 MAC / HMAC

```
MAC（Message Authentication Code）：
  - 用密钥 + 消息生成固定长度标签
  - 验证方用同一密钥验证
  - 提供完整性 + 来源认证

HMAC（Hash-based MAC）：
  HMAC(K, m) = H((K ⊕ opad) ‖ H((K ⊕ ipad) ‖ m))
  - opad = 0x5c5c...（外部填充）
  - ipad = 0x3636...（内部填充）
  - 常用：HMAC-SHA256、HMAC-SHA512

加密 vs 认证：
  仅加密 ≠ 消息完整 → 密文可被篡改
  ✓ AEAD（GCM/ChaCha20-Poly1305）= 加密 + 认证，一次完成
  ✓ Encrypt-then-MAC = 先加密，再对密文做 MAC（推荐模式）
```

### 5.2 数字签名

```
数字签名原理：
  1. 签名者用私钥对消息哈希值进行"签名"
  2. 验证者用签名者的公钥验证签名
  3. 提供：完整性 + 来源认证 + 不可否认性

常见算法：
  RSA-PKCS#1 v1.5  → 传统，仍有使用（不推荐新系统）
  RSA-PSS           → RSA 签名推荐方案
  ECDSA             → 椭圆曲线数字签名（需安全随机数！）
  EdDSA (Ed25519)   → ★ 现代首选，确定性签名

ECDSA 的随机数风险：
  ECDSA 签名需要随机数 k（nonce）
  如果 k 重复使用或可预测 → 私钥可以被恢复
  Sony PS3（2010 年）因固定 k 值导致私钥泄露
  EdDSA 使用确定性签名，避免此问题
```

## 六、密码学在真实协议中的应用

### 6.1 TLS 1.3 中的密码学

```
TLS 1.3 握手密码学流程：
  1. 客户端发送：随机数 + 支持的套件 + ECDHE 公钥
  2. 服务器回应：随机数 + ECDHE 公钥 + 证书（含签名公钥）
     + 用证书私钥对握手消息签名
  3. 双方验证证书链 → 计算 ECDH 共享密钥 → 派生会话密钥
  4. 应用数据使用 AEAD（AES-128-GCM 或 ChaCha20-Poly1305）加密

TLS 1.3 强制要求：
  ✓ 仅 AEAD 对称加密（无 CBC）
  ✓ 仅 ECDHE/DHE 密钥交换（无 RSA 密钥交换）
  ✓ 前向安全为默认
  ✓ 握手消息加密（ServerHello 之后）
```

### 6.2 SSH 中的密码学

```
SSH 连接建立：
  1. 密钥协商：Diffie-Hellman（或 ECDH）协商会话密钥
  2. 主机认证：服务器用主机私钥签名协商消息
     客户端比对 known_hosts 中的公钥
  3. 用户认证：
     - password：明文密码通过加密通道传输
     - publickey：客户端用私钥签名挑战值

推荐算法：
  密钥交换：curve25519-sha256
  主机密钥：ssh-ed25519
  对称加密：chacha20-poly1305@openssh.com, aes256-gcm@openssh.com
```

### 6.3 密码存储（应用层）

```
✗ 错误做法：
  明文存储                    → 数据库泄露即密码泄露
  MD5/SHA-256 无盐             → 彩虹表攻击
  SHA-256 + 固定盐             → 可并行暴力破解
  单次 SHA-256 迭代            → GPU 暴力破解可达 10^9 次/秒

✓ 正确做法：
  使用专门的密码哈希算法（慢哈希 + 自适应）：
  
  Argon2id（★ 首选，PHC 比赛冠军）：
    - 参数：内存 m、迭代 t、并行度 p
    - 抗 GPU/ASIC/FPGA
    - 内存困难函数
  
  bcrypt（传统推荐）：
    - 基于 Blowfish
    - cost 参数调节计算量（指数增长）
    - 自动加盐
  
  scrypt（内存困难）：
    - 抗 GPU 暴力破解
    - 可同时调节 CPU 和内存成本
  
  PBKDF2-HMAC-SHA256（兼容性好）：
    - HMAC 迭代
    - 需足够大的迭代次数（建议 100000+）

推荐参数（2025 年）：
  Argon2id: m=64MB, t=3, p=4
  bcrypt: cost=12
  PBKDF2: iterations=600000
```

## 七、密码存储最佳实践

### 7.1 密码存储检查清单

```
□ 使用 Argon2id > bcrypt > scrypt > PBKDF2
□ 每个密码使用独立的随机盐（16+ 字节）
□ 盐与哈希一起存储（盐不是秘密）
□ 足够高的计算参数（目标：验证耗时 0.5-1 秒）
□ 版本字段：记录算法和参数，便于未来升级
□ 哈希后再加密（如 HSM/ KMS 加密哈希值）
□ 传输层使用 HTTPS（防止中间人窃取）
□ 实施速率限制（防暴力破解）
□ 监控异常登录模式
□ 支持多因素认证（MFA）
```

## 八、常见误用与陷阱

| 错误做法 | 问题 | 正确做法 | 严重度 |
| :--- | :--- | :--- | :---: |
| ECB 模式加密 | 相同明文块→相同密文块，模式可见 | 使用 CBC/CTR/GCM | ⭐⭐⭐⭐⭐ |
| 固定/可预测 IV | 破坏语义安全，选择明文攻击 | 每次加密使用安全随机 IV | ⭐⭐⭐⭐⭐ |
| 仅加密不做认证 | 密文可被篡改，padding oracle | 使用 AEAD 或 Encrypt-then-MAC | ⭐⭐⭐⭐⭐ |
| 弱随机数生成 | 密钥/IV/nonce 可被预测 | 使用系统 CSPRNG | ⭐⭐⭐⭐⭐ |
| GCM nonce 重用 | 泄露认证密钥和明文异或 | 随机 96-bit nonce，保证不重复 | ⭐⭐⭐⭐⭐ |
| RSA 直接加密长明文 | 速度慢，长度受限 | 混合加密：RSA + 对称会话密钥 | ⭐⭐⭐⭐ |
| MD5/SHA-1 签名哈希 | 存在碰撞，可伪造签名 | SHA-256 及以上 | ⭐⭐⭐⭐ |
| 密码直接 SHA-256 存库 | 彩虹表/GPU 暴力破解 | bcrypt/Argon2 + salt | ⭐⭐⭐⭐⭐ |
| 自研加密算法 | 难以抵御高级攻击 | 使用经过审计的标准库 | ⭐⭐⭐⭐⭐ |
| 硬编码密钥 | 代码泄露即密钥泄露 | KMS/HSM/Vault 密钥管理 | ⭐⭐⭐⭐⭐ |
| 密钥通过不安全通道传输 | 密钥被窃听 | DH 密钥协商/非对称加密传输 | ⭐⭐⭐⭐ |
| 不使用证书验证 | 中间人攻击 | TLS 证书链验证 + Certificate Pinning | ⭐⭐⭐⭐ |

## 九、侧信道与后量子密码学

### 9.1 侧信道攻击

```
侧信道攻击类型：
  - 时序攻击（Timing）：通过运算时间推断密钥位
  - 功耗分析（Power Analysis）：通过功耗曲线推断密钥
  - 电磁辐射（EM）：通过电磁信号推断
  - 缓存攻击（Cache）：Spectre/Meltdown 等
  - 声学攻击：通过设备声音推断

防御措施：
  ✓ 恒定时间算法（Constant-time）
  ✓ 掩码技术（Masking）
  ✓ 硬件安全模块（HSM/TPM/Secure Enclave）
  ✓ 随机化处理顺序
```

### 9.2 后量子密码学（PQC）

```
威胁模型：
  Shor 算法可在大型量子计算机上破解 RSA/ECC
  Grover 算法可将暴力破解复杂度降低为平方根
  → AES-256 在量子计算下等效 128 bit 安全（仍可接受）
  → RSA/ECC 将完全被破解

NIST PQC 标准化算法：
  公钥加密/KEM：
    CRYSTALS-Kyber（格密码）
  
  数字签名：
    CRYSTALS-Dilithium（格密码）
    FALCON（格密码）
    SPHINCS+（哈希签名，保守备用）

现状：
  TLS 1.3 已开始试验性部署 Kyber
  Signal、Apple iMessage 已引入 PQC 保护
  建议：开始规划 PQC 迁移，但暂不需要急于替换现有 RSA/ECC
```

### 💡 知识巧记口诀

#### 1. 三大密码学分支
> **"对称加密保机密、非对称解决密钥分发、哈希做摘要和签名"**

#### 2. AES 工作模式
> **"ECB 禁用、CBC 需 IV、CTR 计数器、GCM 最推荐"**
>
> 安全口诀：**"GCM 是首选，ECB 永不用，CBC 防 Padding Oracle"**

#### 3. 前向安全
> **"ECDHE 临时密钥，泄露长期私钥也不怕——会话密钥用过就扔"**

#### 4. 密码存储
> **"Argon2 是冠军、bcrypt 是老将、PBKDF2 是备胎——SHA-256 存密码是外行"**
>
> 三要素：**"慢哈希 + 随机盐 + 高迭代"**

#### 5. 哈希安全
> **"MD5 已死、SHA-1 已废、SHA-256 是主流、SHA-3 是备胎"**

#### 6. TLS 1.3 密码学
> **"只用 AEAD 加密、只用 ECDHE 协商、握手消息全加密"**

#### 7. 数字签名算法选择
> **"Ed25519 最好、ECDSA 还行（但要随机数）、RSA-PSS 可以用、RSA-PKCS1v1.5 过时了"**

#### 8. 常见误用
> **"ECB 是新手杀手、硬编码密钥是自掘坟墓、自研算法是安全黑洞"**

### 🎯 考试陷阱提醒

| 陷阱 | 正确理解 |
| :--- | :--- |
| "AES-256 比 AES-128 更安全" | ⚠️ 不完全正确！在当前计算能力下，AES-128 已足够安全。256 主要考虑合规和抗量子 |
| "RSA 2048 和 ECC P-256 安全级别不同" | ❌ 错误！两者等效约 128 bit 安全级别 |
| "哈希可以用于加密" | ❌ 错误！哈希是单向函数，不可逆 |
| "使用 HTTPS 就是端到端加密" | ❌ 错误！HTTPS 是传输层加密，CDN/反向代理可以解密 |
| "数字签名 = 加密的哈希值" | ❌ 不完全！签名不等于"用私钥加密哈希"，ECDSA/EdDSA 根本不是加密操作 |
| "SHA-256 存密码就够了" | ❌ 错误！SHA-256 太快，易被 GPU 暴力破解。必须用 bcrypt/Argon2 |
| "base64 是加密算法" | ❌ 错误！base64 是编码（Encoding），不是加密（Encryption），无密钥 |
| "量子计算机已经可以破解所有密码" | ❌ 错误！目前尚无足够大的量子计算机。对称加密加倍密钥长度即可抵抗 |

## 学习建议

1. 📖 **入门书籍**：《深入浅出密码学》或《现代密码学：原理与协议》
2. 💻 **动手实践**：用 Python cryptography 库实现 AES-GCM 加解密、Ed25519 签名、bcrypt 哈希
3. 🔬 **对比测试**：用 `openssl speed` 对比不同算法与套件性能
4. 📜 **阅读 RFC**：TLS 1.3 RFC 8446、NIST SP 800-52r2
5. 🔍 **跟踪漏洞**：关注 CVE（Heartbleed、ROCA、Curveball、Terrapin 等），理解真实协议漏洞
