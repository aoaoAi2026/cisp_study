## 一、电子政务密码应用

### 1.1 政务密码应用场景

```
政务信息化密码应用全景：

电子证照：
  签发：CA机构用SM2签名 → 电子证照 = 数据 + SM2数字签名
  验证：公众端验证SM2签名 → 确认证照真实/未被篡改

电子印章：
  《电子签名法》电子印章 = 电子签名 + 可视化图章
  国密实现：
    印章数据 → SM3哈希 → SM2私钥签名 → 盖章完成
    验章 → SM2公钥验签 → SM3哈希比对

电子公文：
  发文：SM2签名(发文机关)+SM2签名(经办人)
  加密传输：SM4(TLCP加密通道)
  归档：SM2签名 + SM3完整性校验值

政务数据共享：
  跨部门数据交换：
    → 国密TLCP安全通道
    → 数据SM4加密 + SM2数字信封
    → 共享接口SM2签名认证(来源真实性)
```

## 二、金融密码应用

### 2.1 金融密码体系

```
金融行业密码标准：

PBOC (中国人民银行) 系列：
  PBOC 3.0 (最新) — 金融IC卡规范
  └── SM系列算法已纳入PBOC 3.0
  └── 金融IC卡 → 国密芯片 + SM2/SM3/SM4

银联：
  UPI (银联) 标准 → 支持SM系列
  双算法支持：SM2+RSA并行

网联：
  网联清算平台 → 支持国密
  线上支付清算 → SM2签名/SM4加密通道

CFCA (中国金融认证中心)：
  国密证书签发 → 金融机构服务器证书/客户端证书
```

### 2.2 数字人民币密码

```
数字人民币(e-CNY) 密码体系：

核心技术：
  1. 数字人民币APP → SM2密钥对(每用户)
  2. 交易签名 → SM2数字签名(交易发起方)
  3. 通信加密 → SM4(TLCP通道 → 央行/运营机构)
  4. 账本保护 → SM3(哈希链) + SM2(运营机构签名)

"可控匿名"的密码实现：
  - 对交易对手匿名(使用一次性SM2地址)
  - 对央行完全可追溯(密钥托管+监管密钥)
  
安全硬件：
  - 手机内置SE(Secure Element)存储SM2私钥
  - 或云端HSM托管(软钱包方案)
```

### 2.3 金融密码机部署

```
金融数据密码机典型部署：

ATM/POS → 前置服务器 → 密码机(HSM)：
  1. PIN 加密 → SM4加密(个人密码)
  2. MAC 计算 → SM3-HMAC(交易完整性)
  3. 密钥管理 → ZMK(主密钥) → ZPK(普通密钥) 层次化

金融密码机高可用：
  主密码机 + 备密码机 (Active-Standby)
  → 密钥手动同步(安全考虑，不自动同步)
  → 心跳检测 + 自动切换
  → 切换时重新加载密钥(GmbH PKCS#11接口)
```

## 三、密评典型整改案例

### 案例1: 某政务网站国密改造

```
改造前（不满足密评）：
  - HTTPS使用标准TLS(RSA+AES-GCM+SHA256)
  - 登录密码MD5存储
  - 日志无完整性保护

改造方案：
  1. Nginx → 铜锁版Nginx (TLS+TLCP双栈)
  2. CA证书 → CFCA国密SM2双证书
  3. 密码存储 → SM3(密码+随机盐)
  4. 日志完整性 → ELK日志系统增加SM3-HMAC字段
  5. 重要操作 → 增加SM2数字签名(审批流)
  
改造周期：4-6周
密评结果：通过 (分数85+，高风险项清零)
```

### 案例2: 某银行支付系统改造

```
改造前：
  - 国密+国际算法混用
  - 密码机型号证书临近过期
  - 密钥管理不规范

改造方案：
  1. 密码机升级+型号证书续期
  2. 全链路国密：APP端→TLCP→前置→加密机SM4→核心
  3. KMS密钥管理系统部署
  4. 密钥定期轮换自动化(每季度→自动+审批)

改造周期：8-12周
结果：密评通过+人行现场检查通过
```
