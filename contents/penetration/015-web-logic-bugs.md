# Web 逻辑漏洞挖掘实战

> 与 "传统注入 / XSS / CSRF 等"已知"漏洞相比，业务逻辑漏洞往往不依赖特定技术栈，藏在业务流程的各种"角落"里——它们不触发 WAF、也不会被 SQL 语法特征匹配到，但其危害往往最直接影响业务核心（如任意用户密码重置、绕过支付、金额篡改、优惠券滥用等）。

## 1. 业务逻辑漏洞的本质

业务逻辑漏洞是"系统按设计执行了某些操作，但**在特定条件下被攻击者可以绕过或滥用业务规则**。它们不依赖特定语言或框架，所以也最难被自动化工具发现——需要人工梳理业务流程与权限模型，手动测试。

典型的业务逻辑漏洞地图：

| 大类 | 代表漏洞 |
|------|--------|
| **越权访问 (IDOR | 水平越权、垂直越权、IDOR |
| **认证与会话** | 密码重置逻辑缺陷、会话固定、弱会话令牌 |
| **支付与交易** | 价格篡改、优惠券滥用、积分操纵 |
| **竞争条件 (Race Condition)** | 优惠券 / 库存扣减 / 提现超额 |
| **验证码 / OTP** | 验证码复用、爆破、复用、绕过 |
| **短信 / 邮件轰炸** | 短信轰炸、邮件轰炸 |
| **注册 / 登录** | 注册重复注册、邮箱验证绕过、邀请码滥用 |
| **两步验证** | 2FA 绕过、恢复令牌劫持 |
| **文件 / 内容** | 文件上传后处理逻辑缺陷 |
| **API / 接口** | 接口越权、接口未授权、批量调用 |

## 2. 越权访问（IDOR / 水平越权 / 垂直越权）

**越权漏洞的本质**：A 用户使用自己的合法身份，去访问 / 修改 / 删除 **B 用户的资源**。核心是 "资源标识符（ID / uid / orderId / userId ）未做所属校验缺失。

### 2.1 测试步骤

```
1. 注册两个账号：user_a / user_b
2. user_a 创建 / 获取资源 X
3. user_b 登录，观察浏览器开两个浏览器 / 使用 user_a / user_b 交替操作
4. 观察请求 user_a 修改 / 查询 / 删除请求中，替换 user_a 对应资源的请求参数（如 id=1001 → id=1002）
5. 或使用 user_b 的身份 token 去访问 user_a 的资源
6. 是否可以成功/修改成功 → 存在越权
```

### 2.2 常见参数位置

```
?user_id=1001              → 改为 1002 (他人)
/api/user/1001                  → /api/user/1002
/order?orderNo=A1234        → A1235
POST { "uid": 1001 }             → { "uid": 1002 }
Cookie: user=1001                  → Cookie: user=1002
X-User-Id: 1001                 → 1002
/attachment/download?fileId=999        → 枚举 fileId 998 / 1000
/api/v2/info (不检查权限，但 /api/v1/info 检查) → 版本差异
```

### 2.3 IDOR 绕过技巧

- **改变请求方法：`GET /api/order/1234` 被拒，而 `POST /api/order/1234` / `DELETE /api/order/1234` 却通过
- **改路径变体**：`/api/order/1234` 被拒，但 `/api/order/1234/` / `/api/order/1234/.json` 可能漏过
- **改 Content-Type**：`application/json` 改为 `application/x-www-form-urlencoded`
- **改 HTTP Header**：增加 `X-HTTP-Method-Override: DELETE`
- **改 Referer**：Referer 包含正确 / 去掉
- **参数重复 /api/v2
- **GUID 猜测**：顺序自增 ID / UUID v1 vs v4：16 进制自增 ID 被猜到后，尝试批量枚举
- **接口差异**：后台接口 /api/admin/orders/admin 检查权限，/api/user/orders/admin 不检查

## 3. 密码重置逻辑漏洞

### 3.1 经典案例

```
1. 忘记密码 → 输入邮箱 → 后端生成 reset_token
2. 前端返回 reset_token 到响应包（而非仅邮件）
3. 攻击者用自己账户请求密码重置
4. 将返回 reset_token 替换为他人的账户 ID，带入后续重置链接
5. 即可重置他人密码
```

### 3.2 常见缺陷形式

| 缺陷 | 测试方法 |
|------|---------|
| **Token 泄漏到前端 | 抓包观察是否在响应 / Cookie / HTML 中返回 reset_token |
| **Token 可预测** | 连续申请多个 reset_token，观察规律 (时间戳、自增、弱随机 |
| **Token 不绑定用户** | 申请攻击者的 token 去重置他人密码 |
| **Token 无过期 / 可多次使用** | 申请后未过期时再次使用 |
| **邮箱枚举** | `email=victim@a.com&email=attacker@a.com 两个邮箱（两次响应差异** |
| **Host 头投毒** | 请求头 Host: evil.com，邮件链接包含 evil.com |
| **密码重置链接 / OTP 验证码** | 短信 OTP 无次数限制，可爆破 4~6 位 OTP 爆破 |
| **用户名枚举** | 注册 / 登录 / 忘记密码接口是否泄漏是否存在 |

## 4. 支付 / 交易 / 订单逻辑漏洞

### 4.1 价格篡改

```
POST /api/order
{
  "product_id": 1,
  "price": 100.00,     → 攻击者改为 0.01 或 -1
  "amount": 1
}
后端只检查库存数量 ×
```

### 4.2 数量与金额负数值负数

```
POST /api/order        → 商品数量设为 -1，金额设为 -100
{
  "product_id": 1,
  "amount": -100         → 负数值造成总价 = 单价 × 数量 = 负数
}
```

### 4.3 优惠券 / 折扣码 重复使用

```
1. 多次提交同一张优惠券 → 未校验已使用状态
→ 若后端未正确扣减库存 / 优惠券状态
```

### 4.4 退款 / 退货逻辑

```
下单 → 支付 → 立即退款 → 继续使用优惠码 / 礼品卡

→ 服务仍然可用
```

## 5. 竞争条件（Race Condition）

利用后端处理过程中，**同一请求并发提交多次，**在完成前并发提交**：

```
1. 优惠券扣减库存 → 200 并发提交 200 × 优惠券在**同一优惠券**同一**多个**
→ 结果：同一张优惠券被使用 200 次 / 同一积分被扣除 200 次

200 ×

典型场景：

- 优惠券 / 积分扣除
- 礼品卡 / 充值 / 提现
- 库存扣减
- 抽奖 / 红包 / 签到奖励发放

### 5.1 测试方法

使用 Burp Intruder ** Pitchfork + Grep - Extract 并发：

1. 用 Burp 的 `Do 同一请求并发提交 20-50 次，观察响应是否多次成功次数 > 1

```bash
# 使用并发工具
for i in {1..50}; do curl -X POST "http://target/api/coupon/use -H "Cookie: session=xxx" -d '{"coupon":"ABC123"}' & done
wait
```

## 6. 短信 / 邮件轰炸

- **短信轰炸**：注册 / 登录 / 忘记密码 / 通知等接口，**参数 `mobile` 后可多次提交未做频率限制
  - 对策：图形验证码、IP+手机号 频控、短信内容校验、频控粒度为"分钟级
- **邮件轰炸**：类似，邮件地址同样思路

## 7. 验证码绕过 / 2FA 绕过

```
1. 图形验证码**：
   - 验证码未刷新：`captcha=1234` 可多次使用
   - 验证码在前端**多次使用：攻击者多次提交

2. **短信 OTP**：
   - OTP 无**未过期时间，可多次爆破 / 无次数限制
   - OTP 长度较短（4 位），未做频控，可暴力枚举
   - OTP 多次提交后仍有效

3. **2FA 绕过**：
   - 直接访问 `step3（绕过 step2 的 2FA 校验 step）
   - `session` 中已验证 2FA → 直接访问其他页面
   - 把 2FA 的 TOTP 无时间戳 ×
```

## 8. 注册 / 登录逻辑漏洞

```
1. **邀请码机制**
   - 邀请码被硬编码在前端 / 邀请码无过期 / 可多次使用
   - 邀请码未与邀请码为空时，攻击者可以任意注册

2. **注册后自动登录**
   - 注册后自动设置 session，导致攻击者可以直接登录他人账户

3. **邮箱验证**
   - 邮箱验证链接未过期 → 攻击者可以用已验证邮箱×其他用户注册相同邮箱

4. **登录后 2FA**
   - 登录后自动跳转到 2FA 页面 / 此时 session 已登录
   - 直接访问其他页面绕过 2FA

5. **会话固定（Session Fixation）
   - 攻击者先访问登录前获取 sessionId → 诱导受害者登录 → 攻击者仍持有 sessionId
```

## 9. 密码策略与密码重置链接

1. **密码重置链接未过期 / 未绑定 IP / 浏览器
2. **重置链接多次使用
3. **重置链接在重置 / 密码修改后旧 session 未失效

## 10. Web 逻辑漏洞挖掘方法清单（实战清单

### 10.1 核心流程清单

```
注册 → 登录 → 2FA → 忘记密码 → 重置密码 → 下单 → 支付 → 发货 → 退款 → 提现 → 抽奖 → 发奖

| 流程 | 测试要点 |
|------|---------|
| 注册 | 邀请码 / 邮箱验证 / 短信 OTP / 图形验证码 / 是否可重复注册同一邮箱 |
| 登录 | 密码错误次数限制 / 登录后 session 是否固定 / 2FA 绕过 / 凭据是否可被暴力破解 |
| 忘记密码 | Token 泄漏 / 可预测 / 绑定用户 / 无过期 / 邮箱投毒 / 用户名枚举 |
| 下单 / 支付 | 价格篡改 / 数量负数 / 优惠券重复 / 库存并发 / 支付状态篡改 / 金额为 0 绕过 |
| 积分 / 钱包 | 提现负数 / 并发 / 积分重复 / 未校验交易状态 |
| 个人信息 | 越权查看 / 修改 / 删除他人信息 |
| 文件下载 | 越权下载 / 路径遍历 / 文件包含 |
| API 接口 | 未授权访问 / 越权 / 批量调用 / 参数未校验所属用户 |
| 后台管理 | 越权访问后台接口（/admin/api/...） |
| 通知 / 消息 | 越权查看他人通知 / XSS / 越权发送通知他人 |
| 上传文件 | 上传后处理逻辑（如解析 / OSS 后处理）是否有逻辑缺陷 |
| 兑换码 | 兑换码是否可预测 / 未绑定用户 / 未过期 / 重复使用 |

### 10.2 实用技巧

```
1. **两账号对比**：用两个权限不同的账号同时操作，**互操作**
2. **前端 / 后端版本对比**：同一接口 /api/v1 / /api/v2 对比
3. **改请求方法**：GET / POST / PUT / DELETE / PATCH
4. **改 Content-Type**：JSON / form-urlencoded / multipart / XML
5. **重复提交**：同一请求多次提交
6. **并发提交**：竞争条件
7. **参数替换**：把自己的 ID 替换成他人的
8. **跳过中间步骤**：直接访问 step3 / final / step
9. **前端校验 × 前端未做校验**：绕过前端验证
10. **观察响应差异**：观察响应状态码 / 响应体差异来判断是否成功

### 10.3 Burp 实用技巧

- **Compare (Comparer)**：比较两账号请求与响应差异
- **Autorize**：自动替换 cookie / session 后重放请求
- **Backslash Powered Scanner**：自动扫描
- **Burp Suite Pro Active Scans**
- **Param Miner**：发现隐藏参数
