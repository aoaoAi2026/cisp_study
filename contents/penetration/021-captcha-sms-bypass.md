# 验证码绕过与短信/邮箱轰炸测试

> 验证码（图形 / 短信 / 邮件 / 滑块 / 行为）是防护体系的最后一道防线，也是业务逻辑漏洞高发区。本文系统梳理验证码绕过的常见手法，以及短信 / 邮箱轰炸的测试流程。

## 1. 图形验证码的典型漏洞

### 1.1 验证码前端生成 / 本地校验

这是最常见也最容易被忽视的漏洞。验证码图片由前端 JS 动态生成、或服务端只返回答案在响应里：

```javascript
// 前端生成，答案在 JS 变量里
var captcha = Math.floor(Math.random() * 9000 + 1000);
document.cookie = "captcha=" + captcha;
```

**检测方法**：F12 在 Response / Cookie / Response Header / HTML `meta` / `<script>` 中搜索 `captcha`、`code`、`verify`、`answer`、`validate` 等关键字。

### 1.2 万能验证码（后门）

开发阶段可能残留了万能验证码，如 `000000`、`8888`、`admin`、`1111`：

```
POST /login
username=admin&password=xxx&captcha=000000
```

### 1.3 验证码在响应中明文返回

```
HTTP/1.1 200 OK
Set-Cookie: captcha=4219        ← 答案在 Cookie 里
X-Captcha: 4219                 ← 答案在 Header 里

响应体里也可能直接带：
{"captcha":"4219"}
```

### 1.4 验证码永久有效 / 可多次使用

- 提交一次错误后验证码未失效；
- 同一张验证码图片能多次提交；
- 直接抓已有验证码的请求，重放 n 次仍然通过。

### 1.5 验证码过于简单 / 可预测

- 纯数字 4 位：1 万次可爆破；
- 基于时间戳：`captcha=md5(timestamp)`；
- 算法可预测：`captcha = current_minute % 10000`。

### 1.6 图形验证码可被 OCR / 机器学习识别

- 简单字符 + 无干扰线 → `tesseract`、ddddocr 轻松识别；
- 滑块 / 行为验证码 → 可以通过自动化脚本模拟滑动轨迹；
- 点选式验证码 → 可训练模型识别图片语义。

```python
# ddddocr 示例（仅演示，未授权场景禁止使用）
import ddddocr
ocr = ddddocr.DdddOcr(show_ad=False)
with open("captcha.png", "rb") as f:
    code = ocr.classification(f.read())
print(code)
```

### 1.7 空验证码 / 任意值通过

- `captcha=` 留空；
- `captcha=1` 任意值；
- 直接删掉整个 captcha 字段；
- 改成 `captcha=xxxx&captcha=yyyy` 参数污染。

### 1.8 验证码的 session 绑定错误

- 用户 A 的验证码能用于用户 B；
- 攻击者请求验证码 → 用验证码以受害者身份提交；
- 验证码仅绑定 IP 不绑定 session → 攻击者只需同出口 IP。

## 2. 短信 / 邮件验证码的典型漏洞

### 2.1 短信验证码的安全要点

- 是否限制 **单手机号 / 单 IP / 单设备** 的发送频率？
- 验证码是否有 **有效期**（通常 5~15 分钟）？
- 验证码校验 **失败后是否立即失效**？
- 是否存在 **白名单手机号**（测试号）？
- 是否存在 **并发绕过**（同一验证码重放 n 次）？

### 2.2 短信轰炸

短信轰炸 = 在短时间内让目标手机收到大量短信。常见原因：

- 短信发送接口无频率限制；
- 频率限制仅前端做（JS countdown）；
- 频率限制只限制手机号 / 或 IP，但攻击者可：
  - 切换 Header：`X-Forwarded-For`、`X-Real-IP`；
  - 切换手机号：在数字末尾加 `+86`、`0086`、`-`、空格；
  - 切换 API 路径：`/api/sms/send`、`/api/v2/sms/send`、`/api/login/sms`；
  - 同产品多个短信接口（登录、注册、改密、注销、邀请）都能触发；
  - POST 中带多个手机号：`phone=138...,139...`；
  - JSON 数组：`["13800000000","13800000001"]`。

### 2.3 短信轰炸检测脚本（概念版，仅演示判断逻辑）

```bash
#!/bin/bash
TARGET="https://target.com/api/sms/send"
PHONE="13800000000"

for i in $(seq 1 5); do
  resp=$(curl -sk -X POST "$TARGET" \
              -H 'Content-Type: application/json' \
              -d "{\"phone\":\"$PHONE\"}" -m 5 -w "\nHTTP:%{http_code}")
  echo "$i: $resp"
done
# 如果 5 次连续成功发送，说明频率限制缺失
```

### 2.4 邮箱轰炸与短信轰炸类似

- `/api/mail/send` 无频率限制；
- 邮箱白名单、邮箱格式绕过：`attacker@mail.com`、`attacker+tag1@mail.com`、`attacker%2bfoo@mail.com`；
- `Content-Type` 从 `application/json` 改 `application/x-www-form-urlencoded`，参数解析差异；
- 邮件模板可注入：`email=<script>alert(1)</script>@mail.com`，部分邮件客户端会执行。

## 3. 验证码 / 短信的业务逻辑链攻击

### 3.1 密码重置业务流程漏洞

```
1. 输入手机号/邮箱
2. 发送验证码
3. 输入验证码
4. 设置新密码

典型漏洞：
- 第 2 步返回的响应里包含验证码；
- 第 3 步的 `uid` / `phone` 参数可被修改为他人；
- 第 4 步可直接跳过第 3 步（请求 `POST /reset/step3` 直接带新密码）；
- 第 3 步验证码可无限次尝试（暴力破解 4 位数字只需 1 万次）；
- 重置链接的 token 可预测 / 未绑定用户 / 可复用。
```

### 3.2 短信验证码并发绕过

用 Turbo Intruder / Burp Intruder 并发发送 `n` 条带有同一验证码的请求：

- 如果后端校验后 **不立即失效** 验证码 → 验证码被多次使用；
- 这可以用于 **并发支付**、**并发领取优惠券**、**并发创建管理员** 等场景；
- 也可用于 **验证码爆破**（并发提交 1 万个 4 位数字）。

### 3.3 验证码复用 / 跨账户

- 用户 A 申请的验证码，可用于用户 B 的密码重置；
- 攻击者只需要先给自己手机申请一次验证码，再用该验证码修改目标用户密码；
- 根本原因：验证码仅绑定 session 但 **未绑定 user_id**。

### 3.4 短信回调 / 第三方短信服务签名绕过

部分业务在客户端调用短信发送的 SDK，其中 sign 参数可通过 jadx / frida 提取：

```
POST /sms/send
phone=13800000000&sign=md5(phone+appkey+secret)

# 如果 secret 在 APP 中可提取，攻击者可自算合法签名
# 如果 sign 仅用 appkey，无需 secret，则接口完全可以自调用
```

## 4. 典型测试清单

### 4.1 图形验证码

- [ ] 验证码答案是否在 HTML / Cookie / Header / Response 中出现？
- [ ] 是否存在万能验证码（0000 / 8888 / admin）？
- [ ] 验证码使用一次后是否失效？
- [ ] 验证码是否有时效？（建议 ≤ 5 分钟）
- [ ] 是否支持重复提交同一验证码？
- [ ] 是否可用 OCR / ddddocr 识别？
- [ ] 留空、任意值、删除 captcha 字段是否都能通过？
- [ ] 是否存在 "仅前端校验"（如 429 或错误提示是 JS 弹的）？

### 4.2 短信 / 邮件验证码

- [ ] 单手机号 / 单 IP 发送频率是否限制？（建议 ≤ 1 条/分钟，≤ 10 条/天）
- [ ] 频率限制是否只在前端？
- [ ] 是否可通过 `X-Forwarded-For` / `X-Real-IP` 伪造 IP 绕过限制？
- [ ] 是否有多个 API 路径 / 多个业务场景（登录/注册/忘记密码/注销）分别可触发短信？
- [ ] 短信模板是否支持注入 HTML / JS？
- [ ] 是否支持同时给多个号码发（数组、逗号分隔）？
- [ ] 验证码错误一次后是否立即失效？
- [ ] 验证码过期是否有时间限制？
- [ ] 验证码绑定 session / user / phone 是否正确？
- [ ] 是否存在 `+86`、`0086`、空格、`-` 等格式绕过限制？

### 4.3 业务逻辑链

- [ ] 密码重置流程中 `uid` / `phone` / `email` 参数是否可被篡改？
- [ ] 步骤间是否跳过某一步（如 step2 → step4）？
- [ ] 重置 token 是否可预测 / 是否绑定用户？
- [ ] 并发发送同一验证码是否造成多次使用？
- [ ] 是否能用自己申请的验证码重置他人密码？
- [ ] 短信/邮件回调的签名是否可伪造？

## 5. 绕过姿势速查表

| 场景 | 绕过姿势 |
|------|----------|
| 频率限制（仅前端） | 直接请求接口 / 用 curl / 用 Intruder |
| 频率限制（基于 IP） | `X-Forwarded-For: 127.0.0.1` / `X-Real-IP` / 多出口 IP |
| 频率限制（基于手机号） | `+86138` / `0086138` / `138 0000 0000` / JSON 数组 |
| 图形验证码 | ddddocr / 万能验证码 / 答案在 Cookie / 留空 |
| 短信验证码 | 并发重放 / 多接口触发 / 白名单手机号 / 参数污染 |
| 验证码绑定错误 | 用自己的验证码重置他人账号 |
| 密码重置链 | step2 → step4 跳步骤；`uid` 篡改 |
| 邮箱轰炸 | `name+tag@mail.com` / 数组 / 多个接口 |

## 6. 防御建议

1. **验证码**：服务端生成并校验，绑定 session + user；一次使用后立即失效；设置有效期（≤ 5 分钟）；OCR 强度足够或采用滑块/行为式；
2. **频率限制**：单 IP + 单手机号 + 单账号 + 单设备 多维限流；服务端限流，前端只是体验；
3. **业务链**：每步严格校验上一步的 token；token 必须关联 user_id；
4. **短信/邮件签名**：如果客户端需要调用签名，sign 必须带 secret（且 secret 不在客户端硬编码）；
5. **日志与告警**：单 IP 3 分钟内发送 > 10 条短信 → 告警；
6. **安全培训**：让产品/开发理解业务逻辑链与并发的风险。

## 7. 实战案例简述

```
某网站密码重置流程：
1. POST /api/reset/send {phone:138...} → 发送短信
2. POST /api/reset/verify {phone:138, code:1234} → 校验，返回 {"token":"abc"}
3. POST /api/reset/password {token:"abc", newPwd:"123456"}

漏洞：
- 第 2 步的 phone 字段是客户端提交的；把 phone 改成他人号码 + 用自己收到的 code，
  服务端仅校验 code 正确性，不校验 phone 和 code 的绑定。
- 攻击者可以：先用自己手机申请验证码 → 把 phone 改成目标 → 验证通过拿到 token → 重置目标密码。

修复：
- 服务端把 code 与 phone 绑定，校验 code 时同时校验 code 是否属于当前 phone / session。
```

验证码与业务逻辑链的组合往往会产生非常严重的漏洞，务必在每个项目中重点测试。
