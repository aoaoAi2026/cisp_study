# 渗透测试报告撰写规范（模板 + 示例）

## 1. 报告结构

1. 封面
2. 摘要（Executive Summary）
3. 测试范围与方法
4. 风险评级说明
5. 漏洞详情列表
6. 漏洞明细（每个漏洞一章节）
7. 附录：工具清单、原始证据链、补充资料

## 2. 摘要示例（非技术向）

> 本次渗透测试针对贵单位 `XXX` 业务系统（域名：`target.com`）进行，自 yyyy-mm-dd 开始，至 yyyy-mm-dd 结束。测试期间共发现 X 个漏洞，其中 **Y 个严重**、**Z 个高危**，主要风险集中在未授权访问、SQL 注入、弱口令与文件上传四个方面。攻击者可利用这些漏洞获得后台管理员权限、读取或篡改数据库数据，并在极端情况下控制服务器。建议贵单位在 **30 天内**完成严重/高危漏洞修复，并在修复后安排复测。

## 3. 风险评级说明

采用 **CVSS v3.1** 基础指标进行评分，并映射为：

| 等级 | CVSS 得分 | 修复时限 |
|------|----------|---------|
| 严重 Critical | 9.0 – 10.0 | 7 天 |
| 高 High | 7.0 – 8.9 | 30 天 |
| 中 Medium | 4.0 – 6.9 | 90 天 |
| 低 Low | 0.1 – 3.9 | 下一版本修复 |
| 信息 Info | 0 | 建议 |

## 4. 漏洞详情章节模板

```
【漏洞标题】
漏洞等级：严重 / 高 / 中 / 低 / 信息
CVSS v3.1：AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H (9.8)
发现时间：yyyy-mm-dd
复测状态：未修复 / 已修复 / 接受风险
影响资产：https://target.com/page?id=

【漏洞描述】
XX 应用 /XX 接口未对参数 id 进行严格过滤，导致攻击者可构造
恶意 SQL 语句，实现任意数据库读取、写入与命令执行。

【复现步骤 / PoC】
1. 访问：https://target.com/page?id=1'
2. 观察到页面返回 MySQL 语法错误
3. 使用 sqlmap 验证：
   sqlmap -u "https://target.com/page?id=1" --dbs --batch
4. 成功列出数据库：information_schema / cms / ...

【影响 / 风险】
- 可读取数据库中所有用户信息、订单、凭据
- 可写入 WebShell，后续获取服务器权限
- 可作为内网横向入口

【修复建议】
1. 使用预编译 SQL（PreparedStatement / 参数化查询）
2. 输入白名单校验：仅允许数字
3. 数据库账号最小权限：撤销 FILE、SUPER 等权限
4. 上线 WAF 规则 + 虚拟补丁

【参考资料】
- OWASP SQL Injection: https://owasp.org/www-community/attacks/SQL_Injection
- PortSwigger: https://portswigger.net/web-security/sql-injection

【证据 / 截图】
[截图 1：注入点错误]
[截图 2：sqlmap 列出数据库]
[截图 3：读取到的敏感数据（已脱敏）]
```

## 5. 漏洞列表（表格）

| 编号 | 漏洞名称 | 位置 | 等级 | 状态 |
|------|---------|------|------|------|
| 001 | SQL 注入 | `/page?id=` | 严重 | 未修复 |
| 002 | 后台弱口令 | `/admin/login` | 高 | 未修复 |
| 003 | 文件上传绕过 | `/upload` | 高 | 未修复 |
| 004 | 越权查看订单 | `/api/order?id=` | 中 | 未修复 |
| 005 | 目录浏览 | `/static/` | 低 | 未修复 |
| 006 | 泄露服务器版本 | 响应头 Server | 信息 | 未修复 |

## 6. 复测报告

复测报告应在原报告基础上：
- 列出原漏洞编号、修复状态、复测结果
- 新增发现的漏洞（如整改引入新问题）
- 给出最终结论与签字

## 7. 撰写规范

- **证据**：每张截图需标注时间戳、URL；敏感数据需脱敏
- **可复现**：每一个漏洞需提供复现步骤，测试人员外的工程师也能按步骤复现
- **修复建议可落地**：区分「紧急临时方案」与「长期根治方案」
- **语言**：客观、专业，不含攻击性/指责性词汇
- **安全**：敏感信息（含 PoC / 漏洞截图）需加密或单独交付

## 8. 常见错误示例（需避免）

- 「该系统存在大量漏洞」→ 模糊，无数据支撑
- 「建议加强安全」→ 无具体可执行动作
- 截图无时间、无 URL → 无法证明真实性
- PoC 直接贴敏感数据 → 违反保密要求
