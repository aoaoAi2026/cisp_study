# 漏洞管理闭环：发现 - 修复 - 验证全流程

护网实战中，**漏洞**是红队最喜欢的"入口"。大量案例显示，被红队最终攻破的系统，其"第一滴血"往往是**一个 1 个月前就已知、但迟迟未修复** 的高危漏洞（比如常见的 Thinkphp RCE、Log4j、Spring Framework、Shiro 反序列化、Exchange SSRF、Redis 未授权等）。漏洞管理的本质是把"发现漏洞 → 确认风险 → 推动修复 → 复测验证 → 长期回顾"这条链路彻底跑通。

## 1. 漏洞管理全流程总览

一个完整的漏洞管理闭环应包含以下 6 个阶段：

```text
   ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
   │   1 发现      │ ──▶│   2 定级      │ ──▶│  3 分派修复  │
   │（扫描/情报/  │     │（CVSS/业务  │     │（工单+SLA）   │
   │  红队/上报）  │     │  影响评估）   │     │              │
   └──────────────┘     └──────────────┘     └──────────────┘
          ▲                                                  │
          │                                                  ▼
   ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
   │  6 持续度量  │ ◀───│ 5 豁免/归档  │ ◀───│   4 验证     │
   │（看板/复盘）  │     │（不可修时 ）  │     │（复测+签名）│
   └──────────────┘     └──────────────┘     └──────────────┘
```

## 2. 阶段一：漏洞发现（Source of Truth）

### 2.1 漏洞发现来源

| 来源 | 说明 | 频率 |
|------|------|------|
| **自动化扫描**（漏扫器 / 平台） | 覆盖 Web / 主机 / 数据库 / 中间件 | 每周 / 每次版本发布后 |
| **红队攻击 / 护网演练** | 真人模拟攻击，发现"逻辑漏洞 / 弱配置 / 默认口令" | 半年一次 + 护网前突击 |
| **威胁情报订阅**（CVE / CNVD / CNNVD） | 高危 0day / Nday 漏洞通告 | 实时 |
| **渗透测试 / 代码审计** | 针对核心系统的深度人工检测 | 每年一次核心系统 |
| **外部通报**（监管 / 厂商安全公告） | 监管单位主动通知或甲方自查清单 | 按通知频率 |
| **员工 / 外部白帽上报**（漏洞奖励计划） | 白帽子 / 内部员工通过指定渠道上报 | 持续 |

### 2.2 护网前的"突击扫描"策略

护网开始前 **14~30 天**，建议做一轮：

- 全量资产覆盖扫描（资产清单必须是"活的"）
- 针对关键系统的 Web 深度扫描（含认证扫描）
- 针对中间件 / 组件版本的 SBOM 扫描（识别 Log4j 等问题）
- 针对所有对外 IP 的端口最小化核查

```bash
# 示例：对外资产端口核查（思路）
# 资产清单 in /tmp/asset_list.txt（每行一个 IP/域名）
```

## 3. 阶段二：漏洞定级与风险评估

### 3.1 定级基础（CVSS + 业务影响）

单一 **CVSS 分数** 并不足以判断真实风险，必须叠加：

- **业务重要性**：核心系统 / 一般系统 / 办公系统
- **可利用性**：是否有公开 EXP / 工具化脚本 / 被在野利用
- **暴露面**：是否对公网 / 仅内网 / 仅办公网暴露
- **当前是否存在真实攻击**：威胁情报是否命中
- **已有防护**：WAF / IPS / RASP 是否能拦截
- **历史数据**：过去是否被成功利用过

### 3.2 实用分级（建议）

| 等级 | 典型场景（示例） | 修复 SLA |
|------|------------------|----------|
| P0 紧急 | 已被在野利用的严重 RCE（如 Log4j 早期），且我方资产暴露 | 24 小时 |
| P1 高危 | 严重远程代码执行 / SQL 注入，有公开 EXP | 7 天 |
| P2 中危 | 一般 RCE / 越权 / 信息泄露 / 暴力破解成功 | 30 天 |
| P3 低危 | 弱加密、配置不规范、信息暴露但无直接影响 | 90 天 |
| P4 提示 | 规范建议、编码风格问题、版本过旧但无直接利用路径 | 下一版本 |

## 4. 阶段三：分派修复（工单系统 + SLA）

### 4.1 关键要素

- **工单必须明确"责任人和资产"**：不能说"某某系统有个漏洞"，而要说"IP 10.1.2.33（业务：门户）责任人：张 XX，部门：门户组"。
- **工单状态可跟踪**：新建 → 已分派 → 修复中 → 复测中 → 关闭 / 豁免。
- **逾期自动升级**：SLA 到期仍未修复，自动升级到上一级主管。

```yaml
# 示例：工单字段
vuln_id:          V-2025-0918-0017
title:            "门户系统 /xxx 路径存在 SQL 注入（手工测试确认）"
asset:            10.1.2.33 (prod-web-03)
severity:         P1
cvss:             8.5
responsible:      张 XX (部门：门户产品组)
reported_by:      外部红队
discovered_at:    2025-09-18 11:20
sla_deadline:     2025-09-25 18:00
status:           修复中
evidence:
  - screenshot_1.png
  - payload_sample.txt
reproduce_steps: >
  1) 访问 /api/v1/products?id=1' and sleep(3)-- -
  2) 响应延迟约 3 秒，确认存在时间盲注
```

### 4.2 升级机制

- 到 SLA 50%：邮件提醒责任人
- 到 SLA 80%：短信 + 直属主管
- 到 SLA 120%：升级 CISO / CTO + 纳入部门考核

## 5. 阶段四：验证与复测

漏洞"改完"不等于"安全"。必须做：

| 验证动作 | 谁执行 | 产出 |
|----------|--------|------|
| **同工具复测** | 安全团队 | 原始扫描器再次扫描，漏洞应消失 |
| **手工验证** | 安全团队 / 第三方 | 复现原 payload，确认不可利用 |
| **其他相关资产复核** | 安全团队 | 同类系统是否也有相同漏洞（批量修复） |
| **代码层 / 配置层审计** | 安全 + 研发 | 确认修复方案未引入新风险 |
| **补丁回滚预案** | 运维 | 如补丁导致业务异常，可快速回滚 |

```text
【关闭标准】必须全部满足：
  1) 漏洞在原暴露面不可被利用（手工验证）
  2) 同类系统已完成批量修复或确认不受影响
  3) 有书面 / 工单的修复说明与签字（或电子记录）
  4) 相关防护规则（WAF/IDS）已同步更新（可选）
  5) 已完成一次生产回归测试，业务未受影响
```

## 6. 阶段五：豁免与归档

不是所有漏洞都能按 SLA 修复。要建立**正式的豁免流程**，避免"安全团队单方面拍板"。

```text
【漏洞豁免申请模板】
- 漏洞编号：V-2025-xxxx
- 申请人：业务负责人（必须是总监级或以上）
- 无法修复原因：
  □ 厂商已停服，无补丁
  □ 修复将导致核心业务停产超过 X 小时，需大版本升级
  □ 其他：（详细说明）
- 已采取的补偿性措施：
  [ ] 该资产已从公网撤回，仅内网暴露
  [ ] 防火墙 / WAF / RASP 已配置临时拦截规则
  [ ] 已开启主机白名单 / EDR 增强监测
  [ ] 已增加蜜罐 / 高交互监测
  [ ] 其他：
- 豁免有效期：3 个月 / 6 个月 / 永久（一般不允许永久）
- 签字：业务负责人 + 安全负责人
```

## 7. 阶段六：持续度量与看板

### 7.1 核心指标

| 指标 | 说明 | 目标值（示意） |
|------|------|----------------|
| 漏洞数量 | 按 P0~P4 统计 | P0=0、P1 持续下降 |
| 平均修复时长（MTTR） | 漏洞生命周期（从发现到关闭） | P1 < 7 天 |
| SLA 达成率 | 在 SLA 内关闭的漏洞占比 | ≥ 90% |
| 重开率 | 被关闭后因"修复不彻底"被重开的比例 | < 5% |
| 豁免占比 | 豁免漏洞 / 总漏洞 | < 5% |
| 扫描覆盖率 | 已扫资产 / 总资产 | ≥ 95% |

### 7.2 护网可视化看板示例

```text
# 护网看板（Grafana / 同类）
- 指标卡：P0 P1 P2 P3 P4 当前数量（与上周同比）
- 趋势图：近 30 天按严重等级的漏洞数量变化
- 热力图：按业务系统 / 部门分布的 P0+P1 漏洞热力图
- TopN：Top10 风险资产（按漏洞数加权）
- SLA：各部门 SLA 达成率
- 红队：护网期间红队新发现漏洞、已修复比例
```

## 8. 常见坑点与反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| "漏洞只扫描，不跟进" | 报告越积越多，没人看 | 每个漏洞必须有责任人 + SLA |
| "修完就算了" | 修复不彻底，被再次利用 | 必须复测 + 归档 |
| "安全团队自己修" | 责任错配，研发/运维完全不参与 | 安全团队"管"，业务研发/运维"修" |
| "只看数量不看质量" | 大量低危被优先修复，高危拖 SLA | 按 P0→P1→P2 优先级修复 |
| "豁免无期限" | "豁免一次，永远不修" | 豁免必须设有效期，并要求补偿性措施 |
| "一次扫完就结束" | 新漏洞、新资产不断出现，静态管理无效 | 周期性扫描 + 资产清单持续更新 |

## 9. 总结：漏洞管理的真正目标

**漏洞管理不是"把漏洞修完"**（现实中永远修不完），而是做到：

- **P0/P1 漏洞必须 100% 在 SLA 内处置**（修复或补偿性防护）
- **暴露面持续收敛**，尤其是公网暴露资产
- **研发流程左移**，把漏洞发现提前到代码 / CI 阶段
- **业务负责人真正承担起责任**，安全不再是"安全团队一个人的事"

当护网开始时，一支真正把漏洞管理跑通的团队，面对的将是：**P0=0、P1 极低、暴露面收敛、已知风险都有补偿性防护** —— 这正是"被攻破概率"最低的安全状态。
