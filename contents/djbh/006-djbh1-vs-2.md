# 等保 1.0 vs 等保 2.0 核心差异对比

## 1. 政策与标准演进总览

等保 1.0（以下简称 1.0）以 2007 年《信息安全等级保护管理办法》和 2008 年发布的 GB/T 22239 等基础标准为标志，构建了"五级分类、三级管理"的信息安全等级保护基础框架。等保 2.0（以下简称 2.0）以 2017 年《网络安全法》的正式实施为法律背景，以 2019 年 GB/T 22239-2019、GB/T 28448-2019、GB/T 25070-2019 等一系列新标准发布为标志，将信息安全等级保护升级为**网络安全等级保护**，并作为《网络安全法》第二十一条规定的法定制度。

两者在核心定位上发生了本质变化：1.0 是**行业推荐性管理制度**，2.0 是**法定强制性义务**，第三级及以上系统的运营者必须定期开展等级测评并向公安机关备案，未履行义务将承担相应法律责任。

## 2. 保护对象扩展：从"信息系统"到"网络空间"

### 2.1 保护对象对比

| 维度 | 等保 1.0 | 等保 2.0 |
|------|----------|----------|
| 保护对象 | 信息系统（计算机信息系统） | 网络、信息系统、数据、应用、云平台、物联网、工业控制系统、移动互联网 |
| 定级粒度 | 以信息系统为单位 | 以安全保护等级对象（网络/系统/平台/云/物/工控）为单位 |
| 云计算 | 无专门要求 | 明确云平台扩展要求（GB/T 22239-2019 附录 A） |
| 移动互联网 | 无专门要求 | 明确移动互联扩展要求（附录 B） |
| 物联网 | 无专门要求 | 明确物联网扩展要求（附录 C） |
| 工业控制系统 | 无专门要求 | 明确工业控制系统扩展要求（附录 D） |
| 使用大数据的系统 | 未提及 | 纳入"大数据"安全管理扩展要求（附录 E） |

### 2.2 一个定级判定的例子

某企业同时部署了"门户网站"、"内部 OA"、"数据仓库"和"基于云平台的生产系统"四个系统。

- 在 1.0 时期：仅需对上述四个"信息系统"分别定级
- 在 2.0 时期：需要额外关注云平台自身的等保等级（通常不得低于承载业务系统等级），并在基本要求基础上叠加"云平台扩展要求"

## 3. 技术要求结构变化："一中心三要素"

等保 1.0 的技术要求按"物理安全、网络安全、主机安全、应用安全、数据安全"五类横向划分；2.0 改为"安全物理环境（S1）、安全通信网络（S2）、安全区域边界（S3）、安全计算环境（S4）、安全管理中心（S5）"的**"一中心三要素"** 架构：

```
           ┌──────────────────────────────────────┐
           │            安全管理中心 (S5)         │
           │ 系统管理 / 审计管理 / 安全管理      │
           └────────┬─────────────┬───────────────┘
                    │ 集中管控    │
   ┌────────────────┴───┬─────────┴──────────────────┐
   │ 安全通信网络 (S2)  │  安全区域边界 (S3)         │
   │ 网络架构、可信验证 │  边界防护、入侵防范、审计   │
   └────────────┬───────┴───────┬────────────────────┘
                │               │
          ┌─────┴───────────────┴──────┐
          │   安全计算环境 (S4)         │
          │ 主机/应用/数据/身份与访问   │
          └────────────┬───────────────┘
                       │
          ┌────────────┴───────────────┐
          │     安全物理环境 (S1)       │
          └────────────────────────────┘
```

新增控制点示例：

- S4.1 身份鉴别：第三级明确要求"采用口令、密码技术、生物技术等两种或两种以上组合的鉴别技术"（即双因子认证 MFA）
- S3.2 边界防护：明确"应能够对非授权设备私自联到内部网络的行为进行检查或限制"
- S5.1 安全管理中心：1.0 只在第四级要求，2.0 **第三级即要求集中管控**

## 4. 管理要求与评价指标对比

| 项目 | 等保 1.0 | 等保 2.0 |
|------|----------|----------|
| 管理结构 | 安全管理制度、机构、人员、建设、运维（5 大类） | 安全管理制度（M1）、机构（M2）、人员（M3）、建设（M4）、运维（M5）（相同骨架，内容扩充） |
| 测评周期 | 三级每年一次；二级每两年一次 | 三级**每年一次**；二级每两年一次，要求不变 |
| 关键指标 | 符合项数量 | 引入"高风险项一票否决"，存在高风险项不能通过 |
| 密码要求 | 仅模糊提及 | 明确引入《密码法》背景，要求使用经过国家密码管理部门认可的商用密码 |
| 个人信息保护 | 未提及 | 明确要求遵循《个人信息保护法》相关原则 |
| 外包服务 | 未明确 | 明确服务商管理与外包安全协议 |

## 5. 典型升级改造清单（从 1.0 合规到 2.0 合规）

### 5.1 技术改造清单

1. **身份鉴别升级**：关键系统增加双因素认证（短信/Token/生物特征）
2. **安全管理中心**：部署统一日志审计平台（SIEM）、堡垒机（运维审计）、集中账号管理
3. **边界防护增强**：下一代防火墙（NGFW）/IPS、上网行为管理、DLP、EDR
4. **可信验证**：服务器、网络设备启动过程进行可信验证（可选第三级，要求级第四级）
5. **数据与传输**：全站 HTTPS、关键数据加密存储（国密 SM2/SM4）
6. **云平台**：评估使用中的云服务（IaaS/PaaS/SaaS）是否具备相应等保证明
7. **密码应用**：使用合规商密产品，开展密码应用安全性评估（密评）

### 5.2 管理改造清单

1. 重新梳理并定级：确认保护对象列表是否完整（含云、移动、工控）
2. 补充《个人信息保护法》相关制度与隐私政策
3. 补充密码管理制度（密钥管理、密评周期）
4. 补充外包/云服务商安全协议与责任条款
5. 完善应急演练流程（年度至少一次完整演练）
6. 完善漏洞管理与补丁管理制度
7. 完善安全审计制度（日志保存 ≥ 6 个月）

## 6. 升级建议

```
Step 1  对照 22239-2019 基本要求 + 扩展要求，重新评估定级对象清单
Step 2  生成 Gap Analysis，识别高风险项
Step 3  优先治理高风险项（MFA、日志审计、边界防护）
Step 4  升级管理制度体系，补齐 2.0 新增要求
Step 5  引入商密方案并启动密评
Step 6  联系测评机构启动正式测评
```

> 等保 2.0 的核心变化不是"条款数量增多"，而是**理念升级**：从合规驱动走向风险驱动，从单点保护走向体系防御，从静态防护走向持续运营。
