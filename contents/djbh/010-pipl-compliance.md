# 个人信息保护法与等保测评：合规衔接与落地

## 1. 两部法律的定位与关系

2021 年 11 月 1 日起《中华人民共和国个人信息保护法》（PIPL）正式施行。与《网络安全法》《数据安全法》共同构成中国数据与个人信息保护领域的"三驾马车"。

| 法律 | 定位 | 与等保的关系 |
|------|------|-------------|
| 网络安全法（2017） | 网络空间的基础性法律 | 第二十一条规定"国家实行网络安全等级保护制度"，等保 2.0 的法律依据 |
| 数据安全法（2021） | 数据处理与安全的基本法律 | 要求重要数据识别、分类分级、出境安全评估；与等保 2.0 的数据安全要求协同 |
| 个人信息保护法（2021） | 个人信息权益保护的专门法律 | 对"可识别自然人"的信息进行全生命周期保护，在等保 2.0 中体现为"个人信息保护"控制点 |

等保测评中"个人信息保护"控制点（见 GB/T 22239-2019 S4 安全计算环境部分）是 PIPL 在技术领域的具体落地。

## 2. PIPL 关键条款梳理

### 2.1 个人信息与敏感个人信息定义

| 分类 | 定义 | 典型场景 |
|------|------|---------|
| 个人信息 | 以电子或其他方式记录的与已识别或可识别自然人有关的各种信息，不包括匿名化处理后的信息 | 姓名、手机号、身份证号、住址、邮箱、生物识别信息、健康信息、金融账号、位置信息、浏览记录 |
| 敏感个人信息 | 一旦泄露或非法使用，容易导致自然人人格尊严受到侵害或人身财产安全受到危害的个人信息 | 生物识别、宗教信仰、特定身份、医疗健康、金融账户、行踪轨迹等 + 不满十四周岁未成年人的个人信息 |

### 2.2 核心权利义务框架

```
个人信息主体的权利：
  1. 知情同意权       ── 可撤回同意
  2. 查询权 / 复制权   ── 可要求复制个人信息
  3. 更正 / 补充权     ── 发现不准确可要求更正
  4. 删除权（被遗忘权）── 可要求删除
  5. 解释说明权       ── 可要求说明规则
  6. 救济权           ── 投诉、诉讼

个人信息处理者的义务：
  1. 采取必要措施保障个人信息安全
  2. 不得过度收集个人信息
  3. 明确告知处理规则并取得同意
  4. 开展个人信息保护影响评估（PIA）
  5. 指定个人信息保护负责人（DPO）
  6. 敏感个人信息：单独同意 + 必要性 + 安全评估
  7. 跨境传输：安全评估 / 标准合同 / 认证
```

## 3. 等保 2.0 中与个人信息保护相关的控制点

GB/T 22239-2019 中 S4（安全计算环境）的数据安全类控制点与 PIPL 直接相关：

| 控制点编号 | 要求摘要 | 对应 PIPL 要求 |
|-----------|----------|---------------|
| S4.3.3 数据保密性 | 应采用加密或其他有效措施实现传输和存储过程中数据的保密性 | PIPL 第二十一条（加密与去标识化） |
| S4.3.4 数据完整性 | 应采用校验技术或密码技术保证重要数据在传输和存储过程中的完整性 | PIPL 第二十一条 |
| S4.3.5 数据备份恢复 | 应对重要数据进行备份和恢复，且备份数据与主数据应分离存储 | PIPL 第二十一条 |
| S4.3.6 剩余信息保护 | 应保证鉴别信息所在的存储空间在释放或再分配前被完全清除 | PIPL 第四十七条（删除权） |
| S4.3.7 个人信息保护 | 应仅采集和保存业务必需的最小范围个人信息；应不允许非授权访问个人信息 | PIPL 第六条（最小必要） + 第二十八条（敏感个人信息） |
| M4.5.1 软件开发 | 应保证开发环境与实际运行环境安全隔离 | PIPL 第二十一条（安全措施） |
| M5.8.2 数据处置 | 应按规定对多余、过期重要数据进行处理 | PIPL 第四十七条（删除权） |

## 4. 落地实操：个人信息处理合规清单

### 4.1 个人信息资产梳理

第一步：画出个人信息处理流程全图

```
收集（注册/填表/SDK） → 传输（HTTPS/脱敏） → 存储（加密/最小化）
     │
     ▼
使用（业务处理、分析） → 共享/委托处理（签署协议） → 跨境传输（评估/合同）
     │
     ▼
删除/销毁（业务终止或用户请求）
```

第二步：形成《个人信息资产清单》

| 字段 | 内容示例 |
|------|---------|
| 数据类型 | 手机号（个人信息）、身份证号（敏感）、指纹（生物识别） |
| 处理目的 | 登录验证、实名认证、防欺诈 |
| 存储位置 | 用户中心数据库（IP 10.x.x.x） |
| 存储格式 | 哈希（SM3）、加密（SM4）、脱敏（138****0000） |
| 保留期限 | 账号存续期 + 180 天 |
| 访问主体 | 客服系统、订单系统、审计系统 |
| 出境情况 | 否 / 是（香港 AWS，已进行标准合同备案） |

### 4.2 敏感字段处理建议

```python
# 脱敏示例（中国手机号 + 身份证号）
def mask_phone(phone: str) -> str:
    return phone[:3] + "****" + phone[-4:] if phone and len(phone) >= 7 else phone

def mask_id_card(id_card: str) -> str:
    if not id_card or len(id_card) < 10:
        return id_card
    return id_card[:6] + "********" + id_card[-4:]

# 存储：使用 SM4 字段级加密
#   key 存在 KMS / 密码机，不在应用代码中
# 展示：脱敏展示，后端不再返回明文
# 分析：匿名化（去掉可识别字段）后用于统计分析
```

### 4.3 隐私政策要点

隐私政策（个人信息处理规则）须满足 PIPL 第十七条：

- 以显著方式、清晰易懂的语言真实、准确、完整地告知
- 明示处理目的、方式、范围、存储期限
- 明示个人行使权利的方式和程序
- 明示处理者名称、联系方式
- 不得包含"免责任条款"规避法定义务

### 4.4 个人信息保护影响评估（PIA）

PIPL 第五十五条规定，以下情形应当事前进行个人信息保护影响评估：

1. 处理敏感个人信息
2. 利用个人信息进行自动化决策
3. 委托处理、向第三方提供、公开个人信息
4. 跨境提供个人信息
5. 其他对个人权益有重大影响的处理活动

评估须形成《个人信息保护影响评估报告》，保存不少于 3 年。

## 5. 等保测评中个人信息保护常见问题整改

| 常见问题 | 风险等级 | 整改建议 |
|---------|---------|---------|
| 前端明文传输手机号/身份证号 | 高风险 | 强制 HTTPS；传输前脱敏或使用国密 SM4 |
| 数据库敏感字段明文存储 | 高风险 | 启用字段级加密（SM4）；禁止开发人员以生产数据调试 |
| 应用日志输出明文身份证号 | 高风险 | 脱敏后再写入日志；日志访问须审计 |
| 过度收集个人信息（要求填住址却用于营销） | 高风险 | 遵循最小必要原则 |
| 无隐私政策 / 隐私政策含糊其辞 | 中风险 | 重写隐私政策，征得明示同意 |
| 外包客服直接访问明文个人信息 | 中风险 | 最小授权 + 脱敏 + 操作审计 |
| 个人信息出境未评估 | 高风险 | 启动数据出境安全评估 / 标准合同 / 认证 |
| 未指定个人信息保护负责人（DPO） | 中风险 | 正式任命并公开联系方式 |

## 6. 违规后果与应急

| 违规情形 | 法律责任（节选） |
|---------|----------------|
| 违反 PIPL 处理个人信息 | 责令改正，警告，没收违法所得；并处一百万元以下罚款 |
| 情节严重 | 责令暂停相关业务、停业整顿、吊销相关业务许可、五千万元以下或者上一年度营业额百分之五以下罚款 |
| 责任人 | 一万元以上十万元以下罚款 |
| 构成犯罪 | 依法追究刑事责任 |

等保测评中发现个人信息保护相关高风险项时，应同时启动个人信息保护合规专项整改，避免被网信部门、市监部门双轨问责。

> PIPL 合规不是等保测评的"附属要求"，而是**与等保 2.0 平行的独立法定义务**。将"最小必要、加密存储、脱敏展示、最小授权、PIA 评估、DPO 负责"这六大原则落到实处，是通过等保测评 + 满足 PIPL 合规的共同路径。
